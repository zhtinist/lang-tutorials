# LLD 面试框架 · LLD Interview Framework

> 低层设计面试考的是**把模糊需求落成可扩展的类设计**。固定 5 步：澄清 → 实体 → 类与接口 → 实现 → 异常/并发。模式是结果，不是起点。

---

## 一、什么是 LLD？

| | HLD（高层） | LLD（低层） |
|--|-------------|-------------|
| 焦点 | 服务、存储、流量、可用性 | 类、接口、方法、关系 |
| 产出 | 架构图、API、数据流 | 类图、核心代码骨架 |
| 时长 | 常 45–60 分钟 | 常 30–45 分钟 |

固定 5 步，改写成可执行清单：

---

## 二、五步框架

```
① 澄清需求（Clarification）
② 识别实体（Entities）
③ 类设计 + 接口（Design）
④ 实现关键方法（Implement）
⑤ 异常 / 并发 / 扩展（Edge & Scale-out of design）
```

### ① 澄清需求（约 5 分钟）

先别画类。问清楚：

| 类别 | 示例问题 |
|------|----------|
| 功能范围 | 要支持哪些核心用例？MVP 是什么？ |
| 用户角色 | 司机/管理员/会员？权限差异？ |
| 约束 | 单机还是分布式？要持久化吗？ |
| 规模（轻问） | 大概多少实体？实时性要求？ |
| 非功能 | 线程安全？计费精度？ |

**口诀**：范围不说清，设计必返工。

输出物：3–6 条 **Functional Requirements** + 可选 NFR 列表。

### ② 识别实体（约 5 分钟）

启发式：

```
名词 → 候选类
动词 → 候选方法
形容词/规则变化 → 策略/状态/配置
```

例：停车场

| 名词 | 类 |
|------|-----|
| 停车场、楼层、车位 | ParkingLot, Floor, Spot |
| 车 | Vehicle（及子类） |
| 票据、费率 | Ticket, FeeStrategy |

过滤：不要把「时长」「金额」都升级成类；值对象/基本类型即可。

### ③ 类设计 + 接口（约 10–12 分钟）

1. 画 ASCII / 白板类图（见 [uml-basics.md](uml-basics.md)）
2. 标组合关系（谁拥有谁）
3. 找 **axis of change（变化轴）** → 抽接口：
   - 计费规则会变 → `FeeStrategy`
   - 分配车位策略会变 → `SpotAssignmentStrategy`
   - 通知渠道会变 → `Notifier`
4. SOLID 自检（见 [solid-principles.md](solid-principles.md)）
5. **组合优于继承**：车型可用枚举+策略，不一定深继承树

### ④ 实现关键方法（约 10 分钟）

挑 2–3 个主路径写代码骨架：

- 停车场：`park` / `leave`
- 图书馆：`borrow` / `return_book`
- 限流：`allow_request`

展示：数据结构选择、接口调用、错误返回。不必写满所有 getter。

### ⑤ 异常 / 并发 / 扩展（约 5 分钟）

主动提，体现完整度：

| 话题 | 例子 |
|------|------|
| 异常 | 满位、重复车牌、书已借出 |
| 并发 | 同一车位双分；限流器锁/原子 |
| 扩展 | 多层、预约、会员价、分布式限流 |
| 持久化 | 内存 → 仓储接口（DIP） |

## 三、时间预算（45 分钟版）

| 阶段 | 时间 | 交付 |
|------|------|------|
| 澄清 | 5 min | 需求列表 |
| 实体 | 5 min | 名词表 |
| 类图 | 10–12 min | 关系 + 接口 |
| 编码 | 10–12 min | 核心方法 |
| 边界与扩展 | 5 min | 并发/异常 |
| Buffer | 3–5 min | 答追问 |

30 分钟场：压缩编码，类图与主路径伪代码优先。

---

## 四、评分维度（面试官在看什么）

| 维度 | 高分表现 | 低分表现 |
|------|----------|----------|
| 需求理解 | 主动划 MVP | 上来就写代码 |
| 抽象 | 变化处有接口 | 到处 `if typ` |
| 关系 | 组合/依赖清晰 | God Class |
| 正确性 | 主路径+边界 | 只讲 happy path |
| 沟通 | 边画边讲 | 沉默猛写 |
| 演进 | 能说扩展点 | 「再加 if」 |

## 五、模式触发信号表

把需求里的「味道」映射到模式（先设计后贴标签）：

| 你听到 / 想到 | 考虑模式 |
|---------------|----------|
| 多种计费/限流/排序算法 | Strategy |
| 对象行为随状态变 | State |
| 接入第三方不一致 API | Adapter |
| 加日志/重试/权限包装 | Decorator / Proxy |
| 子系统调用很烦 | Facade |
| 树形菜单/文件系统 | Composite |
| 创建逻辑分叉多 | Factory Method |
| 构造参数爆炸 | Builder |
| 事件通知多人 | Observer |
| 审批/过滤器链路 | Chain of Responsibility |
| 全局唯一配置 | Singleton（慎用） |
| 固定流程不同步骤 | Template Method |

## 六、口述开场模板（可背）

> 「我按五步来：先确认范围与用例，再抽实体，然后画类图并在变化点定义接口，接着实现 park/leave 主路径，最后讨论满位、并发占位和后续扩展。当前假设是单机内存、支持多种车型与按小时计费，可以吗？」

## 七、变化轴练习（核心肌肉）

对每个系统问：

```
1. 什么最可能变？（费率、分配规则、渠道……）
2. 谁在用这些变化？（哪个类会 if？）
3. 能否把变化封进接口，调用方只依赖抽象？
```

这与 OCP / DIP 一致。

### 反例 → 正例

```python
# 反例：变化轴写死在服务里
def fee(hours: int, kind: str) -> float:
    if kind == "hourly":
        return hours * 5
    if kind == "daily":
        return -(-hours // 24) * 40  # 向上取整
    raise ValueError(kind)

# 正例：FeeStrategy 接口 + 注入
```

## 八、常见翻车点

| 翻车 | 改法 |
|------|------|
| 一上来写 Singleton 数据库 | 先用例与实体 |
| 继承树表达「有一个」 | 改组合 |
| 类图没有方法 | 补动词 |
| 代码与类图对不上 | 先改图再写 |
| 忽略并发 | 至少说清临界区 |
| 模式堆砌 | 每个模式说清动机 |

---

## 九、练习法

1. 选 [exercises.md](exercises.md) 一题，限时 35 分钟自己走 5 步
2. 对照案例文（停车场/图书馆/限流）看差异
3. 只改进「变化轴」与接口，不追求代码行数

推荐顺序：Parking Lot → Library → Rate Limiter → Elevator → Ticket Booking。
