# 基础语法:与 C/C++ 对照

Java 的表层语法刻意向 C 家族靠拢——`{}` 划分块、`;` 结尾语句、`if/for/while` 的写法、算术与逻辑运算符——这是 Gosling 团队的明确设计目标:让 C/C++ 程序员几乎不需要重新学习记号系统,注意力可以直接放到语义差异上。本章因此不逐条重复这些通识,只讲 Java 在语法与语言结构上与 C/C++ 真正分道扬镳的地方,以及每处分歧背后的设计取舍。

先给出一段体现常见记号的代码,建立整体印象:

```java
public class Basics {
    public static void main(String[] args) {
        int age = 20;
        double price = 9.9;
        String name = "Ada";
        boolean ok = true;

        int sum = age + 5;
        System.out.println(sum);                              // 25
        System.out.println("name=" + name + ", age=" + age);  // name=Ada, age=20

        if (age >= 18) {
            System.out.println("adult");
        } else {
            System.out.println("minor");
        }

        for (int i = 0; i < 3; i++) {
            System.out.println("i=" + i);   // i=0  i=1  i=2
        }
    }
}
```

注释用 `//` 与 `/* */`,与 C 相同。下面逐项讨论结构性差异。

---

## 一、一切都在 class 里;程序入口

C 允许自由函数和全局变量;Java **没有自由函数、没有全局变量**——一切代码必须挂在类(或接口/枚举/记录)之下。

```java
public class App {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}
```

**为什么这样设计**:Java 的对象模型继承自 Smalltalk 一脉的"纯粹面向对象"理念,刻意不留自由函数和可变全局状态这两个逃逸口。这不只是审美选择——全局可变状态是多线程程序里最难推理的东西(任何线程都可能在任何时刻读写它,给静态分析、并发正确性、后来的安全沙箱模型都带来麻烦);强制一切代码归属某个类,相当于强制它归属某个明确的命名空间和访问控制边界。Java 早期设计目标之一是可安全地从网络加载并运行不受信任的代码(applet 模型),一个没有全局可写状态的语言明显更容易做到这一点。

对照 C 的 `int main(int argc, char** argv)`:
- **`main` 返回 `void`**:JVM 启动器用反射按固定签名 `public static void main(String[])` 查找入口,只关心"能否找到并调用",不关心业务返回值——进程退出码是另一个独立通道,要设置显式调用 `System.exit(int)`。把"函数返回值"和"进程退出状态"解耦,使得入口签名可以对所有程序保持统一,不必为"这个程序想返回什么值"发明五花八门的 `main` 签名。
- **参数是 `String[] args`,不含程序名**:C 的 `argv[0]` 是程序名,这是操作系统 `exec` 系列调用传参的历史遗留(需要程序自己知道被以什么名字调用,例如 busybox 式多路复用二进制)。JVM 已经知道要运行哪个类,不需要再把这个信息塞进用户可见的参数数组,`args` 里因此只有真正的调用参数,数组自带 `length`,不需要额外的 `argc`。
- **必须 `static`**:JVM 加载完类、还没有创建任何实例时就要能调用入口,`static` 方法不依赖实例存在,这是"先有类、后有实例"这一生命周期顺序的直接推论(实例方法的调用逻辑上必须先有一个对象)。

> JDK 21 起以预览形式提供了省略 class 包裹的精简入口写法,并在 **JDK 25(JEP 512)转为正式特性**——但这只是给教学/脚本场景的语法糖,`public static void main(String[])` 背后的加载—反射—调用机制并未改变,正式项目仍以经典写法为主。

## 二、包与 import:没有预处理器

C/C++ 用 `#include` 做**文本粘贴**:头文件里的声明在编译期被原样复制进每个引用它的源文件,`#define` 宏做纯文本替换。这套机制的代价是显著的——同一份声明在每个编译单元里重复解析、宏展开不遵守作用域规则(全局生效、可被任意重定义)、`#ifdef` 导致同一份源码在不同编译条件下实际上是不同的程序、忘记 include guard 会重复包含。

Java 用**包(package)+ import** 取代这一整套机制:

```java
package com.example;

import java.util.List;          // 引入单个类
import java.util.*;             // 引入整个包(不递归子包)
import static java.lang.Math.*; // 静态导入

public class Demo { /* ... */ }
```

`import` **不做任何文本插入**,它只是告诉编译器"某个简单名对应哪个全限定名",真正的类型解析发生在编译期查符号表、运行期由类加载器按需加载 `.class` 文件(见[「运行模型」](01-运行模型.md))。

**为什么能省掉头文件和前向声明**:根源在于 `.class` 文件本身是**自描述**的——它包含完整的方法签名、字段类型、常量池等元数据,编译器或类加载器可以直接从这份二进制里读出一个类的完整契约,不需要像 C++ 那样额外维护一份人写的 `.h` 声明去"预告"实现里有什么。这也是为什么类之间可以互相引用、循环依赖——编译器按类为单位解析符号,不要求"用之前必须已经见过声明"的严格文本顺序,而 C++ 单遍编译模型必须先看到声明才能使用,这是两种完全不同的编译与链接模型的自然结果:C++ 编译到目标文件后由链接器做符号拼接;Java 编译到带完整元数据的字节码后由类加载器在运行时按需解析。

省略宏还带来一个经常被低估的好处:**没有宏就没有宏这一整类 bug**——宏无视作用域和类型系统,`#define MAX(a,b) ((a)>(b)?(a):(b))` 这种写法在参数有副作用时会重复求值,而 Java 用普通方法(会被 JIT 内联,见[「运行模型」](01-运行模型.md))替代"函数宏",方法遵守正常的作用域、类型检查和求值顺序规则。常量用 `static final`,平台差异靠运行时判断或构建工具处理,而不是编译期条件展开出"实际上不同的源码"。

## 三、强静态类型;`var` 局部推断

```java
int count = 10;
double ratio = 0.5;
String name = "Ada";
long big = 9_000_000_000L;   // 超 int 范围要加 L 后缀
```

和 C++ 一样是强静态类型:每个变量的类型在编译期确定,类型不匹配编译报错——这是为了让 JIT/AOT 能在编译期而非运行期确定方法调度、内存布局,类型错误在开发阶段而非用户手上暴露。

JDK 10 引入的 `var` 做局部变量类型推断,语义上对应 C++11 的 `auto`:

```java
var count = 10;                        // 推断为 int
var list = new ArrayList<String>();    // 推断为 ArrayList<String>
```

**为什么 Java 到第 10 版才加这个、且只能用在局部变量上**:泛型(Java 5)之后,类型声明经常变得很长(`Map<String, List<Integer>> m = new HashMap<>();`),纯粹是视觉噪音,`var` 是针对这个具体痛点的编译期语法糖——类型仍在编译期定死,`var` 不改变 Java 是静态类型语言这一事实,字节码里就是具体类型,与手写类型名完全等价,运行期零差异。刻意把它限制在"方法内、且必须有初始化表达式"的局部变量范围,是因为字段类型和方法签名是**公开契约的一部分**(既是给调用方看的文档,也是二进制接口的一部分),用 `var` 会让读者/调用方失去查看契约的直接途径;而局部变量纯属实现细节,推断掉不影响任何人对外部契约的理解。这是一条深思熟虑的边界,不是任意限制。

## 四、运算符

绝大多数运算符与 C 相同:算术 `+ - * / %`、关系 `== != < > <= >=`、逻辑 `&& || !`、位运算 `& | ^ ~ << >>`、复合赋值、自增自减,写法照搬即可。真正值得深究的是下面几处分歧,每一处都对应一个具体的设计取舍:

**`if` 条件必须是 `boolean`,不接受整数**:C 里"非零即真",`if (a = b)` 这个把 `==` 误写成 `=` 的经典 bug 之所以能通过编译,是因为赋值表达式的值(`b`)可以隐式转成条件所需的"真值"。Java 要求条件必须是 `boolean`,`int` 不能隐式转 `boolean`,这类笔误在 Java 里直接编译不通过——用类型系统在语言层面根除了一整类 C 程序员都吃过亏的 bug,代价是牺牲了"整数当真值用"的灵活性(比如不能再写 `if (count)` 判断非零)。

**无符号右移 `>>>` 与算术右移 `>>` 分开**:Java 故意不设计 unsigned 整数类型(只有 `byte/short/int/long` 这些带符号类型),以换取更简单、无歧义的溢出与符号扩展规则——不必像 C 那样处理"有符号和无符号混合运算时哪个提升到哪个"这类实现定义行为。但省去 unsigned 类型后,仍然存在"把最高位当纯 bit、不当符号位"的真实需求(位运算、哈希、模拟无符号算术),`>>>` 就是为此专门开的口子:`>>` 补符号位(算术右移),`>>>` 补 0(逻辑右移)。C 里是否补符号位取决于类型本身是否 unsigned;Java 用两个运算符把这个选择显式化。

**整数除零抛异常而非未定义行为**:C 标准把整数除零定为未定义行为,允许不同硬件平台有不同的处理方式(有的直接触发 CPU 异常、有的返回垃圾值)。JVM 作为跨硬件的托管运行时,必须保证同一段字节码在任何平台上行为一致,于是把"除零"定义为**确定性抛出 `ArithmeticException`**,可被 `catch` 处理。这是"语言规范保证可移植的确定性行为"与"贴近硬件、允许平台差异"这两种设计哲学的直接分野。浮点除零则遵循 IEEE-754 本身的语义,得到 `Infinity`/`NaN`,不抛异常——因为 IEEE-754 标准自己就定义了这些值,不需要 JVM 额外兜底。

**移位计数按位宽取模**:`int` 移位时位移量隐式 `& 31`,`long` 按 `& 63`,所以 `1 << 32` 结果是 `1` 而不是 `0`。这同样是"消除平台相关的未定义行为"的例子——C 里移位量超过类型位宽是未定义行为,Java 显式规定了取模规则,任何平台结果都一样。

**`+` 是唯一被"重载"的运算符**:只要有一侧操作数是 `String`,`+` 就变成拼接,由编译器特殊处理(内部常转换为 `StringBuilder.append`,见[「数据类型」](04-数据类型.md))。这是刻意开的**唯一**一个口子——Java 语言层面不支持用户自定义运算符重载(理由见第八节),但字符串拼接是高频到值得为它破例的场景。

实测这几处:

```java
System.out.println(1 << 32);        // 1  (32 & 31 == 0, 即 1<<0)
System.out.println(-8 >> 1);        // -4   算术右移,补符号位
System.out.println(-8 >>> 28);      // 15   逻辑右移,补0

try { int x = 1 / 0; }
catch (ArithmeticException e) { System.out.println(e.getMessage()); } // / by zero
System.out.println(1.0 / 0);        // Infinity(IEEE-754 语义,不抛异常)

System.out.println("sum=" + 1 + 2); // sum=12(从左到右求值,先变成字符串就一路拼接)
System.out.println(1 + 2 + "=sum"); // 3=sum(遇到字符串前仍是普通加法)
```

## 五、字符串:不可变的类,不是字符数组

```java
String s = "hello";
int n = s.length();               // O(1) 字段,不是 strlen 那样扫描找 \0
boolean same = s.equals("hello"); // 比内容用 equals,不是 ==
String up = s.toUpperCase();      // 返回新串,s 本身不变
```

`String` 是类,不是 `char*` 也不是 `char[]`,且**不可变**——任何"修改"方法都返回新对象,原对象内容永不改变:

```java
String s = "hello";
String up = s.toUpperCase();
System.out.println(s);         // hello   原串没变
System.out.println(s == up);   // false   up 是另一个对象
```

**为什么设计成不可变**,这不是随意的选择,而是同时服务于三个目标:
1. **安全共享无需防御性拷贝**:多个引用可以放心指向同一个 `String` 对象而不必担心某处代码悄悄改了内容——这对经常被当作 `HashMap` 键、类名、文件路径、网络地址传递的字符串尤其重要;若字符串可变,理论上会出现"校验时是一个值、使用时已被改成另一个值"的检查—使用时间差漏洞(time-of-check-to-time-of-use)。
2. **可安全缓存哈希值**:不可变对象的哈希值算一次就永远有效,`String.hashCode()` 内部缓存结果,使字符串作为 `HashMap` 键的开销大幅降低。
3. **免锁线程安全**:不可变对象天然线程安全,多个线程读同一个 `String` 不需要任何同步。

代价是清楚的:频繁拼接会不断产生新对象,循环里用 `+` 拼接是 O(n²) 的,这是不可变性的直接代价,解法是用可变的 `StringBuilder`(内部是可增长的 `char`/`byte` 数组,类似手动管理的缓冲区,细节见[「数据类型」](04-数据类型.md))。

## 六、控制台 IO

```java
System.out.println("整行带换行");
System.out.printf("%d 和 %.2f%n", 10, 3.14159);  // 类 printf,%n 是平台无关换行

Scanner sc = new Scanner(System.in);
int x = sc.nextInt();
```

`System.out`/`System.err`/`System.in` 分别是包装 stdout/stderr/stdin 的对象。`printf` 风格的格式化在 Java 里做**运行期类型检查**——格式符与实参类型不符会抛异常,而不是像 C 的 `printf` 那样,参数类型与格式符不符属于未定义行为(这正是 C 历史上一整类"格式化字符串攻击"漏洞的根源:攻击者构造的格式串可以让 `printf` 按错误的类型解释栈上数据,造成信息泄露甚至任意写)。Java 用"类型检查失败就抛异常"替代"类型不符就是 UB",与前面除零、移位等处理思路一致:**能确定性地失败,就不要留不确定的行为**。

## 七、注释与 Javadoc

```java
/**
 * Javadoc 文档注释
 * @param n 输入数量
 * @return 计算结果
 */
public int compute(int n) { return n * 2; }
```

`@param`/`@return`/`@throws` 等标签是编译器和工具链都认识的**语言级约定**,`javadoc` 工具据此生成 HTML API 文档,IDE 也据此显示悬浮提示。C/C++ 没有这一层语言内建的文档系统,Doxygen 一类工具是外部约定,编译器本身并不理解这些注释的结构。把文档格式变成语言生态共识的一部分,换来的是整个生态的 API 文档风格高度统一。

## 八、与 C++ 的结构性差异,及背后的取舍

| 维度 | C++ | Java |
|------|-----|------|
| 继承 | 多继承 | 单继承类 + 多实现接口 |
| 指针 | 显式指针语法 `*`/`&`/`->` | 无显式指针语法,引用是受管指针 |
| 内存释放 | `delete`/`free` 手动 | GC 自动回收 |
| 运算符重载 | 支持 | 不支持(仅 `+` 拼接字符串例外) |
| 自由函数 | 支持 | 无,一切在类里 |
| 数组 | 栈/堆连续内存,无边界检查 | 是带 `length` 的对象,有边界检查 |
| 头文件 | 需要 | 无,无预处理器/宏 |
| 默认参数 | 支持 | 不支持,用方法重载替代 |
| 析构 | RAII 确定性析构 | 无确定性析构,用 try-with-resources |
| `const` | 修饰广泛 | `final`(语义窄得多) |

逐条说明取舍,而不是止于罗列差异:

- **单继承 + 多接口,而非多继承**:C++ 的多继承会遇到菱形继承问题——同一个基类经由两条路径被继承两次时,基类成员到底是共享一份还是各自一份需要 `virtual` 继承之类的额外机制来消歧,规则复杂且容易出错。Java 的解法是把"继承实现"和"继承类型契约"拆开:类只能单继承(避免了实现层面的菱形问题),但可以实现任意多个接口(纯契约,默认没有状态,JDK 8 起允许默认方法后才引入了新的、但范围小得多的冲突消解规则)。这是"牺牲一部分表达力换取规则简单性"的经典取舍。
- **不支持运算符重载**:C++ 允许几乎任何运算符被重载,这带来了表达力(比如矩阵用 `*` 直接相乘),但也是长期被诟病的复杂度来源——`a + b` 可能悄悄调用一个开销巨大、有副作用的自定义函数,读代码时无法从记号本身判断发生了什么。Java 选择"方法调用永远是方法调用"这一可预测性:`a.equals(b)`、`bigInt.add(other)` 都写成显式方法,代价是啰嗦,换来的是"看到什么记号就知道大致会发生什么"的确定性。
- **无确定性析构,用 try-with-resources 替代 RAII**:C++ 的析构在作用域退出时确定性调用,这依赖于对象的生命周期能被静态确定。Java 的 GC 为了能做分代回收、可能移动对象、多线程并发回收,把对象生命周期的终止时间交给运行时决定,不再是确定性的——`finalize()` 机制因此已被废弃。但文件句柄、网络连接这类**非内存资源**仍然需要确定性关闭,Java 用 try-with-resources(见[「错误处理与资源管理」](09-错误处理与资源管理.md))把"退出作用域就关闭"这一 RAII 的核心价值,在不依赖确定性析构的前提下重新实现出来。
- **数组永远做边界检查**:C 数组访问越界是未定义行为,这是内存安全问题的重灾区。Java 数组在语言层面就是带 `length` 字段的堆对象,每次下标访问都做边界检查,越界抛 `ArrayIndexOutOfBoundsException` 而不是读写到不属于该数组的内存。代价是每次访问多一次比较,JIT 通常能消除循环内可证明安全的冗余检查,把运行时代价压到接近零。

## 小结

- **语法记号**:`{}`、`;`、`if/for/while`、算术与逻辑运算符与 C 一致,可直接照搬;`var`(JDK 10+)提供局部类型推断但不改变静态类型的本质。
- **一切在类里**:无自由函数、无全局变量,源于纯面向对象设计与安全沙箱的历史目标;`main` 必须 `static` 是"类先于实例存在"这一加载顺序的直接结果。
- **无预处理器**:`.class` 文件自描述完整签名,使 import 只是名字解析而非文本插入,省去头文件、前向声明、宏,也就避免了宏带来的整类 bug。
- **几处运算符分歧**(`if` 需 boolean、`>>>`、除零抛异常、移位取模)背后是同一条原则:能确定性地报错,就不留未定义行为。
- **`String` 不可变**是为共享安全、哈希缓存、免锁并发三个目标联合服务的设计,代价是拼接需要 `StringBuilder`。
- 与 C++ 的结构性差异(单继承多接口、无运算符重载、无确定性析构、数组强制边界检查)每一处都是"简单性/安全性"与"表达力/极致性能"之间的具体取舍,不是任意省略。
