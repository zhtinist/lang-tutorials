# 控制流

Java 的分支和循环写法与 C 高度相似（`if/for/while/switch` 可直接照抄），本章聚焦**差异点**与**字节码层面的机制**：switch 的三种形态与其字节码实现、增强 for 背后的迭代器、模式匹配、文本块。

## 一、if

和 C 基本一样，唯一硬性差异：**条件必须是 `boolean`**（见[「基础语法」](02-基础语法.md)）。`if (n)` 这种"非零即真"在 Java 编译不过，必须写 `if (n != 0)`。好处是根除了 `if (a = b)` 误写赋值的经典 bug——`if (a = b)` 里 `a = b` 若非 boolean 类型直接编译错误。

字节码上，`if` 就是条件比较指令（`if_icmpge`、`ifnull`、`ifeq` 等）加跳转，和[第 1 章](01-运行模型.md)看到的一样，没有惊喜。

## 二、switch：三种形态 + 字节码

switch 一共有三种写法：传统 switch、`->` 表达式 switch、模式匹配 switch，一次比一次解决更具体的问题。

### 2.1 传统 switch：fall-through 陷阱依旧

和 C 一模一样：不写 `break` 会**贯穿（fall-through）**到下一个 case。

```java
int day = 3;
switch (day) {
    case 1:
    case 2:
    case 3:
        System.out.println("工作日前半");
        break;                       // 不写就贯穿，和 C 同坑
    case 6:
    case 7:
        System.out.println("周末");
        break;
    default:
        System.out.println("其它");
}
```

传统 switch 支持的类型：整型（及可拆箱到 int 的包装类）、`char`、`enum`、以及 **`String`**（C 不支持 switch 字符串）。String 的 switch 由编译器降级为"先比 `hashCode` 定位再 `equals` 确认"两级判断实现。

### 2.2 表达式 switch（`->`，JDK14+）：无贯穿、可返回值

新语法用 `->`，**不贯穿、无需 break**，还能作为**表达式求值**：

```java
int day = 3;
String kind = switch (day) {
    case 1, 2, 3, 4, 5 -> "工作日";       // 多标签逗号并列
    case 6, 7          -> "周末";
    default            -> "非法";
};                                        // 注意作为表达式，末尾有分号

// 需要多条语句时用块 + yield 返回值
int score = switch (grade) {
    case 'A' -> 100;
    case 'B' -> { int base = 80; yield base + 5; }  // yield 交出块的值
    default  -> 0;
};
```

要点：
- `->` 分支**执行完自动跳出**，没有 fall-through，从语法上消灭了忘写 break 的 bug。
- 作为表达式时，编译器要求**穷尽所有可能**（对 enum 覆盖全部枚举值，或提供 `default`），否则编译错误——比 C 更安全。
- 多语句分支用 `{}` 块，用 `yield` 交出值（`return` 在这里会返回整个方法，别混淆）。

**为什么要专门引入表达式 switch（JEP 361，JDK 14 转正）**：传统 switch 有两个长期被诟病的问题。其一是 fall-through 默认贯穿——`break` 是"防止贯穿"的手段而非"进入下一分支"的手段，语义上反直觉，每个 case 忘写一次 `break` 就是一个真实 bug，这类失误在生产代码里反复出现，多到成为静态检查工具的标配规则。其二是传统 switch 只是**语句**，不能作为表达式直接产出一个值——想按某个值分支算出一个结果，得先声明一个变量、在每个 case 里赋值、还要在每条路径上都记得 `break`，样板代码量与逻辑复杂度不成比例，还容易漏掉某个分支忘赋值。表达式 switch 一次性解决这两个问题：`->` 语法让每个分支默认不贯穿（要贯穿必须显式用块与旧语法），同时整个 switch 可以直接作为表达式赋值给变量或作为返回值，编译器还强制穷尽性检查，堵住"漏写某个分支"这类疏漏。这不是语法糖式的美化，而是把"分支求值"这一常见需求从"语句 + 手工赋值 + break 纪律"降级为一个内建、结构性防错的表达式。

### 2.3 模式匹配 switch（JDK21+）：按类型分派

switch 可以对**类型**做匹配并绑定变量，配合 sealed 类型能穷尽检查：

```java
Object obj = ...;
String desc = switch (obj) {
    case Integer i when i > 0 -> "正整数 " + i;   // when 是守卫条件
    case Integer i            -> "非正整数 " + i;
    case String s             -> "字符串长度 " + s.length();
    case null                 -> "空值";          // 可显式匹配 null
    default                   -> "其它类型";
};
```

这相当于把一串 `if (obj instanceof Integer i) ... else if (obj instanceof String s) ...` 收拢成结构化写法，是 Java 向"代数数据类型 + 模式匹配"靠拢的产物。与 C 的 switch 完全不同层次。

下面是可运行的完整版，`describe` 对几种实际类型分别分派：

```java
public class PatternSwitch {
    static String describe(Object obj) {
        return switch (obj) {
            case Integer i when i > 0 -> "正整数 " + i;
            case Integer i            -> "非正整数 " + i;
            case String s             -> "字符串长度 " + s.length();
            case null                 -> "空值";
            default                   -> "其它类型";
        };
    }
    public static void main(String[] args) {
        System.out.println(describe(42));     // 正整数 42
        System.out.println(describe(-3));     // 非正整数 -3
        System.out.println(describe("hello")); // 字符串长度 5
        System.out.println(describe(null));   // 空值
        System.out.println(describe(3.14));   // 其它类型
    }
}
```

注意 case 的**顺序有意义**：带 `when` 守卫的 `Integer i when i > 0` 必须排在无守卫的 `Integer i` 之前；若把无守卫的放前面，它会先吃掉所有 `Integer`，编译器会因后面的 case "已被前面覆盖、不可达"而直接报错。

**为什么要引入按类型分派的模式匹配 switch（JEP 441，JDK 21 转正）**：在此之前，"按运行时类型分派"这种极常见的需求只能写成一串 `if (obj instanceof Integer) { Integer i = (Integer) obj; ... } else if (obj instanceof String) { String s = (String) obj; ... }`。这段代码有两层重复劳动：先判断类型，再手动强转一次——`instanceof` 已经在运行时确认了类型，强转在逻辑上是多余的信息重复，纯粹是语言表达力不足导致的样板；而且这一串 `if-else` 没有编译器帮你检查"是否覆盖了所有可能类型"，遗漏一个分支只会在运行时才暴露成未处理的默认路径。模式匹配 switch 把"类型判断 + 绑定变量 + 可选值匹配"整合进一个结构，配合 sealed 类型（见[「面向对象与类型系统」](08-面向对象与类型系统.md)）还能让编译器**穷尽检查**——如果你为一个 sealed 接口的每个实现类都写了 case，穷尽性由编译器验证，不需要手写 `default` 兜底，新增一个实现类而忘记处理会在编译期直接报错而不是运行时悄悄漏判。这本质上是 Java 在向函数式语言里"代数数据类型 + 模式匹配"这一成熟机制靠拢，用于解决"按结构/类型分支"这类此前只能靠啰嗦且不安全的 `instanceof`+强转链手写的问题。

### 2.4 字节码：tableswitch vs lookupswitch

switch 在字节码里有两条指令，编译器按 case 值的分布**自动选择**——这是 switch 常比等价 if-else 链更快的原因：

| 指令 | 适用 | 机制 | 复杂度 |
|------|------|------|--------|
| `tableswitch` | case 值**连续或接近连续** | 用值作**跳转表索引**直接跳 | O(1) |
| `lookupswitch` | case 值**稀疏** | 存 (值,偏移) 有序对，**二分查找** | O(log n) |

`tableswitch` 就像 C 编译器对密集 case 生成的跳转表（jump table）：把 `case 1/2/3` 编成一张连续偏移表，运行时 `目标 = 表[值 - 低界]`，一次寻址完成，无需逐个比较。`lookupswitch` 用于 `case 1/100/10000` 这种稀疏值，存排序好的键值对做二分。可用 `javap -c` 观察：

```
javap -c YourClass     // 会看到 tableswitch { ... } 或 lookupswitch { ... }
```

连续小整数 case → tableswitch（最快）；稀疏 → lookupswitch。这提示：性能敏感场景，让 case 值尽量连续有利于生成跳转表。

下面用一个真例对比两条指令。`dense` 的 case 连续（0~3），`sparse` 的 case 稀疏（1/100/10000）：

```java
public class SwitchByteCode {
    static int dense(int x) {   // case 0,1,2,3 连续
        switch (x) {
            case 0: return 10;  case 1: return 11;
            case 2: return 12;  case 3: return 13;
            default: return -1;
        }
    }
    static int sparse(int x) {  // case 1,100,10000 稀疏
        switch (x) {
            case 1: return 10;  case 100: return 11;
            case 10000: return 12;
            default: return -1;
        }
    }
    public static void main(String[] args) {}
}
```

`javac SwitchByteCode.java && javap -c SwitchByteCode.class` 的关键输出（实测）：

```
static int dense(int);
     0: iload_0
     1: tableswitch   { // 0 to 3
                   0: 32
                   1: 35
                   2: 38
                   3: 41
             default: 44
        }

static int sparse(int);
     0: iload_0
     1: lookupswitch  { // 3
                   1: 36
                 100: 39
               10000: 42
             default: 45
        }
```

`dense` 编成 `tableswitch`——一张 `0..3` 的连续偏移表，运行期直接按索引跳；`sparse` 编成 `lookupswitch`——排好序的键值对做二分。同样是 switch，case 值的分布决定了用哪条指令。

## 三、循环

`for` / `while` / `do-while` 三者与 C 语义完全一致，不赘述：

```java
for (int i = 0; i < n; i++) { ... }
while (cond) { ... }
do { ... } while (cond);   // 至少执行一次
```

Java 还有一种 C 没有的**增强 for（for-each）**，专门用来"遍历一个东西的每个元素"，不用自己管下标：

```java
int[] arr = {1, 2, 3};
for (int x : arr) { ... }              // 遍历数组

List<String> list = List.of("a","b");
for (String s : list) { ... }          // 遍历集合
```

`for`/`while`/`do-while` 的字节码就是循环判断 + `goto` 回边（见[第 1 章](01-运行模型.md)的 `goto 4`）。回边计数正是 HotSpot 判定"热循环"触发 JIT 的依据之一。

**增强 for** 的底层机制取决于被遍历者：
- **数组**：编译器降级为普通带下标的 for 循环（用 length 控制、下标访问），**没有**迭代器对象，零额外开销。
- **实现了 `Iterable` 的集合**：编译器降级为调用 `iterator()` 取迭代器，循环 `hasNext()`/`next()`。等价展开：

```java
// for (String s : list) {...} 实际编译成 ↓
for (Iterator<String> it = list.iterator(); it.hasNext(); ) {
    String s = it.next();
    ...
}
```

对照 C++ 的 range-based for（`for (auto& x : container)` 靠 `begin()/end()`）几乎一一对应：Java 靠 `Iterable.iterator()` + `Iterator` 接口。注意用增强 for 遍历集合时**不能在循环体里增删该集合**，否则迭代器检测到结构变化抛 `ConcurrentModificationException`（要删除得用迭代器自己的 `remove()` 或用 `removeIf`）。集合与迭代器详见[「复合数据结构」](07-复合数据结构.md)。

## 四、带标签的 break / continue

Java 没有 `goto`（是保留字但不用），但提供**带标签的 break/continue**来跳出/继续**外层**循环，覆盖了 C 里用 goto 跳出多重循环的常见场景：

```java
outer:
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        if (matrix[i][j] == target) {
            found = true;
            break outer;        // 直接跳出外层两重循环
        }
        if (skip) continue outer;  // 直接进入外层的下一轮
    }
}
```

`break outer` 结束标签所指的整个循环，`continue outer` 跳到该循环的下一次迭代。比 C 用 flag 变量或 goto 更结构化、更可读。字节码上仍是 `goto` 到相应偏移。

## 五、三元运算符

```java
int max = (a > b) ? a : b;
```

与 C 相同。注意 Java 里三元表达式的两个分支要有**兼容的类型**（会做类型提升 / 装箱），比如 `cond ? 1 : 2.0` 整体是 double。混用包装类与基本类型时可能触发自动拆箱导致意外 NPE，留意。

## 六、instanceof 模式匹配

传统 `instanceof` 判类型后还要手动强转，模式匹配（JDK16+）一步到位并绑定变量：

```java
// 传统写法
if (obj instanceof String) {
    String s = (String) obj;    // 还得再强转
    System.out.println(s.length());
}

// 模式匹配写法
if (obj instanceof String s) {  // 判定成功即把 obj 绑定到 s（已是 String 类型）
    System.out.println(s.length());
}

// 可在同一条件里继续用绑定变量
if (obj instanceof String s && s.length() > 3) { ... }
```

`s` 的作用域是"能确定它一定是 String 的地方"（流敏感作用域）。这与[第 8 章](08-面向对象与类型系统.md)的类型系统、以及本章的模式匹配 switch 是同一套语言演进方向。相当于把 C++ 的 `if (auto* p = dynamic_cast<String*>(obj))` 做得更顺手。

## 七、文本块（Text Block，JDK15+）

多行字符串字面量，免去手写 `\n` 和拼接，类似其它语言的 heredoc：

```java
String json = """
        {
          "name": "Ada",
          "age": 36
        }
        """;                    // 三引号包裹；编译期确定缩进基线并去掉公共前导空白
```

要点：
- 它仍然是一个普通的 `String`（不可变、进常量池），只是**书写方式**不同，运行期与普通字符串无异。
- 编译器按结束 `"""` 的缩进作为基线，剥掉每行的公共前导空白，方便你在代码里对齐缩进而不污染内容。
- 适合嵌 JSON / SQL / HTML。对照 C 里只能用一堆 `"..."` 相邻拼接或行尾 `\`，文本块干净得多。

## 小结

- `if` 条件必须 boolean，杜绝 `if(a=b)` 类误写。
- switch 三形态一次比一次解决更具体的问题：传统 switch（有 fall-through，源于 C 式默认贯穿的历史包袱）、`->` 表达式 switch（消灭忘写 break 的 bug，还能作为表达式并强制穷尽性检查）、模式匹配 switch（用编译器验证的穷尽性取代啰嗦且不安全的 `instanceof`+强转链）；String/enum 均可 switch。
- 字节码 **tableswitch（连续值→跳转表，O(1)）vs lookupswitch（稀疏值→二分，O(log n)）**，编译器自动选，可用 `javap -c` 查看，这是 switch 有时比等价 if-else 链更快的原因。
- 增强 for：数组降级为下标循环、集合靠 `Iterable.iterator()`；遍历中改集合抛 `ConcurrentModificationException`。
- 带标签 break/continue 替代跨层 goto。
