# 并发模型

C/C++ 里并发靠 `std::thread`、`pthread`、原子、`memory_order` 手动控制。Java 把这一整套抽象进 **JMM(Java Memory Model)** 和 `java.util.concurrent`。**本章依然是先讲「怎么用」——开线程、加锁、用线程池,这些 API 不需要你懂底层就能直接抄着写;再讲「底层与 C 对比」——JVM 层面线程怎么映射到内核、JMM 到底约束了什么、锁在对象头里如何升级,以及 JDK21 的虚拟线程原理。** 只想能跑起来的话读每节前半段;想弄懂为什么会有数据竞争、为什么要加锁,再看后半段。

---

## 先会用:开个线程速览

开一个线程、等它跑完,是 Java 并发最基础的动作:

```java
public class SimpleThreadDoc {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            System.out.println("子线程在跑");
        });
        t.start();     // 启动新线程,异步执行
        t.join();      // 阻塞当前线程,直到 t 跑完
        System.out.println("子线程结束,主线程继续");
    }
}
// 子线程在跑
// 子线程结束,主线程继续
```

`new Thread(Runnable)` 接一个 lambda 当作线程要跑的任务;`start()` 真正把它扔给系统去异步执行;`join()` 则是"等它"——不调用的话主线程可能先于子线程结束。

如果两个线程会同时改同一个变量,直接改会出错(下一节详细说为什么),但先记住"解法"很简单:把改变量的代码块用 `synchronized` 包起来,同一时刻只允许一个线程进入:

```java
public class SimpleSyncDemo {
    static int counter = 0;

    static synchronized void increment() {   // synchronized 方法:同一时刻只有一个线程能执行它
        counter++;
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable job = () -> {
            for (int i = 0; i < 100_000; i++) increment();
        };
        Thread t1 = new Thread(job);
        Thread t2 = new Thread(job);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter); // 200000,每次运行都一样
    }
}
// counter = 200000
```

两个线程各自把 `counter` 加 10 万次,只要 `increment()` 加了 `synchronized`,结果就总是精确的 20 万,不会少。至于为什么"不加就可能少"、`synchronized` 内部怎么实现的,是下面要讲的内容。

这些够你写出正确的多线程小程序了。下面开始进入底层——先理解"问题从哪来",再逐个看每个工具怎么解决它。

## 开始深入:多线程为什么会出问题

这几节讲的是:**为什么多个线程读写同一个变量会出问题**。不涉及具体 API 语法,是纯概念,不需要任何预备知识——理解了它们,后面 `volatile`、`synchronized`、`Atomic*` 为什么存在也就自然清楚了。

## 线程创建与生命周期

### 怎么用

创建线程要跑的"任务",Java 里有三种写法:

```java
// 1. 继承 Thread(不推荐:单继承被占用,任务与线程耦合)
class MyThread extends Thread {
    @Override public void run() { System.out.println("run in " + getName()); }
}

// 2. Runnable:无返回值、无受检异常,最常用
Runnable r = () -> System.out.println("runnable");

// 3. Callable<V>:有返回值 V,可抛受检异常,配合 Future 使用(见 java.util.concurrent 一节)
Callable<Integer> c = () -> 42;

public class ThreadDemo {
    public static void main(String[] args) throws Exception {
        Thread t1 = new MyThread();
        t1.start();            // start() 让 JVM 新开一个线程去跑 run()
        Thread t2 = new Thread(r);
        t2.start();
        t1.join();             // 当前线程阻塞直到 t1 终止,类似 pthread_join
        t2.join();
    }
}
// 两个线程谁先打印不确定,但 join() 保证 main 一定等它们都跑完再结束
```

> 关键区分:`start()` 让 JVM 新建一个线程并异步回调 `run()`;如果你手滑直接调用 `run()`,那只是一次普通的方法调用,代码在当前线程里**同步**执行,根本没有开新线程。一个 `Thread` 对象只能 `start()` 一次,再调第二次会抛 `IllegalThreadStateException`。

### 底层与 C 对比

Java 的 `Thread` 在 HotSpot 上默认是 **1:1 映射到内核线程**(OS thread),与 C++ `std::thread` 一样重。调用 `start()` 会触发一次 `pthread_create`,每个线程默认分到约 1MB 栈空间(`-Xss` 可调)。也就是说,`new Thread(...).start()` 在系统层面的代价,和 C 里手写 `pthread_create` 是同一个数量级——线程不是免费的,开几千个平台线程就会感到吃力(这也是本章最后"虚拟线程"要解决的问题)。

`Thread.State` 六态,与 OS 的线程状态不是一一对应:

```
    NEW --start()--> RUNNABLE <----+
                      |  ^          | notify/超时/锁获得
    进入同步块拿不到锁 |  |          |
                      v  |          |
                   BLOCKED          |
    wait()/park()   |               |
                      v             |
    WAITING / TIMED_WAITING --------+
                      |
                   TERMINATED（run 返回）
```

- `RUNNABLE` 同时涵盖"正在 CPU 上跑"和"就绪等调度",Java 不区分(OS 才区分)。
- `BLOCKED`:等待 `synchronized` 监视器锁。
- `WAITING`:`Object.wait()`、`Thread.join()`、`LockSupport.park()` 无限等待。
- `TIMED_WAITING`:带超时的上述调用、`Thread.sleep(ms)`。

## Java 内存模型 JMM

JMM 是一份**规范**(JLS 第17章),不是硬件。它定义多线程读写共享变量时,什么值是合法可见的——这正是上一节"为什么会出问题"的正式答案。类比 C++11 内存模型,但 JMM 更早(2004, JSR-133)且默认更强。

### 主内存与工作内存

JMM 抽象出:每个线程有**工作内存**(working memory),共享变量存在**主内存**(main memory)。线程操作变量必须先从主内存拷贝到工作内存,改完再写回。

```
   +----------------+        +----------------+
   |   线程 A        |        |   线程 B        |
   | 工作内存        |        | 工作内存        |
   | [x=0 副本]      |        | [x=0 副本]      |
   +-------+--------+        +--------+-------+
           |  read/write             |
           v                         v
   +-------------------------------------------+
   |            主内存  x = 0                    |
   +-------------------------------------------+
```

这是**抽象模型**,不代表真实内存布局。实际中"工作内存"对应 CPU 寄存器、store buffer、各级缓存。核心问题:A 改了 `x`,B 不一定立刻看到——这就是**可见性**问题,根源是缓存 + 编译器/CPU 重排。

### 三个特性

| 特性 | 含义 | 靠什么保证 |
|------|------|-----------|
| 原子性 | 操作不可分割 | 基本读写(long/double 除外)天然原子;复合操作需 synchronized/CAS |
| 可见性 | 一个线程的写对另一个可见 | volatile、synchronized、final |
| 有序性 | 指令不被重排到破坏语义 | volatile、synchronized、happens-before |

> `long`/`double` 的坑:JMM 只保证 32 位读写原子,64 位变量在 32 位平台上可能被拆成两次写导致读到"半个值",加 `volatile` 保证 64 位原子。HotSpot 在 64 位 JVM 上实际都原子,但规范不保证。

## happens-before

JMM 的核心工具。若操作 A happens-before B,则 A 的结果对 B 可见,且 A 排在 B 前。规则:

1. **程序顺序**:同一线程内,前面的操作 hb 后面的。
2. **监视器锁**:对锁的 `unlock` hb 后续对同一锁的 `lock`。
3. **volatile**:对 volatile 变量的写 hb 后续的读。
4. **传递性**:A hb B, B hb C ⇒ A hb C。
5. **线程启动**:`t.start()` hb 线程内的任何操作。
6. **线程终止**:线程内操作 hb 其他线程的 `t.join()` 返回。

```java
int a = 0;
volatile boolean flag = false;

// 线程1
a = 1;              // (1)
flag = true;        // (2) volatile 写

// 线程2
if (flag) {         // (3) volatile 读,读到 true 则 (2) hb (3)
    System.out.println(a); // 必然是 1:(1)hb(2)hb(3),传递 ⇒ (1)hb(4)
}
```

没有 `volatile`,`a` 的写可能对线程2不可见,或 (1)(2) 被重排,线程2看到 `flag==true` 但 `a==0`。

## 指令重排

编译器和 CPU 为性能会重排指令,只要**单线程语义(as-if-serial)**不变。多线程下就出问题。经典是双重检查锁定(DCL):

```java
public class Singleton {
    private static volatile Singleton instance; // volatile 不可省
    public static Singleton getInstance() {
        if (instance == null) {                  // 第一次检查(无锁)
            synchronized (Singleton.class) {
                if (instance == null) {          // 第二次检查
                    instance = new Singleton();  // 见下方拆解
                }
            }
        }
        return instance;
    }
}
```

`instance = new Singleton()` 非原子,分三步:(1) 分配内存 (2) 调用构造器初始化 (3) 将 `instance` 指向内存。无 `volatile` 时 (2)(3) 可能重排为 (1)(3)(2),另一线程在第一次检查看到 `instance != null` 却拿到未初始化的对象。`volatile` 禁止此重排(插入内存屏障),这是 JDK5 后 DCL 才正确的原因。

## volatile

### 怎么用

`volatile` 最常见的用法是"一写多读的状态标志"——一个线程写,其他线程读,保证读到的是最新值:

```java
public class VolatileFlagDemo {
    static volatile boolean ready = false;
    static int data = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread writer = new Thread(() -> {
            data = 42;         // 先写普通变量
            ready = true;      // 再写 volatile,充当"发布"信号
        });
        Thread reader = new Thread(() -> {
            while (!ready) {   // 忙等,直到看到 writer 的更新
                // 自旋
            }
            System.out.println("data = " + data); // 保证看到 42,而不是 0
        });
        reader.start();
        writer.start();
        writer.join();
        reader.join();
        System.out.println("done");
    }
}
// data = 42
// done
```

去掉 `volatile` 的话,`reader` 线程理论上可能一直看不到 `ready` 变成 `true`(编译器/JIT 可能把 `while (!ready)` 优化成只读一次寄存器),或者看到 `ready==true` 但 `data` 还是 0(重排)。加上 `volatile` 后这两个问题都不会发生。

### 底层与 C 对比

`volatile` 提供两个保证,类比 C++ 的 acquire/release 语义(但不完全等同)——注意 volatile 变量的**单次读、单次写本身就是原子的**(它甚至让 `long`/`double` 的读写也变原子),真正**不原子的是 `count++` 这类"读-改-写"复合操作**:

1. **可见性**:一个线程对 volatile 变量的写,保证随后读它的线程都能看到最新值。实现上靠**内存屏障**(store/load fence)配合 CPU 的缓存一致性协议,而**不是**字面意义上"每次读写都绕过缓存、直刷主内存"(那是旧教材的简化说法)。
2. **禁止重排**:写前的操作不能重排到写后,读后的操作不能重排到读前(内存屏障)。

它**不保证复合操作原子**:

```java
volatile int count = 0;

void bump() {
    count++;  // 读-改-写三步,volatile 不能防止两线程都读到同一值后各自 +1 丢更新
}
```

两个线程各对同一个 `volatile int count` 调用 10 万次 `count++`,结果几乎必然小于 20 万——`volatile` 保证了每次读写都可见,却挡不住"线程A读到100、线程B也读到100、两者都写回101"这种交叠:

```
count = 157682 (expected 200000, usually less)   // 多次运行,结果每次都不同,但总是明显小于 200000
```

`count++` 需要 `synchronized` 或 `AtomicInteger` 才能保证原子。volatile 适合"一写多读的状态标志",不适合"多线程都要修改的计数器"。

## synchronized 与对象头

### 怎么用

前面「先会用」小节已经见过最基本的用法——给方法加 `synchronized`,同一时刻只允许一个线程执行它。`synchronized` 也可以包一段代码块,并指定锁哪个对象:

```java
synchronized (lock) {        // monitorenter 字节码
    // 临界区
}                            // monitorexit 字节码(异常路径也会 exit)
```

配合锁还有 `wait`/`notify`:一个线程在满足某个条件前先"等着",另一个线程改完条件后"叫醒"它。两者都必须在 `synchronized` 块内调用:

```java
public class WaitNotifyDemo {
    static final Object lock = new Object();
    static boolean condition = false;

    public static void main(String[] args) throws InterruptedException {
        Thread waiter = new Thread(() -> {
            synchronized (lock) {
                while (!condition) {                 // 用 while 而不是 if,防止虚假唤醒
                    try { lock.wait(); } catch (InterruptedException e) { }
                }
                System.out.println("waiter: condition met");
            }
        });
        waiter.start();
        Thread.sleep(100); // 让 waiter 大概率先进入 wait(不影响正确性,只是让演示更直观)
        synchronized (lock) {
            condition = true;
            lock.notifyAll();                        // 唤醒等待者(仍需重新竞争锁)
        }
        waiter.join();
        System.out.println("main: done");
    }
}
// waiter: condition met
// main: done
```

`lock.wait()` 会释放锁并让当前线程进入等待,直到别的线程调用 `lock.notify()`/`notifyAll()` 才有机会被唤醒重新竞争锁——所以 `wait`/`notify` 必须配对使用同一个锁对象。

### 底层与 C 对比

`synchronized` 是**可重入的监视器锁**(monitor)。每个 Java 对象都能作锁,锁状态记录在**对象头的 mark word** 里。

64 位 HotSpot 对象头布局(未压缩,普通对象):

```
对象内存布局:
+------------------+------------------+-------------+---------+
| Mark Word (8B)   | Klass Pointer(4/8B)| 实例字段    | 对齐填充 |
+------------------+------------------+-------------+---------+
```

Mark Word 复用同一块 8 字节存不同信息(锁标志位决定解释方式):

| 锁状态 | Mark Word 内容(部分) | 标志位 |
|--------|----------------------|--------|
| 无锁 | hashCode、GC 分代年龄 | 01 |
| 偏向锁 | 持锁线程 ID | 01(偏向位1) |
| 轻量级锁 | 指向栈中 Lock Record 指针 | 00 |
| 重量级锁 | 指向 monitor(ObjectMonitor)指针 | 10 |
| GC 标记 | — | 11 |

### 锁升级

为避免每次都走重量级(OS 互斥量,涉及用户态/内核态切换,昂贵),HotSpot 做了分级:

```
无锁 --第一个线程进入--> 偏向锁(记录线程ID,后续同线程CAS都免了)
      --别的线程竞争--> 轻量级锁(自旋+CAS,乐观,适合短临界区)
      --自旋失败/竞争激烈--> 重量级锁(挂起线程,进OS等待队列)
```

- **偏向锁**:假设锁总被同一线程获取,首次记下线程 ID,之后无竞争则无需 CAS。JDK15 起默认禁用并计划移除(维护成本高、现代应用收益小)。
- **轻量级锁**:多线程交替执行(非同时竞争)。线程在自己栈帧建 Lock Record,用 CAS 把 mark word 换成指向它的指针。失败则自旋。
- **重量级锁**:真正竞争。膨胀为 `ObjectMonitor`,竞争线程被 OS 挂起,类比 `pthread_mutex`。

## CAS 与 Atomic*

### 怎么用

`AtomicInteger` 这类原子类,提供一组"看起来是普通方法、实际上整体原子"的操作,不需要自己加锁:

```java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicBasicDemo {
    public static void main(String[] args) {
        AtomicInteger ai = new AtomicInteger(0);
        System.out.println(ai.incrementAndGet()); // 1
        System.out.println(ai.incrementAndGet()); // 2
        boolean ok = ai.compareAndSet(2, 100);     // 期望当前值是2,设为100
        System.out.println(ok);                    // true
        System.out.println(ai.get());               // 100
    }
}
// 1
// 2
// true
// 100
```

多个线程并发调用 `incrementAndGet()`,不会像普通 `int` 的 `count++` 那样丢更新——这背后就是 CAS。

### 底层与 C 对比

CAS(Compare-And-Swap)是无锁并发基石,对应 x86 的 `cmpxchg` 指令。语义:`if (内存值==预期) { 内存值=新值; return true }`,整个过程硬件原子。

```java
ai.incrementAndGet();               // 底层 CAS 自旋:do { old=get(); } while(!compareAndSet(old, old+1))
boolean ok = ai.compareAndSet(1, 2); // 期望1则设2

// JDK8+ 高并发计数器,分段减少 CAS 竞争(热点分散到多个 Cell)
LongAdder adder = new LongAdder();
adder.increment();
long sum = adder.sum();
```

`Atomic*` 内部字段用 `volatile` + `Unsafe`/`VarHandle` 的 CAS 实现。

**ABA 问题**:值从 A 变 B 又变回 A,CAS 察觉不到。用 `AtomicStampedReference`(加版本号)解决,类比无锁栈里的 tagged pointer。

可运行对比:8 个线程各自 `count++` 十万次。无保护的 `int` 因"读-改-写"非原子而**丢更新**,`AtomicInteger` 靠 CAS 始终正确。用 `join` 等全部结束,结果确定:

```java
import java.util.concurrent.atomic.AtomicInteger;

public class LostUpdate {
    static int plain = 0;                       // 无保护
    static final AtomicInteger atomic = new AtomicInteger(0);

    public static void main(String[] args) throws Exception {
        int threads = 8, perThread = 100_000;
        Runnable job = () -> {
            for (int i = 0; i < perThread; i++) {
                plain++;                         // 读-改-写,非原子:丢更新
                atomic.incrementAndGet();        // CAS:原子
            }
        };
        Thread[] ts = new Thread[threads];
        for (int i = 0; i < threads; i++) ts[i] = new Thread(job);
        for (Thread t : ts) t.start();
        for (Thread t : ts) t.join();            // 等全部结束,结果确定

        int expected = threads * perThread;      // 800000
        System.out.println("expected      = " + expected);
        System.out.println("atomic == exp : " + (atomic.get() == expected));
        System.out.println("plain  <  exp : " + (plain < expected));
    }
}
// expected      = 800000
// atomic == exp : true
// plain  <  exp : true    —— plain 几乎必然 < 800000(丢更新),atomic 恒等于期望值
```

（这段代码已连续多次运行验证:`atomic == exp` 每次都是 `true`,`plain < exp` 每次都是 `true`,现象稳定可复现。）

## java.util.concurrent

### ReentrantLock

#### 怎么用

`ReentrantLock` 是 `synchronized` 的显式版本:自己 `lock()`、自己 `unlock()`,一定要把 `unlock()` 放进 `finally`,否则异常路径会导致锁永远不释放:

```java
import java.util.concurrent.locks.ReentrantLock;

public class LockBasicDemo {
    static final ReentrantLock lock = new ReentrantLock();
    static int counter = 0;

    static void increment() {
        lock.lock();
        try {
            counter++;
        } finally {
            lock.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable job = () -> { for (int i = 0; i < 100_000; i++) increment(); };
        Thread t1 = new Thread(job);
        Thread t2 = new Thread(job);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter); // 200000
    }
}
// counter = 200000
```

比 `synchronized` 多出来的能力是可以带超时地尝试加锁,拿不到就走别的分支,不会一直傻等:

```java
if (lock.tryLock(1, TimeUnit.SECONDS)) {   // 最多等1秒
    try {
        // 临界区
    } finally {
        lock.unlock();
    }
} else {
    // 没拿到锁,做别的事(比如放弃、重试)
}
```

#### 底层与 C 对比

`ReentrantLock` 比 `synchronized` 更灵活:可中断、可超时、可公平、可多条件变量。基于 **AQS(AbstractQueuedSynchronizer)**——一个 `volatile int state` + CLH 变体等待队列。

`new ReentrantLock(true)` 构造公平锁(按请求顺序排队,吞吐略低但不会有线程一直抢不到)。

| 对比 | synchronized | ReentrantLock |
|------|--------------|---------------|
| 获取方式 | 隐式(JVM) | 显式 lock/unlock |
| 释放 | 自动 | 手动(finally) |
| 可中断 | 否 | lockInterruptibly() |
| 超时 | 否 | tryLock(timeout) |
| 公平锁 | 否 | 可选 |
| 条件变量 | 单个(wait/notify) | 多个 Condition |

### ConcurrentHashMap

#### 怎么用

并发场景下别用普通 `HashMap`(会出数据错乱甚至死循环),用 `ConcurrentHashMap`,API 用法和 `HashMap` 一样:

```java
import java.util.concurrent.ConcurrentHashMap;

public class CHMBasicDemo {
    public static void main(String[] args) {
        ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
        map.put("a", 1);
        map.putIfAbsent("a", 99);         // "a" 已存在,不覆盖
        map.computeIfAbsent("b", k -> 2); // "b" 不存在,计算并放入
        System.out.println(map.get("a")); // 1
        System.out.println(map.get("b")); // 2
    }
}
// 1
// 2
```

#### 底层与 C 对比

JDK8 重写:**数组 + 链表/红黑树**,并发控制用 **CAS + synchronized(锁桶头结点)**,而非 JDK7 的分段锁(Segment)。

- 空桶插入:CAS 直接放,无锁。
- 桶非空:`synchronized` 锁住该桶头结点,只锁一个桶,并发度 = 桶数。
- 链表长度 ≥8 且表容量 ≥64 时转红黑树,查询从 O(n) 降到 O(log n)。
- `size()` 用 `baseCount` + `CounterCell[]`(类似 LongAdder)避免热点。

读操作基本无锁(结点 val/next 为 volatile),写只锁单桶,吞吐远高于 `Collections.synchronizedMap`(锁整表)。

### 线程池 ThreadPoolExecutor

#### 怎么用

比起每次任务都 `new Thread`,通常用线程池复用线程。最简单的用法是拿到一个 `ExecutorService`,反复 `submit` 任务:

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class PoolBasicDemo {
    public static void main(String[] args) throws Exception {
        ExecutorService es = Executors.newFixedThreadPool(2);
        Future<Integer> f = es.submit(() -> 1 + 1);
        System.out.println("result = " + f.get()); // 2,阻塞获取结果
        es.shutdown();                              // 用完记得关闭线程池
    }
}
// result = 2
```

`submit` 把任务交给线程池异步执行,立刻返回一个 `Future`;`f.get()` 阻塞等待任务跑完并拿到返回值;`shutdown()` 通知线程池不再接受新任务、跑完手头的就退出。

#### 底层与 C 对比

复用线程能避免频繁创建(创建 OS 线程昂贵)。生产环境更常见的是自己 `new ThreadPoolExecutor` 把参数写明白,而不是用 `Executors` 的工厂方法(下面会解释为什么):

```java
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.Executors;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2,                              // corePoolSize:核心线程数,常驻
    4,                              // maximumPoolSize:最大线程数
    60L, TimeUnit.SECONDS,          // keepAliveTime:非核心线程空闲存活时间
    new ArrayBlockingQueue<>(100),  // workQueue:任务队列
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
);
```

任务提交逻辑(务必理解):

```
提交任务
  ├─ 运行线程数 < core?      → 新建核心线程执行
  ├─ 否,队列未满?           → 入队等待
  ├─ 否,线程数 < max?       → 新建非核心线程执行
  └─ 否                      → 触发拒绝策略
```

拒绝策略:`AbortPolicy`(抛异常,默认)、`CallerRunsPolicy`(提交者自己跑,反压)、`DiscardPolicy`、`DiscardOldestPolicy`。

> 生产环境避免用 `Executors.newFixedThreadPool`/`newCachedThreadPool`:前者队列无界(可能 OOM),后者线程数无界。手动 `new ThreadPoolExecutor` 明确参数。

### Future 与 CompletableFuture

#### 怎么用

上面 `PoolBasicDemo` 里 `es.submit(...)` 返回的 `Future` 就是最基础的异步结果占位符——`get()` 阻塞拿结果。`CompletableFuture` 则可以像搭积木一样把多个异步步骤串起来,不必每一步都手动 `get()` 阻塞:

```java
import java.util.concurrent.CompletableFuture;

public class CFBasicDemo {
    public static void main(String[] args) {
        CompletableFuture<Integer> cf = CompletableFuture
            .supplyAsync(() -> 10)     // 异步产出一个值
            .thenApply(x -> x * 2);    // 对结果做变换,链式返回新的 CompletableFuture
        System.out.println(cf.join()); // 20,阻塞等待整条链跑完再取值
    }
}
// 20
```

更常见的链式写法还包括消费结果、处理异常:

```java
CompletableFuture
    .supplyAsync(() -> 10)                    // 异步产出
    .thenApply(x -> x * 2)                    // 变换(同线程链式)
    .thenCompose(x -> CompletableFuture.supplyAsync(() -> x + 1)) // 扁平化嵌套 future
    .thenAccept(System.out::println)          // 消费,打印 21
    .exceptionally(ex -> { ex.printStackTrace(); return null; })
    .join(); // 等链跑完 —— 公共线程池是守护线程,main 不等待就可能提前退出、看不到打印
```

#### 底层与 C 对比

类似 JS 的 Promise / C++ 的 `std::future` + continuation。可运行:`thenCombine` 让两个任务并发跑再汇合;`exceptionally` 从链中异常恢复,后续步骤继续:

```java
import java.util.concurrent.CompletableFuture;

public class CFCompose {
    public static void main(String[] args) throws Exception {
        CompletableFuture<Integer> a = CompletableFuture.supplyAsync(() -> 3);
        CompletableFuture<Integer> b = CompletableFuture.supplyAsync(() -> 4);

        // 组合:a、b 并发执行,完成后合并
        int hypotSq = a.thenCombine(b, (x, y) -> x * x + y * y).join(); // 25

        // 链式 + 异常恢复
        String r = CompletableFuture.supplyAsync(() -> 10)
            .thenApply(x -> x / 0)                    // 抛 ArithmeticException
            .exceptionally(ex -> -1)                  // 恢复为 -1
            .thenApply(x -> "result=" + x)
            .join();

        System.out.println("hypotSq = " + hypotSq);
        System.out.println(r);
    }
}
// hypotSq = 25
// result=-1
```

### 协调工具

```java
// CountDownLatch:一次性,等 N 个事件完成(不可重置),类似 barrier 单向
CountDownLatch latch = new CountDownLatch(3);
// worker: latch.countDown();
latch.await();   // 主线程等计数归零

// Semaphore:信号量,限流/限并发数
Semaphore sem = new Semaphore(2);   // 最多2个同时进入
sem.acquire();
try { /* 受限资源 */ } finally { sem.release(); }

// CyclicBarrier:可重用,N 个线程互相等待汇合后一起继续
```

## 死锁与避免

死锁四条件(与 OS 教材一致):互斥、持有并等待、不可剥夺、循环等待。典型:

```java
// 线程1: 先锁 A 再锁 B;线程2: 先锁 B 再锁 A → 循环等待
synchronized (A) { synchronized (B) { } }  // 线程1
synchronized (B) { synchronized (A) { } }  // 线程2
```

避免手段:
- **锁排序**:所有线程按固定全局顺序获取锁(破坏循环等待)。
- **tryLock 超时**:`ReentrantLock.tryLock` 拿不到就回退释放。
- **减小锁粒度/持有时间**,尽量无锁(CAS、不可变对象)。
- 检测:`jstack <pid>` 能报 "Found one Java-level deadlock"。

## 虚拟线程(Loom,JDK21)

### 怎么用

前面看到的平台线程(`new Thread(...)`)1:1 绑内核线程,开几千个就吃力。JDK21 起可以用**虚拟线程**,语法和平台线程几乎一样,但极轻,能开到百万级:

```java
import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class VirtualBasicDemo {
    public static void main(String[] args) throws Exception {
        Thread vt = Thread.ofVirtual().start(() -> System.out.println("hello from virtual thread"));
        vt.join();
        System.out.println("is virtual: " + vt.isVirtual()); // true

        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            Future<Integer> f = executor.submit(() -> 1 + 1);
            System.out.println("result = " + f.get()); // 2
        }
    }
}
// hello from virtual thread
// is virtual: true
// result = 2
```

`Thread.ofVirtual().start(...)` 直接起一个虚拟线程;`Executors.newVirtualThreadPerTaskExecutor()` 则是"每提交一个任务就用一个新虚拟线程去跑"的线程池,`submit`/`get` 用法和普通线程池完全一样。

### 底层与 C 对比

平台线程 1:1 绑内核线程,昂贵(栈 ~1MB,上下文切换走内核)。**虚拟线程**是 JVM 调度的**用户态线程**(类比 Go goroutine),极轻(栈按需增长,存堆上),可开百万级。

```java
// 每个任务一个虚拟线程,阻塞 IO 时自动卸载,不占内核线程
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1)); // 阻塞不占载体线程
            return null;
        });
    }
}
```

可运行:一次开 10 万个虚拟线程,每个"阻塞" 10ms。放在几个平台线程(载体)上照样几十毫秒跑完——同样数量的平台线程根本开不出来。用 `CountDownLatch` 等全部报到,`AtomicInteger` 计数:

```java
import java.util.concurrent.CountDownLatch;
import java.util.concurrent.Executors;
import java.util.concurrent.atomic.AtomicInteger;

public class VThreads {
    public static void main(String[] args) throws Exception {
        int n = 100_000;
        AtomicInteger done = new AtomicInteger();
        CountDownLatch latch = new CountDownLatch(n);

        // 每任务一个虚拟线程,共 10 万个,各自短暂阻塞
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < n; i++) {
                executor.submit(() -> {
                    Thread.sleep(10);         // 阻塞卸载载体,而非占用内核线程
                    done.incrementAndGet();
                    latch.countDown();
                    return null;
                });
            }
            latch.await();                    // 等全部任务报到
        }
        System.out.println("tasks done      = " + done.get());
        System.out.println("current is virtual: " + Thread.currentThread().isVirtual());
    }
}
// tasks done      = 100000
// current is virtual: false   —— main 是平台线程;10 万虚拟线程由少量载体线程调度
```

（该例已实测:10 万虚拟线程 + 每个 10ms 阻塞,总耗时不到 0.2 秒——如果换成 10 万个平台线程,光是创建开销和内存就足以拖垮大多数机器。）

### 挂载与卸载

虚拟线程运行时**挂载(mount)**到一个平台线程(**载体线程 carrier**,来自 ForkJoinPool)。遇到阻塞操作(如网络 IO、sleep)时**卸载(unmount)**:虚拟线程的栈(continuation)被保存到堆,载体线程释放去跑别的虚拟线程。阻塞结束后再挂载到某个载体继续。

```
平台线程池(载体, ~CPU核数个)
  Carrier-1: [VT-A 运行中] --VT-A遇到IO阻塞--> 卸载VT-A, 挂载 VT-B
  Carrier-2: [VT-C 运行中]
             堆: [VT-A 栈快照 待唤醒] [VT-D 栈快照] ...
```

对比:平台线程阻塞时,整个内核线程被 OS 挂起;虚拟线程阻塞时,内核线程被解放去干活。因此"一请求一线程"的同步写法也能有异步的吞吐。

| 维度 | 平台线程 | 虚拟线程 |
|------|---------|---------|
| 映射 | 1:1 内核线程 | M:N,JVM 调度到载体 |
| 数量级 | 数千 | 数百万 |
| 栈 | 固定 ~1MB | 按需,堆上可增减 |
| 阻塞代价 | 占用内核线程 | 卸载,释放载体 |
| 适用 | CPU 密集 | IO 密集、高并发 |

**pinning(固定)陷阱**:虚拟线程阻塞时若无法卸载,就会**钉住**载体线程,削弱优势。**在 JDK 21–23**,在 `synchronized` 块内阻塞会 pin,所以当时的建议是把 IO 相关临界区改用 `ReentrantLock`;但 **JDK 24 起(JEP 491)已基本消除 `synchronized` 造成的 pinning**,不必再为此刻意替换。调用 native(JNI)等少数情形仍可能 pin。

关于 `volatile`/CAS 的底层内存屏障,可回顾 [03-变量与内存模型](03-变量与内存模型.md)。线程池提交的任务与异常传播见 [09-错误处理与资源管理](09-错误处理与资源管理.md)。

## 小结

- **用法层面**:`new Thread(() -> ...).start()` 开线程,`join()` 等它跑完;两个线程改同一个变量就用 `synchronized` 包起来,或用 `ReentrantLock`/`AtomicInteger`;批量任务用线程池 `submit`,拿 `Future`/`CompletableFuture` 取结果;`ConcurrentHashMap` 代替并发场景下的 `HashMap`;`Thread.ofVirtual()`/`newVirtualThreadPerTaskExecutor()` 开虚拟线程,语法和平台线程一样但能开到百万级。
- **底层层面**:多线程出问题的根源是 JMM 抽象里"工作内存/主内存"不同步——可见性、有序性、原子性三个特性分别靠 volatile/synchronized/happens-before 和 CAS 保证。`synchronized` 锁状态记在对象头 mark word 里,会经历偏向锁→轻量级锁→重量级锁的升级;CAS 是无锁并发的硬件基石(但有 ABA 问题);`ReentrantLock` 基于 AQS;`ConcurrentHashMap` 靠"空桶 CAS + 非空桶锁单桶头结点"把并发度提到桶级;线程池要理解核心/最大线程数、队列、拒绝策略的提交流程;虚拟线程是 JVM 调度的用户态线程,阻塞时卸载载体线程而不占用内核线程,但要避开会导致 pinning 的场景。
