# 变量与内存模型：`a = b` 到底发生了什么 ⭐

这是全书最重要的一章。本章先讲最基本的事——Java 里怎么**声明和赋值**一个变量，这部分和你写过的任何一门语言差不多，不需要任何内存背景。然后才进入本章真正的核心：C/C++ 程序员切换到 Java 时，几乎所有"诡异 bug"都源于误解**赋值、比较、传参**时内存里到底发生了什么，这部分要把栈帧、堆、对象头彻底讲清。只想会写的话，读完第一节就够；想搞懂"为什么改了一个变量另一个也跟着变"这类灵异事件，请把全篇读完。

---

## 先会用：声明与赋值怎么写

Java 里声明一个变量就是 `类型 变量名 = 值;`，和 C 几乎一样：

```java
public class VarDemo {
    public static void main(String[] args) {
        // 基本类型:变量里直接是值
        int age = 20;
        double price = 9.9;

        // 引用类型:变量里是"指向对象的引用"
        String name = "Ada";              // String 对象
        int[] nums = new int[]{1, 2, 3};  // 数组对象

        System.out.println(age);    // 20
        System.out.println(price);  // 9.9
        System.out.println(name);   // Ada
        System.out.println(nums[0]);// 1

        // 赋值:把等号右边的值/引用复制给左边变量
        int age2 = age;      // 复制的是数值
        age2 = 99;
        System.out.println(age);     // 20,不受影响

        int[] nums2 = nums;  // 复制的是"引用"
        nums2[0] = 100;
        System.out.println(nums[0]); // 100!两个变量指向同一个数组
    }
}
```

注意到最后两组赋值的区别了吗：改 `age2` 不影响 `age`，但改 `nums2[0]` 却让 `nums[0]` 也变了。这不是巧合，也不是 bug——这正是本章接下来要讲透的核心机制。

## 开始深入：基本类型 vs 引用类型，`a = b` 到底拷贝了什么

从这里开始，我们不再满足于"能写"，而要搞清楚"为什么"。核心结论先摆出来，后面逐条论证：

> Java 的变量分**两大阵营**：**基本类型（primitive）**变量里直接装"值"；**引用类型（reference）**变量里装的是"引用值"——一个受 JVM 管理的指针，指向堆上的对象。`a = b` 永远是"把 b 里的比特拷给 a"：对基本类型是拷贝值本身，对引用类型是拷贝那根指针（两变量从此指向同一个堆对象）。

## 一、两大阵营

| 阵营 | 成员 | 变量里存的是 | 数据在哪 |
|------|------|------------|---------|
| 基本类型 | `byte short int long float double char boolean` | **值本身**（比特位） | 变量所在处（栈帧 slot 或对象字段内联） |
| 引用类型 | 类、接口、数组、`String`、包装类、枚举… | **引用值**（≈指针） | 引用指向的对象在**堆**上 |

用 C 的话讲：
- 基本类型变量 ≈ `int a;`，`a` 这块内存直接就是那个整数。
- 引用类型变量 ≈ `Foo* p;`，`p` 这块内存装的是地址，真正的 `Foo` 在堆上（由 `new` 分配）。**但你不能对这根"指针"做算术、不能解引用取地址、不能强转成整数**——它是 JVM 托管的、被约束的指针。

> HotSpot 实现细节：一个引用在 64 位 JVM 上通常占 8 字节，但开启**压缩指针（CompressedOops）**后压到 4 字节（用一个 35 位可寻址的编码，默认对 <32GB 堆生效）。见[「数据类型」](04-数据类型.md)。语言规范不规定引用的具体表示，只规定它的语义。

## 二、局部变量表 slot：变量住在哪

每次方法调用，JVM 在当前线程的 **JVM 栈**上压一个**栈帧**，帧里有一张**局部变量表（Local Variable Table）**，由若干 **slot** 组成（编译期就定好数量，见[「运行模型」](01-运行模型.md)的 `javap`）。

- 一个 slot 是 32 位。`int/float/引用` 各占 1 个 slot；`long/double` 占 2 个连续 slot。
- 方法参数和方法内声明的局部变量都住在这张表里。

对基本类型局部变量，slot 里就是那个值。对引用类型局部变量，slot 里是那根引用（指向堆）。**对象本体永远不在 slot 里，永远在堆上**（逃逸分析除外，见文末）。

## 三、`a = b`：核心中的核心

### 3.1 基本类型：拷贝值

```java
int b = 100;
int a = b;   // 把 b 的比特(100)拷贝一份放进 a 的 slot
a = 200;     // 只改 a 自己的 slot，b 毫无影响
// 此刻 a=200, b=100
```

内存图（栈帧局部变量表）：

```
 赋值前              a = b 之后           a = 200 之后
┌──────────┐       ┌──────────┐        ┌──────────┐
│ b │ 100  │       │ b │ 100  │        │ b │ 100  │
├──────────┤       ├──────────┤        ├──────────┤
│ a │  ??  │       │ a │ 100  │←拷贝    │ a │ 200  │
└──────────┘       └──────────┘        └──────────┘
     两个独立的 slot，互不影响。等价于 C 的 int a=b;
```

这和 C 完全一致，没有惊喜。

### 3.2 引用类型：拷贝引用（拷贝的是指针，不是对象！）

```java
int[] b = new int[]{1, 2, 3};  // 堆上建数组对象，b 里存指向它的引用
int[] a = b;                    // 拷贝"引用值"！a 和 b 指向同一个数组
a[0] = 99;                      // 通过 a 改堆上对象
System.out.println(b[0]);       // 输出 99！因为 b 指的是同一个对象
```

**这是 C 程序员最需要建立的直觉**：`int[] a = b;` 不是拷贝数组，而是拷贝那根指针。它等价于 C 的 `int* a = b;`（浅拷贝指针），而**不是** `memcpy`。

内存图：

```
      栈帧局部变量表                 堆 Heap
   ┌────────────────┐         ┌───────────────────────┐
   │ b │  ref ●──────┼────────▶│ int[3] 对象            │
   ├────────────────┤    ┌───▶│ [0]=99 [1]=2 [2]=3     │
   │ a │  ref ●──────┼────┘    │ (a[0]=99 改的就是这里)  │
   └────────────────┘         └───────────────────────┘
      a 和 b 是两根指针，指向同一个堆对象（别名 aliasing）
```

对照基本类型的图，差异一目了然：

```
   Object a = b  （引用类型）           int a = b  （基本类型）
   ┌───────┐                          ┌───────┐
   │ b │●──┼──┐                       │ b │100│
   ├───────┤  │   堆                   ├───────┤
   │ a │●──┼──┴──▶[ 对象 ]             │ a │100│ ← 独立副本
   └───────┘  两指针共享一个对象         └───────┘  两个独立的值
```

一句话记牢：**基本类型赋值克隆"数据"，引用类型赋值克隆"指针"。** 后者产生**别名（aliasing）**——通过任一变量修改对象，另一变量都看得见。这与 C 里两个指针指向同一块 `malloc` 内存的行为完全一致。

### 3.3 想要真正的副本怎么办

Java 不会自动深拷贝（没有 C++ 的拷贝构造函数在赋值时触发）。要独立副本得显式做：

```java
int[] a = b.clone();               // 数组浅拷贝，得到新对象
int[] a2 = Arrays.copyOf(b, b.length);
List<T> copy = new ArrayList<>(src);  // 集合的拷贝构造
```

注意这些多是**浅拷贝**：容器/数组本身是新的，但里面若装的是引用，元素引用仍共享同一批对象。深拷贝需要你逐层复制。

实测这个"浅"字——二维数组 `clone()` 只复制外层引用，内层数组仍是共享的：

```java
int[][] grid = {{1, 2}, {3, 4}};
int[][] copy = grid.clone();   // clone() 是浅拷贝：只复制外层引用
copy[0][0] = 99;               // 穿透到共享的内层数组
System.out.println(grid[0][0]);          // 99  <- 原对象也被改了！
System.out.println(copy[0] == grid[0]);  // true -- 内层是同一个数组对象

// 真正的深拷贝必须逐个复制内层数组
int[][] deep = new int[grid.length][];
for (int i = 0; i < grid.length; i++) deep[i] = grid[i].clone();
deep[1][0] = -1;
System.out.println(grid[1][0]);          // 3   <- 这次原对象安全了
```

## 四、`==` vs `equals`

理解了两大阵营，这个经典陷阱就顺理成章了。

### `==`：比"变量里的比特"
- 基本类型：比**值**。`3 == 3` → true。符合直觉。
- 引用类型：比**引用**——即"是不是同一个堆对象"（地址是否相同），**不看内容**。

```java
String x = new String("hi");
String y = new String("hi");
System.out.println(x == y);        // false！两个不同的堆对象，两根不同的指针
System.out.println(x.equals(y));   // true！比内容
```

内存图：

```
 ┌───────┐        堆
 │ x │●──┼──▶[ String "hi" 对象①]
 ├───────┤
 │ y │●──┼──▶[ String "hi" 对象②]   x==y 比的是 ●vs●，指向不同→false
 └───────┘                          x.equals(y) 逐字符比→true
```

### `equals`：比"内容"（语义相等）
`equals` 是定义在 `Object` 上的**方法**，可被重写。默认实现（`Object.equals`）就是 `==`（比引用）；但 `String`、包装类、集合等**重写**了它，改为比逻辑内容。所以：
- 比较字符串/对象内容，**永远用 `equals`**。
- `==` 只在你确实想问"是不是同一个对象"、或比较基本类型、或比较 `null` 时用。

字面量还有个"字符串常量池"的细节：同一个字面量在池中复用同一个对象，而 `new String(...)` 强制建新对象——这恰好把 `==`（比引用）和 `equals`（比内容）的差别照得一清二楚：

```java
String p = "hi";
String q = "hi";              // 同一字面量 -> 同一个池中对象
String n = new String("hi");  // new 强制新建一个堆对象
System.out.println(p == q);         // true  -- 都指向池中那个对象
System.out.println(p == n);         // false -- n 是另一个对象
System.out.println(p == n.intern());// true  -- intern() 返回池中那个
System.out.println(p.equals(n));    // true  -- 内容始终相等
```

> 与 `equals` 配套的还有 `hashCode`：约定"equals 相等则 hashCode 必须相等"，否则放进 `HashMap/HashSet` 会出错。详见[「复合数据结构」](07-复合数据结构.md)。C++ 里你为 `unordered_map` 提供 `operator==` 和 `hash`，是一回事。

### 一个反直觉的坑：包装类缓存

```java
Integer a = 100, b = 100;
System.out.println(a == b);   // true  —— 100 在缓存区间
Integer c = 200, d = 200;
System.out.println(c == d);   // false —— 200 超出缓存，是两个对象
```

`Integer` 对 `[-128, 127]` 有对象缓存，装箱时复用同一对象，故 `==` 恰好为 true；超出则新建对象，`==` 为 false。**这正说明 `==` 比的是引用**。比较包装类的值请一律用 `equals` 或先拆箱。细节见[「数据类型」](04-数据类型.md)。

## 五、`null`

`null` 是引用类型的"空引用"，等价于 C 的空指针 `nullptr`——变量不指向任何对象。

```java
String s = null;         // s 的 slot 里是"空引用"值
s.length();              // NullPointerException！对空引用"解引用"
```

区别于 C：解引用空指针在 C 里是未定义行为（多半段错误），在 Java 里**确定性地抛 `NullPointerException`（NPE）**，可被捕获、带堆栈信息（Helpful NPE / JEP 358：JDK 14 引入但默认关闭，JDK 15 起默认开启，能提示是哪个变量为 null）。基本类型**不能**为 `null`（它装的是值不是引用），`int x = null;` 编译错误。这也是包装类 `Integer` 拆箱成 `int` 时若为 null 会 NPE 的原因。

## 六、`final` 变量语义

`final` 常被拿来对标 C++ 的 `const`，但要精确区分：**`final` 约束的是"变量这个 slot 能否再被赋值"，而不是它指向的对象能否被改。**

```java
final int x = 10;
// x = 20;                 // 编译错误：不能重新赋值

final int[] arr = {1, 2, 3};
arr[0] = 99;               // 合法！改的是堆上对象的内容，没动 arr 这根引用
// arr = new int[5];       // 编译错误：不能让 arr 重新指向别的对象
```

内存图理解：`final` 锁住的是左边那个 slot（不能换指向），锁不住右边堆里的对象内容。

```
 final int[] arr ●═══▶[ 1, 2, 3 ]   ═══ 表示这根引用被锁死不可改指
                        ↑ 但对象内容照样能改：arr[0]=99 ok
```

- 想要"对象也不可变"，得靠对象自身设计（如 `String` 天生不可变，或用不可变集合 `List.copyOf(...)`）。这与 C++ `const int* p`（不能改所指内容）vs `int* const p`（不能改指向）的区分类似：**Java 的 `final` 只提供后者那半**——锁指向，不锁内容。
- `final` 字段：必须在构造完成前赋值一次，之后不可变；对多线程可见性有保证（构造完成后其它线程能看到正确的 final 字段值）。
- `final` 局部变量 / 参数：常用于被 lambda 捕获（lambda 只能捕获"事实上 final"的局部变量，见[「函数与调用栈」](06-函数与调用栈.md)）。

## 七、把它们放进对象字段

上面讲的是局部变量（在栈帧 slot 里）。**对象的字段**遵循同样的两阵营规则，只是存储位置在**堆上的对象内部**：

```java
class Node {
    int value;      // 基本类型字段：值直接内联在 Node 对象的内存里
    Node next;      // 引用字段：存一根引用，指向另一个堆上的 Node（或 null）
}
```

内存图（一个 `Node` 对象在堆上）：

```
堆上的 Node 对象
┌─────────────────────────┐
│ [对象头 mark+klass]      │  ← 见第4章对象布局
│ value : 42              │  ← int 值内联
│ next  : ref ●───────────┼──▶ 另一个 Node 对象（或 null）
└─────────────────────────┘
```

这正是链表/树在 Java 里的样子：结构和 C 的 `struct Node { int value; struct Node* next; };` 完全同构，区别仅在于 `next` 是托管引用、对象在 GC 堆上、不用你 `free`。

## 八、逃逸分析：语言语义 vs HotSpot 实现

语言规范说"对象都在堆上"。但**在 HotSpot 中**，JIT 的**逃逸分析**（见[「运行模型」](01-运行模型.md)）如果证明某个 `new` 出来的对象不会逃出当前方法，就可能做**标量替换**：把对象拆成几个基本值直接放进栈帧 slot 甚至寄存器，**完全不进堆、无需 GC**。

```java
int distance() {
    var p = new Point(3, 4);   // p 不逃逸出本方法
    return p.x * p.x + p.y * p.y;
    // HotSpot 优化后：可能根本没 new，x、y 变成两个栈上局部变量
}
```

所以精确表述是：
- **语义层面**：引用类型对象逻辑上在堆，赋值拷贝引用，别名成立。
- **实现层面**：不逃逸的对象可能被**标量替换**（拆成基本值放进栈帧 slot/寄存器，而不是把整个对象搬上栈），接近 C 里"栈上放 struct"的效率——但这对你写的代码**语义透明**，你无需也无法直接控制。

这解释了为什么 Java 里大量创建短命小对象往往没有想象中那么慢。

## 小结（务必内化）

**用法层面**：声明变量用 `类型 名字 = 值;`；赋值 `a = b` 直接照抄语法即可写出程序；但一定记住"先会用"里那个对比实验——改基本类型的副本不影响原变量，改引用类型副本指向的对象内容，原变量会看到变化。

**底层层面**：

1. 变量两大阵营：**基本类型装值、引用类型装指针（引用值）**。
2. `a = b` 永远拷贝"变量里的比特"：基本类型是**拷值**（独立副本），引用类型是**拷指针**（产生别名，共享同一堆对象）。想要副本必须显式 clone/拷贝构造，且默认是浅拷贝。
3. `==` 比比特（引用类型即比是否同一对象），`equals` 比内容——比对象内容永远用 `equals`。
4. `null` 是空引用，解引用**确定性抛 NPE**（非 UB）；基本类型不能为 null。
5. `final` 只锁"变量指向"，不锁"对象内容"（相当于 C++ 的 `T* const`，而非 `const T*`）。
6. 对象字段同样遵循两阵营规则，只是存在堆对象内部；链表/树与 C 的 struct 同构，只是引用托管、GC 回收。
7. HotSpot 逃逸分析可能对不逃逸对象做**标量替换**（拆成标量放栈帧/寄存器，而非把整个对象栈上分配），但这对语义透明。
