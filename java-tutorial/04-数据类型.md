# 数据类型与内存布局

本章承接[「变量与内存模型」](03-变量与内存模型.md)的两阵营划分，讲清楚每种类型在内存里占多少、长什么样，处处与 C 对照。

## 一、8 种基本类型：定长、跨平台、无 unsigned

字面量与后缀，和 C 类似：`10L`（long）、`3.14f`（float）、`0x1F`（十六进制）、`0b1010`（二进制）、`017`（八进制，前导 0）。Java 7+ 允许**下划线分隔**提高可读性：

```java
long   creditCard = 1234_5678_9012_3456L;
int    million     = 1_000_000;
int    bytes       = 0b1101_0010;
double pi          = 3.141_592_653;
```

C 的整型宽度是**实现定义**的：`int` 可能 16/32/64 位，`long` 在不同平台不同（LP64 vs LLP64），你得靠 `<stdint.h>` 的 `int32_t` 才能确定。Java 把宽度**写进语言规范**，所有平台一致：

| 类型 | 位宽 | 范围 | C 近似对应 | 默认值 |
|------|------|------|-----------|--------|
| `byte` | 8 | -128 ~ 127 | `int8_t` | 0 |
| `short` | 16 | -32768 ~ 32767 | `int16_t` | 0 |
| `int` | 32 | ±2.1e9 | `int32_t` | 0 |
| `long` | 64 | ±9.2e18 | `int64_t` | 0L |
| `float` | 32 | IEEE754 单精度 | `float` | 0.0f |
| `double` | 64 | IEEE754 双精度 | `double` | 0.0d |
| `char` | 16 | **UTF-16 码元** 0~65535 | 无（`wchar_t` 不可靠对应） | `'\u0000'` |
| `boolean` | 未规定 | true/false | 无（C 用 int/`_Bool`） | false |

三个必须记住的 Java 特性：

1. **没有 unsigned**。所有整型都是**有符号**的（`char` 是唯一的无符号 16 位量，但它语义上是字符不是数）。需要无符号运算时用更宽的类型或 `Integer.toUnsignedLong`、`Integer.divideUnsigned` 等静态方法。位移的逻辑右移用 `>>>`（见[「基础语法」](02-基础语法.md)）。
2. **`char` 是 16 位 UTF-16 码元**，不是 C 的 8 位 `char`。一个 `char` 存不下所有 Unicode（增补平面字符需两个 char，即代理对）。要处理"一个字符"的语义，用 `int` 码点（code point）相关 API。字符串编码见后文。
3. **`boolean` 大小未规定**。JVM 规范不定义 boolean 占几位，HotSpot 里单个 boolean 变量通常按 int 处理，`boolean[]` 数组里每元素占 1 字节。别指望位打包（要位图用 `long` + 位运算或 `BitSet`）。

浮点遵循 IEEE754：有 `Double.NaN`、`Double.POSITIVE_INFINITY`；`0.0/0.0` 得 NaN，`1.0/0.0` 得 Infinity（不像整数除零抛异常）。`NaN != NaN` 为 true 的坑与 C 一致。

## 二、包装类、自动装箱与 `Integer` 缓存

每个基本类型都有一个对应的**包装类（wrapper）**——把值包成堆上对象，以便放进只接受对象的容器（泛型集合不能装 `int`，只能装 `Integer`，见[「复合数据结构」](07-复合数据结构.md)）：

| 基本 | 包装类 |
|------|--------|
| int | `Integer` |
| long | `Long` |
| double | `Double` |
| char | `Character` |
| boolean | `Boolean` |
| … | （byte/short/float 同理） |

**自动装箱（autoboxing）/拆箱（unboxing）**：编译器自动在基本类型与包装类之间转换。

```java
Integer boxed = 42;          // 自动装箱：编译器插入 Integer.valueOf(42)
int prim = boxed;            // 自动拆箱：编译器插入 boxed.intValue()
List<Integer> list = new ArrayList<>();
list.add(7);                 // 装箱：int 7 -> Integer 对象放进堆
int v = list.get(0);         // 拆箱
```

代价（对性能敏感者注意）：装箱要在**堆上 new 一个对象**，一根引用指向它——比栈上一个 `int` 昂贵得多，还制造 GC 压力。密集数值计算别用 `List<Integer>`，用 `int[]` 或 `IntStream`（见[「标准库算法」](11-标准库算法.md)）。

#### `Integer` 缓存 `[-128, 127]` 与 `==` 陷阱

`Integer.valueOf(int)` 对 **-128 到 127** 的值返回**缓存的同一个对象**（可用 `-XX:AutoBoxCacheMax` 调上界）；范围外每次新建对象。于是：

```java
Integer a = 127, b = 127;
System.out.println(a == b);      // true  —— 命中缓存，同一对象
Integer c = 128, d = 128;
System.out.println(c == d);      // false —— 各自新建，不同对象
System.out.println(c.equals(d)); // true  —— 比值，正确
```

结合[第 3 章](03-变量与内存模型.md)的 `==` 语义：`==` 比引用，缓存让小整数恰好共享对象故为 true，大整数则否。**结论：包装类比较值一律用 `equals` 或先拆箱成基本类型比。** 拆箱还有 NPE 风险：`Integer x = null; int y = x;` 运行时 NPE。

另一个自动拆箱的隐蔽坑藏在**三元表达式**里：当两个分支是不同的包装类型时，编译器会把两边**都拆箱**再做二元数值提升，最后再装箱——结果类型可能和你想的不一样。

```java
public class Ternary {
    public static void main(String[] args) {
        Object x = true ? Integer.valueOf(1) : Double.valueOf(2);
        System.out.println(x);                            // 1.0   —— 不是 1！
        System.out.println(x.getClass().getSimpleName()); // Double
    }
}
```

`true` 明明选了 `Integer.valueOf(1)`，但因为另一分支是 `Double`，整个三元表达式被提升为 `double`：`Integer` 先拆箱成 `int` → 提升为 `double` `1.0` → 再装箱成 `Double`。这是 JLS 三元表达式类型规则叠加自动拆装箱的产物，C 的 `?:` 只有算术提升、没有装箱这一层。

## 三、对象内存布局（HotSpot）

这是 C 程序员最关心的部分：一个 Java 对象在堆上到底长什么样？在 HotSpot 中，对象由三部分构成：

```
┌────────────────────────────────────────────────┐
│ 对象头 Object Header                             │
│  ┌──────────────────────────────────────────┐  │
│  │ Mark Word   (64位JVM 通常 8 字节)          │  │  哈希码、GC 分代年龄、
│  │                                            │  │  锁状态/偏向锁指针等
│  ├──────────────────────────────────────────┤  │
│  │ Klass 指针  (指向方法区/元空间的类元数据)   │  │  开启压缩指针时 4 字节
│  │                                            │  │  ≈ C++ 的 vptr 作用之一
│  └──────────────────────────────────────────┘  │
│  [ 若是数组，还有一个 length 字段 (4 字节) ]     │
├────────────────────────────────────────────────┤
│ 实例字段 Instance Data                           │  各字段按类型排布，
│  int / long / 引用 ...                           │  HotSpot 会重排以省填充
├────────────────────────────────────────────────┤
│ 对齐填充 Padding                                 │  补齐到 8 字节的整数倍
└────────────────────────────────────────────────┘
```

对照 C++：C++ 的 `struct`/`class` 没有 Mark Word；有虚函数时有一个 `vptr` 指向 vtable。Java 的 **Klass 指针**兼具"找到类元数据、方法表"的职责（虚方法分派见[「函数与调用栈」](06-函数与调用栈.md)），Mark Word 则是 C++ 完全没有的东西——它承载了对象哈希、GC 年龄、以及内建锁（`synchronized`）状态。

**为什么 Mark Word 要把哈希码、GC 年龄、锁状态这几件互不相干的事挤进同一个字**：因为这几件事在对象的生命周期里几乎从不同时需要，把它们分开设三个独立字段，等于让每个对象都白白多背几个字节，而 Java 里对象数量级往往是千万到亿级，每个对象多 8 字节就是几十上百 MB 的整体开销。Mark Word 的解法是**按需复用同一块比特位、靠低位标志区分当前含义**：一个新建对象默认处于无锁或偏向锁状态，此时高位可能还没被用到；一旦调用 `hashCode()`，JVM 才把哈希值写进这块区域并标记"已计算"；一旦发生 `synchronized` 竞争，同一块区域又被重新解释成指向线程栈上锁记录的指针，随竞争程度从偏向锁→轻量级锁→重量级锁逐级"膨胀"，每次膨胀都在这同一个字里切换编码格式。GC 也需要在这里记一个分代年龄计数器，决定对象何时从新生代晋升到老年代（见[「错误处理与资源管理」](09-错误处理与资源管理.md)的分代假说）。这是一种时间复用（同一块内存在不同阶段承载不同信息）而非空间复用，用状态机式的编码复杂度换取每个对象更小的固定开销——这类空间敏感的位打包设计，在 C 里对应的是手写 bitfield 或者把多个标志位压进一个 `uint64_t` 自己解析，Java 把这套逻辑内建进对象模型，业务代码完全看不见。

**对齐**和 C 的 `struct` 一样存在：HotSpot 默认对象按 **8 字节对齐**，末尾补 padding。字段之间也可能有填充。区别是：C 里字段顺序严格按声明排（除非 pragma pack），而 **HotSpot 会自行重排字段顺序**以最小化填充（把宽类型放前面、把引用聚在一起方便 GC 扫描），你不能假设声明顺序等于内存顺序。

一个空对象（无字段）在 64 位 + 压缩指针下通常是 16 字节（8 Mark + 4 Klass + 4 padding）。

### 压缩指针 CompressedOops

前面[第 3 章](03-变量与内存模型.md)提过：64 位下一根引用本应 8 字节，太占空间且伤缓存。HotSpot 默认对 **小于 32GB 的堆**开启 **压缩普通对象指针（Compressed Ordinary Object Pointers）**：把引用编码成 32 位，访问时按对象 8 字节对齐做左移 3 位还原出真实地址。

**为什么 4 字节能寻址本该需要 8 字节才够的地址空间**：关键在于上一段提到的"对齐"这个事实——既然堆里每个对象的起始地址都保证是 **8 字节的整数倍**（对齐要求所致），那么真实地址的最低 3 位永远是 0，存储这 3 位就是纯粹的浪费。压缩指针索性不存这 3 位，只存地址右移 3 位后的结果；这样一个 32 位无符号数能表示的 `2^32` 个编码值，每个都对应一个 8 字节对齐的真实地址，等效寻址范围就是 `2^32 × 8 = 32GB`，而不是 32 位地址本身能直接表示的 4GB。代价是每次解引用都要做一次左移换算（对现代 CPU 而言几乎免费，一条移位指令），换来的是引用字段、Klass 指针普遍减半，对象更小、缓存命中率更高——对象越小，同样大小的 CPU cache line 能装下越多对象，GC 扫描和字段访问都更快。堆一旦超过 32GB，压缩后的编码空间不够用，JVM 自动退回 8 字节裸指针（这是"别把堆调到 32GB 出头"这条调优经验的直接原因：过 32GB 门槛不仅没有变大反而所有指针都变宽了）。这是纯 HotSpot 实现细节，C 里没有对应概念（C 指针就是裸地址，不做这层编码）。

## 四、数组是对象

```java
int[] a = new int[5];      // 分配一个长度为 5 的数组，元素默认 0
System.out.println(a.length);   // 5，length 是内建字段，不是方法调用
a[0] = 10;
System.out.println(a[0]);       // 10
```

访问越界会抛异常，而不是像 C 那样悄悄踩坏内存：

```java
a[10] = 1;                 // ArrayIndexOutOfBoundsException
```

Java 数组是**堆上的对象**，不是 C 那种"一段裸连续内存 + 首地址"。内存布局：数组对象 = 对象头（含额外的 **length 字段**）+ 连续的元素区 + 对齐填充。

```
int[5] 在堆上：
┌──────────┬────────┬──────────────────────────────┐
│ 对象头    │ length=5│ [0][1][2][3][4]  连续 int 元素 │
└──────────┴────────┴──────────────────────────────┘
```

要点对照：
- **元素连续存储**，`int[]` 的元素区就是 5 个连续 int——这点和 C 一样，缓存友好。
- **带边界检查**：每次 `a[i]` 访问 JVM 都校验 `0 <= i < length`，越界抛异常而非未定义行为。JIT 常能证明循环内索引安全从而**消除边界检查**，所以热点循环没有想象中的开销。
- **引用数组存的是引用**：`String[]` / `Object[]` 的元素区是一排引用，各自指向堆上真正的对象（可能为 null）——**不是**对象内联。这与 C 的 `struct T arr[]`（值内联）不同，更像 `T* arr[]`（指针数组）。要值内联的连续对象数组，Java 目前没有（Valhalla 项目的 value class 未来可能提供）。
- **多维数组是"数组的数组"**：`int[][]` 是一个元素为"int[] 引用"的数组，各行可不等长（锯齿数组），内存上**不保证**整体连续。C 的 `int m[3][4]` 是真正连续的行主序块——这是重要差异，做数值计算追求局部性时常改用一维 `int[]` 手动索引。

## 五、String：不可变 + 常量池 + intern

```java
String s = "abc";
System.out.println(s.length());        // 3
System.out.println(s.charAt(0));        // a
System.out.println(s.substring(1));     // bc
System.out.println(s.concat("d"));      // abcd（返回新串，s 本身不变）
```

`String` 是引用类型、**不可变**对象。内部（JDK9+ 的 Compact Strings）用一个 `byte[]` 存字符 + 一个 `coder` 标记编码（Latin-1 或 UTF-16）——**只要整串所有字符的码点都 ≤ 0xFF（能用 Latin-1 表示，含拉丁扩展、常见西文重音字母）就每字符 1 字节；一旦出现任一码点 > 0xFF 的字符（中文、emoji 等），整串改用 UTF-16、每字符 2 字节**。它**不以 `\0` 结尾**，长度是显式字段。

#### 不可变意味着什么

任何"修改"方法都返回**新对象**，原串不动：

```java
String s = "abc";
s.concat("d");            // 返回新串 "abcd"，但没接收，s 还是 "abc"
s = s.toUpperCase();      // "ABC" 是新对象，s 改指向它，旧的 "abc" 待回收
```

不可变的好处：可安全共享（多线程无需同步）、可缓存哈希、可放心用作 Map 的 key。代价：拼接产生大量中间对象——所以要用 `StringBuilder`。

#### 字符串常量池与 intern

**字符串字面量**（源码里的 `"hello"`）由 JVM 放进一个**字符串常量池**（driven by 运行时常量池，位于堆中）。相同字面量**共享同一个对象**：

```java
String a = "hi";
String b = "hi";
System.out.println(a == b);          // true！同一个池中对象

String c = new String("hi");         // new 强制在堆上另建新对象
System.out.println(a == c);          // false！不同对象
System.out.println(a == c.intern()); // true！intern() 返回池中的规范对象
```

内存图：

```
常量池（堆内）          普通堆
[ "hi" 对象P ]◀──a      [ "hi" 对象Q ]◀── c (new 出来的)
       ▲────── b
c.intern() 查池，返回 P；故 a == c.intern() 为 true
```

- `==` 对字面量常为 true 只是因为共享池对象——**别依赖它**，比内容永远 `equals`（呼应[第 3 章](03-变量与内存模型.md)）。
- `intern()` 把一个串登记/查找到池里，返回规范实例。大量重复字符串可借此省内存，但滥用会撑大池，慎用。
- 编译期能确定的常量拼接（`"a" + "b"`）会被 javac 直接折叠成 `"ab"` 字面量入池。

下面这个可运行例子把"编译期折叠"与"运行期拼接"的区别看得一清二楚：

```java
public class Folding {
    public static void main(String[] args) {
        String s           = "ab";
        String compileTime = "a" + "b";   // 两个字面量，javac 折叠成 "ab" 入池
        String c           = "a";         // 非 final 局部变量
        String runtime     = c + "b";     // 运行期用 StringConcatFactory 拼接，新对象
        System.out.println(s == compileTime);      // true  —— 同一池对象
        System.out.println(s == runtime);          // false —— 运行期新建
        System.out.println(s == runtime.intern()); // true  —— intern 取回池中规范对象
    }
}
```

只要参与拼接的都是编译期常量（字面量、`final` 常量），结果就在编译期折叠成一个字面量、进常量池；只要有一个运行期才知道的变量（如上面非 final 的 `c`），拼接就发生在运行期、产生堆上新对象。

#### StringBuilder：可变缓冲

需要频繁拼接、循环构造字符串时用 `StringBuilder`（非线程安全，单线程用它；线程安全版是 `StringBuffer`）：

```java
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 100; i++) {
    sb.append(i).append(',');   // 就地修改内部 char/byte 缓冲，不建中间 String
}
String result = sb.toString();  // 最后一次性生成 String
```

它内部维护一个可扩容的字节数组（类似 C 里手动 `realloc` 的缓冲区），`append` 就地追加，容量不够才扩容（通常翻倍）。**编译器会把 `"a" + b + "c"` 这类拼接自动优化**：JDK 9+（JEP 280）默认编成 `invokedynamic` + `StringConcatFactory`（`javap -c` 可见；JDK 8 及更早才是显式 `StringBuilder`）。但**循环里的 `+=` 每轮都是一次独立拼接、各自产生中间 `String`**，累积是 O(n²)，所以循环拼接务必手动复用一个 `StringBuilder`。

## 小结

- 8 种基本类型**定长跨平台、无 unsigned**；`char` 是 16 位 UTF-16 码元；`boolean` 大小未规定。
- 包装类把值装箱成堆对象；`Integer` 缓存 `[-128,127]`，导致 `==` 时小值 true、大值 false——比值用 `equals`。
- HotSpot 对象 = **对象头（Mark Word + Klass 指针）+ 实例字段 + 对齐填充**；Mark Word 按需复用同一块比特位承载哈希、GC 年龄、锁状态，用编码复杂度换每个对象更小的固定开销；字段会被重排；**压缩指针**利用"对象按 8 字节对齐、地址低 3 位恒为 0"这一事实，把引用压到 4 字节（<32GB 堆）。
- 数组是**堆上对象**：有 `length` 字段、有边界检查；元素连续，但引用数组存的是引用、多维是数组的数组（不保证整体连续）。
- `String` 不可变，内部 `byte[]`+编码标记、无 `\0`；**字面量共享常量池**，`intern()` 取规范实例；频繁拼接用 `StringBuilder`。
