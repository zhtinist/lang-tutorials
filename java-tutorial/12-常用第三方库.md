# 常用第三方库

C/C++ 的依赖管理长期是痛点:手写 Makefile/CMake,头文件 + 静态/动态库,vcpkg/Conan 才算晚近的统一方案。Java 生态早就靠 **Maven/Gradle + 中央仓库**统一了依赖分发。本章讲构建工具、坐标机制,以及主流库的生态定位——重点是"什么场景用什么",而非教 API。

## 构建与依赖管理

### 坐标与仓库

每个库由 **GAV 坐标**唯一标识:`groupId:artifactId:version`。类比 C++ 里"命名空间 + 库名 + 版本",但这里是全球统一命名空间(Maven Central)。

```
com.google.guava : guava : 33.0.0-jre
     |               |         |
   groupId       artifactId  version
```

**Maven**(`pom.xml`,XML,约定优于配置):

```xml
<dependencies>
    <dependency>
        <groupId>com.google.guava</groupId>
        <artifactId>guava</artifactId>
        <version>33.0.0-jre</version>
    </dependency>
</dependencies>
```

**Gradle**(`build.gradle`,Groovy/Kotlin DSL,更灵活、增量构建快):

```groovy
dependencies {
    implementation 'com.google.guava:guava:33.0.0-jre'
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.0'
}
```

构建工具从**仓库**下载 jar 到本地缓存(Maven `~/.m2/repository`),默认仓库是 **Maven Central**,可加私服(Nexus/Artifactory)。这一步取代了 C++ 里手动找库、配 include/link 路径。

### 传递依赖与 scope

关键机制:**传递依赖**。你依赖 A,A 依赖 B,构建工具自动拉取 B——无需你显式声明。类比 C++ 若有全自动的头文件+库依赖解析。

```
你的项目 ──> Jackson-databind ──> Jackson-core
                              └──> Jackson-annotations   (全自动拉取)
```

**依赖冲突**:不同库依赖同一库的不同版本时,Maven 用"最近路径优先 + 先声明优先"仲裁,Gradle 默认取最高版本。可用 `mvn dependency:tree` / `gradle dependencies` 查看依赖树,用 `<exclusions>` 排除。

**scope/配置**决定依赖在哪个阶段可见:

| Maven scope | Gradle 配置 | 含义 |
|-------------|-------------|------|
| compile(默认) | implementation/api | 编译+运行都需要 |
| provided | compileOnly | 编译需要,运行由容器提供(如 Servlet API) |
| runtime | runtimeOnly | 仅运行需要(如 JDBC 驱动) |
| test | testImplementation | 仅测试 |

> `implementation` vs `api`(Gradle):`implementation` 不把该依赖传递暴露给下游(封装,类比 PIMPL 隐藏实现),`api` 会暴露。合理用 `implementation` 能减少重编译范围、避免依赖泄漏。

## 生态定位

下面按用途分组,记住"选型"即可,API 用时再查。

### 通用工具:Guava、Apache Commons

- **Guava**(Google):`Preconditions` 参数校验、不可变集合 `ImmutableList`、`Multimap`/`BiMap`/`Cache`(本地缓存)、`Splitter/Joiner`、并发的 `ListenableFuture`。工程化程度高,现代项目首选。
- **Apache Commons**:`commons-lang3`(`StringUtils`、`ObjectUtils`)、`commons-collections4`、`commons-io`。历史悠久,很多老项目依赖。JDK 新版已内置不少同类功能(如 `String.strip`、`Files.readString`),按需取用。

### JSON:Jackson、Gson

序列化/反序列化对象与 JSON,靠**反射**读写字段。

```java
// Jackson:功能最全,Spring 默认,支持流式/树/数据绑定三种模式
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(person);        // 对象 -> JSON
Person p = mapper.readValue(json, Person.class);        // JSON -> 对象

// Gson(Google):API 更简洁,依赖少,轻量场景
Gson gson = new Gson();
String j = gson.toJson(person);
Person p2 = gson.fromJson(j, Person.class);
```

选型:Spring 项目直接用 Jackson(已集成);独立小工具用 Gson。

### 企业框架:Spring / Spring Boot

Java 后端事实标准。核心是 **IoC 容器 + DI + AOP**。

**IoC(控制反转)/ DI(依赖注入)**:对象不自己 `new` 依赖,而是声明需要什么,由容器创建并注入。类比"不自己管理生命周期,交给一个全局工厂+作用域管理器"。

```java
@Service                              // 声明为容器管理的 Bean
public class OrderService {
    private final PayClient payClient;
    // 构造器注入:容器启动时找到 PayClient 类型的 Bean 传进来
    public OrderService(PayClient payClient) { this.payClient = payClient; }
}
```

容器在启动时扫描注解、实例化 Bean、解析依赖图、按序注入(解决依赖顺序、单例复用、循环依赖检测)。

**AOP(面向切面)**:把横切关注点(事务、日志、权限)从业务代码里抽出,声明式织入。

```java
@Transactional                        // 方法自动包裹事务:开始->执行->提交/回滚
public void placeOrder() { /* 纯业务,无事务样板代码 */ }
```

底层靠**动态代理**:Spring 为 Bean 生成代理对象,方法调用先过代理(执行切面逻辑)再转发。两种实现:
- **JDK 动态代理**:目标实现了接口时,用 `java.lang.reflect.Proxy` 运行期生成接口的代理类。
- **CGLIB**:无接口时,运行期生成目标类的**子类**(字节码增强)覆盖方法。因此被代理方法不能是 `final`(无法覆盖)。

这套机制依赖**反射**(运行期读取类型信息、调用方法)——C++ 没有内建反射,Java 靠 `.class` 元数据 + `Class`/`Method` API 实现,代价是运行期开销和难以静态优化(Spring 启动慢部分源于此;GraalVM 原生镜像即为缓解此点)。

### ORM:Hibernate / JPA

- **JPA**:规范(接口),定义对象-关系映射的注解与 API。
- **Hibernate**:JPA 最主流的实现。

```java
@Entity                               // 映射到数据库表
public class User {
    @Id @GeneratedValue Long id;      // 主键,自增
    @Column(nullable = false) String name;
}
// 用 EntityManager/Repository 操作对象,框架翻译成 SQL
```

把关系表映射为对象,自动生成 SQL、管理会话与缓存。省去手写 JDBC 样板,但要警惕 N+1 查询、懒加载陷阱等。

### 测试:JUnit5 + Mockito

```java
// JUnit5:单元测试框架
class CalcTest {
    @Test
    void add() {
        assertEquals(3, new Calc().add(1, 2));
    }
    @ParameterizedTest
    @ValueSource(ints = {1, 2, 3})
    void positive(int n) { assertTrue(n > 0); }
}

// Mockito:mock 依赖,隔离被测单元
UserRepo repo = mock(UserRepo.class);
when(repo.findById(1L)).thenReturn(new User());   // 打桩:规定 mock 行为
UserService svc = new UserService(repo);
svc.load(1L);
verify(repo).findById(1L);                        // 验证交互发生
```

### 编译期增强:Lombok

用注解在**编译期生成字节码**,消除 getter/setter/构造器/`toString` 等样板代码。

```java
@Data                                 // 生成 getter/setter/equals/hashCode/toString
@AllArgsConstructor
public class Point {
    private int x;
    private int y;
    // 源码里没有任何方法,编译后字节码里都有了
}
```

原理:Lombok 是一个 **注解处理器(annotation processor)**,挂进 javac 编译流程,在编译期修改 AST(抽象语法树)注入方法。它不是运行期反射,零运行开销。代价:IDE 需装插件才能识别生成的方法;对 javac 内部 API 有依赖,JDK 大版本升级偶有兼容问题。

### 日志:SLF4J + Logback

**SLF4J** 是日志**门面**(接口),`Logback`/`Log4j2` 是**实现**。代码只依赖门面,运行期绑定具体实现——类比"面向接口编程",可换实现不改代码。

```java
private static final Logger log = LoggerFactory.getLogger(MyClass.class);
log.info("user {} logged in", userId);   // 占位符 {} 惰性拼接,级别不够时不构造字符串,省开销
log.error("failed", exception);
```

配置文件(`logback.xml`)控制级别、输出格式、滚动策略,无需改代码。

> "门面依赖接口、运行期绑定实现"的底层机制,就是 JDK 自带的 **SPI(Service Provider Interface)**:`java.util.ServiceLoader`。SLF4J 正是靠它在运行期发现并绑定 `Logback` 等实现。下面用纯 JDK 演示这套机制——**无需任何第三方库**。

调用方只依赖接口 `Greeter`,实现类通过 `META-INF/services/` 下的配置文件注册,运行期由 `ServiceLoader` 发现,`new`/`import` 都不出现在调用方代码里:

```java
// spi/Greeter.java —— "门面":调用方只依赖它
package spi;
public interface Greeter { String greet(String name); }

// spi/EnglishGreeter.java —— 实现之一
package spi;
public class EnglishGreeter implements Greeter {
    public String greet(String name) { return "Hello, " + name; }
}

// spi/FrenchGreeter.java —— 实现之二
package spi;
public class FrenchGreeter implements Greeter {
    public String greet(String name) { return "Bonjour, " + name; }
}

// spi/Main.java —— 调用方:运行期发现实现,不引用任何实现类
package spi;
import java.util.ServiceLoader;
public class Main {
    public static void main(String[] args) {
        ServiceLoader<Greeter> loader = ServiceLoader.load(Greeter.class);
        for (Greeter g : loader) {   // 运行期发现,调用方不 new/import 具体实现
            System.out.println(g.getClass().getSimpleName() + " -> " + g.greet("Ada"));
        }
    }
}
```

注册文件(纯文本,每行一个实现类全名),放在类路径的 `META-INF/services/` 下,**文件名 = 接口全名**:

```
文件:META-INF/services/spi.Greeter
spi.EnglishGreeter
spi.FrenchGreeter
```

编译运行(`out` 目录同时含 class 与 `META-INF/services/`):

```bash
javac -d out spi/*.java
java -cp out spi.Main
# EnglishGreeter -> Hello, Ada
# FrenchGreeter -> Bonjour, Ada
```

新增一种实现只需加一个类 + 在注册文件里加一行,调用方**零改动**——这正是"面向接口编程 + 可插拔实现"的 JDK 原生支撑,也是 JDBC 驱动、SLF4J 绑定、`java.nio.file.spi` 等的共同底座。

### 高性能网络:Netty

异步事件驱动的网络框架,基于 Java NIO(多路复用 epoll/kqueue),用 **Reactor 模式** + 少量线程处理海量连接。RPC 框架(gRPC-Java、Dubbo)、消息中间件底层多用它。

- **EventLoop**:一个线程绑一个 selector,轮询多路 channel 事件——类比 C 里手写 epoll 事件循环,但封装成 pipeline。
- **ChannelPipeline**:一串 handler,数据入站/出站依次经过(编解码、业务处理),类似责任链。
- 池化的 `ByteBuf`(堆外内存 + 引用计数),减少 GC 与拷贝——接近 C 里手动管理缓冲区,但需注意手动 `release` 防泄漏。

适用:自研高并发长连接服务。普通 HTTP 服务用 Spring Boot(其内嵌的 Tomcat/Netty)即可,不必直接写 Netty。

## 打包:jar 与 fat-jar

- **普通 jar**:只含本项目 class + 资源,不含依赖。运行需在 classpath 手动列出所有依赖 jar,类比只编译出 `.o` 还没链接库。
- **fat-jar / uber-jar**:把所有传递依赖打进一个可执行 jar,`java -jar app.jar` 直接跑。Spring Boot 用 `spring-boot-maven-plugin`,普通项目用 `maven-shade-plugin` 或 Gradle `shadow` 插件。

```bash
mvn package                       # 产出 target/app.jar
java -jar target/app.jar          # fat-jar 直接运行
java -cp "app.jar:libs/*" Main    # 普通 jar 需指定 classpath
```

> fat-jar 里多个依赖含同名资源(如各自的 `META-INF/services`)会冲突,shade 插件的 transformer 负责合并。

## JNI:调用 C 的代价

Java 通过 **JNI(Java Native Interface)** 调 C/C++(如需复用现有本地库、做 CPU 密集计算)。但代价不小,通常是最后手段:

```java
public class NativeLib {
    static { System.loadLibrary("mylib"); }   // 加载 libmylib.so/.dll
    public native int compute(int[] data);    // native 方法,实现在 C 侧
}
```

代价来源:
- **调用开销**:每次跨越 JNI 边界有固定成本(参数封送、状态切换),不适合高频细粒度调用。
- **对象封送**:Java 对象/数组在托管堆,C 侧访问需 `GetArrayElements` 等,可能触发拷贝或**钉住(pin)对象**阻碍 GC 移动/压缩。
- **安全性**:native 代码崩溃(段错误)会直接**带崩整个 JVM**,失去 JVM 的内存安全保障。
- **可移植性**:需为每个平台编译本地库,破坏 Java "一次编写到处运行"。

> JDK 正推 **FFM API(Foreign Function & Memory,JEP 454,JDK22 正式)** 作为 JNI 的现代替代,纯 Java 声明本地调用与堆外内存,无需写 C 胶水代码,开销更低、更安全。新项目优先考虑它。
