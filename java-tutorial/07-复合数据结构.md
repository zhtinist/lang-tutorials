# 集合框架的内部实现

C++ 里你用 `std::vector`、`std::list`、`std::unordered_map`、`std::map`。Java 的集合框架(Java Collections Framework, JCF)在概念上一一对应,但**内部实现细节、内存布局、扩容策略与迭代器语义**都值得逐一掰开看。本章的核心是:每种容器在堆上到底摆了什么,拷贝的是值还是引用,复杂度从哪来。

关于"引用 vs 值"的基础,参见[「变量与内存模型」](03-变量与内存模型.md);对象头布局参见[「数据类型与内存布局」](04-数据类型.md)。

## 一、数组:一切的地基

Java 数组是**真正的对象**,在堆上分配,不像 C 那样只是一段裸内存 + 指针算术。

```java
int[] a = new int[4];        // 堆上分配:对象头 + length 字段 + 4 个连续 int
String[] s = new String[3];  // 堆上:3 个连续的“引用槽”,初值 null
```

### 内存布局(HotSpot,64 位,压缩指针)

```
int[4] 在堆上:
┌──────────┬──────────┬────────┬────┬────┬────┬────┐
│ mark word│ klass ptr│ length │ [0]│ [1]│ [2]│ [3]│
│  8 bytes │  4 bytes │ 4 bytes│int │int │int │int │
└──────────┴──────────┴────────┴────┴────┴────┴────┘
             对象头共 16 字节        元素连续存放,cache 友好
```

与 C 的关键区别:

| 特性 | C/C++ 数组 | Java 数组 |
|------|-----------|-----------|
| 本质 | 裸内存,`a[i]` = `*(a+i)` | 堆上对象,含 length 字段 |
| 长度 | 编译期常量或需另存 | `a.length` 随身携带 |
| 边界检查 | 无,越界即 UB | 每次访问检查,越界抛 `ArrayIndexOutOfBoundsException` |
| 存放对象 | 可存对象值(连续) | 只能存引用(`Object[]` 是引用数组) |
| 协变 | 无 | **协变**:`Object[] o = new String[3];` 合法 |

### 数组协变的坑

Java 数组协变意味着 `String[]` IS-A `Object[]`,这在编译期通过,但运行时每次写入都要做类型检查:

```java
Object[] arr = new String[3];
arr[0] = "ok";           // 正常
arr[1] = Integer.valueOf(1); // 编译通过!运行时抛 ArrayStoreException
```

C++ 的模板是不变(invariant)的,`vector<string>` 与 `vector<Object>` 毫无关系。Java 数组这个设计是历史包袱(泛型出现前为了让 `Arrays.sort(Object[])` 能用),代价是运行时 store check。泛型集合(下文)则是不变的。

### 边界检查开销与消除

每次 `a[i]` 都插入 `0 <= i < a.length` 判断。JIT 有 **范围检查消除**(Range Check Elimination):在 `for (int i=0;i<a.length;i++)` 这类可证明安全的循环里,C2 会把检查提到循环外或删除,使性能逼近 C。所以规范层面有开销,实现层面常常被优化掉。

## 二、Collection 体系全景

```
                 Iterable
                     │
                Collection ───────────────┐
                 │      │                  │
               List    Set               Queue
              /  │  \    │  \              │   \
     ArrayList  │  Vector│   \          ArrayDeque Deque
     LinkedList │       HashSet         PriorityQueue
     (LinkedList 同时是 Queue/Deque)  LinkedHashSet
                          TreeSet(SortedSet)

  Map(不属于 Collection,单独一支)
   │    │        │
 HashMap TreeMap LinkedHashMap
 (SortedMap = TreeMap)
```

要点:`Map` **不**继承 `Collection`,是平行的一支。`List`/`Set`/`Queue` 都是 `Collection`。

## 三、ArrayList vs LinkedList

### ArrayList:动态数组

内部就是一个 `Object[] elementData` + `int size`:

```java
// 简化的 JDK 源码结构
transient Object[] elementData; // 真正存元素的数组
private int size;               // 逻辑大小(<= elementData.length 容量)
```

内存图:

```
ArrayList 对象(堆)          elementData 数组(堆)
┌───────────────┐          ┌────┬────┬────┬────┬────┬────┐
│ elementData ──┼────────▶ │ e0 │ e1 │ e2 │null│null│null│
│ size = 3      │          └────┴────┴────┴────┴────┴────┘
└───────────────┘           capacity = 6, size = 3
```

注意存的是**引用**(`Object[]`),对象本体散落在堆各处;不像 C++ `vector<Point>` 把 Point 值连续内联。所以 Java 的 `ArrayList<Point>` 遍历时是"数组连续 → 逐个跳到堆各处",cache 局部性不如 C++ 的值语义 vector。要连续存基本类型,得用原生 `int[]` 或第三方原生集合。

**扩容**:默认容量 10,满了按 **1.5 倍** 增长(`newCap = oldCap + (oldCap >> 1)`),`Arrays.copyOf` 分配新数组并拷贝引用(浅拷贝,只拷指针值,不复制对象本体)。

```
size=10 已满,add 第 11 个:
新容量 = 10 + 5 = 15,分配 Object[15],拷贝 10 个引用过去,旧数组待 GC
```

对比 C++ `std::vector` 常见 2 倍扩容(libstdc++ 用 2x,MSVC 用 1.5x)。均摊 O(1) 追加。

### LinkedList:双向链表

```java
private static class Node<E> {
    E item;
    Node<E> next;
    Node<E> prev;
}
```

每个节点是独立堆对象:对象头 16 字节 + 3 个引用(item/next/prev,压缩指针各 4 字节)≈ 32 字节/节点,存一个元素开销巨大。内存图:

```
first ─▶┌────┬────┬────┐   ┌────┬────┬────┐   ┌────┬────┬────┐◀─ last
        │prev│item│next┼──▶│prev│item│next┼──▶│prev│item│next│
        │null│ e0 │  ●─┘   │ ●  │ e1 │ ●─┘   │ ●  │ e2 │null│
        └────┴────┴────┘◀──┴────┴────┴────┘◀──┴────┴────┴────┘
```

对应 C++ `std::list`。节点散布堆中,遍历 cache-unfriendly。`get(i)` 是 O(n)(从最近的一端走)。**实践中 `ArrayList` 几乎总是更好**,即便频繁头部插入,`ArrayDeque` 也比 `LinkedList` 快。`LinkedList` 唯一优势是持有节点引用时的 O(1) 删除,但集合 API 里你通常拿不到节点。

### 复杂度对照

| 操作 | ArrayList | LinkedList |
|------|-----------|-----------|
| `get(i)` | O(1) | O(n) |
| `add`(尾) | 均摊 O(1) | O(1) |
| `add(i, e)`(中间) | O(n)(搬移) | O(n)(定位)+O(1) |
| 头部插入 | O(n) | O(1) |
| `contains` | O(n) | O(n) |
| 内存/元素 | 1 个引用 + 容量冗余 | ~32 字节节点 |

## 四、HashMap:重头戏

对应 C++ `std::unordered_map`,但实现细节差别很大。

### 结构:桶数组 + 链表/红黑树

```java
transient Node<K,V>[] table;  // 桶数组,长度总是 2 的幂
static class Node<K,V> {
    final int hash;   // 缓存的扰动后哈希
    final K key;
    V value;
    Node<K,V> next;   // 链表指针(拉链法解决冲突)
}
```

内存图(冲突时拉链):

```
table (Node[16])
┌────┐
│ 0  │──▶ null
│ 1  │──▶ [h,k1,v1] ──▶ [h,k9,v9] ──▶ null   (bucket 1 有两个键冲突)
│ 2  │──▶ null
│... │
│ 5  │──▶ [h,k5,v5] ──▶ null
└────┘
```

### 哈希扰动函数

桶下标 = `hash & (n-1)`(n 是 2 的幂,等价取模但更快)。若直接用 `key.hashCode()`,高位不参与运算(因为 `& (n-1)` 只保留低位),高位不同、低位相同的键会撞。扰动把高 16 位异或到低位:

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

### 冲突加剧时 treeify

同一个桶的链表长度 **≥ 8** 且 **table 长度 ≥ 64** 时,链表转为**红黑树**,把该桶最坏查找从 O(n) 降到 O(log n),防哈希碰撞攻击(DoS)。树节点数退到 **≤ 6** 时退化回链表(留 6/8 的滞后区间避免频繁转换)。

```
链长 >= 8 且 table.length >= 64:
bucket ──▶ 链表 ─treeify─▶ 红黑树(TreeNode)
若 table 还没到 64,优先扩容而非树化
```

### 负载因子与扩容 rehash

- **负载因子 0.75**:`size > capacity * 0.75` 时扩容为**两倍**。0.75 是空间/时间折中(经泊松分布分析,冲突概率与内存浪费平衡)。
- **扩容 rehash**:分配 2 倍新桶数组,重散列每个元素。因容量是 2 的幂且翻倍,元素新位置只有两种:`原位置` 或 `原位置 + 旧容量`。JDK 8 用这个性质避免重算 hash,只看 `hash & oldCap` 那一位是 0 还是 1,把旧链拆成 lo/hi 两条链,分别放到新表两个位置。O(n) 但常数小。

```
旧 table.length=16,扩到 32:
某桶的链表按 (hash & 16) 拆成两条 →  桶 j 和桶 j+16
```

### 关键契约:equals/hashCode

`HashMap` 定位分两步:先按 hash 找桶,再在桶内用 `equals` 逐个比。所以自定义 key **必须**同时正确重写 `equals` 和 `hashCode`,否则相等的对象散到不同桶就找不回来。详见[「面向对象与类型系统」中的 equals/hashCode 契约](08-面向对象与类型系统.md)。C++ 里同理:`unordered_map` 需要 `std::hash` 特化 + `operator==`。

### 复杂度

| 操作 | 平均 | 最坏(树化后) |
|------|------|--------------|
| `get`/`put`/`remove` | O(1) | O(log n) |
| 遍历 | O(capacity + size) | 同 |

注意:`HashMap` **非线程安全**;并发下扩容 JDK7 会成环死循环(JDK8 修复但仍可能丢数据)。并发用 `ConcurrentHashMap`,见[「并发模型」](10-并发模型.md)。

## 五、LinkedHashMap / TreeMap

### LinkedHashMap

在 `HashMap` 的 Node 上再加 `before`/`after` 双向链,维护**插入顺序**(或访问顺序 `accessOrder=true`,可实现 LRU 缓存)。哈希查找仍 O(1),额外维护一条贯穿所有节点的链表。

```
HashMap 的桶结构 + 一条横穿所有节点的双向链(记录顺序)
head ──▶ node ──▶ node ──▶ node ──▶ tail
```

### TreeMap

基于**红黑树**(自平衡 BST),对应 C++ `std::map`。按 key 排序(自然序或 `Comparator`)。所有操作 O(log n),提供 `firstKey`/`lastKey`/`floorKey`/`ceilingKey`/`subMap` 等有序视图。key 需可比较(`Comparable` 或传 `Comparator`),见[「标准库算法」](11-标准库算法.md)。

## 六、HashSet / ArrayDeque

- **HashSet**:内部就是一个 `HashMap`,元素当 key,value 是一个共享的 dummy `Object`。所以 `HashSet` 的一切特性等于 `HashMap` 的 keySet。对应 C++ `unordered_set`。
- **LinkedHashSet**:内部 `LinkedHashMap`,保插入序。
- **TreeSet**:内部 `TreeMap`,有序,对应 `std::set`。
- **ArrayDeque**:**循环数组**实现的双端队列,`head`/`tail` 两个索引在环形缓冲上移动,扩容翻倍。作栈或队列都比 `Stack`(过时,继承自 `Vector`,带同步)和 `LinkedList` 快。**首选的栈/队列实现**。不允许 null(用 null 表示空槽)。

```
ArrayDeque 循环数组:
┌───┬───┬───┬───┬───┬───┐
│ e │ e │   │   │ e │ e │   head=4, tail=2 (环绕)
└───┴───┴───┴───┴───┴───┘
```

## 七、泛型集合与类型擦除的影响

`ArrayList<String>` 在运行时就是 `ArrayList`,内部 `Object[]`;泛型只在编译期检查并自动插入强制转换。这就是为什么不能 `new T[]`、集合里存的永远是引用(基本类型被自动装箱)。擦除机制详见[「泛型与类型擦除」](08-面向对象与类型系统.md)。

后果:`ArrayList<Integer>` 每个元素是堆上的 `Integer` 对象 + 数组里的引用,内存与拆装箱开销远大于 C++ `vector<int>`。性能敏感场景用原生数组或第三方原生集合(Eclipse Collections、fastutil)。

## 八、迭代器与 fail-fast

集合内部维护 `modCount`(结构性修改计数)。迭代器创建时记下 `expectedModCount`,每次 `next()` 校验;若期间集合被非迭代器途径修改,抛 `ConcurrentModificationException`。

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3));
for (Integer x : list) {      // 增强 for 底层用 Iterator
    if (x == 2) list.remove(x); // 结构修改 → modCount 变 → 下次 next() 抛异常
}
// 正确做法:用 iterator.remove() 或 list.removeIf(x -> x == 2)
```

fail-fast 是**尽力而为**的错误检测(不保证一定触发),不是并发保护;真正并发用 `CopyOnWriteArrayList`/`ConcurrentHashMap`(弱一致迭代器,不抛此异常)。C++ STL 里对应的是"迭代器失效"——但 STL 是 UB,Java 主动抛异常帮你抓早。

## 九、与 C++ STL 容器速查对照

| Java | C++ STL | 底层 |
|------|---------|------|
| `ArrayList` | `vector` | 动态数组(Java 存引用,C++ 存值) |
| `LinkedList` | `list` | 双向链表 |
| `ArrayDeque` | `deque` | 循环数组(C++ deque 是分段数组) |
| `HashMap` | `unordered_map` | 拉链哈希(Java 会树化) |
| `TreeMap` | `map` | 红黑树 |
| `HashSet` | `unordered_set` | 哈希 |
| `TreeSet` | `set` | 红黑树 |
| `PriorityQueue` | `priority_queue` | 二叉堆 |
