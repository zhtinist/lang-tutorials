# Error Handling, Resource Management, and Garbage Collection

Go has no exceptions (no exceptions), no `try/catch`, and no destructors doing RAII. It treats errors as **ordinary return values**, hands resource cleanup to `defer`, and hands memory reclamation to a **concurrent garbage collector**. For a C++ programmer, these are three places where you need to rebuild your muscle memory, and each one corresponds to a concrete design trade-off.

## I. error is an ordinary interface value

`error` is not a keyword — it is just a perfectly ordinary interface in the standard library, no different from any interface you'd define yourself:

```go
type error interface {
    Error() string
}
```

Any type that implements `Error() string` is an error. In memory it is [the two-word interface `iface` covered in Chapter 08](08-面向对象与类型系统.md) — `{itab, data}`. Returning an error involves no stack unwinding and no zero-cost exception tables; it just pushes a 16-byte interface value into the return stack slot, as cheap as returning any other value.

The idiom is "multiple return values + if err != nil":

```go
f, err := os.Open("data.txt")
if err != nil {
    return fmt.Errorf("failed to open config: %w", err) // handle or wrap, then propagate
}
defer f.Close()
// happy path
```

A few ways to create errors:

```go
errors.New("something failed")           // simplest, returns *errorString
fmt.Errorf("port %d in use", port)        // with formatting
fmt.Errorf("read config: %w", err)        // %w wraps the underlying error (keeps the chain)
```

`%w` (wrap) is the key: it embeds `err` as the "wrapped cause" inside the new error, forming an error chain that `errors.Is`/`errors.As` can later unwrap layer by layer. Using `%v` just splices the text in, and the chain breaks.

**Why use a `(result, error)` return value instead of an exception, and what the real difference from a Java checked exception is.** In C++ and most languages with exceptions, error handling is **implicit control flow**: a single `throw` automatically jumps up an arbitrary number of stack frames until it hits a matching `catch`, and nothing in the signature of any function in between reveals that it might fail. Go instead makes errors **explicit data flow**: whether something can fail is part of the function's return type, spelled out right in the signature, and the caller must explicitly receive that return value — `err` just sits there, and the compiler won't complain if you never check it, but code review and static-analysis tools (`errcheck` and the like) can catch the obvious slip of "ignored the err," and more importantly every caller at every layer is forced to visually confront the fact that "this might fail."

It's worth comparing this with Java's checked exceptions, which are actually trying to solve the same problem — forcing the caller to handle the failure path at compile time, and `throws IOException` in a method signature does make "this method might fail" visible too. But the two solutions land very differently in practice: a checked exception is still built on the same implicit-control-flow machinery as any exception (a `throw` still jumps up and unwinds however many stack frames to reach some `catch`), stacking "visibility at the type level" on top of "invisibility at the control-flow level." The result is that Java code is full of boilerplate whose sole purpose is to catch an exception and immediately rethrow it wrapped as an unchecked `RuntimeException`, just to dodge the signature annotation — which is exactly why checked exceptions have been controversial in the Java community for years, to the point that many major frameworks and libraries just throw unchecked exceptions instead. Go's `(result, error)` fuses "might fail" and "control flow" into one thing: an error is just an ordinary value that can be stored, passed around, composed, and wrapped (`%w`), traveling the exact same path as any other return value, with no separate "exception channel" causing out-of-order jumps in control flow. What you get in exchange is control flow that is completely visible, with no hidden jumps and no stack-unwinding overhead; the price is verbosity — `if err != nil { return err }` has to be repeated after nearly every call that might fail, which is the long-standing source of Go's reputation for boilerplate, but it's also this design's uncompromising commitment to the word "explicit."

## II. Sentinel errors, custom errors, Is/As

**Sentinel errors**: predefined error values that can be compared with `==` or `errors.Is`:

```go
var ErrNotFound = errors.New("not found")

func Lookup(k string) (string, error) {
    // ...
    return "", ErrNotFound
}

if errors.Is(err, ErrNotFound) { /* special-case it */ }
```

**Why use `errors.Is` rather than `err == ErrNotFound`?** Because `err` may have been wrapped several layers deep with `%w`; `==` only compares the outermost layer, whereas `errors.Is` walks the `%w` chain, `Unwrap`ping and comparing at every level.

**Custom error types**: if you want to carry structured information, define a type:

```go
type PathError struct {
    Op   string
    Path string
    Err  error
}
func (e *PathError) Error() string {
    return e.Op + " " + e.Path + ": " + e.Err.Error()
}
func (e *PathError) Unwrap() error { return e.Err } // lets Is/As see through it
```

`errors.As` pulls out the first error in the chain that matches the target type:

```go
var pe *PathError
if errors.As(err, &pe) {
    fmt.Println("failing path:", pe.Path) // access the structured field
}
```

| Function | Purpose | Analogy |
|------|------|------|
| `errors.Is(err, target)` | Is there an error in the chain equal to target | Matching by "value" |
| `errors.As(err, &v)` | Is there an error in the chain assignable to v; extract it | Matching by "type" (like C++ `catch(SpecificType&)`) |
| `errors.Unwrap(err)` | Peel off one layer | Walk the chain manually |

`errors.Is`/`errors.As` are both just a loop that repeatedly walks down via the `Unwrap() error` method — not compiler magic: at each layer, if it doesn't match, it calls that layer's `Unwrap()` to get the next one, until the chain ends (`Unwrap` returns nil). `errors.Is` compares with `==` by default, but prefers a type's own `Is(error) bool` if it defines one; `errors.As` uses reflection to check whether each layer's dynamic type in the chain is assignable to the type the target pointer points to. This chain is, at bottom, just a series of "nodes" holding onto each other through interface values, no different in essence from walking a linked list — which is one more confirmation of the "error is just an ordinary value" design: the error chain needs no special runtime support at all; it's assembled entirely out of ordinary interface method calls.

Runnable: %w wrapping + Is + As:

```go
package main

import (
	"errors"
	"fmt"
)

var ErrNotFound = errors.New("not found")

type QueryError struct {
	Query string
	Err   error
}

func (e *QueryError) Error() string { return e.Query + ": " + e.Err.Error() }
func (e *QueryError) Unwrap() error { return e.Err }

func find() error {
	qe := &QueryError{Query: "SELECT", Err: ErrNotFound} // wraps the sentinel error
	return fmt.Errorf("db call: %w", qe)                 // wraps once more with %w
}

func main() {
	err := find()
	fmt.Println(err)                         // db call: SELECT: not found
	fmt.Println(errors.Is(err, ErrNotFound)) // true   ← walks the whole %w/Unwrap chain

	var qe *QueryError
	fmt.Println(errors.As(err, &qe), qe.Query) // true SELECT   ← pulls the concrete type out of the chain
}
```

## III. panic / recover: not an exception

`panic` is for **genuine programming errors that should never happen** (an out-of-bounds index, dereferencing nil, a broken invariant), not a routine error-handling tool. `recover` is effective only when called inside a `defer`; it "catches" the panic and lets the program resume normal execution:

```go
func safeRun() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r) // turn the panic into a returned error
        }
    }()
    riskyOperation()
    return nil
}
```

A typical use: wrap each request handler of a web server in a recover, so that a single request's panic doesn't take down the whole process. **But don't use panic/recover to simulate try/catch for routine flow control** — that's an anti-pattern, slow and against Go convention.

**Why panic/recover exist at all, yet aren't recommended for everyday use.** This is the other half of a design that echoes the previous section: Go's philosophy splits "errors" into two categories of an entirely different nature. **Expected failures** (a missing file, a network timeout, invalid user input) are a normal part of the program's operation, and the caller ought to know they can happen and be ready to handle them — these are handled with `error` return values, as in the previous section. **Programming errors that should never happen** (an out-of-bounds index, dereferencing nil, a broken invariant that was supposed to be asserted) represent a case where the program's own logic has already gone wrong, and the state it's left in downstream can no longer be trusted — this is where `panic` belongs. Mixing the two into a single channel (as many languages do by handling everything with exceptions) leaves the caller unable to tell from the type signature whether "this is an error I expected and need to handle" or "this means the program is already broken and the right move is to stop as soon as possible rather than keep limping along." The reason `recover` exists at all, rather than letting every panic unconditionally kill the process, is to backstop things **at a boundary** rather than to be abused inside everyday business logic: a long-running service shouldn't crash entirely just because a single request triggered an out-of-bounds access that should never have happened — a request-level recover isolates and logs that kind of incident without it spilling over into the other requests currently being handled. But wrapping every single expected failure in a `defer recover` to simulate `try/catch` is both slower than checking `error` (the unwinding machinery is far heavier than ordinary return-value passing) and defeats the very point of "layering errors" — it hides a failure that should have been made explicit in the type signature back behind an implicit jump.

`panic(x)` will: immediately stop the current function's normal execution → **unwind up the call stack layer by layer**, running each layer's registered `defer`s → if not recovered along the way, reach the top of the goroutine, print a stack traceback, and **`exit(2)` to terminate the entire process** (not just kill the current goroutine).

```
panic unwinding:
main → A() → B() → panic!
                    ↓ run B's defers
              ↑ run A's defers
        ↑ run main's defers (if no recover) → process crashes
```

This unwinding process runs `defer`s, but there is **no C++-style type-based catch matching** — recover catches "any panic," without distinguishing types.

`recover()` is effective only when called directly inside a `defer`red function. It **stops the unwinding**, returns the value that was passed to panic, and lets the program resume normal execution at the current stack frame.

Runnable: turning a runtime panic into an error:

```go
package main

import "fmt"

func safeDiv(a, b int) (result int, err error) {
	defer func() {
		if r := recover(); r != nil {
			err = fmt.Errorf("recovered: %v", r) // turn the panic into a returned error
		}
	}()
	return a / b, nil // b==0 triggers a runtime panic before the return completes
}

func main() {
	r, err := safeDiv(10, 2)
	fmt.Println(r, err) // 5 <nil>
	r, err = safeDiv(10, 0)
	fmt.Println(r, err) // 0 recovered: runtime error: integer divide by zero
}
```

Note that `result` is still 0 on the panic branch: the panic fires while evaluating `a / b`, so the return value is never assigned — only the named return `err` gets set inside the deferred `recover`.

For comparison:

| | C++ exceptions | Go panic/recover |
|--|---------|------------------|
| Purpose | Routine error handling | Only fatal errors / broken invariants |
| Trigger | `throw` | `panic()` or a runtime error |
| Catch | `catch(Type&)`, by type | `recover()` catches anything, you decide |
| Uncaught | `std::terminate` | Process `exit(2)` |
| Cleanup | Destructors (RAII) | `defer` |
| Routine errors | Also uses exceptions | Uses return-value errors |

## IV. Resource management: defer instead of RAII

Go has no destructors. Resource cleanup relies on `defer`: registered calls run before the function returns (including during panic unwinding), in **last-in-first-out** order ([see the mechanics in Chapter 05](05-控制流.md)).

```go
fmt.Println("start")
defer fmt.Println("cleanup 1") // LIFO: runs last
defer fmt.Println("cleanup 2") // runs first
fmt.Println("end")
// output order: start end cleanup2 cleanup1
```

In practice it's commonly used to pair "open/close" or "lock/unlock":

```go
f, err := os.Open(name)
if err != nil { return err }
defer f.Close()           // whatever happens next — return or panic — it will be closed

mu.Lock()
defer mu.Unlock()         // pairs lock/unlock, so you never forget to unlock
```

The difference from C++ RAII: RAII is bound to the **object's lifetime/scope** (destructed on leaving the block); `defer` is bound to the **function's return** (not block-level — function-level! A `defer` inside a loop piles up and doesn't run until the function ends, which is a common source of resource leaks). If you need prompt, block-level release, call the cleanup explicitly, or factor the block out into a small function. **Why Go chose function-level `defer` instead of block-level destruction**: RAII's deterministic destruction depends on an object's lifetime being statically determinable at compile time, which holds for C++ stack objects — but in Go, whether an object ends up on the heap and when its lifetime ends is decided jointly by escape analysis and the GC (see below), so the compiler can't insert a destructor call at the precise instant of "leaving these braces" the way C++ can. `defer` settles for the next best thing: a moment that can always be pinned down at compile time — **the function's return** — a moment every function passes through whether it returns normally or unwinds via panic, which is enough to cover the core need of "resource management," namely "whatever you open, you must close" — it's just coarser-grained than block scope.

> `runtime.SetFinalizer` can register a "finalizer to run when the GC reclaims an object," but its timing is nondeterministic and it's only guaranteed to run once, so it **cannot substitute for defer/an explicit Close** in managing deterministic resources like files or locks.

## V. Garbage collection (GC)

Go is a garbage-collected language. You don't write `free`/`delete`, and you don't manage lifetimes yourself; the compiler's [escape analysis (Chapter 03)](03-变量与内存模型.md) decides whether a variable lives on the stack or the heap — stack ones get reclaimed along with the stack frame, heap ones are reclaimed by the GC. Writing everyday business code, you basically never have to think about the GC — only in the rare case where tuning is needed do these "knobs" come into play:

```go
debug.SetGCPercent(200) // turn it up → less GC, higher memory use, better throughput
debug.SetGCPercent(50)  // turn it down → more frequent GC, lower memory use
runtime.GC()            // manually force one (blocking; generally only for tests/special cases)
```

**Concurrent tri-color mark-sweep.** Go's GC is **concurrent, non-moving, tri-color mark-sweep**, and its core goal is an **extremely short STW (stop-the-world)** — and that goal is itself the product of a design trade-off: Go targets building long-running, latency-sensitive server-side programs (network services, infrastructure components), and what these programs fear most isn't "slightly lower average throughput" but "an occasional stall of tens to hundreds of milliseconds." So the GC is designed to happily burn more CPU on concurrent marking if that's what it takes to drive pause times as low as possible — a different trade-off direction from many throughput-first or batch-oriented GCs (such as some of Java's older collectors).

The tri-color abstraction:

- **White**: not yet visited, a candidate for collection.
- **Gray**: visited, but the objects it references have not all been scanned yet (sitting in the pending queue).
- **Black**: visited, and every object it references has already been scanned.

The flow: starting from the roots (stacks, global variables), color the roots gray; repeatedly take an object from the gray set, color the white objects it references gray, and turn itself black; keep going until there is no gray left. Whatever white remains is garbage, and gets swept. **The mark phase runs concurrently with user goroutines** (mark workers use a portion of the CPU) — it is not a long pause.

**Write barrier.** The trouble with concurrent marking: as the user program keeps mutating pointers while marking runs concurrently, you can end up with "a black object comes to point at a white object, while the only gray reference that pointed to that white object gets deleted" — the white object is missed by the mark and wrongly collected.

The fix is the **write barrier**: during marking, the compiler inserts a small hook of code at every pointer write, re-graying the objects involved so nothing gets missed. Go uses a hybrid write barrier (a Dijkstra + Yuasa hybrid), which means **stacks need almost no STW re-scan**. The write barrier is only active during the GC's mark phase, and is off the rest of the time.

```
one round of concurrent GC (simplified):
[extremely short STW: turn on the write barrier] → [concurrent marking, user program runs alongside it, protected by the write barrier] →
[extremely short STW: mark termination] → [concurrent sweep, alongside the user program]
both STWs are typically sub-millisecond to a few milliseconds
```

**Trigger timing and GOGC.** GC is driven by **heap growth ratio**, controlled by the environment variable/parameter `GOGC` (default 100):

> The next round fires when the live heap has grown by `GOGC%` relative to right after the last GC. `GOGC=100` means one GC every time the heap doubles (grows by 100%).

Go 1.19+ also provides a **soft memory limit**, `GOMEMLIMIT` / `debug.SetMemoryLimit`, as a backstop for memory-constrained environments like containers.

**On generations.** **Go's current GC is not generational** (no generational GC), and it doesn't compact or move objects (non-moving, so object addresses stay stable — which is also why it's safe to take addresses freely in Go, and why interop with cgo is relatively straightforward). The community has historically debated adding generations, but since escape analysis already keeps a large share of short-lived objects on the stack (effectively a cheap "generation zero" for free), the payoff from generations is smaller than it would otherwise be; the implementation went with the current low-latency design instead. This may change as the runtime evolves — defer to the runtime documentation.

**The memory allocator: mcache / mcentral / mspan.** How are objects allocated on the heap? Go has a **tcmalloc-style multi-level allocator**, built around **bucketing by size class** to avoid global lock contention:

```
each P (see Chapter 10 on scheduling) has its own mcache (lock-free, thread-local)
        │  caches free objects by size class (about 70 classes: 8B,16B,...,32KB)
        ▼ when the local cache runs short, ask the level above
mcentral (one per size class, locked)
        ▼ when it runs short, ask the level above
mheap (the global heap, managing pages in units of mspan)
        ▼
OS (mmap large chunks of virtual memory)
```

- **mspan**: a set of contiguous pages, sliced into equal-sized blocks of one size class. Small objects (≤32KB) are taken from the span of the matching size class.
- **mcache**: private to each P; allocating a small object **needs no lock** (contrast C's `malloc`, which often has global lock contention) — this is one reason Go's allocation stays fast under high concurrency, and also why the concept of P shows up in both scheduling and memory allocation: as "a context private to each logical processor," it's a natural home for a lock-free local cache.
- Large objects (>32KB) are allocated directly from mheap.

Echoing [Chapter 03](03-变量与内存模型.md): **the more objects escape to the heap, the greater the allocation and GC pressure**. To optimize, cut down on heap allocation — preallocate slice capacity, reuse objects (`sync.Pool`), avoid unnecessary interface boxing/closure captures, use `go build -gcflags=-m` to inspect escape analysis, and use [pprof](12-常用第三方库.md) to find allocation hotspots.

## Summary

- **Usage level**: functions use `(result, error)` multiple return values, and the caller checks with `if err != nil`; `errors.New`/`fmt.Errorf` create errors, `%w` wraps them into an error chain, and `errors.Is`/`errors.As` match along that chain; `defer` registers cleanup that runs before the function returns (last-in-first-out); `panic`/`recover` are only for genuinely exceptional situations, and `recover` must be called inside a `defer`.
- **Design trade-offs**: `error` is an ordinary interface value, `{itab, data}`, two words — returning one is as cheap as returning anything else, which is a direct expression of the principle "errors are explicit data flow, not implicit control flow"; the price is the verbose `if err != nil`, and what you get for it is that the caller can never overlook the failure path, unlike Java's checked exceptions, which try to solve the same problem but remain stuck with the implicit-jump machinery of exceptions; panic/recover exist specifically for the category "the program's own logic has already gone wrong," which is a different nature of error from "an expected failure," and they backstop things only at boundaries rather than replacing everyday error handling; `defer` is bound to a function-level rather than a block-level lifetime, because in Go the endpoint of an object's lifetime is decided by the GC and the compiler can't pin down a block-scope exit point statically the way C++ RAII can; the GC is concurrent tri-color mark-sweep plus a hybrid write barrier, willing to spend extra CPU on concurrent marking to push STW down to the sub-millisecond range for this latency-sensitive setting; the allocator is the multi-level mcache/mcentral/mspan structure, with lock-free local allocation per P, so cutting heap allocation is what lowers GC pressure.
