# Basic Syntax (contrasted with C/C++)

The goal of this chapter is to let you—a C/C++ veteran—build up the syntax-level differences as fast as possible. We won't explain "what a variable is" or "what a function is"; we'll only cover **why Go and C do the same thing differently, and the trade-offs behind it**. Wherever a one-line analogy to C makes it clear, we won't belabor it.

## 1. Program Structure: package and import

C has no language-level concept of a "module"; it relies on `#include` for textual concatenation and the linker for symbol resolution. Go uses the **package** as the basic unit of compilation and encapsulation:

```go
package main // declares this file belongs to package main; the entry package of an executable must be called main

import (
	"fmt"      // standard library
	"os"

	"example.com/myapp/util" // a package within your own module, path comes from go.mod
)

func main() {
	fmt.Println("hello")
	os.Exit(0)
}
```

Key points:

- **One directory = one package.** All `.go` files in the same directory must declare the same package name. Identifiers within a package are visible to one another, with no need for forward declarations or header files—**Go has no header files and no forward declarations**. The compiler sees all source files of the whole package at once, so you can define something later in a file and call it earlier; order doesn't matter.
- `import` imports a **package** (the compiled artifact + exported information), not textual inclusion. The import path is the module path + directory, not a file path.
- **An unused import is a compile error**, not a warning. The same goes for an unused local variable. Go uses compile errors to force your code to stay clean (this annoys C programmers at first, but it eliminates "zombie imports").

## 2. func and main

```go
func add(a int, b int) int { // parameter type comes after the name; the return type comes after the parameter list
	return a + b
}

// Adjacent parameters of the same type can be merged:
func add2(a, b int) int { return a + b }

func main() { /* no parameters, no return value; the process entry point */ }
```

Compared with C's `int add(int a, int b)`, the most jarring thing is that **the type comes after the name**. This isn't just being contrarian; the reasons are covered specifically in Section 5.

The `main` function takes no `argc/argv` and returns no exit code. Command-line arguments come from `os.Args`; the exit code is set with `os.Exit(n)` or by returning normally from `main` (= 0).

## 3. Variable Declaration: var and :=, and the Zero-Value Mechanism

Two ways to declare:

```go
var x int          // explicit declaration, no initial value → automatic zero value 0
var y = 42         // initial value present, type inferred from the right side (here int)
var z int = 42     // fully spelled out, usually redundant

s := "hello"       // short variable declaration: usable only inside functions, = shorthand for var s = "hello"
```

### The Zero-Value Mechanism—One Way It's Safer Than C

In C, a local variable that isn't initialized holds an **undefined garbage value** (leftover on the stack), and reading it is a source of bugs. Go mandates: **any variable, if declared without an explicit initial value, is initialized to that type's "zero value"**, and the zero value is deterministic:

| Type | Zero value |
|------|------|
| Numeric types (int/float/…) | `0` |
| `bool` | `false` |
| `string` | `""` (empty string, not nil) |
| pointer, slice, map, chan, func, interface | `nil` |
| struct | each field recursively takes its zero value |
| array | each element takes its zero value |

This gives rise to a Go design philosophy: **"the zero value is useful."** Many types are deliberately designed so that the zero value is an immediately usable, valid state—for example, `var buf bytes.Buffer` can be used directly with `buf.WriteString(...)` without initialization, and `var mu sync.Mutex` has an unlocked state as its zero value. This contrasts with C++, which needs a constructor to bring an object into a valid state. Details are in [Chapter 3](03-变量与内存模型.md).

Let's just print out the zero values of various types (in C, these uninitialized variables would all be stack garbage):

```go
package main

import "fmt"

type User struct {
	Name string
	Age  int
	Tags []string
}

func main() {
	var i int
	var f float64
	var b bool
	var s string
	var p *int
	var u User // struct: each field recursively takes its zero value

	fmt.Printf("%d %g %t %q %v\n", i, f, b, s, p) // 0 0 false "" <nil>
	fmt.Printf("%+v\n", u)                        // {Name: Age:0 Tags:[]}
	fmt.Println(u.Tags == nil)                    // true
}
```

Note that the zero value of a `string` is the empty string `""`, not `nil`, whereas the zero value of a slice is `nil`.

`:=` can only be used inside functions; package-level variables must use `var`. On the left side of `:=`, at least one variable must be new.

## 4. No Implicit Type Conversion—It Must Be Explicit

This is one of Go's "strictest" rules and one C programmers find most uncomfortable at first. **Go has no implicit numeric conversion whatsoever**:

```go
var i int = 3
var f float64 = 1.5

// var r float64 = i + f  // compile error! int and float64 cannot be added directly
var r float64 = float64(i) + f // must convert explicitly: float64(i)

var a int32 = 1
var b int64 = 2
// a + b            // compile error! int32 and int64 are different types
a + int32(b)        // OK

var p int = 10
// var q int8 = p   // compile error! not allowed even if the value fits
var q int8 = int8(p) // OK, but you're responsible for overflow yourself
```

In C, `int + double`, `char + int`, and mixing signed/unsigned all silently trigger integer promotion and conversion, a breeding ground for countless hidden bugs. Go simply forbids it, turning every conversion into code your eyes can see. **The only exception is "untyped constants"**—a literal constant is "untyped" before it's assigned to a concrete type, and adapts as needed:

```go
var f2 float64 = 3    // OK, the constant 3 is untyped and adapts to float64
const big = 1 << 40   // untyped constant, assignable to int64, etc.
```

The rules for constants are detailed in [Chapter 4](04-数据类型.md).

## 5. Why the Type Is Written After the Name

C's declaration syntax follows "declaration reflects use," resulting in the famous spiral reading hell:

```c
int (*fp)(int, int);        // fp is a "pointer to a function taking two ints and returning int"
char *(*arr[10])(void);     // dizzying to read
```

Go goes the opposite way, adopting a **left-to-right, name-first, type-last** syntax:

```go
var fp func(int, int) int          // fp's type is func(int, int) int, read in one glance
var arr [10]func() *byte           // arr is an array of 10 func() *byte
var p *int                         // p is a *int
```

Two benefits: (1) **linearized readability**—it's always "name + type," and reading left to right is like natural language; (2) **an unambiguous grammar amenable to single-pass fast parsing** (echoing the fast compilation from [Chapter 1](01-运行模型.md)). In C, whether `a * b;` is a multiplication expression or the declaration of a pointer is something the compiler can only tell by consulting the symbol table; in Go a declaration is always led by `var`/`func`/`type` and can be parsed without semantics.

## 6. Multiple Return Values—No More Out-Parameter Pointers

In C a function can return only one value; to return several you must use **out-parameter pointers** or return a struct:

```c
int divmod(int a, int b, int *rem) { *rem = a % b; return a / b; }
```

Go natively supports multiple return values, one of its most practical features and the foundation of the error-handling convention:

```go
func divmod(a, b int) (int, int) {
	return a / b, a % b
}

q, r := divmod(17, 5) // q=3, r=2

// The idiomatic (result, error) pattern:
n, err := fmt.Println("hi")
if err != nil { /* handle the error */ }

// Use _ to discard a return value you don't need:
q2, _ := divmod(10, 3)
```

Under the hood, multiple return values are simply multiple values returned via the stack/registers (Go 1.17+ uses a register-based calling convention), and you no longer have to manually pass out-parameter pointers. The error-handling convention is in [Chapter 9](09-错误处理与资源管理.md).

## 7. Constants and iota

`const` declares compile-time constants. `iota` is a counter that **increments line by line** within a const block (starting from 0), used to concisely generate enumerations:

```go
const (
	Sunday = iota // 0
	Monday        // 1 (the expression carries over from the previous line, iota increments)
	Tuesday       // 2
	Wednesday     // 3
)

// Bit flags (like C's 1<<n enums):
const (
	Read    = 1 << iota // 1 << 0 = 1
	Write               // 1 << 1 = 2
	Execute             // 1 << 2 = 4
)

// Use _ to skip certain values:
const (
	_  = iota             // discard 0
	KB = 1 << (10 * iota) // 1<<10
	MB                    // 1<<20
	GB                    // 1<<30
)
```

`iota` counts independently within each `const (...)` block, incrementing by 1 for each new line (ConstSpec). It's more flexible than C's `enum` (it can carry expressions), but it too is limited to compile-time constants and cannot be used for runtime values.

Let's actually run both idioms above to see how the `iota` expression unfolds line by line:

```go
package main

import "fmt"

const (
	_  = iota             // 0, discarded
	KB = 1 << (10 * iota) // 1 << 10
	MB                    // 1 << 20
	GB                    // 1 << 30
)

const (
	Read    = 1 << iota // 1 << 0
	Write               // 1 << 1
	Execute             // 1 << 2
)

func main() {
	fmt.Println(KB, MB, GB)                        // 1024 1048576 1073741824
	fmt.Println(Read, Write, Execute)              // 1 2 4
	fmt.Printf("%b %b %b\n", Read, Write, Execute) // 1 10 100
}
```

The key point: the expression `1 << (10 * iota)` written on the second line is **carried over** by the subsequent lines, and `iota` increments each line, so 1KB/1MB/1GB are generated automatically; the bit-flag group generates the non-overlapping 1/2/4, which can be combined with bitwise OR.

## 8. Visibility: By First-Letter Case, Not Keywords

C++ uses `public/private`; Go uses one minimalist rule: **an identifier with an uppercase first letter = exported (visible outside the package, equivalent to public); lowercase = unexported (visible only within the package, equivalent to private)**. This rule applies uniformly to package-level functions, types, and variables, as well as struct fields and methods:

```go
package util

func Public()  {} // usable outside the package: util.Public()
func private() {} // usable only within this package

type User struct {
	Name string // exported field, externally readable/writable
	age  int    // unexported field, externally invisible
}
```

There are no `public/private/protected` keywords; visibility is apparent at a glance from the name. **The encapsulation boundary is the "package," not the "class"**—code within the same package can access each other's unexported members, unlike C++ where the class is the boundary.

## 9. No Semicolons: Automatic Semicolon Insertion

In Go's grammar, statements actually end with semicolons, but **you almost never write them**—the lexer inserts them automatically by rule. Simplified version of the rule: **if the last token of a line is an identifier, a literal, `)`/`]`/`}`, or `return`/`break` etc., a semicolon is automatically appended at the end of the line**.

This brings a hard consequence you must remember: **`{` cannot start on a new line**. In C/Java you can:

```c
if (x)
{           // legal in C
}
```

In Go, writing it this way inserts a semicolon after `if x`, turning it into a syntax error. So the opening brace must follow on the same line:

```go
if x {   // the only correct form
}

func f() {   // likewise
}
```

This isn't a style suggestion, it's enforced by the language. And precisely because semicolons are automatic, `gofmt` can confidently unify everyone's layout.

## 10. gofmt: Formatting Is No Longer a Topic of Debate

In the C/C++ world, brace style, tabs vs. spaces for indentation, alignment—every team argues endlessly. Go officially ships a `gofmt` (and `go fmt`) that **mandates the one correct format**: tab indentation, fixed spacing rules, grouped and sorted imports, and so on. The community convention: always `gofmt` before committing. IDEs typically format on save.

The result: **all Go code looks the same**, and you never have to adjust to a new style when reading any open-source library. This isn't a syntax feature, but it's part of Go culture and worth accepting from day one.

## 11. Quick Reference: C vs. Go Differences

| Aspect | C/C++ | Go |
|------|-------|-----|
| Declaration syntax | Type first `int x;` | Type last `var x int` / `x := 0` |
| Uninitialized local variable | Garbage value | Deterministic zero value |
| Implicit numeric conversion | Yes (integer promotion, etc.) | **None, must be explicit** |
| Header files / forward declarations | Needed | Not needed |
| Multiple return values | None (via out-params/struct) | Natively supported |
| Encapsulation boundary | Class (`public`/`private`) | Package (first-letter case) |
| Unused variable/import | Warning or ignored | **Compile error** |
| Semicolons | Hand-written | Auto-inserted (`{` can't wrap) |
| Formatting | Varies by team | Uniformly enforced by `gofmt` |
| Enumerations | `enum` | `const` + `iota` |
| Ternary operator `?:` | Yes | **None** (use if) |
| Pointer arithmetic | Yes | **None** |

With this table you can already read most Go code. The next chapter gets into the most important content of the whole book: what exactly happens in memory in `a = b`—the cornerstone of understanding all data behavior in Go.
