# Runtime Model: Compilation, Linking, and the Runtime

Getting a Go program from source to running takes two steps: compilation and runtime startup—one fewer stage than C (no separate dynamic-linking step, since it doesn't depend on external .so files by default), but with a built-in runtime you now have to understand. This chapter takes a Go binary apart layer by layer, contrasting it with C's compile-link-run model at every step, and answers two questions: exactly what extra gets packed into this binary, and why Go was designed this way.

## 1. Writing a Program: `go run` and `go build`

```go
package main

import "fmt"

func main() {
	fmt.Println("hello, world")
}
```

`package main` declares this is an executable program (as opposed to a library meant to be imported by others); `import "fmt"` pulls in the standard-library package for formatted output; `func main()` is the program's entry point, equivalent to C's `int main(void)`, except Go's `main` takes no arguments and returns nothing—command-line arguments and exit codes are handled separately by `os.Args` and `os.Exit`, covered in [Chapter 2](02-基础语法.md).

`go run` compiles and runs immediately, like running a script:

```bash
$ go run hello.go
hello, world
```

`go build` produces a real executable, the counterpart of C's `gcc -o hello hello.c`:

```bash
$ go build -o hello hello.go
$ ./hello
hello, world
```

Note that `go run` is not interpreted execution—it first compiles the source to native machine code, then puts the binary in a temp directory, runs it, and deletes it when done. It gives you a "script-like experience," not "a script's execution model."

Concurrency has exactly one syntactic entry point: put `go` in front of a function call, and that call is handed to the runtime to be scheduled asynchronously:

```go
package main

import (
	"fmt"
	"time"
)

func sayHi() {
	fmt.Println("hi, from another goroutine")
}

func main() {
	go sayHi()                        // launch a goroutine, don't wait for it to finish before moving on
	time.Sleep(time.Millisecond * 10) // wait a bit so it gets a chance to run
	fmt.Println("main done")
}
```

Output:

```
hi, from another goroutine
main done
```

**Why concurrency primitives are language keywords, instead of being left to the standard library the way C leaves them to pthreads**: C's concurrency runs on library functions—thread creation and mutexes are just ordinary calls, the compiler has no idea two pieces of code might run concurrently, and the language itself knows nothing about "concurrency," so it can't check or guarantee anything at the language level. Go's designers (Pike, Thompson, and others, after years of building highly concurrent servers in C++ inside Google) judged that network services and multi-core CPUs were the default scenario, not the exception, at the time Go was born (around 2007), so they made "launching" (`go`) and "communicating" (`channel`/`<-`) language keywords and built-in types rather than a swappable library. The cost is that you can't swap in a different scheduling model—behind the `go` keyword is always Go's own GMP scheduler; there's no freedom to "pick a different concurrency library." What you get in exchange is deep compiler and runtime support for concurrency: the runtime knows precisely where the pointers are on every goroutine's stack (Chapter 6's growable stacks depend on this), and the race detector can uniformly understand memory accesses across all goroutines. If concurrency were just a library, this kind of language-level unified view wouldn't be possible.

When a project grows and needs to declare dependencies, use `go.mod` to declare a module (single-file `go run`/`go build` doesn't need this step):

```bash
$ go mod init example.com/hello
go: creating new go.mod: module example.com/hello
```

## 2. Binary Composition: Code + Runtime + Standard Library, Statically Packaged

As a C/C++ programmer, you already have a clear mental model of how a program becomes something runnable: preprocess → compile to `.o` → link (static or dynamic) → produce an ELF/Mach-O executable → loaded by the kernel and bootstrapped into `main` by libc's `_start`. Go's model has things in common with this and things that differ. What's the same: the final artifact is a native machine-code executable, with no virtual machine and no bytecode interpreter. What's different: this binary additionally statically links in a runtime, which is already running before your `main` even starts.

| Dimension | C/C++ (gcc/clang) | Go (gc toolchain) | JVM (Java) |
|------|-------------------|----------------|------------|
| Final artifact | Native machine code | Native machine code | Bytecode `.class`/`.jar` |
| Execution | Directly executed by the CPU | Directly executed by the CPU | JVM interpretation + JIT |
| Has a runtime? | Extremely thin (crt0 + libc, usually dynamically linked) | **Yes, fairly thick, and statically linked in by default** | Yes, a separately installed JVM |
| Default linking | Dynamically links libc | **Static** (no libc dependency on Linux with pure Go; macOS still links libSystem) | N/A |
| Deployment | Requires a matching .so on the target machine | **A single self-contained file—copy it over and it runs** | Requires a JRE on the target machine |
| GC | None (manual `malloc/free`) | Yes (concurrent tricolor mark-and-sweep) | Yes |
| Threading model | 1:1 mapping to OS threads | M:N, goroutines multiplexed onto OS threads | Platform threads / virtual threads |

The key mental model: **Go sits between C and the JVM**. Like C, it compiles to machine code executed directly by the CPU (no interpretation, no JIT warm-up), but like the JVM it ships with a runtime responsible for scheduling, GC, and memory allocation. The difference is that this runtime isn't an external process—it's **statically linked into your binary, existing as ordinary functions and code segments**.

```
   Address space of a C program      Address space of a Go program
 ┌────────────────────┐         ┌────────────────────┐
 │  your .text        │         │  your .text        │
 │  libc (often dyn.) │         │  runtime .text     │  ← scheduler/GC/allocator
 │  crt0 bootstrap    │         │  stdlib .text      │  (all compiled in statically)
 ├────────────────────┤         ├────────────────────┤
 │  heap (malloc/free)│         │  heap (runtime-mgd)│
 │  one/more thread    │         │  many goroutine    │  ← ~2KB each to start, growable
 │  stacks             │         │  stacks             │
 └────────────────────┘         └────────────────────┘
```

## 3. Static Linking: Trading Binary Size for Deployment Simplicity

The previous section already used `go build`. What's more worth digging into for a C programmer is that it **statically links by default** (more precisely, this mainly applies to the **Linux + `CGO_ENABLED=0`** case). Write a pure-Go `hello`, and on Linux `ldd hello` will tell you `not a dynamic executable`—it hasn't linked libc, it doesn't even need `libc.so.6`. (**macOS is the exception**: Apple doesn't support statically linking libSystem, so a Go binary on macOS always dynamically links `/usr/lib/libSystem.B.dylib`, visible with `otool -L`.)

**Why static linking by default, instead of dynamic linking the way C does**: the historical motivation for dynamic linking was saving resources—multiple processes share the same copy of libc in memory, and libc can get a security patch without recompiling every application. That benefit assumes the target machine happens to have a version-compatible .so, and that's exactly the most troublesome part of distributed deployment (dependency hell, `GLIBC_2.34 not found`). The scenario Go was born into was thousands of machines inside Google running the same batch of services, and later containerized deployment—"can this binary run directly on any machine of the same architecture" mattered far more than "saving a few dozen KB of disk." So Go tips the trade-off the other way: accept a larger binary (compile in whatever runtime and standard-library packages get used—a `hello world` is typically 1–2 MB, whereas a dynamically-linked C version might be only a few dozen KB but depends on the several MB of libc already on the system), in exchange for the certainty that "compile once, copy to any machine of the same architecture, and it runs"—in the extreme case, an empty `FROM scratch` Docker image with just this one file in it will run. This isn't a universally superior approach, it's a trade-off suited to a particular deployment scenario: where you need to patch libc without recompiling the application (say, some long-running system that can't easily be redeployed), static linking is actually a disadvantage.

### When Does Go Link Dynamically?

It's not absolutely static. Two common situations introduce dynamic linking:

1. **cgo**: once you use `import "C"` to call a C library (such as certain resolution paths in `net` or `os/user`, or a third-party C dependency), libc gets linked in and a dynamic dependency appears. You can force cgo off with `CGO_ENABLED=0` to go back to fully static.
2. Explicit `-linkmode=external` or linking against system libraries.

```bash
CGO_ENABLED=0 go build -o hello   # force fully static, commonly used when building Docker images
```

## 4. What's Actually Inside the Runtime

C's "runtime" is so thin it's almost nothing but crt0's bootstrap plus the library functions libc provides; control is handed to your `main` right away, and after that there's no "background magic" unless you call into a library. Go is different: before your `main.main` is even called, the runtime has already initialized a whole set of subsystems, and it **keeps working in the background** for as long as your program runs. Three main pieces:

### 1. The Goroutine Scheduler (GMP Model)

Go doesn't map goroutines directly onto OS threads (that's C's 1:1 pthread model); instead it uses **M:N scheduling**: thousands upon thousands of goroutines are multiplexed onto a small number of OS threads. Three core abstractions:

```
G (goroutine)  ── one goroutine, with its own stack, PC, and state. Extremely lightweight.
M (machine)    ── one OS thread, actually running code on the CPU.
P (processor)  ── a logical processor/scheduling context, count defaults to GOMAXPROCS (usually = #CPU cores).
                  P holds a local runnable-G queue; an M must bind to a P to execute a G.

     M0 ── P0 ── [G,G,G] local queue
     M1 ── P1 ── [G,G]
                 [G,G,G,...] global queue (a P steals from here when its own queue is empty)
```

When a goroutine blocks in a system call, the runtime detaches the M from its P, letting another M take over that P and keep running other Gs—so a small number of threads can sustain massive concurrency. See [Chapter 10, "The Concurrency Model"](10-并发模型.md) for details. By contrast, spinning up a thread in C costs several MB of stack plus kernel scheduling overhead, so a few thousand is already a strain; spinning up hundreds of thousands of goroutines in Go is entirely routine.

The `go sayHi()` above was just syntax; you can see the scheduler's existence more directly with the `runtime` package. Pin the number of Ps to 2, then spawn 1000 goroutines—all of them multiplexed onto those 2 logical processors:

```go
package main

import (
	"fmt"
	"runtime"
)

func main() {
	runtime.GOMAXPROCS(2)              // pin logical processors P to 2 (deterministic result)
	fmt.Println(runtime.GOMAXPROCS(0)) // 2   (passing 0 = query only, don't change)

	fmt.Println(runtime.NumGoroutine()) // 1   (only the main goroutine so far)

	ch := make(chan struct{})
	for i := 0; i < 1000; i++ {
		go func() { <-ch }() // 1000 goroutines, all parked waiting on ch
	}
	for runtime.NumGoroutine() < 1001 { // wait until all of them have been created
		runtime.Gosched()
	}
	fmt.Println(runtime.NumGoroutine()) // 1001 (main + 1000), running on just 2 OS threads

	close(ch) // release them all at once
}
```

Output:

```
2
1
1001
```

1001 goroutines coexist peacefully on 2 Ps—with C's 1:1 threads, spinning up 1000 threads would need several GB of stack alone. This is the M:N scheduling model made concrete.

### 2. The Garbage Collector (GC)

In C you `malloc`/`free` yourself, and dangling pointers and memory leaks are entirely up to your own discipline. Go has a **concurrent, tricolor mark-and-sweep** GC running in the background, executing concurrently with your code, with STW (stop-the-world) pauses typically in the sub-millisecond range. This means you almost never have to manually free memory—but you do need to understand **escape analysis** (why a variable ends up on the heap), or you'll end up writing GC-unfriendly code. Escape analysis is covered in [Chapter 3](03-变量与内存模型.md); GC details are in [Chapter 9](09-错误处理与资源管理.md).

### 3. The Memory Allocator (tcmalloc-style)

Go's allocator descends from Google's tcmalloc, using a **multi-level cache** structure, with the core goal of allocating small objects as lock-free as possible:

```
mcache   ── one per P, thread-local, allocates small objects without locking (like tcmalloc's thread cache)
   │ requests more from below when it runs short
mcentral ── a central cache organized by size class, shared across Ps, requires locking
   │ requests more from below when it runs short
mheap    ── the global heap, managing large chunks of memory obtained from the OS (span/arena)
```

Objects are grouped by size into roughly 70 **size classes**; small objects are taken directly from the P-local `mcache` with no global lock contention—this is one reason Go's allocation performance is good under high concurrency. Compared with C's `malloc` (typically a global lock or arena lock), Go's per-P cache is naturally multicore-friendly.

## 5. Why Go Compiles So Fast

A large C++ project can compile slowly enough that you go get a coffee, and you know the reasons well: header files textually included over and over, templates instantiated in every compilation unit, huge amounts of preprocessor expansion, and heavy symbol resolution at link time. Go was designed to deliberately sidestep all of these:

1. **No header files, no preprocessor**. Dependency information is encoded directly in the package; compiling a package only requires reading the **exported information of the packages it directly imports** (the compiled object carries compact export data), and does **not** require recursively expanding the source of the entire dependency tree. In C, `#include <a.h>` where a.h includes a bunch more, and every .c re-parses them all over again.
2. **Imports are package-level and acyclic**. Go mandates that the package dependency graph have no cycles, and imports the compiled artifact rather than source, so the compiler need only read one summary.
3. **The grammar is designed for fast parsing**. Go's grammar can essentially be parsed in a single pass without consulting semantics (see automatic semicolon insertion in [Chapter 2](02-基础语法.md)); there's none of C++'s ambiguity where "you need the symbol table to tell whether `a * b` is a multiplication or a pointer declaration."
4. **Parallel compilation**: different packages can be compiled in parallel.

The cost: fast compilation comes at the price of "a thicker runtime + a larger binary + GC overhead," plus trade-offs in language expressiveness (no generics early on, etc.—covered in Section 8).

## 6. Common Commands: `go run` / `build` / `mod`

The earlier sections already used `go run`, `go build`, and `go mod init`. The full set of common commands:

```bash
go run main.go        # compile to a temp directory and run immediately (leaves no binary), for quick experiments
go build ./...        # compile all packages in the current module, produce an executable (when there's a main package)
go install ./cmd/foo  # compile and install the binary into $GOBIN (default ~/go/bin)
go test ./...         # compile and run tests
go vet ./...          # static analysis for suspicious code
```

### go modules: Dependency Management

Go uses **modules** to manage dependencies (replacing the old GOPATH mode):

```bash
go mod init example.com/myapp   # generate go.mod, declaring the module path
go get github.com/foo/bar@v1.2.3 # add/upgrade a dependency, writing it to go.mod and go.sum
go mod tidy                      # add/remove dependencies to match the source imports
```

- `go.mod`: declares the module path, Go version, and dependencies with their versions.
- `go.sum`: records hash checksums of dependencies, guaranteeing reproducible builds (like a lock file).

Dependencies get downloaded to a local cache (`$GOPATH/pkg/mod`) and are statically compiled into your binary at build time. **There's no C-style "go find the .so at runtime" problem**—dependencies are frozen in at compile time.

## 7. Cross-Compilation: Just Two Environment Variables

Cross-compilation in C is notoriously painful: you need a cross-toolchain for the target platform (a cross gcc), the target platform's libc, a sysroot, and so on. Because Go by default doesn't depend on libc, and the toolchain ships with the runtime for every target platform, cross-compilation is just setting two environment variables:

```bash
# On macOS/arm64, build for Linux/amd64:
GOOS=linux GOARCH=amd64 go build -o hello-linux-amd64

# Build for Windows:
GOOS=windows GOARCH=amd64 go build -o hello.exe

# Build for a Raspberry Pi (Linux/arm64):
GOOS=linux GOARCH=arm64 go build -o hello-pi

go tool dist list   # list all supported GOOS/GOARCH combinations
```

- `GOOS`: target operating system (`linux`/`darwin`/`windows`/`freebsd`...).
- `GOARCH`: target CPU architecture (`amd64`/`arm64`/`386`/`riscv64`...).

**The prerequisite is pure Go (`CGO_ENABLED=0`)**. Once you use cgo, cross-compilation degenerates back into the painful C-world mode of "needing a C cross-toolchain for the target platform."

## 8. Minimalist Philosophy: Why Exceptions, Early Generics, and Class Inheritance Were Cut

Go deliberately cut three things out of the common C++/Java feature set: exceptions, generics (in the early versions), and class inheritance. This isn't a lack of capability—it's an intentional design choice, rooted in real engineering problems that designers like Pike, Griesemer, and Thompson observed inside Google: a team of thousands of engineers, with code read, modified, and maintained for decades by people who've never met each other. At that scale, the accumulated combination of C++ features often becomes a burden rather than a help.

- **No exceptions**: C++/Java exceptions let control flow implicitly jump out of many stack frames at any level—reading a piece of code, you can't tell from the function signature alone whether it might throw, where it might throw from, or who's responsible for catching it. Go turns errors into ordinary return values (`(result, error)`, covered in detail in [Chapter 9](09-错误处理与资源管理.md)); the error-handling path and the normal path are written in the same visible place, and "this function might fail" is part of the signature rather than a hidden contract buried in documentation. The cost is verbosity—you write `if err != nil` after every call, without the convenience exceptions give you of throwing all the way up and catching once at the top. What you get in exchange is **no implicit non-local control-flow jumps**, which is also the fundamental reason Go has no `try/catch/finally` and instead expresses cleanup logic with `defer` (see [Chapter 6](06-函数与调用栈.md)).

- **No generics early on (Go 1.0–1.17)**: C++ templates provide enormous expressiveness (compile-time specialization, SFINAE), at the cost of famously unreadable compile errors (template instantiation stacks running hundreds of lines), slower compilation, and code that requires specialized training to read once templates are used heavily. Go initially judged that interfaces (duck typing + runtime polymorphism) already covered most scenarios that need "polymorphism," and was willing to sacrifice compile-time monomorphization performance and container type safety in exchange for a simpler language and faster compilation. That judgment later turned out not to be thorough enough—the lack of generics led to large amounts of boilerplate (hand-writing one version per container type, or falling back to `interface{}` and losing type safety), so Go 1.18 (2022) added type parameters, though the design remains restrained: no specialization, no template metaprogramming. This is itself an example of the "expressiveness vs. simplicity" trade-off line being recalibrated, not overturned.

- **No class inheritance**: C++ class inheritance (especially multiple inheritance) brings the diamond inheritance problem and the fragile base class problem (a seemingly harmless change in one spot of a base class can quietly break every subclass's assumptions). Go has no classes and no inheritance—only structs (data) + interfaces (contracts) + composition (embedding, see [Chapter 4](04-数据类型.md)). This is the software-engineering consensus of "composition over inheritance" written directly into the language—you cannot inherit an implementation, you can only explicitly embed one type inside another; dependencies are always visible fields, never a parent-child chain hidden inside a vtable.

Behind all three cuts is the same principle: **reading code takes far more time than writing it, so Go deliberately gives up some expressiveness in exchange for the long-term maintainability of anyone being able to understand unfamiliar code in a reasonable amount of time**. The complexity that comes from language features combining with each other grows super-linearly—every feature you add means one more layer of interaction to think through with every existing feature; cutting unnecessary features is what keeps the mental burden of understanding unfamiliar code manageable.

## 9. Final Contrast with C: A Single Self-Contained Binary

**What Go aims for is "compile once, get a self-contained native binary that can be copied to another machine and just run."**

- Versus C/C++: you're spared shared-library version hell (things like `GLIBC_2.34 not found`) and the hassle of scraping together the right .so files at deploy time; the cost is a larger binary, plus a runtime (GC/scheduler) you can't fully turn off.
- Versus the JVM: you don't need a JRE installed on the target machine, and startup has no JIT warm-up (fast cold start), but you give up the JVM's deep runtime introspection and peak JIT performance.

Once you understand that "there's a runtime hidden inside the binary that keeps running the whole time," a lot of behavior in later chapters has a foundation: why goroutines are cheap (scheduler + growable small stacks), why some variables end up on the heap (escape analysis + GC), why you never have to free a slice or a map.

## Summary

- **At the usage level**: `go run file.go` compiles and runs immediately; `go build -o out file.go` produces an executable; putting the `go` keyword before a function call launches a goroutine that runs in the background; for multi-file projects with dependencies, declare a module with `go mod init`; `GOOS=xxx GOARCH=xxx go build` cross-compiles.
- **At the underlying level**: a Go binary = your code + runtime (scheduler + GC + allocator) + the standard-library packages you use, statically packaged into a single native machine-code file; it doesn't depend on libc by default (except with cgo), which is a trade of "binary size" for "deployment simplicity"; the GMP model lets massive numbers of goroutines multiplex onto a small number of OS threads, which is also the direct payoff of making concurrency a language keyword instead of a library function; a concurrent tricolor mark-and-sweep GC runs in the background; the allocator uses a mcache/mcentral/mheap multi-level cache; no header files or template instantiation makes compilation far faster than C++; and cutting exceptions, holding off on generics early, and cutting class inheritance are all expressions of that same philosophy of trading minimalism for long-term maintainability.
