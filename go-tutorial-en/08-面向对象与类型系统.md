# The Type System: Structs, Interfaces, Method Sets

Go has no `class`, no inheritance, no constructors/destructors, no `virtual` keyword, and no access modifiers (`public`/`private` are determined by the case of an identifier's first letter). It composes a simpler, more "structural" type system out of **struct + methods + composition + interfaces**. Every layer has a clear, predictable underlying representation — no hidden vtable pointer, no hidden `this`. For C++ programmers, the two things most in need of recalibration are: **inheritance is replaced by composition**, and **interfaces are implemented implicitly**.

## I. Methods: functions with a receiver

A method is just an ordinary function whose first parameter (the receiver) is written up front. There are two kinds: a value receiver (operates on a copy) and a pointer receiver (can modify in place).

```go
type Point struct{ X, Y int }

func (p Point) Add(q Point) Point {   // value receiver: p is a copy of the Point
	return Point{p.X + q.X, p.Y + q.Y}
}
func (p *Point) Scale(f int) {        // pointer receiver: can modify in place
	p.X *= f
	p.Y *= f
}

p1 := Point{1, 2}
p2 := Point{3, 4}
sum := p1.Add(p2)  // {4 6}
p1.Scale(10)       // p1 becomes {10 20}, Scale modified it in place
```

A method can be defined on **any named type defined within the current package**, not just structs:

```go
type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }
```

There is no magic underneath: `p.Add(q)` is equivalent to `Point.Add(p, q)`, and the receiver is just an ordinary passed argument—a value receiver copies the whole `Point`, a pointer receiver copies one pointer. There is no C++-style hidden `this`-pointer semantic difference; everything is explicit. ([Recall the value receiver vs. pointer receiver discussion in Chapter 06](06-函数与调用栈.md))

## II. Composition: embedding and method promotion

Go doesn't reuse via inheritance; it uses **embedding**: put an anonymous field (just the type, no field name) inside a struct. The embedded type's methods and fields get "promoted" to the outer type, so they look like the outer type's own:

```go
type Engine struct{ Power int }
func (e Engine) Start() { fmt.Println("engine start", e.Power) }

type Car struct {
	Engine       // embedding (anonymous field), not inheritance!
	Brand string
}

c := Car{Engine{100}, "Toyota"}
c.Start()             // method promotion: equivalent to c.Engine.Start()
fmt.Println(c.Power)  // field promotion: equivalent to c.Engine.Power
```

**Why Go has no classes or inheritance, only struct composition + interfaces.** Classic C++ inheritance hierarchies encourage hoisting common logic further and further up into base classes, and as a system evolves it easily grows a deep inheritance tree; deep inheritance trees are notoriously fragile — a subclass's correctness implicitly depends on the base class's current implementation details (not just its interface), and a base class, in order to "leave room for subclasses to extend," often makes internal state and methods `protected`, so subclass and base class end up tightly coupled at the implementation level. This is the "fragile base class problem": changing a seemingly unrelated implementation detail in the base class can silently break the behavior of every subclass, and this kind of breakage usually can't be caught at compile time. The software-engineering community's advice to "favor composition over inheritance" has been around for a long time, but in languages that allow inheritance it remains just a customary best practice, followed or not depending on team discipline. Go's approach is more radical: it simply **doesn't provide implementation inheritance** at all — composition (embedding) is the only mechanism for code reuse. This isn't a missing feature; it's a deliberate elevation of a "best practice" into "the only way the language supports," which stamps out the entire class of deep-inheritance-tree problems at the language-design level, at the cost of giving up the expressiveness inheritance hierarchies offer—define once, reuse everywhere, and let subclasses override implementation details at any layer.

Memory layout: embedding simply inlines the entire `Engine` as one field of `Car`, no different from an ordinary field:

```
Car in memory:
┌──────────────┬───────────────┐
│ Engine.Power │ Brand(string) │   Engine's fields expand in place within Car
└──────────────┴───────────────┘
```

This is not C++ inheritance (no base-class subobject vptr, no is-a relationship). The compiler merely does "name forwarding": `c.Start()` is rewritten at compile time to `c.Engine.Start()`. **No runtime overhead, no vtable**.

| Dimension | C++ inheritance | Go embedding |
|------|---------|---------|
| Relationship | is-a (Car is a kind of Engine) | has-a + forwarding (Car has an Engine) |
| Polymorphism entry | `virtual` + vtable | Via interfaces (see below) |
| Override | Virtual-function override | An outer method with the same name "shadows" the inner one |
| Accessing the base | `Base::foo()` | `c.Engine.Start()` explicit path |
| Diamond problem of multiple inheritance | Requires virtual inheritance | Embed multiple types; same-name conflicts require explicit qualification, the compiler reports ambiguity |

Method shadowing example:

```go
func (c Car) Start() { fmt.Println("car:", c.Brand); c.Engine.Start() }
// now c.Start() calls the outer one; to call the inner one write c.Engine.Start() explicitly
```

Note: embedding is **static forwarding**, not virtual dispatch. If `Engine.Start` internally calls another method, it still calls `Engine`'s own, and **will not** call back into `Car`'s overriding version—fundamentally different from C++ virtual functions where "a derived-class override gets called by the base class." For that kind of callback you need interfaces.

Runnable: promotion, shadowing, and static forwarding:

```go
package main

import "fmt"

type Engine struct{ Power int }

func (e Engine) Start() { fmt.Println("engine start", e.Power) }
func (e Engine) Boot()  { e.Start() } // always calls Engine.Start

type Car struct {
	Engine        // embedded (anonymous field)
	Brand string
}

func (c Car) Start() { fmt.Println("car start", c.Brand) } // shadows Engine.Start

func main() {
	c := Car{Engine{100}, "Toyota"}
	c.Start()            // car start Toyota   (outer shadows the promoted method)
	c.Engine.Start()     // engine start 100   (explicit path to the inner one)
	fmt.Println(c.Power) // 100                (field promotion)
	c.Boot()             // engine start 100   (static forwarding: NOT "car start")
}
```

The last line is the crux: `Boot` belongs to `Engine` and calls `Engine.Start` directly—it does **not** dispatch to `Car.Start`. That is static forwarding, the opposite of a C++ virtual call.

## III. Interfaces: implicitly implemented structural types

An interface defines a set of method signatures. **As long as a type implements all of the interface's methods, it automatically satisfies that interface—no `implements`, no explicit declaration needed.**

```go
type Stringer interface {
    String() string
}

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

var s Stringer = Celsius(37.5)  // automatically satisfied, no need to declare "Celsius implements Stringer"
```

This is called **structural typing**—a statically-checked version of duck typing. **Why interfaces are designed to be implemented implicitly, rather than requiring an explicit `implements` or inheriting from an abstract base class the way Java/C++ do.** The core reason is one of Go's design maxims: **interfaces should be defined by consumers, not producers**. In an explicit-declaration model, for a type to satisfy an interface it must know, **at the point where it's defined**, that this interface exists and declare conformance to it—which means the interface usually has to be predefined by the package that implements it (or some shared upstream of theirs). Adding a new interface viewpoint often requires going back and modifying the implementer's code, or is simply impossible for a type you don't control (say, a type from the standard library or a third-party library). Implicit implementation breaks this coupling: any consumer can define, in its own package, an interface containing only the handful of methods it actually needs, without needing the implementer's cooperation or having to modify the implementer's code—as long as the concrete type "happens" to have those methods, the match is automatic. The result is that the right to define an interface moves down to the consumer, and the direction of dependency between modules is simplified along with it—the implementer's package never needs to import the package that defines the interface, which reduces coupling between packages and the risk of import cycles. The price is that the compiler can't tell you, at the moment a type is defined, which interfaces it implements; the match is only checked by the compiler when a value actually needs to be assigned to an interface variable, and it's also harder for IDEs/documentation to list at a glance "which interfaces this type implements."

Other benefits include: an interface can be defined "after the fact" for a type you can't modify (even a standard-library type), decoupling the definer from the user. C++20's concept is somewhat similar, but interfaces are runtime polymorphism (with a vtable), closer to C++'s abstract base class + virtual functions.

The underlying representation of an interface value: two words. An interface value in memory is **two machine words** ([mentioned in Chapter 04](04-数据类型.md)). There are two kinds:

**`iface`—a non-empty interface (an interface with methods)**:

```go
type iface struct {
    tab  *itab          // type info + method table (equivalent to a vtable)
    data unsafe.Pointer  // points to the actual data
}
```

**`eface`—the empty interface `any` (i.e. `interface{}`)**:

```go
type eface struct {
    _type *_type         // dynamic type info
    data  unsafe.Pointer  // points to the actual data
}
```

`itab` is the key: it pairs an "interface type" with a "concrete type" and caches the method address table:

```go
type itab struct {
    inter *interfacetype // interface type
    _type *_type         // concrete type
    hash  uint32
    fun   [1]uintptr     // method address array (variable-length), equivalent to a vtable
}
```

```
var s Stringer = Celsius(37.5)

s(interface value, 16 bytes):          itab (one per (interface, type) combination, globally cached):
┌──────────┐               ┌────────────────────────┐
│ tab  ────┼──────────────▶│ inter: Stringer type    │
│ data ────┼───┐           │ _type: Celsius type     │
└──────────┘   │           │ fun[0]: &Celsius.String │◀─ method address
               │           └────────────────────────┘
               ▼
        the Celsius value 37.5 on the heap/stack (boxing: the value is copied to where data points)
```

**The key difference from a C++ vtable**: a C++ object stores a vptr inside itself, and the vtable is uniquely determined by the class; Go's method table is **inside the interface value** (`itab`), the same concrete type has different `itab`s for different interfaces, and `itab`s are generated on demand and globally cached. Calling `s.String()` = look up `tab.fun[0]` to get the address → call it with `data` as the receiver, one indirect jump, comparable in cost to a virtual call. This difference directly mirrors the "implicit implementation" design, too: a C++ vtable must be fixed at class-definition time (because the vptr is part of the object's own layout), whereas Go moves the method table for "type T satisfies interface I" into the interface value and generates it on demand—which naturally supports assembling, after the fact, an `itab` that makes an existing type satisfy some interface, without touching the type's own definition.

**Boxing**: when you stuff a concrete value into an interface, if the value is not a pointer, it usually must be copied to the heap so `data` can point at it (escape, see [Chapter 03](03-变量与内存模型.md)). So `var x any = 42` may trigger a small allocation (the runtime has caching optimizations for small integers, etc.). This resembles Java/C# boxing, with no counterpart in C++.

### The nil-interface trap (a must-know)

An interface value equals nil **if and only if both words—`tab` (or `_type`) and `data`—are nil**. If the interface holds a pointer whose type is non-nil but whose value is nil, the interface itself is **not equal to nil**:

```go
func doErr() error {
    var p *MyError = nil   // a nil pointer of a concrete type
    return p               // boxed into the error interface: tab=*MyError, data=nil
}

func main() {
    err := doErr()
    if err != nil {        // surprisingly true!!
        fmt.Println("non-nil:", err == nil) // non-nil: false
    }
}
```

**Why this trap exists, and why it's an inevitable consequence of the two-word representation rather than a bug that could simply be fixed.** Whether an interface value is nil is, by definition, decided by looking at only one word—`tab`/`_type`—and has nothing to do with whatever value happens to be sitting in the other word, `data`. This isn't an oversight—it's a logical consequence of the representation itself, "an interface value is two separate fields": the line `return p` boxes a **concrete type, `*MyError`** (even though its value is nil) into the `error` interface, and what the boxing operation does is "record that this is a `*MyError`, and the data pointer just happens to be nil," not "turn the interface itself into nil"—these are two entirely different things: the former is "the type is known, and the value happens to be empty," the latter is "there isn't even any type information." As long as an interface is represented with this explicit two-word structure of "type pointer + data pointer" (rather than a single, type-agnostic sentinel value meaning "empty"), this distinction necessarily exists, and so "a nil pointer of a concrete type ≠ a nil interface" necessarily arises. **Why choose this two-word representation over a single "universal nil" word**: the two-word structure keeps an interface's type information and data stored separately, so an `itab` can be reused across multiple boxings of the same type, a method call needs only one indirect jump, and the call path never has to repeatedly check at runtime "is this actually one of those special nils." If you switched to a single-word representation with some sentinel value uniformly meaning "nil of any type," both interface calls and type checks would get more complicated to implement, and there's no way around that while keeping the current performance characteristics—so this is a trade-off between "a simple, efficient implementation" and "one fewer beginner trap," and Go chose the former, turning this trap into a rule you have to remember rather than a runtime cost. **Rule: when a function returns `error`, either return a genuine nil (`return nil`), or don't return a nil pointer of a concrete type.** ([Chapter 09 on error revisits this point](09-错误处理与资源管理.md))

Runnable:

```go
package main

import "fmt"

type MyError struct{}

func (e *MyError) Error() string { return "boom" }

func doErr() error {
	var p *MyError = nil // a typed nil pointer
	return p             // boxed into error as {type:*MyError, data:nil}
}

func returnsNil() error { return nil } // a genuine nil interface

func main() {
	fmt.Println(doErr() == nil)      // false  ← the trap: interface holds a non-nil type
	fmt.Println(returnsNil() == nil) // true
}
```

### Type assertions and type switch

Retrieving the concrete type from an interface:

```go
var i any = "hello"
s := i.(string)         // assertion, panics on failure
s, ok := i.(string)     // comma-ok form, ok=false on failure, no panic
```

At the low level: compare the interface's `_type`/`itab` against the target type; if they match, return `data` (dereferencing/copying as necessary).

A type switch dispatches on the dynamic type:

```go
switch v := i.(type) {
case int:
    fmt.Println("int", v)     // v is an int
case string:
    fmt.Println("string", v)  // v is a string
case fmt.Stringer:
    fmt.Println(v.String())   // can also match an interface
default:
    fmt.Println("other")
}
```

## IV. Method sets: can a value or a pointer satisfy the interface?

Which methods "belong to" a type determines whether it can satisfy an interface. The rule:

| Receiver defined as | `T`'s method set | `*T`'s method set |
|-------------|-------------|--------------|
| `func (t T)` | ✓ | ✓ |
| `func (t *T)` | ✗ | ✓ |

That is: **the pointer type `*T` has all methods; the value type `T` has only value-receiver methods.**

```go
type Speaker interface{ Speak() string }
type Dog struct{ Name string }
func (d *Dog) Speak() string { return d.Name + " says woof" } // pointer receiver

d := &Dog{Name: "Rex"}
var s Speaker = d          // OK: *Dog satisfies Speaker
fmt.Println(s.Speak())     // Rex says woof
```

Practical advice: **use all value receivers or all pointer receivers, don't mix**. If you need to modify state, the struct is large, or you want `*T` to satisfy an interface, use a pointer receiver.

The reason behind the rule: given a `*T` you can automatically dereference to get a value (the compiler inserts one dereference), but given a `T` you can't necessarily obtain a stable address (e.g., a map element, or a temporary value returned from a function) from which to call a pointer method. If `T`'s method set were allowed to include pointer methods, a call like `Dog{}.Speak()`—which has no addressable lvalue—would require the compiler to secretly manufacture a temporary variable and take its address; that temporary becomes invalid the instant the method returns, and if the method tried to save that pointer for later use, things would break. Go chooses to forbid this situation outright at the method-set level, rather than allow it while leaving behind a landmine that could misfire at any time.

```go
type Speaker interface{ Speak() }
type Dog struct{}
func (d *Dog) Speak() {}      // pointer receiver

var s Speaker = &Dog{}        // OK
// var s Speaker = Dog{}      // compile error: Dog does not implement Speaker (Speak is on *Dog)
```

### Method values vs method expressions

```go
d := Dog{}
f := d.Speak            // method value: binds the receiver d, type func()
f()                     // equivalent to d.Speak()

g := (*Dog).Speak       // method expression: the receiver becomes an explicit first parameter
g(&d)                   // equivalent to d.Speak()
```

A method value captures the receiver (which may cause it to escape to the heap, like a closure); a method expression merely treats the method as an ordinary function, with the receiver passed explicitly.

## V. Generics (Go 1.18+): type parameters and constraints

Go long lacked generics; 1.18 introduced type parameters. The syntax: declare type parameters in square brackets, paired with a **constraint**, and write the function once so it works across multiple types:

```go
func Sum[T int | float64](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

Sum([]int{1, 2, 3})        // 6, the type argument int is inferred automatically
Sum([]float64{1.5, 2.5})   // 4
```

A constraint can also be defined up front as a named interface that lists the set of allowed types:

```go
// constraint: an interface listing the allowed set of types
type Number interface {
    ~int | ~int64 | ~float64   // ~T means "all types whose underlying type is T"
}

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}
```

`comparable` is a built-in constraint (types usable with `==`/`!=`), commonly used for map-key-like generics:

```go
func Keys[K comparable, V any](m map[K]V) []K { /* ... */ }
```

The essence of a constraint: it is simply an interface, which can list methods (requiring the implementation of a method) as well as **a set of types** (`| ~T`).

Runnable, showing what `~` means (it admits named types whose underlying type matches):

```go
package main

import "fmt"

type Number interface {
	~int | ~int64 | ~float64
}

func Sum[T Number](xs []T) T {
	var total T
	for _, x := range xs {
		total += x
	}
	return total
}

type Celsius float64 // underlying type is float64, so ~float64 admits it

func main() {
	fmt.Println(Sum([]int{1, 2, 3}))      // 6   (T inferred as int)
	fmt.Println(Sum([]float64{1.5, 2.5})) // 4
	fmt.Println(Sum([]Celsius{1.5, 2.5})) // 4   (~float64 admits the named type Celsius)
}
```

### Underlying implementation: GC shape stenciling + dictionaries

This is where Go differs most from C++ templates. A C++ template is **fully monomorphized**: `vector<int>` and `vector<string>` generate two entirely separate copies of machine code, with zero runtime overhead but code bloat and slow compilation. Go takes a middle path:

**Generate code grouped by "GC shape" + pass a dictionary.** Specifically:

- The compiler generates code grouped by **GC shape**. The current implementation's grouping rule is: two types belong to the same shape **if and only if their underlying types are the same, or both are pointers**. So all pointer types (`*int`, `*string`, ...) share one instantiation; whereas `int` and `int64` have **different underlying types, so even though both are 8 bytes and neither contains pointers, each generates its own copy** (not shared).
- Each call additionally passes a hidden **dictionary** parameter, which holds the concrete type's meta-information (type descriptor, needed itabs, etc.), for the runtime to look up the concrete size and method addresses.

```
C++ templates (monomorphization):          Go generics (shape stenciling):
Sum<int>    → code copy A     Sum[int]     → int's shape code + int dictionary
Sum<int64>  → code copy B     Sum[int64]   → int64's shape code + int64 dictionary (not shared with int)
Sum<*T>×N   → N copies        Sum[*T]×N    → 1 copy of pointer shape code + individual dictionaries
```

**Why not monomorphize directly the way C++ does**: when designing generics, the Go team was explicitly wary of repeating the mistakes of C++ templates—monomorphization means slow compilation and binary bloat, and that cost is especially high given Go's emphasis on fast compilation and single static binaries. **Why not go with pure type erasure the way Java does, either**: type erasure means every generic argument has to be boxed into an interface, and every access then costs an indirect jump and a potential allocation, which is a bigger runtime cost. GC shape stenciling is a middle ground between the two: types of the same shape share code, avoiding the bloat of monomorphization; but what's shared is machine code, not the type information itself—the type details are filled in at runtime via the dictionary, avoiding the cost that type erasure incurs by having to box everywhere. Conclusion: **somewhere between monomorphization and boxing**—faster than Java's type erasure + boxing (which boxes everywhere), more code-frugal than C++ monomorphization; but dictionary lookups and indirect access through pointers do bring a bit of runtime overhead, so in some scenarios it's not as good as hand-written concrete types, nor as extreme as C++ templates.

| | C++ templates | Go generics | Java generics |
|--|---------|---------|----------|
| Mechanism | Monomorphization | shape stenciling + dictionary | Type erasure + boxing |
| Code size | Large (one per type) | Medium (one per shape) | Small (one) |
| Runtime overhead | None | Slight (dictionary/indirection) | Large boxing overhead |
| Compile-time computation | Strong (template metaprogramming) | Weak (just parameterization) | None |

Don't do generics for the sake of generics. Good fits: containers/algorithms (the `slices`, `maps` packages, see [Chapter 11](11-标准库算法.md)), performing the same operation on multiple numeric types. In many scenarios an `any` + type assertion or an ordinary interface is clearer instead.

## Summary

- **Usage level**: `func (recv T) Name(...)` defines a method—a value receiver operates on a copy, a pointer receiver can modify in place; embedding (an anonymous field) lets fields/methods be "promoted" to the outer type; an interface is automatically satisfied as soon as the method signatures match, with no explicit implementation declaration needed; `T` and `*T` have different method sets—a pointer-receiver method belongs only to `*T`; generics declare type parameters with `[T Constraint]`, letting one piece of code support multiple types.
- **Design trade-offs**: having no classes or inheritance, only composition + interfaces, elevates "favor composition over inheritance" from a best practice into a language-level mandate—the cost is giving up the expressiveness of deep inheritance hierarchies, in exchange for eliminating the fragile base class problem; implicit interface implementation hands the right to define interfaces to consumers rather than producers, reducing coupling between modules, at the cost that the compiler can't list which interfaces a type implements right at its definition site; an interface value is two machine words (`itab`/`_type` + `data`), and this two-word representation itself is what makes "a nil pointer of a concrete type ≠ a nil interface" an inevitable trap, in exchange for efficient, boxing-free method dispatch; the method-set rule (`*T` gets everything, `T` gets only value-receiver methods) follows from the fact that a value type doesn't always have an addressable lvalue; generics are neither monomorphization nor pure boxing, but code generated grouped by GC shape plus a passed-in dictionary—a compromise between "compile-time bloat" and "boxing overhead."
