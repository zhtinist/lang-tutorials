# Algorithms and Data Structures in the Standard Library

C++ programmers are familiar with the STL: `std::sort`, `std::vector`, `std::map`, `std::priority_queue`, iterators decoupled from algorithms. Go's standard library follows a different philosophy: early on there were no generics, so containers were either built in (slice/map) or relied on **interfaces + hand-written implementations**; not until Go 1.18/1.21 did the generic utility functions arrive. This chapter walks through sorting, the generic `slices`/`maps`, the `container/*` trio, and string/numeric utilities, and explains the trade-offs behind this historical shift "from no generics to generics."

## 1. The sort package: interface-based sorting

### sort.Slice — the most commonly used

```go
people := []Person{{"Bob", 31}, {"Alice", 25}, {"Eve", 40}}
sort.Slice(people, func(i, j int) bool {   // less(i,j): should i come before j
    return people[i].Age < people[j].Age
})
```

`sort.Slice` uses reflection to get the slice and swap elements, and the `less` closure defines the order. It is not stable; for stability use `sort.SliceStable`.

### sort.Sort — interface-based (the classic pre-generics approach)

Sort any collection, as long as it implements the three methods of `sort.Interface`. This is very "Go 1.0 style" — **polymorphism through interfaces rather than templates**:

```go
type sort.Interface interface {
    Len() int
    Less(i, j int) bool
    Swap(i, j int)
}

type ByAge []Person
func (a ByAge) Len() int           { return len(a) }
func (a ByAge) Less(i, j int) bool { return a[i].Age < a[j].Age }
func (a ByAge) Swap(i, j int)      { a[i], a[j] = a[j], a[i] }

sort.Sort(ByAge(people))
```

Convenience functions: `sort.Ints`, `sort.Strings`, `sort.Float64s`.

### The internal algorithm: pattern-defeating quicksort (pdqsort)

Since Go 1.19, `sort` internally uses **pdqsort** (pattern-defeating quicksort) rather than naive quicksort. **Why not just use textbook quicksort**: when naive quicksort picks its pivot with a fixed strategy (say, always the first or middle element), there exist inputs you can construct — already sorted, reverse sorted, or data an attacker deliberately crafts — that make every partition wildly unbalanced, degrading to O(n²). For a standard-library utility that "any caller might pass any data to," that's an unacceptable risk, especially when the sort key comes from untrusted input (e.g. a network request): the degenerate behavior itself becomes an exploitable denial-of-service vector. pdqsort is a hybrid sort designed specifically to address this:

- **The body is quicksort**, but with smarter pivot selection (median-of-samples) to avoid degeneration as much as possible.
- **Small arrays (about ≤12 elements) switch to insertion sort** — better constants and cache-friendlier at small scale.
- **On too many "bad partitions" it falls back to heapsort** — pdqsort maintains a `limit` counter (initial value about `log₂n`), decremented each time it hits a badly unbalanced pivot partition; when it reaches zero it switches to heapsort (note this is **not** classic introsort's "recursion depth 2·log n" criterion). This prevents adversarially constructed inputs from dragging quicksort into O(n²), guaranteeing **worst case O(n log n)** — which is exactly the direct answer to the "worried about degeneration" problem: once the algorithm detects that partitions keep coming out unbalanced (meaning it may be facing adversarial input), it proactively switches to heapsort as a fallback that doesn't share this weakness.
- It can recognize "patterns" such as already-sorted or many duplicate elements, taking a fast path.

This shares a lineage with C++ STL's `std::sort` (introsort: quicksort + heapsort + insertion sort); pdqsort is a more recent improvement, and both are solutions to the same underlying demand: giving a general-purpose library function a worst-case guarantee.

| Function | Stable | Average | Worst |
|------|------|------|------|
| `sort.Sort`/`sort.Slice` | no (pdqsort) | O(n log n) | O(n log n) |
| `sort.Stable`/`sort.SliceStable` | yes (in-place merge) | O(n log n) comparisons | O(n log² n) (more swaps, in-place needs no O(n) extra space) |

### sort.Search — general binary search

```go
// in already-ascending a, find the first index >= target (equivalent to lower_bound)
i := sort.Search(len(a), func(i int) bool { return a[i] >= target })
if i < len(a) && a[i] == target { /* found */ }
```

`sort.Search` finds "the first position where the predicate goes from false to true," O(log n). The predicate must be monotonic. Less error-prone than a hand-written binary search.

## 2. The generic slices / maps packages (Go 1.21)

Go 1.21 officially brought the generic utility functions into the standard library as `slices` and `maps` — type-safe, no reflection, no need to write `sort.Interface`. This is a belated response to C++'s `<algorithm>`.

### The slices package

```go
import "slices"

s := []int{3, 1, 2}
slices.Sort(s)                     // ascending, internally pdqsort, no reflection (faster than sort.Slice)
slices.SortFunc(people, func(a, b Person) int {
    return cmp.Compare(a.Age, b.Age) // returns <0/0/>0 (note: not bool!)
})
i, found := slices.BinarySearch(s, 2)  // binary search, returns index and whether it hit
ok := slices.Contains(s, 2)
idx := slices.Index(s, 2)              // returns -1 if not found
mx := slices.Max(s)                    // Min also exists
eq := slices.Equal(s, []int{1, 2, 3})
s = slices.Insert(s, 1, 99)            // insert at index 1
s = slices.Delete(s, 0, 1)            // delete [0,1)
r := slices.Clone(s)                   // shallow copy (see the aliasing issue in Chapter 07)
s = slices.Compact(s)                  // remove adjacent duplicates
```

Note that `SortFunc`'s comparison function returns **`int` (three-state: negative/zero/positive)**, not the `bool` of `sort.Slice` — consistent with C's `qsort` comparator, able to express equality. A `bool`-based `less` function can only answer "is a strictly less than b"; when you need multi-level sorting (by age first, then by name when ages tie), the `bool` interface forces you to shoehorn the third state, "equal," into two separate `less` calls, whereas a three-state `int` can directly express "equal, move on to compare the next field" — which pairs more directly with the `cmp` package's `cmp.Compare`/`cmp.Or`.

A runnable example: `slices.Sort`, `slices.BinarySearch`, and a multi-level `slices.SortFunc` built with `cmp.Compare`:

```go
package main

import (
	"cmp"
	"fmt"
	"slices"
)

type Person struct {
	Name string
	Age  int
}

func main() {
	nums := []int{3, 1, 4, 1, 5, 9, 2, 6}
	slices.Sort(nums) // ascending, no reflection
	fmt.Println(nums)
	// [1 1 2 3 4 5 6 9]

	i, found := slices.BinarySearch(nums, 5) // requires sorted input
	fmt.Println(i, found)
	// 5 true

	people := []Person{{"Bob", 31}, {"Alice", 25}, {"Eve", 25}}
	slices.SortFunc(people, func(a, b Person) int {
		if c := cmp.Compare(a.Age, b.Age); c != 0 {
			return c // by age first
		}
		return cmp.Compare(a.Name, b.Name) // then by name when age ties
	})
	fmt.Println(people)
	// [{Alice 25} {Eve 25} {Bob 31}]
}
```

### The maps package

A map is itself unordered and has no built-in "get all keys" operation (see [Chapter 07](07-复合数据结构.md)'s explanation of this unorderedness); the `maps` package fills the gap:

```go
import "maps"

m := map[string]int{"a": 1, "b": 2}
ks := maps.Keys(m)      // Go 1.23: returns an iterator iter.Seq[K]
vs := maps.Values(m)
m2 := maps.Clone(m)     // shallow copy
maps.Copy(dst, src)     // copy src into dst
eq := maps.Equal(m, m2)
```

> Version note: the signatures of `maps.Keys`/`Values` were changed in Go 1.23 to return `iter.Seq`, along with range-over-func iterators. To get sorted keys, a common idiom is: `ks := slices.Sorted(maps.Keys(m))` (1.23+). Defer to the docs for your Go version.

A runnable example: `maps.Keys` returns an iterator, `slices.Sorted` collects and sorts it, giving a deterministic traversal order:

```go
package main

import (
	"fmt"
	"maps"
	"slices"
)

func main() {
	m := map[string]int{"banana": 3, "apple": 5, "cherry": 1}

	// maps.Keys returns an iterator (Go 1.23+); slices.Sorted collects and sorts
	for _, k := range slices.Sorted(maps.Keys(m)) {
		fmt.Printf("%s=%d\n", k, m[k])
	}
	// apple=5
	// banana=3
	// cherry=1

	total := 0
	for v := range maps.Values(m) {
		total += v
	}
	fmt.Println("total:", total)
	// total: 9
}
```

## 3. The container/* trio

The standard library's only three "container" packages, all **interface-based**, non-generic old designs (not yet made generic to this day).

### container/heap — binary heap (priority queue)

Not a ready-to-use class, but a set of **algorithms** operating on `heap.Interface`. You must first implement the three `sort.Interface` methods **plus** `Push`/`Pop`, five methods in total, backed by a slice storing the heap:

```go
type IntHeap []int
func (h IntHeap) Len() int            { return len(h) }
func (h IntHeap) Less(i, j int) bool  { return h[i] < h[j] } // min-heap
func (h IntHeap) Swap(i, j int)       { h[i], h[j] = h[j], h[i] }
func (h *IntHeap) Push(x any)         { *h = append(*h, x.(int)) }
func (h *IntHeap) Pop() any {
    old := *h; n := len(old)
    x := old[n-1]; *h = old[:n-1]
    return x
}

h := &IntHeap{5, 2, 8}
heap.Init(h)              // O(n) build
heap.Push(h, 1)          // O(log n)
min := heap.Pop(h)       // O(log n), pops the minimum
```

Compared to C++'s `std::priority_queue`: the same functionality, but Go makes you hand-write 5 methods (the cost of the pre-generics era), and `Push`/`Pop` box through `any`. This is the most blunt sample of "early Go's lack of generics" as a pain point: the standard library needs to offer a heap algorithm that works on **any** element type, and with no type parameters available, there are only two roads left — either use interfaces as here (the algorithm depends only on 5 methods and doesn't care about the concrete type, at the cost of every user having to rewrite the same boilerplate and boxing), or give up on generality and duplicate a dedicated implementation for each element type. `container/heap` chose the former.

| Operation | Complexity |
|------|--------|
| Init | O(n) |
| Push / Pop | O(log n) |
| Peek the top (h[0]) | O(1) |

A runnable example: a complete min-heap; after `Init`/`Push`/`Pop`, popping in turn yields ascending order:

```go
package main

import (
	"container/heap"
	"fmt"
)

// IntHeap is a min-heap implementing the 5 methods of heap.Interface.
type IntHeap []int

func (h IntHeap) Len() int           { return len(h) }
func (h IntHeap) Less(i, j int) bool { return h[i] < h[j] } // < means min-heap
func (h IntHeap) Swap(i, j int)      { h[i], h[j] = h[j], h[i] }
func (h *IntHeap) Push(x any)        { *h = append(*h, x.(int)) }
func (h *IntHeap) Pop() any {
	old := *h
	n := len(old)
	x := old[n-1]
	*h = old[:n-1]
	return x
}

func main() {
	h := &IntHeap{5, 2, 8, 1}
	heap.Init(h)                 // build the heap in O(n)
	heap.Push(h, 3)              // O(log n)
	fmt.Println("top:", (*h)[0]) // the top is always the minimum

	var out []int
	for h.Len() > 0 {
		out = append(out, heap.Pop(h).(int)) // pop in turn -> ascending
	}
	fmt.Println("popped:", out)
	// top: 1
	// popped: [1 2 3 5 8]
}
```

### container/list — doubly linked list

```go
l := list.New()
e := l.PushBack(1)       // append at tail, returns *list.Element
l.PushFront(0)           // insert at head
l.InsertAfter(2, e)
for e := l.Front(); e != nil; e = e.Next() {
    fmt.Println(e.Value) // Value is any, assert on use
}
```

Elements are nodes (holding prev/next pointers + an `any` value); insert/delete is O(1). But in practice **a slice is faster in most scenarios** — modern CPU performance depends heavily on cache locality: a slice's elements sit contiguously in memory, so sequential traversal gets a far higher cache-hit rate than the scattered, random-access nodes of a linked list spread across the heap; even though a linked list's asymptotic complexity (O(1) insertion) is theoretically better, actual throughput is still usually higher for a slice. A linked list only pays off when you frequently insert/delete in the middle and hold node references — the same trade-off as C++'s `std::list` vs `std::vector`.

### container/ring — circular linked list

A fixed circular-buffer doubly-linked ring; `r.Next()`/`r.Prev()` cycle around, `r.Move(n)`. Rarely used; occasionally handy for a fixed-length sliding window.

## 4. Strings and bytes: strings / bytes / strconv

### strings

`string` is a read-only byte sequence ([Chapter 04](04-数据类型.md)); the `strings` package provides operations that **return new strings (immutable)**:

```go
strings.Contains(s, "go")     strings.HasPrefix / HasSuffix
strings.Split(s, ",")         strings.Join(parts, "-")
strings.ToUpper / ToLower     strings.TrimSpace / Trim
strings.ReplaceAll(s, "a", "b")   strings.Fields(s) // split on whitespace
strings.Index(s, "x")             strings.Repeat("ab", 3)
```

For heavy concatenation use `strings.Builder` (an internally growable `[]byte`, avoiding allocating a new string on every concatenation — like using C++'s `string::reserve` + `+=`):

```go
var b strings.Builder
for i := 0; i < 100; i++ { b.WriteString("x") }
result := b.String()
```

Because a `string` is immutable, writing `s += "x"` has to allocate a brand-new string, copy the old contents over, then append, every single time — doing that in a loop is O(n²). `strings.Builder` internally maintains a growable `[]byte` buffer, and the only possible conversion happens once you finally call `String()`; it's the same idea as a slice's growth strategy — **as long as the underlying buffer can grow in place, you don't need to reallocate on every "modification" of an immutable type**.

### bytes

An API almost mirroring `strings`, but operating on `[]byte` (mutable). `bytes.Buffer` is a read-write buffer, often used as an `io.Reader`/`io.Writer`. Use it for handling binary, assembling network protocols, and avoiding `string`↔`[]byte` copies.

### strconv

Convert between strings and numbers/booleans (faster than `fmt`, no reflection):

```go
n, err := strconv.Atoi("42")            // string → int
s := strconv.Itoa(42)                    // int → string
f, err := strconv.ParseFloat("3.14", 64)
b, err := strconv.ParseBool("true")
s = strconv.FormatInt(255, 16)           // "ff"
s = strconv.Quote(`he said "hi"`)        // add escaped quotes
```

Why `strconv` is faster than `fmt.Sprintf("%d", n)`: to support arbitrary types and format verbs, the `fmt` family of functions has to use reflection at runtime to determine an argument's concrete type before deciding how to format it; each `strconv` function serves exactly one concrete, statically-known type conversion, so the compiler can determine which code path to take at compile time, with none of reflection's runtime overhead.

## 5. math / math/rand

```go
math.MaxInt64, math.Pi, math.Inf(1), math.NaN()
math.Sqrt, math.Pow, math.Floor, math.Abs, math.Max/Min
math.MaxInt, math.MaxUint32 ...          // boundary constants for the various types
```

Random numbers:

```go
import "math/rand/v2"                     // the new version, Go 1.22+, recommended
n := rand.IntN(100)                       // [0,100)
f := rand.Float64()                       // [0.0,1.0)
rand.Shuffle(len(s), func(i, j int) { s[i], s[j] = s[j], s[i] })
```

> `math/rand` is **not cryptographically secure** (it is predictable). For secure randomness (tokens, keys, salts) use `crypto/rand`. **Why split this into two separate packages**: `math/rand` optimizes for speed and reproducibility (given a seed, the sequence is entirely deterministic — an advantage rather than a drawback for testing, simulation, and games); internally it's a deterministic pseudo-random-number algorithm, so once the internal state is known, every subsequent output can be predicted. `crypto/rand`, by contrast, trades away speed and reproducibility, reading genuinely unpredictable random data from an OS-provided entropy source. Mixing the two up — using `math/rand` to generate keys or tokens — makes sensitive data that looks random actually predictable to an attacker; this is a real security-vulnerability pattern that has happened in practice, which is why the standard library deliberately draws the line with two separate package names rather than blurring it as two options within the same package. **As of Go 1.20**, the top-level `math/rand` is randomly seeded by default and `rand.Seed` is deprecated, so you no longer need to seed manually (only pre-1.20 required it); `math/rand/v2` (1.22) is the cleaner new API.

## 6. Complexity cheat sheet

| Structure/operation | Complexity | Notes |
|-----------|--------|------|
| slice index `s[i]` | O(1) | with bounds check |
| slice `append` (enough cap) | O(1) amortized | O(n) copy on growth |
| map lookup/insert/delete | O(1) average | worst-case degrades; order is random |
| `sort.Slice`/`slices.Sort` | O(n log n) | pdqsort, worst case also O(n log n) |
| `sort.Search`/`slices.BinarySearch` | O(log n) | requires sorted input |
| heap Push/Pop | O(log n) | top O(1) |
| list insert/delete (holding the node) | O(1) | traversal O(n) |

## 7. History: why Go once had no generic containers

Early Go (1.0~1.17) **had no generics**, which directly shaped the standard library:

- The only containers were the built-in slice/map/chan (specially supported by the compiler, working on any element type).
- Generic algorithms could only rely on **interfaces + reflection**: `sort.Sort` requires you to implement `sort.Interface`; `container/heap`/`list` store elements as `any` (formerly `interface{}`), requiring a type assertion on extraction, with boxing overhead and no compile-time type checking.
- To get a "type-safe stack/queue/set," you either copy-pasted a concrete-type version, used `interface{}` sacrificing type safety, or used code generation (`go generate`). This was the community's biggest complaint for years.

The designers' reasoning at the time: generics would add language complexity, and they feared repeating C++ templates' complexity and compile-time bloat, preferring to not do it rather than do it wrong — this wasn't procrastination but a deliberate risk trade-off: once generics are added to a language, it's almost impossible to remove or substantially redesign them in later versions, so a flawed generics design would be permanently locked into the language, whereas "no generics" is a shortcoming that can at least be patched incrementally (write a few more concrete-type versions in the standard library, or have users reach for code generation) — of two evils, they chose the lesser one. After many rounds of proposals, **Go 1.18 (2022) introduced type parameters** ([generics mechanism, Chapter 08](08-面向对象与类型系统.md)), and **Go 1.21 (2023)** settled generic utilities into the standard library as `slices`/`maps`/`cmp`, with `min`/`max`/`clear` becoming built-in functions. Only then did Go have type-safe, reflection-free generic algorithms.

The practical impact of the shift:

- New code prefers `slices.Sort` over `sort.Slice` (the former is reflection-free, faster, type-safe).
- Old interface-based containers like `container/heap`/`list` were **not** made generic (backward compatibility), but the community already has plenty of generic container libraries to choose from.
- Generics make it possible to "write a type-safe `Map`/`Filter`/`Reduce`," but the Go team remains restrained, adding only the most basic things to the standard library, not pursuing the vast algorithm families of C++/Rust.

In one sentence: Go evolved from "polymorphism through interfaces, containers via built-ins + reflection" to "a generics + built-ins hybrid," but overall it is still far more minimal than the STL — **enough and simplicity-first** is its consistent trade-off.
