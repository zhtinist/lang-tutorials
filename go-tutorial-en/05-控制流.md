# Control Flow

This chapter starts by walking through **how to write** the control-flow keywords `for`, `if`, `switch`, and `defer`—no background knowledge needed, just follow along and it runs. Then we go under the hood, to the copy semantics and evaluation timing behind `range` and `defer`—this is exactly where C/C++ veterans most often trip up, and it's the focus of the second half of the chapter. If you only want to use these, the first section is enough; if you want to understand "why does this code print in such a strange order," keep reading.

## Just Using It: Loops, Branches, and defer at a Glance

Go has exactly one looping keyword, `for`; one pair of branching keywords, `if`/`switch`; plus a very handy `defer`. First, a small example that uses all of them:

```go
package main

import "fmt"

func main() {
	// for: Go's only looping keyword
	for i := 0; i < 3; i++ {
		fmt.Println("loop", i)
	}

	// range: iterate over slices, arrays, strings, maps
	fruits := []string{"apple", "banana", "cherry"}
	for idx, name := range fruits {
		fmt.Println(idx, name)
	}

	// if / else
	score := 85
	if score >= 60 {
		fmt.Println("pass")
	} else {
		fmt.Println("fail")
	}

	// switch: branch on a value
	switch score / 10 {
	case 10, 9:
		fmt.Println("excellent")
	case 8:
		fmt.Println("good")
	default:
		fmt.Println("average")
	}
}
```

Output:

```
loop 0
loop 1
loop 2
0 apple
1 banana
2 cherry
pass
good
```

Now meet `defer`—it **postpones a call until just before the current function returns**, commonly used for "clean up once you're done with it" work:

```go
package main

import "fmt"

func doWork() {
	fmt.Println("starting work")
	defer fmt.Println("work done, cleaning up") // deferred until doWork returns
	fmt.Println("working...")
}

func main() {
	doWork()
}
```

Output:

```
starting work
working...
work done, cleaning up
```

Notice that "cleaning up" is written in the middle but printed last—that's `defer` at work. That's already enough to write ordinary loops, branches, and cleanup logic. Now let's go under the hood: Go's control flow is deliberately kept "small but exact"—`for` swallows every scenario that C's `while`/`do...while`/complex `for` variants cover, `switch` completely flips the default behavior, and the copy semantics and evaluation timing behind `range` and `defer` are the real focus of this chapter.

## Going Deeper

Go's control flow is deliberately kept "small but exact." C/C++'s `while`, `do...while`, the comma operator, and elaborate `for(;;)` variants are all compressed here into a **single `for` keyword**; `switch` also completely flips its default behavior. The rest of this chapter explains, from the underlying-implementation angle, exactly what each construct becomes after compilation, and the copy semantics behind `range` and `defer`—the two places C/C++ veterans most often get tripped up.

## for: The Only Looping Keyword

Go has no `while` and no `do...while`. Every loop uses `for`, which comes in three forms; after compilation they all boil down to the same jump instructions (condition test + backward jump), no different from C's loops at the machine-code level.

```go
package main

import "fmt"

func main() {
	// Form one: the full three-clause form, same as C's for(i=0;i<n;i++)
	for i := 0; i < 3; i++ {
		fmt.Println("three-clause", i)
	}

	// Form two: condition only, equivalent to C's while(cond)
	n := 3
	for n > 0 {
		fmt.Println("while-like", n)
		n--
	}

	// Form three: no condition, equivalent to C's for(;;) or while(1)
	count := 0
	for {
		if count == 2 {
			break // must break yourself, or it loops forever
		}
		fmt.Println("infinite loop", count)
		count++
	}
}
```

The `i := 0` in the three-clause form declares an `i` whose **scope is limited to that `for` statement** (including the loop body), matching C99's `for(int i...)`. It's normally allocated on the stack; if a closure inside the loop body captures it (see [06-函数与调用栈](06-函数与调用栈.md)), escape analysis moves it to the heap.

`do...while` semantics (execute at least once) are hand-written in Go with `for` + `break`:

```go
for {
	// the loop body runs at least once
	if !condition {
		break
	}
}
```

## range: Iteration Semantics and Copy Behavior

`range` is the construct that most demands you keep an eye on memory. The core fact: **the range iteration variable is a copy of the element being iterated over**—each iteration **copies** the current element into the iteration variable. This matches C++'s `for (auto x : v)` (value copy), not `for (auto& x : v)` (reference).

### range over slice / array

```go
s := []int{10, 20, 30}
for i, v := range s {
	// i is the index, v is a [copy] of s[i]
	// modifying v does not affect s; to change the original array you must use s[i] = ...
	v += 100
	_ = v
	_ = i
}
// s is still {10, 20, 30}
```

Memory picture (during iteration 2):

```
Underlying array (heap or stack):
  s[0]=10  s[1]=20  s[2]=30
                ↑
         range reads s[1]=20
                │  copy
                ▼
   iteration variable v = 20   ← an independent stack variable, changing it doesn't touch the array
```

**Pitfall one: the range expression is evaluated exactly once, before the loop starts.** For a slice, `range` copies the slice's **header (SliceHeader: pointer + length + capacity)**, not the underlying array. So an `append` inside the loop that triggers a reallocation does not change how many times this iteration runs—but if you modify an element **in place** (no reallocation), the change **will be seen** (because the pointer still points at the same array):

```go
s := []int{1, 2, 3}
for i, v := range s {
	if i == 0 {
		s[2] = 99 // no reallocation, modified in place; iteration 3 will see 99
		s = append(s, 4) // the iteration length is already locked at 3; element 4 is never visited
	}
	fmt.Println(i, v)
}
// output: 0 1 / 1 2 / 2 99
```

For **arrays** (not slices) it matters whether you take the element value or not: `for i, v := range arr` first **copies the entire array** before iterating (`v` is taken from that copy), which is costly for large arrays; but `for i := range arr`, which only uses the index and never reads element values, is **not** copied by the compiler. If you want to take values while also avoiding the copy, range over a **pointer to the array**:

```go
var arr [1024]int
for i := range arr {      // index only: doesn't copy these 8KB
	arr[i] = i
}
for i, v := range &arr {  // need values but want to avoid the copy: range over &arr, v reads straight from the original array
	arr[i] = v + i
}
```

### range over map

```go
m := map[string]int{"a": 1, "b": 2}
for k, v := range m {
	// both k and v are copies; iteration order is [random] (the runtime deliberately randomizes the starting bucket)
	_ = k
	_ = v
}
```

Go's map iteration order is randomized at runtime—this is deliberate, to keep code from depending on order. C++'s `std::map` is ordered, and `unordered_map` has an unspecified but stable-within-a-process order; Go doesn't even give you stability. Deleting the current key or a not-yet-visited key during iteration is safe; a newly inserted key **may or may not** be visited—the behavior is undefined.

### range over string: by rune, not byte

In C a string is just a `char*`, one byte at a time. Go's `range` over a `string` **decodes UTF-8 and gives you one rune at a time (a Unicode code point, `int32`)**, with the index being the byte offset where that rune starts:

```go
s := "aé中" // UTF-8: 'a'=1 byte, 'é'=2 bytes, '中'=3 bytes
for i, r := range s {
	fmt.Printf("byte offset=%d rune=%c code point=%d\n", i, r, r)
}
// byte offset=0 rune=a code point=97
// byte offset=1 rune=é code point=233
// byte offset=3 rune=中 code point=20013
```

Notice `i` skips 2 (because `é` occupies 2 bytes). If you want to iterate byte by byte, use an ordinary index `for i := 0; i < len(s); i++ { s[i] }`, where `s[i]` is a `byte`. On an invalid UTF-8 sequence, `range` returns `RuneError` (U+FFFD) and advances 1 byte.

### range over channel

```go
ch := make(chan int)
// ... another goroutine sends values into ch, then calls close(ch) when done
for v := range ch {
	// receives one value from ch (a copy) each time, until ch is closed and drained
	_ = v
}
```

`range ch` is equivalent to repeatedly doing `v, ok := <-ch` until `ok == false` (the channel is closed and empty). See [10-并发模型](10-并发模型.md) for details.

### range over int (Go 1.22+)

Since Go 1.22 you can write `for i := range n` directly, equivalent to `for i := 0; i < n; i++`:

```go
for i := range 5 { // i goes from 0 to 4
	fmt.Println(i)
}
```

### Go 1.22: A Fresh Loop-Variable Instance Each Iteration

This is a **language semantics change** with no counterpart in C/C++. Before Go 1.22, the loop variable in `for i := 0; ...` and `for k, v := range` had **only a single instance for the entire loop**, with every iteration reusing the same memory address. This meant that closures/goroutines capturing it all ended up pointing at the same variable (a classic pitfall):

```go
// Go 1.21 and earlier: all print 3 (sharing the same i; i==3 when the loop ends); Go 1.22+: prints 0 1 2
funcs := []func(){}
for i := 0; i < 3; i++ {
	funcs = append(funcs, func() { fmt.Println(i) })
}
for _, f := range funcs {
	f()
}
```

Since Go 1.22, **each iteration's loop variable is a fresh instance** (logically re-declared every iteration), so each closure captures a different variable, and the output becomes `0 1 2`. This fixes one of the most common historical bugs. See [06-函数与调用栈](06-函数与调用栈.md#closures-capture-variables-not-values) for capture details.

## if with an Init Statement

`if` can run a simple statement before the condition; any variable it declares is scoped **only to the if/else block**:

```go
if v, ok := m["key"]; ok {
	// v and ok are only visible here
	fmt.Println(v)
} else {
	// v and ok are visible in the else branch too
	fmt.Println("missing")
}
// v and ok are no longer visible here
```

This keeps a variable's lifetime very tight; under the hood `v` and `ok` are stack variables scoped to the if statement's block. It's idiomatic for error handling: `if err := doSomething(); err != nil { ... }`.

## switch: No Automatic Fallthrough

In C, `switch` cases **automatically fall through**, and forgetting `break` is a classic bug. Go flips this: **every case automatically breaks by default, with no fallthrough**.

```go
switch x {
case 1:
	fmt.Println("one") // automatically exits after running, won't fall into case 2
case 2, 3: // a single case can match multiple values (comma-separated)
	fmt.Println("two or three")
default:
	fmt.Println("other")
}
```

### The fallthrough Keyword

If you really want to fall through, use the explicit `fallthrough` keyword (it unconditionally enters the **next** case's body, without checking that case's condition):

```go
switch x {
case 1:
	fmt.Println("one")
	fallthrough // force execution to continue into case 2's body
case 2:
	fmt.Println("two") // this also runs when x==1
}
```

### switch with No Expression = an if-else Chain

A `switch` with no expression after it is equivalent to `switch true`, with each case writing a boolean condition—used as a cleaner if-else chain:

```go
switch {
case score >= 90:
	grade = "A"
case score >= 60:
	grade = "B"
default:
	grade = "F"
}
```

### switch with an Init Statement

Like `if`, it can carry an init statement: `switch v := f(); v { ... }`.

### type switch: Runtime Type Dispatch

Dispatches on an interface value's **dynamic type**; under the hood this reads the interface's type pointer (see [08-面向对象与类型系统](08-面向对象与类型系统.md)) and compares it:

```go
func describe(i interface{}) {
	switch v := i.(type) { // in each case, v has that case's concrete type
	case int:
		fmt.Println("int:", v*2) // v is an int
	case string:
		fmt.Println("string len:", len(v)) // v is a string
	case nil:
		fmt.Println("nil")
	default:
		fmt.Printf("unknown type %T\n", v)
	}
}
```

This resembles a chain of C++ `dynamic_cast`s, or `std::variant` + `std::visit`, but is directly backed by the runtime's interface type information, without the vtable-walking overhead of RTTI.

## Labeled break / continue

To break out of an **outer** loop from within nested loops, C uses `goto` or a flag variable. Go provides labeled `break`/`continue`, with the label placed before the loop:

```go
outer:
	for i := 0; i < 3; i++ {
		for j := 0; j < 3; j++ {
			if i*j > 2 {
				break outer // breaks straight out of the outer for, not just the inner one
			}
			if j == 1 {
				continue outer // advances the outer loop to its next iteration of i
			}
			fmt.Println(i, j)
		}
	}
```

`break` can also be used on `switch`/`select`; with a label it breaks out of whichever level that label marks.

## goto

Go keeps `goto`, but with restrictions: it can't jump over a variable declaration, and it can't jump into the middle of a block. It's mainly used to break out of deep nesting or for error cleanup (in places without `defer`):

```go
	i := 0
loop:
	if i < 3 {
		fmt.Println(i)
		i++
		goto loop // jump backward
	}
```

In practice there's almost always a better way to write it (`for`, `defer`), so `goto` is rarely used.

## ⭐ defer: Deferred Execution

`defer` **postpones a function call until just before the current function returns**, and is Go's core mechanism for managing resources (closing files, unlocking, releasing)—it replaces C++'s RAII/destructors. The first section already showed its most basic usage; here we dig into the three underlying facts you must nail down.

### Fact One: LIFO Order

Multiple `defer`s in the same function execute in **last-in-first-out (stack) order**. The runtime pushes each defer record onto the current goroutine's defer linked list (the `_defer` struct), popping them off the head of the list one by one when the function returns:

```go
func main() {
	defer fmt.Println("1") // runs last
	defer fmt.Println("2")
	defer fmt.Println("3") // runs first
	fmt.Println("function body")
}
// output: function body / 3 / 2 / 1
```

Memory diagram (defer linked list, newest inserted at the head):

```
registration order: defer1 → defer2 → defer3
linked list:  head → [defer3] → [defer2] → [defer1] → nil
popped from the head on return, executed as: 3, 2, 1
```

### Fact Two: Arguments Are Evaluated at Registration Time

This is the easiest point to get wrong. `defer f(x)` copies the **current value** of `x` into the defer record **at the moment the defer statement executes**—**not** when the call actually happens later:

```go
func main() {
	i := 0
	defer fmt.Println("i seen by defer =", i) // i=0 is copied into the record right now
	i = 100
	fmt.Println("i at end of function body =", i) // 100
}
// output: i at end of function body = 100 / i seen by defer = 0
```

What's deferred is the **execution of the call**, not the **evaluation** of its arguments. If you want it to read the latest value at call time, wrap the logic in a closure (a closure captures the variable—see the next chapter):

```go
defer func() { fmt.Println(i) }() // reads i at call time, prints 100
```

Putting LIFO and "evaluated at registration time" together makes `defer` in a loop intuitive:

```go
package main

import "fmt"

func main() {
	for i := 0; i < 3; i++ {
		defer fmt.Println("defer:", i) // each iteration copies the current i into a record
	}
	fmt.Println("function body")
}
// output:
// function body
// defer: 2
// defer: 1
// defer: 0
```

The three defers each remember the `i` at the moment they were registered (0, 1, 2), and run in LIFO order when the function returns—so they print 2, 1, 0.

### Fact Three: Interaction with Named Return Values / return

`return x` is, under the hood, **two steps**: first write the return value into the return slot, then run the defers, and only then actually return. If the function has **named return values**, a defer can modify that return slot during the second step, thereby changing the final return value:

```go
func inc() (result int) { // named return value result
	defer func() {
		result++ // the defer modifies the return slot, changing the final return value
	}()
	return 10 // result is set to 10 first, then defers run, then result becomes 11
}
// inc() returns 11
```

Contrast this with anonymous return values: with `return 10` the value is already copied to a temporary return location, so changing a local variable inside the defer cannot affect it. This mechanism is commonly used to uniformly wrap an error inside a defer (see [09-错误处理与资源管理](09-错误处理与资源管理.md)).

### Fact Four: The Pitfall of defer in Loops

`defer` is tied to **the function returning**, not to **a block or loop ending**. Deferring inside a loop lets calls pile up until the function ends, which can exhaust resources:

```go
// Bad: 1000 file handles all stay open until the function ends
func bad(files []string) {
	for _, name := range files {
		f, _ := os.Open(name)
		defer f.Close() // all pile up, none close during the loop!
		// ...
	}
} // only here do all 1000 finally get closed at once

// Good: extract the loop body into a function, so the defer fires when that function returns
func good(files []string) {
	for _, name := range files {
		func() {
			f, _ := os.Open(name)
			defer f.Close() // closes at the end of each iteration
			// ...
		}()
	}
}
```

### defer and panic/recover

`defer` is also the only opportunity to catch a `panic`: only calling `recover()` inside a deferred function can intercept a panic and resume normal flow. This "exception" mechanism differs fundamentally from C++'s `try/catch` and stack unwinding—covered in detail in [09-错误处理与资源管理](09-错误处理与资源管理.md#iii.-panic-recover-not-an-exception).

## Summary

- **At the usage level**: loops only use `for` (its three forms cover while/do-while/infinite loops); `range` iterates over slices/arrays/strings/maps/channels; `if`/`switch` can carry an init statement; `switch` doesn't fall through by default, and a single case can list multiple values; `defer` postpones a call, in one line, until just before the function returns, commonly used to close resources.
- **At the underlying level**: the `range` iteration variable is a **copy**; the range expression is evaluated exactly once before the loop starts (a slice's header is copied; whether an array's entire contents get copied depends on whether you take element values); `range` over a string decodes by rune; since Go 1.22 the loop variable is a **fresh instance every iteration**, fixing the historical closure-capture pitfall; `defer` executes in **LIFO** order, its arguments are **evaluated at registration time**, and it can modify the final return value via a named return value.
