# The Internal Implementation of Composite Data Structures

This chapter is the heart of the book. slice, map, and channel are the three siblings in Go that "look like reference types." Understanding them comes down to a single sentence: **each is itself a very small, value-copied "header" struct, and the header hides a pointer to the real data on the heap.** ([Recall the value-copy discussion in "Variables and the Memory Model"](03-变量与内存模型.md))

Once you understand the header's fields, you can predict exactly what each assignment, argument pass, and `append` actually copies, whether it allocates, and whether it produces aliasing.

## I. array: a genuine value type

Let's first establish a point of comparison. Go's array `[N]T` has **fixed length and value semantics**, and the length is part of the type—`[3]int` and `[4]int` are different types.

```go
var a [3]int          // 24 bytes, entirely in the current stack frame, zero value {0,0,0}
b := a                // copies all 24 bytes byte by byte (same as C struct assignment)
b[0] = 100            // does not affect a
fmt.Println(a, b)     // [0 0 0] [100 0 0]
```

This matches C's `int a[3]; int b[3]; b = a;` (actually C doesn't allow direct array assignment, you'd need `memcpy`)—**it copies the whole block of memory**. Argument passing also copies the whole block: passing a `[1024]int` into a function copies 8KB. So almost nobody in Go passes arrays directly; they all pass slices.

```
array [3]int in memory:
┌────────┬────────┬────────┐
│ int[0] │ int[1] │ int[2] │   contiguous 24 bytes, which is the data itself
└────────┴────────┴────────┘
```

## II. slice: a three-tuple header

The slice is the "dynamic array" you use daily. Its runtime representation is `runtime.slice` (source `runtime/slice.go`):

```go
type slice struct {
    array unsafe.Pointer // points to the first element of the underlying array
    len   int            // current length
    cap   int            // capacity from array onward
}
```

Three machine words (24 bytes on 64-bit). **A slice variable itself is only these 24 bytes**, and the data is elsewhere.

```
s := []int{10, 20, 30}

slice header (on the stack, 24 bytes)      underlying array (usually on the heap)
┌───────────┐                      ┌────┬────┬────┐
│ array ────┼─────────────────────▶│ 10 │ 20 │ 30 │
│ len = 3   │                      └────┴────┴────┘
│ cap = 3   │
└───────────┘
```

### Assignment / argument passing: copy the header, not the data

```go
s := []int{10, 20, 30}
t := s          // copies the 24-byte header; t.array and s.array point to the same array!
t[0] = 999      // modifies the shared underlying array
fmt.Println(s)  // [999 20 30] — s changed too (aliasing)
```

This is exactly where C++ programmers stumble most: `t := s` looks like a `std::vector` deep copy, but it's actually more like a `std::span` / raw-pointer copy—sharing the underlying buffer. Argument passing is the same: pass a slice into a function, and the function's modifications to elements are visible to the caller (but modifications to the header itself—such as a reallocation triggered by `append`—are not visible, see below).

### Slicing from an array/slice: `s[low:high]`

The slice operation **does not allocate and does not copy data**; it only makes a new header pointing at a segment of the original array:

```go
a := [5]int{0, 1, 2, 3, 4}
s := a[1:3]     // len=2, cap=4 (from index 1 to the end of the array), array points to &a[1]
```

```
a: ┌───┬───┬───┬───┬───┐
   │ 0 │ 1 │ 2 │ 3 │ 4 │
   └───┴───┴───┴───┴───┘
         ▲
         │ s.array
   s = {array:&a[1], len:2, cap:4}   ← cap counts all the way to the end of the underlying array
```

`len` is the range you can index, `cap` is the range you can `append` into without reallocating. Indexing `s[i]` out of bounds (`i >= len`) panics; this is Go's bounds checking, unlike C's "freely read garbage out of bounds" (the compiler elides the check when it can prove safety).

### append and the growth strategy

`append` is the key to understanding slices. The logic:

1. If `len < cap`: write directly to `array[len]`, `len++`, **no allocation, no copy**, and the returned new header shares the underlying array with the old one.
2. If `len == cap`: allocate a larger new array, `memmove` the old data over, point the new header at the new array, **decoupling from the old slice**.

```go
s := make([]int, 0, 2) // len=0 cap=2
s = append(s, 1)       // len=1, written in place, no allocation
s = append(s, 2)       // len=2, written in place, cap full
s = append(s, 3)       // len=3, cap insufficient → allocate new array (usually cap becomes 4), copy
```

**Growth multiplier (the gc implementation, `runtime.growslice`, tweaked slightly across versions, defer to the runtime)**:

| Old cap | New cap growth strategy |
|--------|----------------|
| Requirement exceeds 2× old cap | Use the required value directly |
| Old cap < 256 | Roughly double (2×) |
| Old cap ≥ 256 | Roughly 1.25×, smoothly approaching 1.25 (to avoid wasting memory on large slices) |

Afterward it also aligns memory by element size and rounds up to a size class (see [Chapter 09, mspan/size class](09-错误处理与资源管理.md)). So the printed `cap` may not be exactly double.

> Rule of thumb: if you can estimate the size, use `make([]T, 0, n)` to reserve capacity and avoid repeated allocation + copying—equivalent to C++'s `vector::reserve`.

### The append aliasing trap (the most classic pitfall)

Because "if cap suffices, write in place; only reallocate when it doesn't," the same underlying array may be shared by multiple slices, and `append` may silently modify someone else's data:

```go
a := []int{1, 2, 3, 4, 5}
s := a[0:2]              // len=2 cap=5, shares the underlying array with a
s = append(s, 99)        // cap is sufficient → write a[2] in place!
fmt.Println(a)           // [1 2 99 4 5] — a was modified!
```

```
before append:            after append(s, 99) (cap sufficient, in place):
a = [1 2 3 4 5]            a = [1 2 99 4 5]   ← the 3rd element is overwritten
s = a[0:2] ────┐          s = {len:3, cap:5}
               │              s[2] wrote into a[2]'s position
```

Conversely, if reallocation happened, the two decouple and subsequent modifications no longer affect each other—**the behavior of the same piece of code depends on whether cap is sufficient at runtime**, which is a breeding ground for bugs. Defenses:

- Use a **three-index slice** to cap the capacity: `s := a[0:2:2]` (`s[low:high:max]`, `cap = max-low`). Here cap==len, so the next append must allocate a new array and won't touch `a`.
- For a truly independent copy, use `copy`:

```go
dst := make([]int, len(src))
copy(dst, src)   // element-by-element copy, returns the number copied min(len(dst),len(src))
```

`copy` is `memmove` underneath and allows the source and destination to overlap.

## III. map: a hash table

> **Version note**: The `hmap`/`bmap` described below (buckets + overflow buckets, load factor 6.5, incremental `evacuate` migration) is the **classic implementation of Go ≤ 1.23**. **Since Go 1.24 the built-in map switched to a Swiss Tables design** (derived from Abseil's open addressing + grouped control words/SIMD probing, `internal/runtime/maps`), so the internal layout and details like "overflow buckets" and "6.5 load factor" have changed and it's faster on average; but **the language-level semantics (unordered, not concurrency-safe, `&m[k]` not addressable, etc.) are entirely unchanged**. We still cover the classic implementation here because it better illustrates the general principle of "hash table + buckets/chaining"—just read it as "one" implementation.

The runtime type of a map is `*hmap` (`runtime/map.go`, Go ≤ 1.23)—note that **a map variable is itself a pointer** (one word), which is one difference from a slice header.

```go
type hmap struct {
    count     int    // number of elements, len(m) reads this directly O(1)
    flags     uint8
    B         uint8  // number of buckets = 2^B
    noverflow uint16
    hash0     uint32 // hash seed (random per map, to defend against hash-flooding attacks)
    buckets   unsafe.Pointer // points to the array of 2^B bmap buckets
    oldbuckets unsafe.Pointer // the old bucket array during growth
    nevacuate uintptr
    extra     *mapextra
}
```

Bucket `bmap`: each bucket has a fixed **8 slots**.

```go
type bmap struct {
    tophash [8]uint8 // the high 8 bits of each key's hash value (for fast comparison)
    // the compiler appends afterward: keys[8], values[8], overflow *bmap
}
```

The memory layout is `8 tophash │ 8 keys │ 8 values │ overflow-bucket pointer`. Note that **keys and values are stored contiguously in separate groups** (not interleaved as kv/kv/kv), which avoids padding waste caused by alignment (e.g., interleaving `map[int8]int64` would waste a lot).

```
hmap ──▶ buckets: [ bucket0 | bucket1 | ... | bucket(2^B-1) ]
                       │
                       ▼
  one bmap bucket:
  ┌──────────────────────┐
  │ tophash[0..7]        │  high 8 bits of each slot's hash (also used as empty/evacuated markers)
  ├──────────────────────┤
  │ key0 key1 ... key7   │  8 keys contiguous
  ├──────────────────────┤
  │ val0 val1 ... val7   │  8 values contiguous
  ├──────────────────────┤
  │ overflow ────────────┼──▶ next overflow bmap (chained)
  └──────────────────────┘
```

### The lookup process

`v := m[k]`:

1. `hash = hasher(k, hash0)`.
2. Select the bucket by the low B bits: `bucket = hash & (2^B - 1)`.
3. Use the high 8 bits `tophash` to quickly filter across the bucket's 8 slots (compare just 1 byte, skipping most non-matches directly), and on a hit do a full key comparison.
4. If not in the bucket, follow the `overflow` chain to the overflow bucket.

Average O(1), worst case (many hash collisions) degrades.

### Incremental growth (rehash)

When the load factor gets too high (element count / bucket count > 6.5) or there are too many overflow buckets, growth is triggered: allocate a 2× new bucket array and move `buckets` to `oldbuckets`. **Key point: it does not migrate everything at once**, but on each subsequent write (`insert`/`delete`) it opportunistically migrates 1~2 old buckets (`evacuate`), amortizing away the pause of a one-shot rehash on a large map. Lookups consult both the new and old buckets. This differs from the one-shot rehash of C++'s `std::unordered_map`.

### A few hard constraints on maps (contrary to C/C++ intuition)

**1. Iteration order is random, deliberately.** Each `for k, v := range m` randomizes the starting bucket and starting slot. This is to prevent programs from depending on order (which would break when the implementation changes). To get order you must sort yourself:

```go
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys) // see Chapter 11
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

**2. Elements are not addressable.** `&m[k]` is a compile error. Because growth relocates the value's memory, the pointer would become invalid. So you cannot modify a struct field in place:

```go
m := map[string]Point{"a": {1, 2}}
// m["a"].X = 10        // compile error: cannot assign to struct field
p := m["a"]; p.X = 10; m["a"] = p   // read out → modify → write back
// or store a pointer: map[string]*Point, then m["a"].X = 10 works
```

**3. Reading a missing key returns the zero value + ok.** The comma-ok idiom:

```go
v, ok := m[k] // ok is false means it doesn't exist, v is the zero value
```

**4. Not concurrency-safe.** A concurrent read-write (even one read and one write) is detected by the runtime and crashes directly with `fatal error: concurrent map read and map write` (not a panic, not recoverable). You must add locking yourself, or use `sync.Map` (for read-heavy, write-light scenarios, see [Chapter 10](10-并发模型.md)).

**5. A nil map is readable but not writable.** `var m map[string]int` is nil; reading returns the zero value, writing panics. You must initialize with `make` or a literal.

## IV. channel: a locked ring buffer

The channel is Go's core concurrency primitive, backed by `*hchan` (`runtime/chan.go`); a channel variable is also a pointer.

```go
type hchan struct {
    qcount   uint           // current number of elements in the buffer
    dataqsiz uint           // buffer capacity (the second argument to make)
    buf      unsafe.Pointer // points to the ring buffer array (when buffered)
    elemsize uint16
    closed   uint32
    sendx    uint           // write index of the ring buffer
    recvx    uint           // read index of the ring buffer
    recvq    waitq          // queue of goroutines waiting to receive
    sendq    waitq          // queue of goroutines waiting to send
    lock     mutex          // protects all the fields above
}
```

```
buffered channel  make(chan int, 4):
        ┌─────────────────────────────┐
buf ──▶ │  slot0 slot1 slot2 slot3    │  ring buffer
        └─────────────────────────────┘
          ▲recvx        ▲sendx
  lock  ─ a single mutex, contended on every send/receive
  sendq ─ [G3, G7 ...]  goroutines blocked on send
  recvq ─ [ ]           goroutines blocked on receive
```

### The blocking semantics of send/receive

**Unbuffered channel (`make(chan int)`, dataqsiz=0)**: send and receive must "rendezvous." The sender blocks until a receiver arrives, and the data is **copied directly from the sender goroutine to the receiver goroutine** (not through a buffer), after which both are woken. This is the strongest synchronization: the send/receive is a synchronization point.

**Buffered channel (`make(chan int, n)`)**:

| Operation | Buffer not full / not empty | Buffer full (send) / empty (receive) |
|------|--------------|-------------------|
| Send `ch <- v` | Copy into buf[sendx], sendx++, return immediately | The current goroutine enters sendq, blocks and is suspended |
| Receive `<-ch` | Copy out of buf[recvx], recvx++, return immediately | The current goroutine enters recvq, blocks and is suspended |

"Blocks and is suspended" is not busy-waiting: the goroutine is removed by the scheduler, and the M goes off to run another G (see [Chapter 10, GMP](10-并发模型.md)). When the peer operation happens, a goroutine in the wait queue is woken. This is the foundation of goroutines' cheap concurrency—blocking a goroutine does not block an OS thread.

One optimization: if, when sending, there is already a waiting receiver in recvq, the data **skips the buffer and is copied directly to that receiver** and wakes it, saving one copy.

### close and nil

```go
close(ch)          // close; sending afterward panics; receiving returns the zero value immediately
v, ok := <-ch      // ok==false means the channel is closed and the buffer is empty
for v := range ch  // keep receiving until the channel is closed
```

Sending to or receiving from a nil channel **blocks forever** (commonly used to dynamically "turn off a branch" in a `select`).

A channel transfers **a copy of the element** (value semantics): `ch <- x` copies `x` into the buffer, `<-ch` copies it out. If the element is a large struct, the whole block is copied; passing a pointer copies the pointer (in which case mind the concurrency safety of the shared data).

## V. A comparison of the three

| Type | Size of the variable itself | What assignment/passing copies | Where the data lives | Concurrency-safe |
|------|-------------|------------------|---------|---------|
| `[N]T` array | N×sizeof(T) | The whole block of data | With the variable (stack or heap) | Value semantics, naturally isolated |
| `[]T` slice | 3 words (24B) | The header (with pointer) | The underlying array is usually on the heap | No, shares the underlying array |
| `map` | 1 word (pointer) | The pointer | hmap + buckets on the heap | No, concurrent read/write crashes |
| `chan` | 1 word (pointer) | The pointer | hchan + buf on the heap | Yes, internally locked |

In one sentence: **except for array, which is a true value type, slice/map/chan all copy a "lightweight header/pointer" and share the data behind it**—this is exactly why they "seem like references," yet mechanically they are still thoroughly value copies.
