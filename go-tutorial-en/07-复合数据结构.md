# The Internal Implementation of Composite Data Structures

slice, map, and channel are the three siblings in Go that "look like reference types": assign one to another variable, or pass it into a function, and modifications will affect each other — behavior that looks a lot like a C++ reference or pointer. But at heart they are still **thoroughly value types** — each of them is just a small, fixed-size "header" struct; the header itself is copied by value, and a pointer field inside the header points at the real data on the heap ([recall the value-copy discussion in "Variables and the Memory Model"](03-变量与内存模型.md)). Once you understand the fields of each header, you can predict exactly what every assignment, argument pass, and `append` actually copies, whether it allocates, and whether it produces aliasing — that is the core goal of this chapter.

## I. array: a genuine value type

An array is declared with `[N]T`; `N` is a compile-time constant and is part of the type — `[3]int` and `[4]int` are different types:

```go
var a [3]int          // 24 bytes, entirely in the current stack frame, zero value {0,0,0}
b := a                // copies all 24 bytes byte by byte (same as C struct assignment)
b[0] = 100            // does not affect a
fmt.Println(a, b)     // [0 0 0] [100 0 0]
```

This matches the behavior of C's `int a[3]; int b[3]; b = a;` (C actually doesn't allow direct array assignment; you'd need `memcpy`) — **it copies the whole block of memory**. Argument passing also copies the whole block: passing a `[1024]int` into a function copies 8KB. The array is one of the few types in Go where "what you see is what you get" — no hidden pointers at all — which is exactly why it works as a slice's underlying storage; but precisely because the cost of passing it grows linearly with length, almost nobody in everyday code passes an array directly — everyone passes a slice instead.

```
array [3]int in memory:
┌────────┬────────┬────────┐
│ int[0] │ int[1] │ int[2] │   contiguous 24 bytes, which is the data itself
└────────┴────────┴────────┘
```

## II. slice: a three-tuple header

The slice is the "dynamic array" you use every day. There are three ways to create one: a literal, `make`, or slicing out of an array/slice:

```go
s1 := []int{10, 20, 30}   // literal
s2 := make([]int, 3)      // make(T, len): length 3, elements all zero value [0 0 0]
s3 := make([]int, 0, 5)   // make(T, len, cap): length 0, capacity reserved for 5

s1 = append(s1, 40)       // append an element
fmt.Println(s1)           // [10 20 30 40]
fmt.Println(s1[1])        // 20  -- read by index
s1[1] = 99                // write by index
fmt.Println(s1)           // [10 99 30 40]

sub := s1[1:3]            // slicing: take index 1 (inclusive) through 3 (exclusive)
fmt.Println(sub)          // [99 30]
```

The slice's runtime representation is `runtime.slice` (source `runtime/slice.go`):

```go
type slice struct {
    array unsafe.Pointer // points to the first element of the underlying array
    len   int            // current length
    cap   int            // capacity from array onward
}
```

Three machine words (24 bytes on 64-bit). **A slice variable itself is only these 24 bytes** — the data lives elsewhere:

```
s := []int{10, 20, 30}

slice header (on the stack, 24 bytes)      underlying array (usually on the heap)
┌───────────┐                      ┌────┬────┬────┐
│ array ────┼─────────────────────▶│ 10 │ 20 │ 30 │
│ len = 3   │                      └────┴────┴────┘
│ cap = 3   │
└───────────┘
```

### Assignment / argument passing: copy the header, not the data

```go
s := []int{10, 20, 30}
t := s          // copies the 24-byte header; t.array and s.array point to the same array!
t[0] = 999      // modifies the shared underlying array
fmt.Println(s)  // [999 20 30] -- s changed too (aliasing)
```

This is exactly where C++ programmers stumble most: `t := s` looks like a `std::vector` deep copy, but it's actually more like copying a `std::span` or a raw pointer — sharing the underlying buffer. Argument passing works the same way: pass a slice into a function, and the function's modifications to elements are visible to the caller (but modifications to the header itself — such as a reallocation triggered by `append` — are not, see below). **The design trade-off here**: if slices had deep-copy semantics, every assignment or argument pass would have to copy linearly in the number of elements, and passing a large slice would become as expensive as passing a large array. Go instead chose "shallow-copy the header, share the data," buying O(1) cost for assignment and argument-passing, at the price that programmers have to understand the sharing relationship themselves and can end up writing aliasing side effects like the one above.

### Slicing an array/slice: `s[low:high]`

Slicing **does not allocate and does not copy data** — it only creates a new header pointing at a segment of the original array:

```go
a := [5]int{0, 1, 2, 3, 4}
s := a[1:3]     // len=2, cap=4 (from index 1 to the end of the array), array points to &a[1]
```

```
a: ┌───┬───┬───┬───┬───┐
   │ 0 │ 1 │ 2 │ 3 │ 4 │
   └───┴───┴───┴───┴───┘
         ▲
         │ s.array
   s = {array:&a[1], len:2, cap:4}   ← cap counts all the way to the end of the underlying array
```

`len` is the range you can index into; `cap` is the range you can `append` into without reallocating. Indexing `s[i]` out of bounds (`i >= len`) panics — this is Go's bounds checking, unlike C's "freely read garbage past the end" (the compiler elides the check when it can prove safety).

### append and the growth strategy

`append` is the key to understanding slices. The logic:

1. If `len < cap`: write directly to `array[len]`, `len++`, **no allocation, no copy** — the returned new header shares the underlying array with the old one.
2. If `len == cap`: allocate a larger new array, `memmove` the old data over, point the new header at the new array — **decoupled from the old slice**.

```go
s := make([]int, 0, 2) // len=0 cap=2
s = append(s, 1)       // len=1, written in place, no allocation
s = append(s, 2)       // len=2, written in place, cap full
s = append(s, 3)       // len=3, cap insufficient → allocate new array (usually cap becomes 4), copy
```

**Why the growth curve looks like this (the gc implementation, `runtime.growslice`, tweaked slightly across versions — defer to the runtime for the exact numbers)**: a pure fixed multiplier (say, always doubling) wastes a huge amount of memory once a slice gets large — a slice with hundreds of thousands of elements would over-allocate hundreds of thousands of redundant elements' worth of space on every growth spurt; a pure fixed increment (always +K) is too slow when the slice is small — it degenerates into O(n) reallocations. Go uses a piecewise strategy — "double aggressively while small, slow down while large" — to strike a compromise between the two:

| Old cap | New cap growth strategy |
|--------|----------------|
| Requirement exceeds 2× old cap | Use the required value directly |
| Old cap < 256 | Roughly double (2×) |
| Old cap ≥ 256 | Roughly 1.25×, smoothing gradually toward 1.25 (to avoid wasting memory on large slices) |

Afterward it also aligns by element size and rounds up to a size class (see [Chapter 09, mspan/size class](09-错误处理与资源管理.md)). So the printed `cap` isn't necessarily an exact doubling.

> Rule of thumb: if you can estimate the size, use `make([]T, 0, n)` to reserve capacity up front and avoid repeated allocation-plus-copy — the equivalent of C++'s `vector::reserve`.

### The append aliasing trap (the classic pitfall)

Because "write in place if cap allows, only allocate if it doesn't," the same underlying array can end up shared by multiple slices, and `append` can silently modify someone else's data:

```go
a := []int{1, 2, 3, 4, 5}
s := a[0:2]              // len=2 cap=5, shares the underlying array with a
s = append(s, 99)        // cap is sufficient → write a[2] in place!
fmt.Println(a)           // [1 2 99 4 5] -- a was modified!
```

```
before append:            after append(s, 99) (cap sufficient, in place):
a = [1 2 3 4 5]            a = [1 2 99 4 5]   ← the 3rd element is overwritten
s = a[0:2] ────┐          s = {len:3, cap:5}
               │              s[2] wrote into a[2]'s position
```

Conversely, if a reallocation happens, the two decouple, and subsequent modifications no longer affect each other — **whether the same piece of code behaves one way or the other depends on whether cap happens to be enough at runtime**, which is a breeding ground for bugs, and the root cause is still the same trade-off as above: shallow-copying the header buys performance, but exactly when two slices share data and when they decouple is a hidden runtime decision that the type system gives you no way to see. Defenses:

- Use a **three-index slice** to cap the capacity: `s := a[0:2:2]` (`s[low:high:max]`, `cap = max-low`). Here cap==len, so the next `append` is guaranteed to allocate a new array and can never touch `a`.
- For a genuinely independent copy, use `copy`:

```go
dst := make([]int, len(src))
copy(dst, src)   // element-by-element copy, returns the number copied min(len(dst),len(src))
```

`copy` is `memmove` underneath, and it allows the source and destination to overlap.

### One runnable program tying the whole flow together

In-place aliasing, three-index capping, and decoupling after a reallocation — all in one program:

```go
package main

import "fmt"

func main() {
	a := []int{1, 2, 3, 4, 5}
	s := a[0:2]                 // len=2 cap=5, shares a's underlying array
	fmt.Println(len(s), cap(s)) // 2 5
	s = append(s, 99)           // cap is enough → writes a[2] in place
	fmt.Println(a)              // [1 2 99 4 5]  ← a was modified through s

	b := []int{1, 2, 3, 4, 5}
	t := b[0:2:2]               // three-index slice: len=2 cap=2
	fmt.Println(len(t), cap(t)) // 2 2
	t = append(t, 99)           // cap full → new array, b untouched
	fmt.Println(b)              // [1 2 3 4 5]
	fmt.Println(t)              // [1 2 99]

	c := []int{1, 2, 3}
	d := c[0:1]                 // len=1 cap=3, still shares c
	d = append(d, 20)           // cap enough → writes c[1]
	d = append(d, 30)           // cap enough → writes c[2]
	fmt.Println(c)              // [1 20 30]  ← still coupled
	d = append(d, 40)           // len==cap → reallocate, now decoupled from c
	d[0] = 999
	fmt.Println(c)              // [1 20 30]       ← c no longer affected
	fmt.Println(d)              // [999 20 30 40]
}
```

## III. map: a hash table

A map is a collection of key-value pairs, declared with `map[K]V` and created with `make` or a literal:

```go
m := make(map[string]int)     // empty map
m["a"] = 1                    // write
m["b"] = 2
fmt.Println(m["a"])           // 1     -- read

v, ok := m["c"]               // comma-ok: check whether a key exists
fmt.Println(v, ok)            // 0 false -- c doesn't exist, v is the zero value

delete(m, "a")                // delete key "a"

for k, v := range m {         // iterate (order is not fixed, see below)
	fmt.Println(k, v)
}
```

> **Version note**: the `hmap`/`bmap` design described below (buckets + overflow buckets, load factor 6.5, incremental `evacuate` migration) is the **classic implementation used through Go 1.23**. **Starting with Go 1.24, the built-in map switched to a Swiss Tables design** (derived from Abseil's open-addressing scheme with grouped control bytes / SIMD probing, `internal/runtime/maps`) — the internal layout and details like "overflow buckets" and "load factor 6.5" have all changed, and it's faster on average, but **the language-level semantics (unordered iteration, not concurrency-safe, `&m[k]` not addressable, and so on) are entirely unchanged**. We still cover the classic implementation here because it better illustrates the general principle of "hash table + buckets/chaining" — just read it as "one" implementation among others. This very replacement is itself proof of a design principle discussed below: the language spec never promises an iteration order or an internal layout, which is exactly what lets an implementation be rewritten wholesale without breaking any user code.

The runtime type of a map is `*hmap` (`runtime/map.go`, Go ≤ 1.23) — note that **a map variable is itself a pointer** (one word), which is one difference from a slice header.

```go
type hmap struct {
    count     int    // number of elements, len(m) reads this directly O(1)
    flags     uint8
    B         uint8  // number of buckets = 2^B
    noverflow uint16
    hash0     uint32 // hash seed (random per map, to defend against hash-flooding attacks)
    buckets   unsafe.Pointer // points to the array of 2^B bmap buckets
    oldbuckets unsafe.Pointer // the old bucket array during growth
    nevacuate uintptr
    extra     *mapextra
}
```

Bucket `bmap`: each bucket has a fixed **8 slots**.

```go
type bmap struct {
    tophash [8]uint8 // the high 8 bits of each key's hash value (for fast comparison)
    // the compiler appends afterward: keys[8], values[8], overflow *bmap
}
```

The memory layout is `8 tophash │ 8 keys │ 8 values │ overflow-bucket pointer`. Note that **keys and values are each stored contiguously in their own group** (not interleaved as kv/kv/kv), which avoids the padding waste alignment would otherwise cause (for example, interleaving `map[int8]int64` would waste a lot of space).

```
hmap ──▶ buckets: [ bucket0 | bucket1 | ... | bucket(2^B-1) ]
                       │
                       ▼
  one bmap bucket:
  ┌──────────────────────┐
  │ tophash[0..7]        │  high 8 bits of each slot's hash (also used as empty/evacuated markers)
  ├──────────────────────┤
  │ key0 key1 ... key7   │  8 keys contiguous
  ├──────────────────────┤
  │ val0 val1 ... val7   │  8 values contiguous
  ├──────────────────────┤
  │ overflow ────────────┼──▶ next overflow bmap (chained)
  └──────────────────────┘
```

`v := m[k]` looks it up like this:

1. `hash = hasher(k, hash0)`.
2. Select the bucket using the low B bits: `bucket = hash & (2^B - 1)`.
3. Use the high 8 bits, `tophash`, to quickly filter across the bucket's 8 slots (comparing just 1 byte, so most non-matches are skipped immediately), and on a hit do a full key comparison.
4. If it's not in the bucket, follow the `overflow` chain to the overflow bucket.

Average O(1); worst case (heavy hash collisions) degrades.

When the load factor gets too high (element count / bucket count > 6.5) or there are too many overflow buckets, **incremental growth (rehash)** is triggered: a new bucket array at 2× the size is allocated, and `buckets` is moved to `oldbuckets`. **The key point: it doesn't migrate everything all at once** — instead, on each subsequent write operation (`insert`/`delete`) it opportunistically migrates 1 or 2 old buckets (`evacuate`), amortizing away the pause that a one-shot rehash of a large map would otherwise cause. Lookups check both the new and old buckets while this is underway. This differs from C++'s `std::unordered_map`, which rehashes in one shot — a one-shot rehash makes some particular insert suddenly become very slow (every element has to be rebucketed at once), whereas amortized migration spreads that cost across a number of subsequent operations, avoiding a latency spike on any single operation, which matters a great deal for latency-sensitive services.

### A few hard constraints on maps (contrary to C/C++ intuition)

**1. Iteration order is random — deliberately, and for two overlapping reasons.** Every `for k, v := range m` randomizes its starting bucket and starting slot. There are actually two separate considerations stacked here, worth pulling apart:

- **Preventing programs from depending on implementation details they have no business depending on.** A hash table's natural traversal order (following the physical layout of buckets and slots) is purely a byproduct of the implementation and has nothing to do with insertion order or any other meaningful semantics. If Go let iteration order "happen" to look reasonably stable, the way early implementations did, programmers would unconsciously write code that depends on that order (for example, assuming whatever was inserted first shows up first) — code like that runs fine at the time, but the moment the runtime switches implementations (bucket count changes, growth kicks in, or even a wholesale swap to Swiss Tables as in Go 1.24), the order changes and the program breaks silently. The Go team deliberately randomized the starting point from early in the language's life, so that "you can't depend on the order" is visible from day one, rather than lying dormant until an implementation change detonates it. This also explains the sentence in the version note above: precisely because the semantics never promised an order, Go 1.24 was free to swap the entire hash-table implementation for Swiss Tables without breaking any existing code that follows the language spec.
- **Reducing the payoff of hash-flooding attacks.** `hash0` gives every map instance's hash function a different random seed, and that's the main line of defense — even an attacker who knows the hashing algorithm can't precompute offline a set of "colliding keys" that all land in the same bucket to mount a denial-of-service attack (otherwise an O(1) lookup could be deliberately degraded to O(n) by an attacker). Randomizing iteration order is a natural extension of this defense at the level of observable behavior: if the order were deterministic, an observer could more easily infer internal state just by watching the iteration results.

To get a stable order you have to sort it yourself:

```go
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys) // see Chapter 11
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

**2. Elements are not addressable.** `&m[k]` is a compile error. Because growth relocates a value's memory, a pointer into it would go stale. So you can't modify a struct field in place:

```go
m := map[string]Point{"a": {1, 2}}
// m["a"].X = 10        // compile error: cannot assign to struct field
p := m["a"]; p.X = 10; m["a"] = p   // read out → modify → write back
// or store a pointer: map[string]*Point, then m["a"].X = 10 works
```

Runnable (the commented-out line would be a compile error):

```go
package main

import "fmt"

type Point struct{ X, Y int }

func main() {
	m := map[string]Point{"a": {1, 2}}
	// m["a"].X = 10 // compile error: cannot assign to struct field m["a"].X
	p := m["a"]         // read out
	p.X = 10            // modify the copy
	m["a"] = p          // write back
	fmt.Println(m["a"]) // {10 2}

	pm := map[string]*Point{"a": {1, 2}} // store pointers instead
	pm["a"].X = 10                       // in-place works: the value is a pointer
	fmt.Println(*pm["a"])                // {10 2}

	v, ok := m["missing"] // reading a missing key
	fmt.Println(v, ok)    // {0 0} false
}
```

**3. Reading a missing key returns the zero value plus ok.** The comma-ok idiom:

```go
v, ok := m[k] // ok is false means it doesn't exist, v is the zero value
```

**4. Not concurrency-safe.** A concurrent read and write (even just one read and one write) is detected by the runtime and crashes the program outright with `fatal error: concurrent map read and map write` (not a panic — not recoverable). This isn't the runtime being lazy about adding synchronization; it's a deliberate trade-off: building a lock into every map would make every single-goroutine use case (the overwhelming majority) pay for synchronization it never needs. The runtime instead keeps maps lock-free on the fast path and hands the responsibility for concurrency safety back to the caller — add your own locking if you need it, or use `sync.Map` (for read-heavy, write-light scenarios, see [Chapter 10](10-并发模型.md)). Crashing with `fatal error` the instant a conflict is detected, rather than silently producing garbage data or undefined behavior, surfaces this class of bug as early as possible instead of letting it linger into production as mysterious, intermittent corruption.

**5. A nil map is readable but not writable.** `var m map[string]int` is nil; reading returns the zero value, writing panics. You must initialize it with `make` or a literal.

## IV. channel: a concrete implementation of the CSP model

A channel is a pipe for passing data between goroutines, created with `make(chan T)`, with `<-` for send/receive and `close` to close it:

```go
ch := make(chan int, 2)  // buffered channel with capacity 2
ch <- 1                  // send
ch <- 2
fmt.Println(<-ch)        // 1  -- receive (first in, first out)
fmt.Println(<-ch)        // 2

close(ch)                // close the channel, meaning no more sends will happen
v, ok := <-ch            // closed and fully drained: v is the zero value, ok=false
fmt.Println(v, ok)       // 0 false
```

`for v := range ch` keeps receiving until the channel is closed and everything in it has been drained (see [Chapter 10](10-并发模型.md) for the concurrent picture).

**Why channel is a first-class citizen, rather than "shared memory + a lock."** The default way to communicate across threads in C/C++ is: open up a block of shared memory and guard it with a lock (or atomics). This model requires every site that touches the shared data to remember to take the lock, and correctness depends on every single access point honoring the same implicit convention — miss one lock, and the data can get torn. The channel comes from Hoare's CSP (Communicating Sequential Processes) theory, and Go built it in as a language-level primitive, on the back of an oft-quoted design philosophy:

> **Don't communicate by sharing memory; share memory by communicating.**

Concretely: when you pass data through a channel, what's being transferred is not just the data itself but its **ownership** — once the sender hands a value to the channel, the convention is that it no longer touches it, and the receiver has it exclusively from then on. That way, at any given moment only one goroutine is reading or writing a given piece of data, and you no longer need every access point to remember to take a lock — synchronization correctness moves from "remember to lock at every access site" to "the data structure itself only ever has one path of flow." This doesn't mean Go has no shared memory and locks — `sync.Mutex` and `sync/atomic` are both still there (see [Chapter 10](10-并发模型.md)), for cases where you genuinely just need to protect a small critical section and performance is sensitive; what channels solve is a higher-level problem: structuring a concurrent program as a clear pipeline of data flow, rather than a pile of mutually-locking shared state. The cost is real too: sending and receiving on a channel involves a lock and interaction with the scheduler, so a single operation costs far more than a single atomic increment — protecting a simple counter is still better served by `atomic` or `Mutex`; a channel earns its keep by transferring ownership of data and constructing synchronization points, not by replacing every lock.

The channel's underlying structure is `*hchan` (`runtime/chan.go`); a channel variable is also just a pointer:

```go
type hchan struct {
    qcount   uint           // current number of elements in the buffer
    dataqsiz uint           // buffer capacity (the second argument to make)
    buf      unsafe.Pointer // points to the ring buffer array (when buffered)
    elemsize uint16
    closed   uint32
    sendx    uint           // write index of the ring buffer
    recvx    uint           // read index of the ring buffer
    recvq    waitq          // queue of goroutines waiting to receive
    sendq    waitq          // queue of goroutines waiting to send
    lock     mutex          // protects all the fields above
}
```

```
buffered channel  make(chan int, 4):
        ┌─────────────────────────────┐
buf ──▶ │  slot0 slot1 slot2 slot3    │  ring buffer
        └─────────────────────────────┘
          ▲recvx        ▲sendx
  lock  ─ a single mutex, contended on every send/receive
  sendq ─ [G3, G7 ...]  goroutines blocked on send
  recvq ─ [ ]           goroutines blocked on receive
```

**Unbuffered channel (`make(chan int)`, dataqsiz=0)**: send and receive must "rendezvous." The sender blocks until a receiver arrives, and the data is **copied directly from the sender goroutine to the receiver goroutine** (never passing through a buffer), after which both are woken. This isn't some behavior picked incidentally at the implementation level — it's the direct definition of what an unbuffered channel's semantics mean: it provides no storage at all, and the very definition of "the send has succeeded" is "the receiver has already gotten the value" — so the sender and receiver are necessarily coincident in time, which is exactly what "rendezvous" means, and it is the strongest synchronization guarantee there is: a send happens-before the corresponding receive completes ([Chapter 10](10-并发模型.md) expands on this happens-before rule).

**Buffered channel (`make(chan int, n)`)** relaxes this constraint, using a ring buffer to decouple the sender from the receiver:

| Operation | Buffer not full / not empty | Buffer full (send) / empty (receive) |
|------|--------------|-------------------|
| Send `ch <- v` | Copy into buf[sendx], sendx++, return immediately | The current goroutine enters sendq, blocks and is suspended |
| Receive `<-ch` | Copy out of buf[recvx], recvx++, return immediately | The current goroutine enters recvq, blocks and is suspended |

"Blocks and is suspended" is not busy-waiting: the goroutine is pulled off by the scheduler, and the M goes off to run some other G (see [Chapter 10, GMP](10-并发模型.md)). When the peer's operation happens, a goroutine sitting in the wait queue gets woken up. This is the very foundation of goroutines' cheap concurrency — blocking a goroutine doesn't block an OS thread.

One optimization: if there's already a waiting receiver sitting in recvq at send time, the data **skips the buffer entirely and is copied straight to that receiver**, which is then woken — saving one copy.

```go
close(ch)          // close; sending afterward panics; receiving returns the zero value immediately
v, ok := <-ch      // ok==false means the channel is closed and the buffer is empty
for v := range ch  // keep receiving until the channel is closed
```

Sending to or receiving from a nil channel **blocks forever** (commonly used to dynamically "switch off" a branch inside a `select`).

A channel transfers **a copy of the element** (value semantics): `ch <- x` copies `x` into the buffer, and `<-ch` copies it back out. If the element is a large struct, the whole thing gets copied; if you pass a pointer, only the pointer gets copied (in which case you need to mind the concurrency safety of the shared data it points to).

### Buffered send/receive, close, and range (runnable)

```go
package main

import "fmt"

func main() {
	ch := make(chan int, 2)
	ch <- 1
	ch <- 2
	close(ch)           // close after sending; buffered values remain receivable
	for v := range ch { // drains the buffer in FIFO order, then stops at close
		fmt.Println(v) // prints 1, then 2
	}
	v, ok := <-ch      // closed and drained → zero value, ok=false
	fmt.Println(v, ok) // 0 false
}
```

## V. Comparing the three

| Type | Size of the variable itself | What assignment/passing copies | Where the data lives | Concurrency-safe |
|------|-------------|------------------|---------|---------|
| `[N]T` array | N×sizeof(T) | The whole block of data | With the variable (stack or heap) | Value semantics, naturally isolated |
| `[]T` slice | 3 words (24B) | The header (with pointer) | The underlying array, usually on the heap | No, shares the underlying array |
| `map` | 1 word (pointer) | The pointer | hmap + buckets on the heap | No, concurrent read/write crashes |
| `chan` | 1 word (pointer) | The pointer | hchan + buf on the heap | Yes, internally locked |

In one sentence: **except for array, which is a genuine value type, slice/map/chan all copy a "lightweight header/pointer" and share the data behind it** — that's exactly why they "feel like references," even though mechanically they remain thoroughly value copies.

## Summary

- **At the usage level**: the array `[N]T` has a fixed length and is read/written by index; the slice `[]T` grows with `append`, is inspected with `len`/`cap`, and is cut with `s[low:high]`; the map `map[K]V` is written with `m[k]=v`, read (and checked for existence) with `v, ok := m[k]`, and pruned with `delete(m, k)`; the channel `chan T` sends with `ch <- v`, receives with `<-ch`, closes with `close(ch)`, and is drained with `for range ch`.
- **At the underlying/design-trade-off level**: a slice is a 24-byte three-tuple header (array/len/cap); shallow-copying the header buys O(1) cost for assignment and argument-passing, and `append` writes in place when cap allows or allocates a new array and decouples from the old slice when it doesn't — the root of the aliasing pitfall. A map (Go ≤ 1.23) is an `hmap` plus 8-slot-per-bucket `bmap`s, using tophash to speed up lookups, and growing incrementally once the load factor passes 6.5 in order to amortize the pause; its iteration order is deliberately randomized, for two reasons — first, so that programs can't depend on implementation details (which is exactly what let Go 1.24 switch to Swiss Tables without breaking existing code), and second, paired with a randomized hash seed, to defend against hash-flooding attacks; elements being non-addressable is a direct consequence of growth relocating memory; and crashing outright on concurrent read/write is the trade-off of leaving synchronization responsibility to the caller in order to keep the lock-free fast path. A channel is a locked ring buffer, `hchan`, a concrete implementation of CSP's "share memory by communicating" philosophy, replacing locks-everywhere with a transfer of ownership; the fact that unbuffered send/receive is a direct "rendezvous" is a necessary consequence of its synchronization semantics, and blocking means suspending the goroutine rather than busy-waiting. For all three, the variable itself is either a header (slice) or a pointer (map/chan) — the real data always lives on the heap.
