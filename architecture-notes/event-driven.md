# 事件驱动架构 · Event-Driven Architecture

> **事件** = 状态变化时发出的通知。事件驱动架构用事件通信，处理器高度解耦，天然偏向**异步、可扩展**。

---

## 一、概念

事件驱动架构（event-driven architecture, EDA）以「发生了什么」为中心组织系统，而不是以「谁同步调用谁」为中心。

典型场景：下单成功 → 扣库存、发邮件、记积分、写审计。若全部同步串在一次 HTTP 里，耦合与延迟都会爆炸；改为发出 `OrderPlaced` 事件，各处理器各自消费。

---

## 二、四部分结构

把这套流程想象成一个快递分拣站：包裹（事件）先进**分拣口**（队列），分拣员（分发器）看包裹上的地址决定送去哪条**传送带**（通道），最后由对应的**处理站**（处理器）完成派送；处理站处理完，可能又产生新包裹（新事件）继续往下走。

经典切分为四部分：

| 部件 | 英文 | 职责 |
|------|------|------|
| 事件队列 | Event Queue | 接收事件的入口 |
| 分发器 | Event Mediator | 按类型把事件分到不同逻辑单元 |
| 事件通道 | Event Channel | 分发器与处理器之间的管道 |
| 事件处理器 | Event Processor | 实现业务逻辑；完成后可再发事件 |

```
  Producer
     │  event
     ▼
┌─────────────┐
│ Event Queue │
└──────┬──────┘
       ▼
┌────────────────┐
│ Event Mediator │  决定路由到哪些通道
└──────┬─────────┘
       │
   ┌───┴────┬────────────┐
   ▼        ▼            ▼
 Channel  Channel     Channel
   │        │            │
   ▼        ▼            ▼
Processor Processor  Processor
   │
   └──► 可能再次产生事件
```

以「用户下单后需要发短信通知 + 扣库存 + 记积分」为例，走一遍这四个部件怎么配合：用户提交订单后，`OrderService` 把一个 `OrderPlaced` 事件（包含订单号、用户 ID、金额）丢进**事件队列**——它此时不知道、也不关心谁会处理这个事件；**分发器**从队列里取出 `OrderPlaced`，发现它需要分别送到「短信通知」「库存」「积分」三个**事件通道**（可以理解成三条各自独立的传送带）；短信通道另一头的**处理器**读取用户手机号并调用短信网关发送通知，库存通道的处理器执行扣减库存的 SQL，积分通道的处理器给用户账户加分——三个处理器互不知道对方的存在，也互不等待彼此完成。如果库存处理器扣减失败，它可以再发出一个 `InventoryDeductionFailed` 事件，交给专门的补偿处理器去处理，而不会影响短信和积分这两条通道。

对简单项目：队列、分发器、通道可合并，系统近似变为 **事件代理 + 事件处理器** 两块。

## 三、Mediator vs Broker

打个比方：Mediator 像餐厅的**总服务员**，统一安排上菜顺序、谁先谁后；Broker 像自助餐台，各人各取所需，没有人统一指挥。实践中常见两种拓扑（Richards 亦区分）：

### 3.1 Mediator（中介 / 编排）

中央编排器知道「流程下一步是谁」，类似工作流引擎。

- **优点**：流程可见、易审计  
- **缺点**：中介易成热点与单点；复杂流程逻辑集中

还是下单场景：如果用 Mediator 模式，会有一个「下单流程编排器」明确写着「收到 OrderPlaced 后，先调扣库存，扣库存成功再调发短信，发短信之后再调记积分」——出了问题一看编排器代码就知道流程卡在哪一步。但如果这个编排器本身挂了，或者被写成了一个巨大的 if-else 流程判断，所有下单相关的副作用都会跟着停摆，这就是「中介易成热点与单点」的具体表现。

### 3.2 Broker（代理 / 编排弱化）

事件总线或消息代理只负责投递；处理器订阅自己关心的事件，**无中央业务流程大脑**。

- **优点**：扩展性好、去中心  
- **缺点**：端到端流程难一眼看懂；需要良好的事件目录与可观测性

换成 Broker 模式：短信、库存、积分三个处理器各自订阅 `OrderPlaced` 事件，谁也不通知谁。想加一个「下单送优惠券」的新功能，只需要新写一个处理器订阅同一个事件，完全不用改已有代码——扩展很轻松。但代价是：半年后新来的工程师想知道「下单之后系统到底做了哪些事」，翻遍代码也找不到一个地方能一次性看全，只能靠事件目录文档或者一个个搜索「谁订阅了 OrderPlaced」——这就是「端到端流程难一眼看懂」。

```
Mediator:   A → [Orchestrator] → B → C
Broker:     A → bus ← B
                └──── ← C   （各自订阅）
```

## 四、与消息队列的关系

| 概念 | 说明 |
|------|------|
| 消息队列 (MQ) | 基础设施：持久化、投递、消费组、重试 |
| 事件驱动架构 | 组织方式：用事件解耦生产者与消费者 |

MQ（Kafka、RabbitMQ、云上队列等）是 EDA 的常见**实现载体**，但：

- 用了 MQ ≠ 自动获得良好 EDA（仍可能同步耦合、巨型事件、错误主题设计）
- 进程内事件总线也可做轻量 EDA（单机、同进程插件间）

**事件设计要点**：过去式命名（`OrderPlaced`）、尽量自描述、版本兼容、明确至少一次/恰好一次语义与幂等。

```python
from dataclasses import dataclass
from datetime import datetime
from typing import Protocol


@dataclass(frozen=True)
class OrderPlaced:
    order_id: str
    amount: float
    occurred_at: datetime


class EventBus(Protocol):
    def publish(self, event: object) -> None: ...


class InventoryHandler:
    def __init__(self, bus: EventBus) -> None:
        self._bus = bus

    def on_order_placed(self, event: OrderPlaced) -> None:
        # 扣库存；失败需可重试 + 幂等
        ...
```

## 五、优点

1. **分布式异步**，处理器之间高度解耦，扩展性好  
2. **适用面广**，多种业务都可「发事件 + 订阅」  
3. **性能友好**：异步本质不易把整条链路堵死（在背压可控时）  
4. **处理器可独立加载/卸载**，利于渐进部署  

---

## 六、缺点

1. **异步编程复杂**：超时、乱序、重复投递、部分失败都要设计  
2. **难做跨处理器原子事务**：多步事件难以简单回滚（需 Saga、出箱、补偿）  
3. **测试更难**：端到端依赖时序与基础设施；需要契约测试与模拟总线  

反例：如果消息队列因为网络抖动把同一条 `OrderPlaced` 事件重复投递了两次，而扣库存的处理器没做幂等判断，用户明明只买了一件商品，库存却被扣了两次——这就是「不设计幂等」的直接后果。再比如扣库存成功了，但发短信这一步因为短信网关限流而失败：在单体同步事务里这本可以直接回滚，但在事件驱动里库存已经真实扣减、无法简单撤销，只能靠额外的补偿事件（比如「短信最终失败后退回库存」）来处理，这就是「难做跨处理器原子事务」。

---

## 七、何时使用

| 适合 | 不太适合 |
|------|----------|
| 通知、审计、异步衍生流程 | 强一致的短事务必须一次成功 |
| 流量尖峰需削峰填谷 | 团队对分布式与可观测性准备不足 |
| 多下游对同一事实有兴趣 | 流程极短、同步更简单清晰 |
| 与微服务配合做最终一致 | 调试要求「单线程栈一眼到底」 |

**与分层配合**：表现层收请求 → 业务层写库并**发布领域事件** → 异步处理器处理副作用。同步路径保持短，异步路径可扩展。
