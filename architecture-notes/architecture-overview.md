# 软件架构概览 · Software Architecture Overview

> 架构是软件的**基本结构**：决定模块如何划分、如何通信、如何随负载与需求变化而演进。

---

## 一、什么是软件架构

可以把它想成建筑物的承重墙与管线布局——不是墙纸颜色（局部实现细节），而是决定房子能盖多高、水电怎么走、以后能不能砸墙改造的关键结构。软件架构也是同理。

软件架构（software architecture）描述系统的**基本结构**：主要部件是什么、部件之间如何协作、关键决策如何约束后续实现。具体来说：

- 系统被拆成哪些**可识别单元**（层、服务、插件、处理单元……）
- 单元之间如何**通信**（同步调用、消息、共享内存……）
- 哪些决策**昂贵且难改**（数据归属、部署边界、一致性模型……）

以一个电商网站为例：系统被拆成「面向用户的 Web 前端」「订单服务」「库存服务」「商品数据库」这些可识别单元；Web 前端通过 HTTP 调用订单服务，订单服务与库存服务之间可能用消息队列异步通信而不是互相直接查库。而「订单数据到底存在订单服务自己的数据库里，还是允许库存服务也直接读订单表」这类决策一旦拍板并被各处代码依赖，后续想改就会牵动两个团队的接口、数据迁移与上线计划——这正是「昂贵且难改」的具体含义。

大型软件组织常设**架构师**岗位，通常由资深工程师担任：不是只画图，而是对质量属性与演进成本负责。

---

## 二、为何架构重要

合适的架构是软件成功的关键因素之一。错误的结构往往表现为：

| 症状 | 常见根因 |
|------|----------|
| 改一行要全量发布 | 部署单元过大、边界模糊 |
| 加机器却上不去 | 中央瓶颈（单库、单进程）未消除 |
| 测试像修水管 | 依赖缠绕、无法替换接口 |
| 团队互相踩脚 | 模块边界与组织边界错位 |

架构的价值不在「看起来高级」，而在：**用结构换取可预期的变化成本**——扩容、改需求、换技术栈时，改动落在哪里、影响多大。

## 三、架构 vs 设计

打个比方：架构好比一座城市的**总体规划**——主干道怎么修、水电气管网怎么埋、住宅区和商业区怎么划分；设计则是**某一栋房子内部**怎么装修、家具怎么摆。规划定错了，装修再精致也救不了整座城市；但一栋房子的装修选择通常不会牵动城市结构。

二者常被混用，实用区分如下：

| | 架构 (Architecture) | 设计 (Design) |
|--|---------------------|---------------|
| 关注点 | 系统级结构、边界、通信、部署 | 模块内算法、类、数据结构 |
| 变更成本 | 高：牵涉多方、数据、运维 | 相对低：局部重构即可 |
| 典型产出 | 上下文图、部署拓扑、质量属性权衡 | 类图、接口、具体算法 |
| 例子 | 「订单与库存是否独立部署」 | 「订单状态机怎么实现」 |

具体来说：如果「订单服务是否要和库存服务分开部署」这类架构决策定错了——比如一开始把两者耦合在同一个进程里，后来才发现库存扣减需要独立扩容——想要拆分就要重新设计数据归属、接口协议与部署流水线，往往需要多个团队协调数月；而「订单状态机用 if-else 还是状态模式实现」这类设计决策，改起来只需要在订单模块内部重构，不影响其他团队，通常几天内就能完成，外部调用方甚至无感知。

一句话：**架构决定棋盘，设计决定落子。**  
本系列以架构模式与边界为主；OOD/LLD（类级设计）可另见面向对象设计笔记。

## 四、质量属性（Quality Attributes）

选型时，先问「我们最在乎什么」，而不是「业界最火什么」。常见质量属性：

### 4.1 可扩展性 (Scalability)

好比一家小吃店突然被网红博主带火，门口排起长队——能不能多开几个窗口、多雇几个人，快速把订单消化掉，而不是让顾客等到放弃。系统层面同理：负载增长时，能否通过加资源（水平/垂直）维持可接受性能。

- **垂直扩展**：更大机器——简单，有上限
- **水平扩展**：更多实例——需要无状态或可分片的数据模型

例如一个电商系统在大促前预估流量会翻十倍：如果订单查询接口本身无状态（不依赖某台机器内存里保存的数据），运维只需要多开几台一样的服务器、前面挂个负载均衡，就能水平扩容；但如果「库存扣减」靠同一个进程内的锁来保证不超卖，加机器反而会让多个进程各自扣减、总库存被超卖——瓶颈这时已经转移到共享数据这一层，不是加机器就能解决的。

架构模式对扩展性差异极大：分层常要「层层加机器」；基于空间的架构刻意去掉中央库瓶颈；微服务可按服务独立扩容。

### 4.2 可用性 (Availability)

就像银行 App 半夜转账都不能失败——用户不管你后台是不是在维护，只关心「现在能不能用」。系统在约定时间内可服务的比例（如 99.9%）。涉及冗余、故障隔离、降级、超时与重试策略。分布式架构提高隔离能力，也引入网络分区与部分失败。

例如某电商大促期间数据库主库突然宕机：如果系统提前配置了热备库、能在秒级内自动切换，用户大概率只会感觉到一次刷新失败，重试就好；如果没有任何冗余，整个下单链路会一直中断到运维手动排查并重启数据库为止——这段从故障发生到恢复的时间，就是直接计入可用性指标的「不可用时长」。

### 4.3 性能 (Performance)

延迟、吞吐、资源利用率。异步事件驱动往往利于吞吐；跨服务同步调用则堆叠延迟。注意：**架构只给性能空间，算法与数据访问仍决定常数因子。**

例如「下单后发短信通知 + 扣库存 + 更新积分」这个需求：如果三步都同步顺序执行在同一次 HTTP 请求里，接口响应时间就是三步耗时之和，短信网关一慢，整个下单就跟着慢；如果只让「扣库存」留在同步路径里，短信与积分改成发一个事件、由后台异步处理器完成，用户能感知的响应时间就只剩「扣库存」这一步——这也是异步事件驱动往往对吞吐更友好的具体原因。

### 4.4 可维护性 (Maintainability)

理解、修改、测试的成本。清晰边界、稳定接口、可替换实现（依赖倒置）都服务于可维护性。分层易理解但易成「通层修改」；微核利于功能插件化。

例如分层架构里，「订单支持部分退款」这个新需求往往要同时改表现层的退款表单、业务层的退款规则、持久层的退款记录字段——分层让每一处改动本身很容易定位（一看就知道该改哪一层的哪个类），但并不能减少「一共要改几处」，这正是「易理解但易成通层修改」的具体含义。

### 4.5 其他常一起权衡的属性

| 属性 | 简要含义 |
|------|----------|
| 可测试性 | 能否隔离测、能否替身 (mock) |
| 可部署性 | 变更能否小步、独立上线 |
| 安全性 | 边界、信任域、攻击面 |
| 弹性 (Resilience) | 故障后恢复与降级能力 |

**没有免费午餐**：提高某一属性通常牺牲另一属性（例如强一致 vs 可用性，或独立部署 vs 运维复杂度）。

## 五、从单体到分布式：演进简述

这段演进像一家小吃店的成长故事：一开始老板一人身兼采购、烹饪、收银（一切挤在一个人/一个进程里）；生意大了请专人分管进货和收银，但各管一段、需要互相协调（拆成多个服务）；后来干脆开连锁店，每家分店都是标准化的小档口，想开几家开几家，互不影响（容器化、标准化交付）。软件系统的演进大致遵循同样的直觉：

```
┌─────────────────┐
│  单体 Monolith  │  一切在一个进程/一个部署单元
└────────┬────────┘
         │ 功能膨胀、发布昂贵、无法按模块扩容
         ▼
┌─────────────────┐
│  SOA / 服务化   │  按服务拆分，常跨多机，协议多样
└────────┬────────┘
         │ 部署与环境仍重；需要更轻量的运行单元
         ▼
┌─────────────────┐
│ 容器化微服务    │  服务 ≈ 进程/容器，标准化交付
└─────────────────┘
```

以一个校园二手交易平台为例：最早只是一个 Flask 单体项目，商品、订单、用户全在一个进程、一个数据库里，一个人就能维护和部署；用户多了以后，把订单和用户拆成两个独立服务、各自有自己的数据库，两边团队可以分别开发、分别发布；再往后，把这些服务分别打包成 Docker 容器部署到 Kubernetes，一次上线只需要重新部署改动的那个容器，其他服务照常运行——这正是单体 → SOA → 容器化微服务在真实项目里的对应关系。

要点：

1. **单体并非原罪**：团队小、域清晰、流量可控时，单体往往最优。
2. **拆分是为了独立变化**：独立开发、独立部署、独立扩容——拆了却仍同库同发布，只得到分布式的复杂度。
3. **五种经典模式**（分层、事件驱动、微核、微服务、基于空间）解决不同痛点；可组合，不必互斥。

---

## 六、五种模式一览

| 模式 | 一句话 | 典型痛点它治 |
|------|--------|--------------|
| 分层 Layered | 水平分层，请求自上而下 | 组织分工、结构清晰 |
| 事件驱动 Event-Driven | 用事件异步解耦 | 吞吐、扩展、松耦合 |
| 微核 Microkernel | 小内核 + 插件 | 产品可定制、功能扩展 |
| 微服务 Microservices | 独立部署的服务单元 | 团队自治、局部扩容 |
| 基于空间 Space-Based | 内存处理单元 + 虚拟中间件 | 高并发、去掉中央库瓶颈 |

## 七、架构工作的日常形态

不必神秘化。务实的架构工作包括：

1. **画边界**：谁拥有数据、谁可以调用谁
2. **写约束**：允许/禁止的依赖方向（见整洁架构）
3. **做权衡记录**（ADR）：选了什么、放弃了什么、何时复审
4. **对齐质量属性**：用可测指标说话（P99、错误预算、发布频率）

```python
# 伪示意：把「架构决策」当成可审查的记录，而非口头传说
from dataclasses import dataclass
from datetime import date


@dataclass
class ArchitectureDecision:
    title: str
    status: str  # proposed | accepted | superseded
    context: str
    decision: str
    consequences: str
    decided_on: date
```

一次真实的记录可能长这样：

```python
decision = ArchitectureDecision(
    title="订单服务与库存服务是否共用同一个数据库",
    status="accepted",
    context="大促期间库存扣减并发量上升，与订单查询共用一个数据库导致锁等待变长",
    decision="库存服务拆出独立数据库，订单服务通过 API 调用库存服务，不再直连库存表",
    consequences="库存扣减不再阻塞订单查询；但两边数据不再天然强一致，需要引入补偿逻辑",
    decided_on=date(2025, 3, 1),
)
```
