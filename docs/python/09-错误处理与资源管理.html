错误处理与资源管理

本章两条主线:异常机制(异常也是对象),以及 CPython 的内存管理(引用计数 + 分代 GC + pymalloc 内存池)。后者是 C/C++ 程序员最关心的部分,我们会和手动 malloc/free、RAII、智能指针逐一对照。

1. 异常是对象

Python 的异常不是 C 的错误码,也不是 C++ 里可以抛任意类型的 throw 42。所有异常都是继承自 BaseException 的对象:

Arena  ── 约 1 MiB 一大块,直接向操作系统 mmap 申请
  │  切成若干
Pool   ── 一个内存页大小(x86 常见 4 KB;Apple Silicon 为 16 KB),一个 pool 只服务一种 size class
  │  切成等长
Block  ── 按 16 字节对齐的等长小块:16, 32, 48, …, 512 字节,共约 32 个 size class
             一次小对象分配 = 从对应 pool 的空闲链表摘一个 block

工作方式:

  • 申请小对象时,按大小归入某个”size class”,从对应 pool 的空闲块链表里 O(1) 摘一个 block;释放时归还到空闲链表,不马上还给 OS。
  • 因此小对象的分配/释放几乎不碰系统 malloc,极快;同一 size class 的对象聚在一起,减少碎片。

  • 512 字节的大对象绕过 pymalloc,直接走系统 malloc。

  • 完全空的 pool/arena 才可能归还操作系统。所以你会看到 Python 进程内存”涨上去下不来”——很多是被内存池占着复用,并非泄漏。

对照 C:这相当于 CPython 内建了一个为”海量小对象”优化的对象池 allocator,层级(arena→pool→block)类似很多游戏/高性能 C++ 项目里手写的 slab/pool allocator。

此外还有前面提过的空闲链表(free list):小整数、小 tuple、常用大小的 list/frame 等,销毁后其内存块被挂到专用 free list 里等着复用,连 pymalloc 那一步都省了。

8. 全景小结

机制 作用 C/C++ 对照
引用计数 主力,即时回收,确定性 shared_ptr
分代 GC 兜底回收循环引用 追踪式 GC(Java/Go 才有)
weakref 不增计数的引用 weak_ptr
with / 上下文管理器 确定性资源释放 RAII / 析构函数
__del__ 析构钩子(不可靠,慎用) 析构函数(但时机不确定)
pymalloc 小对象内存池 slab/pool allocator
free list 常用对象内存复用 对象池

一句话:Python 的内存管理是”引用计数(像 shared_ptr,给你确定性)+ 分代 GC(兜底环)+ 多层内存池(给你速度)“的组合;资源释放请用 with,别赌 __del__。
