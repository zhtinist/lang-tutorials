Common Third-Party Libraries

Dependency management has long been a pain point in C/C++: hand-written Makefiles/CMake, header files plus static/dynamic libraries, and only recently a unified approach with vcpkg/Conan. The Java ecosystem unified dependency distribution long ago with Maven/Gradle + a central repository. This chapter covers build tools, the coordinate mechanism, and the ecosystem positioning of mainstream libraries—the focus is “what to use in which scenario,” not teaching APIs.

Build and Dependency Management

Coordinates and Repositories

Every library is uniquely identified by its GAV coordinate: groupId:artifactId:version. Analogous to “namespace + library name + version” in C++, but here it is a single global namespace (Maven Central).

com.google.guava : guava : 33.0.0-jre
     |               |         |
   groupId       artifactId  version

Maven (pom.xml, XML, convention over configuration):

<dependencies>
    <dependency>
        <groupId>com.google.guava</groupId>
        <artifactId>guava</artifactId>
        <version>33.0.0-jre</version>
    </dependency>
</dependencies>

Gradle (build.gradle, Groovy/Kotlin DSL, more flexible with fast incremental builds):

dependencies {
    implementation 'com.google.guava:guava:33.0.0-jre'
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.0'
}

The build tool downloads jars from a repository into a local cache (Maven’s ~/.m2/repository); the default repository is Maven Central, and you can add a private server (Nexus/Artifactory). This step replaces the C++ chore of manually finding libraries and configuring include/link paths.

Transitive Dependencies and Scope

A key mechanism: transitive dependencies. You depend on A, A depends on B, and the build tool automatically pulls in B—no explicit declaration needed. Analogous to C++ if it had fully automatic resolution of header and library dependencies.

Your project ──> Jackson-databind ──> Jackson-core
                                  └──> Jackson-annotations   (pulled in automatically)

Dependency conflicts: when different libraries depend on different versions of the same library, Maven arbitrates by “nearest path first + first-declared first,” while Gradle by default takes the highest version. Use mvn dependency:tree / gradle dependencies to view the dependency tree, and <exclusions> to exclude.

Scope/configuration determines in which phase a dependency is visible:

Maven scope Gradle configuration Meaning
compile (default) implementation/api Needed at both compile and runtime
provided compileOnly Needed at compile time, provided by the container at runtime (e.g., the Servlet API)
runtime runtimeOnly Needed only at runtime (e.g., a JDBC driver)
test testImplementation Test only

implementation vs. api (Gradle): implementation does not expose the dependency transitively to downstream consumers (encapsulation, like PIMPL hiding the implementation), while api does. Using implementation appropriately reduces the recompilation scope and avoids dependency leakage.

Ecosystem Positioning

The following are grouped by purpose; just remember “the selection,” and look up the API when you use it.

General Utilities: Guava, Apache Commons

  • Guava (Google): Preconditions for argument validation, immutable collections ImmutableList, Multimap/BiMap/Cache (local cache), Splitter/Joiner, and the concurrent ListenableFuture. Highly engineered, the first choice for modern projects.
  • Apache Commons: commons-lang3 (StringUtils, ObjectUtils), commons-collections4, commons-io. Long-established, relied on by many legacy projects. Newer JDK versions have built in many equivalents (such as String.strip, Files.readString); use as needed.

JSON: Jackson, Gson

Serialize/deserialize objects and JSON, reading and writing fields via reflection.

// Jackson: the most full-featured, Spring's default, supports streaming/tree/data-binding modes
ObjectMapper mapper = new ObjectMapper();
String json = mapper.writeValueAsString(person);        // object -> JSON
Person p = mapper.readValue(json, Person.class);        // JSON -> object

// Gson (Google): simpler API, fewer dependencies, for lightweight scenarios
Gson gson = new Gson();
String j = gson.toJson(person);
Person p2 = gson.fromJson(j, Person.class);

Selection: for Spring projects use Jackson directly (already integrated); for standalone small tools use Gson.

Enterprise Framework: Spring / Spring Boot

The de facto standard for Java backends. Its core is the IoC container + DI + AOP.

IoC (Inversion of Control) / DI (Dependency Injection): an object does not new its dependencies itself but declares what it needs, and the container creates and injects them. Analogous to “not managing lifetimes yourself, handing it to a global factory + scope manager.”

@Service                              // declared as a container-managed Bean
public class OrderService {
    private final PayClient payClient;
    // Constructor injection: at container startup, a Bean of type PayClient is found and passed in
    public OrderService(PayClient payClient) { this.payClient = payClient; }
}

At startup, the container scans annotations, instantiates Beans, resolves the dependency graph, and injects in order (handling dependency ordering, singleton reuse, and circular-dependency detection).

AOP (Aspect-Oriented Programming): extracts cross-cutting concerns (transactions, logging, authorization) out of business code and weaves them in declaratively.

@Transactional                        // the method is automatically wrapped in a transaction: begin -> execute -> commit/rollback
public void placeOrder() { /* pure business, no transaction boilerplate */ }

Underneath it relies on dynamic proxies: Spring generates a proxy object for the Bean, and a method call goes through the proxy first (executing the aspect logic) then forwards. Two implementations: - JDK dynamic proxy: when the target implements an interface, use java.lang.reflect.Proxy to generate an interface proxy class at runtime. - CGLIB: when there is no interface, generate a subclass of the target class at runtime (bytecode enhancement) that overrides methods. Hence a proxied method cannot be final (it can’t be overridden).

This machinery relies on reflection (reading type information and invoking methods at runtime)—C++ has no built-in reflection; Java achieves it with .class metadata plus the Class/Method API, at the cost of runtime overhead and difficulty in static optimization (part of why Spring startup is slow; the GraalVM native image is meant to mitigate this).

ORM: Hibernate / JPA

  • JPA: a specification (interfaces) defining the annotations and API for object-relational mapping.
  • Hibernate: JPA’s most mainstream implementation.
@Entity                               // mapped to a database table
public class User {
    @Id @GeneratedValue Long id;      // primary key, auto-increment
    @Column(nullable = false) String name;
}
// Operate on objects via EntityManager/Repository; the framework translates to SQL

It maps relational tables to objects, auto-generates SQL, and manages sessions and caching. It saves you hand-written JDBC boilerplate, but beware of N+1 queries, lazy-loading pitfalls, etc.

Testing: JUnit 5 + Mockito

// JUnit 5: a unit-testing framework
class CalcTest {
    @Test
    void add() {
        assertEquals(3, new Calc().add(1, 2));
    }
    @ParameterizedTest
    @ValueSource(ints = {1, 2, 3})
    void positive(int n) { assertTrue(n > 0); }
}

// Mockito: mock dependencies to isolate the unit under test
UserRepo repo = mock(UserRepo.class);
when(repo.findById(1L)).thenReturn(new User());   // stubbing: specify the mock's behavior
UserService svc = new UserService(repo);
svc.load(1L);
verify(repo).findById(1L);                        // verify the interaction happened

Compile-Time Enhancement: Lombok

Uses annotations to generate bytecode at compile time, eliminating boilerplate like getters/setters/constructors/toString.

@Data                                 // generates getter/setter/equals/hashCode/toString
@AllArgsConstructor
public class Point {
    private int x;
    private int y;
    // No methods in the source, but they're all in the bytecode after compilation
}

How it works: Lombok is an annotation processor that hooks into the javac compilation pipeline and modifies the AST (abstract syntax tree) at compile time to inject methods. It is not runtime reflection, so there is zero runtime overhead. The cost: the IDE needs a plugin to recognize the generated methods; and it depends on javac’s internal APIs, so major JDK upgrades occasionally bring compatibility issues.

Logging: SLF4J + Logback

SLF4J is a logging facade (interface), and Logback/Log4j2 are implementations. The code depends only on the facade, and the concrete implementation is bound at runtime—analogous to “programming to an interface,” letting you swap implementations without changing code.

private static final Logger log = LoggerFactory.getLogger(MyClass.class);
log.info("user {} logged in", userId);   // the {} placeholder is concatenated lazily; if the level is too low no string is built, saving overhead
log.error("failed", exception);

A configuration file (logback.xml) controls the level, output format, and rolling strategy, with no code changes needed.

High-Performance Networking: Netty

An asynchronous, event-driven networking framework based on Java NIO (multiplexing via epoll/kqueue), using the Reactor pattern plus a small number of threads to handle a massive number of connections. RPC frameworks (gRPC-Java, Dubbo) and message middleware are often built on it.

  • EventLoop: one thread bound to one selector, polling multiplexed channel events—analogous to a hand-written epoll event loop in C, but packaged into a pipeline.
  • ChannelPipeline: a chain of handlers through which inbound/outbound data passes in sequence (codec, business processing), like a chain of responsibility.
  • Pooled ByteBuf (off-heap memory + reference counting) reduces GC and copies—close to manually managing buffers in C, but you must remember to release manually to prevent leaks.

Suited for: building your own high-concurrency long-connection services. For ordinary HTTP services, Spring Boot (with its embedded Tomcat/Netty) is enough; you don’t need to write Netty directly.

Packaging: jar and fat-jar

  • Ordinary jar: contains only this project’s classes + resources, not the dependencies. To run it you must list all dependency jars on the classpath manually, like compiling only .o files without linking the libraries.
  • fat-jar / uber-jar: packs all transitive dependencies into one executable jar, so java -jar app.jar runs directly. Spring Boot uses spring-boot-maven-plugin; ordinary projects use maven-shade-plugin or Gradle’s shadow plugin.
mvn package                       # produces target/app.jar
java -jar target/app.jar          # a fat-jar runs directly
java -cp "app.jar:libs/*" Main    # an ordinary jar needs an explicit classpath

In a fat-jar, multiple dependencies containing same-named resources (such as each one’s META-INF/services) will conflict; the shade plugin’s transformer is responsible for merging them.

JNI: The Cost of Calling C

Java calls C/C++ via JNI (Java Native Interface) (when you need to reuse an existing native library or do CPU-intensive computation). But the cost is not small, and it is usually a last resort:

public class NativeLib {
    static { System.loadLibrary("mylib"); }   // load libmylib.so/.dll
    public native int compute(int[] data);    // native method, implemented on the C side
}

Sources of cost: - Call overhead: each crossing of the JNI boundary has a fixed cost (parameter marshaling, state switching), unsuitable for high-frequency fine-grained calls. - Object marshaling: Java objects/arrays live in the managed heap, and C-side access needs GetArrayElements etc., which may trigger a copy or pin the object, hindering GC movement/compaction. - Safety: a crash in native code (a segfault) directly brings down the whole JVM, losing the JVM’s memory-safety guarantees. - Portability: you must compile a native library for each platform, breaking Java’s “write once, run anywhere.”

The JDK is promoting the FFM API (Foreign Function & Memory, JEP 454, finalized in JDK 22) as a modern replacement for JNI, declaring native calls and off-heap memory in pure Java with no C glue code, lower overhead, and better safety. Prefer it for new projects.
