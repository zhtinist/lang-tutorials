Variables and the Memory Model: What Really Happens in a = b ⭐

This is the most important chapter in the book. If you only read one chapter, read this one. When C/C++ programmers switch to Java, almost every “weird bug” stems from misunderstanding what actually happens in memory during assignment, comparison, and parameter passing. This chapter makes it completely clear.

Let’s state the core conclusion up front, then argue it point by point:

Java variables fall into two camps: a primitive-type variable holds a “value” directly; a reference-type variable holds a “reference value”—a JVM-managed pointer to an object on the heap. a = b is always “copy the bits in b into a”: for a primitive it copies the value itself, and for a reference type it copies that pointer (from then on both variables point to the same heap object).

1. The Two Camps

Camp Members What the variable stores Where the data is
Primitive types byte short int long float double char boolean the value itself (bits) where the variable is (a stack-frame slot or inlined in an object field)
Reference types classes, interfaces, arrays, String, wrapper classes, enums… a reference value (≈ pointer) the object the reference points to, on the heap

In C terms: - A primitive-type variable ≈ int a;—the memory of a is directly that integer. - A reference-type variable ≈ Foo* p;—the memory of p holds an address, and the actual Foo lives on the heap (allocated by new). But you cannot do arithmetic on this “pointer,” cannot dereference it to take an address, and cannot cast it to an integer—it is a JVM-managed, constrained pointer.

HotSpot implementation detail: a reference on a 64-bit JVM usually occupies 8 bytes, but with compressed pointers (CompressedOops) enabled it shrinks to 4 bytes (using a 35-bit addressable encoding, on by default for heaps < 32 GB). See Data Types. The language specification does not dictate the concrete representation of a reference, only its semantics.

2. Local Variable Table Slots: Where Variables Live

On each method call, the JVM pushes a stack frame onto the current thread’s JVM stack; the frame contains a Local Variable Table, made up of several slots (the count is fixed at compile time, see javap in Runtime Model).

  • A slot is 32 bits. int/float/reference each take 1 slot; long/double take 2 consecutive slots.
  • Both method parameters and local variables declared inside the method live in this table.

For a primitive-type local variable, the slot holds that value directly. For a reference-type local variable, the slot holds the reference (pointing to the heap). The object body is never in a slot, always on the heap (except under escape analysis, see the end of the chapter).

3. a = b: The Core of the Core

3.1 Primitive Types: Copy the Value

int b = 100;
int a = b;   // copy b's bits (100) into a's slot
a = 200;     // changes only a's own slot, no effect on b
// at this moment a=200, b=100

Memory diagram (stack-frame local variable table):

 before assignment    after a = b          after a = 200
┌──────────┐       ┌──────────┐        ┌──────────┐
│ b │ 100  │       │ b │ 100  │        │ b │ 100  │
├──────────┤       ├──────────┤        ├──────────┤
│ a │  ??  │       │ a │ 100  │←copy    │ a │ 200  │
└──────────┘       └──────────┘        └──────────┘
     Two independent slots, unaffecting each other. Equivalent to C's int a=b;

This is exactly the same as C, no surprises.

3.2 Reference Types: Copy the Reference (copying the pointer, not the object!)

int[] b = new int[]{1, 2, 3};  // create the array object on the heap, b holds the reference to it
int[] a = b;                    // copy the "reference value"! a and b point to the same array
a[0] = 99;                      // modify the heap object through a
System.out.println(b[0]);       // prints 99! because b points to the same object

This is the intuition a C programmer most needs to build: int[] a = b; does not copy the array, it copies the pointer. It is equivalent to C’s int* a = b; (a shallow pointer copy), and not memcpy.

Memory diagram:

      stack-frame local var table    Heap
   ┌────────────────┐         ┌───────────────────────┐
   │ b │  ref ●──────┼────────▶│ int[3] object          │
   ├────────────────┤    ┌───▶│ [0]=99 [1]=2 [2]=3     │
   │ a │  ref ●──────┼────┘    │ (a[0]=99 changed this) │
   └────────────────┘         └───────────────────────┘
      a and b are two pointers pointing to the same heap object (aliasing)

Compared with the primitive-type diagram, the difference is obvious at a glance:

   Object a = b  (reference type)      int a = b  (primitive type)
   ┌───────┐                          ┌───────┐
   │ b │●──┼──┐                       │ b │100│
   ├───────┤  │   Heap                ├───────┤
   │ a │●──┼──┴──▶[ object ]          │ a │100│ ← independent copy
   └───────┘  two pointers share      └───────┘  two independent values
              one object

In a sentence: primitive assignment clones the “data,” reference assignment clones the “pointer.” The latter produces aliasing—modifying the object through either variable is visible to the other. This is exactly the same as the behavior of two C pointers pointing to the same malloc’d memory.

3.3 What If You Want a True Copy

Java does not deep-copy automatically (there is no C++ copy constructor triggered on assignment). To get an independent copy you must do it explicitly:

int[] a = b.clone();               // array shallow copy, yields a new object
int[] a2 = Arrays.copyOf(b, b.length);
List<T> copy = new ArrayList<>(src);  // a collection's copy constructor

Note that these are mostly shallow copies: the container/array itself is new, but if it holds references, the element references still share the same set of objects. A deep copy requires you to duplicate layer by layer.

4. == vs equals

Once you understand the two camps, this classic trap follows naturally.

==: Compares “the bits in the variable”

  • Primitive types: compares the value. 3 == 3 → true. Intuitive.
  • Reference types: compares the reference—i.e. “is it the same heap object” (whether the address is the same), regardless of content.
String x = new String("hi");
String y = new String("hi");
System.out.println(x == y);        // false! two different heap objects, two different pointers
System.out.println(x.equals(y));   // true! compares content

Memory diagram:

 ┌───────┐        Heap
 │ x │●──┼──▶[ String "hi" object① ]
 ├───────┤
 │ y │●──┼──▶[ String "hi" object② ]   x==y compares ● vs ●, different targets → false
 └───────┘                              x.equals(y) compares char by char → true

equals: Compares “content” (semantic equality)

equals is a method defined on Object, and it can be overridden. The default implementation (Object.equals) is just == (compares references); but String, the wrapper classes, collections, etc. override it to compare logical content. So: - To compare string/object content, always use equals. - Use == only when you really want to ask “is it the same object,” or when comparing primitives, or comparing against null.

Paired with equals there is also hashCode: the contract is “if equals is true, hashCode must be equal,” otherwise putting the object into a HashMap/HashSet goes wrong. Details in Composite Data Structures. In C++ you provide operator== and hash for unordered_map—same idea.

A Counterintuitive Pitfall: the Wrapper-Class Cache

Integer a = 100, b = 100;
System.out.println(a == b);   // true  —— 100 is in the cache range
Integer c = 200, d = 200;
System.out.println(c == d);   // false —— 200 is beyond the cache, they are two objects

Integer caches objects for [-128, 127], reusing the same object on boxing, so == happens to be true; beyond that range a new object is created, and == is false. This is precisely evidence that == compares references. To compare the values of wrapper classes, always use equals or unbox first. Details in Data Types.

5. null

null is the “null reference” of reference types, equivalent to C’s null pointer nullptr—the variable points to no object.

String s = null;         // s's slot holds the "null reference" value
s.length();              // NullPointerException! "dereferencing" a null reference

The difference from C: dereferencing a null pointer in C is undefined behavior (most likely a segfault); in Java it deterministically throws a NullPointerException (NPE), which can be caught and carries stack information (Helpful NPE / JEP 358: introduced in JDK 14 but off by default, on by default since JDK 15, able to indicate which variable was null). Primitive types cannot be null (they hold a value, not a reference); int x = null; is a compile error. This is also why unboxing a wrapper Integer to an int throws NPE when it is null.

6. final Variable Semantics

final is often held up as the counterpart of C++’s const, but you must distinguish precisely: final constrains whether the variable’s slot can be reassigned, not whether the object it points to can be modified.

final int x = 10;
// x = 20;                 // compile error: cannot reassign

final int[] arr = {1, 2, 3};
arr[0] = 99;               // legal! modifies the heap object's content, not the arr reference
// arr = new int[5];       // compile error: cannot make arr point to a different object

Understanding via the memory diagram: final locks the slot on the left (cannot change what it points to), but cannot lock the object’s content on the heap on the right.

 final int[] arr ●═══▶[ 1, 2, 3 ]   ═══ means this reference is locked and cannot be repointed
                        ↑ but the object content can still be changed: arr[0]=99 ok
  • To make “the object immutable too,” you must rely on the object’s own design (e.g. String is inherently immutable, or use immutable collections like List.copyOf(...)). This resembles C++’s distinction between const int* p (cannot change the pointed-to content) vs int* const p (cannot change the target): Java’s final provides only the latter half—it locks the target, not the content.
  • final fields: must be assigned exactly once before construction completes, and are immutable afterwards; there is a guarantee for multithreaded visibility (after construction completes, other threads can see the correct final-field values).
  • final local variables / parameters: commonly used for capture by a lambda (a lambda can only capture “effectively final” local variables, see Methods and the Call Stack).

7. Putting Them into Object Fields

The above covers local variables (in stack-frame slots). An object’s fields follow the same two-camp rules, only their storage location is inside the object on the heap:

class Node {
    int value;      // primitive field: the value is inlined directly in the Node object's memory
    Node next;      // reference field: holds a reference to another Node on the heap (or null)
}

Memory diagram (a Node object on the heap):

The Node object on the heap
┌─────────────────────────┐
│ [object header mark+klass]│  ← see the object layout in Chapter 4
│ value : 42              │  ← int value inlined
│ next  : ref ●───────────┼──▶ another Node object (or null)
└─────────────────────────┘

This is exactly what a linked list/tree looks like in Java: the structure is completely isomorphic to C’s struct Node { int value; struct Node* next; };, differing only in that next is a managed reference, the object lives on the GC heap, and you don’t free it.

8. Escape Analysis: Language Semantics vs. HotSpot Implementation

The language spec says “objects always live on the heap.” But in HotSpot, the JIT’s escape analysis (see Runtime Model) may perform scalar replacement if it proves that some new’d object will not escape the current method: it breaks the object into a few primitive values placed directly in stack-frame slots or even registers, never entering the heap and needing no GC.

int distance() {
    var p = new Point(3, 4);   // p does not escape this method
    return p.x * p.x + p.y * p.y;
    // after HotSpot optimization: there may be no new at all, x and y become two stack local variables
}

So the precise statement is: - At the semantic level: a reference-type object logically lives on the heap, assignment copies the reference, and aliasing holds. - At the implementation level: a non-escaping object may undergo scalar replacement (broken into primitive values placed in stack-frame slots/registers, rather than moving the whole object onto the stack), approaching the efficiency of C’s “struct on the stack”—but this is semantically transparent to the code you write; you need not, and cannot, directly control it.

This explains why creating lots of short-lived small objects in Java is often not as slow as you’d imagine.

Summary (be sure to internalize)

  1. The two camps of variables: primitives hold values, reference types hold pointers (reference values).
  2. a = b always copies “the bits in the variable”: for a primitive it is a value copy (an independent copy), for a reference type it is a pointer copy (producing aliasing, sharing the same heap object). To get a copy you must explicitly clone / copy-construct, and the default is a shallow copy.
  3. == compares bits (for reference types, whether it’s the same object), equals compares content—always use equals to compare object content.
  4. null is a null reference; dereferencing it deterministically throws NPE (not UB); primitives cannot be null.
  5. final locks only “what the variable points to,” not “the object’s content” (equivalent to C++’s T* const, not const T*).
  6. Object fields follow the same two-camp rules, only they live inside the heap object; linked lists/trees are isomorphic to C structs, only the references are managed and reclaimed by GC.
  7. HotSpot escape analysis may perform scalar replacement on non-escaping objects (breaking them into scalars placed in stack-frame slots/registers, rather than whole-object stack allocation), but this is semantically transparent.
