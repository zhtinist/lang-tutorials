Standard Library Algorithms

C++’s <algorithm>, std::sort (introsort), std::set (red-black tree), and priority_queue (heap) live in the STL. Java’s counterparts are java.util.Collections, Arrays, PriorityQueue, and TreeMap, plus the functional Stream API. This chapter covers the underlying implementations of these tools and the reasons for the choices, rather than listing APIs.

The Collections Utility Class

java.util.Collections is a batch of static methods operating on List/Collection:

List<Integer> list = new ArrayList<>(List.of(3, 1, 2));
Collections.sort(list);                          // in-place sort (internally converts to an array and uses TimSort)
int idx = Collections.binarySearch(list, 2);     // binary search on a sorted List; a linked list (non-RandomAccess) is O(n) list traversals + O(log n) comparisons
Collections.reverse(list);
Collections.shuffle(list);
int max = Collections.max(list);

// Immutable view: returns a wrapper; any modification throws UnsupportedOperationException (like a const reference, but checked at runtime)
List<Integer> ro = Collections.unmodifiableList(list); // still backed by list; changing list changes it too
List<Integer> imm = List.of(1, 2, 3);            // JDK 9+ a truly immutable copy

binarySearch requires the List to be sorted in the same order, or the result is undefined. A negative return -(insertion point)-1 indicates “not found,” which you can use to find the insertion position.

Arrays.sort: Two Algorithms

This is the focus of the chapter. Arrays.sort uses different algorithms for primitive types and object arrays, and this is not arbitrary:

int[] a = {5, 2, 8, 1};
Arrays.sort(a);                    // Dual-Pivot Quicksort

Integer[] b = {5, 2, 8, 1};
Arrays.sort(b);                    // TimSort (a merge-sort variant)
Arrays.sort(b, Comparator.reverseOrder());

Why the Difference

Dimension Primitive int[] Object Object[]
Algorithm Dual-pivot quicksort TimSort
Stability Irrelevant (equal values are wholly identical) Must be stable
Comparison cost Extremely low (machine instructions) High (calls compareTo/compare, possibly a virtual method)
Focus Cache locality, comparison count Fewer comparisons, stability

Why primitives use dual-pivot quicksort: two equal ints are the same value, and swapping them makes no observable difference, so stability is meaningless. Quicksort is in-place and cache-friendly (sequential access has good locality), and comparison is a cheap CPU instruction. Dual-pivot quicksort picks two pivots to split the array into three parts, giving about 5% fewer comparisons than classic single-pivot quicksort and better cache locality—note that its number of swaps is actually higher than single-pivot; the advantage comes from “fewer comparisons + memory-access-friendliness,” not “less moving.”

Why objects use TimSort: 1. Stability: object sorting is often by some field, and the original relative order of equal elements must be preserved (e.g., sort by A first, then by B; when B is equal, A’s order must be maintained). Quicksort is unstable; TimSort (a merge-sort family member) is stable. 2. Comparison is expensive: comparing objects is a method call (possibly an interface virtual call); on nearly-sorted data TimSort’s comparison count approaches O(n), whereas quicksort’s worst case is O(n²). Real-world data is often partially ordered, and TimSort identifies existing sorted “runs” and merges them, greatly reducing comparisons.

TimSort combines merge sort with insertion sort: it scans out increasing/decreasing runs, extends short runs to the minrun length with binary insertion sort, then merges according to invariants maintained on a stack. Worst case O(n log n), already-sorted O(n). It needs O(n) auxiliary space (for merging), whereas quicksort is in-place—but object arrays already have an extra layer of pointer indirection and poor cache behavior, so the extra space is acceptable.

Comparable and Comparator

  • Comparable<T>: natural ordering, implemented by the class itself via compareTo; a class has only one. Analogous to C++’s operator<.
  • Comparator<T>: an external strategy, and you can define arbitrarily many; analogous to a functor passed to std::sort.
record Person(String name, int age) implements Comparable<Person> {
    @Override public int compareTo(Person o) {   // natural order: by age
        return Integer.compare(this.age, o.age);  // don't use this.age-o.age, overflow risk
    }
}

// Comparator combinators (JDK 8+), declarative, like a chained SQL ORDER BY
Comparator<Person> cmp = Comparator
    .comparing(Person::name)                     // by name first
    .thenComparingInt(Person::age)               // if names are equal, then by age
    .reversed();                                 // reverse the whole thing

Comparator<Person> byAgeNullsFirst = Comparator
    .comparingInt(Person::age)
    .reversed();
List<Person> people = new ArrayList<>();
people.sort(cmp);   // List.sort internally uses TimSort

The contract: compare must be reflexive, transitive, and antisymmetric; and consistency with equals is recommended (otherwise TreeMap/TreeSet behavior is counterintuitive). Violating transitivity makes TimSort throw IllegalArgumentException: Comparison method violates its general contract.

PriorityQueue: The Binary Heap

PriorityQueue is an array-based min-heap (by default the smallest element is at the head), analogous to C++’s priority_queue (a max-heap by default).

PriorityQueue<Integer> pq = new PriorityQueue<>();      // min-heap
pq.offer(3); pq.offer(1); pq.offer(2);
pq.peek();     // 1, view the heap top O(1)
pq.poll();     // 1, pop and sift down to rebalance O(log n)

// Max-heap / custom
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());

Internally an Object[] queue, with parent-child index relationships: node i’s children are 2i+1 and 2i+2. offer sifts up (siftUp), and poll moves the last element to the top and sifts down (siftDown).

The heap represented by the array [1,3,2,7,4]:
          1(0)
        /     \
      3(1)     2(2)
      /  \
    7(3) 4(4)

Not thread-safe, and it does not guarantee ordered traversal (only the heap top is ordered); to traverse in order you must repeatedly poll. For concurrent scenarios, use PriorityBlockingQueue.

TreeMap / TreeSet: The Red-Black Tree

TreeMap is backed by a red-black tree (a self-balancing BST), and TreeSet is its wrapper (values used as keys). These correspond to C++’s std::map/std::set. All operations are O(log n), and they are ordered by key.

TreeMap<String, Integer> tm = new TreeMap<>();
tm.put("banana", 2); tm.put("apple", 1); tm.put("cherry", 3);
// Ordered views and navigation methods (HashMap has none)
tm.firstKey();               // "apple"
tm.floorKey("b");            // "apple" (largest key ≤ "b")
tm.ceilingKey("b");          // "banana" (smallest key ≥ "b")
tm.headMap("cherry");        // subview of keys less than cherry (a view; changes are reflected in the original map)
tm.subMap("apple", "cherry");
for (var e : tm.entrySet()) { }  // traverse in ascending key order

The five properties of a red-black tree (nodes are red/black, the root is black, a red node’s children are black, all paths from any node to its leaves have the same black height, etc.) guarantee the longest path is ≤ 2× the shortest, giving approximate balance. Insertion/deletion is maintained via recoloring + rotation, with each adjustment O(log n).

Comparison HashMap TreeMap
Underlying Array + linked list/red-black tree Red-black tree
Ordered Unordered Ordered by key
get/put O(1) amortized O(log n)
Navigation methods None floor/ceiling/subMap, etc.
null key One allowed Not allowed (comparison required)

The Stream API

A Stream is a declarative, lazy, parallelizable data-processing pipeline, analogous to C++20 ranges/views. It is not a container, does not store data, and is “a description of operations on a data source.”

Laziness and Operation Categories

List<String> words = List.of("apple", "banana", "cherry", "avocado");

long count = words.stream()          // source
    .filter(s -> s.startsWith("a"))  // intermediate operation: lazy, returns a new Stream
    .map(String::toUpperCase)        // intermediate operation
    .peek(System.out::println)       // intermediate operation (for debugging)
    .count();                        // terminal operation: triggers actual execution
  • Intermediate operations (filter/map/sorted/distinct/limit): lazy, merely recorded, not executed. They return a Stream and can be chained.
  • Terminal operations (collect/forEach/count/reduce/findFirst): trigger the traversal and execute the whole chain.

The point of laziness: operations are fused, so one element flows through the entire chain before the next is taken (rather than generating a full intermediate collection at each step), saving memory; limit/findFirst can short-circuit to finish early.

// Short-circuit: stop at the first match, without processing the whole stream
Optional<Integer> first = Stream.iterate(1, n -> n + 1)  // infinite stream
    .filter(n -> n % 7 == 0)
    .findFirst();   // Optional[7]

Collectors

The terminal operation collect gathers a stream into a result:

import static java.util.stream.Collectors.*;

List<String> list  = words.stream().collect(toList());
Set<String> set    = words.stream().collect(toSet());
Map<Character, List<String>> byFirst =
    words.stream().collect(groupingBy(s -> s.charAt(0)));   // grouping, like SQL GROUP BY
Map<Boolean, List<String>> parts =
    words.stream().collect(partitioningBy(s -> s.length() > 5));
String joined = words.stream().collect(joining(", ", "[", "]"));
Double avg = words.stream().collect(averagingInt(String::length));

Parallel Streams

parallelStream() or .parallel() splits the stream and processes it divide-and-conquer using the common ForkJoinPool (ForkJoinPool.commonPool(), default thread count = CPU cores - 1).

long sum = LongStream.rangeClosed(1, 1_000_000)
    .parallel()
    .sum();

Considerations: - The data source should be easy to split (arrays and ArrayList are good; LinkedList is bad). - Operations must be stateless and side-effect-free, or you get data races (do not modify shared variables inside a lambda). - For small datasets or cheap operations, the splitting/merging/scheduling overhead of parallelism may not pay off. - All parallel streams share the same commonPool, so one slow task drags down the whole system; don’t use it for IO-bound work.

Optional

Optional<T> expresses “there may be no value,” replacing a bare null and forcing the caller to handle the absence, reducing NPEs. Analogous to C++17’s std::optional (but Java’s is an object, with heap allocation overhead).

Optional<String> opt = Optional.ofNullable(getName());
String s = opt.orElse("default");             // give a default if empty
String t = opt.orElseGet(() -> compute());    // lazy default (computed only if empty)
opt.ifPresent(System.out::println);
String u = opt.map(String::trim).filter(x -> !x.isEmpty()).orElse("-");
opt.orElseThrow(() -> new IllegalStateException("missing"));

Usage convention: use it mainly as a method return value to express “there may be no result”; do not use it for fields, method parameters, or collection elements (needless overhead).

Numeric Utilities

Math.max(a, b); Math.floorMod(-1, 3); // 1 (unlike %, the result's sign follows the divisor)
Math.addExact(a, b);                  // throws ArithmeticException on overflow, rather than silently wrapping

// BigInteger: arbitrary-precision integers (immutable), like a bignum you'd implement yourself; operations return new objects
BigInteger big = BigInteger.valueOf(2).pow(1000);
BigInteger f = big.multiply(BigInteger.TEN).mod(BigInteger.valueOf(7));

// BigDecimal: exact decimals, a must for finance (double cannot represent 0.1 exactly)
BigDecimal price = new BigDecimal("0.10");    // must construct from a String; new BigDecimal(0.1) carries binary error
BigDecimal total = price.multiply(new BigDecimal("3"));  // exact 0.30
System.out.println(new BigDecimal("0.1").add(new BigDecimal("0.2"))); // 0.3 (double would be 0.30000000000000004)
total = total.setScale(2, RoundingMode.HALF_UP);         // explicitly specify rounding

// Random: linear congruential pseudo-random; for concurrency use ThreadLocalRandom; for security use SecureRandom
int r = ThreadLocalRandom.current().nextInt(0, 100);

BigInteger/BigDecimal are immutable, so frequent operations produce many temporary objects; watch out for GC pressure in loops. Compare them with compareTo; BigDecimal’s equals also compares scale (2.0 != 2.00), so use compareTo == 0 to test numeric equality.
