Basic Syntax: Contrasted with C/C++

This chapter first gives you a “copy it and it just works” syntax overview, even with zero background, so you can write simple Java code right away; then it quickly levels the syntactic differences with C/C++ at the structural level and explains the trade-offs behind them. Anything that is the same as C/C++ ({} blocks, ; terminators, // and /* */ comments, + - * /, && || !, if/for/while) we won’t belabor; the focus is on the differences.

Get It Running First: A Java Syntax Overview

You already know how to compile and run from the last chapter. The syntax elements below you can just copy and use—no JVM/memory background required:

public class Basics {
    public static void main(String[] args) {
        // variable declaration and assignment
        int age = 20;
        double price = 9.9;
        String name = "Ada";
        boolean ok = true;

        // arithmetic
        int sum = age + 5;
        System.out.println(sum);                              // 25

        // string concatenation with +
        System.out.println("name=" + name + ", age=" + age);  // name=Ada, age=20

        // conditional branch
        if (age >= 18) {
            System.out.println("adult");
        } else {
            System.out.println("minor");
        }

        // loop
        for (int i = 0; i < 3; i++) {
            System.out.println("i=" + i);   // i=0  i=1  i=2
        }
    }
}

Comments use // for single-line and /* ... */ for multi-line, same as C. This is enough to start writing simple programs. Next comes where Java and C/C++ genuinely diverge.

Going Deeper: Structural Differences from C/C++

1. Everything Lives in a class; the Program Entry Point

C has free functions, and main is a global function. Java has no free functions and no global variables—all code must hang off some class (or interface/enum/record). The minimal runnable program:

public class App {
    // Program entry point. The signature must be public static void main(String[])
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}

Contrasted with C’s int main(int argc, char** argv): - No return value: main returns void. To set the process exit code, use System.exit(int) (the counterpart of C’s exit()). - The parameter is String[] args: only the actual arguments, not including the program name itself (C’s argv[0] is the program name, whereas in Java args[0] is already the first real argument). There is no need for argc either—arrays carry their own length. - static: when the JVM starts there are no object instances yet, so it can only call static methods that don’t depend on an instance; hence the entry point must be static. See the meaning of static in Object Orientation and the Type System.

Since JDK 21, a simpler entry point (omitting the class wrapper, with an instance main) has been available as a preview, and it became a standard feature in JDK 25 (JEP 512, no longer a preview); but for cross-version compatibility, most production projects still use the classic form above.

2. Packages and import: No Header Files, No Preprocessor

This is where Java differs most from C/C++—and also where you’ll feel most relieved.

The C/C++ model: #include does textual pasting, header files separate declaration from source-file definition, and the linker stitches them together; #define macros do textual substitution during preprocessing; this easily leads to duplicate inclusion, macro pollution, and translation-unit bloat.

Java’s model: - package: a namespace corresponding to the directory structure. package com.example.util; means the file lives under com/example/util/. - import: it merely tells the compiler which fully qualified name a simple name maps to; it performs no textual insertion and introduces no code. After import java.util.List; you can write List instead of java.util.List. Compilation and linking happen at the granularity of classes, resolved by the class loader at runtime (see Runtime Model).

package com.example;

import java.util.List;          // import a single class
import java.util.*;             // import a whole package (not recursive into sub-packages)
import static java.lang.Math.*; // static import: afterwards you can write sqrt(2) instead of Math.sqrt(2)

public class Demo { /* ... */ }

What’s absent (and why): - No header files: declaration and implementation are both in the same .class, which carries complete signature information that the compiler reads directly, so you don’t have to maintain a .h. - No preprocessor / macros: there is no #define, #ifdef, or conditional compilation. Constants use static final, “function macros” use ordinary methods (which get inlined by the JIT anyway, see Runtime Model), and platform differences are handled by runtime checks or build tools. Without macros, you also lose the whole class of bugs macros bring. - No forward declarations: classes can reference one another and form circular dependencies; the compiler resolves them in a single pass, so you needn’t worry about declaration order the way you do in C++.

3. Strong Static Typing: You Must Declare; var for Local Inference

How to Use It

Like C++, it is strongly, statically typed: every variable has a type determined at compile time, and a type mismatch is a compile error.

int count = 10;
double ratio = 0.5;
String name = "Ada";
long big = 9_000_000_000L;   // beyond int's range requires the L suffix

Since JDK 10, var supports local variable type inference, semantically equivalent to C++11’s auto:

var count = 10;              // inferred as int
var name = "Ada";            // inferred as String
var list = new ArrayList<String>();  // inferred as ArrayList<String>

Underlying Behavior, Contrasted with C

Key restrictions and clarifications: - var is only for local variables (inside methods, with an initializer). It cannot be used for fields, method parameters, or return types. C++’s auto has a wider scope of applicability. - It is not dynamic typing: the type is fixed at compile time; var just saves you from writing the type name by hand. There is no difference at runtime—the bytecode holds the concrete type. - You cannot write var x; without an initializer (nothing to infer from), nor var x = null; (null has no type).

4. Operators

How to Use Them

The vast majority are the same as C: arithmetic + - * / %, relational == != < > <= >=, logical && || !, bitwise & | ^ ~ << >>, compound assignment += etc., and increment/decrement ++ --—written exactly the same as in C, you can carry them straight over.

Underlying Behavior, Contrasted with C

The real points of difference:

  • Unsigned right shift >>>: Java has no unsigned types (see Data Types), so it adds a >>> for logical right shift (fill high bits with 0); >> is arithmetic right shift (fill with the sign bit). In C, whether the sign is filled depends on whether the type is unsigned; Java distinguishes them by operator.
  • + for string concatenation: "a" + 1 yields "a1". This is the only “overloaded” operator, and it is specially handled by the compiler (see the StringBuilder discussion in Data Types).
  • instanceof: runtime type checking (with a flavor of C++’s dynamic_cast), and it also supports pattern binding, see Control Flow.
  • Integer division by zero: 1/0 throws ArithmeticException (not UB!); but floating-point division by zero yields Infinity/NaN, consistent with IEEE754. In C, integer division by zero is undefined behavior.
  • Shift count taken modulo: when shifting an int, the shift amount is treated as & 31, and for long as & 63. 1 << 32 results in 1, not 0—don’t fall into this trap.
  • No comma operator; the comma in a for header is a syntactic special case.
  • Strict booleans: an if condition must be a boolean; in if (x), x cannot be an int. C’s “nonzero is true” does not hold in Java, which fundamentally rules out the if (a = b) typo.

Putting several of these traps together and actually running them:

// 1) shift count is taken modulo the width: int uses & 31
System.out.println(1 << 32);        // 1  (32 & 31 == 0, so 1<<0)
System.out.println(1 << 33);        // 2  (33 & 31 == 1)

// 2) >>> is logical (zero-fill), >> is arithmetic (sign-fill)
System.out.println(-8 >> 1);        // -4
System.out.println(-8 >>> 28);      // 15

// 3) integer divide-by-zero THROWS; float divide-by-zero does not
try { int x = 1 / 0; }
catch (ArithmeticException e) { System.out.println(e.getMessage()); } // / by zero
System.out.println(1.0 / 0);        // Infinity
System.out.println(0.0 / 0);        // NaN

// 4) with a String operand, '+' concatenates, evaluated left-to-right
System.out.println("sum=" + 1 + 2); // sum=12  (not sum=3)
System.out.println(1 + 2 + "=sum"); // 3=sum

5. Strings

How to Use Them

String is a class, not a char*, nor a char[]. Basic usage:

String s = "hello";
int n = s.length();               // not strlen; not \0-terminated
char c = s.charAt(0);
boolean same = s.equals("hello"); // compare content with equals, not ==!
String up = s.toUpperCase();      // returns a new string, s itself unchanged
String cat = s + " world";        // concatenation
String f = String.format("%d-%s", 42, "x");  // printf-like

Underlying Behavior, Contrasted with C

String is immutable, and its contents live on the heap (internally a byte[] + an encoding flag). Three things a C programmer must remember: 1. Not \0-terminated; the length is an explicit field, and length() is O(1). 2. Compare content with .equals(), not ==. == compares references (whether it’s the same object), which is the number-one beginner trap; Variables and the Memory Model explains this thoroughly at the memory level. 3. Immutable: any “modification” returns a new object. For frequent concatenation use StringBuilder (like a manually managed mutable buffer); details in Data Types.

Testing “immutable”—every “modifying” method returns a new string while the original stays untouched:

String s = "hello";
String up = s.toUpperCase();   // returns a NEW object
String rep = s.replace('l', 'L');
System.out.println(s);         // hello   <- original untouched
System.out.println(up);        // HELLO
System.out.println(rep);       // heLLo
System.out.println(s == up);   // false   <- up is a different object
System.out.println(s.length());// 5       <- length is an O(1) field, no \0 scan

6. Console IO

// Output
System.out.println("a full line with a newline");
System.out.print("no newline");
System.out.printf("%d and %.2f%n", 10, 3.14159);  // printf-like, %n is the platform newline

// Input
import java.util.Scanner;
Scanner sc = new Scanner(System.in);
int x = sc.nextInt();
String line = sc.nextLine();

System.out is a PrintStream object (corresponding to stdout), System.in is an InputStream (stdin), and System.err is stderr. There’s none of the type-unsafety of printf/scanf’s variadic format—printf in Java performs runtime checks and throws an exception on a type mismatch rather than silent UB.

7. Comments and Javadoc

// single line
/* multi line */

/**
 * A Javadoc doc comment, used with the javadoc tool to generate HTML API documentation.
 * @param n the input count
 * @return the computed result
 */
public int compute(int n) { return n * 2; }

Javadoc is like Doxygen built into the toolchain; tags such as @param/@return/@throws are conventions. IDEs also display hover documentation based on them.

8. Quick-Reference Table of Differences from C++

Dimension C++ Java
Inheritance multiple inheritance single class inheritance + multiple interface implementation (see Chapter 8)
Pointers has pointer syntax */&/-> no explicit pointer syntax; references are implicit managed pointers
Memory freeing delete/free manual none, GC reclaims automatically
Operator overloading supported not supported (except the built-in + for string concatenation)
Free functions supported none, everything in a class (use static methods)
Arrays contiguous memory on stack/heap, no bounds checking are objects, on the heap, carry length, have bounds checking
Header files required none, no preprocessor/macros
Default arguments supported not supported, use method overloading instead (see Chapter 6)
Value/reference semantics objects can have value semantics (copy constructor) or reference objects have reference semantics only; primitives have value semantics
Destruction RAII deterministic destruction no deterministic destruction; finalize is deprecated, resources use try-with-resources (see Chapter 9)
Objects on the stack Foo f; directly on the stack at the language level always new’d to the heap; HotSpot escape analysis may do scalar replacement (break the object into scalars placed in registers/stack-frame slots, rather than moving the whole object onto the stack)
const const modifier used broadly final (semantics not fully equivalent, see Chapter 3)

A few points to emphasize: - Arrays are objects: int[] a = new int[10]; lives on the heap, a.length is a field, and going out of bounds throws ArrayIndexOutOfBoundsException rather than trampling memory. This fundamentally eliminates the most dangerous class of C bug—array out-of-bounds—at the cost of a bounds check on each access (the JIT can often eliminate redundant checks inside loops). - No manual delete: once you’re done newing you leave it alone, and the GC handles reclamation. But non-memory resources (files, connections) still need explicit closing—see Chapter 9. - No operator overloading: a.equals(b), a.compareTo(b), bigInt.add(other) are all method calls; you cannot write them as a == b (that compares references) or a + b. This makes code more verbose but unambiguous.

Summary

  • At the usage level: variable declaration Type name = value;, if/for, string concatenation with +, output with System.out.println—copy the examples in “Get It Running First” and you can already write simple programs.
  • At the implementation level:
    • All code lives in classes, the entry point is public static void main(String[]), with no free functions and no globals.
    • Packages + import replace header files and the preprocessor; import inserts no code, it only resolves names.
    • Strong static typing requires declarations; var provides local type inference (fixed at compile time, not dynamic).
    • Operators are broadly the same as C; watch out for >>>, string +, division by zero throwing, and if requiring a boolean.
    • String is an immutable heap object; compare content with equals.
    • Relative to C++: single inheritance, no pointer syntax, no manual freeing, no operator overloading, and arrays are objects with bounds checking.
