Runtime Model: javac, Bytecode, the JVM, and JIT

As a C/C++ engineer, the compilation pipeline you know is: .c source goes through preprocessing, compilation, and assembly to produce an object file .o, which the linker then stitches into an executable (ELF/Mach-O/PE) targeting a specific CPU architecture + specific operating system. The CPU executes the machine code inside it directly.

Java takes a two-stage path: first it compiles to a platform-independent intermediate code (bytecode), which is then executed at runtime by a вҖңvirtual CPUвҖқ (the JVM). Understanding this pipeline is a prerequisite for understanding all the memory behavior that follows.

1. From Source to Bytecode: What javac Does

// Hello.java
public class Hello {
    public static void main(String[] args) {
        int sum = 0;
        for (int i = 0; i < 10; i++) {
            sum += i;
        }
        System.out.println(sum);
    }
}

Compile:

javac Hello.java     # produces Hello.class

The essential difference between javac and gcc: gccвҖҷs output targets a real CPU, with an instruction set of x86-64 / ARM64; javacвҖҷs output targets a hypothetical stack-based virtual machine, whose instruction set is вҖңbytecode.вҖқ Bytecode assumes nothing about the number of registers, the word size, or the byte orderвҖ”it targets only the abstract machine defined by the JVM specification. This is the technical foundation of вҖңwrite once, run anywhereвҖқ: a .class file is byte-for-byte identical on Windows/Linux/macOS and on x86/ARM.

A .class file is a binary container with a strict format (beginning with the magic number 0xCAFEBABE), containing:

Section Contents C/C++ analogy
constant pool symbolic references to strings, class names, method names, literals like .rodata + the symbol table, but more structured
field table the classвҖҷs field declarations struct member layout information
method table each methodвҖҷs bytecode + metadata function bodies, but with full type signatures
attribute table line number table, local variable table, annotations, etc. debug info (like DWARF), but built in

Key point: the .class file retains complete type and symbol information. After you strip a C program almost nothing but machine code remains, with symbols gone; a .class file inherently carries class names, method signatures, and field typesвҖ”this is the material basis on which reflection, dynamic loading, and runtime type checking can exist.

Viewing Bytecode with javap

javap is the disassembler that ships with the JDK, the Java worldвҖҷs equivalent of objdump -d:

javap -c Hello.class

The bytecode of the main method looks roughly like this (comments added by me):

0:  iconst_0        // push the constant 0 onto the operand stack
1:  istore_1        // pop the top of the stack into local variable slot 1 (i.e. sum)
2:  iconst_0        // push the constant 0
3:  istore_2        // store into slot 2 (i.e. i)
4:  iload_2         // load i onto the stack
5:  bipush 10       // push the constant 10
7:  if_icmpge 20    // if i >= 10 jump to offset 20 (exit the loop)
10: iload_1         // load sum
11: iload_2         // load i
12: iadd            // add the top two stack values, result left on the stack
13: istore_1        // store back into sum
14: iinc 2, 1       // increment slot 2 (i) by 1 directly, without going through the operand stack
17: goto 4          // back to the loop test
20: getstatic ...   // fetch System.out
23: iload_1
24: invokevirtual println
27: return

Notice that instructions like iconst/istore/iload/iadd have no register operands. The JVM is a stack machine: the inputs and outputs of operations flow across the вҖңoperand stack,вҖқ rather than naming registers as x86 does with add eax, ebx. The i prefix means the operation is on the int type; there are also l (long), f (float), d (double), and a (reference/address). iinc is one of the rare instructions that modifies the local variable table directly.

Every method declares in its bytecode how large an operand stack it needs (stack=) and how many local variable slots (locals=). These are fixed values computed at compile time, and the JVM allocates stack frames accordinglyвҖ”see Methods and the Call Stack.

2. Class Loading: Getting a .class into the JVM

When a C program starts, the loader maps the ELF segments into memory, does relocation, and binds dynamic-library symbolsвҖ”all at once. JavaвҖҷs class loading is lazy, on-demand, and happens at runtime, in three phases:

1. Loading

Based on a classвҖҷs fully qualified name, find the .class byte stream (from the filesystem, a jar, the network, or even generated at runtime), parse it into an internal data structure in the method area / metaspace, and create a java.lang.Class object on the heap to represent that class. This step is like dlopen opening a library, but at the granularity of a single class.

2. Linking (three sub-steps)

  • Verification: check that the bytecode is legal, that types are safe, and that the stack will not overflow/underflow. C/C++ has no counterpartвҖ”it trusts the machine code produced by the compiler; because the JVM may load bytecode of unknown origin, it must perform this layer of security checking.
  • Preparation: allocate space for static fields and set them to their zero values (intвҶ’0, referenceвҶ’null). Note that user-supplied assignments are not executed at this point.
  • Resolution: replace the symbolic references in the constant pool (вҖңmethod foo of class AвҖқ) with direct references (memory addresses/offsets). This is the equivalent of CвҖҷs link-time symbol resolution, but it happens at runtime and can be deferred until first use.

3. Initialization

Execute the static initializer blocks and the assignments to static fields (i.e.В the <clinit> method). Triggers include: the first new, accessing a static field/method, reflection, etc. The JVM guarantees <clinit> runs exactly once under multithreading (with a lock), which is why the вҖңstatic singleton + class loadingвҖқ idiom is thread-safe.

Parent Delegation Model

The JVM has multiple class loaders, forming a hierarchy:

        Bootstrap ClassLoader   (implemented in C++, loads the java.* core classes)
              в–І  delegate
        Platform ClassLoader    (JDK platform modules)
              в–І  delegate
        Application ClassLoader (your classpath)
              в–І  delegate
        Custom ClassLoader

When loading a class, a child loader first delegates upward to its parent to try; only if the parent cannot load it does the child load it itself. The benefit: core classes (such as java.lang.String) are always loaded by the Bootstrap loader, and you cannot substitute a forged java.lang.String for the system classвҖ”this is a guarantee of security and consistency. In C/C++, if two .so files each carry a symbol of the same name, the behavior depends on link order and is prone to problems; parent delegation mechanically prevents core classes from being вҖңhijacked.вҖқ

3. Execution: Interpreter + JIT Mixed Mode

Once it has the bytecode, how does the JVM execute it? It uses two means together (HotSpotвҖҷs default вҖңmixed modeвҖқ):

Interpretation

The JVM has a built-in bytecode interpreter that reads bytecode one instruction at a time, looks it up in a table, and executes the corresponding C++ logic. It starts fast (no waiting for compilation), but every instruction carries interpretation overhead, making it one to two orders of magnitude slower than native machine code.

Hot-Spot Detection + JIT Compilation

The name HotSpot comes from its strategy: detecting hot code (Hot Spots). The JVM maintains counters for each method (number of method invocations + number of loop back-edges), and once these exceed a threshold, it deems the method вҖңhotвҖқ and hands that bytecode to the JIT (Just-In-Time) compiler, which compiles it at runtime into native machine code; from then on the machine code executes directly.

HotSpot has two JIT compilers (Tiered Compilation uses them cooperatively):

Compiler Role Characteristics
C1 (Client) fast compilation few optimizations, fast to compile, steps in first
C2 (Server) deep optimization aggressive optimizations, slow to compile, used for the hottest code

Typical path: interpretation вҶ’ C1 compilation (with performance counters) вҶ’ C2 recompiles the hottest parts.

This is why Java вҖңgets faster the longer it runs,вҖқ and can sometimes approach or even locally exceed C: the JIT has runtime informationвҖ”which branch is actually taken, that a virtual method actually only ever calls one implementing class, what a loopвҖҷs bounds are. Based on this it can perform speculative optimizations that a static compiler cannot; if it bets wrong, it вҖңde-optimizesвҖқ (deoptimization) back to interpretation. gcc, by contrast, knows nothing about runtime at compile time and can only be conservative.

A Few Key JIT Optimizations (with Large Effects on Memory Behavior)

  • Inlining: embed the calleeвҖҷs method body directly at the call site, eliminating call overhead and opening up the horizon for subsequent optimizations. Equivalent to CвҖҷs inline, but the JIT decides automatically at runtime based on hotness.
  • Escape Analysis: analyze whether a reference to an object вҖңescapesвҖқ the current method (whether it is returned, stored into another object on the heap, or made visible to other threads). If it does not escape, the JVM can do:
    • Scalar Replacement: break apart an object that would otherwise be newвҖҷd on the heap into its individual fields, treating them as ordinary local variables placed directly into stack-frame slots or even registers. Note this is not вҖңmoving the whole object onto the stackвҖқ; rather, the object as a whole no longer exists at allвҖ”so there is neither heap allocation nor GC reclamation. (HotSpot has no true вҖңwhole-object stack allocationвҖқ; the mechanism that can save the heap allocation is scalar replacement.)

This point is important for understanding Java memory: at the language level вҖңobjects always live on the heap,вҖқ but at the HotSpot implementation level, escape analysis may keep short-lived, non-escaping objects out of the heap entirely. This closes part of the performance gap between Java and C (which puts structs on the stack). See Variables and the Memory Model.

A few contrasts with C: - Metaspace lives in native memory, not in the GC heap (this design dates from JDK 8, replacing the old вҖңPermGenвҖқ/permanent generation). - Heap reclamation is automatic: there is no free/delete; the GC reclaims what it determines to be unreachable. You almost never manage heap lifetimes manuallyвҖ”this is the biggest mental difference from C, and it is why in Error Handling and Resource Management, try-with-resources targets only вҖңnon-memory resourcesвҖқ (files/sockets): memory is left to the GC, but handles still need to be closed explicitly. - The semantics of the stack are almost identical to CвҖҷs: each method call pushes a stack frame, which is popped on return; if the stack gets too deep it triggers a StackOverflowError (the counterpart of CвҖҷs stack overflow, but the JVM throws an exception rather than segfaulting directly).

5. AOT: GraalVM native-image

Java can also, like C, be compiled Ahead-Of-Time (AOT) into a native executable. GraalVMвҖҷs native-image performs closure analysis at build time, statically compiling the classes and methods usedвҖ”along with a slimmed-down runtime (including a GC)вҖ”into a single standalone executable:

native-image Hello        # produces a native executable, hello
./hello                   # no JVM needed, millisecond startup

Costs and trade-offs: - Advantages: extremely fast startup, small memory footprint, no JVM to distributeвҖ”suitable for CLI tools and Serverless/function-compute cold starts. Here its distribution model closely resembles CвҖҷs: a single native binary. - Costs: the build must know all the classes that will be used, so reflection, dynamic proxies, and runtime-generated classes need extra configuration; you lose the JITвҖҷs runtime speculative optimizations, so peak throughput may be lower than C2вҖҷs after a long warm-up.

So, as a memory aid: - javac вҶ’ bytecode, cross-platform, relies on JVM + JIT runtime optimization (the mainstream for long-running services). - native-image вҶ’ native code, bound to a platform, fixed at build time (the choice for fast-starting, short-lived processes). - gcc/clang вҶ’ native code, bound to a platform, fixed at compile time (the only path for C/C++).

Summary

  • javac produces platform-independent bytecode, not machine code; executing it requires the JVM, a stack-based virtual machine.
  • Class loading is runtime, lazy, and layered (parent delegation), and retains complete type information.
  • HotSpot interprets first, then JIT-compiles after detecting hot spots (C1/C2, inlining, escape analysis); escape analysis can perform scalar replacement on non-escaping objects (breaking them into scalars placed in stack-frame slots/registers, rather than whole-object stack allocation).
  • Runtime memory is divided into a shared heap/metaspace and per-thread JVM stack/PC/native stack; the heap is reclaimed automatically by the GC.
  • When you need a C-style fast-starting native binary, use GraalVM native-image for AOT, at the cost of runtime dynamism and peak optimization.
