Error Handling and Resource Management: Exceptions and Garbage Collection

C++ uses exceptions for error propagation, RAII (destructors) for resource release, and manual delete/smart pointers for memory management. Java splits these three concerns into: the exception hierarchy, try-with-resources (an approximation of RAII), and automatic garbage collection (GC). This chapter explains how exceptions unwind the stack, how resources are released deterministically, and when, by whom, and by what mechanism heap objects are reclaimed.

The Exception Hierarchy

Everything throwable inherits from Throwable. It has two main branches:

                Throwable
               /         \
           Error        Exception
        (severe JVM-level faults) /       \
     OutOfMemoryError  RuntimeException  others (checked exceptions)
     StackOverflowError  (unchecked)          IOException
                          NullPointerException  SQLException
                          IllegalArgumentException ...
Category Checked? Semantics C++ analogy
Error No Unrecoverable at the JVM level (stack overflow, heap exhaustion); should not be caught Roughly system-level like std::bad_alloc
RuntimeException No Programming errors (null pointer, out of bounds, illegal argument); should be prevented by code Logic errors, like something an assert should have caught
Checked Exception Yes Anticipated external failures (IO, network); the compiler forces you to handle them No C++ equivalent; the deprecated throw() specification is a counterexample

Checked vs. unchecked is a compile-time constraint unique to Java: a checked exception must either be declared in the method signature’s throws clause or be handled locally with try/catch, or the code won’t compile. C++’s early dynamic exception specifications (throw(...)) were deprecated because they were hard to use; Java kept the feature, though it too has been much debated.

import java.io.*;
public class Checked {
    // Checked: must declare throws, otherwise a compile error
    static String readFirst(String path) throws IOException {
        try (BufferedReader r = new BufferedReader(new FileReader(path))) {
            return r.readLine();
        }
    }
    // Unchecked: no declaration needed, the compiler does not stop you
    static int div(int a, int b) { return a / b; } // b=0 throws ArithmeticException
}

try/catch/finally and Stack Unwinding

When an exception is thrown, the JVM starts from the current method and consults its exception table—javac generates one [start PC, end PC, handler PC, caught type] record per try block. If the current frame has no matching handler, that frame is popped (stack unwinding) and the search walks back up the call chain until a matching catch is found or the bottom of the thread stack is reached (at which point the thread terminates and prints a stack trace).

finally executes whether the outcome is normal or exceptional. At the bytecode level, the compiler copies the finally code onto every normal/exceptional exit path (or falls back to the exception table plus athrow); this is how its “always executes” guarantee is implemented.

public class Finally {
    static int tricky() {
        try {
            return 1;          // return value 1 is computed and pushed first
        } finally {
            return 2;          // the finally's return overrides it -> actually returns 2 (anti-pattern!)
        }
    }
    public static void main(String[] args) {
        System.out.println(tricky()); // 2
    }
}

Pitfall: a return or a thrown exception inside finally swallows the return value/exception from try. Never return inside finally.

try-with-resources: Java’s RAII

C++ releases resources deterministically via the destructor when an object leaves scope (RAII). The reclamation timing of a Java object is nondeterministic (GC decides), so you cannot rely on finalize to close a file. JDK 7 introduced try-with-resources: any object implementing AutoCloseable (with close()) is guaranteed to have close() called—in reverse of declaration order—when the try block ends. This is the closest thing Java has to deterministic RAII-style resource management.

import java.io.*;
public class TWR implements AutoCloseable {
    private final String name;
    TWR(String name) { this.name = name; System.out.println("open " + name); }
    @Override public void close() { System.out.println("close " + name); }

    public static void main(String[] args) {
        // Reverse-order close: close b first, then a (like C++ local objects destructed in reverse)
        try (TWR a = new TWR("A"); TWR b = new TWR("B")) {
            System.out.println("body");
            throw new RuntimeException("boom");
        } catch (RuntimeException e) {
            System.out.println("caught: " + e.getMessage());
            // An exception thrown by close() is attached to the primary exception as a "suppressed exception":
            for (Throwable s : e.getSuppressed()) System.out.println("suppressed: " + s);
        }
    }
    // Output: open A / open B / body / close B / close A / caught: boom
}

Difference from C++: the close() call site is instrumented by the compiler at the try exit, whereas the object’s memory itself is still reclaimed later by the GC. In other words, “deterministic release of resources (fd/handle)” and “nondeterministic reclamation of memory” are two separate things.

Multi-Catch and Exception Chaining

public class MultiCatch {
    static void handle(String s) {
        try {
            Integer.parseInt(s);
        } catch (NumberFormatException | NullPointerException e) {
            // Multi-catch: one catch handles multiple types; e is implicitly final, its type is the common supertype
            // Exception chaining: preserve the "cause" to make it easier to locate the original fault
            throw new IllegalStateException("handling failed: " + s, e);
        }
    }
    public static void main(String[] args) {
        try { handle(null); }
        catch (IllegalStateException e) {
            System.out.println(e.getMessage());
            System.out.println("cause: " + e.getCause()); // NullPointerException
        }
    }
}

Exception chaining (initCause / a constructor taking a cause) lets you translate an exception without losing the original stack trace; when printed, a Caused by: section appears.

Garbage Collection (GC)

In C++ you new/delete yourself, with full control over lifetimes. Java’s heap objects are reclaimed automatically by the GC; the price you pay is uncontrollable reclamation timing and occasional pauses (STW). Understanding GC is what lets you diagnose memory problems and tune for latency.

The Generational Hypothesis and Heap Layout

GC is built on the weak generational hypothesis: the vast majority of objects die young; objects that live long rarely point to young objects. So the heap is managed generationally:

                Java heap
 +---------------------------------------------------+
 | Young Gen                                         |
 |  +----------+  +----------+  +----------+         |
 |  |  Eden    |  | Survivor |  | Survivor |         |
 |  | (new obj)|  |   S0     |  |   S1     |         |
 |  +----------+  +----------+  +----------+         |
 +---------------------------------------------------+
 | Old Gen (Tenured)  long-lived objects             |
 +---------------------------------------------------+
   (Metaspace lives in native memory, holds class metadata, not in the heap)
  • New objects are allocated in Eden (pointer bumping, extremely fast, like advancing a stack top).
  • When Eden fills, a Minor GC is triggered: live objects are copied to one of the Survivor spaces and their age is incremented; each survived Minor GC raises the age, and once it exceeds the threshold (default 15, -XX:MaxTenuringThreshold) the object is promoted to the old generation. The two Survivor spaces take turns being the “copy target,” so one is always empty (a copying algorithm that avoids fragmentation).
  • When the old generation fills, a Major GC (old-generation collection) is triggered; a Full GC usually means whole-heap plus Metaspace collection—the slowest, and to be avoided. (Note: “Major GC” is classic terminology for generational collectors; modern collectors like G1/ZGC use different terms—G1 distinguishes young / mixed / full, where Full GC is its degraded fallback whole-heap collection.)
GC type Scope Trigger Pause
Minor GC Young gen Eden full Short
Major GC Old gen Old gen full Long
Full GC Whole heap + Metaspace Old gen/Metaspace insufficient, System.gc() Longest

Reachability Analysis and GC Roots

Java does not use reference counting (which cannot handle circular references); instead it uses reachability analysis: starting from a set of GC Roots, it traverses the reference graph, and any object that is unreachable is eligible for reclamation.

GC Roots include:

  • References in each thread’s stack-frame local variable table (objects referenced by currently executing methods);
  • Objects referenced by static fields;
  • Constant-pool references and JNI global references;
  • Objects held by synchronization locks, etc.
GC Roots
  | thread-stack locals  ---> A ---> B      (A, B reachable, alive)
  | static fields        ---> C
                          C <---> D    (C, D reference each other)
                                E ---> F  (E, F reference each other but no Root reaches them => both reclaimed)

Note the last group: E and F form a reference cycle. Reference counting would wrongly consider them alive, whereas reachability analysis correctly judges them to be garbage. This is exactly why Java does not use reference counting.

Collector Evolution and the STW Trade-off

STW (Stop-The-World): some GC phases must pause all application threads (which must reach a “safepoint”) to obtain a consistent snapshot of the heap. The evolution of collectors is essentially the history of continually shrinking STW.

Collector Traits STW Suited for
Serial Single-threaded, simple STW throughout, relatively long Small heaps, client-side
Parallel (throughput-first) Multithreaded parallel collection STW but parallel Batch processing, throughput-heavy
CMS (deprecated in JDK 9, removed in JDK 14) Concurrent mark-sweep, low-pause pioneer Partly concurrent Low-latency old gen (has fragmentation problems)
G1 (default since JDK 9) Region-partitioned, predictable pause target Short STW, mostly concurrent General-purpose large heaps
ZGC Colored pointers + read barriers Pauses <1ms, independent of heap size Very large heaps, low latency
Shenandoah Concurrent compaction Extremely short pauses Low latency

The core of the trade-off is throughput vs. latency: Parallel maximizes work done per unit time (good for background batch jobs); ZGC/Shenandoah sacrifice a bit of throughput for nearly imperceptible pauses (good for interactive/online services). There is no “best,” only “best fit for the scenario.”

Selection: -XX:+UseG1GC, -XX:+UseZGC, -XX:+UseParallelGC.

The Deprecation of finalize and the Cleaner

C++’s destructor runs deterministically. Java’s Object.finalize() once aimed to play this role, but it runs at an unpredictable time, may never run, slows down GC, and can even resurrect objects. It was marked @Deprecated in JDK 9 and further deprecated starting in JDK 18.

The replacement: java.lang.ref.Cleaner, which registers a cleanup action to be run by a background thread once the object becomes phantom-reachable. But it is still nondeterministic—always release critical resources explicitly with try-with-resources, and use Cleaner only as a last-resort fallback.

import java.lang.ref.Cleaner;
public class CleanerDemo implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();
    // The cleanup action must be a static/standalone class and must never hold a reference to the enclosing object, or the object can never be reclaimed
    static class State implements Runnable {
        public void run() { System.out.println("releasing native resource (fallback)"); }
    }
    private final Cleaner.Cleanable cleanable;
    CleanerDemo() { cleanable = CLEANER.register(this, new State()); }
    @Override public void close() { cleanable.clean(); } // explicit path: deterministic release
}

Reference Strength: Strong / Soft / Weak / Phantom

java.lang.ref provides four levels of reference strength, which determine an object’s “right to survive” in the eyes of the GC:

Strength Class GC behavior Typical use
Strong reference plain Object o = ... Never reclaimed while reachable Default
Soft reference SoftReference Reclaimed only when memory is low Memory-sensitive caches
Weak reference WeakReference Reclaimed at the next GC WeakHashMap, canonicalizing maps
Phantom reference PhantomReference Can never retrieve the object; only serves as a reclamation notification Replacing finalize for cleanup 
import java.lang.ref.*;
public class Refs {
    public static void main(String[] args) {
        ReferenceQueue<byte[]> queue = new ReferenceQueue<>();
        byte[] data = new byte[1024];
        // Weak reference + reference queue: after the object is reclaimed, the ref is enqueued, letting you detect reclamation
        WeakReference<byte[]> weak = new WeakReference<>(data, queue);
        data = null;          // drop the only strong reference
        System.gc();          // suggest a GC (not guaranteed to run immediately)
        System.out.println("get() after reclamation: " + weak.get()); // very likely null
        Reference<?> polled = queue.poll();                 // the reclaimed reference has been enqueued
        System.out.println("enqueued: " + (polled == weak));
    }
}

ReferenceQueue is the accompanying mechanism: when a referenced object is reclaimed, the JVM places the corresponding Reference object into the queue, and your cleanup thread polls it to run the follow-up work (this is exactly how Cleaner is implemented internally).

What a Memory Leak Looks Like in Java

Having GC does not mean there are no memory leaks. A leak = an object that is no longer logically useful but is still reachable from GC Roots, so GC does not reclaim it. Common forms:

  • Long-lived collections that only add, never remove: a static Map/cache is continually put into, and the keys and values stay reachable forever. Use WeakHashMap or cap the capacity (e.g., Guava/Caffeine caches).
  • Listeners/callbacks not deregistered: an observer registers but forgets to removeListener, and stays referenced by the subject.
  • ThreadLocal not removed: thread-pool threads live a long time, so the ThreadLocal’s value stays reachable for the life of the thread. Always remove() when done.
  • Inner classes implicitly holding the outer instance: a non-static inner class/anonymous class holds Outer.this, and if its lifetime is longer than the outer object’s, the outer object cannot be reclaimed (the same caution as in the Cleaner section).

Common OOMs and Tuning Parameters

OutOfMemoryError is an Error, subdivided into different messages:

Error message Meaning Direction
Java heap space The heap can’t hold it Increase -Xmx or hunt for a leak (heap-dump analysis)
GC overhead limit exceeded Spending 98%+ of time in GC but reclaiming very little Usually a leak or too small a heap
Metaspace Class metadata over the limit (lots of dynamically generated classes) -XX:MaxMetaspaceSize, check for classloader leaks
unable to create new native thread Thread count over the system/memory limit Reduce threads, tune -Xss
Direct buffer memory Off-heap direct memory exhausted -XX:MaxDirectMemorySize

Common parameters:

-Xms512m -Xmx4g                  # initial/max heap (in production, set them equal to avoid resize jitter)
-Xss1m                           # per-thread stack size
-XX:MaxMetaspaceSize=256m        # Metaspace upper bound
-XX:+UseG1GC -XX:MaxGCPauseMillis=200  # use G1 and set a pause target
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof  # dump the heap on OOM
-Xlog:gc*                        # print GC logs (unified logging in JDK 9+), for analysis

Diagnostic workflow: use -Xlog:gc* to observe GC frequency and pauses → grab the .hprof on OOM → use MAT/VisualVM to analyze the dominator tree and find who is holding a large object.

Summary: exceptions unwind up the call stack, and checked exceptions are a compile-time contract; try-with-resources provides near-RAII deterministic resource release, while memory reclamation is left to the GC; the GC judges liveness via reachability analysis (not reference counting) starting from GC Roots, managing memory generationally with copying/marking algorithms; the history of collectors is the history of shrinking STW; finalize is deprecated in favor of Cleaner + reference queues; a memory leak in Java shows up as “reachable but useless.” Master these, and you can command the JVM’s heap with the same confidence you manage C++ memory.
