Control Flow

Control flow is mostly identical to C, so this chapter focuses on the differences and the bytecode-level mechanics, rather than re-explaining the common knowledge of if/for/while. The highlights: the three forms of switch and their bytecode, the iterator behind the enhanced for, pattern matching, and text blocks.

1. if

Basically the same as C, with one hard difference: the condition must be a boolean (see “Basic Syntax”). Something like if (n) — “nonzero means true” — won’t compile in Java; you must write if (n != 0). The upside is that it eradicates the classic bug of accidentally writing if (a = b) instead of a comparison — inside if (a = b), if a = b isn’t of boolean type it’s a compile error outright.

At the bytecode level, if is just a conditional comparison instruction (if_icmpge, ifnull, ifeq, etc.) plus a jump, exactly as you saw in Chapter 1 — no surprises.

2. switch: three forms + bytecode

2.1 Classic switch: the fall-through trap is still there

Exactly like C: omitting break falls through to the next case.

int day = 3;
switch (day) {
    case 1:
    case 2:
    case 3:
        System.out.println("First half of the workweek");
        break;                       // omit this and it falls through — same pitfall as C
    case 6:
    case 7:
        System.out.println("Weekend");
        break;
    default:
        System.out.println("Other");
}

Types supported by the classic switch: integer types (and wrapper classes that can unbox to int), char, enum, and String (C doesn’t support switching on strings). Switching on a String is lowered by the compiler into a two-level check: “first compare hashCode to locate, then confirm with equals.”

2.2 Expression switch (->, JDK 14+): no fall-through, returns a value

The new syntax uses ->, has no fall-through and needs no break, and can also be evaluated as an expression:

int day = 3;
String kind = switch (day) {
    case 1, 2, 3, 4, 5 -> "Workday";      // multiple labels separated by commas
    case 6, 7          -> "Weekend";
    default            -> "Illegal";
};                                        // note: as an expression, it ends with a semicolon

// use a block + yield to return a value when you need multiple statements
int score = switch (grade) {
    case 'A' -> 100;
    case 'B' -> { int base = 80; yield base + 5; }  // yield hands out the block's value
    default  -> 0;
};

Key points: - A -> branch exits automatically once executed; there is no fall-through, syntactically killing the “forgot to write break” bug. - When used as an expression, the compiler requires you to cover all possibilities (all enum values for an enum, or provide a default); otherwise it’s a compile error — safer than C. - Multi-statement branches use a {} block and yield to hand out the value (return here would return from the whole method — don’t confuse the two).

2.3 Pattern-matching switch (JDK 21+): dispatch by type

A switch can match on type and bind a variable; combined with sealed types it enables exhaustiveness checking:

Object obj = ...;
String desc = switch (obj) {
    case Integer i when i > 0 -> "positive integer " + i;   // when is a guard condition
    case Integer i            -> "non-positive integer " + i;
    case String s             -> "string of length " + s.length();
    case null                 -> "null value";              // null can be matched explicitly
    default                   -> "other type";
};

This effectively collapses a chain of if (obj instanceof Integer i) ... else if (obj instanceof String s) ... into a structured form — a product of Java moving toward “algebraic data types + pattern matching.” A completely different level from C’s switch.

Here is the runnable full version; describe dispatches on several actual types:

public class PatternSwitch {
    static String describe(Object obj) {
        return switch (obj) {
            case Integer i when i > 0 -> "positive integer " + i;
            case Integer i            -> "non-positive integer " + i;
            case String s             -> "string of length " + s.length();
            case null                 -> "null value";
            default                   -> "other type";
        };
    }
    public static void main(String[] args) {
        System.out.println(describe(42));     // positive integer 42
        System.out.println(describe(-3));     // non-positive integer -3
        System.out.println(describe("hello")); // string of length 5
        System.out.println(describe(null));   // null value
        System.out.println(describe(3.14));   // other type
    }
}

Note that the order of cases matters: the guarded Integer i when i > 0 must come before the unguarded Integer i; if you put the unguarded one first, it would swallow all Integers and the compiler would reject the later case as “already covered, unreachable.”

2.4 Bytecode: tableswitch vs lookupswitch

A switch compiles to one of two instructions in bytecode, the compiler choosing automatically based on the distribution of case values — this is why a switch is often faster than an equivalent if-else chain:

Instruction When Mechanism Complexity
tableswitch case values are contiguous or nearly so uses the value as a jump-table index to jump directly O(1)
lookupswitch case values are sparse stores sorted (value, offset) pairs and does a binary search O(log n)

tableswitch is like the jump table a C compiler generates for dense cases: it encodes case 1/2/3 into a contiguous offset table, and at runtime target = table[value - low], done in a single indexed lookup with no per-case comparison. lookupswitch is used for sparse values like case 1/100/10000, storing sorted key-value pairs for a binary search. You can observe this with javap -c:

javap -c YourClass     // you'll see tableswitch { ... } or lookupswitch { ... }

Contiguous small-integer cases → tableswitch (fastest); sparse → lookupswitch. The takeaway: in performance-sensitive code, keeping case values as contiguous as possible favors generating a jump table.

Here is a concrete example contrasting the two instructions. dense has contiguous cases (0~3), sparse has sparse ones (1/100/10000):

public class SwitchByteCode {
    static int dense(int x) {   // cases 0,1,2,3 contiguous
        switch (x) {
            case 0: return 10;  case 1: return 11;
            case 2: return 12;  case 3: return 13;
            default: return -1;
        }
    }
    static int sparse(int x) {  // cases 1,100,10000 sparse
        switch (x) {
            case 1: return 10;  case 100: return 11;
            case 10000: return 12;
            default: return -1;
        }
    }
    public static void main(String[] args) {}
}

The key output of javac SwitchByteCode.java && javap -c SwitchByteCode.class (verified):

static int dense(int);
     0: iload_0
     1: tableswitch   { // 0 to 3
                   0: 32
                   1: 35
                   2: 38
                   3: 41
             default: 44
        }

static int sparse(int);
     0: iload_0
     1: lookupswitch  { // 3
                   1: 36
                 100: 39
               10000: 42
             default: 45
        }

dense compiles to tableswitch—a contiguous offset table for 0..3, jumped into by index at runtime; sparse compiles to lookupswitch—sorted key-value pairs searched with binary search. Same switch, but the distribution of case values decides which instruction is used.

3. Loops

for / while / do-while

All three have exactly the same semantics as C, so no need to belabor them:

for (int i = 0; i < n; i++) { ... }
while (cond) { ... }
do { ... } while (cond);   // executes at least once

The bytecode is just the loop test plus a goto back-edge (see the goto 4 in Chapter 1). The back-edge count is precisely one of the signals HotSpot uses to decide a loop is “hot” and trigger JIT compilation.

Enhanced for (for-each): based on Iterable/iterator

int[] arr = {1, 2, 3};
for (int x : arr) { ... }              // iterate an array

List<String> list = List.of("a","b");
for (String s : list) { ... }          // iterate a collection

The underlying mechanism depends on what’s being iterated: - Arrays: the compiler lowers this to a plain indexed for loop (controlled by length, accessed by index), with no iterator object and zero extra overhead. - Collections implementing Iterable: the compiler lowers this to calling iterator() to get an iterator, then looping over hasNext()/next(). The equivalent expansion:

// for (String s : list) {...} actually compiles into ↓
for (Iterator<String> it = list.iterator(); it.hasNext(); ) {
    String s = it.next();
    ...
}

Compared with C++’s range-based for (for (auto& x : container), which relies on begin()/end()), this maps almost one-to-one: Java relies on Iterable.iterator() + the Iterator interface. Note that when iterating a collection with an enhanced for, you cannot add or remove elements of that collection inside the loop body; otherwise the iterator detects the structural change and throws ConcurrentModificationException (to remove, use the iterator’s own remove() or use removeIf). Collections and iterators are covered in detail in “Composite Data Structures”.

4. Labeled break / continue

Java has no goto (it’s a reserved word but unused), but it provides labeled break/continue to break out of / continue an outer loop, covering the common C scenario of using goto to escape nested loops:

outer:
for (int i = 0; i < n; i++) {
    for (int j = 0; j < m; j++) {
        if (matrix[i][j] == target) {
            found = true;
            break outer;        // break straight out of both nested loops
        }
        if (skip) continue outer;  // jump straight to the outer loop's next iteration
    }
}

break outer ends the entire loop the label points to; continue outer jumps to that loop’s next iteration. More structured and readable than C’s flag variables or goto. At the bytecode level it’s still a goto to the corresponding offset.

5. The ternary operator

int max = (a > b) ? a : b;

Same as C. Note that in Java the two branches of a ternary expression must have compatible types (type promotion / boxing is applied), e.g. cond ? 1 : 2.0 is a double overall. Mixing wrapper classes with primitives can trigger auto-unboxing and cause an unexpected NPE — watch out.

6. instanceof pattern matching

The classic instanceof checks the type and then requires a manual cast; pattern matching (JDK 16+) does it in one step and binds a variable:

// classic style
if (obj instanceof String) {
    String s = (String) obj;    // still need an explicit cast
    System.out.println(s.length());
}

// pattern-matching style
if (obj instanceof String s) {  // on a successful test, obj is bound to s (already of type String)
    System.out.println(s.length());
}

// you can keep using the bound variable within the same condition
if (obj instanceof String s && s.length() > 3) { ... }

The scope of s is “wherever it’s guaranteed to be a String” (flow-sensitive scoping). This is the same direction of language evolution as the type system in Chapter 8 and the pattern-matching switch in this chapter. It’s essentially a smoother version of C++’s if (auto* p = dynamic_cast<String*>(obj)).

7. Text blocks (JDK 15+)

Multi-line string literals that spare you from hand-writing \n and concatenation, similar to a heredoc in other languages:

String json = """
        {
          "name": "Ada",
          "age": 36
        }
        """;                    // wrapped in triple quotes; the indentation baseline is fixed at compile time and common leading whitespace is stripped

Key points: - It’s still an ordinary String (immutable, interned into the constant pool); only the way you write it differs, and at runtime it’s no different from an ordinary string. - The compiler uses the indentation of the closing """ as the baseline and strips the common leading whitespace from each line, letting you align indentation in your code without polluting the content. - Great for embedding JSON / SQL / HTML. Compared with C, where you can only string together adjacent "..." literals or use trailing \, text blocks are far cleaner.

Summary

  • An if condition must be a boolean, ruling out if(a=b)-style slips.
  • switch has three forms: classic (with fall-through), -> expression (no fall-through, can return a value, requires exhaustiveness), and pattern matching (by type + when guards); both String and enum can be switched on.
  • In bytecode, tableswitch (contiguous values → jump table, O(1)) vs lookupswitch (sparse values → binary search, O(log n)), chosen automatically by the compiler; inspect with javap -c.
  • Enhanced for: arrays lower to indexed loops, collections rely on Iterable.iterator(); mutating the collection during iteration throws ConcurrentModificationException.
  • Labeled break/continue replace cross-level goto.
  • instanceof patterns test the type and bind at once; text blocks write multi-line strings.
