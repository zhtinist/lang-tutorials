Concurrency Model

In C/C++, concurrency relies on std::thread, pthread, atomics, and manually controlled memory_order. Java abstracts this whole stack into the JMM (Java Memory Model) and java.util.concurrent. This chapter again starts with “how to use it” — spinning up threads, adding locks, using thread pools; these APIs can be copied and used without understanding what’s underneath. Then it moves to “underlying implementation vs. C” — how the JVM maps threads onto the kernel, what the JMM actually constrains, how locks get upgraded inside the object header, and how JDK 21’s virtual threads work. If you just want things to run, read the first half of each section; if you want to understand why data races happen and why locking is needed, read the second half too.

Get It Working First: A Quick Look at Threads

Starting a thread and waiting for it to finish is the most basic move in Java concurrency:

public class SimpleThreadDoc {
    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            System.out.println("child thread running");
        });
        t.start();     // start a new thread, runs asynchronously
        t.join();      // block the current thread until t finishes
        System.out.println("child thread done, main thread continues");
    }
}
// child thread running
// child thread done, main thread continues

new Thread(Runnable) takes a lambda as the task the thread should run; start() actually hands it off to the system to run asynchronously; join() is how you “wait for it” — without it, the main thread might finish before the child thread does.

If two threads modify the same variable at the same time, doing it directly is wrong (the next section explains why in detail), but for now just remember the “fix” is simple: wrap the code that changes the variable in synchronized, so only one thread can enter at a time:

public class SimpleSyncDemo {
    static int counter = 0;

    static synchronized void increment() {   // synchronized method: only one thread can execute it at a time
        counter++;
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable job = () -> {
            for (int i = 0; i < 100_000; i++) increment();
        };
        Thread t1 = new Thread(job);
        Thread t2 = new Thread(job);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter); // 200000, same result every run
    }
}
// counter = 200000

Each thread increments counter 100,000 times; as long as increment() is synchronized, the result is always exactly 200,000, never less. Why “not adding it can lose updates,” and how synchronized is actually implemented underneath, is what comes next.

That’s enough to write correct little multithreaded programs. From here on we go underneath — first understanding “where the problem comes from,” then looking at how each tool solves it.

Going Deeper: Why Multithreading Goes Wrong

These next few sections cover why multiple threads reading and writing the same variable causes problems. No specific API syntax is involved — it’s pure concepts, no prerequisites needed. Once you understand them, why volatile, synchronized, and Atomic* exist will make sense naturally.

Thread Creation and Lifecycle

How to Use It

Java has three ways to write the “task” a thread should run:

// 1. Extend Thread (not recommended: single inheritance is used up, and task and thread are coupled)
class MyThread extends Thread {
    @Override public void run() { System.out.println("run in " + getName()); }
}

// 2. Runnable: no return value, no checked exceptions, the most commonly used
Runnable r = () -> System.out.println("runnable");

// 3. Callable<V>: returns a value V, can throw checked exceptions, used with Future (see the java.util.concurrent section)
Callable<Integer> c = () -> 42;

public class ThreadDemo {
    public static void main(String[] args) throws Exception {
        Thread t1 = new MyThread();
        t1.start();            // start() makes the JVM spin up a new thread to run run()
        Thread t2 = new Thread(r);
        t2.start();
        t1.join();             // the current thread blocks until t1 terminates, similar to pthread_join
        t2.join();
    }
}
// which thread prints first is undetermined, but join() guarantees main won't end until both are done

Key distinction: start() makes the JVM create a new thread and call back run() asynchronously; if you slip and call run() directly, that’s just an ordinary method call, and the code runs synchronously on the current thread — no new thread is created at all. A Thread object can only be start()ed once; calling it a second time throws IllegalThreadStateException.

Underlying Implementation vs. C

Java’s Thread on HotSpot is by default mapped 1:1 to a kernel thread (OS thread), just as heavyweight as a C++ std::thread. Calling start() triggers a pthread_create, and each thread gets about 1MB of stack space by default (tunable via -Xss). In other words, the system-level cost of new Thread(...).start() is the same order of magnitude as hand-writing pthread_create in C — threads are not free, and spinning up a few thousand platform threads will start to strain the system (this is exactly the problem “virtual threads,” covered at the end of this chapter, is meant to solve).

Thread.State has six states, which don’t map one-to-one onto OS thread states:

    NEW --start()--> RUNNABLE <----+
                      |  ^          | notify/timeout/lock acquired
    entering sync blk |  |          |
    but can't get lock|  |          |
                      v  |          |
                   BLOCKED          |
    wait()/park()   |               |
                      v             |
    WAITING / TIMED_WAITING --------+
                      |
                   TERMINATED (run returns)
  • RUNNABLE covers both “actually running on the CPU” and “ready and waiting to be scheduled” — Java doesn’t distinguish the two (only the OS does).
  • BLOCKED: waiting on a synchronized monitor lock.
  • WAITING: indefinite waits from Object.wait(), Thread.join(), LockSupport.park().
  • TIMED_WAITING: the same calls above but with a timeout, plus Thread.sleep(ms).

The Java Memory Model (JMM)

The JMM is a specification (JLS Chapter 17), not hardware. It defines, for multiple threads reading and writing shared variables, what values are legally visible — this is exactly the formal answer to “why things go wrong” from the previous section. It’s analogous to the C++11 memory model, but the JMM is older (2004, JSR-133) and stronger by default.

Main Memory and Working Memory

The JMM’s abstraction: each thread has its own working memory, and shared variables live in main memory. A thread operating on a variable must first copy it from main memory into working memory, modify it, then write it back.

   +----------------+        +----------------+
   |   Thread A     |        |   Thread B     |
   | working memory |        | working memory |
   | [copy of x=0]  |        | [copy of x=0]  |
   +-------+--------+        +--------+-------+
           |  read/write             |
           v                         v
   +-------------------------------------------+
   |            main memory  x = 0              |
   +-------------------------------------------+

This is an abstract model, not a description of real memory layout. In practice, “working memory” corresponds to CPU registers, store buffers, and the various cache levels. The core problem: A modifies x, but B won’t necessarily see it right away — that’s the visibility problem, rooted in caching plus compiler/CPU reordering.

Three Properties

Property Meaning Guaranteed by
Atomicity An operation is indivisible Basic reads/writes (except long/double) are naturally atomic; compound operations need synchronized/CAS
Visibility One thread’s write is visible to another volatile, synchronized, final
Ordering Instructions aren’t reordered in a way that breaks semantics volatile, synchronized, happens-before

The long/double gotcha: the JMM only guarantees atomicity for 32-bit reads/writes; a 64-bit variable on a 32-bit platform might get split into two separate writes, so a read could land on “half a value.” Adding volatile guarantees 64-bit atomicity. On a 64-bit JVM, HotSpot is actually atomic in practice, but the spec doesn’t guarantee it.

happens-before

The JMM’s core tool. If operation A happens-before B, then A’s result is visible to B, and A is ordered before B. The rules:

  1. Program order: within the same thread, an earlier operation happens-before a later one.
  2. Monitor lock: an unlock on a lock happens-before a subsequent lock on the same lock.
  3. volatile: a write to a volatile variable happens-before a subsequent read of it.
  4. Transitivity: A hb B, B hb C ⇒ A hb C.
  5. Thread start: t.start() happens-before any operation inside that thread.
  6. Thread termination: operations inside a thread happen-before another thread’s t.join() returning.
int a = 0;
volatile boolean flag = false;

// Thread 1
a = 1;              // (1)
flag = true;        // (2) volatile write

// Thread 2
if (flag) {         // (3) volatile read; if it reads true, then (2) hb (3)
    System.out.println(a); // must be 1: (1)hb(2)hb(3), transitively ⇒ (1)hb(4)
}

Without volatile, the write to a might not be visible to thread 2, or (1) and (2) could be reordered, so thread 2 sees flag==true but a==0.

Instruction Reordering

Compilers and CPUs reorder instructions for performance, as long as single-thread semantics (as-if-serial) are preserved. Under multithreading this becomes a problem. The classic case is double-checked locking (DCL):

public class Singleton {
    private static volatile Singleton instance; // volatile cannot be omitted
    public static Singleton getInstance() {
        if (instance == null) {                  // first check (lock-free)
            synchronized (Singleton.class) {
                if (instance == null) {          // second check
                    instance = new Singleton();  // broken down below
                }
            }
        }
        return instance;
    }
}

instance = new Singleton() is not atomic — it breaks down into three steps: (1) allocate memory (2) call the constructor to initialize (3) point instance at the memory. Without volatile, (2) and (3) might get reordered into (1)(3)(2), so another thread’s first check sees instance != null but gets an uninitialized object. volatile forbids this reordering (by inserting memory barriers) — this is why DCL only became correct starting with JDK 5.

volatile

How to Use It

The most common use of volatile is a “single-writer, multiple-reader status flag” — one thread writes, other threads read, and they’re guaranteed to see the latest value:

public class VolatileFlagDemo {
    static volatile boolean ready = false;
    static int data = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread writer = new Thread(() -> {
            data = 42;         // write the plain variable first
            ready = true;      // then write the volatile one, as a "publish" signal
        });
        Thread reader = new Thread(() -> {
            while (!ready) {   // busy-wait until writer's update is seen
                // spin
            }
            System.out.println("data = " + data); // guaranteed to see 42, not 0
        });
        reader.start();
        writer.start();
        writer.join();
        reader.join();
        System.out.println("done");
    }
}
// data = 42
// done

Without volatile, the reader thread could in theory never see ready become true (the compiler/JIT might optimize while (!ready) into reading a register just once), or it could see ready==true but data still 0 (reordering). With volatile added, neither problem can happen.

Underlying Implementation vs. C

volatile provides two guarantees, analogous to C++’s acquire/release semantics (though not exactly the same) — note that a single read or single write of a volatile variable is itself atomic (it even makes reads/writes of long/double atomic); what’s genuinely not atomic is a “read-modify-write” compound operation like count++:

  1. Visibility: a thread’s write to a volatile variable guarantees any thread that subsequently reads it sees the latest value. This is implemented via memory barriers (store/load fences) working with the CPU’s cache coherence protocol, and is not literally “every read/write bypasses the cache and hits main memory directly” (that’s an oversimplification from older textbooks).
  2. No reordering: operations before the write can’t be reordered after it, and operations after the read can’t be reordered before it (memory barriers).

It does not guarantee atomicity for compound operations:

volatile int count = 0;

void bump() {
    count++;  // read-modify-write in three steps; volatile can't stop two threads reading the same value then each adding 1, losing an update
}

If two threads each call count++ 100,000 times on the same volatile int count, the result is almost certainly less than 200,000 — volatile guarantees every read and write is visible, but that doesn’t stop the interleaving where “thread A reads 100, thread B also reads 100, both write back 101”:

count = 157682 (expected 200000, usually less)   // varies across runs, but always noticeably less than 200000

count++ needs synchronized or AtomicInteger to be atomic. volatile is a good fit for “single-writer, multiple-reader status flags,” but not for “a counter multiple threads all modify.”

synchronized and the Object Header

How to Use It

The earlier “Get It Working First” section already showed the most basic usage — put synchronized on a method, and only one thread at a time is allowed to run it. synchronized can also wrap a block of code and specify which object to lock on:

synchronized (lock) {        // monitorenter bytecode
    // critical section
}                            // monitorexit bytecode (the exception path also exits)

Locks are often paired with wait/notify: one thread waits until some condition is met, another thread changes the condition and then wakes it up. Both must be called inside a synchronized block:

public class WaitNotifyDemo {
    static final Object lock = new Object();
    static boolean condition = false;

    public static void main(String[] args) throws InterruptedException {
        Thread waiter = new Thread(() -> {
            synchronized (lock) {
                while (!condition) {                 // use while, not if, to guard against spurious wakeups
                    try { lock.wait(); } catch (InterruptedException e) { }
                }
                System.out.println("waiter: condition met");
            }
        });
        waiter.start();
        Thread.sleep(100); // makes waiter likely enter wait() first (doesn't affect correctness, just makes the demo clearer)
        synchronized (lock) {
            condition = true;
            lock.notifyAll();                        // wake the waiter (it still has to re-contend for the lock)
        }
        waiter.join();
        System.out.println("main: done");
    }
}
// waiter: condition met
// main: done

lock.wait() releases the lock and puts the current thread into a wait state, until another thread calls lock.notify()/notifyAll(), giving it a chance to wake up and re-contend for the lock — so wait/notify must always be paired on the same lock object.

Underlying Implementation vs. C

synchronized is a reentrant monitor lock. Every Java object can serve as a lock, with the lock state recorded in the mark word of the object header.

64-bit HotSpot object header layout (uncompressed, ordinary object):

Object memory layout:
+------------------+------------------+-------------+---------+
| Mark Word (8B)   | Klass Pointer(4/8B)| instance fields | padding |
+------------------+------------------+-------------+---------+

The Mark Word reuses the same 8 bytes to hold different information (lock flag bits decide how it’s interpreted):

Lock state Mark Word contents (partial) Flag bits
No lock hashCode, GC generational age 01
Biased lock ID of the thread holding it 01 (bias bit 1)
Lightweight lock Pointer to the Lock Record on the stack 00
Heavyweight lock Pointer to the monitor (ObjectMonitor) 10
GC mark — 11

Lock Upgrade

To avoid going heavyweight every time (an OS mutex, involving costly user-mode/kernel-mode switches), HotSpot introduces tiers:

no lock --first thread enters--> biased lock (records the thread ID, later same-thread access skips CAS)
        --another thread contends--> lightweight lock (spin + CAS, optimistic, good for short critical sections)
        --spin fails/heavy contention--> heavyweight lock (suspends threads, into the OS wait queue)
  • Biased lock: assumes the lock is always acquired by the same thread — records the thread ID the first time, and afterward, with no contention, no CAS is needed at all. Disabled by default since JDK 15 and slated for removal (high maintenance cost, little benefit for modern applications).
  • Lightweight lock: threads run in alternation (not truly contending at the same instant). A thread builds a Lock Record in its own stack frame and uses CAS to swap the mark word for a pointer to it. If that fails, it spins.
  • Heavyweight lock: genuine contention. It inflates into an ObjectMonitor; contending threads are suspended by the OS, analogous to pthread_mutex.

CAS and Atomic*

How to Use It

Atomic classes like AtomicInteger provide a set of operations that “look like ordinary methods but are atomic as a whole,” without needing to add your own locks:

import java.util.concurrent.atomic.AtomicInteger;

public class AtomicBasicDemo {
    public static void main(String[] args) {
        AtomicInteger ai = new AtomicInteger(0);
        System.out.println(ai.incrementAndGet()); // 1
        System.out.println(ai.incrementAndGet()); // 2
        boolean ok = ai.compareAndSet(2, 100);     // expects the current value to be 2, sets it to 100
        System.out.println(ok);                    // true
        System.out.println(ai.get());               // 100
    }
}
// 1
// 2
// true
// 100

Multiple threads calling incrementAndGet() concurrently won’t lose updates the way a plain int’s count++ would — CAS is what makes this work.

Underlying Implementation vs. C

CAS (Compare-And-Swap) is the cornerstone of lock-free concurrency, corresponding to x86’s cmpxchg instruction. Semantics: if (memoryValue == expected) { memoryValue = newValue; return true }, with the whole operation atomic in hardware.

ai.incrementAndGet();               // underlying CAS spin: do { old=get(); } while(!compareAndSet(old, old+1))
boolean ok = ai.compareAndSet(1, 2); // if expected is 1, set to 2

// JDK 8+ high-concurrency counter, segmented to reduce CAS contention (hot spot spread across multiple Cells)
LongAdder adder = new LongAdder();
adder.increment();
long sum = adder.sum();

Atomic*’s internal fields are implemented with volatile plus CAS via Unsafe/VarHandle.

The ABA problem: a value goes A → B → back to A, and CAS can’t detect it. Solved with AtomicStampedReference (adds a version number), analogous to a tagged pointer in a lock-free stack.

A runnable comparison: 8 threads each do count++ 100,000 times. The unprotected int loses updates because “read-modify-write” isn’t atomic, while AtomicInteger stays correct via CAS. join waits for all of them, so the result is deterministic:

import java.util.concurrent.atomic.AtomicInteger;

public class LostUpdate {
    static int plain = 0;                       // unprotected
    static final AtomicInteger atomic = new AtomicInteger(0);

    public static void main(String[] args) throws Exception {
        int threads = 8, perThread = 100_000;
        Runnable job = () -> {
            for (int i = 0; i < perThread; i++) {
                plain++;                         // read-modify-write, not atomic: loses updates
                atomic.incrementAndGet();        // CAS: atomic
            }
        };
        Thread[] ts = new Thread[threads];
        for (int i = 0; i < threads; i++) ts[i] = new Thread(job);
        for (Thread t : ts) t.start();
        for (Thread t : ts) t.join();            // wait for all, result is deterministic

        int expected = threads * perThread;      // 800000
        System.out.println("expected      = " + expected);
        System.out.println("atomic == exp : " + (atomic.get() == expected));
        System.out.println("plain  <  exp : " + (plain < expected));
    }
}
// expected      = 800000
// atomic == exp : true
// plain  <  exp : true    -- plain is almost certainly < 800000 (lost updates); atomic always equals the expected value

(This code has been run multiple times to verify: atomic == exp is true every time, plain < exp is true every time — the behavior is stable and reproducible.)

java.util.concurrent

ReentrantLock

How to Use It

ReentrantLock is the explicit version of synchronized: you call lock() and unlock() yourself, and unlock() must go in a finally block, or an exception path will leave the lock held forever:

import java.util.concurrent.locks.ReentrantLock;

public class LockBasicDemo {
    static final ReentrantLock lock = new ReentrantLock();
    static int counter = 0;

    static void increment() {
        lock.lock();
        try {
            counter++;
        } finally {
            lock.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable job = () -> { for (int i = 0; i < 100_000; i++) increment(); };
        Thread t1 = new Thread(job);
        Thread t2 = new Thread(job);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter); // 200000
    }
}
// counter = 200000

What it offers beyond synchronized is the ability to try acquiring a lock with a timeout, and go do something else if it fails, instead of waiting blindly:

if (lock.tryLock(1, TimeUnit.SECONDS)) {   // wait at most 1 second
    try {
        // critical section
    } finally {
        lock.unlock();
    }
} else {
    // didn't get the lock, do something else (e.g. give up, retry)
}

Underlying Implementation vs. C

ReentrantLock is more flexible than synchronized: interruptible, timeable, fair-mode capable, and supports multiple condition variables. It’s based on AQS (AbstractQueuedSynchronizer) — a volatile int state plus a CLH-variant wait queue.

new ReentrantLock(true) constructs a fair lock (queues requests in order — somewhat lower throughput, but no thread gets perpetually starved).

ConcurrentHashMap

How to Use It

In concurrent scenarios, don’t use a plain HashMap (it can corrupt data or even infinite-loop) — use ConcurrentHashMap instead, whose API works just like HashMap:

import java.util.concurrent.ConcurrentHashMap;

public class CHMBasicDemo {
    public static void main(String[] args) {
        ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
        map.put("a", 1);
        map.putIfAbsent("a", 99);         // "a" already exists, not overwritten
        map.computeIfAbsent("b", k -> 2); // "b" doesn't exist, computed and inserted
        System.out.println(map.get("a")); // 1
        System.out.println(map.get("b")); // 2
    }
}
// 1
// 2

Underlying Implementation vs. C

Rewritten in JDK 8: array + linked list/red-black tree, with concurrency control via CAS + synchronized (locking the bucket’s head node), rather than JDK 7’s segment locks (Segment).

  • Empty-bucket insertion: CAS puts it in directly, lock-free.
  • Non-empty bucket: synchronized locks that bucket’s head node — only that one bucket is locked, so concurrency = number of buckets.
  • When a chain’s length reaches ≥8 and the table capacity is ≥64, it converts to a red-black tree, dropping lookups from O(n) to O(log n).
  • size() uses baseCount + CounterCell[] (similar to LongAdder) to avoid a hot spot.

Reads are essentially lock-free (node val/next are volatile), and writes only lock a single bucket, giving throughput far above Collections.synchronizedMap (which locks the entire table).

The Thread Pool: ThreadPoolExecutor

How to Use It

Instead of new Thread for every task, thread pools are typically used to reuse threads. The simplest usage is to get an ExecutorService and repeatedly submit tasks to it:

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class PoolBasicDemo {
    public static void main(String[] args) throws Exception {
        ExecutorService es = Executors.newFixedThreadPool(2);
        Future<Integer> f = es.submit(() -> 1 + 1);
        System.out.println("result = " + f.get()); // 2, blocks to get the result
        es.shutdown();                              // remember to shut down the pool when done
    }
}
// result = 2

submit hands the task to the pool to run asynchronously and immediately returns a Future; f.get() blocks until the task finishes and gets its return value; shutdown() tells the pool to stop accepting new tasks, finish what’s already running, and then exit.

Underlying Implementation vs. C

Reusing threads avoids frequent creation (creating an OS thread is expensive). In production, it’s more common to build your own ThreadPoolExecutor with explicit parameters, rather than using Executors’s factory methods (explained below):

import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.Executors;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2,                              // corePoolSize: number of core threads, kept alive
    4,                              // maximumPoolSize: max thread count
    60L, TimeUnit.SECONDS,          // keepAliveTime: how long idle non-core threads survive
    new ArrayBlockingQueue<>(100),  // workQueue: task queue
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection policy
);

Task submission logic (make sure you understand this):

submit task
  ├─ running threads < core?      → create a new core thread to run it
  ├─ no, queue not full?          → enqueue and wait
  ├─ no, threads < max?           → create a new non-core thread to run it
  └─ no                           → trigger the rejection policy

Rejection policies: AbortPolicy (throws an exception, the default), CallerRunsPolicy (the submitter runs it itself, providing backpressure), DiscardPolicy, DiscardOldestPolicy.

Avoid Executors.newFixedThreadPool/newCachedThreadPool in production: the former has an unbounded queue (risk of OOM), and the latter has an unbounded thread count. Build a new ThreadPoolExecutor manually with explicit parameters instead.

Future and CompletableFuture

How to Use It

The Future returned by es.submit(...) in PoolBasicDemo above is the most basic placeholder for an async result — get() blocks to fetch it. CompletableFuture lets you chain multiple async steps together like building blocks, without needing to block on get() at every step:

import java.util.concurrent.CompletableFuture;

public class CFBasicDemo {
    public static void main(String[] args) {
        CompletableFuture<Integer> cf = CompletableFuture
            .supplyAsync(() -> 10)     // asynchronously produce a value
            .thenApply(x -> x * 2);    // transform the result, chaining a new CompletableFuture
        System.out.println(cf.join()); // 20, blocks until the whole chain finishes, then gets the value
    }
}
// 20

A more common chaining pattern also consumes the result and handles exceptions:

CompletableFuture
    .supplyAsync(() -> 10)                    // asynchronous production
    .thenApply(x -> x * 2)                    // transform (chained on the same thread)
    .thenCompose(x -> CompletableFuture.supplyAsync(() -> x + 1)) // flatten a nested future
    .thenAccept(System.out::println)          // consume, prints 21
    .exceptionally(ex -> { ex.printStackTrace(); return null; })
    .join(); // wait for the chain to finish -- the common pool uses daemon threads, so if main doesn't wait it may exit before the print happens

Underlying Implementation vs. C

Similar to a JS Promise or C++’s std::future plus continuations. Runnable: thenCombine runs two tasks concurrently and merges the results; exceptionally recovers from an exception in the chain, letting later steps continue:

import java.util.concurrent.CompletableFuture;

public class CFCompose {
    public static void main(String[] args) throws Exception {
        CompletableFuture<Integer> a = CompletableFuture.supplyAsync(() -> 3);
        CompletableFuture<Integer> b = CompletableFuture.supplyAsync(() -> 4);

        // combine: a and b run concurrently, then merge once both finish
        int hypotSq = a.thenCombine(b, (x, y) -> x * x + y * y).join(); // 25

        // chain + recover from exception
        String r = CompletableFuture.supplyAsync(() -> 10)
            .thenApply(x -> x / 0)                    // throws ArithmeticException
            .exceptionally(ex -> -1)                  // recover to -1
            .thenApply(x -> "result=" + x)
            .join();

        System.out.println("hypotSq = " + hypotSq);
        System.out.println(r);
    }
}
// hypotSq = 25
// result=-1

Deadlock and Its Avoidance

The four conditions for deadlock (same as in OS textbooks): mutual exclusion, hold-and-wait, no preemption, and circular wait. A classic example:

// Thread 1: locks A then B; Thread 2: locks B then A → circular wait
synchronized (A) { synchronized (B) { } }  // thread 1
synchronized (B) { synchronized (A) { } }  // thread 2

Avoidance techniques: - Lock ordering: all threads acquire locks in a fixed, global order (breaks circular wait). - tryLock with timeout: ReentrantLock.tryLock backs off and releases if it can’t acquire the lock. - Reduce lock granularity/hold time, going lock-free where possible (CAS, immutable objects). - Detection: jstack <pid> can report “Found one Java-level deadlock.”

Virtual Threads (Loom, JDK 21)

How to Use It

The platform threads seen so far (new Thread(...)) are bound 1:1 to kernel threads, and a few thousand of them already strains the system. Starting with JDK 21, you can use virtual threads, whose syntax is nearly identical to platform threads but which are extremely lightweight — you can spin up millions:

import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class VirtualBasicDemo {
    public static void main(String[] args) throws Exception {
        Thread vt = Thread.ofVirtual().start(() -> System.out.println("hello from virtual thread"));
        vt.join();
        System.out.println("is virtual: " + vt.isVirtual()); // true

        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            Future<Integer> f = executor.submit(() -> 1 + 1);
            System.out.println("result = " + f.get()); // 2
        }
    }
}
// hello from virtual thread
// is virtual: true
// result = 2

Thread.ofVirtual().start(...) directly spins up a virtual thread; Executors.newVirtualThreadPerTaskExecutor() is a thread pool that runs “each submitted task on a brand-new virtual thread” — submit/get work exactly the same as with an ordinary thread pool.

Underlying Implementation vs. C

Platform threads are bound 1:1 to kernel threads, which is expensive (stack ~1MB, context switches go through the kernel). Virtual threads are user-mode threads scheduled by the JVM (analogous to Go goroutines), extremely lightweight (stack grows on demand, lives on the heap), and you can spin up millions of them.

// One virtual thread per task; when it hits blocking IO it's automatically unmounted, occupying no kernel thread
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1)); // blocking doesn't occupy the carrier thread
            return null;
        });
    }
}

Runnable: spin up 100,000 virtual threads at once, each “blocking” for 10ms. Run on just a handful of platform threads (carriers), it still finishes in tens of milliseconds — the same number of platform threads simply couldn’t be created at all. Use CountDownLatch to wait for all of them to check in, and AtomicInteger to count:

import java.util.concurrent.CountDownLatch;
import java.util.concurrent.Executors;
import java.util.concurrent.atomic.AtomicInteger;

public class VThreads {
    public static void main(String[] args) throws Exception {
        int n = 100_000;
        AtomicInteger done = new AtomicInteger();
        CountDownLatch latch = new CountDownLatch(n);

        // one virtual thread per task, 100k of them total, each briefly blocking
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < n; i++) {
                executor.submit(() -> {
                    Thread.sleep(10);         // blocking unmounts the carrier rather than occupying a kernel thread
                    done.incrementAndGet();
                    latch.countDown();
                    return null;
                });
            }
            latch.await();                    // wait for every task to check in
        }
        System.out.println("tasks done      = " + done.get());
        System.out.println("current is virtual: " + Thread.currentThread().isVirtual());
    }
}
// tasks done      = 100000
// current is virtual: false   -- main is a platform thread; 100k virtual threads are scheduled onto a small number of carriers

(This example has been measured in practice: 100,000 virtual threads, each blocking for 10ms, finish in under 0.2 seconds total — if you swapped in 100,000 platform threads instead, the creation overhead and memory alone would bring down most machines.)

Mounting and Unmounting

While running, a virtual thread is mounted onto a platform thread (the carrier, drawn from a ForkJoinPool). When it hits a blocking operation (network IO, sleep, etc.) it gets unmounted: the virtual thread’s stack (a continuation) is saved to the heap, and the carrier thread is freed up to run other virtual threads. Once the blocking operation finishes, it mounts back onto some carrier and continues.

Platform thread pool (carriers, ~ number of CPU cores)
  Carrier-1: [VT-A running] --VT-A hits blocking IO--> unmount VT-A, mount VT-B
  Carrier-2: [VT-C running]
             heap: [VT-A stack snapshot awaiting wakeup] [VT-D stack snapshot] ...

Contrast: when a platform thread blocks, the entire kernel thread is suspended by the OS; when a virtual thread blocks, the kernel thread is freed to do other work. That’s why writing “one thread per request” in a synchronous style can still achieve the throughput of an asynchronous design.

Dimension Platform thread Virtual thread
Mapping 1:1 kernel thread M:N, scheduled by the JVM onto carriers
Order of magnitude Thousands Millions
Stack Fixed ~1MB On demand, grows/shrinks on the heap
Blocking cost Occupies a kernel thread Unmounts, freeing the carrier
Best suited for CPU-bound work IO-bound work, high concurrency

The pinning trap: if a virtual thread can’t unmount while blocked, it pins the carrier thread, undercutting the benefit. In JDK 21–23, blocking inside a synchronized block causes pinning, so the advice at the time was to switch IO-related critical sections to ReentrantLock; but starting with JDK 24 (JEP 491), pinning caused by synchronized has largely been eliminated, so there’s no longer a need to make that switch just for this reason. A few cases, like calling into native code (JNI), can still pin.

Summary

  • At the usage level: new Thread(() -> ...).start() starts a thread, join() waits for it to finish; if two threads modify the same variable, wrap it in synchronized, or use ReentrantLock/AtomicInteger; for batches of tasks, use a thread pool’s submit, then get results via Future/CompletableFuture; use ConcurrentHashMap instead of HashMap in concurrent scenarios; Thread.ofVirtual()/newVirtualThreadPerTaskExecutor() start virtual threads, whose syntax matches platform threads but which can scale to millions.
  • At the underlying level: the root cause of multithreading problems is the JMM abstraction’s “working memory vs. main memory” going out of sync — the three properties of visibility, ordering, and atomicity are guaranteed respectively by volatile/synchronized/happens-before and by CAS. synchronized’s lock state is recorded in the object header’s mark word, and it goes through an upgrade path from biased lock → lightweight lock → heavyweight lock; CAS is the hardware cornerstone of lock-free concurrency (though it has the ABA problem); ReentrantLock is built on AQS; ConcurrentHashMap raises concurrency to the bucket level via “CAS on empty buckets + locking just the head node on non-empty buckets”; for thread pools, understand the submission flow around core/max thread counts, the queue, and rejection policies; virtual threads are user-mode threads scheduled by the JVM, unmounting from the carrier when blocked instead of tying up a kernel thread, but you need to watch out for scenarios that cause pinning.
