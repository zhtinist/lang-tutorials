Concurrency Model

In C/C++, concurrency relies on std::thread, pthread, atomics, and manually controlled memory_order, with the memory model being C++11’s std::memory_order_*. Java abstracts this whole stack into the JMM (Java Memory Model) and java.util.concurrent. This chapter covers how threads run at the JVM level, what the JMM actually constrains, how locks are upgraded within the object header, and JDK 21’s virtual threads.

Thread Creation and Lifecycle

Java’s Thread on HotSpot is by default mapped 1:1 to a kernel thread (OS thread), as heavyweight as a C++ std::thread. Creating a platform thread calls pthread_create, with a default stack of about 1MB (-Xss tunable).

Three task carriers:

// 1. Extend Thread (not recommended: single inheritance is used up, and task and thread are coupled)
class MyThread extends Thread {
    @Override public void run() { System.out.println("run in " + getName()); }
}

// 2. Runnable: no return value, no checked exceptions
Runnable r = () -> System.out.println("runnable");

// 3. Callable<V>: returns a value V, can throw checked exceptions, used with Future
Callable<Integer> c = () -> 42;

public class ThreadDemo {
    public static void main(String[] args) throws Exception {
        Thread t1 = new MyThread();
        t1.start();            // start() triggers pthread_create; calling run() directly is just an ordinary method call, no thread is started
        Thread t2 = new Thread(r);
        t2.start();
        t1.join();             // the current thread blocks until t1 terminates, like pthread_join
        t2.join();
    }
}

Key distinction: start() makes the JVM create a new OS thread and call back run(); calling run() directly runs serially on the current thread. A Thread object can be start()ed only once; calling it again throws IllegalThreadStateException.

Thread State Machine

Thread.State has six states, which do not map one-to-one with OS states:

    NEW --start()--> RUNNABLE <----+
                      |  ^          | notify/timeout/lock acquired
    can't get lock    |  |          |
    entering sync blk |  |          |
                      v  |          |
                   BLOCKED          |
    wait()/park()   |               |
                      v             |
    WAITING / TIMED_WAITING --------+
                      |
                   TERMINATED (run returns)
  • RUNNABLE covers both “currently running on a CPU” and “ready, awaiting scheduling”; Java does not distinguish them (only the OS does).
  • BLOCKED: waiting for a synchronized monitor lock.
  • WAITING: indefinite waits in Object.wait(), Thread.join(), LockSupport.park().
  • TIMED_WAITING: the above calls with a timeout, and Thread.sleep(ms).

The Java Memory Model (JMM)

The JMM is a specification (JLS Chapter 17), not hardware. It defines, when multiple threads read and write shared variables, which values are legally visible. It is analogous to the C++11 memory model, but the JMM came earlier (2004, JSR-133) and is stronger by default.

Main Memory and Working Memory

The JMM abstracts: each thread has working memory, and shared variables live in main memory. A thread operating on a variable must first copy it from main memory into working memory, and after modifying it, write it back.

   +----------------+        +----------------+
   |   Thread A     |        |   Thread B     |
   | working memory |        | working memory |
   | [copy of x=0]  |        | [copy of x=0]  |
   +-------+--------+        +--------+-------+
           |  read/write             |
           v                         v
   +-------------------------------------------+
   |            main memory  x = 0             |
   +-------------------------------------------+

This is an abstract model and does not represent the real memory layout. In practice, “working memory” corresponds to CPU registers, store buffers, and the various cache levels. The core problem: A modifies x, but B does not necessarily see it immediately—this is the visibility problem, rooted in caching plus compiler/CPU reordering.

Three Properties

Property Meaning Guaranteed by
Atomicity The operation is indivisible Basic reads/writes (except long/double) are naturally atomic; compound operations need synchronized/CAS
Visibility One thread’s write is visible to another volatile, synchronized, final
Ordering Instructions are not reordered in a way that breaks semantics volatile, synchronized, happens-before

The long/double gotcha: the JMM only guarantees 32-bit reads/writes are atomic; a 64-bit variable on a 32-bit platform may be split into two writes, causing a read of “half a value.” Adding volatile guarantees 64-bit atomicity. On a 64-bit JVM, HotSpot is atomic in practice, but the spec does not guarantee it.

happens-before

The JMM’s core tool. If operation A happens-before B, then A’s result is visible to B, and A is ordered before B. The rules:

  1. Program order: within the same thread, an earlier operation happens-before a later one.
  2. Monitor lock: an unlock on a lock happens-before a subsequent lock on the same lock.
  3. volatile: a write to a volatile variable happens-before a subsequent read of it.
  4. Transitivity: A hb B, B hb C ⇒ A hb C.
  5. Thread start: t.start() happens-before any operation within the thread.
  6. Thread termination: operations within a thread happen-before another thread’s return from t.join().
int a = 0;
volatile boolean flag = false;

// Thread 1
a = 1;              // (1)
flag = true;        // (2) volatile write

// Thread 2
if (flag) {         // (3) volatile read; if it reads true then (2) hb (3)
    System.out.println(a); // necessarily 1: (1)hb(2)hb(3), transitively ⇒ (1)hb(4)
}

Without volatile, the write to a might not be visible to thread 2, or (1)(2) might be reordered so thread 2 sees flag==true but a==0.

Instruction Reordering

The compiler and CPU reorder instructions for performance, as long as single-thread semantics (as-if-serial) are preserved. Under multithreading this causes problems. The classic case is double-checked locking (DCL):

public class Singleton {
    private static volatile Singleton instance; // volatile cannot be omitted
    public static Singleton getInstance() {
        if (instance == null) {                  // first check (lock-free)
            synchronized (Singleton.class) {
                if (instance == null) {          // second check
                    instance = new Singleton();  // see the breakdown below
                }
            }
        }
        return instance;
    }
}

instance = new Singleton() is not atomic; it breaks into three steps: (1) allocate memory (2) call the constructor to initialize (3) point instance at the memory. Without volatile, (2)(3) may be reordered into (1)(3)(2), so another thread at the first check sees instance != null but obtains an uninitialized object. volatile forbids this reordering (by inserting memory barriers); this is why DCL is only correct after JDK 5.

volatile

volatile provides two guarantees, analogous to C++’s acquire/release semantics (but not exactly identical)—note that a single read or single write of a volatile variable is itself atomic (it even makes reads/writes of long/double atomic); what is genuinely not atomic is a “read-modify-write” compound operation like count++:

  1. Visibility: a thread’s write to a volatile variable guarantees that any thread that subsequently reads it sees the latest value. This is implemented via memory barriers (store/load fences) together with the CPU’s cache coherence protocol, and not literally by “every read/write bypassing the cache and going straight to main memory” (that is a simplification from older textbooks).
  2. No reordering: operations before the write cannot be reordered after it, and operations after the read cannot be reordered before it (memory barriers).

It does not guarantee atomicity of compound operations:

volatile int count = 0;
count++;  // read-modify-write in three steps; volatile can't stop two threads reading the same value then each +1, losing an update

count++ requires synchronized or AtomicInteger. volatile suits “single-writer, multiple-reader status flags.”

synchronized and the Object Header

synchronized is a reentrant monitor lock. Every Java object can serve as a lock, and the lock state is recorded in the mark word of the object header.

64-bit HotSpot object header layout (uncompressed, ordinary object):

Object memory layout:
+------------------+------------------+-------------+---------+
| Mark Word (8B)   | Klass Pointer(4/8B)| instance fields | padding |
+------------------+------------------+-------------+---------+

The Mark Word reuses the same 8 bytes to store different information (the lock flag bits decide how it is interpreted):

Lock state Mark Word contents (partial) Flag bits
No lock hashCode, GC generational age 01
Biased lock Holding thread ID 01 (bias bit 1)
Lightweight lock Pointer to Lock Record on the stack 00
Heavyweight lock Pointer to the monitor (ObjectMonitor) 10
GC mark — 11

Lock Upgrade

To avoid going heavyweight every time (an OS mutex, involving costly user/kernel mode switches), HotSpot introduces tiers:

no lock --first thread enters--> biased lock (records thread ID, subsequent same-thread accesses skip CAS)
        --another thread contends--> lightweight lock (spin + CAS, optimistic, good for short critical sections)
        --spin fails/heavy contention--> heavyweight lock (suspend threads, enter OS wait queue)
  • Biased lock: assumes the lock is always acquired by the same thread; records the thread ID on the first acquisition, and afterward, with no contention, needs no CAS. Disabled by default since JDK 15 and slated for removal (high maintenance cost, small benefit for modern applications).
  • Lightweight lock: threads execute in alternation (not contending simultaneously). A thread builds a Lock Record in its own stack frame and uses CAS to swap the mark word for a pointer to it. On failure it spins.
  • Heavyweight lock: genuine contention. It inflates into an ObjectMonitor; contending threads are suspended by the OS, analogous to pthread_mutex.
synchronized (lock) {        // monitorenter bytecode
    // critical section
}                            // monitorexit bytecode (the exceptional path also exits)

wait/notify must be inside synchronized (you must hold the monitor):

synchronized (lock) {
    while (!condition) lock.wait();  // release the lock and enter WAITING; must use while to guard against spurious wakeups
    // ...
}
synchronized (lock) {
    condition = true;
    lock.notifyAll();               // wake waiters (they must still re-contend for the lock)
}

CAS and Atomic*

CAS (Compare-And-Swap) is the cornerstone of lock-free concurrency, corresponding to x86’s cmpxchg instruction. Semantics: if (memoryValue==expected) { memoryValue=newValue; return true }, with the whole operation atomic in hardware.

AtomicInteger ai = new AtomicInteger(0);
ai.incrementAndGet();               // underlying CAS spin: do { old=get(); } while(!compareAndSet(old, old+1))
boolean ok = ai.compareAndSet(1, 2); // if expected 1 then set 2

// JDK 8+ high-concurrency counter, segmented to reduce CAS contention (the hot spot is spread across multiple Cells)
LongAdder adder = new LongAdder();
adder.increment();
long sum = adder.sum();

The internal fields of Atomic* are implemented with volatile plus CAS via Unsafe/VarHandle.

The ABA problem: a value goes A → B → back to A, and CAS can’t detect it. Solve it with AtomicStampedReference (add a version number), analogous to a tagged pointer in a lock-free stack.

java.util.concurrent

ReentrantLock

More flexible than synchronized: interruptible, timed, fair-mode capable, and supports multiple condition variables. Based on AQS (AbstractQueuedSynchronizer)—a volatile int state plus a CLH-variant wait queue.

ReentrantLock lock = new ReentrantLock(); // new ReentrantLock(true) for a fair lock
lock.lock();
try {
    // critical section
} finally {
    lock.unlock();   // must be in finally, or the lock leaks on an exception (synchronized is released automatically by the JVM)
}

// Timed, to avoid deadlock
if (lock.tryLock(1, TimeUnit.SECONDS)) {
    try { /* ... */ } finally { lock.unlock(); }
}

ConcurrentHashMap

Rewritten in JDK 8: array + linked list/red-black tree, with concurrency control via CAS + synchronized (locking the bucket head node), rather than JDK 7’s segment locks (Segment).

  • Empty-bucket insertion: CAS puts it directly, lock-free.
  • Non-empty bucket: synchronized locks that bucket’s head node, locking only one bucket, so concurrency = number of buckets.
  • When a chain length ≥8 and the table capacity ≥64, it converts to a red-black tree, lowering lookup from O(n) to O(log n).
  • size() uses baseCount + CounterCell[] (like LongAdder) to avoid a hot spot.

Reads are largely lock-free (node val/next are volatile), and writes lock only a single bucket, giving throughput far above Collections.synchronizedMap (which locks the whole table).

The Thread Pool ThreadPoolExecutor

Reuses threads to avoid frequent creation (creating OS threads is expensive). Core constructor parameters:

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2,                              // corePoolSize: core thread count, always kept
    4,                              // maximumPoolSize: max thread count
    60L, TimeUnit.SECONDS,          // keepAliveTime: idle survival time for non-core threads
    new ArrayBlockingQueue<>(100),  // workQueue: task queue
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection policy
);

Task submission logic (be sure to understand it):

submit task
  ├─ running threads < core?  → create a core thread to run it
  ├─ else, queue not full?    → enqueue and wait
  ├─ else, threads < max?     → create a non-core thread to run it
  └─ else                     → trigger the rejection policy

Rejection policies: AbortPolicy (throws an exception, the default), CallerRunsPolicy (the submitter runs it itself, backpressure), DiscardPolicy, DiscardOldestPolicy.

In production, avoid Executors.newFixedThreadPool/newCachedThreadPool: the former has an unbounded queue (possible OOM), and the latter has an unbounded thread count. Manually new ThreadPoolExecutor with explicit parameters.

Future and CompletableFuture

ExecutorService es = Executors.newFixedThreadPool(2);
Future<Integer> f = es.submit(() -> 42);
Integer v = f.get();   // blocks to get the result (may throw ExecutionException wrapping the task's exception)

// CompletableFuture: composable, non-blocking callbacks, like a JS Promise / C++ std::future + continuation
CompletableFuture
    .supplyAsync(() -> 10)                    // async production
    .thenApply(x -> x * 2)                    // transform (chained on the same thread)
    .thenCompose(x -> CompletableFuture.supplyAsync(() -> x + 1)) // flatten a nested future
    .thenAccept(System.out::println)          // consume
    .exceptionally(ex -> { ex.printStackTrace(); return null; });

CompletableFuture<Void> all = CompletableFuture.allOf(f1, f2); // wait for all to complete
es.shutdown();

Coordination Tools

// CountDownLatch: one-shot, wait for N events to complete (not resettable), like a one-way barrier
CountDownLatch latch = new CountDownLatch(3);
// worker: latch.countDown();
latch.await();   // the main thread waits for the count to reach zero

// Semaphore: throttling / limiting concurrency
Semaphore sem = new Semaphore(2);   // at most 2 enter simultaneously
sem.acquire();
try { /* limited resource */ } finally { sem.release(); }

// CyclicBarrier: reusable; N threads wait for each other to rendezvous, then continue together

Deadlock and Its Avoidance

The four conditions for deadlock (consistent with OS textbooks): mutual exclusion, hold-and-wait, no preemption, and circular wait. A classic:

// Thread 1: lock A then B; Thread 2: lock B then A → circular wait
synchronized (A) { synchronized (B) { } }  // thread 1
synchronized (B) { synchronized (A) { } }  // thread 2

Avoidance techniques: - Lock ordering: all threads acquire locks in a fixed global order (breaks circular wait). - tryLock with timeout: ReentrantLock.tryLock backs off and releases if it can’t acquire. - Reduce lock granularity/hold time, and go lock-free where possible (CAS, immutable objects). - Detection: jstack <pid> can report “Found one Java-level deadlock”.

Virtual Threads (Loom, JDK 21)

Platform threads are 1:1 bound to kernel threads, expensive (stack ~1MB, context switches go through the kernel), and even a few thousand strain the system. Virtual threads are user-mode threads scheduled by the JVM (like Go goroutines), extremely light (the stack grows on demand and lives on the heap), and you can spawn millions.

// One virtual thread per task; on blocking IO it is automatically unmounted and occupies no kernel thread
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1)); // blocking does not occupy the carrier thread
            return null;
        });
    }
}

Thread vt = Thread.ofVirtual().start(() -> System.out.println("virtual"));
Thread pt = Thread.ofPlatform().start(() -> System.out.println("platform"));

Mounting and Unmounting

While a virtual thread runs, it is mounted onto a platform thread (the carrier, from a ForkJoinPool). On a blocking operation (e.g., network IO, sleep) it unmounts: the virtual thread’s stack (continuation) is saved to the heap, and the carrier thread is freed to run other virtual threads. When the blocking ends, it mounts onto some carrier again and continues.

Platform thread pool (carriers, ~CPU-core count)
  Carrier-1: [VT-A running] --VT-A hits IO block--> unmount VT-A, mount VT-B
  Carrier-2: [VT-C running]
             heap: [VT-A stack snapshot awaiting wakeup] [VT-D stack snapshot] ...

Contrast: when a platform thread blocks, the whole kernel thread is suspended by the OS; when a virtual thread blocks, the kernel thread is freed to do work. That is why the “thread-per-request” synchronous coding style can achieve asynchronous throughput.

Dimension Platform thread Virtual thread
Mapping 1:1 kernel thread M:N, scheduled by the JVM onto carriers
Order of magnitude Thousands Millions
Stack Fixed ~1MB On demand, grows/shrinks on the heap
Blocking cost Occupies a kernel thread Unmounts, freeing the carrier
Suited for CPU-bound IO-bound, high concurrency

The pinning pitfall: if a virtual thread cannot unmount while blocking, it pins the carrier thread, weakening the advantage. In JDK 21–23, blocking inside a synchronized block causes pinning, so the advice at the time was to switch IO-related critical sections to ReentrantLock; but since JDK 24 (JEP 491), pinning caused by synchronized has been largely eliminated, so you no longer need to deliberately make that switch. A few cases such as calling native code (JNI) can still pin.
