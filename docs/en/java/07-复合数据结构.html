Internal Implementation of the Collections Framework

In C++ you use std::vector, std::list, std::unordered_map, std::map. Java’s collections framework (the Java Collections Framework, JCF) corresponds to these one-to-one conceptually, but the internal implementation details, memory layout, growth strategy, and iterator semantics all deserve to be pried apart one by one. The core of this chapter: what each container actually lays out on the heap, whether it copies values or references, and where the complexity comes from.

For the basics of “reference vs value,” see “Variables and the Memory Model”; for object-header layout, see “Data Types and Memory Layout”.

1. Arrays: the bedrock of everything

A Java array is a true object, allocated on the heap, unlike C where it’s just a chunk of raw memory + pointer arithmetic.

int[] a = new int[4];        // heap allocation: object header + length field + 4 contiguous ints
String[] s = new String[3];  // on the heap: 3 contiguous "reference slots", initially null

Memory layout (HotSpot, 64-bit, compressed pointers)

int[4] on the heap:
┌──────────┬──────────┬────────┬────┬────┬────┬────┐
│ mark word│ klass ptr│ length │ [0]│ [1]│ [2]│ [3]│
│  8 bytes │  4 bytes │ 4 bytes│int │int │int │int │
└──────────┴──────────┴────────┴────┴────┴────┴────┘
       object header: 16 bytes total    elements stored contiguously, cache-friendly

The key differences from C:

Feature C/C++ array Java array
Nature raw memory, a[i] = *(a+i) heap object, includes a length field
Length compile-time constant or stored separately a.length carried along
Bounds checking none, out-of-bounds is UB checked on every access, throws ArrayIndexOutOfBoundsException
Storing objects can store object values (contiguous) can only store references (Object[] is an array of references)
Covariance none covariant: Object[] o = new String[3]; is legal

The pitfall of array covariance

Java’s array covariance means String[] IS-A Object[]; this passes at compile time, but every write at runtime requires a type check:

Object[] arr = new String[3];
arr[0] = "ok";           // fine
arr[1] = Integer.valueOf(1); // compiles! throws ArrayStoreException at runtime

C++ templates are invariant; vector<string> has nothing to do with vector<Object>. This Java design is historical baggage (before generics existed, it let Arrays.sort(Object[]) work), at the cost of a runtime store check. Generic collections (below) are, by contrast, invariant.

Bounds-check cost and its elimination

Every a[i] inserts a 0 <= i < a.length check. The JIT has Range Check Elimination: in a loop like for (int i=0;i<a.length;i++) that’s provably safe, C2 hoists the check out of the loop or removes it, bringing performance close to C. So there’s a cost at the specification level, but it’s often optimized away at the implementation level.

3. ArrayList vs LinkedList

ArrayList: a dynamic array

Internally it’s just an Object[] elementData + an int size:

// simplified JDK source structure
transient Object[] elementData; // the array that actually stores elements
private int size;               // logical size (<= elementData.length capacity)

Memory diagram:

ArrayList object (heap)     elementData array (heap)
┌───────────────┐          ┌────┬────┬────┬────┬────┬────┐
│ elementData ──┼────────▶ │ e0 │ e1 │ e2 │null│null│null│
│ size = 3      │          └────┴────┴────┴────┴────┴────┘
└───────────────┘           capacity = 6, size = 3

Note that what’s stored are references (Object[]), with the objects themselves scattered across the heap — unlike C++ vector<Point>, which inlines the Point values contiguously. So iterating a Java ArrayList<Point> means “contiguous array → jumping to scattered heap locations one by one,” with worse cache locality than a value-semantics C++ vector. To store primitives contiguously, you need a native int[] or a third-party primitive collection.

Growth: default capacity 10; when full it grows by 1.5× (newCap = oldCap + (oldCap >> 1)), with Arrays.copyOf allocating a new array and copying the references (a shallow copy — only the pointer values are copied, not the objects themselves).

size=10 is full, adding the 11th:
new capacity = 10 + 5 = 15, allocate Object[15], copy 10 references over, old array awaits GC

Compare with C++ std::vector’s common 2× growth (libstdc++ uses 2x, MSVC uses 1.5x). Amortized O(1) append.

Peeking at the real length of the underlying elementData array via reflection shows this 1.5× growth curve directly (capacity is an implementation detail, so reflection needs --add-opens):

import java.lang.reflect.*;
import java.util.*;
public class GrowDemo {
    static int cap(ArrayList<?> list) throws Exception {
        Field f = ArrayList.class.getDeclaredField("elementData");
        f.setAccessible(true);
        return ((Object[]) f.get(list)).length;
    }
    public static void main(String[] args) throws Exception {
        ArrayList<Integer> list = new ArrayList<>();
        System.out.println("initial capacity = " + cap(list));
        int last = -1;
        for (int i = 0; i < 40; i++) {
            list.add(i);
            int c = cap(list);
            if (c != last) { System.out.println("size=" + list.size() + " -> capacity=" + c); last = c; }
        }
    }
}
// Run: java --add-opens java.base/java.util=ALL-UNNAMED GrowDemo.java
// initial capacity = 0     <- empty ctor allocates lazily; the length-10 array is built on first add
// size=1  -> capacity=10
// size=11 -> capacity=15   <- 10 + (10>>1) = 15
// size=16 -> capacity=22   <- 15 + (15>>1) = 22
// size=23 -> capacity=33
// size=34 -> capacity=49

LinkedList: a doubly linked list

private static class Node<E> {
    E item;
    Node<E> next;
    Node<E> prev;
}

Each node is an independent heap object: 16-byte object header + 3 references (item/next/prev, 4 bytes each with compressed pointers) ≈ 32 bytes/node, a huge overhead to store one element. Memory diagram:

first ─▶┌────┬────┬────┐   ┌────┬────┬────┐   ┌────┬────┬────┐◀─ last
        │prev│item│next┼──▶│prev│item│next┼──▶│prev│item│next│
        │null│ e0 │  ●─┘   │ ●  │ e1 │ ●─┘   │ ●  │ e2 │null│
        └────┴────┴────┘◀──┴────┴────┴────┘◀──┴────┴────┴────┘

Corresponds to C++ std::list. Nodes are scattered across the heap, so traversal is cache-unfriendly. get(i) is O(n) (walking from whichever end is closer). In practice ArrayList is almost always better; even for frequent head insertion, ArrayDeque is faster than LinkedList. LinkedList’s only advantage is O(1) removal when you hold a node reference, but the collection API generally doesn’t give you the node.

Complexity comparison

Operation ArrayList LinkedList
get(i) O(1) O(n)
add (tail) amortized O(1) O(1)
add(i, e) (middle) O(n) (shift) O(n) (locate) + O(1)
Head insertion O(n) O(1)
contains O(n) O(n)
Memory/element 1 reference + capacity slack ~32-byte node

4. HashMap: the main event

Corresponds to C++ std::unordered_map, but the implementation details differ greatly.

Structure: a bucket array + linked lists/red-black trees

transient Node<K,V>[] table;  // the bucket array, length always a power of 2
static class Node<K,V> {
    final int hash;   // the cached, perturbed hash
    final K key;
    V value;
    Node<K,V> next;   // linked-list pointer (chaining to resolve collisions)
}

Memory diagram (chaining on collision):

table (Node[16])
┌────┐
│ 0  │──▶ null
│ 1  │──▶ [h,k1,v1] ──▶ [h,k9,v9] ──▶ null   (bucket 1 has two colliding keys)
│ 2  │──▶ null
│... │
│ 5  │──▶ [h,k5,v5] ──▶ null
└────┘

The hash-perturbation function

The bucket index = hash & (n-1) (n being a power of 2, equivalent to modulo but faster). If you used key.hashCode() directly, the high bits wouldn’t participate (because & (n-1) keeps only the low bits), so keys differing in the high bits but equal in the low bits would collide. Perturbation XORs the high 16 bits into the low bits:

static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

treeify when collisions intensify

When a bucket’s chain length is ≥ 8 and the table length is ≥ 64, the chain is converted into a red-black tree, dropping that bucket’s worst-case lookup from O(n) to O(log n) and guarding against hash-collision attacks (DoS). When the tree’s node count falls back to ≤ 6, it degrades back into a chain (the 6/8 hysteresis interval avoids frequent conversion).

chain length >= 8 and table.length >= 64:
bucket ──▶ chain ─treeify─▶ red-black tree (TreeNode)
if the table hasn't reached 64 yet, prefer resizing over treeifying

Load factor and resize rehashing

  • Load factor 0.75: when size > capacity * 0.75, resize to double. 0.75 is a space/time trade-off (per Poisson-distribution analysis, balancing collision probability against memory waste).
  • Resize rehashing: allocate a bucket array 2× larger and rehash every element. Because the capacity is a power of 2 and doubles, an element’s new position is one of only two: original position or original position + old capacity. JDK 8 exploits this property to avoid recomputing the hash, looking only at whether that one bit hash & oldCap is 0 or 1, splitting the old chain into two chains (lo/hi) placed at the two new-table positions. O(n) but with a small constant.
old table.length=16, resizing to 32:
a bucket's chain is split by (hash & 16) into two → bucket j and bucket j+16

The key contract: equals/hashCode

A HashMap locates in two steps: first find the bucket by hash, then compare within the bucket one by one using equals. So a custom key must correctly override both equals and hashCode; otherwise equal objects scattered into different buckets can never be found again. See the equals/hashCode contract in “Object Orientation and the Type System”. Same in C++: unordered_map needs a std::hash specialization + operator==.

Complexity

Operation Average Worst (after treeify)
get/put/remove O(1) O(log n)
Traversal O(capacity + size) same

Note: HashMap is not thread-safe; under concurrency, resizing in JDK 7 could form a cycle and loop forever (JDK 8 fixed this but data can still be lost). For concurrency, use ConcurrentHashMap, see “Concurrency Model”.

5. LinkedHashMap / TreeMap

LinkedHashMap

On top of HashMap’s Node it adds before/after doubly-linked pointers, maintaining insertion order (or access order with accessOrder=true, which can implement an LRU cache). Hash lookups are still O(1); it additionally maintains a linked list threading through all nodes.

HashMap's bucket structure + a doubly-linked list threading through all nodes (recording order)
head ──▶ node ──▶ node ──▶ node ──▶ tail

TreeMap

Based on a red-black tree (a self-balancing BST), corresponding to C++ std::map. Sorted by key (natural order or a Comparator). All operations are O(log n), providing ordered views like firstKey/lastKey/floorKey/ceilingKey/subMap. Keys must be comparable (Comparable, or pass a Comparator), see “Standard Library Algorithms”.

The iteration order of the three Maps makes the difference plain at a glance: HashMap is determined by bucket position (unspecified, looks “random”), LinkedHashMap preserves insertion order, and TreeMap sorts by key:

import java.util.*;
public class MapOrder {
    static void fill(Map<String,Integer> m) {
        for (String k : new String[]{"banana","apple","cherry","date"}) m.put(k, k.length());
    }
    public static void main(String[] args) {
        Map<String,Integer> hash = new HashMap<>();
        Map<String,Integer> linked = new LinkedHashMap<>();
        Map<String,Integer> tree = new TreeMap<>();
        fill(hash); fill(linked); fill(tree);
        System.out.println("HashMap:       " + hash.keySet());
        System.out.println("LinkedHashMap: " + linked.keySet());
        System.out.println("TreeMap:       " + tree.keySet());
    }
}
// HashMap:       [banana, date, apple, cherry]   <- bucket order, unspecified (never rely on it)
// LinkedHashMap: [banana, apple, cherry, date]   <- insertion order
// TreeMap:       [apple, banana, cherry, date]   <- lexicographic order

6. HashSet / ArrayDeque

  • HashSet: internally just a HashMap, with the element as the key and the value being a shared dummy Object. So everything about HashSet equals the keySet of HashMap. Corresponds to C++ unordered_set.
  • LinkedHashSet: internally a LinkedHashMap, preserving insertion order.
  • TreeSet: internally a TreeMap, sorted, corresponding to std::set.
  • ArrayDeque: a double-ended queue implemented with a circular array; the two indices head/tail move around a ring buffer, doubling on growth. As a stack or a queue it’s faster than both Stack (obsolete, extends Vector, carries synchronization) and LinkedList. The preferred stack/queue implementation. It disallows null (null is used to mark an empty slot).
ArrayDeque circular array:
┌───┬───┬───┬───┬───┬───┐
│ e │ e │   │   │ e │ e │   head=4, tail=2 (wraps around)
└───┴───┴───┴───┴───┴───┘

7. Generic collections and the impact of type erasure

ArrayList<String> is just ArrayList at runtime, with an Object[] inside; generics are only checked at compile time, with casts inserted automatically. This is why you can’t new T[], and why a collection always stores references (primitives get auto-boxed). The erasure mechanism is detailed in “Generics and Type Erasure”.

Consequence: each element of an ArrayList<Integer> is an Integer object on the heap + a reference in the array, so the memory and boxing/unboxing overhead is far greater than C++ vector<int>. In performance-sensitive code, use native arrays or third-party primitive collections (Eclipse Collections, fastutil).

8. Iterators and fail-fast

A collection internally maintains a modCount (a count of structural modifications). When an iterator is created it records expectedModCount, and validates it on each next(); if the collection is modified through a non-iterator path in the meantime, it throws ConcurrentModificationException.

List<Integer> list = new ArrayList<>(List.of(1, 2, 3));
for (Integer x : list) {      // the enhanced for uses an Iterator under the hood
    if (x == 2) list.remove(x); // structural modification → modCount changes → the next next() throws
}
// correct approach: use iterator.remove() or list.removeIf(x -> x == 2)

Running it for real shows the exception is indeed thrown, while removeIf is perfectly safe:

import java.util.*;
public class FailFast {
    public static void main(String[] args) {
        List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4));
        try {
            for (Integer x : list) {          // the enhanced for uses an Iterator
                if (x == 2) list.remove(x);   // structural change bumps modCount
            }
        } catch (ConcurrentModificationException e) {
            System.out.println("caught: " + e.getClass().getSimpleName());
        }
        list.removeIf(x -> x == 2);           // correct: goes through the iterator, no CME
        System.out.println("after removeIf: " + list);
    }
}
// caught: ConcurrentModificationException
// after removeIf: [1, 3, 4]

fail-fast is best-effort error detection (not guaranteed to trigger), not concurrency protection; for real concurrency use CopyOnWriteArrayList/ConcurrentHashMap (weakly consistent iterators that don’t throw this exception). The C++ STL counterpart is “iterator invalidation” — but the STL leaves it UB, whereas Java proactively throws to help you catch it early.
