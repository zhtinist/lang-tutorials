Data Types and Memory Layout

This chapter makes clear how much memory each type occupies and what it looks like, contrasting with C throughout. Building on the two-camp division from Variables and the Memory Model, here we go deep into bit widths and the byte-level layout of objects on the heap.

1. The 8 Primitive Types: Fixed-Width, Cross-Platform, No unsigned

C’s integer widths are implementation-defined: int might be 16/32/64 bits, long differs across platforms (LP64 vs LLP64), and you have to rely on <stdint.h>’s int32_t to be sure. Java writes the widths into the language specification, consistent across all platforms:

Type Bit width Range Approx. C equivalent Default
byte 8 -128 ~ 127 int8_t 0
short 16 -32768 ~ 32767 int16_t 0
int 32 ±2.1e9 int32_t 0
long 64 ±9.2e18 int64_t 0L
float 32 IEEE754 single precision float 0.0f
double 64 IEEE754 double precision double 0.0d
char 16 UTF-16 code unit 0~65535 none (wchar_t is not a reliable match) U+0000 (NUL)
boolean unspecified true/false none (C uses int/_Bool) false

Three Java features you must remember:

  1. No unsigned. All integer types are signed (char is the only unsigned 16-bit quantity, but semantically it is a character, not a number). When you need unsigned arithmetic, use a wider type or static methods like Integer.toUnsignedLong, Integer.divideUnsigned, etc. For logical right shift use >>> (see Basic Syntax).
  2. char is a 16-bit UTF-16 code unit, not C’s 8-bit char. A single char cannot hold all of Unicode (supplementary-plane characters need two chars, i.e. a surrogate pair). To handle the semantics of “one character,” use int code-point APIs. String encoding is discussed below.
  3. boolean size is unspecified. The JVM spec does not define how many bits a boolean occupies; in HotSpot a single boolean variable is usually treated as an int, and each element of a boolean[] array takes 1 byte. Don’t count on bit packing (for a bitmap use long + bitwise ops or BitSet).

Literals and suffixes are similar to C: 10L (long), 3.14f (float), 0x1F (hexadecimal), 0b1010 (binary), 017 (octal, leading 0). Java 7+ allows underscore separators to improve readability:

long   creditCard = 1234_5678_9012_3456L;
int    million     = 1_000_000;
int    bytes       = 0b1101_0010;
double pi          = 3.141_592_653;

Floating point follows IEEE754: there is Double.NaN, Double.POSITIVE_INFINITY; 0.0/0.0 yields NaN, 1.0/0.0 yields Infinity (unlike integer division by zero, which throws). The NaN != NaN being true pitfall is the same as in C.

2. Wrapper Classes, Autoboxing, and the Integer Cache

Each primitive type has a corresponding wrapper class—wrapping the value into a heap object, so it can be placed into containers that accept only objects (generic collections cannot hold int, only Integer, see Composite Data Structures):

Primitive Wrapper class
int Integer
long Long
double Double
char Character
boolean Boolean
… (byte/short/float likewise)

Autoboxing / unboxing: the compiler automatically converts between primitives and wrapper classes.

Integer boxed = 42;          // autoboxing: the compiler inserts Integer.valueOf(42)
int prim = boxed;            // auto-unboxing: the compiler inserts boxed.intValue()
List<Integer> list = new ArrayList<>();
list.add(7);                 // boxing: int 7 -> an Integer object put on the heap
int v = list.get(0);         // unboxing

The cost (note for the performance-sensitive): boxing has to new an object on the heap, with a reference pointing to it—far more expensive than an int on the stack, and it creates GC pressure. For intensive numeric computation don’t use List<Integer>; use int[] or IntStream (see Standard Library Algorithms).

The Integer Cache [-128, 127] and the == Trap

Integer.valueOf(int) returns the same cached object for values -128 to 127 (the upper bound is adjustable via -XX:AutoBoxCacheMax); outside the range it creates a new object each time. Hence:

Integer a = 127, b = 127;
System.out.println(a == b);      // true  —— cache hit, same object
Integer c = 128, d = 128;
System.out.println(c == d);      // false —— each newly created, different objects
System.out.println(c.equals(d)); // true  —— compares values, correct

Combined with the == semantics from Chapter 3: == compares references, the cache makes small integers happen to share an object so it’s true, while large integers don’t. Conclusion: to compare the values of wrapper classes, always use equals or unbox to primitives first. Unboxing also carries an NPE risk: Integer x = null; int y = x; throws NPE at runtime.

3. Object Memory Layout (HotSpot)

This is the part C programmers care about most: what does a Java object actually look like on the heap? In HotSpot, an object consists of three parts:

┌────────────────────────────────────────────────┐
│ Object Header                                    │
│  ┌──────────────────────────────────────────┐  │
│  │ Mark Word   (usually 8 bytes on 64-bit JVM)│ │  hash code, GC generational age,
│  │                                            │  │  lock state / biased-lock pointer, etc.
│  ├──────────────────────────────────────────┤  │
│  │ Klass pointer (points to class metadata    │  │  4 bytes when compressed pointers are on
│  │  in method area/metaspace)                 │  │  ≈ one of the roles of C++'s vptr
│  └──────────────────────────────────────────┘  │
│  [ if an array, also a length field (4 bytes) ] │
├────────────────────────────────────────────────┤
│ Instance Data                                    │  fields laid out by type,
│  int / long / reference ...                      │  HotSpot reorders to save padding
├────────────────────────────────────────────────┤
│ Padding                                          │  padded to a multiple of 8 bytes
└────────────────────────────────────────────────┘

Contrasted with C++: C++’s struct/class has no Mark Word; when there are virtual functions there is a vptr pointing to the vtable. Java’s Klass pointer doubles as “finding the class metadata and method table” (virtual method dispatch is in Methods and the Call Stack), while the Mark Word is something C++ has nothing of at all—it carries the object’s hash, GC age, and the state of the built-in lock (synchronized).

Alignment exists just like in C’s struct: HotSpot aligns objects to 8 bytes by default, padding the tail. There may also be padding between fields. The difference: in C fields are laid out strictly in declaration order (unless you use pragma pack), whereas HotSpot reorders field order on its own to minimize padding (placing wide types first, grouping references together for easy GC scanning); you cannot assume declaration order equals memory order.

An empty object (no fields) under 64-bit + compressed pointers is usually 16 bytes (8 Mark + 4 Klass + 4 padding).

CompressedOops

Mentioned earlier in Chapter 3: under 64-bit a reference would take 8 bytes, which wastes space and hurts the cache. HotSpot by default enables Compressed Ordinary Object Pointers for heaps smaller than 32 GB: it encodes references into 32 bits, and on access, since objects are 8-byte aligned, it left-shifts by 3 to restore the real address, so 4 bytes can address 2^32 × 8 = 32 GB. The effect: reference fields and Klass pointers each save half the space, improving cache hit rate. Only when the heap exceeds 32 GB does it fall back to 8-byte references (hence the tuning wisdom of “don’t set the heap to just over 32 GB”). This is a pure implementation detail with no counterpart in C (a C pointer is just a raw address).

4. Arrays Are Objects

Java arrays are objects on the heap, not C’s “a stretch of raw contiguous memory + a base address.”

int[] a = new int[5];      // allocate an array object on the heap
System.out.println(a.length);   // 5, length is a built-in field, not a method
a[10] = 1;                 // ArrayIndexOutOfBoundsException (not trampling memory!)

Memory layout: array object = object header (including an extra length field) + a contiguous element region + padding.

int[5] on the heap:
┌──────────┬────────┬──────────────────────────────┐
│ header   │ length=5│ [0][1][2][3][4] contiguous int elements │
└──────────┴────────┴──────────────────────────────┘

Key contrasts: - Elements stored contiguously: an int[]’s element region is 5 contiguous ints—the same as C, cache-friendly. - With bounds checking: on each a[i] access the JVM verifies 0 <= i < length, throwing an exception on out-of-bounds rather than undefined behavior. The JIT can often prove indices in a loop are safe and thus eliminate the bounds checks, so hot loops don’t have the overhead you’d imagine. - Reference arrays store references: the element region of String[] / Object[] is a row of references, each pointing to the real object on the heap (possibly null)—not inlined objects. This differs from C’s struct T arr[] (values inlined), and is more like T* arr[] (an array of pointers). Java currently has no value-inlined contiguous object array (Project Valhalla’s value classes may provide this in the future). - Multidimensional arrays are “arrays of arrays”: int[][] is an array whose elements are “int[] references,” rows can have unequal lengths (jagged arrays), and it is not guaranteed to be contiguous overall in memory. C’s int m[3][4] is a truly contiguous row-major block—an important difference, and for numeric computation seeking locality one often switches to a one-dimensional int[] with manual indexing.

5. String: Immutable + Constant Pool + intern

String is a reference type, an immutable object. Internally (JDK 9+’s Compact Strings) it stores characters in a byte[] + a coder flag marking the encoding (Latin-1 or UTF-16)—as long as the code points of all characters in the whole string are ≤ 0xFF (representable in Latin-1, including Latin extensions and common Western accented letters), each character takes 1 byte; once any character with a code point > 0xFF appears (Chinese, emoji, etc.), the whole string switches to UTF-16, 2 bytes per character. It is not \0-terminated; the length is an explicit field.

What Immutability Means

Any “modifying” method returns a new object, leaving the original string untouched:

String s = "abc";
s.concat("d");            // returns the new string "abcd", but it's not received, s is still "abc"
s = s.toUpperCase();      // "ABC" is a new object, s is repointed to it, the old "abc" awaits reclamation

Benefits of immutability: safe to share (no synchronization needed across threads), the hash can be cached, and it’s safe to use as a Map key. The cost: concatenation produces lots of intermediate objects—which is why you use StringBuilder.

The String Constant Pool and intern

String literals ("hello" in source) are placed by the JVM into a string constant pool (driven by the runtime constant pool, located in the heap). Identical literals share the same object:

String a = "hi";
String b = "hi";
System.out.println(a == b);          // true! the same object in the pool

String c = new String("hi");         // new forces another new object on the heap
System.out.println(a == c);          // false! different objects
System.out.println(a == c.intern()); // true! intern() returns the canonical object in the pool

Memory diagram:

Constant pool (in heap)   Ordinary heap
[ "hi" object P ]◀──a      [ "hi" object Q ]◀── c (new'd)
       ▲────── b
c.intern() looks up the pool and returns P; so a == c.intern() is true
  • == being true for literals is only because they share the pool object—don’t rely on it; always compare content with equals (echoing Chapter 3).
  • intern() registers/looks up a string in the pool and returns the canonical instance. For lots of duplicate strings this can save memory, but overuse bloats the pool, so use it with care.
  • Constant concatenation determinable at compile time ("a" + "b") is folded by javac directly into the "ab" literal and interned.

StringBuilder: A Mutable Buffer

When you need frequent concatenation or to build strings in a loop, use StringBuilder (not thread-safe, use it single-threaded; the thread-safe version is StringBuffer):

StringBuilder sb = new StringBuilder();
for (int i = 0; i < 100; i++) {
    sb.append(i).append(',');   // modifies the internal char/byte buffer in place, building no intermediate String
}
String result = sb.toString();  // generate the String once at the end

Internally it maintains a growable byte array (like a manually realloc’d buffer in C), append appends in place, and it only grows the capacity when it isn’t enough (usually doubling). The compiler automatically optimizes concatenations like "a" + b + "c": JDK 9+ (JEP 280) compiles them by default to invokedynamic + StringConcatFactory (visible with javap -c; only JDK 8 and earlier used an explicit StringBuilder). But += in a loop is an independent concatenation each iteration, each producing an intermediate String, accumulating to O(n²), so for concatenation in a loop always reuse a single StringBuilder manually.

Summary

  • The 8 primitive types are fixed-width, cross-platform, and have no unsigned; char is a 16-bit UTF-16 code unit; boolean size is unspecified.
  • Wrapper classes box values into heap objects; Integer caches [-128,127], causing == to be true for small values and false for large ones—compare values with equals.
  • A HotSpot object = object header (Mark Word + Klass pointer) + instance fields + padding; fields get reordered; compressed pointers shrink references to 4 bytes (heaps < 32 GB).
  • Arrays are objects on the heap: they have a length field and bounds checking; elements are contiguous, but reference arrays store references, and multidimensional arrays are arrays of arrays (not guaranteed contiguous overall).
  • String is immutable, internally a byte[] + encoding flag, with no \0; literals share the constant pool, intern() fetches the canonical instance; for frequent concatenation use StringBuilder.
