Object Orientation and the Type System

In C++ you’re familiar with classes, inheritance, virtual functions, and templates. Java’s OOP looks similar on the surface, but three points are the core differences: single inheritance + interfaces, every method is virtual, and generic erasure (rather than C++’s template monomorphization). This chapter contrasts the underlying mechanics one by one.

1. Classes, objects, and initialization order

All instance methods of a Java class are “virtual” by default (except private/static/final); objects always live on the heap, and variables hold references (see “Variables and the Memory Model”).

Initialization order (be sure to memorize)

When you new an object, the JVM executes in a fixed order:

1. On first use of the class: static fields to default zero values → static init blocks + static field assignments (in source order, once only)
2. Instance creation: allocate heap memory, all instance fields set to zero values
3. Superclass instance initialization (recursively: superclass fields → superclass instance blocks → superclass constructor body)
4. This class's instance field defaults → instance init blocks + field assignments (in source order)
5. This class's constructor body
class Base {
    int x = init("Base.x");           // ③
    Base() { System.out.println("Base()"); }
    static int s = init("Base.s");    // ① at class load time
    static String init(String m){ System.out.println(m); return null; }
}
class Derived extends Base {
    int y = Base.init("Derived.y");   // ④
    Derived() { super(); System.out.println("Derived()"); } // super() is implicitly first
}
// new Derived() prints: Base.s (first time) → Base.x → Base() → Derived.y → Derived()

Compared with C++: members are constructed in the initializer list in declaration order, base before derived — Java’s order is similar, but there’s no initializer-list syntax, and static initialization is lazy (triggered only on the first active use of the class, executed exactly once in a thread-safe manner guaranteed by the class loader).

Constructors and this/super

  • The first line of a constructor must be this(...) (delegating to another constructor of the same class) or super(...) (calling the superclass); if omitted, the compiler inserts a no-arg super().
  • Calling a virtual method during construction is dangerous: a superclass constructor calling a method overridden by the subclass will, at that point, run when the subclass’s fields aren’t yet initialized (still zero values), reading an uninitialized state. C++ statically binds virtual calls during construction to the current class’s version; Java, by contrast, dynamically binds to the subclass’s version, which is more dangerous.

2. Encapsulation

Access modifiers: private < package-level (default, no keyword) < protected < public. Note that Java’s protected also permits same-package access, a broader semantic than C++’s. There’s no friend. The module system (JPMS) can encapsulate whole packages at a coarser granularity.

3. Inheritance: single inheritance + multiple interfaces

Java classes can only inherit singly (avoiding the complexity of C++’s diamond inheritance and virtual inheritance), but a class can implement any number of interfaces:

class Service extends AbstractBase implements Runnable, AutoCloseable { }

Interfaces once could only have abstract methods; since JDK 8 they can have default methods (with implementations) and static methods, and since JDK 9 they can have private methods. This brings the ability of “multiple inheritance of implementation,” but fields remain constants only (public static final), so there’s no state-diamond problem. If two interfaces have default methods with the same signature, the subclass must explicitly override to resolve the conflict (otherwise a compile error).

4. Polymorphism and the virtual method table (vtable)

Java instance methods dispatch dynamically by default. HotSpot’s implementation is highly similar to C++’s vtable:

Each class has a klass structure in the method area (Metaspace), containing a vtable:
  Klass(Animal)          Klass(Dog extends Animal)
  vtable:                vtable:
   [0] Object.method...    [0] Object.method... (inherited)
   [k] Animal::speak ─┐    [k] Dog::speak    ← overridden, same slot
                      │
object header's klass ptr ─┘ points to its own Klass
calling a.speak():
   1. take a's object-header klass ptr
   2. fetch the method address from the fixed vtable slot
   3. call   —— almost identical to C++

Differences:

  • Interface methods can’t use a fixed vtable slot (a class implements multiple interfaces, so slots can’t be unified); they use a separate itable (interface method table) + runtime lookup, slightly slower than the vtable. The JIT optimizes with an inline cache: it remembers the actual type of the last call, with near-zero overhead in the monomorphic case.
  • Java’s final/private/static methods are statically bound (invokespecial/invokestatic), with no virtual dispatch.
  • The JIT also does devirtualization: if it can prove a call site has only one implementation (CHA, class hierarchy analysis), it inlines directly, and deoptimizes if a new subclass is loaded during a STW pause. C++ needs a manual final or link-time optimization to get a similar effect.

At the bytecode level there are four call instructions: invokevirtual (virtual), invokeinterface (interface), invokespecial (constructor/private/super), invokestatic (static), plus invokedynamic (lambdas, string concatenation). See “Functions and the Call Stack”.

5. Abstract classes vs interfaces

Dimension Abstract class Interface
Instance fields can have (stateful) only public static final constants
Constructor yes no
Inheritance single multiple implementation
Methods abstract + concrete abstract + default + static + private
Semantics “what it is” (is-a) “what it can do” (can-do, capability)

Rule of thumb: use an abstract class to express a type hierarchy sharing state; use interfaces to express capabilities/contracts and where multiple composition is needed. In the JDK, AbstractList is an abstract-class skeleton and List is an interface.

6. Object’s methods and the equals/hashCode contract

Every class implicitly inherits Object (analogous to but stronger than C++ — C++ has no unified root). Key methods: equals, hashCode, toString, getClass, clone, wait/notify (see Concurrency), finalize (deprecated, see GC).

The equals/hashCode contract (violating it wrecks collection behavior)

  1. a.equals(b) is true ⟹ a.hashCode() == b.hashCode() (equal implies same hash).
  2. The converse is not required (equal hashes need not be equal, i.e. collisions are allowed).
  3. equals must be reflexive, symmetric, transitive, consistent, and non-null.

Overriding only equals without hashCode and then putting the object in a HashMap/HashSet will lose it, because it scatters to the wrong bucket (see “The Collections Framework”). The default Object.equals is reference equality (==), and the default hashCode is an identity hash — often mistakenly described as “derived from the object’s address,” but that doesn’t hold up (the hash would change after GC moves the object); modern HotSpot (default since JDK 8) uses a thread-local xorshift pseudo-random number, computed lazily on the first call, stored into the object header’s mark word, and unchanged thereafter (see “Data Types”).

final class Point {
    final int x, y;
    Point(int x, int y){ this.x=x; this.y=y; }
    @Override public boolean equals(Object o){
        if (this == o) return true;
        if (!(o instanceof Point p)) return false; // pattern-matching instanceof
        return x == p.x && y == p.y;
    }
    @Override public int hashCode(){ return Objects.hash(x, y); }
}

A record can generate these automatically (below).

7. Generics and type erasure (the fundamental divide from C++ templates)

This is the most “counterintuitive” part of Java’s type system, and where it differs most from C++.

Erasure: no generic information at runtime

Java generics are a compile-time feature. After compilation the type parameters are “erased”: List<String> and List<Integer> are both the same List class at runtime (the raw type List, with T erased to Object, or to the upper bound if bounded). The compiler inserts casts where needed and performs static type checking.

List<String> ls = new ArrayList<>();
ls.add("hi");
String s = ls.get(0);   // the compiler inserts a (String) cast
// at runtime: ls.getClass() == ArrayList.class, exactly the same as List<Integer>

Compared with C++: templates are monomorphized, so vector<int> and vector<string> compile to two independent copies of code, with their own types at runtime — a zero-cost abstraction but with code bloat. Java has only one copy of code, no bloat, but the type is lost at runtime, giving rise to a series of restrictions.

The four big restrictions from erasure

// 1. Can't new T[] (the runtime type can't be determined, and covariant arrays would miss checks)
// T[] arr = new T[10];              // compile error
T[] arr = (T[]) new Object[10];       // workaround: cast, with an unchecked warning

// 2. Can't new T() (don't know which class to construct)
// T t = new T();                     // compile error, need to pass a Supplier<T> or Class<T>

// 3. instanceof can't carry a type parameter (no such info at runtime)
// if (x instanceof List<String>)     // compile error
if (x instanceof List<?>) { }         // can only use a wildcard

// 4. Static fields can't use type parameters; can't catch a generic exception

Bridge methods

Erasure would break covariant returns and generic overriding, so the compiler generates bridge methods to bridge the gap:

class Node<T> { T get(){...} void set(T t){...} }
class StrNode extends Node<String> {
    @Override String get(){...}        // what you wrote
    // the compiler also secretly generates: Object get(){ return get(); }  ← bridge, matching the parent's erased signature Object get()
}

Otherwise the parent Node erases to Object get(), the subclass has String get() with a different signature, and virtual dispatch wouldn’t line up. Bridge methods are visible with javap -v, marked ACC_BRIDGE ACC_SYNTHETIC.

Wildcards: the PECS principle

Generics are invariant: List<String> is not a List<Object> (the opposite of array covariance, and safer). To loosen this, use wildcards:

  • ? extends T: upper bound, the producer (can only read out T, can’t write in, since the exact subtype is unknown).
  • ? super T: lower bound, the consumer (can only write in T, and reads out only as Object).
  • The mnemonic PECS: Producer-Extends, Consumer-Super.
// read from src, write to dst: src is the producer, use extends; dst is the consumer, use super
static <T> void copy(List<? super T> dst, List<? extends T> src){
    for (T t : src) dst.add(t);
}

C++20 concepts / old-style SFINAE solve a similar problem of “constraining type parameters,” but the mechanism is entirely different (compile-time duck typing vs nominal subtyping + erasure).

8. record: an immutable data carrier

record Point(int x, int y) { }
// auto-generated: private final fields, an all-args constructor, accessors x()/y(), equals/hashCode/toString

A record is implicitly final, with final fields, semantically a “transparent aggregate of data.” Analogous to a C++ aggregate struct + an auto-generated operator==. You can add a compact constructor to do validation:

record Range(int lo, int hi) {
    Range {   // compact constructor, assigns automatically after validation
        if (lo > hi) throw new IllegalArgumentException();
    }
}

9. sealed: sealed types (controlled inheritance)

Restricts who can inherit/implement, working with pattern matching to do exhaustiveness checking:

sealed interface Shape permits Circle, Square { }
record Circle(double r) implements Shape { }
record Square(double s) implements Shape { }

double area(Shape sh) {
    return switch (sh) {          // the compiler knows there are only two, no default needed
        case Circle c -> Math.PI * c.r() * c.r();
        case Square s -> s.s() * s.s();
    };
}

Similar to expressing a closed sum type in C++ with std::variant + std::visit. sealed lets the compiler verify the switch is exhaustive.

10. enum: a full-fledged class

A Java enum is not a C/C++ integer constant; it’s compiled into a class extending java.lang.Enum, with each enum constant being a singleton object (guaranteed by the JVM). It can have fields, methods, and constructors, and even a method body per constant (constant-specific behavior):

enum Planet {
    EARTH(5.97e24, 6.37e6), MARS(6.42e23, 3.39e6);  // calls the constructor
    private final double mass, radius;
    Planet(double m, double r){ mass=m; radius=r; }
    double gravity(){ return 6.67e-11 * mass / (radius*radius); }
}

Being singletons, enums can be safely compared with ==, making them the best way to implement the singleton pattern; EnumMap/EnumSet use the ordinal for a bitmap/array, extremely efficient.

11. Inner classes, anonymous classes, lambdas, and variable capture

Kinds of inner classes

  • Static nested class (static class): no reference to an outer instance, equivalent to a top-level class placed inside.
  • Member inner class: holds a hidden outer this reference, able to access the outer instance’s fields — also a source of memory leaks (a live inner-class instance drags the outer instance along).
  • Local class / anonymous class: defined inside a method.

Captured variables must be effectively final

A local variable captured by an anonymous class/lambda must be final or effectively final (never reassigned after initialization):

int base = 10;                 // effectively final
Runnable r = () -> System.out.println(base + 1);
// base = 20;                  // uncomment this and the line above becomes a compile error

Why? A local variable lives in the stack frame, which is destroyed after the method returns; whereas the lambda/anonymous-class object lives on the heap and may outlive it. Java’s approach is to copy the value of the captured variable into the lambda object (copying the value for primitives, the reference value for reference types), requiring it to be immutable to keep “the original variable on the stack” and “the copy on the heap” semantically consistent, avoiding the confusion of the two being out of sync.

Compared with C++ lambdas: you explicitly choose [=] capture-by-value or [&] capture-by-reference, and by-reference can capture mutable variables — but once the stack variable becomes invalid it’s a dangling reference (UB), and the responsibility is yours. Java uses effectively final to plug this hole at the language level (by-value only, and the value doesn’t change). Capturing an instance field is different: the lambda captures this and accesses the field through it, and the field is mutable.

Under the hood of lambdas: invokedynamic

A lambda is not syntactic sugar for an anonymous class. javac generates an invokedynamic instruction, and on first execution LambdaMetafactory dynamically generates, at runtime, a class implementing the functional interface. Benefit: a capture-free lambda can reuse the same instance (stateless), avoiding the anonymous-class bloat of one class file per site. Method references (String::length) work the same way.
