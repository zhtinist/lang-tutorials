Variables and the Memory Model

This is the most important chapter in the book. It first covers how to write the code—how var/:= declare and assign, how variables get passed into functions—material a complete beginner can follow with no trouble. Then it goes underneath the hood: if you come from C/C++, you’ll find that Go’s memory semantics are far simpler than C++’s, but there’s one axiom you must first burn into your brain: in Go, assignment and argument passing are always a value copy. There is no pass-by-reference, no copy constructor, no move semantics. Everything that “looks like a reference,” once taken apart, is “a small struct containing a pointer, copied byte for byte.” If you just want to know how to use it, reading the first section is enough; if you want to understand the black magic behind “why did changing a also change b,” keep reading past that.

Start Using It: A Quick Look at Assignment and Argument Passing

Assignment: b := a copies a’s value into b. After that, a and b are two independent variables—changing one doesn’t affect the other:

package main

import "fmt"

func main() {
    a := 10
    b := a // copy a's value into b
    b = 99
    fmt.Println(a, b) // 10 99 —— changing b doesn't affect a
}

Passing a variable to a function also gives the function a copy; modifying it inside the function doesn’t affect the variable outside:

package main

import "fmt"

func addOne(x int) {
    x = x + 1 // only modifies the function's local copy
}

func main() {
    n := 5
    addOne(n)
    fmt.Println(n) // 5 —— unchanged
}

To let a function actually modify the outer variable, you need a pointer: &n takes n’s address, and *x reads/writes the value it points to through that address:

package main

import "fmt"

func addOnePtr(x *int) {
    *x = *x + 1 // modifies the outer variable through its address
}

func main() {
    n := 5
    addOnePtr(&n)
    fmt.Println(n) // 6 —— changed
}

These three examples already lay out the core behavior of Go variables: assignment and argument passing are copies; to change something across a function boundary, pass an address. Now let’s go underneath the hood—turning this rule into an axiom and explaining why it accounts for nearly every “aliasing” phenomenon you’ll run into.

Going Deeper

If you come from C/C++, by the end of this chapter you’ll find that Go’s memory semantics are far simpler than C++’s—but there’s one axiom you must first burn into your brain: in Go, assignment and argument passing are always a value copy. There is no pass-by-reference, no copy constructor, no move semantics. Everything that “looks like a reference,” once taken apart, is “a small struct containing a pointer, copied byte for byte.” Master this one point, and 90% of the later “why did changing a also change b?” ceases to be a mystery.

Axiom: Assignment and Argument Passing Are Always a Value Copy

In C++, what happens in b = a depends on a’s type and a pile of rules: built-in types copy byte for byte, class types invoke a copy constructor / copy assignment operator, rvalues trigger a move, reference types are merely aliases… a heavy cognitive burden.

Go cuts this down to a single rule:

b = a, as well as “passing a as an argument into a function,” both copy a’s machine representation byte for byte.

There is no user-overridable copy behavior, no hidden constructor call. However many bytes a occupies, that many bytes are copied. The only difference is—whether those bytes contain a pointer.

  • If a is an int, bool, float64, array, or struct (a value type), then what’s copied is all the data itself.
  • If a is a pointer, slice, map, chan, interface, or func (containing a pointer internally), what’s copied is that few-machine-words “header/handle”; the pointer inside the header is copied, so a and b point to the same underlying data.

So in Go there is fundamentally no “reference type” in the C++ sense. So-called reference types are merely value types whose header is small and contains a pointer. Remember: what’s copied is always a “value”; the only question is “whether that value contains a pointer.”

Value Types: Copy the Whole Block of Memory Byte for Byte

First, pure value types. The Point below is a struct with no pointer fields:

package main

import "fmt"

type Point struct {
    X, Y int64
}

func main() {
    a := Point{X: 1, Y: 2}
    b := a // byte-for-byte copy of 16 bytes; b now has nothing to do with a

    b.X = 100
    fmt.Println(a.X, b.X) // 1 100 —— changing b doesn't affect a
}

b := a at the underlying level is a single “copy 16 bytes” instruction (the gc compiler will use an instruction like MOVUPS to move it). This is exactly the same as C’s struct Point b = a;: a whole block of POD memory copied as-is, with no constructor involved.

Memory diagram (two completely independent blocks of memory):

a: [ X=1 | Y=2 ]        b: [ X=100 | Y=2 ]
   ^independent 16 bytes   ^another independent 16 bytes

Argument passing works the same way. The addOne example earlier was the int version; here’s the struct version, which likewise receives a copy:

func move(p Point) { // p is a copy of the argument
    p.X += 10 // only modifies the copy
}

func main() {
    a := Point{1, 2}
    move(a)
    fmt.Println(a.X) // 1 —— the outer a doesn't budge
}

This is identical to C’s void move(struct Point p): pass-by-value, and what the function modifies is the copy on the stack. To change something outside, you must pass a pointer—just like C. Placing “pass by value” and “pass by pointer” side by side makes the difference obvious:

package main

import "fmt"

type Point struct{ X, Y int64 }

func moveVal(p Point)  { p.X += 10 } // modifies a copy
func movePtr(p *Point) { p.X += 10 } // modifies the original through the pointer

func main() {
    a := Point{X: 1, Y: 2}

    moveVal(a)
    fmt.Println(a.X) // 1  —— unchanged, the function got a copy

    movePtr(&a)
    fmt.Println(a.X) // 11 —— changed, the function got the address
}

Note that movePtr writes p.X, not (*p).X—Go automatically dereferences a pointer to a struct, sparing you C’s dedicated arrow operator p->X.

Arrays are also value types, and copying copies the entire array (C programmers should be especially careful here—in C, passing an array as an argument decays to a pointer; in Go it doesn’t):

arr := [3]int{1, 2, 3}
cp := arr // copies all 3 elements, 24 bytes
cp[0] = 99
fmt.Println(arr[0], cp[0]) // 1 99

C-pitfall contrast: in C, void f(int a[3]) is equivalent to void f(int *a), passing the address of the first element; in Go, func f(a [3]int) passes a copy of the entire array. The length is part of the type, and [3]int and [4]int are different types. For C-style “pass by reference,” pass a *[3]int or use a slice instead.

Pointers: There Are Pointers, but No Pointer Arithmetic, and They’re Safer

Go has pointers: & takes an address and * dereferences, with the same syntax as C—you already used this in the first section’s example:

a := 42
p := &a  // p's type is *int
*p = 100 // change a through the pointer
fmt.Println(a) // 100

At the underlying level, p is simply a machine word holding the address of a (8 bytes on a 64-bit platform), with exactly the same representation as a C pointer. But Go’s pointers observe a few disciplines:

Feature C/C++ Go
Take address / dereference & / * & / * (same)
Pointer arithmetic p+1, p[i] Allowed Forbidden (except via unsafe.Pointer)
Null pointer NULL / nullptr nil
Dangling pointer Common bug GC guarantees: as long as a pointer points to an object, it isn’t collected
Point to a local variable and return it Undefined behavior Legal, the variable automatically escapes to the heap

The last one is a major liberation for C programmers:

func newCounter() *int {
    n := 0     // local variable
    return &n  // a disaster in C, perfectly legal in Go
}

In C, returning the address of a local variable is a classic dangling-pointer bug: after the function returns the stack frame is destroyed and the pointer points to garbage. In Go, when the compiler finds that n’s address escapes the function scope, it automatically allocates n on the heap, with the GC managing its lifetime. This leads to the next core topic—escape analysis.

Escape Analysis: Stack or Heap Is Not Decided by new/&

A C++ programmer’s intuition is: int x; is on the stack, new int is on the heap, decided literally by the code you write. Go is not like this. Whether a variable is on the stack or the heap is decided by the compiler’s escape analysis, and has no necessary connection to whether you used & or new.

The core of the rule is a single question: will references to this variable outlive the stack frame of the function it’s in?

  • Won’t escape → put on the stack, reclaimed automatically when the function returns, zero GC cost.
  • Will escape (address is returned, stored into a heap object, stored into an interface, lifetime uncertain) → put on the heap, handed to the GC.

Use go build -gcflags='-m' (add -m -m for more detailed reasoning) to have the compiler print its decisions. Consider this example:

package main

func stackAlloc() int {
    x := new(int) // don't be fooled by new
    *x = 42
    return *x // returns only the value; x's address doesn't get out
}

func heapAlloc() *int {
    y := 0
    return &y // the address escapes
}

func main() {
    _ = stackAlloc()
    _ = heapAlloc()
}

Compile and observe (the exact line numbers and wording printed can vary slightly between Go versions and depending on inlining, but the conclusion is consistent):

$ go build -gcflags='-m -l' escape.go
./escape.go:4:10: new(int) does not escape
./escape.go:10:2: moved to heap: y

The conclusion is counterintuitive: new(int) does not escape and is instead allocated on the stack; while the perfectly ordinary local variable y, because its address is returned, is “moved to heap.”

This mental-model update for C/C++ programmers is crucial:

In Go, new/& express “I want an addressable object / I want its address,” not “allocate on the heap.” Stack vs. heap placement is computed by the compiler based on escape analysis. You don’t need to—and can’t—decide it manually (except by changing your code structure to influence the escape result).

Common escape triggers: returning the address of a local variable; stuffing a pointer into a slice/map/struct field (where that container escapes); putting a value into an interface{} (e.g. fmt.Println(x) often causes x to escape); a closure capturing and modifying an outer variable; an object whose size is unknown at compile time.

A practical mindset: don’t agonize everywhere over “avoiding heap allocation”; write clear code first, then use -gcflags='-m' and pprof to pinpoint what actually escapes when you find a hotspot. Go’s philosophy is to let the compiler worry about stack/heap placement for you.

new vs make

Two built-in allocation functions that C programmers easily confuse; distinguished in one sentence:

new(T) make(T, …)
Applicable types any type T only slice / map / chan
Return value *T (a pointer to a zero value) T itself (not a pointer)
What it does allocates one block of zeroed memory allocates and initializes the internal data structure (header + underlying array/hash table/ring buffer) 
p := new([]int) // p is *[]int, pointing to a nil slice header; the underlying array doesn't exist yet, not directly usable
s := make([]int, 3) // s is []int, length 3, the underlying array is ready, usable

new(T) is like C’s calloc(1, sizeof(T)) followed by taking the address—it just gives you a block of zeroed memory. make, on the other hand, is a “constructor” dedicated to slice/map/chan, because these three need their internal header and underlying structure initialized before they can work (map especially: writing to an uninitialized nil map panics). In practice new is rarely used; most of the time a composite literal &T{...} is clearer.

The Zero-Value-Is-Useful Philosophy

In C/C++, reading an uninitialized variable is undefined behavior, with random garbage on the stack. Go guarantees every variable is initialized to its type’s “zero value,” and these zero values are designed to be “usable out of the box”:

Type Zero value
Numeric (int/float/…) 0
bool false
string "" (empty string, not nil)
pointer / slice / map / chan / func / interface nil
struct each field’s own zero value (recursively)

Zero-value-is-useful means many types can be used just by declaring them, with no constructor:

var buf bytes.Buffer // the zero value is a usable empty buffer, no initialization needed
buf.WriteString("hi")

var mu sync.Mutex // the zero value is the unlocked state, Lock directly
mu.Lock()

var s []int    // nil slice, but append works on it directly
s = append(s, 1) // OK, append allocates the underlying array as needed

Note one exception: a nil map can only be read, not written. var m map[string]int reading m["x"] returns 0 (the zero value), but m["x"] = 1 panics, because the underlying hash table hasn’t been made yet. This is an important difference in zero-value behavior between map and slice.

The Truth About “Reference Types”: A Small Header Containing a Pointer

Now let’s expose that so-called “reference type.” slice, map, and chan are often called reference types in tutorials, as if they were a different species from value types. The truth: they are still values, and a = b is still a byte-for-byte copy—it’s just that their value is a “small header struct,” and the header contains a pointer to the underlying data. Copy the header → the pointer is copied → the two variables share the same underlying data.

Take slice as an example. Its header is, at runtime, a 3-word struct like this (see the runtime’s reflect.SliceHeader):

type sliceHeader struct {
    Data uintptr // pointer to the underlying array
    Len  int     // length
    Cap  int     // capacity
}

24 bytes (on a 64-bit platform). When you write b := a (a is a slice), what’s copied is these 24 header bytes, and the Data pointer is copied as-is. So:

a := []int{1, 2, 3}
b := a       // copies the header, b.Data == a.Data, sharing the same underlying array
b[0] = 99    // modifies an element through the shared underlying array
fmt.Println(a[0]) // 99 —— a changed too! this is "aliasing"

This is exactly the point that puzzles C++ programmers: it’s clearly a value copy, so how does changing b affect a? Because the pointer inside the copied header points to the same block of heap memory. This is the same principle as C’s struct { int *data; int len; } b = a; after which b.data[0] = 99 affects a—the struct was copied, but not the buffer it points to.

But this “sharing” is conditional—once append triggers a reallocation, the underlying array is reallocated and the sharing quietly breaks. This is the pitfall beginners hit most often:

package main

import "fmt"

func main() {
    a := []int{1, 2, 3}
    b := a // copies the 24-byte header; b shares a's underlying array

    b[0] = 99
    fmt.Println(a) // [99 2 3]  —— shared array, a sees the change too

    b = append(b, 4) // len==cap==3 is full, append reallocates: b moves to its own new array
    b[1] = 88
    fmt.Println(a) // [99 2 3]     —— unchanged, the sharing is now broken
    fmt.Println(b) // [99 88 3 4]
}

Before the append, modifying b reaches a; after the append, modifying b no longer reaches a—two behaviors within the same snippet of code, rooted in that one reallocation. When growth happens and by how much depends on capacity; details are in 07-复合数据结构.md.

map and chan go even further: the variable itself is a pointer (to the runtime’s hmap / hchan struct). b := a copies this pointer, and both operate on the same hash table / same channel.

Here we only build intuition. Details like a slice’s len/cap, the growth and reallocation of append, the map’s hash structure, the chan’s ring buffer, and so on are covered in 07-复合数据结构.md. For now just keep this unified picture firmly in mind: there are no reference types, only “value types whose header contains a pointer”; assignment copies the header and shares the underlying data.

ASCII Contrast Diagram: struct Value Copy vs. slice Copy

struct value copy (no sharing): b := a copies the whole block of data, and the two never interfere with each other again.

struct value type      a := Point{1,2}; b := a

  a ┌─────────┐        b ┌─────────┐
    │ X = 1   │          │ X = 1   │   two independent blocks of memory
    │ Y = 2   │          │ Y = 2   │   changing b doesn't affect a
    └─────────┘          └─────────┘

slice header copy (shared underlying): b := a copies the 24-byte header, and the Data of both headers points to the same array.

slice "reference type" a := []int{1,2,3}; b := a

  a ┌──────────┐       b ┌──────────┐
    │ Data ●───┼──┐      │ Data ●───┼──┐   headers are each independent (on the stack)
    │ Len  3   │  │      │ Len  3   │  │
    │ Cap  3   │  │      │ Cap  3   │  │
    └──────────┘  │      └──────────┘  │
                  └──────┬─────────────┘
                         ▼   both headers point to the same underlying array (on the heap)
              ┌────┬────┬────┐
              │  1 │  2 │  3 │  ← modifying b[0] modifies this, and a[0] changes with it
              └────┴────┴────┘

To sum up in one sentence: when you see a = b, first ask yourself whether b’s bytes contain a pointer. No → two independent copies of the data; yes → independent headers, shared underlying. This criterion explains nearly all aliasing phenomena in Go.

Summary

  • At the usage level: var/:= declare variables; b := a copies a’s value into b; passing a variable to a function gives the function a copy; to let a function modify the outer variable, pass a pointer (&/*).
  • At the underlying level: assignment and argument passing are always a value copy—value types copy all their data, and “reference types” (slice/map/chan/pointer/interface) copy a header containing a pointer, which produces aliasing; whether a variable lives on the stack or the heap is decided by escape analysis, independent of new/&; new only allocates zeroed memory, while make is dedicated to initializing the internal structure of slice/map/chan; zero values are designed to be usable out of the box, but a nil map is read-only, not writable.
