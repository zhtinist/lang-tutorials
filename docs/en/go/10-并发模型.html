The Concurrency Model: goroutines and channels

Concurrency is Go’s signature feature. This chapter first covers how to write and use goroutines/channels/select — understanding ordinary concurrent code is enough for that — and then moves on to “underlying implementation vs C”: how the runtime schedules hundreds of thousands of goroutines onto a handful of OS threads. For C++ programmers: forget std::thread first (a 1:1 mapping to OS threads, several MB of stack each, expensive to create) — later on you’ll see that Go’s goroutine is a user-space coroutine that the runtime multiplexes onto a small number of OS threads, cheap enough to spin up hundreds of thousands.

First, the basics: a quick tour of goroutines, channels, and select

No scheduler details yet — just how to write these three things:

package main

import "fmt"

func say(msg string, done chan bool) {
    fmt.Println(msg)
    done <- true // tell the outside world via a channel: I'm done
}

func main() {
    done := make(chan bool)
    go say("in goroutine", done) // start a goroutine, keep going immediately (don't wait for it)
    <-done                       // block until the goroutine signals

    ch := make(chan int, 1)
    ch <- 100 // send

    select { // select waits on several channels at once, proceeding with whichever is ready
    case v := <-ch:
        fmt.Println("received:", v) // received: 100
    default:
        fmt.Println("no value")
    }
}

A quick summary:

  • go someCall(...) starts a goroutine — a “lightweight concurrent task” — and returns immediately without waiting for it to finish.
  • Goroutines pass data/signals to each other through channels: ch <- v to send, v := <-ch to receive (recap the basics of channels in Chapter 07).
  • select watches several channels at once and takes whichever branch becomes ready first; default keeps it from blocking.

That much is enough to write concurrent code that runs. Now let’s go deeper — why Go can casually spin up hundreds of thousands of goroutines, and how the scheduler puts them on the CPU.

Now let’s go deeper: why goroutines are cheap

From here on we talk about the underlying mechanics. To understand the performance characteristics of Go concurrency, start with this: a goroutine is not an OS thread — it is a user-space coroutine managed by the runtime itself, multiplexed onto a small number of OS threads by the Go runtime’s scheduler (the GMP model).

I. goroutine: an extremely lightweight coroutine

How to use it

The go keyword plus a function call starts a goroutine:

go f(x, y)        // start a goroutine running f(x,y), return immediately, do not wait for it
go func() {
    fmt.Println("in goroutine")
}()

Pair it with sync.WaitGroup to wait for a whole group of goroutines to finish (covered in detail in the “sync package” section below):

var wg sync.WaitGroup
results := make([]int, 3)

for i := 0; i < 3; i++ {
    wg.Add(1)
    go func(id int) { // start a goroutine
        defer wg.Done()
        results[id] = id * id
    }(i)
}
wg.Wait() // wait for all goroutines to finish

fmt.Println(results) // [0 1 4]

When does a goroutine end: it ends when its function body returns. When main returns, the whole program exits and does not wait for other goroutines — if you need to wait, use a channel or sync.WaitGroup.

Comparison with C

Why they are cheap:

OS thread (C++ std::thread) goroutine
Initial stack fixed 1~8 MB (virtual memory reserved) 2 KB, grows/shrinks on demand
Create/destroy system call, microseconds, expensive user-space, nanoseconds, practically free
Scheduling kernel scheduling, switching traps into the kernel runtime user-space scheduling, extremely fast switching
Scale thousands is already a strain hundreds of thousands, even millions, are feasible

Stack growth: a goroutine stack starts at 2KB; when that is not enough, the runtime allocates a larger stack, copies the contents over, and rewrites all pointers into that stack (contiguous stacks, Chapter 06). Precisely because the runtime fixes up pointers when the stack “moves house,” you can still take the address of a local variable as usual (&local), return it, or pass it elsewhere: escape analysis puts the truly escaping variables on the heap and leaves the rest on the stack, entirely transparently to you. This is exactly the opposite of C — in C, returning or long-term holding the address of a stack variable is a dangling-pointer bug, whereas in Go it is safe.

II. channels and CSP

How to use it

Go embraces the CSP (Communicating Sequential Processes) philosophy:

Don’t communicate by sharing memory; share memory by communicating.

This means: prefer using a channel to transfer ownership of data between goroutines, rather than having multiple goroutines contend over a shared piece of memory guarded by a lock.

ch := make(chan int)        // unbuffered
ch := make(chan int, 10)    // buffered, capacity 10

ch <- 42                    // send
v := <-ch                   // receive
v, ok := <-ch               // ok=false means closed and empty
close(ch)                   // close (the sender closes it, not the receiver)
for v := range ch { }       // until the channel is closed

A common pattern: use a channel to wait for a goroutine to finish.

done := make(chan struct{})   // struct{} is zero-size, used only as a signal
go func() {
    work()
    close(done)               // or done <- struct{}{}
}()
<-done                        // block until work completes

Comparison with C

The channel’s underlying structure (hchan: a ring buffer + a lock + wait queues) was already covered in detail in Chapter 07. One more point about runtime behavior: send/receive on an unbuffered channel is a rendezvous, naturally forming a happens-before relationship (expanded on in the “Go memory model” section below), and it is often used as a semaphore/synchronization point — this is exactly why the “wait for a goroutine to finish” pattern above works.

III. select: multiplexing

How to use it

select waits on multiple channel operations at once, proceeding with whichever is ready (if several are ready, one is chosen at random to prevent starvation):

select {
case v := <-ch1:
    fmt.Println("from ch1", v)
case ch2 <- x:
    fmt.Println("sent to ch2")
case <-time.After(2 * time.Second):
    fmt.Println("timeout")          // timeout pattern
default:
    fmt.Println("none ready, non-blocking") // with a default, it does not block
}
  • No default: blocks until at least one case is ready.
  • With default: if no case is ready, proceed to default immediately — non-blocking send/receive.
  • time.After returns a channel that only sends data at the deadline — the timeout idiom.
  • A case on a nil channel is never ready, which can be used to dynamically “disable” a branch.

A runnable example: multiplexing two channels plus a time.After timeout fallback. Two goroutines each send once at their deadline, and select exits only after collecting both. The arrival order is undefined, so we assert the set rather than an exact order:

package main

import (
    "fmt"
    "time"
)

func main() {
    nums := make(chan int)
    strs := make(chan string)

    go func() {
        time.Sleep(10 * time.Millisecond)
        nums <- 42
    }()
    go func() {
        time.Sleep(20 * time.Millisecond)
        strs <- "hi"
    }()

    got := map[string]bool{}
    timeout := time.After(100 * time.Millisecond)
    for len(got) < 2 {
        select {
        case n := <-nums:
            got[fmt.Sprintf("num:%d", n)] = true
        case s := <-strs:
            got[fmt.Sprintf("str:%s", s)] = true
        case <-timeout:
            fmt.Println("timed out")
            return
        }
    }
    fmt.Println("got num:", got["num:42"])
    fmt.Println("got str:", got["str:hi"])
    // got num: true
    // got str: true
}

IV. The GMP scheduling model

The basic usage of goroutine/channel/select ends here. This section is purely internal runtime machinery — it explains why all of the above is cheap and pleasant to use.

How to use it: GOMAXPROCS

Writing everyday concurrent code, you basically never need to worry about the scheduler; the one knob you’ll actually touch is the degree of parallelism:

runtime.GOMAXPROCS(4)         // set/query the number of Ps (i.e. the parallelism cap)
n := runtime.GOMAXPROCS(0)    // passing 0 means query only, no change

Comparison with C

The Go runtime uses the G-M-P trio to multiplex goroutines onto OS threads.

  • G (goroutine): a coroutine, holding a stack, program counter, and state.
  • M (machine): an OS thread, which actually executes code on the CPU.
  • P (processor): a scheduling context/logical processor, holding a local run queue of runnable Gs. The number of Ps = GOMAXPROCS (default = number of available CPU cores; as of Go 1.25, inside a container it is aware of the cgroup CPU limit and rounds to that limit, no longer always equal to the physical core count). An M must be bound to a P to execute a G.
              global run queue GRQ: [G G G ...]
                     │(when a local queue empties, steal/take from here)
   ┌─────────────┐   ┌─────────────┐
   │  P0         │   │  P1         │   number of Ps = GOMAXPROCS
   │ local queue │   │ local queue │
   │ LRQ [G G G] │   │ LRQ [G G]   │
   └──────┬──────┘   └──────┬──────┘
          │bound            │bound
      ┌───▼───┐         ┌───▼───┐
      │  M0   │         │  M1   │   M = OS thread, count as needed (may be > P)
      │(thread)│        │(thread)│
      └───────┘         └───────┘
           actually run on CPU cores

Scheduling essentials

Local queues + work-stealing: each P has a local queue; newly created Gs preferentially go to the local queue (lock-free, fast). When a P’s local queue empties, it first takes from the global queue, then steals half from another P’s queue. This balances load automatically while minimizing global lock contention as much as possible.

Why have P at all (instead of G binding directly to M): the P layer decouples “parallelism (GOMAXPROCS Ps)” from “thread count (M can be more).” When one M blocks on a system call, its P can be handed off to another M to keep running other Gs — see below.

P/M hand-off during system calls: when a G issues a blocking system call (e.g., reading a file), the M it is on traps into the kernel and blocks. The runtime detaches that M’s P and hands it to another (idle or newly created) M, so the remaining Gs on the P keep running on other cores, not held up by one blocking call. After the system call returns, the original M tries to reacquire a P; if it cannot get one, it puts the G back on a queue and puts itself to sleep.

Network I/O uses the netpoller: network reads/writes do not actually block an M; instead they register with an epoll/kqueue-based netpoller. The goroutine is suspended, the M goes off to run other Gs, and when the I/O is ready the goroutine is woken. This is how you get to write “one goroutine per connection, synchronous blocking style” yet obtain epoll-level scalability — the underlying reason Go network services are so pleasant.

Preemptive scheduling: early Go was cooperative (yielding only at function calls), so a pure compute loop could starve the other Gs on the same P. Since Go 1.14, it is signal-based asynchronous preemption: the runtime can signal a G that has not yielded for a long time, forcing a switch at a safe point. So nowadays you need not worry about for {} jamming the scheduler.

Compared to C++: std::thread is a 1:1 kernel thread, scheduled by the OS, and switching enters the kernel; Go is M:N (M Gs running on N threads), with most scheduling done in user space and far lower switching cost.

V. The sync package: traditional locks and synchronization primitives

How to use it

Not every scenario suits channels. For a shared counter or protecting a small critical section, a lock is more direct and faster.

var mu sync.Mutex
mu.Lock()
count++
mu.Unlock()          // often paired with defer mu.Unlock()
Primitive Purpose C++ counterpart
sync.Mutex mutual-exclusion lock std::mutex
sync.RWMutex read-write lock, many readers or one writer std::shared_mutex
sync.WaitGroup wait for a group of goroutines to finish no direct equivalent (manual join)
sync.Once run some code only once (lazy initialization) std::call_once
sync/atomic atomic operations (Add/Load/Store/CAS) std::atomic
sync.Map concurrency-safe map, optimized for read-heavy workloads none (concurrent_map class)
sync.Pool temporary object pool, reduces allocation/GC pressure object pool

WaitGroup idiom:

var wg sync.WaitGroup
for i := 0; i < 10; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        work(id)
    }(i)                  // note: pass i in as a parameter (the loop-variable-reuse trap before Go 1.22)
}
wg.Wait()                 // block until the count reaches zero

Since Go 1.22, for loop variables are independent per iteration, so the classic “closure captures the loop variable” bug in old code mostly disappears; but explicitly passing the argument is still a clear, good habit.

sync.Mutex is usable as its zero value (no initialization needed) and is non-copyable (copying gives you two separate locks; go vet warns). So structs containing a Mutex are usually passed by pointer.

A runnable example: 100 goroutines each add 1 a thousand times; both Mutex and atomic.Int64 reach a deterministic 100000. If you drop the mu.Lock/Unlock and use a bare count++, go run -race reports a DATA RACE, and the result also falls below 100000 due to lost updates:

package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

func main() {
    const goroutines, per = 100, 1000
    want := int64(goroutines * per)

    // 1) a plain counter guarded by sync.Mutex
    var mu sync.Mutex
    var muCount int64
    var wg sync.WaitGroup
    for i := 0; i < goroutines; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for j := 0; j < per; j++ {
                mu.Lock()
                muCount++
                mu.Unlock()
            }
        }()
    }
    wg.Wait()

    // 2) an atomic add with sync/atomic
    var atCount atomic.Int64
    for i := 0; i < goroutines; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for j := 0; j < per; j++ {
                atCount.Add(1)
            }
        }()
    }
    wg.Wait()

    fmt.Println("want  :", want)
    fmt.Println("mutex :", muCount, muCount == want)
    fmt.Println("atomic:", atCount.Load(), atCount.Load() == want)
    // want  : 100000
    // mutex : 100000 true
    // atomic: 100000 true
}

Comparison with C

sync.Mutex itself is small (conceptually a state word plus a semaphore, 8 bytes total); locking has two paths: when there’s no contention, a single atomic CAS grabs the lock directly (the fast path, near-zero overhead); when that fails, it spins briefly a few times, and if it still can’t get the lock, it parks the goroutine via the runtime’s semaphore, waking it on unlock — a parked goroutine is taken off the scheduler and does not tie up an M, using the same “park/wake” mechanism as ordinary channel blocking. This differs from C++’s std::mutex, which typically wraps a kernel futex wait directly: Go’s lock is part of the runtime’s scheduling framework, and what blocks is the goroutine, not the thread itself. The details evolve across versions — defer to the runtime source sync/mutex.go.

VI. The Go memory model and happens-before

On multicore, concurrent reads/writes without synchronization have no ordering guarantees (the compiler/CPU may reorder, and caches may be invisible). The Go memory model uses happens-before to define visibility: if event A happens-before B, then A’s writes are visible to B. Ways to establish happens-before:

  • channel: a send happens-before the corresponding receive completes; a channel’s close happens-before a receive returning the zero value.
  • Mutex: the nth Unlock happens-before the (n+1)th Lock.
  • WaitGroup: before Wait returns, all writes prior to every Done are visible.
  • Once: the initialization inside Do happens-before any return of Do.
  • atomic: provides ordering guarantees (since Go 1.19, sync/atomic has explicit typed APIs like atomic.Int64).

Without these synchronizations, you cannot assume one goroutine’s write is visible to another. This is the same class of problem as C++’s std::memory_order; Go simplifies it to “use channels/locks, don’t share raw memory.”

VII. context: cancellation, timeout, value passing

How to use it

context.Context is the standard way to pass a cancellation signal, a deadline, and request-scoped values across goroutines/APIs, and it is the first parameter of nearly every blocking standard-library/framework API.

ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()                        // must call cancel to release resources

select {
case res := <-doAsync(ctx):
    use(res)
case <-ctx.Done():                    // timed out or canceled
    fmt.Println("cancellation reason:", ctx.Err()) // context.DeadlineExceeded / Canceled
}

ctx.Done() returns a channel that is closed on cancellation, used together with select. The cancellation signal propagates down the context tree — cancel in one place and all derived contexts receive it — used to “the request is over, stop all the downstream goroutines it launched together.” context.WithValue passes request-scoped data (such as a trace id), but do not use it to pass function arguments.

A runnable example: a single cancel() broadcasts to 3 workers, whose ctx.Done() channels all become ready and exit at once. A WaitGroup converges deterministically, and we assert the exit count and ctx.Err():

package main

import (
    "context"
    "fmt"
    "sync"
    "time"
)

func main() {
    ctx, cancel := context.WithCancel(context.Background())
    var wg sync.WaitGroup
    const n = 3
    stopped := make(chan int, n)

    for id := 0; id < n; id++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for {
                select {
                case <-ctx.Done(): // cancellation received, exit
                    stopped <- id
                    return
                default:
                    time.Sleep(time.Millisecond) // simulate work
                }
            }
        }(id)
    }

    time.Sleep(20 * time.Millisecond)
    cancel() // broadcast cancel: every worker's ctx.Done() becomes ready
    wg.Wait()
    close(stopped)

    fmt.Println("ctx.Err():", ctx.Err())
    fmt.Println("stopped  :", len(stopped))
    // ctx.Err(): context canceled
    // stopped  : 3
}

VIII. Common concurrency patterns

Worker Pool (a fixed number of workers consuming tasks):

jobs := make(chan int, 100)
results := make(chan int, 100)
var wg sync.WaitGroup
for w := 0; w < 4; w++ {              // 4 workers
    wg.Add(1)
    go func() {
        defer wg.Done()
        for j := range jobs {         // exits automatically when jobs is closed
            results <- j * j
        }
    }()
}
for i := 1; i <= 20; i++ { jobs <- i }
close(jobs)                            // close → worker loops end
go func() { wg.Wait(); close(results) }()
for r := range results { fmt.Println(r) }

Pipeline (stages chained together, one/several goroutines per stage, connected by channels):

gen() ──chan──▶ square() ──chan──▶ print()

Fan-out / Fan-in: fan-out = multiple goroutines reading from the same channel (sharing the work); fan-in = merging multiple channels into one.

// fan-in: merge multiple input channels
func merge(cs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, c := range cs {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c { out <- v }
        }(c)
    }
    go func() { wg.Wait(); close(out) }()
    return out
}

IX. Data races and the -race detector

Data race: two goroutines concurrently access the same memory, at least one is a write, and there is no synchronization. The behavior is undefined — you may read a torn value, or crash. Go does not automatically eliminate races for you (except that maps proactively crash with an error).

Go has a built-in race detector; add -race at build time:

go run -race main.go
go test -race ./...

It instruments every memory access at runtime, and on finding an unsynchronized concurrent read/write it prints a detailed report (which two goroutines, which line, the call stacks). The performance overhead is large (several times slower, memory grows), so use it only in tests/CI, not in production. This is Go’s most useful concurrency-debugging tool; be sure to run your tests through it when writing concurrent code.

WARNING: DATA RACE
Write at 0x00c000... by goroutine 7:
  main.main.func1() main.go:12
Previous read at 0x00c000... by goroutine 6:
  main.main.func2() main.go:15

Summary

  • Usage level: go f(x,y) starts a goroutine without waiting for it to finish; channels send/receive/synchronize via make/<-/close/range; select waits on multiple channels at once, paired with default for non-blocking and time.After for timeouts; sync.Mutex/WaitGroup/atomic handle shared state; context carries cancellation/timeout; worker pool, pipeline, and fan-in/out are common skeletons; run -race when writing concurrent code.
  • Underlying level: a goroutine is a coroutine starting at 2KB, scheduled in user space, with a growable/shrinkable stack, M:N multiplexed onto OS threads; the GMP trio — G the coroutine, M the thread, P the context (= GOMAXPROCS) — implements scheduling via local queues + work-stealing + preemption + system-call hand-off, with network I/O relying on the netpoller instead of blocking a thread; sync.Mutex is an atomic-CAS fast path plus a runtime park/wake slow path, blocking the goroutine rather than the OS thread; correctness is guaranteed by happens-before — without synchronization, you cannot assume a write is visible to another goroutine.
