The Type System: Structs, Interfaces, Method Sets

Go has no class, no inheritance, no constructors/destructors, no virtual keyword, and no access modifiers (public/private are determined by the first letter’s case of the identifier). Yet with struct + methods + composition + interfaces it composes a simpler and more “structural” type system. For C++ programmers, the two things most in need of recalibration are: inheritance is replaced by composition, and interfaces are implemented implicitly.

I. Methods: functions with a receiver

A method is just an ordinary function with its first parameter (the receiver) written up front. (Recall the value receiver vs pointer receiver from Chapter 06)

type Point struct{ X, Y int }

func (p Point) Add(q Point) Point {   // value receiver: p is a copy of the Point
    return Point{p.X + q.X, p.Y + q.Y}
}
func (p *Point) Scale(f int) {        // pointer receiver: can modify in place
    p.X *= f
    p.Y *= f
}

There is no magic underneath: p.Add(q) is equivalent to Point.Add(p, q), and the receiver is just an ordinary passed argument—a value receiver copies the whole Point, a pointer receiver copies one pointer. There is no C++-style hidden this-pointer semantic difference; everything is explicit.

A method can be defined on any named type defined within the current package, not just structs:

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

II. Composition: embedding and method promotion

Go doesn’t reuse via inheritance; it uses embedding: put an anonymous field (just the type, no field name) inside a struct.

type Engine struct{ Power int }
func (e Engine) Start() { fmt.Println("engine start", e.Power) }

type Car struct {
    Engine       // embedding (anonymous field), not inheritance!
    Brand string
}

c := Car{Engine{100}, "Toyota"}
c.Start()        // method promotion: equivalent to c.Engine.Start()
fmt.Println(c.Power) // field promotion: equivalent to c.Engine.Power

Memory layout: embedding simply inlines the entire Engine as one field of Car, no different from an ordinary field:

Car in memory:
┌──────────────┬───────────────┐
│ Engine.Power │ Brand(string) │   Engine's fields expand in place within Car
└──────────────┴───────────────┘

This is not C++ inheritance (no base-class subobject vptr, no is-a relationship). The compiler merely does “name forwarding”: c.Start() is rewritten at compile time to c.Engine.Start(). No runtime overhead, no vtable.

Comparison with C++ inheritance

Dimension C++ inheritance Go embedding
Relationship is-a (Car is a kind of Engine) has-a + forwarding (Car has an Engine)
Polymorphism entry virtual + vtable Via interfaces (see below)
Override Virtual-function override An outer method with the same name “shadows” the inner one
Accessing the base Base::foo() c.Engine.Start() explicit path
Diamond problem of multiple inheritance Requires virtual inheritance Embed multiple types; same-name conflicts require explicit qualification, the compiler reports ambiguity

Method shadowing example:

func (c Car) Start() { fmt.Println("car:", c.Brand); c.Engine.Start() }
// now c.Start() calls the outer one; to call the inner one write c.Engine.Start() explicitly

Note: embedding is static forwarding, not virtual dispatch. If Engine.Start internally calls another method, it still calls Engine’s own, and will not call back into Car’s overriding version—fundamentally different from C++ virtual functions where “a derived-class override gets called by the base class.” For that kind of callback you need interfaces.

III. Interfaces: implicitly implemented structural types

An interface defines a set of method signatures. As long as a type implements all of the interface’s methods, it automatically satisfies that interface—no implements, no explicit declaration needed.

type Stringer interface {
    String() string
}

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

var s Stringer = Celsius(37.5)  // automatically satisfied, no need to declare "Celsius implements Stringer"

This is called structural typing / a static version of duck typing. The benefits: an interface can be defined “after the fact” for a type you cannot modify (even a standard-library type); it decouples the definer from the user. C++20’s concept is somewhat similar, but interfaces are runtime polymorphism (with a vtable), closer to C++’s abstract base class + virtual functions.

The low level of an interface value: two words

An interface value in memory is two machine words (mentioned in Chapter 04). There are two kinds:

iface—a non-empty interface (an interface with methods):

type iface struct {
    tab  *itab          // type info + method table (equivalent to a vtable)
    data unsafe.Pointer  // points to the actual data
}

eface—the empty interface any (i.e. interface{}):

type eface struct {
    _type *_type         // dynamic type info
    data  unsafe.Pointer  // points to the actual data
}

itab is the key: it pairs an “interface type” with a “concrete type” and caches the method address table:

type itab struct {
    inter *interfacetype // interface type
    _type *_type         // concrete type
    hash  uint32
    fun   [1]uintptr     // method address array (variable-length), equivalent to a vtable
}
var s Stringer = Celsius(37.5)

s (interface value, 16 bytes):    itab (one per (interface, type) combination, globally cached):
┌──────────┐               ┌────────────────────────┐
│ tab  ────┼──────────────▶│ inter: Stringer type    │
│ data ────┼───┐           │ _type: Celsius type     │
└──────────┘   │           │ fun[0]: &Celsius.String │◀─ method address
               │           └────────────────────────┘
               ▼
        the Celsius value 37.5 on the heap/stack (boxing: the value is copied to where data points)

The key difference from a C++ vtable: a C++ object stores a vptr inside itself, and the vtable is uniquely determined by the class; Go’s method table is inside the interface value (itab), the same concrete type has different itabs for different interfaces, and itabs are generated on demand and globally cached. Calling s.String() = look up tab.fun[0] to get the address → call it with data as the receiver, one indirect jump, comparable in cost to a virtual call.

Boxing: when you stuff a concrete value into an interface, if the value is not a pointer, it usually must be copied to the heap so data can point at it (escape, see Chapter 03). So var x any = 42 may trigger a small allocation (the runtime has caching optimizations for small integers, etc.). This resembles Java/C# boxing, with no counterpart in C++.

The nil-interface trap (a must-know for interviews)

An interface value == nil if and only if both words, tab (or _type) and data, are nil. If the interface holds a “non-empty type but nil value” pointer, the interface itself is not equal to nil:

func doErr() error {
    var p *MyError = nil   // a nil pointer of a concrete type
    return p               // boxed into the error interface: tab=*MyError, data=nil
}

func main() {
    err := doErr()
    if err != nil {        // surprisingly true!!
        fmt.Println("non-nil:", err == nil) // non-nil: false
    }
}

Because the returned interface value is {tab: *MyError, data: nil}, and tab is non-nil, err != nil. This is one of the most common bugs among newcomers. Rule: when a function returns error, either return a genuine nil (return nil), or don’t return a nil pointer of a concrete type. (Chapter 09 on error revisits this)

Type assertions and type switch

Retrieving the concrete type from an interface:

var i any = "hello"
s := i.(string)         // assertion, panics on failure
s, ok := i.(string)     // comma-ok form, ok=false on failure, no panic

At the low level: compare the interface’s _type/itab against the target type; if they match, return data (dereferencing/copying as necessary).

A type switch dispatches on the dynamic type:

switch v := i.(type) {
case int:
    fmt.Println("int", v)     // v is an int
case string:
    fmt.Println("string", v)  // v is a string
case fmt.Stringer:
    fmt.Println(v.String())   // can also match an interface
default:
    fmt.Println("other")
}

IV. Method sets: can a value or a pointer satisfy an interface?

Which methods “belong to” a type determines whether it can satisfy an interface. The rule:

Receiver defined as T’s method set *T’s method set
func (t T) ✓ ✓
func (t *T) ✗ ✓

That is: the pointer type *T has all methods; the value type T has only value-receiver methods. Because with a *T you can automatically dereference to a value, but with a T you cannot necessarily obtain a stable address (e.g. a map element or a temporary value) to call a pointer method.

type Speaker interface{ Speak() }
type Dog struct{}
func (d *Dog) Speak() {}      // pointer receiver

var s Speaker = &Dog{}        // OK
// var s Speaker = Dog{}      // compile error: Dog does not implement Speaker (Speak is on *Dog)

Practical advice: use all value receivers or all pointer receivers, don’t mix. If you need to modify state, the struct is large, or you want *T to satisfy an interface, use a pointer receiver.

Method values vs method expressions

d := Dog{}
f := d.Speak            // method value: binds the receiver d, type func()
f()                     // equivalent to d.Speak()

g := (*Dog).Speak       // method expression: the receiver becomes an explicit first parameter
g(&d)                   // equivalent to d.Speak()

A method value captures the receiver (which may cause it to escape to the heap, like a closure); a method expression merely treats the method as an ordinary function, with the receiver passed explicitly.

V. Generics (Go 1.18+): type parameters and constraints

Go long lacked generics; 1.18 introduced type parameters. Syntax: declare type parameters in square brackets, paired with a constraint.

// constraint: an interface listing the allowed set of types
type Number interface {
    ~int | ~int64 | ~float64   // ~T means "all types whose underlying type is T"
}

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

Sum([]int{1, 2, 3})        // the type argument int is inferred automatically
Sum([]float64{1.5, 2.5})

comparable is a built-in constraint (types usable with ==/!=), commonly used for map-key-like generics:

func Keys[K comparable, V any](m map[K]V) []K { /* ... */ }

The essence of a constraint: it is simply an interface, which can list methods (requiring the implementation of a method) as well as a set of types (| ~T).

The low-level implementation: GC shape stenciling + dictionaries

This is where Go differs most from C++ templates. A C++ template is fully monomorphized: vector<int> and vector<string> generate two entirely separate copies of machine code, with zero runtime overhead but code bloat and slow compilation. Go takes a middle path:

Generate code grouped by “GC shape” + pass a dictionary. Specifically:

  • The compiler generates code grouped by GC shape. The current implementation’s grouping rule is: two types belong to the same shape if and only if their underlying types are the same, or both are pointers. So all pointer types (*int, *string, …) share one instantiation; whereas int and int64 have different underlying types, so even though both are 8 bytes and neither contains pointers, each generates its own copy (not shared).
  • Each call additionally passes a hidden dictionary parameter, which holds the concrete type’s meta-information (type descriptor, needed itabs, etc.), for the runtime to look up the concrete size and method addresses.
C++ template (monomorphization):    Go generics (shape stenciling):
Sum<int>    → code copy A     Sum[int]     → int's shape code + int dictionary
Sum<int64>  → code copy B     Sum[int64]   → int64's shape code + int64 dictionary (not shared with int)
Sum<*T>×N   → N copies        Sum[*T]×N    → 1 copy of pointer shape code + individual dictionaries

Conclusion: somewhere between monomorphization and boxing. Faster than Java’s type erasure + boxing (which boxes everywhere), more code-frugal than C++ monomorphization; but dictionary lookups and indirect access through pointers bring a bit of runtime overhead, and in some scenarios it’s not as good as hand-written concrete types nor as extreme as C++ templates.

C++ template Go generics Java generics
Mechanism Monomorphization shape stenciling + dictionary Type erasure + boxing
Code size Large (one per type) Medium (one per shape) Small (one)
Runtime overhead None Slight (dictionary/indirection) Large boxing overhead
Compile-time computation Strong (template metaprogramming) Weak (just parameterization) None

When to use generics

Don’t do generics for the sake of generics. Good fits: containers/algorithms (the slices, maps packages, see Chapter 11), performing the same operation on multiple numeric types. In many scenarios an any + type assertion or an ordinary interface is clearer instead.
