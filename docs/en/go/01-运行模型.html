Runtime Model: Compilation, Linking, Runtime

This chapter starts by getting a Go program written and running—no background knowledge required, just follow along and it works. Once it runs, we’ll come back and ask the question every C/C++ programmer naturally asks: what exactly does this binary turn into, and what’s inside it while it’s running? If you only care about using it, read the first section and jump to the next chapter; if you want to understand “what’s happening under the hood” and how it differs from C’s compile-link-run model, keep reading.

1. Just Using It: Write a Go Program, Run It

You don’t need to know any underlying principles to write and run a Go program with the steps below.

Step one: write the code. Create a file hello.go:

package main

import "fmt"

func main() {
    fmt.Println("hello, world")
}

The three lines: package main declares this is an executable program (as opposed to a library meant to be imported by others); import "fmt" pulls in the standard-library package for formatted output; func main() is the program’s entry point, equivalent to C’s int main(void), except Go’s main takes no arguments and returns nothing. The syntax details come chapter by chapter starting next chapter—for now just copy it and get it running.

Step two: run it directly. The easiest way is go run, which runs it like a script:

$ go run hello.go
hello, world

Step three: compile a real executable, similar to C’s gcc -o hello hello.c:

$ go build -o hello hello.go
$ ./hello
hello, world

Step four: try out “concurrency.” Put a go keyword before a function call and that call gets thrown into the background to run asynchronously—this is the syntactic entry point to Go’s concurrency support (just look at the syntax for now; the behavioral details come in the second half of this chapter):

package main

import (
    "fmt"
    "time"
)

func sayHi() {
    fmt.Println("hi, from another goroutine")
}

func main() {
    go sayHi()                        // launch a goroutine, don't wait for it to finish before moving on
    time.Sleep(time.Millisecond * 10) // crudely wait a bit so it gets a chance to run
    fmt.Println("main done")
}

Output:

hi, from another goroutine
main done

Step five: when the project grows, declare a module with go.mod (single-file go run/go build doesn’t need this step, but bringing in dependencies or organizing a multi-file project does):

$ go mod init example.com/hello
go: creating new go.mod: module example.com/hello

These few steps are already enough to start writing and running code. Now let’s go under the hood—the same plain hello.go, and exactly what gets packed into the compiled binary beyond what a C version would have.

2. Going Deeper: What Exactly Is Sealed Inside the Binary

As a C/C++ programmer, you have a very clear mental model of “how a program becomes something runnable”: preprocess → compile to .o → link (static or dynamic) → produce an ELF/Mach-O executable → loaded by the kernel and bootstrapped into main by libc’s _start. Go’s model is both like and unlike this one. What’s the same: the final artifact is a native machine-code executable—no virtual machine, no bytecode interpreter. What’s different: this binary additionally statically links in a runtime, which has already started running before your main.

From here on, this chapter peels this binary apart layer by layer, contrasting it with C’s model at every step.

3. Overview: A Go Binary = Your Code + Runtime + Standard Library, Statically Packaged

First the conclusion, in a comparison table:

Dimension C/C++ (gcc/clang) Go (gc toolchain) JVM (Java)
Final artifact Native machine code Native machine code Bytecode .class/.jar
Execution Directly executed by the CPU Directly executed by the CPU JVM interpretation + JIT
Has a runtime? Extremely thin (crt0 + libc, usually dynamically linked) Yes, fairly thick, and statically linked in by default Yes, a separately installed JVM
Default linking Dynamically links libc Static (no libc dependency on Linux + pure Go; macOS still links libSystem) N/A
Deployment Requires a matching .so on the target machine A single self-contained file—copy it over and it runs Requires a JRE on the target machine
GC None (manual malloc/free) Yes (concurrent tricolor mark) Yes
Threading model 1:1 mapping to OS threads M:N, goroutines multiplexed onto OS threads Platform threads / virtual threads

The key mental model: Go sits between C and the JVM. Like C, it compiles to machine code executed directly by the CPU (no interpretation, no JIT warm-up), but like the JVM it ships with a runtime responsible for scheduling, GC, and memory allocation. The difference is that this runtime is not an external process, but is statically linked into your binary, existing as ordinary functions and code segments.

   Address space of a C program      Address space of a Go program
 ┌────────────────────┐         ┌────────────────────┐
 │  your .text        │         │  your .text        │
 │  libc (often dyn.) │         │  runtime .text     │  ← scheduler/GC/allocator
 │  crt0 bootstrap    │         │  stdlib .text      │  (all compiled in statically)
 ├────────────────────┤         ├────────────────────┤
 │  heap (malloc/free)│         │  heap (runtime-mgd) │
 │  one/more thread   │         │  many goroutine    │  ← ~2KB each to start, growable
 │  stacks            │         │  stacks            │
 └────────────────────┘         └────────────────────┘

4. go build: The Truth About Static Linking

The first section already used go build. What’s more worth digging into for a C programmer is that it statically links by default (more precisely, this mainly refers to the Linux + CGO_ENABLED=0 case). Write a pure-Go hello and on Linux, ldd hello will tell you not a dynamic executable—it hasn’t linked libc, it doesn’t even need libc.so.6. (macOS is an exception: Apple does not support statically linking libSystem, so a Go binary on macOS will always dynamically link /usr/lib/libSystem.B.dylib, visible with otool -L.) This means (on Linux):

  • You can scp this binary directly to another Linux machine of the same architecture and run it, even if that machine has no runtime libraries installed at all (in the extreme case: put just this one file in an empty FROM scratch Docker image and it runs).
  • The cost is size: a hello world Go binary is typically 1–2 MB, because the runtime and the standard-library packages you use are all compiled in. A dynamically-linked C hello might be only a few dozen KB (but it depends on the several MB of libc on the system).

When Does Go Link Dynamically?

It’s not absolutely static. Two common situations introduce dynamic linking:

  1. cgo: once you use import "C" to call a C library (such as certain resolution paths in net or os/user, or a third-party C dependency), libc gets linked and produces a dynamic dependency. You can force cgo off with CGO_ENABLED=0 to return to fully static.
  2. Explicit -linkmode=external or linking against system libraries.
CGO_ENABLED=0 go build -o hello   # force fully static, commonly used when building Docker images

5. What’s Actually Inside the Runtime

C’s “runtime” is so thin it’s almost only crt0’s bootstrap plus the library functions provided by libc; control is handed to your main very quickly, and after that there’s no “background magic” unless you call into a library. Go is different: before your main.main is called, the runtime has already initialized a whole set of subsystems, and it keeps working in the background for the duration of your program. Three main pieces:

1. The Goroutine Scheduler (GMP Model)

Go does not map goroutines directly onto OS threads (that’s C’s 1:1 pthread model); instead it uses M:N scheduling: thousands upon thousands of goroutines are multiplexed onto a small number of OS threads. Three core abstractions:

G (goroutine)  ── one goroutine, with its own stack, PC, and state. Extremely lightweight.
M (machine)    ── one OS thread, actually running code on the CPU.
P (processor)  ── a logical processor/scheduling context, count defaults to GOMAXPROCS (usually = #CPU cores).
                  P holds a local runnable-G queue; an M must bind to a P to execute a G.

     M0 ── P0 ── [G,G,G] local queue
     M1 ── P1 ── [G,G]
                 [G,G,G,...] global queue (a P steals from here when its queue is empty)

When a goroutine blocks in a system call, the runtime detaches the M from the P, letting another M take over that P and keep running other Gs—so a few threads can sustain massive concurrency. For details see Chapter 10, “The Concurrency Model”. By contrast, spinning up a thread in C costs several MB of stack plus kernel scheduling overhead, so a few thousand is already a strain; running hundreds of thousands of goroutines in Go is entirely routine.

The go sayHi() in the first section was just syntax; you can see the scheduler’s existence more directly with the runtime package. Pin the number of Ps to 2, then spawn 1000 goroutines—all of them multiplexed onto those 2 logical processors:

package main

import (
    "fmt"
    "runtime"
)

func main() {
    runtime.GOMAXPROCS(2)              // pin logical processors P to 2 (deterministic)
    fmt.Println(runtime.GOMAXPROCS(0)) // 2   (passing 0 = query only, don't change)

    fmt.Println(runtime.NumGoroutine()) // 1   (only the main goroutine so far)

    ch := make(chan struct{})
    for i := 0; i < 1000; i++ {
        go func() { <-ch }() // 1000 goroutines, all parked waiting on ch
    }
    for runtime.NumGoroutine() < 1001 { // wait until all have been created
        runtime.Gosched()
    }
    fmt.Println(runtime.NumGoroutine()) // 1001 (main + 1000), running on just 2 OS threads

    close(ch) // release them all at once
}

Output:

2
1
1001

1001 goroutines coexist happily on 2 Ps—with C’s 1:1 threads, 1000 threads would need several GB of stack alone. This is the M:N scheduling model made concrete.

2. The Garbage Collector (GC)

In C you malloc/free yourself, and dangling pointers and memory leaks all rely on self-discipline. Go has a concurrent, tricolor mark-and-sweep GC running in the background, executing concurrently with your code, with STW (stop-the-world) pauses typically in the sub-millisecond range. This means you almost never manually free memory—but you do need to understand escape analysis (why a variable ends up on the heap), or you’ll write GC-unfriendly code. Escape analysis is covered in Chapter 3; GC details are in Chapter 9.

3. The Memory Allocator (tcmalloc-style)

Go’s allocator descends from Google’s tcmalloc, using a multi-level cache structure, with the core goal of allocating small objects as lock-free as possible:

mcache   ── one per P, thread-local, allocates small objects without locking (like tcmalloc's thread cache)
   │ requests downward when short
mcentral ── a central cache organized by size class, shared across Ps, requires locking
   │ requests downward when short
mheap    ── the global heap, managing large chunks of memory obtained from the OS (span/arena)

Objects are grouped by size into roughly 70 size classes; small objects are taken directly from the P-local mcache with no global lock contention—this is one reason Go’s allocation performance is good under high concurrency. Compared with C’s malloc (typically a global lock or arena lock), Go’s per-P cache is naturally multicore-friendly.

6. Why Go Compiles So Fast

A large C++ project can compile slowly enough that you go get a coffee, and you know the reasons well: header files textually included over and over, templates instantiated in every compilation unit, huge amounts of preprocessor expansion, and heavy symbol resolution at link time. Go was designed to deliberately sidestep all of these:

  1. No header files, no preprocessor. Dependency information is encoded directly in the package; compiling a package only requires reading the exported information of the packages it directly imports (the compiled object carries compact export data), and does not require recursively expanding the source of the entire dependency tree. In C, #include <a.h> where a.h includes a bunch more, and every .c re-parses them all over again.
  2. Imports are package-level and acyclic. Go mandates that the package dependency graph have no cycles, and imports the compiled artifact rather than source, so the compiler need only read one summary.
  3. The grammar is designed for fast parsing. Go’s grammar can essentially be parsed in a single pass without consulting semantics (see automatic semicolon insertion in Chapter 2); there is no C++-style ambiguity where “you need the symbol table to decide whether a * b is a multiplication or a pointer declaration.”
  4. Parallel compilation: different packages can be compiled in parallel.

The cost: fast compilation is bought with “a thicker runtime + a larger binary + GC overhead,” plus trade-offs in language expressiveness (no generics early on, etc.).

7. Common Commands: go run / build / mod

The first section already used go run, go build, and go mod init. The full set of common commands:

go run main.go        # compile to a temp directory and run immediately (leaves no binary), for quick experiments
go build ./...        # compile all packages in the current module, produce an executable (when there's a main package)
go install ./cmd/foo  # compile and install the binary into $GOBIN (default ~/go/bin)
go test ./...         # compile and run tests
go vet ./...          # static analysis for suspicious code

Note that go run is not interpreted execution—it really does compile to native code first and then run, it just puts the binary in a temp directory and deletes it when done. It gives you a “script-like experience,” not “a script’s execution model.”

go modules: Dependency Management

Go uses modules to manage dependencies (replacing the old GOPATH mode):

go mod init example.com/myapp   # generate go.mod, declaring the module path
go get github.com/foo/bar@v1.2.3 # add/upgrade a dependency, writing go.mod and go.sum
go mod tidy                      # add/remove dependencies to match the source imports
  • go.mod: declares the module path, Go version, dependencies and their versions.
  • go.sum: records hash checksums of dependencies, guaranteeing reproducible builds (like a lock file).

Dependencies are downloaded to a local cache ($GOPATH/pkg/mod) and statically compiled into your binary at build time. There’s no C-style “go find the .so at runtime” problem—dependencies are frozen in at compile time.

8. Cross-Compilation: Just Change Two Environment Variables

Cross-compilation in C is notoriously painful: you need a cross-toolchain for the target platform (a cross gcc), the target platform’s libc, a sysroot, and so on. Because Go by default doesn’t depend on libc, and the toolchain ships with the runtime for every target platform, cross-compilation is just setting two environment variables:

# On macOS/arm64, build for Linux/amd64:
GOOS=linux GOARCH=amd64 go build -o hello-linux-amd64

# Build for Windows:
GOOS=windows GOARCH=amd64 go build -o hello.exe

# Build for a Raspberry Pi (Linux/arm64):
GOOS=linux GOARCH=arm64 go build -o hello-pi

go tool dist list   # list all supported GOOS/GOARCH combinations
  • GOOS: target operating system (linux/darwin/windows/freebsd…).
  • GOARCH: target CPU architecture (amd64/arm64/386/riscv64…).

The prerequisite is pure Go (CGO_ENABLED=0). Once you use cgo, cross-compilation degenerates back into the painful C-world mode of “needing a C cross-toolchain for the target platform.”

9. The Final Contrast with C: A Single Self-Contained Binary

To sum up this chapter in one sentence: Go aims for “compile once, get a self-contained native binary that can be copied across machines and run as-is.”

  • Versus C/C++: you’re spared shared-library version hell (GLIBC_2.34 not found and the like), and the hassle of assembling the right .so files at deployment; the cost is a larger binary, plus a runtime (GC/scheduler) you can’t fully turn off.
  • Versus the JVM: you don’t need to install a JRE on the target machine, and startup has no JIT warm-up (fast cold start), but you give up the JVM’s deep runtime introspection and peak JIT performance.

Once you understand “there’s a continuously running runtime hidden inside the binary,” much of the “magic” in later chapters has a foundation: why goroutines are cheap (scheduler + growable small stacks), why some variables end up on the heap (escape analysis + GC), why you don’t have to free slices/maps. In the next chapter we come back down to earth and go through Go’s basic syntax point by point against C.

Summary

  • At the usage level: go run file.go compiles and runs immediately; go build -o out file.go produces an executable; putting the go keyword before a function call launches a goroutine that runs in the background; for multi-file projects with dependencies, declare a module with go mod init; GOOS=xxx GOARCH=xxx go build cross-compiles.
  • At the underlying level: a Go binary = your code + runtime (scheduler + GC + allocator) + the standard-library packages you use, statically packaged into a single native machine-code file; it doesn’t depend on libc by default (except with cgo); the GMP model lets massive numbers of goroutines multiplex onto a small number of OS threads; a concurrent tricolor mark-and-sweep GC runs in the background; the allocator uses a mcache/mcentral/mheap multi-level cache; no header files or template instantiation makes compilation far faster than C++.
