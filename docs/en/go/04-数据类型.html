Data Types

The previous chapter made the memory semantics clear (value copy, pointers, escape). This chapter lays out Go’s type system one type at a time, with the focus still on the underlying memory layout, contrasting with C throughout: which types are fixed-width, how wide int really is, how a struct is aligned, why a string isn’t a char*, and what an interface value actually is.

Fixed-Width Numeric Types

The widths of Go’s numeric types are fixed by the language specification, unlike C where the widths of int/long drift with platform and compiler. This is enormous certainty for anyone writing cross-platform code or protocols.

Type Width Notes
int8 int16 int32 int64 explicit signed, two’s complement
uint8 uint16 uint32 uint64 explicit unsigned
float32 float64 4 / 8 bytes IEEE-754, corresponding to C’s float/double
complex64 complex128 8 / 16 bytes built-in complex numbers, which C needs a library for
int uint platform word size 4 bytes on 32-bit platforms, 8 bytes on 64-bit
uintptr platform word size an integer large enough to hold a pointer, used with unsafe

Key points contrasted with C:

  • int is the platform word size (8 bytes on a 64-bit machine), similar to the width of C’s intptr_t, but with plain-integer semantics. For everyday counting and loop indices, just use int.
  • There is no implicit integer promotion, and no implicit type conversion. In C, int + long and int + double promote automatically; in Go, different types cannot be mixed directly and must be converted explicitly:
var a int32 = 1
var b int64 = 2
// c := a + b        // compile error: mismatched types int32 and int64
c := int64(a) + b    // must convert explicitly
var f float64 = float64(a) // int32 → float64 also requires explicitness

This “verbosity” is deliberate: C’s implicit conversions and integer promotions are a breeding ground for countless overflow, truncation, and sign bugs, and Go bans them outright at the language level.

rune and byte

Two important aliases, integers underneath, but the semantic naming makes code clearer:

Alias Equivalent to Purpose
byte uint8 one byte, for handling binary/ASCII
rune int32 one Unicode code point

rune is int32 rather than one byte because a Unicode code point needs up to 21 bits. Understanding this is crucial when iterating over strings (detailed in the string section below).

var b byte = 'A'   // 65, occupies 1 byte
var r rune = '中'  // code point 20013; rune is int32, so the variable occupies 4 bytes—this is the type width, not the UTF-8 length ('中' is 3 bytes in UTF-8, see Chapter 05)
fmt.Println(b, r)  // 65 20013

Untyped Constants and Implicit Conversion Rules

Since Go forbids implicit type conversion, why does something like var x float64 = 3 compile? Because Go has a unique untyped constant mechanism.

The literals 3, 3.14, 'A', true, "hi" are “untyped” before they acquire a definite type; they exist at arbitrary precision at compile time, and only “default” or convert to the corresponding type per context when they are assigned or participate in an operation that requires a concrete type. This lets constants be used naturally across types, whereas runtime variables cannot:

const k = 3          // untyped integer constant
var a int32 = k      // OK, k adapts to int32
var b float64 = k    // OK, the same k adapts to float64
var c float64 = 3.0  // OK
// var d int32 = a + b // an error if a, b are variables of different types; but constants aren't bound by this

Untyped constants can also retain extremely high precision until they land:

const big = 1 << 62      // arbitrary precision at compile time, legal
const huge = 1 << 100    // the constant itself is legal (arbitrary precision)
// var x int64 = huge    // error: overflows int64, checked only when it lands

Every untyped constant has a default type, used when no context forces one: integers default to int, floats to float64, characters to rune, booleans to bool, strings to string.

const and iota

const declares compile-time constants. iota is a counter that starts at 0 and increments line by line within each const block, specifically for generating enumerations:

type Weekday int

const (
    Sunday Weekday = iota // 0
    Monday                // 1 (the expression carries over from the previous line, iota increments)
    Tuesday               // 2
    Wednesday             // 3
)

// Bit flags: iota combined with shifting
const (
    FlagRead    = 1 << iota // 1 << 0 = 1
    FlagWrite               // 1 << 1 = 2
    FlagExecute             // 1 << 2 = 4
)

Compared with C’s enum, iota is more flexible (it can do shifts, skip values, use expressions), but it too is merely a compile-time constant and doesn’t carry the type safety of C++’s enum class—type safety you get by defining type Weekday int as above.

An array Is a Value, a slice Is Not

Emphasizing again (because C programmers step on this too easily): an array is a value type, fixed-length, and assignment and argument passing copy the entire array; a slice is a header containing a pointer.

func modify(a [3]int)  { a[0] = 99 } // copies the entire array, modifies the copy
func modifyS(s []int)  { s[0] = 99 } // copies the header, shares the underlying array

func main() {
    arr := [3]int{1, 2, 3}
    modify(arr)
    fmt.Println(arr[0]) // 1 —— unchanged

    sl := []int{1, 2, 3}
    modifyS(sl)
    fmt.Println(sl[0]) // 99 —— changed
}
array [N]T slice []T
Length part of the type, fixed at compile time variable at runtime
Assignment/passing copies all elements copies the 3-word header, shares the underlying
Memory representation N contiguous elements {ptr, len, cap} header
C analogy int a[3] (but no decay when passed) struct{int*; int; int;}

The full details of slices are in 07-复合数据结构.md; here just remember that an array is a “whole-block value” and a slice is a “pointer header.”

struct: Memory Layout, Alignment, and Padding

A struct’s fields are laid out contiguously in declaration order in memory, exactly like C’s struct—they likewise obey memory alignment rules and likewise insert padding. A C programmer’s alignment knowledge carries over 100% here.

Alignment rules: each type has an alignment value (usually equal to its size); a field’s offset must be a multiple of its alignment value; and the whole struct’s size must be a multiple of its largest field’s alignment value (to allow contiguous layout in arrays).

unsafe.Sizeof / unsafe.Alignof / unsafe.Offsetof let you verify precisely:

package main

import (
    "fmt"
    "unsafe"
)

// Poor field order: bool, int64, bool
type Bad struct {
    a bool  // offset 0, occupies 1
    // 7 bytes of padding (to align b to 8)
    b int64 // offset 8, occupies 8
    c bool  // offset 16, occupies 1
    // 7 bytes of padding (rounding the whole up to a multiple of 8)
}

// After reordering: int64, bool, bool
type Good struct {
    b int64 // offset 0, occupies 8
    a bool  // offset 8, occupies 1
    c bool  // offset 9, occupies 1
    // 6 bytes of padding
}

func main() {
    fmt.Println(unsafe.Sizeof(Bad{}))  // 24
    fmt.Println(unsafe.Sizeof(Good{})) // 16
    fmt.Println(unsafe.Alignof(int64(0)), unsafe.Offsetof(Bad{}.b))
}

Memory diagram contrast (each cell is 1 byte):

Bad (24 bytes):
offset 0        8               16       24
   [a][×××××××][b b b b b b b b][c][×××××××]
    1B   7pad        8B          1B   7pad

Good (16 bytes):
offset 0               8   9        16
   [b b b b b b b b][a][c][××××××]
        8B          1B 1B   6pad

Merely adjusting field order cuts it from 24 bytes down to 16—put large-alignment fields first and cluster small fields together to reduce padding. This is exactly the same struct-slimming trick as in C. When allocating large numbers of the same struct (say, a million records), this optimization is worth it.

Tip: Go doesn’t offer #pragma pack. Field order is your only means of controlling layout (short of going to unsafe).

Anonymous Fields and Embedding

Go uses embedding in place of C++ inheritance. Writing a type into a struct without a field name is embedding; the outer type automatically “promotes” the inner type’s fields and methods, which you can access directly:

type Base struct {
    ID   int
    Name string
}

func (b Base) Hello() string { return "hi " + b.Name }

type User struct {
    Base        // embedded, anonymous field
    Email string
}

func main() {
    u := User{Base: Base{ID: 1, Name: "Alice"}, Email: "a@x.com"}
    fmt.Println(u.Name)    // field promotion: equivalent to u.Base.Name
    fmt.Println(u.Hello()) // method promotion: equivalent to u.Base.Hello()
}

Under the hood, embedding has no magic at all: Base is just an ordinary field named Base inside User, laid out contiguously in memory, and u.Name is merely syntactic sugar for u.Base.Name. This is composition, not inheritance—there is no parent-child relationship in the vtable sense, and no C++-style base-class-subobject + vptr overhead. The polymorphism of method dispatch is carried by interfaces (see 08-面向对象与类型系统.md).

string: A Read-Only (ptr, len) Header

A C programmer’s biggest misconception about string is to treat it as a char*. Go’s string is not a \0-terminated character pointer, but a two-word read-only header:

type stringHeader struct {
    Data uintptr // pointer to the underlying byte array
    Len  int     // byte length (no trailing 0, and no trailing 0 exists)
}

Key points:

  • The underlying data is a run of UTF-8-encoded bytes, not an array of code points. len(s) returns the byte count, not the character count.
  • Immutable. Once a string is created, its underlying bytes are read-only; you cannot do s[i] = 'x'. This makes copying and slicing strings extremely cheap—just share the underlying bytes, with no worry about them being modified.
  • s := "hello"; t := s copies the 16-byte header, sharing the same underlying bytes, and is safe (because it’s read-only).
s := "héllo"       // é is 2 bytes in UTF-8
fmt.Println(len(s)) // 6, not 5 (byte count)

// Index by byte
fmt.Println(s[0])   // 104, a byte, 'h'

// Iterate by code point: range decodes UTF-8 automatically, i is the byte offset, r is the rune
for i, r := range s {
    fmt.Printf("%d:%c ", i, r) // 0:h 1:é 3:l 4:l 5:o (note the offset jumps)
}
fmt.Println(len([]rune(s))) // 5, the real character count

Converting Between string and []byte Copies

Because string is read-only while []byte is writable, converting between the two must copy the underlying bytes; they cannot share (otherwise you could modify an “immutable” string through the []byte):

b := []byte("hello") // allocates a new array, copies 5 bytes
s := string(b)       // copies again

Frequent string ↔︎ []byte conversions on a hot path incur allocation and copy overhead, a common performance pitfall. (The compiler optimizes certain local cases, e.g. a map[string(b)] lookup won’t actually allocate a string; more in later chapters.) Contrast with C: in C, char* and a string are one and the same, casting doesn’t copy, but you have to manage mutability and bounds yourself—Go trades “one copy” for the safety of immutability.

interface: A (type, data) Two-Word

The interface is the core of Go’s type system, and its runtime representation is a two-word struct: a pointer to type information and a pointer to the data.

// Runtime layout of a non-empty interface (with methods), conceptually:
type iface struct {
    tab  *itab          // type + method table (like a C++ vtable + type metadata)
    data unsafe.Pointer // points to the actual data
}
// The empty interface interface{} / any:
type eface struct {
    _type *_type         // dynamic type information
    data  unsafe.Pointer // points to the actual data
}

Key points for C++ programmers:

  • An interface value = a (dynamic type, data pointer) two-word. The itab contains the method table, and method calls are dispatched indirectly through it—functionally like a vtable, but the vtable is not in the object, it’s in the interface value. A plain struct has no vptr and takes no extra space; the type information is only carried along when you put it into an interface.
  • When you assign a value to an interface, if the value isn’t a pointer, it’s usually copied to the heap (escapes), with data pointing to that copy. So var x any = bigStruct may trigger a heap allocation and copy.
  • A nil interface has both words nil. There’s a classic trap: put a concrete pointer whose value is nil into an interface, and the interface’s _type is non-nil, so iface != nil is true—“the interface isn’t nil but holds a nil pointer inside.”
var i interface{} = 42
fmt.Println(unsafe.Sizeof(i)) // 16 (64-bit: two words)

// Type assertion: retrieve the dynamic type
n := i.(int)      // assertion succeeds, returns 42
m, ok := i.(string) // ok=false, doesn't panic
fmt.Println(n, m, ok)

The interface method-set rules, type assertions, type switch, and interplay with embedding are systematically covered in 08-面向对象与类型系统.md. Here we first establish the underlying picture that “an interface value is a (type, data) two-word.”

Summary: Quick Reference for Memory Layout

Type Underlying representation What’s copied on copy
numeric / bool fixed-width raw bytes all the data
array [N]T N contiguous elements all elements
struct fields in order + padding all fields (including embedded)
pointer *T one machine word the address (shares the target)
string (ptr, len) read-only two-word the header (shares read-only bytes)
slice (ptr, len, cap) three-word the header (shares the underlying array)
map / chan a pointer to a runtime structure the pointer (shares the same structure)
interface (type, data) two-word the two-word (the data it points to depends)
