Control Flow

Go deliberately keeps its control flow “small but correct.” The while, do...while, comma operator, and elaborate for(;;) variants of C/C++ are collapsed into a single for keyword, and switch completely flips its default behavior. This chapter explains, from a low-level perspective, what each construct compiles down to, along with the copy semantics behind range and defer—the two spots where seasoned C/C++ programmers are most likely to trip up.

for: the only loop keyword

Go has no while and no do...while. Every loop uses for, which comes in three forms. After compilation they all reduce to the same set of jump instructions (a condition test plus a backward jump), and at the machine-code level there is no difference from a C loop.

package main

import "fmt"

func main() {
    // Form 1: full three-clause, same as C's for(i=0;i<n;i++)
    for i := 0; i < 3; i++ {
        fmt.Println("three-clause", i)
    }

    // Form 2: condition only, equivalent to C's while(cond)
    n := 3
    for n > 0 {
        fmt.Println("while-like", n)
        n--
    }

    // Form 3: no condition, equivalent to C's for(;;) or while(1)
    count := 0
    for {
        if count == 2 {
            break // you must break yourself, otherwise an infinite loop
        }
        fmt.Println("infinite loop", count)
        count++
    }
}

The i := 0 in the three-clause form declares an i whose scope is limited to that for statement (including the loop body), which matches C99’s for(int i...). It is usually allocated on the stack; if a closure inside the loop body captures it (see 06-Functions and the Call Stack), escape analysis will move it to the heap.

The do...while semantics (execute at least once) are written by hand in Go with for + break:

for {
    // loop body runs at least once
    if !condition {
        break
    }
}

range: iteration semantics and copy behavior

range is the construct where you most need to keep an eye on memory. The core fact: the range iteration variable is a copy of the element being iterated. Each loop pass copies the current element into the iteration variable. This matches C++’s for (auto x : v) (value copy), not for (auto& x : v) (reference).

range slice / array

s := []int{10, 20, 30}
for i, v := range s {
    // i is the index, v is a [copy] of s[i]
    // modifying v does not affect s; to change the original you must use s[i] = ...
    v += 100
    _ = v
    _ = i
}
// s is still {10, 20, 30}

Memory diagram (during the 2nd iteration):

underlying array (heap or stack):
  s[0]=10  s[1]=20  s[2]=30
                ↑
         range reads s[1]=20
                │  copy
                ▼
   iteration variable v = 20   ← an independent stack variable; changing it doesn't touch the array

Key pitfall 1: the range expression is evaluated only once, before the loop begins. For a slice, range copies the slice’s header (SliceHeader: pointer + length + capacity), not the underlying array. So an append that triggers reallocation during the loop does not change the number of iterations this pass makes, but if you modify an element in place without reallocating, that change will be read (because the pointer still points to the same array):

s := []int{1, 2, 3}
for i, v := range s {
    if i == 0 {
        s[2] = 99 // no reallocation, modified in place; the 3rd pass will read 99
        s = append(s, 4) // the iteration length is already locked at 3, so 4 is not iterated
    }
    fmt.Println(i, v)
}
// output: 0 1 / 1 2 / 2 99

For an array (not a slice) you must distinguish whether you read the element value: for i, v := range arr first copies the entire array and then iterates (v comes from that copy), which is expensive for large arrays; but for i := range arr, using only the index without reading element values, causes the compiler to not copy the array. To avoid the copy while still reading values, range over an array pointer:

var arr [1024]int
for i := range arr {      // index only: no copy of these 8KB
    arr[i] = i
}
for i, v := range &arr {  // need values but want to avoid the copy: range over &arr, v reads directly from the original array
    arr[i] = v + i
}

range map

m := map[string]int{"a": 1, "b": 2}
for k, v := range m {
    // both k and v are copies; iteration order is [random] (the runtime deliberately randomizes the starting bucket)
    _ = k
    _ = v
}

Go’s map iteration order is randomized at runtime—this is a deliberate design decision to prevent code from depending on order. C++’s std::map is ordered and unordered_map has an unspecified order that is nonetheless stable within a single process; Go doesn’t even give you that stability. Deleting the current or a not-yet-visited key during iteration is safe; a newly added key may or may not be visited—the behavior is undefined.

range string: by rune, not by byte

In C a string is just a char*, one byte at a time. Go’s range string decodes UTF-8, handing you one rune at a time (a Unicode code point, int32), where the index is the byte offset of that rune’s starting byte:

s := "aé中" // UTF-8: 'a'=1 byte, 'é'=2 bytes, '中'=3 bytes
for i, r := range s {
    fmt.Printf("byte offset=%d rune=%c code point=%d\n", i, r, r)
}
// byte offset=0 rune=a code point=97
// byte offset=1 rune=é code point=233
// byte offset=3 rune=中 code point=20013

Note that i skips 2 (because é occupies 2 bytes). To iterate by byte, use a plain index for i := 0; i < len(s); i++ { s[i] }, where s[i] is a byte. On encountering an invalid UTF-8 sequence, range returns RuneError (U+FFFD) and advances 1 byte.

range channel

ch := make(chan int)
// ... another goroutine sends data to ch, then close(ch) when done
for v := range ch {
    // receive one value (a copy) from ch each time, until ch is closed and the buffer is drained
    _ = v
}

range ch is equivalent to repeatedly doing v, ok := <-ch until ok == false (channel closed and empty), at which point it exits. See 10-The Concurrency Model for details.

range int (Go 1.22+)

Since Go 1.22 you can write for i := range n directly, equivalent to for i := 0; i < n; i++:

for i := range 5 { // i goes from 0 to 4
    fmt.Println(i)
}

Go 1.22: a fresh loop-variable instance each iteration

This is a language semantic change with no C/C++ counterpart. Before Go 1.22, the loop variable in for i := 0; ... and in for k, v := range had a single instance across the whole loop, reusing the same memory address every iteration. This caused closures/goroutines capturing it to all point at the same variable (the classic trap):

// Go 1.21 and earlier: all print 3 (sharing the same i); Go 1.22+: print 0 1 2
funcs := []func(){}
for i := 0; i < 3; i++ {
    funcs = append(funcs, func() { fmt.Println(i) })
}
for _, f := range funcs {
    f()
}

Since Go 1.22, the loop variable of each iteration is a fresh instance (logically redeclared each pass), so closures each capture a different variable and the output becomes 0 1 2. This fixes one of the most common bugs in Go’s history. For capture details see 06-Functions and the Call Stack.

if with an initialization clause

if can execute one simple statement before testing the condition; the variable declared in that statement has a scope limited to the if/else block:

if v, ok := m["key"]; ok {
    // v and ok are visible only here
    fmt.Println(v)
} else {
    // v and ok are also visible in the else
    fmt.Println("missing")
}
// here v and ok are no longer visible

This keeps the variable’s lifetime very tight; at the low level v and ok are stack variables within the if statement block. It is idiomatic for error handling: if err := doSomething(); err != nil { ... }.

switch: no automatic fallthrough

In C, a switch case falls through automatically, and forgetting break is a classic bug. Go reverses this: each case automatically breaks by default and does not fall through.

switch x {
case 1:
    fmt.Println("one") // after running it breaks automatically, doesn't fall into case 2
case 2, 3: // a single case can match multiple values (comma-separated)
    fmt.Println("two or three")
default:
    fmt.Println("other")
}

the fallthrough keyword

If you really want to fall through, use an explicit fallthrough (it unconditionally enters the next case body, without checking that next case’s condition):

switch x {
case 1:
    fmt.Println("one")
    fallthrough // force execution to continue into case 2's body
case 2:
    fmt.Println("two") // when x==1 this also runs
}

expression-less switch = an if-else chain

A switch with no expression is equivalent to switch true. Each case is a boolean condition, serving as a cleaner if-else chain:

switch {
case score >= 90:
    grade = "A"
case score >= 60:
    grade = "B"
default:
    grade = "F"
}

switch with an initialization clause

Like if, it can carry an initializer: switch v := f(); v { ... }.

type switch: runtime type dispatch

Dispatches on an interface value’s dynamic type. At the low level it reads the interface’s type pointer (see 08-The Type System and OOP) and compares:

func describe(i interface{}) {
    switch v := i.(type) { // in each case v has the corresponding concrete type
    case int:
        fmt.Println("int:", v*2) // v is an int
    case string:
        fmt.Println("string len:", len(v)) // v is a string
    case nil:
        fmt.Println("nil")
    default:
        fmt.Printf("unknown type %T\n", v)
    }
}

This resembles a C++ dynamic_cast chain or std::variant + std::visit, but is directly backed by the runtime’s interface type information, without the overhead of walking a vtable like RTTI.

labeled break / continue

To break out of the outer loop in nested loops, C uses goto or a flag variable. Go provides labeled break/continue, with the label placed before the loop:

outer:
    for i := 0; i < 3; i++ {
        for j := 0; j < 3; j++ {
            if i*j > 2 {
                break outer // break out of the outer for directly, not just the inner one
            }
            if j == 1 {
                continue outer // advance the outer loop to its next i
            }
            fmt.Println(i, j)
        }
    }

break can also apply to switch/select, and with a label it can break out of the level marked by that label.

goto

Go keeps goto, but with restrictions: it cannot jump over a variable declaration, and it cannot jump into the middle of a block. Its main use is escaping deep nesting or cleaning up on error (in scenarios without defer):

    i := 0
loop:
    if i < 3 {
        fmt.Println(i)
        i++
        goto loop // jump backward
    }

In practice there is almost always a better approach (for, defer), and goto is rarely used.

⭐ defer: deferred execution

defer postpones a function call until just before the current function returns. It is Go’s core mechanism for managing resources (closing files, unlocking, releasing), replacing C++’s RAII/destructors. Understanding it requires nailing down three low-level facts.

Fact 1: LIFO order

Multiple defers in the same function execute in last-in-first-out (stack) order. The runtime pushes each defer record onto the current goroutine’s defer linked list (the _defer struct), and when the function returns it pops them off from the head one by one:

func main() {
    defer fmt.Println("1") // runs last
    defer fmt.Println("2")
    defer fmt.Println("3") // runs first
    fmt.Println("function body")
}
// output: function body / 3 / 2 / 1

Memory sketch (the defer list, with new entries inserted at the head):

registration order: defer1 → defer2 → defer3
list:  head → [defer3] → [defer2] → [defer1] → nil
on return, pop from head and execute: 3, 2, 1

Fact 2: arguments are evaluated at registration time

This is the point people get wrong most often. defer f(x) copies the current value of x into the defer record the moment the defer statement executes, not when the call actually happens:

func main() {
    i := 0
    defer fmt.Println("i seen by defer =", i) // i=0 is copied into the record right now
    i = 100
    fmt.Println("i at end of function body =", i) // 100
}
// output: i at end of function body = 100 / i seen by defer = 0

What is deferred is the execution of the call, not the evaluation of the arguments. To defer reading the latest value until call time, wrap the logic in a closure (a closure captures the variable, see Fact 4):

defer func() { fmt.Println(i) }() // reads i at call time, outputs 100

Putting LIFO and “evaluated at registration” together, a defer inside a loop is the most vivid demonstration:

package main

import "fmt"

func main() {
    for i := 0; i < 3; i++ {
        defer fmt.Println("defer:", i) // each iteration copies the current i into the record
    }
    fmt.Println("function body")
}
// output:
// function body
// defer: 2
// defer: 1
// defer: 0

Each of the three defers remembers the i at its moment of registration (0, 1, 2), and on return they run in LIFO reverse order—so it prints 2, 1, 0.

Fact 3: interaction with named return values / return

return x is two steps at the low level: first it writes the return value into the return slot, then it runs the defers, then it actually returns. If the function uses named return values, a defer can modify this return slot in the second step and thereby change the final returned value:

func inc() (result int) { // named return value result
    defer func() {
        result++ // defer modifies the return slot, changing the final return value
    }()
    return 10 // first sets result to 10, then runs the defer, making result 11
}
// inc() returns 11

Contrast this with anonymous return values: with return 10 the value is already copied to a temporary return location, so changing a local variable inside the defer cannot affect it. This mechanism is commonly used in a defer to uniformly wrap an error (see 09-Error Handling and Resource Management).

Fact 4: the defer-in-a-loop trap

defer is tied to the function return, not to the end of a block/loop. Defers inside a loop accumulate and all run when the function ends, which may exhaust resources:

// bad example: 1000 file handles stay open until the function ends
func bad(files []string) {
    for _, name := range files {
        f, _ := os.Open(name)
        defer f.Close() // all pile up, none closed during the loop!
        // ...
    }
} // only here do all 1000 get closed at once

// good example: extract the loop body into a function, so defer fires when that function returns
func good(files []string) {
    for _, name := range files {
        func() {
            f, _ := os.Open(name)
            defer f.Close() // closed at the end of each iteration
            // ...
        }()
    }
}

defer with panic/recover

defer is also the only opportunity to catch a panic: only by calling recover() inside a deferred function can you intercept a panic and resume normal flow. This “exception” mechanism differs fundamentally from C++’s try/catch and stack unwinding—we cover it in detail in 09-Error Handling and Resource Management.
