Functions and the Call Stack (⭐ the truth about parameter passing)

This is the single most important chapter of the whole tutorial. Go’s functions look like C syntactically, but their memory semantics rest on several facts you must absolutely nail down: everything is passed by value, a method receiver is also a copy of a value or a pointer, a closure captures the variable rather than the value, and a goroutine’s stack is growable. C/C++ veterans who treat these as “the same as C” will keep falling into traps.

Functions are first-class citizens

Functions are values in Go: you can assign them to variables, pass them as arguments, return them, and put them in a slice/map. At the low level a function value is a pointer to a funcval struct (containing the code address, plus captured variables when it’s a closure), so a function variable itself is just one machine word (pointer-sized).

package main

import "fmt"

func add(a, b int) int { return a + b }

func main() {
    var op func(int, int) int = add // assign a function to a variable
    fmt.Println(op(3, 4))            // 7

    apply := func(f func(int, int) int, x, y int) int {
        return f(x, y) // function as an argument
    }
    fmt.Println(apply(add, 10, 20)) // 30
}

This resembles a C function pointer, but a Go function value can carry state (a closure), making it closer to C++’s std::function / lambda—only without that layer of type-erasure heap allocation (unless the closure variables escape).

Multiple return values: the idiomatic (result, error)

Go natively supports multiple return values. Implementation detail: since the register-based ABI in Go 1.17, return values are passed preferentially through registers (only the portion exceeding the available registers spills to the stack), and the callee writes them directly, with no need for C-style output parameters (pointers) or packing into a struct.

func divmod(a, b int) (int, int) {
    return a / b, a % b
}

func main() {
    q, r := divmod(17, 5) // q=3, r=2
    fmt.Println(q, r)
    q, _ = divmod(10, 3)  // use _ to discard a return value you don't need
    _ = q
}

Go’s most important idiom is (result, error)—the last return value is an error, and the caller checks it immediately:

func parse(s string) (int, error) {
    // ...
    return 0, fmt.Errorf("parse failed: %q", s)
}

func main() {
    n, err := parse("abc")
    if err != nil {
        fmt.Println("error:", err)
        return
    }
    fmt.Println(n)
}

This replaces C’s errno / return-code conventions, and also replaces C++’s use of exceptions as the ordinary error path. See 09-Error Handling and Resource Management for details.

⭐ Everything is passed by value

Go has no pass-by-reference. On every function call, all actual arguments are copied into the callee’s stack frame. To modify a caller’s variable inside a function, you must explicitly pass a pointer (the pointer value is copied, but it points to the same object).

func incValue(x int)  { x++ }      // modifies the copy, invisible to the caller
func incPtr(x *int)   { *x++ }     // modifies the original variable through the pointer

func main() {
    a := 10
    incValue(a)
    fmt.Println(a) // still 10

    incPtr(&a)
    fmt.Println(a) // 11
}

Memory diagram:

main's stack frame:      incValue's stack frame:
  a = 10  ──copy──────▶   x = 10   (x++ only changes this, destroyed after return)

  a = 11  ◀──*x++──┐     incPtr's stack frame:
        │          └──   x = &a   (x is a pointer to a, the address is copied)
        └── a is modified

This rule holds uniformly for all types, including structs and arrays. Passing a large struct copies it wholesale:

type Big struct{ data [1024]int } // 8KB

func mutate(b Big)  { b.data[0] = 1 } // copies 8KB, the change has no effect
func mutateP(b *Big) { b.data[0] = 1 } // copies an 8-byte pointer, the change takes effect

slice / map / channel are “headers containing pointers”

Beginners often mistakenly believe slice/map are “passed by reference.” The truth: they are also copied by value, it’s just that their value is a small header containing a pointer. Copying the header = copying the pointer, so the two headers point to the same underlying data, producing an aliasing effect:

func fill(s []int) {
    s[0] = 99 // modifies the shared underlying array through the pointer in the header — visible to the caller
    s = append(s, 7) // append may swap the underlying array; even if it doesn't, this reassigns the [local header], so the caller's header length is unchanged
}

func main() {
    s := []int{1, 2, 3}
    fill(s)
    fmt.Println(s) // [99 2 3] — s[0] changed, but the length is still 3
}

Memory diagram (copying a slice header):

main's s (header):         fill's s (copied header):
  ptr ──┐                    ptr ──┐   ← the two ptrs are equal, pointing at the same array
  len=3 │                    len=3 │
  cap=3 │                    cap=3 │
        └──────┬─────────────┘
               ▼
    underlying array (heap): [99, 2, 3]   ← s[0]=99 is visible on both sides
Type What is copied when passed Are element modifications visible to the caller?
int/struct/array The entire value (possibly large) No
pointer The pointer (1 word) Yes (through dereference)
slice The header (ptr+len+cap, 3 words) Modifying an element: yes; append/reslice: no
map The header pointer (1 word) Yes
channel The header pointer (1 word) Yes (send/receive)
string The header (ptr+len, read-only) Strings are immutable

For the detailed memory layouts of map/slice/channel, see 07-Composite Data Structures.

Methods: value receiver vs pointer receiver

A method is just a function whose first parameter (the receiver) has special syntax. The receiver likewise follows the pass-by-value rule.

type Counter struct{ n int }

// value receiver: c is a [copy] of the whole Counter, changes don't affect the original
func (c Counter) IncValue() { c.n++ }

// pointer receiver: c is a copy of the pointer, through which the original can be changed
func (c *Counter) IncPtr() { c.n++ }

func main() {
    var c Counter
    c.IncValue()
    fmt.Println(c.n) // 0 — modified the copy

    c.IncPtr()
    fmt.Println(c.n) // 1 — modified the original
}

Two low-level reasons for choosing the receiver type:

  1. Need to modify the receiver → you must use a pointer receiver (otherwise you modify a copy).
  2. The receiver is a large struct → use a pointer to avoid copying the whole struct on every call (a value receiver pushes the entire struct onto the stack).

Small, value-semantic types (such as a Point{x,y int} representing coordinates) are simpler and more cache-friendly with a value receiver. Convention: a given type’s methods should all use value receivers or all use pointer receivers—don’t mix them.

Addressability and automatic address-taking

In c.IncPtr(), c is a value, but the compiler automatically rewrites it as (&c).IncPtr()—provided that c is addressable. Local variables, slice elements, and dereferenceable pointer targets are all addressable; whereas map elements and temporary values (function return values) are not addressable, so you cannot call a pointer-receiver method on them:

type Counter struct{ n int }
func (c *Counter) IncPtr() { c.n++ }

m := map[string]Counter{"a": {}}
// m["a"].IncPtr() // compile error: m["a"] is not addressable, cannot take its address

Method sets

A type’s method set determines which interfaces it can satisfy:

  • The method set of type T: all value-receiver methods.
  • The method set of type *T: value-receiver + pointer-receiver methods (all of them).

In other words, pointer-receiver methods do not belong to T’s method set. Therefore, if a value type T wants to satisfy some interface whose methods use pointer receivers, then only *T satisfies the interface and T does not. This rule, combined with addressability, is a common source of interface-assignment errors. See 08-The Type System and OOP for details.

A closure captures the variable, not the value

When a closure (anonymous function) captures an outer variable, it captures the variable itself (its storage location), not its value at that moment. If a captured variable is still referenced by the closure after the enclosing function returns, escape analysis moves it from the stack to the heap, so the closure and the outer scope share the same storage.

func counter() func() int {
    n := 0 // escapes to the heap because the returned closure holds it
    return func() int {
        n++ // each call shares and modifies the same n
        return n
    }
}

func main() {
    c := counter()
    fmt.Println(c(), c(), c()) // 1 2 3
}

This has the same semantics as a C++ lambda’s [&n] (capture by reference), not [n] (by value). C has no closures and can only simulate this by passing a struct pointer.

The classic loop-variable trap and the Go 1.22 semantic change

Because it captures the variable, in Go 1.21 and earlier the loop variable had a single instance for the whole loop, and all closures captured the same variable:

// Go 1.21-: prints 3 3 3 (sharing the same i, which is 3 when the loop ends)
// Go 1.22+: prints 0 1 2 (each i is a fresh instance, each closure captures a different variable)
func main() {
    funcs := make([]func(), 0, 3)
    for i := 0; i < 3; i++ {
        funcs = append(funcs, func() { fmt.Print(i, " ") })
    }
    for _, f := range funcs {
        f()
    }
}

Since Go 1.22, a for loop variable is a fresh instance each iteration (see 05-Control Flow), and this near-universal trap is fixed at the language level. In 1.21 and earlier, the old style relied on an explicit per-iteration copy i := i to work around it. The same applies when using goroutines (go func(){ ... i ... }()); the behavior differs before and after 1.22.

Variadic …T

...T declares a variadic parameter, which the function treats internally as a []T (slice). At the low level: the caller packs these arguments into a new slice (usually allocating the underlying array on the stack when it doesn’t escape) and passes it in.

func sum(nums ...int) int { // nums has type []int
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

func main() {
    fmt.Println(sum(1, 2, 3)) // pass 3 → packed into []int{1,2,3}
    fmt.Println(sum())        // pass 0 → nums is a nil slice, len=0

    xs := []int{4, 5, 6}
    fmt.Println(sum(xs...)) // use ... to spread an existing slice; here [no copy] happens, xs's underlying array is reused directly
}

Note that sum(xs...) reuses xs’s underlying array, so modifying nums[i] inside the function would change xs. Compared to C’s va_list (untyped, no length, requires a sentinel), this is type-safe and carries its own length.

Named return values + defer modifying the return value

Combined with the defer from 05-Control Flow, named return values are a powerful combination. return first writes the value into the named return slot, then runs the defers, and a defer can modify this slot:

func readConfig() (cfg string, err error) {
    defer func() {
        if r := recover(); r != nil {
            // convert the panic into an error returned to the caller
            err = fmt.Errorf("recovered from panic: %v", r)
        }
    }()
    panic("config file corrupted")
    // unreachable, but the defer will set err and return normally
}

At the low level, named return values are simply named variables in the callee’s stack frame, and return x = “copy x into the return slot + run the defers + actually return.” This is the standard technique for converting a panic into an error. See 09-Error Handling and Resource Management for details.

The goroutine stack: a growable contiguous stack

This is one of the most elegant designs in the Go runtime, and completely different from C/C++ thread stacks.

A C/C++ thread stack has a fixed size at creation time (typically 1–8MB), with virtual address space reserved by the OS. Spawning too many threads exhausts the address space; recursing too deeply overflows the stack (segfault).

Every Go goroutine has its own stack, initially only about 2KB, managed by the Go runtime (not the OS). This makes it easy to create hundreds of thousands of goroutines—2KB × 1 million ≈ 2GB, which threads cannot match.

How the stack grows: morestack and stack copying

The compiler inserts a small piece of stack-check (stack guard) code at each function’s entry: it checks whether the current stack pointer is about to hit the bottom. If space is insufficient, it calls the runtime’s morestack:

  1. Allocate a larger new stack (usually double).
  2. Copy the entire contents of the old stack to the new stack (a contiguous stack).
  3. Adjust all pointers that pointed into the old stack (the runtime can precisely scan stack pointers, which relies on Go’s precise-GC type information)—this step is impossible in C, because C cannot know which words on the stack are pointers.
  4. Free the old stack, and the function continues executing.

The stack can also shrink on demand during GC.

before growth (2KB stack nearly full):   after growth (copied to a 4KB new stack):
 ┌──────────┐ high address                ┌──────────┐
 │ frame A  │               │ frame A  │  ← contents copied byte by byte
 │ frame B  │   morestack   │ frame B  │
 │ frame C ←SP │  ══════════▶  │ frame C  │
 └──────────┘ hit bottom          │ frame D ←SP │  ← now there's room to continue
                            │  ...     │
                            └──────────┘ low address

Impact on code

  • Do not assume the address of a stack variable is stable: after stack growth the address changes (but pointers to it are corrected by the runtime, so it’s transparent to you). This is also why taking the address of a local variable is perfectly safe in Go—even if it escapes, the runtime/compiler handles it correctly (the escaped one moves to the heap, the non-escaped one moves with the stack).
  • Deep recursion does not easily segfault in Go, but the stack keeps growing until it reaches the limit (64-bit default maxstacksize ≈ 1GB, adjustable); on exceeding it the runtime directly fatal error: stack overflow and terminates the process—this is a fatal error that cannot be recovered, fundamentally different from an ordinary panic.
  • Goroutine scheduling and stacks, together with channels, form the foundation of Go’s concurrency model. See 10-The Concurrency Model.

Summary

  • Functions are first-class values, backed by a funcval pointer, carrying captured variables when they are closures.
  • Multiple return values are natively supported, idiomatically (result, error).
  • Everything is passed by value; to modify, pass a pointer. slice/map/chan copy a pointer-containing header, hence the aliasing effect.
  • Method receivers are likewise by value: a value receiver copies the whole receiver, a pointer receiver can modify it and avoids a large copy; mind method sets and addressability.
  • A closure captures the variable, not the value; Go 1.22 fixed the loop-variable capture trap.
  • A variadic ...T is a slice internally; xs... spreads and reuses the underlying array.
  • A goroutine stack starts at 2KB and, as a contiguous stack, can grow/shrink—a world apart from C’s fixed thread stack.
