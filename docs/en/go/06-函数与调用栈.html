Functions and the Call Stack (⭐ The Truth About Parameter Passing)

This chapter starts by walking through how to define a function, how to call it, how to write multiple return values, and how to write methods and closures—no background knowledge needed. Then it moves into the most important underlying material in this whole tutorial: Go’s functions look like C syntactically, but their memory semantics rest on several facts you must absolutely nail down—everything is passed by value, a method receiver is also a copy of a value or a pointer, a closure captures the variable rather than the value, and a goroutine’s stack is growable. C/C++ veterans who treat these as “the same as C” will keep falling into traps. If you only want to write functions, the first section is enough; if you want to understand the truth about parameter passing and the call stack, keep reading.

Just Using It: Functions, Methods, and Closures at a Glance

Defining and calling a function looks about the same as in most languages—except the return type is written after the parameter list:

package main

import "fmt"

func add(a, b int) int { // define a function: two int parameters, returns an int
    return a + b
}

func divmod(a, b int) (int, int) { // can return multiple values at once
    return a / b, a % b
}

func main() {
    fmt.Println(add(3, 4)) // call the function: 7

    q, r := divmod(17, 5) // receive both return values at once
    fmt.Println(q, r)     // 3 2
}

A method is just a function “bound to a type”, with an extra “receiver” written before the function name:

package main

import "fmt"

type Counter struct {
    n int
}

func (c *Counter) Inc() { // method: c is the receiver, like "this method belongs to *Counter"
    c.n++
}

func main() {
    c := Counter{}
    c.Inc()
    c.Inc()
    fmt.Println(c.n) // 2
}

A closure is an anonymous function that “remembers” outer variables:

package main

import "fmt"

func makeAdder(base int) func(int) int {
    return func(x int) int { // this anonymous function remembers the outer base
        return base + x
    }
}

func main() {
    add10 := makeAdder(10)
    fmt.Println(add10(5)) // 15
}

That’s already enough to write ordinary functions, methods, and closures. Now let’s go under the hood—what actually happens in memory with parameter passing, receivers, and closure capture. This is the most important material in this chapter, and in the whole book.

Going Deeper

We start from the most fundamental question—“how exactly are parameters passed”—and nail down, one fact at a time, the real memory behavior of functions, methods, and closures.

Functions Are First-Class Citizens

Functions in Go are values: they can be assigned to variables, passed as arguments, returned as results, and stored in slices/maps. Under the hood, a function value is a pointer to a funcval struct (containing the code address, plus captured variables when it’s a closure), so a function variable itself is only a pointer-sized word.

package main

import "fmt"

func add(a, b int) int { return a + b }

func main() {
    var op func(int, int) int = add // assign a function to a variable
    fmt.Println(op(3, 4))            // 7

    apply := func(f func(int, int) int, x, y int) int {
        return f(x, y) // pass a function as an argument
    }
    fmt.Println(apply(add, 10, 20)) // 30
}

This resembles a C function pointer, but Go’s function values can carry state (closures), making them closer to C++’s std::function/lambdas—just without that layer of type-erasure heap allocation overhead (unless the closure’s captured variables escape).

Multiple Return Values: The (result, error) Idiom

The first section already used multiple return values. Under the hood: since the register-based ABI introduced in Go 1.17, return values are preferentially passed via registers (only the part exceeding the available register count spills to the stack), written directly by the callee, without needing output parameters (pointers) or a packed struct the way C does.

Go’s most important idiom is (result, error)—the last return value is an error, which the caller checks immediately:

func parse(s string) (int, error) {
    // ...
    return 0, fmt.Errorf("parse failed: %q", s)
}

func main() {
    n, err := parse("abc")
    if err != nil {
        fmt.Println("error:", err)
        return
    }
    fmt.Println(n)
}

This replaces C’s errno/return-code convention, and also replaces C++’s use of exceptions as the routine error path. See 09-错误处理与资源管理 for details.

⭐ Everything Is Passed by Value

There is no pass-by-reference in Go. On every function call, all arguments are copied into the callee’s stack frame. To modify the caller’s variable from inside the function, you must explicitly pass a pointer (the pointer value itself is copied, but it points at the same object).

func incValue(x int)  { x++ }      // modifies the copy; the caller never sees it
func incPtr(x *int)   { *x++ }     // modifies the original variable through the pointer

func main() {
    a := 10
    incValue(a)
    fmt.Println(a) // still 10

    incPtr(&a)
    fmt.Println(a) // 11
}

Memory picture:

main's stack frame:            incValue's stack frame:
  a = 10  ──copy───────▶   x = 10   (x++ only changes this; destroyed on return)

  a = 11  ◀──*x++──┐     incPtr's stack frame:
        │          └──   x = &a   (x is a pointer to a; the address is copied)
        └── a gets modified

This rule holds uniformly for every type, including structs and arrays. Passing a large struct copies the whole thing:

type Big struct{ data [1024]int } // 8KB

func mutate(b Big)  { b.data[0] = 1 } // copies 8KB; the change has no effect
func mutateP(b *Big) { b.data[0] = 1 } // copies an 8-byte pointer; the change takes effect

slice / map / channel Are “Headers Containing a Pointer”

Beginners often mistakenly believe slices/maps are “passed by reference.” The truth: they too are passed by value copy—it’s just that their value is a small header containing a pointer. Copying the header means copying the pointer, so the two headers end up pointing at the same underlying data, producing an aliasing effect:

func fill(s []int) {
    s[0] = 99 // modifies the shared underlying array through the pointer in the header — visible to the caller
    s = append(s, 7) // append may swap in a new underlying array; even if it doesn't, this just reassigns the [local header], leaving the caller's header length unchanged
}

func main() {
    s := []int{1, 2, 3}
    fill(s)
    fmt.Println(s) // [99 2 3] — s[0] changed, but the length is still 3
}

Memory picture (slice header copy):

main's s (header):          fill's s (copied header):
  ptr ──┐                    ptr ──┐   ← both ptrs are equal, pointing at the same array
  len=3 │                    len=3 │
  cap=3 │                    cap=3 │
        └──────┬─────────────┘
               ▼
    underlying array (heap): [99, 2, 3]   ← s[0]=99 is visible from both sides
Type What gets copied when passed Is modifying an element visible to the caller?
int/struct/array the entire value (potentially large) No
pointer the pointer (1 word) Yes (via dereference)
slice the header (ptr+len+cap, 3 words) Modifying an element: yes; append/reslice: no
map a header pointer (1 word) Yes
channel a header pointer (1 word) Yes (send/receive)
string the header (ptr+len, read-only) strings are immutable

See 07-复合数据结构 for the detailed memory layout of map/slice/channel.

Methods: Value Receivers vs. Pointer Receivers

A method is just a function whose first parameter (the receiver) has special syntax. The receiver follows the same pass-by-value rule. The first section’s Inc used a pointer receiver; here’s a value receiver and a pointer receiver side by side:

type Counter struct{ n int }

// value receiver: c is a [copy] of the whole Counter; changes don't affect the original
func (c Counter) IncValue() { c.n++ }

// pointer receiver: c is a copy of a pointer, through which the original can be modified
func (c *Counter) IncPtr() { c.n++ }

func main() {
    var c Counter
    c.IncValue()
    fmt.Println(c.n) // 0 — modified the copy

    c.IncPtr()
    fmt.Println(c.n) // 1 — modified the original
}

Two underlying reasons for choosing a receiver type:

  1. You need to modify the receiver → you must use a pointer receiver (otherwise you’re modifying a copy).
  2. The receiver is a large struct → use a pointer to avoid copying the whole thing on every call (a value receiver pushes the entire struct onto the stack).

For small, immutable-in-spirit types (like a Point{x,y int} representing coordinates), a value receiver is simpler and more cache-friendly. Convention: methods on the same type should either all use value receivers or all use pointer receivers—don’t mix.

Addressability and Automatic Address-Of

In c.IncPtr(), c is a value, but the compiler automatically rewrites this as (&c).IncPtr()—provided c is addressable. Local variables, slice elements, and dereferenceable pointer targets are all addressable; but map elements and temporary values (function return values) are not addressable, and you can’t call a pointer-receiver method on them:

type Counter struct{ n int }
func (c *Counter) IncPtr() { c.n++ }

m := map[string]Counter{"a": {}}
// m["a"].IncPtr() // compile error: m["a"] is not addressable, its address can't be taken

Method Sets

A type’s method set determines which interfaces it can satisfy:

  • The method set of type T: all value-receiver methods.
  • The method set of type *T: value-receiver plus pointer-receiver methods (all of them).

In other words, pointer-receiver methods do not belong to T’s method set. So if a value type T needs to satisfy an interface whose method uses a pointer receiver, only *T satisfies the interface—T does not. This rule, combined with addressability, is a common source of interface-assignment errors; see 08-面向对象与类型系统 for details.

Closures Capture Variables, Not Values

The first section’s makeAdder already demonstrated the basic use of closures. When a closure (an anonymous function) captures an outer variable, what it captures is the variable itself (its storage location), not the value at that moment. If the captured variable is still referenced by the closure after the enclosing function returns, escape analysis moves it from the stack to the heap, so the closure and the enclosing function share the same storage.

func counter() func() int {
    n := 0 // escapes to the heap, because the returned closure holds onto it
    return func() int {
        n++ // each call shares and modifies the same n
        return n
    }
}

func main() {
    c := counter()
    fmt.Println(c(), c(), c()) // 1 2 3
}

This matches the semantics of a C++ lambda’s [&n] (capture by reference), not [n] (capture by value). C has no closures at all—you can only simulate one by passing a pointer to a struct.

The most convincing evidence that “what’s captured is the variable” is having two closures capture the same variable—one writes, one reads, and they see the same storage:

package main

import "fmt"

func newAccount() (deposit func(int), balance func() int) {
    bal := 0
    deposit = func(n int) { bal += n }
    balance = func() int { return bal }
    return
}

func main() {
    dep, bal := newAccount()
    dep(100)
    dep(50)
    fmt.Println(bal()) // 150, the two closures share the same bal
}

bal escapes to the heap, and both deposit and balance hold a reference to it, forming a piece of encapsulated private state (closure = data + behavior)—exactly what, in C, you’d have to cobble together with “a struct plus a set of function pointers that operate on it.”

The Classic Loop-Variable Pitfall and the Go 1.22 Semantic Change

Because what’s captured is the variable, in Go 1.21 and earlier the loop variable had only a single instance for the entire loop, so all closures captured the same variable:

// Go 1.21-: prints 3 3 3 (sharing the same i; i==3 when the loop ends)
// Go 1.22+: prints 0 1 2 (each iteration's i is a fresh instance, each closure captures a different variable)
func main() {
    funcs := make([]func(), 0, 3)
    for i := 0; i < 3; i++ {
        funcs = append(funcs, func() { fmt.Print(i, " ") })
    }
    for _, f := range funcs {
        f()
    }
}

Since Go 1.22, for’s loop variable is a fresh instance every iteration (see 05-控制流), fixing a pitfall that nearly everyone has stepped on. In 1.21 and earlier, the traditional workaround was to explicitly create a copy each iteration with i := i. The same applies with goroutines (go func(){ ... i ... }())—behavior differs before and after 1.22.

Variadic Parameters: …T

...T declares a variadic parameter, which the function body treats as a []T (slice). Under the hood: the caller packs these arguments into a new slice (typically allocating the underlying array on the stack, when it doesn’t escape) and passes that in.

func sum(nums ...int) int { // nums has type []int
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

func main() {
    fmt.Println(sum(1, 2, 3)) // pass 3 values → packed into []int{1,2,3}
    fmt.Println(sum())        // pass 0 values → nums is a nil slice, len=0

    xs := []int{4, 5, 6}
    fmt.Println(sum(xs...)) // use ... to spread an existing slice; this [does not copy], it directly reuses xs's underlying array
}

Note that sum(xs...) reuses xs’s underlying array, so if the function modifies nums[i], it changes xs too. Compared with C’s va_list (untyped, no length, needs a sentinel), this is type-safe and carries its own length.

Named Return Values + Modifying the Return Value with defer

Combined with the defer covered in 05-控制流, named return values form a powerful pair. return first writes the value into the named return slot, then runs the defers, and the defers can modify that slot:

func readConfig() (cfg string, err error) {
    defer func() {
        if r := recover(); r != nil {
            // turn the panic into an error returned to the caller
            err = fmt.Errorf("recovered from panic: %v", r)
        }
    }()
    panic("config file corrupted")
    // unreachable, but the defer sets err and then returns normally
}

Under the hood, a named return value is just a named variable in the callee’s stack frame; return x means “copy x into the return slot + run the defers + actually return.” This is the standard technique for converting a panic into an error; see 09-错误处理与资源管理 for details.

Goroutine Stacks: Growable Contiguous Stacks

This is one of the most elegant designs in the Go runtime, and is completely different from C/C++ thread stacks.

A C/C++ thread’s stack has a fixed size set at creation time (typically 1–8MB), with the OS reserving virtual address space for it. Creating too many threads exhausts address space; recursing too deep causes a stack overflow (segfault).

Each Go goroutine has its own stack, starting at only about 2KB, managed by the Go runtime (not the OS). This makes it easy to create hundreds of thousands of goroutines—2KB × 1 million ≈ 2GB, which threads simply can’t do.

How the Stack Grows: morestack and Stack Copying

The compiler inserts a small stack-check (stack guard) snippet at the entry of every function: it checks whether the current stack pointer is close to the bottom. If there isn’t enough room, it calls the runtime’s morestack:

  1. Allocate a larger new stack (typically double the size).
  2. Copy the entire contents of the old stack to the new one (a contiguous stack).
  3. Adjust every pointer that pointed into the old stack (the runtime can precisely scan pointers on the stack, relying on Go’s precise GC type information)—this is a step C cannot do, because C has no way to know which values on the stack are pointers.
  4. Free the old stack; the function continues executing.

The stack can also shrink on demand during GC.

Before growth (2KB stack nearly full):     After growth (copied to a new 4KB stack):
 ┌──────────┐ high address        ┌──────────┐
 │ frame A  │               │ frame A  │  ← contents copied byte for byte
 │ frame B  │   morestack   │ frame B  │
 │ frame C ←SP │  ══════════▶  │ frame C  │
 └──────────┘ hit bottom        │ frame D ←SP │  ← now there's room to keep going
                            │  ...     │
                            └──────────┘ low address

Effect on Your Code

  • Don’t assume a stack variable’s address is stable: after a stack grows, the address changes (but the runtime fixes up any pointers to it, so this is transparent to you). This is also why taking the address of a local variable in Go is completely safe—even if it escapes, the runtime/compiler handles it correctly (an escaping variable moves to the heap; a non-escaping one moves along with the stack).
  • Deep recursion in Go doesn’t easily segfault, but it keeps growing the stack until it hits a limit (on 64-bit, the default maxstacksize is about 1GB, adjustable); once exceeded, the runtime issues a fatal error: stack overflow and terminates the process directly—this is a fatal error that cannot be recovered, fundamentally different from an ordinary panic.
  • Goroutine scheduling and stacks, together with channels, form the foundation of Go’s concurrency model; see 10-并发模型.

Summary

  • At the usage level: functions are defined with func name(params types) returnType { ... } and can return multiple values at once; a method has a receiver (c *T) written before the function name; a closure is an anonymous function that remembers outer variables, written as func(params) returnType { ... }.
  • At the underlying level: functions are first-class values, backed by a funcval pointer, carrying captured variables when they’re closures. Everything is passed by value; to modify something, pass a pointer. slice/map/chan copy a header containing a pointer, which is why they alias. Method receivers are likewise passed by value: a value receiver copies the entire receiver, a pointer receiver can modify it and avoids a large copy; watch out for method sets and addressability. Closures capture variables, not values; Go 1.22 fixed the loop-variable capture pitfall. Variadic ...T is a slice internally; xs... spreads it while reusing the underlying array. Goroutine stacks start at 2KB and are contiguous stacks that can grow/shrink—a completely different world from C’s fixed thread stacks.
