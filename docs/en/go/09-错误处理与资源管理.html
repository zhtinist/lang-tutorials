Error Handling and Garbage Collection

Go has no exceptions, no try/catch, and no destructors doing RAII. It treats errors as ordinary return values, hands resource cleanup to defer, and hands memory reclamation to a concurrent garbage collector. This chapter first covers how to write and use these three things — understanding ordinary Go code is enough for that — and then moves on to “underlying implementation vs C”: what an error interface value actually is, what really happens when a panic unwinds, and how the GC marks concurrently without long pauses. For C++ programmers, these are three places where you need to rebuild muscle memory.

First, the basics: a quick tour of error, defer, and panic/recover

No underlying mechanics yet — just how to write these things:

package main

import (
    "errors"
    "fmt"
)

func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("division by zero") // construct an error
    }
    return a / b, nil // a nil second return value means "no error"
}

func main() {
    result, err := divide(10, 2)
    if err != nil {
        fmt.Println("error:", err)
    } else {
        fmt.Println("result:", result) // result: 5
    }

    _, err = divide(10, 0)
    if err != nil {
        fmt.Println("error:", err) // error: division by zero
    }

    // defer: registers a call that is guaranteed to run before the function returns
    defer fmt.Println("printed just before main ends")
    fmt.Println("main running normally")
}

A quick summary:

  • Go functions often return two values: (result, error). The caller must check it itself with if err != nil — an error does not automatically bubble up the way an exception would.
  • errors.New("...") builds the simplest possible error; err == nil means nothing went wrong.
  • A defer statement registers a call that runs just before the function returns, whether the function returns normally or via an error — commonly used for “opened it, so must close it” cleanup (closing a file, unlocking a mutex, and so on).

Now for panic/recover — used for genuinely unexpected situations (an out-of-bounds index, an explicitly triggered fatal error), not as an everyday error-handling tool:

func mayPanic() {
    panic("something went wrong") // deliberately trigger a panic
}

func safe() {
    defer func() {
        if r := recover(); r != nil { // recover only works inside a defer
            fmt.Println("recovered:", r)
        }
    }()
    mayPanic()
    fmt.Println("this line never runs") // once panic fires, the rest of the function stops running
}

func main() {
    safe()
    fmt.Println("program keeps running") // after recover, main carries on normally
}

That much is enough to write structurally complete Go code. Now let’s go deeper — what an error really is in memory, the full mechanics of panic unwinding, and how the GC works concurrently.

Now let’s go deeper: what an error really is

From here on we talk about the underlying mechanics. To understand the cost and behavior of error/panic/GC, start with this: error is not a keyword — it is just a perfectly ordinary interface in the standard library, no different from any interface you’d define yourself.

type error interface {
    Error() string
}

Any type that implements Error() string is an error. In memory it is the two-word interface iface covered in Chapter 08 — {itab, data}. Returning an error involves no stack unwinding and no zero-cost exception tables; it just pushes a 16-byte interface value into the return stack slot, as cheap as returning any other value.

I. error is an ordinary interface value

How to use it

The idiom is “multiple return values + if err != nil”:

f, err := os.Open("data.txt")
if err != nil {
    return fmt.Errorf("failed to open config: %w", err) // handle or wrap, then propagate
}
defer f.Close()
// happy path

A few ways to create errors:

errors.New("something failed")           // simplest, returns *errorString
fmt.Errorf("port %d in use", port)        // with formatting
fmt.Errorf("read config: %w", err)        // %w wraps the underlying error (keeps the chain)

%w (wrap) is the key: it embeds err as the “wrapped cause” inside the new error, forming an error chain that errors.Is/errors.As can later unwrap layer by layer. Using %v just splices the text in, and the chain breaks.

Comparison with C

In C++, errors are “implicit control flow” (exceptions bubble up automatically); in Go, errors are “explicit data flow” — every layer must visibly check and pass them along. The verbosity is what it gets criticized for, but in exchange you get fully visible control flow, no hidden jumps, and no stack-unwinding overhead.

II. Sentinel errors, custom errors, Is/As

How to use it

Sentinel errors: predefined error values that can be compared with == or errors.Is:

var ErrNotFound = errors.New("not found")

func Lookup(k string) (string, error) {
    // ...
    return "", ErrNotFound
}

if errors.Is(err, ErrNotFound) { /* special-case it */ }

Why use errors.Is rather than err == ErrNotFound? Because err may have been wrapped several layers deep with %w; == only compares the outermost layer, whereas errors.Is walks the %w chain, calling Unwrap and comparing at each level.

Custom error types: if you want to carry structured information, define a type:

type PathError struct {
    Op   string
    Path string
    Err  error
}
func (e *PathError) Error() string {
    return e.Op + " " + e.Path + ": " + e.Err.Error()
}
func (e *PathError) Unwrap() error { return e.Err } // lets Is/As see through it

errors.As pulls out the first error in the chain that matches the target type:

var pe *PathError
if errors.As(err, &pe) {
    fmt.Println("failing path:", pe.Path) // access the structured field
}
Function Purpose Analogy
errors.Is(err, target) Is there an error in the chain equal to target Match by “value”
errors.As(err, &v) Is there an error in the chain assignable to v; extract it Match by “type” (like C++ catch(SpecificType&))
errors.Unwrap(err) Peel off one layer Walk the chain manually

Comparison with C

errors.Is/errors.As are both just a loop that repeatedly walks down via the Unwrap() error method — not compiler magic: at each layer, if it doesn’t match, it calls that layer’s Unwrap() to get the next one, until the chain ends (Unwrap returns nil). errors.Is compares with == by default, but prefers a type’s own Is(error) bool if it defines one; errors.As uses reflection to check whether each layer’s dynamic type in the chain is assignable to the type the target pointer points to. This chain is fundamentally just a series of “nodes” holding onto each other through interface values — no different in essence from walking a linked list.

Runnable: %w wrapping + Is + As

package main

import (
    "errors"
    "fmt"
)

var ErrNotFound = errors.New("not found")

type QueryError struct {
    Query string
    Err   error
}

func (e *QueryError) Error() string { return e.Query + ": " + e.Err.Error() }
func (e *QueryError) Unwrap() error { return e.Err }

func find() error {
    qe := &QueryError{Query: "SELECT", Err: ErrNotFound} // wraps the sentinel
    return fmt.Errorf("db call: %w", qe)                 // wraps again with %w
}

func main() {
    err := find()
    fmt.Println(err)                         // db call: SELECT: not found
    fmt.Println(errors.Is(err, ErrNotFound)) // true   ← walks the whole %w/Unwrap chain

    var qe *QueryError
    fmt.Println(errors.As(err, &qe), qe.Query) // true SELECT   ← pulls the typed error out
}

III. panic / recover: not an exception

How to use it

panic is for genuine programming errors that should never happen (array out of bounds, dereferencing nil, a broken invariant), not for routine error handling. recover is effective only when called inside a defer; it “catches” the panic and lets the program resume normal execution:

func safeRun() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r) // turn the panic into a returned error
        }
    }()
    riskyOperation()
    return nil
}

Typical use: wrap each request handler of a web server in a recover, so a single request’s panic does not bring down the whole process. But do not use panic/recover to simulate try/catch for routine flow control — that is an anti-pattern, slow and against Go convention.

Comparison with C

panic(x) will: immediately stop the current function’s normal execution → unwind up the call stack layer by layer, running each layer’s registered defers → if not recovered along the way, reach the top of the goroutine, print a stack traceback, and exit(2) to terminate the entire process (not just kill the current goroutine).

panic unwinding:
main → A() → B() → panic!
                    ↓ run B's defers
              ↑ run A's defers
        ↑ run main's defers (if no recover) → process crashes

This unwinding process runs defers, but there is no C++-style type-based catch matching — recover catches “any panic” without distinguishing types.

recover() is effective only when called directly inside a deferred function. It stops the unwinding, returns the value passed to panic, and lets the program resume normally at the current stack frame.

Runnable: turning a runtime panic into an error:

package main

import "fmt"

func safeDiv(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r) // turn the panic into a returned error
        }
    }()
    return a / b, nil // b==0 raises a runtime panic before the return completes
}

func main() {
    r, err := safeDiv(10, 2)
    fmt.Println(r, err) // 5 <nil>
    r, err = safeDiv(10, 0)
    fmt.Println(r, err) // 0 recovered: runtime error: integer divide by zero
}

Note result stays 0 in the panic case: the panic fires while evaluating a / b, so the return value is never assigned — only the named err is set inside the deferred recover.

For comparison:

C++ exceptions Go panic/recover
Purpose Routine error handling Only fatal / broken invariants
Trigger throw panic() or a runtime error
Catch catch(Type&) by type recover() catches anything, you decide
Uncaught std::terminate Process exit(2)
Cleanup Destructors (RAII) defer
Routine errors Also use exceptions Use return-value errors

IV. Resource management: defer instead of RAII

How to use it

Go has no destructors. Resource cleanup relies on defer: registered calls run before the function returns (including during panic unwinding), in last-in-first-out order (see the mechanics in Chapter 05).

fmt.Println("start")
defer fmt.Println("cleanup 1") // LIFO: runs last
defer fmt.Println("cleanup 2") // runs first
fmt.Println("end")
// output order: start end cleanup 2 cleanup 1

In practice it’s often used to pair “open/close” or “lock/unlock”:

f, err := os.Open(name)
if err != nil { return err }
defer f.Close()           // whatever return/panic follows, it will be closed

mu.Lock()
defer mu.Unlock()         // paired lock/unlock, prevents forgetting to unlock

Comparison with C

The difference from C++ RAII: RAII is bound to the object’s lifetime/scope (destructed on leaving the block); defer is bound to the function return (not block-level, function-level! A defer inside a loop accumulates until the function ends — a common resource-leak trap). If you need prompt block-level release, call it explicitly or factor out a small function.

runtime.SetFinalizer can register a “finalizer to run when the GC reclaims an object,” but its timing is nondeterministic and it is guaranteed to run only once, so it cannot replace defer/explicit Close for managing deterministic resources like files and locks.

V. Garbage collection (GC)

How to use it

Go is a garbage-collected language. You write no free/delete and manage no lifetimes; the compiler’s escape analysis (Chapter 03) decides whether a variable lives on the stack or the heap — stack ones are reclaimed with the stack frame, heap ones are reclaimed by the GC. Writing everyday business code, you basically never need to think about the GC — only in the rare case where you need to tune it do these “knobs” come into play:

debug.SetGCPercent(200) // raise it → less GC, higher memory usage, better throughput
debug.SetGCPercent(50)  // lower it → more frequent GC, less memory
runtime.GC()            // manually force one (blocking; generally only for tests/special cases)

Comparison with C

Concurrent tri-color mark-and-sweep. Go’s GC is a concurrent, non-moving, tri-color mark-and-sweep collector, whose core goal is an extremely short STW (stop-the-world).

The tri-color abstraction:

  • White: not yet visited, a candidate for collection.
  • Gray: visited, but the objects it references have not all been scanned (it is in the pending queue).
  • Black: visited, and all referenced objects have been scanned.

The flow: starting from the roots (stacks, global variables), color the roots gray; keep taking objects from the gray set, color the white objects they reference gray, and turn themselves black; continue until there are no gray objects left. The remaining white objects are garbage and get swept. The mark phase runs concurrently with user goroutines (mark workers use a fraction of the CPU), rather than being a long pause.

Write barrier. The trouble with concurrent marking: while the user program mutates pointers as marking runs, you can get a situation where “a black object comes to point to a white object, while the only gray reference to that white object is deleted” — the white object gets missed by the mark and wrongly collected.

The solution is the write barrier: during marking, the compiler inserts a small hook of code before every pointer write, re-coloring the involved objects gray to guarantee nothing is missed. Go uses a hybrid write barrier (a Dijkstra + Yuasa hybrid), which lets stacks require almost no STW rescan. The write barrier is active only during the GC mark phase and is off the rest of the time.

one round of concurrent GC (simplified):
[very short STW: enable write barrier] → [concurrent marking, user program runs alongside, protected by write barrier] →
[very short STW: mark termination] → [concurrent sweep, alongside the user program]
both STWs are typically sub-millisecond to millisecond

Trigger timing and GOGC. GC is driven by the heap growth ratio, controlled by the environment variable/parameter GOGC (default 100):

The next round triggers when the live heap grows to GOGC% relative to the amount after the last GC. GOGC=100 means GC once whenever the heap doubles (grows 100%).

Go 1.19+ also provides a soft memory limit GOMEMLIMIT / debug.SetMemoryLimit, a backstop for memory-constrained environments like containers.

On generations. Go’s current GC is not generational (no generational GC), and it does no compaction/moving (non-moving, so object addresses are stable — which is why in Go you can freely take addresses and interop with cgo is relatively simple). The community historically discussed generations, but because escape analysis already keeps most short-lived objects on the stack (effectively a cheap “generation zero”), the payoff from generations is smaller; the implementation chose the current low-latency design instead. This may evolve across versions — defer to the runtime docs.

The memory allocator: mcache / mcentral / mspan. How are objects allocated on the heap? Go has a tcmalloc-style multi-level allocator, centered on bucketing by size class to avoid global lock contention:

each P (see Chapter 10 scheduling) has an mcache (lock-free, thread-local)
        │  caches free objects by size class (about 70 classes: 8B,16B,...,32KB)
        ▼ when the local cache runs low, request from above
mcentral (one per size class, locked)
        ▼ when it runs short, request from above
mheap (global heap, manages pages in units of mspan)
        ▼
OS (mmap large chunks of virtual memory)
  • mspan: a set of contiguous pages, sliced into equal-size blocks of one size class. Small objects (≤32KB) are taken from the span of the corresponding size class.
  • mcache: private to each P; allocating small objects requires no lock (contrast C’s malloc, which often has global lock contention) — this is one reason Go allocation is fast under high concurrency.
  • Large objects (>32KB) are allocated directly from mheap.

Echoing Chapter 03: the more objects escape to the heap, the greater the allocation and GC pressure. To optimize, reduce heap allocation — preallocate slice capacity, reuse objects (sync.Pool), avoid unnecessary interface boxing/closure captures, use go build -gcflags=-m to inspect escape analysis, and use pprof to find allocation hotspots.

Summary

  • Usage level: functions return (result, error) as multiple values, and the caller checks with if err != nil; errors.New/fmt.Errorf create errors, %w wraps them into an error chain, and errors.Is/errors.As match along that chain; defer registers cleanup that runs before the function returns (last-in-first-out); panic/recover are only for genuinely exceptional situations, and recover must be called inside a defer.
  • Underlying level: error is just an ordinary interface value — {itab, data}, two words in memory — so returning one is as cheap as returning any other value, with no exception-table overhead; errors.Is/As are fundamentally a loop of comparison/reflection along the Unwrap() chain; once triggered, panic unwinds the call stack layer by layer, running each layer’s defers, and recover stops the unwinding — if nothing recovers it, the process calls exit(2); defer is bound to the function’s lifetime rather than a block’s, unlike C++ RAII’s scope-based destruction; the GC is concurrent tri-color mark-and-sweep plus a hybrid write barrier, with STW pushed down to sub-millisecond levels, driven by GOGC/heap growth, non-generational and non-moving; the allocator is the multi-level mcache/mcentral/mspan structure, with lock-free per-P local allocation — cutting heap allocation is what lowers GC pressure.
