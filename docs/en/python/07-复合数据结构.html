Internal Implementation of Composite Data Structures

This chapter covers Python’s most commonly used “containers” — list, tuple, dict, set, and deque. We’ll first cover how to create each container, how to add/remove/look up/modify its contents, and which methods you’ll reach for most often — all understandable without any low-level knowledge; then we’ll dig into what data structure each one actually is inside CPython, comparing it against C’s arrays, structs, and hash tables one by one. If you just want to know how to use them, reading each section’s “How to Use It” is enough; if you want to understand the memory layout and complexity, keep reading into “Internals vs. C.”

One thread runs through the whole book: see “Variables and the Memory Model” — a Python container never holds “values,” only pointers to PyObject. This comes up again and again in this chapter’s “internals” sections, and it’s what determines a container’s memory layout, copy semantics, and complexity.

0. Get Familiar with Usage First: A Quick Look at list / dict / tuple / set

list: ordered and mutable, the most commonly used container

fruits = ["apple", "banana", "cherry"]
fruits.append("date")           # append at the end
print(fruits)                   # ['apple', 'banana', 'cherry', 'date']

fruits.insert(1, "kiwi")        # insert at index 1
print(fruits)                   # ['apple', 'kiwi', 'banana', 'cherry', 'date']

fruits.remove("banana")         # remove by value (removes the first match)
print(fruits)                   # ['apple', 'kiwi', 'cherry', 'date']

last = fruits.pop()             # pop and return the last element
print(last, fruits)             # date ['apple', 'kiwi', 'cherry']

print(fruits[0], fruits[-1])    # apple cherry  -- positive/negative indices
print(fruits[1:3])              # ['kiwi', 'cherry']  -- slicing
print(len(fruits))              # 3
print("apple" in fruits)        # True

dict: key-value mapping

person = {"name": "Alice", "age": 30}
print(person["name"])           # Alice -- look up by key
person["age"] = 31              # update an existing key
person["city"] = "Boston"       # add a new key
print(person)                   # {'name': 'Alice', 'age': 31, 'city': 'Boston'}

del person["city"]              # delete a key
print(person)                   # {'name': 'Alice', 'age': 31}
print(person.get("job", "unknown"))  # unknown -- a default instead of raising when the key is missing

for k, v in person.items():     # iterate over key-value pairs
    print(k, v)
# name Alice
# age 31

tuple: ordered and immutable

point = (3, 4)
x, y = point                    # unpacking
print(x, y)                     # 3 4
print(point[0])                 # 3
# point[0] = 9                  # TypeError: a tuple can't be modified

set: unordered, deduplicated, supports set operations

a = {1, 2, 3}
a.add(4)
a.remove(2)
print(a)                        # {1, 3, 4}

b = {3, 4, 5}
print(a | b)                    # {1, 3, 4, 5}  union
print(a & b)                    # {3, 4}        intersection

These are the operations you’ll reach for most in everyday code. Below we go through each container one at a time — usage first, internals second.

1. Building Intuition: Arrays of Pointers vs. Arrays of Values

From this section on we head into “internals.” To understand the memory layout of every container that follows, let’s first build an intuitive comparison.

C’s array int a[4] = {1,2,3,4} looks like this in memory:

a:  [ 1 ][ 2 ][ 3 ][ 4 ]        // 16 bytes, values stored contiguously, cache-friendly
     ^0   ^4   ^8   ^12

Python’s a = [1, 2, 3, 4] looks like this:

list object (heap)                each element is a separate heap object
+-----------------+
| ob_refcnt       |
| ob_type = list  |
| ob_size = 4     |----> ob_item (another heap block, an array of pointers)
| allocated = 4   |       +------+------+------+------+
| ob_item ------- |------>| ptr0 | ptr1 | ptr2 | ptr3 |
+-----------------+       +--|---+--|---+--|---+--|---+
                            v      v      v      v
                          PyLong PyLong PyLong PyLong
                           (1)    (2)    (3)    (4)   <- each on the heap, possibly shared elsewhere

So a Python list is an “array of pointers,” not an “array of values.” This one diagram explains almost every phenomenon in this chapter. Keep it in mind — the “internals” subsection of every container below will come back to it.

2. list: A Dynamic Array of Pointers to PyObject

How to Use It

“Get Familiar with Usage First” above already covered append/insert/remove/pop/indexing/slicing. Here are a few more commonly used methods:

nums = [3, 1, 2]
nums.extend([9, 8])    # append elements from another iterable, one by one
print(nums)             # [3, 1, 2, 9, 8]
nums.sort()              # sort in place
print(nums)              # [1, 2, 3, 8, 9]
print(nums.index(8))    # 3 -- index of the first element equal to 8
print(nums.count(2))    # 1 -- number of occurrences
nums.reverse()            # reverse in place
print(nums)              # [9, 8, 3, 2, 1]

Internals vs. C

Memory Layout

The struct for a list in CPython (simplified from Include/cpython/listobject.h):

typedef struct {
    PyObject_VAR_HEAD          // includes ob_refcnt, ob_type, ob_size (=len)
    PyObject **ob_item;        // points to a separately allocated array of pointers
    Py_ssize_t allocated;      // the allocated capacity (capacity >= ob_size)
} PyListObject;

Key points:

  • The list object itself is a fixed-size header; the actual element pointers live in another heap block pointed to by ob_item. That’s two allocations (the list header + the pointer array), analogous to how a C++ std::vector’s vector object and the buffer it new[]s are separate.
  • ob_size is the current length len(lst); allocated is how many pointers the buffer can hold. The relationship is always allocated >= ob_size, and the empty slots in between exist so that append doesn’t have to reallocate every time.
  • Each element occupies one pointer slot in ob_item (8 bytes on a 64-bit platform), pointing at the real object. The element objects themselves are not in the array.

Compared to std::vector<T>: a vector stores the T values contiguously (with T=int, contiguous ints). A Python list always stores pointers, so even for all-small-integers it takes one pointer dereference to reach a value. That’s why numerical computing should use numpy (see “Common Third-Party Libraries”): numpy’s ndarray is a true contiguous-value C array.

Over-Allocation Growth Strategy

On append, if ob_size == allocated, the list must grow. CPython doesn’t simply double; it uses a mild growth formula (list_resize, simplified):

/* Current CPython list_resize: over-allocate about 1/8, then round up to a multiple of 4 */
new_allocated = (new_size + (new_size >> 3) + 6) & ~3;

That is, over-allocate roughly 1/8 plus a small constant, then align. Measured (CPython 3.14), the allocated growth sequence is:

0, 4, 8, 16, 24, 32, 40, 52, 64, 76, 92, 108, 128, ...

Compare that with the 1.5x or 2x growth common to C++’s std::vector. CPython uses 1/8 as a compromise between “wasted memory” and “number of reallocations,” because it assumes a list’s typical usage doesn’t append absurd amounts.

What happens on growth: it calls realloc to swap ob_item for a larger block, copying the old pointer array over bitwise (copying only pointers, not element objects, and not touching reference counts). The element objects stay put, and references to them elsewhere remain valid.

append triggers growth:
old:  ob_item -> [p0][p1][p2][p3]                (allocated=4, size=4)
     realloc to a new buffer (allocated=8):
new:  ob_item -> [p0][p1][p2][p3][p4][ ][ ][ ]    (size=5)
     the element objects (for p0..p4) never moved

Complexity of Each Operation and Why

Operation Complexity Underlying reason
lst[i] index read/write O(1) Random access into the pointer array, ob_item[i]
lst.append(x) Amortized O(1) Write directly if there’s a free slot; realloc only when full, constant when amortized
lst.pop() (tail) O(1) ob_size--, DECREF the last element
lst.insert(0, x) O(n) All following pointers must memmove one slot right
lst.pop(0) O(n) All following pointers shift left; use deque for frequent head operations
del lst[i] / middle deletion O(n) memmove to fill the hole
x in lst O(n) Linear scan comparing each with ==
len(lst) O(1) Directly reads ob_size
slice lst[a:b] O(k) Build a new list, copy k pointers (shallow copy!) and INCREF each

About amortized O(1): the total copy cost of n appends is the sum over all growth copies, and since growth is geometric, the sum is O(n), which amortized per operation is O(1). This is exactly the same analysis as std::vector::push_back.

Shallow Copy: A Trap You Must Remember

lst2 = lst[:], list(lst), and copy.copy(lst) are all shallow copies: they create a new list header and a new pointer array, but the pointers point at the same batch of element objects.

a = [[1, 2], [3, 4]]
b = a[:]              # shallow copy
b[0].append(99)       # modifying the shared inner list
print(a)              # [[1, 2, 99], [3, 4]]  -- a changed too!
a.ob_item -> [p0][p1]
                \    \
                 v    v
             [1,2,99] [3,4]      <- there is only one copy of the inner objects
                 ^    ^
b.ob_item -> [p0][p1]            <- b is a new array, but its pointers point at the same inner objects

For true isolation, use copy.deepcopy, which recursively rebuilds every layer of objects. This is the same “shallow vs. deep copy” issue as in C++, except that Python’s pervasive references by default make it much easier to trip over.

3. tuple: An Immutable Fixed-Length Array

How to Use It

“Get Familiar with Usage First” already showed creation, indexing, and unpacking. One more thing worth stressing: once a tuple is created, its slots can’t be reassigned to point elsewhere, but if a slot itself holds a mutable object, that object’s contents can still change:

t = (1, [2, 3])
t[1].append(4)     # OK, modifies the inner list
print(t)            # (1, [2, 3, 4])
# t[1] = [9]        # TypeError: can't change a tuple's slot

Use a tuple for a heterogeneous, fixed “record” (similar to a C struct, but by position rather than by name); use a list for a homogeneous “collection” that needs insertion and deletion. For named fields, see namedtuple.

Internals vs. C

The tuple struct (PyTupleObject) is simpler than a list’s — it inlines the pointer array at the tail of the object rather than allocating it separately:

typedef struct {
    PyObject_VAR_HEAD
    PyObject *ob_item[1];   // variable-length tail array, allocated once at creation, never changed
} PyTupleObject;

Differences and consequences:

  • No allocated field; the length is fixed after creation. One allocation (the object header and the element pointer array share a single block), so it’s one fewer allocation and one fewer level of indirection than a list, and more compact. Similar to a C flexible array member.
  • “Immutable” means “the element pointer array is immutable” — you can’t make t[0] point at a different object. The example above already confirms this: what changes is the inner list’s contents, not the tuple’s own slot.
  • Because (when all its elements are hashable) a tuple is hashable, it can serve as a dict key or a set element; a list cannot. See the hashing section below.
  • CPython keeps a free list cache for small tuples, so frequent creation and destruction of small tuples can reuse memory blocks, reducing malloc pressure.

4. dict: CPython’s Compact Dictionary

dict is Python’s most central data structure — it’s not only the user’s hash table but also the underlying carrier for object attributes (__dict__), namespaces, and keyword arguments. Its implementation has therefore been optimized again and again.

How to Use It

“Get Familiar with Usage First” already showed lookup, assignment, deletion, and get. Here are a few more common operations:

d = {"a": 1, "b": 2}
d.update({"b": 20, "c": 3})   # batch update / insert
print(d)                      # {'a': 1, 'b': 20, 'c': 3}

print(d.pop("a"))             # 1 -- delete and return the value for that key
print(d)                      # {'b': 20, 'c': 3}

print(d.setdefault("z", 0))   # sets a default if the key is missing and returns it; returns the existing value otherwise
print(d)                      # {'b': 20, 'c': 3, 'z': 0}

squares = {x: x * x for x in range(5)}   # dict comprehension
print(squares)                            # {0: 0, 1: 1, 2: 4, 3: 9, 4: 16}

Internals vs. C

Hash-Table Basics and Open Addressing

Python’s dict uses open addressing, not chaining. That is, on a collision it doesn’t hang a linked list in the bucket but follows a probe sequence to find the next empty slot. All entries live in the same contiguous array, which is more cache-friendly.

  • A key must be hashable: it implements __hash__ and its hash value doesn’t change over its lifetime. Immutable built-in types (int/str/tuple/frozenset) are hashable; list/dict/set are not (because they’re mutable — a content change invalidates the hash and would corrupt the table).
  • Average lookup/insert/delete is O(1); worst case (many collisions) O(n), but that’s extremely rare in practice.

The 3.6+ Compact Dict: Why It Preserves Order

Since Python 3.6, dict remembers insertion order (guaranteed by the language since 3.7). This isn’t a linked list bolted on; it’s a change to the data structure — splitting into two arrays:

indices (the hash table, storing indices into entries; may have empty slots -1):
+----+----+----+----+----+----+----+----+
| -1 |  1 | -1 | -1 |  0 | -1 |  2 | -1 |
+----+----+----+----+----+----+----+----+

entries (compact array, appended in insertion order, no holes):
+---------+--------+---------+
|hash key val|hash key val|hash key val|
|   'a':1   |  'b':2  |  'c':3  |
+---------+--------+---------+
   idx 0      idx 1     idx 2
  • The sparse indices array handles hash positioning; it stores “integer indices into entries,” with empty slots holding a special value. It can be quite sparse (low load factor), but each slot holds only a small integer (1/2/4/8 bytes depending on table size), so the wasted space is small.
  • The compact entries array stores the actual (hash, key, value) triples contiguously in insertion order, with no holes. Iterating a dict is a sequential scan of this array — this is why order is preserved, and iteration is faster than the old version (which scanned the sparse large table).
  • Compared to the pre-3.5 “every slot stores a full entry” sparse large table, the compact dict saves about 20%–25% memory.

Verified: a dict preserves order, and the order is decided by “insertion,” not “sorting.” Deleting a middle key and re-inserting it appends to the end; updating an existing key changes the value in place and leaves the order alone:

d = {}
d["c"] = 1; d["a"] = 2; d["b"] = 3
print(list(d))     # ['c', 'a', 'b']   insertion order, not sorted
del d["a"]         # delete the middle key
d["a"] = 9         # re-insert -- appended to the end of entries
print(list(d))     # ['c', 'b', 'a']
d["c"] = 100       # update an existing key: value changed in place, order unchanged
print(list(d))     # ['c', 'b', 'a']

The lookup process: hash the key → take the low bits as the indices subscript → get the entries index → compare hash and key in entries (compare the integer hash first; only if equal compare with ==, since hashes can collide).

The Probe Sequence

CPython uses a perturbation probing scheme that balances spread and locality (simplified logic):

perturb = hash;
i = hash & mask;
while (slot_used_and_key_differs) {
    perturb >>= 5;
    i = (i * 5 + perturb + 1) & mask;
}

It’s not simple linear probing but continually stirs the high bits of the hash in, making the probe sequence more scattered and reducing clustering.

Growth

When the load factor exceeds 2/3, the dict grows. The new table’s capacity is the smallest power of two that can hold “3 times the currently used entries” (about 2x–4x growth). Growth rebuilds the indices table (re-scattering all keys), but entries are carried over in insertion order. Growth also cleans up the “tombstones” left by deletions along the way.

Complexity Table

Operation Average Worst Notes
d[k] lookup O(1) O(n) Hash + probe
d[k] = v insert/update O(1) amortized O(n) May trigger a growth re-scatter
del d[k] O(1) O(n) Marked deleted; the entries slot left empty (skipped on iteration)
k in d O(1) O(n) Same as lookup
iteration O(n) O(n) Sequential scan of entries
len(d) O(1) O(1) Reads a counter

Compared to C++: std::unordered_map uses chaining (each bucket hangs a linked list/nodes, each node separately new’d, cache-unfriendly); std::map is a red-black tree, O(log n) and sorted by key. Python’s dict is more like a highly optimized open-addressing hash table that additionally guarantees insertion order.

5. set and frozenset

How to Use It

“Get Familiar with Usage First” already showed add/remove/|/&. There are four set operations in total, plus the immutable version and a comprehension:

a = {1, 2, 3, 4}
b = {3, 4, 5, 6}
print(a & b)   # {3, 4}              intersection
print(a | b)   # {1, 2, 3, 4, 5, 6}  union
print(a - b)   # {1, 2}              difference (in a, not in b)
print(a ^ b)   # {1, 2, 5, 6}        symmetric difference (in exactly one)

fs = frozenset([1, 2, 3])    # an immutable set
s = {fs, frozenset([4])}     # a frozenset is hashable, so it can go inside another set
print(len(s))                 # 2

sc = {x % 3 for x in range(10)}   # set comprehension
print(sc)                          # {0, 1, 2}

A common pattern — use a set to check “have I seen this before,” averaging O(1), far faster than a list’s O(n):

seen = set()
data = [1, 2, 2, 3, 1]
for x in data:
    if x not in seen:
        seen.add(x)
print(seen)   # {1, 2, 3}

Internals vs. C

set’s implementation is highly similar to dict’s — essentially a “hash table with keys but no values,” likewise open-addressed and likewise requiring hashable elements. Differences:

  • A set does not guarantee order (it doesn’t introduce a compact entries array to record insertion order the way dict does; it uses a structure closer to the traditional sparse table). Don’t rely on a set’s iteration order.
  • frozenset is the immutable version, so it is itself hashable and can serve as a dict key or go inside another set (as in the example above).
Operation Average complexity
add / remove / in O(1)
intersection a & b O(min(len a, len b)) — iterate the smaller set and probe the other
union a | b O(len a + len b)
difference a - b O(len a)

6. collections.deque: A Doubly-Linked List of Blocks

How to Use It

A list’s head operations are O(n). When you need efficient insertion/removal at both ends (queues, sliding windows), use collections.deque — its usage is very similar to list’s, but with the addition of appendleft/popleft:

from collections import deque

dq = deque([1, 2, 3])
dq.append(4)         # append at the tail
dq.appendleft(0)      # append at the head
print(dq)              # deque([0, 1, 2, 3, 4])

print(dq.pop())        # 4 -- pop from the tail
print(dq.popleft())    # 0 -- pop from the head
print(dq)               # deque([1, 2, 3])

The maxlen argument turns it into a fixed-length ring buffer, commonly used to keep “the most recent N records”:

recent = deque(maxlen=3)
for x in [1, 2, 3, 4, 5]:
    recent.append(x)
print(recent)   # deque([3, 4, 5], maxlen=3) -- overflow automatically pushed out the other end

Internals vs. C

Its implementation is not an ordinary doubly-linked list (that would be one node per element, terrible for cache), but a doubly-linked list of fixed-size blocks:

block (64 pointers each)     block                block
+----------------+       +----------------+     +----------------+
| prev | items...|  <->  | prev | items...| <-> | prev | items...|
| next |         |       | next |         |     | next |         |
+----------------+       +----------------+     +----------------+
   ^ leftblock                                     ^ rightblock
  • Each block is a fixed-length (64) pointer array; blocks are chained into a doubly-linked list via prev/next pointers.
  • append/appendleft/pop/popleft at both ends are all O(1): shift one position in the head/tail block, and chain a new block when full.
  • The cost: random indexing dq[i] is O(n) (you must walk the block chain), so deque isn’t suited to random access.
Operation list deque
tail append/pop O(1) amortized O(1)
head insert/pop O(n) O(1)
random index x[i] O(1) O(n)

8. Summary of the Memory-Model Comparison with C

C/C++ Python Key difference
int a[N] list C stores values contiguously; a list stores pointers, values scattered on the heap
std::vector<T> list vector stores values and can hold small objects on the stack; a list always stores pointers
struct {int; char*;} tuple / namedtuple / class a struct lays out fields by offset; a tuple stores pointers by index
std::unordered_map dict chaining vs. open addressing; dict additionally preserves order
std::unordered_set set same as above
std::deque deque conceptually close, both block-based

Core takeaway: Python containers are containers of references. Copying a container copies only pointers by default (shallow copy), and multiple containers can alias the same batch of elements; for true independence you must deepcopy. If you want C-style contiguous value storage, cache-friendliness, and vectorization, use array/memoryview or numpy.

Summary

  • Usage level: list uses append/insert/remove/pop/slicing for add/remove/lookup/modify; dict uses []/get/update/pop/setdefault, with for k, v in d.items() for iteration; a tuple is immutable once created and is often used for unpacking; set uses add/remove and supports the four set operations &/|/-/^; use collections.deque when you need fast insertion/removal at both ends.
  • Internals level: list is a dynamic array of pointers to PyObject, growing by a mild ~1/8 over-allocation, and its shallow copies duplicate only pointers, not elements; tuple inlines its pointer array at the tail of the object, allocated once and immutable; dict is an open-addressing hash table split into two arrays, indices (for hash positioning) and entries (compact, insertion-ordered); set is a hash table with no values and no ordering guarantee; deque is a doubly-linked list of blocks, O(1) at both ends and O(n) for random access.
