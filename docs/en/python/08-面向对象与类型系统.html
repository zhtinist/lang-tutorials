Object Orientation and the Type System

C++’s object model is “compile-time-fixed memory layout + vtable”; Python’s object model is “runtime-mutable dictionary + descriptor protocol.” This chapter takes apart classes, instances, methods, inheritance, and metaclasses layer by layer, comparing each with C++’s object memory layout. Prerequisite: see “Variables and the Memory Model”—everything is an object, a variable is a reference.

1. class Syntax and the Memory Layout of an Instance

class Point:
    count = 0                 # class attribute (shared by all instances)
    def __init__(self, x, y):
        self.x = x            # instance attribute, written into self.__dict__
        self.y = y
        Point.count += 1

p = Point(3, 4)
print(p.x, p.__dict__)        # 3 {'x': 3, 'y': 4}

1.1 Instance Attributes Live in __dict__

A C++ object is a contiguous block of memory whose fields have fixed offsets fixed at compile time:

C++  struct Point { int x; int y; };
p:  [ x=3 ][ y=4 ]      // accessing p.x is *(int*)(&p + 0); the offset is hardcoded into the instruction

By default, each Python instance carries a __dict__ (a dict), a mapping from attribute names to object references:

Python instance p (heap object)
+------------------+
| ob_refcnt        |
| ob_type -> Point |
| __dict__ ------- |----> {'x': -> PyLong(3), 'y': -> PyLong(4)}
+------------------+

Consequences:

  • Accessing p.x is a dictionary lookup, not a constant-offset dereference, so it’s slower.
  • You can add or remove attributes on any instance at runtime: p.z = 5 just stuffs a key into its __dict__. In C++ an object’s set of fields is fixed at compile time; you can’t do this.
  • Each instance carries an extra dict, a nontrivial memory overhead. To eliminate it, see __slots__ later.

CPython has an optimization called “key-sharing dict”: multiple instances of the same class, if they have the same set of attributes, can share the key portion and each store only a value array, saving memory. This makes “instance dicts” less expensive, but still not as good as fixed offsets.

2. Classes Are Objects Too, and type Is the Metaclass

In Python, a class itself is also an object, whose type is type:

print(type(p))         # <class '__main__.Point'>   —— p's type is Point
print(type(Point))     # <class 'type'>              —— Point's type is type
print(type(type))      # <class 'type'>              —— type's type is itself
  • Point is an object living on the heap; you can assign it, pass it, and put it in a list.
  • The “class” that creates classes is called the metaclass, by default type. class Point: ... is essentially equivalent to calling type(name, bases, namespace) to dynamically build a class object:
Point2 = type('Point2', (object,), {'x': 0})   # equivalent to the class statement

This is entirely unlike C++: a C++ class is only a compile-time concept, and there is no “class object” at runtime (RTTI’s type_info is just read-only metadata and can’t be used as a value to construct new classes). Python’s classes are first-class citizens and can be generated and modified at runtime. Metaclasses are an advanced feature (ORMs and frameworks use them for registration/validation); you don’t write them by hand day to day.

3. Attribute Lookup Order: Instance → Class → MRO

When accessing obj.attr, the interpreter searches in a fixed order. This is the core mechanism for understanding OOP:

  1. Data descriptors (class attributes that define __set__/__delete__, such as property)—take precedence over the instance dict;
  2. The instance __dict__—obj.__dict__['attr'];
  3. The class and its MRO—search each class’s __dict__ along type(obj).__mro__; a non-data descriptor encountered on the way (such as a plain function) triggers the descriptor protocol;
  4. None found → call __getattr__ (if defined) → otherwise AttributeError.

(Detailed order: data descriptor > instance dict > non-data descriptor / class attribute > __getattr__.)

3.1 MRO and C3 Linearization

With multiple inheritance, the “class chain” is not a simple depth-first traversal but a definite order computed by the C3 linearization algorithm, guaranteeing: subclasses before base classes, the relative order of base classes preserved, and a unique result (or an error).

class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print([c.__name__ for c in D.__mro__])
# ['D', 'B', 'C', 'A', 'object']

C3 solves C++’s notorious diamond inheritance problem. C++ requires manual handling of duplicate base classes via virtual inheritance; Python automatically produces a linear order via the MRO, and super() walks along it (see section 6).

4. Method = Function + Descriptor Protocol: Where Bound Methods Come From

A “method” is really just an ordinary function defined in the class’s __dict__. The magic is that a function is a descriptor (it implements __get__).

class Foo:
    def bar(self):
        return self

f = Foo()
print(Foo.bar)      # <function Foo.bar>       —— accessed via the class, it's a plain function
print(f.bar)        # <bound method Foo.bar>   —— accessed via an instance, it's a "bound method"

What happened: f.bar triggers attribute lookup, finds the function object bar in the class, and because a function has __get__, the interpreter calls bar.__get__(f, Foo), which returns a bound method object—a callable wrapper that pre-fills f into the first parameter (self):

f.bar()  is equivalent to  Foo.bar(f)
         the bound method internally stores (__func__ = bar, __self__ = f)
         at call time, __self__ is automatically passed as the first argument

This explains why a method’s first parameter must be written explicitly as self—it’s not a keyword, just the conventional name for “the parameter that gets auto-filled at binding time.” Compared to C++: C++’s this is a hidden pointer implicitly passed by the compiler; Python makes this explicit as a descriptor-protocol step performed at runtime.

5. @property / @staticmethod / @classmethod

These three are all “built-in descriptors” used to change the binding behavior of an attribute/method:

class Circle:
    def __init__(self, r):
        self._r = r

    @property                       # disguises a method as a read-only attribute, called automatically on access
    def area(self):                 # a data descriptor, higher priority than the instance dict
        return 3.14159 * self._r ** 2

    @staticmethod                   # binds neither self nor cls; just a plain function in the class namespace
    def unit():
        return "cm"

    @classmethod                    # binds the "class" rather than an instance; the first parameter is cls
    def from_diameter(cls, d):
        return cls(d / 2)           # cls makes a subclass call return a subclass instance (polymorphic construction)

c = Circle(2)
print(c.area)                       # 6.28...  —— accessed like a field, calls a method underneath
print(Circle.from_diameter(10)._r)  # 5.0
Decorator First parameter When called via an instance Typical use
plain method self (instance) binds the instance regular behavior
@classmethod cls (class) binds the class alternative constructors, manipulating class state
@staticmethod none no binding logic that belongs to the class but needs no self/cls
@property self access triggers the call computed attributes, read-only encapsulation, added validation

@property is language-level support for C++’s “encapsulate a private field with getter/setter”—but it’s transparent to the caller: c.area looks like a field access, so a field can be upgraded to a computed attribute without changing calling code.

6. Inheritance and super

class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return "..."

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)      # calls the previous class's implementation along the MRO
        self.breed = breed
    def speak(self):
        return "Woof"

super() (the argument-less form in Python 3) does not mean “call the parent class” but “find the next class along the current instance’s MRO.” This is essential in multiple inheritance—it ensures that in cooperative multiple inheritance each base class’s method is called exactly once, chained in C3 order. This is more flexible than hand-writing Base::method() (hardcoding a specific base class) in C++, and it avoids a base class being called twice in diamond inheritance.

7. Duck Typing vs. C++ Virtual Functions / Static Typing

C++’s polymorphism relies on a virtual function table (vtable): a vptr hidden in the object header points to the vtable, a virtual call is “look up the table + indirect jump,” and types are checked at compile time—only classes declaring a common base can be polymorphic.

Python is duck typing: when you call x.speak(), the interpreter doesn’t care what type x is at all; at runtime it looks up the speak attribute in x’s type and calls it if found. “If it walks like a duck and quacks like a duck, treat it as a duck.”

class Duck:
    def speak(self): return "Quack"

class Person:
    def speak(self): return "Hello"

def make_it_speak(x):        # needs no common base class or interface declaration
    return x.speak()         # looks up the method by x's actual type at runtime

Comparison:

C++ virtual function Python duck typing
Dispatch timing interface fixed at compile time, vtable looked up at runtime entirely at runtime, looking up the type’s dict
Requires common base yes (inheritance + virtual) no (a same-named method suffices)
Type checking static at compile time at runtime; raises AttributeError only when it’s wrong
Overhead one indirect jump one dictionary lookup (slower)

The cost is “no declared interface, errors deferred to runtime.” For a middle ground, see abstract base classes in section 10, and type annotations + mypy for static checking.

8. __slots__: Replacing the dict with a Fixed Array

The default instance __dict__ is flexible but memory-hungry. If a class will create huge numbers of instances (say, a million Points), you can declare a fixed set of fields with __slots__:

class Point:
    __slots__ = ('x', 'y')      # after this: no more __dict__, fields live in fixed slots
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(3, 4)
# p.z = 5          # AttributeError: can no longer add attributes dynamically
# p.__dict__       # AttributeError: there is no __dict__ anymore

The underlying change: CPython allocates a fixed-offset slot for each slot on the type object (implemented via descriptors); the instance no longer carries a dict but, like a C struct, stores field value pointers contiguously within the object body:

instance with __dict__:  [head][__dict__ ptr] -> {'x':.., 'y':..}   (an extra dict)
instance with __slots__:  [head][x ptr][y ptr]                        (fields inlined, no dict)

The gain: each instance saves a dict (tens to hundreds of bytes), and access becomes a fixed offset (close to C++ field access, fast). The cost: you lose dynamic attribute add/remove, lose __dict__, and slots have restrictions under multiple inheritance. This is the way of writing that comes closest to C struct memory layout in Python’s object model.

9. Magic Methods and Operator Overloading

“Magic methods” (dunder methods, with double underscores) are hooks the interpreter calls back on specific syntax/operations. They let your object integrate with the language’s operators, built-in functions, and protocols.

class Vec:
    def __init__(self, x, y): self.x, self.y = x, y
    def __repr__(self):  return f"Vec({self.x}, {self.y})"   # repr() / interactive display
    def __add__(self, o): return Vec(self.x+o.x, self.y+o.y) # supports v1 + v2
    def __eq__(self, o):  return (self.x, self.y) == (o.x, o.y)
    def __len__(self):    return 2                            # supports len(v)
    def __getitem__(self, i): return (self.x, self.y)[i]     # supports v[0], iteration
    def __hash__(self):   return hash((self.x, self.y))      # can go into a set / dict key

Common magic methods:

Syntax/operation Magic method Corresponding C++
a + b __add__ / __radd__ operator+
a == b __eq__ operator==
a < b __lt__ operator<
len(a) __len__ size() convention
a[i] __getitem__/__setitem__ operator[]
a() __call__ operator() (functor)
str(a)/print __str__ operator<< convention
with a: __enter__/__exit__ RAII (see chapter 9)
for x in a __iter__/__next__ iterator

The power of operator overloading rivals C++’s, but it’s all done at runtime by looking up methods, with no compile-time overload resolution.

10. Abstract Base Classes (abc)

Duck typing is flexible, but sometimes you want to enforce “subclasses must implement certain methods,” or to categorize via isinstance. The abc module provides abstract base classes:

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...    # abstract method, subclasses must implement

class Square(Shape):
    def __init__(self, s): self.s = s
    def area(self): return self.s * self.s

# Shape()      # TypeError: can't instantiate a class with unimplemented abstract methods
Square(3)      # OK
  • Similar to C++’s pure virtual functions (virtual void area() = 0;) + an abstract class that can’t be instantiated.
  • But the check happens at instantiation time (runtime), not at compile time.
  • abc also supports registering “virtual subclasses” and structured isinstance checks; collections.abc defines protocol base classes such as Iterable/Sequence/Mapping, usable for type checks.
  • A lighter, more modern approach is typing.Protocol (structural subtyping / static duck typing), which, together with mypy, verifies interfaces at compile time (check time) without intruding on runtime.

Summary

  • Instance attributes live in __dict__ by default; access is a dictionary lookup, not C++’s fixed offset; __slots__ can switch back to near-struct fixed slots.
  • Classes are runtime objects, and type is the default metaclass.
  • Attribute lookup: data descriptor → instance dict → MRO (C3 linearization).
  • Methods are functions + the descriptor protocol; self / bound methods are produced by __get__ at runtime.
  • Polymorphism relies on runtime method lookup (duck typing), not a compile-time vtable; for interface constraints use abc/Protocol.
