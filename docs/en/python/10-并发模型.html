Concurrency Models: GIL, Threads, Processes, and Coroutines

This chapter first covers how to write and use the three concurrency APIs—threading (threads), multiprocessing (processes), and asyncio (coroutines)—no low-level knowledge required to follow along; then it covers CPython’s own GIL (Global Interpreter Lock). Understand it, and you’ll understand why Python’s multithreading “looks parallel but actually isn’t,” and when to use threads, processes, or coroutines. As a C/C++ engineer, you’re familiar with pthread, std::thread, atomic operations, memory ordering, and mutexes—the second half compares each of these in turn.

0. Get Using It First: threading / multiprocessing / asyncio at a Glance

threading: Spin Up a Thread to Run a Function

import threading

def greet(name):
    print(f"hello, {name}")

t = threading.Thread(target=greet, args=("world",))
t.start()     # start the thread
t.join()      # wait for the thread to finish
print("done")
# hello, world
# done

multiprocessing: Spin Up a Child Process to Run a Function

The usage is almost identical to threading—just swap Thread for Process:

from multiprocessing import Process

def greet(name):
    print(f"hello from process, {name}")

if __name__ == "__main__":   # this line is required (see below)
    p = Process(target=greet, args=("world",))
    p.start()
    p.join()
    print("done")
# hello from process, world
# done

asyncio: Write Coroutines with async/await

import asyncio

async def greet(name):              # async def defines a "coroutine function"
    print(f"start {name}")
    await asyncio.sleep(0.01)       # await: wait for an async operation
    print(f"done {name}")
    return name

async def main():
    result = await greet("world")   # calling a coroutine requires await
    print(result)

asyncio.run(main())                  # start the event loop and run main
# start world
# done world
# world

concurrent.futures: A More Convenient Thread/Process Pool

from concurrent.futures import ThreadPoolExecutor

def square(x):
    return x * x

with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(square, range(5)))
print(results)   # [0, 1, 4, 9, 16]

These four are the most common entry points for concurrent programming. What follows covers each one’s scheduling model and trade-offs—usage first, internals second.

1. Going Deeper: What the GIL Is

From this section on, we move into the internals part of the chapter: you already know the basic usage of threads, processes, and coroutines. What follows explains why “Python’s multithreading doesn’t seem to actually parallelize across multiple cores,” and what each approach is best suited for.

CPython traditionally has a single global mutex: at any moment, only one thread can hold the GIL and execute Python bytecode. It’s not part of the language spec but a product of the CPython implementation (Jython and IronPython have no GIL). (Strictly speaking, since 3.12 the GIL is per-interpreter—multiple sub-interpreters in the same process can each have their own GIL; see PEP 684. The discussion below assumes the most common single-interpreter case.)

import sys
# how often (in "check intervals") the GIL considers a switch (Python 3.2+ switched to a time basis)
print(sys.getswitchinterval())  # default 0.005 seconds, i.e. 5ms

In C mental-model terms, the GIL is like wrapping a big lock around the interpreter’s main loop:

Thread T1                  GIL                     Thread T2
  |                     [ idle ]                     |
  |-- acquire GIL ------>[ T1 ]                      |
  |  run bytecode...      [ T1 ]     <-- blocked ----|
  |  (every 5ms or when blocking)                    |
  |-- release GIL ------>[ idle ]                     |
  |                      [ T2 ]<---- acquire GIL -----|
  |  <-- blocked --------[ T2 ]     run bytecode...   |

2. Why It Exists: Protecting Reference Counting

Recall Chapter 09—CPython manages memory with reference counting. Every object header has an ob_refcnt. When you do a = b or pass an argument, the reference count goes +1; when a variable is destroyed, -1; reaching zero frees it.

Here’s the problem: ob_refcnt++ and ob_refcnt-- are not atomic operations. At the C level they’re a three-step “read-modify-write.” If two threads increment/decrement the same object’s reference count concurrently without a lock:

initial refcnt = 1
T1: read refcnt(1)        T2: read refcnt(1)
T1: +1 gives 2            T2: -1 gives 0
T1: write back 2          T2: write back 0   <-- race! the object is wrongly freed, or never freed

This is exactly the data race you know. There are two solutions:

  1. Add fine-grained locks / use atomic operations on every object’s reference count—but objects number in the millions, and atomic operations trigger cache-coherency overhead on every INCREF/DECREF, crashing single-threaded performance.
  2. Protect the entire interpreter state with a single big lock—this is the GIL.

CPython chose option 2, because it makes single-threaded performance optimal and makes writing C extensions extremely simple (extension authors assume by default that they own the interpreter exclusively, with no need to think about concurrency). The GIL protects not just reference counts but various global interpreter states (the memory allocator, internal caches, etc.).

3. The Consequence: CPU-Bound Multithreading Doesn’t Speed Up

Since only one thread runs bytecode at a time, pure computation (CPU-bound) tasks don’t get faster with more threads—in fact they get slightly slower due to lock contention and context switching.

import time
from threading import Thread

def cpu_bound(n: int) -> int:
    # pure computation: hogs the CPU, never releases the GIL
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 20_000_000

# serial
t0 = time.perf_counter()
cpu_bound(N); cpu_bound(N)
print(f"serial: {time.perf_counter() - t0:.2f}s")

# two threads
t0 = time.perf_counter()
threads = [Thread(target=cpu_bound, args=(N,)) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(f"two threads: {time.perf_counter() - t0:.2f}s")  # won't be twice as fast, may even be slower

Compare with C++: two std::threads running pure computation can saturate two cores, achieving near-linear speedup. In Python, two threads fight over the same GIL, so only one core is ever really working:

CPU-bound Python multithreading on a dual-core CPU:

Core 0: [T1][   ][T1][   ][T2][   ]...   <- the two threads take turns on the same core
Core 1: [idle...........................]  <- the second core is wasted

C++ std::thread on the same task:
Core 0: [T1][T1][T1][T1][T1][T1]...      <- fully loaded
Core 1: [T2][T2][T2][T2][T2][T2]...      <- fully loaded, truly parallel

4. threading: IO-Bound Is Its Home Turf

Key point: when performing blocking IO (reading/writing files, network requests, time.sleep), a thread proactively releases the GIL. Since no bytecode needs to run while waiting on IO, it makes sense to let another thread run. In addition, the interpreter also forces a switch periodically (5ms by default).

So threading is well-suited to IO-bound scenarios: multiple threads spend most of their time waiting on the network/disk, the GIL is released during the wait, and effective concurrency is achieved.

import time, urllib.request
from threading import Thread

def fetch(url: str) -> None:
    # the GIL is released while blocked on the network request, so other threads can run
    with urllib.request.urlopen(url) as r:
        r.read()

urls = ["https://example.com"] * 8

t0 = time.perf_counter()
for u in urls: fetch(u)
print(f"serial: {time.perf_counter() - t0:.2f}s")

t0 = time.perf_counter()
threads = [Thread(target=fetch, args=(u,)) for u in urls]
for t in threads: t.start()
for t in threads: t.join()
print(f"multithreaded: {time.perf_counter() - t0:.2f}s")  # significant speedup, IO waits are overlapped

Thread Synchronization Primitives

As in C++, shared mutable state needs protection. Note: even with the GIL, compound operations (like x += 1) are not atomic—they consist of multiple bytecodes, and a switch may happen in between.

from threading import Lock, Thread

counter = 0
lock = Lock()

def worker():
    global counter
    for _ in range(100_000):
        with lock:          # equivalent to std::lock_guard, auto-released on leaving scope
            counter += 1     # read-modify-write, must be locked

threads = [Thread(target=worker) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)  # with lock: 400000; without lock: less than that (lost updates)

threading also provides RLock (reentrant), Semaphore, Event, Condition, and Barrier, with semantics largely matching their C++ standard-library counterparts.

5. multiprocessing: Bypassing the GIL for True Parallelism

To saturate multiple cores on CPU-bound tasks, you must spawn multiple processes. Each process has its own Python interpreter and its own GIL, independent of the others, scheduled by the OS onto different cores.

import time
from multiprocessing import Process

def cpu_bound(n: int) -> int:
    return sum(i * i for i in range(n))

if __name__ == "__main__":   # Windows/spawn mode requires this guard
    N = 20_000_000
    t0 = time.perf_counter()
    ps = [Process(target=cpu_bound, args=(N,)) for _ in range(2)]
    for p in ps: p.start()
    for p in ps: p.join()
    print(f"two processes: {time.perf_counter() - t0:.2f}s")  # close to half the serial time

The cost is that processes don’t share memory (just like after a fork, each writes its own copy). There are two paths for passing data:

Method Mechanism Notes
Arguments/return values/queues pickle serialization objects are serialized into a byte stream, sent to the child process via a pipe, then deserialized. There’s copy overhead, and the object must be picklable
Value/Array/shared_memory shared memory maps the same physical memory directly, zero-copy, but you must lock and synchronize yourself 
from multiprocessing import shared_memory
import numpy as np

# create a shared memory block; multiple processes can map the same physical pages, no copying needed
shm = shared_memory.SharedMemory(create=True, size=1024)
buf = np.ndarray((256,), dtype=np.int32, buffer=shm.buf)
buf[0] = 42
# another process attaches to the same memory using shm.name
shm.close(); shm.unlink()

Start methods (start method): fork (copies the parent’s memory, fast, but unsafe when combined with multithreading), spawn (a fresh interpreter, clean but slow), and forkserver. The default varies by platform/version: macOS and Windows default to spawn; Linux historically defaulted to fork, but since 3.14 defaults to forkserver (because fork is prone to problems in multithreaded programs). Don’t keep memorizing “Unix defaults to fork.”

6. asyncio: A Single-Threaded Event Loop

asyncio is a different route: instead of multiple threads/processes, it achieves high-concurrency IO within a single thread using an event loop + coroutines. It’s essentially cooperative multitasking, in contrast to threads’ preemptive scheduling.

  • Coroutine: defined with async def. Calling it doesn’t execute immediately but returns a coroutine object (like a lazy task description).
  • await: proactively yields control to the event loop here. Only when it hits an await on an operation that isn’t ready does the current coroutine pause, and the event loop goes to run another coroutine.
  • Event loop: a while loop that maintains a queue of ready tasks, working with the OS’s epoll/kqueue/IOCP to monitor IO events, and resuming the corresponding coroutine whenever its IO becomes ready.
import asyncio, time

async def fetch(name: str, delay: float) -> str:
    print(f"{name} started")
    await asyncio.sleep(delay)   # yields control! the event loop runs another coroutine
    print(f"{name} done")
    return name

async def main():
    t0 = time.perf_counter()
    # gather schedules multiple coroutines concurrently; total time ≈ the longest one, not the sum
    results = await asyncio.gather(
        fetch("A", 2), fetch("B", 1), fetch("C", 3),
    )
    print(results, f"took {time.perf_counter() - t0:.2f}s")  # about 3s, not 6s

asyncio.run(main())   # create the event loop, run main to completion, then close the loop

Mental model of the event loop:

Event loop (single thread):
  ┌─> take a coroutine from the ready queue, run it to the next await
  │      coroutine A hits await sleep(2) --> register a timer, suspend A
  │      coroutine B hits await sleep(1) --> register a timer, suspend B
  │      queue empty, epoll waits for IO/timers to become ready
  │      1s later B's timer fires --> put B back on the ready queue
  └──── resume B and continue after its await ...

Print the concurrent interleaving explicitly—every await is a yield point, and you can watch the two coroutines take turns advancing at each switch:

import asyncio

async def worker(name: str, steps: int) -> None:
    for i in range(steps):
        print(f"{name}-{i}")
        await asyncio.sleep(0)   # yield control; the event loop runs the other coroutine

async def main():
    await asyncio.gather(worker("A", 3), worker("B", 3))

asyncio.run(main())
# A-0
# B-0
# A-1
# B-1
# A-2
# B-2   —— the two coroutines alternate at each await (cooperative, deterministic order)

Contrast with Threads

Dimension threading asyncio
Scheduling preemptive, OS decides when to switch cooperative, switches only at await
Switch points any bytecode boundary (uncontrollable) explicit await (controllable, reasoned about)
Parallelism limited by the GIL, no true parallelism single-threaded, no true parallelism
Races needs locks, hard to debug switch points are known, most operations within the thread need no locks
Overhead MB-scale stack per thread + OS scheduling a few KB per coroutine, tens of thousands possible
Fits IO-bound + blocking libraries IO-bound + a fully async ecosystem

Key pitfall: calling a blocking function inside a coroutine (e.g. synchronous requests.get, time.sleep) freezes the entire event loop—because there’s no await, control is never yielded. You must use async libraries (aiohttp, httpx) or offload the blocking call to a thread pool: await loop.run_in_executor(None, blocking_fn).

import asyncio, time

def blocking_io(x: int) -> int:   # a synchronous blocking function (e.g. sync requests.get)
    time.sleep(0.05)
    return x * 2

async def main():
    loop = asyncio.get_running_loop()
    # offload to the default thread pool; the blocking happens on another thread, the event loop stays alive
    results = await asyncio.gather(
        loop.run_in_executor(None, blocking_io, 1),
        loop.run_in_executor(None, blocking_io, 2),
    )
    print(results)   # [2, 4]

asyncio.run(main())

7. concurrent.futures: A Unified High-Level Interface

concurrent.futures provides a unified API for thread pools and process pools, and is the most commonly used entry point day to day (the “get using it first” section already showed the basic usage of ThreadPoolExecutor). The Future object is like C++’s std::future.

from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def task(x: int) -> int:
    return x * x

# IO-bound: use a thread pool
with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(task, range(10)))

# CPU-bound: use a process pool (bypass the GIL)
if __name__ == "__main__":
    with ProcessPoolExecutor(max_workers=4) as ex:
        futures = [ex.submit(task, i) for i in range(10)]
        for f in futures:
            print(f.result())   # blocks until that task completes

Rule of thumb: CPU-bound → ProcessPool; IO-bound → ThreadPool or asyncio.

8. free-threaded (no-GIL): The GIL’s Twilight?

CPython introduced a free-threaded build as an experimental feature in 3.13 (released 2024) (PEP 703), i.e. “no-GIL”; in 3.14 it became officially supported, but still not the default. Using techniques like biased locks + lock-free (biased) reference counting + per-object locks, it makes reference counting safe under multithreading, thereby removing the GIL and achieving genuinely parallel multi-core threads.

Key points:

  • It requires a dedicated build (e.g. python3.13t / python3.14t); regular distributions still default to the GIL build.
  • Single-threaded performance regresses somewhat (a few percent early on, continually being optimized).
  • C extensions need adapting (declaring free-threading support); ecosystem migration takes time.
  • 3.13 experimental → 3.14 officially supported (but not default); the GIL and free-threaded builds will coexist for a long time.

If this route succeeds, future Python’s threading will be able to saturate multiple cores like C++’s std::thread, and many of multiprocessing’s pickle/shared-memory pain points will ease. But for the foreseeable future, the selection principles above still hold.

Summary

  • Usage level: threading.Thread(target=...) spins up a thread and multiprocessing.Process(target=...) spins up a process, with almost identical syntax; async def/await define and drive coroutines, and asyncio.run(main()) starts the event loop; concurrent.futures’s ThreadPoolExecutor/ProcessPoolExecutor provide a more convenient pooled interface (.map/.submit).
  • Internals level: the GIL is CPython’s interpreter-level big lock (per-interpreter since 3.12); only one thread executes bytecode at a time, and its fundamental reason for existing is to protect non-atomic reference counting and other internal interpreter state. CPU-bound multithreading doesn’t speed up; IO-bound multithreading is effective, because blocking IO releases the GIL. multiprocessing achieves true parallelism with separate processes + separate GILs, at the cost of passing data via pickle or shared memory. asyncio uses a single-threaded event loop + cooperative coroutines, yielding explicitly at await, well-suited to high-concurrency IO; never call a blocking function inside it. The free-threaded build (experimental in 3.13, official but not default in 3.14) is removing the GIL and is the long-term direction of evolution.
