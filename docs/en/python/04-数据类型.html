Data Types

In the previous chapter we established the worldview that “a variable is a reference to an object.” This chapter takes the common built-in data types apart one by one and looks at what they actually look like inside CPython—what bytes follow the PyObject header, and how they differ from the C fixed-width int, the IEEE-754 double, and char* you know.

The sentence running through this chapter: in C the type determines how many bytes a variable occupies; in Python the type is a struct in the object heap, and the variable is just a pointer to it. So the question “how many bytes does an int take” in Python is really asking “how big is this PyLongObject struct,” not “what’s the machine word size.”

4.1 Everything Is an Object: the Unified Object Header

Recall see “Variables and the Memory Model”: in CPython every object begins on the heap with a PyObject (or the variable-length PyVarObject):

// CPython source Include/object.h (simplified)
typedef struct _object {
    Py_ssize_t    ob_refcnt;    // reference count
    PyTypeObject *ob_type;      // pointer to the type object (equivalent to C++'s vptr)
} PyObject;

typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;         // "number of elements" for variable-length objects
} PyVarObject;

ob_type is the analog of C++’s virtual table pointer (vptr): it points to a PyTypeObject that stores “how this type does addition, how it prints, how it takes its length,” and a whole pile of function pointers (called tp_slots). The difference is that C++’s vtable is fixed at compile time, whereas Python’s type object is an ordinary object modifiable at runtime.

On 64-bit CPython this header is 16 bytes (PyObject) or 24 bytes (PyVarObject). Any Python object is at least this big—which is why Python’s int easily runs to twenty or thirty bytes while C’s int is only 4.

import sys
print(sys.getsizeof(0))      # 28 (64-bit CPython 3.12+; earlier versions 24)
print(sys.getsizeof(1.0))    # 24

4.2 int: the Arbitrary-Precision, Variable-Length Big Integer

4.2.1 Semantics: Integers in the Mathematical Sense, No Overflow

In C, int32_t x = INT_MAX; x + 1; overflows and wraps into a negative number. Python’s int is arbitrary precision:

x = 2 ** 100
print(x)            # 1267650600228229401496703205376, no overflow
print(x * x)        # keeps growing with no problem

This isn’t magic; it stores the integer as an array of “digits,” expanding the number of digits on demand like long multiplication in grade school—essentially the bignum you’d hand-write in C (what libraries like GMP do), just built in.

4.2.2 Memory Layout: PyLongObject and 30-bit Digits

CPython’s integer is the variable-length object PyLongObject:

// simplified conceptual structure
typedef struct {
    PyObject_VAR_HEAD          // ob_refcnt, ob_type, ob_size
    digit ob_digit[1];         // variable-length array, each digit holds 30 bits
} PyLongObject;

Key points:

  • Each digit is a 32-bit unsigned integer (uint32_t), but only 30 of its bits are used (PYLONG_BITS_IN_DIGIT == 30). The spare bits make carry and borrow operations convenient without overflowing C’s 32 bits.
  • The absolute value of ob_size is the digit count, and its sign indicates whether the integer is positive or negative; ob_size == 0 represents the value 0. This is a “sign-magnitude” representation, not two’s complement.

Note: since CPython 3.12 the internals use an lv_tag packing ob_size plus a flag bit (not covered by any PEP; an implementation detail), but the “30-bit digit array + sign” model is unchanged.

An ASCII diagram representing 2**60 (needs 3 30-bit digits, because 60 bits don’t fit into two full 30-bit digits… actually 2**60 has exactly bit 61 set):

value = digit[0] * 2^0 + digit[1] * 2^30 + digit[2] * 2^60

+-------------------+
| ob_refcnt         |  \
| ob_type  --> int  |   > object header (PyVarObject)
| ob_size = 3       |  /   (positive sign + 3 digits)
+-------------------+
| ob_digit[0] (30b) |
| ob_digit[1] (30b) |
| ob_digit[2] (30b) |
+-------------------+

Compared with C:

C int32_t Python int
Size fixed 4 bytes variable, grows with the value
Overflow wraparound (UB/implementation-defined) none, grows automatically
Negative representation two’s complement sign bit + magnitude (digit array)
Location can be on the stack/in a register always on the heap; the variable holds a pointer
Operation a single CPU instruction a function call, processed digit by digit

Observe the growth with sys.getsizeof:

import sys
print(sys.getsizeof(0))          # 28 (empty-digit case, depends on version)
print(sys.getsizeof(2**29))      # 28  —— 1 digit
print(sys.getsizeof(2**30))      # 32  —— needs 2 digits
print(sys.getsizeof(2**60))      # 36  —— 3 digits
print(sys.getsizeof(2**300))     # bigger still

Each extra 30-bit digit adds about 4 bytes, which matches the structure above exactly.

4.2.3 Small-Integer Cache

At startup CPython pre-creates all integer objects in the range [-5, 256], permanently resident. Every reference to these values points to the same object:

a = 100
b = 100
print(a is b)       # True —— the same cached object

x = 1000
y = 1000
print(x is y)                       # usually True within one compilation unit: the literal 1000 is reused in co_consts (constant deduplication, not caching)
print(int("1000") is int("1000"))   # False: created separately at runtime, so they truly fall outside the cache range

This is a pure CPython optimization (unrelated to a C++ compiler’s handling of small literals; more like a shared constant pool). Never use is to compare numeric values; use ==.

4.2.4 Viewing Integer Literals with dis

import dis
dis.dis(compile("x = 2 + 3", "<s>", "exec"))

You’ll see LOAD_CONST 5—compile-time constant folding has already computed 2 + 3 into 5, and no addition happens at runtime. This is similar to a C compiler’s constant folding.

4.3 bool: a Subclass of int

In Python bool inherits from int, and True and False are two singleton integer objects with values 1 and 0:

print(isinstance(True, int))   # True
print(True + True)             # 2 —— treated as an integer in arithmetic
print(True == 1)               # True
print(True is 1)               # False —— values equal but not the same object

Under the hood True/False are two pre-created, globally unique PyLongObjects (with the bool type). This is entirely unlike C’s bool (_Bool, usually 1 byte): C’s bool is one byte on the stack, whereas Python’s bool is a heap object inheriting from int.

Practical consequence: True can serve as an array index, and you can sum booleans to count:

data = [1, -2, 3, -4, 5]
positives = sum(x > 0 for x in data)   # summing True/False as 1/0 = 3

4.4 float: a Boxed C double

Python’s float is simply an IEEE-754 double-precision float, boxed inside an object:

typedef struct {
    PyObject_HEAD
    double ob_fval;      // a genuine C double
} PyFloatObject;

So a float object = 16-byte object header + 8-byte double = 24 bytes:

import sys
print(sys.getsizeof(3.14))   # 24

Compared with C: C’s double x = 3.14; is an 8-byte raw value on the stack, and arithmetic goes straight to FPU instructions. In Python each float is a heap object, and a + b must dereference two pointers to fetch ob_fval, add them, and then allocate a new float object to hold the result. This is the root of pure Python being slow and memory-hungry in numeric-intensive computation, and the reason NumPy exists—NumPy packs a hundred million doubles into one contiguous C array rather than a hundred million 24-byte objects (see “Common Third-Party Libraries”).

All the floating-point “gotchas” are exactly the same as in C, because the underlying thing is the same double:

print(0.1 + 0.2)            # 0.30000000000000004
print(0.1 + 0.2 == 0.3)     # False

Special values are consistent too: float('inf'), float('-inf'), float('nan'), and nan != nan holds.

When you need exact decimals (finance) use the standard library decimal.Decimal; for exact fractions use fractions.Fraction—they’re implemented internally with integers, not doubles.

4.5 complex: Two doubles

complex is simply an object holding two C doubles (the real and imaginary parts):

typedef struct {
    PyObject_HEAD
    double real;
    double imag;
} PyComplexObject;
z = 3 + 4j
print(z.real, z.imag)   # 3.0 4.0
print(abs(z))           # 5.0

The imaginary literal suffix is j. It’s a built-in type; no import needed.

For trigonometric/exponential operations on complex numbers, use the standard library cmath (not math, which only accepts reals):

import cmath
z = 1 + 1j
print(cmath.phase(z))        # 0.7853981633974483 —— argument (π/4 radians)
print(abs(z))                # 1.4142135623730951 —— magnitude
r, phi = cmath.polar(z)      # to polar coordinates: (magnitude, argument)
print(cmath.rect(r, phi))    # (1.0000000000000002+1j) —— back again, float error visible
print((-1 + 0j) ** 0.5)      # (6.123233995736766e-17+1j) —— √-1 ≈ i (real part is float noise)

Note that (-1) ** 0.5 on a float raises or returns nan, but on a complex it takes the complex square root, so √-1 gives approximately i (the 6e-17 real part is floating-point rounding noise, ideally 0).

4.6 str: PEP 393 Compact Unicode

Strings are one of the areas where C and Python differ most. C’s char* is a run of bytes terminated by \0; Python’s str is an immutable sequence of Unicode code points, and CPython chooses the most economical internal encoding based on the content.

4.6.1 PEP 393: Choose Storage Width by the Largest Code Point

Since Python 3.3 (PEP 393), CPython no longer fixes 2 or 4 bytes per character; instead it looks at the largest code point in the string and picks one of three compact representations:

Internal kind Bytes per char Representable range Example
Latin-1 (PyUnicode_1BYTE_KIND) 1 U+0000–U+00FF "hello", "café"
UCS-2 (PyUnicode_2BYTE_KIND) 2 U+0000–U+FFFF "你好"
UCS-4 (PyUnicode_4BYTE_KIND) 4 U+0000–U+10FFFF "😀" (needs a supplementary plane)

As soon as one high-code-point character appears in the string, the entire string is promoted to the wider representation. Observe:

import sys
print(sys.getsizeof(""))        # 41 (64-bit CPython 3.12+; 3.11 and earlier ~49, the header shrank after PEP 623 removed wstr)
print(sys.getsizeof("a"))       # 42   —— +1 byte (Latin-1)
print(sys.getsizeof("ab"))      # 43   —— +2 bytes
print(sys.getsizeof("你"))      # 60 —— UCS-2, 2 bytes per char and a larger baseline
print(sys.getsizeof("😀"))      # 64 —— UCS-4, 4 bytes per char

Conceptual memory layout (compact form) for "café" (all within the Latin-1 range):

+---------------------------+
| ob_refcnt / ob_type       |  object header
| length = 4                |  number of code points (not bytes)
| hash (cached)             |
| kind = 1BYTE, flags...    |  storage kind
+---------------------------+
| 'c' 'a' 'f' 'é' '\0'      |  contiguous code-point data right after the object
+---------------------------+

Key differences compared with C:

C char* Python str
Unit byte Unicode code point
Length via \0 terminator, O(n) to compute length stored explicitly, O(1)
len("😀") by byte count (4, UTF-8) 1 (one code point)
Mutability mutable immutable
Indexing s[i] fetches a byte s[i] fetches a code point, O(1)

len() counts code points, not bytes, a common trap for those used to C:

s = "café😀"
print(len(s))            # 5 code points
print(len(s.encode()))   # 9 UTF-8 bytes: c/a/f are 1 each + é (2) + 😀 (4)

Don’t conflate two different things. .encode() produces a variable-length UTF-8 encoding (each character takes 1–4 bytes by code point), which has nothing to do with how the string is stored in memory — hence 9 bytes. The “the whole string is promoted to 4 bytes/char” from 4.6.1 refers to the str object’s fixed-width in-memory buffer (here UCS-4, ≈ 5×4 = 20 bytes, as reflected by sys.getsizeof). Same string: stored fixed-width in memory (so s[i] indexing is O(1)), serialized variable-width as UTF-8 by .encode() (compact for storage/transport) — the two byte counts are simply different.

4.6.2 Immutability and Hash Caching

str is immutable; any “modification” produces a new object. Because it’s immutable, a string can cache its hash value (the hash field in the layout above), which makes it an efficient dict key (see “Compound Data Structures”).

s = "hello"
# s[0] = 'H'   # TypeError: 'str' object does not support item assignment
s2 = s.replace("h", "H")   # a new object
print(s is s2)             # False

4.6.3 String Interning

CPython interns short string literals that look like identifiers (into a global pool) so that is compares quickly and memory is saved:

a = "hello"
b = "hello"
print(a is b)          # usually True —— literals in the same compilation unit are interned

c = "".join(["h", "e", "l", "l", "o"])
print(a is c)          # usually False —— constructed at runtime, not interned
print(a == c)          # True —— contents equal

The same refrain: compare contents with ==; is only tests whether it’s the same object. You can use sys.intern() to manually intern frequently used strings.

4.7 bytes and bytearray: These Are “C’s char Array”

If you want C’s “a run of raw bytes,” the correspondents are bytes (immutable) and bytearray (mutable), not str.

b = b"hello"          # bytes literal, prefix b
print(b[0])           # 104 —— what you get is an int (byte value), not a character
print(type(b[0]))     # <class 'int'>
  • bytes: an immutable byte sequence, internally just a contiguous char (unsigned char) buffer + length. Closest to C’s const char* + length.
  • bytearray: the mutable version, internally like C++’s std::vector<uint8_t>—it has capacity and supports in-place insertion/deletion:
ba = bytearray(b"hi")
ba.append(33)         # append '!' (0x21)
ba[0] = 72            # modify in place -> 'H'
print(ba)             # bytearray(b'Hi!')

If you want C’s “point into a buffer and read/write a run of bytes with zero copies,” the correspondent is memoryview: it copies no underlying bytes, it’s just a view over an existing buffer, and slicing is likewise zero-copy (analogous to C++’s std::span / string_view):

ba = bytearray(b"hello world")
mv = memoryview(ba)          # zero-copy view, no copy of the underlying bytes
chunk = mv[6:11]             # a slice is still a view, still pointing into the original buffer (no new allocation)
chunk[0] = ord('W')          # modify byte 6 in place through the view
print(ba)                    # bytearray(b'hello World') —— the original object changed
print(bytes(mv[:5]))         # b'hello' —— convert to bytes explicitly when a real copy is needed

When processing large binary blobs (network packets, images, files) this avoids the “slice once, copy once” cost of data[a:b].

Conversion between str and bytes must specify an encoding explicitly, which Python 3 enforces (Python 2’s implicit conversion was a historical minefield):

s = "café"
data = s.encode("utf-8")     # str -> bytes
print(data)                  # b'caf\xc3\xa9'
back = data.decode("utf-8")  # bytes -> str
print(back)                  # café

Compared with C: in C char* is both text and bytes, and encoding is entirely a matter of convention; Python forces you to encode/decode explicitly at the boundaries (files, network, system calls), with str for text and bytes for binary.

Parsing a binary protocol often means assembling several bytes into an integer with a given endianness—in C you’d shift and |; in Python use int.from_bytes / int.to_bytes directly, with the byte order stated explicitly:

data = b'\x00\x00\x01\x00'
print(int.from_bytes(data, "big"))     # 256    —— big-endian: 0x00000100
print(int.from_bytes(data, "little"))  # 65536  —— little-endian: 0x00010000
print((1000).to_bytes(4, "big"))       # b'\x00\x00\x03\xe8' —— pack into 4 big-endian bytes
print((1000).to_bytes(4, "big").hex()) # 000003e8 —— view as hex

byteorder must be spelled out as "big" or "little"; it does not default to the machine’s byte order as in C—which actually removes an implicit cross-platform trap.

4.8 Singletons: None, NotImplemented, Ellipsis

These three are globally unique singleton objects, similar to a global sentinel you might define in C++.

None

Represents “no value / empty,” analogous to C’s NULL, but it’s a genuine object (type NoneType), not a 0 pointer:

x = None
print(x is None)        # the standard way to test for empty, always use is
print(type(None))       # <class 'NoneType'>

None has only one instance for the entire program, so use is None rather than == None—both faster and immune to a custom __eq__ interfering. A function with no return returns None by default (see “Functions and the Call Stack”).

NotImplemented

In operator overloading, when a type doesn’t know how to operate with the other operand, return it (don’t raise an exception), letting Python try the other side’s reflected method:

class Vec:
    def __init__(self, x): self.x = x
    def __add__(self, other):
        if not isinstance(other, Vec):
            return NotImplemented    # hand off to the other operand
        return Vec(self.x + other.x)

Ellipsis (...)

The literal is just three dots .... It has almost no built-in semantics of its own, and is often used as a placeholder, or by NumPy for multidimensional slicing (arr[..., 0]):

def todo(): ...          # a legal function-body placeholder (equivalent to pass)
print(... is Ellipsis)   # True

4.9 Numeric and String Conversion

For explicit construction/conversion, use the type name as a function (it’s actually calling the type object):

int("42")           # 42        —— str -> int
int("ff", 16)       # 255       —— specifying the base
int(3.9)            # 3         —— float truncation (toward zero, not rounding)
float("3.14")       # 3.14
float("inf")        # inf
str(255)            # "255"
bin(255)            # "0b11111111"
hex(255)            # "0xff"
oct(255)            # "0o377"

Formatting (f-string, 3.6+) is the most common way to convert numbers to strings:

n = 255
print(f"{n:d} {n:b} {n:x} {n:08b}")   # 255 11111111 ff 11111111
pi = 3.14159
print(f"{pi:.2f}")                    # 3.14
print(f"{1234567:,}")                 # 1,234,567 —— thousands separator

A failed conversion raises ValueError, rather than silently returning 0 like C’s atoi:

try:
    int("abc")
except ValueError as e:
    print("conversion failed:", e)

This “raise an exception on error” orientation is a major style difference from C’s return-error-codes approach; see see “Error Handling and Resource Management”.

Summary

  • All types are heap objects sharing the PyObject header (reference count + type pointer, the type pointer acting like a vptr).
  • int is a variable-length bignum (30-bit digit array + sign), no overflow; [-5,256] has cached singletons.
  • bool is a subclass of int; True/False are singletons of value 1/0.
  • float/complex are boxed C doubles (one/two), and operations must allocate new objects.
  • str is immutable Unicode, stored compactly in 1/2/4 bytes chosen by the largest code point (PEP 393), and len counts code points; bytes/bytearray are what correspond to C’s byte array.
  • None/NotImplemented/Ellipsis are three whole-program singletons; test equality with is.
  • Convert by calling the type name; failure raises ValueError rather than being silent.
