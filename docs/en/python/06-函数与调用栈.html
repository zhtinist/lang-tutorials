Functions and the Call Stack

This is the chapter that most requires “rewiring your brain.” In C/C++, argument passing has only two physical mechanisms: by value (copy into the callee’s stack frame) and by pointer (copy an address). C++ adds references (&, aliasing the same block of memory). The “value vs. pointer vs. reference” model in your head is crystal clear—but it cannot be applied directly to Python. Python has only one way to pass arguments, and we must say precisely what it is.

def Creates an Object

First, correct an intuition: in C a function is the address of a chunk of code (a function pointer pointing into the text segment), fixed at compile time. In Python, def is a statement executed at runtime; when it runs it creates a function object on the heap, then binds that object to a name:

def add(a, b):
    return a + b

print(type(add))          # <class 'function'>
print(add.__code__)       # the compiled code object (where the bytecode lives)
print(add.__name__)       # 'add'
add_alias = add           # just one more name pointing at the same function object

A function object holds: __code__ (the immutable bytecode and metadata), __defaults__ (default argument values), __globals__ (the global namespace it can see), and __closure__ (variables captured by a closure). Functions are “first-class citizens”—they can be assigned, passed as arguments, returned, and stuffed into containers, because they are just ordinary objects.

The Truth About Argument Passing: Call by Object Reference (Call by Sharing)

Python’s argument-passing mechanism is called call by object reference, also known as call by sharing. In one sentence: the argument and the parameter point to the same object on the heap; what the passing process copies is the reference (a pointer), not the object itself.

This sounds like C’s “pass a pointer,” but there’s a crucial difference: in C when you pass a pointer, reassigning the parameter p = ... inside the function doesn’t affect the caller, but you can *p = ... to change the contents; in Python you have no “pointer to a pointer” at all, so you cannot make the caller’s name point elsewhere. This is clearest along two dimensions:

Dimension Python’s behavior
Can it change what the caller’s variable points to? No. Reassigning a parameter only rebinds the local name
Can it change the object’s contents? Yes, but only if the object is mutable (list/dict/set/custom objects)

Dimension One: Reassigning a Parameter — Never Affects the Caller

def rebind(x):
    # x initially points to the same object as the caller's argument;
    # the line below makes the local name x point to a new object, unbeknownst to the caller
    x = 999
    print("inside:", x)

n = 1
rebind(n)
print("outside:", n)     # 1 —— unchanged

Memory diagram (at the moment of the call, n and x point to the same int object; after assignment, x points to a new object):

At call time (n and x are two names for one object):   After x = 999:
  n ─┐                                                  n ───► [int 1]      ← n still points to the old object
     ├──► [int 1]                                       x ───► [int 999]    ← x now points to a new object
  x ─┘

This is exactly why Python cannot write C’s void swap(int *a, int *b) style “change the caller’s variables through parameters” function. To swap, you can only return new values: a, b = b, a, received by the caller.

Dimension Two: Mutating a Mutable Object In Place — The Caller Sees It

def mutate(lst):
    # Doesn't rebind lst; calls a method to modify the object it points to
    lst.append(4)      # in-place modification, produces no new list
    print("inside:", lst)

data = [1, 2, 3]
mutate(data)
print("outside:", data)   # [1, 2, 3, 4] —— it changed!

Memory diagram (only one list object, with both names pointing at it):

  data --\
          >---> [list: 1,2,3] --append--> [list: 1,2,3,4]
  lst  --/

This has the same effect as changing contents via *p after passing a pointer in C. The only difference is that in Python “whether you can change it” depends on whether the object itself is mutable, not on how you declared the parameter.

Mutable vs. Immutable Arguments Side by Side

The same “modify the argument” code behaves oppositely for mutable and immutable arguments:

def try_change(obj):
    if isinstance(obj, list):
        obj += [99]        # list's += is an in-place extend (changes contents)
    else:
        obj += 99          # int's += creates a new object, then rebinds the local obj

a = [1, 2]
try_change(a)
print(a)          # [1, 2, 99] —— the mutable object was changed

b = 10
try_change(b)
print(b)          # 10 —— an immutable object; the caller is unaffected

Here’s the trap: obj += [99] and obj += 99 both look like +=, but for a list it’s an in-place modification (equivalent to obj.extend; __iadd__ returns self), and for an int it’s “create new + rebind” (int has no in-place addition, and when __iadd__ is absent it degrades to obj = obj + 99). Mutability decides everything.

A few more examples to nail it down

Example 1: both dimensions in a single call — the same function reassigns a (invisible) and mutates b in place (visible):

def f(a, b):
    a = a + [100]     # for a: build a new list, rebind local name a → caller can't see it
    b.append(100)     # for b: call a method to modify the object in place → caller sees it
    print("inside a =", a, " b =", b)   # [1, 100]  [1, 100]

x = [1]
y = [1]
f(x, y)
print("outside x =", x)   # [1]        —— unchanged (only a local binding changed)
print("outside y =", y)   # [1, 100]   —— changed (the object's contents changed)

Example 2: custom objects work the same — changing an attribute = changing contents (visible); rebinding the parameter = only a local name (invisible):

class Account:
    def __init__(self, bal):
        self.balance = bal

def touch(acc):
    acc.balance += 50     # changing the object's attribute is changing contents → visible
    acc = Account(0)      # rebinding local name acc to a new object → caller is unaware
    print("inside acc.balance =", acc.balance)   # 0

a = Account(100)
touch(a)
print("outside a.balance =", a.balance)   # 150 —— the attribute was changed; a still points to the original object

Example 3: so how can I change what the caller’s variable points to? Two routes:

# Route 1: return the new value and let the caller rebind it (the most Pythonic way)
def older(age):
    return age + 1

age = 30
age = older(age)     # the caller takes the return value and rebinds it
print(age)           # 31

# Route 2: wrap it in a mutable container and change the element (like C's int**: passing a "pointer to a pointer")
def set_value(box, val):
    box[0] = val     # this changes container contents, so it's visible
holder = [None]
set_value(holder, 42)
print(holder[0])     # 42

Example 4: the most classic confusion — lst = lst + [x] vs lst += [x] differ by a hair, with opposite results:

def rebind(lst):
    lst = lst + [9]   # first compute a NEW list, then rebind local name lst → caller can't see it
def inplace(lst):
    lst += [9]        # equivalent to lst.extend([9]), modifies the same object in place → caller sees it

m = [1]
rebind(m)
print(m)              # [1]      —— unchanged
n = [1]
inplace(n)
print(n)              # [1, 9]   —— changed

lst + [9] “makes a new object,” while lst += [9] “changes in place” — a one-character difference, a world apart. This is the other face of the += trap above.

Mnemonic: What’s passed is always a reference; what’s copied is always the reference itself (a pointer), never the object. Rebinding doesn’t propagate; changing contents shows through if it’s mutable.

Kinds of Parameters: Positional, Keyword, Default, *args, **kwargs

Python’s parameter system is far more flexible than C’s. The complete form of the various symbols in a function signature:

def f(pos1, pos2, /, both, *args, kw_only, **kwargs):
    #   ^^^^^^^^^^   /   ^^^^  ^^^^^  ^^^^^^^   ^^^^^^^^
    ...
  • Before /: positional-only parameters (the caller cannot pass pos1=...), 3.8+;
  • Between / and *: positional or keyword (the most common);
  • *args: collects extra positional arguments into a tuple;
  • After * (or after *args): keyword-only parameters, which must be written as kw_only=...;
  • **kwargs: collects extra keyword arguments into a dict.
def demo(a, b, *args, sep="-", **kwargs):
    print(f"a={a} b={b}")
    print("args =", args)        # tuple
    print("kwargs =", kwargs)    # dict
    print("sep =", sep)

demo(1, 2, 3, 4, sep="|", x=10, y=20)
# a=1 b=2
# args = (3, 4)
# kwargs = {'x': 10, 'y': 20}
# sep = |

Under the hood, *args allocates a new tuple to hold those extra arguments, and **kwargs allocates a new dict. The * / ** at the call site are the reverse operation (unpacking):

nums = [1, 2, 3]
print(*nums)             # equivalent to print(1, 2, 3), unpacks the list into positional arguments
opts = {"sep": "/"}
print("a", "b", **opts)  # unpacks the dict into keyword arguments -> sep="/"

Default Arguments: Evaluated Once When def Runs (a Classic Trap)

C++’s default arguments are evaluated on every call. Python is the opposite: the default value is evaluated once, when the def statement runs, and the result is stored in the function object’s __defaults__, shared by every subsequent call. This is fine for immutable defaults, but for mutable defaults it’s the famous pitfall:

def bad(item, bucket=[]):     # this [] is created only once, at definition time!
    bucket.append(item)
    return bucket

print(bad(1))    # [1]
print(bad(2))    # [1, 2] —— surprise! the same list is reused over and over
print(bad(3))    # [1, 2, 3]

print(bad.__defaults__)   # ([1, 2, 3],) —— the default value lives here and got mutated

Reason: the default value of bucket is that one list object created at definition time, stored in __defaults__. Each time you omit bucket, the parameter binds to this same object, and append keeps adding to it.

The correct way is to use a None sentinel and create a new one inside the function body:

def good(item, bucket=None):
    if bucket is None:
        bucket = []           # a fresh one on every call, mutually independent
    bucket.append(item)
    return bucket

print(good(1))   # [1]
print(good(2))   # [2]

Likewise, don’t use things like time.time(), {}, or [] as defaults expecting them to be “recomputed on each call”—they’re computed only once.

Returning Multiple Values: It’s Actually a Tuple

C returns multiple values via output parameters (pointers) or by wrapping them in a struct. Python appears to “return multiple values,” but under the hood it packs them into one tuple, then unpacks it:

def divmod2(a, b):
    return a // b, a % b      # a comma builds a tuple, equivalent to return (a//b, a%b)

q, r = divmod2(17, 5)         # the right side is a tuple; the left side is sequence unpacking
print(q, r)                   # 3 2

t = divmod2(17, 5)
print(type(t), t)             # <class 'tuple'> (3, 2)

There is no true “multiple return values,” only “return one tuple + unpacking assignment.” So its cost is allocating a small tuple (CPython has a caching optimization for small tuples).

Closures and Cell Variables

When an inner function references a local variable of an outer function, a closure forms. Here Python again parts ways with C: C’s local variables live on the stack frame and become invalid when the function returns, so you cannot safely return a pointer to one. In Python, a variable captured by a closure is placed in a cell object on the heap, and the variable stays alive even after the outer function has returned:

def make_counter():
    count = 0                 # referenced by the inner function, promoted to a cell variable
    def inc():
        nonlocal count        # declares: modify the outer count, don't create a new local
        count += 1
        return count
    return inc                # after the outer returns, count is still alive

c = make_counter()
print(c(), c(), c())          # 1 2 3

print(c.__closure__)          # (<cell at ...: int object at ...>,)
print(c.__closure__[0].cell_contents)   # 3 —— the cell holds the current value

Memory diagram: count is not on the stack but in a cell, and the closure inc holds a reference to that cell through __closure__:

   After make_counter returns, its stack frame is destroyed, but:

   c (function inc) --__closure__--> ( cell )
                                       |
                                       v
                                    [int 3]   <- on the heap, still alive

nonlocal declares “I want to write a variable in an enclosing scope”; global declares “I want to write a module-level global variable.” Without either, assigning directly makes Python treat it as a new local variable (because assignment creates a local binding by default), raising UnboundLocalError—a common beginner mistake:

x = 0
def wrong():
    x += 1          # UnboundLocalError: x is treated as a local but isn't bound yet when read
def right():
    global x
    x += 1          # OK

A closure captures the cell (the variable), not the value at that moment—this is the classic trap when building closures in a loop. The three lambdas below share the same i cell; by the time they’re actually called the loop has long finished with i stuck at 2, so they all return 2:

funcs = [lambda: i for i in range(3)]
print([f() for f in funcs])   # [2, 2, 2] —— all three lambdas share one i cell, reading its final value

# Fix: use a default argument to "snapshot" the current i into each lambda's own i at definition time
funcs = [lambda i=i: i for i in range(3)]
print([f() for f in funcs])   # [0, 1, 2] —— the default is fixed when def is evaluated (see the previous section on defaults)

Contrast with C++: a C++ lambda capturing [i] by value copies the current i immediately, whereas [&i] by reference shares it as Python does. A Python closure always “shares the cell by reference”; there is no by-value capture syntax—to snapshot, you rely on the default-argument trick.

The Call Stack: frame Objects

A C stack frame is a chunk of contiguous memory with a compile-time-fixed layout, popped when the function returns. CPython’s call stack is made of frame objects; each call creates a frame containing a local-variable table, a link to the previous frame, and so on—it’s a heap object (since CPython 3.11 frames have an inlining optimization, but semantically you can still access them as objects), and it can even be reflected upon at runtime:

import sys

def inner():
    frame = sys._getframe()           # the current frame
    print("current function:", frame.f_code.co_name)      # 'inner'
    print("locals:", frame.f_locals)             # {'frame': ...}
    print("caller:", frame.f_back.f_code.co_name)  # 'outer' —— the previous frame

def outer():
    inner()

outer()
frame attribute Meaning (compared to C)
f_locals Dict of local variables (in C, an unnamed chunk of stack memory)
f_globals The global namespace of its module
f_back Points to the caller’s frame (i.e., one level up the call stack)
f_code The code object being executed
f_lineno The current line number

This reflectable frame structure is exactly what lets tracebacks, debuggers, and logging print a complete call chain. The cost is being heavier than a C stack frame.

Recursion Depth Limit

In C, recursing too deep is a stack overflow (a segfault). To prevent the C-level stack from actually overflowing, CPython sets a soft limit; exceeding it raises RecursionError instead of crashing:

import sys
print(sys.getrecursionlimit())    # usually 1000

def rec(n):
    return 1 if n == 0 else rec(n - 1) + 1

# rec(100000) raises RecursionError instead of segfaulting the process
sys.setrecursionlimit(3000)       # can be raised, but setting it too high may still blow the C stack

Python has no tail-call optimization; for deep recursion, rewrite it as iteration or use an explicit stack.

First-Class Functions and How Decorators Work

Because functions are objects, they can be received, wrapped, and returned by other functions. A decorator is syntactic sugar for exactly this ability: @deco on def foo is equivalent to foo = deco(foo)—replacing the original name with deco’s return value (typically a wrapper function).

import functools, time

def timer(func):
    @functools.wraps(func)        # copy func's __name__/__doc__ onto the wrapper
    def wrapper(*args, **kwargs):  # use *args/**kwargs to pass through any arguments
        t0 = time.perf_counter()
        result = func(*args, **kwargs)   # call the wrapped original function
        print(f"{func.__name__} took {time.perf_counter()-t0:.6f}s")
        return result
    return wrapper                 # return the wrapper, replacing the original name

@timer
def work(n):
    return sum(range(n))

# The above is fully equivalent to: work = timer(work)
work(1_000_000)
print(work.__name__)   # 'work' —— thanks to functools.wraps, otherwise it would be 'wrapper'

Seen clearly under the hood: when @timer runs, it immediately calls timer(work), obtaining the new function object wrapper, then binds it to the name work. From then on, calling work(...) actually calls wrapper, which internally forwards to the original function (captured via the closure __closure__ as func). That’s “wrapping”—no magic, all first-class functions + closures.

functools.wraps matters: without it, work.__name__ becomes 'wrapper', and tracebacks and docs point at the wrapper function, causing confusion during debugging.

A decorator that takes arguments adds one more layer: in @repeat(3), repeat(3) runs first and returns the actual decorator, which then decorates the function. So it’s three nested layers—the outermost takes the decorator argument, the middle takes the function being decorated, and the innermost takes the call arguments:

import functools

def repeat(n):                       # layer 1: takes the decorator argument n
    def deco(func):                  # layer 2: takes the function being decorated (this is the real decorator)
        @functools.wraps(func)
        def wrapper(*args, **kwargs):  # layer 3: takes the per-call arguments
            for _ in range(n):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return deco

@repeat(3)                           # equivalent to greet = repeat(3)(greet)
def greet(name):
    print(f"Hi {name}")
    return name

greet("Ann")
# Hi Ann
# Hi Ann
# Hi Ann

See the evaluation order clearly: the @repeat(3) line first calls repeat(3) to get deco, then runs deco(greet) to get wrapper, and finally binds wrapper to greet. The argument n stays remembered by wrapper through the closure.

Summary

  • def creates a function object at runtime; functions are first-class citizens.
  • Argument passing is call by object reference / call by sharing: it copies a reference (pointer), not the object. Rebinding a parameter doesn’t propagate; mutating an object’s contents in place propagates only for mutable objects. This two-dimensional model is the core of understanding Python’s argument passing.
  • Default arguments are evaluated once when def runs; a mutable default is reused across calls—avoid it with a None sentinel.
  • Returning “multiple values” is really returning a tuple and unpacking it; closure variables live in a cell on the heap, held via __closure__, with nonlocal/global controlling writes to enclosing/global scopes.
  • The call stack is made of reflectable frame objects (f_locals/f_back), has a recursion soft limit, and no tail-call optimization; a decorator = syntactic sugar over first-class functions + closures.
