Error Handling and Resource Management

This chapter has two main threads: the first is the exception mechanism (how to write try/except, and how exceptions are objects too); the second is CPython’s memory management (reference counting + generational GC + the pymalloc memory pool). The first half covers try/except/finally, custom exceptions, and how to write and use the with statement — no low-level knowledge required to follow along; the second half covers the low-level protocols behind exception objects and context managers, plus the more low-level memory-management thread, comparing it point by point with manual malloc/free, RAII, and smart pointers.

0. Get Familiar with Usage First: A Quick Tour of try / except / finally and with

Use try/except to catch code that might fail; the code in finally runs whether or not an error occurred:

try:
    x = int("abc")           # this line raises ValueError
except ValueError as e:
    print("conversion failed:", e)   # conversion failed: invalid literal for int() with base 10: 'abc'
finally:
    print("cleanup done")            # runs whether or not an error occurred

The else block runs only when the try block does not raise an exception:

def divide(a, b):
    try:
        result = a / b
    except ZeroDivisionError:
        print("cannot divide by zero")
        return None
    else:
        print("computed successfully")
        return result

print(divide(10, 2))    # computed successfully \n 5.0
print(divide(10, 0))    # cannot divide by zero \n None

Define your own exception type and raise it explicitly with raise:

class MyError(Exception):        # custom exception: just inherit from Exception
    pass

def check(x):
    if x < 0:
        raise MyError(f"x cannot be negative: {x}")
    return x

try:
    check(-1)
except MyError as e:
    print("caught custom exception:", e)   # caught custom exception: x cannot be negative: -1

For resources you need to “close once you’re done with them” — open files, network connections — use the with statement: leaving the with block closes it automatically, no need to call close() by hand:

with open("data.txt", "w") as f:
    f.write("hello")
# by the time we leave the with block, the file has already been closed
with open("data.txt") as f:
    print(f.read())       # hello

These four patterns are the ones you’ll reach for most often in everyday error handling and resource management. Below we go into their details and the underlying mechanics — usage first, internals second.

1. Exceptions Are Objects

How to Use It

When writing except, keep one practical rule in mind: prefer except Exception over a bare except: or except BaseException — otherwise you’ll also catch signals that business code should never swallow, such as the user pressing Ctrl-C (KeyboardInterrupt) or sys.exit() (SystemExit):

print(issubclass(KeyboardInterrupt, Exception))     # False
print(issubclass(KeyboardInterrupt, BaseException)) # True

Internals vs. C

Python’s exceptions are not C error codes, nor are they C++’s “throw anything” (throw 42). Every exception is an object that inherits from BaseException, carrying information (args) and a traceback. The hierarchy looks roughly like this:

BaseException            ← Top level; except Exception won't catch these direct subclasses
├── SystemExit           ← Raised by sys.exit(); don't casually except it away
├── KeyboardInterrupt    ← Ctrl-C; don't casually except it away
├── GeneratorExit
└── Exception            ← Base class for "ordinary" errors; your own exceptions should inherit from it
    ├── ValueError / TypeError / KeyError / IndexError / ...
    ├── ArithmeticError → ZeroDivisionError
    ├── OSError → FileNotFoundError / PermissionError / ...
    └── RuntimeError / StopIteration / ...

Key takeaway: except Exception won’t catch KeyboardInterrupt or SystemExit — because they hang directly under BaseException, deliberately keeping “the user wants to interrupt the program” from being accidentally swallowed by business code.

Compared with C++: C++ can throw any type (throw 42; is legal); Python can only raise instances of BaseException. The type system is unified, and except matches via isinstance.

2. try / except / else / finally

How to Use It

A single except can also catch several types at once (written as a tuple); multiple except clauses are matched in order:

def parse(s, data):
    try:
        x = int(s)               # may raise ValueError
        y = data[x]               # may raise IndexError / KeyError
    except (ValueError, KeyError) as e:   # one except catches multiple types (a tuple)
        print("bad input:", e)   # e is the exception object; e.args is the tuple of arguments
    except IndexError:
        print("out of range")
    else:
        print("runs only if no error:", y)   # runs when try raises nothing
    finally:
        print("always runs")     # cleanup, always runs—even if try does return/break

parse("abc", [1, 2, 3])   # bad input: invalid literal ... \n always runs
parse("5", [1, 2, 3])     # out of range \n always runs
parse("1", [1, 2, 3])     # runs only if no error: 2 \n always runs

Internals vs. C

  • except handles the first branch that matches via isinstance, scanning top to bottom; it stops at the first match. So subclass exceptions must be listed before their parent classes.
  • else runs only when the try block raises no exception (separating the “normal path” from “code that may raise”).
  • finally always runs, commonly used to release resources. Note: if finally contains a return, it will override the return value from try/except (unlike C++ destructors, this is explicit control flow and easy to trip over).

Under the hood: CPython’s exception handling relies on an exception table in the bytecode (zero-cost try since 3.11) — when no exception occurs, try incurs almost no extra overhead; only an actual raise triggers a table lookup and jump. This mirrors C++’s “zero-cost EH” (no overhead on the normal path).

3. raise, Custom Exceptions, and Exception Chaining

How to Use It

Using raise ... from ... lets you preserve the original cause when wrapping an exception, so the full chain is visible when you’re debugging:

class ConfigError(Exception):        # custom exception: just inherit from Exception
    pass

def load(path):
    try:
        return open(path).read()
    except OSError as e:
        raise ConfigError(f"failed to read config: {path}") from e   # exception chain: preserve the cause

try:
    load("/no/such/file")
except ConfigError as e:
    print("caught:", e)            # caught: failed to read config: /no/such/file
    print("cause:", e.__cause__)   # cause: [Errno 2] No such file or directory: '/no/such/file'

Internals vs. C

  • raise ValueError("msg") raises; a bare raise (inside an except) re-raises the current exception, preserving the original traceback.
  • raise NewError from cause explicitly sets __cause__; when printed, you see The above exception was the direct cause of.... Without from, re-raising inside an except automatically records __context__ (an implicit chain), printing During handling of the above exception....
  • To cut the chain: raise NewError from None.

Compared with C++: this is similar to “catch a low-level exception, wrap it in a domain exception, and re-raise,” but Python’s from makes the cause chain visible in the traceback (stored on the __cause__/__context__ attributes), which is friendlier for debugging.

4. Context Managers: with vs. RAII

How to Use It

“Get Familiar with Usage First” already showed the built-in open() paired with with closing a file automatically. You can also implement your own context manager just by defining __enter__ and __exit__:

import time

class Timer:
    def __enter__(self):
        self.t0 = time.perf_counter()
        return self
    def __exit__(self, exc_type, exc_val, exc_tb):
        elapsed = time.perf_counter() - self.t0
        print(f"finished within {elapsed:.4f}s")
        return False            # return False: don't swallow the exception, let it propagate as usual

with Timer():
    total = sum(range(1000))
print(total)   # 499500

A more concise approach is to use contextlib.contextmanager to turn a generator function into a context manager:

from contextlib import contextmanager

@contextmanager
def opened(path, mode="r"):
    f = open(path, mode)
    try:
        yield f                 # before yield = __enter__, after = __exit__
    finally:
        f.close()

with opened("data.txt") as f:
    print(f.read())    # hello

Internals vs. C

C++ does RAII with destructors + stack objects — leaving scope releases automatically. Python has no deterministic destruction (see __del__ below); it uses the with statement + context managers to achieve the same “deterministic release”:

with open("data.txt") as f:      # entering: calls f.__enter__()
    process(f)
# leaving the with block (normally or via exception): calls f.__exit__(exc_type, exc_val, exc_tb), closing the file

__exit__ receives the exception information (all three are None on normal exit); returning a truthy value can swallow the exception — Timer.__exit__ above returning False is an explicit way of saying “don’t swallow it.”

Compared with RAII: with is RAII with an explicitly delimited scope — more verbose than C++’s “destruct at the closing brace,” but the release timing is equally deterministic (not dependent on GC). This is the preferred way to manage files, locks, and connections in Python.

From here on, we move into the chapter’s second thread — how CPython manages memory automatically. This part introduces no new syntax (you already know del); just compare the mechanics directly against the C malloc/free and C++ shared_ptr you’re already familiar with.

5. Reference Counting: The Primary Reclamation Mechanism

The workhorse of CPython memory management is reference counting: every object header contains an ob_refcnt; a new reference to it does +1, a disappearing reference (del, rebinding a variable, leaving scope, removal from a container) does -1; dropping to 0 releases it immediately — deterministic and instant, like C++’s shared_ptr, but because the GIL protects it, incrementing/decrementing the count requires no atomic operations (one of the reasons the GIL exists; see Concurrency Models).

import sys
a = object()
print(sys.getrefcount(a))   # e.g. 2: note that the argument to getrefcount is itself a temporary reference, so it's 1 higher than intuition suggests
b = a                       # another reference
print(sys.getrefcount(a))   # 3
del b                       # reference -1

Compared with C: in C, you call free manually and control the timing; CPython automatically calls free for you “the moment the last reference disappears,” deterministically and immediately — closer to the C/C++ mental model than the tracing GC of Java/Go (whose reclamation timing is nondeterministic).

Verified deterministic, immediate reclamation: the instant the refcount hits zero, __del__ runs — the timing is fully predictable:

class Res:
    def __init__(self, name): self.name = name
    def __del__(self): print(f"{self.name} released")

a = Res("a")
b = Res("b")
print("before del a")
del a               # a's last reference disappears -> count hits 0 -> reclaimed at once
print("after del a")
# before del a
# a released        <- right at the del, not at script end
# after del a
# b released        <- when the script ends and b leaves scope

There’s only one thing reference counting can’t handle: reference cycles.

6. Generational GC, __del__, and weakref

a references b, and b references a; even when nothing external uses them, neither count can drop to 0 — reference counting misses this cycle. CPython therefore adds a tracing generational garbage collector (the gc module) on top, dedicated to reclaiming cyclic garbage:

import gc
print(gc.get_threshold())   # (2000, 10, 10): a generation-0 alloc-minus-free difference of 2000 triggers a collection (thresholds vary by version)
gc.collect()                # manually run a full collection; returns the number of unreachable objects reclaimed
  • Three generations (0/1/2): new objects start in generation 0; surviving one collection “promotes” them to an older generation; older generations are scanned less frequently. Based on the generational hypothesis that “most objects die young” (the same idea as Java/Go’s generational approach).
  • Only container objects that could form cycles (list/dict/set/instances, etc.) are tracked by the gc; atomic objects like int and str that can’t form cycles are not tracked, relying solely on reference counting.

Verified: pure reference cycles can’t be reclaimed by reference counting — they need the gc. Below, a<->b reference each other; after deleting the external references the objects stay alive until gc.collect() reclaims them:

import gc
class Node:
    def __del__(self): print("Node freed")

gc.collect()          # clear interpreter startup leftovers first, for a clean count
gc.disable()          # turn off automatic collection to observe clearly
a = Node(); b = Node()
a.ref = b; b.ref = a  # create a reference cycle
del a; del b          # external references gone, but they still hold each other -> count never hits 0
print("after del, still alive")   # both objects are still not reclaimed here
print("collected:", gc.collect()) # manually run tracing collection on the cycle
# after del, still alive
# Node freed
# Node freed
# collected: 2        <- the gc reclaimed 2 unreachable objects

__del__ (the destructor hook) is unreliable; don’t use it for RAII:

class Res:
    def __del__(self):          # called when the object is reclaimed
        print("released")       # but "when" this happens is not guaranteed
  • The trigger timing depends on when the reference count reaches zero / when the gc runs — nondeterministic; it may not even be called at interpreter exit.
  • Since Python 3.4 (PEP 442), reference cycles containing __del__ can also be reclaimed by the gc (calling the destructors correctly), no longer getting stuck in gc.garbage as in older versions. Even so, resource release should still use with — don’t gamble on __del__.

weakref — a reference that doesn’t increment the reference count (the counterpart to C++’s weak_ptr):

import weakref
class C: pass
c = C()
r = weakref.ref(c)     # r does not bump c's count by +1
print(r() is c)        # True: the target is still alive, so r() returns it
del c                  # the only strong reference disappears, c is reclaimed
print(r())             # None: the weak reference is automatically invalidated

Uses: caching (you don’t want the cache to keep objects from being reclaimed), observers, and breaking reference cycles.

7. pymalloc: CPython’s Memory Pool

Python programs frantically create and destroy small objects (every integer, every string is a heap object). If each one called C’s malloc/free, the overhead and fragmentation would be intolerable. CPython therefore ships with a layered memory allocator, pymalloc, dedicated to small objects ≤ 512 bytes:

Arena  ── ~1 MiB chunk, requested directly from the OS via mmap
  │  split into several
Pool   ── one memory page in size (commonly 4 KB on x86; 16 KB on Apple Silicon); a pool serves only one size class
  │  split into equal-length
Block  ── equal-length small blocks aligned to 16 bytes: 16, 32, 48, …, 512 bytes, about 32 size classes total
             a small-object allocation = grab a block from the corresponding pool's free list

How it works:

  • When allocating a small object, it’s binned by size into a “size class,” and an O(1) block is grabbed from the corresponding pool’s free-block list; on release, the block returns to the free list rather than being returned to the OS immediately.
  • Small-object allocation/release therefore almost never touches the system malloc, making it extremely fast; objects of the same size class cluster together, reducing fragmentation.
  • Objects > 512 bytes bypass pymalloc and go straight to the system malloc.
  • Only a completely empty pool/arena can be returned to the OS. So you’ll see a Python process’s memory “go up and not come back down” — much of it is held and reused by memory pools, not a leak.
  • The constants above (arena≈1 MiB, pool = system page size, 16-byte alignment, ~32 size classes) can be checked live with python3 -c "import sys; sys._debugmallocstats()"; values vary slightly by version/platform.

Compared with C: this is like CPython having a built-in object-pool allocator optimized for “massive numbers of small objects,” with a hierarchy (arena→pool→block) resembling the slab/pool allocators hand-written in many game/high-performance C++ projects.

There are also free lists: small integers, small tuples, common-sized lists/frames, etc. — once destroyed, their memory blocks are hung on a dedicated free list waiting for reuse, skipping even the pymalloc step.

8. The Big Picture

Mechanism Role C/C++ Counterpart
Reference counting Primary, instant reclamation, deterministic shared_ptr
Generational GC Backstop for reclaiming reference cycles Tracing GC (only in Java/Go)
weakref Reference that doesn’t bump the count weak_ptr
with / context managers Deterministic resource release RAII / destructors
__del__ Destructor hook (unreliable, use with caution) Destructor (but timing is nondeterministic)
pymalloc Small-object memory pool slab/pool allocator
free list Memory reuse for common objects object pool

In one sentence: Python’s memory management is a combination of “reference counting (like shared_ptr, giving you determinism) + generational GC (backstop for cycles) + multi-layer memory pools (giving you speed)”; use with to release resources, and don’t gamble on __del__.

Summary

  • Usage level: try/except/else/finally respectively handle “code that might error,” “the matched error type,” “code that runs only when nothing went wrong,” and “cleanup code that always runs”; except handles the first matching type scanning top to bottom; custom exceptions inherit from Exception; raise ... from ... wraps an exception while preserving the cause; for resource release, prefer with (whether the built-in kind or one you implement yourself via __enter__/__exit__ or @contextmanager).
  • Internals level: exceptions are instances of BaseException, implemented via a zero-cost exception table since 3.11; CPython’s memory management relies on reference counting (immediate and deterministic, like shared_ptr) as a backstop plus generational GC (which specifically handles reference cycles), with __del__ firing at an unreliable time; pymalloc uses a three-tier arena→pool→block structure to speed up small objects.
