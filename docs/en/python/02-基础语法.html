Basic Syntax: An Adaptation Guide for Those Coming from the Land of Braces

You can already follow the execution model in Chapter 01 and know that every value is a heap object. This chapter deals with surface syntax: the things that strike you as “off” at first glance—no braces, no semicolons, no type declarations, no main. Behind these surface differences usually lie mechanisms that C/C++ either omits or rewrites.

1. Indentation Is the Block: Scope Enforced at the Syntax Level

C/C++ uses {} to delimit blocks; indentation is purely cosmetic and the compiler ignores it:

if (x > 0) {
    foo();
    bar();
}

Python uses indentation itself to delimit blocks—no braces—and statements end at a newline (no semicolons):

if x > 0:
    foo()
    bar()
# dedenting exits the block

Rules: - A colon : introduces a block, and the block body must be indented (4 spaces by convention). - All statements within the same block must be indented identically, or you get an IndentationError. - Do not mix tabs and spaces—CPython will error out directly. Configure your editor to convert tabs to spaces.

Under-the-hood view: during lexical analysis, indentation is turned into INDENT / DEDENT tokens (recall the tokenize stage in Chapter 01), and their role is exactly that of C’s { and }. So indentation isn’t a “style suggestion”; it’s a syntactic token, just like a missing brace causing a compile error.

One important semantic difference: C’s {} introduces a new lexical scope (variables declared in the block cease to exist on exit). Python’s indented blocks do not introduce a new scope—names bound inside an if/for/while block remain visible after the block:

if True:
    y = 10
print(y)   # 10, legal! y wasn't "destroyed on block exit"

In Python, the only things that truly introduce a new scope are functions, classes, modules, and comprehensions, not every indented block. Scope details are in Chapter 06.

2. Dynamic Typing, Strong Typing, No Declarations Needed

Three orthogonal concepts—don’t conflate them:

  • No declaration needed: variables aren’t declared with a type in advance; they “come into being” on first assignment. x = 3 is enough; there’s no int x.
  • Dynamic typing: the type belongs to the object, not the variable. The same name can point first to an int, then to a str. Type checking happens at runtime (recall from Chapter 01: ob_type hangs on the object).
  • Strong typing: no implicit, reckless conversions. 1 + "2" raises TypeError outright, unlike weakly-typed languages that hand you a "12" or 3.
x = 42          # x now points to an int object
x = "hello"     # x now points to a str object; perfectly legal
x = [1, 2, 3]   # now points to a list object

# Strong typing: no implicit conversion
# 1 + "2"       # TypeError: unsupported operand type(s)
1 + int("2")    # 3, works only with an explicit conversion

Compared to C++: C++ is static + strong; once int x is fixed, x = "hi" won’t compile. Python defers type checking to runtime, buying flexibility at the cost of many errors that can’t be caught at compile time (remediable with type annotations + mypy; see Chapter 08).

Note what “reassigning a variable” is under the hood: x = "hello" does not overwrite the int object’s memory with a string; it makes the name x point to a new heap object, and the old int object’s reference count drops by one and may be reclaimed. This is a central theme of the whole book, and Chapter 03 explains it thoroughly with memory diagrams.

3. No main: Modules Execute Top to Bottom

A C/C++ program’s entry point is main. Python has no mandatory entry—when a module is loaded, its top-level statements execute in order, top to bottom. A function definition def is itself a statement that “creates a function object when executed,” not a compile-time declaration.

So how do you distinguish “run directly as a script” from “imported by someone else”? Via this idiom:

def main():
    ...

if __name__ == "__main__":   # __name__ == "__main__" when run directly
    main()

When imported, the module’s __name__ is the module name, the condition is false, and main() doesn’t run—so importing it gives you only the function definitions without triggering side effects. This is the standard “library/script dual-use” pattern, functionally corresponding to C’s int main() entry point, but mechanically entirely different: it’s just an ordinary runtime if.

4. print / input and Comments

print("hello", "world")          # space-separated by default, newline at end -> hello world
print("no newline", end="")      # custom ending
print(a, b, sep=", ")            # custom separator
name = input("name? ")           # reads a line, strips the newline; return value is always str

# Comments
# single-line comments use the hash sign
"""Triple-quoted strings are often used as multi-line comments,
but they're really string-literal objects."""

print is a function (since Python 3), not a statement. input returns str; for a number you must do int(input()) yourself—there’s no scanf("%d")-style formatted reading.

5. Operators: A Few You Haven’t Seen

The semantics of most operators (+ - * / % == < > && ||) are familiar, but Python uses the keywords and / or / not in place of && || ! (& | ^ ~ << >> are still bitwise operators). A few unfamiliar ones:

Operator Meaning C/C++ analogy
/ True division, always returns float 5/2 == 2.5 (not C’s integer 2!)
// Floor division Like C integer division but rounds toward negative infinity
** Power No equivalent; you’d use pow()
@ Matrix multiplication None; defined by the object’s __matmul__, used by NumPy
and/or Logical and/or (short-circuiting) && / \|\|

Be sure to remember the / pitfall:

5 / 2     # 2.5  (in C 5/2 == 2, not here!)
5 // 2    # 2
-5 // 2   # -3   (floor, toward negative infinity; in C -5/2 == -2, truncated toward zero)
5 % 2     # 1
-5 % 2    # 1    (sign follows the divisor, unlike C! In C -5%2 == -1)
2 ** 10   # 1024

The rounding/sign rules for // and % differ from C, a classic source of bugs when porting numeric code. Python guarantees a == (a // b) * b + (a % b), and the remainder’s sign follows the divisor.

Integers have another killer feature: arbitrary precision. Python’s int doesn’t overflow; 2 ** 200 computes directly, because it’s a variable-length heap object (Chapter 01 mentioned that PyLongObject carries variable-length digits after its header). There’s no such thing as int32 overflow wraparound.

x = 2 ** 200          # no overflow; computes all 61 decimal digits directly
print(x)              # 1606938044258990275541962092341162602522202993782792835301376
print(type(x))        # <class 'int'>  still an int, no int64 ceiling
print(x.bit_length()) # 201

6. f-strings: Formatted Strings

name, n = "Ann", 3
print(f"{name} has {n} apples")        # Ann has 3 apples
print(f"{n} squared is {n**2}")        # any expression can go inside the braces
print(f"pi = {3.14159:.2f}")           # pi = 3.14  (:.2f format spec)
print(f"{n=}")                         # n=3        (for debugging, auto-includes the variable name)
print(f"{255:#x}")                     # 0xff

At compile time, an f-string is broken down into string-concatenation bytecode (verifiable with dis), so it’s efficient at runtime and is the preferred formatting method.

Verify the “compile-time breakdown” yourself: there’s no runtime parsing of a format string; it compiles straight to “format each piece + concatenate.”

import dis
name, n = "Ann", 3
dis.dis('f"{name} has {n}"')

Output (3.14): each {} becomes one FORMAT_SIMPLE, and the final BUILD_STRING 3 joins the three pieces (name, ' has ', n) into one str:

  0           RESUME                   0

  1           LOAD_NAME                0 (name)
              FORMAT_SIMPLE
              LOAD_CONST               0 (' has ')
              LOAD_NAME                1 (n)
              FORMAT_SIMPLE
              BUILD_STRING             3
              RETURN_VALUE

It supersedes the old % formatting and str.format(). It’s analogous to C’s printf/snprintf, but expressions are embedded directly and types are handled automatically, with no %d/%s placeholder matching needed.

7. Booleans, None, and Truthiness Testing

flag = True          # capitalized! True / False, not true / false
nothing = None       # the null-value singleton, analogous to nullptr / NULL (but it's an object)
  • True / False are of type bool; in CPython they’re actually a subclass of int, with True == 1 and False == 0.
  • None is a globally unique singleton object meaning “no value,” analogous to nullptr—but it’s a full-fledged heap object, and you test equality with is: if x is None: (not == None).

Truthiness testing: any object can go in an if, not just a bool. In C “nonzero is true”; Python is richer:

Falsy Truthy
False, None everything else
0, 0.0, 0j nonzero numbers
"", [], {}, (), set() non-empty containers 
items = []
if items:                # an empty list is falsy; no need to write len(items) == 0
    print("in stock")
else:
    print("empty")       # this branch runs

Under the hood: if x calls the object’s __bool__ (falling back to __len__, testing whether it’s 0). This is runtime method dispatch through ob_type, in the same vein as the protocols in Chapter 08.

8. Chained Comparisons and Multiple Assignment

Chained comparisons—absolutely nonexistent in C, and one of the easiest pitfalls for C porters:

x = 5
if 0 < x < 10:           # reads as (0 < x) and (x < 10)
    print("in range")

# In C, 0 < x < 10 parses as (0 < x) < 10, i.e. (true) < 10 == 1 < 10 == true, always true!
# Python doesn't do that; it's genuinely chained in the mathematical sense

Multiple assignment / unpacking:

a, b = 1, 2              # bind simultaneously
a, b = b, a             # swap, no temporary variable needed
x = y = z = 0           # all three names point to the same 0 object (mind the aliasing; see Chapter 03)
first, *rest = [1, 2, 3, 4]   # first=1, rest=[2,3,4], the star collects the rest

Under the hood of a, b = b, a: the two values on the right are each evaluated and pushed onto the stack first, then bound to the left, so no temporary is needed and there’s no evaluation-order trap. CPython optimizes small swaps away: 3.11–3.12 use a single SWAP instruction (SWAP 2); 3.13+ goes further with the super-instructions LOAD_FAST_LOAD_FAST + STORE_FAST_STORE_FAST (dropping even SWAP). Only earlier versions or larger tuples fall back to BUILD_TUPLE + UNPACK_SEQUENCE. Measured on 3.14 here:

import dis

def swap(a, b):
    a, b = b, a

dis.dis(swap)

The output confirms that even SWAP/tuple-packing is gone—one super-instruction to read, one to write:

  3           RESUME                   0

  4           LOAD_FAST_LOAD_FAST     16 (b, a)
              STORE_FAST_STORE_FAST   16 (b, a)
              LOAD_CONST               0 (None)
              RETURN_VALUE

9. C/C++ ↔︎ Python Syntax Cheat Sheet

Topic C/C++ Python
Block delimiter { } indentation (INDENT/DEDENT)
Statement end ; newline
Variable declaration int x; required just x = ...
Type static, bound to the variable dynamic, bound to the object
Entry point main() top-level statements; if __name__=="__main__"
Logical operators && \|\| ! and or not
Division / truncated integer division between ints always float; use // for integer division
Modulo sign follows the dividend follows the divisor
Integer overflow yes (wraparound/UB) none, arbitrary precision
Ternary c ? a : b a if c else b
++/-- yes no, use x += 1
Null value NULL/nullptr None (an object, tested with is)
Boolean literals true/false True/False
Increment/decrement i++ doesn’t exist
switch switch/case match/case (3.10+, see Chapter 05)
Constants const none enforced; by naming convention (ALL CAPS)

Two easily forgotten points: - No ++/--. x++ is a syntax error; use x += 1. - The ternary operator is value1 if condition else value2, in a different order from C: y = a if x > 0 else b.

10. Summary

  • Indentation is a syntactic token (turned into INDENT/DEDENT at compile time), replacing braces; but indented blocks don’t introduce a new scope—only functions/classes/modules plus comprehensions and generator expressions do.
  • Dynamically and strongly typed, no declaration needed: types are bound to objects and checked at runtime, but no implicit conversions happen.
  • No main; a module’s top level executes in order; if __name__ == "__main__" distinguishes script from library.
  • Operator pitfalls: / is always float, // floors, %’s sign follows the divisor, integers are arbitrary precision, and there’s no ++.
  • Truthiness testing is richer than C (empty containers are falsy); chained comparisons are genuinely chained; a, b = b, a is optimized by CPython (3.11–3.12 use SWAP, 3.13+ use LOAD_FAST_LOAD_FAST/STORE_FAST_STORE_FAST, earlier versions use tuple pack/unpack).
