运行模型：javac、字节码、JVM 与 JIT

作为 C/C++ 工程师，你熟悉的编译链路是：.c 源码经预处理、编译、汇编生成目标文件 .o，再由链接器拼成一个针对特定 CPU 架构 + 特定操作系统的可执行文件（ELF/Mach-O/PE）。CPU 直接执行其中的机器码。

Java 走的是一条两段式的路：先编成一种平台无关的中间码（字节码），再由一台”虚拟 CPU”（JVM）在运行时执行。理解这条链路，是理解后面所有内存行为的前提。

一、从源码到字节码：javac 干了什么

// Hello.java
public class Hello {
    public static void main(String[] args) {
        int sum = 0;
        for (int i = 0; i < 10; i++) {
            sum += i;
        }
        System.out.println(sum);
    }
}

编译：

javac Hello.java     # 产出 Hello.class

javac 与 gcc 的本质区别：gcc 的输出面向真实 CPU，指令集是 x86-64 / ARM64；javac 的输出面向一台假想的栈式虚拟机，指令集就是”字节码”（bytecode）。字节码不假设寄存器数量、不假设字长、不假设字节序——它只面向 JVM 规范定义的抽象机器。这就是”一次编写，到处运行”的技术根基：.class 文件在 Windows/Linux/macOS、x86/ARM 上是逐字节相同的。

.class 文件是一个有严格格式的二进制容器（魔数 0xCAFEBABE 开头），里面包含：

区段 内容 C/C++ 类比
常量池 constant pool 字符串、类名、方法名、字面量的符号引用 类似 .rodata + 符号表，但更结构化
字段表 类的字段声明 struct 成员布局信息
方法表 每个方法的字节码 + 元数据 函数体，但带完整类型签名
属性表 行号表、局部变量表、注解等 调试信息（类似 DWARF），但内置

关键点：.class 保留了完整的类型与符号信息。C 程序 strip 之后几乎只剩机器码，符号丢失；而 .class 天然携带类名、方法签名、字段类型——这是反射、动态加载、运行时类型检查得以存在的物质基础。

用 javap 看字节码

javap 是 JDK 自带的反汇编器，相当于 Java 世界的 objdump -d：

javap -c Hello.class

main 方法的字节码大致如下（注释我加的）：

0:  iconst_0        // 把常量 0 压入操作数栈
1:  istore_1        // 弹出栈顶存入局部变量表 slot 1（即 sum）
2:  iconst_0        // 常量 0 入栈
3:  istore_2        // 存入 slot 2（即 i）
4:  iload_2         // 加载 i 入栈
5:  bipush 10       // 常量 10 入栈
7:  if_icmpge 20    // 若 i >= 10 跳到偏移 20（退出循环）
10: iload_1         // 加载 sum
11: iload_2         // 加载 i
12: iadd            // 栈顶两值相加，结果留在栈上
13: istore_1        // 存回 sum
14: iinc 2, 1       // 直接对 slot 2（i）自增 1，不经过操作数栈
17: goto 4          // 回到循环判断
20: getstatic ...   // 取 System.out
23: iload_1
24: invokevirtual println
27: return

注意 iconst/istore/iload/iadd 这些指令没有寄存器操作数。JVM 是栈式机器：运算的输入输出都在”操作数栈”上流转，而不是像 x86 那样 add eax, ebx 指名寄存器。i 前缀表示操作 int 类型，还有 l(long)、f(float)、d(double)、a(引用 address)。iinc 是少见的能直接改局部变量表的指令。

每个方法在字节码里都声明了它需要多大的操作数栈（stack=）和多少个局部变量 slot（locals=）。这些是编译期就算好的固定值，JVM 据此分配栈帧——见「函数与调用栈」。

二、类加载：把 .class 装进 JVM

C 程序启动时，加载器（loader）把 ELF 段映射进内存，重定位、绑定动态库符号，一次性搞定。Java 的类加载是惰性的、按需的、运行期的，分三个阶段：

1. 加载 Loading

根据类的全限定名找到 .class 字节流（从文件系统、jar、网络、甚至运行时生成），解析成方法区/元空间里的一个内部数据结构，并在堆上创建一个代表该类的 java.lang.Class 对象。这一步类似 dlopen 打开一个库，但粒度是单个类。

2. 链接 Linking（三小步）

  • 验证 Verification：检查字节码是否合法、类型是否安全、栈是否会溢出/下溢。C/C++ 没有对应物——它信任编译器产出的机器码；JVM 因为可能加载来路不明的字节码，必须做这层安全校验。
  • 准备 Preparation：为静态字段分配空间并置零值（int→0，引用→null）。注意此时不执行用户赋值。
  • 解析 Resolution：把常量池里的符号引用（“类 A 的方法 foo”）替换为直接引用（内存地址/偏移）。这相当于 C 的链接期符号解析，但发生在运行时，且可以延迟到第一次使用时才做。

3. 初始化 Initialization

执行静态初始化块和静态字段的赋值（即 <clinit> 方法）。触发时机：首次 new、访问静态字段/方法、反射等。JVM 保证 <clinit> 在多线程下只执行一次（带锁），这也是”静态单例 + 类加载”惯用法线程安全的原因。

双亲委派模型 Parent Delegation

JVM 里有多个类加载器，形成层级：

        Bootstrap ClassLoader   (C++ 实现，加载 java.* 核心类)
              ▲  委派
        Platform ClassLoader    (JDK 平台模块)
              ▲  委派
        Application ClassLoader (你的 classpath)
              ▲  委派
        自定义 ClassLoader

加载一个类时，子加载器先向上委派给父加载器尝试，父加载不了才自己加载。好处：核心类（如 java.lang.String）永远由 Bootstrap 加载，你没法用一个伪造的 java.lang.String 顶替系统类——这是一种安全与一致性保证。C/C++ 里若两个 .so 各自带同名符号，行为取决于链接顺序，容易出问题；双亲委派从机制上避免了核心类被”劫持”。

三、执行：解释 + JIT 混合模式

拿到字节码后，JVM 怎么执行？两种手段并用（HotSpot 的默认”混合模式”）：

解释执行

JVM 内置一个字节码解释器，逐条读取字节码、查表、执行对应的 C++ 逻辑。启动快（不用等编译），但每条指令都有解释开销，比原生机器码慢一到两个数量级。

热点探测 + JIT 编译

HotSpot 这个名字就来自它的策略：探测热点代码（Hot Spot）。JVM 为每个方法维护计数器（方法调用次数 + 循环回边次数），一旦超过阈值，就认定它”热”，交给 JIT（Just-In-Time）编译器把这段字节码在运行时编译成本地机器码，之后直接执行机器码。

HotSpot 有两个 JIT 编译器（分层编译 Tiered Compilation 会协同使用）：

编译器 定位 特点
C1（Client） 快速编译 优化少，编译快，先顶上
C2（Server） 深度优化 激进优化，编译慢，用于最热的代码

典型路径：解释执行 → C1 编译（带性能计数）→ C2 重编译最热部分。

这就是 Java “越跑越快”、且有时能逼近甚至局部超过 C 的原因：JIT 掌握运行时信息——实际走哪个分支、某个虚方法实际只调用了一个实现类、某个循环边界是多少。它能据此做静态编译器做不到的投机优化（speculative optimization），赌错了再”去优化”（deoptimization）退回解释执行。而 gcc 编译时对运行时一无所知，只能保守。

几个关键 JIT 优化（对内存行为影响很大）

  • 方法内联 Inlining：把被调方法体直接嵌入调用处，消除调用开销，还打开了后续优化的视野。等价于 C 的 inline，但 JIT 是运行时基于热度自动决定的。
  • 逃逸分析 Escape Analysis：分析一个对象的引用是否”逃逸”出当前方法（是否被返回、被存入堆上其它对象、被其它线程看到）。若不逃逸，JVM 可以做：
    • 标量替换（Scalar Replacement）：把本该在堆上 new 的对象拆散成它的各个字段，当作普通局部变量直接放进栈帧 slot 甚至寄存器。注意这不是”把整个对象搬到栈上”，而是对象作为一个整体根本不再存在——因此既没有堆分配、也无需 GC 回收。（HotSpot 并没有真正的”整对象栈上分配”，能省掉堆分配的机制就是标量替换。）

这一点对你理解 Java 内存很重要：Java 语言层面”对象一律在堆上”，但 HotSpot 实现层面，逃逸分析可能让短命的、不逃逸的对象根本不进堆。这弥合了 Java 与 C（栈上放 struct）的部分性能差距。详见「变量与内存模型」。

四、运行时内存区域

一个运行中的 JVM 把内存划成几块（JVM 规范定义，HotSpot 实现）：

┌───────────────────────────────────────────────────────┐
│                         JVM 进程                         │
│                                                         │
│  所有线程共享 ↓                                          │
│  ┌─────────────────────┐   ┌───────────────────────┐   │
│  │        堆 Heap        │   │  方法区 / 元空间         │   │
│  │  所有对象、数组       │   │  类元数据、方法字节码    │   │
│  │  GC 主战场            │   │  运行时常量池           │   │
│  │  (new 出来的都在这)   │   │  (元空间在本地内存)      │   │
│  └─────────────────────┘   └───────────────────────┘   │
│                                                         │
│  每个线程私有 ↓                                          │
│  ┌───────────┐ ┌───────────┐ ┌──────────┐ ┌──────────┐ │
│  │ JVM 栈     │ │ PC 寄存器  │ │本地方法栈 │ │ ...      │ │
│  │ 一堆栈帧   │ │ 当前指令   │ │JNI 用     │ │每线程一份 │ │
│  └───────────┘ └───────────┘ └──────────┘ └──────────┘ │
└───────────────────────────────────────────────────────┘
区域 谁的 存什么 C/C++ 类比
堆 Heap 共享 所有对象、数组实例 malloc 的堆，但由 GC 自动回收
方法区/元空间 Metaspace 共享 类结构、方法字节码、运行时常量池、静态字段的元信息（字段的值存放在堆上的 Class 镜像对象里，不在元空间） 代码段 + 只读数据段（但可运行时增长）
JVM 栈 每线程 栈帧（局部变量表、操作数栈） 线程调用栈，逻辑一致
PC 寄存器 每线程 当前执行到的字节码地址 程序计数器 PC/RIP
本地方法栈 每线程 执行 native（JNI）方法时的栈 C 侧的原生栈

几点与 C 的对照： - 元空间在本地内存（native memory），不在 GC 堆里（这是 JDK8 起的设计，取代了旧的”永久代 PermGen”）。 - 堆的回收是自动的：没有 free/delete，GC 判定不可达就回收。你几乎从不手动管理堆生命周期——这是与 C 最大的心智差异，也是「错误处理与资源管理」里 try-with-resources 只针对”非内存资源”（文件/socket）的原因：内存交给 GC，句柄仍需显式关。 - 栈的语义几乎和 C 一样：每次方法调用压一个栈帧，返回时弹出；栈太深触发 StackOverflowError（对应 C 的栈溢出，但 JVM 会抛异常而非直接段错误）。

五、AOT：GraalVM native-image

Java 也能像 C 一样提前编译（Ahead-Of-Time）成原生可执行文件。GraalVM 的 native-image 在构建期做闭包分析，把用到的类、方法、乃至一个精简的运行时（含 GC）静态编译打包成一个独立可执行文件：

native-image Hello        # 产出一个原生可执行文件 hello
./hello                   # 无需 JVM，毫秒级启动

代价与取舍： - 优点：启动极快、内存占用小、无需分发 JVM——适合 CLI 工具、Serverless/函数计算冷启动。这时它和 C 的分发模型很像：一个原生二进制。 - 代价：构建期必须知道全部会用到的类，因此反射、动态代理、运行时生成类需要额外配置；失去了 JIT 的运行时投机优化，峰值吞吐可能不如长时间预热后的 C2。

所以对照记忆： - javac → 字节码，跨平台，靠 JVM + JIT 运行时优化（长跑服务的主流）。 - native-image → 原生码，绑定平台，构建期定死（快启动短命进程的选择）。 - gcc/clang → 原生码，绑定平台，编译期定死（C/C++ 的唯一路径）。

小结

  • javac 产出的是平台无关字节码，不是机器码；执行它需要 JVM 这台栈式虚拟机。
  • 类加载是运行时、惰性、分层（双亲委派）的，且保留完整类型信息。
  • HotSpot 先解释、探测热点后 JIT（C1/C2、内联、逃逸分析）；逃逸分析能对不逃逸对象做标量替换（拆成标量放栈帧/寄存器，而非整体栈上分配）。
  • 运行时内存分共享的堆/元空间与每线程的 JVM 栈/PC/本地栈；堆由 GC 自动回收。
  • 需要 C 式的快启动原生二进制时，用 GraalVM native-image 做 AOT，代价是牺牲运行时动态性与峰值优化。
