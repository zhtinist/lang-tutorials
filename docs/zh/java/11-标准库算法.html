标准库算法

C++ 的 <algorithm>、std::sort(introsort)、std::set(红黑树)、priority_queue(堆)在 STL 里。Java 对应的是 java.util.Collections、Arrays、PriorityQueue、TreeMap,加上函数式的 Stream API。

Collections 工具类

java.util.Collections 是一堆静态方法,操作 List/Collection:

List<Integer> list = new ArrayList<>(List.of(3, 1, 2));
Collections.sort(list);                          // 原地排序(内部转数组用 TimSort)
int idx = Collections.binarySearch(list, 2);     // 已排序 List 上二分;链表(非 RandomAccess)是 O(n) 次链表遍历 + O(log n) 次比较
Collections.reverse(list);
Collections.shuffle(list);
int max = Collections.max(list);

// 不可变视图:返回包装,任何修改抛 UnsupportedOperationException(类比 const 引用,但运行期检查)
List<Integer> ro = Collections.unmodifiableList(list); // 底层仍指向 list,list 变它也变
List<Integer> imm = List.of(1, 2, 3);            // JDK9+ 真正不可变副本

binarySearch 要求二分查找的前提——数组已经按同一顺序排好——必须由调用者自己保证,标准库不会替你检查。原因很直接:检查”是否已排序”本身是 O(n) 的,如果每次调用都验证一遍,就抵消了二分查找 O(log n) 的意义;把这个前提作为契约交给调用者,才能让 binarySearch 本身保持 O(log n)。违反契约(传入未排序数组)不会报错,只会返回一个没有意义的结果。返回负数 -(插入点)-1 表示未找到,可用于求插入位置。

Arrays.sort:两种算法

Arrays.sort 对基本类型和对象数组用不同算法,这不是随意为之:

int[] a = {5, 2, 8, 1};
Arrays.sort(a);                    // Dual-Pivot Quicksort(双轴快排)

Integer[] b = {5, 2, 8, 1};
Arrays.sort(b);                    // TimSort(归并变体)
Arrays.sort(b, Comparator.reverseOrder());
维度 基本类型 int[] 对象 Object[]
算法 双轴快排 TimSort
稳定性 无所谓(值相等即完全相同) 必须稳定
比较代价 极低(机器指令) 高(调 compareTo/compare,可能虚方法)
关注点 缓存局部性、比较次数 减少比较次数、稳定性

基本类型用双轴快排的理由:两个 int 相等就是同一个值,交换它们无可观测差别,所以稳定性无意义——这是选择快排的前提条件,而不是快排本身的优点,快排在对象排序里同样快,只是不稳定这一点在基本类型上完全不构成代价。快排原地、缓存友好(顺序访问局部性好),比较是廉价的 CPU 指令,这意味着”比较次数”在这里不是瓶颈,“访存模式”才是。双轴快排选两个 pivot 把数组分三段,比经典单轴快排减少约 5% 比较、缓存局部性更好——注意它的交换次数其实比单轴更多,优势来自”比较更少 + 访存友好”,而非”少移动”。

对象用 TimSort 的理由,直接对应上表两个关注点: 1. 稳定性:对象排序常按某字段,相等元素的原相对顺序必须保留(比如先按 A 排、再按 B 排,B 相等时要维持 A 的顺序)。快排在划分过程中会把值相等的元素任意换位,不保证原始顺序,这对基本类型无所谓,对多字段排序的对象却是错误的结果。TimSort(归并类算法天然稳定)满足这个要求。 2. 比较昂贵:对象比较是方法调用(可能是接口虚调用,还可能内部又比较多个字段),不是一条机器指令能完成的操作,这意味着”减少比较次数”比”减少数据移动次数”更值得优化。TimSort 利用真实数据常见的局部有序性:扫描出已经有序的片段(“run”),run 之间做归并而不是重新比较每一对元素,已排序输入下比较次数逼近 O(n);快排不识别这种结构,最坏情况仍是 O(n²) 次比较。

TimSort 结合归并排序与插入排序:扫描出递增/递减的 run,短 run 用二分插入排序扩展到 minrun 长度,再按栈上不变式归并。最坏 O(n log n),已有序 O(n)。它需要 O(n) 辅助空间(归并),而快排原地——对象数组本身就是一层指针间接(数组存的是引用,不是对象本体),缓存局部性天然不如基本类型数组,这种情况下再多付出一点空间换稳定性和更少比较,是合理的取舍。

Comparable 与 Comparator

  • Comparable<T>:自然顺序,类自己实现 compareTo,一个类只有一种。类比 C++ 的 operator<。
  • Comparator<T>:外部策略,可定义任意多种,类比传给 std::sort 的仿函数。

两者并存而非只保留一个,是因为它们回答的是两个不同的问题:“这个类型本身有没有唯一自然的顺序”(比如数值大小、字符串字典序,Comparable 表达这种类型内建的排序语义),与”我现在这一次想按什么临时规则排”(同一个 Person 类型,有的地方想按年龄排,有的地方想按姓名排,这些都不构成”这个类型的自然顺序”,只是调用方的一次性需求,Comparator 把它表达成一个独立的、可以随时替换的策略对象,而不必侵入类定义)。

下面例子用到两个可能没见过的语法:record 是 Java 16+ 的简写,一行声明一个只存数据的不可变类(类比 C++ 里一个只有字段、自动生成构造函数和 getter 的 struct);Person::name 是方法引用,是 p -> p.name() 的简写形式,可以直接理解成”取 name 字段的 lambda”。

record Person(String name, int age) implements Comparable<Person> {
    @Override public int compareTo(Person o) {   // 自然顺序:按年龄
        return Integer.compare(this.age, o.age);  // 勿用 this.age-o.age,溢出风险
    }
}

// Comparator 组合器(JDK8+),声明式,类似链式 SQL ORDER BY
Comparator<Person> cmp = Comparator
    .comparing(Person::name)                     // 先按名字
    .thenComparingInt(Person::age)               // 名字相同再按年龄
    .reversed();                                 // 整体反转

Comparator<Person> byAgeNullsFirst = Comparator
    .comparingInt(Person::age)
    .reversed();
List<Person> people = new ArrayList<>();
people.sort(cmp);   // List.sort 内部走 TimSort

契约:compare 须满足自反、传递、反对称;且与 equals 一致性建议保持(否则 TreeMap/TreeSet 行为反直觉)。违反传递性会让 TimSort 抛 IllegalArgumentException: Comparison method violates its general contract——这不是随意加的检查,而是因为归并排序的正确性证明本身依赖比较关系满足传递性(算法假设”已知 A ≤ B、B ≤ C 就能推出 A ≤ C”,不再需要重新比较 A 和 C),一旦这个前提不成立,算法内部的一致性检测会发现矛盾并主动报错,而不是默默输出一个错误但不崩溃的排序结果。

可运行:先按 name 升序,name 相同再按 age 降序——thenComparing + reversed 声明式组合:

import java.util.*;

public class CmpDemo {
    record Person(String name, int age) {}

    public static void main(String[] args) {
        List<Person> people = new ArrayList<>(List.of(
            new Person("Bob", 30), new Person("Ann", 30),
            new Person("Bob", 25), new Person("Ann", 20)));

        // 先按 name 升序,name 相同再按 age 降序
        people.sort(Comparator.comparing(Person::name)
                              .thenComparing(Comparator.comparingInt(Person::age).reversed()));

        for (Person p : people) System.out.println(p.name() + " " + p.age());
    }
}
// Ann 30
// Ann 20
// Bob 30
// Bob 25

PriorityQueue:二叉堆

PriorityQueue 是基于数组的小顶二叉堆(默认最小元素在队首),类比 C++ priority_queue(默认大顶)。

PriorityQueue<Integer> pq = new PriorityQueue<>();      // 小顶堆
pq.offer(3); pq.offer(1); pq.offer(2);
pq.peek();     // 1,查看堆顶 O(1)
pq.poll();     // 1,弹出并下沉调整 O(log n)

// 大顶堆 / 自定义
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());

为什么用堆而不是排序数组或平衡树:堆解决的问题只有”反复拿最值、反复插入”,不需要”任意位置查找”或”完整有序遍历”这两种能力,这个更窄的需求让它能比通用有序结构更便宜。排序数组能 O(1) 取最值,但插入一个新元素要挪动数据、O(n);红黑树(TreeMap/TreeSet)插入和取最值都是 O(log n),看起来和堆一样,但红黑树要维护”任意两个元素之间的相对顺序”这个更强的不变式(为了支持中序遍历得到全序、支持任意 key 的 floor/ceiling 查询),堆放弃了这个不变式,只维护”父节点 ≤ 子节点”这一条弱得多的性质(不要求兄弟节点之间有序),换来的是插入/删除时只需要沿一条路径上浮或下沉,不需要红黑树那样的变色和旋转维护,常数因子更小、实现更简单。这是”越具体的需求,越能用越便宜的结构满足”的一个直接例子——需要完整有序或任意查找就该用 TreeMap,只需要反复拿最值就选堆。

内部 Object[] queue,父子索引关系:节点 i 的孩子是 2i+1、2i+2(用数组下标运算模拟树形指针,省去每个节点存左右子指针的空间,也让节点在内存里连续排列,缓存友好)。offer 上浮(siftUp),poll 把末尾元素移到堆顶再下沉(siftDown)。

数组 [1,3,2,7,4] 表示的堆:
          1(0)
        /     \
      3(1)     2(2)
      /  \
    7(3) 4(4)

非线程安全,也不保证遍历有序(只有堆顶有序——这正是上面说的”放弃了任意两元素间的顺序”这一弱不变式的直接后果);要有序遍历得反复 poll。并发场景用 PriorityBlockingQueue。

可运行:用大小为 k 的小顶堆求 Top-K 最大值——堆顶恰是这 k 个里最小的,超出就弹掉,单次 O(log k),这正是利用了堆”只需要知道最值、不需要维护整体顺序”的特性,比对全部数据排序(O(n log n))更省:

import java.util.*;

public class HeapTopK {
    public static void main(String[] args) {
        int[] data = {5, 1, 9, 3, 7, 8, 2, 6, 4};
        int k = 3;

        // 维护 k 个最大值:小顶堆的堆顶 = 当前 top-k 中最小者
        PriorityQueue<Integer> pq = new PriorityQueue<>();  // 小顶堆
        for (int x : data) {
            pq.offer(x);
            if (pq.size() > k) pq.poll();                   // 弹掉最小,O(log k)
        }

        // 逐个 poll:小顶堆按升序出队
        List<Integer> topK = new ArrayList<>();
        while (!pq.isEmpty()) topK.add(pq.poll());
        System.out.println("top-" + k + " ascending = " + topK);
    }
}
// top-3 ascending = [7, 8, 9]

TreeMap / TreeSet:红黑树

TreeMap 底层是红黑树(自平衡 BST),TreeSet 是其包装(值当 key)。对应 C++ std::map/std::set。所有操作 O(log n),且按 key 有序。

TreeMap<String, Integer> tm = new TreeMap<>();
tm.put("banana", 2); tm.put("apple", 1); tm.put("cherry", 3);
// 有序视图与导航方法(HashMap 没有)
tm.firstKey();               // "apple"
tm.floorKey("b");            // "apple"(≤"b" 的最大 key)
tm.ceilingKey("b");          // "banana"(≥"b" 的最小 key)
tm.headMap("cherry");        // 小于 cherry 的子视图(视图,改动会反映到原 map)
tm.subMap("apple", "cherry");
for (var e : tm.entrySet()) { }  // 按 key 升序遍历

红黑树的五条性质(节点红/黑、根黑、红节点子黑、任一节点到叶黑高相同等)保证最长路径 ≤ 2× 最短,近似平衡而不要求严格平衡。为什么选红黑树而不是更严格平衡的 AVL 树:AVL 树要求任意节点左右子树高度差 ≤1,查找路径更短、查找更快,但插入和删除后为了维持这个更严格的约束,平均需要更多次旋转来恢复平衡;红黑树放宽约束(允许最长路径达到最短路径的 2 倍),换来插入/删除时旋转次数的上界更低(插入最多两次旋转,删除最多三次)。TreeMap 作为通用关联容器,插入删除和查找都常见,红黑树在”查找稍慢一点点、但增删更便宜”这个折中点上更适合通用场景,这也是 C++ std::map、Linux 内核调度器等大量选择红黑树而非 AVL 的共同理由。插入/删除靠变色 + 旋转维持,单次调整 O(log n)。

对比 HashMap TreeMap
底层 数组+链表/红黑树 红黑树
有序 无序 按 key 排序
get/put O(1) 均摊 O(log n)
导航方法 无 floor/ceiling/subMap 等
null key 允许一个 不允许(需比较)

HashMap 用 O(1) 换掉了顺序;TreeMap 用 O(log n) 换回顺序和导航能力——选哪个取决于”要不要按 key 排序或做区间查询”,不需要就用更快的 HashMap,需要就付出 log 因子换 TreeMap 的导航方法。

Stream API

最简单的用法:对一个 List 先筛选、再计数:

List<String> names = List.of("Ann", "Bob", "Al", "Ben");
long count = names.stream()               // 1. 从集合开一条"流水线"
    .filter(s -> s.startsWith("A"))        // 2. 只留下满足条件的元素
    .count();                              // 3. 数一数还剩几个
System.out.println(count);                 // 2

集合.stream() 开启流水线,中间可以链式接任意多个”加工步骤”(这里是 filter),最后用一个”取结果的操作”(这里是 count)结束整条链子——源、中间操作、终止操作这个三段结构不是随意的 API 风格选择,下面解释它具体解决什么问题。

Stream 是声明式、惰性、可并行的数据处理管道,类比 C++20 ranges/views。不是容器,不存数据,是”对数据源的操作描述”。

惰性与操作分类

List<String> words = List.of("apple", "banana", "cherry", "avocado");

long count = words.stream()          // 源
    .filter(s -> s.startsWith("a"))  // 中间操作(intermediate):惰性,返回新 Stream
    .map(String::toUpperCase)        // 中间操作
    .peek(System.out::println)       // 中间操作(调试用)
    .count();                        // 终止操作(terminal):触发实际执行
  • 中间操作(filter/map/sorted/distinct/limit):惰性,只是记录,不执行。返回 Stream 可链式。
  • 终止操作(collect/forEach/count/reduce/findFirst):触发遍历,执行整条链。

为什么要分成”惰性中间操作”和”及早求值终止操作”这两类,而不是每个操作都立即执行:如果 filter、map 这类操作都立即求值,每一步都必须产出一个完整的中间集合交给下一步——filter 先把满足条件的元素收集成一个新 List,再交给 map 生成第二个新 List,再交给下一个操作……n 步链式操作就要分配 n 个中间集合,每个都是 O(元素数) 的内存和一次完整的遍历。这在链条变长或数据量变大时,内存分配和多趟遍历的开销会线性累加。

Stream 的设计反过来:中间操作只是把”要做什么”记录成一个函数,并不真正遍历数据,直到终止操作被调用,才会驱动数据一个元素一个元素地穿过整条链——同一个元素在同一次”访问”里依次经过 filter 判断、map 转换、peek 打印,再取下一个元素重复,整条链只对数据源做一次遍历,并且不需要为任何中间步骤分配集合(sorted/distinct 等需要看到全部数据才能判断的操作是例外,这类操作本身必须缓冲)。这正是”操作融合”(fusion):多个逻辑步骤在执行时被合并成一次遍历里的一系列函数调用,而不是 n 次遍历 + n 次分配。

惰性还带来另一个直接收益:短路。像 limit/findFirst 这类操作一旦拿到需要的结果就能让整条流水线提前停止,不需要处理完整个数据源——如果操作是及早求值的,filter 已经把所有满足条件的元素都算出来放进集合了,findFirst 就算只要第一个也已经太迟。

// 短路:找到第一个就停,不会处理整个流
Optional<Integer> first = Stream.iterate(1, n -> n + 1)  // 无限流
    .filter(n -> n % 7 == 0)
    .findFirst();   // Optional[7]

这个例子把惰性的必要性推到极端:数据源是无限流,如果 filter 试图及早求值,程序根本不会终止——只有”直到终止操作明确要求某个结果之前,什么都不做”这套惰性机制,才能让”对无限序列做有限次查询”这件事本身可行。

Collectors

终止操作 collect 把流收集成结果:

import static java.util.stream.Collectors.*;

List<String> list  = words.stream().collect(toList());
Set<String> set    = words.stream().collect(toSet());
Map<Character, List<String>> byFirst =
    words.stream().collect(groupingBy(s -> s.charAt(0)));   // 分组,类似 SQL GROUP BY
Map<Boolean, List<String>> parts =
    words.stream().collect(partitioningBy(s -> s.length() > 5));
String joined = words.stream().collect(joining(", ", "[", "]"));
Double avg = words.stream().collect(averagingInt(String::length));

为什么 collect 接受一个 Collector 对象,而不是像 toList()/toSet()/groupingBy() 那样直接在 Stream 接口上开一堆重载方法:Collector<T,A,R> 本质上是把”如何归约”打包成三个函数(supplier 造一个可变的中间容器、accumulator 把一个元素并入容器、combiner 把两个并行处理出来的容器合并成一个)加上一个收尾步骤(finisher)。如果改用方法重载,toList、toSet、toMap(带合并函数的重载)、groupingBy(可以嵌套指定”分组之后再怎么收集”的下游收集器)这些组合会迅速爆炸成数十个签名相近却语义不同的重载,而且重载方法之间没有办法互相组合——而 Collector 是一个值,可以作为参数传递、可以嵌套(groupingBy(分类函数, 下游 Collector) 就是把一个 Collector 作为另一个 Collector 的一部分),collectingAndThen 可以在任意收集器外面再包一层后处理。这是”把行为封装成可组合的对象”而不是”穷举所有组合的方法名”的设计——组合数是几个正交维度的乘积,封装成对象后只需要为每个维度写一个函数,重载法却要为每个组合都写一个方法。

combiner(合并两个中间容器)存在的意义还不止于此:它是并行流能够工作的前提——数据被拆成若干块,每块各自跑 supplier+accumulator 产出一个局部结果,最后要靠 combiner 把这些局部结果两两合并成一个。方法重载没有天然的地方放这个”合并”步骤,而 Collector 接口把它作为一等公民,同一个 Collector 因此能同时支持串行和并行流,不需要为并行场景另写一套。

可运行:peek 观察惰性——findFirst 短路,只拉到第一个匹配就停,不遍历整个流;groupingBy 按首字母分组:

import java.util.*;
import java.util.stream.*;
import static java.util.stream.Collectors.*;

public class StreamDemo {
    public static void main(String[] args) {
        List<String> words = List.of("apple", "banana", "cherry", "avocado", "blueberry");

        // 惰性 + 短路:peek 只会打印真正被拉取的元素
        System.out.println("--- lazy pipeline (findFirst short-circuits) ---");
        Optional<String> first = words.stream()
            .peek(w -> System.out.println("seen: " + w))   // 副作用,用于观察遍历
            .filter(w -> w.startsWith("b"))
            .findFirst();
        System.out.println("first b-word = " + first.get());

        // groupingBy:按首字母分组,用 TreeMap 让输出有序稳定
        Map<Character, List<String>> byFirst = words.stream()
            .collect(groupingBy(w -> w.charAt(0), TreeMap::new, toList()));
        System.out.println("--- groupingBy ---");
        System.out.println(byFirst);
    }
}
// --- lazy pipeline (findFirst short-circuits) ---
// seen: apple
// seen: banana
// first b-word = banana
// --- groupingBy ---
// {a=[apple, avocado], b=[banana, blueberry], c=[cherry]}

输出证实了惰性:peek 只打印到 banana 就停(findFirst 短路),cherry、avocado、blueberry 根本没有被访问——如果流是及早求值的,filter 会先扫完整个列表,peek 应该打印全部 5 个元素。

并行流

parallelStream() 或 .parallel() 把流拆分,底层用公共 ForkJoinPool(ForkJoinPool.commonPool(),默认线程数 = CPU 核数-1)分治处理。选择 ForkJoinPool 而非普通线程池,是因为它用工作窃取(work-stealing)调度:每个工作线程有自己的任务双端队列,做完自己的任务后去”偷”别的线程队列尾部的任务,不需要中心调度器协调,天然适合”一个大任务被递归拆成大量不均匀子任务”这种分治场景。

long sum = LongStream.rangeClosed(1, 1_000_000)
    .parallel()
    .sum();

注意事项: - 数据源要易拆分(数组、ArrayList 好,支持按下标快速二分;LinkedList 差,拆分本身要遍历,抵消并行收益)。 - 操作须无状态、无副作用,否则数据竞争(不要在 lambda 里改共享变量)。 - 小数据集或廉价操作,并行的拆分/合并/调度开销可能得不偿失——并行流的收益只有在”单元素处理成本 × 元素数量”显著大于”拆分+调度+合并”的固定开销时才划算。 - 所有并行流共享同一 commonPool,一个慢任务(尤其是阻塞 IO)会占住线程拖累全局其他并行流任务;IO 密集场景不要用它。

Optional

Optional<T> 表达”可能没有值”,替代裸 null,强制调用者处理缺失,减少 NPE。类比 C++17 std::optional(但 Java 的是对象,有堆分配开销)。

Optional 相比”直接返回可能是 null 的 T”解决的具体问题是:null 在类型签名上和正常值毫无区别(String getName() 既可能返回名字,也可能悄悄返回 null,调用方从签名上看不出来,只能靠文档或经验),调用方一旦忘记判空就是运行时 NPE,且往往在离”真正缺值”那处代码很远的地方才爆出来。Optional<String> 把”可能没有值”这个事实写进了方法签名本身,调用方被迫要么显式处理”空”这个分支(orElse/map/ifPresent),要么显式选择 get()/orElseThrow() 承担风险——把”这里可能没有值”从一条容易被忽略的隐性约定,变成编译器能看见的类型信息。

最基础的用法:Optional.of 包一个确定存在的值,Optional.empty() 表示确定没有值,isPresent() 判断是哪种情况:

Optional<String> opt1 = Optional.of("hello");   // 明确有值
Optional<String> opt2 = Optional.empty();       // 明确没有值
System.out.println(opt1.isPresent());           // true
System.out.println(opt2.isPresent());           // false

实际代码里更常见的是用 Optional.ofNullable 包一个”可能是 null 的值”(比如一个可能查不到结果的方法返回值),再配合下面这些方法处理”有”或”没有”两种情况,而不必显式写 if (x != null):

Optional<String> opt = Optional.ofNullable(getName());
String s = opt.orElse("default");             // 无值给默认
String t = opt.orElseGet(() -> compute());    // 惰性默认(仅无值时算)
opt.ifPresent(System.out::println);
String u = opt.map(String::trim).filter(x -> !x.isEmpty()).orElse("-");
opt.orElseThrow(() -> new IllegalStateException("missing"));

用法约定:主要用作方法返回值表达”可能无结果”;不要用于字段、方法参数或集合元素——这几个场景里 Optional 本身也要占一次堆分配,而字段/参数是否可为空通常有别的手段表达(文档、@Nullable 注解、直接判空),为它们套一层 Optional 只是徒增一次对象分配和一层解包动作,没有换来签名上的额外清晰度。

数值工具

日常用 Math 的静态方法做基本运算;当 int/long/double 精度或范围不够用时(比如需要超过 64 位的整数,或金融计算不能有浮点误差),再用 BigInteger/BigDecimal。

Math.max(a, b); Math.floorMod(-1, 3); // 2(与 % 不同,结果符号随除数,恒为非负)
Math.addExact(a, b);                  // 溢出抛 ArithmeticException,而非静默回绕

// BigInteger:任意精度整数(不可变),类比自己实现的大数;运算返回新对象
BigInteger big = BigInteger.valueOf(2).pow(1000);
BigInteger f = big.multiply(BigInteger.TEN).mod(BigInteger.valueOf(7));

// BigDecimal:精确十进制,金融必用(double 无法精确表示 0.1)
BigDecimal price = new BigDecimal("0.10");    // 必须用 String 构造,new BigDecimal(0.1) 会带二进制误差
BigDecimal total = price.multiply(new BigDecimal("3"));  // 精确 0.30
System.out.println(new BigDecimal("0.1").add(new BigDecimal("0.2"))); // 0.3(double 会是 0.30000000000000004)
total = total.setScale(2, RoundingMode.HALF_UP);         // 显式指定舍入

BigDecimal 必须用字符串构造而不是 double 字面量,原因在浮点数的表示本身:double 是二进制浮点数,0.1 这样的十进制小数在二进制里是无限循环小数,只能被近似表示——new BigDecimal(0.1) 拿到的不是数学意义上的 0.1,而是这个 double 近似值本身精确对应的二进制小数(展开出一长串非零位),BigDecimal 只是如实呈现了这个已经存在的误差,并没有引入新误差。用字符串构造则是直接按十进制精确解析,避免了”先转换成有损的二进制浮点数、再转回来”这一步。

// Random:线性同余伪随机;并发用 ThreadLocalRandom;安全场景用 SecureRandom
int r = ThreadLocalRandom.current().nextInt(0, 100);

BigInteger/BigDecimal 不可变,频繁运算产生大量临时对象,循环中注意 GC 压力。它们的比较用 compareTo,BigDecimal 的 equals 还会比较 scale(2.0 != 2.00),用 compareTo == 0 判数值相等。

小结

  • 用法:
    • Collections 是操作 List/Collection 的静态方法集合:sort/binarySearch/reverse/shuffle/max;不可变视图用 unmodifiableXxx(包装原集合)或 List.of(真正的不可变副本)。
    • Arrays.sort 对基本类型数组和对象数组调用方式一样,内部自动选算法。
    • 给对象排序:类自己实现 Comparable(只有一种自然顺序),或传入外部的 Comparator(可以有任意多种),JDK8+ 用 Comparator.comparing().thenComparing().reversed() 声明式组合。
    • PriorityQueue 就是堆,默认小顶堆,offer/peek/poll 增删查堆顶。
    • TreeMap/TreeSet 按 key 天然有序,多了 firstKey/floorKey/ceilingKey/headMap/subMap 等 HashMap 没有的导航方法。
    • Stream 的骨架是”源 .stream() → 若干中间操作(filter/map/sorted)→ 一个终止操作(collect/count/forEach)“;collect 配合 Collectors 做分组、分区、拼接、统计。
    • Optional<T> 用来表达”可能没有值”,主要当方法返回值用,配合 orElse/map/ifPresent 处理,不要用在字段或参数上。
    • 超出 long 范围用 BigInteger,需要精确十进制(金融场景)用 BigDecimal——BigDecimal 必须用字符串构造,比较数值相等要用 compareTo == 0 而不是 equals。
  • 原理:
    • Arrays.sort:基本类型用双轴快排(相等的值本就无差别,稳定性无意义,原地、缓存友好);对象用 TimSort(归并+插入的混合,识别已排序的 run,保证稳定,最坏 O(n log n)、已有序数据 O(n),需要 O(n) 辅助空间)——两种选择分别对应”比较廉价、访存主导”与”比较昂贵、稳定性刚需”这两种截然不同的成本结构。
    • Comparator/Comparable 的契约要求自反、传递、反对称,违反传递性会让 TimSort 直接抛 IllegalArgumentException,因为归并排序的正确性证明本身依赖比较关系的传递性。
    • PriorityQueue 底层是数组表示的二叉堆(节点 i 的孩子是 2i+1/2i+2),只维护”父≤子”这一条弱不变式,比红黑树维护更少信息、常数更小,适合”只需反复取最值”的场景;offer 上浮、poll 下沉,均为 O(log n),非线程安全。
    • TreeMap 底层是红黑树,靠”根黑、红节点子必黑、任一路径黑高相同”等性质保证近似平衡(而非 AVL 那样的严格平衡),用查找路径稍长换取插入删除时更少的旋转次数,所有操作 O(log n)。
    • Stream 的中间操作全部惰性,直到终止操作才真正执行:这避免了每步操作都分配一个中间集合(操作能在一次遍历里融合),且能短路(limit/findFirst 提前结束,这也是处理无限流的前提);Collector 被设计成”供应—累加—合并—收尾”四个函数打包的对象而非方法重载,是为了让分组、收集策略可以任意组合、可以嵌套,同时天然支持并行流的局部结果合并。并行流靠公共 ForkJoinPool 的工作窃取调度分治,要求数据源易拆分、操作无状态无副作用。
    • BigInteger/BigDecimal 都是不可变对象,每次运算生成新对象,循环里大量使用要留意 GC 压力;BigDecimal 必须用字符串构造是因为要避免十进制小数先经过 double 的二进制近似这一步有损转换。
