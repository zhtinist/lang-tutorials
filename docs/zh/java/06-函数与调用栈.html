函数与调用栈:方法与参数传递的真相

在 C++ 里你有全局函数、成员函数、静态成员函数、函数指针、引用参数、std::move……Java 把这一切收窄成一个概念:方法(method)。方法永远属于某个类,没有游离于类之外的函数。但收窄不代表简单——方法调用背后的栈帧布局、分派机制和只有值传递这条铁律,才是本章要拆开的底层真相。

方法定义与重载

方法签名 = 方法名 + 参数类型列表(不含返回类型,这点和 C++ 一致)。重载(overload)在编译期根据实参的静态类型选定目标,与 C++ 的重载决议同理,但 Java 没有默认参数,只能靠多个重载或可变参数模拟。

public class Overload {
    static void f(int x)    { System.out.println("int: " + x); }
    static void f(long x)   { System.out.println("long: " + x); }
    static void f(Object x) { System.out.println("Object: " + x); }

    public static void main(String[] args) {
        f(1);        // int:任何时候优先精确匹配
        f(1L);       // long
        f("hi");     // Object
        // 重载决议顺序:精确匹配 → 拓宽(widening) → 装箱(boxing) → 可变参数
        byte b = 5;
        f(b);        // int:byte 拓宽成 int,而不是装箱成 Object
    }
}

关键:重载决议在编译期完成,javac 直接把调用点绑定到某个具体签名。运行期的对象实际类型对重载毫无影响——这与下文的重写(override)是两码事。

Java 只有值传递(反复强调)

C++ 里你可以显式区分值传递、指针传递、引用传递。Java 只有一种:值传递(pass-by-value)。争议全部来自”引用类型传的是什么值”。

答案:传的是引用值(reference,可理解为对象地址)的一份副本。不是对象本身,也不是 C++ 意义上的”引用别名”。

  • 基本类型:拷贝那 4/8 字节的数值。
  • 引用类型:拷贝那个”指向堆对象的引用值”(HotSpot 64 位默认开启指针压缩,通常是 4 字节的 oop)。

副本和原变量指向同一个堆对象,所以你能改对象内容;但重新赋值副本,原变量纹丝不动。

swap 反例:证明改不了调用方的指向

public class Swap {
    static class Box { int v; Box(int v) { this.v = v; } }

    // 试图交换两个引用——注定失败
    static void swap(Box a, Box b) {
        Box t = a; a = b; b = t;   // 只交换了副本 a、b 的指向
    }

    // 能改对象内容——成功
    static void fill(Box x) {
        x.v = 999;                 // 通过副本引用写堆里的字段
    }

    public static void main(String[] args) {
        Box p = new Box(1), q = new Box(2);
        swap(p, q);
        System.out.println(p.v + " " + q.v); // 1 2:调用方指向没变
        fill(p);
        System.out.println(p.v);             // 999:对象内容被改了
    }
}

内存图(swap 执行到 b = t; 时):

   main 栈帧                     swap 栈帧
 +----------+                 +----------+
 |  p ------|----+       +----|  a ------|--> 指回 Box(2)
 |  q ------|--+ |       |    |  b ------|--> 指回 Box(1)
 +----------+  | |       |    |  t ------|--> Box(1)
               | |       |    +----------+
               | +-------|-----------+
               |         |           |
               v         |           v
        堆: Box(1) <-----+    堆: Box(2)
             v=999            v=2

main 里的 p、q 从头到尾指向不变;swap 交换的是它自己栈帧里的 a、b。方法返回后 swap 栈帧连同 a、b、t 一起弹栈销毁。

想在 Java 里实现真交换?只能返回新值、传入可变容器(数组/持有者对象),或用字段。没有 C++ 的 Box*& 或 Box&。

对照表:

场景 C++ Java
传基本类型 值/引用/指针可选 只有值(拷贝数值)
传对象 值(深/浅拷贝构造)/引用/指针 只有值(拷贝引用值,不拷贝对象)
改对象内容影响调用方 引用/指针可以 可以(共享同一堆对象)
改指向影响调用方 引用/指针可以 不可能

栈帧结构

每次方法调用,JVM 在当前线程的Java 虚拟机栈上压入一个栈帧(frame)。这与 C++ 的调用栈概念一致,但结构是 JVM 规范明确规定的:

线程栈(高地址在下,向下增长示意)
 +--------------------------------------+
 | 栈帧 A (main)                        |
 |  局部变量表 Local Variable Table     |  <- 类似 C++ 局部变量+参数区
 |  操作数栈    Operand Stack            |  <- 字节码计算用的工作区(C++ 用寄存器/隐式栈)
 |  帧数据:返回地址、指向常量池的引用   |
 +--------------------------------------+
 | 栈帧 B (被 A 调用的方法)             |  <- 栈顶,当前活动帧
 |  ...                                 |
 +--------------------------------------+
  • 局部变量表:数组式结构,按 slot 编号。long/double 占 2 个 slot,其余占 1 个。实例方法的 slot 0 是 this。方法的最大局部变量数在编译期算好,写进 Code 属性的 locals。
  • 操作数栈:JVM 是栈式虚拟机(不同于 x86 的寄存器机)。iadd 从操作数栈弹两个 int、相加、压回结果。最大深度也在编译期确定(stack)。
  • 帧数据:返回地址(调用者的下一条指令)、异常表引用等。

用 javap 看真相:

// Frame.java
public class Frame {
    static int add(int a, int b) { int c = a + b; return c; }
}
$ javac Frame.java && javap -c -v Frame.class
static int add(int, int);
  Code:
   stack=2, locals=3, args_size=2   // 操作数栈深2,局部变量表3个slot(a,b,c)
     0: iload_0        // 把 slot0(a)压入操作数栈
     1: iload_1        // 把 slot1(b)压入
     2: iadd           // 弹两个,相加,压回
     3: istore_2       // 弹出存入 slot2(c)
     4: iload_2        // 再压 c
     5: ireturn        // 弹出作为返回值

返回值通过操作数栈传递:被调方 ireturn 弹出栈顶,JVM 将其压入调用方栈帧的操作数栈。参数传递则相反:调用方把实参压在自己操作数栈上,invoke* 指令消费它们并填入新帧的局部变量表低位 slot。

方法分派:静态绑定 vs 动态绑定

C++ 里非虚函数是静态绑定,虚函数走 vtable。Java 的对应关系更规整,由四条字节码指令体现:

指令 用途 绑定时机
invokestatic 静态方法 静态(编译期)
invokespecial 构造器、private、super. 调用 静态
invokevirtual 普通实例方法 动态(运行期查 vtable)
invokeinterface 接口方法 动态(查 itable)
invokedynamic lambda、方法引用、动态语言 运行期首次链接后固化

静态绑定

static、private、final 方法,以及重载决议,都在编译期定死。final 方法虽然用 invokevirtual,但 JIT 可以去虚化(devirtualize)并内联,因为它不可能被重写。

动态绑定与 vtable

普通实例方法调用编译成 invokevirtual。运行期 JVM 通过对象头里的类型指针(klass pointer)找到实际类的方法元数据,在其虚方法表(vtable)中按固定索引取出目标方法入口。这与 C++ 的 vptr → vtable 机制几乎一模一样,区别在于:

  • C++:对象里存 vptr,vtable 在只读数据段。
  • HotSpot:对象头(mark word + klass pointer)里的 klass pointer 指向元空间(Metaspace)的 InstanceKlass,vtable 紧跟其后。
对象(堆)              InstanceKlass(元空间)
+-------------+       +------------------+
| mark word   |       | ... 元数据 ...   |
| klass ptr --|------>| vtable[0] -> ...  |
+-------------+       | vtable[1] -> Dog.speak
| 实例字段    |       | vtable[2] -> ...  |
+-------------+       +------------------+
public class Dispatch {
    static class Animal { void speak() { System.out.println("..."); } }
    static class Dog extends Animal { @Override void speak() { System.out.println("Woof"); } }

    public static void main(String[] args) {
        Animal a = new Dog();
        a.speak();   // invokevirtual:静态类型 Animal,运行期查到 Dog 的 vtable 槽 -> "Woof"
    }
}

invokeinterface 稍慢:接口方法在实现类里的 vtable 索引不固定,需查 itable(interface method table)或做搜索,JIT 会用内联缓存(inline cache)优化。

递归与 StackOverflowError

每层递归压一个栈帧。Java 栈容量由 -Xss 决定(默认约 512KB~1MB,平台相关)。栈耗尽抛 StackOverflowError(注意是 Error 不是 Exception,见第 9 章)。JVM 不做尾调用优化(TCO),即便是尾递归也照样吃栈——这是与许多函数式语言、部分 C++ 编译器的重要区别。

public class Recur {
    static long depth = 0;
    static void dive() { depth++; dive(); }   // 无终止,故意撑爆栈
    public static void main(String[] args) {
        try { dive(); }
        catch (StackOverflowError e) {
            System.out.println("爆栈深度约: " + depth); // 每帧越大,层数越少
        }
    }
}

深递归应改写为迭代 + 显式 Deque 模拟栈,把帧数据从线程栈搬到堆上。

可变参数(varargs)

T... 在字节码里就是 T[]。编译器在调用点自动把散列实参打包成一个堆上的数组——所以每次调用都可能有一次数组分配。

public class Varargs {
    static int sum(int... xs) {          // 编译为 int[] xs
        int s = 0; for (int x : xs) s += x; return s;
    }
    public static void main(String[] args) {
        System.out.println(sum(1, 2, 3)); // 编译器生成 new int[]{1,2,3}
        System.out.println(sum());        // 传空数组 new int[]{},不是 null
        int[] arr = {4, 5};
        System.out.println(sum(arr));     // 直接传数组,不再包一层
    }
}

重载 + varargs 时,固定参数版本优先于 varargs 版本(varargs 是决议的最后一档)。

lambda 与方法引用:invokedynamic 的舞台

C++ 的 lambda 在编译期生成一个匿名闭包类(带 operator())。Java 没有在编译期为每个 lambda 生成 .class,而是用 invokedynamic 把实现推迟到运行期。

import java.util.function.IntBinaryOperator;
public class Lambda {
    public static void main(String[] args) {
        IntBinaryOperator add = (a, b) -> a + b;   // 编译成一个 invokedynamic 调用点
        System.out.println(add.applyAsInt(3, 4));
    }
}

javap -c 会看到 invokedynamic #N, 0 与一个引导方法(bootstrap method)LambdaMetafactory.metafactory。机制:

  1. lambda 体本身被 javac 编译成宿主类里的一个私有静态/实例方法(如 lambda$main$0)。
  2. 首次执行到该 invokedynamic 时,JVM 调用引导方法,由 LambdaMetafactory 在运行期动态生成一个实现了 IntBinaryOperator 的类,其抽象方法转调那个私有方法。
  3. 生成的实现被绑定到该调用点的 CallSite,后续调用直接命中,近乎零开销。

好处:避免”每个 lambda 一个类文件”的膨胀,把生成策略留给运行时(未来可换成更优实现,而字节码不用变)。方法引用(String::length、System.out::println、ArrayList::new)走完全相同的机制,只是目标方法直接指向已有方法而非合成方法。

捕获变量(capture)方面:被 lambda 捕获的局部变量必须是 effectively final(事实最终)。因为捕获的是变量值的副本(引用类型则是引用值副本),在生成实例时作为构造参数存入——这和 C++ 的 [=] 按值捕获语义相近,而 Java 不允许 [&] 式的按引用捕获局部变量(局部变量在栈上,方法返回即失效,捕获引用会悬空)。

小结:方法属于类;传参永远是值传递(引用类型传引用值的副本);栈帧由局部变量表 + 操作数栈 + 帧数据构成;invokestatic/special 静态绑定,invokevirtual/interface 走 vtable/itable 动态分派,invokedynamic 承载 lambda 的运行期生成。下一章我们把镜头转向异常与内存回收——当栈帧因异常提前展开、当堆对象无人引用时,JVM 如何收场。
