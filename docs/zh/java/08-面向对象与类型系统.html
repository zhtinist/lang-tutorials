面向对象与类型系统

你在 C/C++ 里可能从没写过 class,或者写过但没深究过”一个对象在内存里到底是什么”。本章先讲「怎么用」——怎么定义一个类、怎么 new 一个对象、怎么写继承和接口、怎么声明最简单的泛型和 lambda,不需要你懂任何底层就能上手;再进入「底层与 C 对比」——看对象在堆上到底摆了什么,虚方法怎么分派,泛型运行时剩下什么。 只想会用的话,读每节的前半段即可;想搞懂运行机制,再往下看后半段。

这是全书里 OOP 概念最集中的一章,涉及类、继承、多态、泛型、record、sealed、enum、lambda 十来个话题。别指望一遍读完全部记住——先把「怎么用」串起来,能写基本的类和接口,再逐节回来啃”底层与 C 对比”。

先会用:类与对象速览

定义一个类,new 一个对象

Java 的类和 C 的 struct 最大的不同:类里可以直接放函数(叫”方法”),new 出来的对象永远在堆上,变量里存的是指向它的引用(细节见「变量与内存模型」)。

class Point {
    int x, y;                          // 字段(成员变量)
    Point(int x, int y) {              // 构造器:和类同名,没有返回类型
        this.x = x;                    // this 指当前对象,区分字段和参数
        this.y = y;
    }
    int sum() {                        // 方法
        return x + y;
    }
}

public class PointBasics {
    public static void main(String[] args) {
        Point p = new Point(3, 4);     // new 出一个实例
        System.out.println(p.sum());  // 调用方法:7
        System.out.println(p.x);      // 直接访问字段:3
    }
}
7
3

class Point { ... } 只是”图纸”;new Point(3, 4) 才真正在堆上造出一个对象,p 里存的是这个对象的引用。同一张图纸能 new 出无数个对象,彼此字段互不影响。

最简单的继承与接口

extends 表示”继承一个类”(单继承,只能写一个),implements 表示”实现一个或多个接口”:

interface Greeter {
    String greet();                    // 接口方法:只有签名,没有实现
}

class Animal {
    String name = "animal";
    String speak() {
        return name + " makes a sound";
    }
}

class Dog extends Animal implements Greeter {
    Dog() { name = "dog"; }
    @Override public String greet() { return "woof from " + name; }
}

public class InheritBasics {
    public static void main(String[] args) {
        Dog d = new Dog();
        System.out.println(d.speak());   // 继承来的方法:dog makes a sound
        System.out.println(d.greet());   // 实现接口的方法:woof from dog
    }
}
dog makes a sound
woof from dog

Dog 既拿到了 Animal 的字段和方法(继承),又必须把 Greeter 承诺的 greet() 写出来(实现接口)——@Override 是给编译器和读者的标记,表示”这是在覆盖父类/接口的方法”,拼错方法名编译器会立刻报错。

最简单的泛型:Box<T>

类名后面的 <T> 表示”这个类可以装任意类型,具体装什么由使用者决定”:

class Box<T> {
    T value;
    Box(T value) { this.value = value; }
    T get() { return value; }
}

public class BoxBasics {
    public static void main(String[] args) {
        Box<String> b = new Box<>("hello");
        System.out.println(b.get());       // hello

        Box<Integer> n = new Box<>(42);
        System.out.println(n.get());       // 42
    }
}
hello
42

Box<String> 和 Box<Integer> 是同一份代码,T 在使用处被替换成具体类型;编译器负责检查类型对不对,你不需要为每种类型各写一个 Box。

最简单的 lambda

Runnable 是一个只有一个方法 run() 的接口(这类接口叫”函数式接口”);lambda () -> ... 就是给这个方法直接写一段实现,不用专门定义一个类:

public class LambdaBasics {
    public static void main(String[] args) {
        Runnable r = () -> System.out.println("running!");
        r.run();                          // 像调用普通方法一样调用
    }
}
running!

这些够你开始读懂大多数 Java 代码了。下面进入”底层”——对象在堆上到底长什么样,虚方法怎么分派,泛型运行时还剩下什么。

开始深入:对象在内存里到底是什么

从这里开始讲底层。先记住一件事,后面每一节都会用到:Java 对象一律在堆上分配,对象开头有一个隐藏的”对象头”,其中一部分指向它所属的类描述结构(HotSpot 里叫 klass)——这个指针的作用和 C++ 里的 vptr 很像:决定了调用虚方法时该跳到哪段代码。变量本身不存对象,只存这个引用(值/引用语义的细节见「变量与内存模型」;对象头的具体字节布局见「数据类型」)。

带着这个模型,我们逐一拆开:初始化顺序、封装、继承、多态(vtable)、抽象类 vs 接口、Object、泛型擦除、record、sealed、enum、内部类与 lambda。

一、类、对象、初始化顺序

怎么用

构造器可以用 this(...) 委托本类的另一个构造器,也可以用 super(...) 显式调用父类构造器(不写的话编译器会自动插入无参 super()):

class Person {
    String name;
    int age;
    Person(String name) { this(name, 0); }          // this(...):委托本类另一个构造器
    Person(String name, int age) { this.name = name; this.age = age; }
}
class Student extends Person {
    String school;
    Student(String name, int age, String school) {
        super(name, age);                            // super(...):调用父类构造器
        this.school = school;
    }
}
public class ConstructorBasics {
    public static void main(String[] args) {
        Student s = new Student("Ann", 20, "MIT");
        System.out.println(s.name + " " + s.age + " " + s.school);  // Ann 20 MIT

        Person p = new Person("Bob");                // 走 this(...) 委托,age 默认 0
        System.out.println(p.name + " " + p.age);     // Bob 0
    }
}
Ann 20 MIT
Bob 0

底层与 C 对比

Java 类的所有实例方法默认是”虚”的(除 private/static/final)。new 一个对象时,JVM 按固定顺序执行:

1. 类首次使用时:静态字段默认零值 → 静态初始化块 + 静态字段赋值(按源码顺序,仅一次)
2. 实例创建:分配堆内存,所有实例字段置零值
3. 父类实例初始化(递归:父类字段 → 父类实例块 → 父类构造器体)
4. 本类实例字段默认值 → 实例初始化块 + 字段赋值(按源码顺序)
5. 本类构造器体
class Base {
    String x = init("Base.x");        // ③
    Base() { System.out.println("Base()"); }
    static String s = init("Base.s");  // ① 类加载时
    static String init(String m){ System.out.println(m); return null; }
}
class Derived extends Base {
    String y = Base.init("Derived.y"); // ④
    Derived() { super(); System.out.println("Derived()"); } // super() 隐式在最前
}
// new Derived() 输出:Base.s(首次) → Base.x → Base() → Derived.y → Derived()
Base.s
Base.x
Base()
Derived.y
Derived()

对比 C++:成员按声明顺序在初始化列表构造,基类先于派生类——Java 顺序类似,但没有初始化列表语法,且静态初始化是懒惰的(类首次主动使用时才触发,由类加载器保证线程安全地执行一次)。

构造器第一行必须是 this(...) 或 super(...);不写则编译器插入无参 super()。构造期间调用虚方法很危险:父类构造器里调被子类重写的方法,此时子类字段尚未初始化(还是零值),会读到未初始化状态。C++ 在构造期虚调用会静态绑定到当前类版本;Java 则动态绑定到子类版本,更危险。

亲眼看这个坑:父类构造器调用虚方法 size(),分派到子类版本,但此时子类字段 value 还是零值:

public class CtorVirtual {
    static class Base {
        Base() { System.out.println("Base ctor sees size = " + size()); } // 虚调用
        int size() { return 0; }
    }
    static class Derived extends Base {
        int value = 42;                 // 在 super() 之后才初始化
        @Override int size() { return value; }
    }
    public static void main(String[] args) {
        Derived d = new Derived();
        System.out.println("after ctor  size = " + d.size());
    }
}
Base ctor sees size = 0    <- 动态分派到 Derived.size(),但 value 尚未赋值(零值)
after ctor  size = 42      <- 构造完成后才是 42

二、封装

怎么用

访问修饰符控制”谁能访问这个字段/方法”,最常见的用法是把字段设为 private,只暴露 getter/setter:

class Account {
    private double balance;   // 外部不可直接访问

    Account(double initial) { balance = initial; }

    public double getBalance() {     // 只能通过方法读取
        return balance;
    }
    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // 借机会做校验
    }
}

public class EncapBasics {
    public static void main(String[] args) {
        Account a = new Account(100);
        a.deposit(50);
        System.out.println(a.getBalance());  // 150.0
        // a.balance = -999;    // 编译错:balance 是 private,外部访问不到
    }
}
150.0

底层与 C 对比

访问修饰符从窄到宽:private < 包级(默认,无关键字) < protected < public。注意 Java 的 protected 还允许同包访问,语义比 C++ 宽。无 friend。模块系统(JPMS)可在更粗粒度上封装整个包。

三、继承:单继承 + 多接口

怎么用

一个类只能 extends 一个父类,但可以 implements 任意多个接口,用逗号分隔:

abstract class AbstractBase {
    void base() { System.out.println("base"); }
}
class Service extends AbstractBase implements Runnable, AutoCloseable {
    @Override public void run() { System.out.println("running"); }
    @Override public void close() { System.out.println("closed"); }
}
public class ServiceCheck {
    public static void main(String[] args) {
        Service s = new Service();
        s.base();
        s.run();
        s.close();
    }
}
base
running
closed

底层与 C 对比

Java 类只能单继承(避免 C++ 菱形继承的复杂性与虚继承),但可实现任意多个接口。

接口曾经只能有抽象方法,JDK 8 起可有 default 方法(带实现)和 static 方法,JDK 9 起可有 private 方法。这带来”多继承实现”的能力,但字段仍只有常量(public static final),所以没有状态菱形问题。若两个接口有同签名 default 方法,子类必须显式覆盖解决冲突,否则编译错:

interface A2 { default String who() { return "A"; } }
interface B2 { default String who() { return "B"; } }
class C2 implements A2, B2 { }   // 编译错
error: types A2 and B2 are incompatible
  class C2 inherits unrelated defaults for who() from types A2 and B2

用 接口名.super.方法名() 显式指明走哪个接口的默认实现来解决冲突:

interface A { default String who() { return "A"; } }
interface B { default String who() { return "B"; } }
class C implements A, B {
    @Override public String who() { return A.super.who() + B.super.who(); } // 显式解决冲突
}
AB

四、多态与虚方法表(vtable)

怎么用

子类重写(@Override)父类的方法后,即使用父类类型的引用去调用,实际执行的也是子类版本——这就是”动态分派”:

class Bird {
    String sound() { return "..."; }
}
class Duck extends Bird {
    @Override String sound() { return "quack"; }
}

public class PolyBasics {
    public static void main(String[] args) {
        Bird b = new Duck();          // 父类型引用指向子类对象
        System.out.println(b.sound()); // quack —— 运行时按实际对象类型分派,而非引用声明类型
    }
}
quack

底层与 C 对比

Java 实例方法默认动态分派。HotSpot 的实现和 C++ 的 vtable 高度相似:

每个类在方法区(Metaspace)有一个 klass 结构,内含 vtable:
  Klass(Animal)          Klass(Dog extends Animal)
  vtable:                vtable:
   [0] Object.方法...      [0] Object.方法...(继承)
   [k] Animal::speak ─┐    [k] Dog::speak    ← 覆盖,槽位相同
                      │
对象头的 klass ptr ───┘ 指向自己的 Klass
调用 a.speak():
   1. 取 a 的对象头 klass ptr
   2. 到 vtable 固定槽位取方法地址
   3. 调用   —— 与 C++ 几乎一致

差异:

  • 接口方法不能用固定 vtable 槽位(一个类实现多接口,槽位无法统一),用单独的 itable(interface method table)+ 运行时查找,比 vtable 稍慢。JIT 用内联缓存(inline cache)优化:记住上次调用的实际类型,单态时几乎零开销。
  • Java 的 final/private/static 方法是静态绑定(invokespecial/invokestatic),无虚分派。
  • JIT 还会做去虚化(devirtualization):若能证明某调用点只有一个实现(CHA,类层级分析),直接内联,STW 时若加载了新子类再退优化。C++ 需要手动 final 或链接期优化才有类似效果。

字节码层面四条调用指令:invokevirtual(虚)、invokeinterface(接口)、invokespecial(构造器/私有/super)、invokestatic(静态),外加 invokedynamic(lambda、字符串拼接)。详见「方法与调用栈」。

五、抽象类 vs 接口

怎么用

abstract class 可以混合”抽象方法”(没有方法体,子类必须实现)和”具体方法”(有实现,直接继承用);接口在 Java 8 之前只能有抽象方法:

abstract class Shape {
    abstract double area();          // 抽象方法,没有方法体
    void describe() {                // 具体方法,可以有实现
        System.out.println("area = " + area());
    }
}
class Circle extends Shape {
    double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
}

interface Movable {
    void move();                     // 抽象方法
}
class Car implements Movable {
    @Override public void move() { System.out.println("car moves"); }
}

public class AbstractVsInterfaceBasics {
    public static void main(String[] args) {
        new Circle(2).describe();
        new Car().move();
    }
}
area = 12.566370614359172
car moves

底层与 C 对比

维度 抽象类 接口
实例字段 可有(带状态) 只有 public static final 常量
构造器 有 无
继承 单继承 多实现
方法 抽象 + 具体 抽象 + default + static + private
语义 “是什么”(is-a) “能做什么”(can-do,能力)

经验:表达类型层级共享状态用抽象类;表达能力/契约、需要多重组合用接口。JDK 里 AbstractList 是抽象类骨架,List 是接口。

六、Object 的方法与 equals/hashCode 契约

怎么用

所有类隐式继承 Object。最常重写的三个方法是 equals(判断”值相等”)、hashCode(配合 HashMap/HashSet)、toString(打印时看到的文字):

import java.util.Objects;

class Point2 {
    int x, y;
    Point2(int x, int y) { this.x = x; this.y = y; }

    @Override public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point2 p)) return false;
        return x == p.x && y == p.y;
    }
    @Override public int hashCode() { return Objects.hash(x, y); }
    @Override public String toString() { return "Point2(" + x + ", " + y + ")"; }
}

public class EqualsBasics {
    public static void main(String[] args) {
        Point2 a = new Point2(1, 2);
        Point2 b = new Point2(1, 2);
        System.out.println(a.equals(b));   // true
        System.out.println(a == b);        // false —— 不同对象
        System.out.println(a);             // Point2(1, 2) —— toString 被调用
    }
}
true
false
Point2(1, 2)

== 永远比较的是引用(是不是同一个对象);要比较”内容是否相等”,必须重写 equals。System.out.println(a) 会自动调用 a.toString()。

底层与 C 对比

Object 类比但强于 C++——C++ 没有统一根。关键方法:equals、hashCode、toString、getClass、clone、wait/notify(见并发)、finalize(已弃用,见GC)。

equals/hashCode 契约(违反会导致集合行为错乱)

  1. a.equals(b) 为 true ⟹ a.hashCode() == b.hashCode()(相等必同哈希)。
  2. 反之不要求(哈希相同不必相等,即允许冲突)。
  3. equals 需自反、对称、传递、一致、非 null。

只重写 equals 不重写 hashCode,放进 HashMap/HashSet 就会因散到错误的桶而丢失(见「集合框架」)。默认 Object.equals 是引用相等(==),默认 hashCode 是标识哈希(identity hash)——常被误说成”由对象地址派生”,但那站不住脚(GC 移动对象后哈希就会变);现代 HotSpot(JDK8+ 默认)用一个线程本地的 xorshift 伪随机数生成,首次调用时惰性算出、存进对象头 mark word,此后不变(见「数据类型」)。

final class Point {
    final int x, y;
    Point(int x, int y){ this.x=x; this.y=y; }
    @Override public boolean equals(Object o){
        if (this == o) return true;
        if (!(o instanceof Point p)) return false; // 模式匹配 instanceof
        return x == p.x && y == p.y;
    }
    @Override public int hashCode(){ return Objects.hash(x, y); }
}

用 record 可自动生成这些(下文)。

违反契约的代价可直接看到:只重写 equals、不重写 hashCode,相等对象散到不同桶,HashSet 里就找不回来:

import java.util.*;
public class HashContract {
    // 重写了 equals,但没重写 hashCode —— 违反契约
    static class Bad {
        final int id; Bad(int id){ this.id = id; }
        @Override public boolean equals(Object o){ return o instanceof Bad b && b.id == id; }
    }
    // 两个都重写 —— 正确
    static class Good {
        final int id; Good(int id){ this.id = id; }
        @Override public boolean equals(Object o){ return o instanceof Good g && g.id == id; }
        @Override public int hashCode(){ return Integer.hashCode(id); }
    }
    public static void main(String[] args) {
        System.out.println("Bad  equal? " + new Bad(1).equals(new Bad(1)));
        Set<Bad> bad = new HashSet<>(); bad.add(new Bad(1));
        System.out.println("Bad  contains equal obj? " + bad.contains(new Bad(1)));

        Set<Good> good = new HashSet<>(); good.add(new Good(1));
        System.out.println("Good contains equal obj? " + good.contains(new Good(1)));
    }
}
Bad  equal? true                 <- 两个 Bad(1) 相等
Bad  contains equal obj? false   <- 但默认 hashCode 不同,散到别的桶,找不到!
Good contains equal obj? true    <- 正确重写后才能找回

七、泛型与类型擦除(与 C++ 模板的根本分野)

怎么用

声明泛型类用 class 类名<T>(上文的 Box<T> 已经见过);泛型方法把 <T> 写在返回类型前面,类型参数由调用时传入的实参自动推断:

public class GenericMethodBasics {
    static <T> T firstOf(T[] arr) {     // 泛型方法:类型参数写在返回类型前
        return arr[0];
    }
    public static void main(String[] args) {
        String[] names = {"Ann", "Bob"};
        System.out.println(firstOf(names));   // Ann

        Integer[] nums = {10, 20};
        System.out.println(firstOf(nums));    // 10
    }
}
Ann
10

这是 Java 类型系统最”反直觉”的部分,也是与 C++ 差异最大之处,下面展开讲运行时到底发生了什么。

底层与 C 对比

擦除:运行时没有泛型信息

Java 泛型是编译期特性。编译后类型参数被”擦除”:List<String> 和 List<Integer> 运行时都是同一个 List 类(裸类型 List,T 擦除为 Object,有界则擦到上界)。编译器在需要处插入强制转换并做静态类型检查。

List<String> ls = new ArrayList<>();
ls.add("hi");
String s = ls.get(0);   // 编译器插入 (String) 转换
// 运行时:ls.getClass() == ArrayList.class,与 List<Integer> 完全相同

对比 C++:模板是单态化(monomorphization),vector<int> 与 vector<string> 编译出两份独立代码,运行时有各自的类型,zero-cost 抽象但代码膨胀。Java 只有一份代码,无膨胀,但运行时丢失了类型,由此产生一系列限制。

运行时两种泛型实例是同一个 Class 对象;裸类型更能暴露”运行时不再有类型检查”:

import java.util.*;
public class Erasure {
    public static void main(String[] args) {
        List<String>  ls = new ArrayList<>();
        List<Integer> li = new ArrayList<>();
        System.out.println(ls.getClass());                 // 都擦除成 ArrayList
        System.out.println(li.getClass());
        System.out.println("same Class object? " + (ls.getClass() == li.getClass()));
        // 运行时不保留类型参数,下面这行能存进去,直到取出来用才会暴露:
        List raw = li;                 // 裸类型
        raw.add("oops");               // 此处不抛 ClassCastException(仅编译告警)
        System.out.println("li now holds: " + li.get(0));  // 尽管声明是 List<Integer>
    }
}
class java.util.ArrayList
class java.util.ArrayList
same Class object? true
li now holds: oops

编译期在 raw.add("oops") 处会有一条 unchecked 警告,但不会阻止编译或运行时报错。

擦除带来的四大限制

// 1. 不能 new T[](无法确定运行时类型,协变数组会漏检)
// T[] arr = new T[10];              // 编译错
T[] arr = (T[]) new Object[10];       // 变通:强转,有 unchecked warning

// 2. 不能 new T()(不知道构造哪个类)
// T t = new T();                     // 编译错,需传 Supplier<T> 或 Class<T>

// 3. instanceof 不能带类型参数(运行时无此信息)
// if (x instanceof List<String>)     // 编译错
if (x instanceof List<?>) { }         // 只能用通配符

// 4. 静态字段不能用类型参数;不能 catch 泛型异常

桥接方法(bridge method)

擦除会破坏协变返回和泛型覆盖,编译器生成桥接方法弥合:

class Node<T> { T get(){...} void set(T t){...} }
class StrNode extends Node<String> {
    @Override String get(){...}        // 你写的
    // 编译器还偷偷生成:Object get(){ return get(); }  ← 桥接,匹配父类擦除后签名 Object get()
}

否则父类 Node 擦除后是 Object get(),子类 String get() 签名不同,虚分派对不上。桥接方法用 javap -v 可见,标记 ACC_BRIDGE ACC_SYNTHETIC——实测 StrNode.class 里确实能看到两个 get:一个 ()Ljava/lang/String;(你写的),一个 ()Ljava/lang/Object; 带 ACC_BRIDGE, ACC_SYNTHETIC 标志(编译器生成的)。

通配符:PECS 原则

泛型是不变的:List<String> 不是 List<Object>(和数组协变相反,更安全)。要放松用通配符:

  • ? extends T:上界,生产者(只能读出 T,不能写入,因为不知道确切子类型)。
  • ? super T:下界,消费者(只能写入 T,读出只能当 Object)。
  • 口诀 PECS:Producer-Extends, Consumer-Super。
// 从 src 读、往 dst 写:src 是生产者用 extends,dst 是消费者用 super
static <T> void copy(List<? super T> dst, List<? extends T> src){
    for (T t : src) dst.add(t);
}

C++20 concepts / 老式 SFINAE 解决的是类似”约束类型参数”的问题,但机制完全不同(编译期鸭子类型 vs 名义子类型 + 擦除)。

八、record:不可变数据载体

怎么用

record 一行声明字段,编译器自动生成构造器、访问器(名字就是字段名,不加 get 前缀)、equals/hashCode/toString:

record Point3(int x, int y) { }

public class RecordBasics {
    public static void main(String[] args) {
        Point3 p = new Point3(3, 4);
        System.out.println(p.x());      // 3   —— 访问器,不叫 getX()
        System.out.println(p.y());      // 4
        System.out.println(p);          // Point3[x=3, y=4] —— 自动生成的 toString
    }
}
3
4
Point3[x=3, y=4]

底层与 C 对比

record Point(int x, int y) { }
// 自动生成:private final 字段、全参构造器、访问器 x()/y()、equals/hashCode/toString

record 是隐式 final 的、字段 final、语义上是”透明的数据聚合”。类比 C++ 的聚合 struct + 自动生成的 operator==。可加紧凑构造器做校验:

record Range(int lo, int hi) {
    Range {   // 紧凑构造器,校验后自动赋值
        if (lo > hi) throw new IllegalArgumentException();
    }
}
Range r = new Range(1, 5);
System.out.println(r);           // Range[lo=1, hi=5]
new Range(5, 1);                 // 抛 IllegalArgumentException —— 紧凑构造器里的校验生效

九、sealed:密封类型(受控继承)

怎么用

sealed interface ... permits ... 明确列出”谁能实现它”,不在列表里的类无法实现:

sealed interface Vehicle permits Car2, Bike { }
record Car2(int wheels) implements Vehicle { }
record Bike() implements Vehicle { }

public class SealedBasics {
    public static void main(String[] args) {
        Vehicle v = new Car2(4);
        System.out.println(v);   // Car2[wheels=4]
    }
}
Car2[wheels=4]

底层与 C 对比

限制谁能继承/实现,配合模式匹配做穷尽性检查:

sealed interface Shape permits Circle, Square { }
record Circle(double r) implements Shape { }
record Square(double s) implements Shape { }

double area(Shape sh) {
    return switch (sh) {          // 编译器知道只有两种,无需 default
        case Circle c -> Math.PI * c.r() * c.r();
        case Square s -> s.s() * s.s();
    };
}
System.out.println(area(new Circle(2)));  // 12.566370614359172
System.out.println(area(new Square(3)));  // 9.0

类似 C++ 里用 std::variant + std::visit 表达封闭的和类型(sum type)。sealed 让编译器验证 switch 穷尽——少覆盖一种子类型,switch 表达式会编译报错而不是留一个运行时才暴露的漏洞。

十、enum:是完整的类

怎么用

最简单的枚举就是列一串常量名,像 C 里的 enum 但更安全——不能和整数混用,switch/== 都能直接用:

enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

public class EnumBasics {
    public static void main(String[] args) {
        Day d = Day.MON;
        System.out.println(d);            // MON
        System.out.println(d == Day.MON); // true —— 枚举常量可以直接用 == 比较

        for (Day day : Day.values()) {    // 遍历所有常量
            System.out.print(day + " ");
        }
        System.out.println();
    }
}
MON
true
MON TUE WED THU FRI SAT SUN

底层与 C 对比

Java 枚举不是 C/C++ 的整数常量,而是编译成继承 java.lang.Enum 的类,每个枚举常量是一个单例对象(JVM 保证)。可有字段、方法、构造器,甚至每个常量各自的方法体(常量特定行为):

enum Planet {
    EARTH(5.97e24, 6.37e6), MARS(6.42e23, 3.39e6);  // 调用构造器
    private final double mass, radius;
    Planet(double m, double r){ mass=m; radius=r; }
    double gravity(){ return 6.67e-11 * mass / (radius*radius); }
}
for (Planet p : Planet.values()) {
    System.out.println(p + " gravity = " + p.gravity());
}
EARTH gravity = 9.813440652193735
MARS gravity = 3.726159709713629

因是单例,枚举可安全用 == 比较,是实现单例模式的最佳方式,EnumMap/EnumSet 用序号做位图/数组,极高效。

十一、内部类、匿名类、lambda 与变量捕获

怎么用

函数式接口(只有一个抽象方法的接口,如内置的 Runnable)可以直接用 lambda 赋值,不需要专门写一个实现类:

Runnable r = () -> System.out.println("running!");
r.run();

底层与 C 对比

内部类种类

  • 静态嵌套类(static class):无外部实例引用,等价顶层类放里面。
  • 成员内部类:持有隐藏的 outer this 引用,能访问外部实例字段——这也是内存泄漏源(内部类实例存活会拖住外部实例)。
  • 局部类 / 匿名类:定义在方法内。

变量捕获必须 effectively final

匿名类/lambda 捕获的局部变量必须是 final 或 effectively final(初始化后不再赋值):

int base = 10;                 // effectively final
Runnable r = () -> System.out.println(base + 1);
// base = 20;                  // 若取消注释,上面编译错

为什么? 局部变量在栈帧里,方法返回后栈帧销毁;而 lambda/匿名类对象在堆上,可能活得更久。Java 的做法是把捕获的变量值拷贝进 lambda 对象(基本类型拷值,引用类型拷引用值),要求其不变以保证”栈上的原变量”和”堆里的拷贝”语义一致,避免二者不同步的困惑。

对比 C++ lambda:显式选择 [=] 按值捕获或 [&] 按引用捕获,按引用可捕获可变变量,但栈变量失效后就是悬垂引用(UB)——责任在你。Java 用 effectively final 从语言层堵死了这个坑(只能按值,且值不变)。捕获实例字段则不同:lambda 捕获的是 this,通过它访问字段,字段可变。

lambda 的底层:invokedynamic

lambda 不是匿名类语法糖。javac 生成 invokedynamic 指令,首次执行时由 LambdaMetafactory 在运行时动态生成实现函数式接口的类。好处:无捕获的 lambda 可复用同一实例(无状态),避免匿名类那种每处一个 class 文件的膨胀。方法引用(String::length)同理。

小结

  • 用法层面:class 类名 { 字段; 构造器; 方法; } 定义类,new 类名(...) 造对象,. 访问字段/调用方法;extends 单继承一个类,implements 实现任意多个接口;class Box<T> 声明泛型类,<T> T method(...) 声明泛型方法;函数式接口变量可以直接用 () -> ... 的 lambda 赋值;private 字段配 getter 是封装的基本写法;重写方法要加 @Override,子类引用父类类型的变量调用方法时按实际对象类型分派;abstract class 混合抽象和具体方法,接口表达”能做什么”;重写 equals 记得同时重写 hashCode;record Point(int x, int y) 一行声明不可变数据类;sealed interface ... permits ... 限定实现者;enum 列常量,配 switch/== 使用。
  • 底层层面:对象一律在堆上,对象头里的 klass 指针类比 C++ 的 vptr;初始化顺序是”父类静态 → 父类实例 → 父类构造器 → 本类实例 → 本类构造器”,构造期调用虚方法会踩到子类字段未初始化的坑;protected 比 C++ 语义宽,允许同包访问;接口 default 方法带来”多继承实现”但无状态菱形问题,冲突需显式覆盖解决;多态靠 vtable(类方法)+ itable(接口方法)实现,JIT 用内联缓存和去虚化优化;Object.hashCode 默认是线程本地伪随机生成的标识哈希,不是地址派生;泛型在编译期做类型检查,运行时被擦除成裸类型,由此产生不能 new T[]/new T()、instanceof 不能带类型参数等限制,编译器还会生成桥接方法弥合擦除导致的签名不一致;record/sealed/enum 都是编译器生成大量样板代码的语法糖,enum 常量本质是单例对象;lambda 捕获局部变量必须 effectively final(值拷贝语义),底层靠 invokedynamic + LambdaMetafactory 运行时生成实现类,而非匿名类语法糖。
