并发模型

C/C++ 里并发靠 std::thread、pthread、原子类型、memory_order 手动控制,程序员要自己在每处共享变量访问上决定用什么内存序。Java 把这一整套抽象进 JMM(Java Memory Model) 和 java.util.concurrent:语言规范统一定义了”什么样的读写顺序在多线程下是安全的”,工具类把常见并发模式(锁、计数器、线程池、异步编排)封装成可直接调用的组件。

线程创建与生命周期

创建线程要跑的”任务”,Java 里有三种写法:

// 1. 继承 Thread(不推荐:单继承名额被占用,任务与线程强耦合)
class MyThread extends Thread {
    @Override public void run() { System.out.println("run in " + getName()); }
}

// 2. Runnable:无返回值、无受检异常,最常用
Runnable r = () -> System.out.println("runnable");

// 3. Callable<V>:有返回值 V,可抛受检异常,配合 Future 使用(见 java.util.concurrent 一节)
Callable<Integer> c = () -> 42;

public class ThreadDemo {
    public static void main(String[] args) throws Exception {
        Thread t1 = new MyThread();
        t1.start();            // start() 让 JVM 新开一个线程去跑 run()
        Thread t2 = new Thread(r);
        t2.start();
        t1.join();             // 当前线程阻塞直到 t1 终止,类似 pthread_join
        t2.join();
    }
}
// 两个线程谁先打印不确定,但 join() 保证 main 一定等它们都跑完再结束

start() 让 JVM 新建一个线程并异步回调 run();如果直接调用 run(),那只是一次普通的方法调用,代码在当前线程里同步执行,根本没有开新线程。一个 Thread 对象只能 start() 一次,再调第二次会抛 IllegalThreadStateException。

Java 的 Thread 在 HotSpot 上默认1:1 映射到内核线程(OS thread),与 C++ std::thread 一样重。调用 start() 会触发一次 pthread_create,每个线程默认分到约 1MB 栈空间(-Xss 可调)。new Thread(...).start() 在系统层面的代价和 C 里手写 pthread_create 是同一个数量级——线程本身不是免费的资源,这一点是本章最后”虚拟线程”要解决的根本问题的起点。

Thread.State 六态,与 OS 的线程状态不是一一对应:

    NEW --start()--> RUNNABLE <----+
                      |  ^          | notify/超时/锁获得
    进入同步块拿不到锁 |  |          |
                      v  |          |
                   BLOCKED          |
    wait()/park()   |               |
                      v             |
    WAITING / TIMED_WAITING --------+
                      |
                   TERMINATED（run 返回）
  • RUNNABLE 同时涵盖”正在 CPU 上跑”和”就绪等调度”,Java 不区分(OS 才区分)。
  • BLOCKED:等待 synchronized 监视器锁。
  • WAITING:Object.wait()、Thread.join()、LockSupport.park() 无限等待。
  • TIMED_WAITING:带超时的上述调用、Thread.sleep(ms)。

两个线程同时改同一个变量会出错,先看一个具体后果,再解释根源:

public class SimpleSyncDemo {
    static int counter = 0;

    static synchronized void increment() {   // synchronized 方法:同一时刻只有一个线程能执行它
        counter++;
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable job = () -> {
            for (int i = 0; i < 100_000; i++) increment();
        };
        Thread t1 = new Thread(job);
        Thread t2 = new Thread(job);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter); // 200000,每次运行都一样
    }
}
// counter = 200000

两个线程各自把 counter 加 10 万次,加了 synchronized 结果就总是精确的 20 万。去掉 synchronized,counter++ 本身是”读取—加一—写回”三步,两个线程的这三步可能交叠,结果会不确定地小于 20 万——这背后的机制是接下来几节要讲的内容。

Java 内存模型(JMM)

JMM 是一份规范(JLS 第 17 章),不是某种硬件行为的直接描述。它定义多线程读写共享变量时,什么值是合法可见的。

为什么需要一份形式化规范,而不是让每个人凭直觉判断”看起来对就行”:直觉建立在单线程的顺序执行模型上——“代码从上到下执行,一个变量赋值之后就能被看到”。但编译器和 CPU 对单线程程序只保证 as-if-serial(重排后的结果和顺序执行的结果一致),这个保证的验证范围仅限于”这一个线程自己观察自己”。跨线程时,一个线程重排、缓存、延迟写回的操作,可能在任意时刻才对另一个线程可见,而 as-if-serial 完全不约束这种跨线程可见性——这是留给硬件和编译器自由优化的空间,不是它们的疏漏。凭直觉写的并发代码,在 x86(内存序较强)上可能”恰好”每次都跑对,换到 ARM(内存序弱得多,重排更激进)上或者换一个更激进内联/重排的 JIT 版本就可能出错——这种”在你的机器上能跑”、无法用常规测试稳定复现的 bug,只有一份精确规定”哪些顺序被保证、哪些不被保证”的规范才能事先预防,单靠测试无法证伪”这段代码在所有平台上都正确”。这正是 JMM(JSR-133,2004)存在的理由:2004 年之前 Java 的内存模型定义本身有内在矛盾(允许一些违反直觉、连规范作者都认为不应该发生的重排),JSR-133 的工作就是把”什么可以重排、什么不可以”重新形式化,让编译器实现者和应用开发者对着同一份契约做事,而不是各凭经验。

JMM 抽象出:每个线程有工作内存(working memory),共享变量存在主内存(main memory)。线程操作变量必须先从主内存拷贝到工作内存,改完再写回。

   +----------------+        +----------------+
   |   线程 A        |        |   线程 B        |
   | 工作内存        |        | 工作内存        |
   | [x=0 副本]      |        | [x=0 副本]      |
   +-------+--------+        +--------+-------+
           |  read/write             |
           v                         v
   +-------------------------------------------+
   |            主内存  x = 0                    |
   +-------------------------------------------+

这是抽象模型,不代表真实内存布局——实际中”工作内存”对应 CPU 寄存器、store buffer、各级缓存,JMM 刻意不绑定到具体硬件实现,只规定可观察的行为契约,这样规范才能同时约束 x86、ARM 等内存序完全不同的硬件。核心问题:线程 A 改了 x,线程 B 不一定立刻看到——根源是缓存这一层客观存在,加上编译器/CPU 为性能会重排指令。

JMM 定义了三个必须分别保证的特性,注意它们是三件不同的事,分别有不同的成因和不同的解决工具,这也是后面 volatile/synchronized/Atomic 分工的直接依据:

特性 含义 不保证时的后果 靠什么保证
原子性 操作不可分割,不会被其他线程的操作插入中间状态 复合操作(如 count++)被交叠执行,丢更新 基本读写(long/double 除外)天然原子;复合操作需 synchronized/CAS
可见性 一个线程的写,另一个线程能读到 一个线程写了新值,另一个线程永远读到旧值(甚至陷入死循环) volatile、synchronized、final
有序性 指令不被重排到破坏跨线程可观察的语义 依赖顺序的两步操作被重排,另一线程看到”结果先于原因” volatile、synchronized、happens-before

三者独立的原因很直接:可见性解决的是”写了之后别人看不看得到”(缓存问题),有序性解决的是”写的顺序会不会被打乱”(重排问题),原子性解决的是”一个逻辑操作会不会被拆成能被其他线程看到中间状态的若干步”(操作粒度问题)——三个问题成因不同,自然没有一个工具能同时以最低代价解决全部三个,这就是为什么 Java 并发工具箱不是一把”万能锁”,而是分层的:volatile 只解决可见性和有序性,synchronized/Atomic 才解决原子性(下文详述)。

long/double 的坑:JMM 只保证 32 位读写原子,64 位变量在 32 位平台上可能被拆成两次写导致读到”半个值”(高 32 位是新值、低 32 位是旧值的错位组合),加 volatile 保证 64 位读写整体原子。HotSpot 在主流 64 位 JVM 上实际都按原子处理,但规范本身不做这个保证,写跨平台代码不能依赖具体实现的这一巧合。

happens-before

JMM 不要求你去推理”缓存什么时候刷新”“指令怎么重排”这些实现细节,而是提供一套更高层的契约:happens-before。若操作 A happens-before 操作 B,则 A 的结果对 B 可见,且 A 在语义上排在 B 之前——这是一个可以直接拿来推理程序正确性的契约,不需要理解 CPU 缓存一致性协议。规则:

  1. 程序顺序:同一线程内,前面的操作 hb 后面的。
  2. 监视器锁:对锁的 unlock hb 后续对同一锁的 lock。
  3. volatile:对 volatile 变量的写 hb 后续的读。
  4. 传递性:A hb B, B hb C ⇒ A hb C。
  5. 线程启动:t.start() hb 线程内的任何操作。
  6. 线程终止:线程内操作 hb 其他线程的 t.join() 返回。
int a = 0;
volatile boolean flag = false;

// 线程1
a = 1;              // (1)
flag = true;        // (2) volatile 写

// 线程2
if (flag) {         // (3) volatile 读,读到 true 则 (2) hb (3)
    System.out.println(a); // 必然是 1:(1)hb(2)hb(3),传递 ⇒ (1)hb(4)
}

没有 volatile,a 的写可能对线程 2 不可见,或者 (1)(2) 被重排,导致线程 2 看到 flag == true 但 a == 0。

指令重排与双重检查锁定

编译器和 CPU 为性能会重排指令,前提只是单线程语义(as-if-serial)不变——这个前提在多线程下不再成立,是并发 bug 最隐蔽的来源之一,因为重排造成的错误结果在单线程视角下”看起来完全正常”。经典例子是双重检查锁定(Double-Checked Locking, DCL):

public class Singleton {
    private static volatile Singleton instance; // volatile 不可省
    public static Singleton getInstance() {
        if (instance == null) {                  // 第一次检查(无锁)
            synchronized (Singleton.class) {
                if (instance == null) {          // 第二次检查
                    instance = new Singleton();  // 见下方拆解
                }
            }
        }
        return instance;
    }
}

具体拆解 instance = new Singleton() 为什么会在没有 volatile 时出错:这一行字节码层面不是一步完成的,而是三步——(1) 分配一块内存 (2) 在这块内存上调用构造器初始化字段 (3) 把 instance 引用指向这块内存。只要编译器或 CPU 判断”单线程内这三步谁先谁后不影响最终结果”,就有权重排成 (1)(3)(2)——即先把 instance 指向内存,再执行构造器。对执行这段代码的线程本身,这个重排确实不可观察(等它执行完 (2),instance 已经指向一个初始化好的对象,和顺序执行的结果一样)。但对另一个线程而言,重排的中间状态是可观察的:如果线程 A 正处在重排后的 (1)(3) 之间(instance 已非 null,但对象字段还是默认值),线程 B 同时执行到第一次检查 if (instance == null),会看到 instance != null 而直接返回——但它拿到的是一个构造函数还没跑完的半成品对象,后续对这个对象的字段访问会读到默认值(比如本该是某个业务对象却读到全 0/null 的字段),这个故障只在多线程同时争抢”首次创建”的窗口期出现,窗口极窄,单元测试和日常运行几乎不可能触发,只会在生产环境高并发下偶发,而且每次复现的字段状态还可能不同——这正是并发 bug 比顺序执行 bug 难排查得多的地方。

volatile 修饰 instance 之后,规则里的”写前的操作不能重排到写后”直接禁止了 (2)(3) 被重排成 (3)(2):对 instance 的写(第 3 步)之前的操作(第 2 步:构造初始化)必须先完成。这正是 JDK5(JSR-133 定义 volatile 语义之后)DCL 才正确的原因——JDK5 之前 volatile 只保证可见性、不禁止重排,DCL 在那之前即使加了 volatile 也不安全。

volatile

volatile 最常见的用法是”一写多读的状态标志”——一个线程写,其他线程读,保证读到的是最新值:

public class VolatileFlagDemo {
    static volatile boolean ready = false;
    static int data = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread writer = new Thread(() -> {
            data = 42;         // 先写普通变量
            ready = true;      // 再写 volatile,充当"发布"信号
        });
        Thread reader = new Thread(() -> {
            while (!ready) {   // 忙等,直到看到 writer 的更新
                // 自旋
            }
            System.out.println("data = " + data); // 保证看到 42,而不是 0
        });
        reader.start();
        writer.start();
        writer.join();
        reader.join();
        System.out.println("done");
    }
}
// data = 42
// done

去掉 volatile,reader 线程理论上可能一直看不到 ready 变成 true(编译器/JIT 可能把 while (!ready) 优化成只读一次寄存器,因为它看不出这个字段会被别的线程改),或者看到 ready == true 但 data 还是 0(重排)。加上 volatile 后这两个问题都不会发生。

volatile 提供的正是可见性和有序性这两个保证(类比 C++ 的 acquire/release 语义,但不完全等同):

  1. 可见性:一个线程对 volatile 变量的写,保证随后读它的线程都能看到最新值。实现上靠内存屏障(store/load fence)配合 CPU 的缓存一致性协议,而不是字面意义上”每次读写都绕过缓存、直刷主内存”(那是过度简化的说法)。
  2. 禁止重排:写前的操作不能重排到写后,读后的操作不能重排到读前。

为什么 volatile 不保证复合操作的原子性,而不是干脆做成”读写都原子”来一步到位:volatile 的实现代价刻意压得很低——它只在读写点插入内存屏障,不涉及任何互斥机制(不排队、不阻塞、不需要操作系统介入),这正是它比锁轻量得多的原因。但”读取旧值—计算新值—写回”这种多步操作,内存屏障管的是”每一步单独发生时的顺序和可见性”,管不住”两个线程各自的这几步互相穿插”——屏障不是互斥量,它不会让第二个线程在第一个线程完成整个读改写序列之前干等着:

volatile int count = 0;

void bump() {
    count++;  // 读-改-写三步,volatile 不能防止两线程都读到同一值后各自 +1 丢更新
}

两个线程各对同一个 volatile int count 调用 10 万次 count++,结果几乎必然小于 20 万——volatile 保证了每次读写单独发生时都可见,却挡不住”线程 A 读到 100、线程 B 也读到 100(A 的写还没发生或 A 还没写),两者都基于 100 算出 101 各自写回”这种交叠:

count = 157682 (expected 200000, usually less)   // 多次运行,结果每次都不同,但总是明显小于 200000

volatile / synchronized / Atomic 的分工由此而定:volatile 只解决单个变量的可见性+有序性,代价最低;要让”读改写”这类复合操作整体原子,要么用 synchronized/ReentrantLock 做互斥(让整个临界区串行执行,一次只有一个线程能进,能覆盖任意多行代码、任意多个变量),要么用 AtomicInteger 这类基于 CAS 的原子类(只对单个变量的单步操作做原子性,靠硬件指令而非阻塞,见下节)。三者按”要解决的问题粒度”和”愿意付出的代价”分层,不是互相替代关系。count++ 需要 synchronized 或 AtomicInteger 才能保证原子;volatile 适合”一写多读的状态标志”,不适合”多线程都要修改的计数器”。

synchronized 与对象头

给方法加 synchronized,同一时刻只允许一个线程执行它;synchronized 也可以包一段代码块,并指定锁哪个对象:

synchronized (lock) {        // monitorenter 字节码
    // 临界区
}                            // monitorexit 字节码(异常路径也会 exit)

配合锁还有 wait/notify:一个线程在满足某个条件前先”等着”,另一个线程改完条件后”叫醒”它。两者都必须在 synchronized 块内调用:

public class WaitNotifyDemo {
    static final Object lock = new Object();
    static boolean condition = false;

    public static void main(String[] args) throws InterruptedException {
        Thread waiter = new Thread(() -> {
            synchronized (lock) {
                while (!condition) {                 // 用 while 而不是 if,防止虚假唤醒
                    try { lock.wait(); } catch (InterruptedException e) { }
                }
                System.out.println("waiter: condition met");
            }
        });
        waiter.start();
        Thread.sleep(100); // 让 waiter 大概率先进入 wait(不影响正确性,只是让演示更直观)
        synchronized (lock) {
            condition = true;
            lock.notifyAll();                        // 唤醒等待者(仍需重新竞争锁)
        }
        waiter.join();
        System.out.println("main: done");
    }
}
// waiter: condition met
// main: done

lock.wait() 会释放锁并让当前线程进入等待,直到别的线程调用 lock.notify()/notifyAll() 才有机会被唤醒重新竞争锁——所以 wait/notify 必须配对使用同一个锁对象。

synchronized 是可重入的监视器锁(monitor)。每个 Java 对象都能作锁,锁状态记录在对象头的 mark word 里。64 位 HotSpot 对象头布局(未压缩,普通对象):

对象内存布局:
+------------------+------------------+-------------+---------+
| Mark Word (8B)   | Klass Pointer(4/8B)| 实例字段    | 对齐填充 |
+------------------+------------------+-------------+---------+

Mark Word 复用同一块 8 字节存不同信息(锁标志位决定解释方式):

锁状态 Mark Word 内容(部分) 标志位
无锁 hashCode、GC 分代年龄 01
偏向锁 持锁线程 ID 01(偏向位1)
轻量级锁 指向栈中 Lock Record 指针 00
重量级锁 指向 monitor(ObjectMonitor)指针 10
GC 标记 — 11

锁升级

为避免每次都走重量级(OS 互斥量,涉及用户态/内核态切换,昂贵),HotSpot 做了分级,本质是”绝大多数锁其实没有真实竞争”这一观察的直接产物——如果一开始就无脑走重量级锁,会为完全不存在的竞争支付切换到内核态的成本:

无锁 --第一个线程进入--> 偏向锁(记录线程ID,后续同线程CAS都免了)
      --别的线程竞争--> 轻量级锁(自旋+CAS,乐观,适合短临界区)
      --自旋失败/竞争激烈--> 重量级锁(挂起线程,进OS等待队列)
  • 偏向锁:假设锁总被同一线程获取,首次记下线程 ID,之后无竞争则无需 CAS。JDK15 起默认禁用并计划移除(现代应用里线程池、并发框架让”同一线程反复加锁”的假设不再普遍成立,维护这层复杂度换来的收益变小)。
  • 轻量级锁:多线程交替执行(非同时竞争)。线程在自己栈帧建 Lock Record,用 CAS 把 mark word 换成指向它的指针。失败则自旋。
  • 重量级锁:真正竞争。膨胀为 ObjectMonitor,竞争线程被 OS 挂起,类比 pthread_mutex。

CAS 与 Atomic*

AtomicInteger 这类原子类,提供一组”看起来是普通方法、实际上整体原子”的操作,不需要自己加锁:

import java.util.concurrent.atomic.AtomicInteger;

public class AtomicBasicDemo {
    public static void main(String[] args) {
        AtomicInteger ai = new AtomicInteger(0);
        System.out.println(ai.incrementAndGet()); // 1
        System.out.println(ai.incrementAndGet()); // 2
        boolean ok = ai.compareAndSet(2, 100);     // 期望当前值是2,设为100
        System.out.println(ok);                    // true
        System.out.println(ai.get());               // 100
    }
}
// 1
// 2
// true
// 100

多个线程并发调用 incrementAndGet(),不会像普通 int 的 count++ 那样丢更新——这背后就是 CAS(Compare-And-Swap),无锁并发的基石,对应 x86 的 cmpxchg 指令。语义:if (内存值==预期) { 内存值=新值; return true },整个过程硬件原子——它和”锁”解决同一个”原子性”问题,但走的是完全不同的路径:锁靠互斥(让第二个线程等待),CAS 靠”乐观重试”(不阻塞任何线程,失败就重新读最新值再试一次),因此完全没有线程被挂起/唤醒的开销,在竞争不激烈时比锁快得多;竞争激烈时反复重试也会浪费 CPU,这是它的代价边界。

ai.incrementAndGet();               // 底层 CAS 自旋:do { old=get(); } while(!compareAndSet(old, old+1))
boolean ok = ai.compareAndSet(1, 2); // 期望1则设2

// JDK8+ 高并发计数器,分段减少 CAS 竞争(热点分散到多个 Cell)
LongAdder adder = new LongAdder();
adder.increment();
long sum = adder.sum();

Atomic* 内部字段用 volatile + Unsafe/VarHandle 的 CAS 实现——这也解释了为什么 Atomic* 能弥补前面说的”volatile 不保证复合操作原子”这个缺口:它没有让 volatile 变得更强,而是在 volatile 提供的可见性之上,用 CAS 这条硬件指令把”比较并交换”合并成单一不可分割的操作,从根本上避免了”读到旧值—计算—写回”这几步被别的线程插入的可能。

ABA 问题:值从 A 变 B 又变回 A,CAS 察觉不到——因为 CAS 只比较”当前值是否等于预期值”,不关心这期间值有没有被改过又改回来,在无锁栈、无锁链表这类结构里,ABA 可能导致节点被错误地复用或释放。用 AtomicStampedReference(给值绑定一个版本号,一并比较)解决,类比无锁栈里的 tagged pointer。

可运行对比:8 个线程各自 count++ 十万次。无保护的 int 因”读-改-写”非原子而丢更新,AtomicInteger 靠 CAS 始终正确:

import java.util.concurrent.atomic.AtomicInteger;

public class LostUpdate {
    static int plain = 0;                       // 无保护
    static final AtomicInteger atomic = new AtomicInteger(0);

    public static void main(String[] args) throws Exception {
        int threads = 8, perThread = 100_000;
        Runnable job = () -> {
            for (int i = 0; i < perThread; i++) {
                plain++;                         // 读-改-写,非原子:丢更新
                atomic.incrementAndGet();        // CAS:原子
            }
        };
        Thread[] ts = new Thread[threads];
        for (int i = 0; i < threads; i++) ts[i] = new Thread(job);
        for (Thread t : ts) t.start();
        for (Thread t : ts) t.join();            // 等全部结束,结果确定

        int expected = threads * perThread;      // 800000
        System.out.println("expected      = " + expected);
        System.out.println("atomic == exp : " + (atomic.get() == expected));
        System.out.println("plain  <  exp : " + (plain < expected));
    }
}
// expected      = 800000
// atomic == exp : true
// plain  <  exp : true    —— plain 几乎必然 < 800000(丢更新),atomic 恒等于期望值

（该代码已实测:atomic == exp 恒为 true,plain < exp 恒为 true,现象稳定可复现。）

java.util.concurrent

ReentrantLock

ReentrantLock 是 synchronized 的显式版本:自己 lock()、自己 unlock(),一定要把 unlock() 放进 finally,否则异常路径会导致锁永远不释放:

import java.util.concurrent.locks.ReentrantLock;

public class LockBasicDemo {
    static final ReentrantLock lock = new ReentrantLock();
    static int counter = 0;

    static void increment() {
        lock.lock();
        try {
            counter++;
        } finally {
            lock.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable job = () -> { for (int i = 0; i < 100_000; i++) increment(); };
        Thread t1 = new Thread(job);
        Thread t2 = new Thread(job);
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("counter = " + counter); // 200000
    }
}
// counter = 200000

比 synchronized 多出来的能力是可以带超时地尝试加锁,拿不到就走别的分支,不会一直傻等:

if (lock.tryLock(1, TimeUnit.SECONDS)) {   // 最多等1秒
    try {
        // 临界区
    } finally {
        lock.unlock();
    }
} else {
    // 没拿到锁,做别的事(比如放弃、重试)
}

ReentrantLock 比 synchronized 更灵活:可中断、可超时、可公平、可多条件变量。基于 AQS(AbstractQueuedSynchronizer)——一个 volatile int state + CLH 变体等待队列,state 的可见性靠 volatile,竞争时的排队和阻塞才靠 AQS 自己的等待队列机制实现。new ReentrantLock(true) 构造公平锁(按请求顺序排队,吞吐略低但不会有线程一直抢不到)。

对比 synchronized ReentrantLock
获取方式 隐式(JVM) 显式 lock/unlock
释放 自动 手动(finally)
可中断 否 lockInterruptibly()
超时 否 tryLock(timeout)
公平锁 否 可选
条件变量 单个(wait/notify) 多个 Condition

ConcurrentHashMap

并发场景下别用普通 HashMap(多线程同时触发扩容可能导致链表成环、get 死循环),用 ConcurrentHashMap,API 用法和 HashMap 一样:

import java.util.concurrent.ConcurrentHashMap;

public class CHMBasicDemo {
    public static void main(String[] args) {
        ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();
        map.put("a", 1);
        map.putIfAbsent("a", 99);         // "a" 已存在,不覆盖
        map.computeIfAbsent("b", k -> 2); // "b" 不存在,计算并放入
        System.out.println(map.get("a")); // 1
        System.out.println(map.get("b")); // 2
    }
}
// 1
// 2

JDK8 重写:数组 + 链表/红黑树,并发控制用 CAS + synchronized(锁桶头结点),而非 JDK7 的分段锁(Segment)。选择”锁桶头结点”而非锁整个表,是为了让并发度和桶数量成正比而不是被压缩成 1——这正是它相比 Collections.synchronizedMap(锁整表,任何时刻只有一个线程能操作)吞吐高得多的根本原因:

  • 空桶插入:CAS 直接放,无锁。
  • 桶非空:synchronized 锁住该桶头结点,只锁一个桶,并发度 = 桶数。
  • 链表长度 ≥8 且表容量 ≥64 时转红黑树,查询从 O(n) 降到 O(log n)。
  • size() 用 baseCount + CounterCell[](类似 LongAdder)避免热点。

读操作基本无锁(结点 val/next 为 volatile),写只锁单桶,吞吐远高于 Collections.synchronizedMap。

线程池 ThreadPoolExecutor

比起每次任务都 new Thread,通常用线程池复用线程——创建 OS 线程本身昂贵(前面讲过,栈分配、pthread_create、内核调度都有实打实的成本),线程池把这份成本摊到多次任务执行上。最简单的用法是拿到一个 ExecutorService,反复 submit 任务:

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class PoolBasicDemo {
    public static void main(String[] args) throws Exception {
        ExecutorService es = Executors.newFixedThreadPool(2);
        Future<Integer> f = es.submit(() -> 1 + 1);
        System.out.println("result = " + f.get()); // 2,阻塞获取结果
        es.shutdown();                              // 用完记得关闭线程池
    }
}
// result = 2

submit 把任务交给线程池异步执行,立刻返回一个 Future;f.get() 阻塞等待任务跑完并拿到返回值;shutdown() 通知线程池不再接受新任务、跑完手头的就退出。

生产环境更常见的是自己 new ThreadPoolExecutor 把参数写明白,而不是用 Executors 的工厂方法:

import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.Executors;
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    2,                              // corePoolSize:核心线程数,常驻
    4,                              // maximumPoolSize:最大线程数
    60L, TimeUnit.SECONDS,          // keepAliveTime:非核心线程空闲存活时间
    new ArrayBlockingQueue<>(100),  // workQueue:任务队列
    Executors.defaultThreadFactory(),
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
);

任务提交逻辑(务必理解):

提交任务
  ├─ 运行线程数 < core?      → 新建核心线程执行
  ├─ 否,队列未满?           → 入队等待
  ├─ 否,线程数 < max?       → 新建非核心线程执行
  └─ 否                      → 触发拒绝策略

拒绝策略:AbortPolicy(抛异常,默认)、CallerRunsPolicy(提交者自己跑,反压)、DiscardPolicy、DiscardOldestPolicy。

生产环境避免用 Executors.newFixedThreadPool/newCachedThreadPool:前者队列无界(任务堆积可能 OOM 而不是触发拒绝策略),后者线程数无界(瞬时压力下可能建出海量线程拖垮系统)。手动 new ThreadPoolExecutor 明确参数,把”容量到底有没有上限”这个决策显式做出来,而不是依赖工厂方法背后不易察觉的默认值。

Future 与 CompletableFuture

上面 PoolBasicDemo 里 es.submit(...) 返回的 Future 就是最基础的异步结果占位符——get() 阻塞拿结果,拿不到结果就什么都做不了,多个异步步骤之间没法声明式地串联,只能一层层手动 get() 阻塞。CompletableFuture 解决的正是这个组合性问题:可以像搭积木一样把多个异步步骤串起来,不必每一步都手动阻塞:

import java.util.concurrent.CompletableFuture;

public class CFBasicDemo {
    public static void main(String[] args) {
        CompletableFuture<Integer> cf = CompletableFuture
            .supplyAsync(() -> 10)     // 异步产出一个值
            .thenApply(x -> x * 2);    // 对结果做变换,链式返回新的 CompletableFuture
        System.out.println(cf.join()); // 20,阻塞等待整条链跑完再取值
    }
}
// 20

更常见的链式写法还包括消费结果、处理异常:

CompletableFuture
    .supplyAsync(() -> 10)                    // 异步产出
    .thenApply(x -> x * 2)                    // 变换(同线程链式)
    .thenCompose(x -> CompletableFuture.supplyAsync(() -> x + 1)) // 扁平化嵌套 future
    .thenAccept(System.out::println)          // 消费,打印 21
    .exceptionally(ex -> { ex.printStackTrace(); return null; })
    .join(); // 等链跑完 —— 公共线程池是守护线程,main 不等待就可能提前退出、看不到打印

类似 JS 的 Promise / C++ 的 std::future + continuation。可运行:thenCombine 让两个任务并发跑再汇合;exceptionally 从链中异常恢复,后续步骤继续:

import java.util.concurrent.CompletableFuture;

public class CFCompose {
    public static void main(String[] args) throws Exception {
        CompletableFuture<Integer> a = CompletableFuture.supplyAsync(() -> 3);
        CompletableFuture<Integer> b = CompletableFuture.supplyAsync(() -> 4);

        // 组合:a、b 并发执行,完成后合并
        int hypotSq = a.thenCombine(b, (x, y) -> x * x + y * y).join(); // 25

        // 链式 + 异常恢复
        String r = CompletableFuture.supplyAsync(() -> 10)
            .thenApply(x -> x / 0)                    // 抛 ArithmeticException
            .exceptionally(ex -> -1)                  // 恢复为 -1
            .thenApply(x -> "result=" + x)
            .join();

        System.out.println("hypotSq = " + hypotSq);
        System.out.println(r);
    }
}
// hypotSq = 25
// result=-1

死锁与避免

死锁四条件(与 OS 教材一致):互斥、持有并等待、不可剥夺、循环等待。典型:

// 线程1: 先锁 A 再锁 B;线程2: 先锁 B 再锁 A → 循环等待
synchronized (A) { synchronized (B) { } }  // 线程1
synchronized (B) { synchronized (A) { } }  // 线程2

避免手段: - 锁排序:所有线程按固定全局顺序获取锁(破坏循环等待)。 - tryLock 超时:ReentrantLock.tryLock 拿不到就回退释放。 - 减小锁粒度/持有时间,尽量无锁(CAS、不可变对象)。 - 检测:jstack <pid> 能报 “Found one Java-level deadlock”。

虚拟线程(Loom,JDK21)

前面看到的平台线程(new Thread(...))1:1 绑内核线程,栈约 1MB,创建走 pthread_create。这在CPU 密集型场景问题不大(线程数量本来就该接近核数),但在IO 密集型服务(比如典型的”一请求一线程”Web 服务器)里会撞上一个结构性矛盾:处理一个 HTTP 请求时,线程绝大部分时间花在等数据库返回、等下游 RPC 响应上——这些等待里线程本身不做任何计算,但只要它是一个平台线程,就一直占着一个宝贵的内核线程和约 1MB 栈内存,直到 IO 完成。

为什么不能靠”加大线程池”解决:数字算一下就清楚——线程池开到 10 万个平台线程,仅栈内存就要 10 万 × 1MB = 100GB,大多数机器根本扛不住,这还没算每个线程在内核态的调度成本;操作系统调度器面对成千上万个线程时,上下文切换本身(保存/恢复寄存器、TLB 刷新、缓存局部性被破坏)的开销会显著增长,吞吐反而下降。这不是”参数没调好”的问题,而是”一个等待中的请求占用一个完整内核线程”这个模型在资源消耗上的天花板——加大线程池只是把这个矛盾往后推,不会消除它。

在虚拟线程之前,业界的解法是异步/回调式编程(Java 的 NIO + CompletableFuture、Netty 的 Reactor 模型):不让线程阻塊等 IO,而是注册一个回调,IO 就绪后再在某个线程上执行回调,同一个线程可以在等待期间去处理别的连接。这确实解决了资源问题(Netty 用几个线程就能扛住数万并发连接),但代价是编程模型变了——业务逻辑被拆成一堆通过回调链接起来的片段,原本”读数据库→算个结果→再查一次→返回”这种线性逻辑变成层层嵌套或链式的 thenCompose/thenApply,一次异常需要在每个环节显式处理,调试时的调用栈也不再对应”这个请求实际经过的调用路径”,而是对应”这个回调在哪个线程池的哪次调度里被触发”——排查问题的直觉被打断了。

虚拟线程要解决的正是这个矛盾:既要摆脱”一个阻塞请求占一个内核线程”的资源模型,又要保留”从上到下顺序写代码、用普通 try/catch、调用栈就是真实调用路径”这套简单直觉的编程模型。JDK21 起可以用虚拟线程,语法和平台线程几乎一样,但极轻,能开到百万级:

import java.util.concurrent.Executors;
import java.util.concurrent.Future;

public class VirtualBasicDemo {
    public static void main(String[] args) throws Exception {
        Thread vt = Thread.ofVirtual().start(() -> System.out.println("hello from virtual thread"));
        vt.join();
        System.out.println("is virtual: " + vt.isVirtual()); // true

        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            Future<Integer> f = executor.submit(() -> 1 + 1);
            System.out.println("result = " + f.get()); // 2
        }
    }
}
// hello from virtual thread
// is virtual: true
// result = 2

Thread.ofVirtual().start(...) 直接起一个虚拟线程;Executors.newVirtualThreadPerTaskExecutor() 则是”每提交一个任务就用一个新虚拟线程去跑”的线程池,submit/get 用法和普通线程池完全一样。

虚拟线程是 JVM 调度的用户态线程(类比 Go goroutine):栈按需增长、存在堆上,创建和切换不经过操作系统,可以开到百万级,平台线程 1:1 绑内核线程做不到这一点。

// 每个任务一个虚拟线程,阻塞 IO 时自动卸载,不占内核线程
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 1_000_000; i++) {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1)); // 阻塞不占载体线程
            return null;
        });
    }
}

可运行:一次开 10 万个虚拟线程,每个”阻塞” 10ms。放在几个平台线程(载体)上照样几十毫秒跑完——同样数量的平台线程根本开不出来。用 CountDownLatch 等全部报到,AtomicInteger 计数:

import java.util.concurrent.CountDownLatch;
import java.util.concurrent.Executors;
import java.util.concurrent.atomic.AtomicInteger;

public class VThreads {
    public static void main(String[] args) throws Exception {
        int n = 100_000;
        AtomicInteger done = new AtomicInteger();
        CountDownLatch latch = new CountDownLatch(n);

        // 每任务一个虚拟线程,共 10 万个,各自短暂阻塞
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < n; i++) {
                executor.submit(() -> {
                    Thread.sleep(10);         // 阻塞卸载载体,而非占用内核线程
                    done.incrementAndGet();
                    latch.countDown();
                    return null;
                });
            }
            latch.await();                    // 等全部任务报到
        }
        System.out.println("tasks done      = " + done.get());
        System.out.println("current is virtual: " + Thread.currentThread().isVirtual());
    }
}
// tasks done      = 100000
// current is virtual: false   —— main 是平台线程;10 万虚拟线程由少量载体线程调度

（该例已实测:10 万虚拟线程 + 每个 10ms 阻塞,总耗时约 0.15 秒——如果换成 10 万个平台线程,光是创建开销和栈内存就足以拖垮大多数机器。）

挂载与卸载

虚拟线程运行时挂载(mount)到一个平台线程(载体线程 carrier,来自 ForkJoinPool)。遇到阻塞操作(如网络 IO、sleep)时卸载(unmount):虚拟线程的栈(以 continuation 形式)被保存到堆上,载体线程释放去跑别的虚拟线程。阻塞结束后再挂载到某个载体继续——这正是它同时兼顾”资源效率”与”同步编程模型”的机制核心:业务代码仍然是普通的顺序阻塞调用(Thread.sleep、阻塞式 IO API),调用栈仍然是这个请求真实的调用路径,但运行时把”阻塞”翻译成了”释放底层资源、稍后恢复”,这一步对写代码的人完全透明。

平台线程池(载体, ~CPU核数个)
  Carrier-1: [VT-A 运行中] --VT-A遇到IO阻塞--> 卸载VT-A, 挂载 VT-B
  Carrier-2: [VT-C 运行中]
             堆: [VT-A 栈快照 待唤醒] [VT-D 栈快照] ...

对比:平台线程阻塞时,整个内核线程被 OS 挂起(占着不用也不能给别人用);虚拟线程阻塞时,内核线程被解放去干活。因此”一请求一线程”的同步写法也能有异步的吞吐,不必再牺牲编程模型换资源效率。

维度 平台线程 虚拟线程
映射 1:1 内核线程 M:N,JVM 调度到载体
数量级 数千 数百万
栈 固定 ~1MB 按需,堆上可增减
阻塞代价 占用内核线程 卸载,释放载体
适用 CPU 密集 IO 密集、高并发

pinning(固定)陷阱:虚拟线程阻塞时若无法卸载,就会钉住载体线程,削弱优势。根源在于监视器锁的实现方式——在 JDK 21–23,synchronized 块底层的 ObjectMonitor 把锁的持有者记录为当前操作系统线程,而不是虚拟线程本身;如果一个虚拟线程在持锁期间被卸载、之后又被挂载到另一个载体上恢复执行,锁的持有者身份就和实际运行它的 OS 线程对不上了,这会破坏监视器实现依赖的”持锁者=某个确定 OS 线程”这一底层假设。为了避免这种不一致,JVM 在 JDK 21–23 里选择让虚拟线程在 synchronized 块内阻塞时不卸载、直接钉住载体,所以当时的建议是把涉及阻塞 IO 的临界区改用 ReentrantLock(它的锁状态记录在 Java 对象里,不绑定 OS 线程身份)。JDK 24 起(JEP 491)重写了监视器实现,把锁持有者的记录方式改为跟踪虚拟线程本身而非底层 OS 线程,消除了这个绑定,synchronized 造成的 pinning 因此基本消失,不必再为此刻意替换成 ReentrantLock。调用 native 方法(JNI)时仍会 pin——原因不同:native 栈帧是这块内存实实在在地属于当前 OS 线程的调用栈,JVM 没有能力把一段正在执行的本地代码连同它的栈帧迁移到另一个载体线程上,这是操作系统级线程模型的硬约束,不是 JVM 实现选择的问题,因此这一类 pinning 目前无法消除。

关于 volatile/CAS 的底层内存屏障,可回顾 03-变量与内存模型。线程池提交的任务与异常传播见 09-错误处理与资源管理。

小结

  • 用法:new Thread(() -> ...).start() 开线程,join() 等它跑完;两个线程改同一个变量就用 synchronized 包起来,或用 ReentrantLock/AtomicInteger;批量任务用线程池 submit,拿 Future/CompletableFuture 取结果;ConcurrentHashMap 代替并发场景下的 HashMap;Thread.ofVirtual()/newVirtualThreadPerTaskExecutor() 开虚拟线程,语法和平台线程一样但能开到百万级。
  • 原理:多线程出问题的根源是 JMM 抽象里”工作内存/主内存”不同步,而这不是实现疏漏,是编译器/CPU 只承诺单线程 as-if-serial 语义的必然代价——可见性、有序性、原子性三个独立问题分别靠 volatile/synchronized/happens-before 和 CAS 保证,没有一个工具能同时以最低代价解决全部三个。synchronized 锁状态记在对象头 mark word 里,会经历偏向锁→轻量级锁→重量级锁的升级;CAS 是无锁并发的硬件基石(但有 ABA 问题);ReentrantLock 基于 AQS;ConcurrentHashMap 靠”空桶 CAS + 非空桶锁单桶头结点”把并发度提到桶级;线程池要理解核心/最大线程数、队列、拒绝策略的提交流程;虚拟线程用 M:N 调度和挂载/卸载机制,在保留同步编程模型的同时解决了”阻塞 IO 占用整个内核线程”这一结构性资源问题,但要避开会导致 pinning 的场景(JDK24 前的 synchronized 阻塞、native 调用)。
