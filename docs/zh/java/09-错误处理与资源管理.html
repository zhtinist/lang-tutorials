错误处理与资源管理:异常与垃圾回收

C++ 用异常做错误传播,用 RAII(析构函数)做资源释放,用手动 delete/智能指针管内存。Java 把这三件事拆成:异常体系、try-with-resources(近似 RAII)、以及自动垃圾回收(GC)。

一、try/catch/finally 基本写法

抛出并捕获一个异常(除以 0 会抛 ArithmeticException):

public class QuickTry {
    public static void main(String[] args) {
        try {
            int x = 10 / 0;          // 抛出 ArithmeticException
            System.out.println("不会执行到这里: " + x);
        } catch (ArithmeticException e) {
            System.out.println("捕获异常: " + e.getMessage());
        } finally {
            System.out.println("finally 总会执行");
        }
        System.out.println("程序继续往下跑");
    }
}
// 捕获异常: / by zero
// finally 总会执行
// 程序继续往下跑

要点:try 包住可能出错的代码;catch (类型 变量) 按类型匹配来接住异常;finally 里的代码不管有没有异常都会跑一遍(常用来做扫尾清理)。异常发生后,try 块里从出错点开始的代码不会再执行(上例里 x 那行的 println 不会跑),直接跳到匹配的 catch。

关闭一次资源(try-with-resources):C 里打开文件要记得 fclose,Java 里只要资源实现了 AutoCloseable,写 try (...) 就能保证用完自动关闭,不用手写 finally { close(); }:

public class QuickTWR {
    // 一个最简单的可关闭资源:实现 AutoCloseable 即可
    static class SimpleResource implements AutoCloseable {
        SimpleResource() { System.out.println("打开资源"); }
        void use() { System.out.println("使用资源"); }
        @Override public void close() { System.out.println("关闭资源"); }
    }

    public static void main(String[] args) {
        try (SimpleResource r = new SimpleResource()) {
            r.use();
        }
        System.out.println("try 块结束后 close() 已经被自动调用");
    }
}
// 打开资源
// 使用资源
// 关闭资源
// try 块结束后 close() 已经被自动调用

try (SimpleResource r = new SimpleResource()) 括号里声明的资源,在 try 块(不管正常结束还是抛异常)结束时,close() 都会被自动调用一次——这是 Java 里最接近 C++ RAII/析构函数的写法。

还有几个问题需要搞清楚:异常抛出后 JVM 到底怎么找到该由哪个 catch 接住?finally 为什么”总会执行”?try-with-resources 关闭多个资源时顺序是怎样的、如果 close() 本身也抛异常怎么办?下面依次展开。

二、异常体系:受检 vs 非受检

Java 异常分两大类,写代码时的实际区别是:受检异常编译器强制你处理,非受检异常不强制。

import java.io.*;

public class QuickChecked {
    // 受检异常:方法签名必须写 throws,否则编译不过
    static void mightFail(boolean fail) throws IOException {
        if (fail) throw new IOException("模拟IO失败");
        System.out.println("读取成功");
    }

    // 非受检异常:不需要 throws 声明
    static int divide(int a, int b) {
        return a / b; // b=0 时抛 ArithmeticException,编译器不会强制处理
    }

    public static void main(String[] args) {
        try {
            mightFail(true);
        } catch (IOException e) {
            System.out.println("捕获受检异常: " + e.getMessage());
        }

        try {
            divide(1, 0);
        } catch (ArithmeticException e) {
            System.out.println("捕获非受检异常: " + e.getMessage());
        }
    }
}
// 捕获受检异常: 模拟IO失败
// 捕获非受检异常: / by zero

如果 mightFail 声明了 throws IOException,调用它的代码要么用 try/catch 接住,要么自己也在方法签名上写 throws IOException 往外抛——否则编译器直接报错。divide 抛的 ArithmeticException 没有这个约束,你可以选择捕获,也可以不管,编译器不拦。

一切可抛出对象继承自 Throwable。它有两条主线:

                Throwable
               /         \
           Error        Exception
        (JVM 层严重故障)   /       \
     OutOfMemoryError  RuntimeException  其他(受检异常)
     StackOverflowError  (非受检)          IOException
                          NullPointerException  SQLException
                          IllegalArgumentException ...
类别 是否受检 语义 C++ 类比
Error 否 JVM 层面不可恢复(栈溢出、堆耗尽),不该 catch 近似 std::bad_alloc 之类系统级
RuntimeException 否 编程错误(空指针、越界、非法参数),应靠代码避免 逻辑错误,类似 assert 该拦住的
受检 Exception 是 可预期的外部失败(IO、网络),编译器强制处理 C++ 无对应,已废弃的 throw() 规格是反例

受检 vs 非受检是 Java 独有的编译期约束:受检异常必须在方法签名 throws 声明,或就地 try/catch,否则编译不过。C++ 早期的动态异常规格(throw(...))因难用被废弃,Java 保留了但也饱受争议。

import java.io.*;
public class Checked {
    // 受检:必须声明 throws,否则编译错误
    static String readFirst(String path) throws IOException {
        try (BufferedReader r = new BufferedReader(new FileReader(path))) {
            return r.readLine();
        }
    }
    // 非受检:无需声明,编译器不拦
    static int div(int a, int b) { return a / b; } // b=0 抛 ArithmeticException
}

三、try/catch/finally 与栈展开

finally 块不管 try 里是正常返回还是抛了异常,都会执行一遍——上一节 QuickTry 已经演示过:即使 10 / 0 抛出了异常,finally 里的 println 照样跑。这让它很适合放”不管成不成功都要做”的收尾动作,比如打日志、释放非 AutoCloseable 的资源。

抛出异常时,JVM 从当前方法开始查其 异常表(exception table)——javac 为每个 try 块生成一条 [起始PC, 结束PC, 处理器PC, 捕获类型] 记录。若当前帧无匹配处理器,该帧被弹出(栈展开,stack unwinding),沿调用链回溯,直到找到匹配的 catch 或到达线程栈底(此时线程终止并打印栈轨迹)。

finally 无论正常还是异常都执行。字节码层面,编译器会把 finally 的代码复制到每条正常/异常出口路径上(或用异常表 + athrow 兜底),这是它”必然执行”的实现方式。

public class Finally {
    static int tricky() {
        try {
            return 1;          // 先算出返回值 1 压栈
        } finally {
            return 2;          // finally 的 return 覆盖它 -> 实际返回 2(反模式!)
        }
    }
    public static void main(String[] args) {
        System.out.println(tricky()); // 2
    }
}

陷阱:finally 里 return 或抛异常会吞掉 try 中的返回值/异常。切勿在 finally 里 return。

四、try-with-resources:Java 的 RAII

上一节 QuickTWR 已经展示了最基本的用法:资源类实现 AutoCloseable,写在 try (...) 括号里,close() 就会在 try 块结束时自动调用——不管块里是正常跑完还是中途抛了异常。

C++ 靠对象离开作用域时析构函数确定性释放资源(RAII)。Java 对象的回收时机不确定(GC 说了算),所以不能靠 finalize 关文件。JDK7 引入 try-with-resources:任何实现 AutoCloseable(close())的对象,在 try 块结束时保证按声明的逆序调用 close()——这是 Java 最接近 RAII 的确定性资源管理。

import java.io.*;
public class TWR implements AutoCloseable {
    private final String name;
    TWR(String name) { this.name = name; System.out.println("open " + name); }
    @Override public void close() { System.out.println("close " + name); }

    public static void main(String[] args) {
        // 逆序关闭:先 close b,再 close a(类似 C++ 局部对象逆序析构)
        try (TWR a = new TWR("A"); TWR b = new TWR("B")) {
            System.out.println("body");
            throw new RuntimeException("boom");
        } catch (RuntimeException e) {
            System.out.println("caught: " + e.getMessage());
            // close 抛出的异常会作为“被抑制异常”挂在主异常上:
            for (Throwable s : e.getSuppressed()) System.out.println("suppressed: " + s);
        }
    }
    // 输出:open A / open B / body / close B / close A / caught: boom
}

区别于 C++:close() 的调用点由编译器插桩在 try 出口,而对象内存本身仍由 GC 稍后回收。即”资源(fd/句柄)确定性释放”与”内存非确定性回收”是分离的两件事。

上面的 close() 不抛异常,所以 getSuppressed() 是空的。当 try 体和 close() 都抛异常时,才看得到”被抑制异常”机制:主异常是 try 体抛的,两个 close() 的异常按逆序挂到主异常上:

public class Suppressed implements AutoCloseable {
    private final String name;
    Suppressed(String name){ this.name = name; }
    @Override public void close() {
        System.out.println("close " + name);
        throw new RuntimeException("close-fail-" + name);   // close() 自己也抛
    }
    public static void main(String[] args) {
        try (Suppressed a = new Suppressed("A"); Suppressed b = new Suppressed("B")) {
            throw new RuntimeException("body-fail");         // 主异常
        } catch (RuntimeException e) {
            System.out.println("primary   : " + e.getMessage());
            for (Throwable s : e.getSuppressed())            // close 抛的异常挂在这里
                System.out.println("suppressed: " + s.getMessage());
        }
    }
}
// close B                   <- 逆序关闭
// close A
// primary   : body-fail     <- try 体的异常胜出为主异常
// suppressed: close-fail-B  <- close 的异常被"抑制",逆序附加
// suppressed: close-fail-A

五、多重捕获与异常链

一个 catch 用 | 分隔多个类型,就能用同一段处理逻辑接住不同种类的异常,不用写重复代码:

public class QuickMultiCatch {
    static void parse(String s) {
        if (s == null) s.length();   // 对 null 调方法,直接触发 NullPointerException
        Integer.parseInt(s);         // 格式不对时抛 NumberFormatException
    }

    public static void main(String[] args) {
        String[] inputs = { null, "abc" };
        for (String s : inputs) {
            try {
                parse(s);
            } catch (NumberFormatException | NullPointerException e) {
                // 一个 catch 同时处理两种类型
                System.out.println("解析失败: " + e.getClass().getSimpleName());
            }
        }
    }
}
// 解析失败: NullPointerException
// 解析失败: NumberFormatException
public class MultiCatch {
    static void handle(String s) {
        try {
            Integer.parseInt(s);
        } catch (NumberFormatException | NullPointerException e) {
            // 多重捕获:一个 catch 处理多种类型,e 是隐式 final,类型为共同父类
            // 异常链:保留“根因(cause)”,便于定位原始故障
            throw new IllegalStateException("处理失败: " + s, e);
        }
    }
    public static void main(String[] args) {
        try { handle(null); }
        catch (IllegalStateException e) {
            System.out.println(e.getMessage());
            System.out.println("根因: " + e.getCause());
            // 根因: java.lang.NumberFormatException: Cannot parse null string
            // 注:现代 JDK 里 Integer.parseInt(null) 内部已显式判空并直接抛
            // NumberFormatException("Cannot parse null string"),而不是 NPE——
            // 这里 catch 两种类型仍然有意义:换一种解析方式(比如先手动取 s.length()),
            // 空指针路径就会走 NullPointerException 分支。
        }
    }
}

异常链(initCause / 带 cause 的构造器)让你在转译异常时不丢原始堆栈,打印时会出现 Caused by: 段落。

六、垃圾回收(GC)

C 和 C++ 里,内存归你自己管:malloc/free,new/delete,忘了 free 就是泄漏,多 free 一次就是崩溃。Java 里没有 delete——你 new 出来的对象,什么时候释放、被谁释放,完全交给一个后台机制:垃圾回收器(Garbage Collector,GC)。它会在合适的时候扫一遍堆,把”没人再用得到”的对象内存收回来。代价是:你不知道确切的回收时刻,而且回收本身可能让程序短暂卡顿(即”停顿”/STW)。理解 GC 的原理,才能在出问题时看懂日志、调对参数。

GC 全程自动运行,你能直接调用的 API 不多,主要是 WeakReference/SoftReference/Cleaner/System.gc() 这几个——它们仍然遵循”先怎么用、再底层”的顺序;而”分代假设”“收集器怎么演进”这类纯概念内容,直接进底层讲。

七、分代假设与堆布局

这部分是纯理解性内容——GC 自己运行,你不需要写代码触发它,但看懂堆是怎么分区的,才能看懂后面”收集器怎么选”“调优参数改哪个”。

GC 建立在弱分代假设(weak generational hypothesis)上:绝大多数对象朝生夕死;活得久的对象很少指向年轻对象。于是把堆分代管理:

                Java 堆
 +---------------------------------------------------+
 | 新生代 Young Gen                                  |
 |  +----------+  +----------+  +----------+         |
 |  |  Eden    |  | Survivor |  | Survivor |         |
 |  | (新对象) |  |   S0     |  |   S1     |         |
 |  +----------+  +----------+  +----------+         |
 +---------------------------------------------------+
 | 老年代 Old Gen (Tenured)  长寿对象               |
 +---------------------------------------------------+
   (元空间 Metaspace 在本地内存,存类元数据,不在堆内)
  • 新对象在 Eden 分配(指针碰撞,极快,类似栈顶推进)。
  • Eden 满触发 Minor GC:存活对象复制到某个 Survivor,年龄 +1;每熬过一次 Minor GC 年龄增长,超过阈值(默认 15,-XX:MaxTenuringThreshold)晋升到老年代。两个 Survivor 轮流做”复制目标”,始终有一个为空(复制算法,避免碎片)。
  • 老年代满触发 Major GC(老年代回收);Full GC 通常指整堆 + 元空间回收,最慢,应尽量避免。(注:“Major GC”是分代式收集器的经典术语;G1/ZGC 等现代收集器说法不同——G1 分 young / mixed / full,其中 Full GC 是它退化时的兜底全堆回收。)
GC 类型 回收范围 触发 停顿
Minor GC 新生代 Eden 满 短
Major GC 老年代 老年代满 长
Full GC 整堆 + 元空间 老年代/元空间不足、System.gc() 最长

八、可达性分析与 GC Roots

Java 不用引用计数(无法处理循环引用),而用可达性分析(reachability analysis):从一组 GC Roots 出发遍历引用图,凡不可达的对象即可回收。

GC Roots 包括:

  • 各线程栈帧局部变量表中的引用(正在执行的方法引用的对象);
  • 静态字段引用的对象;
  • 常量池引用、JNI 全局引用;
  • 同步锁持有的对象等。
GC Roots
  | 线程栈局部变量  ---> A ---> B      (A、B 可达,存活)
  | 静态字段        ---> C
                          C <---> D    (C、D 互相引用)
                                E ---> F  (E、F 互相引用但无 Root 可达 => 都回收)

注意最后一组:E、F 循环引用,引用计数法会误判为存活,而可达性分析正确判定它们为垃圾。这正是 Java 不用引用计数的原因。

九、收集器演进与 STW 权衡

STW(Stop-The-World):GC 的某些阶段必须暂停所有应用线程(到达”安全点 safepoint”),以获得一致的堆快照。收集器的进化史,本质是不断缩短 STW的历史。

收集器 特点 STW 适用
Serial 单线程,简单 全程 STW,较长 小堆、客户端
Parallel(吞吐优先) 多线程并行回收 STW 但并行 批处理,重吞吐
CMS(JDK9 弃用、JDK14 已移除) 并发标记清除,低停顿先驱 部分并发 老年代低延迟(有碎片问题)
G1(JDK9+ 默认) Region 分区、可预测停顿目标 短 STW,大部分并发 大堆通用
ZGC 染色指针 + 读屏障 停顿 <1ms,与堆大小无关 超大堆、低延迟
Shenandoah 并发压缩 停顿极短 低延迟

权衡的核心是吞吐量 vs 延迟:Parallel 追求单位时间处理最多工作(适合后台批处理);ZGC/Shenandoah 牺牲一点吞吐换取几乎无感的停顿(适合交互/在线服务)。没有”最好”,只有”最匹配场景”。

选择:-XX:+UseG1GC、-XX:+UseZGC、-XX:+UseParallelGC。

十、finalize 弃用与 Cleaner

释放资源永远优先用你已经会的 try-with-resources/显式 close()——它是确定性的,用完立刻执行。Cleaner 只是”万一你忘了调用 close()“的兜底网,不该当成主要手段:

import java.lang.ref.Cleaner;

public class QuickCleaner implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();
    private final Cleaner.Cleanable cleanable;

    QuickCleaner() {
        // 注册一个"忘记 close 时的兜底清理动作"
        cleanable = CLEANER.register(this, () -> System.out.println("兜底清理被触发"));
    }

    @Override
    public void close() {
        System.out.println("显式 close() 被调用"); // 首选路径:确定性、立即执行
        cleanable.clean();                          // 手动触发清理,之后兜底动作不会再重复执行
    }

    public static void main(String[] args) {
        try (QuickCleaner r = new QuickCleaner()) {
            System.out.println("使用资源");
        }
        // 因为 try-with-resources 已经调用了 close(),兜底的 Cleaner 动作只打印一次
    }
}
// 使用资源
// 显式 close() 被调用
// 兜底清理被触发

close() 里主动调用 cleanable.clean(),清理动作立刻执行一次;如果你忘了写 try-with-resources、这个对象最终变得不可达,GC 之后 Cleaner 内部的后台线程也会自动跑一次同样的动作——但那是”最后一道保险”,不该是你依赖的主路径。

C++ 的析构函数确定性执行。Java 的 Object.finalize() 曾想扮演这个角色,但它执行时机不确定、可能永不执行、拖慢 GC、还能复活对象,已在 JDK9 标记 @Deprecated,JDK18 起进一步废弃。

替代方案:java.lang.ref.Cleaner,注册一个清理动作,当对象变得幻象可达时由后台线程执行。但它仍非确定性——关键资源永远用 try-with-resources 显式释放,Cleaner 只作最后兜底。

import java.lang.ref.Cleaner;
public class CleanerDemo implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();
    // 清理动作必须是静态/独立类,绝不能持有外层对象引用,否则对象永不可回收
    static class State implements Runnable {
        public void run() { System.out.println("释放本地资源(兜底)"); }
    }
    private final Cleaner.Cleanable cleanable;
    CleanerDemo() { cleanable = CLEANER.register(this, new State()); }
    @Override public void close() { cleanable.clean(); } // 显式路径:确定性释放
}

十一、引用强度:强/软/弱/虚

平时写的 Object o = new Object() 都是强引用——只要它可达,GC 绝不会碰它。如果你想要”这个对象我还想用,但内存紧张时可以先牺牲它”这种效果,把它包一层 WeakReference:

import java.lang.ref.WeakReference;

public class QuickWeakRef {
    public static void main(String[] args) {
        Object obj = new Object();
        WeakReference<Object> weak = new WeakReference<>(obj); // 包一层弱引用
        System.out.println("get() 之前: " + (weak.get() != null)); // true —— 还有强引用 obj

        obj = null;      // 去掉唯一的强引用
        System.gc();      // 建议 JVM 做一次垃圾回收(不保证立即执行)
        System.out.println("get() 之后: " + weak.get()); // 很可能是 null
    }
}
// get() 之前: true
// get() 之后: null

weak.get() 拿到的是被引用的对象,或者 null(已被回收)。只要 obj 这个强引用还在,weak.get() 就一直有效;把 obj 置空、并且再没有别的强引用指着它之后,GC 就有权回收它,weak.get() 随之变成 null。

java.lang.ref 提供四种引用强度,决定对象在 GC 眼中的”生存权”:

强度 类 GC 行为 典型用途
强引用 普通 Object o = ... 可达就绝不回收 默认
软引用 SoftReference 内存不足时才回收 内存敏感缓存
弱引用 WeakReference 下次 GC 即回收 WeakHashMap、规范化映射
虚引用 PhantomReference 永远取不到对象,仅作回收通知 替代 finalize 做清理 
import java.lang.ref.*;
public class Refs {
    public static void main(String[] args) {
        ReferenceQueue<byte[]> queue = new ReferenceQueue<>();
        byte[] data = new byte[1024];
        // 弱引用 + 引用队列:对象被回收后,ref 会被入队,可据此感知回收
        WeakReference<byte[]> weak = new WeakReference<>(data, queue);
        data = null;          // 去掉唯一强引用
        System.gc();          // 建议 GC(不保证立即执行)
        System.out.println("回收后 get(): " + weak.get()); // 很可能是 null
        Reference<?> polled = queue.poll();                 // 已回收的引用被入队
        System.out.println("入队: " + (polled == weak));
    }
}

ReferenceQueue 是配套机制:当被引用对象被回收,对应的 Reference 对象被 JVM 放入队列,你的清理线程 poll 到后执行善后(Cleaner 内部正是这么实现的)。

弱引用最常见的落地是 WeakHashMap:key 只被弱引用持有,一旦外部再无强引用,GC 后条目自动消失——天然适合做规范化缓存,避免”只加不删”的泄漏:

import java.util.*;
public class WeakMap {
    static void seed(Map<Object,String> m) {          // key 只存活在这个栈帧里
        m.put(new Object(), "v");
        m.put(new Object(), "v");
    }
    public static void main(String[] args) throws Exception {
        Map<Object,String> weak = new WeakHashMap<>();
        seed(weak);
        System.out.println("before GC  weak.size = " + weak.size());
        System.gc();
        Thread.sleep(100);
        System.out.println("after  GC  weak.size = " + weak.size());
    }
}
// before GC  weak.size = 2
// after  GC  weak.size = 0    <- key 无强引用,GC 回收后条目被清除
// (注意:若把 key 留在 main 的局部变量里,它会成为 GC Root 而不被回收)

十二、Java 里内存泄漏长什么样

有 GC 不代表没内存泄漏。泄漏 = 对象逻辑上没用了,但仍从 GC Roots 可达,GC 因而不回收。常见形态:

  • 长生命周期集合只加不删:static Map/缓存不断 put,键值永远可达。用 WeakHashMap 或设容量上限(如 Guava/Caffeine 缓存)。
  • 监听器/回调未注销:观察者注册后忘记 removeListener,被主题对象一直引用。
  • ThreadLocal 未 remove:线程池线程长期存活,ThreadLocal 的 value 随线程一直可达。用完必须 remove()。
  • 内部类隐式持有外部实例:非静态内部类/匿名类持有 Outer.this,若其生命周期比外部对象长,外部对象无法回收(Cleaner 那节的告诫同源)。

十三、常见 OOM 与调优参数

OutOfMemoryError 是 Error,细分为不同信息:

错误信息 含义 方向
Java heap space 堆放不下 增大 -Xmx 或查泄漏(堆转储分析)
GC overhead limit exceeded 花 98%+ 时间 GC 却回收很少 通常是泄漏或堆太小
Metaspace 类元数据超限(动态生成大量类) -XX:MaxMetaspaceSize,查类加载器泄漏
unable to create new native thread 线程数超系统/内存限制 减少线程、调 -Xss
Direct buffer memory 堆外直接内存耗尽 -XX:MaxDirectMemorySize

常用参数:

-Xms512m -Xmx4g                  # 初始/最大堆(生产建议两者相等,避免动态扩容抖动)
-Xss1m                           # 每线程栈大小
-XX:MaxMetaspaceSize=256m        # 元空间上限
-XX:+UseG1GC -XX:MaxGCPauseMillis=200  # 用 G1 并设停顿目标
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof  # OOM 时导出堆快照
-Xlog:gc*                        # 打印 GC 日志(JDK9+ 统一日志),用于分析

诊断流程:-Xlog:gc* 看 GC 频率与停顿 → OOM 时拿 .hprof → 用 MAT/VisualVM 分析支配树(dominator tree)找出谁在撑着大对象不放。

小结

  • 用法层面:try/catch 按类型接住异常,finally 保证总执行;try-with-resources 让实现了 AutoCloseable 的资源用完自动 close();catch (A | B e) 一次接住多种类型;受检异常必须在方法签名写 throws 或就地捕获,非受检异常没有这个约束;GC 自动运行不用你操心,但 WeakReference/SoftReference/Cleaner 是你能主动用的几个相关 API,资源释放仍优先用显式 close(),Cleaner 只当兜底。
  • 底层层面:异常靠方法自带的异常表定位处理器,找不到就逐帧栈展开;finally 靠编译期代码复制实现”必然执行”;try-with-resources 按声明逆序调用 close(),close() 抛出的异常会作为 suppressed 挂在主异常上;异常链靠 cause 保留根因。GC 不用引用计数(处理不了循环引用),靠可达性分析从 GC Roots 出发判活;堆按弱分代假设分 Eden/Survivor/老年代,Minor/Major/Full GC 回收范围递增;收集器从 Serial 到 G1/ZGC 的演进史就是压缩 STW 停顿的历史;finalize 已弃用,Cleaner + PhantomReference/ReferenceQueue 是它的替代机制;内存泄漏在 Java 里表现为”逻辑上没用但仍可达”。掌握这些,你就能像管理 C++ 内存那样,自信地驾驭 JVM 的堆。
