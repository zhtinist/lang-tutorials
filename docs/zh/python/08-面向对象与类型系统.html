面向对象与类型系统

本章讲 Python 的类、继承、方法这套面向对象机制在 CPython 里的内部结构,和 C++ 的对象模型逐一对照。C++ 的对象模型是”编译期定死的内存布局 + vtable”;Python 的对象模型是”运行时可变的字典 + 描述符协议”。前提是「变量与内存模型」里确立的原则:一切皆对象、变量即引用,这一章把它用到”类”和”实例”这两个概念上。

一、class 语法与实例的内存布局

class Dog:
    def __init__(self, name, breed):
        self.name = name       # 实例属性
        self.breed = breed

    def bark(self):            # 实例方法,第一个参数总是 self
        return f"{self.name} says Woof!"

d = Dog("Rex", "Labrador")
print(d.name, d.bark())         # Rex Rex says Woof!

类还可以有类属性——写在类体里、方法之外,所有实例共享:

class Point:
    count = 0                 # 类属性(所有实例共享)
    def __init__(self, x, y):
        self.x = x            # 实例属性,每个实例各自一份
        self.y = y
        Point.count += 1      # 通过类名修改类属性

p = Point(3, 4)
print(p.x, p.__dict__)        # 3 {'x': 3, 'y': 4}
q = Point(1, 1)
print(Point.count)             # 2 —— 两个实例共享同一个 count

C++ 的对象是一块连续内存,字段在编译期就有固定偏移:

C++  struct Point { int x; int y; };
p:  [ x=3 ][ y=4 ]      // 访问 p.x 就是 *(int*)(&p + 0),偏移写死在指令里

Python 默认下每个实例带一个 __dict__(一个 dict),是属性名到对象引用的映射:

Python 实例 p(堆对象)
+------------------+
| ob_refcnt        |
| ob_type -> Point |
| __dict__ ------- |----> {'x': -> PyLong(3), 'y': -> PyLong(4)}
+------------------+

为什么实例属性默认存在 dict 里,而不是像 C++ 那样在编译期分配固定偏移:C++ 能这样做的前提是编译器在编译期就完全知道一个类有哪些字段——struct Point { int x; int y; }; 一旦写定,任何 Point 实例的字段集合终生不变。Python 的类定义在运行期执行(见「运行模型」),而且实例的属性集合允许在创建后继续变化——p.z = 5 这种运行时动态加属性在 Python 里是合法操作,编译器不可能预先算出一个”还没被写出来的属性”该在哪个固定偏移。既然属性集合本身是运行期开放的,存储结构就只能选一个支持动态增删键的容器,也就是 dict——这与「变量与内存模型」里”函数局部变量能用 fast locals 数组、而模块全局命名空间不能”是同一条推理:能在编译期确定的集合才能用固定偏移优化,运行期开放的集合只能退回字典。

后果是具体的:访问 p.x 是一次字典查找(见「复合数据结构」里的哈希表),不是常数偏移解引用,更慢;但换来的是可以在运行时给任意实例增删属性,C++ 里对象的字段集合是编译期固定的,做不到。每个实例多背一个 dict,内存开销不小——CPython 因此有个优化叫 “key-sharing dict”:同一个类的多个实例,若属性集合相同,可以共享 key 部分,只各存 value 数组,省下重复存储字段名字符串的开销。这让”实例 dict”没那么贵,但仍然是一次字典查找,不如固定偏移。真正要把这份代价压到接近 C struct 的写法是 __slots__,见第八节。

二、类也是对象,type 是元类

print(type(p))         # <class '__main__.Point'>   —— p 的类型是 Point
print(type(42))        # <class 'int'>
print(isinstance(p, Point))   # True

在 Python 里,类本身也是一个对象,它的类型是 type:

print(type(Point))     # <class 'type'>              —— Point 的类型是 type
print(type(type))      # <class 'type'>              —— type 的类型是它自己

Point 是一个存在于堆上的对象,你可以把它赋值、传参、放进 list。创建类的”类”叫元类(metaclass),默认是 type。class Point: ... 本质上等价于调用 type(name, bases, namespace) 动态造出一个类对象:

Point2 = type('Point2', (object,), {'x': 0})   # 和 class 语句等价

这与 C++ 完全不同:C++ 的类只是编译期概念,运行时不存在”类对象”(RTTI 的 type_info 只是只读元数据,不能当值来构造新类)。为什么 Python 要让类本身也是一等公民:这与上一节同一个根源——Python 的类定义是运行期执行的一条语句(class 关键字触发的是运行时的求值和赋值),既然创建类这件事本身发生在运行期,类的产物(类对象)顺理成章也是一个可以被赋值、传参、内省的运行期对象,而不是像 C++ 那样在编译完成后就从运行时世界里消失的纯编译期记号。这个统一性是”一切皆对象”这条设计原则贯彻到底的结果,不是为类专门开的特例。元类是高级特性(ORM、框架用它做注册/校验),日常不用手写。

三、属性查找顺序:实例 → 类 → MRO

class Base:
    label = "base"

class Child(Base):
    pass

obj = Child()
print(obj.label)     # base —— 实例和 Child 都没有 label,沿着继承链在 Base 里找到
obj.label = "child"  # 给实例自己设置一个同名属性
print(obj.label)      # child —— 实例自己的优先
print(Base().label)   # base —— 没影响到 Base 类本身

访问 obj.attr 时,解释器按固定顺序查找:

  1. 数据描述符(定义了 __set__/__delete__ 的类属性,如 property)——优先于实例 dict;
  2. 实例 __dict__——obj.__dict__['attr'];
  3. 类及其 MRO——沿 type(obj).__mro__ 逐个类查 __dict__,期间遇到非数据描述符(如普通函数)会触发描述符协议;
  4. 都没有 → 调用 __getattr__(若定义)→ 否则 AttributeError。

多继承时,“类链”不是简单的深度优先,而是 C3 线性化算法算出的一个确定顺序,保证:子类在父类前、父类相对顺序保持、且唯一确定(否则报错):

class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print([c.__name__ for c in D.__mro__])
# ['D', 'B', 'C', 'A', 'object']

为什么 Python 要引入 C3 线性化,而不是像 C++ 那样把菱形继承的消歧问题交给程序员用 virtual 继承手动解决:C++ 的多继承里,D(B, C) 若 B、C 都各自继承 A,默认会在 D 的内存布局里放两份 A 子对象——这在语义上通常不是程序员想要的,需要显式声明 virtual 继承才能让 B、C 共享同一份 A 子对象,而这又要求 A 提前预知自己可能被虚继承(调整内存布局和构造顺序的规则),把决定权和维护负担都留给了写这几个类的人,规则复杂且容易出错,是 C++ 里公认的复杂度重灾区。Python 从设计上就没有”每个基类各一份子对象”这个选项——继承关系的实质是”到哪些类里按什么顺序查找属性/方法”,而不是内存布局的拼接,所以只需要计算一个唯一确定的类查找顺序即可。C3 线性化就是解决”如何从多个可能冲突的继承顺序里唯一确定一个合法顺序”这个问题的通用算法:它保证结果既尊重每个类自己声明的继承顺序,又保证子类永远排在父类前面,且矛盾时报错而不是给出一个模糊的默认行为(这与后面「运行模型」和「基础语法」反复出现的”能确定性地报错就不留不确定行为”是一致的设计脉络)。代价是程序员需要理解 C3 计算出的顺序,而不能像 C++ 那样精确控制”这个字段到底存了几份、构造顺序具体是什么样”——但对属性/方法查找而言,C3 给出的自动化顺序在几乎所有场景下已经足够。

四、方法 = 函数 + 描述符协议

class Foo:
    def bar(self, n):
        return f"self is {self}, n is {n}"

f = Foo()
print(f.bar(10))     # self is <__main__.Foo object at ...>, n is 10

一个”方法”其实就是定义在类 __dict__ 里的普通函数。神奇之处在于:函数是一个描述符(实现了 __get__)。

print(Foo.bar)      # <function Foo.bar at ...>       —— 通过类拿到的是纯函数
print(f.bar)        # <bound method Foo.bar of ...>   —— 通过实例拿到的是"绑定方法"

f.bar 触发属性查找,在类里找到函数对象 bar,因为函数有 __get__,解释器调用 bar.__get__(f, Foo),它返回一个绑定方法对象——一个把 f 预先塞进第一个参数(self)的可调用包装:

f.bar()  等价于  Foo.bar(f)
         绑定方法内部保存了 (__func__ = bar, __self__ = f)
         调用时把 __self__ 作为第一个实参自动传入

这解释了为什么方法第一个参数必须显式写 self——它不是关键字,只是”绑定时被自动填入的那个参数”的约定名。对照 C++:C++ 的 this 是编译器隐式传入的隐藏指针,在编译期就已经确定”这次调用绑定哪个对象”;Python 把这件事显式化为描述符协议,self 究竟绑定谁是运行时通过 __get__ 现算出来的——这与整章的主题一致:C++ 把对象模型的机制固化进编译产物,Python 把同样的机制暴露成运行时可以拦截、重写的协议(__get__/__set__),第 5 节的 property 正是利用了这一点。

五、@property / @staticmethod / @classmethod

这三个装饰器改变方法的绑定行为,分别对应”像字段一样访问的方法”“不需要 self 的方法”“操作类本身的方法”:

class Circle:
    def __init__(self, r):
        self._r = r

    @property                       # 把方法伪装成只读属性,访问时自动调用
    def area(self):
        return 3.14159 * self._r ** 2

    @staticmethod                   # 不绑定 self/cls,就是个放在类命名空间的普通函数
    def unit():
        return "cm"

    @classmethod                    # 绑定的是"类"而非实例,第一个参数是 cls
    def from_diameter(cls, d):
        return cls(d / 2)           # cls 让子类调用时返回子类实例(多态构造)

c = Circle(2)
print(c.area)                       # 12.56636 —— 像字段一样访问,底层调了方法
print(Circle.unit())                # cm
print(Circle.from_diameter(10)._r)  # 5.0
装饰器 第一个参数 通过实例调用时 典型用途
普通方法 self(实例) 绑定实例 常规行为
@classmethod cls(类) 绑定类 备选构造函数、操作类状态
@staticmethod 无 不绑定 逻辑归属该类但不需 self/cls
@property self 访问即调用 计算属性、只读封装、加校验

@property 是 C++ 里”用 getter/setter 封装私有字段”的语言级支持,但对调用方透明:c.area 看起来是字段访问,不用改调用代码就能从字段升级成计算属性。它之所以能拦在实例 __dict__ 前面被优先访问,正是因为 property 实现了 __set__,是第三节说的”数据描述符”——查找顺序里排在实例 dict 之前。

六、继承与 super

class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return "..."

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)      # 调用父类 Animal 的 __init__
        self.breed = breed
    def speak(self):
        return "Woof"

d = Dog("Rex", "Labrador")
print(d.name, d.breed, d.speak())   # Rex Labrador Woof

super()(无参形式)不是”调父类”,而是”沿当前实例的 MRO 找下一个类”。在多继承里这至关重要——它保证协作式多继承时每个基类的方法恰好被调用一次,按 C3 顺序串起来。这比 C++ 里手写 Base::method()(硬编码具体基类)更灵活,也避免了菱形继承里基类被调两次这个真实存在的 bug:

class A:
    def __init__(self): print("A")
class B(A):
    def __init__(self): print("B"); super().__init__()
class C(A):
    def __init__(self): print("C"); super().__init__()
class D(B, C):
    def __init__(self): print("D"); super().__init__()

D()
# D
# B
# C
# A   <- 只有一次;若把 super() 换成硬编码 A.__init__(self),A 会被调两次

super() 之所以能保证 A.__init__ 只被调用一次,是因为它查的不是”我的直接父类是谁”这个静态信息,而是”在 D.__mro__ = [D, B, C, A, object] 这个顺序里,我当前所在位置的下一个是谁”这个动态信息——B.__init__ 里的 super() 知道自己是在为 D() 这次调用服务,MRO 是 [D,B,C,A,object],所以它调的下一个是 C 而非直接跳到 A。这正是 C3 线性化换来的实际收益:多继承协作不再依赖每个类自己知道完整的继承图,只需要知道”MRO 里我的下一个是谁”。

七、鸭子类型:核心哲学而非强制接口

class Duck:
    def speak(self): return "Quack"

class Person:
    def speak(self): return "Hello"

def make_it_speak(x):        # 不需要任何共同基类或接口声明
    return x.speak()         # 运行时按 x 的实际类型查方法

print(make_it_speak(Duck()))     # Quack
print(make_it_speak(Person()))   # Hello

“如果它走起来像鸭子、叫起来像鸭子,那就当它是鸭子”——这就是鸭子类型(duck typing)。C++ 的多态靠虚函数表(vtable):对象头部藏一个 vptr 指向 vtable,虚调用是”查表 + 间接跳转”,且类型在编译期检查——只有声明了共同基类、且该方法被声明为 virtual,才能多态。

为什么鸭子类型是 Python 的核心哲学,而不是”权宜之计”:C++ 的模型要求”能力”必须提前用类型系统声明出来——一个类要支持多态调用 speak(),必须先有一个共同基类把 speak 声明为 virtual,这个声明是运行前就要写死的契约。鸭子类型反过来,把”这个对象能不能做某件事”这个判断完全推迟到真正调用它的那一刻:x.speak() 执行时,解释器只关心此刻的 x 是否恰好有一个叫 speak 的可调用属性,不关心 x 的类型历史上是否”声明”过这个能力。这意味着两个毫不相干、没有共同基类的类(Duck、Person)只要凑巧都定义了同名方法,就能被同一段代码统一处理——不需要为了让这段通用代码工作,而回头给这两个类都加上一个共同的接口声明。这与 Python 动态类型这一根本设计(见「运行模型」)是同一枚硬币的两面:类型信息本来就要推迟到运行期才能确定,“这个对象支持什么操作”自然也就成了运行期才检查的事情,而不是在类型系统里提前登记的静态契约。

对照:

C++ 虚函数 Python 鸭子类型
分派时机 编译期定接口,运行时查 vtable 全在运行时查类型的 dict
需要共同基类 是(继承 + virtual) 否(有同名方法即可)
类型检查 编译期静态 运行时,错了才抛 AttributeError
开销 一次间接跳转 一次字典查找(更慢)

代价同样清楚:接口从没有在任何地方被显式声明过,一个对象到底该实现哪些方法才能被某段代码正确处理,只能靠阅读代码或文档得知,错误也推迟到真正调用那一刻才暴露为 AttributeError。折中方案见第十节的抽象基类,以及类型注解 + mypy 做静态检查——后者是在不改变鸭子类型运行时语义的前提下,把接口检查这件事挪回开发期完成。

八、__slots__:用固定数组替代 dict

默认的实例 __dict__ 灵活但费内存。若一个类要创建海量实例(如百万个 Point),可以用 __slots__ 声明固定字段集合:

class Point:
    __slots__ = ('x', 'y')      # 声明后:不再有 __dict__,字段存在固定槽位
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(3, 4)
# p.z = 5          # AttributeError:不能再动态加属性
# p.__dict__       # AttributeError:没有 __dict__ 了

CPython 在类型对象上为每个 slot 分配一个固定偏移的槽位(通过描述符实现),实例不再挂 dict,而是像 C struct 一样把字段值指针连续存在对象体内:

带 __dict__ 的实例:  [头][__dict__ ptr] -> {'x':.., 'y':..}   (额外一个 dict)
带 __slots__ 的实例:  [头][x ptr][y ptr]                        (字段内联,无 dict)

用 sys.getsizeof 实测两者内存差(CPython 3.14,64 位):

import sys
class WithDict:
    def __init__(self, x, y): self.x, self.y = x, y
class WithSlots:
    __slots__ = ('x', 'y')
    def __init__(self, x, y): self.x, self.y = x, y

a = WithDict(1, 2); b = WithSlots(1, 2)
print(sys.getsizeof(a) + sys.getsizeof(a.__dict__))  # 344  = 实例体 48 + __dict__ 296
print(sys.getsizeof(b))                              # 48   字段内联,无独立 dict
print(hasattr(b, "__dict__"))                        # False

这是本章第一节提出的问题(实例属性默认存 dict、访问是字典查找、每实例多背一个 dict)最直接的解药,也是一次教科书式的空间换速度取舍,只是方向和通常说的”空间换时间”相反——这里是”用更小的空间同时换来更快的访问”,代价是牺牲了灵活性:去掉 __dict__ 省下的是一整个哈希表对象(通常几百字节)的内存,换来的是属性访问从”哈希 + 查找”变成”固定偏移解引用”,两者同时受益是因为 __dict__ 本身既占内存又是访问变慢的直接原因;真正付出的代价是动态性——__slots__ 类的实例不能再 p.z = 5 临时加新属性,也没有 __dict__ 可以内省,多继承时如果多个基类都声明了非空 __slots__ 还会有布局冲突限制。对创建量极大、字段集合完全固定的类型(比如解析大量结构化数据时的记录类),这个取舍通常是划算的;对灵活性优先、实例数量不大的普通类,默认的 __dict__ 仍然是更省心的选择。

九、魔术方法与运算符重载

“魔术方法”(dunder methods,双下划线)是解释器在特定语法/操作时回调的钩子。定义了它们,你的对象就能用 +、==、len()、print() 这些内置语法:

class Vec:
    def __init__(self, x, y): self.x, self.y = x, y
    def __repr__(self):  return f"Vec({self.x}, {self.y})"   # repr()/交互式显示
    def __add__(self, o): return Vec(self.x+o.x, self.y+o.y) # 支持 v1 + v2
    def __eq__(self, o):  return (self.x, self.y) == (o.x, o.y)

v1 = Vec(1, 2)
v2 = Vec(3, 4)
print(v1 + v2)          # Vec(4, 6)   —— 触发 __add__
print(v1 == Vec(1, 2))  # True        —— 触发 __eq__
print(v1)                # Vec(1, 2)  —— 触发 __repr__

常见魔术方法:

语法/操作 魔术方法 对应 C++
a + b __add__ / __radd__ operator+
a == b __eq__ operator==
a < b __lt__ operator<
len(a) __len__ size() 约定
a[i] __getitem__/__setitem__ operator[]
a() __call__ operator()(仿函数)
str(a)/print __str__ operator<< 约定
with a: __enter__/__exit__ RAII(见第 9 章)
for x in a __iter__/__next__ 迭代器

运算符重载的能力和 C++ 相当,但全在运行时通过查方法完成,没有编译期重载决议:v1 + v2 本质是解释器发现 + 号,去 type(v1) 里查 __add__ 并调用——和第四节方法调用是同一套查找机制,没有额外魔法,这也是鸭子类型思路的延伸:解释器不关心 v1 是不是”声明支持加法”的某个类型,只关心此刻它有没有 __add__。

十、抽象基类(abc)

鸭子类型灵活,但有时你想强制”子类必须实现某些方法”。abc 模块提供抽象基类:

from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...    # 抽象方法,子类必须实现

class Square(Shape):
    def __init__(self, s): self.s = s
    def area(self): return self.s * self.s

# Shape()      # TypeError:不能实例化含未实现抽象方法的类
print(Square(3).area())   # 9

这类似 C++ 的纯虚函数(virtual void area() = 0;)+ 抽象类不可实例化,但检查时机是实例化时(运行时),不是编译期——这正是鸭子类型哲学下能做到的最强承诺:Python 无法在”定义子类”那一刻就静态验证契约(没有编译期类型检查这个阶段),只能退而求其次,在”创建实例”这个运行时时刻做一次检查,把”忘记实现某个方法”这类错误至少提早到构造对象的时候暴露,而不是拖到真正调用那个缺失方法时才暴露。abc 还支持”虚拟子类”注册和结构化的 isinstance 检查;collections.abc 定义了 Iterable/Sequence/Mapping 等协议基类,可用于类型判断。更轻量的现代做法是 typing.Protocol(结构化子类型 / 静态鸭子类型),配合 mypy 在编译期(检查期)验证接口,而不侵入运行时——它让鸭子类型”只要有同名方法就能用”这条运行时规则,同时获得一份可选的静态检查,两者并不冲突。

小结

  • 实例属性默认存 __dict__:因为类定义和属性集合都在运行期才能确定,编译期无法像 C++ 那样分配固定偏移,访问因此是字典查找而非常数偏移解引用。类本身也是运行时对象,type 是默认元类,这是”一切皆对象”贯彻到底的结果。
  • 属性查找顺序是数据描述符 → 实例 dict → MRO(C3 线性化);C3 存在的意义是把 C++ 需要程序员用 virtual 继承手动解决的菱形继承问题,变成一个自动计算、确定唯一顺序的算法。方法是函数 + 描述符协议,self/绑定方法由 __get__ 在运行时产生;super() 沿 MRO 找下一个类而非硬编码父类,保证协作式多继承里每个基类恰好被初始化一次。
  • 鸭子类型是核心哲学而非权宜之计:能力检查被推迟到调用那一刻,不需要提前用共同接口声明”我支持什么操作”,代价是错误只能在运行时暴露;abc 和 typing.Protocol 是在不放弃这一哲学的前提下,把契约检查分别挪到实例化时和开发期的两种折中。
  • __slots__ 是一次实打实的空间换速度:去掉实例 __dict__ 换来更小的内存和固定偏移的属性访问,代价是不能再动态增删属性。
