运行模型：CPython 是怎么把 .py 变成行为的

作为 C/C++ 工程师，你脑子里的心智模型大概是：gcc 把 .c 编译成汇编、再汇编成目标文件、链接成 ELF/Mach-O 可执行文件，操作系统把它 exec 进内存，CPU 直接执行机器码。这里没有”运行时”在中间翻译，int x = 3; 就是往栈上某个偏移写 4 字节。

Python 完全不是这样。理解它的运行模型，是理解后面所有”诡异”行为（别名、GIL、动态类型开销）的地基。本章只讲一件事：从源码到行为，中间隔着什么。

一、一句话对照

阶段 C/C++ CPython
源码 .c / .cpp .py
前端 预处理 + 编译 词法/语法分析 → AST
中间产物 .o 目标文件（机器码） .pyc 字节码（bytecode）
链接 链接器生成可执行文件 无独立链接步骤
执行 CPU 直接跑机器码 解释器（虚拟机）求值循环逐条解释字节码
类型检查 编译期 运行期

关键区别：C 编译完就没编译器什么事了，程序自己在裸机上跑。CPython 编译出的字节码不是机器码，需要一个常驻的”解释器”（一个 C 写的巨大循环）来逐条执行。这个解释器就是 python 这个可执行文件本身。

二、CPython 是”C 写的解释器”

先分清两个概念：

  • Python 是一门语言，是一份语言规范（语义）。
  • CPython 是这门语言的参考实现，用 C 写成。你 brew install python 装的、python3 跑起来的，99% 情况就是 CPython。

还有别的实现：PyPy（带 JIT）、Jython（跑在 JVM 上）、GraalPy 等。本教程凡涉及实现细节，一律以 CPython 为准，并显式标注”在 CPython 中”。语言语义与实现细节的区分，对你这种要抠底层的人极其重要——很多”Python 的特性”其实只是”CPython 的实现选择”。

三、四个阶段：源码 → AST → 字节码 → 求值

当你运行 python foo.py，CPython 内部大致经历：

foo.py (源码文本)
   │  词法分析 (tokenize)
   ▼
Token 流
   │  语法分析 (parse)
   ▼
AST (抽象语法树)          <- 可用 ast 模块查看
   │  编译 (compile)
   ▼
Code Object (含字节码)    <- .pyc 里存的就是它
   │  解释执行 (ceval.c 的求值循环)
   ▼
程序行为 / 副作用

3.1 AST：语法树

Python 暴露了编译中间态，这在 C 世界里很奢侈。

import ast

src = "x = a + b * 2"
tree = ast.parse(src)          # 解析成 AST
print(ast.dump(tree, indent=2))

你会看到 Assign 节点，目标是 Name(id='x')，值是 BinOp(left=Name('a'), op=Add, right=BinOp(...))。这跟编译原理课上的语法树一模一样。CPython 内部就是把这棵树编译成字节码。

3.2 字节码与 .pyc

编译产物是 code object，其中 co_code 是一串字节码指令。当你 import 一个模块时，CPython 会把编译结果缓存到 __pycache__/模块名.cpython-<版本>.pyc（版本号随解释器,如 CPython 3.14 是 cpython-314.pyc），下次导入若源码没改（比对 mtime/hash）就直接加载 .pyc，跳过重新编译。

注意： - .pyc 不是机器码，不能脱离解释器运行。它只是”编译好的字节码”，省的是重新解析编译的时间，省不了解释执行的开销。 - 直接 python foo.py 运行的主脚本不会生成 .pyc（只有被 import 的模块才缓存）。 - 这类比 C 的 .o：都是中间产物、都可缓存复用，但 .o 里是真机器码，.pyc 里是给虚拟机吃的字节码。

3.3 用 dis 看字节码

dis 模块把字节码反汇编成人类可读形式，是你窥探底层的显微镜。

import dis

def add(a, b):
    return a + b

dis.dis(add)

输出（Python 3.12，不同版本指令略有差异）：

  2           RESUME                   0

  3           LOAD_FAST                0 (a)
              LOAD_FAST                1 (b)
              BINARY_OP                0 (+)
              RETURN_VALUE

逐条解释： - LOAD_FAST 0 (a)：把局部变量 a 压入求值栈。（FAST 指走的是数组下标而非字典查找，详见第03章。） - LOAD_FAST 1 (b)：把 b 压栈。 - BINARY_OP 0 (+)：弹出栈顶两个对象，执行加法，结果压栈。 - RETURN_VALUE：弹出栈顶作为返回值。

这里的核心洞察：CPython 是一台栈式虚拟机（stack machine），不是寄存器机。所有运算都围绕一个”求值栈”（value stack）做压栈/弹栈。这跟 JVM 类似，跟 x86 那种寄存器 + 内存的机器模型不同。你看到的每条 LOAD_* 都在往这个栈推数据。

3.4 求值循环（the eval loop）

CPython 执行字节码的核心，是 Python/ceval.c 里一个巨大的循环，伪代码：

for (;;) {
    opcode = NEXT_INSTRUCTION();
    switch (opcode) {
        case LOAD_FAST:  ... push(...); break;
        case BINARY_OP:  ... ; break;
        case RETURN_VALUE: ...; return;
        ...
    }
}

每次转一圈解释一条指令。对比 CPU 的取指-译码-执行：概念一样，但这层是软件模拟的，所以慢。一条 Python a + b 要经过：取字节码 → switch 分派 → 查 a 的类型 → 找 __add__ → 调用 → 装箱结果。而 C 的 a + b 可能就是一条 add 指令。这就是 Python “慢” 的根本来源——不是语言笨，是多了一整层解释。

四、为什么”一切皆对象，且都在堆上”

这是与 C 最本质的分野，务必吃透。

在 C 里：

int x = 3;   // x 是栈上一个 4 字节的槽，里面就是整数 3 的补码

x 是一块内存，值直接内联在里面。取地址 &x 得到栈地址。

在 Python 里，x = 3 完全是另一回事。CPython 中每个值都是一个 PyObject，分配在堆上。简化后的结构（见 Include/object.h）：

typedef struct _object {
    Py_ssize_t ob_refcnt;   // 引用计数
    PyTypeObject *ob_type;  // 指向类型对象的指针
} PyObject;
  • ob_refcnt：有多少个名字/容器正引用着我。归零就释放（详见第03章）。
  • ob_type：指向”类型对象”，回答”我是谁”。整数指向 int 类型对象，其中记录了怎么加、怎么打印等。

这个 ob_type 指针，你完全可以类比 C++ 对象里的 vtable 指针：对象自己带着”我该怎么被操作”的信息，方法调用是运行期通过它分派的。区别是 Python 里连整数、None 这种都带，且分派完全动态。

一个具体的 int（PyLongObject）在 PyObject 头之后还跟着实际数值的数字位。所以内存里：

栈上的名字 x            堆上的 PyLongObject
┌─────────┐            ┌──────────────────┐
│ 指针   ─┼──────────► │ ob_refcnt = 1    │
└─────────┘            │ ob_type   ──► int │
   (一个机器字)         │ 数值: 3          │
                       └──────────────────┘

对照 C 的 int x = 3：

C：栈上           Python：栈上名字 + 堆上对象
┌───────┐        ┌───────┐        ┌───────────┐
│  3    │        │ 指针 ─┼──────► │ ...3(对象) │
└───────┘        └───────┘        └───────────┘
4 字节，值即数据   一个指针         一次堆分配

所以： - Python 的”变量”永远是指向堆对象的引用（本质是个指针），从不”内联存值”。x = 3 是”让名字 x 指向那个表示 3 的堆对象”。 - 没有”栈上的值类型”。连 1、True、None 都是堆对象（None 是全局唯一单例）。 - 一个整数对象在 64 位 CPython 上要占约 28 字节，而不是 4 字节——因为要带对象头。这解释了为什么 Python 数值密集计算内存和速度都吃亏，也是 NumPy 存在的理由（见第12章）。

语义 vs 实现：语言只保证”一切皆对象”的行为语义。“都在堆上”“对象头是这两个字段”是 CPython 的实现。PyPy 用了不同的对象表示和逃逸分析，很多短命对象根本不真正堆分配。

小整数其实有缓存优化（[-5, 256]），字符串有驻留，这些也是 CPython 的实现细节，放到第03章专门拆解，那里才是本书内存模型的主战场。

五、CPython vs PyPy：解释 vs JIT

既然慢的根源是”逐条软件解释”，自然想到：能不能把热点字节码即时编译成机器码？这正是 PyPy 做的事。

CPython PyPy
执行方式 纯解释求值循环 追踪式 JIT：热点代码编译为机器码
首次/冷代码 无预热 有预热开销
长循环/热点 慢 可快数倍到数十倍
C 扩展兼容 原生（CPython API） 较弱（cffi 更佳）
对象表示 每个对象老实堆分配 逃逸分析后可优化掉

JIT 的思路你不陌生：跟 JVM 的 HotSpot 一样，先解释、统计热点、再把热路径编成机器码并做特化（比如假定循环里 a、b 一直是 int，就省掉每次的类型查找，一旦假设失败再回退）。CPython 从 3.11 起也引入了”专门化自适应解释器”（specializing adaptive interpreter，PEP 659），在字节码层面做轻量特化，但那还不是完整 JIT（3.13+ 有实验性 JIT）。

对你的实践含义：先用 CPython（生态、C 扩展最全）；纯 Python 的 CPU 密集长任务且不依赖重 C 扩展时，PyPy 可能白捡几倍性能。

六、GIL：先埋个伏笔

你肯定会问：这个求值循环是线程安全的吗？多核能并行跑吗？

简短答案：在 CPython 中有个全局解释器锁（GIL），同一进程内任一时刻只有一个线程在执行 Python 字节码。它的存在是为了让引用计数（ob_refcnt 的增减）等操作无需处处加锁，实现简单且单线程快。代价是：纯 Python 计算无法靠多线程吃满多核。

这对习惯了 pthread / std::thread 真并行的你是个大坑。但它是 CPython 的实现选择（PyPy 也有，Jython 没有），且 Python 3.13 起有实验性的 no-GIL 构建。完整机制、绕过手段（多进程、C 扩展释放 GIL、asyncio）留到第10章专门讲，这里只需记住：GIL 是解释器级别的锁，不是语言语义。

七、小结

  • CPython 把 .py 编译成字节码（缓存为 .pyc），再由 C 写的栈式虚拟机求值循环逐条解释执行——多了一整层软件翻译，这是慢的根源。
  • 用 ast.parse 看语法树，用 dis.dis 看字节码，随时能验证底层发生了什么。
  • 一切皆对象，在 CPython 中都分配在堆上，每个对象带 ob_refcnt（引用计数）和 ob_type（类似 vtable 指针）。变量是指向堆对象的引用，绝不内联存值——这与 C 的栈上值类型根本不同。
  • 区分 Python（语义）与 CPython（实现）；PyPy 用 JIT 提速；GIL 是 CPython 的解释器级锁，细节见第10章。
