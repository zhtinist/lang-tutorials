变量与内存模型 ⭐

这是全书最重要的一章。如果你只读一章，读这章。

对 C/C++ 工程师来说，Python 90% 的”灵异事件”——改了一个列表另一个跟着变、函数里改了参数外面也变、默认参数记住了上次的值——根源都是同一件事：Python 的变量不是内存槽，而是名字到堆对象的引用绑定。 一旦这个模型在你脑中扎根，一切诡异都变得可预测。

本章先给你一段”变量怎么赋值”的最基础用法——这部分你可能已经会了；然后从第二节开始进入本书最核心的底层模型，我们会用 id()、is、sys.getrefcount 反复做实证，多画内存图。

一、先会用：变量怎么赋值

Python 里给变量赋值，不需要像 C 那样先声明类型，直接写：

x = 3
print(x)          # 3

x = "hello"       # 重新赋值，甚至换了"类型"也没问题
print(x)          # hello

a, b = 1, 2       # 一行同时给两个变量赋值
print(a, b)       # 1 2

y = x             # 把 x 现在的值"给" y
print(y)          # hello

这些操作你可能已经很熟了：赋值、重新赋值、一行赋多个值。但 y = x 这一行，在 Python 里到底发生了什么——是把值拷贝了一份给 y，还是别的什么？这正是本章要彻底讲清楚的问题。下面开始进入底层。

二、开始深入：变量不是盒子，是名签

C 的心智模型是”盒子”：int x 就是一块命名的内存，x = 3 往盒子里塞值，x = 5 覆盖它。变量就是那块内存。

Python 完全不是。回顾 第01章：所有值都是堆上的 PyObject。变量只是命名空间里的一个名字，绑定到某个堆对象上。你可以把它想成一个”贴在对象上的便利贴”，或者——用你熟悉的话说——变量是一个持有对象地址的指针，但你永远不能对它取地址、也做不了指针算术。

C 的模型：变量即内存         Python 的模型：名字指向堆对象
┌─────────┐                名字 x         堆
│  x: 3   │  ← 盒子即值     ┌────┐        ┌──────────────┐
└─────────┘                │ ●──┼──────► │ int 对象: 3  │
                           └────┘        └──────────────┘
                           便利贴          真正的值在这

x = 3 的准确读法不是”把 3 存进 x”，而是”让名字 x 绑定到那个表示 3 的对象”。这个区别是本章一切的地基。

用 id() 观察对象身份

id(obj) 返回对象的身份标识；在 CPython 中它就是对象的内存地址（语言只保证同一生命周期内唯一且恒定，地址是实现细节）。它就是你想要的 &——只不过是只读的、针对对象而非变量的。

x = 3
print(id(x))   # 例如 4310721234，就是那个 int 对象的堆地址

三、a = b 到底拷贝了什么

看这段，先猜结果：

a = [1, 2, 3]
b = a
b.append(4)
print(a)        # ???

答案是 [1, 2, 3, 4]。C++ 工程师第一反应是”b = a 不是拷了一份吗？改 b 怎么动了 a？”

因为 b = a 既不拷贝对象，也不拷贝数据，只拷贝了那个指针值。执行后，a 和 b 是贴在同一个堆对象上的两张便利贴。

a = [1, 2, 3]
b = a

  名字 a ─┐
          ├──► ┌──────────────────────┐
  名字 b ─┘    │ list 对象            │
               │ [ ●, ●, ● ]          │ ← 同一个对象
               └──┼──┼──┼─────────────┘
                  ▼  ▼  ▼
                  1  2  3   (各自也是堆上的 int 对象)

b.append(4) 是通过 b 找到那个 list 对象，原地修改它。a 指向的是同一个对象，所以从 a 看过去当然也变了。这不是拷贝语义的 bug，是别名（aliasing）——两个名字是同一对象的两个别名。

类比 C++：b = a 更像 int* b = a;（拷贝指针），而绝不是 std::vector<int> b = a;（拷贝构造，深拷贝一份）。Python 的赋值永远是”拷贝引用”，从不调用什么拷贝构造函数。

用 is / == / id 证明别名

  • == 比较值是否相等（调用 __eq__）。
  • is 比较是否同一个对象（比较 id），等价于 C 的 a == b 比较两个指针。
a = [1, 2, 3]
b = a
c = [1, 2, 3]          # 另建一个内容相同的列表

print(a == b)          # True，值相等
print(a is b)          # True，同一对象（别名）
print(id(a) == id(b))  # True

print(a == c)          # True，值相等
print(a is c)          # False！不同对象
print(id(a), id(c))    # 两个不同地址

a is b 为真而 a is c 为假，白纸黑字证明：b = a 制造了别名，而 c = [1,2,3] 是全新对象。判断”是不是同一个东西”用 is，“值一不一样”用 ==。（is None 的惯用法正来自这里——None 是单例，比身份最合适。）

四、引用计数：谁在指着我

回顾 第01章：每个 PyObject 头里有 ob_refcnt。每多一个名字/容器指向它就 +1，少一个就 -1，归零立即回收（CPython 的主要内存管理机制，另有循环垃圾回收器处理引用环，见 第09章）。这就是 Python 版的”自动 free”——你不用 free，引用计数替你做。

import sys

x = object()
print(sys.getrefcount(x))   # 例如 2

y = x                        # 多一个引用
print(sys.getrefcount(x))   # 3

del y                        # 去掉一个引用（del 解除名字绑定，不是删对象！）
print(sys.getrefcount(x))   # 2 回落

两个关键点： 1. sys.getrefcount(x) 的结果总比你直觉多 1，因为把 x 作为实参传给函数这个动作本身又临时加了一次引用。 2. del y 删除的是名字绑定，不是对象。对象在 refcnt 归零时才真正释放。这跟 C++ 的 delete 语义完全不同——del 更像”松开这只手”，最后一只手松开对象才没。

del 的语义                  refcnt 归零才回收
名字 y ──╳ (解绑)           object ← 只剩 x 指着，refcnt=1，还活着
名字 x ─────► object        当 x 也 del，refcnt=0 → 立即 free

五、可变 vs 不可变：决定一切别名行为

现在回到那个”改 b 动 a”的问题。为什么下面这段没有同样的”问题”？

a = 1
b = a
b = b + 1
print(a, b)     # 1 2   —— a 没变！

区别在于：list 是可变（mutable）的，int 是不可变（immutable）的。

  • 可变对象（list、dict、set、以及大多数自定义类实例）：可以原地修改内容。b.append(4)、b[0] = 9 改的是对象本身，所有别名都看得到。
  • 不可变对象（int、float、str、tuple、bool、frozenset、None）：内容不能改。任何”修改”操作其实是造一个新对象并让名字改指过去。

b = b + 1 的底层：b + 1 求出一个新的 int 对象 2，然后 b = 让名字 b 改绑到这个新对象。原来那个 1 对象纹丝没动，a 还指着它。

a = 1; b = a                b = b + 1

 a ─┐                        a ────► [ int 1 ]   ← a 仍指旧对象
    ├──► [ int 1 ]           b ────► [ int 2 ]   ← b 改指新对象
 b ─┘                        (新分配的对象)

对比可变对象的原地修改：

a = [1,2,3]; b = a          b.append(4)

 a ─┐                        a ─┐
    ├──► [1,2,3]                ├──► [1,2,3,4]   ← 同一对象被改
 b ─┘                        b ─┘                  两个名字都看到变化

核心规律： - 对可变对象做原地修改（append/b[0]=/.sort()），影响所有别名。 - 对任何对象做重新绑定（b = ...），只改 b 这个名字的指向，不影响别人。 - 不可变对象没有”原地修改”这条路，所以它们的别名永远安全——这也是为什么”用不可变类型更省心”。

一个坑：+= 对可变和不可变行为不同。b += [4] 对 list 是原地扩展（等价 extend，影响别名）；b += 1 对 int 是重新绑定（不影响别名）。因为可变对象实现了原地版的 __iadd__，不可变的退回 b = b + 1。

眼见为实——同样是 +=，别名跟不跟随取决于对象可变与否：

a = [1, 2]
b = a
a += [3]        # list 的 += 是原地扩展(__iadd__),还是同一对象
print(b)        # [1, 2, 3]   别名 b 跟着变

a = a + [4]     # + 造一个新 list,再把 a 重新绑定过去
print(b)        # [1, 2, 3]   b 不跟随了——a 已指向别处

六、小整数缓存与字符串驻留

这两个是纯 CPython 实现细节，但你用 is 玩的时候一定会撞见，不讲清会一头雾水。

小整数缓存 [-5, 256]

CPython 启动时预先创建好 -5 到 256 的所有整数对象并缓存复用。所以这个范围内相同的值是同一个对象：

a = 100
b = 100
print(a is b)      # True！都指向缓存里那个唯一的 100 对象

x = 257
y = 257
print(x is y)      # True！——但这不是靠小整数缓存

# 在同一编译单元里，字面量 257 在字节码的 co_consts 常量表中只存一份，
# x、y 复用同一个常量对象，所以哪怕 257 超出 [-5, 256] 缓存范围，x is y 仍是 True。
# 要真正演示“超出缓存就各自新建”，得让对象在运行时构造、绕开 co_consts 复用：
m = int("257")
n = int("257")
print(m is n)      # False！两次 int() 各自在堆上新建了一个 257 对象
print(m == n)      # True，值当然相等

为什么缓存？小整数用得极频繁，预建好避免反复分配释放，省内存省时间。范围 [-5, 256] 是写死的实现选择，别的实现或版本可能不同。

教训：判断数值相等永远用 ==，绝不用 is。 is 碰巧对小整数”能用”，一出范围就骗你。

字符串驻留（interning）

CPython 会把某些字符串（尤其是看起来像标识符的字面量、编译期常量）驻留到一张全局表里，相同内容共享同一对象——这样比较标识符、字典查键时能先比指针（is）快速判等。

a = "hello"
b = "hello"
print(a is b)      # 通常 True，被驻留了

s = "hello world!"
t = "hello world!"
print(s is t)      # True——但同样是“同一编译单元里两个相同字面量共享一个 co_consts 常量”，
                   # 并不代表运行时把它 intern 了；含空格/标点的串默认不会被自动驻留

# 运行时构造、绕开字面量常量复用，才看得到“内容相同但不是同一对象”：
u = "hello world!".upper().lower()
v = "hello world!".upper().lower()
print(u is v)      # 通常 False

import sys
print(sys.intern(u) is sys.intern(v))   # True——手动 intern 后共享同一对象

驻留是优化，触发条件依版本和上下文而变，永远不要依赖它。同样：字符串判等用 ==。

七、命名空间即字典，但局部变量是 fast locals

你可能听过”Python 的命名空间就是字典（dict）“。基本对：模块的全局命名空间、对象的属性，底层确实是 dict（key 是名字字符串，value 是对象引用）。

x = 10
print(globals()["x"])    # 10，全局命名空间就是个 dict

但函数内的局部变量是个重要例外，这正是底层优化点。CPython 编译函数时，能静态数出局部变量有几个、各是谁，于是把它们放进一个定长数组（“fast locals”），按下标访问，而不是每次去 dict 里按字符串查。回顾 第01章 里 dis 看到的 LOAD_FAST：

import dis

g = 1
def f(a):
    b = a + g       # a、b 是局部（fast），g 是全局
    return b

dis.dis(f)

本机 3.14 实测输出：

  4           RESUME                   0

  5           LOAD_FAST_BORROW         0 (a)
              LOAD_GLOBAL              0 (g)
              BINARY_OP                0 (+)
              STORE_FAST               1 (b)

  6           LOAD_FAST_BORROW         1 (b)
              RETURN_VALUE

（LOAD_FAST_BORROW 是 3.14 对读取局部变量的优化变体——“借用”引用、免去 refcnt 增减，本质仍是按数组下标取局部；写入仍是 STORE_FAST，全局仍是 LOAD_GLOBAL。）

关键指令对比： - LOAD_FAST 0 (a) / STORE_FAST 1 (b)：局部变量走数组下标，O(1)，无哈希、无字符串比较。快。 - LOAD_GLOBAL 1 (g)：全局变量走 dict 查找（先本模块 globals 再 builtins），慢些。

函数 f 的栈帧（frame）
┌───────────────────────────┐
│ fast locals 数组:          │
│   [0] a ──► 对象           │  ← LOAD_FAST 0，按下标取，快
│   [1] b ──► 对象           │
├───────────────────────────┤
│ 求值栈 (value stack)       │  ← 字节码运算用（见第01章栈式虚拟机）
└───────────────────────────┘
        全局 g 在模块的 globals dict 里，LOAD_GLOBAL 查字典

这解释了一个常见的性能技巧：热循环里把频繁访问的全局/属性用局部变量”缓存”一下，就是把 LOAD_GLOBAL（查字典）换成 LOAD_FAST（查数组）。也解释了函数内为什么读写全局要 global 声明——编译器靠”函数里有没有对该名字赋值”来决定把它当局部还是全局。作用域与栈帧的完整故事见 第06章。

注意区分：局部变量用 fast locals 数组是 CPython 实现；“函数有独立作用域”是语言语义。

八、浅拷贝 vs 深拷贝

既然 b = a 只拷指针，那想真拷一份怎么办？用 copy 模块。但要分清浅和深——这跟 C++ 里”拷贝构造是浅还是深、要不要写自己的深拷贝”是同一个问题。

import copy

original = [[1, 2], [3, 4]]

alias    = original                 # 别名，同一对象
shallow  = copy.copy(original)      # 浅拷贝：新外层，内层元素仍共享
deep     = copy.deepcopy(original)  # 深拷贝：递归复制所有层

original[0].append(99)              # 修改内层第一个子列表

print(alias)    # [[1, 2, 99], [3, 4]]   跟着变（就是同一对象）
print(shallow)  # [[1, 2, 99], [3, 4]]   跟着变！内层是共享的
print(deep)     # [[1, 2], [3, 4]]       独立，没变

内存图：

original ─► [ ●, ● ]            外层 list
              │  │
              ▼  ▼
           [1,2] [3,4]          内层两个 list

shallow = copy.copy(original):
shallow ─► [ ●, ● ]  ← 新的外层对象
             │  │
             └──┴──► 指向同样的 [1,2] 和 [3,4]  （内层没复制！）

deep = copy.deepcopy(original):
deep ─► [ ●, ● ]  ← 新外层
          │  │
          ▼  ▼
       [1,2]'[3,4]'  ← 连内层也是新对象，完全独立
  • 浅拷贝：只新建最外层容器，里面装的还是原来那些对象的引用。改内层可变元素，浅拷贝也受影响。
  • 深拷贝：递归地把每一层都复制成新对象，彻底独立（还能正确处理循环引用）。

其他造浅拷贝的常见写法：list(x)、x[:]、dict(x)、x.copy()。它们和 copy.copy 一样都是浅的。要完全独立必须 deepcopy。深拷贝有成本（递归 + 分配），别无脑用。

九、可变默认参数陷阱（预告）

有了本章的模型，你现在能理解一个 Python 最著名的坑长什么样：

def add_item(item, bucket=[]):    # 危险！默认值 [] 只创建一次
    bucket.append(item)
    return bucket

print(add_item(1))   # [1]
print(add_item(2))   # [1, 2]  ← 你可能期望 [2]，但它记住了上次！

原因就是本章讲的：默认值 [] 是在 def 执行时创建一次的一个可变对象，被函数对象一直持有；每次不传 bucket 都复用同一个 list，原地 append 自然累积。这是”可变对象 + 别名 + 只创建一次”三者合流的必然结果。

正确写法和完整机制（默认值存在哪、def 何时求值）留到 第06章 展开——但你现在已经看懂它为什么会这样了，这正是本章模型的威力。

十、小结

  • 用法层面：赋值 x = 值 不用声明类型；y = x 语法上就是一次普通赋值。
  • 底层层面（把这几条刻进肌肉）：
    1. 变量是名字到堆对象的引用绑定，不是内存盒子；类比只读、不能算术的指针。
    2. a = b 只拷贝引用（指针值），从不拷贝对象或数据；结果是别名。用 is/id() 可证明是同一对象，== 只比值。
    3. 引用计数（ob_refcnt）管回收，sys.getrefcount 可观测（结果多 1）；del 解绑名字而非删对象。
    4. 可变 vs 不可变决定别名行为：原地修改可变对象影响所有别名；重新绑定只动当前名字；不可变对象的”修改”都是造新对象。
    5. 小整数 [-5,256] 缓存、字符串驻留是 CPython 优化——判等一律用 ==，别用 is。
    6. 命名空间是 dict，但函数局部变量用 fast locals 数组（LOAD_FAST），是重要性能点。
    7. copy.copy 浅、copy.deepcopy 深；b=a/切片/list() 都只是浅层。
    8. 可变默认参数陷阱是本章模型的直接推论，细节见 第06章。
