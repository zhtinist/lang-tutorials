基础语法:从大括号世界过来的适应指南

第01章已经建立了”每个值都是堆对象”的运行模型。这一章处理表层语法——没有大括号、没有分号、没有类型声明、没有 main——以及每处差异背后的设计考虑。

先看一段代码建立整体印象:

name = "Ann"
age = 20

print("Hello,", name)          # Hello, Ann

if age >= 18:
    print("成年人")
else:
    print("未成年")

for i in range(3):
    print(i)

def add(a, b):
    return a + b

print(add(2, 3))

name = "Ann" 是变量赋值,不用像 C 那样先声明类型;if/else 用冒号加缩进代替 {};for i in range(3) 循环 3 次;def add(a, b): return a + b 定义函数。下面逐条对照 C/C++ 拆解这些写法背后的机制与取舍。

一、缩进即块:语法层强制的作用域

C/C++ 用 {} 界定块,缩进纯属排版习惯,编译器不检查:

if (x > 0) {
    foo();
    bar();
}

Python 用缩进本身界定块,没有大括号,语句以换行结束(无分号):

if x > 0:
    foo()
    bar()

冒号 : 引导一个块,块体必须缩进(约定 4 空格);同一块内所有语句缩进必须严格一致,否则 IndentationError;不能混用 Tab 和空格,CPython 会直接报错。

缩进在词法分析阶段被转成 INDENT/DEDENT token(回顾第01章的 tokenize 阶段),扮演的正是 C 里 {/} 的角色——不是风格建议,是语法记号,漏掉正确的缩进和 C 里少写一个大括号是同一类错误。

为什么用缩进强制块结构、而不是像 C 那样把缩进和语法结构解耦:这是 Python 设计哲学”可读性即语法”的直接体现——C 允许缩进和实际的块结构不一致(比如误以为某行属于 if 块,实际上因为漏了 {} 它并不属于),这类”看起来在块里、实际不在”的错位是真实存在的一类 bug 来源(历史上苹果著名的 goto fail SSL 漏洞正是因为多出的一行代码在视觉缩进上像在 if 块内、实际因为没有大括号而在块外)。Python 把缩进本身定义为语法的一部分,消除了”视觉结构”和”实际结构”不一致的可能性,代价是牺牲了一部分排版自由(不能把整个 if 语句写在一行里,除非用特殊的单行写法)。

一个重要的语义差异:C 的 {} 引入新的词法作用域,块内声明的变量出块即失效。Python 的缩进块不引入新作用域——if/for/while 块里绑定的名字,出块后依然可见:

if True:
    y = 10
print(y)   # 10,合法,y 没有"出块销毁"

Python 里真正引入新作用域的只有函数、类、模块、推导式,不是每个缩进块。这不是疏漏,而是”缩进只表达执行流程的分组,不表达命名空间边界”这一设计选择——命名空间的粒度定在函数而非任意代码块,原因见「函数与调用栈」。

二、动态类型、强类型、无需声明

三个正交的概念:

  • 无需声明:变量不预先声明类型,第一次赋值即”诞生”。
  • 动态类型:类型属于对象,不属于变量。同一个名字可以先指 int 再指 str,类型检查发生在运行期(回顾第01章:ob_type 挂在对象上而非变量上)。
  • 强类型:不做隐式的乱来转换。1 + "2" 直接抛 TypeError。
x = 42          # x 指向一个 int 对象
x = "hello"     # x 指向一个 str 对象;完全合法
x = [1, 2, 3]   # 又指向 list 对象

# 1 + "2"       # TypeError: unsupported operand type(s)
1 + int("2")    # 3,显式转换才行

对照 C++:C++ 是静态 + 强类型,int x 一旦定下,x = "hi" 编译不过。Python 把类型检查推迟到运行时,换来”同一段代码可以处理任意类型,只要它支持所需的操作”这种灵活性(鸭子类型的基础),代价是很多类型错误编译期发现不了,只能靠运行时触发或用类型注解 + mypy 做静态检查弥补(见「面向对象与类型系统」)。强类型而非弱类型这一点同样是刻意选择:C 的隐式转换规则(如 int 到 float 自动提升)简化了书写,但也制造了大量”看起来能算、结果却出乎意料”的场景;Python 选择”宁可报错,也不做没有明确规则的自动转换”,与前面提到的”能确定性地报错就不留不确定行为”是同一条设计脉络。

变量重新赋值的底层实质:x = "hello" 不是把 int 对象的内存改写成字符串,而是让名字 x 改指向一个新的堆对象,旧对象引用计数减一、可能被回收。这是全书重点,第03章会用内存图讲透。

三、没有 main:模块从上到下执行

C/C++ 程序入口是 main。Python 没有强制入口——模块被加载时,顶层语句从上到下依次执行,def 本身也是一条”执行时创建函数对象”的语句,不是编译期声明。

def main():
    ...

if __name__ == "__main__":   # 直接运行时 __name__ == "__main__"
    main()

被 import 时,该模块的 __name__ 是模块名而非 "__main__",条件为假,main() 不执行——导入它只拿到函数定义,不触发副作用。

为什么用这种运行期判断而不是像 C 那样有专门的入口语法:Python 的定位从一开始就是”每个文件既可以直接当脚本运行,也可以被当作库 import”,而不是强制”一个程序恰好一个入口”。这个设计要求语言提供一种方式,让同一份代码在两种使用场景下有不同行为——if __name__ == "__main__" 正是利用”模块级代码本来就会执行”这个既有机制实现的,不需要为”脚本”和”库”发明两套语法,用一个普通的 if 就解决了,这是”少设计新概念、复用已有机制”这一 Python 一贯风格的例子。

四、print / input 与注释

print("hello", "world")          # 默认空格分隔、末尾换行 -> hello world
print("no newline", end="")      # 自定义结尾
name = input("name? ")           # 读一行,去掉换行;返回值永远是 str

print(Python 3 起)和 input 都是普通函数,不是语言关键字或语句——这本身是一个设计取舍:Python 2 里 print 是语句(print "hello",没有括号),3.0 把它改成函数,换来的是 print 可以像任何函数一样被赋值、传递、重写,不需要为它单独开一条语法规则。input 返回值永远是 str,要数字须显式 int(input())——没有 scanf("%d") 那种按格式符解析输入的机制,这与前面”不做隐式转换”的强类型立场一致:转换必须显式、可见。

五、运算符:与 C 分歧最大的几处

大部分运算符(+ - * / % == < > && ||)语义相通,但 Python 用关键字 and/or/not 代替 && || !(& | ^ ~ << >> 仍是按位运算)。

运算符 含义 C/C++ 对照
/ 真除法,总返回 float 5/2 == 2.5(C 里整数间是 2)
// 向下取整除法(floor) C 整数除法是向零截断
** 幂 无对应,C 要用 pow()
and/or 逻辑与或(短路) &&/\|\| 
5 / 2     # 2.5
5 // 2    # 2
-5 // 2   # -3   向下取整,向负无穷
5 % 2     # 1
-5 % 2    # 1    符号跟除数,C 里 -5%2 == -1
2 ** 10   # 1024

为什么 / 恒返回 float、需要单独的 // 做整除:Python 2 的 / 曾经和 C 一样,两个整数相除做截断整除,这被认为是一个显著的设计缺陷——当一段代码里把某个变量从 int 改成 float(或反过来),/ 的行为会跟着悄悄改变,而调用点的写法完全不变,这种”类型一变、语义跟着变”的隐式行为是错误的温床。Python 3 的 PEP 238 把 / 固定为真除法(总返回 float 或按值类型),需要整除时必须显式写 //——用两个不同的记号表达两种不同的意图,让”我到底要不要截断”这件事在代码里可见,而不是取决于操作数的类型。

为什么取模符号跟随除数、不像 C 那样跟随被除数:这和 // 的定义方式直接相关。Python 的 // 是向下取整(floor,向负无穷方向),C 的整数除法是向零截断(truncate)。选择哪种取整方式,决定了余数的符号约定,因为语言保证恒等式 a == (a // b) * b + (a % b) 成立。Python 选择向下取整(与 Knuth《计算机程序设计艺术》里的偏好一致),换来的好处是模运算在”环绕”语义下更符合直觉——比如用 (i - 1) % n 计算数组下标的前一个位置,即使 i 是 0,结果也会正确地环绕到 n - 1;C 的截断除法在 i 为负时会得到负余数,环绕计算需要额外的条件判断修正。

整数还有任意精度这个特性:2 ** 200 直接算出准确值,不会像 C 的 int32/int64 那样溢出回绕(第01章提过 PyLongObject 是堆上变长对象)。为什么要做任意精度而不是像 C 那样限定固定位宽:固定位宽是为了让整数直接映射到 CPU 寄存器、用一条机器指令完成运算,这是 C 追求的贴近硬件、可预测的存储代价;Python 选择永不溢出,把”这个数到底需要多少位来表示”这个问题从程序员手上拿走,代价是每次整数运算都要走一遍变长数字位的通用算法,而不是一条 CPU 指令,速度慢了一个数量级,换来的是”数值计算永远不会因为选错类型宽度而静默出错”。

x = 2 ** 200
print(x)              # 1606938044258990275541962092341162602522202993782792835301376
print(x.bit_length()) # 201

六、f-string:编译期展开的格式化

name, n = "Ann", 3
print(f"{name} has {n} apples")        # Ann has 3 apples
print(f"pi = {3.14159:.2f}")           # pi = 3.14
print(f"{n=}")                         # n=3,调试用,自动带变量名

f-string 在编译期就被拆成”取值 + 格式化 + 拼接”的字节码,不是运行时解析格式字符串:

import dis
dis.dis('f"{name} has {n}"')
  0           RESUME                   0

  1           LOAD_NAME                0 (name)
              FORMAT_SIMPLE
              LOAD_CONST               0 (' has ')
              LOAD_NAME                1 (n)
              FORMAT_SIMPLE
              BUILD_STRING             3
              RETURN_VALUE

每个 {} 对应一条 FORMAT_SIMPLE,最后 BUILD_STRING 把各段拼成一个字符串。为什么要在编译期而非运行期解析格式串:老式的 % 格式化和 str.format() 都需要在每次调用时重新解析格式字符串(找到 %s/{} 占位符、匹配参数),这是可避免的重复工作。f-string 把这一步挪到编译期一次性完成,运行时只是按已经确定好的顺序取值拼接,省掉了每次调用都要重新解析的开销,这是它现在被作为首选格式化方式推荐的直接原因,而不只是写法上更紧凑。

七、布尔、None 与真值测试

flag = True          # 首字母大写
nothing = None        # 空值单例

True/False 是 bool 类型,在 CPython 中是 int 的子类(True == 1);None 是全局唯一单例对象,判等要用 is(if x is None,不是 == None)——这与前面”变量是引用”的模型一致:is 比较的是”是不是同一个对象”,None 只有一份,身份比较既准确又比调用 __eq__ 快。

真值测试:任何对象都能放进 if。C 里”非零即真”,Python 更丰富:

假值(falsy) 真值(truthy)
False、None 其余一切
0、0.0 非零数字
""、[]、{}、() 非空容器 
items = []
if items:                # 空列表为假,不用写 len(items) == 0
    print("有货")
else:
    print("空的")        # 走这里

if x 会调用对象的 __bool__(没有则退回 __len__,判断是否为 0)。为什么把”空容器”也定义成假值:这是在语言层面把”这个容器有没有内容”和”这个数值是不是零”统一成同一套”有没有意义的内容”的语义,让 if items: 这种写法能直接表达”如果有东西”的意图,不需要为每种容器类型单独查长度——本质上是把 C 里”非零即真”这条规则,从”数值是否非零”泛化到”对象是否非空/有意义”,覆盖面更广但保持了同一条直觉。完整的真值测试顺序和自定义示例见「控制流」。

八、链式比较与多重赋值

x = 5
if 0 < x < 10:           # 读作 (0 < x) and (x < 10)
    print("范围内")

C 里 0 < x < 10 会被解析成 (0 < x) < 10,即 (结果为0或1) < 10,几乎总是真——这是 C 运算符结合律带来的一个真实陷阱。Python 把链式比较定义成真正的数学意义上的链式(等价于用 and 连接每一对相邻比较,且每个子表达式只求值一次),消除了这个陷阱。

a, b = 1, 2               # 同时绑定
a, b = b, a                # 交换,无需临时变量
first, *rest = [1, 2, 3, 4]   # first=1, rest=[2,3,4]

a, b = b, a 的底层:右侧两个值先各自求值压栈,再分别绑定给左侧,不需要临时变量,也没有 C 里”先赋值给 a 会不会影响后面读 b”这种求值顺序疑虑。CPython 对这种小规模交换做了专门优化:3.11–3.12 用一条 SWAP 指令;3.13 起进一步用超级指令 LOAD_FAST_LOAD_FAST + STORE_FAST_STORE_FAST,连 SWAP 都省了,只有更早版本或元素更多时才退回到通用的 BUILD_TUPLE + UNPACK_SEQUENCE。本机 3.14 实测:

import dis
def swap(a, b):
    a, b = b, a
dis.dis(swap)
  3           RESUME                   0

  4           LOAD_FAST_LOAD_FAST     16 (b, a)
              STORE_FAST_STORE_FAST   16 (b, a)
              LOAD_CONST               0 (None)
              RETURN_VALUE

九、C/C++ ↔︎ Python 语法差异速查

主题 C/C++ Python
块界定 { } 缩进(INDENT/DEDENT)
语句结束 ; 换行
变量声明 int x; 必须 直接 x = ...
类型 静态、绑在变量 动态、绑在对象
入口 main() 顶层语句;if __name__=="__main__"
逻辑运算 && \|\| ! and or not
除法 / 整数间截断取整 总是真除法,整除用 //
取模符号 随被除数 随除数
整数溢出 有(回绕/UB) 无,任意精度
三目 c ? a : b a if c else b
++/-- 有 没有,用 x += 1
空值 NULL/nullptr None(对象,用 is 判)

为什么没有 ++/--:C 的自增自减运算符可以嵌在表达式内部产生副作用(a[i++] = i),而”前置/后置的值不同”“同一表达式里多次自增顺序未定义”是 C 里长期被诟病的陷阱来源。Python 的设计原则是”一条语句只做一件清楚的事”,干脆不提供这个可以嵌入表达式、自带副作用的运算符,自增写成明确的语句 x += 1,牺牲了一点书写简洁,换来了不再有”这个自增到底在表达式的哪个时间点生效”的疑问。

小结

  • 缩进是语法记号(编译期转成 INDENT/DEDENT),代替大括号,把”视觉结构”和”实际结构”强制统一,消除了 C 里两者可能不一致的一类 bug;但缩进块本身不引入新作用域,只有函数/类/模块/推导式才引入。
  • 动态强类型:类型绑在对象上、运行期检查、不做隐式转换——用运行时灵活性换编译期安全性,配合类型注解可部分找回静态检查能力。
  • / 恒为真除法、// 单独表示整除、取模符号随除数,都是”语义在类型变化时不该悄悄改变”和”floor 除法让模运算在负数下更符合直觉”这两条原则的产物;整数任意精度是”永不因类型宽度选错而溢出”换来的通用性,代价是运算不再是单条机器指令。
  • f-string 在编译期而非运行期解析格式串,省去重复解析的开销;真值测试把”空容器”泛化进”非零即真”的直觉;链式比较和 a, b = b, a 都消除了 C 里因运算符结合律或求值顺序产生的具体陷阱。
