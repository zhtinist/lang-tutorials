复合数据结构的内部实现

本章讲 list、tuple、dict、set、deque 这几种最常用容器在 CPython 里的内部结构,和 C 的数组、结构体、哈希表逐一对照,重点是每处实现选择解决了什么问题、放弃了什么。「变量与内存模型」确立的前提贯穿全文:Python 容器里存的从来不是”值”本身,而是指向 PyObject 的指针——这一点决定了下面每种容器的内存布局、拷贝语义与复杂度。

一、指针数组 vs 值数组

C 的数组 int a[4] = {1,2,3,4} 在内存里是这样的:

a:  [ 1 ][ 2 ][ 3 ][ 4 ]        // 16 字节,值连续存放,cache 友好
     ^0   ^4   ^8   ^12

Python 的 a = [1, 2, 3, 4] 是这样的:

list 对象(堆)                    每个元素是一个独立的堆对象
+-----------------+
| ob_refcnt       |
| ob_type = list  |
| ob_size = 4     |----> ob_item(另一块堆内存,指针数组)
| allocated = 4   |       +------+------+------+------+
| ob_item ------- |------>| ptr0 | ptr1 | ptr2 | ptr3 |
+-----------------+       +--|---+--|---+--|---+--|---+
                            v      v      v      v
                          PyLong PyLong PyLong PyLong
                           (1)    (2)    (3)    (4)   ← 各自在堆上,还可能被别处共享

Python 的 list 是”指针的数组”,不是”值的数组”。这不是随意的实现,而是”一切皆对象、变量即引用”这一运行模型(见「运行模型」)在容器层面的必然延伸——数组的每个槽位必须是统一大小,才能用 ob_item[i] 这种固定步长直接算地址;int、str、自定义实例的大小完全不同,数组唯一能统一存放的东西就是”指向它们的指针”。这个结论贯穿本章:每次访问元素都多一次指针解引用,拷贝容器默认只拷指针不拷内容,同一个数组里可以混装任意不同类型的元素(因为槽位里存的只是统一大小的指针,类型差异被推到了被指向的对象那一侧)。

二、list:为什么是动态数组而不是链表

fruits = ["apple", "banana", "cherry"]
fruits.append("date")           # 末尾追加
fruits.insert(1, "kiwi")        # 在下标 1 处插入
fruits.remove("banana")         # 按值删除(删第一个匹配项)
last = fruits.pop()             # 弹出并返回末尾元素
print(fruits[0], fruits[-1])    # apple kiwi   —— 正数/负数下标
print(fruits[1:3])              # ['kiwi', 'cherry']  —— 切片
print("apple" in fruits, len(fruits))   # True 3

nums = [3, 1, 2]
nums.extend([9, 8])      # 把另一个可迭代对象的元素逐个追加
nums.sort()               # 原地排序
print(nums)               # [1, 2, 3, 8, 9]
print(nums.index(8), nums.count(2))     # 3 1
nums.reverse()
print(nums)               # [9, 8, 3, 2, 1]

CPython 中 list 的结构体(简化自 Include/cpython/listobject.h):

typedef struct {
    PyObject_VAR_HEAD          // 含 ob_refcnt、ob_type、ob_size(=len)
    PyObject **ob_item;        // 指向一块单独分配的指针数组
    Py_ssize_t allocated;      // 已分配的容量(容量 >= ob_size)
} PyListObject;

list 对象本身是固定大小的头部,真正的元素指针存在 ob_item 指向的另一块堆内存里——这是两次分配(list 头 + 指针数组),类似 C++ 里 std::vector 的 vector 对象和它 new[] 出来的缓冲区分离。ob_size 是当前长度,allocated 是缓冲区能容纳的指针个数,恒有 allocated >= ob_size,中间的空槽是为了 append 不用每次都重新分配。

为什么用连续的指针数组而不是链表(每个节点存一个元素和 next 指针,理论上中间插入删除是 O(1)):这是对”哪种操作更常见”的一个判断,而不是任意选择。实际代码里按下标随机访问(lst[i])、遍历、尾部追加是压倒性高频的操作,在中间任意位置插入删除相对少见。数组把随机访问做到 O(1)——ob_item + i 直接算地址;链表的随机访问是 O(n)——必须从头沿指针一步步走到第 i 个节点。如果选链表,牺牲的是几乎所有代码都在用的操作,换来的只是一个用得少得多的操作变快,这笔账在通用容器的默认实现里明显不划算。链表还有一个经常被忽视的代价:每个节点是一次独立的堆分配,元素在内存里彼此不相邻,遍历时 cache miss 远比连续数组频繁,而现实中”遍历”几乎是任何容器都逃不掉的操作。真正需要”两端 O(1)“这个特性时,标准库把它做成了一个专门的类型(collections.deque,见后文),而不是让 list 本身承担这个权衡。

对照 std::vector<T>:vector 直接把 T 的值连续存放(T=int 就是连续的 int)。Python list 永远存指针,哪怕全是小整数,也要经过一次指针解引用才能拿到值——这就是为什么数值计算要用 numpy(见「常用第三方库」):numpy 的 ndarray 才是真正连续存值的 C 数组。

append 时若 ob_size == allocated,需要扩容。CPython 不是简单翻倍,而是用一个温和的增长公式(list_resize,简化):

/* 现行 CPython list_resize:多分配约 1/8,再向上对齐到 4 的倍数 */
new_allocated = (new_size + (new_size >> 3) + 6) & ~3;

实测(在 3.9.6 上用 sys.getsizeof 反推,64 位平台 list 头占 56 字节、每指针 8 字节)allocated 的增长序列是:

0, 4, 8, 16, 24, 32, 40, 52, 64, 76, 92, 108, 128, ...

对比 C++ std::vector 常见的 1.5x 或 2x 增长,CPython 只多分配约 1/8。这是”内存浪费”和”重分配次数”之间的另一层取舍:翻倍增长重分配次数更少,但每次浪费的空间可能高达已用空间的一倍;1/8 增长把浪费压得更低,代价是同样数量的元素需要更多次 realloc——CPython 押注 list 的典型追加量不会特别夸张,愿意多付出几次 realloc 换取更省内存。扩容时调用 realloc 换一块更大内存,把旧的指针数组按位拷贝过去(只拷指针,不拷元素对象,也不动引用计数),元素对象原地不动,别处对它们的引用依然有效:

append 触发扩容:
旧:  ob_item -> [p0][p1][p2][p3]                (allocated=4, size=4)
     realloc 到新缓冲区(allocated=8):
新:  ob_item -> [p0][p1][p2][p3][p4][ ][ ][ ]    (size=5)
     元素对象 (对应 p0..p4) 全程没动

各操作复杂度与原因:

操作 复杂度 底层原因
lst[i] 索引读写 O(1) 指针数组随机访问,ob_item[i]
lst.append(x) 摊还 O(1) 有空槽直接写;满了才 realloc,均摊到每次是常数
lst.pop()(尾部) O(1) ob_size--,DECREF 尾元素
lst.insert(0, x) O(n) 后面所有指针要 memmove 右移一格
lst.pop(0) O(n) 后面所有指针左移;要频繁头部操作请用 deque
del lst[i] / 中间删除 O(n) memmove 补洞
x in lst O(n) 线性扫描逐个 == 比较
len(lst) O(1) 直接读 ob_size
切片 lst[a:b] O(k) 新建 list,拷贝 k 个指针(浅拷贝!)并各 INCREF

这张表就是上面那个设计取舍的量化版本:随机访问和尾部操作是 O(1),中间插入删除是 O(n)——list 把复杂度预算优先分配给了前者。摊还 O(1) 的证明和 std::vector::push_back 完全一样:n 次 append 的总拷贝成本是所有扩容拷贝之和,因增长是几何级数,总和为 O(n),平摊到每次即 O(1)。

lst2 = lst[:]、list(lst)、copy.copy(lst) 都是浅拷贝:新建一个 list 头和新的指针数组,但指针指向的是同一批元素对象。

a = [[1, 2], [3, 4]]
b = a[:]              # 浅拷贝
b[0].append(99)       # 修改的是共享的内层 list
print(a)              # [[1, 2, 99], [3, 4]]  —— a 也变了!
a.ob_item -> [p0][p1]
                \    \
                 v    v
             [1,2,99] [3,4]      ← 内层对象只有一份
                 ^    ^
b.ob_item -> [p0][p1]            ← b 是新数组,但指针指向同一内层对象

要彻底隔离得用 copy.deepcopy,它递归重建每一层对象。这与 「变量与内存模型」里”赋值只拷引用、浅拷贝只复制一层”是同一条设计原则在容器上的体现:廉价操作作为默认,昂贵操作必须显式请求。

三、tuple:不可变的定长记录

point = (3, 4)
x, y = point                    # 解包
print(point[0])                 # 3
# point[0] = 9                  # TypeError:tuple 不能修改

t = (1, [2, 3])
t[1].append(4)     # OK,改的是内层 list
print(t)            # (1, [2, 3, 4])
# t[1] = [9]        # TypeError:不能改 tuple 的槽位

tuple 存在的意义不只是”不可变的 list”。list 表达的是同构、变长的集合,下标只是位置,元素类型通常一致;tuple 表达的是异构、定长的记录(record),下标对应固定含义的字段——(x, y) 里下标 0 永远是 x、下标 1 永远是 y,和 C 的 struct { int x; int y; } 是同一个语义,只是按位置而非按名字访问。这是两种不同的设计意图,不可变性是用来强化”这是个定长记录,不该在这插入删除元素”这一语义的手段,而不是它存在的唯一理由。想要带名字的字段而不是纯位置,见 namedtuple。

为什么 tuple 不可变这件事还能让它当 dict 的 key、放进 set:「变量与内存模型」已经论证过,不可变性带来的好处之一正是”可以安全用作 dict 键”。哈希表的正确性依赖一个前提:key 的哈希值在它存活期间必须保持不变,因为哈希值决定了 key 在表内的物理位置,以后查找时还要用同一个哈希值把它找回来。如果 tuple 可变,某个元素被改了之后哈希值会跟着变,而这个 key 早已经按旧哈希值存进了表的某个槽位——新旧哈希值对应的槽位对不上,这个 key 实际上就”丢”在表里,再也查不到。list 允许原地修改,所以语言干脆不给它定义 __hash__,禁止作 key,而不是留一个”能用但一改内容就出诡异 bug”的陷阱。tuple 只有在所有元素都可哈希时才可哈希(哈希值由元素哈希组合而成),这也是为什么 (1, [2, 3]) 这种含 list 的 tuple 不能作 dict 键。

tuple 的结构体(PyTupleObject)比 list 更简单——它把指针数组内联在对象尾部,而不是单独分配:

typedef struct {
    PyObject_VAR_HEAD
    PyObject *ob_item[1];   // 变长尾部数组,创建时一次分配好,永不改变
} PyTupleObject;

没有 allocated 字段,长度创建后固定,一次分配(对象头和元素指针数组在同一块内存里),比 list 少一次分配、少一层间接,内存更紧凑——类似 C 里的柔性数组成员(flexible array member)。这个更紧凑的布局正是”长度固定、永不改变”这个约定换来的:list 必须留出增长空间、必须能整体 realloc,tuple 放弃了这些能力,换回了创建成本更低、访问时少一次指针跳转的实现。CPython 对小 tuple 还有空闲链表(free list)缓存,频繁创建销毁小 tuple 时能复用内存块,减少 malloc 压力。

四、dict:紧凑数组 + 索引表如何在保序的同时保持哈希表本质

dict 是 Python 最核心的数据结构——它不仅是用户的哈希表,还是对象属性(__dict__)、模块命名空间、关键字参数的底层载体,因此它的实现被反复优化。

d = {"a": 1, "b": 2}
d.update({"b": 20, "c": 3})   # 批量更新/新增
print(d.pop("a"))             # 1 —— 删除并返回该键的值
print(d.setdefault("z", 0))   # 键不存在则设为默认值并返回;存在则返回原值
squares = {x: x * x for x in range(5)}   # 字典推导式
print(squares)                            # {0: 0, 1: 1, 2: 4, 3: 9, 4: 16}

Python dict 用开放寻址(open addressing),不是链地址法(chaining):冲突时不在桶里挂链表,而是按一个探测序列去找下一个空槽,所有条目都住在同一块连续数组里,cache 更友好。key 必须可哈希——实现了 __hash__ 且哈希值在对象生命周期内不变;不可变内置类型(int/str/tuple/frozenset)可哈希,list/dict/set 不可哈希,原因与上一节 tuple 部分完全一致:可变对象的哈希值会随内容漂移,破坏哈希表的正确性。平均查找/插入/删除 O(1),最坏(大量冲突)O(n),现实中极少见。

Python 3.6 起 dict 记忆插入顺序(3.7 起是语言保证)。为什么这个保证没有靠额外挂一条链表来实现——比如 Java 的 LinkedHashMap 就是在普通哈希表的每个 entry 上再穿一条双向链表记录插入顺序,这个做法需要给每个 entry 额外存两个指针,遍历时还要沿链表跳,cache 不友好。CPython 3.6 的紧凑字典换了一个思路:把”哈希定位”和”数据存储”这两件事从同一个数组里拆开,变成两个数组:

indices(哈希表,存的是到 entries 的下标,可能有空槽 -1):
+----+----+----+----+----+----+----+----+
| -1 |  1 | -1 | -1 |  0 | -1 |  2 | -1 |
+----+----+----+----+----+----+----+----+

entries(紧凑数组,按插入顺序追加,无空洞):
+---------+--------+---------+
|hash key val|hash key val|hash key val|
|   'a':1   |  'b':2  |  'c':3  |
+---------+--------+---------+
   idx 0      idx 1     idx 2

稀疏的 indices 数组只负责”给定 key,找到它在 entries 里的下标”这一件事,可以很稀疏(装载因子低),但每个槽只占一个小整数(视表大小 1/2/4/8 字节),浪费的空间小;真正的 (hash, key, value) 三元组集中放进紧凑的 entries 数组,按插入顺序连续追加、无空洞。遍历字典就是顺序扫这个 entries 数组——保序完全是这个物理排列的自然副产品,不需要为它专门维护链表指针,还顺带让遍历本身更快(连续内存扫描 vs 跳链表)。相比 3.5 之前”每个槽都存完整 entry”的稀疏大表,这个拆分还额外省了约 20%–25% 内存,是一个”顺便”拿到的收益,不是设计的主要目的。

实测:dict 保序,且顺序由”插入”而非”排序”决定。删除中间的键再重新插入,会追加到末尾;更新已有键则原地改值、顺序不动:

d = {}
d["c"] = 1; d["a"] = 2; d["b"] = 3
print(list(d))     # ['c', 'a', 'b']   按插入序,不是排序
del d["a"]         # 删掉中间的键
d["a"] = 9         # 重新插入 —— 追加到 entries 末尾
print(list(d))     # ['c', 'b', 'a']
d["c"] = 100       # 更新已存在的键:原地改 value,顺序不动
print(list(d))     # ['c', 'b', 'a']

查找过程:对 key 求哈希 → 取低位作 indices 下标 → 拿到 entries 下标 → 去 entries 比对 hash 和 key(先比 hash 整数,相等再比 ==,因为哈希可能碰撞)。CPython 用一种扰动探测(perturbation probing)找下一个候选槽,而不是简单的线性探测:

perturb = hash;
i = hash & mask;
while (slot_used_and_key_differs) {
    perturb >>= 5;
    i = (i * 5 + perturb + 1) & mask;
}

简单线性探测(i = i + 1)在装载因子升高时容易产生”聚集”(cluster)——一旦某个区域被占满,后续冲突的 key 都得排队探测过这一长串占用的槽,查找退化明显。扰动探测不断把哈希值的高位”搅”进下一个候选位置的计算里,让探测序列更分散,减少聚集,这是拿一点计算开销换取更均匀的探测分布。

装载因子超过 2/3 时扩容。新表容量取”能容纳当前已用条目 3 倍”的最小的 2 的幂(增长约 2x–4x)。扩容时重建 indices 表(重新散布所有 key,因为表容量变了、每个 key 的哈希对新掩码取模的结果也变了),但 entries 保持插入顺序被搬过去,扩容期间还会顺便清理删除留下的”墓碑”。

操作 平均 最坏 说明
d[k] 查 O(1) O(n) 哈希 + 探测
d[k] = v 插入/改 O(1) 摊还 O(n) 可能触发扩容重散布
del d[k] O(1) O(n) 标记删除,entries 位置留空(遍历会跳过)
k in d O(1) O(n) 同查找
遍历 O(n) O(n) 顺序扫 entries
len(d) O(1) O(1) 读计数

与 C++ 对照:std::unordered_map 是链地址法(每桶挂链表/节点,节点各自 new,cache 不友好);std::map 是红黑树 O(log n) 且按 key 排序。Python dict 是一个开放寻址哈希表,并且用”紧凑 entries + 稀疏 indices”这个拆分额外换到了插入序保证,这在纯粹的开放寻址或链地址实现里都不是自动获得的。

五、set 与 frozenset

a = {1, 2, 3, 4}
b = {3, 4, 5, 6}
print(a & b)   # {3, 4}              交集
print(a | b)   # {1, 2, 3, 4, 5, 6}  并集
print(a - b)   # {1, 2}              差集(在 a 不在 b)
print(a ^ b)   # {1, 2, 5, 6}        对称差(恰在其一)

fs = frozenset([1, 2, 3])    # 不可变集合
s = {fs, frozenset([4])}     # frozenset 可哈希,能放进另一个 set
print(len(s))                 # 2

sc = {x % 3 for x in range(10)}   # 集合推导式
print(sc)                          # {0, 1, 2}

一个常见用法——用 set 给”是否见过”做去重判断,平均 O(1) 远快于 list 的 O(n):

seen = set()
data = [1, 2, 2, 3, 1]
for x in data:
    if x not in seen:
        seen.add(x)
print(seen)   # {1, 2, 3}

set 的实现和 dict 高度相似——本质是”只有 key 没有 value 的哈希表”,同样开放寻址、同样要求元素可哈希。为什么 set 没有沿用 dict 那套”indices + entries”的拆分:那套拆分的全部意义在于用紧凑数组的物理顺序换取插入序保证,而 set 从语义上就不承诺顺序——它建模的是数学意义上的集合,元素之间没有先后关系,没有必要为一个不需要的保证多付内存和一层间接。所以 set 用的是更接近传统的稀疏表(entry 直接是 hash + key),不引入 dict 那套双数组结构。别依赖 set 的迭代顺序,它没有任何保证。

为什么需要 frozenset:set 本身可变,因此不可哈希,不能放进另一个 set 或作 dict 键——这与 tuple/list 那条规则完全一致。但”需要一个不可变、可哈希的集合”是真实存在的需求,比如把一组标签整体当作缓存的 key,或者对”集合的集合”去重。frozenset 就是为这个需求专门开的口子:创建后不能再 add/remove,因此可以安全地计算并缓存哈希值,能够作为 dict 的 key 或放进另一个 set(如上面的例子)。

操作 平均复杂度
add / remove / in O(1)
交集 a & b O(min(len a, len b))——遍历较小集合去查另一个
并集 a | b O(len a + len b)
差集 a - b O(len a)

六、collections.deque:分块双向链表

list 的头部操作是 O(n)。当你需要两端都高效增删(队列、滑动窗口),用 collections.deque——用法和 list 很像,但多了 appendleft/popleft:

from collections import deque

dq = deque([1, 2, 3])
dq.append(4)         # 尾部追加
dq.appendleft(0)      # 头部追加
print(dq.pop())        # 4 —— 弹出尾部
print(dq.popleft())    # 0 —— 弹出头部
print(dq)               # deque([1, 2, 3])

maxlen 参数能做成固定长度的环形缓冲区,常用于”最近 N 条记录”:

from collections import deque

recent = deque(maxlen=3)
for x in [1, 2, 3, 4, 5]:
    recent.append(x)
print(recent)   # deque([3, 4, 5], maxlen=3) —— 超出的自动从另一端挤出

它的实现不是普通双向链表(那样每个元素一个节点、cache 极差),而是分块双向链表(doubly-linked list of fixed-size blocks):

block(每块 64 个指针)      block                block
+----------------+       +----------------+     +----------------+
| prev | 元素... |  <->  | prev | 元素... | <-> | prev | 元素... |
| next |         |       | next |         |     | next |         |
+----------------+       +----------------+     +----------------+
   ^ leftblock                                     ^ rightblock

为什么不像标准链表那样每个元素一个节点:纯链表的每个节点是一次独立的堆分配,元素在内存里彼此不相邻,遍历时 cache miss 严重,而且每个节点要多存 prev/next 两个指针,这份开销对小整数这类元素而言占比相当可观。deque 把若干元素打包进一个固定大小(64)的 block,链表结构只在 block 这一级存在,block 内部则是连续数组——相当于”链表套数组”,用更粗的链接粒度换回大部分的 cache 局部性,同时保留两端 O(1) 增删(在头/尾 block 里挪一个位置,满了就再链一个新 block,不需要像 list 那样整体搬移)。代价是随机索引 dq[i] 变成 O(n)(要沿 block 链走),这是”优化两端、放弃中间随机访问”这个取舍的直接体现——deque 从设计上就不是给随机访问用的容器。

操作 list deque
尾部 append/pop O(1) 摊还 O(1)
头部 insert/pop O(n) O(1)
随机索引 x[i] O(1) O(n)

八、与 C 的内存模型对照小结

C/C++ Python 关键差异
int a[N] list C 连续存值;list 存指针,值散落堆上
std::vector<T> list vector 存值且可存栈上小对象;list 永远存指针
struct {int; char*;} tuple / namedtuple / 类 struct 按偏移排布字段;tuple 按下标存指针

核心记忆点:Python 容器是引用的容器。拷贝容器默认只拷指针(浅拷贝),多个容器可以别名同一批元素;要真正独立必须 deepcopy。想要 C 那样连续存值、cache 友好、可向量化的数组,去用 array/memoryview 或 numpy。

小结

  • list 是指向 PyObject 的指针动态数组:选数组而非链表,是因为随机访问和尾部追加远比中间插入删除常见,数组把前者做到 O(1) 而链表做不到;1/8 温和扩容在”重分配次数”与”内存浪费”间取折中;浅拷贝只拷指针不拷元素。
  • tuple 把指针数组内联在对象尾部、一次分配、不可变;不可变性是它能作 dict 键的前提——可变对象的哈希值会随内容漂移,破坏哈希表定位,这与第 03 章的可变性讨论完全呼应。
  • dict 用”稀疏 indices(哈希定位)+ 紧凑 entries(数据存储、保插入序)“两个数组实现开放寻址哈希表:保序是紧凑数组物理排列的副产品,不需要像 Java LinkedHashMap 那样额外挂链表。
  • set 是没有 value、也不保证顺序的哈希表,因此不需要 dict 那套双数组结构;frozenset 为”需要不可变、可哈希的集合”这个真实需求存在。
  • deque 是分块双向链表:用固定大小的 block 换回链表结构下的 cache 局部性,两端 O(1)、随机访问 O(n)。
