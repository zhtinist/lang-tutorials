并发模型:GIL、线程、进程与协程

本章先讲 threading(线程)、multiprocessing(进程)、asyncio(协程)这三套并发 API 怎么写、怎么用——不需要任何底层知识就能看懂;再讲 CPython 特有的 GIL(全局解释器锁),理解它你就理解了为什么 Python 的多线程”看起来能并行,实际上不能”,以及何时该用线程、进程还是协程。作为 C/C++ 工程师,你熟悉 pthread、std::thread、原子操作、内存序、互斥量,后半部分会逐一和它们对照。

零、先会用:threading / multiprocessing / asyncio 速览

threading:开一个线程跑函数

import threading

def greet(name):
    print(f"hello, {name}")

t = threading.Thread(target=greet, args=("world",))
t.start()     # 启动线程
t.join()      # 等待线程结束
print("done")
# hello, world
# done

multiprocessing:开一个子进程跑函数

用法和 threading 几乎一样,只是把 Thread 换成 Process:

from multiprocessing import Process

def greet(name):
    print(f"hello from process, {name}")

if __name__ == "__main__":   # 这行是必须的(见下文)
    p = Process(target=greet, args=("world",))
    p.start()
    p.join()
    print("done")
# hello from process, world
# done

asyncio:用 async/await 写协程

import asyncio

async def greet(name):              # async def 定义一个"协程函数"
    print(f"start {name}")
    await asyncio.sleep(0.01)       # await:等待一个异步操作
    print(f"done {name}")
    return name

async def main():
    result = await greet("world")   # 调用协程要用 await
    print(result)

asyncio.run(main())                  # 启动事件循环运行 main
# start world
# done world
# world

concurrent.futures:更省心的线程池/进程池

from concurrent.futures import ThreadPoolExecutor

def square(x):
    return x * x

with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(square, range(5)))
print(results)   # [0, 1, 4, 9, 16]

这四个是并发编程最常用的入口。下面讲它们各自的调度模型和取舍——先用法,后底层。

一、开始深入:GIL 是什么

从这一节起,进入本章的底层部分:线程、进程、协程的基本用法你已经会了,以下解释为什么”Python 多线程好像并不能并行跑满多核”,以及每种方式各自适合什么场景。

CPython 传统上有一把全局互斥锁:任何时刻,只有一个线程能持有 GIL 并执行 Python 字节码。它不是语言规范的一部分,而是 CPython 这个实现的产物(Jython、IronPython 没有 GIL)。(严格说,自 3.12 起 GIL 是每解释器(per-interpreter)的——同一进程里的多个子解释器可各有一把 GIL,见 PEP 684;下文默认讨论最常见的单解释器情形。)

import sys
# GIL 每隔多少个"检查间隔"考虑一次切换(Python 3.2+ 改为基于时间)
print(sys.getswitchinterval())  # 默认 0.005 秒,即 5ms

用 C 的心智模型来看,GIL 相当于在解释器主循环里包了一把大锁:

线程 T1                    GIL                     线程 T2
  |                     [ 空闲 ]                     |
  |-- acquire GIL ------>[ T1 ]                      |
  |  执行字节码...        [ T1 ]     <-- 阻塞等待 ----|
  |  (每 5ms 或阻塞时)                                |
  |-- release GIL ------>[ 空闲 ]                     |
  |                      [ T2 ]<---- acquire GIL -----|
  |  <-- 阻塞等待 --------[ T2 ]     执行字节码...     |

二、为什么存在:保护引用计数

回顾第 09 章——CPython 用引用计数管理内存。每个对象头里有 ob_refcnt。当你 a = b 或函数传参时,引用计数 +1;变量销毁时 -1,归零即释放。

问题来了:ob_refcnt++ 和 ob_refcnt-- 不是原子操作。在 C 层面它是”读-改-写”三步。如果两个线程同时对同一对象增减引用计数而不加锁:

初始 refcnt = 1
T1: 读 refcnt(1)          T2: 读 refcnt(1)
T1: +1 得 2               T2: -1 得 0
T1: 写回 2                T2: 写回 0   <-- 竞态!对象被误释放,或永不释放

这正是你熟悉的 data race。解决方案有两种:

  1. 给每个对象的引用计数加细粒度锁/用原子操作——但对象数以百万计,原子操作在每次 INCREF/DECREF 都触发缓存一致性开销,单线程性能暴跌。
  2. 用一把大锁保护整个解释器状态——这就是 GIL。

CPython 选了方案 2,因为它让单线程性能最优,且让 C 扩展的编写变得极其简单(扩展作者默认自己独占解释器,无需考虑并发)。GIL 保护的不止引用计数,还有解释器的各种全局状态(内存分配器、内部缓存等)。

三、后果:CPU 密集型多线程不加速

由于同一时刻只有一个线程跑字节码,纯计算(CPU 密集)任务开多线程不会变快,反而因为锁竞争和上下文切换略微变慢。

import time
from threading import Thread

def cpu_bound(n: int) -> int:
    # 纯计算:狂占 CPU,不释放 GIL
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 20_000_000

# 串行
t0 = time.perf_counter()
cpu_bound(N); cpu_bound(N)
print(f"串行:{time.perf_counter() - t0:.2f}s")

# 双线程
t0 = time.perf_counter()
threads = [Thread(target=cpu_bound, args=(N,)) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(f"双线程:{time.perf_counter() - t0:.2f}s")  # 不会快一倍,可能还更慢

对比 C++:两个 std::thread 跑纯计算能吃满两个核心,近乎线性加速。Python 里两个线程在抢同一把 GIL,始终只有一个核心在真正工作:

双核 CPU 上的 CPU 密集型 Python 多线程:

Core 0: [T1][   ][T1][   ][T2][   ]...   <- 两线程轮流用同一核心
Core 1: [空闲............................]  <- 第二个核心浪费

C++ std::thread 同任务:
Core 0: [T1][T1][T1][T1][T1][T1]...      <- 满载
Core 1: [T2][T2][T2][T2][T2][T2]...      <- 满载,真并行

四、threading:IO 密集型才是主场

关键点:执行阻塞式 IO(读写文件、网络请求、time.sleep)时,线程会主动释放 GIL。因为等 IO 期间不需要执行字节码,让别的线程跑正合适。此外解释器也会定时(默认 5ms)强制切换。

所以 threading 适合 IO 密集型 场景:多个线程大部分时间在等网络/磁盘,GIL 在等待期间被释放,实现了有效的并发。

import time, urllib.request
from threading import Thread

def fetch(url: str) -> None:
    # 网络请求阻塞期间释放 GIL,其他线程可运行
    with urllib.request.urlopen(url) as r:
        r.read()

urls = ["https://example.com"] * 8

t0 = time.perf_counter()
for u in urls: fetch(u)
print(f"串行:{time.perf_counter() - t0:.2f}s")

t0 = time.perf_counter()
threads = [Thread(target=fetch, args=(u,)) for u in urls]
for t in threads: t.start()
for t in threads: t.join()
print(f"多线程:{time.perf_counter() - t0:.2f}s")  # 显著加速,IO 等待被重叠

线程同步原语

和 C++ 一样,共享可变状态需要保护。注意:即使有 GIL,复合操作(如 x += 1)也不是原子的——它由多条字节码组成,中间可能被切换。

from threading import Lock, Thread

counter = 0
lock = Lock()

def worker():
    global counter
    for _ in range(100_000):
        with lock:          # 相当于 std::lock_guard,退出作用域自动释放
            counter += 1     # 读-改-写,必须加锁

threads = [Thread(target=worker) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)  # 有锁:400000;去掉锁:小于该值(丢失更新)

threading 还提供 RLock(可重入)、Semaphore、Event、Condition、Barrier,语义与 C++ 标准库对应物基本一致。

五、multiprocessing:绕过 GIL 实现真并行

要在 CPU 密集型任务上吃满多核,就得开多个进程。每个进程有独立的 Python 解释器和独立的 GIL,互不干扰,由 OS 调度到不同核心。

import time
from multiprocessing import Process

def cpu_bound(n: int) -> int:
    return sum(i * i for i in range(n))

if __name__ == "__main__":   # Windows/spawn 模式必须有此保护
    N = 20_000_000
    t0 = time.perf_counter()
    ps = [Process(target=cpu_bound, args=(N,)) for _ in range(2)]
    for p in ps: p.start()
    for p in ps: p.join()
    print(f"双进程:{time.perf_counter() - t0:.2f}s")  # 接近串行的一半

代价是进程间不共享内存(和 fork 后各写各的一样)。数据传递有两条路:

方式 机制 说明
参数/返回值/队列 pickle 序列化 对象被序列化成字节流,经管道传给子进程再反序列化。有拷贝开销,且对象必须可 pickle
Value/Array/shared_memory 共享内存 直接映射同一块物理内存,零拷贝,但需自己加锁同步 
from multiprocessing import shared_memory
import numpy as np

# 创建一块共享内存,多个进程可映射同一块物理页,无需拷贝
shm = shared_memory.SharedMemory(create=True, size=1024)
buf = np.ndarray((256,), dtype=np.int32, buffer=shm.buf)
buf[0] = 42
# 另一进程用 shm.name 附着到同一块内存
shm.close(); shm.unlink()

启动方式(start method):fork(拷贝父进程内存,快,但与多线程共用时不安全)、spawn(全新解释器,干净但慢)、forkserver。默认值随平台/版本而变:macOS、Windows 默认 spawn;Linux 早期默认 fork,但 3.14 起改为默认 forkserver(因为 fork 在多线程程序里易出问题)。别再记成”Unix 默认 fork”。

六、asyncio:单线程事件循环

asyncio 是另一条路线:不靠多线程/多进程,而是在单个线程内用事件循环 + 协程实现高并发 IO。本质是协作式多任务(cooperative multitasking),对比线程的抢占式(preemptive)调度。

  • 协程(coroutine):用 async def 定义。调用它不会立即执行,而是返回一个协程对象(类似惰性的任务描述)。
  • await:在此处主动让出控制权给事件循环。只有遇到 await 一个未就绪的操作时,当前协程才暂停,事件循环去跑别的协程。
  • 事件循环(event loop):一个 while 循环,维护就绪任务队列,配合 OS 的 epoll/kqueue/IOCP 监听 IO 事件,哪个 IO 就绪就恢复对应协程。
import asyncio, time

async def fetch(name: str, delay: float) -> str:
    print(f"{name} 开始")
    await asyncio.sleep(delay)   # 让出控制权!事件循环去跑别的协程
    print(f"{name} 完成")
    return name

async def main():
    t0 = time.perf_counter()
    # gather 并发调度多个协程,总耗时约等于最长的那个,而非累加
    results = await asyncio.gather(
        fetch("A", 2), fetch("B", 1), fetch("C", 3),
    )
    print(results, f"耗时 {time.perf_counter() - t0:.2f}s")  # 约 3s,非 6s

asyncio.run(main())   # 创建事件循环,跑 main 直到完成,再关闭循环

事件循环的心智模型:

事件循环(单线程):
  ┌─> 从就绪队列取一个协程,运行到下一个 await
  │      协程 A 跑到 await sleep(2) --> 注册定时器,挂起 A
  │      协程 B 跑到 await sleep(1) --> 注册定时器,挂起 B
  │      队列空,epoll 等待 IO/定时器就绪
  │      1s 后 B 的定时器到期 --> 把 B 放回就绪队列
  └──── 恢复 B 从 await 之后继续 ...

把并发的执行交错显式打印出来——每个 await 都是一次让出,可以看到两个协程在切换点轮流推进:

import asyncio

async def worker(name: str, steps: int) -> None:
    for i in range(steps):
        print(f"{name}-{i}")
        await asyncio.sleep(0)   # 让出控制权,事件循环去跑另一个协程

async def main():
    await asyncio.gather(worker("A", 3), worker("B", 3))

asyncio.run(main())
# A-0
# B-0
# A-1
# B-1
# A-2
# B-2   —— 两协程在每个 await 处交替推进(协作式,顺序确定)

与线程的对照

维度 threading asyncio
调度 抢占式,OS 决定何时切换 协作式,只在 await 处切换
切换点 任意字节码边界(不可控) 显式 await(可控、可推理)
并行度 受 GIL 限制,无真并行 单线程,无真并行
竞态 需要锁,难调试 切换点确定,同线程内多数操作无需锁
开销 每线程 MB 级栈 + OS 调度 每协程几 KB,可开数万个
适用 IO 密集 + 有阻塞库 IO 密集 + 全异步生态

关键陷阱:在协程里调用阻塞函数(如同步的 requests.get、time.sleep)会卡死整个事件循环——因为没有 await,不会让出控制权。必须用异步库(aiohttp、httpx)或把阻塞调用丢到线程池:await loop.run_in_executor(None, blocking_fn)。

import asyncio, time

def blocking_io(x: int) -> int:   # 同步阻塞函数(比如同步版 requests.get)
    time.sleep(0.05)
    return x * 2

async def main():
    loop = asyncio.get_running_loop()
    # 丢进默认线程池执行,阻塞发生在别的线程,事件循环不被冻结
    results = await asyncio.gather(
        loop.run_in_executor(None, blocking_io, 1),
        loop.run_in_executor(None, blocking_io, 2),
    )
    print(results)   # [2, 4]

asyncio.run(main())

七、concurrent.futures:统一的高层接口

concurrent.futures 提供了线程池和进程池的统一 API,是日常最常用的入口(「先会用」一节已经展示了 ThreadPoolExecutor 的基本用法)。Future 对象类似 C++ 的 std::future。

from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def task(x: int) -> int:
    return x * x

# IO 密集用线程池
with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(task, range(10)))

# CPU 密集用进程池(绕过 GIL)
if __name__ == "__main__":
    with ProcessPoolExecutor(max_workers=4) as ex:
        futures = [ex.submit(task, i) for i in range(10)]
        for f in futures:
            print(f.result())   # 阻塞直到该任务完成

选型口诀:CPU 密集 → ProcessPool;IO 密集 → ThreadPool 或 asyncio。

八、free-threaded(no-GIL):GIL 的黄昏?

CPython 3.13(2024 年发布)以实验性形式引入了 free-threaded 构建(PEP 703),即”no-GIL”;到 3.14 转为官方正式支持,但仍不是默认。它通过偏向锁 + 无锁引用计数(biased reference counting)+ 每对象锁等技术,让引用计数在多线程下安全,从而移除 GIL,实现真正的多核并行线程。

要点:

  • 需要专门的构建版本(如 python3.13t / python3.14t),普通发行版默认仍是带 GIL 的构建。
  • 单线程性能有一定回退(早期约百分之几,持续优化中)。
  • C 扩展需要适配(声明支持 free-threading),生态迁移需时间。
  • 3.13 实验性 → 3.14 正式支持(但非默认);带 GIL 与 free-threaded 两种构建将长期共存。

如果这一路线成功,未来 Python 的 threading 将能像 C++ 的 std::thread 一样吃满多核,multiprocessing 的很多 pickle/共享内存痛点也会缓解。但在可预见的将来,上面讲的选型原则仍然成立。

小结

  • 用法层面:threading.Thread(target=...) 起线程,multiprocessing.Process(target=...) 起进程,写法几乎一样;async def/await 定义并驱动协程,asyncio.run(main()) 启动事件循环;concurrent.futures 的 ThreadPoolExecutor/ProcessPoolExecutor 提供更省心的池化接口(.map/.submit)。
  • 底层层面:GIL 是 CPython 的解释器级大锁(3.12 起为每解释器 per-interpreter),一次只有一个线程执行字节码,存在的根本原因是保护非原子的引用计数等解释器内部状态。CPU 密集型多线程不加速;IO 密集型多线程有效,因为阻塞 IO 时释放 GIL。multiprocessing 用独立进程+独立 GIL 实现真并行,代价是数据靠 pickle 或共享内存传递。asyncio 用单线程事件循环 + 协作式协程,await 处显式让出,适合高并发 IO;切勿在其中调用阻塞函数。free-threaded 构建(3.13 实验、3.14 正式但非默认)正在移除 GIL,是长期演进方向。
