错误处理与资源管理

本章两条主线:第一条是异常机制(try/except 怎么写、异常也是对象);第二条是 CPython 的内存管理(引用计数 + 分代 GC + pymalloc 内存池)。前半部分先讲 try/except/finally、自定义异常、with 语句怎么写、怎么用——不需要任何底层知识就能看懂;后半部分讲异常对象和上下文管理器的底层协议,以及内存管理这条更偏底层的主线,会和手动 malloc/free、RAII、智能指针逐一对照。

0. 先会用:try / except / finally 与 with 速览

用 try/except 捕获可能出错的代码,finally 里的代码无论是否出错都会执行:

try:
    x = int("abc")           # 这行会抛出 ValueError
except ValueError as e:
    print("转换失败:", e)    # 转换失败: invalid literal for int() with base 10: 'abc'
finally:
    print("清理完成")         # 无论是否出错都会执行

else 块只在 try 里没有抛出异常时才执行:

def divide(a, b):
    try:
        result = a / b
    except ZeroDivisionError:
        print("不能除以零")
        return None
    else:
        print("计算成功")
        return result

print(divide(10, 2))    # 计算成功 \n 5.0
print(divide(10, 0))    # 不能除以零 \n None

自己定义异常类型,用 raise 主动抛出:

class MyError(Exception):        # 自定义异常:继承 Exception 即可
    pass

def check(x):
    if x < 0:
        raise MyError(f"x 不能为负数: {x}")
    return x

try:
    check(-1)
except MyError as e:
    print("捕获自定义异常:", e)   # 捕获自定义异常: x 不能为负数: -1

打开文件、网络连接这类需要”用完就关”的资源,用 with 语句——离开 with 块会自动关闭,不需要手动写 close():

with open("data.txt", "w") as f:
    f.write("hello")
# 离开 with 块时,文件已经自动关闭
with open("data.txt") as f:
    print(f.read())       # hello

这四个是日常错误处理和资源管理最常用的写法。下面展开讲它们的细节和底层原理——先用法,后底层。

1. 异常是对象

怎么用

写 except 时,记住一条实用规则:捕获异常尽量写 except Exception,而不是裸的 except: 或 except BaseException——否则会连用户按 Ctrl-C(KeyboardInterrupt)、sys.exit()(SystemExit)这些”本不该被业务代码吞掉”的信号也一起捕获了:

print(issubclass(KeyboardInterrupt, Exception))     # False
print(issubclass(KeyboardInterrupt, BaseException)) # True

底层与 C 对比

Python 的异常不是 C 的错误码,也不是 C++ 里可以抛任意类型的 throw 42。所有异常都是继承自 BaseException 的对象,携带信息(args)与回溯(traceback)。层级大致如下:

BaseException            ← 顶层,except Exception 抓不到它的这几个直接子类
├── SystemExit           ← sys.exit() 触发,别顺手 except 掉
├── KeyboardInterrupt    ← Ctrl-C,别顺手 except 掉
├── GeneratorExit
└── Exception            ← 「普通」错误的基类,你自己的异常应继承它
    ├── ValueError / TypeError / KeyError / IndexError / ...
    ├── ArithmeticError → ZeroDivisionError
    ├── OSError → FileNotFoundError / PermissionError / ...
    └── RuntimeError / StopIteration / ...

关键推论:except Exception 抓不到 KeyboardInterrupt、SystemExit——因为它们直接挂在 BaseException 下,故意让”用户想中断程序”不被业务代码误吞。

对照 C++:C++ 能 throw 任意类型(throw 42; 合法);Python 只能抛 BaseException 的实例,类型体系统一,except 按 isinstance 匹配。

2. try / except / else / finally

怎么用

一个 except 也能同时捕获多种类型(写成元组),多个 except 按顺序匹配:

def parse(s, data):
    try:
        x = int(s)               # 可能抛 ValueError
        y = data[x]               # 可能抛 IndexError / KeyError
    except (ValueError, KeyError) as e:   # 一个 except 抓多种类型(元组)
        print("坏输入:", e)      # e 是异常对象,e.args 是参数元组
    except IndexError:
        print("越界")
    else:
        print("没出错才执行:", y)   # try 未抛异常时运行
    finally:
        print("无论如何都执行")     # 清理,总会运行——即使 try 里 return/break

parse("abc", [1, 2, 3])   # 坏输入: invalid literal ... \n 无论如何都执行
parse("5", [1, 2, 3])     # 越界 \n 无论如何都执行
parse("1", [1, 2, 3])     # 没出错才执行: 2 \n 无论如何都执行

底层与 C 对比

  • except 按从上到下第一个 isinstance 匹配的分支处理,匹配到就停。所以子类异常要写在父类前面。
  • else 只在 try 块没有异常时运行(把”正常路径”和”可能抛错的代码”分开)。
  • finally 总会执行,常用于释放资源。注意:若 finally 里有 return,会覆盖 try/except 里的返回值(和 C++ 析构不同,这是显式控制流,容易踩)。

底层:CPython 的异常处理靠字节码里的异常表(exception table,3.11 起零成本 try)——没有异常时 try 几乎不产生额外开销,只有真正抛出时才查表跳转。这与 C++ 的”零成本异常”(zero-cost EH,正常路径无开销)思路一致。

3. raise、自定义异常与异常链

怎么用

用 raise ... from ... 可以在包装异常时保留原始原因,排查问题时能看到完整链条:

class ConfigError(Exception):        # 自定义异常:继承 Exception 即可
    pass

def load(path):
    try:
        return open(path).read()
    except OSError as e:
        raise ConfigError(f"读配置失败: {path}") from e   # 异常链:保留原因

try:
    load("/no/such/file")
except ConfigError as e:
    print("捕获:", e)             # 捕获: 读配置失败: /no/such/file
    print("原因:", e.__cause__)   # 原因: [Errno 2] No such file or directory: '/no/such/file'

底层与 C 对比

  • raise ValueError("msg") 抛出;裸 raise(在 except 内)重新抛出当前异常,保留原始 traceback。
  • raise NewError from cause 显式设置 __cause__,打印时出现 The above exception was the direct cause of...;若不写 from,在 except 内再抛也会自动记录 __context__(隐式链),打印 During handling of the above exception...。
  • 想切断链:raise NewError from None。

对照 C++:类似”捕获底层异常、包装成领域异常再抛”,但 Python 的 from 让原因链在 traceback 里可见(存在 __cause__/__context__ 属性上),排错更友好。

4. 上下文管理器:with 与 RAII

怎么用

「先会用」已经展示了用内置的 open() 配合 with 自动关闭文件。你也可以自己实现一个上下文管理器,只要定义 __enter__ 和 __exit__:

import time

class Timer:
    def __enter__(self):
        self.t0 = time.perf_counter()
        return self
    def __exit__(self, exc_type, exc_val, exc_tb):
        elapsed = time.perf_counter() - self.t0
        print(f"耗时 {elapsed:.4f}s 以内")
        return False            # 返回 False:不吞异常,照常向外传播

with Timer():
    total = sum(range(1000))
print(total)   # 499500

更简单的写法是用 contextlib.contextmanager 把一个生成器函数变成上下文管理器:

from contextlib import contextmanager

@contextmanager
def opened(path, mode="r"):
    f = open(path, mode)
    try:
        yield f                 # yield 之前 = __enter__,之后 = __exit__
    finally:
        f.close()

with opened("data.txt") as f:
    print(f.read())    # hello

底层与 C 对比

C++ 靠析构函数 + 栈对象做 RAII——离开作用域自动释放。Python 没有确定性析构(见下文 __del__),它用 with 语句 + 上下文管理器达到同样的”确定性释放”:

with open("data.txt") as f:      # 进入:调用 f.__enter__()
    process(f)
# 离开 with 块(正常或异常):调用 f.__exit__(exc_type, exc_val, exc_tb),关闭文件

__exit__ 会收到异常信息(正常退出时三个都是 None);返回真值可吞掉异常,上面 Timer.__exit__ 返回 False 就是明确表示”不吞异常”。

对照 RAII:with 是显式划定作用域的 RAII——比 C++ “离开花括号即析构”更啰嗦,但释放时机同样确定(不依赖 GC)。这是 Python 管理文件、锁、连接的首选方式。

从这里开始,进入本章第二条主线——CPython 怎么自动管理内存。这部分不再引入新语法(del 你已经会用),直接对照你熟悉的 C malloc/free、C++ shared_ptr 看底层机制即可。

5. 引用计数:主力回收机制

CPython 内存管理的主力是引用计数:每个对象头里有一个 ob_refcnt,有新引用指向它就 +1,引用消失(del、变量重绑定、离开作用域、容器移除)就 -1;减到 0 立刻释放——确定性、即时,类似 C++ 的 shared_ptr,但因为有 GIL 保护,计数增减不需要原子操作(这也是 GIL 存在的原因之一,见「并发模型」)。

import sys
a = object()
print(sys.getrefcount(a))   # 例如 2:注意 getrefcount 的实参本身也算一个临时引用,故比直觉多 1
b = a                       # 又一个引用
print(sys.getrefcount(a))   # 3
del b                       # 引用 -1

对照 C:C 里 free 由你手动调用、时机由你掌控;CPython 帮你在”最后一个引用消失的那一刻”自动 free,确定且即时——这点比 Java/Go 的追踪式 GC(回收时机不确定)更接近 C/C++ 的心智模型。

实测确定性即时回收:引用计数归零的瞬间 __del__ 就被调用,时机完全可预测:

class Res:
    def __init__(self, name): self.name = name
    def __del__(self): print(f"{self.name} released")

a = Res("a")
b = Res("b")
print("before del a")
del a               # a 的最后一个引用消失 -> 计数归零 -> 立即回收
print("after del a")
# before del a
# a released        <- 就在 del 这一刻,不是等到脚本结束
# after del a
# b released        <- 脚本结束、b 离开作用域时

引用计数搞不定的,只有一种情况:循环引用。

6. 分代 GC、__del__ 与 weakref

a 引用 b、b 又引用 a,即使外部都不用了,两者的计数也降不到 0——引用计数漏了这种环。CPython 因此再叠一层追踪式的分代垃圾回收器(gc 模块)专门回收循环垃圾:

import gc
print(gc.get_threshold())   # (2000, 10, 10):第 0 代分配-释放差达 2000 触发一次回收(阈值随版本变)
gc.collect()                # 手动跑一次完整回收,返回回收的不可达对象数
  • 三代(0/1/2):新对象在第 0 代,挺过一次回收就”晋升”到更老的代;越老的代扫描越不频繁。基于”大多数对象很快就死”的分代假设(和 Java/Go 的分代思路一致)。
  • 只有可能形成环的容器对象(list/dict/set/实例等)才被 gc 追踪;int、str 这类不可能成环的原子对象不进 gc,只靠引用计数。

实测:纯循环引用引用计数回收不掉,要靠 gc。下面构造 a<->b 互引,删除外部引用后对象仍活着,直到 gc.collect() 才被回收:

import gc
class Node:
    def __del__(self): print("Node freed")

gc.collect()          # 先清掉解释器启动残留,让计数干净
gc.disable()          # 关掉自动回收,便于观察
a = Node(); b = Node()
a.ref = b; b.ref = a  # 制造循环引用
del a; del b          # 外部引用没了,但彼此还引着 -> 计数不归零
print("after del, still alive")   # 此刻两个对象都没被回收
print("collected:", gc.collect()) # 手动追踪式回收环
# after del, still alive
# Node freed
# Node freed
# collected: 2        <- gc 回收了 2 个不可达对象

__del__(析构钩子)不可靠,别拿它当 RAII:

class Res:
    def __del__(self):          # 对象被回收时调用
        print("released")       # 但"何时"调用不保证
  • 触发时机取决于引用计数何时归零 / gc 何时跑,不确定;解释器退出时甚至可能不调用。
  • 自 Python 3.4(PEP 442) 起,带 __del__ 的循环引用也能被 gc 回收(会正确调用析构),不再像旧版那样卡在 gc.garbage 里。但即便如此,资源释放仍应用 with,不要赌 __del__。

weakref——不增加引用计数的引用(对标 C++ weak_ptr):

import weakref
class C: pass
c = C()
r = weakref.ref(c)     # r 不让 c 的计数 +1
print(r() is c)        # True:目标还活着,r() 返回它
del c                  # 唯一强引用消失,c 被回收
print(r())             # None:弱引用自动失效

用途:缓存(不想因为缓存住而阻止对象回收)、观察者、以及打破循环引用。

7. pymalloc:CPython 的内存池

Python 程序疯狂创建销毁小对象(每个整数、每个字符串都是堆对象)。如果每次都调 C 的 malloc/free,开销和碎片都受不了。CPython 因此自带一个分层内存分配器 pymalloc,专门服务 ≤ 512 字节的小对象:

Arena  ── 约 1 MiB 一大块,直接向操作系统 mmap 申请
  │  切成若干
Pool   ── 一个内存页大小(x86 常见 4 KB;Apple Silicon 为 16 KB),一个 pool 只服务一种 size class
  │  切成等长
Block  ── 按 16 字节对齐的等长小块:16, 32, 48, …, 512 字节,共约 32 个 size class
             一次小对象分配 = 从对应 pool 的空闲链表摘一个 block

工作方式:

  • 申请小对象时,按大小归入某个”size class”,从对应 pool 的空闲块链表里 O(1) 摘一个 block;释放时归还到空闲链表,不马上还给 OS。
  • 因此小对象的分配/释放几乎不碰系统 malloc,极快;同一 size class 的对象聚在一起,减少碎片。
  • > 512 字节的大对象绕过 pymalloc,直接走系统 malloc。
  • 完全空的 pool/arena 才可能归还操作系统。所以你会看到 Python 进程内存”涨上去下不来”——很多是被内存池占着复用,并非泄漏。
  • 上述常量(arena≈1 MiB、pool=系统页大小、16 字节对齐、约 32 个 size class)可用 python3 -c "import sys; sys._debugmallocstats()" 现场核对,数值随版本/平台略有差异。

对照 C:这相当于 CPython 内建了一个为”海量小对象”优化的对象池 allocator,层级(arena→pool→block)类似很多游戏/高性能 C++ 项目里手写的 slab/pool allocator。

此外还有空闲链表(free list):小整数、小 tuple、常用大小的 list/frame 等,销毁后其内存块被挂到专用 free list 里等着复用,连 pymalloc 那一步都省了。

8. 全景小结

机制 作用 C/C++ 对照
引用计数 主力,即时回收,确定性 shared_ptr
分代 GC 兜底回收循环引用 追踪式 GC(Java/Go 才有)
weakref 不增计数的引用 weak_ptr
with / 上下文管理器 确定性资源释放 RAII / 析构函数
__del__ 析构钩子(不可靠,慎用) 析构函数(但时机不确定)
pymalloc 小对象内存池 slab/pool allocator
free list 常用对象内存复用 对象池

一句话:Python 的内存管理是”引用计数(像 shared_ptr,给你确定性)+ 分代 GC(兜底环)+ 多层内存池(给你速度)“的组合;资源释放请用 with,别赌 __del__。

小结

  • 用法层面:try/except/else/finally 分别处理”可能出错的代码”/“匹配到的错误类型”/“没出错时才跑的代码”/“总会跑的清理代码”;except 按从上到下第一个匹配的类型处理;自定义异常继承 Exception;raise ... from ... 包装异常并保留原因;资源释放优先用 with(内置的或自己用 __enter__/__exit__、@contextmanager 实现的都行)。
  • 底层层面:异常是 BaseException 的实例,3.11 起用零成本异常表实现;CPython 内存管理靠引用计数(即时确定,类似 shared_ptr)兜底加分代 GC(专治循环引用),__del__ 触发时机不可靠;pymalloc 用 arena→pool→block 三层结构给小对象加速。
