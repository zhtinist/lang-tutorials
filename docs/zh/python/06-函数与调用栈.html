函数与调用栈

C/C++ 里参数传递只有两种物理机制:按值(拷贝一份到被调栈帧)和按指针(拷贝一个地址);C++ 又加了引用(&,给同一块内存起别名)。Python 只有一种传参方式,且它既不是这三种里的任何一种,也不是三者的简单混合,必须精确地重新说清它是什么。

一、函数定义与调用

定义函数用 def,缩进的部分是函数体,return 返回结果:

def greet(name):
    return f"Hello, {name}!"

message = greet("Ann")
print(message)            # Hello, Ann!

def add(a, b=10):          # b 有默认值,调用时可以不传
    return a + b

print(add(3))              # 13 —— 不传 b,用默认值 10
print(add(3, 5))           # 8  —— 传了 b,覆盖默认值

def show(*args, **kwargs):  # 收集多余的位置/关键字实参
    print(args, kwargs)

show(1, 2, x=3)              # (1, 2) {'x': 3}

二、函数也是对象

C 里函数是一段代码的地址(函数指针指向 text 段),编译期就定死,要把函数当参数传递需要专门的函数指针语法。Python 里 def 是一条运行时执行的语句,它在执行时创建一个 function 对象放到堆上,再把这个对象绑定到名字:

def add(a, b):
    return a + b

print(type(add))          # <class 'function'>
print(add.__code__)       # 编译好的 code 对象(字节码在这)
print(add.__name__)       # 'add'
add_alias = add           # 只是多一个名字指向同一个函数对象

一个 function 对象里装着:__code__(不可变的字节码及元数据)、__defaults__(默认参数值)、__globals__(它能看到的全局命名空间)、__closure__(闭包捕获的变量)。

为什么函数在 Python 里是一等对象,能赋值、能当参数传、能当返回值、能塞进容器,而不像 C 那样需要一套独立的函数指针语法:这是「运行模型」里”一切皆对象”这条哲学延伸到函数本身的直接结果,不是为函数专门开的特例。C 的类型系统里,函数和数据是两个不同范畴的东西——数据有类型、能被变量持有,函数只是代码段里的一个入口地址,要把它当”值”使用(赋值、传参、存进结构体)必须借助函数指针这一单独的类型语法(int (*fp)(int, int)),因为 C 的变量模型是”类型决定内存布局”,而函数不是一种可以被值语义安放进变量槽里的数据。Python 的变量从不内联存值,永远是指向堆对象的引用(见「变量与内存模型」);既然一切值都以”堆对象 + 引用”的统一方式存在,def 语句创建的函数没有理由是例外——它和 list、int、自定义类实例一样,只是一种恰好可以被 () 调用的对象。这带来的直接好处是:把函数传给另一个函数(回调、排序的 key、装饰器)、把函数存进字典(分发表)、把函数当返回值(工厂函数、闭包)都是普通的赋值和引用传递,不需要为”这是一个函数值”设计额外语法;代价是每个函数对象都要背对象头、__code__/__globals__ 等属性槽这一份内存开销,比 C 的裸函数指针(一个机器字)重得多。

三、参数传递的真相:按对象引用(按共享传递)

Python 的传参机制叫 call by object reference,也叫 call by sharing(按共享传递)。一句话:实参和形参指向堆上的同一个对象;传递过程拷贝的是引用(指针),不是对象本身。

这听起来像 C 的”传指针”,但有个关键差异:在 C 里传指针,你在函数内对形参 p = ... 重新赋值不影响调用方,但你能 *p = ... 改内容;而 Python 里你手上根本没有”指针的指针”,你无法让调用方的名字改指向别处。用两个维度看最清楚:

维度 Python 的行为
能否改变调用方变量的指向? 不能。形参重新赋值只改本地名字绑定
能否改变对象的内容? 能,但前提是对象可变(list/dict/set/自定义对象)

维度一:重新赋值形参,永远不影响调用方

def rebind(x):
    # x 原本和调用方的实参指向同一对象;
    # 下面这行让本地名字 x 改指向一个新对象,调用方毫不知情
    x = 999
    print("函数内:", x)

n = 1
rebind(n)
print("函数外:", n)     # 1 —— 没变

内存图(调用瞬间,n 和 x 指向同一个 int 对象;赋值后 x 改指向新对象):

调用时(n、x 是同一对象的两个名字):    x = 999 之后:
  n ─┐                                 n ───► [int 1]      ← n 仍指旧对象
     ├──► [int 1]                      x ───► [int 999]    ← x 改指新对象
  x ─┘

这正是为什么 Python 无法写出 C 的 void swap(int *a, int *b) 那种”通过参数改调用方变量”的函数。想交换,只能返回新值:a, b = b, a 再由调用方接收。

维度二:原地修改可变对象,调用方看得见

def mutate(lst):
    # 没有重新绑定 lst,而是调用它的方法修改它指向的那个对象
    lst.append(4)      # 原地修改,不产生新 list
    print("函数内:", lst)

data = [1, 2, 3]
mutate(data)
print("函数外:", data)   # [1, 2, 3, 4] —— 变了!

内存图(只有一个 list 对象,两个名字都指向它):

  data --\
          >---> [list: 1,2,3] --append--> [list: 1,2,3,4]
  lst  --/

这和 C 传指针后 *p 改内容效果一致。区别只在于:Python 里”能不能改”取决于对象本身可不可变,而不是取决于你怎么声明参数。

可变 vs 不可变实参对照

同一段”改一下参数”的代码,对可变和不可变实参表现相反:

def try_change(obj):
    if isinstance(obj, list):
        obj += [99]        # list 的 += 是原地 extend(改内容)
    else:
        obj += 99          # int 的 += 生成新对象再绑定给本地 obj

a = [1, 2]
try_change(a)
print(a)          # [1, 2, 99] —— 可变对象被改了

b = 10
try_change(b)
print(b)          # 10 —— 不可变对象,调用方不受影响

陷阱就在这里:obj += [99] 和 obj += 99 看着都是 +=,但对 list 是原地修改(等价 obj.extend,__iadd__ 返回自身),对 int 是”新建 + 重绑定”(int 没有原地加法,__iadd__ 缺失时退化为 obj = obj + 99)。可变性决定一切。

再来几个例子,彻底看透

例 1:一次调用里同时演示两个维度——同一个函数,对 a 重新赋值(看不见),对 b 原地改(看得见):

def f(a, b):
    a = a + [100]     # 对 a:新建一个 list 再绑定给本地名 a → 调用方看不到
    b.append(100)     # 对 b:调方法原地改它指向的对象 → 调用方看得到
    print("函数内 a =", a, " b =", b)   # [1, 100]  [1, 100]

x = [1]
y = [1]
f(x, y)
print("函数外 x =", x)   # [1]        —— 没变(只改了本地绑定)
print("函数外 y =", y)   # [1, 100]   —— 变了(改了对象内容)

例 2:自定义对象也一样——改属性 = 改内容(可见);重新绑定形参 = 只改本地名(不可见):

class Account:
    def __init__(self, bal):
        self.balance = bal

def touch(acc):
    acc.balance += 50     # 改对象的属性,就是改内容 → 可见
    acc = Account(0)      # 让本地名 acc 改指向一个新对象 → 调用方无感
    print("函数内 acc.balance =", acc.balance)   # 0

a = Account(100)
touch(a)
print("函数外 a.balance =", a.balance)   # 150 —— 属性被改了;a 仍指向原来那个对象

例 3:那我怎样才能真的改调用方变量的指向?两条路:

# 路子一:把新值 return 回去,由调用方自己重新绑定(最 Pythonic)
def older(age):
    return age + 1

age = 30
age = older(age)     # 调用方接收返回值,自己改指向
print(age)           # 31

# 路子二:包一层可变容器,改容器里的元素(相当于 C 的 int**:传"指针的指针")
def set_value(box, val):
    box[0] = val     # 改的是容器内容,可见
holder = [None]
set_value(holder, 42)
print(holder[0])     # 42

例 4:最经典的一处易混——lst = lst + [x] 与 lst += [x] 只差一点,结果相反:

def rebind(lst):
    lst = lst + [9]   # 先算出一个【新 list】,再绑定给本地名 lst → 调用方看不到
def inplace(lst):
    lst += [9]        # 等价 lst.extend([9]),原地改同一个对象 → 调用方看得到

m = [1]
rebind(m)
print(m)              # [1]      —— 没变
n = [1]
inplace(n)
print(n)              # [1, 9]   —— 变了

lst + [9] 是”造新对象”,lst += [9] 是”原地改”——一字之差,天壤之别。这正是「变量与内存模型」里 += 陷阱的另一面。

记忆口诀:传的永远是引用,拷贝的永远是引用本身(一个指针),从不拷贝对象。改指向不外传,改内容看可变。

四、参数种类:位置、关键字、默认、*args、**kwargs

Python 的参数系统比 C 灵活得多。函数签名里各种符号的完整形态:

def f(pos1, pos2, /, both, *args, kw_only, **kwargs):
    #   ^^^^^^^^^^   /   ^^^^  ^^^^^  ^^^^^^^   ^^^^^^^^
    ...
  • / 之前:仅限位置参数(caller 不能用 pos1=... 传),3.8+;
  • / 与 * 之间:位置或关键字都行(最常见);
  • *args:把多余的位置实参收集成一个元组;
  • * 之后(或 *args 之后):仅限关键字参数,必须写 kw_only=...;
  • **kwargs:把多余的关键字实参收集成一个字典。
def demo(a, b, *args, sep="-", **kwargs):
    print(f"a={a} b={b}")
    print("args =", args)        # 元组
    print("kwargs =", kwargs)    # 字典
    print("sep =", sep)

demo(1, 2, 3, 4, sep="|", x=10, y=20)
# a=1 b=2
# args = (3, 4)
# kwargs = {'x': 10, 'y': 20}
# sep = |

底层上,*args 会分配一个新元组装那些额外实参,**kwargs 会分配一个新字典。调用点的 */** 是反向操作(解包):

nums = [1, 2, 3]
print(*nums)             # 等价 print(1, 2, 3),把列表解包成位置实参
opts = {"sep": "/"}
print("a", "b", **opts)  # 把字典解包成关键字实参 -> sep="/"

为什么 Python 用 *args/**kwargs 这种运行时收集的语法处理可变数量参数,而不是像 C++ 变参模板那样在编译期为每种实参组合展开出一份特化代码:两者对应的是完全不同的类型模型。C++ 模板变参(template<typename... Args>)之所以能在编译期展开,前提是编译器在编译期就知道调用点每个实参的具体类型和个数,并据此实例化出一份专门代码——这本身就是静态类型系统提供的信息。Python 的函数在定义时根本不知道将来会被传几个、什么类型的实参,*args/**kwargs 收集的是运行期才确定的一组对象引用,唯一能做的就是在调用发生的那一刻,把多出来的位置实参打包进一个元组、多出来的关键字实参打包进一个字典——这是动态类型语言在没有编译期实参形状信息的前提下,处理”参数个数可变”这一需求的必然实现方式,而不是刻意在两种语言之间制造语法差异。代价是每次调用都要付出构造元组/字典的运行时分配开销,不像 C++ 模板展开那样在编译期就已经生成了参数个数固定的具体代码、运行期零额外开销。

五、默认参数:在 def 执行时求值一次

C++ 的默认实参在每次调用时求值。Python 相反:默认值在 def 语句执行时求值一次,结果存进函数对象的 __defaults__,以后每次调用共享这同一个对象。理解这一点,要先弄清楚 def 本身在做什么:回顾第二节,def 是一条运行时语句,执行到它时,Python 会依次求值参数列表里所有的默认值表达式(此刻,也只有此刻,这些表达式被求值一次),把结果打包存进即将创建的 function 对象的 __defaults__ 属性,然后把这个函数对象绑定到函数名——def 语句执行完毕后,默认值已经是几个具体的、固定的对象,以后每次调用这个函数,不传对应参数时就直接从 __defaults__ 里取出同一个对象来用,不会重新执行那行默认值表达式。

对不可变默认值这没问题,对可变默认值就是著名的坑:

def bad(item, bucket=[]):     # 这个 [] 只在定义时创建一次!
    bucket.append(item)
    return bucket

print(bad(1))    # [1]
print(bad(2))    # [1, 2] —— 惊了!同一个 list 被反复复用
print(bad(3))    # [1, 2, 3]

print(bad.__defaults__)   # ([1, 2, 3],) —— 默认值就存在这里,被改花了

正确写法是用 None 哨兵,在函数体内新建:

def good(item, bucket=None):
    if bucket is None:
        bucket = []           # 每次调用都新建,互不干扰
    bucket.append(item)
    return bucket

print(good(1))   # [1]
print(good(2))   # [2]

为什么 Python 不干脆把默认值设计成每次调用都重新求值,像 C++ 那样避免这整类坑:这背后是一个关于”默认值到底是什么”的语义选择,而不是实现疏忽。如果默认值每次调用都重新求值,def f(t=time.time()): 这种写法的含义就变得依赖上下文——你没法从 def 这一行代码本身判断 time.time() 到底是”函数定义时的那一刻”还是”每次调用发生的那一刻”才被调用,这个含糊会渗透进任何在默认值位置写函数调用表达式的场景。Python 选择让默认值在语义上和函数体内的其余代码彻底分开:def 语句本身在执行时求值一次并固定下来,函数体内的代码则每次调用都重新执行——这两者是两个不同时间点发生的事情,def 语句求值默认值这一次的时机和创建函数对象绑定名字是同一个动作、不可拆分。代价就是可变默认值会跨调用共享同一个对象,而好处是默认值的求值时机变得完全可预测、且只需要付出一次求值成本(比如一个默认值是耗时的初始化调用,不会在每次函数调用时都重新执行一遍)——None 哨兵模式正是利用了这个可预测性:用一个不可变、廉价的单例做默认值占位,把真正的可变对象构造挪到函数体内部、每次调用都重新执行的那部分代码里。

同理,别把 time.time()、{}、[] 之类当默认值期望”每次调用重新算”——它只算一次。

六、返回多值:其实是一个元组

C 要返回多个值得靠输出参数(指针)或包一个 struct。Python 看着能”返回多个值”,底层其实是打包成一个元组再解包:

def divmod2(a, b):
    return a // b, a % b      # 逗号建元组,等价 return (a//b, a%b)

q, r = divmod2(17, 5)         # 右边是元组,左边序列解包
print(q, r)                   # 3 2

t = divmod2(17, 5)
print(type(t), t)             # <class 'tuple'> (3, 2)

没有真正的”多返回值”,只有”返回一个元组 + 解包赋值”。所以它的成本是分配一个小元组(CPython 对小元组有缓存优化)。

七、闭包:内层函数记住外层变量

当内层函数引用了外层函数的局部变量,就形成闭包。最简单的例子:

def make_adder(n):
    def add(x):
        return x + n     # add 记住了外层的 n
    return add

add5 = make_adder(5)
print(add5(10))    # 15
print(add5(1))     # 6

make_adder(5) 早就执行完返回了,但 add5 依然”记得” n = 5——这就是闭包:内层函数携带着它用到的外层变量,而不只是一段代码。

如果内层函数要修改外层变量(而不只是读取),需要显式声明 nonlocal:

def make_counter():
    count = 0
    def inc():
        nonlocal count    # 声明:要修改的是外层的 count,不是新建本地
        count += 1
        return count
    return inc

c = make_counter()
print(c(), c(), c())          # 1 2 3

C 的局部变量在栈帧上,函数返回即失效,你没法安全返回指向它的指针。Python 里被闭包捕获的变量会被放进堆上的 cell 对象,即使外层函数已返回,变量依然存活。接着上面的 c(已经调用过三次,count 停在 3):

print(c.__closure__)          # (<cell at ...: int object at ...>,)
print(c.__closure__[0].cell_contents)   # 3 —— cell 里存着当前值

内存图:count 不在栈上,而在一个 cell 里,闭包 inc 通过 __closure__ 持有对该 cell 的引用:

   make_counter 返回后,栈帧销毁,但:

   c (function inc) --__closure__--> ( cell )
                                       |
                                       v
                                    [int 3]   <- 在堆上,存活

nonlocal 声明”我要写外层作用域的变量”;global 声明”我要写模块级全局变量”。不写这两个而直接赋值,Python 会把它当成新建本地变量(因为赋值默认创建本地绑定),从而报 UnboundLocalError——这是新手常见错误:

x = 0
def wrong():
    x += 1          # UnboundLocalError:x 被当成本地变量,但读时还没绑定
def right():
    global x
    x += 1          # OK

为什么闭包捕获的是变量本身(通过 cell 对象共享),而不是把外层变量当时的值拷贝一份存进内层函数:这取决于闭包真正要解决的问题是什么。如果闭包只是”记住定义那一刻某个值”,拷贝一份快照当然够用、实现也更简单(不需要额外的 cell 间接层)。但闭包真实存在的一大用途,恰恰是让内层函数和外层作用域共享同一个可变状态——上面的计数器 make_counter 就是最直接的例子:每次调用 inc() 都要在上一次的基础上继续累加,count 必须是内层函数和外层作用域共同读写的同一个变量,而不是内层函数自己攥着的一份独立拷贝。如果闭包捕获的是值的快照,每个 inc 调用看到的都会是创建时那个雪藏的 count,累加根本无从谈起,计数器这一类”闭包充当有状态对象”的模式就不成立了。Python 用一层间接的 cell 对象解决这个问题:count 本身不直接存在某个栈帧里,而是存在一个独立分配、在堆上、可以被多方共享引用的 cell 里,外层函数的局部变量名和内层函数的自由变量名,本质上都只是指向同一个 cell 的两个入口;函数返回、栈帧销毁,并不影响 cell 的存活,因为闭包对象的 __closure__ 还引用着它。这正是「变量与内存模型」“名字绑定而非内存槽”这套模型在跨函数场景下的延伸——只是这次共享的名字绑定关系跨越了两个不同的函数作用域。

代价体现在循环里造闭包这个经典陷阱上:闭包捕获的是 cell,不是当时的值。下面三个 lambda 共享同一个 i cell,等到真正调用时循环早已结束、i 停在 2,于是全返回 2:

funcs = [lambda: i for i in range(3)]
print([f() for f in funcs])   # [2, 2, 2] —— 三个 lambda 共享同一个 i cell,读到的都是最终值

# 修正:用默认参数,在定义那一刻就把当前 i 的值"快照"进各自的 i
funcs = [lambda i=i: i for i in range(3)]
print([f() for f in funcs])   # [0, 1, 2] —— 默认值在 def 求值时定死(见上一节的默认参数)

这个修正之所以有效,正是因为上一节讲过的”默认值在 def/lambda 执行时求值一次”——每次循环创建 lambda 时,i=i 里右边的 i 立刻被求值并存进这个新 lambda 自己的 __defaults__,不再依赖那个共享的、会继续变化的 i cell。

对照 C++:C++ 的 lambda 用 [i] 按值捕获会立刻拷贝当时的 i,而 [&i] 按引用捕获才会像 Python 这样共享。Python 闭包永远是”按引用共享 cell”,没有按值捕获语法——想要快照就得靠默认参数这个技巧。

八、调用栈:frame 对象

每调用一次函数,都得有个地方记着”这次调用的局部变量是什么、执行到第几行、返回后该回到哪里”——这就是一次函数调用的”记账单”。C 的栈帧是编译期固定布局的一块连续内存,函数返回即弹出。CPython 的调用栈由 frame 对象组成,每次调用创建一个 frame,里面有局部变量表、指向上一帧的链接等——它是堆上的对象(CPython 3.11 起对帧做了内联优化,但语义上仍可当对象访问),甚至能在运行时反射:

import sys

def inner():
    frame = sys._getframe()           # 当前帧
    print("当前函数:", frame.f_code.co_name)      # 'inner'
    print("本地变量:", frame.f_locals)             # {'frame': ...}
    print("调用者:", frame.f_back.f_code.co_name)  # 'outer' —— 上一帧

def outer():
    inner()

outer()
frame 属性 含义(对照 C)
f_locals 本地变量字典(C 里是栈上一块无名内存)
f_globals 所在模块的全局命名空间
f_back 指向调用者的帧(即调用栈的上一层)
f_code 正在执行的 code 对象
f_lineno 当前行号

这套可反射的帧结构正是 traceback、调试器、logging 能打印出完整调用链的基础。代价是比 C 栈帧重——C 的栈帧退出即失效、连结构本身都不需要在运行期被查询,CPython 的 frame 却要在整个调用期间维持一份可被外部代码读取的元数据。

递归深度限制

C 递归深到爆是栈溢出(段错误)。CPython 为防止 C 层栈真的溢出,设了一个软限制,超过就抛 RecursionError 而不是崩溃:

import sys
print(sys.getrecursionlimit())    # 通常 1000

def rec(n):
    return 1 if n == 0 else rec(n - 1) + 1

# rec(100000) 会抛 RecursionError,而不是让进程段错误
sys.setrecursionlimit(3000)       # 可调大,但调太大仍可能真的爆 C 栈

Python 没有尾调用优化,深递归请改写成迭代或显式用栈。

九、一等函数与装饰器原理

跟在 def 上面一行、以 @ 开头的写法叫装饰器。标准库自带的 functools.lru_cache,加一行就能让函数自动缓存结果:

import functools

@functools.lru_cache
def slow_square(x):
    return x * x

print(slow_square(4))   # 16
print(slow_square(4))   # 16(第二次直接从缓存拿,不重新计算)

因为函数是对象(第二节),它能被别的函数接收、包装、返回。装饰器就是这套能力的语法糖:@deco 加在 def foo 上,等价于 foo = deco(foo)——用 deco 的返回值(通常是个包装函数)替换原名字。下面自己实现一个装饰器,看清它的原理:

import functools, time

def timer(func):
    @functools.wraps(func)        # 把 func 的 __name__/__doc__ 拷到 wrapper
    def wrapper(*args, **kwargs):  # 用 *args/**kwargs 透传任意参数
        t0 = time.perf_counter()
        result = func(*args, **kwargs)   # 调用被包装的原函数
        print(f"{func.__name__} 耗时 {time.perf_counter()-t0:.6f}s")
        return result
    return wrapper                 # 返回包装函数,替换掉原来的名字

@timer
def work(n):
    return sum(range(n))

# 上面完全等价于:work = timer(work)
work(1_000_000)
print(work.__name__)   # 'work' —— 多亏 functools.wraps,否则会是 'wrapper'

底层看清楚:@timer 执行时立刻调用 timer(work),得到 wrapper 这个新函数对象,再把它绑定到名字 work。此后调用 work(...) 实际调用的是 wrapper,它内部再转调原函数(通过闭包 __closure__ 抓住的 func)。这就是”包装”——没有魔法,全是一等函数加闭包,*args/**kwargs 负责把调用点的任意参数原样透传给被包装的原函数。

functools.wraps 很重要:不加它,work.__name__ 会变成 'wrapper',traceback 和文档都会指向包装函数,给调试带来困惑。

带参数的装饰器再套一层:@repeat(3) 里的 repeat(3) 先执行,返回一个真正的装饰器,再拿它去装饰函数。所以是三层嵌套——最外层收装饰器参数,中间层收被装饰函数,最内层收调用实参:

import functools

def repeat(n):                       # 第 1 层:收装饰器参数 n
    def deco(func):                  # 第 2 层:收被装饰的函数(这才是真正的装饰器)
        @functools.wraps(func)
        def wrapper(*args, **kwargs):  # 第 3 层:收每次调用的实参
            for _ in range(n):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return deco

@repeat(3)                           # 等价于 greet = repeat(3)(greet)
def greet(name):
    print(f"Hi {name}")
    return name

greet("Ann")
# Hi Ann
# Hi Ann
# Hi Ann

看清求值顺序:@repeat(3) 这一行先调用 repeat(3) 得到 deco,再执行 deco(greet) 得到 wrapper,最后把 wrapper 绑定到 greet。参数 n 靠闭包一直被 wrapper 记着——这与第七节”闭包共享 cell”是同一套机制:n 存在 deco 创建 wrapper 时捕获的 cell 里,wrapper 每次被调用都要经这个 cell 读出当前的 n。

小结

  • 函数在 Python 里是一等对象,是”一切皆对象”延伸到函数本身的直接结果,不需要 C 那样单独的函数指针语法。
  • 传参是按对象引用/按共享:拷贝引用(指针),不拷贝对象。改形参指向不外传;原地改对象内容,可变对象才外传。
  • *args/**kwargs 是动态类型语言在编译期不知道实参形状时,处理可变参数个数的运行时收集方案,与 C++ 变参模板的编译期展开是两种类型模型下的不同必然选择。
  • 默认参数在 def 执行时求值一次并固定存入 __defaults__,这是”默认值求值时机必须可预测”这一设计选择的结果,可变默认值因此会跨调用共享,应以 None 哨兵规避。
  • 返回”多值”实为返回元组再解包。
  • 闭包捕获的是变量所在的堆上 cell,而不是值的快照——只有这样才能实现”内层函数与外层作用域共享同一可变状态”这个真实需求(如计数器闭环);循环体里直接用 lambda 捕获循环变量,拿到的是同一个 cell 的最终值,须用默认参数在定义时刻显式快照。
  • 调用栈由可反射的 frame 对象组成(f_locals/f_back),有递归软限制,无尾调用优化;装饰器 = 一等函数 + 闭包的语法糖,@deco 等价于 foo = deco(foo)。
