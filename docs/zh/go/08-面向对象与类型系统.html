类型系统:结构体、接口、方法集

Go 没有 class,没有继承,没有构造/析构函数,没有 virtual 关键字,没有访问修饰符(public/private 靠标识符首字母大小写)。它用 struct + 方法 + 组合 + 接口 组合出一套更简单、也更”结构化”的类型系统。每一层都有明确、可预测的底层表示——没有隐藏的虚表指针、没有隐藏的 this。对 C++ 程序员来说,最需要重新校准的两点是:继承换成组合,接口是隐式实现的。

一、方法:带接收者的函数

方法就是第一个参数(接收者 receiver)写在前面的普通函数。分两种:值接收者(拷贝一份操作)和指针接收者(能就地修改)。

type Point struct{ X, Y int }

func (p Point) Add(q Point) Point {   // 值接收者: p 是 Point 的拷贝
    return Point{p.X + q.X, p.Y + q.Y}
}
func (p *Point) Scale(f int) {        // 指针接收者: 能就地修改
    p.X *= f
    p.Y *= f
}

p1 := Point{1, 2}
p2 := Point{3, 4}
sum := p1.Add(p2)  // {4 6}
p1.Scale(10)       // p1 变成 {10 20}, Scale 就地修改了它

方法可以定义在任何本包内定义的具名类型上,不限于 struct:

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

底层没有魔法:p.Add(q) 等价于 Point.Add(p, q),receiver 就是普通传参——值接收者拷贝整个 Point,指针接收者拷贝一个指针。没有 C++ 那种隐藏的 this 指针语义差异,一切显式。(回顾第06章值接收者 vs 指针接收者)

二、组合:嵌入(embedding)与方法提升

Go 不用继承复用,用嵌入:在 struct 里放一个匿名字段(只写类型不写字段名)。嵌入类型的方法和字段会被”提升”到外层,看起来像是外层自己的:

type Engine struct{ Power int }
func (e Engine) Start() { fmt.Println("engine start", e.Power) }

type Car struct {
    Engine       // 嵌入(匿名字段), 不是继承!
    Brand string
}

c := Car{Engine{100}, "Toyota"}
c.Start()             // 方法提升: 等价于 c.Engine.Start()
fmt.Println(c.Power)  // 字段提升: 等价于 c.Engine.Power

为什么 Go 没有类和继承,只有 struct 组合 + interface。C++ 的经典继承体系鼓励把公共逻辑不断上提到基类,随着系统演化容易长出很深的继承树;深继承树是出了名的脆弱——子类的正确性隐含依赖基类此刻的实现细节(而不只是接口),基类为了”预留给子类扩展”经常把内部状态和方法设成 protected,子类和基类因此在实现层面紧密耦合,这就是”脆弱基类问题(fragile base class problem)“:修改基类一个看似无关的实现细节,可能悄悄破坏所有子类的行为,而这种破坏往往编译期发现不了。软件工程界”组合优于继承(favor composition over inheritance)“的建议由来已久,但在允许继承的语言里,这始终只是一条约定俗成的最佳实践,遵守与否取决于团队自律。Go 的做法更彻底:直接不提供实现继承,组合(嵌入)是唯一的代码复用机制。这不是功能缺失,而是刻意把”最佳实践”提升为”语言唯一支持的方式”——从语言设计层面杜绝了深继承树这一整类问题,代价是牺牲了继承体系里”一次定义、到处复用并可被子类覆盖任意层实现细节”的表达力。

内存布局:嵌入就是把 Engine 整个作为 Car 的一个字段内联进去,和普通字段没区别:

Car 在内存中:
┌──────────────┬───────────────┐
│ Engine.Power │ Brand(string) │   Engine 的字段就地展开在 Car 里
└──────────────┴───────────────┘

这不是 C++ 的继承(没有基类子对象的虚表指针、没有 is-a 关系)。编译器只是做了”名字转发”:c.Start() 在编译期被改写成 c.Engine.Start()。没有运行时开销,没有虚表。

维度 C++ 继承 Go 嵌入
关系 is-a(Car 是一种 Engine) has-a + 转发(Car 有个 Engine)
多态入口 virtual + vtable 靠接口(见下)
覆盖(override) 虚函数覆盖 外层定义同名方法即”遮蔽”内层
访问基类 Base::foo() c.Engine.Start() 显式路径
多重继承菱形问题 需 virtual 继承 嵌入多个类型,同名冲突需显式限定,编译器报歧义

方法遮蔽(shadowing)示例:

func (c Car) Start() { fmt.Println("car:", c.Brand); c.Engine.Start() }
// 现在 c.Start() 调外层的; 想调内层显式写 c.Engine.Start()

注意:嵌入是静态转发,不是虚派发。若 Engine.Start 内部调用另一个方法,它调的还是 Engine 自己的,不会回调到 Car 覆盖的版本——这与 C++ 虚函数的”派生类覆盖被基类调用到”根本不同。需要那种回调,得用接口。

可运行:方法提升、遮蔽与静态转发:

package main

import "fmt"

type Engine struct{ Power int }

func (e Engine) Start() { fmt.Println("engine start", e.Power) }
func (e Engine) Boot()  { e.Start() } // 永远调 Engine.Start

type Car struct {
    Engine        // 嵌入(匿名字段)
    Brand string
}

func (c Car) Start() { fmt.Println("car start", c.Brand) } // 遮蔽 Engine.Start

func main() {
    c := Car{Engine{100}, "Toyota"}
    c.Start()            // car start Toyota   (外层遮蔽了提升来的方法)
    c.Engine.Start()     // engine start 100   (显式路径调内层)
    fmt.Println(c.Power) // 100                (字段提升)
    c.Boot()             // engine start 100   (静态转发: 不是 "car start")
}

最后一行是关键:Boot 是 Engine 的方法,直接调 Engine.Start,不会派发到 Car.Start。这就是静态转发,与 C++ 虚调用相反。

三、接口:隐式实现的结构化类型

接口定义一组方法签名。只要一个类型实现了接口的全部方法,它就自动满足该接口——不需要 implements、不需要显式声明。

type Stringer interface {
    String() string
}

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

var s Stringer = Celsius(37.5)  // 自动满足, 无需声明"Celsius implements Stringer"

这叫结构化类型(structural typing) / 鸭子类型的静态版本。为什么接口设计成隐式实现,而不像 Java/C++ 那样要求显式 implements/继承抽象基类。核心原因是 Go 的一条设计箴言:接口应当由使用方定义,而不是由实现方定义(interfaces should be defined by consumers, not producers)。在显式声明的模型里,一个类型要满足某个接口,必须在定义时就知道这个接口存在并声明遵从它——这意味着接口通常得由实现方所在的包(或它们共同的上游)预先定义好,新增一个接口视角往往需要回头修改实现方代码,或者干脆没法给不受你控制的类型(比如标准库或第三方库的类型)追加一个新接口。隐式实现打破了这种耦合:任何调用方都可以在自己的包里定义一个只包含它需要的那几个方法的接口,不必得到实现方的配合、不必修改实现方代码,只要具体类型”恰好”有这些方法,匹配就自动成立。结果是接口的定义权下放到了使用方,模块之间的依赖方向也随之简化——实现方的包完全不需要 import 定义接口的包,减少了包之间的耦合与循环依赖风险。代价是编译器不能在”定义一个类型”这一刻就告诉你它实现了哪些接口,匹配关系只有在真正需要赋值给接口变量时才被编译器检查,IDE/文档也更难一眼列出”这个类型实现了哪些接口”。

好处还包括:接口可以为你无法修改的类型(甚至标准库类型)“事后”定义;解耦定义方和使用方。C++20 的 concept 有点像,但接口是运行时多态(带 vtable),更接近 C++ 的抽象基类 + 虚函数。

接口值的底层:两个字。接口值在内存里是两个机器字(第04章已提过)。分两种:

iface——非空接口(带方法的接口):

type iface struct {
    tab  *itab          // 类型信息 + 方法表(相当于 vtable)
    data unsafe.Pointer  // 指向实际数据
}

eface——空接口 any(即 interface{}):

type eface struct {
    _type *_type         // 动态类型信息
    data  unsafe.Pointer  // 指向实际数据
}

itab 是关键,它把”接口类型”和”具体类型”配对,并缓存了方法地址表:

type itab struct {
    inter *interfacetype // 接口类型
    _type *_type         // 具体类型
    hash  uint32
    fun   [1]uintptr     // 方法地址数组(变长), 相当于 vtable
}
var s Stringer = Celsius(37.5)

s(接口值, 16 字节):          itab(每个 (接口,类型) 组合一份, 全局缓存):
┌──────────┐               ┌────────────────────────┐
│ tab  ────┼──────────────▶│ inter: Stringer 类型    │
│ data ────┼───┐           │ _type: Celsius 类型     │
└──────────┘   │           │ fun[0]: &Celsius.String │◀─ 方法地址
               │           └────────────────────────┘
               ▼
        堆/栈上的 Celsius 值 37.5(装箱: 值被拷到 data 指向处)

与 C++ vtable 的关键差异:C++ 对象自己内部存一个 vptr,vtable 由类唯一确定;Go 的方法表在接口值里(itab),同一个具体类型对不同接口有不同 itab,itab 按需生成并全局缓存。调用 s.String() = 查 tab.fun[0] 得地址 → 以 data 为 receiver 调用,一次间接跳转,和虚调用开销相当。这个差异也直接对应”隐式实现”这条设计:C++ 的 vtable 必须在类定义时就确定(因为 vptr 是对象自身布局的一部分),而 Go 把”类型 T 满足接口 I”这件事的方法表挪到接口值里按需生成,天然支持”事后”为已有类型配出一个满足某接口的 itab,不需要改动类型定义本身。

装箱:把具体值塞进接口时,若值不是指针,通常要把它拷到堆上让 data 指向(逃逸,见第03章)。所以 var x any = 42 可能触发一次小分配(运行时对小整数等有缓存优化)。这点像 Java/C# 的装箱,C++ 里没有对应物。

nil 接口坑(必考)

接口值 == nil,当且仅当 tab(或 _type)和 data 两个字都为 nil。如果接口装了一个”类型非空但值为 nil 的指针”,接口本身不等于 nil:

func doErr() error {
    var p *MyError = nil   // 具体类型的 nil 指针
    return p               // 装进 error 接口: tab=*MyError, data=nil
}

func main() {
    err := doErr()
    if err != nil {        // 竟然为 true!!
        fmt.Println("非 nil:", err == nil) // 非 nil: false
    }
}

为什么会有这个坑,以及为什么这是双字表示的必然结果,而不是可以简单修掉的 bug。接口值的”是否为 nil”这个判断,定义上只看 tab/_type 这一个字,和 data 那一个字里到底装的是什么值完全无关。这不是疏忽——它是”接口值是两个独立字段”这个表示方式本身带来的逻辑推论:return p 这行代码把一个具体类型 *MyError(哪怕它的值是 nil)装进 error 接口,装箱操作要做的是”记录下这是个 *MyError,数据指针恰好是 nil”,而不是”把接口本身变成 nil”——这是两件完全不同的事:前者是”类型确定、值恰好为空”,后者是”连类型信息都没有”。只要接口用”类型指针 + 数据指针”这种显式的两字结构表示(而不是用一个单一的、类型无关的哨兵值表示”空”),这个区分就必然存在,也就必然会出现”具体类型的 nil 指针 ≠ 接口的 nil”这种情况。为什么要选这种双字表示而不是单字的”万能 nil”:两字结构让接口的类型信息和数据分离存放,itab 可以被同类型的多次装箱复用、方法调用只需一次间接跳转,调用路径不需要在运行时反复判断”这到底是不是个特殊的 nil”;要是改用单字表示、用某个哨兵值统一表达”任何类型的 nil”,接口调用和类型判断的实现都会变复杂,且无法在保持当前性能特征的前提下解决——所以这是”实现简单高效”和”少一个新手坑”之间的取舍,Go 选择了前者,把这个坑变成需要记住的规矩,而不是运行时开销。规矩:函数返回 error 时,要么返回真正的 nil(return nil),不要返回一个具体类型的 nil 指针。(第09章 error 会再强调)

可运行:

package main

import "fmt"

type MyError struct{}

func (e *MyError) Error() string { return "boom" }

func doErr() error {
    var p *MyError = nil // 具体类型的 nil 指针
    return p             // 装进 error: {type:*MyError, data:nil}
}

func returnsNil() error { return nil } // 真正的 nil 接口

func main() {
    fmt.Println(doErr() == nil)      // false  ← 坑: 接口装了非空类型
    fmt.Println(returnsNil() == nil) // true
}

类型断言与 type switch

从接口取回具体类型:

var i any = "hello"
s := i.(string)         // 断言, 失败会 panic
s, ok := i.(string)     // 逗号 ok 形式, 失败 ok=false 不 panic

底层:比较接口里的 _type/itab 与目标类型是否一致,一致就返回 data(必要时解引用/拷贝)。

type switch 按动态类型分派:

switch v := i.(type) {
case int:
    fmt.Println("int", v)     // v 是 int
case string:
    fmt.Println("string", v)  // v 是 string
case fmt.Stringer:
    fmt.Println(v.String())   // 也能匹配接口
default:
    fmt.Println("其他")
}

四、方法集:值还是指针能满足接口?

哪些方法”属于”一个类型,决定它能否满足接口。规则:

接收者定义为 T 的方法集 *T 的方法集
func (t T) ✓ ✓
func (t *T) ✗ ✓

即:指针类型 *T 拥有全部方法;值类型 T 只拥有值接收者方法。

type Speaker interface{ Speak() string }
type Dog struct{ Name string }
func (d *Dog) Speak() string { return d.Name + " says woof" } // 指针接收者

d := &Dog{Name: "Rex"}
var s Speaker = d          // OK: *Dog 满足 Speaker
fmt.Println(s.Speak())     // Rex says woof

实践建议:要么全用值接收者,要么全用指针接收者,别混。需要修改状态、结构体较大、或想让 *T 满足接口,就用指针接收者。

规则背后的原因:因为拿到 *T 能自动取值(编译器插入一次解引用),拿到 T 却不一定能取到稳定地址(比如 map 元素、函数返回的临时值)去调指针方法——若允许 T 的方法集包含指针方法,Dog{}.Speak() 这类没有可取地址左值的调用就需要编译器偷偷造一个临时变量再取地址,这个临时变量在方法返回后立刻失效,若方法内部试图保存这个指针供之后使用就会出错。Go 选择在方法集层面直接禁止这种情况,而不是允许它但留下一个随时可能出错的隐患。

type Speaker interface{ Speak() }
type Dog struct{}
func (d *Dog) Speak() {}      // 指针接收者

var s Speaker = &Dog{}        // OK
// var s Speaker = Dog{}      // 编译错误: Dog 未实现 Speaker(Speak 在 *Dog 上)

方法值 vs 方法表达式

d := Dog{}
f := d.Speak            // 方法值: 绑定了 receiver d, 类型 func()
f()                     // 等价 d.Speak()

g := (*Dog).Speak       // 方法表达式: receiver 变成显式第一个参数
g(&d)                   // 等价 d.Speak()

方法值会捕获 receiver(可能导致其逃逸到堆,像闭包);方法表达式只是把方法当成普通函数,receiver 显式传。

五、泛型(Go 1.18+):类型参数与约束

Go 长期没有泛型,1.18 引入类型参数。语法:方括号声明类型参数,配 约束(constraint),写一次函数就能对多种类型工作:

func Sum[T int | float64](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

Sum([]int{1, 2, 3})        // 6, 类型实参 int 自动推导
Sum([]float64{1.5, 2.5})   // 4

约束还可以先定义成一个具名接口,列出允许的类型集合:

// 约束: 一个接口, 列出允许的类型集合
type Number interface {
    ~int | ~int64 | ~float64   // ~T 表示"底层类型是 T 的所有类型"
}

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

comparable 是内建约束(可用 ==/!= 的类型),常用于 map 键类泛型:

func Keys[K comparable, V any](m map[K]V) []K { /* ... */ }

约束的本质:它就是一个接口,既能列方法(要求实现某方法),也能列类型集合(| ~T)。

可运行,展示 ~ 的意义(它接纳底层类型匹配的具名类型):

package main

import "fmt"

type Number interface {
    ~int | ~int64 | ~float64
}

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

type Celsius float64 // 底层类型是 float64, 所以 ~float64 接纳它

func main() {
    fmt.Println(Sum([]int{1, 2, 3}))      // 6   (T 推导为 int)
    fmt.Println(Sum([]float64{1.5, 2.5})) // 4
    fmt.Println(Sum([]Celsius{1.5, 2.5})) // 4   (~float64 接纳具名类型 Celsius)
}

底层实现:GC shape stenciling + 字典

这是和 C++ 模板差异最大的地方。C++ 模板是完全单态化(monomorphization):vector<int> 和 vector<string> 生成两份完全独立的机器码,零运行时开销但代码膨胀、编译慢。Go 走了折中路线:

按 “GC shape” 分组生成代码 + 传字典(dictionary)。 具体:

  • 编译器按 GC shape 分组生成代码。当前实现的分组规则是:两个类型同属一个 shape,当且仅当它们底层类型相同,或都是指针。所以所有指针类型(*int、*string……)共享一份实例;而 int 与 int64 底层类型不同,即便大小都是 8 字节、都不含指针,也各自生成一份(不共享)。
  • 每次调用额外传一个隐藏的字典参数,里面放该具体类型的元信息(类型描述符、需要的 itab 等),供运行时查具体大小、方法地址。
C++ 模板(单态化):          Go 泛型(shape stenciling):
Sum<int>    → 代码副本A     Sum[int]     → int 的 shape 代码 + int 字典
Sum<int64>  → 代码副本B     Sum[int64]   → int64 的 shape 代码 + int64 字典(与 int 不共享)
Sum<*T>×N   → N 份副本      Sum[*T]×N    → 1 份指针 shape 代码 + 各自字典

为什么不像 C++ 一样直接单态化:Go 团队在设计泛型时明确担心重蹈 C++ 模板的覆辙——单态化编译慢、二进制膨胀,尤其在 Go 强调的快速编译、单一静态二进制这两个目标下代价过高。为什么也不像 Java 一样纯类型擦除:类型擦除意味着所有泛型实参都要装箱成接口,每次访问都有一次间接跳转和潜在分配,运行时代价更高。GC shape stenciling 是两者之间的折中:同 shape 的类型共享代码,避免了单态化的膨胀;但共享的是机器码而非类型信息本身,通过字典在运行时补齐类型细节,避免了类型擦除必须无处不装箱的代价。结论:介于单态化与装箱之间——比 Java 的类型擦除+装箱快(无处不装箱),比 C++ 单态化省代码;但字典查找、经指针间接访问会带来一点运行时开销,某些场景不如手写具体类型或不如 C++ 模板极致。

C++ 模板 Go 泛型 Java 泛型
机制 单态化 shape stenciling + 字典 类型擦除 + 装箱
代码体积 大(每类型一份) 中(每 shape 一份) 小(一份)
运行时开销 无 少量(字典/间接) 装箱开销大
编译期计算 强(模板元编程) 弱(只是参数化) 无

不要为了泛型而泛型。适合:容器/算法(slices、maps 包,见第11章)、对多种数值类型做同一操作。很多场景一个 any + 类型断言或普通接口反而更清晰。

小结

  • 用法层面:func (recv T) Name(...) 定义方法,值接收者拷贝、指针接收者能就地修改;嵌入(匿名字段)让字段/方法被”提升”到外层;接口只要方法签名匹配就自动满足,不用显式声明实现;T/*T 方法集不同——指针接收者的方法只属于 *T;泛型用 [T 约束] 声明类型参数,一份代码支持多种类型。
  • 设计取舍:没有类和继承、只有组合+接口,是把”组合优于继承”从最佳实践升级为语言强制,代价是牺牲了深继承体系的表达力,换来的是杜绝脆弱基类问题;接口隐式实现让接口定义权下放给使用方而非实现方,降低模块耦合,代价是编译器无法在类型定义处直接列出它实现了哪些接口;接口值是两个机器字(itab/_type + data),这个双字表示本身导致了”具体类型的 nil 指针≠接口 nil”这个必然的坑,换来的是免装箱开销的高效方法分派;方法集规则(*T 全有、T 只有值接收者方法)源于值类型不一定有可取地址的左值这一事实;泛型不是单态化也不是纯装箱,而是按 GC shape 分组生成代码 + 传字典,是”编译膨胀”与”装箱开销”之间的折中。
