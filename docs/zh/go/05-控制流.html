控制流

本章先带你把 for、if、switch、defer 这些控制流关键字怎么写过一遍——零基础也能看懂,照着写就能跑;然后再进入底层——range、defer 背后的拷贝语义和求值时机,这是 C/C++ 老手最容易踩坑的地方,也是本章后半段的重点。只想会用,读完第一节就够;想搞懂”为什么这段代码打印的顺序这么怪”,继续往下看。

先会用:循环、分支、defer 速览

Go 只有一个循环关键字 for,一个分支关键字 if/switch,外加一个很实用的 defer。先看一个把它们全用上的小例子:

package main

import "fmt"

func main() {
    // for:Go 唯一的循环关键字
    for i := 0; i < 3; i++ {
        fmt.Println("循环", i)
    }

    // range:遍历切片、数组、字符串、map
    fruits := []string{"apple", "banana", "cherry"}
    for idx, name := range fruits {
        fmt.Println(idx, name)
    }

    // if / else
    score := 85
    if score >= 60 {
        fmt.Println("及格")
    } else {
        fmt.Println("不及格")
    }

    // switch:按值分支
    switch score / 10 {
    case 10, 9:
        fmt.Println("优秀")
    case 8:
        fmt.Println("良好")
    default:
        fmt.Println("一般")
    }
}

运行结果:

循环 0
循环 1
循环 2
0 apple
1 banana
2 cherry
及格
良好

再认识 defer——它把一句调用推迟到当前函数返回前才执行,常用来做”用完就关闭/清理”这类收尾工作:

package main

import "fmt"

func doWork() {
    fmt.Println("开始工作")
    defer fmt.Println("工作结束,清理一下") // 推迟到 doWork 返回前执行
    fmt.Println("工作中...")
}

func main() {
    doWork()
}

运行结果:

开始工作
工作中...
工作结束,清理一下

注意”清理一下”这句虽然写在中间,却是最后打印的——这就是 defer 的效果。这些足够你写出正常的循环、分支、收尾逻辑了。下面开始进入底层:Go 的控制流刻意做得”少而正”,for 吃掉了 C 里 while/do...while/复杂 for 变体的所有场景,switch 也彻底翻转了默认行为,而 range、defer 的拷贝与求值时机,是本章真正的重点。

开始深入

Go 的控制流刻意做得”少而正”。C/C++ 里的 while、do...while、逗号运算符、复杂的 for(;;) 变体在这里被压缩成唯一一个 for 关键字;switch 也彻底翻转了默认行为。本章接下来从底层视角讲清楚每个构造在编译后变成了什么,以及 range、defer 背后的拷贝语义——这两个是 C/C++ 老手最容易踩坑的地方。

for:唯一的循环关键字

Go 没有 while,也没有 do...while。所有循环都用 for,它有三种形态,编译后本质上都是同一套跳转指令(条件判断 + 向后跳转),和 C 的循环在机器码层面没有区别。

package main

import "fmt"

func main() {
    // 形态一:完整三段式,和 C 的 for(i=0;i<n;i++) 一致
    for i := 0; i < 3; i++ {
        fmt.Println("三段式", i)
    }

    // 形态二:只有条件,等价于 C 的 while(cond)
    n := 3
    for n > 0 {
        fmt.Println("类 while", n)
        n--
    }

    // 形态三:无条件,等价于 C 的 for(;;) 或 while(1)
    count := 0
    for {
        if count == 2 {
            break // 必须自己 break,否则死循环
        }
        fmt.Println("无限循环", count)
        count++
    }
}

三段式里的 i := 0 声明的 i 作用域仅限于该 for 语句(包括循环体),这点和 C99 的 for(int i...) 一致。它通常分配在栈上;若循环体内有闭包捕获它(见 06-函数与调用栈),逃逸分析会把它挪到堆上。

do...while 的语义(至少执行一次)在 Go 里用 for + break 手写:

for {
    // 循环体至少执行一次
    if !condition {
        break
    }
}

range:遍历的语义与拷贝行为

range 是最需要盯住内存的构造。核心事实:range 迭代变量是被遍历元素的副本,每轮循环把当前元素拷贝一份赋给迭代变量。这与 C++ 里 for (auto x : v)(值拷贝)一致,而不是 for (auto& x : v)(引用)。

range slice / array

s := []int{10, 20, 30}
for i, v := range s {
    // i 是下标,v 是 s[i] 的【拷贝】
    // 修改 v 不影响 s;要改原数组必须用 s[i] = ...
    v += 100
    _ = v
    _ = i
}
// s 仍是 {10, 20, 30}

内存图(遍历第 2 轮时):

底层数组(堆或栈):
  s[0]=10  s[1]=20  s[2]=30
                ↑
         range 读到 s[1]=20
                │  拷贝
                ▼
   迭代变量 v = 20   ← 独立的栈变量,改它不动数组

关键陷阱一:range 表达式在循环开始前只求值一次。 对 slice 而言,range 会拷贝 slice 的表头(SliceHeader:指针+长度+容量),而不是底层数组。所以循环中 append 导致的扩容不会改变本次遍历的次数,但若不扩容而原地改元素,会被读到(因为指针还指向同一块数组):

s := []int{1, 2, 3}
for i, v := range s {
    if i == 0 {
        s[2] = 99 // 未扩容,原地改;第 3 轮会读到 99
        s = append(s, 4) // 遍历长度已锁定为 3,不会遍历到 4
    }
    fmt.Println(i, v)
}
// 输出:0 1 / 1 2 / 2 99

对数组(非 slice)要区分取不取元素值:for i, v := range arr 会先把整个数组拷贝一份再遍历(v 取自那个副本),大数组代价很大;但 for i := range arr 只用下标、不读元素值时,编译器并不会拷贝数组。想在取值时也避免拷贝,就对数组指针做 range:

var arr [1024]int
for i := range arr {      // 只用下标:不拷贝这 8KB
    arr[i] = i
}
for i, v := range &arr {  // 需要取值又想省拷贝:对 &arr range,v 直接读自原数组
    arr[i] = v + i
}

range map

m := map[string]int{"a": 1, "b": 2}
for k, v := range m {
    // k、v 都是拷贝;遍历顺序【随机】(runtime 故意随机化起始桶)
    _ = k
    _ = v
}

Go 的 map 遍历顺序是运行时随机的——这是故意设计,防止代码依赖顺序。C++ 的 std::map 有序、unordered_map 顺序不定但同一进程内稳定;Go 连稳定都不给你。遍历中删除当前或未遍历的键是安全的;新增的键可能被遍历到也可能不,行为未定义。

range string:按 rune 而非 byte

C 里字符串就是 char*,一次一个字节。Go 的 range string 按 UTF-8 解码,一次给你一个 rune(Unicode 码点,int32),下标是该 rune 起始字节的偏移:

s := "aé中" // UTF-8: 'a'=1字节, 'é'=2字节, '中'=3字节
for i, r := range s {
    fmt.Printf("字节偏移=%d rune=%c 码点=%d\n", i, r, r)
}
// 字节偏移=0 rune=a 码点=97
// 字节偏移=1 rune=é 码点=233
// 字节偏移=3 rune=中 码点=20013

注意 i 跳过了 2(因为 é 占 2 字节)。若想按字节遍历,用普通下标 for i := 0; i < len(s); i++ { s[i] },s[i] 是 byte。遇到非法 UTF-8 序列,range 会返回 RuneError(U+FFFD)并前进 1 字节。

range channel

ch := make(chan int)
// ... 另一 goroutine 往 ch 发数据,发完 close(ch)
for v := range ch {
    // 每次从 ch 接收一个值(拷贝),直到 ch 被 close 且缓冲耗尽
    _ = v
}

range ch 等价于反复 v, ok := <-ch,直到 ok == false(channel 关闭且空)才退出。详见 10-并发模型。

range int(Go 1.22+)

Go 1.22 起可以直接 for i := range n,等价于 for i := 0; i < n; i++:

for i := range 5 { // i 从 0 到 4
    fmt.Println(i)
}

Go 1.22:循环变量每轮新实例

这是一个语言语义变更,C/C++ 里没有对应物。Go 1.22 之前,for i := 0; ... 和 for k, v := range 中的循环变量在整个循环中只有一个实例,每轮复用同一块内存地址。这导致闭包/goroutine 捕获时全都指向同一个变量(经典坑):

// Go 1.21 及之前:全部打印 3(共享同一个 i);Go 1.22+:打印 0 1 2
funcs := []func(){}
for i := 0; i < 3; i++ {
    funcs = append(funcs, func() { fmt.Println(i) })
}
for _, f := range funcs {
    f()
}

Go 1.22 起,每轮迭代的循环变量都是一个新实例(逻辑上每轮重新声明),闭包各自捕获到不同的变量,输出变为 0 1 2。这解决了历史上最常见的 bug 之一。捕获细节见 06-函数与调用栈。

if 带初始化子句

if 可以先执行一条简单语句再判断,该语句声明的变量作用域仅限 if/else 块:

if v, ok := m["key"]; ok {
    // v、ok 只在这里可见
    fmt.Println(v)
} else {
    // else 里也能看到 v、ok
    fmt.Println("缺失")
}
// 这里 v、ok 已不可见

这把变量的生命周期收得很紧,底层上 v、ok 是 if 语句块内的栈变量。惯用于错误处理:if err := doSomething(); err != nil { ... }。

switch:无自动 fallthrough

C 里 switch 的 case 会自动贯穿(fallthrough),忘记 break 是经典 bug。Go 反过来:每个 case 默认自动 break,不贯穿。

switch x {
case 1:
    fmt.Println("一") // 执行完自动跳出,不会掉进 case 2
case 2, 3: // 一个 case 可以匹配多个值(逗号)
    fmt.Println("二或三")
default:
    fmt.Println("其他")
}

fallthrough 关键字

若确实想贯穿,用显式 fallthrough(它无条件进入下一个 case 体,且不检查下一个 case 的条件):

switch x {
case 1:
    fmt.Println("一")
    fallthrough // 强制继续执行 case 2 的体
case 2:
    fmt.Println("二") // x==1 时这里也会执行
}

无表达式的 switch = if-else 链

switch 后不写表达式,等价于 switch true,每个 case 写布尔条件,当作更清爽的 if-else 链:

switch {
case score >= 90:
    grade = "A"
case score >= 60:
    grade = "B"
default:
    grade = "F"
}

switch 带初始化子句

和 if 一样可带初始化:switch v := f(); v { ... }。

type switch:运行时类型分派

对接口值按动态类型分派,底层是读取接口的类型指针(见 08-面向对象与类型系统)并比较:

func describe(i interface{}) {
    switch v := i.(type) { // v 在每个 case 里是对应的具体类型
    case int:
        fmt.Println("int:", v*2) // v 是 int
    case string:
        fmt.Println("string len:", len(v)) // v 是 string
    case nil:
        fmt.Println("nil")
    default:
        fmt.Printf("未知类型 %T\n", v)
    }
}

这类似 C++ 的 dynamic_cast 链或 std::variant + std::visit,但由运行时的接口类型信息直接支持,无需 RTTI 开销那样的虚表遍历。

带标签的 break / continue

嵌套循环里想跳出外层,C 用 goto 或标志变量。Go 提供带标签的 break/continue,标签放在循环前:

outer:
    for i := 0; i < 3; i++ {
        for j := 0; j < 3; j++ {
            if i*j > 2 {
                break outer // 直接跳出外层 for,而非只跳内层
            }
            if j == 1 {
                continue outer // 外层进入下一轮 i
            }
            fmt.Println(i, j)
        }
    }

break 还能用在 switch/select 上,带标签可跳出被标签标记的那一层。

goto

Go 保留了 goto,但限制:不能跳过变量声明,不能跳进块内部。主要用于跳出深层嵌套或错误清理(在没有 defer 的场景):

    i := 0
loop:
    if i < 3 {
        fmt.Println(i)
        i++
        goto loop // 向后跳
    }

实务中几乎总有更好的写法(for、defer),goto 很少用。

⭐ defer:延迟执行

defer 把一个函数调用推迟到当前函数返回前执行,是 Go 管理资源(关文件、解锁、释放)的核心机制,替代了 C++ 的 RAII/析构函数。第一节已经见过它最基本的用法;这里深入它必须抓住的三个底层事实。

事实一:LIFO 顺序

同一函数里多个 defer 按后进先出(栈)执行。运行时把每个 defer 记录压入当前 goroutine 的 defer 链表(_defer 结构),函数返回时从链表头依次弹出:

func main() {
    defer fmt.Println("1") // 最后执行
    defer fmt.Println("2")
    defer fmt.Println("3") // 最先执行
    fmt.Println("函数体")
}
// 输出:函数体 / 3 / 2 / 1

内存示意(defer 链表,新的插在头部):

注册顺序: defer1 → defer2 → defer3
链表:  head → [defer3] → [defer2] → [defer1] → nil
返回时从 head 弹出执行: 3, 2, 1

事实二:注册时即求值参数

这是最容易错的点。defer f(x) 在执行到 defer 语句那一刻就把 x 的当前值拷贝进 defer 记录,而不是等到真正调用时才取值:

func main() {
    i := 0
    defer fmt.Println("defer 看到的 i =", i) // 此刻 i=0 被拷贝进记录
    i = 100
    fmt.Println("函数体结束时 i =", i) // 100
}
// 输出:函数体结束时 i = 100 / defer 看到的 i = 0

延迟的是调用的执行,不是参数的求值。若想延迟到调用时才读最新值,把逻辑包进闭包(闭包捕获变量,见事实四):

defer func() { fmt.Println(i) }() // 调用时才读 i,输出 100

把 LIFO 和”注册时求值”合起来看,循环里的 defer 最直观:

package main

import "fmt"

func main() {
    for i := 0; i < 3; i++ {
        defer fmt.Println("defer:", i) // 每轮把当前 i 拷进记录
    }
    fmt.Println("函数体")
}
// 输出:
// 函数体
// defer: 2
// defer: 1
// defer: 0

三个 defer 各自记住注册那一刻的 i(0、1、2),函数返回时按 LIFO 逆序执行——于是打印 2、1、0。

事实三:与命名返回值 / return 的交互

return x 在底层是两步:先把返回值写入返回槽,再执行 defer,最后真正返回。若函数用了命名返回值,defer 能在第二步里修改这个返回槽,从而改变最终返回值:

func inc() (result int) { // 命名返回值 result
    defer func() {
        result++ // defer 修改返回槽,最终返回值被改
    }()
    return 10 // 先把 result 置为 10,再跑 defer,result 变 11
}
// inc() 返回 11

对比匿名返回值:return 10 时值已拷贝到临时返回位置,defer 里改局部变量影响不到它。这个机制常用于在 defer 里统一包装 error(见 09-错误处理与资源管理)。

事实四:循环中 defer 的坑

defer 绑定的是函数返回,不是块/循环结束。在循环里 defer 会累积到函数结束才一起执行,可能耗尽资源:

// 反例:1000 个文件句柄会一直开到函数结束
func bad(files []string) {
    for _, name := range files {
        f, _ := os.Open(name)
        defer f.Close() // 全部堆积,循环期间不关闭!
        // ...
    }
} // 到这里才一次性关 1000 个

// 正例:把循环体抽成函数,defer 随该函数返回而触发
func good(files []string) {
    for _, name := range files {
        func() {
            f, _ := os.Open(name)
            defer f.Close() // 每轮迭代结束即关闭
            // ...
        }()
    }
}

defer 与 panic/recover

defer 还是捕获 panic 的唯一时机:只有在 defer 的函数里调用 recover() 才能拦截 panic、恢复正常流程。这套”异常”机制与 C++ 的 try/catch、栈展开(stack unwinding)有本质区别——留到 09-错误处理与资源管理 详解。

小结

  • 用法层面:循环只用 for(三种形态覆盖 while/do-while/无限循环);range 遍历切片/数组/字符串/map/channel;if/switch 可带初始化子句;switch 默认不贯穿,一个 case 可写多个值;defer 一句话推迟到函数返回前执行,常用来关闭资源。
  • 底层层面:range 迭代变量是副本;range 表达式在循环开始前只求值一次(slice 拷贝表头,数组按是否取值决定拷不拷贝整个数组);字符串 range 按 rune 解码;Go 1.22 起循环变量每轮新实例,修复了闭包捕获的历史坑;defer 按 LIFO 执行,参数在注册时求值,能通过命名返回值修改最终返回结果。
