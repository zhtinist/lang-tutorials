函数与调用栈(⭐ 参数传递的真相)

本章先带你把函数怎么定义、怎么调用、怎么写多返回值、怎么写方法和闭包过一遍——零基础也能看懂;然后进入这本教程最重要的底层内容:Go 的函数在语法上像 C,但在内存语义上有几个必须彻底钉死的事实——一切都是按值传递、方法接收者也是值或指针的拷贝、闭包捕获的是变量而非值、goroutine 栈是可增长的。C/C++ 老手若把这些当成”和 C 一样”就会持续踩坑。只想会写函数,读完第一节就够;想搞懂参数传递和调用栈的真相,继续往下看。

先会用:函数、方法、闭包速览

定义并调用一个函数,写法和大多数语言差不多——只是返回类型写在参数列表后面:

package main

import "fmt"

func add(a, b int) int { // 定义一个函数:两个 int 参数,返回一个 int
    return a + b
}

func divmod(a, b int) (int, int) { // 可以一次返回多个值
    return a / b, a % b
}

func main() {
    fmt.Println(add(3, 4)) // 调用函数:7

    q, r := divmod(17, 5) // 一次接收两个返回值
    fmt.Println(q, r)     // 3 2
}

方法就是”绑定到某个类型”的函数,多了一个写在函数名前面的”接收者”:

package main

import "fmt"

type Counter struct {
    n int
}

func (c *Counter) Inc() { // 方法:c 是接收者,类似"这个方法属于 *Counter"
    c.n++
}

func main() {
    c := Counter{}
    c.Inc()
    c.Inc()
    fmt.Println(c.n) // 2
}

闭包是一个”记得住外部变量”的匿名函数:

package main

import "fmt"

func makeAdder(base int) func(int) int {
    return func(x int) int { // 这个匿名函数记住了外面的 base
        return base + x
    }
}

func main() {
    add10 := makeAdder(10)
    fmt.Println(add10(5)) // 15
}

这些足够你写出正常的函数、方法、闭包了。下面开始进入底层——参数传递、接收者、闭包捕获,在内存里到底发生了什么,这是本章、也是全书最重要的内容。

开始深入

下面从”参数到底是怎么传的”这个最根本的问题开始,逐条钉死函数、方法、闭包在内存里的真实行为。

函数是一等公民

函数在 Go 里是值:可以赋给变量、作为参数、作为返回值、放进 slice/map。底层上,一个函数值是一个指向 funcval 结构的指针(内含代码地址,闭包时还含捕获的变量),因此函数变量本身只是一个指针大小的字长。

package main

import "fmt"

func add(a, b int) int { return a + b }

func main() {
    var op func(int, int) int = add // 函数赋给变量
    fmt.Println(op(3, 4))            // 7

    apply := func(f func(int, int) int, x, y int) int {
        return f(x, y) // 函数作为参数
    }
    fmt.Println(apply(add, 10, 20)) // 30
}

这类似 C 的函数指针,但 Go 的函数值可以携带状态(闭包),更接近 C++ 的 std::function / lambda——只是没有那层类型擦除的堆分配开销(除非闭包变量逃逸)。

多返回值:惯用 (result, error)

第一节已经用过多返回值。底层实现:自 Go 1.17 的寄存器 ABI,返回值优先通过寄存器传递(超过可用寄存器数量的部分才落到栈上),被调函数直接写入,无需像 C 那样用输出参数(指针)或打包结构体。

Go 最重要的惯用法是 (result, error)——最后一个返回值是 error,调用方立刻检查:

func parse(s string) (int, error) {
    // ...
    return 0, fmt.Errorf("解析失败: %q", s)
}

func main() {
    n, err := parse("abc")
    if err != nil {
        fmt.Println("出错:", err)
        return
    }
    fmt.Println(n)
}

这取代了 C 的 errno / 返回码约定,也取代了 C++ 的异常作为常规错误路径。详见 09-错误处理与资源管理。

⭐ 全部按值传递

Go 里没有引用传递。 每次函数调用,所有实参都被拷贝一份到被调函数的栈帧。要在函数里改动调用方的变量,必须显式传指针(拷贝的是指针值,但它指向同一对象)。

func incValue(x int)  { x++ }      // 改的是拷贝,调用方看不到
func incPtr(x *int)   { *x++ }     // 通过指针改到原变量

func main() {
    a := 10
    incValue(a)
    fmt.Println(a) // 仍是 10

    incPtr(&a)
    fmt.Println(a) // 11
}

内存图:

main 的栈帧:            incValue 的栈帧:
  a = 10  ──拷贝──────▶   x = 10   (x++ 只改这里,返回后销毁)

  a = 11  ◀──*x++──┐     incPtr 的栈帧:
        │          └──   x = &a   (x 是指向 a 的指针,拷贝的是地址)
        └── a 被改到

这条规则对所有类型统一成立,包括 struct、数组。传一个大 struct 会整块拷贝:

type Big struct{ data [1024]int } // 8KB

func mutate(b Big)  { b.data[0] = 1 } // 拷贝 8KB,改动无效
func mutateP(b *Big) { b.data[0] = 1 } // 拷贝 8 字节指针,改动有效

slice / map / channel 是”含指针的表头”

初学者常误以为 slice/map “按引用传递”。真相是:它们也是按值拷贝,只不过它们的值是一个含指针的小表头。拷贝表头 = 复制指针,于是两份表头指向同一底层数据,产生别名(aliasing)效果:

func fill(s []int) {
    s[0] = 99 // 通过表头里的指针改到共享的底层数组 —— 调用方可见
    s = append(s, 7) // append 可能换底层数组;即便不换,这是给【局部表头】重新赋值,调用方的表头长度不变
}

func main() {
    s := []int{1, 2, 3}
    fill(s)
    fmt.Println(s) // [99 2 3] —— s[0] 改了,但长度仍是 3
}

内存图(slice 表头拷贝):

main 的 s (表头):          fill 的 s (拷贝的表头):
  ptr ──┐                    ptr ──┐   ← 两个 ptr 相等,指向同一数组
  len=3 │                    len=3 │
  cap=3 │                    cap=3 │
        └──────┬─────────────┘
               ▼
    底层数组(堆): [99, 2, 3]   ← s[0]=99 两边都看得到
类型 传递时拷贝的是什么 修改元素调用方可见?
int/struct/array 整个值(可能很大) 否
pointer 指针(1 字长) 是(通过解引用)
slice 表头(ptr+len+cap,3 字长) 改元素:是;append/重切:否
map 表头指针(1 字长) 是
channel 表头指针(1 字长) 是(收发)
string 表头(ptr+len,只读) string 不可变

理解 map/slice/channel 的详细内存布局见 07-复合数据结构。

方法:值接收者 vs 指针接收者

方法只是第一个参数(接收者)有特殊语法的函数。接收者同样遵循按值传递规则。第一节的 Inc 用的是指针接收者;下面把值接收者和指针接收者并排对比:

type Counter struct{ n int }

// 值接收者:c 是整个 Counter 的【拷贝】,改动不影响原对象
func (c Counter) IncValue() { c.n++ }

// 指针接收者:c 是指针的拷贝,通过它能改到原对象
func (c *Counter) IncPtr() { c.n++ }

func main() {
    var c Counter
    c.IncValue()
    fmt.Println(c.n) // 0 —— 改的是拷贝

    c.IncPtr()
    fmt.Println(c.n) // 1 —— 改到原对象
}

选择接收者类型的两条底层理由:

  1. 要修改接收者 → 必须用指针接收者(否则改的是拷贝)。
  2. 接收者是大 struct → 用指针避免每次调用整块拷贝(值接收者会把整个 struct 压栈)。

小的、不可变语义的类型(如表示坐标的 Point{x,y int})用值接收者更简单、更缓存友好。惯例:同一类型的方法要么全用值、要么全用指针,不要混。

可寻址性与自动取地址

c.IncPtr() 里 c 是值,但编译器自动改写成 (&c).IncPtr()——前提是 c 可寻址(addressable)。局部变量、slice 元素、可解引用的指针目标都可寻址;而 map 元素、临时值(函数返回值)不可寻址,不能对它们调用指针接收者方法:

type Counter struct{ n int }
func (c *Counter) IncPtr() { c.n++ }

m := map[string]Counter{"a": {}}
// m["a"].IncPtr() // 编译错误:m["a"] 不可寻址,无法取地址

方法集

一个类型的方法集决定它能满足哪些接口:

  • 类型 T 的方法集:所有值接收者方法。
  • 类型 *T 的方法集:值接收者 + 指针接收者方法(全部)。

也就是说,指针接收者方法不属于 T 的方法集。因此一个值类型 T 若想满足某接口,而该接口方法用了指针接收者,则只有 *T 满足接口,T 不满足。这条规则与可寻址性配合,是接口赋值报错的常见根源,详见 08-面向对象与类型系统。

闭包捕获的是变量,不是值

第一节的 makeAdder 已经展示了闭包的基本用法。闭包(匿名函数)捕获外层变量时,捕获的是变量本身(其存储位置),不是当时的值。若被捕获的变量在函数返回后仍被闭包引用,逃逸分析会把它从栈搬到堆,让闭包和外层共享同一份存储。

func counter() func() int {
    n := 0 // 逃逸到堆,因为返回的闭包持有它
    return func() int {
        n++ // 每次调用共享并修改同一个 n
        return n
    }
}

func main() {
    c := counter()
    fmt.Println(c(), c(), c()) // 1 2 3
}

这与 C++ lambda 的 [&n](按引用捕获)语义相同,而非 [n](按值)。C 没有闭包,只能靠传结构体指针模拟。

“捕获的是变量”最有说服力的证据,是让两个闭包捕获同一个变量——一个写、一个读,它们看到的是同一份存储:

package main

import "fmt"

func newAccount() (deposit func(int), balance func() int) {
    bal := 0
    deposit = func(n int) { bal += n }
    balance = func() int { return bal }
    return
}

func main() {
    dep, bal := newAccount()
    dep(100)
    dep(50)
    fmt.Println(bal()) // 150,两个闭包共享同一个 bal
}

bal 逃逸到堆,deposit 和 balance 都持有指向它的引用,于是形成了一份被封装的私有状态(闭包 = 数据 + 行为)——这正是 C 里要靠”结构体 + 一组操作它的函数指针”才能拼出来的东西。

经典循环变量坑与 Go 1.22 语义变化

因为捕获的是变量,在 Go 1.21 及以前,循环变量整个循环只有一个实例,所有闭包捕获到同一个变量:

// Go 1.21-:输出 3 3 3(共享同一个 i,循环结束时 i=3)
// Go 1.22+:输出 0 1 2(每轮 i 是新实例,各闭包捕获不同变量)
func main() {
    funcs := make([]func(), 0, 3)
    for i := 0; i < 3; i++ {
        funcs = append(funcs, func() { fmt.Print(i, " ") })
    }
    for _, f := range funcs {
        f()
    }
}

Go 1.22 起,for 的循环变量每轮迭代都是新实例(见 05-控制流),这个几乎人人踩过的坑被语言层面修掉。在 1.21 及以前,老写法要靠每轮 i := i 显式创建副本来规避。用 goroutine 时同理(go func(){ ... i ... }()),1.22 前后行为不同。

变参 …T

...T 声明变参,函数内部把它当作 []T(slice)使用。底层:调用方把这些实参打包成一个新 slice(通常在栈上分配底层数组,不逃逸时)传入。

func sum(nums ...int) int { // nums 的类型是 []int
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

func main() {
    fmt.Println(sum(1, 2, 3)) // 传 3 个 → 打包成 []int{1,2,3}
    fmt.Println(sum())        // 传 0 个 → nums 是 nil slice,len=0

    xs := []int{4, 5, 6}
    fmt.Println(sum(xs...)) // 用 ... 把已有 slice 展开;此时【不拷贝】,直接复用 xs 的底层数组
}

注意 sum(xs...) 复用 xs 的底层数组,若函数内修改 nums[i] 会改到 xs。这与 C 的 va_list(无类型、无长度、需哨兵)相比,类型安全且自带长度。

命名返回值 + defer 修改返回值

结合 05-控制流 的 defer,命名返回值是一个强大组合。return 先把值写入命名的返回槽,再跑 defer,defer 可改这个槽:

func readConfig() (cfg string, err error) {
    defer func() {
        if r := recover(); r != nil {
            // 把 panic 转成 error 返回给调用者
            err = fmt.Errorf("恢复自 panic: %v", r)
        }
    }()
    panic("配置文件损坏")
    // 到达不了,但 defer 会把 err 设好后正常返回
}

底层上命名返回值就是被调函数栈帧里的具名变量,return x = “把 x 拷进返回槽 + 执行 defer + 真正返回”。这是把 panic 转换为 error 的标准手法,详见 09-错误处理与资源管理。

goroutine 栈:可增长的连续栈

这是 Go 运行时最漂亮的设计之一,和 C/C++ 线程栈截然不同。

C/C++ 的线程栈在创建时固定大小(典型 1–8MB),由 OS 预留虚拟地址空间。开太多线程会吃光地址空间;递归太深会栈溢出(segfault)。

Go 的每个 goroutine 有自己的栈,初始只有约 2KB,由 Go 运行时管理(不是 OS)。因此可以轻松创建几十万个 goroutine——2KB × 100 万 ≈ 2GB,而线程做不到。

栈如何增长:morestack 与栈拷贝

编译器在每个函数入口插入一小段栈检查(stack guard)代码:检查当前栈指针是否快触底。若空间不足,调用运行时的 morestack:

  1. 分配一块更大的新栈(通常翻倍)。
  2. 把旧栈的全部内容拷贝到新栈(连续栈,contiguous stack)。
  3. 调整所有指向旧栈的指针(运行时能精确扫描栈上的指针,这依赖 Go 的精确 GC 类型信息)——这一步 C 做不到,因为 C 无法知道栈上哪些是指针。
  4. 释放旧栈,函数继续执行。

栈也会在 GC 时按需收缩。

增长前(2KB 栈快满):        增长后(拷贝到 4KB 新栈):
 ┌──────────┐ 高地址        ┌──────────┐
 │ 帧 A     │               │ 帧 A     │  ← 内容逐字节拷贝
 │ 帧 B     │   morestack   │ 帧 B     │
 │ 帧 C ←SP │  ══════════▶  │ 帧 C     │
 └──────────┘ 触底           │ 帧 D ←SP │  ← 现在有空间继续
                            │  ...     │
                            └──────────┘ 低地址

对代码的影响

  • 不要假设栈变量地址稳定:栈增长后地址会变(但指向它的指针会被运行时修正,所以对你透明)。这也是为什么 Go 里取局部变量地址完全安全——即使逃逸,运行时/编译器会正确处理(逃逸的搬到堆,不逃逸的随栈移动)。
  • 深递归在 Go 里不易 segfault,但会持续增长栈直至触及上限(64 位默认 maxstacksize ≈ 1GB,可调);超限时 runtime 直接 fatal error: stack overflow 终止进程——这是不可 recover 的 fatal error,和普通 panic 有本质区别。
  • goroutine 的调度与栈,配合 channel,构成 Go 并发模型的基础,见 10-并发模型。

小结

  • 用法层面:函数用 func 名(参数 类型) 返回类型 { ... } 定义,可一次返回多个值;方法在函数名前加接收者 (c *T);闭包是能记住外部变量的匿名函数,写法是 func(参数) 返回类型 { ... }。
  • 底层层面:函数是一等值,底层是 funcval 指针,闭包时携带捕获变量。一切按值传递;要改就传指针。slice/map/chan 拷贝的是含指针的表头,故有别名效果。方法接收者同样按值:值接收者拷贝整个接收者,指针接收者可改且避免大拷贝;注意方法集与可寻址性。闭包捕获变量而非值;Go 1.22 修复了循环变量捕获坑。变参 ...T 内部是 slice;xs... 展开复用底层数组。goroutine 栈从 2KB 起,连续栈可增长/收缩,与 C 固定线程栈是两个世界。
