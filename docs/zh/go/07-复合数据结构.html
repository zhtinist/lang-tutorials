复合数据结构的内部实现

本章是全书重点。slice、map、channel 是 Go 里”看起来像引用类型”的三兄弟。理解它们的关键只有一句话:它们本身都是一个很小的、按值拷贝的”表头(header)“结构,表头里藏着指向堆上真实数据的指针。(回顾「变量与内存模型」中的值拷贝)

搞清了表头的字段,你就能预测每一次赋值、传参、append 到底拷贝了什么、有没有分配、会不会产生别名(aliasing)。

一、array:真正的值类型

先立一个对照物。Go 的数组 [N]T 是定长、值语义的,长度是类型的一部分——[3]int 和 [4]int 是不同类型。

var a [3]int          // 24 字节,全部在当前栈帧里,零值 {0,0,0}
b := a                // 逐字节拷贝整整 24 字节(和 C 的 struct 赋值一样)
b[0] = 100            // 不影响 a
fmt.Println(a, b)     // [0 0 0] [100 0 0]

这和 C 里 int a[3]; int b[3]; b = a;(其实 C 不允许数组直接赋值,得 memcpy)行为一致——拷贝整块内存。传参也是拷贝整块:把 [1024]int 传进函数会复制 8KB。所以 Go 里几乎没人直接传数组,都传 slice。

array [3]int 在内存中:
┌────────┬────────┬────────┐
│ int[0] │ int[1] │ int[2] │   连续 24 字节,就是数据本身
└────────┴────────┴────────┘

二、slice:三元组表头

slice 才是日常用的”动态数组”。它的运行时表示是 runtime.slice(源码 runtime/slice.go):

type slice struct {
    array unsafe.Pointer // 指向底层数组首元素
    len   int            // 当前长度
    cap   int            // 底层数组从 array 起的容量
}

三个机器字(64 位下 24 字节)。slice 变量本身只有这 24 字节,数据在别处。

s := []int{10, 20, 30}

slice header(栈上, 24 字节)        底层数组(通常在堆上)
┌───────────┐                      ┌────┬────┬────┐
│ array ────┼─────────────────────▶│ 10 │ 20 │ 30 │
│ len = 3   │                      └────┴────┴────┘
│ cap = 3   │
└───────────┘

赋值 / 传参:拷贝表头,不拷贝数据

s := []int{10, 20, 30}
t := s          // 拷贝 24 字节表头;t.array 与 s.array 指向同一块数组!
t[0] = 999      // 改的是共享的底层数组
fmt.Println(s)  // [999 20 30] —— s 也变了(别名 aliasing)

这就是 C++ 程序员最容易踩的地方:t := s 看着像 std::vector 的深拷贝,其实更像 std::span / 裸指针拷贝——共享底层缓冲。传参同理,把 slice 传进函数,函数内对元素的修改对调用方可见(但对表头本身的修改——比如 append 导致重新分配——不可见,见下文)。

从数组/slice 切片:s[low:high]

切片操作不分配、不拷贝数据,只造一个新表头指向原数组的一段:

a := [5]int{0, 1, 2, 3, 4}
s := a[1:3]     // len=2, cap=4(从下标1到数组末尾), array 指向 &a[1]
a: ┌───┬───┬───┬───┬───┐
   │ 0 │ 1 │ 2 │ 3 │ 4 │
   └───┴───┴───┴───┴───┘
         ▲
         │ s.array
   s = {array:&a[1], len:2, cap:4}   ← cap 一直数到底层数组末尾

len 是你能索引的范围,cap 是不重新分配就能 append 到的范围。s[i] 越界(i >= len)会 panic;这是 Go 的边界检查,和 C 的”随便越界读脏数据”不同(编译器能证明安全时会消除检查)。

append 与扩容策略

append 是理解 slice 的核心。逻辑:

  1. 若 len < cap:直接写到 array[len],len++,不分配、不拷贝,返回的新表头与旧的共享底层数组。
  2. 若 len == cap:分配一块更大的新数组,把旧数据 memmove 过去,新表头指向新数组,与旧 slice 脱钩。
s := make([]int, 0, 2) // len=0 cap=2
s = append(s, 1)       // len=1, 原地写, 无分配
s = append(s, 2)       // len=2, 原地写, cap 用满
s = append(s, 3)       // len=3, cap 不够 → 分配新数组(通常 cap 变 4)、拷贝

扩容倍数(gc 实现,runtime.growslice,随版本微调,以运行时为准):

旧 cap 新 cap 增长策略
需求超过 2×旧 cap 直接用需求值
旧 cap < 256 约翻倍(2×)
旧 cap ≥ 256 约 1.25×,逐步平滑到 1.25(避免大 slice 浪费内存)

之后还会按元素大小做内存对齐、向 size class 取整(见第09章 mspan/size class)。所以打印出的 cap 未必正好是翻倍值。

经验:能预估大小就 make([]T, 0, n) 预留容量,避免多次分配+拷贝——等价于 C++ vector::reserve。

append 别名陷阱(最经典的坑)

因为”cap 够就原地写、不够才分配”,同一底层数组可能被多个 slice 共享,append 可能悄悄改到别人:

a := []int{1, 2, 3, 4, 5}
s := a[0:2]              // len=2 cap=5, 与 a 共享底层数组
s = append(s, 99)        // cap 足够 → 原地写 a[2]!
fmt.Println(a)           // [1 2 99 4 5] —— a 被改了!
append 前:                append(s, 99) 后(cap 够, 原地):
a = [1 2 3 4 5]            a = [1 2 99 4 5]   ← 第3个元素被覆盖
s = a[0:2] ────┐          s = {len:3, cap:5}
               │              s[2] 写进了 a[2] 的位置

反过来,如果扩容发生了,两者又会脱钩,后续修改互不影响——同一段代码的行为取决于运行时 cap 够不够,这是 bug 温床。防御手段:

  • 用三索引切片限制 cap:s := a[0:2:2](s[low:high:max],cap = max-low)。此时 cap==len,下一次 append 必然分配新数组,不会踩到 a。
  • 想要真正独立的副本,用 copy:
dst := make([]int, len(src))
copy(dst, src)   // 逐元素拷贝, 返回拷贝个数 min(len(dst),len(src))

copy 底层是 memmove,允许源和目标重叠。

三、map:哈希表

版本提示:下面描述的 hmap/bmap(桶 + 溢出桶、负载因子 6.5、渐进 evacuate 迁移)是 Go ≤ 1.23 的经典实现。Go 1.24 起内置 map 改用 Swiss Tables 设计(源自 Abseil 的开放寻址 + 分组控制字/SIMD 探测,internal/runtime/maps),内部布局与”溢出桶”“6.5 负载因子”等细节都变了、平均更快;但语言层面的语义(无序、并发不安全、&m[k] 不可取址等)完全不变。这里仍讲经典实现,因为它更能说明”哈希表 + 桶/拉链”的通用原理——把它当作”一种”实现来读即可。

map 的运行时类型是 *hmap(runtime/map.go,Go ≤ 1.23)——注意 map 变量本身就是一个指针(一个字),这是它和 slice 表头的一个差异。

type hmap struct {
    count     int    // 元素个数, len(m) 直接读它 O(1)
    flags     uint8
    B         uint8  // 桶数量 = 2^B
    noverflow uint16
    hash0     uint32 // 哈希种子(每个 map 随机, 防 hash flooding 攻击)
    buckets   unsafe.Pointer // 指向 2^B 个 bmap 桶的数组
    oldbuckets unsafe.Pointer // 扩容时的旧桶数组
    nevacuate uintptr
    extra     *mapextra
}

桶 bmap:每个桶固定 8 个槽位。

type bmap struct {
    tophash [8]uint8 // 每个 key 哈希值的高 8 位(快速比较用)
    // 编译器在后面追加: keys[8], values[8], overflow *bmap
}

内存布局是 8 个 tophash │ 8 个 key │ 8 个 value │ 溢出桶指针。注意 key 和 value 分别连续存放(不是 kv/kv/kv 交错),这样能避免因对齐产生的填充浪费(比如 map[int8]int64 交错会浪费很多)。

hmap ──▶ buckets: [ bucket0 | bucket1 | ... | bucket(2^B-1) ]
                       │
                       ▼
  一个 bmap 桶:
  ┌──────────────────────┐
  │ tophash[0..7]        │  每槽哈希高8位(也用作空/已迁移标记)
  ├──────────────────────┤
  │ key0 key1 ... key7   │  8 个 key 连续
  ├──────────────────────┤
  │ val0 val1 ... val7   │  8 个 value 连续
  ├──────────────────────┤
  │ overflow ────────────┼──▶ 下一个溢出 bmap(链式)
  └──────────────────────┘

查找过程

v := m[k]:

  1. hash = hasher(k, hash0)。
  2. 用低 B 位选桶:bucket = hash & (2^B - 1)。
  3. 用高 8 位 tophash 在桶内 8 个槽快速筛选(只比 1 字节,大多数不匹配的直接跳过),命中再完整比较 key。
  4. 桶内没有就沿 overflow 链找溢出桶。

平均 O(1),最坏(大量哈希冲突)退化。

增量扩容(rehash)

当负载因子过高(元素数 / 桶数 > 6.5)或溢出桶太多时,触发扩容:分配 2× 的新桶数组,把 buckets 挪到 oldbuckets。关键:不一次性搬完,而是在后续每次写操作(insert/delete)时顺手迁移 1~2 个旧桶(evacuate),摊销掉大 map 一次性 rehash 的停顿。查找时会同时看新旧桶。这与 C++ std::unordered_map 的一次性 rehash 不同。

map 的几个硬约束(和 C/C++ 直觉不同)

1. 迭代顺序是随机的,故意的。 每次 for k, v := range m 的起始桶和起始槽都随机化。这是为了防止程序依赖顺序(以后实现变了就崩)。要有序必须自己排序:

keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys) // 见第11章
for _, k := range keys {
    fmt.Println(k, m[k])
}

2. 元素不可取地址。 &m[k] 编译错误。因为扩容会搬动 value 的内存位置,指针会失效。所以不能就地改结构体字段:

m := map[string]Point{"a": {1, 2}}
// m["a"].X = 10        // 编译错误: cannot assign to struct field
p := m["a"]; p.X = 10; m["a"] = p   // 取出→改→写回
// 或者存指针: map[string]*Point, 这样 m["a"].X = 10 可以

3. 读缺失键返回零值 + ok。 逗号 ok 惯用法:

v, ok := m[k] // ok 为 false 表示不存在, v 是零值

4. 并发不安全。 并发读写(哪怕一读一写)会被运行时检测到并直接 fatal error: concurrent map read and map write 崩溃(不是 panic,不可 recover)。需要自己加锁,或用 sync.Map(读多写少场景,见第10章)。

5. nil map 可读不可写。 var m map[string]int 是 nil,读返回零值,写 panic。必须 make 或字面量初始化。

四、channel:带锁的环形队列

channel 是 Go 并发的核心原语,底层是 *hchan(runtime/chan.go),channel 变量也是一个指针。

type hchan struct {
    qcount   uint           // 当前缓冲区里的元素数
    dataqsiz uint           // 缓冲区容量(make 时的第二个参数)
    buf      unsafe.Pointer // 指向环形缓冲数组(有缓冲时)
    elemsize uint16
    closed   uint32
    sendx    uint           // 环形缓冲的写下标
    recvx    uint           // 环形缓冲的读下标
    recvq    waitq          // 等待接收的 goroutine 队列
    sendq    waitq          // 等待发送的 goroutine 队列
    lock     mutex          // 保护上面所有字段
}
有缓冲 channel  make(chan int, 4):
        ┌─────────────────────────────┐
buf ──▶ │  slot0 slot1 slot2 slot3    │  环形缓冲
        └─────────────────────────────┘
          ▲recvx        ▲sendx
  lock  ─ 一把互斥锁, 每次收发都要抢
  sendq ─ [G3, G7 ...]  被阻塞在发送上的 goroutine
  recvq ─ [ ]           被阻塞在接收上的 goroutine

收发的阻塞语义

无缓冲 channel(make(chan int),dataqsiz=0):发送和接收必须”会合(rendezvous)“。发送方阻塞,直到有接收方到来,数据直接从发送方 goroutine 拷贝到接收方 goroutine(不经过缓冲),然后两者都被唤醒。是最强的同步:收发是一个同步点。

有缓冲 channel(make(chan int, n)):

操作 缓冲未满/非空 缓冲满(发)/空(收)
发送 ch <- v 拷进 buf[sendx],sendx++,立即返回 当前 goroutine 入 sendq,阻塞挂起
接收 <-ch 从 buf[recvx] 拷出,recvx++,立即返回 当前 goroutine 入 recvq,阻塞挂起

“阻塞挂起”不是忙等:goroutine 被调度器摘下,M 去跑别的 G(见第10章 GMP)。当对端操作发生时,把等待队列里的 goroutine 唤醒。这就是 goroutine 廉价并发的基础——阻塞一个 goroutine 不阻塞 OS 线程。

一个优化:若发送时 recvq 里已有等待的接收者,数据跳过缓冲区直接拷给那个接收者并唤醒它,省一次拷贝。

close 与 nil

close(ch)          // 关闭; 之后发送 panic; 接收立即返回零值
v, ok := <-ch      // ok==false 表示 channel 已关闭且缓冲已空
for v := range ch  // 一直收到 channel 关闭为止

对 nil channel 收发永久阻塞(常用于在 select 中动态”关掉某个分支”)。

channel 传递的是元素的拷贝(值语义):ch <- x 把 x 拷进缓冲,<-ch 拷出。如果元素是大结构体就拷整块;传指针则拷指针(此时要注意共享数据的并发安全)。

五、三者对照小结

类型 变量本身大小 赋值/传参拷贝什么 数据在哪 并发安全
[N]T array N×sizeof(T) 整块数据 随变量(栈或堆) 值语义,天然隔离
[]T slice 3 字(24B) 表头(含指针) 底层数组通常在堆 否,共享底层数组
map 1 字(指针) 指针 hmap+桶在堆 否,并发读写崩溃
chan 1 字(指针) 指针 hchan+buf 在堆 是,内部有锁

一句话:除了 array 是真值类型,slice/map/chan 拷贝的都是”轻表头/指针”,共享背后的数据——这正是它们”像引用”的原因,但机制上仍是彻底的值拷贝。
