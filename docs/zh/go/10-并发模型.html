并发模型:goroutine 与 channel

并发是 Go 的招牌。本章先讲 goroutine/channel/select 这几样东西「怎么写、怎么用」——普通的并发代码看得懂就够;再讲「底层与 C 对比」——运行时怎么把几十万个 goroutine 塞进几个 OS 线程里调度。 对 C++ 程序员:先忘掉 std::thread(1:1 映射 OS 线程、每个几 MB 栈、创建昂贵),后面你会看到 Go 的 goroutine 是运行时在少量 OS 线程上多路复用的用户态协程,便宜到可以开几十万个。

先会用:goroutine、channel、select 速览

不涉及任何调度器细节,先看这三样东西怎么写:

package main

import "fmt"

func say(msg string, done chan bool) {
    fmt.Println(msg)
    done <- true // 通过 channel 告诉外面: 我干完了
}

func main() {
    done := make(chan bool)
    go say("in goroutine", done) // 启动一个 goroutine, 立即往下走(不等它)
    <-done                       // 阻塞, 直到 goroutine 发来信号

    ch := make(chan int, 1)
    ch <- 100 // 发送

    select { // select 同时等待多个 channel, 哪个就绪走哪个
    case v := <-ch:
        fmt.Println("received:", v) // received: 100
    default:
        fmt.Println("no value")
    }
}

简单总结:

  • go 函数调用(...) 启动一个 goroutine——一个”轻量级的并发任务”,立即返回,不等它跑完。
  • goroutine 之间用 channel 传数据/发信号:ch <- v 发送,v := <-ch 接收(回顾第07章 channel 的基本用法)。
  • select 同时盯着几个 channel,哪个先就绪就走哪个分支;default 让它不阻塞。

这几个够你写出能跑的并发代码了。下面进入”底层”——为什么 Go 能轻松开几十万个 goroutine、调度器怎么把它们安排到 CPU 上。

开始深入:为什么 goroutine 便宜

从这一节起讲底层。要理解 Go 并发的性能特征,先记住一句话:goroutine 不是 OS 线程,是运行时自己管理的用户态协程,由 Go 运行时的调度器(GMP 模型)多路复用到少数几个 OS 线程上。

一、goroutine:极轻量的协程

怎么用

go 关键字加一个函数调用就启动一个 goroutine:

go f(x, y)        // 启动一个 goroutine 跑 f(x,y), 立即返回, 不等它
go func() {
    fmt.Println("in goroutine")
}()

配合 sync.WaitGroup 等一组 goroutine 都跑完(下面「sync 包」一节详细介绍):

var wg sync.WaitGroup
results := make([]int, 3)

for i := 0; i < 3; i++ {
    wg.Add(1)
    go func(id int) { // 启动一个 goroutine
        defer wg.Done()
        results[id] = id * id
    }(i)
}
wg.Wait() // 等所有 goroutine 完成

fmt.Println(results) // [0 1 4]

goroutine 什么时候结束:函数体返回就结束。main 函数返回则整个程序退出,不等其他 goroutine——需要等待就用 channel 或 sync.WaitGroup。

底层与 C 对比

为什么便宜:

OS 线程(C++ std::thread) goroutine
初始栈 固定 1~8 MB(虚拟内存预留) 2 KB,按需增长/收缩
创建/销毁 系统调用,微秒级,昂贵 用户态,纳秒级,几乎白送
调度 内核调度,切换要陷入内核 运行时用户态调度,切换极快
数量级 几千个就吃力 几十万、上百万可行

栈增长:goroutine 栈起始 2KB,不够时运行时分配更大的栈、把内容拷过去、并改写所有指向该栈的指针(第06章连续栈)。正因为运行时会在栈”搬家”时修正指针,你照样可以像平常一样对局部变量取地址(&local)、把它返回或传给别处:逃逸分析会把真正逃逸的变量放到堆上,其余留在栈上,你完全无感。这点恰恰和 C 相反——C 里返回/长期持有栈变量地址是悬垂指针 bug,Go 里则安全。

二、channel 与 CSP

怎么用

Go 奉行 CSP(Communicating Sequential Processes)哲学:

不要通过共享内存来通信,而要通过通信来共享内存。 Don’t communicate by sharing memory; share memory by communicating.

意思是:优先用 channel 在 goroutine 间传递数据的所有权,而不是多个 goroutine 抢一块共享内存靠锁保护。

ch := make(chan int)        // 无缓冲
ch := make(chan int, 10)    // 缓冲 10

ch <- 42                    // 发送
v := <-ch                   // 接收
v, ok := <-ch               // ok=false 表示已关闭且空
close(ch)                   // 关闭(由发送方关, 别由接收方关)
for v := range ch { }       // 收到 channel 关闭为止

常见用法:用 channel 等待一个 goroutine 完成。

done := make(chan struct{})   // struct{} 零大小, 只当信号
go func() {
    work()
    close(done)               // 或 done <- struct{}{}
}()
<-done                        // 阻塞直到 work 完成

底层与 C 对比

channel 的底层结构(hchan:环形缓冲 + 一把锁 + 等待队列)已经在第07章详细讲过。这里补充一点运行时行为:无缓冲 channel 的发送/接收是一次会合(rendezvous),天然构成 happens-before(下面「Go 内存模型」一节详细展开),常用作信号量/同步点——这正是上面”等待 goroutine 完成”模式能工作的原因。

三、select:多路复用

怎么用

select 同时等待多个 channel 操作,哪个就绪走哪个(多个就绪则随机选一个,防饥饿):

select {
case v := <-ch1:
    fmt.Println("from ch1", v)
case ch2 <- x:
    fmt.Println("sent to ch2")
case <-time.After(2 * time.Second):
    fmt.Println("超时")            // 超时模式
default:
    fmt.Println("都没就绪, 非阻塞") // 有 default 就不阻塞
}
  • 无 default:阻塞到至少一个 case 就绪。
  • 有 default:所有 case 都没就绪时立即走 default —— 非阻塞收发。
  • time.After 返回一个到点才发数据的 channel —— 超时惯用法。
  • 对 nil channel 的 case 永远不就绪,可用于动态”禁用”某分支。

可运行示例:两路 channel 多路复用 + time.After 超时兜底。两个 goroutine 到点各发一次,select 收满两条才退出。到达顺序不定,所以断言集合而非精确顺序:

package main

import (
    "fmt"
    "time"
)

func main() {
    nums := make(chan int)
    strs := make(chan string)

    go func() {
        time.Sleep(10 * time.Millisecond)
        nums <- 42
    }()
    go func() {
        time.Sleep(20 * time.Millisecond)
        strs <- "hi"
    }()

    got := map[string]bool{}
    timeout := time.After(100 * time.Millisecond)
    for len(got) < 2 {
        select {
        case n := <-nums:
            got[fmt.Sprintf("num:%d", n)] = true
        case s := <-strs:
            got[fmt.Sprintf("str:%s", s)] = true
        case <-timeout:
            fmt.Println("timed out")
            return
        }
    }
    fmt.Println("got num:", got["num:42"])
    fmt.Println("got str:", got["str:hi"])
    // got num: true
    // got str: true
}

四、GMP 调度模型

goroutine/channel/select 的基本用法到此为止。这一节是纯粹的运行时内部机制——解释为什么上面这些便宜、好用。

怎么用:GOMAXPROCS

日常写并发代码基本不需要关心调度器,唯一常打交道的旋钮是并行度:

runtime.GOMAXPROCS(4)         // 设置/查询 P 的数量(即并行度上限)
n := runtime.GOMAXPROCS(0)    // 传 0 表示只查询不修改

底层与 C 对比

Go 运行时用 G-M-P 三元组把 goroutine 多路复用到 OS 线程上。

  • G(goroutine):一个协程,含栈、程序计数器、状态。
  • M(machine):一个 OS 线程,真正在 CPU 上执行代码。
  • P(processor):调度上下文/逻辑处理器,持有一个本地可运行 G 队列。P 的数量 = GOMAXPROCS(默认 = 可用 CPU 核数;Go 1.25 起在容器中会感知 cgroup 的 CPU 限额、按限额取整,不再一律等于物理核数)。M 必须绑定一个 P 才能执行 G。
              全局运行队列 GRQ: [G G G ...]
                     │(本地空了来偷/拿)
   ┌─────────────┐   ┌─────────────┐
   │  P0         │   │  P1         │   P 数量 = GOMAXPROCS
   │ 本地队列LRQ │   │ 本地队列LRQ │
   │ [G G G]     │   │ [G G]       │
   └──────┬──────┘   └──────┬──────┘
          │绑定             │绑定
      ┌───▼───┐         ┌───▼───┐
      │  M0   │         │  M1   │   M = OS 线程, 数量按需(可能 > P)
      │(线程) │         │(线程) │
      └───────┘         └───────┘
           实际跑在 CPU 核上

调度要点

本地队列 + work-stealing(工作窃取):每个 P 有本地队列,新建的 G 优先放本地(无锁,快)。P 本地空了,先去全局队列拿,再从别的 P 队列偷一半。这让负载自动均衡,又尽量避免全局锁竞争。

为什么要有 P(而不是 G 直接绑 M):P 这层解耦了”并行度(GOMAXPROCS 个 P)“与”线程数(M 可以更多)“。当一个 M 因系统调用阻塞时,它的 P 可以交给别的 M 继续跑其他 G——见下。

系统调用时 P 与 M 解绑(hand-off):某个 G 发起阻塞式系统调用(如读文件),它所在的 M 会陷入内核阻塞。运行时把这个 M 的 P 摘下,交给另一个(空闲或新建的)M,让 P 上其余的 G 继续在别的核上跑,不被一个阻塞调用拖住。系统调用返回后,原 M 尝试重新获取一个 P,拿不到就把 G 放回队列、自己休眠。

网络 I/O 用 netpoller:网络读写不会真的阻塞 M,而是注册到基于 epoll/kqueue 的 netpoller,goroutine 挂起、M 去跑别的 G,I/O 就绪时再唤醒。所以你能用”每连接一个 goroutine、同步阻塞式写法”却获得 epoll 的伸缩性——这是 Go 网络服务爽点的底层原因。

抢占式调度:早期 Go 是协作式(只在函数调用处让出),一个纯计算死循环会饿死同 P 的其他 G。Go 1.14 起是基于信号的异步抢占:运行时可向长时间不让出的 G 发信号,在安全点强制切换。所以现在不用担心 for {} 卡死调度器。

对照 C++:std::thread 是 1:1 内核线程,调度归 OS,切换要进内核;Go 是 M:N(M 个 G 跑在 N 个线程上),大部分调度在用户态完成,切换成本低得多。

五、sync 包:传统锁与同步原语

怎么用

不是所有场景都适合 channel。共享计数器、保护一小段临界区,用锁更直接、更快。

var mu sync.Mutex
mu.Lock()
count++
mu.Unlock()          // 常配 defer mu.Unlock()
原语 用途 对照 C++
sync.Mutex 互斥锁 std::mutex
sync.RWMutex 读写锁,多读单写 std::shared_mutex
sync.WaitGroup 等一组 goroutine 完成 无直接对应(手动 join)
sync.Once 某代码只执行一次(懒初始化) std::call_once
sync/atomic 原子操作(Add/Load/Store/CAS) std::atomic
sync.Map 并发安全 map,读多写少优化 无(concurrent_map 类)
sync.Pool 临时对象池,减少分配/GC 压力 对象池

WaitGroup 惯用法:

var wg sync.WaitGroup
for i := 0; i < 10; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        work(id)
    }(i)                  // 注意: 把 i 作为参数传进去(Go1.22 前循环变量复用坑)
}
wg.Wait()                 // 阻塞到计数归零

Go 1.22 起 for 循环变量每轮独立,老代码里”闭包捕获循环变量”的经典 bug 大多消失;但显式传参仍是清晰的好习惯。

sync.Mutex 零值即可用(不用初始化),不可拷贝(拷了就是两把锁,go vet 会警告)。所以含 Mutex 的 struct 通常用指针传递。

可运行示例:100 个 goroutine 各自 +1 一千次,Mutex 与 atomic.Int64 都得到确定的 100000。若把 mu.Lock/Unlock 去掉、改成裸 count++,go run -race 会报 DATA RACE,结果也会因丢更新小于 100000:

package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

func main() {
    const goroutines, per = 100, 1000
    want := int64(goroutines * per)

    // 1) sync.Mutex 保护普通计数器
    var mu sync.Mutex
    var muCount int64
    var wg sync.WaitGroup
    for i := 0; i < goroutines; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for j := 0; j < per; j++ {
                mu.Lock()
                muCount++
                mu.Unlock()
            }
        }()
    }
    wg.Wait()

    // 2) sync/atomic 原子加
    var atCount atomic.Int64
    for i := 0; i < goroutines; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for j := 0; j < per; j++ {
                atCount.Add(1)
            }
        }()
    }
    wg.Wait()

    fmt.Println("want  :", want)
    fmt.Println("mutex :", muCount, muCount == want)
    fmt.Println("atomic:", atCount.Load(), atCount.Load() == want)
    // want  : 100000
    // mutex : 100000 true
    // atomic: 100000 true
}

底层与 C 对比

sync.Mutex 本身很小(概念上是一个状态字 + 一个信号量,共 8 字节),加锁分两条路径:无竞争时靠一次原子 CAS 直接拿到锁(fast path,几乎零开销);抢不到才会短暂自旋几次,还拿不到就通过运行时的信号量把 goroutine 挂起(park),解锁时唤醒——挂起的 goroutine 会被调度器摘下,不占用 M,和普通 channel 阻塞是同一套”挂起-唤醒”机制。这与 C++ std::mutex 通常直接包一层内核 futex 等待不同:Go 的锁是运行时调度框架内的一部分,阻塞的是 goroutine 而不是线程本身。细节随版本演进,以运行时源码 sync/mutex.go 为准。

六、Go 内存模型与 happens-before

多核下,没有同步的并发读写没有顺序保证(编译器/CPU 会重排、缓存不可见)。Go 内存模型用 happens-before 定义可见性:若事件 A happens-before B,则 A 的写对 B 可见。建立 happens-before 的手段:

  • channel:一次发送 happens-before 对应的接收完成;channel 的关闭 happens-before 接收到零值。
  • Mutex:第 n 次 Unlock happens-before 第 n+1 次 Lock。
  • WaitGroup:Wait 返回前,所有 Done 之前的写可见。
  • Once:Do 里的初始化 happens-before 任何 Do 返回。
  • atomic:提供顺序保证(Go 1.19 起 sync/atomic 有明确的类型化 API atomic.Int64 等)。

没有这些同步,就不能假设一个 goroutine 的写另一个能看到。 这和 C++ 的 std::memory_order 是同一类问题,Go 把它简化成”用 channel/锁,不要裸共享”。

七、context:取消、超时、传值

怎么用

context.Context 是跨 goroutine/跨 API 传递取消信号、截止时间、请求域值的标准方式,几乎所有阻塞型标准库/框架 API 第一个参数都是它。

ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()                        // 必须调 cancel 释放资源

select {
case res := <-doAsync(ctx):
    use(res)
case <-ctx.Done():                    // 超时或被取消
    fmt.Println("取消原因:", ctx.Err()) // context.DeadlineExceeded / Canceled
}

ctx.Done() 返回一个在取消时被关闭的 channel,配合 select 用。取消信号会沿 context 树向下传播,一处 cancel,所有派生 context 都收到——用于”请求结束了,把它启动的所有下游 goroutine 一起停掉”。context.WithValue 传请求域数据(如 trace id),但别拿它传函数参数。

可运行示例:一次 cancel() 广播给 3 个 worker,它们各自的 ctx.Done() 同时就绪并退出。用 WaitGroup 确定性收敛,断言退出计数与 ctx.Err():

package main

import (
    "context"
    "fmt"
    "sync"
    "time"
)

func main() {
    ctx, cancel := context.WithCancel(context.Background())
    var wg sync.WaitGroup
    const n = 3
    stopped := make(chan int, n)

    for id := 0; id < n; id++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for {
                select {
                case <-ctx.Done(): // 收到取消信号, 退出
                    stopped <- id
                    return
                default:
                    time.Sleep(time.Millisecond) // 模拟干活
                }
            }
        }(id)
    }

    time.Sleep(20 * time.Millisecond)
    cancel() // 广播取消: 所有 worker 的 ctx.Done() 一起就绪
    wg.Wait()
    close(stopped)

    fmt.Println("ctx.Err():", ctx.Err())
    fmt.Println("stopped  :", len(stopped))
    // ctx.Err(): context canceled
    // stopped  : 3
}

八、常见并发模式

Worker Pool(固定数量 worker 消费任务):

jobs := make(chan int, 100)
results := make(chan int, 100)
var wg sync.WaitGroup
for w := 0; w < 4; w++ {              // 4 个 worker
    wg.Add(1)
    go func() {
        defer wg.Done()
        for j := range jobs {         // 自动随 jobs 关闭而退出
            results <- j * j
        }
    }()
}
for i := 1; i <= 20; i++ { jobs <- i }
close(jobs)                            // 关闭 → worker 循环结束
go func() { wg.Wait(); close(results) }()
for r := range results { fmt.Println(r) }

Pipeline(阶段串联,每阶段一个/多个 goroutine,用 channel 连接):

gen() ──chan──▶ square() ──chan──▶ print()

Fan-out / Fan-in:fan-out = 多个 goroutine 从同一个 channel 读(分摊活);fan-in = 把多个 channel 合并成一个。

// fan-in: 合并多个输入 channel
func merge(cs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, c := range cs {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c { out <- v }
        }(c)
    }
    go func() { wg.Wait(); close(out) }()
    return out
}

九、data race 与 -race 检测器

数据竞争(data race):两个 goroutine 并发访问同一内存,至少一个是写,且无同步。行为未定义——可能读到撕裂值、可能崩溃。Go 不会自动帮你消除竞争(除了 map 会主动崩溃报错)。

Go 内置竞争检测器,编译时加 -race:

go run -race main.go
go test -race ./...

它在运行时插桩每次内存访问,发现无同步的并发读写就打印详细报告(哪两个 goroutine、哪一行、调用栈)。性能开销大(慢几倍、内存涨),只在测试/CI 用,不上生产。这是 Go 并发调试最有用的工具,写并发代码务必用它跑一遍测试。

WARNING: DATA RACE
Write at 0x00c000... by goroutine 7:
  main.main.func1() main.go:12
Previous read at 0x00c000... by goroutine 6:
  main.main.func2() main.go:15

小结

  • 用法层面:go f(x,y) 启动 goroutine,不等它跑完;channel 用 make/<-/close/range 收发和同步;select 同时等待多个 channel,配 default 做非阻塞、配 time.After 做超时;sync.Mutex/WaitGroup/atomic 处理共享状态;context 传取消/超时;worker pool、pipeline、fan-in/out 是常见骨架;写并发代码要用 -race 跑一遍。
  • 底层层面:goroutine 是 2KB 起步、用户态调度的协程,栈可增长/收缩,M:N 多路复用到 OS 线程;GMP 三元组——G 协程、M 线程、P 上下文(=GOMAXPROCS)——靠本地队列 + work-stealing + 抢占 + 系统调用 hand-off 实现调度,网络 I/O 靠 netpoller 而不阻塞线程;sync.Mutex 是原子 CAS 快路径 + 运行时挂起/唤醒的慢路径,阻塞的是 goroutine 不是 OS 线程;正确性由 happens-before 保证,没有同步就不能假设写操作对另一个 goroutine 可见。
