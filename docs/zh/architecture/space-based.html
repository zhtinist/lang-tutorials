基于空间的架构 · Space-Based Architecture

这种模式有时也称云架构 (cloud architecture)：通过去掉中央数据库瓶颈，用可复制的内存处理单元实现极致水平扩展。即 Space-Based Architecture。

一、概念与要解决的问题

高并发网站常见瓶颈不在 Web 层，而在中央共享数据库：连接打满、锁竞争、磁盘 I/O。

例如一次秒杀活动，如果沿用普通分层架构「每次请求都直接查/改中央数据库里的库存字段」：Web 层可以轻松加到 50 台服务器分摊流量，但这 50 台服务器最终都要抢同一行库存记录的锁，数据库连接池很快被打满，大量请求卡在等锁上，哪怕再加 50 台 Web 服务器也无济于事——瓶颈根本不在 Web 层，加了也是白加。

基于空间的架构思路：

  1. 不把每次请求都打到中央库
  2. 把业务处理封装为可复制的处理单元 (Processing Unit)
  3. 数据尽量驻留在单元内存中，并在单元间复制/同步
  4. 访问量升则启新单元，降则关单元——弹性伸缩

名称里的「空间 (space)」来自分布式共享内存 / tuple space 传统：多个节点共享一个逻辑数据空间。阮文用「云架构」强调其云上弹性伸缩气质；本教程两种叫法并用，结构对齐 Richards。

二、两大组成部分

可以把这套结构想成大型演唱会的入场安检：如果所有观众都挤到同一个安检口（中央数据库），必然大排长龙；更聪明的做法是设很多条安检通道（处理单元），每条通道能独立处理一批观众，通道之间再由统一调度（虚拟中间件）协调进度、按人流增减开放的通道数。

三、虚拟中间件的四个组件

对齐阮文 / Richards 四分法：

3.1 Messaging Grid（消息中间件）

管理用户请求与 session；请求进入后决定交给哪一个处理单元。

3.2 Data Grid（数据中间件）

把数据复制到各处理单元，做数据同步，使相关单元看到一致（或可接受一致级别）的内存视图。

3.3 Processing Grid（处理中间件，可选）

当一次请求需要多种类型处理单元协作时，负责协调编排。

3.4 Deployment Manager（部署管理）

启动/关闭处理单元；监控负载与响应时间——负载升则扩，降则缩。

以电商秒杀抢购为例走一遍：活动开始瞬间涌入 10 万个「抢购某商品」的请求，Messaging Grid 先接住这些请求，按用户 ID 把每个人的请求分配给某一个具体的处理单元（比如用户 ID 尾号 0-9 分别路由到 10 个处理单元之一），而不是让所有请求挤到同一个地方排队；每个处理单元 (PU) 在自己的内存里保存这部分商品的剩余库存，直接在内存中做「库存是否够、扣减」的判断，不必每次都往中央数据库打一次请求；Data Grid 负责把「商品总库存被扣减了多少」在各个处理单元之间同步，避免不同处理单元各自以为库存还很多、一起超卖；如果监控发现请求量还在飙升，Deployment Manager 会自动再启动几个新的处理单元分担压力，活动结束后再把多余的处理单元关掉。

Request ─► Messaging Grid ─► PU
                │
                ├── Data Grid（内存数据复制）
                ├── Processing Grid（跨 PU 协调，可选）
                └── Deployment Manager（弹性伸缩）

四、数据与一致性直觉

策略 含义
内存为主 读路径不打中央库，延迟低
复制 多 PU 持有数据副本，扩展读/算力
异步持久化 常在后台写入磁盘/库，防崩溃丢数
一致性权衡 强一致复制成本高；常接受有限过期或冲突解决

实现上会用到分布式缓存、数据网格产品、或自研复制协议——复杂度与成本显著高于普通分层单体。

五、优点

  1. 高负载下扩展性极好——消除中央库这一最大瓶颈（在适用模型内）
  2. 动态部署：按负载增减处理单元
  3. 适合「大量并发用户、工作集可放进内存网格」的在线系统

六、缺点

  1. 实现复杂、成本高（中间件 + 复制 + 运维）
  2. 主要适合网站类 / 会话型在线应用；不适合「超大吞吐、必须以大型集中式数据库为事实来源」的分析与重事务场景（阮文亦指出）
  3. 测试困难：涉及分布式状态、复制延迟、故障注入

七、何时使用

适合 不太适合
突发高并发、读多、工作集可控 海量冷数据必须频繁扫盘
可接受最终一致或精细冲突处理 简单业务、团队小、无运维预算
已有或愿意引入数据网格能力 强依赖复杂多表事务且难拆

现实对照：许多互联网系统用「无状态应用 + 分布式缓存 + 分库分表 + 异步」逼近类似目标，而不一定完整实现教科书式四组件中间件；但去掉同步路径上的中央瓶颈这一思想是相通的。
