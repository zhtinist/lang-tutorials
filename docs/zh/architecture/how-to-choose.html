架构选型 · Choosing an Architecture

选型不是选「最酷的名字」，而是在质量属性与成本之间做可撤回的权衡。模式可组合；反模式往往是过早分布式。

一、选型前先问的问题

就像装修房子要先想清楚自己家住几口人、预算多少、住多久，而不是看邻居用了什么网红建材就照抄——选架构也要先想清楚自己的处境，再挑模式，而不是看什么技术火就用什么。

  1. 主要矛盾：是吞吐、发布频率、团队并行，还是模型复杂度？
  2. 谁会变：UI 常变？业务规则常变？流量尖峰？
  3. 组织现实：几个人？有无平台/ SRE？能否独立发布？
  4. 一致性需求：要不要跨模块强事务？
  5. 可撤回性：选错了，回退成本有多大？

先定质量属性优先级，再映射到架构特征（扩展、弹性、可测、可部署……）。

三、按痛点速查

你的痛点 优先考虑 谨慎
不知道用啥，团队按技术分层 分层 一上来微服务
通知/审计/多下游、要削峰 事件驱动（可挂在分层或微服务上） 用分布式事务硬扛同步
同一产品多客户功能组合 微核 / 插件 为每个客户 Fork 整仓
多团队抢发版、模块负载不均 微服务（边界清时） 无自动化仍硬拆
中央库已被打爆、工作集可进内存 空间架构或「缓存+分片」近似 只加 Web 机器
领域术语混乱、规则极多 先 DDD 边界 + 整洁依赖 先选框架再补模型

四、决策口诀（可张贴）

1. 能单体则模块化单体；痛点未现不拆进程
2. 边界对齐限界上下文，而非对齐数据库表
3. 同步路径要短；能异步则事件
4. 扩展插件化能力用微核；扩展流量用水平单元
5. 每引入一次网络调用，就引入一次失败模式
6. 先有 CI/CD、监控、契约测试，再谈微服务
7. 写 ADR：选了什么、没选什么、何时复审

口诀浓缩：

属性排序 → 模式匹配 → 组合使用 → 小步验证；
过早微服务是经典踩坑，复杂度要买在刀刃上。

五、反模式

5.1 过早微服务

症状：三人团队、十几个仓库、本地起不来、改字段要开五个 PR。
处方：回到模块化单体；用包边界模拟上下文；真痛了再拆。

5.2 分布式单体

症状：服务很多，却同库、同步强依赖、必须一起发布。
处方：要么真拆数据与发布，要么诚实合并回单体。

5.3 为架构而架构

症状：简历驱动技术选型；质量属性从未写明。
处方：用一页纸列出 Top 3 质量属性与约束。

5.4 事件风暴却无幂等

症状：MQ 一上，重复消费、乱序把库存打穿。
处方：先设计幂等键、去重、顺序范围，再铺主题。

5.5 聚合/服务切得过碎

症状：一次下单跨 20 次 RPC；Saga 比业务还长。
处方：按不变式与变更频率合并；接受稍大的服务。

六、组合示例（现实中更常见）

┌──────────────────────────────────────────┐
│  API (REST) 表现适配                      │
├──────────────────────────────────────────┤
│  模块化单体 / 若干微服务（按上下文）        │
│    └─ 内：整洁依赖 + DDD 聚合             │
│    └─ 旁：领域事件 → MQ → 异步处理器       │
├──────────────────────────────────────────┤
│  插件点（可选微核）：支付、渠道、规则包      │
└──────────────────────────────────────────┘

很少有系统是「纯五种之一」；更多是分层/整洁做内核，事件做集成，微服务做组织缩放，插件做产品变体。
