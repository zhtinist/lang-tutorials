领域驱动设计入门 · DDD Basics

DDD 用领域模型驱动设计：先把业务语言与边界谈清，再谈类与服务——战略定边界，战术定模型零件。

一、DDD 是什么

开会时业务专家说「这份保单要理赔」「那个客户要升级成 VIP」，如果代码里对应的类叫 Table3Row、字段叫 flag2，那每次改需求都要在人脑里先做一遍「业务话术 → 代码术语」的翻译，翻译错一次就可能是一个线上 bug。DDD 想解决的就是这种「语言不通」的问题：让代码直接说业务的话。

领域驱动设计（Domain-Driven Design）由 Eric Evans 系统阐述，Vaughn Vernon 等补充实践。它不是某种部署架构，而是：

  • 让软件结构跟随业务结构
  • 用统一语言消除业务与代码的翻译损耗
  • 在复杂域里用模型保护不变式

适合：业务复杂度高的系统。简单 CRUD 不必硬上全套战术模式。

二、战略设计 vs 战术设计

战略 (Strategic) 战术 (Tactical)
问题 系统如何切分、上下文如何协作 上下文内部模型怎么建
关键产出 限界上下文、上下文映射、统一语言 实体、值对象、聚合、仓储、领域服务
失败代价 边界错 → 微服务/模块永远纠缠 模型弱 → 不变式散落、bug 多

先战略后战术：边界错了，聚合拆得再漂亮也救不了。

三、统一语言 (Ubiquitous Language)

同一限界上下文内，业务人员与开发者共用一套术语——会议里怎么说，代码里怎么命名。

反例 正例方向
代码 User，业务说「投保人/承运人/管理员」混用 在上下文内用精确词：PolicyHolder
表字段 status=3 无人能解 模型与文档同用「已承保 / 已退保」

统一语言是活的：术语变了，代码与测试命名应跟着变，而不是靠口头翻译层。

反例：产品经理在需求文档里写「保单进入宽限期后仍可补缴」，工程师把这句话翻译成代码里的 status = 3；半年后新来的同事要修一个宽限期相关的 bug，翻遍代码只看到一堆 if status == 3，既不知道 3 代表什么业务含义，也不敢确定改动会不会影响到「已失效」（说不定也是某个数字）的判断逻辑，只能回头去找产品经理或翻会议记录确认——这一趟来回就是「翻译损耗」的具体样子。如果代码里直接是 status == PolicyStatus.GRACE_PERIOD，业务人员和工程师对着屏幕就能核对，不需要额外的翻译环节。

四、限界上下文 (Bounded Context)

模型的适用边界：同一个词在不同上下文可以有不同含义。

┌─────────────────────┐     ┌─────────────────────┐
│  销售上下文          │     │  履约上下文          │
│  「订单」= 报价与成交 │     │  「订单」= 拣货发货   │
│  Order, Quote       │────►│  FulfillmentOrder   │
└─────────────────────┘     └─────────────────────┘
         集成：事件 / ACL / 开放主机服务等

要点：

  1. 不要追求企业级「一个大对象模型打天下」
  2. 跨上下文集成要显式（防腐层 ACL、共享内核需极谨慎）
  3. 上下文边界是微服务拆分的重要候选（见下文）

反例：如果销售团队和履约团队图省事，共用同一张 Order 数据库表和同一个 Order 类——销售团队眼里的「订单」在客户下单那一刻就该存在，字段以「报价、优惠、成交价」为主；履约团队眼里的「订单」要等支付确认后才该出现，字段以「仓库、物流单号、发货状态」为主。两边对着同一张表加字段、改状态枚举，销售团队上一个需求可能刚好把履约团队依赖的某个状态值语义改掉，导致发货逻辑莫名出错。拆成两个限界上下文、各自维护 Order（销售）和 FulfillmentOrder（履约）、之间用事件显式集成，两边就能各自演进而不互相绊倒。

五、战术积木

5.1 实体 (Entity)

有稳定身份 (Identity)，属性可变；靠 ID 区分，而非靠属性值。

@dataclass
class Customer:
    id: str          # 身份
    name: str        # 可变
    email: str

5.2 值对象 (Value Object)

无独立身份，由属性值定义；不可变更佳；可替换、可比较相等。

@dataclass(frozen=True)
class Money:
    amount: int      # 分
    currency: str

    def add(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("currency mismatch")
        return Money(self.amount + other.amount, self.currency)

地址、金额、日期区间常是值对象。

5.3 聚合 (Aggregate) 与 聚合根 (Aggregate Root)

一组相关对象的一致性边界：外部只能通过聚合根修改内部，一次事务只改一个聚合（经典启发式）。

        Order (Root)
        ├── OrderLine
        ├── OrderLine
        └── ShippingAddress (VO)

设计问题：聚合太大 → 并发冲突多；太小 → 跨聚合事务变多。按不变式划边界，而非按「方便 join」。

例如把「客户」也塞进 Order 这个聚合（一个订单对象里直接嵌套整个客户对象），会导致两个毫不相干的操作——「用户改自己的收货地址」和「另一个人查看这笔订单详情」——都要对同一个聚合加锁，高峰期两个操作互相等待，这是「聚合太大」。反过来，如果把 OrderLine（订单里的每一行商品）都各自设计成独立聚合，那么「提交订单」这一个操作就要跨好几个聚合分别开事务、分别保证一致——一旦其中一步失败，回滚会很麻烦，这是「聚合太小」。判断标准是「哪些字段必须在同一个事务里一起保持正确」：订单总金额必须和订单行金额之和一致，这条不变式决定了 OrderLine 应该在 Order 聚合内部，而客户信息只需要引用客户 ID，不必打包进同一个聚合。

5.4 仓储 (Repository)

对聚合根提供「类似集合」的持久化抽象：get / save / 按条件查找。
实现放基础设施；领域只依赖接口（与整洁架构一致）。

from abc import ABC, abstractmethod


class OrderRepository(ABC):
    @abstractmethod
    def get(self, order_id: str) -> "Order": ...

    @abstractmethod
    def save(self, order: "Order") -> None: ...

5.5 其他常见战术概念（点到为止）

概念 一句话
领域服务 不适合放进实体/值对象的领域操作
应用服务 / 用例 事务边界、权限、编排，薄于领域
领域事件 聚合内重要事实，供下游集成
工厂 创建复杂聚合，保证一开始就满足不变式

六、与微服务拆分的关系

DDD 微服务
限界上下文 服务边界的首选对齐单位
上下文映射 服务间集成方式（同步 API、事件、ACL）
每上下文自己的模型 每服务自己的数据与 Schema

注意：

  • 不是「一个实体一个服务」
  • 不是「有了 DDD 就必须微服务」——模块化单体里同样需要限界上下文
  • 上下文很大时，服务可以仍是一个；上下文稳定后再拆进程

先有清晰上下文与统一语言，再决定是否独立部署。

七、与整洁架构如何放一起

  Interface Adapters / Frameworks
            │
            ▼
     Application (Use Cases)     ← 可对应应用服务
            │
            ▼
     Domain (Entities, VOs,      ← DDD 战术模型住这里
             Aggregates, Domain Services)

DDD 管「模型长什么样」；整洁架构管「依赖往哪指」。二者经常一起用。

八、何时使用 / 口诀

适合 不太适合
规则多、术语乱、长期演进 报表拼装、简单后台
多团队易误解同一名词 一两周的小工具
准备模块化或拆服务 无业务专家可对话

统一语言定术语，限界上下文定边界；
实体有身份值对象不可变，聚合保一致仓储存根。
