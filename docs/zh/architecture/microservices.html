微服务架构 · Microservices Architecture

微服务 = 独立部署的服务单元（常以容器为载体）：SOA 的轻量化演进。解耦与扩容换来的是分布式复杂度。

一、从单体到微服务（通俗演进）

好比一家公司从「一人身兼数职的夫妻店」，长成「各部门独立、甚至各开分公司的大集团」：规模变大后，职责必须拆给不同的人，拆出去的部门再各自独立运作、独立对外负责。

1.1 单体 (Monolith)

所有功能写在一起，单一部署单元。

痛点 表现
耦合 改 A 可能伤 B，整体难管
发布成本 改一行也要整包构建部署
协作模型 难以按功能独立开发测试，易被迫瀑布

1.2 面向服务架构 (SOA)

把大型单体拆成独立服务：后台持续运行的功能单元，经网络协议协作。

优点包括：功能单一好开发、可独立部署升级、易扩容、故障隔离较好、可跨语言。传统 SOA 往往「一服务一机」，环境重。

1.3 容器化之后：微服务

Docker 等容器让「每个服务一个轻量运行环境」成为日常：不必新开物理机，新建容器（进程）即可。

通俗说法：微服务 ≈ 采用容器技术的 SOA——服务仍是功能单元，但实现轻量，故称「微」。

一个微服务是一个独立进程：可本机、可他机、可云函数。标准化镜像让「在哪跑都一样」，也催生大量可组合的云服务。

Monolith          SOA                Microservices

二、核心定义（架构模式视角）

  • 每个服务是独立部署单元 (separately deployed unit)
  • 服务分布式、互相解耦
  • 通过远程协议通信（REST、消息、gRPC 等）

「微」不指代码行数少，而指可独立发布的边界足够小且内聚。

三、三种实现模式

Richards / 阮文归纳的三种常见实现：

3.1 RESTful API 模式

服务以 API 形式对外提供能力；公有云 API、BaaS 等多属此类。

  • 同步、资源导向、易被多客户端复用
  • 需认真做版本、限流、鉴权（见 RESTful）

3.2 RESTful 应用模式

经传统网络/应用协议暴露；背后常是功能较完整的应用，常见于企业内部系统集成。

  • 比「纯资源 API」更接近「可部署的业务应用」
  • 边界可能比公有 API 服务更粗

3.3 集中消息模式 (Message Broker)

经消息代理通信：队列、负载均衡、统一日志与异常处理等可集中。

优点 风险
异步解耦、削峰 代理成单点 → 常需集群
横切能力集中 拓扑与运维复杂度上升

实践中常与事件驱动结合：同步用 API，跨服务副作用用消息。

                 └──────┬──────┘
          ┌─────────────┼─────────────┐
          ▼             ▼             ▼

         │            │            │
         ▼            ▼            ▼
      DB_order     DB_inv        DB_user   ← 理想：每服务自有数据

数据归属：尽量「服务拥有自己的数据」；多服务直连同一库会把微服务打回「分布式单体」。

反例：假设 Order 服务和 Inventory 服务表面上是两个独立部署的进程，但都直接连着同一张 inventory 表——Order 服务下单时自己写 SQL 扣库存，Inventory 服务也有自己的扣库存逻辑。某天 Inventory 团队想给这张表加一个字段、调整一下库存计算规则，却发现 Order 服务里也有一段直接依赖这张表结构的代码，必须两个团队一起改、一起测、一起上线——名义上是两个服务，实际发布节奏完全绑死，这就是「分布式单体」：拆开了进程，却没拆开耦合。

五、优点

  1. 扩展性好，服务间低耦合，可按热点服务扩容
  2. 易部署：从单一部署单元拆成多个可部署单元
  3. 易开发与持续交付：可并行开发、独立上线、不停机升级（配合良好流水线）
  4. 易测：可对单服务做契约测试与独立测试

六、缺点与陷阱

6.1 拆分过细

强调独立与低耦合时，服务可能拆得过碎：依赖图凌乱、调用链过长、性能与运维变差。

例如把「下单」这一个动作拆成 CreateOrder、ValidateOrder、PriceOrder、ReserveInventory、ShipOrder 五个独立微服务，一次用户下单就要串行发起五次网络调用；只要其中一次因为网络抖动多花 200ms，或者某个服务临时扩容不及时排队变长，端到端延迟就会从单体里的几十毫秒暴涨到接近一秒，而且任何一次调用失败都要考虑重试与补偿——排查一次下单失败要翻五个服务的日志，比在单体里定位一个函数调用麻烦得多。

6.2 服务间通信成本

一旦服务 A 依赖 B，架构复杂度陡增。对通用 Utility：

  • 有人选择复制小段代码到各服务（冗余换简单）
  • 有人抽共享库（版本地狱风险）
  • 有人做成平台服务（又引入运行时耦合）

无银弹，按变更频率与稳定性权衡。

6.3 分布式事务

跨服务难以像单体一样本地事务回滚。需 Saga、事务消息、出箱模式、幂等等——最终一致是常态，强一致要额外设计。

例如下单需要依次调用「扣库存」「扣款」两个服务：在单体里这是一个数据库事务，扣款失败直接整体回滚，库存自动恢复。但拆成两个服务后，如果库存已经扣成功、扣款却因为余额不足失败，系统不能像本地事务那样简单回滚——必须显式发一个「扣款失败，请恢复库存」的补偿消息给库存服务，由库存服务自己执行「加回库存」的操作。这一串「正向操作 + 失败后补偿」的编排就是 Saga；一旦某个补偿步骤也失败或者消息重复投递，还需要额外的重试与幂等设计，否则库存可能被恢复两次。

6.4 运维与可观测性

日志、链路追踪、配置、密钥、服务发现、熔断限流——成本从「应用内」外移到「平台」。

七、服务边界与 DDD（点到为止）

拆服务最难的是切哪里，不是用什么框架。

  • 限界上下文 (Bounded Context) 是天然候选边界：同一套统一语言、同一模型含义一致的范围
  • 上下文之间用显式翻译/防腐，而不是共享巨大 ORM 模型
  • 详见 DDD 入门

口诀级提醒：先定领域边界，再定进程边界。

八、何时使用

适合 不太适合
多团队需独立发布节奏 小团队、域未稳定
不同模块负载差异极大 尚无自动化部署与监控
需按业务能力扩容/隔离故障 「为了微服务而微服务」
组织已具备平台工程能力 强一致交易且无分布式经验

更稳妥路径：模块化单体 → 看清边界与痛点 → 再拆高频变化模块。

九、口诀

独立部署才叫服务，容器让 SOA 变轻；
拆分宁粗勿过碎，事务一致要想清。
