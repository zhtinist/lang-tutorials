# Internal Implementation of the Collections Framework

In C++ you use `std::vector`, `std::list`, `std::unordered_map`, `std::map`. Java's collections framework (the Java Collections Framework, JCF) corresponds to these one-to-one conceptually, but the **internal implementation details, memory layout, growth strategy, and iterator semantics** all deserve to be pried apart one by one. **This chapter starts with "how to use it" — the basic operations of arrays and a handful of the most common collections, no underlying knowledge required to get going; then it moves on to "underlying implementation vs. C" — what each container actually lays out on the heap, whether it copies values or references, and where the complexity comes from.** If you just want to know how to use them, reading the first half of each section is enough; if you want to understand performance and memory, keep reading into the second half.

For the basics of "reference vs value," see [“Variables and the Memory Model”](03-变量与内存模型.md); for object-header layout, see [“Data Types and Memory Layout”](04-数据类型.md).

---

## Get it working first: arrays and collections at a glance

Before diving into how Java works under the hood, get acquainted with how a few of the most common "containers" are declared and accessed. The code below doesn't touch on any internals — just copy it and run it:

```java
import java.util.*;

public class QuickStart {
    public static void main(String[] args) {
        // arrays: declare and access
        int[] nums = new int[3];
        nums[0] = 10;
        nums[1] = 20;
        nums[2] = 30;
        System.out.println(nums[1]);     // 20
        System.out.println(nums.length); // 3

        // ArrayList: an auto-growing "list"
        ArrayList<String> names = new ArrayList<>();
        names.add("Alice");
        names.add("Bob");
        System.out.println(names.get(0)); // Alice
        System.out.println(names.size()); // 2

        // HashMap: key-value pairs
        HashMap<String, Integer> ages = new HashMap<>();
        ages.put("Alice", 30);
        ages.put("Bob", 25);
        System.out.println(ages.get("Alice")); // 30

        // enhanced for loop: use this to iterate over any collection
        for (String name : names) {
            System.out.println("name: " + name);
        }
    }
}
```

```
20
3
Alice
2
30
name: Alice
name: Bob
```

A few impressions worth keeping for now:

- An array `int[]` has a fixed length once declared — after `new int[3]` its size can never change; indices start at 0, and going out of bounds throws an error (it won't silently read garbage memory the way C does).
- `ArrayList` is a list that grows automatically: `add` appends, `get(i)` fetches by index, and `size()` gives the current element count.
- `HashMap` stores a "key -> value" mapping: `put` writes, `get` reads by key.
- To iterate over any collection (arrays, `ArrayList`, `HashMap`'s key-value pairs, …) you can use the enhanced for loop `for (elementType var : collection)` — no manual indexing required.

That's enough to start writing code. Now let's go under the hood — what these containers actually look like in memory, and how they compare to their C/C++ counterparts, where they're fast and where they're slow.

## Going deeper: internal implementation of the collections framework

From here on we go under the hood. The core question for this chapter: what does each container actually lay out on the heap, does it copy values or references, and where does the complexity come from?

## 1. Arrays: the bedrock of everything

### How to use it

An array stores a group of elements of the **same type**. You declare one with `type[]` and create it with `new type[length]`; once the length is set, it can never change:

```java
int[] a = new int[4];       // 4 ints, all initialized to 0
a[0] = 100;
a[3] = 400;
System.out.println(a[0]);      // 100
System.out.println(a.length);  // 4 (note: it's a field, no parentheses — not a.length())

String[] s = new String[3]; // 3 reference slots, all initialized to null
s[0] = "hi";
System.out.println(s[1]);      // null (not assigned yet)
```

Accessing out of bounds throws an exception immediately; it won't silently read or write other memory the way C does:

```java
int[] b = new int[2];
System.out.println(b[5]); // throws ArrayIndexOutOfBoundsException, terminating the program (unless you catch it)
```

You can also give all the elements directly at declaration time:

```java
int[] c = {1, 2, 3};        // length is automatically set to 3
```

### Underlying implementation vs. C

A Java array is a **true object**, allocated on the heap, unlike C where it's just a chunk of raw memory + pointer arithmetic.

```java
int[] a = new int[4];        // heap allocation: object header + length field + 4 contiguous ints
String[] s = new String[3];  // on the heap: 3 contiguous "reference slots", initially null
```

#### Memory layout (HotSpot, 64-bit, compressed pointers)

```
int[4] on the heap:
┌──────────┬──────────┬────────┬────┬────┬────┬────┐
│ mark word│ klass ptr│ length │ [0]│ [1]│ [2]│ [3]│
│  8 bytes │  4 bytes │ 4 bytes│int │int │int │int │
└──────────┴──────────┴────────┴────┴────┴────┴────┘
       object header: 16 bytes total    elements stored contiguously, cache-friendly
```

The key differences from C:

| Feature | C/C++ array | Java array |
|------|-----------|-----------|
| Nature | raw memory, `a[i]` = `*(a+i)` | heap object, includes a length field |
| Length | compile-time constant or stored separately | `a.length` carried along |
| Bounds checking | none, out-of-bounds is UB | checked on every access, throws `ArrayIndexOutOfBoundsException` |
| Storing objects | can store object values (contiguous) | can only store references (`Object[]` is an array of references) |
| Covariance | none | **covariant**: `Object[] o = new String[3];` is legal |

#### The pitfall of array covariance

Java's array covariance means `String[]` IS-A `Object[]`; this passes at compile time, but every write at runtime requires a type check:

```java
Object[] arr = new String[3];
arr[0] = "ok";           // fine
arr[1] = Integer.valueOf(1); // compiles! throws ArrayStoreException at runtime
```

C++ templates are invariant; `vector<string>` has nothing to do with `vector<Object>`. This Java design is historical baggage (before generics existed, it let `Arrays.sort(Object[])` work), at the cost of a runtime store check. Generic collections (below) are, by contrast, invariant.

#### Bounds-check cost and its elimination

Every `a[i]` inserts a `0 <= i < a.length` check. The JIT has **Range Check Elimination**: in a loop like `for (int i=0;i<a.length;i++)` that's provably safe, C2 hoists the check out of the loop or removes it, bringing performance close to C. So there's a cost at the specification level, but it's often optimized away at the implementation level.

## 2. A panorama of the Collection hierarchy

Java organizes its non-array containers into one big inheritance diagram. You don't need to memorize it — it's just a reference "map": whenever you want to know which category a container belongs to, or what sibling implementations exist, come back and take a look:

```
                 Iterable
                     │
                Collection ───────────────┐
                 │      │                  │
               List    Set               Queue
              /  │  \    │  \              │   \
     ArrayList  │  Vector│   \          ArrayDeque Deque
     LinkedList │       HashSet         PriorityQueue
     (LinkedList is also Queue/Deque) LinkedHashSet
                          TreeSet (SortedSet)

  Map (not part of Collection, a separate branch)
   │    │        │
 HashMap TreeMap LinkedHashMap
 (SortedMap = TreeMap)
```

Key point: `Map` does **not** extend `Collection`; it's a parallel branch. `List`/`Set`/`Queue` are all `Collection`s.

## 3. ArrayList vs LinkedList

### How to use it

`ArrayList` is the most commonly used "list"; it supports indexed access, appending, and removal:

```java
List<String> list = new ArrayList<>();
list.add("a");
list.add("b");
list.add("c");
System.out.println(list.get(1));   // b
list.remove("b");                  // removes by value (there's also an overload remove(int) that removes by index)
System.out.println(list);          // [a, c]
System.out.println(list.size());   // 2
```

`LinkedList` is used almost identically to `ArrayList` (both implement the `List` interface), but additionally offers methods like `addFirst`/`addLast` for operating on both ends:

```java
LinkedList<String> linked = new LinkedList<>();
linked.add("x");
linked.addFirst("w");
System.out.println(linked);        // [w, x]
```

As a beginner, just defaulting to `ArrayList` is good enough for now — the rest of this section explains why.

### Underlying implementation vs. C

#### ArrayList: a dynamic array

Internally it's just an `Object[] elementData` + an `int size`:

```java
// simplified JDK source structure
transient Object[] elementData; // the array that actually stores elements
private int size;               // logical size (<= elementData.length capacity)
```

Memory diagram:

```
ArrayList object (heap)     elementData array (heap)
┌───────────────┐          ┌────┬────┬────┬────┬────┬────┐
│ elementData ──┼────────▶ │ e0 │ e1 │ e2 │null│null│null│
│ size = 3      │          └────┴────┴────┴────┴────┴────┘
└───────────────┘           capacity = 6, size = 3
```

Note that what's stored are **references** (`Object[]`), with the objects themselves scattered across the heap — unlike C++ `vector<Point>`, which inlines the Point values contiguously. So iterating a Java `ArrayList<Point>` means "contiguous array → jumping to scattered heap locations one by one," with worse cache locality than a value-semantics C++ vector. To store primitives contiguously, you need a native `int[]` or a third-party primitive collection.

**Growth**: default capacity 10; when full it grows by **1.5×** (`newCap = oldCap + (oldCap >> 1)`), with `Arrays.copyOf` allocating a new array and copying the references (a shallow copy — only the pointer values are copied, not the objects themselves).

```
size=10 is full, adding the 11th:
new capacity = 10 + 5 = 15, allocate Object[15], copy 10 references over, old array awaits GC
```

Compare with C++ `std::vector`'s common 2× growth (libstdc++ uses 2x, MSVC uses 1.5x). Amortized O(1) append.

Peeking at the real length of the underlying `elementData` array via reflection shows this 1.5× growth curve directly (capacity is an implementation detail, so reflection needs `--add-opens`):

```java
import java.lang.reflect.*;
import java.util.*;
public class GrowDemo {
    static int cap(ArrayList<?> list) throws Exception {
        Field f = ArrayList.class.getDeclaredField("elementData");
        f.setAccessible(true);
        return ((Object[]) f.get(list)).length;
    }
    public static void main(String[] args) throws Exception {
        ArrayList<Integer> list = new ArrayList<>();
        System.out.println("initial capacity = " + cap(list));
        int last = -1;
        for (int i = 0; i < 40; i++) {
            list.add(i);
            int c = cap(list);
            if (c != last) { System.out.println("size=" + list.size() + " -> capacity=" + c); last = c; }
        }
    }
}
// Run: java --add-opens java.base/java.util=ALL-UNNAMED GrowDemo.java
// initial capacity = 0     <- empty ctor allocates lazily; the length-10 array is built on first add
// size=1  -> capacity=10
// size=11 -> capacity=15   <- 10 + (10>>1) = 15
// size=16 -> capacity=22   <- 15 + (15>>1) = 22
// size=23 -> capacity=33
// size=34 -> capacity=49
```

#### LinkedList: a doubly linked list

```java
private static class Node<E> {
    E item;
    Node<E> next;
    Node<E> prev;
}
```

Each node is an independent heap object: 16-byte object header + 3 references (item/next/prev, 4 bytes each with compressed pointers) ≈ 32 bytes/node, a huge overhead to store one element. Memory diagram:

```
first ─▶┌────┬────┬────┐   ┌────┬────┬────┐   ┌────┬────┬────┐◀─ last
        │prev│item│next┼──▶│prev│item│next┼──▶│prev│item│next│
        │null│ e0 │  ●─┘   │ ●  │ e1 │ ●─┘   │ ●  │ e2 │null│
        └────┴────┴────┘◀──┴────┴────┴────┘◀──┴────┴────┴────┘
```

Corresponds to C++ `std::list`. Nodes are scattered across the heap, so traversal is cache-unfriendly. `get(i)` is O(n) (walking from whichever end is closer). **In practice `ArrayList` is almost always better**; even for frequent head insertion, `ArrayDeque` is faster than `LinkedList`. `LinkedList`'s only advantage is O(1) removal when you hold a node reference, but the collection API generally doesn't give you the node.

#### Complexity comparison

| Operation | ArrayList | LinkedList |
|------|-----------|-----------|
| `get(i)` | O(1) | O(n) |
| `add` (tail) | amortized O(1) | O(1) |
| `add(i, e)` (middle) | O(n) (shift) | O(n) (locate) + O(1) |
| Head insertion | O(n) | O(1) |
| `contains` | O(n) | O(n) |
| Memory/element | 1 reference + capacity slack | ~32-byte node |

## 4. HashMap: the main event

### How to use it

`HashMap` stores a "key -> value" mapping: `put` writes, `get` looks up by key, `containsKey` checks whether a key exists; to iterate, use `entrySet()` to get the key-value pairs:

```java
Map<String, Integer> scores = new HashMap<>();
scores.put("Alice", 90);
scores.put("Bob", 85);
System.out.println(scores.get("Alice"));           // 90
System.out.println(scores.containsKey("Carol"));   // false
for (Map.Entry<String, Integer> e : scores.entrySet()) {
    System.out.println(e.getKey() + " -> " + e.getValue());
}
```

Note: `HashMap` does not guarantee iteration order (the next section shows the printed order actually differing from insertion order); for a fixed order, see `LinkedHashMap`/`TreeMap` in the next section.

### Underlying implementation vs. C

Corresponds to C++ `std::unordered_map`, but the implementation details differ greatly.

#### Structure: a bucket array + linked lists/red-black trees

```java
transient Node<K,V>[] table;  // the bucket array, length always a power of 2
static class Node<K,V> {
    final int hash;   // the cached, perturbed hash
    final K key;
    V value;
    Node<K,V> next;   // linked-list pointer (chaining to resolve collisions)
}
```

Memory diagram (chaining on collision):

```
table (Node[16])
┌────┐
│ 0  │──▶ null
│ 1  │──▶ [h,k1,v1] ──▶ [h,k9,v9] ──▶ null   (bucket 1 has two colliding keys)
│ 2  │──▶ null
│... │
│ 5  │──▶ [h,k5,v5] ──▶ null
└────┘
```

#### The hash-perturbation function

The bucket index = `hash & (n-1)` (n being a power of 2, equivalent to modulo but faster). If you used `key.hashCode()` directly, the high bits wouldn't participate (because `& (n-1)` keeps only the low bits), so keys differing in the high bits but equal in the low bits would collide. Perturbation XORs the high 16 bits into the low bits:

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

#### treeify when collisions intensify

When a bucket's chain length is **≥ 8** and the **table length is ≥ 64**, the chain is converted into a **red-black tree**, dropping that bucket's worst-case lookup from O(n) to O(log n) and guarding against hash-collision attacks (DoS). When the tree's node count falls back to **≤ 6**, it degrades back into a chain (the 6/8 hysteresis interval avoids frequent conversion).

```
chain length >= 8 and table.length >= 64:
bucket ──▶ chain ─treeify─▶ red-black tree (TreeNode)
if the table hasn't reached 64 yet, prefer resizing over treeifying
```

#### Load factor and resize rehashing

- **Load factor 0.75**: when `size > capacity * 0.75`, resize to **double**. 0.75 is a space/time trade-off (per Poisson-distribution analysis, balancing collision probability against memory waste).
- **Resize rehashing**: allocate a bucket array 2× larger and rehash every element. Because the capacity is a power of 2 and doubles, an element's new position is one of only two: `original position` or `original position + old capacity`. JDK 8 exploits this property to avoid recomputing the hash, looking only at whether that one bit `hash & oldCap` is 0 or 1, splitting the old chain into two chains (lo/hi) placed at the two new-table positions. O(n) but with a small constant.

```
old table.length=16, resizing to 32:
a bucket's chain is split by (hash & 16) into two → bucket j and bucket j+16
```

#### The key contract: equals/hashCode

A `HashMap` locates in two steps: first find the bucket by hash, then compare within the bucket one by one using `equals`. So a custom key **must** correctly override both `equals` and `hashCode`; otherwise equal objects scattered into different buckets can never be found again. See [the equals/hashCode contract in “Object Orientation and the Type System”](08-面向对象与类型系统.md). Same in C++: `unordered_map` needs a `std::hash` specialization + `operator==`.

#### Complexity

| Operation | Average | Worst (after treeify) |
|------|------|--------------|
| `get`/`put`/`remove` | O(1) | O(log n) |
| Traversal | O(capacity + size) | same |

Note: `HashMap` is **not thread-safe**; under concurrency, resizing in JDK 7 could form a cycle and loop forever (JDK 8 fixed this but data can still be lost). For concurrency, use `ConcurrentHashMap`, see [“Concurrency Model”](10-并发模型.md).

## 5. LinkedHashMap / TreeMap

### How to use it

`LinkedHashMap` and `TreeMap` have `put`/`get` just like `HashMap`; the only difference is **iteration order**:

```java
Map<String, Integer> linked = new LinkedHashMap<>();
linked.put("banana", 1);
linked.put("apple", 2);
System.out.println(linked.get("apple")); // 2
System.out.println(linked);              // {banana=1, apple=2}

Map<String, Integer> tree = new TreeMap<>();
tree.put("banana", 1);
tree.put("apple", 2);
System.out.println(tree);                // {apple=2, banana=1}
```

How to choose among the three Maps: use `HashMap` if you don't care about order (it's the fastest); use `LinkedHashMap` if you want to iterate in insertion order; use `TreeMap` if you want to iterate sorted by key.

### Underlying implementation vs. C

#### LinkedHashMap

On top of `HashMap`'s Node it adds `before`/`after` doubly-linked pointers, maintaining **insertion order** (or access order with `accessOrder=true`, which can implement an LRU cache). Hash lookups are still O(1); it additionally maintains a linked list threading through all nodes.

```
HashMap's bucket structure + a doubly-linked list threading through all nodes (recording order)
head ──▶ node ──▶ node ──▶ node ──▶ tail
```

#### TreeMap

Based on a **red-black tree** (a self-balancing BST), corresponding to C++ `std::map`. Sorted by key (natural order or a `Comparator`). All operations are O(log n), providing ordered views like `firstKey`/`lastKey`/`floorKey`/`ceilingKey`/`subMap`. Keys must be comparable (`Comparable`, or pass a `Comparator`), see [“Standard Library Algorithms”](11-标准库算法.md).

The iteration order of the three Maps makes the difference plain at a glance: `HashMap` is determined by bucket position (unspecified, looks "random"), `LinkedHashMap` preserves insertion order, and `TreeMap` sorts by key:

```java
import java.util.*;
public class MapOrder {
    static void fill(Map<String,Integer> m) {
        for (String k : new String[]{"banana","apple","cherry","date"}) m.put(k, k.length());
    }
    public static void main(String[] args) {
        Map<String,Integer> hash = new HashMap<>();
        Map<String,Integer> linked = new LinkedHashMap<>();
        Map<String,Integer> tree = new TreeMap<>();
        fill(hash); fill(linked); fill(tree);
        System.out.println("HashMap:       " + hash.keySet());
        System.out.println("LinkedHashMap: " + linked.keySet());
        System.out.println("TreeMap:       " + tree.keySet());
    }
}
// HashMap:       [banana, date, apple, cherry]   <- bucket order, unspecified (never rely on it)
// LinkedHashMap: [banana, apple, cherry, date]   <- insertion order
// TreeMap:       [apple, banana, cherry, date]   <- lexicographic order
```

## 6. HashSet / ArrayDeque

### How to use it

`HashSet` is a collection that "only stores keys, no values"; `add` inserts, `contains` checks membership, and duplicate elements are automatically ignored:

```java
Set<String> set = new HashSet<>();
set.add("a");
set.add("b");
set.add("a");                          // duplicate, ignored
System.out.println(set.size());        // 2
System.out.println(set.contains("a"));  // true
```

`ArrayDeque` is a double-ended queue: use it as a stack with `push`/`pop` (LIFO), or as a queue with `offer`/`poll` (FIFO):

```java
ArrayDeque<Integer> stack = new ArrayDeque<>();
stack.push(1);
stack.push(2);
System.out.println(stack.pop());    // 2 — LIFO

ArrayDeque<Integer> queue = new ArrayDeque<>();
queue.offer(1);
queue.offer(2);
System.out.println(queue.poll());   // 1 — FIFO
```

### Underlying implementation vs. C

- **HashSet**: internally just a `HashMap`, with the element as the key and the value being a shared dummy `Object`. So everything about `HashSet` equals the keySet of `HashMap`. Corresponds to C++ `unordered_set`.
- **LinkedHashSet**: internally a `LinkedHashMap`, preserving insertion order.
- **TreeSet**: internally a `TreeMap`, sorted, corresponding to `std::set`.
- **ArrayDeque**: a double-ended queue implemented with a **circular array**; the two indices `head`/`tail` move around a ring buffer, doubling on growth. As a stack or a queue it's faster than both `Stack` (obsolete, extends `Vector`, carries synchronization) and `LinkedList`. **The preferred stack/queue implementation**. It disallows null (null is used to mark an empty slot).

```
ArrayDeque circular array:
┌───┬───┬───┬───┬───┬───┐
│ e │ e │   │   │ e │ e │   head=4, tail=2 (wraps around)
└───┴───┴───┴───┴───┴───┘
```

## 7. Generic collections and the impact of type erasure

### How to use it

When declaring a collection, use angle brackets `<Type>` to specify "what it's allowed to hold"; the compiler catches type errors for you at compile time, and you don't need a cast to get elements back out:

```java
ArrayList<String> names = new ArrayList<>();
names.add("Alice");
// names.add(42);       // compile error: type mismatch, caught by the compiler before you even run it
String first = names.get(0); // no cast needed
System.out.println(first);   // Alice
```

By contrast, if you omit the angle brackets (the old-style `ArrayList names = new ArrayList();`), `get(0)` returns an `Object`, which you'd have to cast manually — and it might still blow up at runtime. So in modern Java, always include the type inside the angle brackets.

### Underlying implementation vs. C

`ArrayList<String>` is just `ArrayList` at runtime, with an `Object[]` inside; generics are only checked at compile time, with casts inserted automatically. This is why you can't `new T[]`, and why a collection always stores references (primitives get auto-boxed). The erasure mechanism is detailed in [“Generics and Type Erasure”](08-面向对象与类型系统.md).

Consequence: each element of an `ArrayList<Integer>` is an `Integer` object on the heap + a reference in the array, so the memory and boxing/unboxing overhead is far greater than C++ `vector<int>`. In performance-sensitive code, use native arrays or third-party primitive collections (Eclipse Collections, fastutil).

## 8. Iterators and fail-fast

### How to use it

Using the enhanced for loop to iterate over a collection is the most common approach; under the hood, the compiler is actually calling an `Iterator` for you:

```java
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String x : list) {
    System.out.println(x);
}
```

**There's a hard rule here**: don't call the collection's own `add`/`remove` directly to modify it while iterating (regardless of whether it's the element you meant to remove) — doing so will most likely throw an exception. To remove elements while iterating, use `removeIf`, or explicitly grab the `Iterator` and call its own `remove()`. The "Underlying implementation vs. C" part below explains why.

### Underlying implementation vs. C

A collection internally maintains a `modCount` (a count of structural modifications). When an iterator is created it records `expectedModCount`, and validates it on each `next()`; if the collection is modified through a non-iterator path in the meantime, it throws `ConcurrentModificationException`.

```java
List<Integer> list = new ArrayList<>(List.of(1, 2, 3));
for (Integer x : list) {      // the enhanced for uses an Iterator under the hood
    if (x == 2) list.remove(x); // structural modification → modCount changes → the next next() throws
}
// correct approach: use iterator.remove() or list.removeIf(x -> x == 2)
```

Running it for real shows the exception is indeed thrown, while `removeIf` is perfectly safe:

```java
import java.util.*;
public class FailFast {
    public static void main(String[] args) {
        List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4));
        try {
            for (Integer x : list) {          // the enhanced for uses an Iterator
                if (x == 2) list.remove(x);   // structural change bumps modCount
            }
        } catch (ConcurrentModificationException e) {
            System.out.println("caught: " + e.getClass().getSimpleName());
        }
        list.removeIf(x -> x == 2);           // correct: goes through the iterator, no CME
        System.out.println("after removeIf: " + list);
    }
}
// caught: ConcurrentModificationException
// after removeIf: [1, 3, 4]
```

fail-fast is **best-effort** error detection (not guaranteed to trigger), not concurrency protection; for real concurrency use `CopyOnWriteArrayList`/`ConcurrentHashMap` (weakly consistent iterators that don't throw this exception). The C++ STL counterpart is "iterator invalidation" — but the STL leaves it UB, whereas Java proactively throws to help you catch it early.

## Summary

- **At the usage level**: declare a fixed-length array with `int[] a = new int[n]`, access it with `a[i]`, and get its length with `a.length`; use `add`/`get`/`remove`/`size` on `ArrayList`/`LinkedList`; use `put`/`get`/`containsKey` on `HashMap`/`LinkedHashMap`/`TreeMap`, paired with `entrySet()` for iteration; use `add`/`contains` on `HashSet`; use `push`/`pop` on `ArrayDeque` as a stack and `offer`/`poll` as a queue; always include the generic type in angle brackets when declaring a collection; don't structurally modify a collection directly while iterating — remove elements with `removeIf` or `Iterator.remove()` instead.
- **At the underlying level**: an array is a heap object carrying a length field, throwing on out-of-bounds access rather than invoking UB, though covariance leaves a runtime store-check pitfall; `ArrayList` is a resizable reference array (growing by 1.5×), `LinkedList` is a doubly linked list (high per-node overhead, cache-unfriendly traversal), and the complexity differences between them are stark; `HashMap` is a bucket array plus linked lists/red-black trees, propped up to amortized O(1) by hash perturbation, a 0.75 load factor, and JDK 8's rehashing optimization; `LinkedHashMap`/`TreeMap` maintain order via a doubly linked list and a red-black tree, respectively; `HashSet`/`TreeSet`/`LinkedHashSet` are essentially thin wrappers around their corresponding Maps; generics are erased at runtime, and primitives in collections are always boxed; iterators use `modCount` for fail-fast detection, which is best-effort, not concurrency protection.
