# Object Orientation and the Type System

Maybe you've never written a `class` in C/C++, or you have but never dug into "what an object actually looks like in memory." **This chapter starts with "how to use it"** — how to define a class, `new` an object, write inheritance and interfaces, and declare the simplest generics and lambdas, no low-level knowledge required to get going. **Then it moves on to "underlying implementation vs. C"** — what's actually sitting on the heap for an object, how virtual methods get dispatched, and what's left of generics at runtime. If you just want to use the language, reading the first half of each section is enough; if you want to understand the machinery, keep reading into the second half.

This is the most OOP-concept-dense chapter in the book, touching on classes, inheritance, polymorphism, generics, records, sealed types, enums, and lambdas — a dozen-odd topics. Don't expect to read it once and remember everything. First get the "how to use it" parts strung together so you can write basic classes and interfaces, then come back section by section and chew through "underlying implementation vs. C."

## Get it working first: a quick tour of classes and objects

### Define a class, `new` an object

The biggest difference between a Java class and a C `struct`: a class can hold functions directly (called "methods"), and every object created with `new` always lives on the heap — the variable just holds a reference to it (details in [“Variables and the Memory Model”](03-变量与内存模型.md)).

```java
class Point {
    int x, y;                          // fields (member variables)
    Point(int x, int y) {              // constructor: same name as the class, no return type
        this.x = x;                    // this refers to the current object, disambiguating field from parameter
        this.y = y;
    }
    int sum() {                        // a method
        return x + y;
    }
}

public class PointBasics {
    public static void main(String[] args) {
        Point p = new Point(3, 4);     // new creates an instance
        System.out.println(p.sum());  // call a method: 7
        System.out.println(p.x);      // access a field directly: 3
    }
}
```

```
7
3
```

`class Point { ... }` is just the "blueprint"; `new Point(3, 4)` is what actually builds an object on the heap, and `p` holds a reference to that object. The same blueprint can produce countless objects, each with its own independent fields.

### The simplest inheritance and interfaces

`extends` means "inherit from a class" (single inheritance — only one allowed), and `implements` means "implement one or more interfaces":

```java
interface Greeter {
    String greet();                    // interface method: signature only, no body
}

class Animal {
    String name = "animal";
    String speak() {
        return name + " makes a sound";
    }
}

class Dog extends Animal implements Greeter {
    Dog() { name = "dog"; }
    @Override public String greet() { return "woof from " + name; }
}

public class InheritBasics {
    public static void main(String[] args) {
        Dog d = new Dog();
        System.out.println(d.speak());   // inherited method: dog makes a sound
        System.out.println(d.greet());   // interface method: woof from dog
    }
}
```

```
dog makes a sound
woof from dog
```

`Dog` gets `Animal`'s fields and methods for free (inheritance), but it must also write out the `greet()` that `Greeter` promises (implementing the interface) — `@Override` is a marker for the compiler and for readers, saying "this overrides a method from a parent class or interface"; misspell the method name and the compiler catches it immediately.

### The simplest generic: `Box<T>`

The `<T>` after the class name means "this class can hold any type — the caller decides which one":

```java
class Box<T> {
    T value;
    Box(T value) { this.value = value; }
    T get() { return value; }
}

public class BoxBasics {
    public static void main(String[] args) {
        Box<String> b = new Box<>("hello");
        System.out.println(b.get());       // hello

        Box<Integer> n = new Box<>(42);
        System.out.println(n.get());       // 42
    }
}
```

```
hello
42
```

`Box<String>` and `Box<Integer>` are the same piece of code — `T` gets replaced with a concrete type wherever it's used, and the compiler checks that the types line up; you don't need to write a separate `Box` for every type.

### The simplest lambda

`Runnable` is an interface with just one method, `run()` (this kind of interface is called a "functional interface"); the lambda `() -> ...` writes an implementation for that method directly, without defining a whole class:

```java
public class LambdaBasics {
    public static void main(String[] args) {
        Runnable r = () -> System.out.println("running!");
        r.run();                          // called just like an ordinary method
    }
}
```

```
running!
```

That's enough to start reading most Java code. Now let's go underneath — what an object actually looks like on the heap, how virtual methods get dispatched, and what's left of generics at runtime.

## Going deeper: what an object actually is in memory

From here on we're talking about the underlying mechanics. Keep one thing in mind, since every section below relies on it: **Java objects are always allocated on the heap**, and every object begins with a hidden "object header," part of which points to the class-description structure it belongs to (in HotSpot this is called the `klass`) — this pointer plays much the same role as a vptr in C++: it decides which code to jump to when a virtual method is called. The variable itself doesn't hold the object, only this reference (details on value vs. reference semantics are in [“Variables and the Memory Model”](03-变量与内存模型.md); the exact byte layout of the object header is in [“Data Types”](04-数据类型.md)).

With that model in hand, we'll take things apart one at a time: initialization order, encapsulation, inheritance, polymorphism (the vtable), abstract classes vs. interfaces, `Object`, generic erasure, records, sealed types, enums, and inner classes and lambdas.

## 1. Classes, objects, and initialization order

### How to use it

A constructor can delegate to another constructor of the same class with `this(...)`, or explicitly call the superclass constructor with `super(...)` (if you don't write one, the compiler inserts a no-arg `super()` automatically):

```java
class Person {
    String name;
    int age;
    Person(String name) { this(name, 0); }          // this(...): delegate to another constructor of this class
    Person(String name, int age) { this.name = name; this.age = age; }
}
class Student extends Person {
    String school;
    Student(String name, int age, String school) {
        super(name, age);                            // super(...): call the superclass constructor
        this.school = school;
    }
}
public class ConstructorBasics {
    public static void main(String[] args) {
        Student s = new Student("Ann", 20, "MIT");
        System.out.println(s.name + " " + s.age + " " + s.school);  // Ann 20 MIT

        Person p = new Person("Bob");                // goes through this(...) delegation, age defaults to 0
        System.out.println(p.name + " " + p.age);     // Bob 0
    }
}
```

```
Ann 20 MIT
Bob 0
```

### Underlying implementation vs. C

All instance methods of a Java class are "virtual" by default (except `private`/`static`/`final`). When you `new` an object, the JVM executes in a fixed order:

```
1. On first use of the class: static fields to default zero values → static init blocks + static field assignments (in source order, once only)
2. Instance creation: allocate heap memory, all instance fields set to zero values
3. Superclass instance initialization (recursively: superclass fields → superclass instance blocks → superclass constructor body)
4. This class's instance field defaults → instance init blocks + field assignments (in source order)
5. This class's constructor body
```

```java
class Base {
    String x = init("Base.x");        // ③
    Base() { System.out.println("Base()"); }
    static String s = init("Base.s");  // ① at class load time
    static String init(String m){ System.out.println(m); return null; }
}
class Derived extends Base {
    String y = Base.init("Derived.y"); // ④
    Derived() { super(); System.out.println("Derived()"); } // super() is implicitly first
}
// new Derived() prints: Base.s (first time) → Base.x → Base() → Derived.y → Derived()
```

```
Base.s
Base.x
Base()
Derived.y
Derived()
```

Compared with C++: members are constructed in the initializer list in **declaration order**, base before derived — Java's order is similar, but there's no initializer-list syntax, and static initialization is lazy (triggered only on the first active use of the class, executed exactly once in a thread-safe manner guaranteed by the class loader).

The first line of a constructor must be `this(...)` or `super(...)`; if omitted, the compiler inserts a no-arg `super()`. **Calling a virtual method during construction is dangerous**: if a superclass constructor calls a method the subclass overrides, at that point the subclass's fields aren't yet initialized (still zero values), so it reads uninitialized state. C++ statically binds virtual calls during construction to the current class's version; **Java, by contrast, dynamically binds to the subclass's version**, which is more dangerous.

See the pitfall for yourself: the superclass constructor calls the virtual method `size()`, which dispatches to the subclass version, but at that point the subclass field `value` is still its zero value:

```java
public class CtorVirtual {
    static class Base {
        Base() { System.out.println("Base ctor sees size = " + size()); } // virtual call
        int size() { return 0; }
    }
    static class Derived extends Base {
        int value = 42;                 // only initialized after super() runs
        @Override int size() { return value; }
    }
    public static void main(String[] args) {
        Derived d = new Derived();
        System.out.println("after ctor  size = " + d.size());
    }
}
```

```
Base ctor sees size = 0    <- dynamically dispatched to Derived.size(), but value isn't set yet (zero)
after ctor  size = 42      <- only 42 once construction completes
```

## 2. Encapsulation

### How to use it

Access modifiers control who can access a field or method; the most common pattern is to make a field `private` and only expose getters/setters:

```java
class Account {
    private double balance;   // not directly accessible from outside

    Account(double initial) { balance = initial; }

    public double getBalance() {     // only readable through a method
        return balance;
    }
    public void deposit(double amount) {
        if (amount > 0) balance += amount;  // a chance to validate
    }
}

public class EncapBasics {
    public static void main(String[] args) {
        Account a = new Account(100);
        a.deposit(50);
        System.out.println(a.getBalance());  // 150.0
        // a.balance = -999;    // compile error: balance is private, not accessible from outside
    }
}
```

```
150.0
```

### Underlying implementation vs. C

Access modifiers from narrowest to widest: `private` < package-level (default, no keyword) < `protected` < `public`. Note that Java's `protected` also permits **same-package** access, a broader semantic than C++'s. There's no `friend`. The module system (JPMS) can encapsulate whole packages at a coarser granularity.

## 3. Inheritance: single inheritance + multiple interfaces

### How to use it

A class can `extends` only one superclass, but can `implements` any number of interfaces, separated by commas:

```java
abstract class AbstractBase {
    void base() { System.out.println("base"); }
}
class Service extends AbstractBase implements Runnable, AutoCloseable {
    @Override public void run() { System.out.println("running"); }
    @Override public void close() { System.out.println("closed"); }
}
public class ServiceCheck {
    public static void main(String[] args) {
        Service s = new Service();
        s.base();
        s.run();
        s.close();
    }
}
```

```
base
running
closed
```

### Underlying implementation vs. C

Java **classes can only inherit singly** (avoiding the complexity of C++'s diamond inheritance and virtual inheritance), but a class can implement any number of interfaces.

Interfaces once could only have abstract methods; since JDK 8 they can have **default methods** (with implementations) and **static methods**, and since JDK 9 they can have `private` methods. This brings the ability of "multiple inheritance of implementation," but fields remain constants only (`public static final`), so there's no state-diamond problem. If two interfaces have default methods with the same signature, the subclass must explicitly override to resolve the conflict, otherwise it's a compile error:

```java
interface A2 { default String who() { return "A"; } }
interface B2 { default String who() { return "B"; } }
class C2 implements A2, B2 { }   // compile error
```

```
error: types A2 and B2 are incompatible
  class C2 inherits unrelated defaults for who() from types A2 and B2
```

Use `InterfaceName.super.methodName()` to explicitly say which interface's default implementation to use, resolving the conflict:

```java
interface A { default String who() { return "A"; } }
interface B { default String who() { return "B"; } }
class C implements A, B {
    @Override public String who() { return A.super.who() + B.super.who(); } // explicitly resolves the conflict
}
```

```
AB
```

## 4. Polymorphism and the virtual method table (vtable)

### How to use it

Once a subclass overrides (`@Override`) a parent method, calling it through a reference of the parent's type still runs the subclass's version — this is "dynamic dispatch":

```java
class Bird {
    String sound() { return "..."; }
}
class Duck extends Bird {
    @Override String sound() { return "quack"; }
}

public class PolyBasics {
    public static void main(String[] args) {
        Bird b = new Duck();          // a parent-typed reference pointing at a subclass object
        System.out.println(b.sound()); // quack -- dispatched at runtime based on the actual object type, not the reference's declared type
    }
}
```

```
quack
```

### Underlying implementation vs. C

Java instance methods dispatch dynamically by default. HotSpot's implementation is highly similar to C++'s vtable:

```
Each class has a klass structure in the method area (Metaspace), containing a vtable:
  Klass(Animal)          Klass(Dog extends Animal)
  vtable:                vtable:
   [0] Object.method...    [0] Object.method... (inherited)
   [k] Animal::speak ─┐    [k] Dog::speak    ← overridden, same slot
                      │
object header's klass ptr ─┘ points to its own Klass
calling a.speak():
   1. take a's object-header klass ptr
   2. fetch the method address from the fixed vtable slot
   3. call   —— almost identical to C++
```

Differences:

- **Interface methods** can't use a fixed vtable slot (a class can implement multiple interfaces, so slots can't be unified across them); they use a separate **itable** (interface method table) + runtime lookup, slightly slower than the vtable. The JIT optimizes with an **inline cache**: it remembers the actual type from the last call, with near-zero overhead in the monomorphic case.
- Java's `final`/`private`/`static` methods are **statically bound** (invokespecial/invokestatic), with no virtual dispatch.
- The JIT also does **devirtualization**: if it can prove a call site has only one implementation (via CHA, class hierarchy analysis), it inlines directly, and deoptimizes if a new subclass gets loaded during a STW pause. C++ needs a manual `final` or link-time optimization to get a similar effect.

At the bytecode level there are four call instructions: `invokevirtual` (virtual), `invokeinterface` (interface), `invokespecial` (constructor/private/super), `invokestatic` (static), plus `invokedynamic` (lambdas, string concatenation). See [“Functions and the Call Stack”](06-函数与调用栈.md).

## 5. Abstract classes vs. interfaces

### How to use it

An `abstract class` can mix "abstract methods" (no body, subclasses must implement) with "concrete methods" (have an implementation, just inherit and use). Before Java 8, interfaces could only have abstract methods:

```java
abstract class Shape {
    abstract double area();          // abstract method, no body
    void describe() {                // concrete method, has an implementation
        System.out.println("area = " + area());
    }
}
class Circle extends Shape {
    double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
}

interface Movable {
    void move();                     // abstract method
}
class Car implements Movable {
    @Override public void move() { System.out.println("car moves"); }
}

public class AbstractVsInterfaceBasics {
    public static void main(String[] args) {
        new Circle(2).describe();
        new Car().move();
    }
}
```

```
area = 12.566370614359172
car moves
```

### Underlying implementation vs. C

| Dimension | Abstract class | Interface |
|------|--------|------|
| Instance fields | can have (stateful) | only `public static final` constants |
| Constructor | yes | no |
| Inheritance | single | multiple implementation |
| Methods | abstract + concrete | abstract + default + static + private |
| Semantics | "what it is" (is-a) | "what it can do" (can-do, capability) |

Rule of thumb: use an abstract class to express a type hierarchy sharing state; use interfaces to express capabilities/contracts, especially where multiple composition is needed. In the JDK, `AbstractList` is an abstract-class skeleton, and `List` is an interface.

## 6. `Object`'s methods and the equals/hashCode contract

### How to use it

Every class implicitly inherits `Object`. The three methods you'll override most often are `equals` (test "value equality"), `hashCode` (works with `HashMap`/`HashSet`), and `toString` (the text you see when printing):

```java
import java.util.Objects;

class Point2 {
    int x, y;
    Point2(int x, int y) { this.x = x; this.y = y; }

    @Override public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Point2 p)) return false;
        return x == p.x && y == p.y;
    }
    @Override public int hashCode() { return Objects.hash(x, y); }
    @Override public String toString() { return "Point2(" + x + ", " + y + ")"; }
}

public class EqualsBasics {
    public static void main(String[] args) {
        Point2 a = new Point2(1, 2);
        Point2 b = new Point2(1, 2);
        System.out.println(a.equals(b));   // true
        System.out.println(a == b);        // false -- different objects
        System.out.println(a);             // Point2(1, 2) -- toString gets called
    }
}
```

```
true
false
Point2(1, 2)
```

`==` always compares references (are these the same object); to compare "is the content equal," you must override `equals`. `System.out.println(a)` automatically calls `a.toString()`.

### Underlying implementation vs. C

`Object` is analogous to but stronger than C++ — C++ has no unified root. Key methods: `equals`, `hashCode`, `toString`, `getClass`, `clone`, `wait`/`notify` (see [Concurrency](10-并发模型.md)), `finalize` (deprecated, see [GC](09-错误处理与资源管理.md)).

#### The equals/hashCode contract (violating it wrecks collection behavior)

1. `a.equals(b)` is true ⟹ `a.hashCode() == b.hashCode()` (**equal implies same hash**).
2. The converse is not required (equal hashes need not be equal — collisions are allowed).
3. `equals` must be reflexive, symmetric, transitive, consistent, and non-null.

Override only `equals` without `hashCode`, and putting the object in a `HashMap`/`HashSet` will lose it, because it scatters to the wrong bucket (see [“The Collections Framework”](07-复合数据结构.md)). The default `Object.equals` is reference equality (`==`), and the default `hashCode` is an **identity hash** — often mistakenly described as "derived from the object's address," but that doesn't hold up (the hash would change once GC moves the object). Modern HotSpot (default since JDK 8) generates it with a **thread-local xorshift pseudo-random number**, computed lazily on the first call and stored into the object header's mark word, unchanged thereafter (see [“Data Types”](04-数据类型.md)).

```java
final class Point {
    final int x, y;
    Point(int x, int y){ this.x=x; this.y=y; }
    @Override public boolean equals(Object o){
        if (this == o) return true;
        if (!(o instanceof Point p)) return false; // pattern-matching instanceof
        return x == p.x && y == p.y;
    }
    @Override public int hashCode(){ return Objects.hash(x, y); }
}
```

A `record` can generate all of this automatically (see below).

The cost of violating the contract is directly observable: override only `equals`, not `hashCode`, and equal objects scatter to different buckets — a `HashSet` can no longer find them:

```java
import java.util.*;
public class HashContract {
    // equals overridden, but hashCode NOT -- violates the contract
    static class Bad {
        final int id; Bad(int id){ this.id = id; }
        @Override public boolean equals(Object o){ return o instanceof Bad b && b.id == id; }
    }
    // both overridden -- correct
    static class Good {
        final int id; Good(int id){ this.id = id; }
        @Override public boolean equals(Object o){ return o instanceof Good g && g.id == id; }
        @Override public int hashCode(){ return Integer.hashCode(id); }
    }
    public static void main(String[] args) {
        System.out.println("Bad  equal? " + new Bad(1).equals(new Bad(1)));
        Set<Bad> bad = new HashSet<>(); bad.add(new Bad(1));
        System.out.println("Bad  contains equal obj? " + bad.contains(new Bad(1)));

        Set<Good> good = new HashSet<>(); good.add(new Good(1));
        System.out.println("Good contains equal obj? " + good.contains(new Good(1)));
    }
}
```

```
Bad  equal? true                 <- the two Bad(1) are equal
Bad  contains equal obj? false   <- but the default hashCode differs, so they land in different buckets: not found!
Good contains equal obj? true    <- only found once hashCode is correctly overridden
```

## 7. Generics and type erasure (the fundamental divide from C++ templates)

### How to use it

You declare a generic class with `class ClassName<T>` (you've already seen `Box<T>` above); a generic method puts `<T>` before the return type, and the type parameter is inferred automatically from the argument passed at the call site:

```java
public class GenericMethodBasics {
    static <T> T firstOf(T[] arr) {     // generic method: type parameter goes before the return type
        return arr[0];
    }
    public static void main(String[] args) {
        String[] names = {"Ann", "Bob"};
        System.out.println(firstOf(names));   // Ann

        Integer[] nums = {10, 20};
        System.out.println(firstOf(nums));    // 10
    }
}
```

```
Ann
10
```

This is the most "counterintuitive" part of Java's type system, and where it differs most from C++ — let's dig into what actually happens at runtime.

### Underlying implementation vs. C

#### Erasure: no generic information at runtime

Java generics are a **compile-time** feature. After compilation the type parameters are "erased": `List<String>` and `List<Integer>` are both the same `List` class at runtime (the raw type `List`, with `T` erased to `Object`, or to the upper bound if bounded). The compiler inserts **casts** where needed and performs static type checking.

```java
List<String> ls = new ArrayList<>();
ls.add("hi");
String s = ls.get(0);   // the compiler inserts a (String) cast
// at runtime: ls.getClass() == ArrayList.class, exactly the same as List<Integer>
```

Compared with C++: templates are **monomorphized** — `vector<int>` and `vector<string>` compile to two independent copies of code, each with its own type at runtime; a zero-cost abstraction, but with code bloat. Java has only one copy of code, no bloat, but loses the type at runtime, giving rise to a series of restrictions.

At runtime the two generic instances are the **same** `Class` object; a raw type exposes even more clearly that "there is no type check at runtime":

```java
import java.util.*;
public class Erasure {
    public static void main(String[] args) {
        List<String>  ls = new ArrayList<>();
        List<Integer> li = new ArrayList<>();
        System.out.println(ls.getClass());                 // both erase to ArrayList
        System.out.println(li.getClass());
        System.out.println("same Class object? " + (ls.getClass() == li.getClass()));
        // the runtime keeps no type argument, so this line stores fine, and only breaks once you pull it out and use it:
        List raw = li;                 // raw type
        raw.add("oops");               // no ClassCastException here (only a compile-time warning)
        System.out.println("li now holds: " + li.get(0));  // stored despite the declared type List<Integer>
    }
}
```

```
class java.util.ArrayList
class java.util.ArrayList
same Class object? true
li now holds: oops
```

At compile time there's an unchecked warning at `raw.add("oops")`, but it doesn't block compilation or throw at runtime.

#### Four big restrictions from erasure

```java
// 1. Can't new T[] (the runtime type can't be determined, and covariant arrays would miss checks)
// T[] arr = new T[10];              // compile error
T[] arr = (T[]) new Object[10];       // workaround: cast, with an unchecked warning

// 2. Can't new T() (don't know which class to construct)
// T t = new T();                     // compile error, need to pass a Supplier<T> or Class<T>

// 3. instanceof can't carry a type parameter (no such info at runtime)
// if (x instanceof List<String>)     // compile error
if (x instanceof List<?>) { }         // only a wildcard is allowed

// 4. Static fields can't use type parameters; can't catch a generic exception
```

#### Bridge methods

Erasure would break covariant returns and generic overriding, so the compiler generates **bridge methods** to bridge the gap:

```java
class Node<T> { T get(){...} void set(T t){...} }
class StrNode extends Node<String> {
    @Override String get(){...}        // what you wrote
    // the compiler also secretly generates: Object get(){ return get(); }  ← bridge, matching the parent's erased signature Object get()
}
```

Otherwise the parent `Node` erases to `Object get()`, the subclass has `String get()` with a different signature, and virtual dispatch wouldn't line up. Bridge methods are visible with `javap -v`, marked `ACC_BRIDGE ACC_SYNTHETIC` — running it on `StrNode.class` really does show two `get` methods: one `()Ljava/lang/String;` (the one you wrote), and one `()Ljava/lang/Object;` carrying the `ACC_BRIDGE, ACC_SYNTHETIC` flags (the compiler-generated one).

#### Wildcards: the PECS principle

Generics are **invariant**: `List<String>` is not a `List<Object>` (the opposite of array covariance, and safer). To loosen this, use wildcards:

- `? extends T`: upper bound, the **producer** (can only read out T, can't write in, since the exact subtype is unknown).
- `? super T`: lower bound, the **consumer** (can only write in T, and reads out only as Object).
- The mnemonic **PECS**: Producer-Extends, Consumer-Super.

```java
// read from src, write to dst: src is the producer, use extends; dst is the consumer, use super
static <T> void copy(List<? super T> dst, List<? extends T> src){
    for (T t : src) dst.add(t);
}
```

C++20 concepts / old-style SFINAE solve a similar problem of "constraining type parameters," but the mechanism is entirely different (compile-time duck typing vs. nominal subtyping + erasure).

## 8. record: an immutable data carrier

### How to use it

A `record` declares its fields in one line, and the compiler automatically generates a constructor, accessors (named after the fields, no `get` prefix), and `equals`/`hashCode`/`toString`:

```java
record Point3(int x, int y) { }

public class RecordBasics {
    public static void main(String[] args) {
        Point3 p = new Point3(3, 4);
        System.out.println(p.x());      // 3   -- accessor, not called getX()
        System.out.println(p.y());      // 4
        System.out.println(p);          // Point3[x=3, y=4] -- auto-generated toString
    }
}
```

```
3
4
Point3[x=3, y=4]
```

### Underlying implementation vs. C

```java
record Point(int x, int y) { }
// auto-generated: private final fields, an all-args constructor, accessors x()/y(), equals/hashCode/toString
```

A `record` is **implicitly final**, with final fields, semantically a "transparent aggregate of data." Analogous to a C++ aggregate struct plus an auto-generated `operator==`. You can add a compact constructor to do validation:

```java
record Range(int lo, int hi) {
    Range {   // compact constructor: validates, then assigns automatically
        if (lo > hi) throw new IllegalArgumentException();
    }
}
```

```java
Range r = new Range(1, 5);
System.out.println(r);           // Range[lo=1, hi=5]
new Range(5, 1);                 // throws IllegalArgumentException -- the compact constructor's check kicks in
```

## 9. sealed: sealed types (controlled inheritance)

### How to use it

`sealed interface ... permits ...` explicitly lists who is allowed to implement it — any class not on the list can't:

```java
sealed interface Vehicle permits Car2, Bike { }
record Car2(int wheels) implements Vehicle { }
record Bike() implements Vehicle { }

public class SealedBasics {
    public static void main(String[] args) {
        Vehicle v = new Car2(4);
        System.out.println(v);   // Car2[wheels=4]
    }
}
```

```
Car2[wheels=4]
```

### Underlying implementation vs. C

Restricts who can inherit/implement, working with pattern matching to do exhaustiveness checking:

```java
sealed interface Shape permits Circle, Square { }
record Circle(double r) implements Shape { }
record Square(double s) implements Shape { }

double area(Shape sh) {
    return switch (sh) {          // the compiler knows there are only two, so no default is needed
        case Circle c -> Math.PI * c.r() * c.r();
        case Square s -> s.s() * s.s();
    };
}
```

```java
System.out.println(area(new Circle(2)));  // 12.566370614359172
System.out.println(area(new Square(3)));  // 9.0
```

Similar to expressing a closed sum type in C++ with `std::variant` + `std::visit`. `sealed` lets the compiler verify the switch is exhaustive — miss covering one subtype and the `switch` expression fails to compile, instead of leaving a hole that only shows up at runtime.

## 10. enum: a full-fledged class

### How to use it

The simplest enum is just a list of constant names — like C's `enum`, but safer: it can't be mixed with integers, and both `switch` and `==` work directly:

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

public class EnumBasics {
    public static void main(String[] args) {
        Day d = Day.MON;
        System.out.println(d);            // MON
        System.out.println(d == Day.MON); // true -- enum constants can be compared directly with ==

        for (Day day : Day.values()) {    // iterate all the constants
            System.out.print(day + " ");
        }
        System.out.println();
    }
}
```

```
MON
true
MON TUE WED THU FRI SAT SUN
```

### Underlying implementation vs. C

A Java enum is **not** a C/C++ integer constant; it's compiled into a class extending `java.lang.Enum`, with each enum constant being a **singleton object** (guaranteed by the JVM). It can have fields, methods, and constructors, and even a per-constant method body (constant-specific behavior):

```java
enum Planet {
    EARTH(5.97e24, 6.37e6), MARS(6.42e23, 3.39e6);  // calls the constructor
    private final double mass, radius;
    Planet(double m, double r){ mass=m; radius=r; }
    double gravity(){ return 6.67e-11 * mass / (radius*radius); }
}
```

```java
for (Planet p : Planet.values()) {
    System.out.println(p + " gravity = " + p.gravity());
}
```

```
EARTH gravity = 9.813440652193735
MARS gravity = 3.726159709713629
```

Because each constant is a singleton, enums can be safely compared with `==`, making them the best way to implement the singleton pattern; `EnumMap`/`EnumSet` use the ordinal as an index into a bitmap/array, extremely efficient.

## 11. Inner classes, anonymous classes, lambdas, and variable capture

### How to use it

A functional interface (an interface with just one abstract method, like the built-in `Runnable`) can be assigned a lambda directly, with no need to write a dedicated implementation class:

```java
Runnable r = () -> System.out.println("running!");
r.run();
```

### Underlying implementation vs. C

#### Kinds of inner classes

- **Static nested class** (`static class`): no reference to an outer instance, equivalent to a top-level class placed inside.
- **Member inner class**: holds a hidden `outer this` reference, able to access the outer instance's fields — also a source of memory leaks (a live inner-class instance drags the outer instance along with it).
- **Local class / anonymous class**: defined inside a method.

#### Captured variables must be effectively final

A local variable captured by an anonymous class/lambda must be **final or effectively final** (never reassigned after initialization):

```java
int base = 10;                 // effectively final
Runnable r = () -> System.out.println(base + 1);
// base = 20;                  // uncommenting this makes the line above a compile error
```

**Why?** A local variable lives in a **stack frame**, which is destroyed once the method returns; the lambda/anonymous-class object, on the other hand, lives on the **heap** and may well outlive it. Java's approach is to **copy the value** of the captured variable into the lambda object (copying the value itself for primitives, the reference value for reference types), and requiring it to be unchanging keeps "the original on the stack" and "the copy on the heap" semantically consistent, avoiding confusion from the two drifting apart.

Compared with C++ lambdas: you explicitly choose `[=]` capture-by-value or `[&]` capture-by-reference; by-reference lets you capture a mutable variable, but once the stack variable's lifetime ends it's a dangling reference (UB) — the responsibility is on you. Java plugs this hole at the language level with effectively final (value-only, and the value never changes). Capturing an instance field works differently: the lambda captures `this` and accesses the field through it, and the field itself is still mutable.

#### Under the hood of lambdas: invokedynamic

A lambda is **not** syntactic sugar for an anonymous class. javac generates an `invokedynamic` instruction, and on first execution `LambdaMetafactory` dynamically generates, at runtime, a class implementing the functional interface. Benefit: a capture-free lambda can reuse the same instance (it's stateless), avoiding the anonymous-class bloat of one class file per call site. Method references (`String::length`) work the same way.

## Summary

- **At the usage level**: `class ClassName { fields; constructor; methods; }` defines a class, `new ClassName(...)` builds an object, and `.` accesses fields/calls methods; `extends` gives single inheritance from one class, `implements` implements any number of interfaces; `class Box<T>` declares a generic class, `<T> T method(...)` declares a generic method; a functional-interface variable can be assigned directly with a `() -> ...` lambda; a `private` field paired with a getter is the basic form of encapsulation; overriding a method should carry `@Override`, and calling a method through a parent-typed reference dispatches based on the actual object type; `abstract class` mixes abstract and concrete methods, while an interface expresses "what it can do"; if you override `equals`, remember to override `hashCode` too; `record Point(int x, int y)` declares an immutable data class in one line; `sealed interface ... permits ...` restricts who can implement it; `enum` lists constants for use with `switch`/`==`.
- **At the underlying level**: objects always live on the heap, and the klass pointer in the object header is analogous to C++'s vptr; initialization order is "superclass statics → superclass instance → superclass constructor → this class's instance → this class's constructor," and calling a virtual method during construction can hit the pitfall of a subclass field not yet being initialized; `protected` is broader than in C++, permitting same-package access; interface default methods bring "multiple inheritance of implementation" without a state-diamond problem, and conflicts must be resolved with an explicit override; polymorphism relies on the vtable (class methods) plus the itable (interface methods), with the JIT optimizing via inline caches and devirtualization; `Object.hashCode`'s default is a thread-local pseudo-random identity hash, not something derived from an address; generics are type-checked at compile time and **erased** to raw types at runtime, which is why you can't `new T[]`/`new T()` or write `instanceof` with a type parameter, and why the compiler generates bridge methods to paper over signature mismatches caused by erasure; `record`/`sealed`/`enum` are all syntactic sugar where the compiler generates a pile of boilerplate for you, and an `enum` constant is fundamentally a singleton object; a lambda capturing a local variable requires it to be effectively final (value-copy semantics), and under the hood it's built by `invokedynamic` + `LambdaMetafactory` generating an implementation class at runtime, rather than being sugar for an anonymous class.
