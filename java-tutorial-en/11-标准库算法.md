# Standard Library Algorithms

C++'s `<algorithm>`, `std::sort` (introsort), `std::set` (red-black tree), and `priority_queue` (heap) live in the STL. Java's counterparts are `java.util.Collections`, `Arrays`, `PriorityQueue`, and `TreeMap`, plus the functional Stream API.

**Every section in this chapter starts with a snippet of code you can copy, run, and tweak right away, to get "how to use it" out of the way first — then it moves on to "why it's designed this way, and how it works underneath." You don't need to understand the JVM's internals or bytecode up front; just copy the code at the top of each section and you can start using these tools. If you want to dig into questions like "why does `Arrays.sort` use different algorithms for `int[]` and `Integer[]`," keep reading into the sections on the underlying mechanics that follow.**

## The Collections Utility Class

`java.util.Collections` is a batch of static methods operating on `List`/`Collection`:

```java
List<Integer> list = new ArrayList<>(List.of(3, 1, 2));
Collections.sort(list);                          // in-place sort (internally converts to an array and uses TimSort)
int idx = Collections.binarySearch(list, 2);     // binary search on a sorted List; a linked list (non-RandomAccess) is O(n) list traversals + O(log n) comparisons
Collections.reverse(list);
Collections.shuffle(list);
int max = Collections.max(list);

// Immutable view: returns a wrapper; any modification throws UnsupportedOperationException (like a const reference, but checked at runtime)
List<Integer> ro = Collections.unmodifiableList(list); // still backed by list; changing list changes it too
List<Integer> imm = List.of(1, 2, 3);            // JDK 9+ a truly immutable copy
```

> `binarySearch` requires the List to be sorted in the same order, or the result is undefined. A negative return `-(insertion point)-1` indicates "not found," which you can use to find the insertion position.

## Arrays.sort: Two Algorithms

This is the focus of the chapter. `Arrays.sort` uses different algorithms for **primitive types** and **object arrays**, and this is not arbitrary:

```java
int[] a = {5, 2, 8, 1};
Arrays.sort(a);                    // Dual-Pivot Quicksort

Integer[] b = {5, 2, 8, 1};
Arrays.sort(b);                    // TimSort (a merge-sort variant)
Arrays.sort(b, Comparator.reverseOrder());
```

### Why the Difference

| Dimension | Primitive int[] | Object Object[] |
|------|--------------|--------------|
| Algorithm | Dual-pivot quicksort | TimSort |
| Stability | Irrelevant (equal values are wholly identical) | Must be stable |
| Comparison cost | Extremely low (machine instructions) | High (calls compareTo/compare, possibly a virtual method) |
| Focus | Cache locality, comparison count | Fewer comparisons, stability |

**Why primitives use dual-pivot quicksort**: two equal `int`s are the same value, and swapping them makes no observable difference, so **stability is meaningless**. Quicksort is in-place and cache-friendly (sequential access has good locality), and comparison is a cheap CPU instruction. Dual-pivot quicksort picks two pivots to split the array into three parts, giving **about 5% fewer comparisons** than classic single-pivot quicksort and better cache locality—note that its **number of swaps is actually higher** than single-pivot; the advantage comes from "fewer comparisons + memory-access-friendliness," not "less moving."

**Why objects use TimSort**:
1. **Stability**: object sorting is often by some field, and the original relative order of equal elements must be preserved (e.g., sort by A first, then by B; when B is equal, A's order must be maintained). Quicksort is unstable; TimSort (a merge-sort family member) is stable.
2. **Comparison is expensive**: comparing objects is a method call (possibly an interface virtual call); on nearly-sorted data TimSort's comparison count approaches O(n), whereas quicksort's worst case is O(n²). Real-world data is often partially ordered, and TimSort identifies existing sorted "runs" and merges them, greatly reducing comparisons.

> TimSort combines merge sort with insertion sort: it scans out increasing/decreasing runs, extends short runs to the minrun length with binary insertion sort, then merges according to invariants maintained on a stack. Worst case O(n log n), already-sorted O(n). It needs O(n) auxiliary space (for merging), whereas quicksort is in-place—but object arrays already have an extra layer of pointer indirection and poor cache behavior, so the extra space is acceptable.

## Comparable and Comparator

- `Comparable<T>`: **natural ordering**, implemented by the class itself via `compareTo`; a class has only one. Analogous to C++'s `operator<`.
- `Comparator<T>`: an **external strategy**, and you can define arbitrarily many; analogous to a functor passed to `std::sort`.

The example below uses two bits of syntax you may not have seen before: `record` is a Java 16+ shorthand for declaring, in one line, an immutable class that just holds data (think of a C++ `struct` with only fields, plus an auto-generated constructor and getters); `Person::name` is a **method reference**, shorthand for `p -> p.name()` — you can just read it as "a lambda that grabs the `name` field."

```java
record Person(String name, int age) implements Comparable<Person> {
    @Override public int compareTo(Person o) {   // natural order: by age
        return Integer.compare(this.age, o.age);  // don't use this.age-o.age, overflow risk
    }
}

// Comparator combinators (JDK 8+), declarative, like a chained SQL ORDER BY
Comparator<Person> cmp = Comparator
    .comparing(Person::name)                     // by name first
    .thenComparingInt(Person::age)               // if names are equal, then by age
    .reversed();                                 // reverse the whole thing

Comparator<Person> byAgeNullsFirst = Comparator
    .comparingInt(Person::age)
    .reversed();
List<Person> people = new ArrayList<>();
people.sort(cmp);   // List.sort internally uses TimSort
```

The contract: `compare` must be reflexive, transitive, and antisymmetric; and consistency with `equals` is recommended (otherwise `TreeMap`/`TreeSet` behavior is counterintuitive). Violating transitivity makes TimSort throw `IllegalArgumentException: Comparison method violates its general contract`.

Runnable: sort by `name` ascending, then, when names are equal, by `age` descending—a declarative `thenComparing` + `reversed` composition:

```java
import java.util.*;

public class CmpDemo {
    record Person(String name, int age) {}

    public static void main(String[] args) {
        List<Person> people = new ArrayList<>(List.of(
            new Person("Bob", 30), new Person("Ann", 30),
            new Person("Bob", 25), new Person("Ann", 20)));

        // by name ascending, then by age descending when names are equal
        people.sort(Comparator.comparing(Person::name)
                              .thenComparing(Comparator.comparingInt(Person::age).reversed()));

        for (Person p : people) System.out.println(p.name() + " " + p.age());
    }
}
// Ann 30
// Ann 20
// Bob 30
// Bob 25
```

## PriorityQueue: The Binary Heap

`PriorityQueue` is an **array-based min-heap** (by default the smallest element is at the head), analogous to C++'s `priority_queue` (a max-heap by default).

```java
PriorityQueue<Integer> pq = new PriorityQueue<>();      // min-heap
pq.offer(3); pq.offer(1); pq.offer(2);
pq.peek();     // 1, view the heap top O(1)
pq.poll();     // 1, pop and sift down to rebalance O(log n)

// Max-heap / custom
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());
```

Internally an `Object[] queue`, with parent-child index relationships: node `i`'s children are `2i+1` and `2i+2`. `offer` sifts up (siftUp), and `poll` moves the last element to the top and sifts down (siftDown).

```
The heap represented by the array [1,3,2,7,4]:
          1(0)
        /     \
      3(1)     2(2)
      /  \
    7(3) 4(4)
```

**Not thread-safe**, and it **does not guarantee ordered traversal** (only the heap top is ordered); to traverse in order you must repeatedly `poll`. For concurrent scenarios, use `PriorityBlockingQueue`.

Runnable: find the Top-K largest with a size-k **min-heap**—the heap top is the smallest of the current top-k, so anything smaller is evicted, each `offer`/`poll` being `O(log k)`:

```java
import java.util.*;

public class HeapTopK {
    public static void main(String[] args) {
        int[] data = {5, 1, 9, 3, 7, 8, 2, 6, 4};
        int k = 3;

        // keep the k largest: a min-heap's top = the smallest of the current top-k
        PriorityQueue<Integer> pq = new PriorityQueue<>();  // min-heap
        for (int x : data) {
            pq.offer(x);
            if (pq.size() > k) pq.poll();                   // evict the smallest, O(log k)
        }

        // drain: a min-heap dequeues in ascending order
        List<Integer> topK = new ArrayList<>();
        while (!pq.isEmpty()) topK.add(pq.poll());
        System.out.println("top-" + k + " ascending = " + topK);
    }
}
// top-3 ascending = [7, 8, 9]
```

## TreeMap / TreeSet: The Red-Black Tree

`TreeMap` is backed by a **red-black tree** (a self-balancing BST), and `TreeSet` is its wrapper (values used as keys). These correspond to C++'s `std::map`/`std::set`. All operations are O(log n), and they are **ordered by key**.

```java
TreeMap<String, Integer> tm = new TreeMap<>();
tm.put("banana", 2); tm.put("apple", 1); tm.put("cherry", 3);
// Ordered views and navigation methods (HashMap has none)
tm.firstKey();               // "apple"
tm.floorKey("b");            // "apple" (largest key ≤ "b")
tm.ceilingKey("b");          // "banana" (smallest key ≥ "b")
tm.headMap("cherry");        // subview of keys less than cherry (a view; changes are reflected in the original map)
tm.subMap("apple", "cherry");
for (var e : tm.entrySet()) { }  // traverse in ascending key order
```

The five properties of a red-black tree (nodes are red/black, the root is black, a red node's children are black, all paths from any node to its leaves have the same black height, etc.) guarantee the longest path is ≤ 2× the shortest, giving approximate balance. Insertion/deletion is maintained via recoloring + rotation, with each adjustment O(log n).

| Comparison | HashMap | TreeMap |
|------|---------|---------|
| Underlying | Array + linked list/red-black tree | Red-black tree |
| Ordered | Unordered | Ordered by key |
| get/put | O(1) amortized | O(log n) |
| Navigation methods | None | floor/ceiling/subMap, etc. |
| null key | One allowed | Not allowed (comparison required) |

## The Stream API

The simplest use case: filter a `List`, then count what's left:

```java
List<String> names = List.of("Ann", "Bob", "Al", "Ben");
long count = names.stream()               // 1. open a "pipeline" from the collection
    .filter(s -> s.startsWith("A"))        // 2. keep only the elements that match
    .count();                              // 3. count what's left
System.out.println(count);                 // 2
```

`collection.stream()` opens a pipeline; in the middle you can chain any number of "processing steps" (here, `filter`), and finish the chain with one "get the result" operation (here, `count`). Keep this three-part shape in mind — source → intermediate operations → terminal operation — everything below builds on this same skeleton.

A Stream is a **declarative, lazy, parallelizable** data-processing pipeline, analogous to C++20 ranges/views. It is not a container, does not store data, and is "a description of operations on a data source."

### Laziness and Operation Categories

```java
List<String> words = List.of("apple", "banana", "cherry", "avocado");

long count = words.stream()          // source
    .filter(s -> s.startsWith("a"))  // intermediate operation: lazy, returns a new Stream
    .map(String::toUpperCase)        // intermediate operation
    .peek(System.out::println)       // intermediate operation (for debugging)
    .count();                        // terminal operation: triggers actual execution
```

- **Intermediate operations** (`filter`/`map`/`sorted`/`distinct`/`limit`): lazy, merely recorded, not executed. They return a Stream and can be chained.
- **Terminal operations** (`collect`/`forEach`/`count`/`reduce`/`findFirst`): trigger the traversal and execute the whole chain.

**The point of laziness**: operations are **fused**, so one element flows through the entire chain before the next is taken (rather than generating a full intermediate collection at each step), saving memory; `limit`/`findFirst` can **short-circuit** to finish early.

```java
// Short-circuit: stop at the first match, without processing the whole stream
Optional<Integer> first = Stream.iterate(1, n -> n + 1)  // infinite stream
    .filter(n -> n % 7 == 0)
    .findFirst();   // Optional[7]
```

### Collectors

The terminal operation `collect` gathers a stream into a result:

```java
import static java.util.stream.Collectors.*;

List<String> list  = words.stream().collect(toList());
Set<String> set    = words.stream().collect(toSet());
Map<Character, List<String>> byFirst =
    words.stream().collect(groupingBy(s -> s.charAt(0)));   // grouping, like SQL GROUP BY
Map<Boolean, List<String>> parts =
    words.stream().collect(partitioningBy(s -> s.length() > 5));
String joined = words.stream().collect(joining(", ", "[", "]"));
Double avg = words.stream().collect(averagingInt(String::length));
```

Runnable: `peek` reveals laziness—`findFirst` short-circuits, stopping at the first match without traversing the whole stream; `groupingBy` buckets words by first letter:

```java
import java.util.*;
import java.util.stream.*;
import static java.util.stream.Collectors.*;

public class StreamDemo {
    public static void main(String[] args) {
        List<String> words = List.of("apple", "banana", "cherry", "avocado", "blueberry");

        // laziness + short-circuit: peek prints only the elements actually pulled
        System.out.println("--- lazy pipeline (findFirst short-circuits) ---");
        Optional<String> first = words.stream()
            .peek(w -> System.out.println("seen: " + w))   // side-effect to observe traversal
            .filter(w -> w.startsWith("b"))
            .findFirst();
        System.out.println("first b-word = " + first.get());

        // groupingBy: bucket by first letter, into a TreeMap for stable ordered output
        Map<Character, List<String>> byFirst = words.stream()
            .collect(groupingBy(w -> w.charAt(0), TreeMap::new, toList()));
        System.out.println("--- groupingBy ---");
        System.out.println(byFirst);
    }
}
// --- lazy pipeline (findFirst short-circuits) ---
// seen: apple
// seen: banana
// first b-word = banana
// --- groupingBy ---
// {a=[apple, avocado], b=[banana, blueberry], c=[cherry]}
```

### Parallel Streams

`parallelStream()` or `.parallel()` splits the stream and processes it divide-and-conquer using the **common ForkJoinPool** (`ForkJoinPool.commonPool()`, default thread count = CPU cores - 1).

```java
long sum = LongStream.rangeClosed(1, 1_000_000)
    .parallel()
    .sum();
```

Considerations:
- The data source should be **easy to split** (arrays and `ArrayList` are good; `LinkedList` is bad).
- Operations must be **stateless and side-effect-free**, or you get data races (do not modify shared variables inside a lambda).
- For small datasets or cheap operations, the splitting/merging/scheduling overhead of parallelism may not pay off.
- All parallel streams share the same commonPool, so one slow task drags down the whole system; don't use it for IO-bound work.

## Optional

`Optional<T>` expresses "there may be no value," replacing a bare `null` and forcing the caller to handle the absence, reducing NPEs. Analogous to C++17's `std::optional` (but Java's is an object, with heap allocation overhead).

The most basic usage: `Optional.of` wraps a value known to exist, `Optional.empty()` represents a definite absence of value, and `isPresent()` tells you which case you're in:

```java
Optional<String> opt1 = Optional.of("hello");   // definitely has a value
Optional<String> opt2 = Optional.empty();       // definitely has no value
System.out.println(opt1.isPresent());           // true
System.out.println(opt2.isPresent());           // false
```

In real code it's more common to use `Optional.ofNullable` to wrap a "value that might be null" (say, the return of a method that might not find a result), and then handle the "present" or "absent" case with the methods below, instead of writing an explicit `if (x != null)`:

```java
Optional<String> opt = Optional.ofNullable(getName());
String s = opt.orElse("default");             // give a default if empty
String t = opt.orElseGet(() -> compute());    // lazy default (computed only if empty)
opt.ifPresent(System.out::println);
String u = opt.map(String::trim).filter(x -> !x.isEmpty()).orElse("-");
opt.orElseThrow(() -> new IllegalStateException("missing"));
```

Usage convention: use it mainly as a **method return value** to express "there may be no result"; **do not** use it for fields, method parameters, or collection elements (needless overhead).

## Numeric Utilities

Day to day, use `Math`'s static methods for basic arithmetic; when `int`/`long`/`double` precision or range isn't enough (say you need integers wider than 64 bits, or a financial calculation that can't tolerate floating-point error), reach for `BigInteger`/`BigDecimal`.

```java
Math.max(a, b); Math.floorMod(-1, 3); // 2 (unlike %, the sign of the result follows the divisor — always non-negative here)
Math.addExact(a, b);                  // throws ArithmeticException on overflow, rather than silently wrapping

// BigInteger: arbitrary-precision integers (immutable), like a bignum you'd implement yourself; operations return new objects
BigInteger big = BigInteger.valueOf(2).pow(1000);
BigInteger f = big.multiply(BigInteger.TEN).mod(BigInteger.valueOf(7));

// BigDecimal: exact decimals, a must for finance (double cannot represent 0.1 exactly)
BigDecimal price = new BigDecimal("0.10");    // must construct from a String; new BigDecimal(0.1) carries binary error
BigDecimal total = price.multiply(new BigDecimal("3"));  // exact 0.30
System.out.println(new BigDecimal("0.1").add(new BigDecimal("0.2"))); // 0.3 (double would be 0.30000000000000004)
total = total.setScale(2, RoundingMode.HALF_UP);         // explicitly specify rounding

// Random: linear congruential pseudo-random; for concurrency use ThreadLocalRandom; for security use SecureRandom
int r = ThreadLocalRandom.current().nextInt(0, 100);
```

> `BigInteger`/`BigDecimal` are immutable, so frequent operations produce many temporary objects; watch out for GC pressure in loops. Compare them with `compareTo`; `BigDecimal`'s `equals` also compares scale (`2.0` != `2.00`), so use `compareTo == 0` to test numeric equality.

## Complexity Cheat Sheet

| Operation | ArrayList | LinkedList | HashMap | TreeMap | PriorityQueue |
|------|-----------|-----------|---------|---------|---------------|
| Random-access get | O(1) | O(n) | — | — | — |
| Head insertion | O(n) | O(1) | — | — | — |
| Tail insertion | O(1) amortized | O(1) | — | — | O(log n) |
| Lookup by key | O(n) | O(n) | O(1) amortized | O(log n) | — |
| View extremum | O(n) | O(n) | — | O(log n) | O(1) |
| Ordered traversal | Sort first | Sort first | Unsupported | O(n) naturally ordered | Unsupported |

| Algorithm | Average | Worst | Stable | Space |
|------|------|------|------|------|
| Dual-pivot quicksort (primitives) | O(n log n) | O(n²) | No | O(log n) |
| TimSort (objects) | O(n log n) | O(n log n) | Yes | O(n) |
| Binary search | O(log n) | O(log n) | — | O(1) |
| Heap operation (insert/delete) | O(log n) | O(log n) | — | O(1) |

## Summary

- **At the usage level**:
  - `Collections` is a bag of static methods for operating on `List`/`Collection`: `sort`/`binarySearch`/`reverse`/`shuffle`/`max`; for an immutable view, use `unmodifiableXxx` (a wrapper around the original collection) or `List.of` (a truly immutable copy).
  - `Arrays.sort` is called the same way for primitive arrays and object arrays — it picks the algorithm internally.
  - To sort objects: either have the class implement `Comparable` itself (only one natural order), or pass in an external `Comparator` (as many as you like); JDK 8+ lets you compose them declaratively with `Comparator.comparing().thenComparing().reversed()`.
  - `PriorityQueue` is just a heap — a min-heap by default — with `offer`/`peek`/`poll` for adding, peeking at, and removing the top.
  - `TreeMap`/`TreeSet` are naturally ordered by key, adding navigation methods `HashMap` lacks: `firstKey`/`floorKey`/`ceilingKey`/`headMap`/`subMap`.
  - A Stream's skeleton is "source `.stream()` → some intermediate operations (`filter`/`map`/`sorted`) → one terminal operation (`collect`/`count`/`forEach`)"; `collect` paired with `Collectors` handles grouping, partitioning, joining, and statistics.
  - `Optional<T>` expresses "there may be no value"; use it mainly as a method return value, handled via `orElse`/`map`/`ifPresent`, and avoid it for fields or parameters.
  - Beyond `long`'s range, use `BigInteger`; when you need exact decimals (finance), use `BigDecimal` — it must be constructed from a `String`, and comparing for numeric equality means `compareTo == 0`, not `equals`.
- **At the underlying level**:
  - `Arrays.sort`: primitives use **dual-pivot quicksort** (equal values are indistinguishable anyway, so stability is meaningless; it's in-place and cache-friendly); objects use **TimSort** (a merge/insertion hybrid that identifies existing sorted runs, guarantees stability, worst case O(n log n), O(n) on already-sorted data, and needs O(n) auxiliary space).
  - The `Comparator`/`Comparable` contract requires reflexivity, transitivity, and antisymmetry; violating transitivity makes TimSort throw `IllegalArgumentException` outright.
  - `PriorityQueue` is backed by an **array-represented binary heap** (node `i`'s children are `2i+1`/`2i+2`); `offer` sifts up and `poll` sifts down, both O(log n), and it's not thread-safe.
  - `TreeMap` is backed by a **red-black tree**, kept approximately balanced by properties like "the root is black, a red node's children must be black, every path has the same black height," with all operations O(log n).
  - A Stream's intermediate operations are all **lazy** and don't actually run until the terminal operation fires; the payoff is that operations can be fused (saving memory) and can **short-circuit** (`limit`/`findFirst` finish early); parallel streams divide-and-conquer via the common `ForkJoinPool`, which requires an easily-splittable data source and stateless, side-effect-free operations.
  - `BigInteger`/`BigDecimal` are both **immutable** — every operation produces a new object, so watch for GC pressure when using them heavily inside loops.
