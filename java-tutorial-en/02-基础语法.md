# Basic Syntax: Contrasted with C/C++

Java's surface syntax deliberately hews close to the C family — `{}` delimits blocks, `;` ends statements, `if/for/while` are written the same way, and so are the arithmetic and logical operators. This was an explicit design goal of the Gosling team: let C/C++ programmers relearn almost no notation at all, so their attention goes straight to the semantic differences. This chapter therefore doesn't rehash that common ground; it covers only where Java's syntax and language structure genuinely part ways with C/C++, and the design trade-off behind each divergence.

Here's a piece of code showing the common notation, to build an overall impression:

```java
public class Basics {
    public static void main(String[] args) {
        int age = 20;
        double price = 9.9;
        String name = "Ada";
        boolean ok = true;

        int sum = age + 5;
        System.out.println(sum);                              // 25
        System.out.println("name=" + name + ", age=" + age);  // name=Ada, age=20

        if (age >= 18) {
            System.out.println("adult");
        } else {
            System.out.println("minor");
        }

        for (int i = 0; i < 3; i++) {
            System.out.println("i=" + i);   // i=0  i=1  i=2
        }
    }
}
```

Comments use `//` and `/* */`, the same as C. Below we go through the structural differences one at a time.

---

## 1. Everything Lives in a class; the Program Entry Point

C allows free functions and global variables; Java **has neither** — all code must hang off some class (or interface/enum/record).

```java
public class App {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}
```

**Why it's designed this way**: Java's object model inherits the Smalltalk lineage's "pure object orientation" philosophy, deliberately leaving out both free functions and mutable global state as escape hatches. This isn't just an aesthetic choice — global mutable state is the hardest thing to reason about in a multithreaded program (any thread can read or write it at any moment, which trips up static analysis, concurrency correctness, and later on, sandboxed security models); forcing all code to belong to some class amounts to forcing it into a well-defined namespace and access-control boundary. One of Java's early design goals was to safely load and run untrusted code fetched over the network (the applet model), and a language with no globally writable state is plainly much easier to make safe in that setting.

Contrasted with C's `int main(int argc, char** argv)`:
- **`main` returns `void`**: the JVM launcher looks up the entry point by reflection against the fixed signature `public static void main(String[])`; it only cares whether such a method exists and can be invoked, not about any business return value — the process exit code is a separate channel, set explicitly via `System.exit(int)`. Decoupling "the function's return value" from "the process's exit status" lets the entry-point signature stay uniform across every program, instead of inventing a different `main` signature for whatever value each program happens to want to return.
- **The parameter is `String[] args`, and it doesn't include the program name**: C's `argv[0]` is the program name, a historical artifact of how the OS's `exec` family of calls passes arguments (the program needs to know what name it was invoked under — e.g. a busybox-style multiplexed binary). The JVM already knows which class it's running, so it doesn't need to stuff that information into the user-visible argument array; `args` therefore contains only the real invocation arguments, and the array carries its own `length`, with no need for a separate `argc`.
- **It must be `static`**: the JVM has to be able to call the entry point right after loading the class, before any instance exists; a `static` method doesn't depend on an instance existing, which is the direct consequence of the lifecycle ordering "the class exists before any instance does" (an instance method logically requires an object to already exist).

> Since JDK 21, a preview form has offered a slimmed-down entry point that omits the class wrapper, and it **became a standard feature in JDK 25 (JEP 512)** — but this is just syntactic sugar aimed at teaching and scripting scenarios; the loading–reflection–invocation machinery behind `public static void main(String[])` hasn't changed, and production projects still mostly use the classic form.

## 2. Packages and import: No Preprocessor

C/C++ uses `#include` to do **textual pasting**: the declarations in a header file are copied verbatim, at compile time, into every source file that references it, and `#define` macros do pure text substitution. This mechanism has real costs — the same declaration gets re-parsed in every translation unit, macro expansion doesn't respect scoping rules (it applies globally and can be redefined arbitrarily), `#ifdef` means the same source is actually a different program under different build conditions, and forgetting an include guard causes duplicate inclusion.

Java replaces this entire mechanism with **packages + import**:

```java
package com.example;

import java.util.List;          // import a single class
import java.util.*;             // import a whole package (not recursive into sub-packages)
import static java.lang.Math.*; // static import

public class Demo { /* ... */ }
```

`import` **inserts no text at all**; it merely tells the compiler "which fully-qualified name a given simple name maps to." Actual type resolution happens at compile time by looking up the symbol table, and at runtime the class loader loads `.class` files on demand (see [Runtime Model](01-运行模型.md)).

**Why header files and forward declarations can be dispensed with**: the root cause is that a `.class` file is itself **self-describing** — it contains the complete method signatures, field types, constant pool, and other metadata, so the compiler or class loader can read a class's full contract straight out of that binary, with no need to separately maintain a hand-written `.h` declaration to "announce" what the implementation contains, the way C++ does. This is also why classes can reference each other and form circular dependencies — the compiler resolves symbols at the granularity of a class, and doesn't require the strict textual ordering of "you must have already seen the declaration before using it." C++'s single-pass compilation model must see a declaration before it can be used; the two behaviors are the natural consequence of two entirely different compilation-and-linking models: C++ compiles to object files and lets the linker stitch symbols together; Java compiles to bytecode carrying complete metadata and lets the class loader resolve it on demand at runtime.

Dropping macros brings a frequently underrated benefit too: **no macros means no macro bugs** as a category. Macros ignore scoping and the type system — `#define MAX(a,b) ((a)>(b)?(a):(b))` evaluates an argument with side effects more than once — whereas Java uses ordinary methods (which get inlined by the JIT anyway, see [Runtime Model](01-运行模型.md)) in place of "function macros," and a method obeys normal scoping, type checking, and evaluation-order rules. Constants use `static final`, and platform differences are handled by runtime checks or build tooling, rather than compile-time conditional expansion producing what is, in effect, a different source file.

## 3. Strong Static Typing; `var` for Local Inference

```java
int count = 10;
double ratio = 0.5;
String name = "Ada";
long big = 9_000_000_000L;   // beyond int's range requires the L suffix
```

Like C++, Java is strongly and statically typed: every variable's type is fixed at compile time, and a type mismatch is a compile error — this lets the JIT/AOT determine method dispatch and memory layout at compile time rather than at runtime, and it surfaces type errors during development rather than in the user's hands.

`var`, introduced in JDK 10, provides local variable type inference, semantically equivalent to C++11's `auto`:

```java
var count = 10;                        // inferred as int
var list = new ArrayList<String>();    // inferred as ArrayList<String>
```

**Why Java didn't add this until version 10, and why it's restricted to local variables**: since generics arrived (Java 5), type declarations often grow quite long (`Map<String, List<Integer>> m = new HashMap<>();`), and that's pure visual noise; `var` is compile-time syntactic sugar aimed squarely at that pain point — the type is still nailed down at compile time, `var` doesn't change the fact that Java is a statically typed language, and the bytecode holds the concrete type, exactly equivalent to spelling the type out by hand, with zero difference at runtime. The deliberate restriction to "inside a method, and only with an initializer expression" exists because field types and method signatures are **part of a public contract** (both documentation for callers and part of the binary interface); using `var` there would strip readers/callers of a direct way to see that contract, whereas a local variable is purely an implementation detail, and inferring it away doesn't affect anyone's understanding of the external contract. This is a deliberate boundary, not an arbitrary restriction.

## 4. Operators

The vast majority of operators are the same as C: arithmetic `+ - * / %`, relational `== != < > <= >=`, logical `&& || !`, bitwise `& | ^ ~ << >>`, compound assignment, increment/decrement — write them exactly as you would in C. What's genuinely worth digging into are the divergences below, each of which corresponds to a specific design trade-off:

**An `if` condition must be `boolean`, and won't accept an integer**: in C, "nonzero is true," and the classic bug of mistyping `==` as `=` in `if (a = b)` compiles cleanly precisely because the value of an assignment expression (`b`) can be implicitly converted to whatever "truthiness" the condition needs. Java requires the condition to be `boolean`; `int` cannot implicitly convert to `boolean`, so this kind of typo simply fails to compile in Java — the type system eradicates an entire class of bug that has burned plenty of C programmers, at the cost of giving up the flexibility of "using an integer as a truth value" (you can no longer write `if (count)` to test for nonzero).

**Unsigned right shift `>>>` is kept separate from arithmetic right shift `>>`**: Java deliberately has no unsigned integer types (only the signed `byte/short/int/long`), trading away that in exchange for simpler, unambiguous overflow and sign-extension rules — you never have to deal with C's implementation-defined behavior around "which operand gets promoted to which type when mixing signed and unsigned arithmetic." But even without unsigned types, there's still a genuine need to "treat the high bit as a plain bit, not a sign bit" (bitwise operations, hashing, emulating unsigned arithmetic), and `>>>` exists precisely to open that door: `>>` fills with the sign bit (arithmetic right shift), `>>>` fills with 0 (logical right shift). In C, whether the sign bit is filled depends on whether the type itself is unsigned; Java makes that choice explicit via two separate operators.

**Integer division by zero throws an exception rather than being undefined behavior**: the C standard defines integer division by zero as undefined behavior, allowing different hardware platforms to handle it differently (some trigger a CPU exception directly, some return garbage). The JVM, as a managed runtime spanning hardware platforms, must guarantee that the same piece of bytecode behaves identically on every platform, so it defines "division by zero" as **deterministically throwing `ArithmeticException`**, which can be `catch`ed. This is the direct fork in the road between "the language spec guarantees portable, deterministic behavior" and "stay close to the hardware, allow platform variance" as design philosophies. Floating-point division by zero follows IEEE-754's own semantics instead, yielding `Infinity`/`NaN` without throwing — because the IEEE-754 standard already defines those values itself, with no need for the JVM to add a fallback.

**Shift counts are taken modulo the operand's bit width**: shifting an `int` implicitly applies `& 31` to the shift amount, and a `long` applies `& 63`, so `1 << 32` results in `1`, not `0`. This is another instance of "eliminating platform-dependent undefined behavior" — in C, a shift amount exceeding the type's bit width is undefined behavior, while Java explicitly specifies the modulo rule, giving the same result on every platform.

**`+` is the only "overloaded" operator**: as long as one operand is a `String`, `+` becomes concatenation, specially handled by the compiler (internally often converted to `StringBuilder.append`, see [Data Types](04-数据类型.md)). This is a deliberate, and the **only**, exception Java carves out — the language doesn't support user-defined operator overloading at all (see reasons in Section 8), but string concatenation is common enough to be worth an exception.

Testing these points:

```java
System.out.println(1 << 32);        // 1  (32 & 31 == 0, i.e. 1<<0)
System.out.println(-8 >> 1);        // -4   arithmetic right shift, fills the sign bit
System.out.println(-8 >>> 28);      // 15   logical right shift, fills with 0

try { int x = 1 / 0; }
catch (ArithmeticException e) { System.out.println(e.getMessage()); } // / by zero
System.out.println(1.0 / 0);        // Infinity (IEEE-754 semantics, no exception)

System.out.println("sum=" + 1 + 2); // sum=12 (evaluated left to right, once it's a string everything else concatenates)
System.out.println(1 + 2 + "=sum"); // 3=sum (still plain addition before a string is encountered)
```

## 5. Strings: an Immutable Class, Not a char Array

```java
String s = "hello";
int n = s.length();               // O(1) field, not a strlen-style scan for \0
boolean same = s.equals("hello"); // compare content with equals, not ==
String up = s.toUpperCase();      // returns a new string, s itself unchanged
```

`String` is a class, neither a `char*` nor a `char[]`, and it is **immutable** — every "modifying" method returns a new object, and the original object's content never changes:

```java
String s = "hello";
String up = s.toUpperCase();
System.out.println(s);         // hello   original unchanged
System.out.println(s == up);   // false   up is a different object
```

**Why it's designed to be immutable** isn't an arbitrary choice — it serves three goals at once:
1. **Safe sharing with no defensive copies needed**: multiple references can point at the same `String` object without worrying that some piece of code will quietly change its content — this matters a great deal for strings that are constantly used as `HashMap` keys, class names, file paths, and network addresses; if strings were mutable, you could in principle get a time-of-check-to-time-of-use bug — the value is one thing at validation time and has already been changed by the time it's used.
2. **The hash value can be safely cached**: an immutable object's hash only ever needs computing once; `String.hashCode()` caches the result internally, dramatically cutting the cost of using a string as a `HashMap` key.
3. **Lock-free thread safety**: an immutable object is inherently thread-safe — multiple threads reading the same `String` need no synchronization at all.

The cost is clear: frequent concatenation keeps generating new objects, and building a string with `+` in a loop is O(n²) — a direct cost of immutability, solved by using the mutable `StringBuilder` (internally a growable `char`/`byte` array, much like a manually managed buffer; details in [Data Types](04-数据类型.md)).

## 6. Console IO

```java
System.out.println("a full line with a newline");
System.out.printf("%d and %.2f%n", 10, 3.14159);  // printf-like, %n is the platform-independent newline

Scanner sc = new Scanner(System.in);
int x = sc.nextInt();
```

`System.out`/`System.err`/`System.in` are objects wrapping stdout/stderr/stdin respectively. Java's `printf`-style formatting does **runtime type checking** — a mismatch between a format specifier and the actual argument type throws an exception, rather than being undefined behavior the way a mismatch between a format specifier and its argument is in C's `printf` (this is precisely the root of a whole historical class of "format string attack" vulnerabilities in C: an attacker-crafted format string can make `printf` interpret stack data under the wrong type, leaking information or even enabling arbitrary writes). Java replaces "type mismatch is UB" with "a type-check failure throws an exception," the same philosophy as division by zero and shifting above: **wherever failure can be made deterministic, don't leave behavior undefined.**

## 7. Comments and Javadoc

```java
/**
 * Javadoc doc comment
 * @param n the input count
 * @return the computed result
 */
public int compute(int n) { return n * 2; }
```

Tags like `@param`/`@return`/`@throws` are **language-level conventions** that both the compiler and the toolchain understand; the `javadoc` tool generates HTML API documentation from them, and IDEs use them to show hover tooltips too. C/C++ has no such built-in documentation system — tools like Doxygen are an external convention, and the compiler itself understands nothing about the structure of those comments. Making the documentation format part of the language ecosystem's shared understanding buys the entire ecosystem a highly uniform API documentation style.

## 8. Structural Differences from C++, and the Trade-offs Behind Them

| Dimension | C++ | Java |
|------|-----|------|
| Inheritance | multiple inheritance | single class inheritance + multiple interface implementation |
| Pointers | explicit pointer syntax `*`/`&`/`->` | no explicit pointer syntax; references are managed pointers |
| Memory deallocation | manual `delete`/`free` | reclaimed automatically by the GC |
| Operator overloading | supported | not supported (except the built-in `+` for string concatenation) |
| Free functions | supported | none; everything lives in a class |
| Arrays | contiguous memory on stack/heap, no bounds checking | objects with a `length`, bounds checked |
| Header files | required | none, no preprocessor/macros |
| Default arguments | supported | not supported, use method overloading instead |
| Destruction | RAII, deterministic destruction | no deterministic destruction, use try-with-resources |
| `const` | modifies broadly | `final` (much narrower semantics) |

Going through the trade-offs one at a time, rather than stopping at listing the differences:

- **Single inheritance + multiple interfaces, rather than multiple inheritance**: C++'s multiple inheritance runs into the diamond problem — when the same base class is inherited twice via two different paths, whether the base class's members are shared or duplicated requires an extra mechanism like `virtual` inheritance to disambiguate, and the rules are complex and error-prone. Java's solution is to split "inheriting an implementation" from "inheriting a type contract": a class can only single-inherit (avoiding the diamond problem at the implementation level), but it can implement any number of interfaces (a pure contract, stateless by default; only since JDK 8's default methods has a new, but much narrower, conflict-resolution rule been needed). This is the classic trade-off of "giving up some expressive power for simpler rules."
- **No operator overloading**: C++ lets almost any operator be overloaded, which buys expressive power (matrices multiplied directly with `*`, say), but is also a long-criticized source of complexity — `a + b` might quietly invoke an expensive, side-effecting custom function, and you can't tell what's actually happening just by looking at the notation. Java opts instead for the predictability of "a method call is always a method call": `a.equals(b)`, `bigInt.add(other)` are all written as explicit method calls; the cost is verbosity, and the payoff is the certainty of "you know roughly what will happen just by seeing the notation."
- **No deterministic destruction; try-with-resources instead of RAII**: C++'s destructors run deterministically on scope exit, which relies on an object's lifetime being statically determinable. Java's GC, in order to do generational collection, potentially move objects, and collect concurrently across multiple threads, hands the decision of when an object's lifetime ends over to the runtime — it is no longer deterministic, which is why the `finalize()` mechanism has been deprecated. But non-memory resources like file handles and network connections still need deterministic closing, and Java's try-with-resources (see [Error Handling and Resource Management](09-错误处理与资源管理.md)) reimplements the core value of RAII — "close on scope exit" — without depending on deterministic destruction.
- **Arrays are always bounds-checked**: an out-of-bounds C array access is undefined behavior, and a major source of memory-safety problems. A Java array is, at the language level, a heap object carrying a `length` field, and every indexed access is bounds-checked, throwing `ArrayIndexOutOfBoundsException` rather than reading or writing memory that doesn't belong to the array. The cost is one extra comparison per access, but the JIT can usually eliminate provably-safe redundant checks inside a loop, pushing the runtime cost close to zero.

## Summary

- **Syntax notation**: `{}`, `;`, `if/for/while`, and the arithmetic/logical operators match C and can be carried straight over; `var` (JDK 10+) provides local type inference without changing the fact that Java is statically typed.
- **Everything lives in a class**: no free functions, no global variables, stemming from a pure object-oriented design and the historical goal of a safe sandbox; `main` must be `static` as a direct consequence of the loading order "the class exists before any instance does."
- **No preprocessor**: `.class` files are self-describing with complete signatures, which is what lets import be pure name resolution rather than text insertion, dispensing with header files, forward declarations, and macros — and with them, the whole class of bugs macros bring.
- **A handful of operator divergences** (`if` requiring boolean, `>>>`, division by zero throwing, shift-count modulo) all trace back to the same principle: wherever failure can be made deterministic, don't leave behavior undefined.
- **`String` is immutable**, a design that jointly serves safe sharing, hash caching, and lock-free concurrency — at the cost of needing `StringBuilder` for concatenation.
- The structural differences from C++ (single inheritance with multiple interfaces, no operator overloading, no deterministic destruction, mandatory array bounds checking) are each a concrete trade-off between "simplicity/safety" and "expressive power/peak performance," not arbitrary omissions.
