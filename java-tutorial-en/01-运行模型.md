# Runtime Model: javac, Bytecode, the JVM, and JIT

As a C/C++ engineer, the compilation pipeline you know is: `.c` source goes through preprocessing, compilation, and assembly to produce an object file `.o`, which the linker then stitches into an executable (ELF/Mach-O/PE) targeting a **specific CPU architecture + specific operating system**. The CPU executes the machine code inside it directly.

Java takes a **two-stage** path: first it compiles to a platform-independent intermediate code (bytecode), which is then executed at runtime by a "virtual CPU" (the JVM). Understanding this pipeline is a prerequisite for understanding all the memory behavior that follows.

## 1. From Source to Bytecode: What javac Does

```java
// Hello.java
public class Hello {
    public static void main(String[] args) {
        int sum = 0;
        for (int i = 0; i < 10; i++) {
            sum += i;
        }
        System.out.println(sum);
    }
}
```

Compile:

```
javac Hello.java     # produces Hello.class
```

The **essential difference** between `javac` and `gcc`: `gcc`'s output targets a real CPU, with an instruction set of x86-64 / ARM64; `javac`'s output targets a hypothetical **stack-based virtual machine**, whose instruction set is "bytecode." Bytecode assumes nothing about the number of registers, the word size, or the byte order—it targets only the abstract machine defined by the JVM specification. This is the technical foundation of "write once, run anywhere": a `.class` file is **byte-for-byte identical** on Windows/Linux/macOS and on x86/ARM.

A `.class` file is a binary container with a strict format (beginning with the magic number `0xCAFEBABE`), containing:

| Section | Contents | C/C++ analogy |
|------|------|-----------|
| constant pool | symbolic references to strings, class names, method names, literals | like `.rodata` + the symbol table, but more structured |
| field table | the class's field declarations | struct member layout information |
| method table | each method's bytecode + metadata | function bodies, but with full type signatures |
| attribute table | line number table, local variable table, annotations, etc. | debug info (like DWARF), but built in |

Key point: **the `.class` file retains complete type and symbol information**. After you `strip` a C program almost nothing but machine code remains, with symbols gone; a `.class` file inherently carries class names, method signatures, and field types—this is the material basis on which reflection, dynamic loading, and runtime type checking can exist.

The code below relies on this retained information to read a method's exact signature **at runtime**—something impossible on a `strip`'d C binary:

```java
import java.lang.reflect.Method;

Class<?> c = java.util.ArrayList.class;
System.out.println("class = " + c.getName());
Method m = c.getMethod("add", Object.class);
System.out.println("method = " + m.getName());
System.out.println("returns = " + m.getReturnType().getName());
System.out.println("params = " + m.getParameterCount());
// Output:
// class = java.util.ArrayList
// method = add
// returns = boolean
// params = 1
```

### Viewing Bytecode with javap

`javap` is the disassembler that ships with the JDK, the Java world's equivalent of `objdump -d`:

```
javap -c Hello.class
```

The bytecode of the `main` method looks roughly like this (comments added by me):

```
0:  iconst_0        // push the constant 0 onto the operand stack
1:  istore_1        // pop the top of the stack into local variable slot 1 (i.e. sum)
2:  iconst_0        // push the constant 0
3:  istore_2        // store into slot 2 (i.e. i)
4:  iload_2         // load i onto the stack
5:  bipush 10       // push the constant 10
7:  if_icmpge 20    // if i >= 10 jump to offset 20 (exit the loop)
10: iload_1         // load sum
11: iload_2         // load i
12: iadd            // add the top two stack values, result left on the stack
13: istore_1        // store back into sum
14: iinc 2, 1       // increment slot 2 (i) by 1 directly, without going through the operand stack
17: goto 4          // back to the loop test
20: getstatic ...   // fetch System.out
23: iload_1
24: invokevirtual println
27: return
```

Notice that instructions like `iconst`/`istore`/`iload`/`iadd` **have no register operands**. The JVM is a **stack machine**: the inputs and outputs of operations flow across the "operand stack," rather than naming registers as x86 does with `add eax, ebx`. The `i` prefix means the operation is on the int type; there are also `l` (long), `f` (float), `d` (double), and `a` (reference/address). `iinc` is one of the rare instructions that modifies the local variable table directly.

> Every method declares in its bytecode how large an operand stack it needs (`stack=`) and how many local variable slots (`locals=`). These are fixed values computed at compile time, and the JVM allocates stack frames accordingly—see [Methods and the Call Stack](06-函数与调用栈.md).

## 2. Class Loading: Getting a .class into the JVM

When a C program starts, the loader maps the ELF segments into memory, does relocation, and binds dynamic-library symbols—all at once. Java's class loading is **lazy, on-demand, and happens at runtime**, in three phases:

### 1. Loading
Based on a class's fully qualified name, find the `.class` byte stream (from the filesystem, a jar, the network, or even generated at runtime), parse it into an internal data structure in the method area / metaspace, and create a `java.lang.Class` object on the heap to represent that class. This step is like `dlopen` opening a library, but at the granularity of a single class.

### 2. Linking (three sub-steps)
- **Verification**: check that the bytecode is legal, that types are safe, and that the stack will not overflow/underflow. C/C++ has no counterpart—it trusts the machine code produced by the compiler; because the JVM may load bytecode of unknown origin, it must perform this layer of security checking.
- **Preparation**: allocate space for static fields and set them to their **zero values** (int→0, reference→null). Note that user-supplied assignments are not executed at this point.
- **Resolution**: replace the **symbolic references** in the constant pool ("method foo of class A") with **direct references** (memory addresses/offsets). This is the equivalent of C's link-time symbol resolution, but it happens at runtime and can be deferred until first use.

### 3. Initialization
Execute the static initializer blocks and the assignments to static fields (i.e. the `<clinit>` method). Triggers include: the first `new`, accessing a static field/method, reflection, etc. The JVM guarantees `<clinit>` **runs exactly once** under multithreading (with a lock), which is why the "static singleton + class loading" idiom is thread-safe.

The code below confirms "lazy + exactly once": `Config`'s static initialization does not happen until its **first active use**, and a second access does not run it again:

```java
static class Config {
    static { System.out.println("  [Config <clinit> runs]"); }
    static final int READY = compute();
    static int compute() { System.out.println("  [compute() runs]"); return 42; }
}
public static void main(String[] a) {
    System.out.println("main start");
    System.out.println("before touching Config");
    int v = Config.READY;   // first active use -> triggers <clinit>
    System.out.println("Config.READY = " + v);
    int v2 = Config.READY;  // second access -> <clinit> does NOT run again
    System.out.println("Config.READY = " + v2);
}
// Output:
// main start
// before touching Config
//   [Config <clinit> runs]
//   [compute() runs]
// Config.READY = 42
// Config.READY = 42
```

Notice that `[Config <clinit> runs]` appears **after** `before touching Config`—class loading really is deferred to the moment it is used.

### Parent Delegation Model

The JVM has multiple class loaders, forming a hierarchy:

```
        Bootstrap ClassLoader   (implemented in C++, loads the java.* core classes)
              ▲  delegate
        Platform ClassLoader    (JDK platform modules)
              ▲  delegate
        Application ClassLoader (your classpath)
              ▲  delegate
        Custom ClassLoader
```

When loading a class, a child loader **first delegates upward** to its parent to try; only if the parent cannot load it does the child load it itself. The benefit: core classes (such as `java.lang.String`) are always loaded by the Bootstrap loader, and you cannot substitute a forged `java.lang.String` for the system class—this is a guarantee of security and consistency. In C/C++, if two `.so` files each carry a symbol of the same name, the behavior depends on link order and is prone to problems; parent delegation mechanically prevents core classes from being "hijacked."

## 3. Execution: Interpreter + JIT Mixed Mode

Once it has the bytecode, how does the JVM execute it? It uses two means together (HotSpot's default "mixed mode"):

### Interpretation
The JVM has a built-in bytecode interpreter that reads bytecode one instruction at a time, looks it up in a table, and executes the corresponding C++ logic. It starts fast (no waiting for compilation), but every instruction carries interpretation overhead, making it one to two orders of magnitude slower than native machine code.

### Hot-Spot Detection + JIT Compilation
The name HotSpot comes from its strategy: **detecting hot code** (Hot Spots). The JVM maintains counters for each method (number of method invocations + number of loop back-edges), and once these exceed a threshold, it deems the method "hot" and hands that bytecode to the **JIT (Just-In-Time) compiler**, which compiles it **at runtime** into native machine code; from then on the machine code executes directly.

HotSpot has two JIT compilers (Tiered Compilation uses them cooperatively):

| Compiler | Role | Characteristics |
|--------|------|------|
| C1 (Client) | fast compilation | few optimizations, fast to compile, steps in first |
| C2 (Server) | deep optimization | aggressive optimizations, slow to compile, used for the hottest code |

Typical path: interpretation → C1 compilation (with performance counters) → C2 recompiles the hottest parts.

**This is why Java "gets faster the longer it runs," and can sometimes approach or even locally exceed C**: the JIT has **runtime information**—which branch is actually taken, that a virtual method actually only ever calls one implementing class, what a loop's bounds are. Based on this it can perform **speculative optimizations** that a static compiler cannot; if it bets wrong, it "de-optimizes" (deoptimization) back to interpretation. `gcc`, by contrast, knows nothing about runtime at compile time and can only be conservative.

### A Few Key JIT Optimizations (with Large Effects on Memory Behavior)

- **Inlining**: embed the callee's method body directly at the call site, eliminating call overhead and opening up the horizon for subsequent optimizations. Equivalent to C's `inline`, but the JIT decides automatically **at runtime** based on hotness.
- **Escape Analysis**: analyze whether a reference to an object "escapes" the current method (whether it is returned, stored into another object on the heap, or made visible to other threads). If it **does not escape**, the JVM can do:
  - **Scalar Replacement**: **break apart** an object that would otherwise be `new`'d on the heap into its individual fields, treating them as ordinary local variables placed directly into **stack-frame slots or even registers**. Note this is not "moving the whole object onto the stack"; rather, the object as a whole **no longer exists at all**—so there is neither heap allocation nor GC reclamation. (HotSpot has no true "whole-object stack allocation"; the mechanism that can save the heap allocation is scalar replacement.)

This point is important for understanding Java memory: at the language level "objects always live on the heap," but at the HotSpot implementation level, escape analysis may keep short-lived, non-escaping objects out of the heap entirely. This closes part of the performance gap between Java and C (which puts structs on the stack). See [Variables and the Memory Model](03-变量与内存模型.md).

## 4. Runtime Memory Regions

A running JVM divides memory into several regions (defined by the JVM spec, implemented by HotSpot):

```
┌───────────────────────────────────────────────────────┐
│                        JVM process                      │
│                                                         │
│  Shared by all threads ↓                                │
│  ┌─────────────────────┐   ┌───────────────────────┐   │
│  │        Heap          │   │  Method Area / Metaspace│  │
│  │  all objects, arrays │   │  class metadata,       │   │
│  │  GC's main battlefield│  │  method bytecode       │   │
│  │  (everything new'd   │   │  runtime constant pool │   │
│  │   lives here)        │   │  (metaspace in native  │   │
│  │                      │   │   memory)              │   │
│  └─────────────────────┘   └───────────────────────┘   │
│                                                         │
│  Private to each thread ↓                               │
│  ┌───────────┐ ┌───────────┐ ┌──────────┐ ┌──────────┐ │
│  │ JVM stack │ │ PC register│ │Native    │ │ ...      │ │
│  │ a pile of │ │ current    │ │method stk│ │one per   │ │
│  │ stack     │ │ instruction│ │for JNI   │ │thread    │ │
│  │ frames    │ │            │ │          │ │          │ │
│  └───────────┘ └───────────┘ └──────────┘ └──────────┘ │
└───────────────────────────────────────────────────────┘
```

| Region | Whose | Stores | C/C++ analogy |
|------|------|--------|-----------|
| Heap | shared | all object and array instances | the malloc heap, but reclaimed automatically by GC |
| Method Area / Metaspace | shared | class structures, method bytecode, runtime constant pool, the **metadata** of static fields (the **values** of the fields live in the `Class` mirror object on the heap, not in metaspace) | code segment + read-only data segment (but can grow at runtime) |
| JVM stack | per-thread | stack frames (local variable table, operand stack) | the thread call stack, logically identical |
| PC register | per-thread | the bytecode address currently being executed | the program counter PC/RIP |
| Native method stack | per-thread | the stack used when executing native (JNI) methods | the native stack on the C side |

A few contrasts with C:
- **Metaspace lives in native memory**, not in the GC heap (this design dates from JDK 8, replacing the old "PermGen"/permanent generation).
- **Heap reclamation is automatic**: there is no `free`/`delete`; the GC reclaims what it determines to be unreachable. You almost never manage heap lifetimes manually—this is the biggest mental difference from C, and it is why in [Error Handling and Resource Management](09-错误处理与资源管理.md), `try-with-resources` targets only "non-memory resources" (files/sockets): memory is left to the GC, but handles still need to be closed explicitly.
- **The semantics of the stack are almost identical to C's**: each method call pushes a stack frame, which is popped on return; if the stack gets too deep it triggers a `StackOverflowError` (the counterpart of C's stack overflow, but the JVM throws an exception rather than segfaulting directly).

## 5. AOT: GraalVM native-image

Java can also, like C, be **compiled Ahead-Of-Time (AOT)** into a native executable. **GraalVM's `native-image`** performs closure analysis at build time, statically compiling the classes and methods used—along with a slimmed-down runtime (including a GC)—into a single standalone executable:

```
native-image Hello        # produces a native executable, hello
./hello                   # no JVM needed, millisecond startup
```

Costs and trade-offs:
- **Advantages**: extremely fast startup, small memory footprint, no JVM to distribute—suitable for CLI tools and Serverless/function-compute cold starts. Here its distribution model closely resembles C's: a single native binary.
- **Costs**: the build must know all the classes that will be used, so **reflection, dynamic proxies, and runtime-generated classes** need extra configuration; you lose the JIT's runtime speculative optimizations, so peak throughput **may be lower than** C2's after a long warm-up.

So, as a memory aid:
- `javac` → bytecode, cross-platform, relies on JVM + JIT runtime optimization (the mainstream for long-running services).
- `native-image` → native code, bound to a platform, fixed at build time (the choice for fast-starting, short-lived processes).
- `gcc/clang` → native code, bound to a platform, fixed at compile time (the only path for C/C++).

## Summary

- `javac` produces **platform-independent bytecode**, not machine code; executing it requires the JVM, a stack-based virtual machine.
- Class loading is **runtime, lazy, and layered (parent delegation)**, and retains complete type information.
- HotSpot **interprets first, then JIT-compiles after detecting hot spots** (C1/C2, inlining, escape analysis); escape analysis can perform **scalar replacement** on non-escaping objects (breaking them into scalars placed in stack-frame slots/registers, rather than whole-object stack allocation).
- Runtime memory is divided into a **shared heap/metaspace** and **per-thread JVM stack/PC/native stack**; the heap is reclaimed automatically by the GC.
- When you need a C-style fast-starting native binary, use GraalVM `native-image` for AOT, at the cost of runtime dynamism and peak optimization.
