# Runtime Model: javac, Bytecode, the JVM, and JIT

The C/C++ compilation pipeline is single-stage: `gcc`/`clang` compiles `.c` source directly into machine code for the target CPU architecture, and after linking you get an executable bound to a specific OS/architecture. Java is **two-stage**: `javac` first compiles source into a platform-independent intermediate representation (bytecode), and then the JVM turns that bytecode into instructions actually executed on the target machine. This chapter lays out every link in that chain, and why the extra bytecode layer exists at all — it's the foundation for understanding everything that follows about the memory model, the call stack, and concurrency.

## 1. The Minimal Compile-and-Run Workflow

The minimal skeleton of a Java program: a **class** containing a `main` method, and the program starts running from `main`.

```java
// Hello.java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, Java");
    }
}
```

A few rules you can act on right away:

- **The file name must exactly match the name of the `public class`**: this class is called `Hello`, so the file must be called `Hello.java`.
- **Compile**: `javac Hello.java` — produces a `Hello.class` file.
- **Run**: `java Hello` — note this is the class name `Hello`, without the `.class` suffix, and not the file name.

```
javac Hello.java
java Hello
```

Output:

```
Hello, Java
```

One more, with a variable and an arithmetic operation, to cement the "compile → run" workflow:

```java
// Add.java
public class Add {
    public static void main(String[] args) {
        int a = 3;
        int b = 4;
        int sum = a + b;
        System.out.println(sum);
    }
}
```

```
javac Add.java
java Add
```

Output:

```
7
```

`javac` compiles, `java` runs — an IDE just executes these two steps automatically for you. What's behind them is what this chapter takes apart.

## 2. From Source to Bytecode: What javac Produces

```java
// Hello.java
public class Hello {
    public static void main(String[] args) {
        int sum = 0;
        for (int i = 0; i < 10; i++) {
            sum += i;
        }
        System.out.println(sum);
    }
}
```

Compile:

```
javac Hello.java     # produces Hello.class
```

The **essential difference** between `javac` and `gcc`: `gcc`'s output targets a real CPU, with an instruction set of x86-64 / ARM64; `javac`'s output targets a hypothetical **stack-based virtual machine**, whose instruction set is "bytecode." Bytecode assumes nothing about the number of registers, the word size, or the byte order — it targets only the abstract machine defined by the JVM specification. This is the technical foundation of "write once, run anywhere": a `.class` file is **byte-for-byte identical** on Windows/Linux/macOS and on x86/ARM.

A `.class` file is a binary container with a strict format (beginning with the magic number `0xCAFEBABE`), containing:

| Section | Contents | C/C++ analogy |
|------|------|-----------|
| constant pool | symbolic references to strings, class names, method names, literals | like `.rodata` + the symbol table, but more structured |
| field table | the class's field declarations | struct member layout information |
| method table | each method's bytecode + metadata | function bodies, but with full type signatures |
| attribute table | line number table, local variable table, annotations, etc. | debug info (like DWARF), but built in |

**Why insert this extra bytecode layer at all, instead of compiling straight to machine code the way `gcc` does**: this is an explicit trade-off between "portability" and "peak performance." Compiling directly to machine code (AOT, Ahead-Of-Time) lets the compiler see the target CPU's full feature set at compile time and perform deep static optimizations, so the program runs at peak speed from the moment it starts — but the artifact is tightly bound to a specific target CPU architecture, operating system, and ABI; move it to a different CPU and you have to recompile. That's the price of C/C++'s "write once, compile separately for each platform" model. Java defers the step of "translating into concrete machine instructions" to runtime: the compiler only has to target an abstract stack-based virtual machine, so the resulting `.class` file is byte-for-byte identical on every machine that has a JVM installed, buying "write once, run anywhere." The cost is that runtime must do one more round of translation (interpretation or JIT compilation), so the program can't reach native-code speed the instant it starts. Section 4's discussion of JIT and Section 6's GraalVM AOT show concretely how this trade-off gets compensated for — or reversed.

Key point: **the `.class` file retains complete type and symbol information**. After you `strip` a C program, almost nothing but machine code remains, with symbols gone; a `.class` file inherently carries class names, method signatures, and field types — this is the material basis on which reflection, dynamic loading, and runtime type checking can exist.

The code below relies on exactly this retained information to read a method's exact signature **at runtime** — something impossible on a `strip`'d C binary:

```java
import java.lang.reflect.Method;

Class<?> c = java.util.ArrayList.class;
System.out.println("class = " + c.getName());
Method m = c.getMethod("add", Object.class);
System.out.println("method = " + m.getName());
System.out.println("returns = " + m.getReturnType().getName());
System.out.println("params = " + m.getParameterCount());
// Output:
// class = java.util.ArrayList
// method = add
// returns = boolean
// params = 1
```

### Viewing Bytecode with javap

`javap` is the disassembler that ships with the JDK, the Java world's equivalent of `objdump -d`:

```
javap -c Hello.class
```

The bytecode of the `main` method looks roughly like this (comments added by me):

```
0:  iconst_0        // push the constant 0 onto the operand stack
1:  istore_1        // pop the top of the stack into local variable slot 1 (i.e. sum)
2:  iconst_0        // push the constant 0
3:  istore_2        // store into slot 2 (i.e. i)
4:  iload_2         // load i onto the stack
5:  bipush 10       // push the constant 10
7:  if_icmpge 20    // if i >= 10 jump to offset 20 (exit the loop)
10: iload_1         // load sum
11: iload_2         // load i
12: iadd            // add the top two stack values, result left on the stack
13: istore_1        // store back into sum
14: iinc 2, 1       // increment slot 2 (i) by 1 directly, without going through the operand stack
17: goto 4          // back to the loop test
20: getstatic ...   // fetch System.out
23: iload_1
24: invokevirtual println
27: return
```

Notice that instructions like `iconst`/`istore`/`iload`/`iadd` **have no register operands**. The JVM is a **stack machine**: the inputs and outputs of an operation flow across the "operand stack," rather than naming registers the way x86 does with `add eax, ebx`. The `i` prefix means the operation is on `int`; there are also `l` (long), `f` (float), `d` (double), and `a` (reference/address). `iinc` is one of the rare instructions that modifies the local variable table directly.

> Every method declares in its bytecode how large an operand stack it needs (`stack=`) and how many local variable slots (`locals=`). These are fixed values computed at compile time, and the JVM allocates stack frames accordingly — see [Methods and the Call Stack](06-函数与调用栈.md).

## 3. Class Loading: Getting a .class into the JVM

When a C program starts, the loader maps the ELF segments into memory, does relocation, and binds dynamic-library symbols — all at once. Java's class loading is **lazy, on-demand, and happens at runtime**, in three phases.

**Why class loading is designed to be lazy**: a typical Java program depends on hundreds or thousands of classes at compile time (the JDK core library plus third-party libraries). If startup eagerly loaded and resolved the entire dependency graph the way C static linking does, startup time would grow linearly with the size of that graph — and most of those classes are never touched during a given run (an exception-handling class on a branch never taken, a feature module never triggered). Lazy loading spreads the cost of loading/linking/initializing a class to the moment it is *first actually used*; a class that's never touched costs nothing at all. This is a direct engineering choice in service of "fast startup." It has a secondary benefit too: a class can come from somewhere that doesn't exist yet (the network, dynamically generated bytecode, an implementation a plugin only supplies at runtime), as long as its byte stream can be found before it's actually used — something impossible under a static-linking model that must see every symbol at link time.

### 1. Loading
Based on a class's fully qualified name, find the `.class` byte stream (from the filesystem, a jar, the network, or even generated at runtime), parse it into an internal data structure in the method area / metaspace, and create a `java.lang.Class` object on the heap to represent that class. This step is like `dlopen` opening a library, but at the granularity of a single class.

### 2. Linking (Three Sub-steps)
- **Verification**: check that the bytecode is legal, that types are safe, and that the stack won't overflow/underflow. C/C++ has no counterpart here — it trusts the machine code produced by the compiler; because the JVM may load bytecode of unknown provenance, it must perform this layer of safety checking.
- **Preparation**: allocate space for static fields and set them to their **zero values** (int→0, reference→null). User-supplied assignments are not executed at this point.
- **Resolution**: replace the **symbolic references** in the constant pool ("method foo of class A") with **direct references** (memory addresses/offsets). This is the equivalent of C's link-time symbol resolution, but it happens at runtime and can be deferred until first use.

### 3. Initialization
Execute the static initializer blocks and the assignments to static fields (i.e. the `<clinit>` method). Triggers include: the first `new`, accessing a static field/method, reflection, and so on. The JVM guarantees `<clinit>` **runs exactly once** under multithreading (with a lock), which is why the "static singleton via class loading" idiom is thread-safe.

The code below confirms "lazy + exactly once": `Config`'s static initialization does not happen until its **first active use**, and a second access does not run it again:

```java
static class Config {
    static { System.out.println("  [Config <clinit> runs]"); }
    static final int READY = compute();
    static int compute() { System.out.println("  [compute() runs]"); return 42; }
}
public static void main(String[] a) {
    System.out.println("main start");
    System.out.println("before touching Config");
    int v = Config.READY;   // first active use -> triggers <clinit>
    System.out.println("Config.READY = " + v);
    int v2 = Config.READY;  // second access -> <clinit> does NOT run again
    System.out.println("Config.READY = " + v2);
}
// Output:
// main start
// before touching Config
//   [Config <clinit> runs]
//   [compute() runs]
// Config.READY = 42
// Config.READY = 42
```

Notice that `[Config <clinit> runs]` appears **after** `before touching Config` — class loading really is deferred to the moment the class is used.

### Parent Delegation Model

The JVM has multiple class loaders, forming a hierarchy:

```
        Bootstrap ClassLoader   (implemented in C++, loads the java.* core classes)
              ▲  delegate
        Platform ClassLoader    (JDK platform modules)
              ▲  delegate
        Application ClassLoader (your classpath)
              ▲  delegate
        Custom ClassLoader
```

When loading a class, a child loader **first delegates upward** to its parent to try; only if the parent can't load it does the child load it itself. The benefit: core classes (such as `java.lang.String`) are always loaded by the Bootstrap loader, and you cannot substitute a forged `java.lang.String` for the system class — a guarantee of security and consistency. In C/C++, if two `.so` files each carry a symbol of the same name, behavior depends on link order and is a common source of bugs; parent delegation mechanically rules out core classes ever being "hijacked."

## 4. Execution: Interpreter + JIT Mixed Mode

Once it has the bytecode, how does the JVM execute it? It uses two means together (HotSpot's default "mixed mode"):

### Interpretation
The JVM has a built-in bytecode interpreter that reads bytecode one instruction at a time, looks it up in a table, and executes the corresponding C++ logic. It starts fast (no waiting for compilation), but every instruction carries interpretation overhead, making it one to two orders of magnitude slower than native machine code.

### Hot-Spot Detection + JIT Compilation
The name HotSpot comes from its strategy: **detecting hot code** (a "hot spot"). The JVM maintains counters for each method (invocation count + loop back-edge count), and once these exceed a threshold, it deems the method "hot" and hands that bytecode to the **JIT (Just-In-Time) compiler**, which compiles it **at runtime** into native machine code; from then on the machine code executes directly.

HotSpot has two JIT compilers (Tiered Compilation uses them cooperatively):

| Compiler | Role | Characteristics |
|--------|------|------|
| C1 (Client) | fast compilation | few optimizations, fast to compile, steps in first |
| C2 (Server) | deep optimization | aggressive optimizations, slow to compile, used for the hottest code |

Typical path: interpretation → C1 compilation (with profiling counters) → C2 recompiles the hottest parts.

**Why JIT is needed at all, instead of picking either pure interpretation or pure AOT**: all three approaches get stuck at one extreme. A pure interpreter has zero startup latency (no waiting for compilation), but every bytecode pays a "fetch–dispatch–execute" overhead, and long-running code stays slow forever. Pure AOT (like `gcc`) can perform deep optimization ahead of time, but at compile time it cannot see information that's only known at runtime — that a particular virtual call site actually only ever hits one implementing class this run, or that a particular loop actually iterates billions of times and is worth aggressively unrolling — so it can only make conservative optimizations based on static assumptions. JIT combines the strengths of both: it starts running immediately via interpretation (or the fastest tier, C1), avoiding any startup wait, while counters continuously observe runtime behavior; only once a piece of code is proven "hot" does the JVM pay the higher compile-time cost of the more aggressive C2, using observed branch probabilities and type information to make **speculative optimizations** a static compiler wouldn't dare attempt. **Tiered Compilation** is exactly the reconciliation of the tension between "startup latency" and "peak throughput": C1 compiles fast with conservative optimizations, getting code off the slow interpreter as quickly as possible; C2 compiles slowly with aggressive optimizations, invested only in the small number of genuinely hot methods, so compile time is never wasted on code that only runs once. A speculative optimization that guesses wrong can **de-optimize**, falling back to the interpreter to re-collect information — the safety net for the fact that runtime information can be stale or incomplete.

**This is why Java "gets faster the longer it runs," and can sometimes approach or even locally exceed C**: the JIT has **runtime information** — which branch is actually taken, that a virtual method actually only ever calls one implementing class, what a loop's actual bounds are. Based on this it can perform **speculative optimizations** that a static compiler cannot; if it guesses wrong, it "de-optimizes" back to interpretation. `gcc`, by contrast, knows nothing about runtime at compile time and can only stay conservative.

### A Few Key JIT Optimizations (With Large Effects on Memory Behavior)

- **Inlining**: embed the callee's method body directly at the call site, eliminating call overhead and opening up the horizon for further optimizations. Equivalent to C's `inline`, but the JIT decides automatically **at runtime**, based on hotness.
- **Escape Analysis**: analyze whether a reference to an object "escapes" the current method (whether it's returned, stored into another object on the heap, or made visible to another thread). If it **does not escape**, the JVM can do:
  - **Scalar Replacement**: **break apart** an object that would otherwise be `new`'d on the heap into its individual fields, treating them as ordinary local variables placed directly into **stack-frame slots or even registers**. Note this is not "moving the whole object onto the stack" — rather, the object as a whole **no longer exists at all**, so there is neither heap allocation nor GC reclamation. (HotSpot has no true "whole-object stack allocation"; the mechanism that can eliminate the heap allocation is scalar replacement.)

This point matters for understanding Java memory: at the language level, "objects always live on the heap," but at the HotSpot implementation level, escape analysis may keep short-lived, non-escaping objects out of the heap entirely. This closes part of the performance gap between Java and C (which puts structs on the stack). See [Variables and the Memory Model](03-变量与内存模型.md).

## 5. Runtime Memory Regions

A running JVM divides memory into several regions (defined by the JVM spec, implemented by HotSpot):

```
┌───────────────────────────────────────────────────────┐
│                         JVM process                     │
│                                                         │
│  Shared by all threads ↓                                │
│  ┌─────────────────────┐   ┌───────────────────────┐   │
│  │        Heap           │   │  Method Area / Metaspace│ │
│  │  all objects, arrays  │   │  class metadata,        │ │
│  │  GC's main battlefield│   │  method bytecode        │ │
│  │  (everything new'd    │   │  runtime constant pool  │ │
│  │   lives here)         │   │  (metaspace in native   │ │
│  │                       │   │   memory)               │ │
│  └─────────────────────┘   └───────────────────────┘   │
│                                                         │
│  Private to each thread ↓                                │
│  ┌───────────┐ ┌───────────┐ ┌──────────┐ ┌──────────┐ │
│  │ JVM stack  │ │ PC register│ │ Native   │ │ ...      │ │
│  │ a pile of  │ │ current    │ │ method   │ │ one per  │ │
│  │ stack      │ │ instruction│ │ stack    │ │ thread   │ │
│  │ frames     │ │            │ │ (for JNI)│ │          │ │
│  └───────────┘ └───────────┘ └──────────┘ └──────────┘ │
└───────────────────────────────────────────────────────┘
```

| Region | Whose | Stores | C/C++ analogy |
|------|------|--------|-----------|
| Heap | shared | all object and array instances | the malloc heap, but reclaimed automatically by the GC |
| Method Area / Metaspace | shared | class structures, method bytecode, the runtime constant pool, and the **metadata** of static fields (the **values** of static fields live in the `Class` mirror object on the heap, not in metaspace) | code segment + read-only data segment (but can grow at runtime) |
| JVM stack | per-thread | stack frames (local variable table, operand stack) | the thread call stack, logically identical |
| PC register | per-thread | the bytecode address currently being executed | the program counter, PC/RIP |
| Native method stack | per-thread | the stack used while executing native (JNI) methods | the native stack on the C side |

A few contrasts with C:
- **Metaspace lives in native memory**, not in the GC heap (this design dates from JDK 8, replacing the old "permanent generation," PermGen).
- **Heap reclamation is automatic**: there's no `free`/`delete`; the GC reclaims whatever it determines to be unreachable. You almost never manage heap lifetimes by hand — this is the single biggest mental shift from C, and it's why in [Error Handling and Resource Management](09-错误处理与资源管理.md), `try-with-resources` targets only "non-memory resources" (files/sockets): memory is left to the GC, but handles still need to be closed explicitly.
- **The semantics of the stack are almost identical to C's**: each method call pushes a stack frame, popped on return; too deep a stack triggers a `StackOverflowError` (the counterpart of a C stack overflow, but the JVM throws an exception rather than segfaulting outright).

## 6. AOT: GraalVM native-image

Java can also, like C, be **compiled Ahead-Of-Time (AOT)** into a native executable. **GraalVM's `native-image`** performs closure analysis at build time, statically compiling the classes and methods that are used — along with a slimmed-down runtime (including a GC) — into a single, standalone executable:

```
native-image Hello        # produces a native executable, hello
./hello                   # no JVM needed, millisecond startup
```

Costs and trade-offs:
- **Advantages**: extremely fast startup, small memory footprint, no JVM to distribute — well suited to CLI tools and Serverless/function-compute cold starts. Here its distribution model closely resembles C's: a single native binary.
- **Costs**: the build must know all the classes that will ever be used, so **reflection, dynamic proxies, and runtime-generated classes** need extra configuration; you lose the JIT's runtime speculative optimizations, so peak throughput **may not match** C2's after a long warm-up.

So, as a memory aid:
- `javac` → bytecode, cross-platform, relies on the JVM + JIT runtime optimization (the mainstream choice for long-running services).
- `native-image` → native code, bound to a platform, fixed at build time (the choice for fast-starting, short-lived processes).
- `gcc/clang` → native code, bound to a platform, fixed at compile time (the only path C/C++ has).

## Summary

- **Workflow**: `javac FileName.java` compiles (the class name must match the file name) → `java ClassName` (no suffix) runs; an IDE just executes these two steps automatically.
- `javac` produces **platform-independent bytecode**, not machine code; the cost is that the JVM — a stack-based virtual machine — must translate it again at runtime, and the payoff is that the same `.class` runs everywhere; this is the trade-off between portability and AOT's peak performance.
- Class loading is **runtime, lazy, and layered (parent delegation)** — laziness spreads the cost to the moment a class is actually used, sparing any startup cost for classes never touched; parent delegation guarantees core classes can never be displaced by a forged version.
- HotSpot **interprets first, then JIT-compiles once it detects hot spots** (C1 compiles fast and steps in first, C2 does deep optimization on the hottest code), and Tiered Compilation is exactly the reconciliation of "startup latency" vs. "peak throughput"; escape analysis can perform **scalar replacement** on non-escaping objects (breaking them into scalars placed in stack-frame slots/registers, rather than whole-object stack allocation).
- Runtime memory splits into a **shared heap/metaspace** and a **per-thread JVM stack/PC/native stack**; the heap is reclaimed automatically by the GC.
- When you need a C-style fast-starting native binary, use GraalVM `native-image` for AOT, at the cost of runtime dynamism and peak optimization.
