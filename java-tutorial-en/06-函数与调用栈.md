# Functions and the Call Stack: The Truth About Methods and Parameter Passing

In C++ you have free functions, member functions, static member functions, function pointers, reference parameters, `std::move`… Java narrows all of this down to a single concept: the **method**. A method always belongs to some class; there are no functions floating free outside a class. But narrowing doesn't mean simple — the **stack-frame layout**, the **dispatch mechanism**, and the ironclad rule of **pass-by-value only** behind method calls are the underlying truths this chapter takes apart.

## Method definitions and overloading

A method signature = method name + parameter type list (**excluding the return type**, same as C++). Overloading is resolved at **compile time** based on the static types of the arguments, on the same principle as C++'s overload resolution. But Java has no default arguments, so you can only emulate them with multiple overloads or varargs.

```java
public class Overload {
    static void f(int x)    { System.out.println("int: " + x); }
    static void f(long x)   { System.out.println("long: " + x); }
    static void f(Object x) { System.out.println("Object: " + x); }

    public static void main(String[] args) {
        f(1);        // int: an exact match always wins
        f(1L);       // long
        f("hi");     // Object
        // overload resolution order: exact match → widening → boxing → varargs
        byte b = 5;
        f(b);        // int: byte is widened to int rather than boxed to Object
    }
}
```

Key point: overload resolution happens at compile time; `javac` binds the call site directly to a specific signature. The object's actual runtime type has no effect whatsoever on **overloading** — that's a different matter entirely from **overriding**, discussed below.

The fact that "overloading looks at the static type" is easily confused with overriding. Consider this example: the same object, with different declared types, selects a different overload:

```java
public class OverloadStatic {
    static void f(Object o) { System.out.println("Object"); }
    static void f(String s) { System.out.println("String"); }

    public static void main(String[] args) {
        Object x = "hi";        // actual runtime type is String
        f(x);                   // Object —— looks only at x's static type Object
        f("hi");                // String —— the literal's static type is String
        f((Object) "hi");       // Object —— the cast changes the static type
    }
}
```

`x` clearly holds a `String` at runtime, yet `f(x)` still selects `f(Object)` — because overloading is fixed at **compile time** by the variable's **declared type**, with the runtime type playing no part. This is exactly the opposite of overriding (which goes through the vtable by the runtime type): one is decided at compile time by the static type, the other at runtime by the dynamic type.

## Java has only pass-by-value (worth repeating)

In C++ you can explicitly distinguish pass-by-value, pass-by-pointer, and pass-by-reference. Java has only one: **pass-by-value**. All the controversy comes from "what is the value being passed for a reference type."

The answer: **what's passed is a copy of the reference value (which you can think of as the object's address)**. Not the object itself, and not a "reference alias" in the C++ sense.

- Primitive types: copy those 4/8 bytes of numeric data.
- Reference types: copy that "reference value pointing to the heap object" (HotSpot enables pointer compression by default on 64-bit, so it's usually a 4-byte oop).

The copy and the original variable point to **the same heap object**, so you can change the object's contents; but reassigning the copy leaves the original variable untouched.

### The swap counterexample: proof you can't change the caller's target

```java
public class Swap {
    static class Box { int v; Box(int v) { this.v = v; } }

    // trying to swap two references — doomed to fail
    static void swap(Box a, Box b) {
        Box t = a; a = b; b = t;   // only swaps the copies a and b's targets
    }

    // can change the object's contents — succeeds
    static void fill(Box x) {
        x.v = 999;                 // writes the field in the heap through the copied reference
    }

    public static void main(String[] args) {
        Box p = new Box(1), q = new Box(2);
        swap(p, q);
        System.out.println(p.v + " " + q.v); // 1 2: the caller's targets are unchanged
        fill(p);
        System.out.println(p.v);             // 999: the object's contents were changed
    }
}
```

Memory diagram (when `swap` reaches `b = t;`):

```
   main frame                    swap frame
 +----------+                 +----------+
 |  p ------|----+       +----|  a ------|--> points back to Box(2)
 |  q ------|--+ |       |    |  b ------|--> points back to Box(1)
 +----------+  | |       |    |  t ------|--> Box(1)
               | |       |    +----------+
               | +-------|-----------+
               |         |           |
               v         |           v
       heap: Box(1) <----+    heap: Box(2)
             v=999            v=2
```

`p` and `q` in `main` point to the same objects from start to finish; what `swap` swaps are the `a` and `b` in its own frame. Once the method returns, the `swap` frame — along with `a`, `b`, and `t` — is popped off the stack and destroyed.

> Want a real swap in Java? You can only return new values, pass in a mutable container (array / holder object), or use fields. There's no C++ `Box*&` or `Box&`.

Comparison table:

| Scenario | C++ | Java |
|------|-----|------|
| Passing a primitive | value/reference/pointer, your choice | value only (copy the number) |
| Passing an object | value (deep/shallow copy ctor) / reference / pointer | value only (copy the reference value, not the object) |
| Changing object contents affects the caller | reference/pointer can | yes (shared same heap object) |
| Changing the target affects the caller | reference/pointer can | **impossible** |

## Stack-frame structure

On each method call, the JVM pushes a **frame** onto the current thread's **Java virtual machine stack**. This aligns with C++'s notion of a call stack, but the structure is explicitly defined by the JVM specification:

```
thread stack (higher address at bottom; illustrated growing downward)
 +--------------------------------------+
 | frame A (main)                       |
 |  Local Variable Table                |  <- like C++ locals + parameter area
 |  Operand Stack                       |  <- workspace for bytecode computation (C++ uses registers / implicit stack)
 |  Frame data: return address, ref to constant pool |
 +--------------------------------------+
 | frame B (a method called by A)       |  <- top of stack, the current active frame
 |  ...                                 |
 +--------------------------------------+
```

- **Local Variable Table**: an array-like structure indexed by slot. `long`/`double` take 2 slots each, everything else takes 1. In an instance method, slot 0 is `this`. The method's maximum number of local variables is computed at compile time and written into the `locals` of the `Code` attribute.
- **Operand Stack**: the JVM is a **stack-based virtual machine** (unlike x86's register machine). `iadd` pops two ints off the operand stack, adds them, and pushes the result back. Its maximum depth is also determined at compile time (`stack`).
- **Frame data**: the return address (the caller's next instruction), a reference to the exception table, and so on.

Seeing the truth with `javap`:

```java
// Frame.java
public class Frame {
    static int add(int a, int b) { int c = a + b; return c; }
}
```

```
$ javac Frame.java && javap -c -v Frame.class
static int add(int, int);
  Code:
   stack=2, locals=3, args_size=2   // operand stack depth 2, local variable table 3 slots (a,b,c)
     0: iload_0        // push slot0 (a) onto the operand stack
     1: iload_1        // push slot1 (b)
     2: iadd           // pop two, add, push back
     3: istore_2       // pop and store into slot2 (c)
     4: iload_2        // push c again
     5: ireturn        // pop as the return value
```

Return values are passed through the operand stack: the callee's `ireturn` pops the top of stack, and the JVM pushes it onto the caller frame's operand stack. Parameter passing is the reverse: the caller pushes the arguments onto its own operand stack, and the `invoke*` instruction consumes them and fills them into the low slots of the new frame's local variable table.

## Method dispatch: static binding vs dynamic binding

In C++, non-virtual functions are statically bound while virtual functions go through the vtable. Java's counterpart is more regular, embodied by **four bytecode instructions**:

| Instruction | Purpose | Binding time |
|------|------|----------|
| `invokestatic` | static methods | static (compile time) |
| `invokespecial` | constructors, `private`, `super.` calls | static |
| `invokevirtual` | ordinary instance methods | dynamic (vtable lookup at runtime) |
| `invokeinterface` | interface methods | dynamic (itable lookup) |
| `invokedynamic` | lambdas, method references, dynamic languages | linked on first use at runtime, then fixed |

### Static binding

`static`, `private`, and `final` methods, as well as **overload resolution**, are all fixed at compile time. A `final` method still uses `invokevirtual`, but the JIT can devirtualize and inline it, since it can never be overridden.

### Dynamic binding and the vtable

Ordinary instance-method calls compile to `invokevirtual`. At runtime the JVM uses the **klass pointer** in the object header to find the actual class's method metadata, and takes the target method's entry from a fixed index in its **virtual method table (vtable)**. This is almost identical to C++'s vptr → vtable mechanism, with the difference:

- C++: the object stores a `vptr`, and the vtable lives in the read-only data segment.
- HotSpot: the klass pointer in the object header (mark word + klass pointer) points to the `InstanceKlass` in Metaspace, with the vtable following right after.

```
object (heap)         InstanceKlass (Metaspace)
+-------------+       +------------------+
| mark word   |       | ... metadata ... |
| klass ptr --|------>| vtable[0] -> ...  |
+-------------+       | vtable[1] -> Dog.speak
| instance    |       | vtable[2] -> ...  |
| fields      |       +------------------+
+-------------+
```

```java
public class Dispatch {
    static class Animal { void speak() { System.out.println("..."); } }
    static class Dog extends Animal { @Override void speak() { System.out.println("Woof"); } }

    public static void main(String[] args) {
        Animal a = new Dog();
        a.speak();   // invokevirtual: static type Animal, at runtime looks up Dog's vtable slot -> "Woof"
    }
}
```

`invokeinterface` is slightly slower: an interface method's vtable index isn't fixed across implementing classes, so it requires an itable (interface method table) lookup or a search; the JIT optimizes this with an inline cache.

## Recursion and StackOverflowError

Each level of recursion pushes a frame. The Java stack size is determined by `-Xss` (default around 512KB–1MB, platform-dependent). Exhausting the stack throws `StackOverflowError` (note it's an `Error`, not an `Exception`, see [Chapter 9](09-错误处理与资源管理.md)). The JVM **does not perform tail-call optimization** (TCO), so even tail recursion consumes stack — an important difference from many functional languages and some C++ compilers.

```java
public class Recur {
    static long depth = 0;
    static void dive() { depth++; dive(); }   // no termination — deliberately blows the stack
    public static void main(String[] args) {
        try { dive(); }
        catch (StackOverflowError e) {
            System.out.println("Blown at depth about: " + depth); // the larger each frame, the fewer levels
        }
    }
}
```

Deep recursion should be rewritten as iteration + an explicit `Deque` simulating the stack, moving the frame data off the thread stack and onto the heap.

## Varargs

`T...` is just `T[]` in bytecode. At the call site the compiler automatically packs the scattered arguments into an **array on the heap** — so every call may incur an array allocation.

```java
public class Varargs {
    static int sum(int... xs) {          // compiles to int[] xs
        int s = 0; for (int x : xs) s += x; return s;
    }
    public static void main(String[] args) {
        System.out.println(sum(1, 2, 3)); // the compiler generates new int[]{1,2,3}
        System.out.println(sum());        // passes an empty array new int[]{}, not null
        int[] arr = {4, 5};
        System.out.println(sum(arr));     // passes the array directly, no extra wrapping
    }
}
```

When overloading is combined with varargs, the fixed-parameter version takes priority over the varargs version (varargs is the last tier of resolution).

## Lambdas and method references: the stage for invokedynamic

A C++ lambda generates an anonymous closure class (with `operator()`) at compile time. Java **does not** generate a `.class` for each lambda at compile time; instead it uses `invokedynamic` to defer the implementation to **runtime**.

```java
import java.util.function.IntBinaryOperator;
public class Lambda {
    public static void main(String[] args) {
        IntBinaryOperator add = (a, b) -> a + b;   // compiles to an invokedynamic call site
        System.out.println(add.applyAsInt(3, 4));
    }
}
```

`javap -c` will show `invokedynamic #N, 0` with a bootstrap method `LambdaMetafactory.metafactory`. The mechanism:

1. The lambda body itself is compiled by `javac` into a **private static/instance method** in the enclosing class (e.g. `lambda$main$0`).
2. The first time execution reaches that `invokedynamic`, the JVM calls the bootstrap method, and `LambdaMetafactory` **dynamically generates at runtime** a class implementing `IntBinaryOperator` whose abstract method forwards to that private method.
3. The generated implementation is bound to that call site's `CallSite`, and subsequent calls hit it directly, with near-zero overhead.

Benefit: it avoids the bloat of "one class file per lambda" and leaves the generation strategy to the runtime (which can later be swapped for a better implementation without changing the bytecode). Method references (`String::length`, `System.out::println`, `ArrayList::new`) go through exactly the same mechanism, except the target method points directly at an existing method rather than a synthesized one.

On variable capture: a local variable captured by a lambda must be **effectively final**. Because what's captured is a **copy of the variable's value** (a copy of the reference value for reference types), stored as a constructor argument when the instance is created — this is close in semantics to C++'s `[=]` capture-by-value, whereas Java does not allow `[&]`-style capture-by-reference of local variables (locals live on the stack and become invalid once the method returns, so a captured reference would dangle).

An intuitive runnable check: each iteration of the loop stores `i` into a fresh `cur` (a new, effectively-final variable per iteration), and each lambda captures its own copy of that value:

```java
import java.util.*;
import java.util.function.Supplier;
public class Capture {
    public static void main(String[] args) {
        List<Supplier<Integer>> fns = new ArrayList<>();
        for (int i = 0; i < 3; i++) {
            int cur = i;              // a new, effectively-final variable each iteration
            fns.add(() -> cur);       // captures the value of cur at that moment
        }
        for (var f : fns) System.out.print(f.get() + " ");  // 0 1 2
        System.out.println();
    }
}
```

The three lambdas print `0 1 2` respectively, because each captured a **copy of `cur`'s value at creation time**, not a shared variable. This contrasts with JavaScript's `var` closures all capturing the same final `i` (often printing `3 3 3`) — Java's "capture a value copy + effectively final" avoids this shared-mutable-state pitfall at the root.

---

Summary: methods belong to classes; parameter passing is always by value (reference types pass a copy of the reference value); a stack frame consists of the local variable table + operand stack + frame data; `invokestatic/special` are statically bound, `invokevirtual/interface` go through the vtable/itable for dynamic dispatch, and `invokedynamic` carries the runtime generation of lambdas.
