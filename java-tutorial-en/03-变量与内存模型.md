# Variables and the Memory Model: What Really Happens in `a = b`

Most of the "spooky bugs" C/C++ programmers run into after switching to Java trace back to a misunderstanding of what actually happens in memory during **assignment, comparison, and parameter passing**. This chapter thoroughly explains stack frames, the heap, and the two camps that variables fall into.

## 1. The Basic Syntax of Declaration and Assignment

Declaring a variable in Java is `Type name = value;`, almost identical to C:

```java
public class VarDemo {
    public static void main(String[] args) {
        // primitive types: the variable directly holds the value
        int age = 20;
        double price = 9.9;

        // reference types: the variable holds a "reference to an object"
        String name = "Ada";              // a String object
        int[] nums = new int[]{1, 2, 3};  // an array object

        System.out.println(age);    // 20
        System.out.println(price);  // 9.9
        System.out.println(name);   // Ada
        System.out.println(nums[0]);// 1

        // assignment: copy the value/reference on the right of = into the variable on the left
        int age2 = age;      // copies the number
        age2 = 99;
        System.out.println(age);     // 20, unaffected

        int[] nums2 = nums;  // copies the "reference"
        nums2[0] = 100;
        System.out.println(nums[0]); // 100! both variables point to the same array
    }
}
```

Changing `age2` doesn't affect `age`, yet changing `nums2[0]` also changes `nums[0]`. This is neither a coincidence nor a bug — it's the core mechanism this chapter is about to lay out in full.

The core conclusion up front, argued point by point below:

> Java variables fall into **two camps**: a **primitive-type** variable holds a "value" directly; a **reference-type** variable holds a "reference value" — a JVM-managed pointer to an object on the heap. `a = b` is always "copy the bits in b into a": for a primitive that copies the value itself, and for a reference type it copies that pointer (from then on both variables point to the same heap object).

## 2. The Two Camps

| Camp | Members | What the variable stores | Where the data is |
|------|------|------------|---------|
| Primitive types | `byte short int long float double char boolean` | **the value itself** (bits) | wherever the variable lives (a stack-frame slot or inlined in an object field) |
| Reference types | classes, interfaces, arrays, `String`, wrapper classes, enums… | **a reference value** (≈ pointer) | the object the reference points to lives on the **heap** |

In C terms:
- A primitive-type variable ≈ `int a;` — the memory of `a` is directly that integer.
- A reference-type variable ≈ `Foo* p;` — the memory of `p` holds an address, and the actual `Foo` lives on the heap (allocated by `new`). **But you cannot do arithmetic on this "pointer," cannot dereference it to take an address, and cannot cast it to an integer** — it's a JVM-managed, constrained pointer.

> HotSpot implementation detail: a reference on a 64-bit JVM usually occupies 8 bytes, but with **compressed pointers (CompressedOops)** enabled it shrinks to 4 bytes (using a 35-bit addressable encoding, on by default for heaps under 32 GB). See [Data Types](04-数据类型.md). The language spec doesn't dictate the concrete representation of a reference, only its semantics.

**Why split into two camps at all, instead of an "everything is an object" model like Python/Smalltalk**: this is a trade-off between performance and uniformity. The "everything is an object" model is conceptually cleaner — every value has a uniform method-call interface, and a container can hold anything without distinction — but the cost is that every integer, every boolean, must be an object on the heap, so even a loop counter's `i++` goes through an object allocation (or at least a boxing operation and a cache lookup) plus a pointer indirection, while a CPU register-level integer add takes a single clock cycle. Java's trade-off is to pull the eight highest-frequency, fixed-size types that are semantically "just a value" (`byte/short/int/long/float/double/char/boolean`) out of the object system entirely, letting them exist directly as values in a stack-frame slot or an object field, with no reference indirection, no extra object-header overhead (see the object-header cost in [Data Types](04-数据类型.md)), and directly mappable by the JIT onto registers for arithmetic. The cost is that primitives lose the uniformity of "everything is an object" — they can't be placed into a generic container that requires `Object` (`List<int>` is illegal), and only once boxed into a wrapper type like `Integer` can they join the object system, which is exactly where the boxing overhead and wrapper-cache issues discussed in [Data Types](04-数据类型.md) come from. This buys "no object overhead for 99% of high-frequency arithmetic" at the cost of "one extra type system that needs manual boxing/unboxing" — rather than making every value uniform but slow.

## 3. Local Variable Table Slots: Where Variables Live

On each method call, the JVM pushes a **stack frame** onto the current thread's **JVM stack**; the frame contains a **Local Variable Table**, made up of several **slots** (the count is fixed at compile time — see `javap` in [Runtime Model](01-运行模型.md)).

- A slot is 32 bits. `int/float/reference` each take 1 slot; `long/double` take 2 consecutive slots.
- Both method parameters and local variables declared inside the method live in this table.

For a primitive-type local variable, the slot holds that value directly. For a reference-type local variable, the slot holds the reference (pointing to the heap). **The object body is never in a slot — always on the heap** (except under escape analysis, see the end of this chapter).

## 4. `a = b`: The Core of the Core

### 4.1 Primitive Types: Copy the Value

```java
int b = 100;
int a = b;   // copy b's bits (100) into a's slot
a = 200;     // changes only a's own slot, no effect on b
// at this moment a=200, b=100
```

Memory diagram (stack-frame local variable table):

```
 before assignment    after a = b          after a = 200
┌──────────┐       ┌──────────┐        ┌──────────┐
│ b │ 100  │       │ b │ 100  │        │ b │ 100  │
├──────────┤       ├──────────┤        ├──────────┤
│ a │  ??  │       │ a │ 100  │←copy    │ a │ 200  │
└──────────┘       └──────────┘        └──────────┘
     Two independent slots, unaffected by each other. Equivalent to C's int a=b;
```

This is exactly the same as C, no surprises.

### 4.2 Reference Types: Copy the Reference (copying the pointer, not the object!)

```java
int[] b = new int[]{1, 2, 3};  // creates the array object on the heap, b holds the reference to it
int[] a = b;                    // copies the "reference value"! a and b point to the same array
a[0] = 99;                      // modifies the heap object through a
System.out.println(b[0]);       // prints 99! because b points to the same object
```

**This is the intuition a C programmer most needs to build**: `int[] a = b;` does not copy the array — it copies the pointer. It's equivalent to C's `int* a = b;` (a shallow pointer copy), and **not** `memcpy`.

Memory diagram:

```
      stack-frame local var table    Heap
   ┌────────────────┐         ┌───────────────────────┐
   │ b │  ref ●──────┼────────▶│ int[3] object          │
   ├────────────────┤    ┌───▶│ [0]=99 [1]=2 [2]=3     │
   │ a │  ref ●──────┼────┘    │ (a[0]=99 changed this) │
   └────────────────┘         └───────────────────────┘
      a and b are two pointers pointing to the same heap object (aliasing)
```

Compared with the primitive-type diagram, the difference is obvious at a glance:

```
   Object a = b  (reference type)      int a = b  (primitive type)
   ┌───────┐                          ┌───────┐
   │ b │●──┼──┐                       │ b │100│
   ├───────┤  │   Heap                ├───────┤
   │ a │●──┼──┴──▶[ object ]          │ a │100│ ← independent copy
   └───────┘  two pointers share      └───────┘  two independent values
              one object
```

Remember it in one line: **assigning a primitive clones the "data," assigning a reference clones the "pointer."** The latter produces **aliasing** — modifying the object through either variable, the other sees it too. This is exactly the same behavior as two C pointers pointing into the same `malloc`'d block.

### 4.3 What If You Want a Real Copy

Java doesn't deep-copy automatically (there's no C++ copy constructor triggered by assignment). To get an independent copy, you must do it explicitly:

```java
int[] a = b.clone();               // array shallow copy, produces a new object
int[] a2 = Arrays.copyOf(b, b.length);
List<T> copy = new ArrayList<>(src);  // a collection's copy constructor
```

Note these are mostly **shallow copies**: the container/array itself is new, but if it holds references, the element references still share the same underlying objects. A deep copy requires you to copy layer by layer.

Testing that word "shallow" — a 2D array's `clone()` only copies the outer references, while the inner arrays remain shared:

```java
int[][] grid = {{1, 2}, {3, 4}};
int[][] copy = grid.clone();   // clone() is a shallow copy: only the outer references are copied
copy[0][0] = 99;               // reaches through to the shared inner array
System.out.println(grid[0][0]);          // 99  <- the original was changed too!
System.out.println(copy[0] == grid[0]);  // true -- the inner arrays are the same object

// a real deep copy must copy the inner arrays one by one
int[][] deep = new int[grid.length][];
for (int i = 0; i < grid.length; i++) deep[i] = grid[i].clone();
deep[1][0] = -1;
System.out.println(grid[1][0]);          // 3   <- the original is safe this time
```

## 5. `==` vs `equals`

Once you understand the two camps, this classic trap follows naturally.

### `==`: Compares "the bits in the variable"
- Primitive types: compares the **value**. `3 == 3` → true. Matches intuition.
- Reference types: compares the **reference** — i.e. "is it the same heap object" (whether the addresses are equal), **without looking at the content**.

```java
String x = new String("hi");
String y = new String("hi");
System.out.println(x == y);        // false! two different heap objects, two different pointers
System.out.println(x.equals(y));   // true! compares content
```

Memory diagram:

```
 ┌───────┐        Heap
 │ x │●──┼──▶[ String "hi" object① ]
 ├───────┤
 │ y │●──┼──▶[ String "hi" object② ]   x==y compares ● vs ●, different targets → false
 └───────┘                              x.equals(y) compares character by character → true
```

### `equals`: Compares "content" (semantic equality)
`equals` is a **method** defined on `Object`, and it can be overridden. The default implementation (`Object.equals`) is just `==` (compares references); but `String`, the wrapper classes, collections, and so on **override** it to compare logical content instead. So:
- To compare string/object content, **always use `equals`**.
- Use `==` only when you really mean "is it the same object," or when comparing primitive types, or comparing against `null`.

Literals add one more wrinkle — the "string constant pool": the same literal reuses the same pooled object, whereas `new String(...)` forces a brand-new object. This throws the difference between `==` (compares references) and `equals` (compares content) into sharp relief:

```java
String p = "hi";
String q = "hi";              // same literal -> same pooled object
String n = new String("hi");  // new forces a new heap object
System.out.println(p == q);         // true  -- both point to the pooled object
System.out.println(p == n);         // false -- n is a different object
System.out.println(p == n.intern());// true  -- intern() returns the pooled one
System.out.println(p.equals(n));    // true  -- content is always equal
```

> Paired with `equals` there's also `hashCode`: the contract is "if equals is true, hashCode must be equal," otherwise putting the object into a `HashMap`/`HashSet` breaks. Details in [Composite Data Structures](07-复合数据结构.md). In C++ you provide `operator==` and `hash` for `unordered_map` — same idea.

### A Counterintuitive Pitfall: the Wrapper-Class Cache

```java
Integer a = 100, b = 100;
System.out.println(a == b);   // true  —— 100 is within the cache range
Integer c = 200, d = 200;
System.out.println(c == d);   // false —— 200 is beyond the cache, two objects
```

`Integer` caches objects for `[-128, 127]`, reusing the same object on boxing, so `==` happens to come out true; outside that range a new object is created, so `==` is false. **This is exactly what proves `==` compares references.** Always use `equals`, or unbox first, when comparing the values of wrapper classes. Details in [Data Types](04-数据类型.md).

## 6. `null`

`null` is the "null reference" of reference types, equivalent to C's null pointer `nullptr` — the variable points to no object.

```java
String s = null;         // s's slot holds the "null reference" value
s.length();              // NullPointerException! "dereferencing" a null reference
```

The difference from C: dereferencing a null pointer in C is undefined behavior (usually a segfault); in Java it **deterministically throws a `NullPointerException` (NPE)**, which can be caught and which carries stack information (Helpful NPE / JEP 358: introduced in JDK 14 but off by default, on by default since JDK 15, able to point out exactly which variable was null). Primitive types **cannot** be `null` (they hold a value, not a reference); `int x = null;` is a compile error. This is also why unboxing a wrapper `Integer` into an `int` throws NPE if it's null.

## 7. `final` Variable Semantics

`final` is often held up as C++'s `const`, but the two must be distinguished precisely: **`final` constrains whether the variable's slot can be reassigned, not whether the object it points to can be modified.**

```java
final int x = 10;
// x = 20;                 // compile error: cannot reassign

final int[] arr = {1, 2, 3};
arr[0] = 99;               // legal! modifies the heap object's content, doesn't touch the arr reference
// arr = new int[5];       // compile error: cannot repoint arr to a different object
```

Understanding this via the memory diagram: `final` locks the slot on the left (it can't be repointed), but it cannot lock the object's content in the heap on the right.

```
 final int[] arr ●═══▶[ 1, 2, 3 ]   ═══ means this reference is locked and cannot be repointed
                        ↑ but the object's content can still be changed: arr[0]=99 is fine
```

- To make "the object itself immutable too," you need the object's own design to provide that (e.g. `String` is inherently immutable, or use an immutable collection like `List.copyOf(...)`). This resembles C++'s distinction between `const int* p` (can't change the pointed-to content) and `int* const p` (can't change the target): **Java's `final` provides only the latter half** — it locks the target, not the content.
- `final` fields: must be assigned exactly once before construction completes, and are immutable afterward; there's a guarantee for multithreaded visibility (once construction is complete, other threads see the correct final-field values).
- `final` local variables/parameters: commonly used so they can be captured by a lambda (a lambda can only capture "effectively final" local variables, see [Methods and the Call Stack](06-函数与调用栈.md)).

**Why `final` locks only the reference and not the content, rather than offering deep freezing the way some other languages do**: the root cause is still the same unifying rule for `a = b` — assignment always copies "the bits in the variable," and all `final` can constrain is whether those bits can subsequently be overwritten; it governs **the variable's storage location**, not the content of whatever heap memory that location points to. Whether an object's content on the heap can change depends on whether the class itself exposes a "mutator" (an array exposes `arr[i]=`, `ArrayList` exposes `add`, while `String` simply provides no mutating method at all) — that's each type's own contract, and there's no uniform, language-level way to "freeze" an arbitrarily shaped object graph (an object may reach other mutable objects recursively through its fields, and deep freezing would require recursively walking and copying the whole graph while also handling cycles). Restricting `final` to this one uniform, cheap, statically verifiable semantics — "lock the storage location" — buys the compiler the ability to perform definite-assignment analysis and multithreaded safe-publication guarantees consistently across **any** type, without having to separately define what "deep immutability" means for every type. When you genuinely need deep immutability, it's left to the specific type to implement it (`String`'s immutability, a `record` paired with immutable collections, `List.copyOf`) — more reliable than the language forcing a generic but semantically fuzzy "deep freeze" on everything.

## 8. Putting Them Into Object Fields

Everything above covers local variables (in stack-frame slots). **An object's fields** follow the same two-camp rules, just with the storage location **inside the object on the heap**:

```java
class Node {
    int value;      // primitive field: the value is inlined directly in the Node object's memory
    Node next;      // reference field: holds a reference to another Node on the heap (or null)
}
```

Memory diagram (a `Node` object on the heap):

```
The Node object on the heap
┌─────────────────────────┐
│ [object header mark+klass]│  ← see the object layout in Chapter 4
│ value : 42              │  ← int value inlined
│ next  : ref ●───────────┼──▶ another Node object (or null)
└─────────────────────────┘
```

This is exactly what a linked list/tree looks like in Java: the structure is completely isomorphic to C's `struct Node { int value; struct Node* next; };`, differing only in that `next` is a managed reference, the object lives on the GC heap, and you don't `free` it.

## 9. Escape Analysis: Language Semantics vs. HotSpot Implementation

The language spec says "objects are always on the heap." But **in HotSpot**, the JIT's **escape analysis** (see [Runtime Model](01-运行模型.md)) may perform **scalar replacement** if it proves that a given `new`'d object never escapes the current method: it breaks the object into a few primitive values placed directly in stack-frame slots or even registers — **never entering the heap at all, needing no GC**.

```java
int distance() {
    var p = new Point(3, 4);   // p does not escape this method
    return p.x * p.x + p.y * p.y;
    // after HotSpot optimization: there may be no new at all — x and y become two stack-based local variables
}
```

So the precise statement is:
- **At the semantic level**: a reference-type object logically lives on the heap, assignment copies the reference, and aliasing holds.
- **At the implementation level**: a non-escaping object may undergo **scalar replacement** (broken into primitive values placed in stack-frame slots/registers, rather than the whole object being moved onto the stack), approaching the efficiency of C's "struct on the stack" — but this is **semantically transparent** to the code you write; you neither need to, nor can, control it directly.

This explains why creating large numbers of short-lived small objects in Java is often not nearly as slow as you'd imagine.

## Summary

1. Variables fall into two camps: **primitives hold values, reference types hold pointers (reference values)**; splitting them this way is a trade-off between performance (avoiding object/boxing overhead for high-frequency arithmetic) and uniformity.
2. `a = b` always copies "the bits in the variable": for a primitive it's a **value copy** (an independent copy), for a reference type it's a **pointer copy** (producing aliasing, sharing the same heap object). To get a copy you must explicitly clone or copy-construct, and the default is a shallow copy.
3. `==` compares bits (for reference types, whether it's the same object); `equals` compares content — always use `equals` to compare object content.
4. `null` is a null reference; dereferencing it **deterministically throws NPE** (not UB); primitive types cannot be null.
5. `final` locks only "what the variable points to," not "the object's content" — because it constrains the bit pattern of the storage location, not the heap memory that location refers to; deep immutability is left to the type itself to implement (`String`, immutable collections), buying a uniform, statically verifiable semantics across any type.
6. Object fields follow the same two-camp rules, just stored inside the heap object; linked lists/trees are isomorphic to C structs, only the references are managed and reclaimed by the GC.
7. HotSpot escape analysis may perform **scalar replacement** on non-escaping objects (breaking them into scalars placed in stack-frame slots/registers, rather than allocating the whole object on the stack), but this is semantically transparent.
