# Error Handling and Resource Management: Exceptions and Garbage Collection

C++ uses exceptions for error propagation, RAII (destructors) for resource release, and manual `delete`/smart pointers for memory management. Java splits these three concerns apart: the exception hierarchy, try-with-resources (an approximation of RAII), and **automatic garbage collection (GC)**. This chapter has two big parts—**exception handling** and **garbage collection**—and each part follows the same order: **first how to write and use it** (throw, catch, close a resource—that's already enough to write business logic), **then what's underneath** (how the exception table works, how the stack unwinds, how objects on the heap get judged as garbage). If you just want to get through day-to-day coding, read the first half of each section; if you want to tune performance or diagnose memory issues, read the second half too.

---

## Getting It Working First: A Quick Look at try/catch

You don't need to understand terms like "exception table" or "stack unwinding" yet to write Java code that throws and catches exceptions right now.

**Throw and catch an exception** (dividing by 0 throws `ArithmeticException`):

```java
public class QuickTry {
    public static void main(String[] args) {
        try {
            int x = 10 / 0;          // throws ArithmeticException
            System.out.println("this line never runs: " + x);
        } catch (ArithmeticException e) {
            System.out.println("caught exception: " + e.getMessage());
        } finally {
            System.out.println("finally always runs");
        }
        System.out.println("program keeps going");
    }
}
// caught exception: / by zero
// finally always runs
// program keeps going
```

The key points: `try` wraps code that might fail; `catch (Type var)` matches by type to grab the exception; the code in `finally` runs whether or not an exception happened (commonly used for cleanup). Once an exception is thrown, execution inside the `try` block never returns to the point where it failed (in the example above, the `println` on the `x` line never runs)—it jumps straight to the matching `catch`.

**Closing a resource once** (`try-with-resources`): in C you have to remember to `fclose` an open file; in Java, as long as a resource implements `AutoCloseable`, writing `try (...)` guarantees it gets closed automatically when you're done—no need to hand-write `finally { close(); }`:

```java
public class QuickTWR {
    // The simplest possible closeable resource: just implement AutoCloseable
    static class SimpleResource implements AutoCloseable {
        SimpleResource() { System.out.println("opening resource"); }
        void use() { System.out.println("using resource"); }
        @Override public void close() { System.out.println("closing resource"); }
    }

    public static void main(String[] args) {
        try (SimpleResource r = new SimpleResource()) {
            r.use();
        }
        System.out.println("close() was already called automatically after the try block ended");
    }
}
// opening resource
// using resource
// closing resource
// close() was already called automatically after the try block ended
```

The resource declared inside the parentheses of `try (SimpleResource r = new SimpleResource())` has its `close()` called automatically exactly once when the `try` block ends—whether it ended normally or via an exception. This is the closest thing Java has to C++'s RAII/destructor pattern.

These two examples are already enough to start writing Java code that "handles errors and manages resources." From here, we go under the hood of exceptions and GC.

## Going Deeper: How Exceptions Propagate Under the Hood

From this point on we're going under the hood. You already know how to write `try/catch/finally` and `try-with-resources`, but a few questions remain unanswered: once an exception is thrown, how does the JVM actually find which `catch` should handle it? Why does `finally` "always run," and how is that implemented at the bytecode level? When `try-with-resources` closes multiple resources, in what order does that happen, and what if `close()` itself throws? The sections below work through these one by one—each feature still starts with a minimal "how to use it" example before digging into the internals.

## The Exception Hierarchy

### How to Use It

Java exceptions split into two big categories, and the practical difference when you're writing code is: **checked exceptions are enforced by the compiler; unchecked exceptions are not**.

```java
import java.io.*;

public class QuickChecked {
    // Checked exception: the method signature must declare throws, or it won't compile
    static void mightFail(boolean fail) throws IOException {
        if (fail) throw new IOException("simulated IO failure");
        System.out.println("read succeeded");
    }

    // Unchecked exception: no throws declaration required
    static int divide(int a, int b) {
        return a / b; // throws ArithmeticException when b=0; the compiler doesn't force you to handle it
    }

    public static void main(String[] args) {
        try {
            mightFail(true);
        } catch (IOException e) {
            System.out.println("caught checked exception: " + e.getMessage());
        }

        try {
            divide(1, 0);
        } catch (ArithmeticException e) {
            System.out.println("caught unchecked exception: " + e.getMessage());
        }
    }
}
// caught checked exception: simulated IO failure
// caught unchecked exception: / by zero
```

If `mightFail` declares `throws IOException`, code that calls it must either catch it with `try/catch` or itself declare `throws IOException` on its own signature to propagate it further—otherwise the compiler flat-out rejects it. The `ArithmeticException` thrown by `divide` has no such constraint: you can catch it or ignore it, and the compiler won't stop you either way.

### Under the Hood, vs. C++

Everything throwable inherits from `Throwable`. It has two main branches:

```
                Throwable
               /         \
           Error        Exception
        (severe JVM-level faults) /       \
     OutOfMemoryError  RuntimeException  others (checked exceptions)
     StackOverflowError  (unchecked)          IOException
                          NullPointerException  SQLException
                          IllegalArgumentException ...
```

| Category | Checked? | Semantics | C++ analogy |
|------|----------|------|----------|
| `Error` | No | Unrecoverable at the JVM level (stack overflow, heap exhaustion); **should not be caught** | Roughly system-level like `std::bad_alloc` |
| `RuntimeException` | No | Programming errors (null pointer, out of bounds, illegal argument); should be prevented by code | Logic errors, like something an `assert` should have caught |
| Checked `Exception` | Yes | Anticipated external failures (IO, network); **the compiler forces you to handle them** | No C++ equivalent; the deprecated `throw()` specification is a counterexample |

**Checked vs. unchecked** is a compile-time constraint unique to Java: a checked exception must either be declared in the method signature's `throws` clause or be handled locally with `try/catch`, or the code won't compile. C++'s early dynamic exception specifications (`throw(...)`) were deprecated because they were hard to use; Java kept the feature, though it too has been much debated.

```java
import java.io.*;
public class Checked {
    // Checked: must declare throws, otherwise a compile error
    static String readFirst(String path) throws IOException {
        try (BufferedReader r = new BufferedReader(new FileReader(path))) {
            return r.readLine();
        }
    }
    // Unchecked: no declaration needed, the compiler does not stop you
    static int div(int a, int b) { return a / b; } // b=0 throws ArithmeticException
}
```

## try/catch/finally and Stack Unwinding

### How to Use It

The `finally` block runs whether the `try` returns normally or throws—the `QuickTry` example above already demonstrated this: even though `10 / 0` threw an exception, the `println` in `finally` still ran. That makes it a good place for "do this regardless of success or failure" cleanup, like logging or releasing a resource that isn't `AutoCloseable`.

### Under the Hood, vs. C++

When an exception is thrown, the JVM starts from the current method and consults its **exception table**—`javac` generates one [start PC, end PC, handler PC, caught type] record per `try` block. If the current frame has no matching handler, that frame is popped (stack unwinding) and the search walks back up the call chain until a matching `catch` is found or the bottom of the thread stack is reached (at which point the thread terminates and prints a stack trace).

`finally` executes whether the outcome is normal or exceptional. At the bytecode level, the compiler **copies** the `finally` code onto every normal/exceptional exit path (or falls back to the exception table plus `athrow`); this is how its "always executes" guarantee is implemented.

```java
public class Finally {
    static int tricky() {
        try {
            return 1;          // return value 1 is computed and pushed first
        } finally {
            return 2;          // the finally's return overrides it -> actually returns 2 (anti-pattern!)
        }
    }
    public static void main(String[] args) {
        System.out.println(tricky()); // 2
    }
}
```

> Pitfall: a `return` or a thrown exception inside `finally` swallows the return value/exception from `try`. Never `return` inside `finally`.

## try-with-resources: Java's RAII

### How to Use It

The `QuickTWR` example earlier already showed the most basic usage: a resource class implements `AutoCloseable`, gets written inside the `try (...)` parentheses, and its `close()` is automatically called when the `try` block ends—regardless of whether the block ran to completion normally or threw partway through.

### Under the Hood, vs. C++

C++ releases resources deterministically via the **destructor** when an object leaves scope (RAII). The reclamation timing of a Java object is nondeterministic (GC decides), so you cannot rely on `finalize` to close a file. JDK 7 introduced **try-with-resources**: any object implementing `AutoCloseable` (with `close()`) is **guaranteed** to have `close()` called—in reverse of declaration order—when the try block ends. This is the closest thing Java has to deterministic RAII-style resource management.

```java
import java.io.*;
public class TWR implements AutoCloseable {
    private final String name;
    TWR(String name) { this.name = name; System.out.println("open " + name); }
    @Override public void close() { System.out.println("close " + name); }

    public static void main(String[] args) {
        // Reverse-order close: close b first, then a (like C++ local objects destructed in reverse)
        try (TWR a = new TWR("A"); TWR b = new TWR("B")) {
            System.out.println("body");
            throw new RuntimeException("boom");
        } catch (RuntimeException e) {
            System.out.println("caught: " + e.getMessage());
            // An exception thrown by close() is attached to the primary exception as a "suppressed exception":
            for (Throwable s : e.getSuppressed()) System.out.println("suppressed: " + s);
        }
    }
    // Output: open A / open B / body / close B / close A / caught: boom
}
```

Difference from C++: the `close()` call site is instrumented by the compiler at the try exit, whereas **the object's memory itself** is still reclaimed later by the GC. In other words, "deterministic release of resources (fd/handle)" and "nondeterministic reclamation of memory" are two separate things.

The `close()` above doesn't throw, so `getSuppressed()` is empty. Only when **both the try body and `close()` throw** do you see the "suppressed exception" mechanism: the primary exception is the one thrown by the try body, and the two `close()` exceptions are attached to it in reverse order:

```java
public class Suppressed implements AutoCloseable {
    private final String name;
    Suppressed(String name){ this.name = name; }
    @Override public void close() {
        System.out.println("close " + name);
        throw new RuntimeException("close-fail-" + name);   // close() itself throws too
    }
    public static void main(String[] args) {
        try (Suppressed a = new Suppressed("A"); Suppressed b = new Suppressed("B")) {
            throw new RuntimeException("body-fail");         // primary exception
        } catch (RuntimeException e) {
            System.out.println("primary   : " + e.getMessage());
            for (Throwable s : e.getSuppressed())            // close's exceptions attach here
                System.out.println("suppressed: " + s.getMessage());
        }
    }
}
// close B                   <- closed in reverse order
// close A
// primary   : body-fail     <- the try body's exception wins as the primary
// suppressed: close-fail-B  <- close's exceptions are "suppressed", attached in reverse order
// suppressed: close-fail-A
```

## Multi-Catch and Exception Chaining

### How to Use It

A single `catch` can list several types separated by `|`, letting one block of handling logic catch different kinds of exceptions without duplicating code:

```java
public class QuickMultiCatch {
    static void parse(String s) {
        if (s == null) s.length();   // calling a method on null triggers NullPointerException directly
        Integer.parseInt(s);         // throws NumberFormatException when the format is wrong
    }

    public static void main(String[] args) {
        String[] inputs = { null, "abc" };
        for (String s : inputs) {
            try {
                parse(s);
            } catch (NumberFormatException | NullPointerException e) {
                // one catch handles both types
                System.out.println("parse failed: " + e.getClass().getSimpleName());
            }
        }
    }
}
// parse failed: NullPointerException
// parse failed: NumberFormatException
```

### Under the Hood, vs. C++

```java
public class MultiCatch {
    static void handle(String s) {
        try {
            Integer.parseInt(s);
        } catch (NumberFormatException | NullPointerException e) {
            // Multi-catch: one catch handles multiple types; e is implicitly final, its type is the common supertype
            // Exception chaining: preserve the "cause" to make it easier to locate the original fault
            throw new IllegalStateException("handling failed: " + s, e);
        }
    }
    public static void main(String[] args) {
        try { handle(null); }
        catch (IllegalStateException e) {
            System.out.println(e.getMessage());
            System.out.println("cause: " + e.getCause());
            // cause: java.lang.NumberFormatException: Cannot parse null string
            // Note: in modern JDKs, Integer.parseInt(null) explicitly null-checks internally and throws
            // NumberFormatException("Cannot parse null string") directly, rather than NPE—
            // catching both types here still makes sense: with a different parsing approach (e.g. calling
            // s.length() by hand first), the null-pointer path would instead go down the
            // NullPointerException branch.
        }
    }
}
```

Exception chaining (`initCause` / a constructor taking a cause) lets you translate an exception without losing the original stack trace; when printed, a `Caused by:` section appears.

## Garbage Collection (GC)

In C and C++, memory is entirely your responsibility: `malloc`/`free`, `new`/`delete`—forget a `free` and you've got a leak, `free` twice and you've got a crash. Java has no `delete`—the object you `new` up gets released whenever and by whomever an underlying mechanism decides: the **Garbage Collector (GC)**. At the right moment it sweeps the heap and reclaims the memory of objects "nobody can reach anymore." The price is that you don't know the exact moment of reclamation, and collection itself can briefly stall your program (a "pause," aka STW). Understanding how GC works is what lets you read its logs and tune its parameters when something goes wrong.

GC runs automatically the whole time; there isn't much API you call directly—mainly `WeakReference`/`SoftReference`/`Cleaner`/`System.gc()`—and those still follow the "how to use it, then under the hood" order. Purely conceptual material like "the generational hypothesis" or "how collectors have evolved" goes straight into the under-the-hood sections, since there's no code to write to trigger it.

## The Generational Hypothesis and Heap Layout

This section is purely conceptual—GC runs on its own, so you don't write code to trigger it, but understanding how the heap is partitioned is what lets you follow "which collector to pick" and "which tuning knob to turn" later on.

GC is built on the **weak generational hypothesis**: the vast majority of objects die young; objects that live long rarely point to young objects. So the heap is managed generationally:

```
                Java heap
 +---------------------------------------------------+
 | Young Gen                                         |
 |  +----------+  +----------+  +----------+         |
 |  |  Eden    |  | Survivor |  | Survivor |         |
 |  | (new obj)|  |   S0     |  |   S1     |         |
 |  +----------+  +----------+  +----------+         |
 +---------------------------------------------------+
 | Old Gen (Tenured)  long-lived objects             |
 +---------------------------------------------------+
   (Metaspace lives in native memory, holds class metadata, not in the heap)
```

- New objects are allocated in **Eden** (pointer bumping, extremely fast, like advancing a stack top).
- When Eden fills, a **Minor GC** is triggered: live objects are copied to one of the Survivor spaces and their age is incremented; each survived Minor GC raises the age, and once it exceeds the threshold (default 15, `-XX:MaxTenuringThreshold`) the object is promoted to the old generation. The two Survivor spaces take turns being the "copy target," so one is always empty (a copying algorithm that avoids fragmentation).
- When the old generation fills, a **Major GC** (old-generation collection) is triggered; a **Full GC** usually means whole-heap plus Metaspace collection—the slowest, and to be avoided. (Note: "Major GC" is **classic terminology for generational collectors**; modern collectors like G1/ZGC use different terms—G1 distinguishes young / mixed / full, where `Full GC` is its degraded fallback whole-heap collection.)

| GC type | Scope | Trigger | Pause |
|---------|----------|------|------|
| Minor GC | Young gen | Eden full | Short |
| Major GC | Old gen | Old gen full | Long |
| Full GC | Whole heap + Metaspace | Old gen/Metaspace insufficient, `System.gc()` | Longest |

## Reachability Analysis and GC Roots

Java **does not use reference counting** (which cannot handle circular references); instead it uses **reachability analysis**: starting from a set of **GC Roots**, it traverses the reference graph, and any object that is unreachable is eligible for reclamation.

GC Roots include:

- References in each thread's **stack-frame local variable table** (objects referenced by currently executing methods);
- Objects referenced by **static fields**;
- Constant-pool references and JNI global references;
- Objects held by synchronization locks, etc.

```
GC Roots
  | thread-stack locals  ---> A ---> B      (A, B reachable, alive)
  | static fields        ---> C
                          C <---> D    (C, D reference each other)
                                E ---> F  (E, F reference each other but no Root reaches them => both reclaimed)
```

Note the last group: `E` and `F` form a reference cycle. Reference counting would wrongly consider them alive, whereas reachability analysis correctly judges them to be garbage. This is exactly why Java does not use reference counting.

## Collector Evolution and the STW Trade-off

**STW (Stop-The-World)**: some GC phases must pause all application threads (which must reach a "safepoint") to obtain a consistent snapshot of the heap. The evolution of collectors is essentially the history of **continually shrinking STW**.

| Collector | Traits | STW | Suited for |
|--------|------|-----|------|
| Serial | Single-threaded, simple | STW throughout, relatively long | Small heaps, client-side |
| Parallel (throughput-first) | Multithreaded parallel collection | STW but parallel | Batch processing, throughput-heavy |
| CMS (deprecated in JDK 9, **removed in JDK 14**) | Concurrent mark-sweep, low-pause pioneer | Partly concurrent | Low-latency old gen (has fragmentation problems) |
| **G1** (default since JDK 9) | Region-partitioned, predictable pause target | Short STW, mostly concurrent | General-purpose large heaps |
| ZGC | Colored pointers + read barriers | Pauses <1ms, independent of heap size | Very large heaps, low latency |
| Shenandoah | Concurrent compaction | Extremely short pauses | Low latency |

The core of the trade-off is **throughput vs. latency**: Parallel maximizes work done per unit time (good for background batch jobs); ZGC/Shenandoah sacrifice a bit of throughput for nearly imperceptible pauses (good for interactive/online services). There is no "best," only "best fit for the scenario."

Selection: `-XX:+UseG1GC`, `-XX:+UseZGC`, `-XX:+UseParallelGC`.

## The Deprecation of finalize and the Cleaner

### How to Use It

For releasing resources, always prefer the `try-with-resources`/explicit `close()` you already know—it's deterministic and runs the instant you're done. `Cleaner` is only a safety net for "in case you forgot to call `close()`"; it shouldn't be your primary mechanism:

```java
import java.lang.ref.Cleaner;

public class QuickCleaner implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();
    private final Cleaner.Cleanable cleanable;

    QuickCleaner() {
        // register a "fallback cleanup action for if close() gets forgotten"
        cleanable = CLEANER.register(this, () -> System.out.println("fallback cleanup triggered"));
    }

    @Override
    public void close() {
        System.out.println("explicit close() called"); // preferred path: deterministic, immediate
        cleanable.clean();                              // trigger cleanup manually; the fallback action won't run again afterward
    }

    public static void main(String[] args) {
        try (QuickCleaner r = new QuickCleaner()) {
            System.out.println("using resource");
        }
        // because try-with-resources already called close(), the fallback Cleaner action only ever prints once
    }
}
// using resource
// explicit close() called
// fallback cleanup triggered
```

Calling `cleanable.clean()` inside `close()` runs the cleanup action immediately, once. If you forget to write `try-with-resources` and the object eventually becomes unreachable, `Cleaner`'s internal background thread will also run that same action automatically after GC—but that's a "last-resort safety net," not something you should rely on as your main path.

### Under the Hood, vs. C++

C++'s destructor runs deterministically. Java's `Object.finalize()` once aimed to play this role, but it **runs at an unpredictable time, may never run, slows down GC, and can even resurrect objects**. It was marked `@Deprecated` in JDK 9 and further deprecated starting in JDK 18.

The replacement: **`java.lang.ref.Cleaner`**, which registers a cleanup action to be run by a background thread once the object becomes phantom-reachable. But it is still nondeterministic—**always release critical resources explicitly with try-with-resources**, and use Cleaner only as a last-resort fallback.

```java
import java.lang.ref.Cleaner;
public class CleanerDemo implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();
    // The cleanup action must be a static/standalone class and must never hold a reference to the enclosing object, or the object can never be reclaimed
    static class State implements Runnable {
        public void run() { System.out.println("releasing native resource (fallback)"); }
    }
    private final Cleaner.Cleanable cleanable;
    CleanerDemo() { cleanable = CLEANER.register(this, new State()); }
    @Override public void close() { cleanable.clean(); } // explicit path: deterministic release
}
```

## Reference Strength: Strong / Soft / Weak / Phantom

### How to Use It

An ordinary `Object o = new Object()` is a **strong reference**—as long as it's reachable, GC will never touch it. If you want the effect of "I might still want this object, but it's fine to sacrifice it first when memory gets tight," wrap it in a `WeakReference`:

```java
import java.lang.ref.WeakReference;

public class QuickWeakRef {
    public static void main(String[] args) {
        Object obj = new Object();
        WeakReference<Object> weak = new WeakReference<>(obj); // wrap it in a weak reference
        System.out.println("before get(): " + (weak.get() != null)); // true -- the strong reference obj still exists

        obj = null;      // drop the only strong reference
        System.gc();      // suggest the JVM run a GC (not guaranteed to happen immediately)
        System.out.println("after get(): " + weak.get()); // very likely null
    }
}
// before get(): true
// after get(): null
```

`weak.get()` returns either the referenced object or `null` (if it's already been reclaimed). As long as the strong reference `obj` still exists, `weak.get()` keeps working; once `obj` is set to null and no other strong reference points to it, GC is free to reclaim it, and `weak.get()` turns into `null`.

### Under the Hood, vs. C++

`java.lang.ref` provides four levels of reference strength, which determine an object's "right to survive" in the eyes of the GC:

| Strength | Class | GC behavior | Typical use |
|------|-----|---------|----------|
| Strong reference | plain `Object o = ...` | Never reclaimed while reachable | Default |
| Soft reference | `SoftReference` | Reclaimed only **when memory is low** | Memory-sensitive caches |
| Weak reference | `WeakReference` | Reclaimed at the next GC | `WeakHashMap`, canonicalizing maps |
| Phantom reference | `PhantomReference` | Can never retrieve the object; only serves as a reclamation notification | Replacing finalize for cleanup |

```java
import java.lang.ref.*;
public class Refs {
    public static void main(String[] args) {
        ReferenceQueue<byte[]> queue = new ReferenceQueue<>();
        byte[] data = new byte[1024];
        // Weak reference + reference queue: after the object is reclaimed, the ref is enqueued, letting you detect reclamation
        WeakReference<byte[]> weak = new WeakReference<>(data, queue);
        data = null;          // drop the only strong reference
        System.gc();          // suggest a GC (not guaranteed to run immediately)
        System.out.println("get() after reclamation: " + weak.get()); // very likely null
        Reference<?> polled = queue.poll();                 // the reclaimed reference has been enqueued
        System.out.println("enqueued: " + (polled == weak));
    }
}
```

**ReferenceQueue** is the accompanying mechanism: when a referenced object is reclaimed, the JVM places the corresponding `Reference` object into the queue, and your cleanup thread `poll`s it to run the follow-up work (this is exactly how Cleaner is implemented internally).

The most common real-world use of weak references is `WeakHashMap`: the key is held only weakly, so once nothing else strongly references it, the entry disappears automatically after a GC—a natural fit for canonicalizing caches, avoiding the "add-only, never-remove" leak:

```java
import java.util.*;
public class WeakMap {
    static void seed(Map<Object,String> m) {          // keys live only in this stack frame
        m.put(new Object(), "v");
        m.put(new Object(), "v");
    }
    public static void main(String[] args) throws Exception {
        Map<Object,String> weak = new WeakHashMap<>();
        seed(weak);
        System.out.println("before GC  weak.size = " + weak.size());
        System.gc();
        Thread.sleep(100);
        System.out.println("after  GC  weak.size = " + weak.size());
    }
}
// before GC  weak.size = 2
// after  GC  weak.size = 0    <- keys have no strong reference; entries are cleared after GC reclaims them
// (Note: if you keep the keys in main's local variables, they become GC Roots and won't be reclaimed)
```

## What a Memory Leak Looks Like in Java

Having GC does not mean there are no memory leaks. A leak = **an object that is no longer logically useful but is still reachable from GC Roots**, so GC does not reclaim it. Common forms:

- **Long-lived collections that only add, never remove**: a `static Map`/cache is continually `put` into, and the keys and values stay reachable forever. Use `WeakHashMap` or cap the capacity (e.g., Guava/Caffeine caches).
- **Listeners/callbacks not deregistered**: an observer registers but forgets to `removeListener`, and stays referenced by the subject.
- **`ThreadLocal` not `remove`d**: thread-pool threads live a long time, so the `ThreadLocal`'s value stays reachable for the life of the thread. Always `remove()` when done.
- **Inner classes implicitly holding the outer instance**: a non-static inner class/anonymous class holds `Outer.this`, and if its lifetime is longer than the outer object's, the outer object cannot be reclaimed (the same caution as in the Cleaner section).

## Common OOMs and Tuning Parameters

`OutOfMemoryError` is an `Error`, subdivided into different messages:

| Error message | Meaning | Direction |
|----------|------|------|
| `Java heap space` | The heap can't hold it | Increase `-Xmx` or hunt for a leak (heap-dump analysis) |
| `GC overhead limit exceeded` | Spending 98%+ of time in GC but reclaiming very little | Usually a leak or too small a heap |
| `Metaspace` | Class metadata over the limit (lots of dynamically generated classes) | `-XX:MaxMetaspaceSize`, check for classloader leaks |
| `unable to create new native thread` | Thread count over the system/memory limit | Reduce threads, tune `-Xss` |
| `Direct buffer memory` | Off-heap direct memory exhausted | `-XX:MaxDirectMemorySize` |

Common parameters:

```
-Xms512m -Xmx4g                  # initial/max heap (in production, set them equal to avoid resize jitter)
-Xss1m                           # per-thread stack size
-XX:MaxMetaspaceSize=256m        # Metaspace upper bound
-XX:+UseG1GC -XX:MaxGCPauseMillis=200  # use G1 and set a pause target
-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof  # dump the heap on OOM
-Xlog:gc*                        # print GC logs (unified logging in JDK 9+), for analysis
```

Diagnostic workflow: use `-Xlog:gc*` to observe GC frequency and pauses → grab the `.hprof` on OOM → use MAT/VisualVM to analyze the dominator tree and find who is holding a large object.

## Summary

- **At the usage level**: `try/catch` catches exceptions by type, and `finally` is guaranteed to always run; `try-with-resources` makes any `AutoCloseable` resource call `close()` automatically when you're done with it; `catch (A | B e)` catches multiple types in one block; checked exceptions must be declared with `throws` on the method signature or caught locally, while unchecked exceptions have no such constraint; GC runs automatically without your intervention, but `WeakReference`/`SoftReference`/`Cleaner` are the handful of related APIs you can actively use—resource release should still go through explicit `close()` first, with `Cleaner` only as a fallback.
- **Under the hood**: exceptions are located via the exception table each method carries, and if no handler matches, the stack unwinds frame by frame; `finally`'s "always executes" guarantee is implemented via compile-time code duplication; try-with-resources calls `close()` in reverse declaration order, and any exception thrown by `close()` gets attached to the primary exception as suppressed; exception chaining preserves the root cause via `cause`. GC doesn't use reference counting (which can't handle cycles), instead judging liveness via reachability analysis starting from GC Roots; the heap is split into Eden/Survivor/old generation based on the weak generational hypothesis, with Minor/Major/Full GC covering progressively larger scopes; the evolution of collectors from Serial to G1/ZGC is the history of shrinking STW pauses; `finalize` is deprecated, replaced by `Cleaner` plus `PhantomReference`/`ReferenceQueue`; a memory leak in Java shows up as "logically useless but still reachable." Master these, and you can command the JVM's heap with the same confidence you manage C++ memory.
