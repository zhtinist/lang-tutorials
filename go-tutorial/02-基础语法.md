# 基础语法(与 C/C++ 对照)

Go 的表层语法和 C 有明显差异——包声明、类型写在变量名之后、`:=` 短声明——这些不是随手挑的记号,每一条都对应一个具体的设计取舍。本章逐条对照 C,讲清楚 Go 为什么在这些地方与 C 分道扬镳。

```go
package main // 每个 .go 文件第一行:声明属于哪个包

import "fmt" // 引入标准库里的 fmt 包,用来做格式化输出

func greet(name string) string { // 函数:名字在前,参数名+参数类型,再是返回类型
	return "hello, " + name
}

func main() { // 程序入口
	var year int = 2026    // 声明一个 int 变量,显式给类型和初值
	message := greet("Go") // 用 := 声明并推导类型(更常见的写法)

	fmt.Println(message)         // hello, Go
	fmt.Println("当前年份:", year) // 当前年份: 2026
}
```

运行结果:

```
hello, Go
当前年份: 2026
```

下面逐项讨论这套记号背后的差异。

## 一、程序结构:package 与 import

C 里没有"模块"这个语言级概念,靠 `#include` 做文本拼接、靠链接器解析符号。Go 用 **package** 作为编译和封装的基本单位:

```go
package main // 声明本文件属于 main 包;可执行程序的入口包必须叫 main

import (
	"fmt"      // 标准库
	"os"

	"example.com/myapp/util" // 自己模块内的包,路径来自 go.mod
)

func main() {
	fmt.Println("hello")
	os.Exit(0)
}
```

**一个目录 = 一个包**。同目录下所有 `.go` 文件必须声明同一个 package 名。包内的标识符互相可见,不需要前置声明或头文件——Go 没有头文件,也没有前向声明。编译器一次性看到整个包的所有源文件,所以你可以在文件后面定义、在前面调用,顺序无所谓。`import` 导入的是**包**(编译产物 + 导出信息),不是文本包含,导入路径是模块路径 + 目录,不是文件路径。

**未使用的 import 和未使用的局部变量是编译错误,不是警告。** 这条规则在其他语言里通常只是 linter 或编译器警告,Go 把它提到编译错误级别,是刻意的。理由是**代码腐化(code rot)**的现实成因:未使用的 import/变量之所以会长期存在,恰恰是因为它们不影响程序运行,没有任何东西强迫开发者去清理。警告会在嘈杂的编译输出里被无视,CI 通常也不会因为警告而失败;一旦某个模式"能跑就行",团队协作下这类残留会持续累积——半年后没人知道某个 import 是否还有用,删还是不删全凭猜测,静态分析工具也难以区分"确实没用"和"预留给未来"。Go 把这类判断从"最好清理"变成"必须清理",消灭的是判断本身的模糊地带:能编译,就意味着每个 import 和每个局部变量都在被使用,阅读代码时不需要额外去确认"这个 import 是不是死代码"。代价是开发时的摩擦感——写代码写到一半、变量还没用上时会被打断;这也是为什么几乎所有 Go 编辑器插件都在保存时自动补全/删除 import,把这次摩擦转嫁给工具而不是人。

## 二、`func` 与 `main`

```go
func add(a int, b int) int { // 参数类型在名字之后;返回类型在参数列表之后
	return a + b
}

// 相同类型的相邻参数可合并写:
func add2(a, b int) int { return a + b }

func main() { /* 无参数、无返回值;进程入口 */ }
```

命令行参数从 `os.Args` 读取(`os.Args[0]` 是程序自身路径,之后是各个参数):

```go
package main

import (
	"fmt"
	"os"
)

func main() {
	fmt.Println(len(os.Args)) // 1 个可执行文件路径 + N 个命令行参数
	if len(os.Args) > 1 {
		fmt.Println("第一个参数:", os.Args[1])
	}
}
```

用 `./prog hello world` 运行,输出:

```
3
第一个参数: hello
```

和 C 的 `int add(int a, int b)` 相比,最扎眼的是**类型写在名字后面**。这不是标新立异,第七节专门讲原因。`main` 函数不接收 `argc/argv`,也不返回退出码——命令行参数从 `os.Args` 取,退出码用 `os.Exit(n)` 或从 `main` 正常返回(=0)。

## 三、变量声明:`var` 与 `:=`,以及零值机制

两种声明方式:

```go
var x int          // 显式声明,未赋初值 → 自动零值 0
var y = 42         // 有初值,类型由右侧推导(此处为 int)
var z int = 42     // 全写,通常冗余

s := "hello"       // 短变量声明:仅函数内可用,= var s = "hello" 的简写
```

`:=` 只能在函数内部用;包级变量必须用 `var`。`:=` 左侧至少要有一个是新变量。

### 零值机制(zero value)

C 里局部变量不初始化就是未定义的垃圾值(栈上残留),读它是 bug 之源。Go 规定:任何变量声明后若无显式初值,一律被初始化为该类型的"零值",且零值是确定的:

| 类型 | 零值 |
|------|------|
| 数值类型 (int/float/…) | `0` |
| `bool` | `false` |
| `string` | `""`(空串,不是 nil) |
| 指针、slice、map、chan、func、interface | `nil` |
| struct | 各字段递归取零值 |
| array | 各元素取零值 |

这催生了 Go 的一个设计哲学:**"零值可用"(zero value is useful)**。很多类型被刻意设计成零值就是一个立即可用的合法状态,比如 `var buf bytes.Buffer` 不用初始化就能直接 `buf.WriteString(...)`,`var mu sync.Mutex` 零值就是未锁定状态。这与 C++ 需要构造函数把对象带到合法状态形成对比。细节在[第 3 章](03-变量与内存模型.md)展开。

直接把各类型的零值打印出来看看(在 C 里这些没初始化的变量全是栈上垃圾):

```go
package main

import "fmt"

type User struct {
	Name string
	Age  int
	Tags []string
}

func main() {
	var i int
	var f float64
	var b bool
	var s string
	var p *int
	var u User // struct:各字段递归取零值

	fmt.Printf("%d %g %t %q %v\n", i, f, b, s, p) // 0 0 false "" <nil>
	fmt.Printf("%+v\n", u)                        // {Name: Age:0 Tags:[]}
	fmt.Println(u.Tags == nil)                    // true
}
```

注意 `string` 的零值是空串 `""` 而不是 `nil`,而 slice 的零值是 `nil`。

## 四、无隐式类型转换——必须显式

数值类型之间的转换,用"目标类型名当函数"调用:

```go
var i int = 3
f := float64(i) // 显式把 int 转成 float64
fmt.Println(f)  // 3
```

这是 Go 最"严格"、最让 C 程序员一开始不适应的规则之一。**Go 不存在任何隐式数值转换**:

```go
var i int = 3
var f float64 = 1.5

// var r float64 = i + f  // 编译错误!int 和 float64 不能直接相加
var r float64 = float64(i) + f // 必须显式转换 float64(i)

var a int32 = 1
var b int64 = 2
// a + b            // 编译错误!int32 和 int64 是不同类型
a + int32(b)        // OK

var p int = 10
// var q int8 = p   // 编译错误!即使数值放得下也不行
var q int8 = int8(p) // OK,但你要自己对溢出负责
```

C 里 `int + double`、`char + int`、有符号无符号混用都会悄悄发生整型提升和转换,是无数隐蔽 bug 的温床。Go 直接禁止,把每一次转换都变成你眼睛看得见的代码。**唯一的例外是"无类型常量"**——字面量常量在被赋给具体类型前是"无类型"的,能按需适配:

```go
var f2 float64 = 3    // OK,常量 3 是无类型的,适配成 float64
const big = 1 << 40   // 无类型常量,可赋给 int64 等
```

常量规则详见[第 4 章](04-数据类型.md)。

## 五、类型为什么写在名字后面

第三节里 `var x int`、`x := 0` 这种"名字在前"的写法,其实是精心设计的结果。C 的声明语法遵循"声明模仿使用"(declaration reflects use),结果是著名的螺旋式阅读地狱:

```c
int (*fp)(int, int);        // fp 是"指向 接收两个 int、返回 int 的函数"的指针
char *(*arr[10])(void);     // 读到头晕
```

Go 反其道而行,采用**从左到右、名字在前、类型在后**的语法:

```go
var fp func(int, int) int          // fp 的类型是 func(int, int) int,一眼读完
var arr [10]func() *byte           // arr 是 10 个 func() *byte 的数组
var p *int                         // p 是 *int
```

好处有二:(1) **可读性线性化**,总是"名字 + 类型",从左往右念就是自然语言;(2) **语法无歧义,利于单遍快速解析**(呼应[第 1 章](01-运行模型.md)讲的编译快)。C 里 `a * b;` 到底是乘法表达式还是声明一个指针,编译器必须查符号表才知道;Go 里声明必有 `var`/`func`/`type` 引导,不看语义就能解析。

## 六、多返回值——不再靠出参指针

```go
func divmod(a, b int) (int, int) {
	return a / b, a % b
}

q, r := divmod(17, 5) // q=3, r=2

// 惯用的 (结果, error) 模式:
n, err := fmt.Println("hi")
if err != nil { /* 处理错误 */ }

// 用 _ 丢弃不需要的返回值:
q2, _ := divmod(10, 3)
```

C 里函数只能返回一个值,想返回多个就得用**出参指针**或返回 struct:

```c
int divmod(int a, int b, int *rem) { *rem = a % b; return a / b; }
```

Go 原生支持多返回值,这是它最实用的特性之一,也是错误处理惯例的基础。底层上,多返回值就是通过栈/寄存器返回多个值(Go 1.17 起用基于寄存器的调用约定),你不必再手动传出参指针。为什么这是 Go 的设计选择而不只是语法糖,以及它如何撑起 `(result, error)` 这套惯用法,详见[第 6 章](06-函数与调用栈.md)和[第 9 章](09-错误处理与资源管理.md)。

## 七、常量与 `iota`

`const` 声明编译期常量。`iota` 是一个在 const 块内**逐行自增**的计数器(从 0 开始),用来简洁地生成枚举:

```go
const (
	Sunday = iota // 0
	Monday        // 1(表达式沿用上一行,iota 自增)
	Tuesday       // 2
	Wednesday     // 3
)

// 位标志(类似 C 的 1<<n 枚举):
const (
	Read    = 1 << iota // 1 << 0 = 1
	Write               // 1 << 1 = 2
	Execute             // 1 << 2 = 4
)

// 跳过某些值用 _:
const (
	_  = iota             // 丢弃 0
	KB = 1 << (10 * iota) // 1<<10
	MB                    // 1<<20
	GB                    // 1<<30
)
```

把上面两种用法真跑一遍,看看 `iota` 的表达式是如何逐行展开的:

```go
package main

import "fmt"

const (
	_  = iota             // 0,丢弃
	KB = 1 << (10 * iota) // 1 << 10
	MB                    // 1 << 20
	GB                    // 1 << 30
)

const (
	Read    = 1 << iota // 1 << 0
	Write               // 1 << 1
	Execute             // 1 << 2
)

func main() {
	fmt.Println(KB, MB, GB)                        // 1024 1048576 1073741824
	fmt.Println(Read, Write, Execute)              // 1 2 4
	fmt.Printf("%b %b %b\n", Read, Write, Execute) // 1 10 100
}
```

运行结果:

```
1024 1048576 1073741824
1 2 4
1 10 100
```

`iota` 在每个 `const (...)` 块里独立计数,每新起一行(ConstSpec)加 1。它比 C 的 `enum` 更灵活(可带表达式),但也仅限编译期常量,不能用于运行时值。关键在于:上面第 2 行写的 `1 << (10 * iota)` 表达式被后续行**沿用**,而 `iota` 每行自增,于是自动生成了 1KB/1MB/1GB;位标志那组则生成了互不重叠的 1/2/4,可以按位或组合。

## 八、可见性:靠首字母大小写,而非关键字

```go
package util

func Public()  {} // 包外可用:util.Public()
func private() {} // 仅本包内可用

type User struct {
	Name string // 导出字段,外部可读写
	age  int    // 未导出字段,外部不可见
}
```

C++ 用 `public/private`,Go 用一个极简规则:**标识符首字母大写 = 导出(包外可见,相当于 public);小写 = 未导出(仅包内可见,相当于 private)**。这个规则对包级函数、类型、变量、以及 struct 字段、方法一律适用。

没有 `public/private/protected` 关键字,可见性一眼从名字看出。**封装边界是"包",不是"类"**——同包内的代码可以访问彼此的未导出成员,这和 C++ 以类为边界不同。

## 九、无分号:自动分号插入

Go 的语法里语句结尾其实有分号,但**你几乎从不写它**——词法分析器按规则自动插入。规则简化版:**如果一行的最后一个 token 是标识符、字面量、`)`/`]`/`}`、或 `return`/`break` 等,就在行尾自动补一个分号**。

这带来一个必须记住的硬性后果:**`{` 不能另起一行**。C/Java 里可以:

```c
if (x)
{           // C 里合法
}
```

Go 里这样写会在 `if x` 后被插入分号,变成语法错误。所以左花括号必须跟在同一行:

```go
if x {   // 唯一正确写法
}

func f() {   // 同理
}
```

这不是风格建议,是语言强制。也正因为分号是自动的,`gofmt` 才能放心地统一所有人的排版。

## 十、`gofmt`:格式不再是争论话题

C/C++ 世界里,大括号风格、缩进用 tab 还是空格、对齐方式,每个团队吵个没完——这类争论在代码审查(code review)里格外消耗精力,因为它们和代码正确性毫无关系,却占据了大量评论和往返。Go 官方带一个 `gofmt`(以及 `go fmt`),**规定了唯一正确的格式**:tab 缩进、固定的空格规则、import 分组排序等,没有配置项、没有风格可选。

这条设计的直接目标是**从代码审查里彻底移除格式话题**:review 只讨论逻辑是否正确、设计是否合理,不会有人在 PR 里评论"这里该用 2 个空格"。它同时消灭了 diff 噪音——一个人重新格式化了整个文件,不会在 diff 里产生和实际改动无关的大片变更,因为所有人写出来的格式本就一致。社区约定俗成:提交前一律 `gofmt`,IDE 通常保存即格式化。结果是:**所有 Go 代码看起来都一样**,你读任何开源库都无需适应新风格。这不是语法特性,但它是 Go 文化的一部分,值得从第一天就接受。

## 十一、C 与 Go 差异速查表

| 方面 | C/C++ | Go |
|------|-------|-----|
| 声明语法 | 类型在前 `int x;` | 类型在后 `var x int` / `x := 0` |
| 未初始化局部变量 | 垃圾值 | 确定的零值 |
| 隐式数值转换 | 有(整型提升等) | **无,必须显式** |
| 头文件/前向声明 | 需要 | 不需要 |
| 多返回值 | 无(靠出参/struct) | 原生支持 |
| 封装边界 | 类 (`public`/`private`) | 包(首字母大小写) |
| 未用变量/import | 警告或忽略 | **编译错误** |
| 分号 | 手写 | 自动插入(`{` 不能换行) |
| 格式 | 团队各异 | `gofmt` 强制统一 |
| 枚举 | `enum` | `const` + `iota` |
| 三目运算符 `?:` | 有 | **没有**(用 if) |
| 指针算术 | 有 | **没有** |

## 小结

- **用法层面**:文件以 `package` 开头、用 `import` 引入依赖;函数用 `func 名(参数 类型) 返回类型` 定义;变量用 `var`/`:=` 声明;`const` + `iota` 生成枚举;导出靠首字母大小写;`{` 必须跟在同一行。
- **底层层面**:类型写在名字后是为了可读性线性化与单遍快速解析;零值机制让"零值可用"成为可能;无隐式转换从语言层面消灭了 C 的整型提升类 bug;多返回值靠寄存器 ABI 实现,取代出参指针;可见性以包为边界而非类;未用 import/变量强制编译报错是为了消灭代码腐化里"删还是不删"的模糊地带;`gofmt` 强制统一格式,是为了把格式话题从代码审查里彻底移除。
