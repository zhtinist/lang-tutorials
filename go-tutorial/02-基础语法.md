# 基础语法(与 C/C++ 对照)

这一章的目标是让你——一个 C/C++ 老手——用最快速度把语法层面的差异建立起来。我们不讲"什么是变量、什么是函数",只讲**Go 和 C 在同一件事上做法为何不同、以及背后的取舍**。凡是能用一句 C 类比说清的,就不啰嗦。

## 一、程序结构:package 与 import

C 里没有"模块"这个语言级概念,靠 `#include` 做文本拼接、靠链接器解析符号。Go 用 **package** 作为编译和封装的基本单位:

```go
package main // 声明本文件属于 main 包;可执行程序的入口包必须叫 main

import (
	"fmt"      // 标准库
	"os"

	"example.com/myapp/util" // 自己模块内的包,路径来自 go.mod
)

func main() {
	fmt.Println("hello")
	os.Exit(0)
}
```

关键点:

- **一个目录 = 一个包**。同目录下所有 `.go` 文件必须声明同一个 package 名。包内的标识符互相可见,不需要前置声明或头文件——**Go 没有头文件,也没有前向声明**。编译器一次性看到整个包的所有源文件,所以你可以在文件后面定义、在前面调用,顺序无所谓。
- `import` 导入的是**包**(编译产物 + 导出信息),不是文本包含。导入路径是模块路径 + 目录,不是文件路径。
- **未使用的 import 是编译错误**,不是警告。未使用的局部变量同理。Go 用编译错误强制你保持代码干净(这点常让 C 程序员一开始很烦躁,但它消灭了"僵尸 import")。

## 二、`func` 与 `main`

```go
func add(a int, b int) int { // 参数类型在名字之后;返回类型在参数列表之后
	return a + b
}

// 相同类型的相邻参数可合并写:
func add2(a, b int) int { return a + b }

func main() { /* 无参数、无返回值;进程入口 */ }
```

和 C 的 `int add(int a, int b)` 相比,最扎眼的是**类型写在名字后面**。这不是标新立异,后面第五节专门讲原因。

`main` 函数不接收 `argc/argv`,也不返回退出码。命令行参数从 `os.Args` 取,退出码用 `os.Exit(n)` 或从 `main` 正常返回(=0)。

## 三、变量声明:`var` 与 `:=`,以及零值机制

两种声明方式:

```go
var x int          // 显式声明,未赋初值 → 自动零值 0
var y = 42         // 有初值,类型由右侧推导(此处为 int)
var z int = 42     // 全写,通常冗余

s := "hello"       // 短变量声明:仅函数内可用,= var s = "hello" 的简写
```

### 零值机制(zero value)——比 C 安全的一点

C 里局部变量不初始化就是**未定义的垃圾值**(栈上残留),读它是 bug 之源。Go 规定:**任何变量声明后若无显式初值,一律被初始化为该类型的"零值"**,且零值是确定的:

| 类型 | 零值 |
|------|------|
| 数值类型 (int/float/…) | `0` |
| `bool` | `false` |
| `string` | `""`(空串,不是 nil) |
| 指针、slice、map、chan、func、interface | `nil` |
| struct | 各字段递归取零值 |
| array | 各元素取零值 |

这催生了 Go 的一个设计哲学:**"零值可用"(zero value is useful)**。很多类型被刻意设计成零值就是一个立即可用的合法状态,比如 `var buf bytes.Buffer` 不用初始化就能直接 `buf.WriteString(...)`,`var mu sync.Mutex` 零值就是未锁定状态。这与 C++ 需要构造函数把对象带到合法状态形成对比。细节在[第 3 章](03-变量与内存模型.md)展开。

`:=` 只能在函数内部用;包级变量必须用 `var`。`:=` 左侧至少要有一个是新变量。

## 四、无隐式类型转换——必须显式

这是 Go 最"严格"、最让 C 程序员一开始不适应的规则之一。**Go 不存在任何隐式数值转换**:

```go
var i int = 3
var f float64 = 1.5

// var r float64 = i + f  // 编译错误!int 和 float64 不能直接相加
var r float64 = float64(i) + f // 必须显式转换 float64(i)

var a int32 = 1
var b int64 = 2
// a + b            // 编译错误!int32 和 int64 是不同类型
a + int32(b)        // OK

var p int = 10
// var q int8 = p   // 编译错误!即使数值放得下也不行
var q int8 = int8(p) // OK,但你要自己对溢出负责
```

C 里 `int + double`、`char + int`、有符号无符号混用都会悄悄发生整型提升和转换,是无数隐蔽 bug 的温床。Go 直接禁止,把每一次转换都变成你眼睛看得见的代码。**唯一的例外是"无类型常量"**——字面量常量在被赋给具体类型前是"无类型"的,能按需适配:

```go
var f2 float64 = 3    // OK,常量 3 是无类型的,适配成 float64
const big = 1 << 40   // 无类型常量,可赋给 int64 等
```

常量规则详见[第 4 章](04-数据类型.md)。

## 五、类型为什么写在名字后面

C 的声明语法遵循"声明模仿使用"(declaration reflects use),结果是著名的螺旋式阅读地狱:

```c
int (*fp)(int, int);        // fp 是"指向 接收两个 int、返回 int 的函数"的指针
char *(*arr[10])(void);     // 读到头晕
```

Go 反其道而行,采用**从左到右、名字在前、类型在后**的语法:

```go
var fp func(int, int) int          // fp 的类型是 func(int, int) int,一眼读完
var arr [10]func() *byte           // arr 是 10 个 func() *byte 的数组
var p *int                         // p 是 *int
```

好处有二:(1) **可读性线性化**,总是"名字 + 类型",从左往右念就是自然语言;(2) **语法无歧义,利于单遍快速解析**(呼应[第 1 章](01-运行模型.md)讲的编译快)。C 里 `a * b;` 到底是乘法表达式还是声明一个指针,编译器必须查符号表才知道;Go 里声明必有 `var`/`func`/`type` 引导,不看语义就能解析。

## 六、多返回值——不再靠出参指针

C 里函数只能返回一个值,想返回多个就得用**出参指针**或返回 struct:

```c
int divmod(int a, int b, int *rem) { *rem = a % b; return a / b; }
```

Go 原生支持多返回值,这是它最实用的特性之一,也是错误处理惯例的基础:

```go
func divmod(a, b int) (int, int) {
	return a / b, a % b
}

q, r := divmod(17, 5) // q=3, r=2

// 惯用的 (结果, error) 模式:
n, err := fmt.Println("hi")
if err != nil { /* 处理错误 */ }

// 用 _ 丢弃不需要的返回值:
q2, _ := divmod(10, 3)
```

底层上,多返回值就是通过栈/寄存器返回多个值(Go 1.17 起用基于寄存器的调用约定),你不必再手动传出参指针。错误处理惯例见[第 9 章](09-错误处理与资源管理.md)。

## 七、常量与 `iota`

`const` 声明编译期常量。`iota` 是一个在 const 块内**逐行自增**的计数器(从 0 开始),用来简洁地生成枚举:

```go
const (
	Sunday = iota // 0
	Monday        // 1(表达式沿用上一行,iota 自增)
	Tuesday       // 2
	Wednesday     // 3
)

// 位标志(类似 C 的 1<<n 枚举):
const (
	Read    = 1 << iota // 1 << 0 = 1
	Write               // 1 << 1 = 2
	Execute             // 1 << 2 = 4
)

// 跳过某些值用 _:
const (
	_  = iota             // 丢弃 0
	KB = 1 << (10 * iota) // 1<<10
	MB                    // 1<<20
	GB                    // 1<<30
)
```

`iota` 在每个 `const (...)` 块里独立计数,每新起一行(ConstSpec)加 1。它比 C 的 `enum` 更灵活(可带表达式),但也仅限编译期常量,不能用于运行时值。

## 八、可见性:靠首字母大小写,而非关键字

C++ 用 `public/private`,Go 用一个极简规则:**标识符首字母大写 = 导出(包外可见,相当于 public);小写 = 未导出(仅包内可见,相当于 private)**。这个规则对包级函数、类型、变量、以及 struct 字段、方法一律适用:

```go
package util

func Public()  {} // 包外可用:util.Public()
func private() {} // 仅本包内可用

type User struct {
	Name string // 导出字段,外部可读写
	age  int    // 未导出字段,外部不可见
}
```

没有 `public/private/protected` 关键字,可见性一眼从名字看出。**封装边界是"包",不是"类"**——同包内的代码可以访问彼此的未导出成员,这和 C++ 以类为边界不同。

## 九、无分号:自动分号插入

Go 的语法里语句结尾其实有分号,但**你几乎从不写它**——词法分析器按规则自动插入。规则简化版:**如果一行的最后一个 token 是标识符、字面量、`)`/`]`/`}`、或 `return`/`break` 等,就在行尾自动补一个分号**。

这带来一个必须记住的硬性后果:**`{` 不能另起一行**。C/Java 里可以:

```c
if (x)
{           // C 里合法
}
```

Go 里这样写会在 `if x` 后被插入分号,变成语法错误。所以左花括号必须跟在同一行:

```go
if x {   // 唯一正确写法
}

func f() {   // 同理
}
```

这不是风格建议,是语言强制。也正因为分号是自动的,`gofmt` 才能放心地统一所有人的排版。

## 十、`gofmt`:格式不再是争论话题

C/C++ 世界里,大括号风格、缩进用 tab 还是空格、对齐方式,每个团队吵个没完。Go 官方带一个 `gofmt`(以及 `go fmt`),**规定了唯一正确的格式**:tab 缩进、固定的空格规则、import 分组排序等。社区约定俗成:提交前一律 `gofmt`。IDE 通常保存即格式化。

结果是:**所有 Go 代码看起来都一样**,你读任何开源库都无需适应新风格。这不是语法特性,但它是 Go 文化的一部分,值得从第一天就接受。

## 十一、C 与 Go 差异速查表

| 方面 | C/C++ | Go |
|------|-------|-----|
| 声明语法 | 类型在前 `int x;` | 类型在后 `var x int` / `x := 0` |
| 未初始化局部变量 | 垃圾值 | 确定的零值 |
| 隐式数值转换 | 有(整型提升等) | **无,必须显式** |
| 头文件/前向声明 | 需要 | 不需要 |
| 多返回值 | 无(靠出参/struct) | 原生支持 |
| 封装边界 | 类 (`public`/`private`) | 包(首字母大小写) |
| 未用变量/import | 警告或忽略 | **编译错误** |
| 分号 | 手写 | 自动插入(`{` 不能换行) |
| 格式 | 团队各异 | `gofmt` 强制统一 |
| 枚举 | `enum` | `const` + `iota` |
| 三目运算符 `?:` | 有 | **没有**(用 if) |
| 指针算术 | 有 | **没有** |

有了这张表,你已经能读懂大部分 Go 代码了。下一章进入全书最重要的内容:`a = b` 在内存里到底发生了什么——这是理解 Go 一切数据行为的基石。
