# 复合数据结构的内部实现

slice、map、channel 是 Go 里"看起来像引用类型"的三兄弟:把它们赋值给另一个变量、或传进函数,修改会互相影响,行为很像 C++ 的引用或指针。但它们本质上仍是**彻底的值类型**——每一种都只是一个很小的、固定大小的"表头(header)"结构,表头本身按值拷贝,表头里的指针字段指向堆上真正的数据([回顾「变量与内存模型」中的值拷贝](03-变量与内存模型.md))。搞清每种表头的字段,就能精确预测每次赋值、传参、`append` 到底拷贝了什么、有没有分配、会不会产生别名(aliasing)——这是本章的核心目标。

## 一、array:真正的值类型

数组用 `[N]T` 声明,`N` 是编译期常量,是类型的一部分——`[3]int` 和 `[4]int` 是不同类型:

```go
var a [3]int          // 24 字节,全部在当前栈帧里,零值 {0,0,0}
b := a                // 逐字节拷贝整整 24 字节(和 C 的 struct 赋值一样)
b[0] = 100            // 不影响 a
fmt.Println(a, b)     // [0 0 0] [100 0 0]
```

这和 C 里 `int a[3]; int b[3]; b = a;`(C 其实不允许数组直接赋值,得 `memcpy`)行为一致——**拷贝整块内存**。传参也是拷贝整块:把 `[1024]int` 传进函数会复制 8KB。数组是 Go 里少数几个"看到什么就是什么"的类型,没有任何隐藏指针,这是它作为 slice 底层存储的基础,但也正因为传参代价随长度线性增长,日常代码里几乎没人直接传数组,都传 slice。

```
array [3]int 在内存中:
┌────────┬────────┬────────┐
│ int[0] │ int[1] │ int[2] │   连续 24 字节,就是数据本身
└────────┴────────┴────────┘
```

## 二、slice:三元组表头

slice 是日常用的"动态数组"。创建方式有三种:字面量、`make`、从数组/slice 切出:

```go
s1 := []int{10, 20, 30}   // 字面量
s2 := make([]int, 3)      // make(T, len): 长度3, 元素全为零值 [0 0 0]
s3 := make([]int, 0, 5)   // make(T, len, cap): 长度0, 预留容量5

s1 = append(s1, 40)       // 追加元素
fmt.Println(s1)           // [10 20 30 40]
fmt.Println(s1[1])        // 20  —— 按下标读
s1[1] = 99                // 按下标写
fmt.Println(s1)           // [10 99 30 40]

sub := s1[1:3]            // 切片: 取下标 1(含)到 3(不含)
fmt.Println(sub)          // [99 30]
```

slice 的运行时表示是 `runtime.slice`(源码 `runtime/slice.go`):

```go
type slice struct {
    array unsafe.Pointer // 指向底层数组首元素
    len   int            // 当前长度
    cap   int            // 底层数组从 array 起的容量
}
```

三个机器字(64 位下 24 字节)。**slice 变量本身只有这 24 字节**,数据在别处:

```
s := []int{10, 20, 30}

slice header(栈上, 24 字节)        底层数组(通常在堆上)
┌───────────┐                      ┌────┬────┬────┐
│ array ────┼─────────────────────▶│ 10 │ 20 │ 30 │
│ len = 3   │                      └────┴────┴────┘
│ cap = 3   │
└───────────┘
```

### 赋值 / 传参:拷贝表头,不拷贝数据

```go
s := []int{10, 20, 30}
t := s          // 拷贝 24 字节表头;t.array 与 s.array 指向同一块数组!
t[0] = 999      // 改的是共享的底层数组
fmt.Println(s)  // [999 20 30] —— s 也变了(别名 aliasing)
```

这是 C++ 程序员最容易踩的地方:`t := s` 看着像 `std::vector` 的深拷贝,其实更像 `std::span` / 裸指针拷贝——共享底层缓冲。传参同理,把 slice 传进函数,函数内对元素的修改对调用方可见(但对表头本身的修改——比如 `append` 导致重新分配——不可见,见下文)。**设计上的取舍**:如果 slice 是深拷贝语义,每次赋值/传参都要按元素数量线性拷贝,大 slice 传参会变得像传大数组一样昂贵;Go 选择"表头浅拷贝、数据共享",换来 O(1) 的赋值和传参代价,代价是程序员必须自己理解共享关系,写出上面这种别名副作用。

### 从数组/slice 切片:`s[low:high]`

切片操作**不分配、不拷贝数据**,只造一个新表头指向原数组的一段:

```go
a := [5]int{0, 1, 2, 3, 4}
s := a[1:3]     // len=2, cap=4(从下标1到数组末尾), array 指向 &a[1]
```

```
a: ┌───┬───┬───┬───┬───┐
   │ 0 │ 1 │ 2 │ 3 │ 4 │
   └───┴───┴───┴───┴───┘
         ▲
         │ s.array
   s = {array:&a[1], len:2, cap:4}   ← cap 一直数到底层数组末尾
```

`len` 是你能索引的范围,`cap` 是不重新分配就能 `append` 到的范围。`s[i]` 越界(`i >= len`)会 panic;这是 Go 的边界检查,和 C 的"随便越界读脏数据"不同(编译器能证明安全时会消除检查)。

### append 与扩容策略

`append` 是理解 slice 的核心。逻辑:

1. 若 `len < cap`:直接写到 `array[len]`,`len++`,**不分配、不拷贝**,返回的新表头与旧的共享底层数组。
2. 若 `len == cap`:分配一块更大的新数组,把旧数据 `memmove` 过去,新表头指向新数组,**与旧 slice 脱钩**。

```go
s := make([]int, 0, 2) // len=0 cap=2
s = append(s, 1)       // len=1, 原地写, 无分配
s = append(s, 2)       // len=2, 原地写, cap 用满
s = append(s, 3)       // len=3, cap 不够 → 分配新数组(通常 cap 变 4)、拷贝
```

**扩容倍数为什么是这套曲线(gc 实现,`runtime.growslice`,随版本微调,以运行时为准)**:纯粹的固定倍数(比如永远翻倍)在 slice 变大后会浪费大量内存——一个几十万元素的 slice 每次扩容都多分配几十万个元素的冗余空间;纯粹的固定增量(每次 +K)在 slice 小的时候又太慢——退化成 O(n) 次重新分配。Go 用"小 slice 激进翻倍、大 slice 放缓增速"的分段策略,在两者之间取折中:

| 旧 cap | 新 cap 增长策略 |
|--------|----------------|
| 需求超过 2×旧 cap | 直接用需求值 |
| 旧 cap < 256 | 约翻倍(2×) |
| 旧 cap ≥ 256 | 约 1.25×,逐步平滑到 1.25(避免大 slice 浪费内存) |

之后还会按元素大小做内存对齐、向 size class 取整(见[第09章 mspan/size class](09-错误处理与资源管理.md))。所以打印出的 `cap` 未必正好是翻倍值。

> 经验:能预估大小就 `make([]T, 0, n)` 预留容量,避免多次分配+拷贝——等价于 C++ `vector::reserve`。

### append 别名陷阱(最经典的坑)

因为"cap 够就原地写、不够才分配",同一底层数组可能被多个 slice 共享,`append` 可能悄悄改到别人:

```go
a := []int{1, 2, 3, 4, 5}
s := a[0:2]              // len=2 cap=5, 与 a 共享底层数组
s = append(s, 99)        // cap 足够 → 原地写 a[2]!
fmt.Println(a)           // [1 2 99 4 5] —— a 被改了!
```

```
append 前:                append(s, 99) 后(cap 够, 原地):
a = [1 2 3 4 5]            a = [1 2 99 4 5]   ← 第3个元素被覆盖
s = a[0:2] ────┐          s = {len:3, cap:5}
               │              s[2] 写进了 a[2] 的位置
```

反过来,如果扩容发生了,两者又会脱钩,后续修改互不影响——**同一段代码的行为取决于运行时 cap 够不够**,这是 bug 温床,根源仍是上面那条取舍:表头浅拷贝换来了性能,但"什么时候共享、什么时候脱钩"这件事本身是运行时的隐藏决策,不体现在类型里。防御手段:

- 用**三索引切片**限制 cap:`s := a[0:2:2]`(`s[low:high:max]`,`cap = max-low`)。此时 cap==len,下一次 append 必然分配新数组,不会踩到 `a`。
- 想要真正独立的副本,用 `copy`:

```go
dst := make([]int, len(src))
copy(dst, src)   // 逐元素拷贝, 返回拷贝个数 min(len(dst),len(src))
```

`copy` 底层是 `memmove`,允许源和目标重叠。

### 一个可运行程序串起全流程

原地别名、三索引限容、扩容后脱钩——都在一个程序里:

```go
package main

import "fmt"

func main() {
	a := []int{1, 2, 3, 4, 5}
	s := a[0:2]                 // len=2 cap=5, 与 a 共享底层数组
	fmt.Println(len(s), cap(s)) // 2 5
	s = append(s, 99)           // cap 够 → 原地写 a[2]
	fmt.Println(a)              // [1 2 99 4 5]  ← a 被 s 改了

	b := []int{1, 2, 3, 4, 5}
	t := b[0:2:2]               // 三索引切片: len=2 cap=2
	fmt.Println(len(t), cap(t)) // 2 2
	t = append(t, 99)           // cap 用满 → 新数组, b 不受影响
	fmt.Println(b)              // [1 2 3 4 5]
	fmt.Println(t)              // [1 2 99]

	c := []int{1, 2, 3}
	d := c[0:1]                 // len=1 cap=3, 仍与 c 共享
	d = append(d, 20)           // cap 够 → 写 c[1]
	d = append(d, 30)           // cap 够 → 写 c[2]
	fmt.Println(c)              // [1 20 30]  ← 仍耦合
	d = append(d, 40)           // len==cap → 扩容, 此时与 c 脱钩
	d[0] = 999
	fmt.Println(c)              // [1 20 30]       ← c 不再受影响
	fmt.Println(d)              // [999 20 30 40]
}
```

## 三、map:哈希表

map 是键值对集合,用 `map[K]V` 声明,`make` 或字面量创建:

```go
m := make(map[string]int)     // 空 map
m["a"] = 1                    // 写入
m["b"] = 2
fmt.Println(m["a"])           // 1     —— 读取

v, ok := m["c"]               // 逗号 ok: 判断键是否存在
fmt.Println(v, ok)            // 0 false —— c 不存在, v 是零值

delete(m, "a")                // 删除键 "a"

for k, v := range m {         // 遍历(顺序不固定, 见下)
	fmt.Println(k, v)
}
```

> **版本提示**:下面描述的 `hmap`/`bmap`(桶 + 溢出桶、负载因子 6.5、渐进 `evacuate` 迁移)是 **Go ≤ 1.23** 的经典实现。**Go 1.24 起内置 map 改用 Swiss Tables 设计**(源自 Abseil 的开放寻址 + 分组控制字/SIMD 探测,`internal/runtime/maps`),内部布局与"溢出桶""6.5 负载因子"等细节都变了、平均更快;但**语言层面的语义(无序、并发不安全、`&m[k]` 不可取址等)完全不变**。这里仍讲经典实现,因为它更能说明"哈希表 + 桶/拉链"的通用原理——把它当作"一种"实现来读即可。这次替换本身也印证了下面要讲的一条设计原则:语言规范从不承诺遍历顺序或内部布局,实现才能在不破坏用户代码的前提下彻底重写。

map 的运行时类型是 `*hmap`(`runtime/map.go`,Go ≤ 1.23)——注意 **map 变量本身就是一个指针**(一个字),这是它和 slice 表头的一个差异。

```go
type hmap struct {
    count     int    // 元素个数, len(m) 直接读它 O(1)
    flags     uint8
    B         uint8  // 桶数量 = 2^B
    noverflow uint16
    hash0     uint32 // 哈希种子(每个 map 随机, 防 hash flooding 攻击)
    buckets   unsafe.Pointer // 指向 2^B 个 bmap 桶的数组
    oldbuckets unsafe.Pointer // 扩容时的旧桶数组
    nevacuate uintptr
    extra     *mapextra
}
```

桶 `bmap`:每个桶固定 **8 个槽位**。

```go
type bmap struct {
    tophash [8]uint8 // 每个 key 哈希值的高 8 位(快速比较用)
    // 编译器在后面追加: keys[8], values[8], overflow *bmap
}
```

内存布局是 `8 个 tophash │ 8 个 key │ 8 个 value │ 溢出桶指针`。注意 **key 和 value 分别连续存放**(不是 kv/kv/kv 交错),这样能避免因对齐产生的填充浪费(比如 `map[int8]int64` 交错会浪费很多)。

```
hmap ──▶ buckets: [ bucket0 | bucket1 | ... | bucket(2^B-1) ]
                       │
                       ▼
  一个 bmap 桶:
  ┌──────────────────────┐
  │ tophash[0..7]        │  每槽哈希高8位(也用作空/已迁移标记)
  ├──────────────────────┤
  │ key0 key1 ... key7   │  8 个 key 连续
  ├──────────────────────┤
  │ val0 val1 ... val7   │  8 个 value 连续
  ├──────────────────────┤
  │ overflow ────────────┼──▶ 下一个溢出 bmap(链式)
  └──────────────────────┘
```

`v := m[k]` 的查找过程:

1. `hash = hasher(k, hash0)`。
2. 用低 B 位选桶:`bucket = hash & (2^B - 1)`。
3. 用高 8 位 `tophash` 在桶内 8 个槽快速筛选(只比 1 字节,大多数不匹配的直接跳过),命中再完整比较 key。
4. 桶内没有就沿 `overflow` 链找溢出桶。

平均 O(1),最坏(大量哈希冲突)退化。

当负载因子过高(元素数 / 桶数 > 6.5)或溢出桶太多时,触发**增量扩容(rehash)**:分配 2× 的新桶数组,把 `buckets` 挪到 `oldbuckets`。**关键:不一次性搬完**,而是在后续每次写操作(`insert`/`delete`)时顺手迁移 1~2 个旧桶(`evacuate`),摊销掉大 map 一次性 rehash 的停顿。查找时会同时看新旧桶。这与 C++ `std::unordered_map` 的一次性 rehash 不同——一次性 rehash 会造成某次插入操作突然变得极慢(所有元素都要重新分桶),而摊销迁移把这笔代价平摊到之后若干次操作里,避免了单次操作的延迟尖峰,这对延迟敏感的服务尤其重要。

### map 的几个硬约束(和 C/C++ 直觉不同)

**1. 迭代顺序是随机的,故意的,而且是双重设计。** 每次 `for k, v := range m` 的起始桶和起始槽都随机化。这里其实叠着两层不同的考虑,值得分开看:

- **不让程序依赖不该依赖的实现细节**。哈希表的自然遍历顺序(按桶/槽的物理布局)完全是实现细节的副产品,和"插入顺序"或任何有意义的语义都无关。如果 Go 像早期实现那样让遍历顺序"恰好"表现得比较固定,程序员会不自觉地写出依赖这个顺序的代码(比如假设先插入的先出现)——这类代码在当时能跑,但一旦运行时切换实现(桶数量变化、扩容发生、乃至像 Go 1.24 那样整体换成 Swiss Tables),顺序就变了,程序静默出错。Go 团队从语言早期就故意把起始点随机化,让"顺序不可依赖"从一开始就以看得见的方式暴露出来,而不是留到实现变更时才暴雷。这也解释了上面版本提示里的那句话:正因为语义上从不承诺顺序,Go 1.24 才能把整个哈希表实现换成 Swiss Tables 而不破坏任何遵循语言规范的现有代码。
- **降低哈希碰撞攻击(hash flooding)的收益**。`hash0` 让每个 map 实例的哈希函数都带不同随机种子,这是防御的主体——攻击者即使知道哈希算法,也无法离线算出一组能让所有槽位都命中同一个桶的"碰撞键集合"来发起拒绝服务攻击(否则 O(1) 的查找会被攻击者蓄意退化成 O(n))。遍历顺序随机化是这层防御在可观察行为上的自然延伸:如果顺序是确定的,观察者更容易通过遍历结果反推内部状态。

要有序必须自己排序:

```go
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
sort.Strings(keys) // 见第11章
for _, k := range keys {
    fmt.Println(k, m[k])
}
```

**2. 元素不可取地址。** `&m[k]` 编译错误。因为扩容会搬动 value 的内存位置,指针会失效。所以不能就地改结构体字段:

```go
m := map[string]Point{"a": {1, 2}}
// m["a"].X = 10        // 编译错误: cannot assign to struct field
p := m["a"]; p.X = 10; m["a"] = p   // 取出→改→写回
// 或者存指针: map[string]*Point, 这样 m["a"].X = 10 可以
```

可运行(注释掉的那行会编译错误):

```go
package main

import "fmt"

type Point struct{ X, Y int }

func main() {
	m := map[string]Point{"a": {1, 2}}
	// m["a"].X = 10 // 编译错误: cannot assign to struct field m["a"].X
	p := m["a"]         // 取出
	p.X = 10            // 改副本
	m["a"] = p          // 写回
	fmt.Println(m["a"]) // {10 2}

	pm := map[string]*Point{"a": {1, 2}} // 改存指针
	pm["a"].X = 10                       // 就地改可行: value 是指针
	fmt.Println(*pm["a"])                // {10 2}

	v, ok := m["missing"] // 读缺失键
	fmt.Println(v, ok)    // {0 0} false
}
```

**3. 读缺失键返回零值 + ok。** 逗号 ok 惯用法:

```go
v, ok := m[k] // ok 为 false 表示不存在, v 是零值
```

**4. 并发不安全。** 并发读写(哪怕一读一写)会被运行时检测到并直接 `fatal error: concurrent map read and map write` 崩溃(不是 panic,不可 recover)。这不是运行时"偷懒"没做同步,而是刻意的取舍:给 map 内置锁会让所有单 goroutine 场景(绝大多数用例)都为用不上的同步付出开销;运行时选择让 map 保持无锁、最快路径,把并发安全的责任交还给调用方——需要就自己加锁,或用 `sync.Map`(读多写少场景,见[第10章](10-并发模型.md))。检测到冲突就直接 `fatal error` 而非静默产生垃圾数据或 UB,是让这类 bug 尽早暴露而不是留到生产环境里变成诡异的偶发损坏。

**5. nil map 可读不可写。** `var m map[string]int` 是 nil,读返回零值,写 panic。必须 `make` 或字面量初始化。

## 四、channel:CSP 模型的具体实现

channel 是 goroutine 之间传数据的管道,`make(chan T)` 创建,`<-` 发送/接收,`close` 关闭:

```go
ch := make(chan int, 2)  // 容量为 2 的有缓冲 channel
ch <- 1                  // 发送
ch <- 2
fmt.Println(<-ch)        // 1  —— 接收(先进先出)
fmt.Println(<-ch)        // 2

close(ch)                // 关闭 channel, 表示不会再发送了
v, ok := <-ch            // 已关闭且已收干: v 是零值, ok=false
fmt.Println(v, ok)       // 0 false
```

`for v := range ch` 会持续接收,直到 channel 被关闭且里面的值都被收完(并发场景见[第10章](10-并发模型.md))。

**为什么 channel 是一等公民,而不是"共享内存 + 锁"**。C/C++ 里多线程通信的默认做法是:开一块共享内存,配一把锁(或原子操作)保护它。这个模型要求每个访问共享数据的地方都记得加锁,正确性依赖于程序员在**所有**访问点都遵守同一套隐式约定——少加一次锁,数据就可能被撕裂。channel 来自 Hoare 的 CSP(Communicating Sequential Processes)理论,Go 把它做成语言内建原语,背后是一句被反复引用的设计哲学:

> **不要通过共享内存来通信,而要通过通信来共享内存。**
> Don't communicate by sharing memory; share memory by communicating.

具体含义:用 channel 传递数据时,传递的不只是数据本身,而是数据的**所有权**——发送方把值交给 channel 之后,约定上就不再触碰它,接收方拿到后独享。这样任意时刻一份数据只有一个 goroutine 在读写,不再需要在每个访问点都记得加锁,同步正确性从"记住在每个访问点加锁"变成"数据结构本身只有一条流转路径"。这不是说 Go 里没有共享内存和锁——`sync.Mutex`、`sync/atomic` 都还在(见[第10章](10-并发模型.md)),用于确实只需要保护一小段临界区、性能更敏感的场景;channel 解决的是更高层次的问题:把并发程序结构化成清晰的数据流管道,而不是一堆互相加锁的共享状态。代价也很实际:channel 的收发涉及锁与调度器交互,单次操作开销比一次原子自增大得多——保护一个简单计数器用 `atomic` 或 `Mutex` 仍然更合适,channel 的价值在于"传递数据所有权"和"构造同步点",不是替代所有锁。

channel 的底层结构是 `*hchan`(`runtime/chan.go`),channel 变量也是一个指针:

```go
type hchan struct {
    qcount   uint           // 当前缓冲区里的元素数
    dataqsiz uint           // 缓冲区容量(make 时的第二个参数)
    buf      unsafe.Pointer // 指向环形缓冲数组(有缓冲时)
    elemsize uint16
    closed   uint32
    sendx    uint           // 环形缓冲的写下标
    recvx    uint           // 环形缓冲的读下标
    recvq    waitq          // 等待接收的 goroutine 队列
    sendq    waitq          // 等待发送的 goroutine 队列
    lock     mutex          // 保护上面所有字段
}
```

```
有缓冲 channel  make(chan int, 4):
        ┌─────────────────────────────┐
buf ──▶ │  slot0 slot1 slot2 slot3    │  环形缓冲
        └─────────────────────────────┘
          ▲recvx        ▲sendx
  lock  ─ 一把互斥锁, 每次收发都要抢
  sendq ─ [G3, G7 ...]  被阻塞在发送上的 goroutine
  recvq ─ [ ]           被阻塞在接收上的 goroutine
```

**无缓冲 channel(`make(chan int)`,dataqsiz=0)**:发送和接收必须"会合(rendezvous)"。发送方阻塞,直到有接收方到来,数据**直接从发送方 goroutine 拷贝到接收方 goroutine**(不经过缓冲),然后两者都被唤醒。这不是实现细节上偶然选的行为,而是无缓冲 channel 语义的直接定义:它不提供任何存储,"发送成功"这件事的定义就是"接收方已经拿到了值"——因此收发双方在时间上必然重合于一点,这正是"会合"一词的含义,也是最强的同步保证:一次发送 happens-before 对应的接收完成([第10章](10-并发模型.md)会展开这条 happens-before 规则)。

**有缓冲 channel(`make(chan int, n)`)** 放松了这个约束,用一段环形缓冲在发送方和接收方之间解耦:

| 操作 | 缓冲未满/非空 | 缓冲满(发)/空(收) |
|------|--------------|-------------------|
| 发送 `ch <- v` | 拷进 buf[sendx],sendx++,立即返回 | 当前 goroutine 入 sendq,阻塞挂起 |
| 接收 `<-ch` | 从 buf[recvx] 拷出,recvx++,立即返回 | 当前 goroutine 入 recvq,阻塞挂起 |

"阻塞挂起"不是忙等:goroutine 被调度器摘下,M 去跑别的 G(见[第10章 GMP](10-并发模型.md))。当对端操作发生时,把等待队列里的 goroutine 唤醒。这就是 goroutine 廉价并发的基础——阻塞一个 goroutine 不阻塞 OS 线程。

一个优化:若发送时 recvq 里已有等待的接收者,数据**跳过缓冲区直接拷给那个接收者**并唤醒它,省一次拷贝。

```go
close(ch)          // 关闭; 之后发送 panic; 接收立即返回零值
v, ok := <-ch      // ok==false 表示 channel 已关闭且缓冲已空
for v := range ch  // 一直收到 channel 关闭为止
```

对 nil channel 收发**永久阻塞**(常用于在 `select` 中动态"关掉某个分支")。

channel 传递的是**元素的拷贝**(值语义):`ch <- x` 把 `x` 拷进缓冲,`<-ch` 拷出。如果元素是大结构体就拷整块;传指针则拷指针(此时要注意共享数据的并发安全)。

### 有缓冲收发、close 与 range(可运行)

```go
package main

import "fmt"

func main() {
	ch := make(chan int, 2)
	ch <- 1
	ch <- 2
	close(ch)           // 发送完再关; 缓冲里的值仍可接收
	for v := range ch { // 按 FIFO 把缓冲收干, 遇 close 停止
		fmt.Println(v) // 依次打印 1、2
	}
	v, ok := <-ch      // 已关闭且已收干 → 零值, ok=false
	fmt.Println(v, ok) // 0 false
}
```

## 五、三者对照小结

| 类型 | 变量本身大小 | 赋值/传参拷贝什么 | 数据在哪 | 并发安全 |
|------|-------------|------------------|---------|---------|
| `[N]T` array | N×sizeof(T) | 整块数据 | 随变量(栈或堆) | 值语义,天然隔离 |
| `[]T` slice | 3 字(24B) | 表头(含指针) | 底层数组通常在堆 | 否,共享底层数组 |
| `map` | 1 字(指针) | 指针 | hmap+桶在堆 | 否,并发读写崩溃 |
| `chan` | 1 字(指针) | 指针 | hchan+buf 在堆 | 是,内部有锁 |

一句话:**除了 array 是真值类型,slice/map/chan 拷贝的都是"轻表头/指针",共享背后的数据**——这正是它们"像引用"的原因,但机制上仍是彻底的值拷贝。

## 小结

- **用法层面**:数组 `[N]T` 固定长度、按下标读写;切片 `[]T` 用 `append` 追加、`len`/`cap` 查询、`s[low:high]` 切片;map `map[K]V` 用 `m[k]=v` 写、`v,ok:=m[k]` 读并判断存在性、`delete(m,k)` 删除;channel `chan T` 用 `ch<-v` 发送、`<-ch` 接收、`close(ch)` 关闭、`for range ch` 遍历。
- **底层与设计取舍**:slice 是 24 字节的三元组表头(array/len/cap),表头浅拷贝换来了赋值传参的 O(1) 代价,`append` 在 cap 够时原地写、不够时分配新数组并与旧 slice 脱钩,这是别名坑的根源;map(Go ≤1.23)是 `hmap` + 8 槽一桶的 `bmap`,靠 tophash 加速查找,负载因子超 6.5 渐进扩容以摊平停顿;map 遍历顺序故意随机化,一是不让程序依赖实现细节(才使得 Go 1.24 换成 Swiss Tables 不破坏现有代码),二是配合哈希种子随机化防哈希碰撞攻击;元素不可取地址是扩容会搬动内存的直接推论;并发读写直接崩溃是把同步责任留给调用方以保持无锁快路径的取舍。channel 是带锁的环形缓冲 `hchan`,是 CSP"通过通信共享内存"哲学的具体实现,用所有权转移替代到处加锁;无缓冲时收发直接"会合"是其同步语义的必然结果,阻塞是把 goroutine 挂起而非忙等。三者变量本身要么是表头(slice)要么是指针(map/chan),真正的数据都在堆上。
