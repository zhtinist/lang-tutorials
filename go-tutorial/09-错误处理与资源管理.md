# 错误处理与垃圾回收

Go 没有异常(no exceptions),没有 `try/catch`,没有析构函数做 RAII。它把错误当**普通的返回值**处理,把资源清理交给 `defer`,把内存回收交给**并发垃圾回收器**。对 C++ 程序员来说,这是三处需要重新建立肌肉记忆的地方。

## 一、error 是一个普通接口值

`error` 不是关键字,就是标准库里一个再普通不过的接口:

```go
type error interface {
    Error() string
}
```

任何实现了 `Error() string` 的类型都是一个 error。它在内存里就是[第08章讲的接口双字 `iface`](08-面向对象与类型系统.md)——`{itab, data}`。返回 error 不涉及栈展开、不涉及零成本异常表,就是压一个 16 字节的接口值到返回栈槽,和返回任何其他值一样廉价。

### 惯用法:多返回值 + if err != nil

```go
f, err := os.Open("data.txt")
if err != nil {
    return fmt.Errorf("打开配置失败: %w", err) // 处理或包装后上抛
}
defer f.Close()
// 正常路径
```

C++ 里错误是"隐式的控制流"(异常自动向上冒泡);Go 里错误是"显式的数据流"——每一层都要肉眼可见地检查和传递。啰嗦是它被诟病的点,但换来了**控制流完全可见、无隐藏跳转、无栈展开开销**。

### 创建错误

```go
errors.New("something failed")           // 最简单, 返回 *errorString
fmt.Errorf("port %d in use", port)        // 带格式化
fmt.Errorf("read config: %w", err)        // %w 包装底层错误(保留链)
```

`%w`(wrap)是关键:它把 `err` 作为"被包装的原因"嵌进新错误,形成一条错误链,可被后面的 `errors.Is`/`errors.As` 逐层解开。用 `%v` 只是把文本拼进去,链就断了。

## 二、哨兵错误、自定义错误、Is/As

### 哨兵错误(sentinel)

预定义的、可用 `==` 或 `errors.Is` 比较的错误值:

```go
var ErrNotFound = errors.New("not found")

func Lookup(k string) (string, error) {
    // ...
    return "", ErrNotFound
}

if errors.Is(err, ErrNotFound) { /* 特判 */ }
```

**为什么用 `errors.Is` 而不是 `err == ErrNotFound`**?因为 err 可能被 `%w` 包装了好几层,`==` 只比最外层;`errors.Is` 会沿着 `%w` 链一路 `Unwrap` 逐层比较。

### 自定义错误类型

想携带结构化信息就定义类型:

```go
type PathError struct {
    Op   string
    Path string
    Err  error
}
func (e *PathError) Error() string {
    return e.Op + " " + e.Path + ": " + e.Err.Error()
}
func (e *PathError) Unwrap() error { return e.Err } // 让 Is/As 能穿透
```

`errors.As` 把链上第一个匹配目标类型的错误取出来:

```go
var pe *PathError
if errors.As(err, &pe) {
    fmt.Println("出错路径:", pe.Path) // 拿到结构化字段
}
```

| 函数 | 作用 | 类比 |
|------|------|------|
| `errors.Is(err, target)` | 链上是否有等于 target 的错误 | 按"值"匹配 |
| `errors.As(err, &v)` | 链上是否有可赋给 v 的类型,取出 | 按"类型"匹配(像 C++ `catch(SpecificType&)`) |
| `errors.Unwrap(err)` | 剥一层 | 手动走链 |

## 三、panic / recover:不是异常

`panic` 用于**真正不该发生的程序错误**(数组越界、解引用 nil、不变量被破坏),不是常规错误处理手段。

### panic 的机制

`panic(x)` 会:立即停止当前函数正常执行 → **沿调用栈向上逐层展开(unwind)**,每层执行已注册的 `defer` → 若一路没被 `recover`,到达 goroutine 顶端,打印栈回溯并 **`exit(2)` 终止整个进程**(不是只挂掉当前 goroutine)。

```
panic 展开:
main → A() → B() → panic!
                    ↓ 执行 B 的 defer
              ↑ 执行 A 的 defer
        ↑ 执行 main 的 defer(若无 recover)→ 进程崩溃
```

这个展开过程会跑 `defer`,但**没有 C++ 那种基于类型的 catch 匹配**——recover 抓的是"任意 panic",不区分类型。

### recover:只能在 defer 里

`recover()` 只有在 `defer` 的函数里直接调用才有效,它会**停止展开**、返回传给 panic 的值,让程序在当前栈帧恢复正常:

```go
func safeRun() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r) // 把 panic 转成 error 返回
        }
    }()
    riskyOperation()
    return nil
}
```

典型用途:web 服务器的每个请求 handler 外面包一层 recover,单个请求 panic 不至于拖垮整个进程。**但不要用 panic/recover 模拟 try/catch 做常规流程控制**——这是反模式,慢且违背 Go 惯例。

对照记忆:

| | C++ 异常 | Go panic/recover |
|--|---------|------------------|
| 用途 | 常规错误处理 | 仅致命/不变量破坏 |
| 触发 | `throw` | `panic()` 或运行时错误 |
| 捕获 | `catch(Type&)` 按类型 | `recover()` 抓任意,自行判断 |
| 不捕获 | `std::terminate` | 进程 `exit(2)` |
| 清理 | 析构函数(RAII) | `defer` |
| 常规错误 | 也用异常 | 用返回值 error |

## 四、资源管理:defer 代替 RAII

Go 没有析构函数。资源清理靠 [`defer`(见第05章机制)](05-控制流.md):注册的调用在函数返回前(含 panic 展开时)按 **后进先出** 执行。

```go
f, err := os.Open(name)
if err != nil { return err }
defer f.Close()           // 无论后面怎么返回/panic, 都会关

mu.Lock()
defer mu.Unlock()         // 配对加解锁, 防止漏解锁
```

和 C++ RAII 的差异:RAII 绑定**对象生命周期/作用域**(离开块即析构);`defer` 绑定**函数返回**(不是块级,是函数级!循环里 `defer` 会攒到函数结束才执行,是常见的资源泄漏坑)。需要块级及时释放就显式调用或拆出小函数。

> `runtime.SetFinalizer` 可给对象注册"GC 回收时的终结器",但它执行时机不确定、只保证跑一次,**不能替代 defer/显式 Close** 管理文件、锁等确定性资源。

## 五、垃圾回收(GC)

Go 是带 GC 的语言。你不写 `free`/`delete`,不管理生命周期;编译器的[逃逸分析(第03章)](03-变量与内存模型.md)决定变量在栈还是堆,栈上的随栈帧回收,堆上的由 GC 回收。

### 并发三色标记-清除

Go 的 GC 是**并发的、非移动的、三色标记-清除(tri-color mark-sweep)**,核心目标是**极短的 STW(stop-the-world)**。

三色抽象:

- **白色**:尚未访问,回收候选。
- **灰色**:已访问,但其引用的对象还没扫完(在待处理队列里)。
- **黑色**:已访问,且引用对象都已扫描完。

流程:从根(栈、全局变量)出发,把根置灰;不断从灰色集合取对象、把它引用的白色对象置灰、自己转黑;直到没有灰色。剩下的白色就是垃圾,清除。**标记阶段与用户 goroutine 并发运行**(mark workers 用一部分 CPU),不是长时间停顿。

### 写屏障(write barrier)

并发标记的麻烦:用户程序一边改指针,标记一边跑,可能"黑色对象新指向一个白色对象、而唯一指向该白色的灰色引用又被删掉"——白色对象被漏标、被错误回收。

解决办法是**写屏障**:在标记期间,每次指针写操作被编译器插入一小段钩子代码,把涉及的对象重新标灰,保证不漏。Go 用的是混合写屏障(Dijkstra + Yuasa 混合),使得**栈几乎不需要 STW 重扫**。写屏障只在 GC 标记阶段激活,平时关闭。

```
并发 GC 一轮(简化):
[极短 STW: 开启写屏障] → [并发标记, 用户程序同时跑, 写屏障保护] →
[极短 STW: 标记终止] → [并发清除, 与用户程序同时]
两次 STW 通常都在亚毫秒~毫秒级
```

### 触发时机与 GOGC

GC 由**堆增长比例**驱动,由环境变量/参数 `GOGC` 控制(默认 100):

> 当堆存活量相对上次 GC 后增长到 `GOGC%` 时触发下一轮。`GOGC=100` 表示堆翻倍(增长 100%)就 GC 一次。

```go
debug.SetGCPercent(200) // 调高 → GC 更少、内存占用更高、吞吐更好
debug.SetGCPercent(50)  // 调低 → GC 更频繁、内存更省
runtime.GC()            // 手动强制一次(阻塞式, 一般只在测试/特殊场景用)
```

Go 1.19+ 还提供**软内存上限** `GOMEMLIMIT` / `debug.SetMemoryLimit`,给容器等内存受限环境兜底。

### 关于分代

**Go 目前的 GC 不是分代式的**(no generational GC),也不做压缩/移动(non-moving,所以对象地址稳定,这也是为什么 Go 里可以放心取地址、和 cgo 交互相对简单)。历史上社区讨论过分代,但由于逃逸分析已经让大量短命对象留在栈上(相当于廉价的"零代"),分代收益没那么大;实现选择了当前这套低延迟方案。这点会随版本演进,以运行时文档为准。

### 内存分配器:mcache / mcentral / mspan

对象在堆上怎么分配?Go 有一套 **tcmalloc 风格的多级分配器**,核心是**按 size class 分档**,避免全局锁竞争:

```
每个 P(见第10章调度) 有一个 mcache(无锁, 线程本地)
        │  按 size class(约 70 档: 8B,16B,...,32KB)缓存空闲对象
        ▼ 本地不够时向上要
mcentral(每个 size class 一个, 带锁)
        ▼ 不够时向上要
mheap(全局堆, 以 mspan 为单位管理页)
        ▼
OS(mmap 大块虚拟内存)
```

- **mspan**:一组连续页,切成同一 size class 的等大小块。小对象(≤32KB)从对应 size class 的 span 里取。
- **mcache**:每个 P 私有,分配小对象时**无需加锁**(对照 C 的 `malloc` 常有全局锁竞争)——这是 Go 高并发分配快的原因之一。
- 大对象(>32KB)直接从 mheap 分配。

呼应[第03章](03-变量与内存模型.md):**逃逸到堆的对象越多,分配和 GC 压力越大**。想优化就减少堆分配——预分配 slice 容量、复用对象(`sync.Pool`)、避免不必要的接口装箱/闭包捕获,用 `go build -gcflags=-m` 看逃逸分析、用 [pprof](12-常用第三方库.md) 看分配热点。

## 小结

- error 是接口值,错误处理是显式数据流,不是异常;`%w` + `Is`/`As` 组织错误链。
- panic/recover 仅用于真正的异常情形;recover 必须在 defer 中。
- 没有 RAII,用 defer 做确定性清理(注意是函数级不是块级)。
- GC 是并发三色标记-清除 + 写屏障,STW 极短,非分代、非移动,由 GOGC/内存增长驱动;分配器 mcache/mcentral/mspan 多级、P 本地无锁。
