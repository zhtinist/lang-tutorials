# 错误处理、资源管理与垃圾回收

Go 没有异常(no exceptions),没有 `try/catch`,没有析构函数做 RAII。它把错误当**普通的返回值**处理,把资源清理交给 `defer`,把内存回收交给**并发垃圾回收器**。对 C++ 程序员来说,这是三处需要重新建立肌肉记忆的地方,每一处都对应一个具体的设计取舍。

## 一、error 是一个普通接口值

`error` 不是关键字,它是标准库里一个再普通不过的接口,和你自己定义的接口没有任何区别:

```go
type error interface {
    Error() string
}
```

任何实现了 `Error() string` 的类型都是一个 error。它在内存里就是[第08章讲的接口双字 `iface`](08-面向对象与类型系统.md)——`{itab, data}`。返回 error 不涉及栈展开、不涉及零成本异常表,就是压一个 16 字节的接口值到返回栈槽,和返回任何其他值一样廉价。

惯用法是"多返回值 + if err != nil":

```go
f, err := os.Open("data.txt")
if err != nil {
    return fmt.Errorf("打开配置失败: %w", err) // 处理或包装后上抛
}
defer f.Close()
// 正常路径
```

创建错误的几种方式:

```go
errors.New("something failed")           // 最简单, 返回 *errorString
fmt.Errorf("port %d in use", port)        // 带格式化
fmt.Errorf("read config: %w", err)        // %w 包装底层错误(保留链)
```

`%w`(wrap)是关键:它把 `err` 作为"被包装的原因"嵌进新错误,形成一条错误链,可被后面的 `errors.Is`/`errors.As` 逐层解开。用 `%v` 只是把文本拼进去,链就断了。

**为什么用 `(result, error)` 返回值而不是异常,以及和 Java checked exception 的真正差异**。C++/大多数带异常的语言里,错误处理是**隐式的控制流**:一处 `throw` 会自动向上跳过任意多层调用栈,直到遇到匹配的 `catch`,中间每一层函数的签名完全看不出它可能失败。Go 选择把错误做成**显式的数据流**:失败与否是函数返回值类型的一部分,写在签名里,调用方必须显式接收这个返回值——`err` 摆在那里,不检查它编译器不会报错,但代码审查、静态检查工具(`errcheck` 等)能揪出"忽略了 err"这种明显的疏漏,更重要的是每一层调用者都被迫在视觉上面对"这里可能失败"这个事实。

值得对照的是 Java 的 checked exception:它其实想解决同一个问题——通过编译器强制调用方处理失败路径,方法签名里的 `throws IOException` 也确实让"这个方法可能失败"变得可见。但两者解决方案的落地效果并不相同:checked exception 仍然是异常那套隐式控制流机制(`throw` 依然会跳过若干层栈帧展开到某个 `catch`),把"类型层面的可见性"叠加在了"控制流层面的不可见性"之上;结果是 Java 代码里大量出现"捕获后原样重新包装成 `RuntimeException` 扔出去"这类为了绕开签名标注而生的样板,checked exception 因此在 Java 社区长期饱受争议,不少大型框架和库干脆只抛未检查异常。Go 的 `(result, error)` 把"可能失败"和"控制流"合而为一:错误就是一个普通值,可以被存储、传递、组合、包装(`%w`),和函数的其他返回值走同一条路径,没有另开一条"异常通道"让控制流产生非顺序的跳转。这换来的是控制流完全可见、无隐藏跳转、无栈展开开销;代价是啰嗦——`if err != nil { return err }` 这行代码要在几乎每次可能失败的调用后重复一遍,这是 Go 长期被诟病的样板代码来源,但也是这套设计对"显式"二字的坚持。

## 二、哨兵错误、自定义错误、Is/As

**哨兵错误(sentinel)**:预定义的、可用 `==` 或 `errors.Is` 比较的错误值:

```go
var ErrNotFound = errors.New("not found")

func Lookup(k string) (string, error) {
    // ...
    return "", ErrNotFound
}

if errors.Is(err, ErrNotFound) { /* 特判 */ }
```

**为什么用 `errors.Is` 而不是 `err == ErrNotFound`**?因为 err 可能被 `%w` 包装了好几层,`==` 只比最外层;`errors.Is` 会沿着 `%w` 链一路 `Unwrap` 逐层比较。

**自定义错误类型**:想携带结构化信息就定义类型:

```go
type PathError struct {
    Op   string
    Path string
    Err  error
}
func (e *PathError) Error() string {
    return e.Op + " " + e.Path + ": " + e.Err.Error()
}
func (e *PathError) Unwrap() error { return e.Err } // 让 Is/As 能穿透
```

`errors.As` 把链上第一个匹配目标类型的错误取出来:

```go
var pe *PathError
if errors.As(err, &pe) {
    fmt.Println("出错路径:", pe.Path) // 拿到结构化字段
}
```

| 函数 | 作用 | 类比 |
|------|------|------|
| `errors.Is(err, target)` | 链上是否有等于 target 的错误 | 按"值"匹配 |
| `errors.As(err, &v)` | 链上是否有可赋给 v 的类型,取出 | 按"类型"匹配(像 C++ `catch(SpecificType&)`) |
| `errors.Unwrap(err)` | 剥一层 | 手动走链 |

`errors.Is`/`errors.As` 都是沿着 `Unwrap() error` 方法反复往下走的一个循环,不是编译器魔法:每一层若不匹配,就调用它的 `Unwrap()` 拿到下一层,直到链尽头(`Unwrap` 返回 nil)。`errors.Is` 默认用 `==` 比较,若某类型自己定义了 `Is(error) bool` 则优先调用它;`errors.As` 则用反射检查链上每一层的动态类型能否赋给目标指针指向的类型。这条链本质上就是一串通过接口值互相持有的"节点",和链表遍历没有本质区别——这也再次印证了"error 只是普通值"这条设计:错误链不需要运行时提供任何特殊支持,完全靠接口方法调用拼出来。

可运行:%w 包装 + Is + As:

```go
package main

import (
	"errors"
	"fmt"
)

var ErrNotFound = errors.New("not found")

type QueryError struct {
	Query string
	Err   error
}

func (e *QueryError) Error() string { return e.Query + ": " + e.Err.Error() }
func (e *QueryError) Unwrap() error { return e.Err }

func find() error {
	qe := &QueryError{Query: "SELECT", Err: ErrNotFound} // 包住哨兵错误
	return fmt.Errorf("db call: %w", qe)                 // 再用 %w 包一层
}

func main() {
	err := find()
	fmt.Println(err)                         // db call: SELECT: not found
	fmt.Println(errors.Is(err, ErrNotFound)) // true   ← 沿整条 %w/Unwrap 链比较

	var qe *QueryError
	fmt.Println(errors.As(err, &qe), qe.Query) // true SELECT   ← 取出链上的具体类型
}
```

## 三、panic / recover:不是异常

`panic` 用于**真正不该发生的程序错误**(数组越界、解引用 nil、不变量被破坏),不是常规错误处理手段。`recover` 只在 `defer` 里调用才有效,用来"接住"panic、让程序恢复正常执行:

```go
func safeRun() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered: %v", r) // 把 panic 转成 error 返回
        }
    }()
    riskyOperation()
    return nil
}
```

典型用途:web 服务器的每个请求 handler 外面包一层 recover,单个请求 panic 不至于拖垮整个进程。**但不要用 panic/recover 模拟 try/catch 做常规流程控制**——这是反模式,慢且违背 Go 惯例。

**为什么 panic/recover 存在,却不推荐日常使用**。这是和上一节相呼应的另一半设计:Go 的哲学把"错误"分成了两个性质完全不同的类别。**可预期的失败**(文件不存在、网络超时、用户输入非法)是程序正常运行的一部分,调用方理应知道它可能发生并做出应对——这类用 `error` 返回值处理,如上一节所述。**不该发生的程序性错误**(数组越界、对 nil 解引用、断言的不变量被打破)则代表程序自身的逻辑已经出错,继续往下执行的状态是不可信的——这类才是 `panic` 的用武之地。把两者混在一条通道里(像很多语言用异常统一处理)会让调用方无法从类型签名分辨"这个错误我预期内、需要处理"还是"这个错误意味着程序已经坏了、该做的是尽快停止而不是继续硬跑"。`recover` 之所以存在而不是让 panic 必然终止进程,是为了在**边界**上兜底而不是在业务逻辑内部滥用它:一个长期运行的服务不该因为单个请求触发了一次不该发生的越界就整体崩溃,请求级别的 recover 让这类事故被隔离、记录,而不至于波及其他正在处理的请求;但对于每一次可预期的失败都套一层 `defer recover` 来模拟 `try/catch`,既比检查 `error` 慢(展开机制比普通返回值传递重得多),也违背了"错误分层"的初衷——把本该在类型签名里显式暴露的失败,重新藏回了隐式跳转里。

`panic(x)` 会:立即停止当前函数正常执行 → **沿调用栈向上逐层展开(unwind)**,每层执行已注册的 `defer` → 若一路没被 `recover`,到达 goroutine 顶端,打印栈回溯并 **`exit(2)` 终止整个进程**(不是只挂掉当前 goroutine)。

```
panic 展开:
main → A() → B() → panic!
                    ↓ 执行 B 的 defer
              ↑ 执行 A 的 defer
        ↑ 执行 main 的 defer(若无 recover)→ 进程崩溃
```

这个展开过程会跑 `defer`,但**没有 C++ 那种基于类型的 catch 匹配**——recover 抓的是"任意 panic",不区分类型。

`recover()` 只有在 `defer` 的函数里直接调用才有效,它会**停止展开**、返回传给 panic 的值,让程序在当前栈帧恢复正常。

可运行:把运行时 panic 转成 error:

```go
package main

import "fmt"

func safeDiv(a, b int) (result int, err error) {
	defer func() {
		if r := recover(); r != nil {
			err = fmt.Errorf("recovered: %v", r) // 把 panic 转成 error 返回
		}
	}()
	return a / b, nil // b==0 会在 return 完成前触发运行时 panic
}

func main() {
	r, err := safeDiv(10, 2)
	fmt.Println(r, err) // 5 <nil>
	r, err = safeDiv(10, 0)
	fmt.Println(r, err) // 0 recovered: runtime error: integer divide by zero
}
```

注意 panic 分支里 `result` 仍是 0:panic 在求值 `a / b` 时就触发了,返回值从未被赋值,只有命名返回 `err` 在 defer 的 `recover` 里被设置。

对照记忆:

| | C++ 异常 | Go panic/recover |
|--|---------|------------------|
| 用途 | 常规错误处理 | 仅致命/不变量破坏 |
| 触发 | `throw` | `panic()` 或运行时错误 |
| 捕获 | `catch(Type&)` 按类型 | `recover()` 抓任意,自行判断 |
| 不捕获 | `std::terminate` | 进程 `exit(2)` |
| 清理 | 析构函数(RAII) | `defer` |
| 常规错误 | 也用异常 | 用返回值 error |

## 四、资源管理:defer 代替 RAII

Go 没有析构函数。资源清理靠 `defer`:注册的调用在函数返回前(含 panic 展开时)按 **后进先出** 执行([见第05章机制](05-控制流.md))。

```go
fmt.Println("开始")
defer fmt.Println("清理1") // 后进先出: 最后执行
defer fmt.Println("清理2") // 先执行
fmt.Println("结束")
// 输出顺序: 开始 结束 清理2 清理1
```

实际场景常用来配对"打开/关闭"或"加锁/解锁":

```go
f, err := os.Open(name)
if err != nil { return err }
defer f.Close()           // 无论后面怎么返回/panic, 都会关

mu.Lock()
defer mu.Unlock()         // 配对加解锁, 防止漏解锁
```

和 C++ RAII 的差异:RAII 绑定**对象生命周期/作用域**(离开块即析构);`defer` 绑定**函数返回**(不是块级,是函数级!循环里 `defer` 会攒到函数结束才执行,是常见的资源泄漏坑)。需要块级及时释放就显式调用或拆出小函数。**为什么 Go 选了函数级的 `defer` 而不是块级的析构**:RAII 的确定性析构依赖于对象生命周期能在编译期静态确定,这对 C++ 的栈对象成立,但 Go 的对象是否在堆上、生命周期何时结束由逃逸分析和 GC 共同决定(见下),编译器无法像 C++ 那样在"离开这对花括号"的精确时刻插入一次析构调用。`defer` 退而求其次,绑定一个总能在编译期确定的时刻——**函数返回**——这个时刻无论函数是正常 return 还是 panic 展开都会经过,足以覆盖"打开就要关闭"这类资源管理的核心需求,只是粒度比块级作用域粗。

> `runtime.SetFinalizer` 可给对象注册"GC 回收时的终结器",但它执行时机不确定、只保证跑一次,**不能替代 defer/显式 Close** 管理文件、锁等确定性资源。

## 五、垃圾回收(GC)

Go 是带 GC 的语言。你不写 `free`/`delete`,不管理生命周期;编译器的[逃逸分析(第03章)](03-变量与内存模型.md)决定变量在栈还是堆,栈上的随栈帧回收,堆上的由 GC 回收。日常写业务代码基本不用管 GC——极少数需要调优的场景,才会用到下面这几个"旋钮":

```go
debug.SetGCPercent(200) // 调高 → GC 更少、内存占用更高、吞吐更好
debug.SetGCPercent(50)  // 调低 → GC 更频繁、内存更省
runtime.GC()            // 手动强制一次(阻塞式, 一般只在测试/特殊场景用)
```

**并发三色标记-清除**。Go 的 GC 是**并发的、非移动的、三色标记-清除(tri-color mark-sweep)**,核心目标是**极短的 STW(stop-the-world)**——这个目标本身也是设计取舍的产物:Go 定位于构建长期运行、对延迟敏感的服务端程序(网络服务、基础设施组件),这类程序最怕的不是"平均吞吐略低",而是"偶尔卡顿几十毫秒到几百毫秒";于是 GC 设计上情愿多用一些 CPU 做并发标记,也要把暂停时间压到最低,这与很多批处理式或吞吐优先的 GC(比如以往 Java 的某些收集器)取舍方向不同。

三色抽象:

- **白色**:尚未访问,回收候选。
- **灰色**:已访问,但其引用的对象还没扫完(在待处理队列里)。
- **黑色**:已访问,且引用对象都已扫描完。

流程:从根(栈、全局变量)出发,把根置灰;不断从灰色集合取对象、把它引用的白色对象置灰、自己转黑;直到没有灰色。剩下的白色就是垃圾,清除。**标记阶段与用户 goroutine 并发运行**(mark workers 用一部分 CPU),不是长时间停顿。

**写屏障(write barrier)**。并发标记的麻烦:用户程序一边改指针,标记一边跑,可能"黑色对象新指向一个白色对象、而唯一指向该白色的灰色引用又被删掉"——白色对象被漏标、被错误回收。

解决办法是**写屏障**:在标记期间,每次指针写操作被编译器插入一小段钩子代码,把涉及的对象重新标灰,保证不漏。Go 用的是混合写屏障(Dijkstra + Yuasa 混合),使得**栈几乎不需要 STW 重扫**。写屏障只在 GC 标记阶段激活,平时关闭。

```
并发 GC 一轮(简化):
[极短 STW: 开启写屏障] → [并发标记, 用户程序同时跑, 写屏障保护] →
[极短 STW: 标记终止] → [并发清除, 与用户程序同时]
两次 STW 通常都在亚毫秒~毫秒级
```

**触发时机与 GOGC**。GC 由**堆增长比例**驱动,由环境变量/参数 `GOGC` 控制(默认 100):

> 当堆存活量相对上次 GC 后增长到 `GOGC%` 时触发下一轮。`GOGC=100` 表示堆翻倍(增长 100%)就 GC 一次。

Go 1.19+ 还提供**软内存上限** `GOMEMLIMIT` / `debug.SetMemoryLimit`,给容器等内存受限环境兜底。

**关于分代**。**Go 目前的 GC 不是分代式的**(no generational GC),也不做压缩/移动(non-moving,所以对象地址稳定,这也是为什么 Go 里可以放心取地址、和 cgo 交互相对简单)。历史上社区讨论过分代,但由于逃逸分析已经让大量短命对象留在栈上(相当于廉价的"零代"),分代收益没那么大;实现选择了当前这套低延迟方案。这点会随版本演进,以运行时文档为准。

**内存分配器:mcache / mcentral / mspan**。对象在堆上怎么分配?Go 有一套 **tcmalloc 风格的多级分配器**,核心是**按 size class 分档**,避免全局锁竞争:

```
每个 P(见第10章调度) 有一个 mcache(无锁, 线程本地)
        │  按 size class(约 70 档: 8B,16B,...,32KB)缓存空闲对象
        ▼ 本地不够时向上要
mcentral(每个 size class 一个, 带锁)
        ▼ 不够时向上要
mheap(全局堆, 以 mspan 为单位管理页)
        ▼
OS(mmap 大块虚拟内存)
```

- **mspan**:一组连续页,切成同一 size class 的等大小块。小对象(≤32KB)从对应 size class 的 span 里取。
- **mcache**:每个 P 私有,分配小对象时**无需加锁**(对照 C 的 `malloc` 常有全局锁竞争)——这是 Go 高并发分配快的原因之一,也是为什么 P 这个概念在调度和内存分配两处都出现:它作为"每个逻辑处理器私有的上下文",天然是无锁本地缓存的载体。
- 大对象(>32KB)直接从 mheap 分配。

呼应[第03章](03-变量与内存模型.md):**逃逸到堆的对象越多,分配和 GC 压力越大**。想优化就减少堆分配——预分配 slice 容量、复用对象(`sync.Pool`)、避免不必要的接口装箱/闭包捕获,用 `go build -gcflags=-m` 看逃逸分析、用 [pprof](12-常用第三方库.md) 看分配热点。

## 小结

- **用法层面**:函数用 `(结果, error)` 多返回值,调用方 `if err != nil` 检查;`errors.New`/`fmt.Errorf` 创建错误,`%w` 包装形成错误链,`errors.Is`/`errors.As` 沿链匹配;`defer` 注册函数返回前执行的清理动作(后进先出);`panic`/`recover` 只用于真正异常的情形,`recover` 必须写在 `defer` 里。
- **设计取舍**:`error` 是普通接口值,`{itab, data}` 两个字,返回它和返回任何值一样廉价——这是"错误是显式数据流而非隐式控制流"这条原则的直接体现,代价是啰嗦的 `if err != nil`,换来的是调用方无法忽视失败路径,与 Java checked exception 想解决同一问题但仍困于异常的隐式跳转机制不同;panic/recover 专门服务于"程序自身逻辑已经出错"这一类和"可预期失败"性质不同的错误,只在边界处兜底而非替代日常错误处理;`defer` 绑定函数级而非块级生命周期,是因为 Go 对象的生命周期终点由 GC 决定、编译器无法像 C++ RAII 那样静态确定块作用域退出时刻;GC 是并发三色标记-清除 + 混合写屏障,为了把 STW 压到亚毫秒级这一延迟敏感场景的目标,情愿多耗费一些 CPU 做并发标记;分配器是 mcache/mcentral/mspan 多级结构,P 本地无锁分配,减少堆分配即可降低 GC 压力。
