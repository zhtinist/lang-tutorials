# 变量与内存模型

这是全书最重要的一章。如果你是 C/C++ 出身,读完本章你会发现 Go 的内存语义比 C++ 简单得多,但有一个必须先刻进大脑的公理:**Go 里赋值和传参永远是值拷贝**。没有引用传参,没有拷贝构造函数,没有移动语义。所有"看起来像引用"的东西,拆开来看都是"含指针的小结构体被逐字节拷贝"。把这一条吃透,后面 90% 的"为什么改了 a 结果 b 也变了"就都不再是玄学。

**赋值:`b := a` 把 a 的值复制一份给 b。** 之后 a、b 是两个独立的变量,改一个不影响另一个:

```go
package main

import "fmt"

func main() {
	a := 10
	b := a // 把 a 的值复制一份给 b
	b = 99
	fmt.Println(a, b) // 10 99 —— 改 b 不影响 a
}
```

**把变量传给函数,函数拿到的也是一份复制品**,在函数内部修改它,不会影响外面的变量:

```go
package main

import "fmt"

func addOne(x int) {
	x = x + 1 // 只改了函数内部的副本
}

func main() {
	n := 5
	addOne(n)
	fmt.Println(n) // 5 —— 没变
}
```

**想让函数真的改到外面的变量,就要用指针**:`&n` 取得 n 的地址,`*x` 通过地址读写它指向的值:

```go
package main

import "fmt"

func addOnePtr(x *int) {
	*x = *x + 1 // 通过地址改到外面的变量
}

func main() {
	n := 5
	addOnePtr(&n)
	fmt.Println(n) // 6 —— 变了
}
```

## 公理:赋值与传参永远是值拷贝

在 C++ 里,`b = a` 到底发生什么取决于 `a` 的类型和一堆规则:内置类型是逐字节拷贝,类类型会调用拷贝构造/拷贝赋值运算符(可能被用户重载,也可能被编译器隐式生成,还可能因为某个成员没有可访问的拷贝构造而被 `= delete`),右值会触发移动构造,引用类型只是别名。同一个 `=` 符号背后可能是完全不同的代码路径,而且哪条路径生效取决于类型定义在别处如何声明——这是 C++ 里"看一行代码猜不出它做了什么"的一个主要来源,也是《C++ Core Guidelines》里"Rule of Three/Five/Zero"存在的原因:一旦你自定义了析构函数,就必须同时想清楚拷贝构造、拷贝赋值、移动构造、移动赋值该怎么处理,否则容易出现二义性或资源双重释放。

Go 把这件事砍到只剩一条规则:

> `b = a` 以及"把 a 作为实参传进函数",**都是把 a 的机器表示逐字节复制一份**。

没有用户可重载的拷贝行为,没有隐藏的构造函数调用。`a` 占多少字节,就复制多少字节。区别只在于——**这些字节里装的是不是指针**。

- 如果 `a` 是 `int`、`bool`、`float64`、数组、结构体(值类型),那么拷贝的就是全部数据本身。
- 如果 `a` 是指针、slice、map、chan、interface、func(内部含指针),拷贝的是那几个字长的"表头/句柄",表头里的指针被复制,于是 `a` 和 `b` 指向同一块底层数据。

**为什么 Go 选择和 C 一样"永远值拷贝"这条最简单的规则,而不是引入引用类型或可重载的拷贝语义**:C++ 的拷贝构造函数机制是为了解决一个真实问题——某些类型(比如管理堆内存的容器)不能简单地逐字节拷贝,否则两个对象会持有同一块内存的所有权,析构时重复释放。但解决这个问题的代价是"赋值"这个操作本身失去了确定性含义,读代码的人必须去查类型定义才知道 `=` 到底做了什么。Go 的应对方式不是让拷贝语义变聪明,而是从根源上让"共享数据"这件事变得**显式**:需要共享,就用含指针的表头类型(slice/map/chan)或者显式指针,类型本身告诉你"这里面有指针,拷贝时表头复制、数据共享";不需要共享,就是纯值类型,拷贝即独立。再加上 GC 兜底内存回收,Go 根本不需要拷贝构造函数解决的那个"所有权"问题——没有所有权语义,就没有必要为拷贝定义特殊行为。这是"简单性"换掉"表达力"的又一个例子:C++ 能让 `+`、`=`、拷贝都表达任意语义,Go 不能,但换来的是任何人看到 `b := a` 都能确定它做了什么,不需要跳转到类型定义去确认。

所以 Go 里根本不存在 C++ 意义上的"引用类型"。所谓引用类型只是**表头小、里面有指针**的值类型而已。记住:拷贝的永远是"值",争议只在于"这个值里有没有指针"。

## 值类型:逐字节复制整块内存

先看纯值类型。下面的 `Point` 是一个结构体,没有任何指针字段:

```go
package main

import "fmt"

type Point struct {
	X, Y int64
}

func main() {
	a := Point{X: 1, Y: 2}
	b := a // 逐字节拷贝 16 字节,b 与 a 从此毫无关系

	b.X = 100
	fmt.Println(a.X, b.X) // 1 100 —— 改 b 不影响 a
}
```

`b := a` 在底层就是一条"复制 16 字节"的指令(gc 编译器会用 `MOVUPS` 之类的指令搬运)。这跟 C 里 `struct Point b = a;` 完全一样:整块 POD 内存原样复制,没有构造函数介入。

内存图(两块完全独立的内存):

```
a: [ X=1 | Y=2 ]        b: [ X=100 | Y=2 ]
   ^独立 16 字节            ^另一块独立 16 字节
```

传参同理。前面 `addOne` 的例子是 int 版本,下面看结构体版本,拿到的同样是副本:

```go
func move(p Point) { // p 是实参的拷贝
	p.X += 10 // 只改副本
}

func main() {
	a := Point{1, 2}
	move(a)
	fmt.Println(a.X) // 1 —— 外面的 a 纹丝不动
}
```

这和 C 里 `void move(struct Point p)` 一模一样:值传参,函数内改的是栈上的副本。想改外面的东西,就得传指针——和 C 一样。把"传值"和"传指针"并排放一起,差别一目了然:

```go
package main

import "fmt"

type Point struct{ X, Y int64 }

func moveVal(p Point)  { p.X += 10 } // 改的是副本
func movePtr(p *Point) { p.X += 10 } // 通过指针改原对象

func main() {
	a := Point{X: 1, Y: 2}

	moveVal(a)
	fmt.Println(a.X) // 1  —— 没变,函数拿到的是副本

	movePtr(&a)
	fmt.Println(a.X) // 11 —— 变了,函数拿到的是地址
}
```

注意 `movePtr` 里写的是 `p.X`,而不是 `(*p).X`——Go 对指向 struct 的指针会自动解引用,省去 C 里 `p->X` 那种专门的箭头运算符。

数组也是值类型,而且拷贝整个数组(这一点 C 程序员要特别当心,C 里数组传参会退化成指针,Go 不会):

```go
arr := [3]int{1, 2, 3}
cp := arr // 拷贝全部 3 个元素,24 字节
cp[0] = 99
fmt.Println(arr[0], cp[0]) // 1 99
```

> C 陷阱对照:C 里 `void f(int a[3])` 等价于 `void f(int *a)`,传的是首元素地址;Go 里 `func f(a [3]int)` 传的是**整个数组的副本**。长度是类型的一部分,`[3]int` 和 `[4]int` 是不同类型。想要 C 那种"传引用"效果,请传 `*[3]int` 或改用 slice。

## 指针:有指针,无指针算术,更安全

Go 有指针,`&` 取地址、`*` 解引用,语法和 C 一样:

```go
a := 42
p := &a  // p 的类型是 *int
*p = 100 // 通过指针改 a
fmt.Println(a) // 100
```

底层上,`p` 就是一个存着 `a` 地址的机器字(64 位平台上 8 字节),和 C 指针的表示完全相同。但 Go 的指针有几条纪律:

| 特性 | C/C++ | Go |
|---|---|---|
| 取地址 / 解引用 | `&` / `*` | `&` / `*`(相同) |
| 指针算术 `p+1`、`p[i]` | 允许 | **禁止**(除非 `unsafe.Pointer`) |
| 空指针 | `NULL` / `nullptr` | `nil` |
| 悬垂指针 | 常见 bug | GC 保证:只要有指针指向,对象就不回收 |
| 指向局部变量并返回 | 未定义行为 | **合法**,变量自动逃逸到堆 |

最后一条对 C 程序员是重大解放:

```go
func newCounter() *int {
	n := 0     // 局部变量
	return &n  // 在 C 里这是灾难,在 Go 里完全合法
}
```

在 C 里返回局部变量地址是经典的悬垂指针 bug,函数返回后栈帧销毁,指针指向垃圾。Go 里编译器发现 `n` 的地址逃出了函数作用域,就自动把 `n` 分配到堆上,由 GC 管理生命周期。这就引出下一个核心话题——**逃逸分析**。

## 逃逸分析:栈还是堆,不由 new/& 决定

C++ 程序员的直觉是:`int x;` 在栈上,`new int` 在堆上,由你写的代码字面决定。**Go 不是这样。** 一个变量在栈还是堆,由编译器的**逃逸分析(escape analysis)**决定,和你用没用 `&`、用没用 `new` 没有必然关系。

**为什么这件事必须交给编译器,而不能像 C++ 那样由程序员在代码里字面决定**:C++ 的"栈上分配还是堆上分配"由语法直接决定(`T x` vs `new T`),这在没有 GC 的语言里是必要的——因为堆对象的生命周期需要程序员自己管理,`new` 和 `delete` 必须配对,这个决定权本来就必须交给写代码的人。但 Go 有 GC,堆对象的生命周期不再需要程序员操心,于是"一个变量该放哪"就单纯变成一个**性能问题**而非**正确性问题**:放栈上更快(函数返回自动回收,没有 GC 扫描成本),放堆上是为了让指向它的指针能活得比函数调用更久。这个判断依赖于**这个变量的地址会不会在函数返回后仍被引用**——而这件事往往需要看到函数内部所有的代码路径(地址有没有被存进全局变量、传给别的 goroutine、装进 interface)才能确定,程序员在写 `n := 0` 那一刻通常并不能、也不需要预判它未来会不会被取地址返回。把这个决定交给编译器,好处是你写代码时只需要关心"逻辑对不对",堆栈归属是一个可以被编译器根据实际数据流优化的实现细节,而不是一个你必须提前想清楚、写错了会导致内存错误的语法选择。

规则的核心是一个问题:**这个变量的引用会不会活得比它所在的函数栈帧更久?**

- 不会逃逸 → 放栈上,函数返回自动回收,零 GC 成本。
- 会逃逸(地址被返回、被存进堆对象、被存进 interface、生命周期不确定)→ 放堆上,交给 GC。

用 `go build -gcflags='-m'`(可加 `-m -m` 看更详细的推理)可以让编译器打印它的决策。看这个例子:

```go
package main

func stackAlloc() int {
	x := new(int) // 别被 new 骗了
	*x = 42
	return *x // 只返回值,x 的地址没跑出去
}

func heapAlloc() *int {
	y := 0
	return &y // 地址逃逸了
}

func main() {
	_ = stackAlloc()
	_ = heapAlloc()
}
```

编译并观察(不同 Go 版本、是否内联,打印的行号和措辞会略有差异,但结论一致):

```
$ go build -gcflags='-m -l' escape.go
./escape.go:4:10: new(int) does not escape
./escape.go:10:2: moved to heap: y
```

结论颠覆直觉:**`new(int)` 没有逃逸,反而分配在栈上**;而普普通通的局部变量 `y`,因为地址被返回,被"moved to heap"。

这条给 C/C++ 程序员的心智更新非常关键:

> 在 Go 里,`new`/`&` 只表达"我要一个可寻址的对象 / 我要它的地址",**不表达"分配在堆上"**。堆栈归属是编译器根据逃逸分析算出来的。你不需要、也无法手动决定(除了通过改变代码结构来影响逃逸结果)。

常见的逃逸触发点:把局部变量地址返回、把指针塞进 slice/map/结构体字段(而该容器逃逸)、把值装进 `interface{}`(如 `fmt.Println(x)` 常导致 x 逃逸)、闭包捕获并修改外层变量、大小在编译期未知的对象。

实用心法:别为了"避免堆分配"到处纠结,先写清楚的代码;发现热点再用 `-gcflags='-m'` 和 pprof 定位真正逃逸的地方。Go 的哲学是让编译器替你操心栈堆归属。

## new vs make

两个内建分配函数,C 程序员容易混淆,一句话区分:

| | new(T) | make(T, ...) |
|---|---|---|
| 适用类型 | 任意类型 T | **只能**用于 slice / map / chan |
| 返回值 | `*T`(指向零值的指针) | `T` 本身(不是指针) |
| 做的事 | 分配一块清零的内存 | 分配并**初始化内部数据结构**(表头+底层数组/哈希表/环形缓冲) |

```go
p := new([]int) // p 是 *[]int,指向一个 nil slice 表头,底层还没数组,不能直接用
s := make([]int, 3) // s 是 []int,长度 3,底层数组已就绪,可用
```

`new(T)` 相当于 C 里 `calloc(1, sizeof(T))` 再取地址——只给你一块清零内存。而 `make` 是 slice/map/chan 专用的"构造器",因为这三者需要初始化内部的表头和底层结构才能工作(map 尤其:未初始化的 nil map 写入会 panic)。实践中 `new` 用得很少,多数时候直接用复合字面量 `&T{...}` 更清楚。

## 零值可用哲学

C/C++ 里读未初始化的变量是未定义行为,栈上是随机垃圾。**Go 保证每个变量都被初始化为它类型的"零值"**,而且这些零值被设计成"开箱可用":

| 类型 | 零值 |
|---|---|
| 数值 (int/float/…) | `0` |
| bool | `false` |
| string | `""`(空串,不是 nil) |
| 指针 / slice / map / chan / func / interface | `nil` |
| struct | 每个字段各自的零值(递归) |

零值可用意味着很多类型不写构造函数、直接声明就能用:

```go
var buf bytes.Buffer // 零值就是一个可用的空缓冲,无需初始化
buf.WriteString("hi")

var mu sync.Mutex // 零值就是未锁状态,直接 Lock
mu.Lock()

var s []int    // nil slice,但 append 能直接用
s = append(s, 1) // OK,append 会按需分配底层数组
```

注意一个例外:**nil map 只能读不能写**。`var m map[string]int` 读 `m["x"]` 返回 0(零值),但 `m["x"] = 1` 会 panic,因为底层哈希表还没被 make 出来。这是 map 和 slice 在零值行为上的重要区别。

## "引用类型"的真相:含指针的小表头

现在拆穿那个所谓的"引用类型"。slice、map、chan 常被教程称为引用类型,好像它们和值类型是两个物种。真相是:**它们仍然是值,`a = b` 仍然是逐字节拷贝**,只不过它们的值是一个"小表头结构",表头里含有指向底层数据的指针。拷贝表头 → 指针被复制 → 两个变量共享同一份底层数据。

以 slice 为例。它的表头在运行时就是这样一个 3 字长的结构(见运行时 `reflect.SliceHeader`):

```go
type sliceHeader struct {
	Data uintptr // 指向底层数组的指针
	Len  int     // 长度
	Cap  int     // 容量
}
```

24 字节(64 位平台)。当你写 `b := a`(a 是 slice),复制的是这 24 字节表头,`Data` 指针被原样复制。于是:

```go
a := []int{1, 2, 3}
b := a       // 拷贝表头,b.Data == a.Data,共享同一底层数组
b[0] = 99    // 通过共享的底层数组改元素
fmt.Println(a[0]) // 99 —— a 也变了!这就是"别名"
```

这正是让 C++ 程序员疑惑的点:明明是值拷贝,怎么改 b 影响 a?因为拷贝的表头里的指针指向同一块堆内存。这跟 C 里 `struct { int *data; int len; } b = a;` 之后 `b.data[0] = 99` 会影响 `a` 是一个道理——拷贝了结构体,但没拷贝它指向的缓冲区。

但这个"共享"是有条件的——一旦 `append` 触发扩容,底层数组被重新分配,共享关系就悄悄断开了。这是新手最常踩的坑:

```go
package main

import "fmt"

func main() {
	a := []int{1, 2, 3}
	b := a // 拷贝 24 字节表头,b 与 a 共享同一底层数组

	b[0] = 99
	fmt.Println(a) // [99 2 3]  —— 共享数组,a 也看到了

	b = append(b, 4) // len==cap==3 已满,append 重新分配:b 换到了自己的新数组
	b[1] = 88
	fmt.Println(a) // [99 2 3]     —— 没变,共享关系已断开
	fmt.Println(b) // [99 88 3 4]
}
```

`append` 之前改 `b` 会波及 `a`,`append` 之后改 `b` 却不再波及 `a`——同一段代码里两种行为,根源就是那次重新分配。什么时候扩容、扩多少,取决于容量,细节见 [07-复合数据结构.md](07-复合数据结构.md)。

map 和 chan 更彻底:它们的变量本身**就是一个指针**(指向运行时的 `hmap` / `hchan` 结构)。`b := a` 拷贝这个指针,两者操作的是同一个哈希表 / 同一个通道。

这里只建立直觉。slice 的 len/cap、append 的扩容与再分配、map 的哈希结构、chan 的环形缓冲等细节,留到 [07-复合数据结构.md](07-复合数据结构.md) 详细展开。现在你只需牢记这条统一图景:**没有引用类型,只有"表头含指针的值类型";赋值拷贝表头,共享底层数据。**

## ASCII 对照图:struct 值拷贝 vs slice 拷贝

**结构体值拷贝(无共享)**:`b := a`,把整块数据复制一份,老死不相往来。

```
struct 值类型          a := Point{1,2}; b := a

  a ┌─────────┐        b ┌─────────┐
    │ X = 1   │          │ X = 1   │   两块独立内存
    │ Y = 2   │          │ Y = 2   │   改 b 不影响 a
    └─────────┘          └─────────┘
```

**slice 表头拷贝(共享底层)**:`b := a`,复制 24 字节表头,两个表头的 Data 指向同一个数组。

```
slice "引用类型"       a := []int{1,2,3}; b := a

  a ┌──────────┐       b ┌──────────┐
    │ Data ●───┼──┐      │ Data ●───┼──┐   表头各自独立(在栈上)
    │ Len  3   │  │      │ Len  3   │  │
    │ Cap  3   │  │      │ Cap  3   │  │
    └──────────┘  │      └──────────┘  │
                  └──────┬─────────────┘
                         ▼   两个表头指向同一底层数组(在堆上)
              ┌────┬────┬────┐
              │  1 │  2 │  3 │  ← 改 b[0] 就是改这里,a[0] 跟着变
              └────┴────┴────┘
```

一句话收束:**看到 `a = b`,先问自己 b 的字节里有没有指针。没有 → 两份独立数据;有 → 表头独立、底层共享。** 这条判据能解释 Go 里几乎所有的别名现象。

## 小结

- **用法层面**:`var`/`:=` 声明变量;`b := a` 把 a 的值复制给 b;把变量传给函数,函数拿到的是副本;想让函数改到外面的变量,传指针(`&`/`*`)。
- **底层层面**:赋值与传参永远是值拷贝——这是 Go 用"简单性"替代 C++ 拷贝构造函数/移动语义那套隐式规则的直接结果;值类型拷贝全部数据,"引用类型"(slice/map/chan/指针/interface)拷贝的是含指针的表头,于是产生别名;变量在栈还是堆由逃逸分析决定,这是把"堆栈归属"从程序员必须操心的语法选择,变成编译器可优化的性能细节;`new` 只清零分配,`make` 专为 slice/map/chan 初始化内部结构;零值被设计成开箱可用,但 nil map 只读不可写。
