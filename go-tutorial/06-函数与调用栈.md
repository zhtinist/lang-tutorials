# 函数与调用栈:参数传递的真相

Go 的函数在语法上像 C,但在内存语义上有几个必须彻底钉死的事实——**一切都是按值传递**、**方法接收者也是值或指针的拷贝**、**闭包捕获的是变量而非值**、**goroutine 栈是可增长的**。C/C++ 老手若把这些当成"和 C 一样"就会持续踩坑。

**定义并调用一个函数,写法和大多数语言差不多——只是返回类型写在参数列表后面**:

```go
package main

import "fmt"

func add(a, b int) int { // 定义一个函数:两个 int 参数,返回一个 int
	return a + b
}

func divmod(a, b int) (int, int) { // 可以一次返回多个值
	return a / b, a % b
}

func main() {
	fmt.Println(add(3, 4)) // 调用函数:7

	q, r := divmod(17, 5) // 一次接收两个返回值
	fmt.Println(q, r)     // 3 2
}
```

**方法就是"绑定到某个类型"的函数**,多了一个写在函数名前面的"接收者":

```go
package main

import "fmt"

type Counter struct {
	n int
}

func (c *Counter) Inc() { // 方法:c 是接收者,类似"这个方法属于 *Counter"
	c.n++
}

func main() {
	c := Counter{}
	c.Inc()
	c.Inc()
	fmt.Println(c.n) // 2
}
```

**闭包是一个"记得住外部变量"的匿名函数**:

```go
package main

import "fmt"

func makeAdder(base int) func(int) int {
	return func(x int) int { // 这个匿名函数记住了外面的 base
		return base + x
	}
}

func main() {
	add10 := makeAdder(10)
	fmt.Println(add10(5)) // 15
}
```

下面从"参数到底是怎么传的"这个最根本的问题开始,逐条钉死函数、方法、闭包在内存里的真实行为。

## 函数是一等公民

函数在 Go 里是值:可以赋给变量、作为参数、作为返回值、放进 slice/map。底层上,一个函数值是一个指向 **funcval 结构**的指针(内含代码地址,闭包时还含捕获的变量),因此函数变量本身只是一个指针大小的字长。

```go
package main

import "fmt"

func add(a, b int) int { return a + b }

func main() {
	var op func(int, int) int = add // 函数赋给变量
	fmt.Println(op(3, 4))            // 7

	apply := func(f func(int, int) int, x, y int) int {
		return f(x, y) // 函数作为参数
	}
	fmt.Println(apply(add, 10, 20)) // 30
}
```

这类似 C 的函数指针,但 Go 的函数值可以携带状态(闭包),更接近 C++ 的 `std::function` / lambda——只是没有那层类型擦除的堆分配开销(除非闭包变量逃逸)。

## 没有函数重载:一个名字一个含义

Go 不允许同一个包内定义两个同名函数,哪怕参数类型或个数不同——这一点和 C++/Java 明显不同:

```go
func Add(a, b int) int       { return a + b }
// func Add(a, b float64) float64 { return a + b } // 编译错误:Add 已声明
```

**为什么不支持重载**:C++ 的重载决议(overload resolution)本身是一套相当复杂的规则——编译器要在一堆候选函数里,按参数类型的隐式转换等级(精确匹配 > 提升 > 标准转换 > 用户定义转换)挑出"最佳匹配",遇到二义性时还会报出一长串候选者列表。这套机制的代价是:看到一处函数调用 `f(x)`,要知道具体调的是哪个重载,有时必须知道 `x` 的精确类型以及所有重载的完整签名——尤其在类型可以隐式转换时,调用可能悄悄命中一个"看起来不对但技术上合法"的重载。Go 选择"一个名字对应一个函数"这条更简单的规则:在某个作用域里看到一个函数名,它的签名是唯一确定的,不需要额外做重载决议这一步心算。

代价是啰嗦:处理不同类型的相似逻辑,要么写多个名字不同的函数(`ParseInt`、`ParseFloat`、`ParseBool`,标准库 `strconv` 包就是这么做的),要么用 `interface{}`/`any` 在运行时分支处理(牺牲编译期类型检查),要么用 Go 1.18 引入的**泛型**(类型参数,见[第 8 章](08-面向对象与类型系统.md))在编译期为不同类型生成同一份逻辑。泛型某种程度上是重载想解决的问题("同一段逻辑套用到不同类型")的一个更受限、但语义更清晰的答案——它不允许你为同一个类型参数写出行为不同的多个版本,调用点永远只有一个函数可能被调用,只是它的类型参数不同。

## 多返回值:惯用 (result, error)

前面已经用过多返回值。底层实现:自 **Go 1.17 的寄存器 ABI**,返回值优先通过**寄存器**传递(超过可用寄存器数量的部分才落到栈上),被调函数直接写入,无需像 C 那样用输出参数(指针)或打包结构体。

**为什么原生支持多返回值,而不是让程序员自己选"出参指针"或"打包成 struct"**:这两种 C 里的替代方案都有明显的摩擦。出参指针(`int divmod(int a, int b, int *rem)`)把"输出"伪装成"输入参数",调用方必须先声明一个变量再取地址传进去,读函数签名时无法一眼分清哪些参数是输入、哪些是输出;打包成 struct 则需要专门为每一种返回值组合定义一个类型,而且调用方要为这个临时的组合类型想名字。这两种方案都在解决同一个问题——"函数天生只能返回一个值"——却把额外的样板工作转嫁给了每一个调用点。Go 直接在语言层面让函数签名支持 `(int, int)` 这样的返回类型,把"多个返回值"处理成语言原生能力而不是靠约定拼凑,调用方用 `a, b := f()` 一行接收,读起来就是"这个函数产出这些东西",没有中间层的样板类型。

这个能力真正的价值在于撑起了 Go 最重要的惯用法——`(result, error)`:最后一个返回值是 `error`,调用方立刻检查:

```go
func parse(s string) (int, error) {
	// ...
	return 0, fmt.Errorf("解析失败: %q", s)
}

func main() {
	n, err := parse("abc")
	if err != nil {
		fmt.Println("出错:", err)
		return
	}
	fmt.Println(n)
}
```

如果没有多返回值,这个惯用法就得靠出参指针(`int parse(char*, int*)`,C 的 errno 传统)或者异常来实现。Go 在[第 1 章](01-运行模型.md#八极简主义哲学为什么去掉异常早期泛型类继承)已经说明了为什么放弃异常:异常会制造看不见的控制流跳转。`(result, error)` 正是多返回值这个语言能力,与"错误应该是显式返回值而非隐式抛出"这个设计立场结合后的自然产物——错误处理路径和正常路径写在同一行代码旁边,调用方无法假装没看到 `err`(编译器虽不强制检查,但 `go vet`、linter 和社区习惯都会盯着未处理的 error)。详见 [09-错误处理与资源管理](09-错误处理与资源管理.md)。

## 全部按值传递

**Go 里没有引用传递。** 每次函数调用,所有实参都被**拷贝**一份到被调函数的栈帧。要在函数里改动调用方的变量,必须显式传**指针**(拷贝的是指针值,但它指向同一对象)。

```go
func incValue(x int)  { x++ }      // 改的是拷贝,调用方看不到
func incPtr(x *int)   { *x++ }     // 通过指针改到原变量

func main() {
	a := 10
	incValue(a)
	fmt.Println(a) // 仍是 10

	incPtr(&a)
	fmt.Println(a) // 11
}
```

内存图:

```
main 的栈帧:            incValue 的栈帧:
  a = 10  ──拷贝──────▶   x = 10   (x++ 只改这里,返回后销毁)

  a = 11  ◀──*x++──┐     incPtr 的栈帧:
        │          └──   x = &a   (x 是指向 a 的指针,拷贝的是地址)
        └── a 被改到
```

**这条规则对所有类型统一成立**,包括 struct、数组。传一个大 struct 会**整块拷贝**:

```go
type Big struct{ data [1024]int } // 8KB

func mutate(b Big)  { b.data[0] = 1 } // 拷贝 8KB,改动无效
func mutateP(b *Big) { b.data[0] = 1 } // 拷贝 8 字节指针,改动有效
```

这与[第 3 章](03-变量与内存模型.md)的公理是同一件事在函数调用场景下的体现:传参和赋值遵守完全相同的拷贝规则,Go 没有为"传参"单独设计一套语义(不像 C++ 的引用参数 `T&` 那样,在语法层面就制造了一种"看起来是值、实际是别名"的特例)。

### slice / map / channel 是"含指针的表头"

初学者常误以为 slice/map "按引用传递"。真相是:它们**也是按值拷贝**,只不过它们的值是一个**含指针的小表头**。拷贝表头 = 复制指针,于是两份表头指向**同一底层数据**,产生别名(aliasing)效果:

```go
func fill(s []int) {
	s[0] = 99 // 通过表头里的指针改到共享的底层数组 —— 调用方可见
	s = append(s, 7) // append 可能换底层数组;即便不换,这是给【局部表头】重新赋值,调用方的表头长度不变
}

func main() {
	s := []int{1, 2, 3}
	fill(s)
	fmt.Println(s) // [99 2 3] —— s[0] 改了,但长度仍是 3
}
```

内存图(slice 表头拷贝):

```
main 的 s (表头):          fill 的 s (拷贝的表头):
  ptr ──┐                    ptr ──┐   ← 两个 ptr 相等,指向同一数组
  len=3 │                    len=3 │
  cap=3 │                    cap=3 │
        └──────┬─────────────┘
               ▼
    底层数组(堆): [99, 2, 3]   ← s[0]=99 两边都看得到
```

| 类型 | 传递时拷贝的是什么 | 修改元素调用方可见? |
| --- | --- | --- |
| int/struct/array | 整个值(可能很大) | 否 |
| pointer | 指针(1 字长) | 是(通过解引用) |
| slice | 表头(ptr+len+cap,3 字长) | **改元素:是**;append/重切:否 |
| map | 表头指针(1 字长) | **是** |
| channel | 表头指针(1 字长) | 是(收发) |
| string | 表头(ptr+len,只读) | string 不可变 |

理解 map/slice/channel 的详细内存布局见 [07-复合数据结构](07-复合数据结构.md)。

## 方法:值接收者 vs 指针接收者

方法只是第一个参数(接收者)有特殊语法的函数。接收者同样遵循**按值传递**规则。前面的 `Inc` 用的是指针接收者;下面把值接收者和指针接收者并排对比:

```go
type Counter struct{ n int }

// 值接收者:c 是整个 Counter 的【拷贝】,改动不影响原对象
func (c Counter) IncValue() { c.n++ }

// 指针接收者:c 是指针的拷贝,通过它能改到原对象
func (c *Counter) IncPtr() { c.n++ }

func main() {
	var c Counter
	c.IncValue()
	fmt.Println(c.n) // 0 —— 改的是拷贝

	c.IncPtr()
	fmt.Println(c.n) // 1 —— 改到原对象
}
```

选择接收者类型的两条底层理由:

1. **要修改接收者** → 必须用指针接收者(否则改的是拷贝)。
2. **接收者是大 struct** → 用指针避免每次调用整块拷贝(值接收者会把整个 struct 压栈)。

小的、不可变语义的类型(如表示坐标的 `Point{x,y int}`)用值接收者更简单、更缓存友好。惯例:同一类型的方法**要么全用值、要么全用指针**,不要混。

### 可寻址性与自动取地址

`c.IncPtr()` 里 `c` 是值,但编译器自动改写成 `(&c).IncPtr()`——前提是 `c` **可寻址(addressable)**。局部变量、slice 元素、可解引用的指针目标都可寻址;而 **map 元素、临时值(函数返回值)不可寻址**,不能对它们调用指针接收者方法:

```go
type Counter struct{ n int }
func (c *Counter) IncPtr() { c.n++ }

m := map[string]Counter{"a": {}}
// m["a"].IncPtr() // 编译错误:m["a"] 不可寻址,无法取地址
```

### 方法集

一个类型的**方法集**决定它能满足哪些接口:

- 类型 `T` 的方法集:所有**值接收者**方法。
- 类型 `*T` 的方法集:值接收者 **+** 指针接收者方法(全部)。

也就是说,指针接收者方法**不**属于 `T` 的方法集。因此一个值类型 `T` 若想满足某接口,而该接口方法用了指针接收者,则只有 `*T` 满足接口,`T` 不满足。这条规则与可寻址性配合,是接口赋值报错的常见根源,详见 [08-面向对象与类型系统](08-面向对象与类型系统.md)。

## 闭包捕获的是变量,不是值

前面的 `makeAdder` 已经展示了闭包的基本用法。闭包(匿名函数)捕获外层变量时,捕获的是**变量本身(其存储位置)**,不是当时的值。若被捕获的变量在函数返回后仍被闭包引用,逃逸分析会把它**从栈搬到堆**,让闭包和外层共享同一份存储。

```go
func counter() func() int {
	n := 0 // 逃逸到堆,因为返回的闭包持有它
	return func() int {
		n++ // 每次调用共享并修改同一个 n
		return n
	}
}

func main() {
	c := counter()
	fmt.Println(c(), c(), c()) // 1 2 3
}
```

这与 C++ lambda 的 `[&n]`(按引用捕获)语义相同,而非 `[n]`(按值)。C 没有闭包,只能靠传结构体指针模拟。

"捕获的是变量"最有说服力的证据,是让**两个**闭包捕获**同一个**变量——一个写、一个读,它们看到的是同一份存储:

```go
package main

import "fmt"

func newAccount() (deposit func(int), balance func() int) {
	bal := 0
	deposit = func(n int) { bal += n }
	balance = func() int { return bal }
	return
}

func main() {
	dep, bal := newAccount()
	dep(100)
	dep(50)
	fmt.Println(bal()) // 150,两个闭包共享同一个 bal
}
```

`bal` 逃逸到堆,`deposit` 和 `balance` 都持有指向它的引用,于是形成了一份被封装的私有状态(闭包 = 数据 + 行为)——这正是 C 里要靠"结构体 + 一组操作它的函数指针"才能拼出来的东西。

### 经典循环变量坑与 Go 1.22 语义变化

因为捕获的是变量,在 Go 1.21 及以前,循环变量整个循环只有一个实例,所有闭包捕获到**同一个**变量:

```go
// Go 1.21-:输出 3 3 3(共享同一个 i,循环结束时 i=3)
// Go 1.22+:输出 0 1 2(每轮 i 是新实例,各闭包捕获不同变量)
func main() {
	funcs := make([]func(), 0, 3)
	for i := 0; i < 3; i++ {
		funcs = append(funcs, func() { fmt.Print(i, " ") })
	}
	for _, f := range funcs {
		f()
	}
}
```

**Go 1.22 起,`for` 的循环变量每轮迭代都是新实例**(见 [05-控制流](05-控制流.md#go-1.22循环变量每轮新实例)),这个几乎人人踩过的坑被语言层面修掉——这条语义变更为什么会发生、又为什么直到 1.22 才发生,05 章已经详细展开:原始设计对应 C 的直觉且省一次分配,但和"goroutine 廉价 + 闭包捕获变量"这两个 Go 自身的特性组合后,变成了被踩得最多的真实 bug,`go.mod` 里的语言版本号让这个破坏性修复得以只作用于新代码。在 1.21 及以前,老写法要靠每轮 `i := i` 显式创建副本来规避。用 goroutine 时同理(`go func(){ ... i ... }()`),1.22 前后行为不同。

## 变参 ...T

`...T` 声明变参,函数内部把它当作 `[]T`(slice)使用。底层:调用方把这些实参**打包成一个新 slice**(通常在栈上分配底层数组,不逃逸时)传入。

```go
func sum(nums ...int) int { // nums 的类型是 []int
	total := 0
	for _, n := range nums {
		total += n
	}
	return total
}

func main() {
	fmt.Println(sum(1, 2, 3)) // 传 3 个 → 打包成 []int{1,2,3}
	fmt.Println(sum())        // 传 0 个 → nums 是 nil slice,len=0

	xs := []int{4, 5, 6}
	fmt.Println(sum(xs...)) // 用 ... 把已有 slice 展开;此时【不拷贝】,直接复用 xs 的底层数组
}
```

注意 `sum(xs...)` 复用 `xs` 的底层数组,若函数内修改 `nums[i]` 会改到 `xs`。这与 C 的 `va_list`(无类型、无长度、需哨兵)相比,类型安全且自带长度。

## 命名返回值 + defer 修改返回值

结合 [05-控制流](05-控制流.md#defer延迟执行) 的 defer,命名返回值是一个强大组合。`return` 先把值写入命名的返回槽,再跑 defer,defer 可改这个槽:

```go
func readConfig() (cfg string, err error) {
	defer func() {
		if r := recover(); r != nil {
			// 把 panic 转成 error 返回给调用者
			err = fmt.Errorf("恢复自 panic: %v", r)
		}
	}()
	panic("配置文件损坏")
	// 到达不了,但 defer 会把 err 设好后正常返回
}
```

底层上命名返回值就是被调函数栈帧里的具名变量,`return x` = "把 x 拷进返回槽 + 执行 defer + 真正返回"。这是把 panic 转换为 error 的标准手法,详见 [09-错误处理与资源管理](09-错误处理与资源管理.md)。

## goroutine 栈:可增长的连续栈

这是 Go 运行时最漂亮的设计之一,和 C/C++ 线程栈截然不同。

C/C++ 的线程栈在创建时**固定大小**(典型 1–8MB),由 OS 预留虚拟地址空间。开太多线程会吃光地址空间;递归太深会栈溢出(segfault)。

Go 的每个 goroutine 有**自己的栈,初始只有约 2KB**,由 Go 运行时管理(不是 OS)。因此可以轻松创建几十万个 goroutine——2KB × 100 万 ≈ 2GB,而线程做不到。

### 栈如何增长:morestack 与栈拷贝

编译器在每个函数入口插入一小段**栈检查(stack guard)**代码:检查当前栈指针是否快触底。若空间不足,调用运行时的 `morestack`:

1. 分配一块**更大的新栈**(通常翻倍)。
2. **把旧栈的全部内容拷贝**到新栈(连续栈,contiguous stack)。
3. **调整所有指向旧栈的指针**(运行时能精确扫描栈上的指针,这依赖 Go 的精确 GC 类型信息)——这一步 C 做不到,因为 C 无法知道栈上哪些是指针。
4. 释放旧栈,函数继续执行。

栈也会在 GC 时按需**收缩**。

```
增长前(2KB 栈快满):        增长后(拷贝到 4KB 新栈):
 ┌──────────┐ 高地址        ┌──────────┐
 │ 帧 A     │               │ 帧 A     │  ← 内容逐字节拷贝
 │ 帧 B     │   morestack   │ 帧 B     │
 │ 帧 C ←SP │  ══════════▶  │ 帧 C     │
 └──────────┘ 触底           │ 帧 D ←SP │  ← 现在有空间继续
                            │  ...     │
                            └──────────┘ 低地址
```

### 对代码的影响

- **不要假设栈变量地址稳定**:栈增长后地址会变(但指向它的指针会被运行时修正,所以对你透明)。这也是为什么 Go 里取局部变量地址完全安全——即使逃逸,运行时/编译器会正确处理(逃逸的搬到堆,不逃逸的随栈移动)。
- 深递归在 Go 里不易 segfault,但会持续增长栈直至触及上限(64 位默认 `maxstacksize` ≈ 1GB,可调);超限时 runtime 直接 `fatal error: stack overflow` **终止进程**——这是**不可 `recover`** 的 fatal error,和普通 panic 有本质区别。
- goroutine 的调度与栈,配合 channel,构成 Go 并发模型的基础,见 [10-并发模型](10-并发模型.md)。

| 维度 | C/C++ 线程栈 | Go goroutine 栈 |
| --- | --- | --- |
| 初始大小 | 固定,1–8MB | 约 2KB |
| 增长 | 不能(溢出即崩) | 自动,morestack + 拷贝 |
| 收缩 | 不能 | GC 时按需收缩 |
| 管理者 | OS | Go 运行时 |
| 可创建数量 | 几千 | 几十万~百万 |
| 栈上指针修正 | 不支持 | 支持(精确 GC) |

## 小结

- **用法层面**:函数用 `func 名(参数 类型) 返回类型 { ... }` 定义,可一次返回多个值;方法在函数名前加接收者 `(c *T)`;闭包是能记住外部变量的匿名函数,写法是 `func(参数) 返回类型 { ... }`。
- **底层层面**:函数是一等值,底层是 funcval 指针,闭包时携带捕获变量。Go 不支持函数重载,一个名字对应一个确定签名,省去重载决议的心智负担,代价靠泛型/多函数名弥补。**一切按值传递**;要改就传指针——多返回值是这条能力的自然延伸,撑起了 `(result, error)` 这套取代异常的错误处理惯用法。slice/map/chan 拷贝的是含指针的表头,故有别名效果。方法接收者同样按值:值接收者拷贝整个接收者,指针接收者可改且避免大拷贝;注意方法集与可寻址性。闭包捕获**变量**而非值;Go 1.22 修复了循环变量捕获坑,原因和取舍见第 5 章。变参 `...T` 内部是 slice;`xs...` 展开复用底层数组。goroutine 栈从 2KB 起,连续栈可增长/收缩,与 C 固定线程栈是两个世界。
