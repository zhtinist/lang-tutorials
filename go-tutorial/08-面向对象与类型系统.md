# 类型系统:结构体、接口、方法集

Go 没有 `class`,没有继承,没有构造/析构函数,没有 `virtual` 关键字,没有访问修饰符(`public`/`private` 靠标识符首字母大小写)。但它用 **struct + 方法 + 组合 + 接口** 组合出了一套更简单、也更"结构化"的类型系统。对 C++ 程序员来说,最需要重新校准的两点是:**继承换成组合**,**接口是隐式实现的**。

## 一、方法:带接收者的函数

方法就是第一个参数(接收者 receiver)写在前面的普通函数。([回顾第06章值接收者 vs 指针接收者](06-函数与调用栈.md))

```go
type Point struct{ X, Y int }

func (p Point) Add(q Point) Point {   // 值接收者: p 是 Point 的拷贝
    return Point{p.X + q.X, p.Y + q.Y}
}
func (p *Point) Scale(f int) {        // 指针接收者: 能就地修改
    p.X *= f
    p.Y *= f
}
```

底层没有魔法:`p.Add(q)` 等价于 `Point.Add(p, q)`,receiver 就是普通传参——值接收者拷贝整个 `Point`,指针接收者拷贝一个指针。没有 C++ 那种隐藏的 `this` 指针语义差异,一切显式。

方法可以定义在任何**本包内定义的具名类型**上,不限于 struct:

```go
type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }
```

## 二、组合:嵌入(embedding)与方法提升

Go 不用继承复用,用**嵌入**:在 struct 里放一个匿名字段(只写类型不写字段名)。

```go
type Engine struct{ Power int }
func (e Engine) Start() { fmt.Println("engine start", e.Power) }

type Car struct {
    Engine       // 嵌入(匿名字段), 不是继承!
    Brand string
}

c := Car{Engine{100}, "Toyota"}
c.Start()        // 方法提升: 等价于 c.Engine.Start()
fmt.Println(c.Power) // 字段提升: 等价于 c.Engine.Power
```

**内存布局:嵌入就是把 `Engine` 整个作为 `Car` 的一个字段内联进去**,和普通字段没区别:

```
Car 在内存中:
┌──────────────┬───────────────┐
│ Engine.Power │ Brand(string) │   Engine 的字段就地展开在 Car 里
└──────────────┴───────────────┘
```

这不是 C++ 的继承(没有基类子对象的虚表指针、没有 is-a 关系)。编译器只是做了"名字转发":`c.Start()` 在编译期被改写成 `c.Engine.Start()`。**没有运行时开销,没有虚表**。

### 与 C++ 继承的对照

| 维度 | C++ 继承 | Go 嵌入 |
|------|---------|---------|
| 关系 | is-a(Car 是一种 Engine) | has-a + 转发(Car 有个 Engine) |
| 多态入口 | `virtual` + vtable | 靠接口(见下) |
| 覆盖(override) | 虚函数覆盖 | 外层定义同名方法即"遮蔽"内层 |
| 访问基类 | `Base::foo()` | `c.Engine.Start()` 显式路径 |
| 多重继承菱形问题 | 需 virtual 继承 | 嵌入多个类型,同名冲突需显式限定,编译器报歧义 |

方法遮蔽(shadowing)示例:

```go
func (c Car) Start() { fmt.Println("car:", c.Brand); c.Engine.Start() }
// 现在 c.Start() 调外层的; 想调内层显式写 c.Engine.Start()
```

注意:嵌入是**静态转发**,不是虚派发。若 `Engine.Start` 内部调用另一个方法,它调的还是 `Engine` 自己的,**不会**回调到 `Car` 覆盖的版本——这与 C++ 虚函数的"派生类覆盖被基类调用到"根本不同。需要那种回调,得用接口。

## 三、接口:隐式实现的结构化类型

接口定义一组方法签名。**只要一个类型实现了接口的全部方法,它就自动满足该接口——不需要 `implements`、不需要显式声明。**

```go
type Stringer interface {
    String() string
}

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

var s Stringer = Celsius(37.5)  // 自动满足, 无需声明"Celsius implements Stringer"
```

这叫**结构化类型(structural typing)** / 鸭子类型的静态版本。好处:接口可以为你无法修改的类型(甚至标准库类型)"事后"定义;解耦定义方和使用方。C++20 的 concept 有点像,但接口是运行时多态(带 vtable),更接近 C++ 的抽象基类 + 虚函数。

### 接口值的底层:两个字

接口值在内存里是**两个机器字**([第04章已提过](04-数据类型.md))。分两种:

**`iface`——非空接口(带方法的接口)**:

```go
type iface struct {
    tab  *itab          // 类型信息 + 方法表(相当于 vtable)
    data unsafe.Pointer  // 指向实际数据
}
```

**`eface`——空接口 `any`(即 `interface{}`)**:

```go
type eface struct {
    _type *_type         // 动态类型信息
    data  unsafe.Pointer  // 指向实际数据
}
```

`itab` 是关键,它把"接口类型"和"具体类型"配对,并缓存了方法地址表:

```go
type itab struct {
    inter *interfacetype // 接口类型
    _type *_type         // 具体类型
    hash  uint32
    fun   [1]uintptr     // 方法地址数组(变长), 相当于 vtable
}
```

```
var s Stringer = Celsius(37.5)

s(接口值, 16 字节):          itab(每个 (接口,类型) 组合一份, 全局缓存):
┌──────────┐               ┌────────────────────────┐
│ tab  ────┼──────────────▶│ inter: Stringer 类型    │
│ data ────┼───┐           │ _type: Celsius 类型     │
└──────────┘   │           │ fun[0]: &Celsius.String │◀─ 方法地址
               │           └────────────────────────┘
               ▼
        堆/栈上的 Celsius 值 37.5(装箱: 值被拷到 data 指向处)
```

**与 C++ vtable 的关键差异**:C++ 对象自己内部存一个 vptr,vtable 由类唯一确定;Go 的方法表在**接口值里**(`itab`),同一个具体类型对不同接口有不同 `itab`,`itab` 按需生成并全局缓存。调用 `s.String()` = 查 `tab.fun[0]` 得地址 → 以 `data` 为 receiver 调用,一次间接跳转,和虚调用开销相当。

**装箱**:把具体值塞进接口时,若值不是指针,通常要把它拷到堆上让 `data` 指向(逃逸,见[第03章](03-变量与内存模型.md))。所以 `var x any = 42` 可能触发一次小分配(运行时对小整数等有缓存优化)。这点像 Java/C# 的装箱,C++ 里没有对应物。

### nil 接口坑(必考)

接口值 == nil,**当且仅当 `tab`(或 `_type`)和 `data` 两个字都为 nil**。如果接口装了一个"类型非空但值为 nil 的指针",接口本身**不等于 nil**:

```go
func doErr() error {
    var p *MyError = nil   // 具体类型的 nil 指针
    return p               // 装进 error 接口: tab=*MyError, data=nil
}

func main() {
    err := doErr()
    if err != nil {        // 竟然为 true!!
        fmt.Println("非 nil:", err == nil) // 非 nil: false
    }
}
```

因为返回的接口值是 `{tab: *MyError, data: nil}`,`tab` 非 nil,所以 `err != nil`。这是新手最常见的 bug 之一。**规矩:函数返回 error 时,要么返回真正的 nil(`return nil`),不要返回一个具体类型的 nil 指针。**([第09章 error 会再强调](09-错误处理与资源管理.md))

### 类型断言与 type switch

从接口取回具体类型:

```go
var i any = "hello"
s := i.(string)         // 断言, 失败会 panic
s, ok := i.(string)     // 逗号 ok 形式, 失败 ok=false 不 panic
```

底层:比较接口里的 `_type`/`itab` 与目标类型是否一致,一致就返回 `data`(必要时解引用/拷贝)。

type switch 按动态类型分派:

```go
switch v := i.(type) {
case int:
    fmt.Println("int", v)     // v 是 int
case string:
    fmt.Println("string", v)  // v 是 string
case fmt.Stringer:
    fmt.Println(v.String())   // 也能匹配接口
default:
    fmt.Println("其他")
}
```

## 四、方法集:值还是指针能满足接口?

哪些方法"属于"一个类型,决定它能否满足接口。规则:

| 接收者定义为 | `T` 的方法集 | `*T` 的方法集 |
|-------------|-------------|--------------|
| `func (t T)` | ✓ | ✓ |
| `func (t *T)` | ✗ | ✓ |

即:**指针类型 `*T` 拥有全部方法;值类型 `T` 只拥有值接收者方法。** 因为拿到 `*T` 能自动取值,拿到 `T` 却不一定能取到稳定地址(比如 map 元素、临时值)去调指针方法。

```go
type Speaker interface{ Speak() }
type Dog struct{}
func (d *Dog) Speak() {}      // 指针接收者

var s Speaker = &Dog{}        // OK
// var s Speaker = Dog{}      // 编译错误: Dog 未实现 Speaker(Speak 在 *Dog 上)
```

实践建议:**要么全用值接收者,要么全用指针接收者,别混**。需要修改状态、结构体较大、或想让 `*T` 满足接口,就用指针接收者。

### 方法值 vs 方法表达式

```go
d := Dog{}
f := d.Speak            // 方法值: 绑定了 receiver d, 类型 func()
f()                     // 等价 d.Speak()

g := (*Dog).Speak       // 方法表达式: receiver 变成显式第一个参数
g(&d)                   // 等价 d.Speak()
```

方法值会捕获 receiver(可能导致其逃逸到堆,像闭包);方法表达式只是把方法当成普通函数,receiver 显式传。

## 五、泛型(Go 1.18+):类型参数与约束

Go 长期没有泛型,1.18 引入类型参数。语法:方括号声明类型参数,配 **约束(constraint)**。

```go
// 约束: 一个接口, 列出允许的类型集合
type Number interface {
    ~int | ~int64 | ~float64   // ~T 表示"底层类型是 T 的所有类型"
}

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

Sum([]int{1, 2, 3})        // 类型实参 int 自动推导
Sum([]float64{1.5, 2.5})
```

`comparable` 是内建约束(可用 `==`/`!=` 的类型),常用于 map 键类泛型:

```go
func Keys[K comparable, V any](m map[K]V) []K { /* ... */ }
```

约束的本质:它就是一个接口,既能列方法(要求实现某方法),也能列**类型集合**(`| ~T`)。

### 底层实现:GC shape stenciling + 字典

这是和 C++ 模板差异最大的地方。C++ 模板是**完全单态化(monomorphization)**:`vector<int>` 和 `vector<string>` 生成两份完全独立的机器码,零运行时开销但代码膨胀、编译慢。Go 走了折中路线:

**按 "GC shape" 分组生成代码 + 传字典(dictionary)。** 具体:

- 编译器按 **GC shape** 分组生成代码。当前实现的分组规则是:两个类型同属一个 shape,**当且仅当它们底层类型相同,或都是指针**。所以所有指针类型(`*int`、`*string`……)共享一份实例;而 `int` 与 `int64` **底层类型不同,即便大小都是 8 字节、都不含指针,也各自生成一份**(不共享)。
- 每次调用额外传一个隐藏的**字典**参数,里面放该具体类型的元信息(类型描述符、需要的 itab 等),供运行时查具体大小、方法地址。

```
C++ 模板(单态化):          Go 泛型(shape stenciling):
Sum<int>    → 代码副本A     Sum[int]     → int 的 shape 代码 + int 字典
Sum<int64>  → 代码副本B     Sum[int64]   → int64 的 shape 代码 + int64 字典(与 int 不共享)
Sum<*T>×N   → N 份副本      Sum[*T]×N    → 1 份指针 shape 代码 + 各自字典
```

结论:**介于单态化与装箱之间**。比 Java 的类型擦除+装箱快(无处不装箱),比 C++ 单态化省代码;但字典查找、经指针间接访问会带来一点运行时开销,某些场景不如手写具体类型或不如 C++ 模板极致。

| | C++ 模板 | Go 泛型 | Java 泛型 |
|--|---------|---------|----------|
| 机制 | 单态化 | shape stenciling + 字典 | 类型擦除 + 装箱 |
| 代码体积 | 大(每类型一份) | 中(每 shape 一份) | 小(一份) |
| 运行时开销 | 无 | 少量(字典/间接) | 装箱开销大 |
| 编译期计算 | 强(模板元编程) | 弱(只是参数化) | 无 |

### 什么时候用泛型

不要为了泛型而泛型。适合:容器/算法(`slices`、`maps` 包,见[第11章](11-标准库算法.md))、对多种数值类型做同一操作。很多场景一个 `any` + 类型断言或普通接口反而更清晰。
