# 类型系统:结构体、接口、方法集

Go 没有 `class`,没有继承,没有构造/析构函数,没有 `virtual` 关键字,没有访问修饰符(`public`/`private` 靠标识符首字母大小写)。但它用 **struct + 方法 + 组合 + 接口** 组合出了一套更简单、也更"结构化"的类型系统。**本章每一部分都先讲「怎么定义、怎么用」——不需要任何底层知识就能看懂;再讲「底层与 C 对比」——看它在内存里到底长什么样。** 对 C++ 程序员来说,最需要重新校准的两点是:**继承换成组合**,**接口是隐式实现的**。

## 先会用:struct、方法、嵌入、接口速览

不涉及任何底层原理,先看这几样东西怎么定义、怎么用:

```go
package main

import "fmt"

// 定义一个结构体: 描述一个"点"
type Point struct {
	X, Y int
}

// 给 Point 定义一个方法(带接收者的函数)
func (p Point) String() string {
	return fmt.Sprintf("(%d, %d)", p.X, p.Y)
}

// 定义一个接口: 只要求有 String() 方法
type Stringer interface {
	String() string
}

func main() {
	p := Point{X: 1, Y: 2}   // 创建并初始化
	fmt.Println(p.X, p.Y)   // 1 2  —— 访问字段
	fmt.Println(p.String()) // (1, 2)  —— 调用方法

	// Point 自动满足 Stringer 接口(不需要声明 implements)
	var s Stringer = p
	fmt.Println(s.String()) // (1, 2)
}
```

简单总结:

- `type Point struct { X, Y int }` 定义结构体,`Point{X: 1, Y: 2}` 或 `Point{1, 2}` 创建实例,`.` 访问字段。
- `func (p Point) String() string { ... }` 给类型定义方法,括号里的 `p Point` 叫**接收者(receiver)**。
- `type Stringer interface { String() string }` 定义接口,只要一个类型实现了接口里列出的所有方法,它就"自动"满足这个接口,不需要像 Java/C++ 那样显式声明 `implements`。

这些够你开始写面向对象风格的 Go 代码了。下面进入"底层"——看方法调用的本质、组合的内存布局、接口值的两个字怎么构成、以及泛型编译成什么样的代码。

## 开始深入:从"怎么定义"到"底层怎么表示"

从这一节起讲底层。Go 的类型系统看起来简单,但每一层都有明确、可预测的底层表示——没有隐藏的虚表指针、没有隐藏的 `this`。搞清这些表示,你就能预测方法调用的开销、组合和接口的内存布局、以及为什么某些类型不能赋给某个接口变量。

## 一、方法:带接收者的函数

### 怎么用

方法就是第一个参数(接收者 receiver)写在前面的普通函数。分两种:值接收者(拷贝一份操作)和指针接收者(能就地修改)。

```go
type Point struct{ X, Y int }

func (p Point) Add(q Point) Point {   // 值接收者: p 是 Point 的拷贝
	return Point{p.X + q.X, p.Y + q.Y}
}
func (p *Point) Scale(f int) {        // 指针接收者: 能就地修改
	p.X *= f
	p.Y *= f
}

p1 := Point{1, 2}
p2 := Point{3, 4}
sum := p1.Add(p2)  // {4 6}
p1.Scale(10)       // p1 变成 {10 20}, Scale 就地修改了它
```

方法可以定义在任何**本包内定义的具名类型**上,不限于 struct:

```go
type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }
```

### 底层与 C 对比

底层没有魔法:`p.Add(q)` 等价于 `Point.Add(p, q)`,receiver 就是普通传参——值接收者拷贝整个 `Point`,指针接收者拷贝一个指针。没有 C++ 那种隐藏的 `this` 指针语义差异,一切显式。([回顾第06章值接收者 vs 指针接收者](06-函数与调用栈.md))

## 二、组合:嵌入(embedding)与方法提升

### 怎么用

Go 不用继承复用,用**嵌入**:在 struct 里放一个匿名字段(只写类型不写字段名)。嵌入类型的方法和字段会被"提升"到外层,看起来像是外层自己的:

```go
type Engine struct{ Power int }
func (e Engine) Start() { fmt.Println("engine start", e.Power) }

type Car struct {
	Engine       // 嵌入(匿名字段), 不是继承!
	Brand string
}

c := Car{Engine{100}, "Toyota"}
c.Start()             // 方法提升: 等价于 c.Engine.Start()
fmt.Println(c.Power)  // 字段提升: 等价于 c.Engine.Power
```

### 底层与 C 对比

**内存布局:嵌入就是把 `Engine` 整个作为 `Car` 的一个字段内联进去**,和普通字段没区别:

```
Car 在内存中:
┌──────────────┬───────────────┐
│ Engine.Power │ Brand(string) │   Engine 的字段就地展开在 Car 里
└──────────────┴───────────────┘
```

这不是 C++ 的继承(没有基类子对象的虚表指针、没有 is-a 关系)。编译器只是做了"名字转发":`c.Start()` 在编译期被改写成 `c.Engine.Start()`。**没有运行时开销,没有虚表**。

### 与 C++ 继承的对照

| 维度 | C++ 继承 | Go 嵌入 |
|------|---------|---------|
| 关系 | is-a(Car 是一种 Engine) | has-a + 转发(Car 有个 Engine) |
| 多态入口 | `virtual` + vtable | 靠接口(见下) |
| 覆盖(override) | 虚函数覆盖 | 外层定义同名方法即"遮蔽"内层 |
| 访问基类 | `Base::foo()` | `c.Engine.Start()` 显式路径 |
| 多重继承菱形问题 | 需 virtual 继承 | 嵌入多个类型,同名冲突需显式限定,编译器报歧义 |

方法遮蔽(shadowing)示例:

```go
func (c Car) Start() { fmt.Println("car:", c.Brand); c.Engine.Start() }
// 现在 c.Start() 调外层的; 想调内层显式写 c.Engine.Start()
```

注意:嵌入是**静态转发**,不是虚派发。若 `Engine.Start` 内部调用另一个方法,它调的还是 `Engine` 自己的,**不会**回调到 `Car` 覆盖的版本——这与 C++ 虚函数的"派生类覆盖被基类调用到"根本不同。需要那种回调,得用接口。

### 可运行:方法提升、遮蔽与静态转发

```go
package main

import "fmt"

type Engine struct{ Power int }

func (e Engine) Start() { fmt.Println("engine start", e.Power) }
func (e Engine) Boot()  { e.Start() } // 永远调 Engine.Start

type Car struct {
	Engine        // 嵌入(匿名字段)
	Brand string
}

func (c Car) Start() { fmt.Println("car start", c.Brand) } // 遮蔽 Engine.Start

func main() {
	c := Car{Engine{100}, "Toyota"}
	c.Start()            // car start Toyota   (外层遮蔽了提升来的方法)
	c.Engine.Start()     // engine start 100   (显式路径调内层)
	fmt.Println(c.Power) // 100                (字段提升)
	c.Boot()             // engine start 100   (静态转发: 不是 "car start")
}
```

最后一行是关键:`Boot` 是 `Engine` 的方法,直接调 `Engine.Start`,**不会**派发到 `Car.Start`。这就是静态转发,与 C++ 虚调用相反。

## 三、接口:隐式实现的结构化类型

### 怎么用

接口定义一组方法签名。**只要一个类型实现了接口的全部方法,它就自动满足该接口——不需要 `implements`、不需要显式声明。**

```go
type Stringer interface {
    String() string
}

type Celsius float64
func (c Celsius) String() string { return fmt.Sprintf("%.1f°C", c) }

var s Stringer = Celsius(37.5)  // 自动满足, 无需声明"Celsius implements Stringer"
```

这叫**结构化类型(structural typing)** / 鸭子类型的静态版本。好处:接口可以为你无法修改的类型(甚至标准库类型)"事后"定义;解耦定义方和使用方。C++20 的 concept 有点像,但接口是运行时多态(带 vtable),更接近 C++ 的抽象基类 + 虚函数。

### 底层与 C 对比

接口值的底层:两个字。接口值在内存里是**两个机器字**([第04章已提过](04-数据类型.md))。分两种:

**`iface`——非空接口(带方法的接口)**:

```go
type iface struct {
    tab  *itab          // 类型信息 + 方法表(相当于 vtable)
    data unsafe.Pointer  // 指向实际数据
}
```

**`eface`——空接口 `any`(即 `interface{}`)**:

```go
type eface struct {
    _type *_type         // 动态类型信息
    data  unsafe.Pointer  // 指向实际数据
}
```

`itab` 是关键,它把"接口类型"和"具体类型"配对,并缓存了方法地址表:

```go
type itab struct {
    inter *interfacetype // 接口类型
    _type *_type         // 具体类型
    hash  uint32
    fun   [1]uintptr     // 方法地址数组(变长), 相当于 vtable
}
```

```
var s Stringer = Celsius(37.5)

s(接口值, 16 字节):          itab(每个 (接口,类型) 组合一份, 全局缓存):
┌──────────┐               ┌────────────────────────┐
│ tab  ────┼──────────────▶│ inter: Stringer 类型    │
│ data ────┼───┐           │ _type: Celsius 类型     │
└──────────┘   │           │ fun[0]: &Celsius.String │◀─ 方法地址
               │           └────────────────────────┘
               ▼
        堆/栈上的 Celsius 值 37.5(装箱: 值被拷到 data 指向处)
```

**与 C++ vtable 的关键差异**:C++ 对象自己内部存一个 vptr,vtable 由类唯一确定;Go 的方法表在**接口值里**(`itab`),同一个具体类型对不同接口有不同 `itab`,`itab` 按需生成并全局缓存。调用 `s.String()` = 查 `tab.fun[0]` 得地址 → 以 `data` 为 receiver 调用,一次间接跳转,和虚调用开销相当。

**装箱**:把具体值塞进接口时,若值不是指针,通常要把它拷到堆上让 `data` 指向(逃逸,见[第03章](03-变量与内存模型.md))。所以 `var x any = 42` 可能触发一次小分配(运行时对小整数等有缓存优化)。这点像 Java/C# 的装箱,C++ 里没有对应物。

### nil 接口坑(必考)

接口值 == nil,**当且仅当 `tab`(或 `_type`)和 `data` 两个字都为 nil**。如果接口装了一个"类型非空但值为 nil 的指针",接口本身**不等于 nil**:

```go
func doErr() error {
    var p *MyError = nil   // 具体类型的 nil 指针
    return p               // 装进 error 接口: tab=*MyError, data=nil
}

func main() {
    err := doErr()
    if err != nil {        // 竟然为 true!!
        fmt.Println("非 nil:", err == nil) // 非 nil: false
    }
}
```

因为返回的接口值是 `{tab: *MyError, data: nil}`,`tab` 非 nil,所以 `err != nil`。这是新手最常见的 bug 之一。**规矩:函数返回 error 时,要么返回真正的 nil(`return nil`),不要返回一个具体类型的 nil 指针。**([第09章 error 会再强调](09-错误处理与资源管理.md))

可运行:

```go
package main

import "fmt"

type MyError struct{}

func (e *MyError) Error() string { return "boom" }

func doErr() error {
	var p *MyError = nil // 具体类型的 nil 指针
	return p             // 装进 error: {type:*MyError, data:nil}
}

func returnsNil() error { return nil } // 真正的 nil 接口

func main() {
	fmt.Println(doErr() == nil)      // false  ← 坑: 接口装了非空类型
	fmt.Println(returnsNil() == nil) // true
}
```

### 类型断言与 type switch

从接口取回具体类型:

```go
var i any = "hello"
s := i.(string)         // 断言, 失败会 panic
s, ok := i.(string)     // 逗号 ok 形式, 失败 ok=false 不 panic
```

底层:比较接口里的 `_type`/`itab` 与目标类型是否一致,一致就返回 `data`(必要时解引用/拷贝)。

type switch 按动态类型分派:

```go
switch v := i.(type) {
case int:
    fmt.Println("int", v)     // v 是 int
case string:
    fmt.Println("string", v)  // v 是 string
case fmt.Stringer:
    fmt.Println(v.String())   // 也能匹配接口
default:
    fmt.Println("其他")
}
```

## 四、方法集:值还是指针能满足接口?

### 怎么用

哪些方法"属于"一个类型,决定它能否满足接口。规则:

| 接收者定义为 | `T` 的方法集 | `*T` 的方法集 |
|-------------|-------------|--------------|
| `func (t T)` | ✓ | ✓ |
| `func (t *T)` | ✗ | ✓ |

即:**指针类型 `*T` 拥有全部方法;值类型 `T` 只拥有值接收者方法。**

```go
type Speaker interface{ Speak() string }
type Dog struct{ Name string }
func (d *Dog) Speak() string { return d.Name + " says woof" } // 指针接收者

d := &Dog{Name: "Rex"}
var s Speaker = d          // OK: *Dog 满足 Speaker
fmt.Println(s.Speak())     // Rex says woof
```

实践建议:**要么全用值接收者,要么全用指针接收者,别混**。需要修改状态、结构体较大、或想让 `*T` 满足接口,就用指针接收者。

### 底层与 C 对比

规则背后的原因:因为拿到 `*T` 能自动取值,拿到 `T` 却不一定能取到稳定地址(比如 map 元素、临时值)去调指针方法。

```go
type Speaker interface{ Speak() }
type Dog struct{}
func (d *Dog) Speak() {}      // 指针接收者

var s Speaker = &Dog{}        // OK
// var s Speaker = Dog{}      // 编译错误: Dog 未实现 Speaker(Speak 在 *Dog 上)
```

### 方法值 vs 方法表达式

```go
d := Dog{}
f := d.Speak            // 方法值: 绑定了 receiver d, 类型 func()
f()                     // 等价 d.Speak()

g := (*Dog).Speak       // 方法表达式: receiver 变成显式第一个参数
g(&d)                   // 等价 d.Speak()
```

方法值会捕获 receiver(可能导致其逃逸到堆,像闭包);方法表达式只是把方法当成普通函数,receiver 显式传。

## 五、泛型(Go 1.18+):类型参数与约束

### 怎么用

Go 长期没有泛型,1.18 引入类型参数。语法:方括号声明类型参数,配 **约束(constraint)**,写一次函数就能对多种类型工作:

```go
func Sum[T int | float64](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}

Sum([]int{1, 2, 3})        // 6, 类型实参 int 自动推导
Sum([]float64{1.5, 2.5})   // 4
```

约束还可以先定义成一个具名接口,列出允许的类型集合:

```go
// 约束: 一个接口, 列出允许的类型集合
type Number interface {
    ~int | ~int64 | ~float64   // ~T 表示"底层类型是 T 的所有类型"
}

func Sum[T Number](xs []T) T {
    var total T
    for _, x := range xs {
        total += x
    }
    return total
}
```

`comparable` 是内建约束(可用 `==`/`!=` 的类型),常用于 map 键类泛型:

```go
func Keys[K comparable, V any](m map[K]V) []K { /* ... */ }
```

约束的本质:它就是一个接口,既能列方法(要求实现某方法),也能列**类型集合**(`| ~T`)。

可运行,展示 `~` 的意义(它接纳底层类型匹配的具名类型):

```go
package main

import "fmt"

type Number interface {
	~int | ~int64 | ~float64
}

func Sum[T Number](xs []T) T {
	var total T
	for _, x := range xs {
		total += x
	}
	return total
}

type Celsius float64 // 底层类型是 float64, 所以 ~float64 接纳它

func main() {
	fmt.Println(Sum([]int{1, 2, 3}))      // 6   (T 推导为 int)
	fmt.Println(Sum([]float64{1.5, 2.5})) // 4
	fmt.Println(Sum([]Celsius{1.5, 2.5})) // 4   (~float64 接纳具名类型 Celsius)
}
```

### 底层实现:GC shape stenciling + 字典

这是和 C++ 模板差异最大的地方。C++ 模板是**完全单态化(monomorphization)**:`vector<int>` 和 `vector<string>` 生成两份完全独立的机器码,零运行时开销但代码膨胀、编译慢。Go 走了折中路线:

**按 "GC shape" 分组生成代码 + 传字典(dictionary)。** 具体:

- 编译器按 **GC shape** 分组生成代码。当前实现的分组规则是:两个类型同属一个 shape,**当且仅当它们底层类型相同,或都是指针**。所以所有指针类型(`*int`、`*string`……)共享一份实例;而 `int` 与 `int64` **底层类型不同,即便大小都是 8 字节、都不含指针,也各自生成一份**(不共享)。
- 每次调用额外传一个隐藏的**字典**参数,里面放该具体类型的元信息(类型描述符、需要的 itab 等),供运行时查具体大小、方法地址。

```
C++ 模板(单态化):          Go 泛型(shape stenciling):
Sum<int>    → 代码副本A     Sum[int]     → int 的 shape 代码 + int 字典
Sum<int64>  → 代码副本B     Sum[int64]   → int64 的 shape 代码 + int64 字典(与 int 不共享)
Sum<*T>×N   → N 份副本      Sum[*T]×N    → 1 份指针 shape 代码 + 各自字典
```

结论:**介于单态化与装箱之间**。比 Java 的类型擦除+装箱快(无处不装箱),比 C++ 单态化省代码;但字典查找、经指针间接访问会带来一点运行时开销,某些场景不如手写具体类型或不如 C++ 模板极致。

| | C++ 模板 | Go 泛型 | Java 泛型 |
|--|---------|---------|----------|
| 机制 | 单态化 | shape stenciling + 字典 | 类型擦除 + 装箱 |
| 代码体积 | 大(每类型一份) | 中(每 shape 一份) | 小(一份) |
| 运行时开销 | 无 | 少量(字典/间接) | 装箱开销大 |
| 编译期计算 | 强(模板元编程) | 弱(只是参数化) | 无 |

### 什么时候用泛型

不要为了泛型而泛型。适合:容器/算法(`slices`、`maps` 包,见[第11章](11-标准库算法.md))、对多种数值类型做同一操作。很多场景一个 `any` + 类型断言或普通接口反而更清晰。

## 小结

- **用法层面**:`func (recv T) Name(...)` 定义方法,值接收者拷贝、指针接收者能就地修改;嵌入(匿名字段)让字段/方法被"提升"到外层;接口只要方法签名匹配就自动满足,不用显式声明实现;`T`/`*T` 方法集不同——指针接收者的方法只属于 `*T`;泛型用 `[T 约束]` 声明类型参数,一份代码支持多种类型。
- **底层层面**:方法调用就是把 receiver 当第一个参数传的普通函数调用,无隐藏 this;嵌入是编译期的名字转发和字段内联,零运行时开销,不是虚派发;接口值是两个机器字(`itab`/`_type` + `data`),`itab` 缓存了方法地址表相当于 vtable,但存在接口值里而非对象里;`nil` 接口的坑源于"接口值只有 tab 和 data 都为 nil 才等于 nil";泛型不是单态化也不是纯装箱,而是按 GC shape 分组生成代码 + 传字典,介于两者之间。
