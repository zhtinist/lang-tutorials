# 运行模型:编译、链接、运行时

Go 程序从源码到运行,要经过编译和运行时启动两步——比 C 少了独立的动态链接阶段(默认不依赖外部 .so),却多了一个必须理解的内置运行时。本章把 Go 二进制逐层拆开,处处与 C 的编译-链接-运行模型对照,并回答两个问题:这个二进制里到底多装了什么,以及 Go 为什么要这样设计。

## 一、写一个程序:`go run` 与 `go build`

```go
package main

import "fmt"

func main() {
	fmt.Println("hello, world")
}
```

`package main` 声明这是可执行程序(而不是被别人 import 的库);`import "fmt"` 引入标准库里管格式化输出的包;`func main()` 是程序入口,作用等价于 C 的 `int main(void)`,区别是 Go 的 `main` 不接收参数、不返回值——命令行参数和退出码分别由 `os.Args`、`os.Exit` 处理,见[第 2 章](02-基础语法.md)。

`go run` 编译并立即执行,像跑脚本一样:

```bash
$ go run hello.go
hello, world
```

`go build` 产出真正的可执行文件,对应 C 的 `gcc -o hello hello.c`:

```bash
$ go build -o hello hello.go
$ ./hello
hello, world
```

注意 `go run` 并非解释执行——它先把源码编译成原生机器码,再把二进制放到临时目录里执行、跑完删除。它给你的是"脚本般的体验",不是"脚本的执行方式"。

并发的语法入口只有一个关键字:把 `go` 放在函数调用前,这次调用就被交给运行时异步调度:

```go
package main

import (
	"fmt"
	"time"
)

func sayHi() {
	fmt.Println("你好,来自另一个 goroutine")
}

func main() {
	go sayHi()                        // 启动一个 goroutine,不等待它跑完就往下走
	time.Sleep(time.Millisecond * 10) // 等一下,让它有机会执行
	fmt.Println("main 结束")
}
```

运行结果:

```
你好,来自另一个 goroutine
main 结束
```

**为什么并发原语是语言关键字,而不是像 C 的 pthread 那样留给标准库**:C 的并发靠库函数——线程创建、互斥锁全是普通调用,编译器不知道两段代码可能并发执行,语言本身对"并发"一无所知,也就无法在语言层面做任何检查或保证。Go 的设计者(Pike、Thompson 等人在 Google 内部用 C++ 构建高并发服务器多年后)判断,网络服务、多核 CPU 是 Go 诞生时(2007 年前后)的默认场景而非特例,于是把"启动"(`go`)和"通信"(`channel`/`<-`)做成语言关键字与内建类型,而不是一个可以被替换掉的库。代价是你无法换一套调度模型——`go` 关键字背后永远是 Go 自己的 GMP 调度器,没有"选用另一个并发库"的自由;换来的是编译器和运行时对并发的深度支持:运行时精确知道每个 goroutine 栈上哪里是指针(第 6 章的可增长栈依赖这一点),竞态检测器(race detector)能统一理解所有 goroutine 间的内存访问。如果并发只是一个库,这种语言级的统一视图做不到。

项目变大、需要声明依赖时,用 `go.mod` 声明一个模块(单文件 `go run`/`go build` 不需要这一步):

```bash
$ go mod init example.com/hello
go: creating new go.mod: module example.com/hello
```

## 二、二进制的构成:代码 + 运行时 + 标准库,静态打包

作为 C/C++ 程序员,你对"一个程序如何变成能跑的东西"有一套清晰的模型:预处理 → 编译成 `.o` → 链接(静态或动态)→ 得到 ELF/Mach-O 可执行文件 → 由内核加载、由 libc 的 `_start` 引导进 `main`。Go 的模型有相同之处,也有不同之处。相同的是:最终产物是一个原生机器码可执行文件,没有虚拟机、没有字节码解释器。不同的是:这个二进制里额外静态链接进了一个运行时(runtime),它在你的 `main` 之前就已经跑起来了。

| 维度 | C/C++ (gcc/clang) | Go (gc 工具链) | JVM (Java) |
|------|-------------------|----------------|------------|
| 最终产物 | 原生机器码 | 原生机器码 | 字节码 `.class`/`.jar` |
| 执行方式 | 直接被 CPU 执行 | 直接被 CPU 执行 | JVM 解释 + JIT |
| 是否有运行时 | 极薄(crt0 + libc,通常动态链接) | **有,较厚,且默认静态链入** | 有,独立安装的 JVM |
| 默认链接方式 | 动态链接 libc | **静态**(Linux+纯 Go 时不依赖 libc;macOS 仍链 libSystem) | 不适用 |
| 部署 | 需目标机有匹配的 .so | **单个自足文件,拷过去就能跑** | 需目标机装 JRE |
| GC | 无(手动 `malloc/free`) | 有(并发三色标记) | 有 |
| 线程模型 | 1:1 映射 OS 线程 | M:N,goroutine 复用 OS 线程 | 平台线程 / 虚拟线程 |

关键心智模型:**Go 处在 C 和 JVM 之间**。它像 C 一样编译成直接被 CPU 执行的机器码(无解释、无 JIT 预热),但又像 JVM 一样自带一个负责调度、GC、内存分配的运行时。区别在于这个运行时不是一个外部进程,而是**被静态链接进你的二进制,作为普通函数和代码段存在**。

```
   C 程序的地址空间                 Go 程序的地址空间
 ┌────────────────────┐         ┌────────────────────┐
 │  你的 .text        │         │  你的 .text        │
 │  libc (常动态)     │         │  runtime .text     │  ← 调度器/GC/分配器
 │  crt0 引导         │         │  标准库 .text      │  (都静态编进来)
 ├────────────────────┤         ├────────────────────┤
 │  堆 (malloc/free)  │         │  堆 (由 runtime 管) │
 │  单个/多个线程栈   │         │  多个 goroutine 栈 │  ← 每个约 2KB 起,可增长
 └────────────────────┘         └────────────────────┘
```

## 三、静态链接:部署简单性与体积的取舍

前面已经用过 `go build`。对 C 程序员更值得深挖的是它**默认静态链接**这件事(准确说主要指 **Linux + `CGO_ENABLED=0`** 的情形)。写一个纯 Go 的 `hello`,在 Linux 上 `ldd hello` 会告诉你 `not a dynamic executable`——它没有链接 libc,连 `libc.so.6` 都不需要。(**macOS 是例外**:苹果不支持静态链接 libSystem,macOS 上的 Go 二进制总会动态链接 `/usr/lib/libSystem.B.dylib`,用 `otool -L` 可见。)

**为什么默认选静态链接,而不是像 C 那样默认动态链接**:动态链接的历史动机是省资源——多个进程共享同一份内存里的 libc,且 libc 打安全补丁时不需要重新编译每个应用。这套好处的前提是目标机器上恰好有版本兼容的 .so,而这恰恰是分布式部署里最麻烦的部分(依赖地狱、`GLIBC_2.34 not found`)。Go 诞生的场景是 Google 内部成千上万台机器跑同一批服务、以及后来的容器化部署,"这个二进制能不能在任意一台同架构机器上直接跑起来"远比"省下几十 KB 磁盘"重要。于是 Go 把取舍拨向另一端:接受更大的二进制体积(把用到的运行时和标准库都编进去,一个 `hello world` 通常 1~2 MB,C 的动态链接版本可能只有几十 KB,但依赖系统里几 MB 的 libc),换来"编译一次、拷贝到任何同架构机器就能跑"的确定性——极端情况下 `FROM scratch` 的空 Docker 镜像里只放这一个文件就能跑。这不是普适更优的方案,而是针对特定部署场景的取舍:需要给 libc 打补丁而不重新编译应用的场景(比如某些长期运行、无法轻易重新部署的系统),静态链接反而是劣势。

### 什么时候 Go 会动态链接?

不是绝对静态。两种常见情况会引入动态链接:

1. **cgo**:一旦用了 `import "C"` 调用 C 库(如某些 `net`、`os/user` 的解析路径,或第三方 C 依赖),就会链接 libc,产生动态依赖。可用 `CGO_ENABLED=0` 强制关闭 cgo,回到纯静态。
2. 显式 `-linkmode=external` 或链接系统库。

```bash
CGO_ENABLED=0 go build -o hello   # 强制纯静态,常用于构建 Docker 镜像
```

## 四、运行时里到底有什么

C 的"运行时"薄到几乎只有 crt0 的引导和 libc 提供的库函数;控制权很快就交给你的 `main`,之后除非你调库,否则没有"背景魔法"。Go 不一样:在你的 `main.main` 被调用之前,运行时已经初始化了一整套子系统,并且在你的程序运行期间**持续在后台工作**。主要三块:

### 1. Goroutine 调度器(GMP 模型)

Go 不把 goroutine 直接映射到 OS 线程(那是 C 的 pthread 1:1 模型),而是 **M:N 调度**:成千上万个 goroutine 被复用到少量 OS 线程上。三个核心抽象:

```
G (goroutine)  ── 一个 goroutine,含自己的栈、PC、状态。极轻量。
M (machine)    ── 一个 OS 线程,真正在 CPU 上跑代码。
P (processor)  ── 逻辑处理器/调度上下文,数量默认 = GOMAXPROCS(通常=CPU 核数)。
                  P 持有一个本地可运行 G 队列,M 必须绑定 P 才能执行 G。

     M0 ── P0 ── [G,G,G] 本地队列
     M1 ── P1 ── [G,G]
                 [G,G,G,...] 全局队列(P 队列空了来这里偷)
```

一个 goroutine 阻塞在系统调用时,运行时会把 M 和 P 解绑,让另一个 M 接管这个 P 继续跑其它 G——所以少数线程能撑起海量并发。细节见[第 10 章「并发模型」](10-并发模型.md)。相比之下,C 里开一个线程要几 MB 栈 + 内核调度开销,开几千个就吃力;Go 开几十万 goroutine 稀松平常。

前面 `go sayHi()` 只是语法;这一点可以用 `runtime` 包更直接地看到调度器的存在。把 P 的数量固定为 2,再开 1000 个 goroutine,它们全都复用在这 2 个逻辑处理器上:

```go
package main

import (
	"fmt"
	"runtime"
)

func main() {
	runtime.GOMAXPROCS(2)              // 把逻辑处理器 P 固定为 2(结果确定)
	fmt.Println(runtime.GOMAXPROCS(0)) // 2   (传 0 = 只查询,不修改)

	fmt.Println(runtime.NumGoroutine()) // 1   (目前只有 main 一个 goroutine)

	ch := make(chan struct{})
	for i := 0; i < 1000; i++ {
		go func() { <-ch }() // 1000 个 goroutine,全部阻塞在 ch 上等待
	}
	for runtime.NumGoroutine() < 1001 { // 等它们全部被创建出来
		runtime.Gosched()
	}
	fmt.Println(runtime.NumGoroutine()) // 1001 (main + 1000),仅跑在 2 个 OS 线程上

	close(ch) // 一次性放行
}
```

运行输出:

```
2
1
1001
```

1001 个 goroutine 挤在 2 个 P 上安然共存——换成 C 的 1:1 线程,开 1000 个线程光栈就要好几 GB。这就是 M:N 调度的直观体现。

### 2. 垃圾回收器(GC)

C 里你自己 `malloc`/`free`,悬垂指针和内存泄漏全靠自律。Go 有一个**并发、三色标记-清除**的 GC 在后台运行,与你的代码并发执行,STW(stop-the-world)暂停通常在亚毫秒级。这意味着你几乎不用手动释放内存——但你要理解**逃逸分析**(变量为何进堆),否则会写出对 GC 不友好的代码。逃逸分析见[第 3 章](03-变量与内存模型.md),GC 细节见[第 9 章](09-错误处理与资源管理.md)。

### 3. 内存分配器(tcmalloc 风格)

Go 的分配器脱胎于 Google 的 tcmalloc,采用**多级缓存**结构,核心是尽量无锁地分配小对象:

```
mcache   ── 每个 P 一个,线程本地,分配小对象无需加锁(类似 tcmalloc 的 thread cache)
   │ 不够了向下要
mcentral ── 按大小规格(size class)组织的中心缓存,跨 P 共享,需加锁
   │ 不够了向下要
mheap    ── 全局堆,管理从 OS 申请来的大块内存(span/arena)
```

对象按大小被归入约 70 个 **size class**,小对象直接从 P 本地的 `mcache` 拿,不用抢全局锁——这是 Go 高并发下分配性能好的原因之一。相比 C 的 `malloc`(通常全局锁或 arena 锁),Go 的 per-P 缓存天然对多核友好。

## 五、为什么 Go 编译这么快

C++ 大工程编译慢到可以去泡咖啡,原因你很清楚:头文件被反复文本包含、模板在每个编译单元里实例化、预处理器展开量巨大、链接期还要做大量符号解析。Go 在设计上刻意规避了这些:

1. **没有头文件、没有预处理器**。依赖信息直接编码在包(package)里,编译一个包时只需读取它**直接导入的包的导出信息**(编译好的对象里带有紧凑的 export data),而**不需要**递归展开整棵依赖树的源码。C 里 `#include <a.h>` 而 a.h 又 include 一堆,每个 .c 都重新解析一遍。
2. **导入是包级、无环**。Go 规定包依赖图不能有循环,且导入的是编译产物而非源码,编译器读一份摘要即可。
3. **语法为快速解析而设计**。Go 的语法基本可以不看语义就单遍解析(见[第 2 章](02-基础语法.md)讲的自动分号插入);没有 C++ 那种"要靠符号表才能判断 `a * b` 是乘法还是声明指针"的歧义。
4. **并行编译**:不同包之间可并行编译。

代价:编译快是以"较厚的运行时 + 较大的二进制 + GC 开销"换来的,还有语言表达力上的取舍(早期没泛型等,详见第八节)。

## 六、常用命令:`go run` / `build` / `mod`

前面已经用过 `go run`、`go build`、`go mod init`。完整的常用命令:

```bash
go run main.go        # 编译到临时目录并立即运行(不留下二进制),用于快速试验
go build ./...        # 编译当前模块所有包,产出可执行文件(有 main 包时)
go install ./cmd/foo  # 编译并把二进制装到 $GOBIN(默认 ~/go/bin)
go test ./...         # 编译并运行测试
go vet ./...          # 静态检查可疑代码
```

### go modules:依赖管理

Go 用 **modules** 管理依赖(取代了老的 GOPATH 模式):

```bash
go mod init example.com/myapp   # 生成 go.mod,声明模块路径
go get github.com/foo/bar@v1.2.3 # 添加/升级依赖,写入 go.mod 和 go.sum
go mod tidy                      # 增删依赖使之与源码 import 一致
```

- `go.mod`:声明模块路径、Go 版本、依赖及其版本。
- `go.sum`:记录依赖的哈希校验和,保证可重现构建(类似锁文件)。

依赖会被下载到本地缓存(`$GOPATH/pkg/mod`),构建时静态编进你的二进制。**没有 C 那种"运行时去系统找 .so"的问题**——依赖在编译期就固化了。

## 七、交叉编译:改两个环境变量而已

C 的交叉编译是出了名的折磨:你得有目标平台的交叉工具链(交叉 gcc)、目标平台的 libc、sysroot……Go 因为默认不依赖 libc、且工具链自带所有目标平台的运行时,交叉编译只是设两个环境变量:

```bash
# 在 macOS/arm64 上,为 Linux/amd64 构建:
GOOS=linux GOARCH=amd64 go build -o hello-linux-amd64

# 为 Windows 构建:
GOOS=windows GOARCH=amd64 go build -o hello.exe

# 为树莓派 (Linux/arm64) 构建:
GOOS=linux GOARCH=arm64 go build -o hello-pi

go tool dist list   # 列出所有支持的 GOOS/GOARCH 组合
```

- `GOOS`:目标操作系统(`linux`/`darwin`/`windows`/`freebsd`...)。
- `GOARCH`:目标 CPU 架构(`amd64`/`arm64`/`386`/`riscv64`...)。

**前提是纯 Go(`CGO_ENABLED=0`)**。一旦用了 cgo,交叉编译就会退化回"需要目标平台 C 交叉工具链"的 C 世界痛苦模式。

## 八、极简主义哲学:为什么去掉异常、早期泛型、类继承

Go 从 C++/Java 的常见特性集里明确砍掉了三样东西:异常(exception)、(早期版本)泛型、类继承。这不是能力不足,而是有意为之的设计选择,根源是 Pike、Griesemer、Thompson 等设计者在 Google 内部观察到的真实工程问题:一个数以千计工程师的团队,代码要被从未谋面的人反复阅读、修改、维护数十年,C++ 积累的语言特性组合常在这种规模下变成负担而非助力。

- **没有异常**:C++/Java 的异常允许控制流在任意一层被隐式地跳出很多层调用栈——读一段代码时,仅凭函数签名无法判断它会不会抛、从哪里抛、谁负责接住。Go 让错误成为普通返回值(`(result, error)`,详见[第 9 章](09-错误处理与资源管理.md)),错误处理路径和正常路径写在同一处、看得见的地方,"这个函数可能出错"是签名的一部分而非深埋文档里的隐藏契约。代价是啰嗦——每次调用后都要写 `if err != nil`,没有异常那种"一路往上抛、顶层统一 catch"的省事,换来的是**没有隐式的非局部控制流跳转**,这也是 Go 之所以没有 `try/catch/finally` 而是用 `defer` 表达清理逻辑的根本原因(见[第 6 章](06-函数与调用栈.md))。

- **早期没有泛型(Go 1.0–1.17)**:C++ 模板给了巨大的表达力(编译期特化、SFINAE),代价是编译错误信息出了名的难读(模板实例化栈动辄几百行)、编译速度被拖慢、重度使用模板的代码需要专门训练才能读懂。Go 最初判断 interface(鸭子类型 + 运行时多态)已经覆盖了大部分需要"多态"的场景,愿意牺牲"编译期单态化的性能"和"容器类型安全"换取更简单的语言和更快的编译。这个判断后来被证明不够彻底——没有泛型导致大量样板代码(每种容器手写一份,或退化成 `interface{}` 丢失类型安全),Go 1.18(2022)加入了类型参数,但设计依然克制:没有特化、没有模板元编程。这本身就是"表达力 vs 简单性"这条取舍线被重新校准、而非推翻的例子。

- **没有类继承**:C++ 的类继承(尤其多继承)带来菱形继承问题、脆弱基类问题(基类一处看似无害的改动可能悄悄破坏所有子类的假设)。Go 没有类、没有继承,只有 struct(数据)+ interface(契约)+ 组合(embedding,见[第 4 章](04-数据类型.md))。这是"组合优于继承"这条软件工程共识被直接写进语言——你无法继承实现,只能显式地把一个类型嵌入另一个类型,依赖关系永远是看得见的字段,而不是隐藏在虚函数表里的父子链条。

这三处删减背后是同一条原则:**读代码的时间远多于写代码,因此主动放弃部分表达力,换取任何人都能在合理时间内看懂陌生代码这一长期可维护性**。语言特性彼此组合带来的复杂度是超线性增长的——每加一个特性,它和已有特性的交互方式就要多想一层;砍掉不必要的特性,理解一段陌生代码所需的心智负担才可控。

## 九、与 C 的最终对照:单一自足二进制

**Go 追求的是"一次编译,得到一个自包含、跨机器可拷贝即运行的原生二进制"**。

- 对比 C/C++:你少了动态库版本地狱(`GLIBC_2.34 not found` 之类),少了部署时凑齐 .so 的麻烦;代价是二进制更大、且带一个你无法完全关掉的运行时(GC/调度器)。
- 对比 JVM:你不需要在目标机装 JRE,启动无 JIT 预热(冷启动快),但你放弃了 JVM 那种运行时深度自省和 JIT 峰值性能。

理解了"二进制里藏着一个一直在跑的运行时",后面章节里许多行为就有了着落:goroutine 为何便宜(调度器 + 可增长小栈)、变量为何有的进堆(逃逸分析 + GC)、slice/map 为何不用你释放。

## 小结

- **用法层面**:`go run file.go` 编译并立即运行;`go build -o out file.go` 产出可执行文件;`go` 关键字放函数调用前即可启动一个后台执行的 goroutine;多文件/带依赖的项目用 `go mod init` 声明模块;`GOOS=xxx GOARCH=xxx go build` 就能交叉编译。
- **底层层面**:Go 二进制 = 你的代码 + 运行时(调度器 + GC + 分配器)+ 用到的标准库,静态打包成一个原生机器码文件;默认不依赖 libc(cgo 除外),这是"部署简单性"换"二进制体积"的取舍;GMP 模型让海量 goroutine 复用少量 OS 线程,这也是并发被做成语言关键字而非库函数的直接收益;并发三色标记 GC 在后台运行;分配器采用 mcache/mcentral/mheap 多级缓存;没有头文件和模板实例化让编译远快于 C++;而去掉异常、早期不上泛型、去掉类继承,都是"极简主义换长期可维护性"这同一条哲学的体现。
