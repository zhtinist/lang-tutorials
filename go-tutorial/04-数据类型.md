# 数据类型

上一章讲清了内存语义(值拷贝、指针、逃逸)。这一章把 Go 的类型系统逐个摊开,重点仍然是**底层内存布局**,并处处与 C 对照:哪些是定长的、int 到底多宽、struct 怎么对齐、string 为什么不是 `char*`、interface 值到底是什么。

## 定长数值类型

Go 的数值类型宽度是**语言规范写死的**,不像 C 里 `int`/`long` 的宽度随平台和编译器漂移。这对写跨平台、写协议的人是巨大的确定性。

| 类型 | 宽度 | 说明 |
|---|---|---|
| `int8` `int16` `int32` `int64` | 明确 | 有符号,二进制补码 |
| `uint8` `uint16` `uint32` `uint64` | 明确 | 无符号 |
| `float32` `float64` | 4 / 8 字节 | IEEE-754,对应 C 的 `float`/`double` |
| `complex64` `complex128` | 8 / 16 字节 | 内建复数,C 里要靠库 |
| `int` `uint` | **平台字长** | 32 位平台 4 字节,64 位平台 8 字节 |
| `uintptr` | 平台字长 | 能容纳指针的整数,配合 unsafe 使用 |

关键点对照 C:

- **`int` 是平台字长**(64 位机上是 8 字节),类似 C 的 `intptr_t` 的宽度,但语义是普通整数。日常计数、循环索引用 `int` 即可。
- **没有隐式整型提升,也没有隐式类型转换**。C 里 `int + long`、`int + double` 会自动提升;Go 里**不同类型不能直接混用**,必须显式转换:

```go
var a int32 = 1
var b int64 = 2
// c := a + b        // 编译错误:mismatched types int32 and int64
c := int64(a) + b    // 必须显式转换
var f float64 = float64(a) // int32 → float64 也要显式
```

这种"啰嗦"是故意的:C 的隐式转换和整型提升是无数溢出、截断、符号 bug 的温床,Go 直接从语言层面禁掉。

### rune 与 byte

两个重要别名,底层就是整数,但语义命名让代码更清楚:

| 别名 | 等价于 | 用途 |
|---|---|---|
| `byte` | `uint8` | 一个字节,处理二进制/ASCII |
| `rune` | `int32` | 一个 Unicode 码点(code point) |

`rune` 是 `int32` 而不是一字节,因为一个 Unicode 码点最大需要 21 位。遍历字符串时理解这点很关键(下文 string 一节详述)。

```go
var b byte = 'A'   // 65,占 1 字节
var r rune = '中'  // 码点 20013;rune 即 int32,变量占 4 字节——这是类型宽度,非 UTF-8 长度('中' 的 UTF-8 是 3 字节,见第05章)
fmt.Println(b, r)  // 65 20013
```

## 无类型常量与隐式转换规则

既然 Go 禁止隐式类型转换,为什么 `var x float64 = 3` 这种写法能通过?因为 Go 有一套独特的**无类型常量(untyped constant)**机制。

字面量 `3`、`3.14`、`'A'`、`true`、`"hi"` 在获得确定类型之前是"无类型"的,它们以任意精度存在于编译期,只在被赋值或参与运算需要具体类型时,才"默认化"或按上下文转成对应类型。这让常量能自然地跨类型使用,而运行期变量不行:

```go
const k = 3          // 无类型整型常量
var a int32 = k      // OK,k 适配为 int32
var b float64 = k    // OK,同一个 k 适配为 float64
var c float64 = 3.0  // OK
// var d int32 = a + b // 若 a,b 是不同类型变量则错;但常量不受此限
```

无类型常量还能保持超高精度,直到落地:

```go
const big = 1 << 62      // 编译期任意精度,合法
const huge = 1 << 100    // 常量本身合法(任意精度)
// var x int64 = huge    // 报错:溢出 int64,落地时才检查
```

每个无类型常量有个**默认类型**,当没有上下文强制时使用:整数默认 `int`,浮点默认 `float64`,字符默认 `rune`,布尔 `bool`,字符串 `string`。

### const 与 iota

`const` 声明编译期常量。`iota` 是一个在每个 const 块里从 0 开始、逐行自增的计数器,专门用来生成枚举:

```go
type Weekday int

const (
	Sunday Weekday = iota // 0
	Monday                // 1(表达式沿用上一行,iota 自增)
	Tuesday               // 2
	Wednesday             // 3
)

// 位标志:iota 配合移位
const (
	FlagRead    = 1 << iota // 1 << 0 = 1
	FlagWrite               // 1 << 1 = 2
	FlagExecute             // 1 << 2 = 4
)
```

比起 C 的 `enum`,`iota` 更灵活(能做移位、跳值、表达式),但也仅仅是编译期常量,不像 C++ `enum class` 那样自带类型安全——类型安全靠你像上面那样定义 `type Weekday int` 来获得。

## array 是值,slice 不是

再强调一次(因为 C 程序员太容易踩):**数组是值类型,定长,赋值和传参拷贝整个数组**;slice 是含指针的表头。

```go
func modify(a [3]int)  { a[0] = 99 } // 拷贝整个数组,改副本
func modifyS(s []int)  { s[0] = 99 } // 拷贝表头,共享底层数组

func main() {
	arr := [3]int{1, 2, 3}
	modify(arr)
	fmt.Println(arr[0]) // 1 —— 没变

	sl := []int{1, 2, 3}
	modifyS(sl)
	fmt.Println(sl[0]) // 99 —— 变了
}
```

| | array `[N]T` | slice `[]T` |
|---|---|---|
| 长度 | 类型的一部分,编译期固定 | 运行时可变 |
| 赋值/传参 | 拷贝**全部元素** | 拷贝 3 字长表头,共享底层 |
| 内存表示 | 连续的 N 个元素 | `{ptr, len, cap}` 表头 |
| 类比 C | `int a[3]`(但传参不退化) | `struct{int*; int; int;}` |

slice 的展开细节见 [07-复合数据结构.md](07-复合数据结构.md);这里只需记住数组是"整块值",slice 是"指针表头"。

## struct:内存布局、对齐与 padding

struct 的字段在内存里**按声明顺序连续排列**,和 C 的 `struct` 完全一致——同样遵守**内存对齐**规则,同样会插入 **padding**。C 程序员的对齐知识在这里 100% 复用。

对齐规则:每个类型有对齐值(通常等于其大小),字段的偏移必须是其对齐值的倍数;整个 struct 的大小必须是其最大字段对齐值的倍数(便于数组中连续排列)。

用 `unsafe.Sizeof` / `unsafe.Alignof` / `unsafe.Offsetof` 可以精确验证:

```go
package main

import (
	"fmt"
	"unsafe"
)

// 字段顺序不佳:bool, int64, bool
type Bad struct {
	a bool  // 偏移 0,占 1
	// 7 字节 padding(为了让 b 对齐到 8)
	b int64 // 偏移 8,占 8
	c bool  // 偏移 16,占 1
	// 7 字节 padding(整体补到 8 的倍数)
}

// 重排后:int64, bool, bool
type Good struct {
	b int64 // 偏移 0,占 8
	a bool  // 偏移 8,占 1
	c bool  // 偏移 9,占 1
	// 6 字节 padding
}

func main() {
	fmt.Println(unsafe.Sizeof(Bad{}))  // 24
	fmt.Println(unsafe.Sizeof(Good{})) // 16
	fmt.Println(unsafe.Alignof(int64(0)), unsafe.Offsetof(Bad{}.b))
}
```

内存图对照(每格 1 字节):

```
Bad (24 字节):
偏移 0        8               16       24
   [a][×××××××][b b b b b b b b][c][×××××××]
    1B   7pad        8B          1B   7pad

Good (16 字节):
偏移 0               8   9        16
   [b b b b b b b b][a][c][××××××]
        8B          1B 1B   6pad
```

**仅仅调整字段顺序,就从 24 字节省到 16 字节**——把大对齐的字段放前面、小字段聚在一起,减少 padding。这和 C 里的结构体瘦身技巧一模一样。在大量分配同种 struct 时(比如百万条记录)这种优化很值。

> 提示:Go 不提供 `#pragma pack`。字段顺序是你唯一能控制布局的手段(除非上 unsafe)。

### 匿名字段与嵌入

Go 用**嵌入(embedding)**替代 C++ 的继承。把一个类型不带字段名地写进 struct,就是嵌入;外层类型自动"提升"内层的字段和方法,可以直接访问:

```go
type Base struct {
	ID   int
	Name string
}

func (b Base) Hello() string { return "hi " + b.Name }

type User struct {
	Base        // 嵌入,匿名字段
	Email string
}

func main() {
	u := User{Base: Base{ID: 1, Name: "Alice"}, Email: "a@x.com"}
	fmt.Println(u.Name)    // 字段提升:等价 u.Base.Name
	fmt.Println(u.Hello()) // 方法提升:等价 u.Base.Hello()
}
```

底层上,嵌入**没有任何魔法**:`Base` 就是 `User` 里一个名叫 `Base` 的普通字段,内存里连续排布,`u.Name` 只是 `u.Base.Name` 的语法糖。这是**组合**,不是继承——没有虚函数表意义上的父子关系,没有 C++ 那种基类子对象+vptr 的开销。方法分发的多态由 interface 承担(见 [08-面向对象与类型系统.md](08-面向对象与类型系统.md))。

## string:只读的 (ptr, len) 头

C 程序员对 string 的最大误解就是把它当 `char*`。**Go 的 string 不是以 `\0` 结尾的字符指针,而是一个两字长的只读表头**:

```go
type stringHeader struct {
	Data uintptr // 指向底层字节数组
	Len  int     // 字节长度(不含结尾 0,也没有结尾 0)
}
```

要点:

- 底层是一段 **UTF-8 编码的字节**,不是码点数组。`len(s)` 返回**字节数**,不是字符数。
- **不可变**。字符串一旦创建其底层字节只读,不能 `s[i] = 'x'`。这让字符串拷贝、切分极廉价——共享底层字节即可,无需担心被改。
- `s := "hello"; t := s` 拷贝的是 16 字节表头,共享同一底层字节,安全(因为只读)。

```go
s := "héllo"       // é 是 2 字节 UTF-8
fmt.Println(len(s)) // 6,不是 5(字节数)

// 按字节索引
fmt.Println(s[0])   // 104,byte,'h'

// 按码点遍历:range 自动 UTF-8 解码,i 是字节偏移,r 是 rune
for i, r := range s {
	fmt.Printf("%d:%c ", i, r) // 0:h 1:é 3:l 4:l 5:o(注意偏移跳了)
}
fmt.Println(len([]rune(s))) // 5,真正的字符数
```

### string 与 []byte 互转会拷贝

因为 string 只读而 `[]byte` 可写,二者互转**必须复制底层字节**,不能共享(否则就能通过 []byte 改动"不可变"的字符串了):

```go
b := []byte("hello") // 分配新数组,拷贝 5 字节
s := string(b)       // 又拷贝一次
```

在热点路径上频繁 `string ↔ []byte` 转换会带来分配和拷贝开销,这是常见性能陷阱。(编译器对某些局部场景有优化,如 `map[string(b)]` 查找不会真的分配字符串;更多见后续章节。)对照 C:C 里 `char*` 和字符串是一回事,强转不拷贝,但你要自己管理可变性和越界——Go 用"拷贝一次"换来了不可变的安全性。

## interface:(type, data) 双字

interface 是 Go 类型系统的核心,它的运行时表示是一个**两字长的结构**:一个指向类型信息的指针,一个指向数据的指针。

```go
// 非空接口(带方法)的运行时布局,概念上:
type iface struct {
	tab  *itab          // 类型 + 方法表(类似 C++ vtable + 类型元数据)
	data unsafe.Pointer // 指向实际数据
}
// 空接口 interface{} / any:
type eface struct {
	_type *_type         // 动态类型信息
	data  unsafe.Pointer // 指向实际数据
}
```

要点对 C++ 程序员:

- interface 值 = **(动态类型, 数据指针)** 双字。`itab` 里含方法表,方法调用通过它间接分发——功能上类似虚函数表,但**vtable 不在对象里**,而在 interface 值里。普通 struct 没有 vptr,不占额外空间;只有把它装进 interface 时才带上类型信息。
- 把一个值赋给 interface 时,如果值不是指针,**通常会把值拷贝到堆上**(逃逸),`data` 指向那份拷贝。所以 `var x any = bigStruct` 可能触发一次堆分配和拷贝。
- `nil` interface 是**双字都为 nil**。有个经典坑:把一个值为 nil 的具体指针装进 interface,interface 的 `_type` 非 nil,于是 `iface != nil` 为真——"接口不是 nil 但里面是 nil 指针"。

```go
var i interface{} = 42
fmt.Println(unsafe.Sizeof(i)) // 16(64 位:两个字)

// 类型断言:取回动态类型
n := i.(int)      // 断言成功返回 42
m, ok := i.(string) // ok=false,不 panic
fmt.Println(n, m, ok)
```

interface 的方法集规则、类型断言、type switch、与嵌入的配合等,在 [08-面向对象与类型系统.md](08-面向对象与类型系统.md) 系统展开。这里先建立"interface 值是 (type, data) 双字"这个底层图景。

### 亲手量一量:各类型的"字长"

前面反复说 string 是双字、slice 是三字、interface 是双字。`unsafe.Sizeof` 只算**表头本身**(不含底层数据),正好能实测这些论断:

```go
package main

import (
	"fmt"
	"unsafe"
)

func main() {
	var p *int
	var s string = "hi"
	var sl []int = []int{1, 2, 3}
	var i interface{} = 42

	fmt.Println(unsafe.Sizeof(p))  // 8,指针:一个机器字
	fmt.Println(unsafe.Sizeof(s))  // 16,string = (ptr, len) 双字
	fmt.Println(unsafe.Sizeof(sl)) // 24,slice = (ptr, len, cap) 三字
	fmt.Println(unsafe.Sizeof(i))  // 16,interface = (type, data) 双字
}
```

注意 `Sizeof(sl)` 恒为 24,和底层数组有 3 个还是 300 万个元素无关——它量的是表头,不是数据。

## 小结:内存布局速查

| 类型 | 底层表示 | 拷贝时复制什么 |
|---|---|---|
| 数值 / bool | 定长原始字节 | 全部数据 |
| array `[N]T` | 连续 N 个元素 | 全部元素 |
| struct | 字段按序 + padding | 全部字段(含嵌入) |
| 指针 `*T` | 一个机器字 | 地址(共享目标) |
| string | `(ptr, len)` 只读双字 | 表头(共享只读字节) |
| slice | `(ptr, len, cap)` 三字 | 表头(共享底层数组) |
| map / chan | 指向运行时结构的指针 | 指针(共享同一结构) |
| interface | `(type, data)` 双字 | 双字(data 指向的数据视情况) |
