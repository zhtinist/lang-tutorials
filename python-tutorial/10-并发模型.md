# 并发模型:GIL、线程、进程与协程

本章讲 `threading`(线程)、`multiprocessing`(进程)、`asyncio`(协程)这三套并发 API,以及 CPython 特有的 **GIL(全局解释器锁)**——理解它,就理解了为什么 Python 的多线程"看起来能并行,实际上不能",以及何时该用线程、进程还是协程。作为 C/C++ 工程师,你熟悉 `pthread`、`std::thread`、原子操作、互斥量,本章会逐一和它们对照。

## 一、GIL 是什么

CPython 传统上有一把**全局互斥锁**:任何时刻,**只有一个线程能持有 GIL 并执行 Python 字节码**。它不是语言规范的一部分,而是 CPython 这个实现的产物(Jython、IronPython 没有 GIL)。(严格说,自 3.12 起 GIL 是**每解释器(per-interpreter)**的——同一进程里的多个子解释器可各有一把 GIL,见 PEP 684;下文默认讨论最常见的单解释器情形。)

```python
import sys
# GIL 每隔多少个"检查间隔"考虑一次切换(Python 3.2+ 改为基于时间)
print(sys.getswitchinterval())  # 默认 0.005 秒,即 5ms
```

用 C 的心智模型来看,GIL 相当于在解释器主循环里包了一把大锁:

```
线程 T1                    GIL                     线程 T2
  |                     [ 空闲 ]                     |
  |-- acquire GIL ------>[ T1 ]                      |
  |  执行字节码...        [ T1 ]     <-- 阻塞等待 ----|
  |  (每 5ms 或阻塞时)                                |
  |-- release GIL ------>[ 空闲 ]                     |
  |                      [ T2 ]<---- acquire GIL -----|
  |  <-- 阻塞等待 --------[ T2 ]     执行字节码...     |
```

## 二、为什么存在:保护非原子的引用计数

回顾[「错误处理与资源管理」](09-错误处理与资源管理.md)——CPython 用**引用计数**管理内存,每个对象头里有 `ob_refcnt`。当你 `a = b` 或函数传参时,引用计数 `+1`;变量销毁时 `-1`,归零即释放。

问题在于:`ob_refcnt++` 和 `ob_refcnt--` **不是原子操作**,在 C 层面它是"读-改-写"三步。如果两个线程同时对同一对象增减引用计数而不加锁:

```
初始 refcnt = 1
T1: 读 refcnt(1)          T2: 读 refcnt(1)
T1: +1 得 2               T2: -1 得 0
T1: 写回 2                T2: 写回 0   <-- 竞态!对象被误释放,或永不释放
```

这正是你熟悉的 data race。解决方案有两种:第一种是给每个对象的引用计数加细粒度锁或用原子操作——但对象数以百万计,原子操作在每次 `INCREF`/`DECREF` 都触发缓存一致性开销,单线程性能会大幅下滑;第二种是用一把大锁保护整个解释器状态,这就是 GIL。CPython 选了第二种,因为它让**单线程性能最优**,且让 C 扩展的编写变得极其简单——扩展作者默认自己独占解释器,无需考虑并发。GIL 保护的不止引用计数,还有解释器的各种全局状态(内存分配器、内部缓存等)。

**这个取舍对日常并发编程实践的直接后果**是本章接下来几节的主题:既然同一时刻只有一个线程能碰 Python 对象,那"多线程"在 CPython 里就不再意味着"多核并行执行 Python 代码",而只意味着"多个执行流交替持有 GIL"——这决定了线程、进程、协程在 Python 里各自该用在什么场景,与它们在 C++ 里的分工完全不同。

## 三、后果:CPU 密集型多线程不加速

由于同一时刻只有一个线程跑字节码,**纯计算(CPU 密集)任务开多线程不会变快,反而因为锁竞争和上下文切换略微变慢**:

```python
import time
from threading import Thread

def cpu_bound(n: int) -> int:
    # 纯计算:狂占 CPU,不释放 GIL
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 20_000_000

# 串行
t0 = time.perf_counter()
cpu_bound(N); cpu_bound(N)
print(f"串行:{time.perf_counter() - t0:.2f}s")

# 双线程
t0 = time.perf_counter()
threads = [Thread(target=cpu_bound, args=(N,)) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(f"双线程:{time.perf_counter() - t0:.2f}s")  # 不会快一倍,可能还更慢
```

对比 C++:两个 `std::thread` 跑纯计算能吃满两个核心,近乎线性加速。Python 里两个线程在抢同一把 GIL,始终只有一个核心在真正工作:

```
双核 CPU 上的 CPU 密集型 Python 多线程:

Core 0: [T1][   ][T1][   ][T2][   ]...   <- 两线程轮流用同一核心
Core 1: [空闲............................]  <- 第二个核心浪费

C++ std::thread 同任务:
Core 0: [T1][T1][T1][T1][T1][T1]...      <- 满载
Core 1: [T2][T2][T2][T2][T2][T2]...      <- 满载,真并行
```

**这就是为什么 CPU 密集型任务要用多进程而不是多线程**:GIL 是解释器级别的锁,一个进程只有一把,无论开多少线程都逃不掉;唯一能让 Python 代码真正跑在多个核心上的办法,是让每个执行流拥有自己独立的解释器和独立的 GIL——也就是开多个**进程**(第五节)。这不是"进程比线程更好"这种笼统的结论,而是 GIL 这个具体机制决定的:凡是瓶颈在于消耗 CPU 周期执行 Python 字节码本身的任务,线程数量再多也只是在同一把锁上排队,只有跳出这把锁的作用域(换进程,或者换到不受 GIL 管的 C 扩展代码)才能真正利用多核。

## 四、`threading`:IO 密集型才是主场

关键点:**执行阻塞式 IO(读写文件、网络请求、`time.sleep`)时,线程会主动释放 GIL**。因为等 IO 期间不需要执行字节码,让别的线程跑正合适。此外解释器也会定时(默认 5ms)强制切换。

所以 `threading` 适合 **IO 密集型** 场景:多个线程大部分时间在等网络/磁盘,GIL 在等待期间被释放,实现了有效的并发:

```python
import time, urllib.request
from threading import Thread

def fetch(url: str) -> None:
    # 网络请求阻塞期间释放 GIL,其他线程可运行
    with urllib.request.urlopen(url) as r:
        r.read()

urls = ["https://example.com"] * 8

t0 = time.perf_counter()
for u in urls: fetch(u)
print(f"串行:{time.perf_counter() - t0:.2f}s")

t0 = time.perf_counter()
threads = [Thread(target=fetch, args=(u,)) for u in urls]
for t in threads: t.start()
for t in threads: t.join()
print(f"多线程:{time.perf_counter() - t0:.2f}s")  # 显著加速,IO 等待被重叠
```

和 C++ 一样,共享可变状态需要保护。注意:即使有 GIL,**复合操作(如 `x += 1`)也不是原子的**——它由多条字节码组成(读取、计算、写回是三步独立指令),中间可能被切换:

```python
from threading import Lock, Thread

counter = 0
lock = Lock()

def worker():
    global counter
    for _ in range(100_000):
        with lock:          # 相当于 std::lock_guard,退出作用域自动释放
            counter += 1     # 读-改-写,必须加锁

threads = [Thread(target=worker) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)  # 有锁:400000;去掉锁:小于该值(丢失更新)
```

`threading` 还提供 `RLock`(可重入)、`Semaphore`、`Event`、`Condition`、`Barrier`,语义与 C++ 标准库对应物基本一致。

## 五、`multiprocessing`:绕过 GIL 实现真并行

要在 CPU 密集型任务上吃满多核,就得**开多个进程**。每个进程有**独立的 Python 解释器和独立的 GIL**,互不干扰,由 OS 调度到不同核心:

```python
import time
from multiprocessing import Process

def cpu_bound(n: int) -> int:
    return sum(i * i for i in range(n))

if __name__ == "__main__":   # Windows/spawn 模式必须有此保护
    N = 20_000_000
    t0 = time.perf_counter()
    ps = [Process(target=cpu_bound, args=(N,)) for _ in range(2)]
    for p in ps: p.start()
    for p in ps: p.join()
    print(f"双进程:{time.perf_counter() - t0:.2f}s")  # 接近串行的一半
```

代价是**进程间不共享内存**(和 `fork` 后各写各的一样)。数据传递有两条路:

| 方式 | 机制 | 说明 |
|------|------|------|
| 参数/返回值/队列 | **pickle 序列化** | 对象被序列化成字节流,经管道传给子进程再反序列化。有拷贝开销,且对象必须可 pickle |
| `Value`/`Array`/`shared_memory` | **共享内存** | 直接映射同一块物理内存,零拷贝,但需自己加锁同步 |

```python
from multiprocessing import shared_memory
import numpy as np

# 创建一块共享内存,多个进程可映射同一块物理页,无需拷贝
shm = shared_memory.SharedMemory(create=True, size=1024)
buf = np.ndarray((256,), dtype=np.int32, buffer=shm.buf)
buf[0] = 42
# 另一进程用 shm.name 附着到同一块内存
shm.close(); shm.unlink()
```

启动方式(`start method`):`fork`(拷贝父进程内存,快,但与多线程共用时不安全)、`spawn`(全新解释器,干净但慢)、`forkserver`。**默认值随平台/版本而变**:macOS、Windows 默认 `spawn`;Linux 早期默认 `fork`,但 **3.14 起改为默认 `forkserver`**(因为 `fork` 在多线程程序里易出问题)。别再记成"Unix 默认 fork"。

## 六、`asyncio`:协作式调度换来的无锁确定性交错

```python
import asyncio

async def greet(name):              # async def 定义一个"协程函数"
    print(f"start {name}")
    await asyncio.sleep(0.01)       # await:等待一个异步操作
    print(f"done {name}")
    return name

async def main():
    result = await greet("world")   # 调用协程要用 await
    print(result)

asyncio.run(main())                  # 启动事件循环运行 main
```

`asyncio` 是**另一条路线**:不靠多线程/多进程,而是在**单个线程内**用**事件循环 + 协程**实现高并发 IO。本质是**协作式多任务**(cooperative multitasking),对比线程的**抢占式**(preemptive)调度。

- **协程(coroutine)**:用 `async def` 定义。调用它不会立即执行,而是返回一个协程对象(类似惰性的任务描述)。
- **`await`**:在此处**主动让出控制权**给事件循环。只有遇到 `await` 一个未就绪的操作时,当前协程才暂停,事件循环去跑别的协程。
- **事件循环(event loop)**:一个 `while` 循环,维护就绪任务队列,配合 OS 的 `epoll`/`kqueue`/`IOCP` 监听 IO 事件,哪个 IO 就绪就恢复对应协程。

```python
import asyncio, time

async def fetch(name: str, delay: float) -> str:
    print(f"{name} 开始")
    await asyncio.sleep(delay)   # 让出控制权!事件循环去跑别的协程
    print(f"{name} 完成")
    return name

async def main():
    t0 = time.perf_counter()
    # gather 并发调度多个协程,总耗时约等于最长的那个,而非累加
    results = await asyncio.gather(
        fetch("A", 2), fetch("B", 1), fetch("C", 3),
    )
    print(results, f"耗时 {time.perf_counter() - t0:.2f}s")  # 约 3s,非 6s

asyncio.run(main())   # 创建事件循环,跑 main 直到完成,再关闭循环
```

**`asyncio` 的协作式调度和线程的抢占式调度,区别不只是"实现方式不同",而是对"什么时候可能被打断"给出了完全不同的保证**。线程的调度由操作系统决定,一个线程可能在任意一条字节码指令执行完之后就被换下 CPU——这个切换点对程序员完全不可见、不可控,这也是为什么共享可变状态几乎总是需要锁:你没法从代码本身看出"这两行之间会不会被别的线程插进来改数据"。协程反过来,只有代码里显式写了 `await` 的地方才可能发生切换,`await` 之前和之后的代码在同一个协程里**保证连续执行,不会被其他协程插入**。这个保证换来的是一种**无锁的确定性交错执行**:多个协程共享状态时,只要在两个 `await` 之间读改写这块状态,就不需要任何锁——因为在下一个 `await` 出现之前,没有任何其他协程能运行、能碰这块状态。事件循环的心智模型:

```
事件循环(单线程):
  ┌─> 从就绪队列取一个协程,运行到下一个 await
  │      协程 A 跑到 await sleep(2) --> 注册定时器,挂起 A
  │      协程 B 跑到 await sleep(1) --> 注册定时器,挂起 B
  │      队列空,epoll 等待 IO/定时器就绪
  │      1s 后 B 的定时器到期 --> 把 B 放回就绪队列
  └──── 恢复 B 从 await 之后继续 ...
```

把并发的执行交错显式打印出来——每个 `await` 都是一次让出,可以看到两个协程在切换点轮流推进,顺序完全确定、可重复:

```python
import asyncio

async def worker(name: str, steps: int) -> None:
    for i in range(steps):
        print(f"{name}-{i}")
        await asyncio.sleep(0)   # 让出控制权,事件循环去跑另一个协程

async def main():
    await asyncio.gather(worker("A", 3), worker("B", 3))

asyncio.run(main())
# A-0
# B-0
# A-1
# B-1
# A-2
# B-2   —— 两协程在每个 await 处交替推进(协作式,顺序确定)
```

对照线程:线程的抢占式切换点不可预测,同一段代码两次运行的交错顺序可能不同,这正是并发 bug 难复现的根源之一;协程的切换点全部由源码里的 `await` 显式标出,同样的代码、同样的调度策略下交错顺序是确定且可重复的,调试和推理都容易得多。这是协作式调度用"必须显式写 `await` 让出控制权"这个约束换来的好处——约束本身也是代价:忘记 `await` 一个真正的异步操作、或者在协程里误用同步阻塞调用,会让当前协程"赖着不走",而抢占式的线程不会有这个问题(操作系统总能强制换下一个线程)。

| 维度 | threading | asyncio |
|------|-----------|---------|
| 调度 | 抢占式,OS 决定何时切换 | 协作式,只在 `await` 处切换 |
| 切换点 | 任意字节码边界(不可控) | 显式 `await`(可控、可推理) |
| 并行度 | 受 GIL 限制,无真并行 | 单线程,无真并行 |
| 竞态 | 需要锁,难调试 | 切换点确定,同线程内多数操作无需锁 |
| 开销 | 每线程 MB 级栈 + OS 调度 | 每协程几 KB,可开数万个 |
| 适用 | IO 密集 + 有阻塞库 | IO 密集 + 全异步生态 |

关键陷阱:**在协程里调用阻塞函数(如同步的 `requests.get`、`time.sleep`)会卡死整个事件循环**——因为没有 `await`,不会让出控制权,协作式调度完全依赖每个协程"自觉"在恰当的地方让出,一旦有协程不遵守这个约定,单线程事件循环上排队的所有其他协程都会被这一个协程堵住。必须用异步库(`aiohttp`、`httpx`)或把阻塞调用丢到线程池:`await loop.run_in_executor(None, blocking_fn)`,本质是把这一小段真正会阻塞的工作挪回抢占式调度的世界(线程池),用一个 `await` 把它重新接回协作式的事件循环。

```python
import asyncio, time

def blocking_io(x: int) -> int:   # 同步阻塞函数(比如同步版 requests.get)
    time.sleep(0.05)
    return x * 2

async def main():
    loop = asyncio.get_running_loop()
    # 丢进默认线程池执行,阻塞发生在别的线程,事件循环不被冻结
    results = await asyncio.gather(
        loop.run_in_executor(None, blocking_io, 1),
        loop.run_in_executor(None, blocking_io, 2),
    )
    print(results)   # [2, 4]

asyncio.run(main())
```

## 七、`concurrent.futures`:统一的高层接口

`concurrent.futures` 提供了线程池和进程池的统一 API,是日常最常用的入口。`Future` 对象类似 C++ 的 `std::future`。

```python
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def task(x: int) -> int:
    return x * x

# IO 密集用线程池
with ThreadPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(task, range(10)))

# CPU 密集用进程池(绕过 GIL)
if __name__ == "__main__":
    with ProcessPoolExecutor(max_workers=4) as ex:
        futures = [ex.submit(task, i) for i in range(10)]
        for f in futures:
            print(f.result())   # 阻塞直到该任务完成
```

选型口诀:**CPU 密集 → ProcessPool;IO 密集 → ThreadPool 或 asyncio。** 这个口诀直接来自第二、三节的论证:GIL 只锁字节码执行,不锁"等待"本身,所以线程能在 IO 等待上有效并发;GIL 锁的恰恰是 CPU 密集代码要做的事(执行字节码),所以只有换成独立 GIL 的进程才能真正并行。

## 八、free-threaded(no-GIL):GIL 的黄昏?

CPython **3.13(2024 年发布)以实验性形式**引入了 **free-threaded 构建**(PEP 703),即"no-GIL";到 **3.14 转为官方正式支持,但仍不是默认**。它通过**偏向锁 + 无锁引用计数(biased reference counting)+ 每对象锁**等技术,让引用计数在多线程下安全,从而**移除 GIL**,实现真正的多核并行线程——本质是把第二节里"用一把大锁保护引用计数"换成了"用更细粒度的锁/原子操作保护引用计数",取舍的方向和当年引入 GIL 时正好相反:当年是牺牲多核并行换单线程性能与 C 扩展简单性,free-threading 是牺牲一部分单线程性能与 C 扩展兼容性、换回多核并行能力。

要点:

- 需要专门的构建版本(如 `python3.13t` / `python3.14t`),普通发行版默认仍是带 GIL 的构建。
- 单线程性能有一定回退(早期约百分之几,持续优化中)——这正是"细粒度锁/原子操作比一把大锁开销更高"这个第二节论证的直接体现。
- C 扩展需要适配(声明支持 free-threading),生态迁移需时间,原因也在第二节:过去 C 扩展作者可以假设"同一时刻只有我的代码在动 Python 对象",no-GIL 下这个假设不再成立。
- 3.13 实验性 → 3.14 正式支持(但非默认);带 GIL 与 free-threaded 两种构建将长期共存。

如果这一路线成功,未来 Python 的 `threading` 将能像 C++ 的 `std::thread` 一样吃满多核,`multiprocessing` 的很多 pickle/共享内存痛点也会缓解。但在可预见的将来,上面讲的选型原则仍然成立。

## 小结

- **GIL** 是 CPython 的解释器级大锁(3.12 起为每解释器 per-interpreter),一次只有一个线程执行字节码,存在的根本原因是保护非原子的引用计数等解释器内部状态,换来单线程性能最优和 C 扩展编写的简单性。
- CPU 密集型多线程**不加速**——GIL 锁的正是字节码执行本身;IO 密集型多线程**有效**——阻塞 IO 时线程会主动释放 GIL。
- **multiprocessing** 用独立进程 + 独立 GIL 实现真并行,代价是数据靠 pickle 或共享内存传递,这是绕开 GIL 唯一可靠的路。
- **asyncio** 用单线程事件循环 + 协作式协程:切换点全部是源码里显式的 `await`,这换来了无锁的确定性交错执行(两次 `await` 之间的代码保证不被打断),代价是一旦某个协程忘记让出(误用阻塞调用),会卡住整个事件循环。
- **free-threaded 构建**(3.13 实验、3.14 正式但非默认)正在用更细粒度的同步机制替换 GIL,取舍方向与当年引入 GIL 时相反。
