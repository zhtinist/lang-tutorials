# 标准库算法:排序、堆、容器与迭代器

C++ 有 STL——`std::sort`(内省排序)、`std::priority_queue`、`std::lower_bound`、各种容器。Python 标准库提供了功能对等、但在底层实现上各有取舍的工具,每一处取舍都对应着标准库设计者对"什么场景更常见"的一个具体判断。

## 一、`sorted` / `list.sort`:为什么是 Timsort

`sorted(iterable)` 返回一个新的、排好序的 list;`list.sort()` 原地排序、不返回新对象。两者都支持 `key`(提取排序依据)和 `reverse`(倒序):

```python
data = [("b", 2), ("a", 2), ("c", 1)]

# key:提取排序键;类似 C++ 的比较器,但只算一次键(装饰-排序-去装饰),更快
s = sorted(data, key=lambda x: x[1])
print(s)  # [('c', 1), ('b', 2), ('a', 2)] —— 稳定:b 仍在 a 前

# reverse 保持稳定性(不是简单反转结果)
print(sorted(data, key=lambda x: x[1], reverse=True))  # [('b',2),('a',2),('c',1)]

nums = [5, 3, 1, 4, 2]
nums.sort()          # 原地排序,返回 None(修改自身)
print(sorted(nums))  # 返回新列表,原列表不动
```

Python 的排序用 **Timsort**——由 Tim Peters 于 2002 年为 CPython 设计,后来被 Java、Android、V8 等采用。它是一种**自适应的、稳定的归并排序**,融合了归并排序和插入排序,最坏情况 O(n log n),不像快排(C++ `std::sort` 用的内省排序底层混合了快排)会退化到 O(n²)。

**为什么标准库选一种稳定排序,而不是像 C++ `std::sort` 默认那样不保证稳定**(C++ 要稳定需要显式用 `std::stable_sort`):这个选择直接服务于 `key=` 这个真实使用场景。实践中经常需要按多个条件依次排序——比如"先按部门分组,同部门内保留原来按入职时间排好的顺序",这种模式要求排序算法保证"相等元素保持原相对顺序",否则第二次排序会把第一次排序建立的顺序打乱。快排在实现上天然不稳定(分区过程会交换不相邻的相等元素),要拿到稳定性通常得多付出额外的内存或比较开销;归并排序天然稳定(合并时先取左边相等的元素)。Python 选择把"稳定"作为排序的默认保证而不是可选项,是判断"多数程序员会默认假设排序是稳定的、且这个假设一旦落空很难在测试里发现"——一个不稳定排序引入的 bug 往往只在极少数相等键的场景下才会现形,排查代价很高,标准库因此宁可让默认排序稍慢一些也要保证稳定。上面的例子里 `key` 相同的 `("b",2)`/`("a",2)` 排序后仍保持 b 在 a 前,就是这个保证的直接体现。

**为什么还要做成"自适应"**:归并排序的经典实现对任何输入都无差别地做 `O(n log n)` 次比较,即便输入本身已经接近有序。但现实数据经常不是随机的——日志按时间追加、增量更新后重新排序的列表——大段已经有序或倒序的情况很常见。Timsort 先扫描数组找出天然递增或递减的连续片段(称为 **run**,降序 run 直接原地反转变成升序),只有长度不足 `minrun`(32~64)时才用二分插入排序补足,再用一个栈按接近斐波那契数列的规则合并相邻 run,合并时还用 galloping(飞驰)模式加速"一方元素连续胜出"的情况。这一整套机制的净效果是:**对已经有序或接近有序的数据,复杂度趋近 O(n)**,而对完全随机的数据退化为普通归并排序的 O(n log n)——识别并利用输入里已有的顺序,不需要每次都从零开始比较。

`key` 函数对每个元素只调用一次,结果缓存起来参与比较(Schwartzian transform,装饰-排序-去装饰),所以用 `key` 远比 `functools.cmp_to_key` 包装的比较函数快——后者在排序过程中每次比较都要重新调用两次比较函数,前者只在排序开始前算一次键。

## 二、`bisect`:有序数组二分查找

`bisect` 在**已排序的 list** 上做二分查找,对应 C++ 的 `std::lower_bound`/`std::upper_bound`:

```python
import bisect

a = [1, 3, 3, 5, 7]
print(bisect.bisect_left(a, 3))   # 1:第一个 >=3 的位置(lower_bound)
print(bisect.bisect_right(a, 3))  # 3:第一个 >3 的位置(upper_bound)

bisect.insort(a, 4)   # 二分找到位置后插入,保持有序
print(a)              # [1, 3, 3, 4, 5, 7]
```

`insort` 的查找是 O(log n),但**插入本身是 O(n)**,因为 list 是连续数组,插入要移动后续元素([见「复合数据结构」](07-复合数据结构.md)里 list 为何是数组而非链表的讨论)——二分只能加速"找到该插入的位置"这一步,搬移元素这一步的成本是数组这个底层结构本身决定的,`bisect` 改变不了它。若需频繁在中间插入,该换的是数据结构本身,而不是查找算法。

一个经典用法——用它把连续值映射到离散等级:

```python
def grade(score: int) -> str:
    breakpoints = [60, 70, 80, 90]
    grades = "FDCBA"
    return grades[bisect.bisect(breakpoints, score)]
print([grade(s) for s in (55, 65, 85, 95)])  # ['F', 'D', 'B', 'A']
```

## 三、`heapq`:数组隐式表示的二叉堆

```python
import heapq

h = []
heapq.heappush(h, 3)      # O(log n):上浮
heapq.heappush(h, 1)
heapq.heappush(h, 2)
print(h[0])               # 1:堆顶恒为最小值,O(1) 查看
print(heapq.heappop(h))   # 1:弹出最小,O(log n),末尾元素补顶后下沉

nums = [5, 3, 8, 1, 9, 2]
heapq.heapify(nums)       # O(n) 原地建堆,比逐个 push 的 O(n log n) 快
print(heapq.nlargest(3, nums))   # [9, 8, 5]:Top-K,用大小为 k 的堆,O(n log k)
print(heapq.nsmallest(2, nums))  # [1, 2]

# merge:惰性合并多个已排序序列,不把全部读入内存(外部归并的基础)
print(list(heapq.merge([1, 4], [2, 3], [0, 5])))  # [0, 1, 2, 3, 4, 5]
```

只有最小堆。要最大堆,取负数,或对元组用 `(-priority, item)`。对比 C++ `std::priority_queue` 默认是最大堆——方向相反,注意别搞混:

```python
import heapq

tasks = [(2, "email"), (1, "backup"), (3, "report")]  # (优先级, 任务)
h = []
for pri, name in tasks:
    heapq.heappush(h, (-pri, name))   # 取负 => 最大优先级落到堆顶
print([heapq.heappop(h)[1] for _ in range(3)])
# ['report', 'email', 'backup']  —— 按优先级 3、2、1 依次出队
```

`heapq` 没有单独的堆类型,而是一组作用于**普通 list** 的函数,把这个 list 当作二叉最小堆来操作。**为什么用数组隐式表示完全二叉树,而不是像教科书里那样用指针连成的二叉树节点**:这与[「复合数据结构」](07-复合数据结构.md)里 `deque` 用分块数组而非逐节点链表是同一条思路——指针树的每个节点是一次独立的堆分配,节点之间在内存里彼此不相邻,遍历/比较时 cache miss 频繁,还要为每个节点多存两个子节点指针。数组表示利用了完全二叉树的一个数学性质:节点 `i` 的孩子必然是 `2i+1` 和 `2i+2`,父子关系可以直接用下标算术算出来,不需要任何显式指针,数据本身连续存放、cache 友好。`push`/`pop` 靠"上浮"(新元素放末尾,和父节点比较、不满足堆序就交换,一路向上)/"下沉"(堆顶弹出后把末尾元素挪到堆顶,和较小的子节点比较、不满足就交换,一路向下)维护堆序,两者都是 O(log n)——堆高度是 log n,是完全二叉树"数组能表示"这个性质本身决定的。

## 四、`collections`:为特定模式优化的专用容器

标准 `dict`/`list`/`set` 之外,`collections` 提供了针对特定模式优化的容器:

```python
from collections import Counter, defaultdict, OrderedDict, deque, namedtuple

# Counter:多重集/频率统计,底层是 dict[element -> count]
c = Counter("abracadabra")
print(c.most_common(2))     # [('a', 5), ('b', 2)]:内部用 heapq.nlargest

# defaultdict:访问缺失键时自动调用工厂创建默认值,免去 setdefault
d = defaultdict(list)
d["x"].append(1)            # "x" 不存在,自动建空 list
print(d)                    # defaultdict(<class 'list'>, {'x': [1]})

# deque:双端队列,底层是分块的双向链表,两端 O(1) 增删
dq = deque([1, 2, 3], maxlen=3)
dq.appendleft(0)            # O(1),而 list.insert(0,x) 是 O(n);maxlen 已满,自动挤掉尾端的 3
dq.append(4)                # maxlen 已满,自动挤掉此刻头端的 0(环形缓冲区)
print(dq)                   # deque([1, 2, 4], maxlen=3)

# namedtuple:带字段名的 tuple,内存与 tuple 一样紧凑(无 __dict__)
Point = namedtuple("Point", ["x", "y"])
p = Point(1, 2)
print(p.x, p[0])            # 1 1:既能按名也能按下标访问
```

`OrderedDict` 记录插入顺序——但自 Python 3.7 起**普通 dict 已保证插入序**([见「复合数据结构」](07-复合数据结构.md)紧凑字典一节),所以 `OrderedDict` 现在只在需要 `move_to_end`、顺序敏感的相等比较等场景才用,不再是"要保序就必须用它"的唯一选择。

| 容器 | 底层结构 | 典型优势操作 |
|------|----------|--------------|
| `Counter` | dict | 计数、`most_common` |
| `defaultdict` | dict | 缺失键自动初始化 |
| `deque` | 双向链表(分块) | 两端 O(1),定长环形缓冲 |
| `namedtuple` | tuple | 命名字段、内存紧凑 |

## 五、`itertools`:惰性迭代器组合子

`itertools` 提供**惰性**(lazy)的迭代器工具。它们返回迭代器而非列表——**用时才算,内存 O(1)**,可处理无限序列,这与生成器一脉相承,类似 C++20 的 ranges/views:

```python
import itertools as it

# 无限迭代器:count / cycle / repeat
counter = it.count(10, 2)          # 10, 12, 14, ... 惰性,不会占满内存
print(list(it.islice(counter, 3))) # [10, 12, 14]:islice 切前 3 个

# 组合学:笛卡尔积、排列、组合
print(list(it.product([0, 1], repeat=2)))     # [(0,0),(0,1),(1,0),(1,1)]
print(list(it.combinations("ABC", 2)))        # [('A','B'),('A','C'),('B','C')]

# chain:串接多个可迭代对象,零拷贝地"拼接"
print(list(it.chain([1, 2], [3, 4])))         # [1, 2, 3, 4]

# groupby:按连续相同的键分组(要求已排序,类似 SQL 分组前需 order by)
data = [("a", 1), ("a", 2), ("b", 3)]
for k, g in it.groupby(data, key=lambda x: x[0]):
    print(k, list(g))              # a [('a',1),('a',2)] / b [('b',3)]

# accumulate:前缀和/扫描
print(list(it.accumulate([1, 2, 3, 4])))      # [1, 3, 6, 10]
```

## 六、`functools`:高阶函数工具

```python
import functools

# lru_cache:记忆化,自动缓存函数结果。底层是 dict + 双向链表实现的 LRU
@functools.lru_cache(maxsize=None)
def fib(n: int) -> int:
    return n if n < 2 else fib(n - 1) + fib(n - 2)
print(fib(50))   # 瞬间完成:无缓存则指数级递归

# reduce:折叠,对应 C++ std::accumulate / 函数式的 fold
print(functools.reduce(lambda a, b: a * b, [1, 2, 3, 4]))  # 24

# partial:偏函数,固定部分参数,类似 std::bind
base2 = functools.partial(int, base=2)
print(base2("1010"))   # 10:相当于 int("1010", base=2)
```

`@functools.cache`(3.9+)是 `lru_cache(maxsize=None)` 的简写。缓存以参数为键,故参数必须可 hash——这是[「复合数据结构」](07-复合数据结构.md)里"可变对象不可哈希"这条规则的又一处体现:缓存本质上是一个 `参数元组 -> 结果` 的 dict,key 必须可哈希。

用 `cache_info()` 能看到命中/未命中次数,直观验证记忆化确实生效:

```python
import functools

@functools.lru_cache(maxsize=None)
def fib(n: int) -> int:
    return n if n < 2 else fib(n - 1) + fib(n - 2)

fib(10)
print(fib.cache_info())
# CacheInfo(hits=8, misses=11, maxsize=None, currsize=11)
# 11 次未命中(n=0..10 各首次计算一次),8 次命中(重复子问题直接查表)
```

## 七、`array` / `struct` / `memoryview`:贴近 C 的连续内存

普通 `list` 是**指针数组**(每个槽存 `PyObject*`)。而这三个模块让你操作**紧凑的原始字节**,内存布局与 C 数组一致:

```python
from array import array

# array:同质类型的紧凑数组,"i" 表示 C 的 int(4字节),无逐元素对象头
a = array("i", [1, 2, 3])
print(a.itemsize)   # 4:每个元素 4 字节连续存放,像 int32_t arr[3]
# 对比 list[int]:每元素是独立 PyObject,再存一个指针,内存大数倍
```

```python
import struct

# struct:在 Python 值与 C 结构体的二进制表示间转换(处理二进制协议/文件)
# "<" 小端, "i" int32, "f" float32, "H" uint16
packed = struct.pack("<iH", 258, 5)   # 打包成字节串
print(packed)                          # b'\x02\x01\x00\x00\x05\x00'
print(struct.unpack("<iH", packed))    # (258, 5):解包还原
print(struct.calcsize("<iH"))          # 6:字节数(注意对齐!用 "<" 关闭填充)
```

`struct` 的对齐规则你很熟悉:默认(native `@`)会按平台对齐插入填充字节;用 `<`/`>`/`=` 前缀切到标准大小、无填充:

```python
# memoryview:零拷贝地访问其他对象的缓冲区(buffer protocol),类似 C++ span
buf = bytearray(b"hello world")
mv = memoryview(buf)
sub = mv[0:5]           # 不拷贝!只是指向同一内存的视图
sub[0] = ord("H")       # 通过视图原地修改
print(buf)              # bytearray(b'Hello world'):底层字节被改
```

`memoryview` 是高性能 IO、图像处理、与 NumPy/C 扩展交互的关键——它让你切片大缓冲区而**不产生拷贝**。

## 八、`math` / `statistics` / `random`

```python
import math, statistics, random

print(math.gcd(12, 18), math.isqrt(17))   # 6 4:整数平方根,无浮点误差
print(math.comb(5, 2), math.perm(5, 2))   # 10 20:组合/排列数
print(math.fsum([0.1] * 10))              # 1.0:补偿求和,避免浮点累积误差

print(statistics.mean([1, 2, 3, 4]))      # 2.5
print(statistics.median([1, 3, 2]))       # 2
print(statistics.stdev([1, 2, 3, 4]))     # 样本标准差

random.seed(42)                            # 底层是 Mersenne Twister(周期 2^19937-1)
print(random.randint(1, 6))                # 掷骰子(注意:非密码学安全!)
# 密码学安全随机用 secrets 模块(基于 OS 的 CSPRNG)
```

`random` 用于模拟/游戏,**不可用于密钥、令牌**——Mersenne Twister 的内部状态可以被足够多的连续输出反推出来,一旦状态泄露就能预测后续所有输出,这对密钥、令牌这类要求不可预测性的场景是致命的;`secrets` 模块基于操作系统提供的密码学安全随机源(CSPRNG),专门为这类场景存在。

## 复杂度速查表

| 操作 | 复杂度 | 底层 |
|------|--------|------|
| `sorted` / `list.sort` | O(n log n),近乎有序时趋近 O(n) | Timsort |
| `bisect.bisect` | O(log n) | 二分 |
| `bisect.insort` | O(n) | 二分 + 数组移位 |
| `heappush` / `heappop` | O(log n) | 二叉堆上浮/下沉 |
| `heapify` | O(n) | 自底向上建堆 |
| `heapq.nlargest(k, n项)` | O(n log k) | 大小 k 的堆 |
| `deque.append` / `appendleft` / `pop` / `popleft` | O(1) | 分块双向链表 |
| `deque` 中间索引 | O(n) | 需遍历 |
| `Counter.most_common(k)` | O(n log k) | heapq |
| `lru_cache` 命中 | O(1) 均摊 | dict + 链表 |

## 小结

- 排序用 **Timsort**:稳定、自适应、最坏 O(n log n)。稳定是为多条件排序(先按 A 排、再稳定地按 B 排)这个真实场景保证的,自适应是为了利用真实数据里常见的局部有序性,对近乎有序数据趋近 O(n);优先用 `key` 而非 `cmp_to_key`。
- `bisect` 二分查找 O(log n),但插入仍受连续数组本身限制为 O(n)——二分只能加速定位,搬移的代价是数组结构决定的。
- `heapq` 把 list 当二叉堆(仅最小堆),用数组下标算术表达父子关系,省去指针树的逐节点分配和 cache miss;Top-K 用 `nlargest`/`nsmallest`。
- `collections` 针对模式优化:`deque` 两端 O(1),`Counter`/`defaultdict`/`namedtuple` 各有专长;`OrderedDict` 因为 dict 已保序而不再是唯一选择。
- `itertools` 惰性组合子,内存 O(1),可处理无限流;`functools` 提供 `lru_cache`/`reduce`/`partial`。
- `array`/`struct`/`memoryview` 让你操作紧凑连续内存,是通往 C 世界与高性能 IO 的桥梁。
