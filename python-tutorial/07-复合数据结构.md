# 复合数据结构的内部实现

> 本章是全教程的重点之一。我们要把 `list`、`tuple`、`dict`、`set`、`deque` 的内部数据结构拆开,和 C 的数组、结构体、哈希表逐一对照。
> 记住贯穿全书的一条:[见「变量与内存模型」](03-变量与内存模型.md),Python 的容器里装的从来不是"值",而是**指向 `PyObject` 的指针**。这一点决定了它们的内存布局、拷贝语义和复杂度。

## 0. 一句话先建立直觉

C 的数组 `int a[4] = {1,2,3,4}` 在内存里是这样的:

```
a:  [ 1 ][ 2 ][ 3 ][ 4 ]        // 16 字节,值连续存放,cache 友好
     ^0   ^4   ^8   ^12
```

Python 的 `a = [1, 2, 3, 4]` 是这样的:

```
list 对象(堆)                    每个元素是一个独立的堆对象
+-----------------+
| ob_refcnt       |
| ob_type = list  |
| ob_size = 4     |----> ob_item(另一块堆内存,指针数组)
| allocated = 4   |       +------+------+------+------+
| ob_item ------- |------>| ptr0 | ptr1 | ptr2 | ptr3 |
+-----------------+       +--|---+--|---+--|---+--|---+
                            v      v      v      v
                          PyLong PyLong PyLong PyLong
                           (1)    (2)    (3)    (4)   ← 各自在堆上,还可能被别处共享
```

所以 Python 的 list 是**"指针的数组"**,不是"值的数组"。这一个图能解释本章几乎所有现象。

## 1. `list`:指向 PyObject 的指针的动态数组

### 1.1 内存布局

CPython 中 list 的结构体(简化自 `Include/cpython/listobject.h`):

```c
typedef struct {
    PyObject_VAR_HEAD          // 含 ob_refcnt、ob_type、ob_size(=len)
    PyObject **ob_item;        // 指向一块单独分配的指针数组
    Py_ssize_t allocated;      // 已分配的容量(容量 >= ob_size)
} PyListObject;
```

关键点:

- list 对象本身是固定大小的头部;真正的元素指针存在 `ob_item` 指向的**另一块堆内存**里。这是两次分配(list 头 + 指针数组),类似 C++ 里 `std::vector` 的 `vector` 对象和它 `new[]` 出来的缓冲区分离。
- `ob_size` 是当前长度 `len(lst)`;`allocated` 是缓冲区能容纳的指针个数。二者关系恒为 `allocated >= ob_size`,中间的空槽是为了 append 不用每次都重新分配。
- 每个元素占 `ob_item` 里的一个指针位(64 位平台上 8 字节),指向真正的对象。**元素对象本身不在数组里**。

对照 `std::vector<T>`:vector 直接把 `T` 的值连续存放(`T=int` 就是连续的 int)。Python list 永远存指针,哪怕全是小整数,也要经过一次指针解引用才能拿到值。这就是为什么数值计算要用 `numpy`([见「常用第三方库」](12-常用第三方库.md)):numpy 的 ndarray 才是真正连续存值的 C 数组。

### 1.2 over-allocation 扩容策略

append 时若 `ob_size == allocated`,需要扩容。CPython 不是简单翻倍,而是用一个温和的增长公式(`list_resize`,简化):

```c
/* 现行 CPython list_resize:多分配约 1/8,再向上对齐到 4 的倍数 */
new_allocated = (new_size + (new_size >> 3) + 6) & ~3;
```

即大约多分配 **1/8** 再加一个小常数并对齐。实测(CPython 3.14)`allocated` 增长序列是:

```
0, 4, 8, 16, 24, 32, 40, 52, 64, 76, 92, 108, 128, ...
```

对比 C++ `std::vector` 常见的 1.5x 或 2x 增长。CPython 用 1/8 是在"内存浪费"和"重分配次数"之间取的折中,因为它假设 list 的典型用法追加量不会特别夸张。

**扩容时发生了什么**:调用 `realloc` 给 `ob_item` 换一块更大的内存,把旧的指针数组按位拷贝过去(只拷指针,不拷元素对象,也不动引用计数)。元素对象原地不动,别处对它们的引用依然有效。

```
append 触发扩容:
旧:  ob_item -> [p0][p1][p2][p3]                (allocated=4, size=4)
     realloc 到新缓冲区(allocated=8):
新:  ob_item -> [p0][p1][p2][p3][p4][ ][ ][ ]    (size=5)
     元素对象 (对应 p0..p4) 全程没动
```

### 1.3 各操作复杂度与原因

| 操作 | 复杂度 | 底层原因 |
|------|--------|---------|
| `lst[i]` 索引读写 | O(1) | 指针数组随机访问,`ob_item[i]` |
| `lst.append(x)` | 摊还 O(1) | 有空槽直接写;满了才 realloc,均摊到每次是常数 |
| `lst.pop()`(尾部) | O(1) | `ob_size--`,DECREF 尾元素 |
| `lst.insert(0, x)` | O(n) | 后面所有指针要 `memmove` 右移一格 |
| `lst.pop(0)` | O(n) | 后面所有指针左移;要频繁头部操作请用 `deque` |
| `del lst[i]` / 中间删除 | O(n) | memmove 补洞 |
| `x in lst` | O(n) | 线性扫描逐个 `==` 比较 |
| `len(lst)` | O(1) | 直接读 `ob_size` |
| 切片 `lst[a:b]` | O(k) | 新建 list,拷贝 k 个指针(浅拷贝!)并各 INCREF |

关于摊还 O(1):n 次 append 总的拷贝成本是所有扩容拷贝之和,因增长是几何级数,总和为 O(n),平摊到每次即 O(1)。这和 `std::vector::push_back` 的分析完全一样。

### 1.4 浅拷贝:一个必须记住的陷阱

`lst2 = lst[:]`、`list(lst)`、`copy.copy(lst)` 都是**浅拷贝**:新建一个 list 头和新的指针数组,但指针指向的是**同一批元素对象**。

```python
a = [[1, 2], [3, 4]]
b = a[:]              # 浅拷贝
b[0].append(99)       # 修改的是共享的内层 list
print(a)              # [[1, 2, 99], [3, 4]]  —— a 也变了!
```

```
a.ob_item -> [p0][p1]
                \    \
                 v    v
             [1,2,99] [3,4]      ← 内层对象只有一份
                 ^    ^
b.ob_item -> [p0][p1]            ← b 是新数组,但指针指向同一内层对象
```

要彻底隔离得用 `copy.deepcopy`,它会递归重建每一层对象。这与 C++ 的"浅拷贝 vs 深拷贝"是同一个问题,只是 Python 默认到处都是引用,更容易踩到。

## 2. `tuple`:不可变的定长数组

tuple 的结构体(`PyTupleObject`)比 list 更简单——它把指针数组**内联**在对象尾部,而不是单独分配:

```c
typedef struct {
    PyObject_VAR_HEAD
    PyObject *ob_item[1];   // 变长尾部数组,创建时一次分配好,永不改变
} PyTupleObject;
```

区别与后果:

- 没有 `allocated` 字段,长度创建后固定。**一次分配**(对象头和元素指针数组在同一块内存里),比 list 少一次分配、少一层间接,内存更紧凑。类似 C 里的柔性数组成员(flexible array member)。
- 不可变指的是"元素指针数组不可变"——你不能改 `t[0]` 指向别的对象。但如果某个元素本身是可变对象(如 list),它的内容仍可变:

```python
t = (1, [2, 3])
t[1].append(4)     # OK,改的是内层 list
# t[1] = [9]       # TypeError:不能改 tuple 的槽位
```

- 因为(在元素都可哈希时)tuple 可哈希,所以能作 `dict` 的 key 或 `set` 的元素;list 不行。见下文哈希部分。
- CPython 对小 tuple 有**空闲链表(free list)**缓存,频繁创建销毁小 tuple 时能复用内存块,减少 malloc 压力。

经验法则:异构、固定的"记录"用 tuple(类似 C 的 `struct`,但按位置而非按名字);同构、需要增删的"集合"用 list。想要带名字的字段,见 [`namedtuple`](11-标准库算法.md)。

## 3. `dict`:CPython 的紧凑字典

`dict` 是 Python 最核心的数据结构——它不仅是用户的哈希表,还是对象属性(`__dict__`)、命名空间、关键字参数的底层载体。所以它的实现被反复优化。

### 3.1 哈希表基础与开放寻址

Python dict 用**开放寻址(open addressing)**,不是链地址法(chaining)。也就是说,冲突时不在桶里挂链表,而是按一个探测序列去找下一个空槽。所有条目都住在同一块连续数组里,cache 更友好。

- key 必须**可哈希**:实现了 `__hash__` 且哈希值在对象生命周期内不变。不可变内置类型(int/str/tuple/frozenset)可哈希;list/dict/set 不可哈希(因为它们可变,内容一变哈希就失效,会破坏哈希表)。
- 平均查找/插入/删除 **O(1)**;最坏(大量冲突)O(n),但现实中极少见。

### 3.2 3.6+ 的紧凑字典:为什么保序

Python 3.6 起 dict 记忆插入顺序(3.7 起是语言保证)。这不是加了个链表,而是改了数据结构——拆成两个数组:

```
indices(哈希表,存的是到 entries 的下标,可能有空槽 -1):
+----+----+----+----+----+----+----+----+
| -1 |  1 | -1 | -1 |  0 | -1 |  2 | -1 |
+----+----+----+----+----+----+----+----+

entries(紧凑数组,按插入顺序追加,无空洞):
+---------+--------+---------+
|hash key val|hash key val|hash key val|
|   'a':1   |  'b':2  |  'c':3  |
+---------+--------+---------+
   idx 0      idx 1     idx 2
```

- **稀疏的 `indices` 数组**负责哈希定位,里面存的是"到 entries 的整数下标",空槽存特殊值。它可以很稀疏(装载因子低),但每个槽只占一个小整数(视表大小 1/2/4/8 字节),浪费的空间小。
- **紧凑的 `entries` 数组**按插入顺序连续存放真正的 `(hash, key, value)` 三元组,无空洞。遍历 dict 就是顺序扫这个数组 —— 这就是保序的原因,而且遍历比旧版(遍历稀疏大表)更快。
- 相比 3.5 之前"每个槽都存完整 entry"的稀疏大表,紧凑字典节省约 **20%–25%** 内存。

**实测:dict 保序,且顺序由"插入"而非"排序"决定**。删除中间的键再重新插入,会追加到末尾;更新已有键则原地改值、顺序不动:

```python
d = {}
d["c"] = 1; d["a"] = 2; d["b"] = 3
print(list(d))     # ['c', 'a', 'b']   按插入序,不是排序
del d["a"]         # 删掉中间的键
d["a"] = 9         # 重新插入 —— 追加到 entries 末尾
print(list(d))     # ['c', 'b', 'a']
d["c"] = 100       # 更新已存在的键:原地改 value,顺序不动
print(list(d))     # ['c', 'b', 'a']
```

查找过程:对 key 求哈希 → 取低位作 `indices` 下标 → 拿到 entries 下标 → 去 entries 比对 hash 和 key(先比 hash 整数,相等再比 `==`,因为哈希可能碰撞)。

### 3.3 探测序列

CPython 用一种扰动探测(perturbation probing),兼顾散布和局部性(简化逻辑):

```c
perturb = hash;
i = hash & mask;
while (slot_used_and_key_differs) {
    perturb >>= 5;
    i = (i * 5 + perturb + 1) & mask;
}
```

它不是简单线性探测,而是不断把 hash 的高位"搅"进来,让探测序列更分散,减少聚集。

### 3.4 扩容

装载因子超过 **2/3** 时扩容。新表容量取"能容纳当前已用条目 3 倍"的最小的 2 的幂(增长约 2x–4x)。扩容时**重建 indices 表**(重新散布所有 key),但 entries 保持插入顺序被搬过去。扩容期间还会顺便清理删除留下的"墓碑"。

### 3.5 复杂度表

| 操作 | 平均 | 最坏 | 说明 |
|------|------|------|------|
| `d[k]` 查 | O(1) | O(n) | 哈希 + 探测 |
| `d[k] = v` 插入/改 | O(1) 摊还 | O(n) | 可能触发扩容重散布 |
| `del d[k]` | O(1) | O(n) | 标记删除,entries 位置留空(遍历会跳过) |
| `k in d` | O(1) | O(n) | 同查找 |
| 遍历 | O(n) | O(n) | 顺序扫 entries |
| `len(d)` | O(1) | O(1) | 读计数 |

> 与 C++ 对照:`std::unordered_map` 是链地址法(每桶挂链表/节点,节点各自 new,cache 不友好);`std::map` 是红黑树 O(log n) 且按 key 排序。Python dict 更像一个高度优化的开放寻址哈希表,且额外保证插入顺序。

## 4. `set` 与 `frozenset`

`set` 的实现和 dict 高度相似——本质是"只有 key 没有 value 的哈希表",同样开放寻址、同样要求元素可哈希。区别:

- set **不保证顺序**(它没有像 dict 那样引入紧凑 entries 数组来记录插入序;它用的是更接近传统稀疏表的结构)。别依赖 set 的迭代顺序。
- 用于成员测试(`x in s` 平均 O(1))、去重、集合运算(`&` 交、`|` 并、`-` 差、`^` 对称差)。
- `frozenset` 是不可变版本,因此**它自己可哈希**,可以作为 dict 的 key 或放进另一个 set。

四种集合运算实测(小整数哈希为自身,故打印顺序稳定):

```python
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}
print(a & b)   # {3, 4}              交集
print(a | b)   # {1, 2, 3, 4, 5, 6}  并集
print(a - b)   # {1, 2}              差集(在 a 不在 b)
print(a ^ b)   # {1, 2, 5, 6}        对称差(恰在其一)
```

```python
seen = set()
for x in data:
    if x not in seen:      # 平均 O(1),比 list 的 O(n) in 快得多
        seen.add(x)
```

| 操作 | 平均复杂度 |
|------|-----------|
| `add` / `remove` / `in` | O(1) |
| 交集 `a & b` | O(min(len a, len b))——遍历较小集合去查另一个 |
| 并集 `a` \| `b` | O(len a + len b) |
| 差集 `a - b` | O(len a) |

## 5. `collections.deque`:分块双向链表

list 的头部操作是 O(n)。当你需要两端高效增删(队列、滑动窗口),用 `collections.deque`。

它的实现**不是**普通双向链表(那样每个元素一个节点、cache 极差),而是**分块双向链表(doubly-linked list of fixed-size blocks)**:

```
block(每块 64 个指针)      block                block
+----------------+       +----------------+     +----------------+
| prev | 元素... |  <->  | prev | 元素... | <-> | prev | 元素... |
| next |         |       | next |         |     | next |         |
+----------------+       +----------------+     +----------------+
   ^ leftblock                                     ^ rightblock
```

- 每个 block 是一个定长(64)的指针数组;block 之间用前后指针连成双向链表。
- 两端 `append`/`appendleft`/`pop`/`popleft` 都是 **O(1)**:在头/尾 block 里挪一个位置,满了就再链一个新 block。
- 代价:随机索引 `dq[i]` 是 **O(n)**(要沿 block 链走),因此 deque 不适合随机访问。
- `maxlen` 参数可做成固定长度的环形缓冲区(超长自动从另一端挤出),很适合"最近 N 条"这类场景。

| 操作 | list | deque |
|------|------|-------|
| 尾部 append/pop | O(1) 摊还 | O(1) |
| 头部 insert/pop | O(n) | O(1) |
| 随机索引 `x[i]` | O(1) | O(n) |

## 6. 全景复杂度对照表

| 结构 | 底层 | 随机访问 | 头部增删 | 尾部增删 | 成员测试 | 有序 | 可作 key |
|------|------|---------|---------|---------|---------|------|---------|
| `list` | 指针动态数组 | O(1) | O(n) | O(1)摊还 | O(n) | 保插入序 | 否 |
| `tuple` | 内联指针数组(不可变) | O(1) | — | — | O(n) | 保序 | 是(元素可哈希时) |
| `dict` | 紧凑开放寻址哈希表 | O(1) 按 key | — | — | O(1) | 保插入序 | 否 |
| `set` | 开放寻址哈希表 | — | — | — | O(1) | 否 | 否 |
| `frozenset` | 同上,不可变 | — | — | — | O(1) | 否 | 是 |
| `deque` | 分块双向链表 | O(n) | O(1) | O(1) | O(n) | 保序 | 否 |

## 7. 与 C 的内存模型对照小结

| C/C++ | Python | 关键差异 |
|-------|--------|---------|
| `int a[N]` | `list` | C 连续存值;list 存指针,值散落堆上 |
| `std::vector<T>` | `list` | vector 存值且可存栈上小对象;list 永远存指针 |
| `struct {int; char*;}` | `tuple` / `namedtuple` / 类 | struct 按偏移排布字段;tuple 按下标存指针 |
| `std::unordered_map` | `dict` | 链地址 vs 开放寻址;dict 额外保序 |
| `std::unordered_set` | `set` | 同上 |
| `std::deque` | `deque` | 概念相近,都是分块 |

核心记忆点:**Python 容器是引用的容器**。拷贝容器默认只拷指针(浅拷贝),多个容器可以别名同一批元素;要真正独立必须 `deepcopy`。想要 C 那样连续存值、cache 友好、可向量化的数组,去用 [`array`/`memoryview`](11-标准库算法.md) 或 [`numpy`](12-常用第三方库.md)。
