# 常用第三方库:生态、包管理与性能

标准库之外,Python 真正的力量来自庞大的第三方生态。作为 C/C++ 工程师,你会发现这里的"包管理"比 CMake + 手动链接舒服得多,但也有自己的坑(全局污染、版本冲突)。本章以**定位与选型**为主——告诉你每个库解决什么问题、底层如何工作,而非堆砌 API。

## 一、生态与包管理

### PyPI、pip、wheel

- **PyPI**(Python Package Index):官方包仓库,类似 npm registry / crates.io。
- **pip**:安装器,`pip install requests` 从 PyPI 下载并安装。
- **wheel**(`.whl`):**预编译的二进制分发格式**。对纯 Python 包只是打包;对含 C 扩展的包(如 NumPy),wheel 里装的是**已编译好的 `.so`/`.pyd`**,免去你本地装编译器和源码构建。这解决了 C/C++ 世界最痛的"编译依赖"问题。
- **sdist**:源码分发,没有对应 wheel 时 pip 会下载 sdist 并现场编译(此时才需要编译工具链)。

### venv:虚拟环境

Python 默认把包装到**全局**解释器,极易造成项目间版本冲突。**虚拟环境**给每个项目一份隔离的依赖副本(类似给每个项目一个独立的 `node_modules`)。

```bash
python -m venv .venv           # 创建虚拟环境(一个含独立 site-packages 的目录)
source .venv/bin/activate      # 激活(Windows: .venv\Scripts\activate)
pip install requests numpy     # 装到该环境,不污染全局
pip freeze > requirements.txt  # 导出精确版本,供他人复现
deactivate                     # 退出
```

### 现代工具:uv / poetry

| 工具 | 定位 |
|------|------|
| **pip + venv** | 标准、最小,但依赖解析弱、无锁文件 |
| **poetry** | 依赖管理 + 打包 + 锁文件(`poetry.lock`),声明式 `pyproject.toml` |
| **uv** | Rust 编写,**极快**(比 pip 快 10-100 倍),兼容 pip 接口,集成虚拟环境/锁文件/Python 版本管理,当前社区强烈推荐 |

`pyproject.toml`(PEP 518/621)已成为标准的项目元数据文件,取代旧的 `setup.py`。

## 二、NumPy:连续内存的向量化

NumPy 是科学计算的基石,理解它的**内存模型**你就懂了它为什么快。

### ndarray vs list:内存布局对照

Python `list[float]` 是**指针数组**——每个槽是 `PyObject*`,指向堆上各自独立、带对象头的 float 对象。数据在内存里**分散**,遍历要不断解引用,缓存命中差。

NumPy `ndarray` 是**一整块连续的 C 数组**,元素是裸机器数(如 float64),无逐元素对象头。

```
Python list[float]:
  [ptr0][ptr1][ptr2]        (指针数组)
    |     |     |
    v     v     v
  {头|1.0}{头|2.0}{头|3.0}   (堆上分散的对象,含引用计数等)

NumPy ndarray (float64):
  [ 1.0 | 2.0 | 3.0 | ... ]  (一块连续内存,8字节一个,像 double[N])
```

```python
import numpy as np

a = np.arange(1_000_000, dtype=np.float64)  # 连续内存,8MB 一整块
print(a.nbytes, a.dtype, a.strides)          # 8000000 float64 (8,)

# 向量化:整个操作在 C 层的紧凑循环里跑完,无 Python 字节码、无 GIL 反复获取
b = a * 2 + 1        # 一行,底层是编译好的 SIMD 循环
# 对比纯 Python:[x*2+1 for x in a] 每次迭代都跑解释器 + 创建对象,慢 10-100 倍
```

### 向量化为何快

1. **无解释器开销**:循环在编译好的 C 代码里执行,不逐元素跑字节码。
2. **缓存友好**:连续内存,预取器和 SIMD 高效工作。
3. **可用 SIMD/多线程**:底层 BLAS(如 OpenBLAS/MKL)对矩阵运算用向量指令和多核。
4. **释放 GIL**:C 循环期间释放 GIL,可与其他线程并行。

`strides`(步长)机制让**切片、转置、reshape 多为零拷贝视图**——只改元数据不动数据,和你玩指针算术是一个道理。

## 三、数据与网络常用库

### pandas:表格数据

建立在 NumPy 之上的 `DataFrame`(二维带标签表格)。每列是一个一维数组、按列存储(缓存友好):pandas 1.x/默认后端下多为 NumPy 数组,但 **pandas 2.x 起也支持 Arrow 列式后端**(`dtype_backend="pyarrow"`),此时列是 Arrow 数组而非 NumPy。用于数据清洗、聚合、连接、时序分析——相当于"内存里的 SQL + Excel"。

```python
import pandas as pd
df = pd.DataFrame({"name": ["a", "b"], "score": [90, 85]})
print(df.groupby("name")["score"].mean())   # 声明式聚合,底层向量化
```

### requests / httpx:HTTP 客户端

- **requests**:最经典的同步 HTTP 库,API 极友好。
- **httpx**:现代替代,API 兼容 requests,但**同时支持同步和 async**(可配合[第 10 章](10-并发模型.md)的 asyncio),还支持 HTTP/2。

```python
import httpx
r = httpx.get("https://example.com")
print(r.status_code, len(r.text))
```

### pydantic:数据校验

用类型注解(见[第 08 章](08-面向对象与类型系统.md))定义数据模型,在**运行时**自动校验、解析、类型转换。核心校验逻辑用 Rust(pydantic v2 的 `pydantic-core`)实现,很快。是 FastAPI 的基石。

```python
from pydantic import BaseModel

class User(BaseModel):
    id: int
    name: str

u = User(id="123", name="Alice")   # "123" 自动转成 int 123;类型不符则抛错
print(u.id, type(u.id))            # 123 <class 'int'>
```

### SQLAlchemy:ORM 与数据库工具

Python 事实标准的数据库工具包。两层:底层 **Core**(SQL 表达式构造器,类型安全地拼 SQL)和上层 **ORM**(对象关系映射,把类映射到表、对象映射到行)。支持连接池、方言抽象,可对接 PostgreSQL/MySQL/SQLite 等。

## 四、测试与 Web 框架

### pytest:测试框架

比标准库 `unittest` 更简洁,用普通 `assert` 即可(它会重写断言,给出详细失败信息),配合 fixture(依赖注入式的测试资源)、参数化、丰富插件生态。

```python
def add(a, b): return a + b

def test_add():
    assert add(2, 3) == 5        # 直接用 assert,失败时 pytest 展示两边的值

import pytest
@pytest.mark.parametrize("a,b,expected", [(1, 1, 2), (2, 3, 5)])
def test_add_many(a, b, expected):
    assert add(a, b) == expected
```

### Web 框架定位

| 框架 | 定位 |
|------|------|
| **Flask** | 轻量微框架,自己拼装组件,适合小型服务、灵活可控 |
| **Django** | "全家桶",自带 ORM、admin 后台、认证、模板,适合大型传统 Web 应用 |
| **FastAPI** | 现代异步 API 框架,基于类型注解 + pydantic 自动做校验和 OpenAPI 文档,原生 async,高性能 API 首选 |

## 五、类型检查:mypy

Python 的类型注解**默认在运行时不强制**(仅是元数据,见[第 08 章](08-面向对象与类型系统.md))。**mypy** 是**静态类型检查器**——在**运行前**像 C++ 编译器那样查类型错误,把动态语言的一部分安全性前移到开发期。

```python
def greet(name: str) -> str:
    return "hi " + name

greet(123)   # 运行时能跑(拼接前才报错),但 mypy 会静态报错:期望 str
```

```bash
mypy myfile.py     # 静态检查,不运行代码。pyright 是另一个流行选择(微软,更快)
```

## 六、性能:测量与下沉到 C

### 先测量:cProfile

优化的第一原则和 C++ 一样——**别猜,先测**。标准库 `cProfile` 统计每个函数的调用次数与耗时。

```bash
python -m cProfile -s cumtime myscript.py   # 按累计耗时排序,找热点
```

行级分析用 `line_profiler`,内存用 `memory_profiler`。找到真正的瓶颈函数,再决定优化它。

### 何时下沉到 C

当纯 Python 的热点确实成为瓶颈(通常是紧密数值循环),几条下沉路径,由易到难:

| 方案 | 机制 | 适用 |
|------|------|------|
| **先试 NumPy** | 向量化,把循环交给 C | 数值数组运算 |
| **Cython** | 给 Python 加类型注解,编译成 C 扩展 | 把现有 Python 热点渐进加速 |
| **ctypes** | 标准库,运行时加载已有 `.so`/`.dll` 调用其函数 | 调用现成 C 库,无需编译胶水 |
| **cffi** | 更友好地声明 C 接口并调用 | 绑定 C 库,比 ctypes 更好维护 |
| **原生 C 扩展 / pybind11** | 直接用 CPython C API 或 pybind11(C++) | 深度集成、性能极致 |
| **PyO3** | 用 Rust 写扩展 | 现代高性能扩展(pydantic-core 即用它) |

```python
# ctypes 示例:直接调用系统 C 库函数
import ctypes
libc = ctypes.CDLL("libc.dylib")   # Linux: "libc.so.6"
print(libc.strlen(b"hello"))        # 5:调用 C 的 strlen
```

关键判断:**先向量化(NumPy),再考虑 Cython/Rust**。多数"Python 慢"的问题其实是算法或没向量化,而非语言本身。记住释放 GIL 的意义——在 C 扩展里做重活时释放 GIL,才能配合多线程真正并行。

---

## 小结

- **包管理**:PyPI 是仓库,pip 是安装器,wheel 是预编译二进制(免编译);用 **venv** 隔离依赖,现代项目首选 **uv** 或 poetry + `pyproject.toml`。
- **NumPy** 用连续 C 内存的 `ndarray` 取代 list 的指针数组,向量化把循环下沉到编译好的 C/SIMD,快 10-100 倍。
- 生态选型:数据用 **pandas**,HTTP 用 **httpx/requests**,校验用 **pydantic**,数据库用 **SQLAlchemy**,测试用 **pytest**,Web 按规模选 Flask/Django/FastAPI。
- **mypy** 把类型检查前移到开发期。
- 性能:先用 **cProfile** 定位,优先 NumPy 向量化,再按需用 Cython/ctypes/cffi/PyO3 下沉到 C/Rust。
