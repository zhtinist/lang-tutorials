# 变量与内存模型 ⭐

对 C/C++ 工程师来说,Python 大部分"灵异事件"——改了一个列表另一个跟着变、函数里改了参数外面也变、默认参数记住了上次的值——根源都是同一件事:**Python 的变量不是内存槽,而是名字到堆对象的引用绑定。** 这个模型一旦确立,后面所有看似诡异的行为都变得可预测,本章用 `id()`、`is`、`sys.getrefcount` 反复做实证并配合内存图讲透它。

## 一、变量不是盒子,是名签

C 的心智模型是"盒子":`int x` 就是一块命名的内存,`x = 3` 往盒子里塞值,`x = 5` 覆盖它,变量**就是**那块内存。

Python 完全不是。回顾[「运行模型」](01-运行模型.md):所有值都是堆上的 `PyObject`。变量只是**命名空间里的一个名字**,绑定到某个堆对象上——可以把它想成"贴在对象上的便利贴",或者说**变量是一个持有对象地址的指针,但你永远不能对它取地址、也做不了指针算术**。

```
C 的模型:变量即内存         Python 的模型:名字指向堆对象
┌─────────┐                名字 x         堆
│  x: 3   │  ← 盒子即值     ┌────┐        ┌──────────────┐
└─────────┘                │ ●──┼──────► │ int 对象: 3  │
                           └────┘        └──────────────┘
                           便利贴          真正的值在这
```

**为什么 Python 要选这种"名字绑定"模型而不是 C 的"内存槽"模型**:根源仍在动态类型——C 里 `int x` 一旦声明,这块内存的大小和解释方式(4 字节、按补码整数读)在编译期就定死了,变量和它存的值天然是同一回事。Python 里同一个名字可以先后指向 `int`、`str`、任意自定义对象,这些类型的大小、内部结构完全不同,不可能让一个固定大小的"盒子"去装任意类型的值——必须让变量只存一个统一大小的东西(指针),真正的数据和类型信息留在被指向的堆对象里。名字绑定模型是动态类型在内存层面的必然实现方式,不是随意的设计偏好。

`id(obj)` 返回对象的身份标识,**在 CPython 中就是对象的内存地址**(语言只保证同一生命周期内唯一且恒定,地址是实现细节)。它相当于 `&`,只不过是只读的、针对对象而非变量的:

```python
x = 3
print(id(x))   # 例如 4310721234,就是那个 int 对象的堆地址
```

## 二、`a = b` 到底拷贝了什么

```python
a = [1, 2, 3]
b = a
b.append(4)
print(a)        # [1, 2, 3, 4]
```

C++ 工程师的第一反应通常是"`b = a` 不是拷了一份吗?改 b 怎么动了 a?"——因为 `b = a` **既不拷贝对象,也不拷贝数据,只拷贝了那个指针值**。执行后,`a` 和 `b` 是贴在**同一个堆对象**上的两张便利贴:

```
a = [1, 2, 3]
b = a

  名字 a ─┐
          ├──► ┌──────────────────────┐
  名字 b ─┘    │ list 对象            │
               │ [ ●, ●, ● ]          │ ← 同一个对象
               └──┼──┼──┼─────────────┘
                  ▼  ▼  ▼
                  1  2  3   (各自也是堆上的 int 对象)
```

`b.append(4)` 通过 b 找到那个 list 对象、**原地修改**它;a 指向同一对象,从 a 看过去当然也变了。这不是拷贝语义的 bug,是**别名(aliasing)**——两个名字是同一对象的两个别名。类比 C++:`b = a` 更像 `int* b = a;`(拷贝指针),而绝不是 `std::vector<int> b = a;`(拷贝构造)。**Python 的赋值永远是"拷贝引用",从不调用拷贝构造函数**——这背后是一个明确的设计取舍:如果赋值默认做深拷贝,任何一次简单的 `b = a` 在 a 是大对象时都会隐含一次昂贵的整体复制,而多数场景下程序员真正需要的只是"给这个对象再起一个名字";Python 把"廉价、总是 O(1)"作为赋值语义的基本承诺,需要真正复制时必须显式调用 `copy`/`deepcopy`(见第八节)——代价前置在语义设计里说清楚,好过默认行为里藏着不确定的性能开销。

`==` 比较**值**是否相等(调用 `__eq__`);`is` 比较**是否同一个对象**(比较 `id`),等价于 C 的两个指针比较:

```python
a = [1, 2, 3]
b = a
c = [1, 2, 3]          # 另建一个内容相同的列表

print(a == b)          # True,值相等
print(a is b)          # True,同一对象(别名)
print(a == c)          # True,值相等
print(a is c)          # False,不同对象
```

`a is b` 为真而 `a is c` 为假,证明 `b = a` 制造了别名,而 `c = [1,2,3]` 是全新对象。判断"是不是同一个东西"用 `is`,"值一不一样"用 `==`(`is None` 的惯用法正来自这里——`None` 是单例,身份比较既准确又免去调用 `__eq__` 的开销)。

## 三、引用计数:谁在指着我

回顾[「运行模型」](01-运行模型.md):每个 `PyObject` 头里有 `ob_refcnt`。每多一个名字/容器指向它就 +1,少一个就 -1,归零立即回收(CPython 的主要内存管理机制,循环引用另有垃圾回收器处理,见[「错误处理与资源管理」](09-错误处理与资源管理.md))。

```python
import sys

x = object()
print(sys.getrefcount(x))   # 例如 2

y = x                        # 多一个引用
print(sys.getrefcount(x))   # 3

del y                        # 去掉一个引用(del 解除名字绑定,不是删对象!)
print(sys.getrefcount(x))   # 2 回落
```

`sys.getrefcount(x)` 的结果总比直觉多 1,因为把 `x` 作为实参传给函数这个动作本身又临时加了一次引用。`del y` 删除的是**名字绑定**,不是对象——对象在 refcnt 归零时才真正释放,这跟 C++ 的 `delete` 语义完全不同,`del` 更像"松开这只手",最后一只手松开对象才没。

**为什么 CPython 选引用计数作为主要回收机制,而不是像 Java/Go 那样纯用追踪式 GC**:引用计数的核心优势是**确定性**——对象在最后一个引用消失的那一刻立即释放,内存占用曲线平滑,不需要专门的 GC 暂停去扫描整个堆;这对 1990 年代内存普遍紧张、且 Python 从一开始就要和大量 C 扩展打交道的场景很重要,C 扩展可以直接用 `Py_INCREF`/`Py_DECREF` 管理引用而不必理解一个复杂的追踪式收集器。代价是显而易见的:引用计数本身有维护开销(几乎每次赋值、传参都要修改计数),且**无法处理循环引用**(两个对象互相引用,谁的计数都不会真正归零),CPython 因此在引用计数之上又叠加了一个专门检测循环的分代垃圾回收器作为补丁,两者共同工作而非二选一。

## 四、可变 vs 不可变:决定别名行为的分界线

```python
a = 1
b = a
b = b + 1
print(a, b)     # 1 2 —— a 没变
```

为什么这段没有"改 b 动 a"的问题?区别在于:list 是**可变(mutable)**的,int 是**不可变(immutable)**的。

- **可变对象**(list、dict、set、大多数自定义类实例):可以原地修改内容,`b.append(4)`、`b[0] = 9` 改的是对象本身,所有别名都看得到。
- **不可变对象**(int、float、str、tuple、bool、frozenset、None):内容不能改,任何"修改"操作其实是**造一个新对象**并让名字改指过去。

`b = b + 1` 的底层:`b + 1` 求出一个**新的** int 对象 `2`,`b =` 让名字 b 改绑到这个新对象;原来那个 `1` 对象纹丝没动,a 还指着它:

```
a = 1; b = a                b = b + 1

 a ─┐                        a ────► [ int 1 ]   ← a 仍指旧对象
    ├──► [ int 1 ]           b ────► [ int 2 ]   ← b 改指新对象
 b ─┘                        (新分配的对象)
```

**为什么 Python 要设计出"不可变对象"这个类别,而不是让一切都可变(更符合"一切皆对象、变量只是引用"的统一模型)**:不可变性带来三个具体好处,值得为它牺牲"万物统一可变"的简洁性——第一,不可变对象天然可以安全共享,多个名字指向同一个 `int`/`str` 对象不需要担心某处代码悄悄改了它,这是小整数缓存、字符串驻留(见第六节)这类共享优化能安全存在的前提;第二,不可变对象可以被安全地用作 `dict` 键、放进 `set`(可变对象的哈希值如果随内容变化,放进哈希表后内容一变就找不到了,Python 索性禁止可变类型作为默认可哈希);第三,函数式编程里"传进去的值不会被意外修改"这一心智负担被免除。代价是需要修改内容时必须构造新对象,这在频繁拼接字符串这类场景下有实打实的性能开销(需要用 `list`/`StringBuilder` 式的可变容器规避,见[「复合数据结构」](07-复合数据结构.md))。

对比可变对象的原地修改:

```
a = [1,2,3]; b = a          b.append(4)

 a ─┐                        a ─┐
    ├──► [1,2,3]                ├──► [1,2,3,4]   ← 同一对象被改
 b ─┘                        b ─┘                  两个名字都看到变化
```

核心规律:对可变对象做**原地修改**影响所有别名;对任何对象做**重新绑定**(`b = ...`)只改 b 这个名字的指向;不可变对象没有"原地修改"这条路,别名永远安全。

一个坑:`+=` 对可变和不可变行为不同。`b += [4]` 对 list 是原地扩展(等价 `extend`,影响别名);`b += 1` 对 int 是重新绑定(不影响别名)。原因是可变对象实现了原地版的 `__iadd__`,不可变对象没有实现、`+=` 退回普通的 `b = b + 1`:

```python
a = [1, 2]
b = a
a += [3]        # list 的 += 是原地扩展(__iadd__),还是同一对象
print(b)        # [1, 2, 3]   别名 b 跟着变

a = a + [4]     # + 造一个新 list,再把 a 重新绑定过去
print(b)        # [1, 2, 3]   b 不跟随了
```

## 五、小整数缓存与字符串驻留

这两个是 CPython 的实现细节,不是语言语义,但用 `is` 时一定会撞见。

CPython 启动时预先创建 `-5` 到 `256` 的所有整数对象并缓存复用,这个范围内相同的值是**同一个对象**:

```python
a = 100
b = 100
print(a is b)      # True,都指向缓存里那个唯一的 100 对象

m = int("257")
n = int("257")
print(m is n)      # False,两次 int() 各自在堆上新建了一个 257 对象
print(m == n)      # True,值当然相等
```

**为什么要做这个缓存**:小整数在实际代码里用得极频繁(循环计数、索引、布尔判断的底层都是小整数),预先建好、永久复用,避免了对同一个常用值反复分配释放对象的开销——这是拿一小块常驻内存换运行时避免大量重复分配的典型时间换空间。范围 `[-5, 256]` 是 CPython 写死的实现选择,别的实现或版本可能不同,**判断数值相等永远用 `==`,绝不用 `is`**,`is` 只是碰巧对小整数"能用",出了范围就会给出误导性的 `False`。

字符串驻留(interning)同理:CPython 把某些字符串(尤其是看起来像标识符的字面量)驻留到一张全局表里,相同内容共享同一对象,这样字典查键、标识符比较能先比指针(`is`)快速判等,而不必逐字符比较内容:

```python
a = "hello"
b = "hello"
print(a is b)      # 通常 True,被驻留了

import sys
u = "hello world!".upper().lower()
v = "hello world!".upper().lower()
print(u is v)                              # 通常 False,运行时构造,未驻留
print(sys.intern(u) is sys.intern(v))      # True,手动 intern 后共享同一对象
```

驻留的触发条件依版本和上下文而变,**永远不要依赖它**,同样只用 `==` 判等。

## 六、命名空间即字典,但局部变量是 fast locals

模块的全局命名空间、对象的属性,底层确实是 dict(key 是名字字符串,value 是对象引用):

```python
x = 10
print(globals()["x"])    # 10,全局命名空间就是个 dict
```

但函数内的局部变量是个重要例外。CPython 编译函数时能静态数出局部变量有几个、各是谁,于是把它们放进一个**定长数组**("fast locals"),按下标访问,而不是每次去 dict 里按字符串查——回顾[「运行模型」](01-运行模型.md)里 `dis` 看到的 `LOAD_FAST`:

```python
import dis

g = 1
def f(a):
    b = a + g       # a、b 是局部(fast),g 是全局
    return b

dis.dis(f)
```

```
  4           RESUME                   0

  5           LOAD_FAST_BORROW         0 (a)
              LOAD_GLOBAL              0 (g)
              BINARY_OP                0 (+)
              STORE_FAST               1 (b)

  6           LOAD_FAST_BORROW         1 (b)
              RETURN_VALUE
```

(`LOAD_FAST_BORROW` 是 3.14 对读取局部变量的优化变体,"借用"引用免去 refcnt 增减,本质仍按数组下标取局部;全局仍是 `LOAD_GLOBAL`。)

`LOAD_FAST`/`STORE_FAST` 走数组下标,O(1),无哈希、无字符串比较;`LOAD_GLOBAL` 走 dict 查找(先本模块 globals 再 builtins),开销明显更大。**为什么函数局部变量要单独优化成数组、而不是像全局命名空间一样统一用 dict**:函数体在编译期是静态已知的——一个函数用到哪些局部变量名、总共几个,编译器扫一遍源码就能确定,不会在运行期动态增减(不像模块全局或对象属性可以在运行时任意添加新名字)。既然名字集合编译期已经固定,就没有必要每次访问都付出字典查找的哈希计算和字符串比较代价,直接分配一个定长数组、把每个名字编译期映射到一个固定下标即可——这是"能在编译期确定的信息,就不要留到运行期重新计算"这一优化原则的具体体现,而模块全局命名空间无法做这个优化,因为它必须支持运行时动态增删名字(比如 `exec` 或者动态 `import`)。

这解释了一个常见性能技巧:热循环里把频繁访问的全局/属性用局部变量"缓存"一下,就是把 `LOAD_GLOBAL`(查字典)换成 `LOAD_FAST`(查数组)。也解释了函数内为什么读写全局要 `global` 声明——编译器靠"函数里有没有对该名字赋值"决定把它当局部还是全局,这个决定在编译期就要做出,无法拖到运行期看情况而定。完整故事见[「函数与调用栈」](06-函数与调用栈.md)。

## 七、浅拷贝 vs 深拷贝

`b = a` 只拷指针,真想拷一份数据要用 `copy` 模块,且要分清浅拷贝和深拷贝——这与 C++ 里"拷贝构造是浅是深、要不要自己写深拷贝"是同一个问题:

```python
import copy

original = [[1, 2], [3, 4]]

alias    = original                 # 别名,同一对象
shallow  = copy.copy(original)      # 浅拷贝:新外层,内层元素仍共享
deep     = copy.deepcopy(original)  # 深拷贝:递归复制所有层

original[0].append(99)

print(alias)    # [[1, 2, 99], [3, 4]]   跟着变(同一对象)
print(shallow)  # [[1, 2, 99], [3, 4]]   跟着变,内层是共享的
print(deep)     # [[1, 2], [3, 4]]       独立,没变
```

```
original ─► [ ●, ● ]            外层 list
              │  │
              ▼  ▼
           [1,2] [3,4]          内层两个 list

shallow = copy.copy(original):
shallow ─► [ ●, ● ]  ← 新的外层对象
             │  │
             └──┴──► 指向同样的 [1,2] 和 [3,4]  (内层没复制)

deep = copy.deepcopy(original):
deep ─► [ ●, ● ]  ← 新外层
          │  │
          ▼  ▼
       [1,2]'[3,4]'  ← 连内层也是新对象,完全独立
```

**为什么默认的浅拷贝只复制一层,而不是干脆总是深拷贝更安全**:嵌套结构可能任意深,盲目深拷贝对大型嵌套对象(比如一棵大树或图)代价很高,而且深拷贝要求正确处理循环引用(对象自己引用自己),实现更复杂;浅拷贝把"要拷贝到多深"这个决定权交还给调用者——大多数场景下容器本身需要独立、但里面装的对象继续共享是可以接受的(比如复制一个"指针列表",指针指向的内容本来就该共享)。这是典型的"提供一个廉价的默认操作,昂贵的操作必须显式请求"的设计惯例,与前面"赋值默认拷引用而非深拷贝"是同一条原则。

其他造浅拷贝的常见写法:`list(x)`、`x[:]`、`dict(x)`、`x.copy()`,都和 `copy.copy` 一样是浅的;要完全独立必须 `deepcopy`,它有递归遍历和额外分配的成本,不应无脑使用。

## 八、可变默认参数陷阱

有了本章的模型,可以理解 Python 最著名的坑之一:

```python
def add_item(item, bucket=[]):    # 危险:默认值 [] 只创建一次
    bucket.append(item)
    return bucket

print(add_item(1))   # [1]
print(add_item(2))   # [1, 2]   可能期望 [2],但它记住了上次
```

原因:默认值 `[]` 是在 `def` **执行时创建一次**的一个可变对象,被函数对象一直持有;每次不传 `bucket` 都复用同一个 list,原地 `append` 自然累积。这是"可变对象 + 别名 + 只创建一次"三者合流的必然结果——`def` 语句本身是运行期执行的一条语句(见[「运行模型」](01-运行模型.md)第三节),默认值表达式只在这条 `def` 语句执行的那一刻求值一次,而不是每次调用函数时重新求值,这个设计的合理性和完整机制留到[「函数与调用栈」](06-函数与调用栈.md)展开。

## 小结

- 变量是名字到堆对象的引用绑定,不是内存盒子——这是动态类型必然导致的内存模型,而非任意选择。
- `a = b` 只拷贝引用,从不拷贝对象或数据,结果是别名;`is`/`id()` 判断是否同一对象,`==` 判断值是否相等。
- 引用计数(`ob_refcnt`)是 CPython 换取"确定性回收"和"C 扩展易于集成"的选择,代价是需要额外的循环垃圾回收器打补丁;`del` 解绑名字而非删对象。
- 可变与不可变的分界决定别名行为:不可变性是为共享安全、可哈希、心智负担低这三个目标换来的,代价是修改内容必须造新对象。
- 小整数缓存、字符串驻留是 CPython 用一小块常驻内存换避免重复分配的实现优化,不是语言保证,判等只用 `==`。
- 函数局部变量用 fast locals 数组而非 dict,是"编译期已知的信息不留到运行期重算"的具体体现;模块全局命名空间因为要支持运行期动态增删名字,无法做同样的优化。
- 浅拷贝只复制一层是"默认操作廉价、深操作显式请求"这一惯例的体现;可变默认参数陷阱是"def 语句只执行一次、默认值只求值一次"的直接推论。
