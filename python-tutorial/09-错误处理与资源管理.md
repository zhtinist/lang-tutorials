# 错误处理与资源管理

本章两条主线:第一条是异常机制——`try`/`except` 怎么写、异常本身是对象、`with` 语句和上下文管理器协议;第二条是 CPython 的内存管理——引用计数、分代 GC、pymalloc 内存池。两条主线都会和你熟悉的 C/C++ 机制逐一对照:错误码、RAII、`malloc`/`free`、智能指针。

## 一、异常是对象,不是错误码

```python
try:
    x = int("abc")           # 这行会抛出 ValueError
except ValueError as e:
    print("转换失败:", e)    # 转换失败: invalid literal for int() with base 10: 'abc'
finally:
    print("清理完成")         # 无论是否出错都会执行
```

Python 的异常不是 C 的错误码,也不是 C++ 里可以抛任意类型的 `throw 42`。所有异常都是继承自 `BaseException` 的**对象**,携带信息(`args`)与回溯(traceback)。层级大致如下:

```
BaseException            ← 顶层,except Exception 抓不到它的这几个直接子类
├── SystemExit           ← sys.exit() 触发,别顺手 except 掉
├── KeyboardInterrupt    ← Ctrl-C,别顺手 except 掉
├── GeneratorExit
└── Exception            ← 「普通」错误的基类,你自己的异常应继承它
    ├── ValueError / TypeError / KeyError / IndexError / ...
    ├── ArithmeticError → ZeroDivisionError
    ├── OSError → FileNotFoundError / PermissionError / ...
    └── RuntimeError / StopIteration / ...
```

**为什么 Python(以及大多数现代语言)选异常而不是 C 那种"返回值里编码错误"的错误码约定**:这是一个关于"错误能不能被默默忽略"的安全性论证,而不只是风格偏好。C 的错误码是函数返回值的一部分,调用方要主动检查它才能知道出没出错——`FILE *f = fopen(path, "r");` 出错时 `f` 是 `NULL`,但语言不强制你检查,一旦漏写 `if (f == NULL)`,程序会拿着一个空指针继续往下跑,直到某处解引用崩溃,而且崩溃的位置往往离真正出错的地方很远,排查成本很高。异常把"出错"这件事从"返回值的一部分"提升为"控制流本身":一旦抛出且没有匹配的 `except`,它会沿调用栈一路向上,中断所有未处理它的函数,最终让程序带着完整的错误信息和调用栈(traceback)终止——**错误不能被静默地什么都不做就继续执行**。这不是说异常绝对安全(一个空的 `except: pass` 一样能把错误吞掉),而是异常机制把"忽略错误"变成了一个必须显式写出来的动作(写一个空的 except 块),而 C 的错误码把"忽略错误"变成了默认行为(什么都不写,就是忽略)——两种设计对"不小心"这件事的容错程度完全不同。

关键推论:`except Exception` 抓不到 `KeyboardInterrupt`、`SystemExit`——因为它们直接挂在 `BaseException` 下,故意让"用户想中断程序"不被业务代码误吞:

```python
print(issubclass(KeyboardInterrupt, Exception))     # False
print(issubclass(KeyboardInterrupt, BaseException)) # True
```

写 `except` 时记住一条实用规则:**捕获异常尽量写 `except Exception`,而不是裸的 `except:` 或 `except BaseException`**,否则连用户按 Ctrl-C、`sys.exit()` 这些"本不该被业务代码吞掉"的信号也会一起被捕获。

## 二、try / except / else / finally

一个 `except` 也能同时捕获多种类型(写成元组),多个 `except` 按顺序匹配:

```python
def parse(s, data):
    try:
        x = int(s)               # 可能抛 ValueError
        y = data[x]               # 可能抛 IndexError / KeyError
    except (ValueError, KeyError) as e:   # 一个 except 抓多种类型(元组)
        print("坏输入:", e)      # e 是异常对象,e.args 是参数元组
    except IndexError:
        print("越界")
    else:
        print("没出错才执行:", y)   # try 未抛异常时运行
    finally:
        print("无论如何都执行")     # 清理,总会运行——即使 try 里 return/break

parse("abc", [1, 2, 3])   # 坏输入: invalid literal ... \n 无论如何都执行
parse("5", [1, 2, 3])     # 越界 \n 无论如何都执行
parse("1", [1, 2, 3])     # 没出错才执行: 2 \n 无论如何都执行
```

`except` 按**从上到下**第一个 `isinstance` 匹配的分支处理,匹配到就停,所以子类异常要写在父类前面。`else` 只在 `try` 块**没有**异常时运行,把"正常路径"和"可能抛错的代码"分开。`finally` **总会执行**,常用于释放资源——但若 `finally` 里有 `return`,会**覆盖** `try`/`except` 里的返回值,这是显式控制流,和 C++ 析构"总在作用域退出时默默运行、不会覆盖返回值"不同,容易踩。

CPython 的异常处理靠字节码里的**异常表(exception table,3.11 起零成本 try)**——没有异常时 `try` 几乎不产生额外开销,只有真正抛出时才查表跳转。这与 C++ 的"零成本异常"(zero-cost EH,正常路径无开销)思路一致:两者都把"检查是否出错"这件事从每次执行的正常路径里挪走,只在真正抛出的那一刻才付出代价。

## 三、`raise`、自定义异常与异常链

用 `raise ... from ...` 可以在包装异常时保留原始原因,排查问题时能看到完整链条:

```python
class ConfigError(Exception):        # 自定义异常:继承 Exception 即可
    pass

def load(path):
    try:
        return open(path).read()
    except OSError as e:
        raise ConfigError(f"读配置失败: {path}") from e   # 异常链:保留原因

try:
    load("/no/such/file")
except ConfigError as e:
    print("捕获:", e)             # 捕获: 读配置失败: /no/such/file
    print("原因:", e.__cause__)   # 原因: [Errno 2] No such file or directory: '/no/such/file'
```

`raise ValueError("msg")` 抛出;裸 `raise`(在 except 内)**重新抛出**当前异常,保留原始 traceback。`raise NewError from cause` 显式设置 `__cause__`,打印时出现 `The above exception was the direct cause of...`;若不写 `from`,在 except 内再抛也会自动记录 `__context__`(隐式链),打印 `During handling of the above exception...`。想切断链用 `raise NewError from None`。

对照 C++:类似"捕获底层异常、包装成领域异常再抛",但 Python 的 `from` 让**原因链**在 traceback 里可见(存在 `__cause__`/`__context__` 属性上),排错更友好——这与第一节的论证是同一条思路的延伸:不仅"出没出错"不能被静默忽略,连"出错的原始原因"也被结构化地保留下来,而不是像很多 C 代码里那样,包装一层错误码之后原始的错误信息就永久丢失了。

## 四、上下文管理器:`with` 是没有确定性析构时的 RAII 替代

内置的 `open()` 配合 `with` 可以自动关闭文件:

```python
with open("data.txt", "w") as f:
    f.write("hello")
# 离开 with 块时,文件已经自动关闭
with open("data.txt") as f:
    print(f.read())       # hello
```

你也可以自己实现一个上下文管理器,只要定义 `__enter__` 和 `__exit__`:

```python
import time

class Timer:
    def __enter__(self):
        self.t0 = time.perf_counter()
        return self
    def __exit__(self, exc_type, exc_val, exc_tb):
        elapsed = time.perf_counter() - self.t0
        print(f"耗时 {elapsed:.4f}s 以内")
        return False            # 返回 False:不吞异常,照常向外传播

with Timer():
    total = sum(range(1000))
print(total)   # 499500
```

更简单的写法是用 `contextlib.contextmanager` 把一个生成器函数变成上下文管理器:

```python
from contextlib import contextmanager

@contextmanager
def opened(path, mode="r"):
    f = open(path, mode)
    try:
        yield f                 # yield 之前 = __enter__,之后 = __exit__
    finally:
        f.close()

with opened("data.txt") as f:
    print(f.read())    # hello
```

C++ 靠**析构函数 + 栈对象**做 RAII——对象离开作用域时,编译器插入的代码会调用析构函数,这个时机是**确定的**,因为 C++ 对象的生命周期能在编译期或至少在语义上被精确追踪(栈对象在作用域结束时销毁)。Python 没有这种确定性析构——见下一节,引用计数虽然大多数时候也是"归零即释放",但对象何时归零本身依赖运行期的引用图,而循环引用的存在意味着"离开作用域"并不必然导致对象被回收(必须等分代 GC 介入,而 GC 何时运行是不确定的)。

**为什么上下文管理器协议(`__enter__`/`__exit__`)是 RAII 思想在没有确定性析构语言里的替代方案**:RAII 的核心价值不是"析构函数"这个具体机制,而是"资源的生命周期与一个语法可见的作用域绑定,离开作用域必然释放"这个保证。C++ 用编译器自动插入析构调用来实现这个保证;Python 做不到编译期插入析构调用(对象生命周期由运行期引用计数/GC 决定,不是作用域决定),于是把"进入/离开一个作用域时应该做什么"做成一个用户可以显式实现的协议——`with` 语句在语法层面划出一个块,进入时调 `__enter__`,不论块内是正常结束还是抛出异常离开,都保证调 `__exit__`。这样"资源与作用域绑定、离开必然释放"这个 RAII 的核心承诺被保留下来了,只是从"编译器自动生成的隐式调用"变成了"语法要求的显式调用"——`with` 比 C++ 的裸花括号啰嗦,但换来的是这个释放时机不依赖于对象的引用计数何时归零,而是直接由 `with` 块的边界决定,是全书里少数几处"确定性"不来自引用计数、而来自语法结构本身的机制:

```
with open("data.txt") as f:      # 进入:调用 f.__enter__()
    process(f)
# 离开 with 块(正常或异常):调用 f.__exit__(exc_type, exc_val, exc_tb),关闭文件
```

`__exit__` 会收到异常信息(正常退出时三个都是 `None`);返回真值可**吞掉**异常,上面 `Timer.__exit__` 返回 `False` 就是明确表示"不吞异常"。这是 Python 管理文件、锁、连接的**首选**方式,原因正是上面这条:它是全书里除了"引用计数即时回收"之外,另一处不依赖 GC、时机完全确定的资源管理机制。

从这里开始,进入本章第二条主线——CPython 怎么自动管理内存。这部分不再引入新语法(`del` 你已经会用),直接对照你熟悉的 C `malloc`/`free`、C++ `shared_ptr` 看底层机制即可。

## 五、引用计数:为什么是默认策略

CPython 内存管理的主力是**引用计数**:每个对象头里有一个 `ob_refcnt`,有新引用指向它就 +1,引用消失(`del`、变量重绑定、离开作用域、容器移除)就 -1;**减到 0 立刻释放**——确定性、即时,类似 C++ 的 `shared_ptr`,但因为有 GIL 保护,计数增减**不需要原子操作**(这也是 GIL 存在的原因之一,见[「并发模型」](10-并发模型.md))。

```python
import sys
a = object()
print(sys.getrefcount(a))   # 例如 2:注意 getrefcount 的实参本身也算一个临时引用,故比直觉多 1
b = a                       # 又一个引用
print(sys.getrefcount(a))   # 3
del b                       # 引用 -1
```

**为什么引用计数是 CPython 的默认策略,而不是像 Java/Go 那样一开始就选纯追踪式 GC**:两种策略解决同一个问题(什么时候可以安全释放一个不再被使用的对象),但对"何时释放"给出的保证完全不同。追踪式 GC 需要周期性地暂停、从一组根对象出发遍历整个可达图,才能判断哪些对象已经不可达——这个"遍历整个堆"的动作决定了它天然是**批量的、时机不确定的**:对象即使已经没有任何引用,也要等到下一次 GC 扫描才会被真正回收,期间它占的内存不会被释放。引用计数反过来是**逐个对象、即时**的:每一次引用变化都立刻更新计数,归零的那一刻就立刻释放,不需要等待任何全局性的扫描。这个即时性直接带来两个具体好处:第一,内存占用曲线更平滑,不会因为等 GC 攒了一堆垃圾才一次性回收而出现"内存陡增陡降";第二,对象析构的时机是**可预测的**(`__del__` 在计数归零那一刻就调用,而不是不确定的将来),这对需要在对象生命周期结束时做清理动作的场景很重要。这两点在 CPython 早年内存普遍紧张、又要和大量 C 扩展打交道的环境下分量很重——C 扩展作者只需要在恰当的地方调用 `Py_INCREF`/`Py_DECREF`,不需要理解和适配一个复杂的追踪式收集器。

**实测确定性即时回收**:引用计数归零的瞬间 `__del__` 就被调用,时机完全可预测:

```python
class Res:
    def __init__(self, name): self.name = name
    def __del__(self): print(f"{self.name} released")

a = Res("a")
b = Res("b")
print("before del a")
del a               # a 的最后一个引用消失 -> 计数归零 -> 立即回收
print("after del a")
# before del a
# a released        <- 就在 del 这一刻,不是等到脚本结束
# after del a
# b released        <- 脚本结束、b 离开作用域时
```

代价也是清楚的:引用计数本身有维护开销——几乎每一次赋值、传参、容器存取都要修改某个对象的计数,这些增减操作累计起来并不便宜;而且引用计数**无法处理循环引用**——两个互相引用的对象即使外部已经不再使用它们,彼此手里都还攥着对方的一个引用,计数永远降不到 0。这是引用计数作为一个策略在数学上的固有盲区,不是实现得不好,CPython 因此必须再叠一层机制专门补这个洞。

## 六、分代 GC:补上引用计数漏掉的循环引用

`a` 引用 `b`、`b` 又引用 `a`,即使外部都不用了,两者的计数也降不到 0。CPython 因此再叠一层**追踪式的分代垃圾回收器**(`gc` 模块)专门回收循环垃圾:

```python
import gc
print(gc.get_threshold())   # (700, 10, 10):第 0 代分配-释放差达 700 触发一次回收(阈值随版本变)
gc.collect()                # 手动跑一次完整回收,返回回收的不可达对象数
```

**为什么这个补丁是"分代"的**:分代 GC 的设计建立在一条从经验观察里总结出来的规律上——**分代假说(generational hypothesis)**:绝大多数对象在分配后很短时间内就会死亡(变得不可达),只有少数对象能活得足够久。这不是 Python 特有的规律,而是几乎所有语言的实际内存使用模式都符合的经验观察(Java、Go 的分代 GC 也基于同一个假说)。既然大多数对象"朝生夕死",那扫描的重点就应该放在"最近分配的这批对象"上,而不是每次都不分青红皂白地扫描整个堆——扫一遍全堆的成本随堆大小线性增长,而大部分被扫到的老对象其实早就证明了自己会一直存活,重复检查它们纯粹是浪费。CPython 把对象分成三代(0/1/2):新对象在第 0 代,挺过一次回收就"晋升"到更老的代;越老的代扫描越不频繁。这样,占多数的短命对象在最便宜的第 0 代扫描里就被处理掉,只有少数真正长期存活的对象才会被拖进代价更高、但频率也更低的老年代扫描——分代假说成立与否直接决定了这套策略是否划算,而实践证明它在绝大多数程序里都成立。

只有**可能形成环的容器对象**(list/dict/set/实例等)才被 gc 追踪;`int`、`str` 这类不可能自己引用自己的原子对象不进 gc,只靠引用计数——这也是分代 GC 作为"补丁"而非"主力"的体现:它只处理引用计数解决不了的那一小撮情况,没有必要把所有对象都纳入追踪范围。

**实测:纯循环引用引用计数回收不掉,要靠 gc**:

```python
import gc
class Node:
    def __del__(self): print("Node freed")

gc.collect()          # 先清掉解释器启动残留,让计数干净
gc.disable()          # 关掉自动回收,便于观察
a = Node(); b = Node()
a.ref = b; b.ref = a  # 制造循环引用
del a; del b          # 外部引用没了,但彼此还引着 -> 计数不归零
print("after del, still alive")   # 此刻两个对象都没被回收
print("collected:", gc.collect()) # 手动追踪式回收环
# after del, still alive
# Node freed
# Node freed
# collected: 4        <- 2 个 Node 实例 + 各自的实例 __dict__(见第 08 章),
#                         共 4 个不可达对象一起被回收
```

**`__del__`(析构钩子)不可靠,别拿它当 RAII**:触发时机取决于引用计数何时归零、gc 何时跑,不确定;解释器退出时甚至可能不调用。自 **Python 3.4(PEP 442)** 起,带 `__del__` 的循环引用**也能被 gc 回收**(会正确调用析构),不再像旧版那样卡在 `gc.garbage` 里。但即便如此,资源释放仍应用第四节的 `with`,不要赌 `__del__`——这正是"确定性释放"(`with`)和"不确定性释放"(`__del__`,依赖计数归零或 GC 扫描的时机)之间的区别,前者才是本章真正推荐的 RAII 替代方案。

**`weakref`——不增加引用计数的引用**(对标 C++ `weak_ptr`):

```python
import weakref
class C: pass
c = C()
r = weakref.ref(c)     # r 不让 c 的计数 +1
print(r() is c)        # True:目标还活着,r() 返回它
del c                  # 唯一强引用消失,c 被回收
print(r())             # None:弱引用自动失效
```

用途:缓存(不想因为缓存住了对象而阻止它被回收)、观察者模式、以及**打破循环引用**——把 `a.ref = b; b.ref = a` 中的一条边换成 `weakref`,两者就不会再形成让引用计数打转的环,不需要等 gc 介入也能被正常回收。

## 七、pymalloc:CPython 的内存池

Python 程序疯狂创建销毁小对象(每个整数、每个字符串都是堆对象)。如果每次都调 C 的 `malloc`/`free`,开销和碎片都受不了。CPython 因此自带一个**分层内存分配器 pymalloc**,专门服务 ≤ 512 字节的小对象:

```
Arena  ── 约 1 MiB 一大块,直接向操作系统 mmap 申请
  │  切成若干
Pool   ── 一个内存页大小(x86 常见 4 KB;Apple Silicon 为 16 KB),一个 pool 只服务一种 size class
  │  切成等长
Block  ── 按 16 字节对齐的等长小块:16, 32, 48, …, 512 字节,共约 32 个 size class
             一次小对象分配 = 从对应 pool 的空闲链表摘一个 block
```

申请小对象时,按大小归入某个"size class",从对应 pool 的空闲块链表里 O(1) 摘一个 block;释放时归还到空闲链表,不马上还给 OS。因此小对象的分配/释放几乎不碰系统 `malloc`,极快;同一 size class 的对象聚在一起,减少碎片。`> 512` 字节的大对象绕过 pymalloc,直接走系统 `malloc`。完全空的 pool/arena 才可能归还操作系统,所以你会看到 Python 进程内存"涨上去下不来"——很多是被内存池占着复用,并非泄漏。上述常量(arena≈1 MiB、pool=系统页大小、16 字节对齐、约 32 个 size class)可用 `python3 -c "import sys; sys._debugmallocstats()"` 现场核对,数值随版本/平台略有差异。

对照 C:这相当于 CPython 内建了一个为"海量小对象"优化的对象池 allocator,层级(arena→pool→block)类似很多游戏/高性能 C++ 项目里手写的 slab/pool allocator——原理是同一个:通用的 `malloc` 为了应对任意大小的请求,内部记账和查找的开销对"海量、等长、频繁生灭"的小对象来说不划算,专门按固定尺寸分类、批量预留再逐个分发,能把这部分开销降到接近于无。此外还有**空闲链表(free list)**:小整数、小 tuple、常用大小的 list/frame 等,销毁后其内存块被挂到专用 free list 里等着复用,连 pymalloc 那一步都省了。

## 八、全景小结

| 机制 | 作用 | C/C++ 对照 |
|------|------|-----------|
| 引用计数 | 主力,即时回收,确定性 | `shared_ptr` |
| 分代 GC | 兜底回收循环引用,基于"大多数对象朝生夕死"的分代假说 | 追踪式 GC(Java/Go 才有) |
| `weakref` | 不增计数的引用 | `weak_ptr` |
| `with` / 上下文管理器 | 确定性资源释放,是没有确定性析构时的 RAII 替代 | RAII / 析构函数 |
| `__del__` | 析构钩子(不可靠,慎用) | 析构函数(但时机不确定) |
| pymalloc | 小对象内存池 | slab/pool allocator |
| free list | 常用对象内存复用 | 对象池 |

一句话:**Python 的内存管理是"引用计数(像 shared_ptr,给你确定性)+ 分代 GC(基于分代假说,兜底环)+ 多层内存池(给你速度)"的组合;资源释放请用 `with`,别赌 `__del__`。**

## 小结

- 异常是 `BaseException` 的实例,不是错误码:错误码可以被默默忽略而程序照常往下跑,异常一旦抛出且无人处理就会中断执行并携带完整 traceback,这是两者在安全性上的根本差异。`except Exception` 按顺序匹配、`finally` 总执行、`raise ... from ...` 保留异常链,3.11 起用零成本异常表实现。
- `with` + `__enter__`/`__exit__` 是 RAII 思想在没有确定性析构的语言里的替代实现:把"离开作用域必然释放"这个保证从编译器生成的隐式调用改造成语法要求的显式协议。
- CPython 内存管理靠引用计数打底(即时确定,类似 `shared_ptr`,代价是无法处理循环引用),分代 GC 基于"大多数对象朝生夕死"的经验规律,只扫描新对象为主来兜底捡漏,`__del__` 触发时机不可靠;pymalloc 用 arena→pool→block 三层结构给小对象加速,原理与手写 slab allocator 一致。
