# 运行模型:CPython 是怎么把 .py 变成行为的

作为 C/C++ 工程师,你脑子里的心智模型大概是:`gcc` 把 `.c` 编译成汇编、再汇编成目标文件、链接成 ELF/Mach-O 可执行文件,操作系统把它 `exec` 进内存,CPU 直接执行机器码。这里没有"运行时"在中间翻译,`int x = 3;` 就是往栈上某个偏移写 4 字节。

Python 完全不是这样。理解它的运行模型,是理解后面所有"诡异"行为(别名、GIL、动态类型开销)的地基。

## 一、跑起来

Python 不需要单独的编译步骤:

```python
print("Hello, Python!")

x = 3          # 赋值,不用像 C 那样先声明类型
x = x + 1
print(x)       # 4
```

存成 `hello.py`,终端执行 `python3 hello.py`——没有"先编译出可执行文件、再运行它"这两步,`python3` 这一个程序读文件、处理、执行是一口气做完的。也可以直接敲 `python3` 进入交互式解释器(REPL),逐行输入立刻看到结果。

这个"没有独立编译步骤"的现象背后,是一个真实的设计选择:Python 从 1991 年诞生起就定位为**动态类型脚本语言**,变量的类型信息只在运行期才确定,`x + 1` 这样的表达式必须在执行到那一刻才能知道 `x` 是整数还是字符串、该调用哪个加法实现。静态编译型语言能把类型检查和方法地址在编译期定死,生成直接操作内存的机器码;动态类型语言做不到这一点,类型分派天然要推迟到运行时,于是需要一个在运行期始终存在、随时准备解释和分派的程序——这就是下面要讲的解释器。

## 二、从源码到行为:CPython 是"C 写的解释器"

先分清两个概念:

- **Python** 是一门语言,是一份语言规范(语义)。
- **CPython** 是这门语言的**参考实现**,用 C 写成。`brew install python` 装的、`python3` 跑起来的,99% 情况就是 CPython。

还有别的实现:PyPy(带 JIT)、Jython(跑在 JVM 上)、GraalPy 等。本教程凡涉及实现细节,一律以 CPython 为准,并显式标注"**在 CPython 中**"。区分语言语义与实现细节很重要——很多所谓"Python 的特性"其实只是"CPython 的实现选择",换一个实现就不成立。

| 阶段 | C/C++ | CPython |
|------|-------|---------|
| 源码 | `.c` / `.cpp` | `.py` |
| 前端 | 预处理 + 编译 | 词法/语法分析 → AST |
| 中间产物 | `.o` 目标文件(机器码) | `.pyc` 字节码(bytecode) |
| 链接 | 链接器生成可执行文件 | 无独立链接步骤 |
| 执行 | CPU 直接跑机器码 | 解释器(虚拟机)求值循环逐条解释字节码 |
| 类型检查 | 编译期 | 运行期 |

C 编译完就没编译器什么事了,程序自己在裸机上跑。CPython 编译出的字节码**不是机器码**,需要一个常驻的解释器(一个 C 写的巨大循环)逐条执行——这个解释器就是 `python` 这个可执行文件本身。**为什么 Python 要多这一层、不直接编译成机器码**:动态类型决定了运行期才能确定"这个加法该怎么做",把源码直接编译成写死类型假设的机器码行不通;字节码是一种折中——先把语法分析这类静态可确定的工作提前做完、编译成紧凑的中间表示,把真正依赖运行期信息的类型分派留给解释循环,兼顾了"不必每次重新解析源码"和"类型决定权留到运行期"两个目标。

## 三、四个阶段:源码 → AST → 字节码 → 求值

```
foo.py (源码文本)
   │  词法分析 (tokenize)
   ▼
Token 流
   │  语法分析 (parse)
   ▼
AST (抽象语法树)          <- 可用 ast 模块查看
   │  编译 (compile)
   ▼
Code Object (含字节码)    <- .pyc 里存的就是它
   │  解释执行 (ceval.c 的求值循环)
   ▼
程序行为 / 副作用
```

### 3.1 AST:语法树

Python 把编译中间态暴露给了用户层,这在 C 的工具链里通常只有编译器内部能看到:

```python
import ast

src = "x = a + b * 2"
tree = ast.parse(src)
print(ast.dump(tree, indent=2))
```

会看到 `Assign` 节点,目标是 `Name(id='x')`,值是 `BinOp(left=Name('a'), op=Add, right=BinOp(...))`——与编译原理课上的语法树一致,CPython 内部正是把这棵树编译成字节码。

### 3.2 字节码与 `.pyc`

编译产物是 **code object**,其中 `co_code` 是一串字节码指令。`import` 一个模块时,CPython 把编译结果缓存到 `__pycache__/模块名.cpython-<版本>.pyc`(版本号随解释器,如 CPython 3.14 是 `cpython-314.pyc`),下次导入若源码未改(比对 mtime/hash)就直接加载 `.pyc`,跳过重新编译。

`.pyc` **不是**机器码,不能脱离解释器运行,只是省去重新解析编译的时间,省不了解释执行的开销——这与 C 的 `.o` 表面相似(都是可缓存的中间产物),但 `.o` 里是真机器码,`.pyc` 里是给虚拟机吃的字节码,两者的复用收益完全不同量级。直接 `python foo.py` 运行的主脚本不会生成 `.pyc`(只有被 import 的模块才缓存,因为主脚本每次运行只执行一次,缓存它没有意义)。

### 3.3 用 `dis` 看字节码

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

输出(Python 3.12,不同版本指令略有差异):

```
  2           RESUME                   0

  3           LOAD_FAST                0 (a)
              LOAD_FAST                1 (b)
              BINARY_OP                0 (+)
              RETURN_VALUE
```

`LOAD_FAST 0 (a)` 把局部变量 `a` 压入**求值栈**(`FAST` 指走数组下标而非字典查找,详见[「变量与内存模型」](03-变量与内存模型.md));`LOAD_FAST 1 (b)` 同理;`BINARY_OP` 弹出栈顶两个对象做加法、结果压栈;`RETURN_VALUE` 弹出栈顶作为返回值。

CPython 是一台**栈式虚拟机**(stack machine),不是寄存器机——所有运算都围绕一个求值栈做压栈/弹栈,这与 JVM 的设计相同,与 x86 寄存器+内存的机器模型不同。**为什么选栈式而不是寄存器式**:栈式字节码的编译器更简单(表达式求值天然对应入栈出栈顺序,不需要寄存器分配这一步),这与 CPython 早期作为教学/胶水语言参考实现优先追求实现简单、正确性易验证的目标一致;代价是同样的计算通常需要更多条指令(寄存器式 VM 如 Lua 能直接引用操作数所在寄存器,省去显式的入栈出栈)。

### 3.4 求值循环(the eval loop)

CPython 执行字节码的核心是 `Python/ceval.c` 里一个巨大的循环,伪代码:

```c
for (;;) {
    opcode = NEXT_INSTRUCTION();
    switch (opcode) {
        case LOAD_FAST:  ... push(...); break;
        case BINARY_OP:  ... ; break;
        case RETURN_VALUE: ...; return;
        ...
    }
}
```

每次转一圈解释一条指令,概念上对应 CPU 的取指-译码-执行,但这层是**软件模拟**的。一条 Python `a + b` 要经过:取字节码 → switch 分派 → 查 `a` 的类型 → 找 `__add__` → 调用 → 装箱结果;C 的 `a + b` 编译期就已经确定是哪种加法,运行时可能就是一条 `add` 指令。多出来的这几步分派开销,正是"多一层解释"的直接代价,也是第二节里"动态类型必须推迟到运行期决定怎么做"这个设计选择的必然后果——不是实现得不够好,是语义要求如此。

## 四、为什么"一切皆对象,且都在堆上"

在 C 里,`int x = 3;` 中 `x` 是栈上一个 4 字节的槽,值直接内联在里面,取地址 `&x` 得到栈地址。

在 Python 里,`x = 3` 是另一回事。CPython 中**每个值都是一个 `PyObject`,分配在堆上**:

```c
typedef struct _object {
    Py_ssize_t ob_refcnt;   // 引用计数
    PyTypeObject *ob_type;  // 指向类型对象的指针
} PyObject;
```

`ob_refcnt` 记录有多少个名字/容器正引用着这个对象,归零就释放(详见[「变量与内存模型」](03-变量与内存模型.md));`ob_type` 指向"类型对象",回答"我是谁",其中记录了怎么加、怎么打印等操作——可以类比 C++ 对象里的 vtable 指针,区别是 Python 里**连整数、`None` 这种"基础值"也带**,且分派完全在运行期查找。

**为什么每个值都要背这个对象头,而不是像 C 那样让小整数直接内联存值**:根源仍是动态类型与统一的运行期自省能力——`type(x)`、`isinstance(x, T)`、给任意值调用 `x.__add__` 这类操作要求"任何值都能回答自己是什么类型、支持哪些操作",C 的 `int` 之所以能省掉这些信息,是因为类型判断已经在编译期被 `gcc` 一次性解决并烧进了机器码里,运行时不再需要"问"。Python 把类型判断推迟到运行期,就必须让每个值随身携带足够的自我描述信息,`PyObject` 头正是这份信息的最小实现。

一个具体的 `int`(`PyLongObject`)在 `PyObject` 头之后还跟着实际数值的数字位:

```
栈上的名字 x            堆上的 PyLongObject
┌─────────┐            ┌──────────────────┐
│ 指针   ─┼──────────► │ ob_refcnt = 1    │
└─────────┘            │ ob_type   ──► int │
   (一个机器字)         │ 数值: 3          │
                       └──────────────────┘
```

对照 C 的 `int x = 3`:

```
C:栈上           Python:栈上名字 + 堆上对象
┌───────┐        ┌───────┐        ┌───────────┐
│  3    │        │ 指针 ─┼──────► │ ...3(对象) │
└───────┘        └───────┘        └───────────┘
4 字节,值即数据   一个指针         一次堆分配
```

Python 的"变量"永远是**指向堆对象的引用**(本质是个指针),从不内联存值;没有"栈上的值类型",连 `1`、`True`、`None` 都是堆对象(`None` 是全局唯一单例)。一个整数对象在 64 位 CPython 上约占 28 字节而不是 4 字节,原因就是对象头的存在——这解释了 Python 数值密集计算为何内存和速度都吃亏,也是 NumPy 存在的理由(把大量数值打包进一块连续 C 数组、绕开逐个对象头的开销,见[「常用第三方库」](12-常用第三方库.md))。

```python
import sys
print(sys.getsizeof(3))         # 28    一个小整数就占 28 字节
print(sys.getsizeof(2 ** 100))  # 40    任意精度:值越大,数字位越多,对象越大
print(sys.getsizeof(2 ** 1000)) # 160
```

> 语义 vs 实现:语言只保证"一切皆对象"这一行为语义,"都在堆上""对象头是这两个字段"是 CPython 的实现。PyPy 用了不同的对象表示,配合逃逸分析,很多短命对象根本不做真正的堆分配——这进一步说明"一切皆对象"是语义约定,"都在堆上"只是其中一种满足这个约定的实现方式。

小整数缓存(`[-5, 256]`)、字符串驻留是 CPython 的另一类实现细节,留到[「变量与内存模型」](03-变量与内存模型.md)专门拆解。

## 五、CPython vs PyPy:解释 vs JIT

慢的根源是"逐条软件解释",自然的优化方向是把热点字节码即时编译成机器码——这正是 **PyPy** 做的事。

| | CPython | PyPy |
|--|---------|------|
| 执行方式 | 纯解释求值循环 | 追踪式 JIT:热点代码编译为机器码 |
| 首次/冷代码 | 无预热 | 有预热开销 |
| 长循环/热点 | 慢 | 可快数倍到数十倍 |
| C 扩展兼容 | 原生(CPython API) | 较弱(cffi 更佳) |
| 对象表示 | 每个对象老实堆分配 | 逃逸分析后可优化掉 |

思路与 JVM 的 HotSpot 一致:先解释、统计热点、把热路径编成机器码并做特化(比如假定循环里 `a`、`b` 一直是 `int`,就省掉每次的类型查找,一旦假设失败再退回通用路径)。CPython 从 3.11 起引入"专门化自适应解释器"(specializing adaptive interpreter,PEP 659),在字节码层面做轻量特化,原理相同但范围小得多——不生成机器码,只是把常见类型组合缓存下来加速分派,3.13+ 才有实验性的完整 JIT。**为什么 CPython 自己多年不做完整 JIT、而是先做这种轻量特化**:JIT 编译器本身是复杂度和维护成本很高的子系统,CPython 数十年来的历史包袱是庞大的 C 扩展生态(NumPy、Pandas 等都直接操作 CPython 的 C API 和对象内存布局),任何要动态生成机器码替换对象表示的方案都可能破坏这些扩展的兼容性,这也是 PyPy 虽然更快、却始终没能取代 CPython 成为主流实现的核心原因——生态兼容性的权重超过了纯粹的执行速度。

## 六、GIL:解释器级别的锁,不是语言语义

这个求值循环是线程安全的吗?多核能并行跑纯 Python 代码吗?

**在 CPython 中有全局解释器锁(GIL)**,同一进程内任一时刻只有一个线程在执行 Python 字节码。**为什么要有 GIL**:CPython 用引用计数做内存管理,`ob_refcnt` 的增减发生在几乎每一次对象传递、赋值、函数调用里,如果多线程真并行执行,这个计数器的读改写就需要在每个对象上加锁,不仅拖慢单线程性能,还会让本来轻量的对象操作变得笨重。1992 年 GIL 被引入时,单核 CPU 是主流硬件,"牺牲多核并行换取单线程简单高效、以及让所有 C 扩展不必自己处理线程安全"是当时合理的取舍——任何用 CPython C API 写的扩展都可以假定"同一时刻只有我的代码在动 Python 对象",不需要额外加锁。

代价是纯 Python 计算无法靠多线程吃满多核,这在多核 CPU 成为标配之后越来越成为一个实际的性能瓶颈,催生了 Python 3.13 起实验性的 no-GIL 构建(PEP 703,即 free-threading)——本质是把引用计数改造成线程安全的(如用原子操作或每线程局部计数),用更细粒度的同步换取真并行,但也带来了单线程场景下的额外开销和 C 扩展兼容性问题,这项工作至今仍在权衡取舍中推进。完整机制与绕过手段(多进程、C 扩展释放 GIL、`asyncio`)留到[「并发模型」](10-并发模型.md)。

## 小结

- CPython 把 `.py` 编译成字节码(缓存为 `.pyc`),再由 C 写的栈式虚拟机求值循环逐条解释执行,这一整层软件翻译是 Python 比 C 慢的根本原因,直接源于动态类型必须把分派推迟到运行期这一语言设计。
- 用 `ast.parse` 看语法树,用 `dis.dis` 看字节码,可以随时验证底层发生了什么。
- 一切皆对象、在 CPython 中都分配在堆上,每个对象带 `ob_refcnt`(引用计数)和 `ob_type`(类似 vtable 指针),这是"任何值都能在运行期回答自己是什么类型"这一要求的直接实现;变量是指向堆对象的引用,绝不内联存值,与 C 的栈上值类型根本不同。
- 区分 Python(语义)与 CPython(实现):PyPy 用 JIT 换速度但牺牲了部分 C 扩展兼容性;GIL 是 CPython 为简化引用计数与 C 扩展线程安全而做的历史取舍,代价是牺牲多核并行,free-threading 正在尝试用更细粒度的同步换回并行能力。
