# 数据类型

在 C 里,类型决定了变量占几个字节、按什么方式解释这几个字节;声明 `int x` 之后,`x` 在编译期就被定死为栈上 4 字节、按补码读写。在 Python 里,类型信息不附着在变量上,而附着在堆上的对象本身——变量永远只是指向某个对象的一个指针(见[「变量与内存模型」](03-变量与内存模型.md))。本章逐个拆解 Python 内置类型在 CPython 里的真实结构,以及每一处与 C 的差异背后具体解决了什么问题、放弃了什么。

## 一、内置类型与字面量

```python
n = 42              # int   整数,任意大小
f = 3.14            # float 浮点数
b = True            # bool  布尔(True / False)
z = 3 + 4j          # complex 复数(虚部后缀 j)
s = "café😀"        # str   字符串(Unicode 文本)
raw = b"hello"      # bytes 字节序列(前缀 b)
nothing = None      # 空值
```

```python
print(7 // 2)       # 3    —— 整除(向下取整)
print(7 % 2)        # 1    —— 取余
print(2 ** 10)      # 1024 —— 幂运算
print("ab" * 3)     # ababab —— 字符串重复
print(len("café"))  # 4    —— 长度(数的是字符,不是字节)
print(type(42))     # <class 'int'> —— 查类型
```

类型名本身可以当函数调用来做转换,因为它们就是类型对象,调用它相当于构造一个该类型的实例:

```python
int("42")           # 42
float("3.14")       # 3.14
str(255)            # "255"
```

## 二、对象头:所有值的公共结构

CPython 里每个值都是堆上的一个对象,开头是一段统一的对象头:

```c
// CPython 源码 Include/object.h(简化)
typedef struct _object {
    Py_ssize_t    ob_refcnt;    // 引用计数
    PyTypeObject *ob_type;      // 指向类型对象的指针(相当于 C++ 的 vptr)
} PyObject;

typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;         // 变长对象的"元素个数"
} PyVarObject;
```

`ob_type` 是 C++ 虚表指针(vptr)的类比:它指向一个 `PyTypeObject`,里面存着"这个类型怎么加法、怎么打印、怎么取长度"等一批函数指针(称为 tp_slots)。区别在于 C++ 的 vtable 在编译期写死进目标文件,而 Python 的类型对象是运行期可以被读取、甚至修改的普通对象——这正是[「运行模型」](01-运行模型.md)里"任何值都要能在运行期回答自己是什么类型"这一要求的具体落地:`ob_type` 就是那份自我描述信息的载体。

在 64 位 CPython 上,这个头本身是 16 字节(`PyObject`)或 24 字节(`PyVarObject`)。这是任何 Python 对象都要背负的固定成本,与对象实际数据大小无关:

```python
import sys
print(sys.getsizeof(0))      # 28(64 位 CPython 3.12+;3.12 之前是 24)
print(sys.getsizeof(1.0))    # 24
```

下面每种类型的内存布局都建立在这个对象头之上。

## 三、`int`:任意精度整数,不是固定宽度的字长

Python 的整数没有大小上限:

```python
x = 2 ** 100
print(x)            # 1267650600228229401496703205376,不溢出
print(x * x)         # 继续放大也没问题

print(10 // 3)      # 3   —— 整除
print(10 % 3)       # 1   —— 取余
print(-7 // 2)      # -4  —— 向下取整(floor),不是向零截断,和 C 不同
print(int("ff", 16))# 255 —— 按 16 进制解析
```

C 的整数除法向零截断(`-7 / 2` 概念上等于 `-3`);Python 的 `//` 向下取整。这不是随意选择:向下取整让 `//` 与 `%` 满足一条统一恒等式 `a == (a // b) * b + a % b` 且余数 `a % b` 的符号总与除数 `b` 一致(方便做取模索引这类场景,比如把负偏移量绕回数组另一端),向零截断则会让余数符号跟着被除数变,恒等式虽然也成立但符号规则不直观、不同语言实现还可能不一致。

Python 之所以能做到"整数不溢出",代价换来的方式是把 `int` 实现成任意精度的 bignum——本质上就是 C 程序员手写大数库(GMP 那一类)会做的事,只不过内置进了语言:

```c
// 简化后的概念结构
typedef struct {
    PyObject_VAR_HEAD          // ob_refcnt, ob_type, ob_size
    digit ob_digit[1];         // 变长数组,每个 digit 存 30 位
} PyLongObject;
```

- 每个 `digit` 是一个 32 位无符号整数(`uint32_t`),但只用其中 30 位(`PYLONG_BITS_IN_DIGIT == 30`)。留出的 2 位是为了让进位、借位运算的中间结果不会溢出 C 层 32 位寄存器,加法链式进位时才不需要额外检测溢出。
- `ob_size` 的绝对值是 digit 个数,符号表示整数正负;`ob_size == 0` 表示数值 0。这是"符号-数值"表示法,不是补码——因为任意精度数没有固定位宽,补码那套"取反加一"依赖位宽才有意义,符号位单独存反而是自然选择。

> 注:CPython 3.12 起内部改用 `ob_size` 与一个标志位打包的 `lv_tag`(属实现细节),但"30-bit digit 数组 + 符号"这个模型不变。

内存布局示意,`2**60`(需要 3 个 30-bit digit):

```
值 = digit[0] * 2^0 + digit[1] * 2^30 + digit[2] * 2^60

+-------------------+
| ob_refcnt         |  \
| ob_type  --> int  |   > 对象头 (PyVarObject)
| ob_size = 3       |  /   (正号 + 3 个 digit)
+-------------------+
| ob_digit[0] (30b) |
| ob_digit[1] (30b) |
| ob_digit[2] (30b) |
+-------------------+
```

**为什么选变长 bignum 而不是像 C 那样固定宽度**:如果坚持固定宽度(比如永远 64 位),要么像 C 一样对溢出保持沉默(结果回绕,C 标准甚至把有符号溢出定义为未定义行为),要么像某些语言那样溢出时抛错,两种做法都要求程序员在数值可能变大的地方(阶乘、组合数、密码学计算、任意精度累加)自己引入一个独立的大数库、并在整数与大数之间手动切换类型。Python 把"数值可以任意大"做成 `int` 唯一的、统一的语义,消除了这一整类"这里会不会溢出"的心智负担和潜在 bug。

代价同样是实打实的:C 的 `int a + b` 在编译期就知道操作数是寄存器宽度的定长整数,编译成一条硬件 `ADD` 指令;Python 的 `a + b` 无法在编译期假设操作数只有一个 digit,必须调用通用的多精度加法算法——按 digit 数组从低位到高位逐位相加、处理进位,结果 digit 数比原来的操作数还可能多一位,于是还要重新分配一个大小合适的新 `PyLongObject`。哪怕 `a`、`b` 只是 `1 + 1` 这种单 digit 的小数值,也要走这同一套通用路径,只是循环只跑一轮。CPython 3.11 起的专门化自适应解释器(PEP 659)能把"这条加法指令上次两个操作数都是 int"这个事实缓存下来,省掉重复的类型分派开销,但省不掉这次要真正执行的多精度加法本身——分派开销和算法开销是两回事,前者可以靠缓存优化,后者是 bignum 语义本身决定的固定成本。这正是数值密集型代码里 Python 比 C 慢一个数量级以上的根本原因之一,也是 NumPy(把数值打包进定长 C 数组、绕开每个元素的 bignum 分派)存在的理由,见[「常用第三方库」](12-常用第三方库.md)。

用 `sys.getsizeof` 直接观察这套变长结构的增长:

```python
import sys
print(sys.getsizeof(0))          # 28(空 digit 情况,依版本)
print(sys.getsizeof(2**29))      # 28  —— 1 个 digit
print(sys.getsizeof(2**30))      # 32  —— 需要 2 个 digit
print(sys.getsizeof(2**60))      # 36  —— 3 个 digit
```

每多一个 30-bit digit,大约多 4 字节,和结构定义完全吻合。

和 C 对照:

| | C `int32_t` | Python `int` |
|---|---|---|
| 大小 | 固定 4 字节 | 变长,随数值增大 |
| 溢出 | 回绕(UB/实现定义) | 不溢出,自动扩容 |
| 负数表示 | 二进制补码 | 符号位 + 数值(digit 数组) |
| 位置 | 可在栈上/寄存器 | 永远在堆上,变量持指针 |
| 运算 | 单条 CPU 指令 | 函数调用,逐 digit 处理 |

**小整数缓存**:CPython 启动时预先创建了 `[-5, 256]` 范围内的所有整数对象,永久驻留,同一数值在这个范围内永远是同一个对象:

```python
a = 100
b = 100
print(a is b)                       # True —— 同一个缓存对象
print(int("1000") is int("1000"))   # False:运行时各自新建,落在缓存范围外
```

这个缓存单纯是为了减少高频小整数(循环计数、索引、布尔值底层)反复分配释放对象的开销,是一种实现细节而非语言语义,**永远不要用 `is` 比较数值**,用 `==`。

## 四、`bool`:`int` 的子类,不是独立类型

```python
flag = True
print(not flag)         # False
print(True and False)   # False
print(3 > 2)            # True —— 比较运算返回 bool
```

Python 里 `bool` **继承自 `int`**,`True` 和 `False` 是值为 1 和 0 的两个单例整数对象:

```python
print(isinstance(True, int))   # True
print(True + True)             # 2 —— 当成整数参与运算
print(True == 1)                # True
```

**为什么 `bool` 是 `int` 的子类,而不是一个独立、和整数无关的类型**:这是历史遗留与工程简化两者共同作用的结果。Python 直到 2.3(2003 年,PEP 285)才引入真正的 `bool` 类型;在那之前,Python(以及几乎所有早期教程和已发布的代码库)一直用普通的 `0`/`1` 表示真假,大量已有代码里写着 `if x == 1`、`flag = 0`、把布尔值当整数累加计数。如果把 `bool` 设计成与 `int` 完全无关的独立类型,`True == 1` 就必须为 `False`,所有依赖"真假就是 0/1"这个既成事实的旧代码会大面积失效。把 `bool` 做成 `int` 的子类,新引入的 `True`/`False` 在任何要求整数的地方都能无缝替代原来的 `1`/`0`,向后兼容性是零成本的;同时,子类化直接复用了 `int` 已经写好的加减乘除、比较、哈希等全部运算逻辑,不需要为 `bool` 单独实现一套"布尔运算表"——`True + True == 2`、`sum(bool 列表)` 能直接当计数用,都是复用 `int` 算术的自然结果,而不是刻意设计的"特性"。

这也带来一个容易被忽视的后果:因为 `bool` 复用 `int` 的相等性和哈希,`1` 和 `True` 在 `dict`/`set` 语境下被视为同一个键:

```python
d = {1: "one", True: "yes"}
print(d)                 # {1: 'yes'} —— True 和 1 哈希、相等,被判成同一个键,后写入的覆盖前面
print(hash(True), hash(1))  # 1 1
```

这与 C 里 `bool`(`_Bool`,通常 1 字节栈上值,和 `int` 是两种不相干的类型)完全不同:Python 的 `bool` 是堆上一个继承自 `int` 的对象,`sys.getsizeof(True)` 和小整数一样是 28 字节,而不是 C 那种紧凑的 1 字节标志位。

## 五、`float`:装箱的双精度浮点

```python
f = 3.14
print(f + 1)            # 4.140000000000001
print(2.0 ** 0.5)       # 1.4142135623730951 —— 开方
print(round(3.14159, 2))# 3.14

print(0.1 + 0.2)        # 0.30000000000000004 —— 浮点误差!
print(0.1 + 0.2 == 0.3) # False —— 别用 == 比浮点
```

要精确的十进制(金融场景)用 `decimal.Decimal`,精确分数用 `fractions.Fraction`。

Python 的 `float` 就是一个 IEEE-754 双精度浮点数,被装箱(boxing)在对象里:

```c
typedef struct {
    PyObject_HEAD
    double ob_fval;      // 一个货真价实的 C double
} PyFloatObject;
```

一个 float 对象 = 16 字节对象头 + 8 字节 double = 24 字节:

```python
import sys
print(sys.getsizeof(3.14))   # 24
```

C 的 `double x = 3.14;` 是栈上 8 字节裸值,运算直接走 FPU 指令。Python 里每个 float 都是堆对象,`a + b` 要先解引用两个指针取出各自的 `ob_fval`、相加,再分配一个新的 `PyFloatObject` 装结果——这与 `int` 的情况类似,都是"一切皆对象"这一统一模型的直接代价,并不需要为 `float` 单独解释。这正是数值密集计算下纯 Python 慢、内存开销大的又一根源,也是 NumPy 存在的理由——把一亿个 double 打包进一块连续 C 数组,而不是一亿个 24 字节对象(见[「常用第三方库」](12-常用第三方库.md))。

浮点的所有"坑"和 C 完全一样,因为底层就是同一个 IEEE-754 double:`0.1 + 0.2 != 0.3` 就是明证,特殊值也一致:`float('inf')`、`float('nan')`,且 `nan != nan`。

## 六、`complex`:复数

虚部字面量后缀是 `j`,内置类型,不需要 import:

```python
z = 3 + 4j
print(z.real, z.imag)   # 3.0 4.0
print(abs(z))            # 5.0 —— 模长
```

要做复数的三角/指数运算,用标准库 `cmath`(而不是 `math`):

```python
import cmath
z = 1 + 1j
print(cmath.phase(z))    # 0.7853981633974483 —— 辐角(π/4)
```

`complex` 内部就是一个装了两个 C `double`(实部、虚部)的对象:

```c
typedef struct {
    PyObject_HEAD
    double real;
    double imag;
} PyComplexObject;
```

和 `float` 同理,只是多存一个 double。这里有一处经常被记错的行为:`(-1) ** 0.5` 并不会像 `math.sqrt(-1)` 那样报错,也不会得到 `nan`,而是**直接返回一个 `complex`**:

```python
print((-1) ** 0.5)          # (6.123233995736766e-17+1j),约等于 i
print(type((-1) ** 0.5))    # <class 'complex'>

import math
try:
    math.sqrt(-1)
except ValueError as e:
    print("math.sqrt 报错:", e)   # math domain error
```

**为什么 `**` 对负底数、非整数次幂要特殊处理成返回复数,而 `math` 模块的同类函数却直接报错**:`**` 运算符由 `float.__pow__` 实现,CPython 在其中显式检查"底数为负且指数不是整数"这一情况——这类输入在实数范围内确实无解(负数没有实数平方根),但在复数范围内有明确、良定义的主值解,于是 `float.__pow__` 主动把计算转交给复数算法,返回一个"看起来奇怪但数学上正确"的复数结果,而不是用 `nan` 掩盖一个其实可以求解的问题。`math.pow`/`math.sqrt` 则完全不做这个转换,因为 `math` 模块从设计上就是**对 C 标准库 `libm` 的一层薄封装**,契约是"只在实数范围内工作",输入超出实数定义域就该抛 `ValueError`,不应该在一个明确只处理实数的模块里悄悄冒出复数返回值——两者体现的是同一条原则的两种应用场景:`**` 作为通用运算符,要覆盖数学上一切有意义的结果;`math` 作为面向实数、追求与 C 语义对齐的数值库,宁可报错也不做类型跳变。真正要处理复数域运算,应显式使用 `cmath`。

## 七、`str`:Unicode 文本与 PEP 393

```python
s = "café😀"
print(s[0])             # c   —— 索引取单个字符
print(s[1:4])            # afé —— 切片
print(s.upper())         # CAFÉ😀
print("a" in s)          # False
print(len(s))             # 5   —— 数的是"字符(码点)",不是字节!

# s[0] = "C"            # TypeError:字符串不可变
s2 = s.replace("c", "C") # 任何"修改"都产生新字符串
```

字符串不可变,原因见[「变量与内存模型」](03-变量与内存模型.md)第四节——不可变性换来安全共享、可哈希、免于防御性拷贝,代价是任何"修改"都要构造新对象。

C 的 `char*` 是一串以 `\0` 结尾的字节;Python 的 `str` 是不可变的 Unicode 码点序列,且 CPython 会根据内容选择最省的内部编码。这套方案(**PEP 393**,Python 3.3 引入)不是唯一可能的设计,理解它之前先看另外两条路为什么被放弃:

- **方案一:字符串内部永远用 4 字节(UCS-4/UTF-32)存每个码点**。好处是任意码点都能存下,且按下标 `s[i]` 永远是 O(1)(定宽,直接乘 4 找偏移)。坏处是绝大多数真实文本以 ASCII 或拉丁字母为主,每个字符本来 1 字节就够,却要为极少数场景(比如字符串里出现一个表情符号)让**全体字符**都背 4 倍内存。
- **方案二:内部直接存变长编码(比如 UTF-8)**。好处是空间效率好,ASCII 字符仍是 1 字节。坏处是变长编码下,第 `i` 个码点在字节缓冲区里的偏移量不是 `i` 的简单函数——UTF-8 里不同字符占 1~4 字节不等,想找第 `i` 个字符必须从头逐字节数过去,`s[i]` 退化成 O(n)。这与 Python 语言里"字符串索引、切片是廉价操作"的既有假设、以及大量现有代码对这一假设的依赖直接冲突。

PEP 393 选择了两者的折中:**按每个字符串自己内容里出现的最大码点,动态选择一种定宽表示**——同一个字符串内部始终是定宽的(保证 O(1) 索引),但不同字符串可能选不同宽度(不会为了个别宽字符串拖累其它字符串的内存开销):

| 内部种类 | 每字符字节 | 可表示范围 | 例子 |
|---|---|---|---|
| Latin-1 | 1 | U+0000–U+00FF | `"hello"`, `"café"` |
| UCS-2 | 2 | U+0000–U+FFFF | `"你好"` |
| UCS-4 | 4 | U+0000–U+10FFFF | `"😀"` |

代价是显式的:构造字符串时要先扫一遍确定最大码点(一次性 O(n) 开销,通常和构造本身的扫描合并,不算额外代价);字符串里只要出现一个更宽的字符,**整个字符串**都要用更宽的表示存,哪怕只有一个字符需要那么宽;把一个 Latin-1 字符串和一个 UCS-4 字符串拼接,还需要把结果重新编码成更宽的表示(分配新缓冲、逐字符搬迁),不是简单的内存拼接。

```python
import sys
print(sys.getsizeof("a"))       # 42 —— Latin-1,每字符 1 字节(3.12+;3.12 前是 50)
print(sys.getsizeof("你"))      # 60 —— UCS-2,每字符 2 字节(3.12+;3.12 前是 76)
print(sys.getsizeof("😀"))      # 64 —— UCS-4,每字符 4 字节(3.12+;3.12 前是 80)
```

3.12 前后的差值来自 PEP 623:早期 CPython 为兼容遗留 C 扩展 API,每个字符串对象里还额外保留一份 `wstr`/`wstr_length` 缓存字段;3.12 移除了这份遗留缓存,单纯是清理历史包袱,与 PEP 393 的核心设计(按最大码点选宽度)无关。

与 C 对照:

| | C `char*` | Python `str` |
|---|---|---|
| 单元 | 字节 | Unicode 码点 |
| 长度 | 靠 `\0` 结尾,O(n) 求长 | 显式存 length,O(1) |
| `len("😀")` | 按字节数(4,UTF-8) | 1(一个码点) |
| 可变性 | 可改 | 不可变 |

`len()` 数的是码点不是字节,这点常让写惯 C 的人踩坑:

```python
s = "café😀"
print(len(s))            # 5 个码点
print(len(s.encode()))   # 9 个 UTF-8 字节:c/a/f 各 1 + é(2) + 😀(4)
```

**别把两件事搞混**:`.encode()` 得到的是 UTF-8 变长编码(每字符按码点占 1–4 字节),这是字符串**序列化到外部**时的表示,和它在内存里怎么存无关;PEP 393 说的"整串按最大码点选宽度"指的是 `str` **对象在内存里的定宽缓冲区**,服务的是 O(1) 索引这一目标。内存里用定宽存(便于 `s[i]`),`.encode()` 序列化用变长存(便于存储/传输)——这是两套各自优化不同目标的编码方案,只是都叫"编码"容易混淆。

因为不可变,字符串可以缓存哈希值,成为高效的 dict 键。CPython 还对看起来像标识符的短字符串字面量做驻留(interning),让 `is` 快速比较、省内存——但比较内容仍然只用 `==`。

## 八、`bytes` / `bytearray`:字节序列,与 `str` 严格分离

如果你想要 C 里那种"一串裸字节",对应的是 `bytes`(不可变)和 `bytearray`(可变),不是 `str`:

```python
b = b"hello"          # bytes 字面量,前缀 b
print(b[0])            # 104 —— 取出的是 int(字节值),不是字符

ba = bytearray(b"hi") # 可变版本
ba.append(33)          # 追加 '!' (0x21)
ba[0] = 72              # 原地修改 -> 'H'
print(ba)               # bytearray(b'Hi!')
```

解析二进制协议时,把字节按大小端拼成整数,不需要像 C 那样手写移位和按位或:

```python
data = b'\x00\x00\x01\x00'
print(int.from_bytes(data, "big"))     # 256   —— 大端
print(int.from_bytes(data, "little"))  # 65536 —— 小端
print((1000).to_bytes(4, "big").hex()) # 000003e8
```

`str` 与 `bytes`/`bytearray` 是两个互不隐式转换的类型,任何跨越必须显式 `encode`/`decode`:

```python
s = "café"
data = s.encode("utf-8")     # str -> bytes: b'caf\xc3\xa9'
back = data.decode("utf-8")  # bytes -> str: café

try:
    b"x" + "y"
except TypeError as e:
    print(e)          # can't concat str to bytes
print(b"x" == "x")    # False —— 不报错,但类型不同直接判为不相等,绝不做隐式解码
```

**为什么要把文本(`str`)和字节(`bytes`)在类型层面严格分开、且转换必须显式**:这是 Python 2 留下的一个真实历史教训。Python 2 里 `str` 本身就是一串字节(没有固定编码),`unicode` 才是文本类型;当 `str` 和 `unicode` 混合运算(拼接、比较)时,Python 2 会**隐式**尝试用默认编码(通常是 ASCII)把 `str` 解码成 `unicode` 再继续。只要程序里流动的字节恰好都是 ASCII,这个隐式转换悄无声息地成功,代码在开发和测试阶段一切正常;一旦某处混入了非 ASCII 字节(用户输入了中文、文件里有一个重音符号),这次隐式解码才会失败,抛出 `UnicodeDecodeError`——而这类输入往往只在生产环境、且是很久之后才出现,导致 bug 潜伏期极长、定位成本极高,是 Python 2 时代被吐槽最多、也是推动社区向 Python 3 迁移的核心痛点之一。Python 3 的修复方式是在类型系统层面彻底堵死这个口子:`str` 就是文本(始终是 Unicode 码点序列),`bytes` 就是字节(不附带任何编码假设),两者之间没有默认编码可言,任何混合操作要么直接 `TypeError`(如拼接),要么虽不报错但按类型不同判定为不相等(如 `==`),绝不会替你偷偷做一次编解码。编解码的编码方式(UTF-8?GBK?)必须由程序员在 `encode`/`decode` 调用里显式指定——这把"这段数据到底是什么编码"这个本该由人决定的问题,从一个可能被忽略、悄悄工作、某天突然爆炸的隐式假设,变成了代码里一处清晰可见、编译不通过就必须处理的显式契约。

`bytes` 是不可变字节序列,内部就是一段连续的 `unsigned char` 缓冲加长度,最接近 C 的 `const char*` + 长度;`bytearray` 是可变版本,内部像 C++ 的 `std::vector<uint8_t>`——有容量、可原地增删。

想要 C 里那种"拿一个指针指进缓冲区、零拷贝读写",对应的是 `memoryview`:它不复制底层字节,只是开在已有 buffer 上的一个视图,切片也零拷贝(类比 C++ 的 `std::span`/`string_view`):

```python
ba = bytearray(b"hello world")
mv = memoryview(ba)          # 零拷贝视图
chunk = mv[6:11]              # 切片仍是视图,指向原缓冲(没有新分配)
chunk[0] = ord('W')            # 通过视图原地改
print(ba)                     # bytearray(b'hello World') —— 原对象被改了
```

处理大块二进制(网络包、图像)时,这能避免 `data[a:b]` 那种"切一次就复制一份"的开销。

## 九、单例:`None`、`NotImplemented`、`Ellipsis`

`None` 表示"无值/空",类比 C 的 `NULL`,但它是真正的对象(类型 `NoneType`),全程只有一个实例:

```python
x = None
print(x is None)        # 判空的标准写法,永远用 is
```

`is None` 而不是 `== None` 有两个理由:一是身份比较(比指针)比调用 `__eq__` 更快;二是自定义类可以重载 `__eq__` 让任何东西都"等于" `None`,而单例的身份永远无法被伪造,`is None` 更可靠。函数没有 `return` 语句时默认返回 `None`。

`NotImplemented` 用在运算符重载里:某类型不知道怎么和对方类型运算时,返回它(而不是抛异常),让 Python 转而尝试对方的反射方法(如 `__radd__`):

```python
class Vec:
    def __init__(self, x): self.x = x
    def __add__(self, other):
        if not isinstance(other, Vec):
            return NotImplemented    # 交给对方处理
        return Vec(self.x + other.x)
```

**为什么是返回一个特殊值而不是直接抛异常**:两个互不相识的类型做运算时(比如 `Vec + Matrix`),`Vec.__add__` 并不知道 `Matrix` 是否有能力和自己相加——如果它抛异常,Python 就没有机会再去问 `Matrix.__radd__` 是否知道怎么处理;`NotImplemented` 让"我不会算"和"这次运算彻底失败"分离成两个独立的判断,前者由类型自己声明,后者由 Python 在两边都声明"不会"之后才真正抛出 `TypeError`。这使得两个完全不认识彼此的类型之间仍有机会协商出一种运算方式,而不必谁提前知道对方的存在。

`Ellipsis`(`...`)就是三个点,常用作占位符,或被 NumPy 用作多维切片(`arr[..., 0]`):

```python
def todo(): ...          # 合法的函数体占位(等价于 pass)
print(... is Ellipsis)   # True
```

## 十、数值与字符串转换

显式构造/转换用类型名当函数(其实是调用类型对象的构造器):

```python
int("42")           # 42        —— str -> int
int("ff", 16)        # 255       —— 指定进制
int(3.9)              # 3         —— float 截断(向零取整)
float("3.14")        # 3.14
str(255)              # "255"
bin(255)              # "0b11111111"
hex(255)              # "0xff"
```

格式化(f-string,3.6+)是最常用的数值转字符串方式:

```python
n = 255
print(f"{n:d} {n:b} {n:x} {n:08b}")   # 255 11111111 ff 11111111
pi = 3.14159
print(f"{pi:.2f}")                     # 3.14
print(f"{1234567:,}")                  # 1,234,567 —— 千位分隔
```

转换失败会抛 `ValueError`,而不是像 C 的 `atoi` 那样静默返回 0:

```python
try:
    int("abc")
except ValueError as e:
    print("转换失败:", e)
```

这种"出错就抛异常、拒绝静默失败"的取向贯穿整个类型系统——除零抛 `ZeroDivisionError`、类型不匹配抛 `TypeError`、转换失败抛 `ValueError`——详见[「错误处理与资源管理」](09-错误处理与资源管理.md)。

## 小结

- 所有值共享 `PyObject`/`PyVarObject` 对象头(引用计数 + 类型指针),这是"任何值都要能在运行期回答自己是什么类型"这一要求在内存布局上的直接体现。
- `int` 是变长 bignum(30-bit digit 数组 + 符号),用"每次运算都走通用多精度算法"的代价换取"永不溢出"的语义;`[-5, 256]` 有缓存单例,判等只用 `==`。
- `bool` 是 `int` 的子类,源于 Python 2.3 才引入 `bool` 时对已有"0/1 表示真假"代码的兼容需求,顺带复用了 `int` 全部的运算与哈希逻辑,代价是 `1` 与 `True` 会被 dict/set 视为同一个键。
- `float`/`complex` 是装箱的一个/两个 C `double`;`**` 对负底数、非整数次幂特意转去算复数结果,`math` 模块则严格限定在实数域直接报错,二者服务不同契约。
- `str` 是不可变 Unicode 码点序列,PEP 393 按每个字符串自身的最大码点动态选 1/2/4 字节定宽存储,在"永远 4 字节省心但费内存"与"变长编码省空间但索引变 O(n)"之间取得 O(1) 索引与空间效率的折中。
- `bytes`/`bytearray` 才对应 C 的字节数组,与 `str` 严格分离、转换必须显式——这是对 Python 2 隐式编解码酿成的生产期 `UnicodeDecodeError` 的直接修正;`memoryview` 提供零拷贝视图。
