# Basic Syntax: An Adaptation Guide for Those Coming from the Land of Braces

[Chapter 01](01-运行模型.md) already established the execution model where "every value is a heap object." This chapter deals with surface syntax—no braces, no semicolons, no type declarations, no `main`—and the design considerations behind every one of those differences.

Start with a snippet to get the overall shape of things:

```python
name = "Ann"
age = 20

print("Hello,", name)          # Hello, Ann

if age >= 18:
    print("adult")
else:
    print("minor")

for i in range(3):
    print(i)

def add(a, b):
    return a + b

print(add(2, 3))
```

`name = "Ann"` is a variable assignment—no need to declare a type first like in C; `if/else` uses a colon plus indentation instead of `{}`; `for i in range(3)` loops 3 times; `def add(a, b): return a + b` defines a function. Below, we go through these constructs one by one against C/C++ and unpack the mechanism and trade-offs behind each.

## 1. Indentation Is the Block: Scope Enforced at the Syntax Level

C/C++ uses `{}` to delimit blocks; indentation is purely cosmetic, and the compiler doesn't check it:

```c
if (x > 0) {
    foo();
    bar();
}
```

Python uses **indentation itself** to delimit blocks—no braces—and a statement ends at a newline (no semicolons):

```python
if x > 0:
    foo()
    bar()
```

A colon `:` introduces a block, and the block body must be indented (4 spaces by convention); every statement within the same block must be indented identically or you get an `IndentationError`; you cannot mix tabs and spaces—CPython errors out directly.

During lexical analysis, indentation gets turned into `INDENT`/`DEDENT` tokens (recall the tokenize stage in [Chapter 01](01-运行模型.md)), playing exactly the role that `{`/`}` play in C—it isn't a style suggestion, it's a syntactic token; missing the correct indentation is the same class of error as forgetting a brace in C.

**Why enforce block structure through indentation instead of decoupling indentation from syntactic structure the way C does?** This is a direct expression of Python's design philosophy that "readability is syntax." C allows indentation and the actual block structure to diverge (a line can look like it belongs to an `if` block while actually not, because a `{}` was left out)—this "looks like it's in the block, actually isn't" mismatch is a real, documented source of bugs (Apple's famous `goto fail` SSL vulnerability was exactly an extra line that visually looked like it was inside the `if` block but, for lack of braces, actually sat outside it). Python makes indentation itself part of the grammar, eliminating the possibility that "visual structure" and "actual structure" diverge, at the cost of some formatting freedom (you can't write a whole `if` statement on one line, except with a special single-line form).

There's one important semantic difference here: C's `{}` introduces a new **lexical scope**—a variable declared inside a block ceases to exist once you leave the block. Python's indented blocks **do not introduce a new scope**—names bound inside an `if`/`for`/`while` block are still visible after the block ends:

```python
if True:
    y = 10
print(y)   # 10, legal; y isn't "destroyed on block exit"
```

In Python, the only things that truly introduce a new scope are functions, classes, modules, and comprehensions—not every indented block. This isn't an oversight; it's the design choice that "indentation only expresses grouping of control flow, not namespace boundaries"—namespace granularity is pinned to the function rather than to an arbitrary code block, for reasons covered in [Functions and the Call Stack](06-函数与调用栈.md).

## 2. Dynamic Typing, Strong Typing, No Declarations Needed

Three orthogonal concepts:

- **No declaration needed**: a variable isn't declared with a type ahead of time; it "comes into being" on first assignment.
- **Dynamic typing**: the type belongs to the **object**, not the variable. The same name can point to an int, then later to a str; type checking happens at runtime (recall from [Chapter 01](01-运行模型.md): `ob_type` hangs on the object, not the variable).
- **Strong typing**: no implicit, reckless conversions. `1 + "2"` raises `TypeError` outright.

```python
x = 42          # x points to an int object
x = "hello"     # x points to a str object; perfectly legal
x = [1, 2, 3]   # now points to a list object

# 1 + "2"       # TypeError: unsupported operand type(s)
1 + int("2")    # 3, works only with an explicit conversion
```

Compared to C++: C++ is static plus strongly typed—once `int x` is fixed, `x = "hi"` won't compile. Python defers type checking to runtime, buying the flexibility that "the same piece of code can handle any type, as long as that type supports the required operations" (the basis of duck typing), at the cost that many type errors can't be caught at compile time and only surface at runtime, or need to be caught statically with type annotations plus `mypy` (see [Object Orientation and the Type System](08-面向对象与类型系统.md)). **Strong typing rather than weak typing** is likewise a deliberate choice: C's implicit conversion rules (e.g. `int` auto-promoting to `float`) make code more concise to write, but also create plenty of scenarios where "it looks like it computes, but the result is unexpected." Python chooses to raise an error rather than perform an automatic conversion with no unambiguous rule—the same design thread as "if it can deterministically be flagged as an error, don't leave it as undefined behavior" seen elsewhere.

What reassigning a variable really means underneath: `x = "hello"` does **not** overwrite the int object's memory with a string; it makes the name `x` point to a new heap object, while the old object's reference count drops by one and it may get reclaimed. This is a central theme of the whole book, and [Chapter 03](03-变量与内存模型.md) walks through it with memory diagrams.

## 3. No main: Modules Execute Top to Bottom

A C/C++ program's entry point is `main`. Python has no mandatory entry point—when a module loads, its top-level statements execute in order from top to bottom, and `def` itself is a statement that "creates a function object when executed," not a compile-time declaration.

```python
def main():
    ...

if __name__ == "__main__":   # __name__ == "__main__" only when run directly
    main()
```

When this module is `import`ed instead, its `__name__` is the module's name rather than `"__main__"`, the condition is false, and `main()` never runs—importing it only gets you the function definitions, without triggering side effects.

**Why use a runtime check like this instead of dedicated entry-point syntax the way C does?** Python was positioned from the start so that any file could either be run directly as a script or imported as a library—rather than enforcing "one program, exactly one entry point." This requires a way for the same piece of code to behave differently depending on how it's used—`if __name__ == "__main__"` achieves exactly this by piggybacking on the existing mechanism that "module-level code executes anyway." There's no need to invent two separate sets of syntax for "script" versus "library"; an ordinary `if` handles it. This is a good example of Python's habitual style of reusing an existing mechanism rather than designing a new concept.

## 4. print / input and Comments

```python
print("hello", "world")          # space-separated by default, newline at end -> hello world
print("no newline", end="")      # custom ending
name = input("name? ")           # reads a line, strips the newline; return value is always str
```

`print` (since Python 3) and `input` are both ordinary functions, not language keywords or statements—this is itself a design trade-off: in Python 2, `print` was a statement (`print "hello"`, no parentheses); 3.0 turned it into a function, which means `print` can now be assigned, passed around, and overridden like any other function, without needing a dedicated grammar rule just for it. `input`'s return value is always `str`; to get a number you must explicitly write `int(input())`—there's no `scanf("%d")`-style format-specifier parsing, consistent with the strong-typing stance from earlier: conversion must be explicit and visible.

## 5. Operators: the Biggest Divergences from C

Most operators (`+ - * / % == < > && ||`) carry familiar meaning, but Python uses the keywords `and`/`or`/`not` in place of `&& || !` (`& | ^ ~ << >>` are still bitwise operators).

| Operator | Meaning | C/C++ comparison |
|--------|------|-----------|
| `/` | True division, always returns float | `5/2 == 2.5` (in C, integer division gives `2`) |
| `//` | Floor division | C's integer division truncates toward zero |
| `**` | Power | No equivalent; C needs `pow()` |
| `and`/`or` | Logical and/or (short-circuiting) | `&&`/`\|\|` |

```python
5 / 2     # 2.5
5 // 2    # 2
-5 // 2   # -3   floors, toward negative infinity
5 % 2     # 1
-5 % 2    # 1    sign follows the divisor; in C, -5 % 2 == -1
2 ** 10   # 1024
```

**Why does `/` always return float, requiring a separate `//` for integer division?** In Python 2, `/` behaved like C: dividing two ints truncated to an int. This was considered a significant design flaw—if a variable somewhere in a piece of code got changed from `int` to `float` (or vice versa), `/`'s behavior would silently change along with it, while every call site stayed exactly the same. This "semantics change silently when the type changes" behavior is fertile ground for bugs. Python 3's [PEP 238](https://peps.python.org/pep-0238/) fixed `/` as true division (always returning a float, or a value-appropriate type), and made `//` mandatory whenever integer division is actually wanted—using two distinct symbols to express two distinct intents, so "do I actually want truncation" is visible right there in the code, instead of depending on the operands' types.

**Why does the modulo's sign follow the divisor instead of the dividend the way it does in C?** This follows directly from how `//` is defined. Python's `//` **floors** (rounds toward negative infinity); C's integer division **truncates** (rounds toward zero). Which rounding rule you pick determines the remainder's sign convention, because the language guarantees the identity `a == (a // b) * b + (a % b)`. Python chooses floor division (matching the preference in Knuth's *The Art of Computer Programming*), which pays off in making modulo arithmetic more intuitive under "wraparound" semantics—for instance, computing the previous index with `(i - 1) % n` correctly wraps around to `n - 1` even when `i` is 0; C's truncating division gives a negative remainder when `i` is negative, and wraparound logic needs an extra conditional to fix it up.

Integers also have arbitrary precision: `2 ** 200` computes the exact value directly, without overflow-wraparound the way C's `int32`/`int64` would ([Chapter 01](01-运行模型.md) mentioned that `PyLongObject` is a variable-length heap object). **Why go with arbitrary precision instead of a fixed bit width like C?** A fixed bit width lets an integer map directly onto a CPU register and complete an operation in a single machine instruction—that's the storage cost C accepts in exchange for staying close to the hardware and predictable. Python chooses never to overflow, taking the question of "how many bits does this number actually need" out of the programmer's hands, at the cost that every integer operation has to run through a general variable-length-digit algorithm instead of a single CPU instruction—an order of magnitude slower—in exchange for numeric computation never silently going wrong because you picked the wrong type width.

```python
x = 2 ** 200
print(x)              # 1606938044258990275541962092341162602522202993782792835301376
print(x.bit_length()) # 201
```

## 6. f-strings: Formatting Expanded at Compile Time

```python
name, n = "Ann", 3
print(f"{name} has {n} apples")        # Ann has 3 apples
print(f"pi = {3.14159:.2f}")           # pi = 3.14
print(f"{n=}")                         # n=3, for debugging, auto-includes the variable name
```

An f-string is broken down into "fetch value + format + concatenate" bytecode at **compile time**, not parsed as a format string at runtime:

```python
import dis
dis.dis('f"{name} has {n}"')
```

```
  0           RESUME                   0

  1           LOAD_NAME                0 (name)
              FORMAT_SIMPLE
              LOAD_CONST               0 (' has ')
              LOAD_NAME                1 (n)
              FORMAT_SIMPLE
              BUILD_STRING             3
              RETURN_VALUE
```

Each `{}` maps to one `FORMAT_SIMPLE`, and the final `BUILD_STRING` joins the pieces into a single string. **Why parse the format string at compile time rather than runtime?** The old `%` formatting and `str.format()` both have to re-parse the format string on every call (locating `%s`/`{}` placeholders, matching them to arguments)—that's avoidable repeated work. f-strings move this step to a one-time job at compile time; at runtime it's just fetching values and concatenating them in an already-determined order, eliminating the "re-parse on every call" overhead. That's the direct reason it's now recommended as the preferred formatting style, not merely that it's more compact to write.

## 7. Booleans, None, and Truthiness Testing

```python
flag = True          # capitalized
nothing = None        # the null-value singleton
```

`True`/`False` are of type `bool`, which in CPython is a subclass of `int` (`True == 1`); `None` is a globally unique singleton object, and equality against it should use `is` (`if x is None`, not `== None`)—consistent with the "variables are references" model from earlier: `is` compares "is this the same object," and since there's only ever one `None`, identity comparison is both correct and faster than invoking `__eq__`.

**Truthiness testing**: any object can go into an `if`. C treats "nonzero as true"; Python is richer:

| Falsy | Truthy |
|--------------|---------------|
| `False`, `None` | everything else |
| `0`, `0.0` | nonzero numbers |
| `""`, `[]`, `{}`, `()` | non-empty containers |

```python
items = []
if items:                # an empty list is falsy; no need to write len(items) == 0
    print("in stock")
else:
    print("empty")       # this branch runs
```

`if x` calls the object's `__bool__` (falling back to `__len__` if absent, testing whether it's 0). **Why define "empty containers" as falsy too?** This unifies, at the language level, "does this container have contents" and "is this number zero" into one shared notion of "is there anything meaningful here," letting `if items:` directly express the intent "if there's something," without having to check the length separately for every container type—in essence it generalizes C's "nonzero is true" rule from "is the numeric value nonzero" to "is the object non-empty/meaningful," covering more ground while preserving the same underlying intuition. The full truthiness-testing order and custom examples are in [Control Flow](05-控制流.md).

## 8. Chained Comparisons and Multiple Assignment

```python
x = 5
if 0 < x < 10:           # reads as (0 < x) and (x < 10)
    print("in range")
```

In C, `0 < x < 10` gets parsed as `(0 < x) < 10`, i.e. `(0 or 1) < 10`, which is almost always true—a real trap arising from C's operator associativity. Python defines chained comparison to be a genuine chain in the mathematical sense (equivalent to joining each adjacent pair of comparisons with `and`, with each subexpression evaluated only once), eliminating this trap.

```python
a, b = 1, 2               # bind simultaneously
a, b = b, a                # swap, no temporary variable needed
first, *rest = [1, 2, 3, 4]   # first=1, rest=[2,3,4]
```

What's underneath `a, b = b, a`: the two values on the right side are each evaluated and pushed onto the stack first, then bound to the names on the left—no temporary variable is needed, and there's none of C's worry about whether "assigning to `a` first affects the later read of `b`." CPython has a dedicated optimization for small swaps like this: 3.11–3.12 use a single `SWAP` instruction; 3.13+ goes further with the super-instructions `LOAD_FAST_LOAD_FAST` + `STORE_FAST_STORE_FAST`, dropping even `SWAP`—only earlier versions, or swaps with more elements, fall back to the general-purpose `BUILD_TUPLE` + `UNPACK_SEQUENCE`. Measured on 3.14 here:

```python
import dis
def swap(a, b):
    a, b = b, a
dis.dis(swap)
```

```
  3           RESUME                   0

  4           LOAD_FAST_LOAD_FAST     16 (b, a)
              STORE_FAST_STORE_FAST   16 (b, a)
              LOAD_CONST               0 (None)
              RETURN_VALUE
```

## 9. C/C++ ↔ Python Syntax Cheat Sheet

| Topic | C/C++ | Python |
|------|-------|--------|
| Block delimiter | `{ }` | Indentation (INDENT/DEDENT) |
| Statement end | `;` | Newline |
| Variable declaration | `int x;` required | Just `x = ...` |
| Type | Static, bound to the variable | Dynamic, bound to the object |
| Entry point | `main()` | Top-level statements; `if __name__=="__main__"` |
| Logical operators | `&& \|\| !` | `and or not` |
| Division `/` | Truncates between integers | Always true division; `//` for integer division |
| Modulo sign | Follows the dividend | Follows the divisor |
| Integer overflow | Yes (wraparound/UB) | None, arbitrary precision |
| Ternary | `c ? a : b` | `a if c else b` |
| `++`/`--` | Yes | None, use `x += 1` |
| Null value | `NULL`/`nullptr` | `None` (an object, tested with `is`) |

**Why no `++`/`--`?** C's increment/decrement operators can be embedded inside an expression and produce a side effect there (`a[i++] = i`), and "pre- vs. post-increment give different values" plus "evaluation order of multiple increments within the same expression is undefined" are long-standing, widely criticized sources of C bugs. Python's design principle is "one statement does one clear thing," so it simply doesn't provide an operator that can be embedded in an expression and carries its own side effect—incrementing is written as the explicit statement `x += 1`. This sacrifices a bit of terseness in exchange for never having to wonder "exactly when in the expression does this increment take effect."

## Summary

- Indentation is a syntactic token (turned into INDENT/DEDENT at compile time) that replaces braces, forcibly unifying "visual structure" with "actual structure" and eliminating a class of bug that C is prone to; but an indented block itself doesn't introduce a new scope—only functions/classes/modules/comprehensions do.
- Dynamically and strongly typed: the type is bound to the object, checked at runtime, with no implicit conversions—trading compile-time safety for runtime flexibility, partially recoverable with type annotations.
- `/` is always true division, `//` is a dedicated integer-division operator, and modulo's sign follows the divisor—all products of two principles: "semantics shouldn't silently change when a type changes" and "floor division makes modulo more intuitive with negative numbers." Arbitrary-precision integers buy generality by never overflowing due to a wrong choice of type width, at the cost that arithmetic is no longer a single machine instruction.
- f-strings parse the format string at compile time rather than runtime, avoiding repeated-parsing overhead; truthiness testing generalizes "nonzero is true" to also cover "empty containers"; chained comparisons and `a, b = b, a` both eliminate specific traps that arise in C from operator associativity or evaluation order.
