# Object Orientation and the Type System

This chapter covers Python's object-oriented machinery: classes, inheritance, and methods. **First we cover how to define a class, how to inherit, and how to use these features—no low-level knowledge required to follow along; then we get into the internals—an object's memory layout, the attribute lookup order (MRO), and the descriptor protocol behind method binding—comparing each one with C++'s object model.** If you just want to be able to write classes, reading each section's "How to Use It" is enough; if you want to understand "why this works" and the performance costs, keep reading "Internals vs. C++" below that.

> C++'s object model is "compile-time-fixed memory layout + vtable"; Python's object model is "runtime-mutable dictionary + descriptor protocol." Prerequisite: [see "Variables and the Memory Model"](03-变量与内存模型.md)—everything is an object, a variable is a reference.

---

## 0. Get Familiar with Usage First: Defining Classes and Inheritance at a Glance

Use `class` to define a class; `__init__` is the constructor method (automatically called when an instance is created); `self` represents "this particular instance":

```python
class Dog:
    def __init__(self, name, breed):
        self.name = name       # instance attribute
        self.breed = breed

    def bark(self):            # instance method, first parameter is always self
        return f"{self.name} says Woof!"

d = Dog("Rex", "Labrador")
print(d.name, d.breed)         # Rex Labrador
print(d.bark())                 # Rex says Woof!
```

Inheritance: a subclass lists its parent class in parentheses, and can override the parent's methods:

```python
class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return "..."

class Cat(Animal):              # Cat inherits from Animal
    def speak(self):            # override the parent's speak
        return f"{self.name} says Meow"

c = Cat("Tom")
print(c.speak())                # Tom says Meow
print(isinstance(c, Animal))    # True —— Cat is also an Animal
```

This is the everyday skeleton for writing classes: `class` + `__init__` + methods + inheritance. Below we go through each feature one at a time—**usage first, internals second**.

## 1. `class` Syntax and Instance Memory Layout

### How to Use It

Besides instance attributes, a class can also have **class attributes**—written in the class body, outside any method, shared by all instances:

```python
class Point:
    count = 0                 # class attribute (shared by all instances)
    def __init__(self, x, y):
        self.x = x            # instance attribute, a separate copy per instance
        self.y = y
        Point.count += 1      # modify the class attribute via the class name

p = Point(3, 4)
print(p.x, p.__dict__)        # 3 {'x': 3, 'y': 4}
q = Point(1, 1)
print(Point.count)             # 2 —— both instances share the same count
```

### Internals vs. C++

C++'s objects are a contiguous block of memory, with fields at **fixed offsets set at compile time**:

```
C++  struct Point { int x; int y; };
p:  [ x=3 ][ y=4 ]      // accessing p.x is *(int*)(&p + 0); the offset is hardcoded into the instruction
```

By default, each Python instance carries a `__dict__` (a dict), a mapping from attribute names to object references:

```
Python instance p (heap object)
+------------------+
| ob_refcnt        |
| ob_type -> Point |
| __dict__ ------- |----> {'x': -> PyLong(3), 'y': -> PyLong(4)}
+------------------+
```

Consequences:

- Accessing `p.x` is a **dictionary lookup** ([see "Composite Data Structures"](07-复合数据结构.md) for the hash table involved), not a constant-offset dereference, so it's slower.
- You can add or remove attributes on any instance at runtime: `p.z = 5` just stuffs a key into its `__dict__`. In C++ an object's set of fields is fixed at compile time; you can't do this.
- Each instance carries an extra dict, a nontrivial memory overhead. To eliminate it, see [`__slots__`](#slots__-replacing-the-dict-with-a-fixed-array) later.

> CPython has an optimization called "key-sharing dict": multiple instances of the same class, if they have the same set of attributes, can share the key portion and each store only a value array, saving memory. This makes "instance dicts" less expensive, but still not as good as fixed offsets.

## 2. Classes Are Objects Too, and `type` Is the Metaclass

### How to Use It

`type(obj)` lets you check what class a value belongs to, which is common when debugging or writing generic code:

```python
print(type(p))         # <class '__main__.Point'>   —— p's type is Point
print(type(42))        # <class 'int'>
print(isinstance(p, Point))   # True
```

### Internals vs. C++

In Python, **a class itself is also an object**, whose type is `type`:

```python
print(type(Point))     # <class 'type'>              —— Point's type is type
print(type(type))      # <class 'type'>              —— type's type is itself
```

- `Point` is an object living on the heap; you can assign it, pass it, and put it in a list.
- The "class" that creates classes is called the **metaclass**, by default `type`. `class Point: ...` is essentially equivalent to calling `type(name, bases, namespace)` to dynamically build a class object:

```python
Point2 = type('Point2', (object,), {'x': 0})   # equivalent to the class statement
```

This is entirely unlike C++: a C++ class is only a compile-time concept, and there is no "class object" at runtime (RTTI's `type_info` is just read-only metadata and can't be used as a value to construct new classes). Python's classes are first-class citizens and can be generated and modified at runtime. Metaclasses are an advanced feature (ORMs and frameworks use them for registration/validation); you don't write them by hand day to day.

## 3. Attribute Lookup Order: Instance → Class → MRO

### How to Use It

Normally, when accessing `obj.attr`, if the instance itself has the attribute it's used; otherwise Python searches the class (and its parents)—this is how "inheritance" shows up for attributes/methods. Day to day you just write code by intuition:

```python
class Base:
    label = "base"

class Child(Base):
    pass

obj = Child()
print(obj.label)     # base —— neither the instance nor Child has label, found in Base along the inheritance chain
obj.label = "child"  # set an attribute of the same name directly on the instance
print(obj.label)      # child —— the instance's own attribute takes priority
print(Base().label)   # base —— didn't affect the Base class itself
```

### Internals vs. C++

When accessing `obj.attr`, the interpreter searches in a fixed order. This is the core mechanism for understanding OOP:

1. **Data descriptors** (class attributes that define `__set__`/`__delete__`, such as `property`)—take precedence over the instance dict;
2. **The instance `__dict__`**—`obj.__dict__['attr']`;
3. **The class and its MRO**—search each class's `__dict__` along `type(obj).__mro__`; a non-data descriptor encountered on the way (such as a plain function) triggers the descriptor protocol;
4. None found → call `__getattr__` (if defined) → otherwise `AttributeError`.

(Detailed order: data descriptor > instance dict > non-data descriptor / class attribute > `__getattr__`.)

### MRO and C3 Linearization

With multiple inheritance, the "class chain" is not a simple depth-first traversal but a definite order computed by the **C3 linearization** algorithm, guaranteeing: subclasses before base classes, the relative order of base classes preserved, and a unique result (or an error).

```python
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

print([c.__name__ for c in D.__mro__])
# ['D', 'B', 'C', 'A', 'object']
```

C3 solves C++'s notorious diamond inheritance problem. C++ requires manual handling of duplicate base classes via `virtual` inheritance; Python automatically produces a linear order via the MRO, and `super()` walks along it (see section 6).

## 4. Method = Function + Descriptor Protocol: Where Bound Methods Come From

### How to Use It

When defining a method, the first parameter is written as `self`; when calling it, you **don't** pass it manually—Python automatically passes in the instance the method was called on:

```python
class Foo:
    def bar(self, n):
        return f"self is {self}, n is {n}"

f = Foo()
print(f.bar(10))     # self is <__main__.Foo object at ...>, n is 10
```

### Internals vs. C++

A "method" is really just an **ordinary function** defined in the class's `__dict__`. The magic is that a function is a **descriptor** (it implements `__get__`).

```python
print(Foo.bar)      # <function Foo.bar at ...>       —— accessed via the class, it's a plain function
print(f.bar)        # <bound method Foo.bar of ...>   —— accessed via an instance, it's a "bound method"
```

What happened: `f.bar` triggers attribute lookup, finds the function object `bar` in the class, and because a function has `__get__`, the interpreter calls `bar.__get__(f, Foo)`, which returns a **bound method object**—a callable wrapper that pre-fills `f` into the first parameter (`self`):

```
f.bar()  is equivalent to  Foo.bar(f)
         the bound method internally stores (__func__ = bar, __self__ = f)
         at call time, __self__ is automatically passed as the first argument
```

This explains why a method's first parameter must be written explicitly as `self`—it's not a keyword, just the conventional name for "the parameter that gets auto-filled at binding time." Compared to C++: C++'s `this` is a hidden pointer implicitly passed by the compiler; Python makes this explicit as a descriptor-protocol step performed at runtime.

## 5. `@property` / `@staticmethod` / `@classmethod`

### How to Use It

These three decorators change a method's "binding behavior," corresponding respectively to "a method accessed like a field," "a method that needs no self," and "a method that operates on the class itself":

```python
class Circle:
    def __init__(self, r):
        self._r = r

    @property                       # disguises a method as a read-only attribute, called automatically on access
    def area(self):
        return 3.14159 * self._r ** 2

    @staticmethod                   # binds neither self nor cls; just a plain function in the class namespace
    def unit():
        return "cm"

    @classmethod                    # binds the "class" rather than an instance; the first parameter is cls
    def from_diameter(cls, d):
        return cls(d / 2)           # cls makes a subclass call return a subclass instance (polymorphic construction)

c = Circle(2)
print(c.area)                       # 12.56636 —— accessed like a field, calls a method underneath
print(Circle.unit())                # cm
print(Circle.from_diameter(10)._r)  # 5.0
```

| Decorator | First parameter | When called via an instance | Typical use |
|--------|-----------|--------------|---------|
| plain method | `self` (instance) | binds the instance | regular behavior |
| `@classmethod` | `cls` (class) | binds the class | alternative constructors, manipulating class state |
| `@staticmethod` | none | no binding | logic that belongs to the class but needs no self/cls |
| `@property` | `self` | access triggers the call | computed attributes, read-only encapsulation, added validation |

### Internals vs. C++

`@property` is language-level support for C++'s "encapsulate a private field with getter/setter"—but it's **transparent** to the caller: `c.area` looks like a field access, so a field can be upgraded to a computed attribute without changing calling code. It's able to intercept access ahead of the instance `__dict__` precisely because `property` implements `__set__`, making it the "data descriptor" from section 3—ranked ahead of the instance dict in the lookup order.

## 6. Inheritance and `super`

### How to Use It

A subclass's `__init__` uses `super().__init__(...)` to call the parent's constructor logic, instead of hardcoding the parent's name:

```python
class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return "..."

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)      # call the parent Animal's __init__
        self.breed = breed
    def speak(self):
        return "Woof"

d = Dog("Rex", "Labrador")
print(d.name, d.breed, d.speak())   # Rex Labrador Woof
```

### Internals vs. C++

`super()` (the argument-less form in Python 3) does not mean "call the parent class" but "find the next class along the current instance's **MRO**." This is essential in multiple inheritance—it ensures that in cooperative multiple inheritance each base class's method is called exactly once, chained in C3 order. This is more flexible than hand-writing `Base::method()` (hardcoding a specific base class) in C++, and it avoids a base class being called twice in diamond inheritance.

**Cooperative multiple inheritance, verified: in a diamond, `A.__init__` is called only once**, as `super()` advances strictly along the MRO `[D, B, C, A]`:

```python
class A:
    def __init__(self): print("A")
class B(A):
    def __init__(self): print("B"); super().__init__()
class C(A):
    def __init__(self): print("C"); super().__init__()
class D(B, C):
    def __init__(self): print("D"); super().__init__()

D()
# D
# B
# C
# A   <- only once; had you hardcoded A.__init__(self) instead of super(), A would run twice
```

## 7. Duck Typing vs. C++ Virtual Functions / Static Typing

### How to Use It

When writing Python functions, you typically don't need to declare a parameter's type or a common base class—as long as the object passed in has the methods/attributes the function uses, it works:

```python
class Duck:
    def speak(self): return "Quack"

class Person:
    def speak(self): return "Hello"

def make_it_speak(x):        # needs no common base class or interface declaration
    return x.speak()         # looks up the method by x's actual type at runtime

print(make_it_speak(Duck()))     # Quack
print(make_it_speak(Person()))   # Hello
```

"If it walks like a duck and quacks like a duck, treat it as a duck"—this is **duck typing**.

### Internals vs. C++

C++'s polymorphism relies on a **virtual function table (vtable)**: a vptr hidden in the object header points to the vtable, a virtual call is "look up the table + indirect jump," and types are checked at compile time—only classes declaring a common base can be polymorphic.

Python is duck typing: when you call `x.speak()`, the interpreter doesn't care what type `x` is at all; it just looks up the `speak` attribute in `x`'s type at runtime and calls it if found.

Comparison:

| | C++ virtual function | Python duck typing |
|---|-----------|----------------|
| Dispatch timing | interface fixed at compile time, vtable looked up at runtime | entirely at runtime, looking up the type's dict |
| Requires common base | yes (inheritance + virtual) | no (a same-named method suffices) |
| Type checking | static at compile time | at runtime; raises `AttributeError` only when it's wrong |
| Overhead | one indirect jump | one dictionary lookup (slower) |

The cost is "no declared interface, errors deferred to runtime." For a middle ground, see abstract base classes in section 10, and [type annotations + mypy](12-常用第三方库.md) for static checking.

## 8. `__slots__`: Replacing the dict with a Fixed Array

### How to Use It

The default instance `__dict__` is flexible but memory-hungry. If a class will create huge numbers of instances (say, a million Points), you can declare a fixed set of fields with `__slots__`:

```python
class Point:
    __slots__ = ('x', 'y')      # after this: no more __dict__, fields live in fixed slots
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = Point(3, 4)
# p.z = 5          # AttributeError: can no longer add attributes dynamically
# p.__dict__       # AttributeError: there is no __dict__ anymore
```

### Internals vs. C++

The underlying change: CPython allocates a **fixed-offset slot** for each slot on the type object (implemented via descriptors); the instance no longer carries a dict but, like a C struct, stores field value pointers contiguously within the object body:

```
instance with __dict__:  [head][__dict__ ptr] -> {'x':.., 'y':..}   (an extra dict)
instance with __slots__:  [head][x ptr][y ptr]                        (fields inlined, no dict)
```

Measure the memory difference with `sys.getsizeof` (CPython 3.14, 64-bit):

```python
import sys
class WithDict:
    def __init__(self, x, y): self.x, self.y = x, y
class WithSlots:
    __slots__ = ('x', 'y')
    def __init__(self, x, y): self.x, self.y = x, y

a = WithDict(1, 2); b = WithSlots(1, 2)
print(sys.getsizeof(a) + sys.getsizeof(a.__dict__))  # 344  = instance body 48 + __dict__ 296
print(sys.getsizeof(b))                              # 48   fields inlined, no separate dict
print(hasattr(b, "__dict__"))                        # False
```

The gain: each instance saves a dict (tens to hundreds of bytes), and access becomes a fixed offset (close to C++ field access, fast). The cost: you lose dynamic attribute add/remove, lose `__dict__`, and slots have restrictions under multiple inheritance. This is the way of writing that comes closest to C struct memory layout in Python's object model.

## 9. Magic Methods and Operator Overloading

### How to Use It

"Magic methods" (dunder methods, with double underscores) are hooks the interpreter calls back on specific syntax/operations. Define them and your object gets to use built-in syntax like `+`, `==`, `len()`, and `print()`:

```python
class Vec:
    def __init__(self, x, y): self.x, self.y = x, y
    def __repr__(self):  return f"Vec({self.x}, {self.y})"   # repr() / interactive display
    def __add__(self, o): return Vec(self.x+o.x, self.y+o.y) # supports v1 + v2
    def __eq__(self, o):  return (self.x, self.y) == (o.x, o.y)

v1 = Vec(1, 2)
v2 = Vec(3, 4)
print(v1 + v2)          # Vec(4, 6)   —— triggers __add__
print(v1 == Vec(1, 2))  # True        —— triggers __eq__
print(v1)                # Vec(1, 2)  —— triggers __repr__
```

### Internals vs. C++

Common magic methods:

| Syntax/operation | Magic method | Corresponding C++ |
|-----------|---------|---------|
| `a + b` | `__add__` / `__radd__` | `operator+` |
| `a == b` | `__eq__` | `operator==` |
| `a < b` | `__lt__` | `operator<` |
| `len(a)` | `__len__` | `size()` convention |
| `a[i]` | `__getitem__`/`__setitem__` | `operator[]` |
| `a()` | `__call__` | `operator()` (functor) |
| `str(a)`/print | `__str__` | `operator<<` convention |
| `with a:` | `__enter__`/`__exit__` | RAII (see chapter 9) |
| `for x in a` | `__iter__`/`__next__` | iterator |

The power of operator overloading rivals C++'s, but it's **all done at runtime by looking up methods**, with no compile-time overload resolution: `v1 + v2` is essentially the interpreter seeing the `+` sign, looking up `__add__` in `type(v1)`, and calling it—the same lookup mechanism as method calls in section 4, no extra magic involved.

## 10. Abstract Base Classes (`abc`)

### How to Use It

Duck typing is flexible, but sometimes you want to force "subclasses must implement certain methods." The `abc` module provides abstract base classes:

```python
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self) -> float: ...    # abstract method, subclasses must implement

class Square(Shape):
    def __init__(self, s): self.s = s
    def area(self): return self.s * self.s

# Shape()      # TypeError: can't instantiate a class with unimplemented abstract methods
print(Square(3).area())   # 9
```

### Internals vs. C++

- Similar to C++'s pure virtual functions (`virtual void area() = 0;`) + an abstract class that can't be instantiated.
- But the check happens **at instantiation time** (runtime), not at compile time.
- `abc` also supports registering "virtual subclasses" and structured `isinstance` checks; `collections.abc` defines protocol base classes such as `Iterable`/`Sequence`/`Mapping`, usable for type checks.
- A lighter, more modern approach is `typing.Protocol` (structural subtyping / static duck typing), which, together with [mypy](12-常用第三方库.md), verifies interfaces at **compile time (check time)** without intruding on runtime.

## Summary

- **Usage level**: `class` + `__init__` + `self` define a class and its instance attributes; list the parent class in parentheses to inherit, override a same-named method for polymorphism; `super().__init__()` calls the parent's constructor; `@property`/`@staticmethod`/`@classmethod` give you computed attributes, unbound methods, and class-level construction respectively; defining `__add__`/`__eq__`/`__repr__` and other magic methods lets an object support built-in operators; `abc.abstractmethod` forces subclasses to implement an interface.
- **Internals level**: instance attributes live in `__dict__` by default; access is a dictionary lookup, not C++'s fixed offset; `__slots__` can switch back to near-struct fixed slots. Classes are runtime objects, and `type` is the default metaclass. Attribute lookup order: data descriptor → instance dict → MRO (C3 linearization). Methods are functions + the descriptor protocol; `self` / bound methods are produced by `__get__` at runtime. Polymorphism relies on runtime method lookup (duck typing), not a compile-time vtable.
