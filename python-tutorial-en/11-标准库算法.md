# Standard Library Algorithms: Sorting, Heaps, Containers, and Iterators

C++ has the STL—`std::sort` (introsort), `std::priority_queue`, `std::lower_bound`, and various containers. The Python standard library offers functionally equivalent tools, each with its own tradeoffs in the underlying implementation. This chapter focuses on the **underlying data structures and algorithms** of each tool and provides a complexity table.

## 1. sorted / list.sort: Timsort

Python's sorting uses **Timsort**—designed by Tim Peters in 2002 for CPython, and later adopted by Java, Android, V8, and others.

### The Essence of Timsort

It's an **adaptive, stable merge sort** that fuses merge sort with insertion sort:

- **Stable**: equal elements keep their original relative order. This is the key difference from C++'s `std::sort` (unstable; you need `std::stable_sort` for stability).
- **Adaptive**: it identifies already-ordered consecutive segments in the data (called **runs**). For **nearly sorted** data it approaches O(n).
- **Worst case O(n log n)**, unlike quicksort which can degrade to O(n²).

Underlying steps (simplified):

```
1. Scan the array, splitting out naturally ascending/descending runs (descending runs are reversed in place)
2. When a run is shorter than minrun (32~64), extend it with binary insertion sort
3. Maintain runs on a stack, merging adjacent runs by specific rules (keeping stack run lengths roughly Fibonacci)
4. During merges, use galloping mode to speed up cases where one side wins consecutively
```

```python
data = [("b", 2), ("a", 2), ("c", 1)]

# key: extracts the sort key; like a C++ comparator, but the key is computed only once (decorate-sort-undecorate), which is faster
s = sorted(data, key=lambda x: x[1])
print(s)  # [('c', 1), ('b', 2), ('a', 2)] —— stable: b is still before a

# reverse preserves stability (it's not a simple reversal of the result)
print(sorted(data, key=lambda x: x[1], reverse=True))  # [('b',2),('a',2),('c',1)]

nums = [5, 3, 1, 4, 2]
nums.sort()          # in-place sort, returns None (modifies itself)
print(sorted(nums))  # returns a new list, the original is untouched
```

The `key` function is called only once per element, and the result is cached for use in comparisons (the Schwartzian transform). So using `key` is far faster than a comparison function wrapped with `functools.cmp_to_key`.

## 2. bisect: Binary Search on a Sorted Array

`bisect` does binary search on a **sorted list**, corresponding to C++'s `std::lower_bound`/`std::upper_bound`.

```python
import bisect

a = [1, 3, 3, 5, 7]
print(bisect.bisect_left(a, 3))   # 1: the first position >=3 (lower_bound)
print(bisect.bisect_right(a, 3))  # 3: the first position >3 (upper_bound)

bisect.insort(a, 4)   # binary-search the position, then insert, keeping order
print(a)              # [1, 3, 3, 4, 5, 7]
```

Note the underlying reality: `insort`'s search is O(log n), but **the insertion itself is O(n)**, because a list is a contiguous array and inserting shifts subsequent elements (see [Chapter 07](07-复合数据结构.md)). If you need frequent middle insertions, use a different structure.

A classic use—mapping continuous values to discrete levels:

```python
def grade(score: int) -> str:
    breakpoints = [60, 70, 80, 90]
    grades = "FDCBA"
    return grades[bisect.bisect(breakpoints, score)]
print([grade(s) for s in (55, 65, 85, 95)])  # ['F', 'D', 'B', 'A']
```

## 3. heapq: A Binary Heap Backed by a list

`heapq` operates on a **plain list as a binary min-heap**—there's no separate heap type, just a set of functions acting on a list. The heap uses an **array to implicitly represent** a complete binary tree: node `i`'s children are `2i+1` and `2i+2`, needing no pointers.

```python
import heapq

h = []
heapq.heappush(h, 3)      # O(log n): sift up
heapq.heappush(h, 1)
heapq.heappush(h, 2)
print(h[0])               # 1: the heap top is always the minimum, O(1) to peek
print(heapq.heappop(h))   # 1: pop the minimum, O(log n), the last element fills the top then sifts down

nums = [5, 3, 8, 1, 9, 2]
heapq.heapify(nums)       # O(n) in-place heap build, faster than pushing one by one at O(n log n)
print(heapq.nlargest(3, nums))   # [9, 8, 5]: Top-K, using a heap of size k, O(n log k)
print(heapq.nsmallest(2, nums))  # [1, 2]

# merge: lazily merge multiple sorted sequences without reading them all into memory (the basis of external merge)
print(list(heapq.merge([1, 4], [2, 3], [0, 5])))  # [0, 1, 2, 3, 4, 5]
```

Only a min-heap exists. For a max-heap, negate the values, or use `(-priority, item)` on tuples. In contrast, C++'s `std::priority_queue` is a max-heap by default—the opposite direction, so don't get confused.

A `(-priority, item)` tuple turns the min-heap into a max-priority queue—the highest priority pops first:

```python
import heapq

tasks = [(2, "email"), (1, "backup"), (3, "report")]  # (priority, task)
h = []
for pri, name in tasks:
    heapq.heappush(h, (-pri, name))   # negate => highest priority lands on top
print([heapq.heappop(h)[1] for _ in range(3)])
# ['report', 'email', 'backup']  —— popped in priority order 3, 2, 1
```

## 4. collections: Specialized Containers

Beyond the standard `dict`/`list`/`set`, `collections` provides containers optimized for specific patterns.

```python
from collections import Counter, defaultdict, OrderedDict, deque, namedtuple

# Counter: multiset/frequency counting, backed by dict[element -> count]
c = Counter("abracadabra")
print(c.most_common(2))     # [('a', 5), ('b', 2)]: internally uses heapq.nlargest

# defaultdict: automatically calls the factory to create a default value on a missing key, avoiding setdefault
d = defaultdict(list)
d["x"].append(1)            # "x" doesn't exist, an empty list is auto-created
print(d)                    # defaultdict(<class 'list'>, {'x': [1]})

# deque: double-ended queue, backed by a chunked doubly-linked list, O(1) add/remove at both ends
dq = deque([1, 2, 3], maxlen=3)
dq.appendleft(0)            # O(1), whereas list.insert(0,x) is O(n)
dq.append(4)                # maxlen is full, the left end is automatically evicted (ring buffer)
print(dq)                   # deque([2, 3, 4], maxlen=3)

# namedtuple: a tuple with field names, as memory-compact as a tuple (no __dict__)
Point = namedtuple("Point", ["x", "y"])
p = Point(1, 2)
print(p.x, p[0])            # 1 1: accessible both by name and by index
```

`OrderedDict` records insertion order—but since Python 3.7, **the plain dict already guarantees insertion order**, so `OrderedDict` is now used only for scenarios needing `move_to_end`, order-sensitive equality comparison, etc.

| Container | Underlying Structure | Typical Strength Operations |
|------|----------|--------------|
| `Counter` | dict | counting, `most_common` |
| `defaultdict` | dict | auto-initialize missing keys |
| `deque` | doubly-linked list (chunked) | O(1) at both ends, fixed-length ring buffer |
| `namedtuple` | tuple | named fields, memory-compact |

## 5. itertools: Lazy Iterator Combinators

`itertools` provides **lazy** iterator tools. They return iterators rather than lists—**computed on demand, O(1) memory**—and can handle infinite sequences. This follows the same lineage as the generators in [Chapter 07](07-复合数据结构.md), similar to C++20's ranges/views.

```python
import itertools as it

# infinite iterators: count / cycle / repeat
counter = it.count(10, 2)          # 10, 12, 14, ... lazy, won't fill memory
print(list(it.islice(counter, 3))) # [10, 12, 14]: islice takes the first 3

# combinatorics: Cartesian product, permutations, combinations
print(list(it.product([0, 1], repeat=2)))     # [(0,0),(0,1),(1,0),(1,1)]
print(list(it.combinations("ABC", 2)))        # [('A','B'),('A','C'),('B','C')]

# chain: concatenate multiple iterables, "joining" them zero-copy
print(list(it.chain([1, 2], [3, 4])))         # [1, 2, 3, 4]

# groupby: group by consecutively identical keys (requires pre-sorting, like needing order by before SQL grouping)
data = [("a", 1), ("a", 2), ("b", 3)]
for k, g in it.groupby(data, key=lambda x: x[0]):
    print(k, list(g))              # a [('a',1),('a',2)] / b [('b',3)]

# accumulate: prefix sums / scan
print(list(it.accumulate([1, 2, 3, 4])))      # [1, 3, 6, 10]
```

## 6. functools: Higher-Order Function Tools

```python
import functools

# lru_cache: memoization, automatically caches function results. Backed by an LRU implemented with dict + doubly-linked list
@functools.lru_cache(maxsize=None)
def fib(n: int) -> int:
    return n if n < 2 else fib(n - 1) + fib(n - 2)
print(fib(50))   # instant: without caching this would be exponential recursion

# reduce: fold, corresponding to C++ std::accumulate / functional fold
print(functools.reduce(lambda a, b: a * b, [1, 2, 3, 4]))  # 24

# partial: partial application, fixing some arguments, like std::bind
base2 = functools.partial(int, base=2)
print(base2("1010"))   # 10: equivalent to int("1010", base=2)
```

`@functools.cache` (3.9+) is shorthand for `lru_cache(maxsize=None)`. The cache is keyed by arguments, so arguments must be hashable.

`cache_info()` shows hit/miss counts, a direct way to confirm memoization really kicks in:

```python
import functools

@functools.lru_cache(maxsize=None)
def fib(n: int) -> int:
    return n if n < 2 else fib(n - 1) + fib(n - 2)

fib(10)
print(fib.cache_info())
# CacheInfo(hits=8, misses=11, maxsize=None, currsize=11)
# 11 misses (n=0..10 each computed once), 8 hits (repeated subproblems served from the table)
```

## 7. array / struct / memoryview: Contiguous Memory Close to C

A plain `list` is an **array of pointers** (each slot holds a `PyObject*`). These three modules, by contrast, let you operate on **compact raw bytes**, with a memory layout matching that of a C array.

```python
from array import array

# array: a compact array of homogeneous type; "i" means C's int (4 bytes), no per-element object header
a = array("i", [1, 2, 3])
print(a.itemsize)   # 4: each element is 4 bytes stored contiguously, like int32_t arr[3]
# compare list[int]: each element is a separate PyObject, plus a stored pointer, several times the memory
```

```python
import struct

# struct: converts between Python values and the binary representation of C structs (for binary protocols/files)
# "<" little-endian, "i" int32, "f" float32, "H" uint16
packed = struct.pack("<iH", 258, 5)   # pack into a byte string
print(packed)                          # b'\x02\x01\x00\x00\x05\x00'
print(struct.unpack("<iH", packed))    # (258, 5): unpack to restore
print(struct.calcsize("<iH"))          # 6: byte count (mind the alignment! use "<" to disable padding)
```

`struct`'s alignment rules are familiar to you: the default (native `@`) inserts padding bytes according to platform alignment; a `<`/`>`/`=` prefix switches to standard sizes with no padding.

```python
# memoryview: zero-copy access to another object's buffer (buffer protocol), like a C++ span
buf = bytearray(b"hello world")
mv = memoryview(buf)
sub = mv[0:5]           # no copy! just a view pointing at the same memory
sub[0] = ord("H")       # modify in place through the view
print(buf)              # bytearray(b'Hello world'): the underlying bytes are changed
```

`memoryview` is key for high-performance IO, image processing, and interop with NumPy/C extensions—it lets you slice a large buffer **without producing a copy**.

## 8. math / statistics / random

```python
import math, statistics, random

print(math.gcd(12, 18), math.isqrt(17))   # 6 4: integer square root, no floating-point error
print(math.comb(5, 2), math.perm(5, 2))   # 10 20: number of combinations/permutations
print(math.fsum([0.1] * 10))              # 1.0: compensated summation, avoiding floating-point accumulation error

print(statistics.mean([1, 2, 3, 4]))      # 2.5
print(statistics.median([1, 3, 2]))       # 2
print(statistics.stdev([1, 2, 3, 4]))     # sample standard deviation

random.seed(42)                            # underlying is the Mersenne Twister (period 2^19937-1)
print(random.randint(1, 6))                # roll a die (note: not cryptographically secure!)
# for cryptographically secure randomness use the secrets module (based on the OS's CSPRNG)
```

`random` is for simulation/games, **not for keys or tokens**—those need `secrets`.

## Complexity Cheat Sheet

| Operation | Complexity | Underlying |
|------|--------|------|
| `sorted` / `list.sort` | O(n log n), approaching O(n) when nearly sorted | Timsort |
| `bisect.bisect` | O(log n) | binary search |
| `bisect.insort` | O(n) | binary search + array shift |
| `heappush` / `heappop` | O(log n) | binary heap sift up/down |
| `heapify` | O(n) | bottom-up heap build |
| `heapq.nlargest(k, n items)` | O(n log k) | heap of size k |
| `deque.append` / `appendleft` / `pop` / `popleft` | O(1) | chunked doubly-linked list |
| `deque` middle indexing | O(n) | requires traversal |
| `Counter.most_common(k)` | O(n log k) | heapq |
| `lru_cache` hit | O(1) amortized | dict + linked list |

---

## Summary

- Sorting uses **Timsort**: stable, adaptive, worst-case O(n log n), extremely fast on nearly sorted data; prefer `key`.
- `bisect` does binary search in O(log n), but insertion is O(n) due to the contiguous-array limitation.
- `heapq` treats a list as a binary heap (min-heap only); use `nlargest`/`nsmallest` for Top-K.
- `collections` is optimized for patterns: `deque` is O(1) at both ends, and `Counter`/`defaultdict`/`namedtuple` each have their specialty.
- `itertools` provides lazy combinators, O(1) memory, capable of handling infinite streams.
- `functools` provides `lru_cache`/`reduce`/`partial`.
- `array`/`struct`/`memoryview` let you operate on compact contiguous memory—the bridge to the C world and high-performance IO.
