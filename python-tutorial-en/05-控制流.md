# Control Flow

The control-flow keywords look much like C's: `if`/`while`/`for`. But the underlying machinery is worlds apart. C's `for (int i=0; i<n; i++)` is a compile-time-generated integer increment plus a compare-and-jump; Python's `for` is a **runtime method-call protocol**, where every iteration sends a message (`__next__`) to an object. **Each construct in this chapter gets two passes: first "how to write it, how to use it"—understandable with zero background—then "under the hood, compared with C," where we see what these "deceptively familiar" constructs are really doing underneath.** If you just want to use them, reading the first half of each section is enough.

---

## 1. Learn to Use It First: A Quick Tour of Control Flow

No background knowledge needed—just look at how these constructs are written:

```python
# if / elif / else
age = 20
if age < 13:
    print("Child")
elif age < 18:
    print("Teenager")
else:
    print("Adult")

# while loop
n = 3
while n > 0:
    print(n)
    n -= 1

# for loop: iterate over a list
for fruit in ["apple", "banana", "orange"]:
    print(fruit)

# break / continue
for i in range(5):
    if i == 3:
        break
    print(i)
```

Output:

```
Adult
3
2
1
apple
banana
orange
0
1
2
```

`if/elif/else` branches by indentation, no parentheses needed; `while condition:` keeps looping as long as the condition is true; `for variable in sequence:` assigns each element of the sequence to the variable in turn; `break` exits a loop early, and `continue` (not demonstrated above) skips the rest of the current iteration and jumps straight to the next one. That's enough to write basic logic. Now let's dig into the underlying mechanics of each construct.

---

## 2. Going Deeper: Truth Testing Is More Than Just 0 and 1

In C, `if (x)` simply tests `x != 0` (the same for integers and pointers). In Python, `if x:` triggers a whole **protocol**; any object can define its own notion of "truthiness."

### How to Use It

First, see which values count as "false" in an `if`, and which count as "true":

```python
if []:
    print("non-empty")
else:
    print("empty list is falsy")     # empty list is falsy

if 0:
    print("nonzero")
else:
    print("0 is falsy")         # 0 is falsy

if "hi":
    print("non-empty string is truthy")  # non-empty string is truthy
```

Here are the built-in falsy values:

| Falsy value | Reason |
|------|------|
| `None` | Singleton, meaning "no value" |
| `False` | It is itself |
| `0`, `0.0`, `0j` | Numeric zero |
| `""`, `b""` | Empty string/bytes |
| `[]`, `()`, `{}`, `set()` | Empty containers |

Aside from the values in this table, every other object is "truthy" in an `if`. That's why you can write `if items:` directly to test whether a list is non-empty, instead of `len(items) == 0`.

### Under the Hood, Compared with C

The order CPython uses to compute a truth value (see `PyObject_IsTrue` in `object.c`):

1. If the object defines `__bool__`, call it; it must return a `bool`.
2. Otherwise, if it defines `__len__`, call it; `len == 0` is false, anything else is true.
3. If neither exists, the object is always true.

```python
class Buffer:
    def __init__(self, data):
        self._data = data
    def __len__(self):
        # No __bool__, so if falls back to __len__
        return len(self._data)

buf = Buffer([])
print(bool(buf))          # False —— length is 0
print(bool(Buffer([1])))  # True

class Always:
    def __bool__(self):
        return False       # __bool__ takes precedence over __len__

print(bool(Always()))     # False
```

This is the underlying reason why empty containers, `0`, `0.0`, `""`, and `None` are all "falsy": either their `__len__` returns 0 or their `__bool__` returns `False`.

Under the hood: `if x:` has no implicit null-pointer check. `None` is Python's "null pointer," but it's a **real singleton object** (with a fixed address), not C's `0` address. To test for it, write `if x is None`, using `is` to compare identity (i.e., pointer equality in C), not `==`.

## 3. if / elif / else

### How to Use It

Syntactically there are no parentheses; it relies on indentation, and there's already an example of that back in "Learn to Use It First." Here's something C doesn't have—the **conditional expression** (Python's version of the ternary operator):

```python
n = 5
label = "positive" if n > 0 else "non-positive"   # a value expression, not a statement
print(label)                        # positive
```

The order differs from C's `c ? a : b`: you write "the value if true" first, then the condition, then "the value if false."

### Under the Hood, Compared with C

It's worth seeing what if/elif/else compile down to as bytecode—pure jumps, just like C:

```python
import dis

def classify(n):
    if n > 0:
        return "positive"
    elif n == 0:
        return "zero"
    else:
        return "negative"

dis.dis(classify)
```

You'll see `COMPARE_OP` followed by `POP_JUMP_IF_FALSE`—if the condition is false, jump to the next branch. `elif` has no bytecode of its own; it's just syntactic sugar for `else: if` that doesn't add an indentation level.

## 4. while Loops

### How to Use It

`while` is nearly identical to C:

```python
n = 10
while n:            # equivalent to while n != 0 (goes through the truth protocol from the previous section)
    n -= 1
print(n)             # 0
```

`break` / `continue` have the same semantics as C. Python has **no** `do-while`; the common substitute is `while True` + `break`.

### Under the Hood, Compared with C

The condition of a `while` goes through the same truth protocol (`__bool__` / `__len__`) covered in the previous section—it's not a simple machine-code comparison. `while n:` works just as well on a custom object, as long as it implements `__bool__` or `__len__`. This differs from C, where `while(p)` can only test whether a pointer is `NULL`.

## 5. for Loops: Fundamentally Different from C's Index-Based for

This is the most important section of the chapter.

### How to Use It

The most common pattern is simply "do something with each element in a sequence":

```python
for x in [10, 20, 30]:
    print(x)             # 10 20 30 (one per line)

for ch in "abc":
    print(ch)             # a b c (one per line)

for i in range(5):
    print(i)              # 0 1 2 3 4 (one per line)
```

No index and no need to know the length—`for variable in sequence:` automatically assigns each element of the sequence to the variable in turn.

### Under the Hood, Compared with C

C's `for` has three parts—initialize, test, increment—where the loop variable is an integer in your hand that you **actively** use to index an array. Python's `for` is nothing like that—behind it lies the **iterator protocol**:

```python
for x in obj:
    ...
```

is equivalent to the following expansion (this is exactly what CPython does):

```python
_it = iter(obj)          # calls obj.__iter__(), getting an iterator object
while True:
    try:
        x = next(_it)    # calls _it.__next__()
    except StopIteration:
        break
    ...
```

Contrasting the two protocol methods:

| Method | Implemented by | Responsibility |
|------|----------|------|
| `__iter__()` | iterable | Returns an iterator |
| `__next__()` | iterator | Returns the next value; raises `StopIteration` when exhausted |

The underlying meaning: every step of the loop is a **method call**—the consumer actively "pulls" the next value from the iterator (calling `__next__` once per step), rather than the iterator "pushing" values at you. That's why Python's `for` can naturally traverse linked lists, trees, file lines, and infinite sequences—as long as the object implements the protocol, no index and no known length are required. It's also why you almost never write an index when iterating.

Hand-write an iterator to see the protocol clearly:

```python
class Countdown:
    def __init__(self, start):
        self.start = start
    def __iter__(self):
        # Return an iterator. Returning a separate object here guarantees repeatable iteration
        return CountdownIter(self.start)

class CountdownIter:
    def __init__(self, cur):
        self.cur = cur
    def __iter__(self):
        return self          # an iterator must itself be iterable
    def __next__(self):
        if self.cur <= 0:
            raise StopIteration   # signal "done" with an exception, not a sentinel value
        self.cur -= 1
        return self.cur + 1

for x in Countdown(3):
    print(x)                 # 3 2 1
```

Note that "iterable" and "iterator" are two distinct roles. `list` is an iterable but not an iterator (it has no `__next__`); the `list_iterator` returned by `iter(mylist)` is the iterator, and it internally holds a C-level index `it_index` that increments on each `__next__`. So you can open multiple independent iterators over the same list, each with its own cursor.

```python
lst = [10, 20, 30]
a = iter(lst)
b = iter(lst)
print(next(a), next(a))   # 10 20
print(next(b))            # 10 —— b is an independent cursor
```

Memory diagram: the list holds only one copy of the data, while the two iterators each hold a reference to it plus an integer cursor:

```
        list_iterator a           list_iterator b
        +-----------+             +-----------+
        | it_seq  ---> [list]     | it_seq  ---> (same list)
        | it_index=2|      ^      | it_index=1|
        +-----------+      |      +-----------+
                           |
        list object +-------+-------+
                   | ob[0]=10 (ptr)|
                   | ob[1]=20 (ptr)|
                   | ob[2]=30 (ptr)|
                   +---------------+
```

`iter` also has a little-known **two-argument form**, `iter(callable, sentinel)`: it calls the zero-argument `callable` repeatedly until the return value equals `sentinel`, then stops (raising `StopIteration`). This is the direct analog of C's `while ((c = getchar()) != EOF)` "keep fetching until a sentinel" idiom, commonly used to read a stream in chunks:

```python
import io
stream = io.StringIO("a\nb\nc\n")
# call stream.readline repeatedly until it returns "" (the end-of-file sentinel)
for line in iter(stream.readline, ""):
    print(repr(line))
# 'a\n'
# 'b\n'
# 'c\n'
```

It compresses "fetch a value + test for the end" into a single iterator, with no `while True` + `break`. It works on real files too: `iter(f.readline, "")`, `iter(lambda: sock.recv(4096), b"")`.

## 6. range: A Lazy Object, Not an Array

### How to Use It

`range` generates a sequence of evenly spaced integers, most commonly used to count with a `for` loop:

```python
for i in range(5):        # 0,1,2,3,4
    print(i)

print(list(range(5)))          # [0, 1, 2, 3, 4]
print(list(range(1, 10, 2)))   # [1, 3, 5, 7, 9] —— start, stop (exclusive), step
```

### Under the Hood, Compared with C

C programmers tend to think of `range(1000000)` as "generating a million-element array." It isn't. `range` is an **immutable lazy sequence object** that stores only three integers—`start`/`stop`/`step`—and takes almost no memory:

```python
import sys
r = range(1_000_000)
print(sys.getsizeof(r))   # about 48 bytes, independent of element count
print(r[500])             # 500 —— computed on demand in O(1), no lookup table
print(len(r))             # 1000000 —— derived from a formula
```

It supports indexing and `len` (it implements the sequence protocol), but the values are **computed on the fly as `start + i*step`** and never materialized into a list. This is entirely unlike C, where "you must allocate the array first, then iterate." To truly materialize it, use `list(range(...))`.

Because everything is computed from a formula, slicing a `range` returns another `range` (still lazy), and `x in range(...)` is an **O(1) arithmetic test**, not a scan over a billion elements:

```python
r = range(0, 100, 7)
print(r[-1])                  # 98 —— negative indexing is computed by formula too
sub = r[2:5]                  # the slice isn't materialized; it returns a new range
print(sub)                    # range(14, 35, 7)
print(list(sub))              # [14, 21, 28]
print(70 in range(0, 1_000_000_000, 7))   # True —— computes whether 70 lies on the arithmetic sequence, instantly
print(71 in range(0, 1_000_000_000, 7))   # False —— 71 is not a multiple of 7
```

Contrast: `70 in [0, 7, 14, ...]` (an actual list) requires a linear scan; `range` decides it in one step via "is `start ≤ x < stop` and `(x-start) % step == 0`."

## 7. Comprehensions: They Have Their Own Scope

### How to Use It

A comprehension is shorthand for "build a new container with a one-line loop":

```python
squares = [x * x for x in range(5)]
print(squares)                       # [0, 1, 4, 9, 16]

evens = [x for x in range(10) if x % 2 == 0]   # with a filter condition
print(evens)                                    # [0, 2, 4, 6, 8]

square_map = {x: x * x for x in range(4)}       # dict comprehension
print(square_map)                                # {0: 0, 1: 1, 2: 4, 3: 9}
```

Compared to writing it by hand:

```python
squares = []
for x in range(5):
    squares.append(x * x)
```

the comprehension is shorter, and it's the more idiomatic (Pythonic) way to write it.

### Under the Hood, Compared with C

Comprehensions are more than syntactic sugar: they have **their own scope**, and the loop variable does not leak to the enclosing scope.

```python
i = "outer"
squares = [i for i in range(3)]
print(i)          # "outer" —— the loop variable i doesn't leak to the outer scope!
```

Exactly how this "own scope" is implemented **depends on the version**:

- **Python ≤ 3.11**: the comprehension body is compiled into an **anonymous function that is immediately called**—the disassembly shows `MAKE_FUNCTION` + `CALL` and an inner code object named `<listcomp>`, relying on a separate stack frame to isolate the variable.
- **Python 3.12+ (PEP 709)**: list/dict/set comprehensions are now **inlined** into the current function; there is **no longer** an anonymous function, no separate stack frame, and no `<listcomp>` code object. In `dis` you see `LOAD_FAST_AND_CLEAR` + `FOR_ITER` + `LIST_APPEND`: on entering the comprehension it "borrows and clears" any same-named outer variable, then **restores** it afterward—achieving both inlining (faster) and non-leaking.

```python
import dis
def make():
    return [i * 2 for i in range(3)]
dis.dis(make)     # 3.12+: no MAKE_FUNCTION, no <listcomp>; only 3.11 and earlier have them
```

This differs from C's block scope, and also from Python 2 (where comprehension variables did leak).

> Note: a plain `for` statement does **not** have its own scope; the loop variable remains in the current scope. Only comprehensions and generator expressions do.

Generator expressions go a step further, not even materializing the result list; they return an iterator that produces values lazily (note: **a generator expression is not inlined**—it is still compiled into a separate `<genexpr>` function, which is why it naturally has its own scope):

```python
gen = (i * 2 for i in range(1_000_000))   # computes no values, allocates no million-element list
print(next(gen))   # 0 —— the first value is computed only now
```

## 8. The Loop else Clause

### How to Use It

Unique to Python: `for`/`while` can carry an `else`, which executes when the loop **exhausts normally** (without being interrupted by `break`). Read it as "if no break, then else":

```python
def find(seq, target):
    for x in seq:
        if x == target:
            print("found", x)
            break
    else:
        # reached only if the loop ran to completion without break
        print("not found")

find([1, 2, 3], 2)   # found 2
find([1, 2, 3], 9)   # not found
```

This eliminates the common C idiom of "set a `found` flag variable, then test it."

### Under the Hood, Compared with C

Under the hood, `break` jumps past the bytecode for the `else` block, while normal completion falls straight through into it—this isn't implemented via a hidden boolean flag variable, but through jump targets arranged at compile time.

## 9. match-case: Structural Pattern Matching (3.10+)

We'll explain `match` from scratch here, because it's the piece of Python syntax that **looks the most familiar yet behaves the least like you'd expect**.

In one sentence: **`match` is not C's `switch`**. C's `switch(x)` means "take the integer value of x and jump through a table" — it only compares integers and needs `break` to avoid fall-through. Python's `match` means "look at the **structure** of a value, and pull the pieces out and **bind them to variables** along the way." Think of it as "unpacking a parcel": first check the box has the right shape, then take the contents out and hand them to you.

It has **no** fall-through (you won't accidentally run into the next branch by forgetting `break`), so it **needs no `break`**: whichever `case` matches runs, and then the whole `match` is done.

### Step 1: The simplest use — matching by "value" (this part is most like a switch)

```python
def http(code):
    match code:               # compare code against each case below
        case 200:             # if code == 200
            return "OK"
        case 404:
            return "Not Found"
        case _:               # _ is the wildcard, matches anything — like switch's default
            return "Unknown"

print(http(200))   # OK
print(http(404))   # Not Found
print(http(500))   # Unknown
```

- When you write a **literal** in `case 200:` (a number, string, `True`/`False`/`None`), it means "match if equal" — consistent with your `switch` intuition.
- The underscore `_` in `case _:` is the **wildcard**: put it last as a catch-all, like `default`.
- The **first** matching case from top to bottom wins; the rest are not checked.

### Step 2: OR patterns — one case matching multiple values

Join several patterns with `|`; a match on any one counts (cleaner than piling up `case 400: case 401:` in C):

```python
def kind(code):
    match code:
        case 200 | 201 | 204:      # any of these three counts as success
            return "success"
        case 400 | 401 | 404:
            return "client error"
        case _:
            return "other"

print(kind(201))   # success
print(kind(404))   # client error
```

### Step 3: Capture patterns — match and "grab a value" at the same time (the essence of `match`)

This is where it pulls away from `switch`. You can put a **variable name** in a pattern; on a successful match it **binds (assigns)** the corresponding value to that variable, ready to use afterward:

```python
def head_tail(seq):
    match seq:
        case []:                       # match an empty list
            return "empty"
        case [x]:                      # match a list of exactly one element, bind it to x
            return f"just one: {x}"
        case [first, *rest]:           # match "at least one element"; first = the first, *rest = the remainder (a list)
            return f"first={first}, rest={rest}"

print(head_tail([]))          # empty
print(head_tail([9]))         # just one: 9
print(head_tail([1, 2, 3, 4]))# first=1, rest=[2, 3, 4]
```

- `[x]`, `[first, *rest]` are **sequence patterns**: they first check "is it a sequence, is the length right," then **destructure** the elements and bind them to variables. `*rest` is like `*args` in a function, collecting the remaining elements.
- Tuples work the same: `case (x, y):` matches "a two-element tuple" and binds the two elements to `x` and `y`.

### ⚠️ Step 4: The biggest beginner trap — a bare name is a "binding," not a "comparison"

This is the one counterintuitive point about `match` that almost everyone trips over, so let's spell it out.

You might think "match if `n` equals that outer variable `value`," and write `case value:`. **Wrong!** A **bare name in a pattern is always a "capture binding,"** not a comparison of its value — it matches **unconditionally** and assigns `n` to a new `value`, shadowing the outer variable of the same name:

```python
value = 3
def bad(n):
    match n:
        case value:                    # trap! this is NOT "n == 3", it's "bind n to value"
            return f"bound value={value}"

print(bad(99))   # bound value=99   ← proof it never compared against the outer 3; it matches anything
```

**To match against a variable's value, there are two correct ways:**

```python
value = 3
def good(n):
    match n:
        case v if v == value:          # way 1: capture as v, then compare in a guard
            return "equals value"
        case _:
            return "not equal"

# way 2: use a "dotted name" (qualified name); the dot tells match this is a constant reference, not a binding
import enum
class Color(enum.Enum):
    RED = 1; GREEN = 2
def name(c):
    match c:
        case Color.RED:                # dotted → compared as a value (not bound)
            return "red"
        case Color.GREEN:
            return "green"
```

Remember one rule: **in a pattern, `abc` is a "binding," and only `X.abc` is a "comparison."**

### Step 5: guards — add an extra condition to a case

Adding `if condition` after a `case` is called a **guard**: it matches only if "the pattern matches **and** the condition is true."

```python
def sign(n):
    match n:
        case 0:
            return "zero"
        case x if x > 0:        # capture as x, then require x > 0
            return "positive"
        case _:
            return "negative"

print(sign(0), sign(5), sign(-2))   # zero positive negative
```

### Step 6: mapping patterns (match a dict) and class patterns (match an object)

`match` can also destructure dicts and custom objects — extremely handy for JSON, ASTs, and events.

**Mapping patterns**: care only about "are certain keys present and what are their values," ignoring extra keys:

```python
def route(req):
    match req:
        case {"method": "GET", "path": p}:   # require method=="GET", bind the value of path to p
            return f"GET {p}"
        case {"method": "POST"}:             # require only method=="POST"; other keys don't matter
            return "POST"
        case _:
            return "unknown request"

print(route({"method": "GET", "path": "/users"}))   # GET /users
```

**Class patterns**: destructure objects by type + attributes. Give a class a `__match_args__` and you can write it positionally, like a tuple:

```python
class Point:
    __match_args__ = ("x", "y")     # declare: for positional matching, 1st → x, 2nd → y
    def __init__(self, x, y):
        self.x, self.y = x, y

def quadrant(p):
    match p:
        case Point(0, 0):                    # positional match: x==0 and y==0
            return "origin"
        case Point(x=0, y=yy):               # keyword match: x==0, and bind y to yy
            return f"on the y-axis, y={yy}"
        case Point(x, y) if x > 0 and y > 0: # positional capture + guard
            return "quadrant I"
        case Point():                        # just needs to be a Point
            return "some other Point"
        case _:
            return "not a Point at all"

print(quadrant(Point(0, 0)))   # origin
print(quadrant(Point(0, 7)))   # on the y-axis, y=7
print(quadrant(Point(1, 2)))   # quadrant I
```

### Cheat sheet: common patterns at a glance

| Pattern | Example | Meaning |
|------|------|------|
| Literal | `case 200:` | equal value |
| Wildcard | `case _:` | matches anything (catch-all) |
| Capture | `case x:` | **always matches**, binds the value to `x` |
| OR | `case 1` \| `2` | any sub-pattern matches |
| Sequence | `case [a, b, *rest]:` | is a sequence, destructured |
| Mapping | `case {"k": v}:` | is a dict containing key `k` |
| Class | `case Point(x, y):` | is that type, destructured by attribute |
| guard | `case x if x > 0:` | pattern matches **and** condition holds |
| Dotted compare | `case Color.RED:` | compares against the value of `Color.RED` (not a binding) |

### Under the hood: what it actually does

`match` is compiled by CPython into a sequence of **dedicated matching bytecodes** (`MATCH_SEQUENCE`, `MATCH_MAPPING`, `MATCH_CLASS`, etc.) plus jumps; matching may call `__len__`, `isinstance`, key/attribute lookups, and so on. So it's **much heavier** than C's `switch` (a single jump-table instruction) — but in return you get expressiveness over **structure**, which `switch` can't give you. Run `import dis; dis.dis(describe)` to see these instructions.

> A one-line summary of `match`: **literals compare by value, bare names bind, `|` is OR, `[]`/`{}`/`Class()` destructure by structure, `if` adds a condition, `_` is the catch-all.**

## Summary

- **Usage level**: `if/elif/else` branches by indentation, and the conditional expression is `a if c else b`; `while`/`for` syntax is close to C's, and `for variable in sequence:` needs no index; a comprehension `[expr for x in sequence if condition]` is shorthand for building a container; `for`/`while` can carry an `else` (runs only if there's no `break`); `match-case` matches on literals/structure and destructures with binding, `_` is the catch-all, and only `X.abc` performs a comparison.
- **Under-the-hood level**:
  - The conditions of `if`/`while` go through the truth protocol (`__bool__` → `__len__` → always true); test for null with `is None`.
  - `for` is syntactic sugar over the iterator protocol (`__iter__`/`__next__`/`StopIteration`); values are actively "pulled" by the consumer repeatedly calling `__next__`, fundamentally unlike C's index-based for.
  - `range` is a lazy object holding only three integers; comprehensions have their own scope (inlined since 3.12+, no more anonymous function) and don't leak the loop variable; generator expressions make even the result lazy.
  - `match-case` compiles to dedicated matching bytecodes (`MATCH_SEQUENCE`/`MATCH_MAPPING`/`MATCH_CLASS`) — structural destructuring, not an integer jump table.
