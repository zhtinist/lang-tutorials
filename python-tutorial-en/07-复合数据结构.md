# Internal Implementation of Composite Data Structures

This chapter looks at the internal structure of Python's most commonly used containers — `list`, `tuple`, `dict`, `set`, and `deque` — inside CPython, comparing each one against C's arrays, structs, and hash tables, with a focus on what problem each implementation choice solves and what it gives up. The premise established in [「Variables and the Memory Model」](03-变量与内存模型.md) runs through the whole chapter: a Python container never holds a "value" itself, only a pointer to a `PyObject` — and this single fact determines the memory layout, copy semantics, and complexity of every container below.

## 1. Arrays of Pointers vs. Arrays of Values

C's array `int a[4] = {1,2,3,4}` looks like this in memory:

```
a:  [ 1 ][ 2 ][ 3 ][ 4 ]        // 16 bytes, values stored contiguously, cache-friendly
     ^0   ^4   ^8   ^12
```

Python's `a = [1, 2, 3, 4]` looks like this:

```
list object (heap)                each element is a separate heap object
+-----------------+
| ob_refcnt       |
| ob_type = list  |
| ob_size = 4     |----> ob_item (another heap block, an array of pointers)
| allocated = 4   |       +------+------+------+------+
| ob_item ------- |------>| ptr0 | ptr1 | ptr2 | ptr3 |
+-----------------+       +--|---+--|---+--|---+--|---+
                            v      v      v      v
                          PyLong PyLong PyLong PyLong
                           (1)    (2)    (3)    (4)   <- each on the heap, possibly shared elsewhere
```

A Python list is an "array of pointers," not an "array of values." This isn't an arbitrary implementation choice — it's the necessary extension, at the container level, of the runtime model where "everything is an object, and a variable is a reference" ([see 「The Runtime Model」](01-运行模型.md)): every slot in an array must be a uniform size so that a fixed stride like `ob_item[i]` can compute an address directly; `int`, `str`, and custom instances all have completely different sizes, so the only thing an array can uniformly hold is "a pointer to them." This conclusion runs through the whole chapter: every element access costs one extra pointer dereference, copying a container by default copies only pointers and not content, and a single array can mix elements of arbitrary different types (because a slot only ever holds a uniformly-sized pointer, and the type difference is pushed onto the object being pointed at).

## 2. `list`: Why a Dynamic Array Instead of a Linked List

```python
fruits = ["apple", "banana", "cherry"]
fruits.append("date")           # append at the end
fruits.insert(1, "kiwi")        # insert at index 1
fruits.remove("banana")         # remove by value (removes the first match)
last = fruits.pop()             # pop and return the last element
print(fruits[0], fruits[-1])    # apple kiwi   -- positive/negative indices
print(fruits[1:3])              # ['kiwi', 'cherry']  -- slicing
print("apple" in fruits, len(fruits))   # True 3

nums = [3, 1, 2]
nums.extend([9, 8])      # append elements from another iterable, one by one
nums.sort()               # sort in place
print(nums)               # [1, 2, 3, 8, 9]
print(nums.index(8), nums.count(2))     # 3 1
nums.reverse()
print(nums)               # [9, 8, 3, 2, 1]
```

The struct for a list in CPython (simplified from `Include/cpython/listobject.h`):

```c
typedef struct {
    PyObject_VAR_HEAD          // includes ob_refcnt, ob_type, ob_size (=len)
    PyObject **ob_item;        // points to a separately allocated array of pointers
    Py_ssize_t allocated;      // the allocated capacity (capacity >= ob_size)
} PyListObject;
```

The list object itself is a fixed-size header; the actual element pointers live in **another heap block** pointed to by `ob_item` — that's two allocations (the list header plus the pointer array), analogous to how a C++ `std::vector`'s `vector` object is separate from the buffer it `new[]`s. `ob_size` is the current length, `allocated` is how many pointers the buffer can hold, and `allocated >= ob_size` always holds; the empty slots in between exist so that append doesn't have to reallocate every time.

**Why a contiguous array of pointers instead of a linked list** (where each node stores one element and a `next` pointer, and insertion/deletion in the middle is, in theory, O(1)): this is a judgment about which operation is more common, not an arbitrary choice. In real code, random access by index (`lst[i]`), iteration, and appending at the tail are overwhelmingly the most frequent operations; inserting or deleting at an arbitrary position in the middle is comparatively rare. An array makes random access O(1) — `ob_item + i` computes the address directly; a linked list's random access is O(n) — you must walk from the head, one pointer at a time, to reach the i-th node. Choosing a linked list would sacrifice an operation that almost all code relies on in exchange for speeding up an operation that's used far less often — a poor trade for the default implementation of a general-purpose container. A linked list also carries a cost that's often overlooked: each node is a separate heap allocation, so elements aren't adjacent in memory, and cache misses during iteration are far more frequent than with a contiguous array — and in practice, "iterating" is an operation almost no container can avoid. When the standard library genuinely needs the "O(1) at both ends" property, it makes that a dedicated type ([`collections.deque`](#collections.deque-a-doubly-linked-list-of-blocks), covered later) instead of making `list` itself carry that trade-off.

Compared to `std::vector<T>`: a vector stores `T` values directly and contiguously (with `T=int`, contiguous ints). A Python list always stores pointers — even for an all-small-integers list, reaching a value costs one pointer dereference. That's why numerical computing uses `numpy` ([see 「Common Third-Party Libraries」](12-常用第三方库.md)): numpy's ndarray is a true contiguous-value C array.

On append, if `ob_size == allocated`, the list must grow. CPython doesn't simply double; it uses a mild growth formula (`list_resize`, simplified):

```c
/* Current CPython list_resize: over-allocate about 1/8, then round up to a multiple of 4 */
new_allocated = (new_size + (new_size >> 3) + 6) & ~3;
```

Measured (derived from `sys.getsizeof` on 3.9.6, where a list header occupies 56 bytes on a 64-bit platform and each pointer is 8 bytes), the growth sequence for `allocated` is:

```
0, 4, 8, 16, 24, 32, 40, 52, 64, 76, 92, 108, 128, ...
```

Compared with the 1.5x or 2x growth common in C++'s `std::vector`, CPython over-allocates only about **1/8**. This is another layer of trade-off between "wasted memory" and "number of reallocations": doubling means fewer reallocations, but each one can waste as much space as is currently in use; growing by 1/8 keeps the waste much lower, at the cost of needing more `realloc` calls for the same number of elements — CPython is betting that a list's typical append volume won't be extreme, and is willing to pay for a few more reallocations in exchange for using less memory. On growth, it calls `realloc` to swap in a larger block and copies the old pointer array over bitwise (copying only the pointers, not the element objects, and without touching any reference counts); the element objects stay exactly where they are, and references to them elsewhere remain valid:

```
append triggers growth:
old:  ob_item -> [p0][p1][p2][p3]                (allocated=4, size=4)
     realloc to a new buffer (allocated=8):
new:  ob_item -> [p0][p1][p2][p3][p4][ ][ ][ ]    (size=5)
     the element objects (for p0..p4) never moved
```

Complexity of each operation, and why:

| Operation | Complexity | Underlying reason |
|------|--------|---------|
| `lst[i]` index read/write | O(1) | Random access into the pointer array, `ob_item[i]` |
| `lst.append(x)` | Amortized O(1) | Direct write if there's a free slot; realloc only when full, amortized to a constant per call |
| `lst.pop()` (tail) | O(1) | `ob_size--`, DECREF the last element |
| `lst.insert(0, x)` | O(n) | Every following pointer must `memmove` one slot to the right |
| `lst.pop(0)` | O(n) | Every following pointer shifts left; use `deque` for frequent head operations |
| `del lst[i]` / middle deletion | O(n) | memmove to fill the hole |
| `x in lst` | O(n) | Linear scan, comparing each element with `==` |
| `len(lst)` | O(1) | Directly reads `ob_size` |
| slice `lst[a:b]` | O(k) | Build a new list, copy k pointers (a shallow copy!) and INCREF each |

This table is the quantified version of the design trade-off above: random access and tail operations are O(1), while inserting or deleting in the middle is O(n) — `list` allocates its complexity budget to the former. The proof of amortized O(1) is exactly the same as for `std::vector::push_back`: the total copying cost of n appends is the sum of all growth-copy costs, and since growth is geometric, that sum is O(n), which amortizes to O(1) per call.

`lst2 = lst[:]`, `list(lst)`, and `copy.copy(lst)` are all shallow copies: they build a new list header and a new pointer array, but the pointers point at the same batch of element objects.

```python
a = [[1, 2], [3, 4]]
b = a[:]              # shallow copy
b[0].append(99)       # modifies the shared inner list
print(a)              # [[1, 2, 99], [3, 4]]  -- a changed too!
```

```
a.ob_item -> [p0][p1]
                \    \
                 v    v
             [1,2,99] [3,4]      <- there's only one copy of the inner objects
                 ^    ^
b.ob_item -> [p0][p1]            <- b is a new array, but its pointers point at the same inner objects
```

For true isolation, use `copy.deepcopy`, which recursively rebuilds every layer of objects. This is the same design principle from [「Variables and the Memory Model」](03-变量与内存模型.md) — "assignment only copies a reference, shallow copy only duplicates one layer" — showing up again at the container level: the cheap operation is the default, and the expensive one must be requested explicitly.

## 3. `tuple`: An Immutable Fixed-Length Record

```python
point = (3, 4)
x, y = point                    # unpacking
print(point[0])                 # 3
# point[0] = 9                  # TypeError: a tuple can't be modified

t = (1, [2, 3])
t[1].append(4)     # OK, modifies the inner list
print(t)            # (1, [2, 3, 4])
# t[1] = [9]        # TypeError: can't reassign a tuple's slot
```

A tuple's reason for existing isn't just "an immutable list." A list expresses a homogeneous, variable-length **collection**, where an index is just a position and elements are usually of the same type; a tuple expresses a heterogeneous, fixed-length **record**, where an index corresponds to a field with a fixed meaning — in `(x, y)`, index 0 is always x and index 1 is always y, the same semantics as C's `struct { int x; int y; }`, just accessed by position instead of by name. These are two different design intents; immutability is a means of reinforcing the semantics of "this is a fixed-length record, elements shouldn't be inserted or removed here," not the sole reason tuple exists. For named fields instead of pure position, see [`namedtuple`](11-标准库算法.md).

**Why tuple's immutability is also what lets it serve as a dict key or go inside a set**: [「Variables and the Memory Model」](03-变量与内存模型.md) already argued that one benefit of immutability is that it can be safely used as a dict key. A hash table's correctness depends on one premise: a key's hash value must stay constant for as long as the key is alive, because the hash value determines the key's physical location in the table, and a later lookup has to use that same hash value to find it again. If a tuple were mutable, changing one of its elements would change its hash value, but the key would already be sitting in some slot chosen according to the old hash — the slots implied by the old and new hash no longer match, and the key is effectively "lost" in the table, never to be found again. A list allows in-place modification, so the language simply doesn't define `__hash__` for it and forbids it from being a key, rather than leaving a trap that "works fine until you mutate the content, and then produces bizarre bugs." A tuple is only hashable when all of its elements are hashable (its hash is combined from its elements' hashes), which is also why a tuple containing a list, like `(1, [2, 3])`, can't be a dict key.

The tuple struct (`PyTupleObject`) is simpler than list's — it **inlines** the pointer array at the tail of the object instead of allocating it separately:

```c
typedef struct {
    PyObject_VAR_HEAD
    PyObject *ob_item[1];   // variable-length tail array, allocated once at creation, never changed
} PyTupleObject;
```

There's no `allocated` field; the length is fixed once the tuple is created, and it's a single allocation (the object header and element pointer array share one block), which is one fewer allocation and one fewer layer of indirection than a list, and more compact — similar to a C flexible array member. This more compact layout is exactly what's bought by the convention "the length is fixed and never changes": a list must leave room to grow and must be able to `realloc` as a whole; a tuple gives up those abilities in exchange for a cheaper creation cost and one fewer pointer hop on access. CPython also keeps a free-list cache for small tuples, so frequently creating and destroying small tuples can reuse memory blocks, reducing malloc pressure.

## 4. `dict`: How a Compact Array Plus an Index Table Preserves Order While Staying a Hash Table

`dict` is Python's most central data structure — it's not only the user-facing hash table, but also the underlying carrier for object attributes (`__dict__`), module namespaces, and keyword arguments, which is why its implementation has been optimized again and again.

```python
d = {"a": 1, "b": 2}
d.update({"b": 20, "c": 3})   # batch update / insert
print(d.pop("a"))             # 1 -- delete and return the value for that key
print(d.setdefault("z", 0))   # sets a default if the key is missing and returns it; returns the existing value otherwise
squares = {x: x * x for x in range(5)}   # dict comprehension
print(squares)                            # {0: 0, 1: 1, 2: 4, 3: 9, 4: 16}
```

Python's dict uses **open addressing**, not chaining: on a collision, it doesn't hang a linked list off the bucket, but instead follows a probe sequence to find the next empty slot, and all entries live in the same contiguous array, which is more cache-friendly. A key must be **hashable** — it implements `__hash__` and its hash value doesn't change over the object's lifetime; immutable built-in types (int/str/tuple/frozenset) are hashable, list/dict/set are not, for exactly the reason covered in the tuple section above: a mutable object's hash value can drift as its content changes, which would corrupt the hash table's correctness. Average lookup/insert/delete is **O(1)**; the worst case (heavy collisions) is O(n), which is extremely rare in practice.

Since Python 3.6, dict has remembered insertion order (a language guarantee since 3.7). **Why this guarantee isn't implemented by bolting an extra linked list on** — the way Java's `LinkedHashMap` threads a doubly-linked list through every entry of an ordinary hash table to record insertion order, which requires storing two extra pointers per entry and jumping along the list on iteration, both cache-unfriendly. CPython 3.6's compact dict took a different approach: it split "hash positioning" and "data storage" — previously one array — into two arrays:

```
indices (the hash table, storing indices into entries; empty slots hold -1):
+----+----+----+----+----+----+----+----+
| -1 |  1 | -1 | -1 |  0 | -1 |  2 | -1 |
+----+----+----+----+----+----+----+----+

entries (compact array, appended in insertion order, no holes):
+---------+--------+---------+
|hash key val|hash key val|hash key val|
|   'a':1   |  'b':2  |  'c':3  |
+---------+--------+---------+
   idx 0      idx 1     idx 2
```

The sparse `indices` array has exactly one job — "given a key, find its index into entries" — so it can be quite sparse (low load factor), yet each slot only takes up a small integer (1/2/4/8 bytes depending on table size), so the waste is small. The real `(hash, key, value)` triples are packed into the compact `entries` array, appended contiguously in insertion order with no holes. Iterating a dict is just a sequential scan of this `entries` array — preserving order is entirely a natural side effect of this physical layout, with no need to maintain linked-list pointers specifically for it, and it also makes iteration itself faster (a contiguous memory scan vs. jumping along a list). Compared to the pre-3.5 sparse large table where "every slot stores a full entry," this split also happens to save about **20%–25%** memory — a bonus gained along the way, not the main goal of the design.

Verified: a dict preserves order, and the order is decided by "insertion," not "sorting." Deleting a key in the middle and re-inserting it appends it to the end; updating an existing key changes its value in place without moving its position:

```python
d = {}
d["c"] = 1; d["a"] = 2; d["b"] = 3
print(list(d))     # ['c', 'a', 'b']   insertion order, not sorted
del d["a"]         # delete the key in the middle
d["a"] = 9         # re-insert -- appended to the end of entries
print(list(d))     # ['c', 'b', 'a']
d["c"] = 100       # update an existing key: value changed in place, order unchanged
print(list(d))     # ['c', 'b', 'a']
```

The lookup process: hash the key → take the low bits as the `indices` subscript → get the entries index → go to entries and compare hash and key (compare the integer hash first, and only compare with `==` if that matches, since hashes can collide). CPython uses perturbation probing to find the next candidate slot, rather than simple linear probing:

```c
perturb = hash;
i = hash & mask;
while (slot_used_and_key_differs) {
    perturb >>= 5;
    i = (i * 5 + perturb + 1) & mask;
}
```

Simple linear probing (`i = i + 1`) tends to produce "clustering" as the load factor rises — once a region fills up, every subsequent colliding key has to probe through that whole long run of occupied slots, and lookup performance degrades noticeably. Perturbation probing continually stirs the high bits of the hash value into the computation of the next candidate position, spreading out the probe sequence and reducing clustering — a small amount of extra computation traded for a more even probe distribution.

When the load factor exceeds **2/3**, the dict grows. The new table's capacity is the smallest power of two that can hold "3 times the currently used entries" (roughly 2x–4x growth). Growth **rebuilds the indices table** (re-scattering every key, since the table capacity changed and each key's hash modulo the new mask changes too), but entries is carried over with its insertion order intact; growth also cleans up the "tombstones" left behind by deletions along the way.

| Operation | Average | Worst | Notes |
|------|------|------|------|
| `d[k]` lookup | O(1) | O(n) | Hash + probe |
| `d[k] = v` insert/update | O(1) amortized | O(n) | May trigger a growth re-scatter |
| `del d[k]` | O(1) | O(n) | Marked deleted; the entries slot left empty (skipped on iteration) |
| `k in d` | O(1) | O(n) | Same as lookup |
| iteration | O(n) | O(n) | Sequential scan of entries |
| `len(d)` | O(1) | O(1) | Reads a counter |

Compared to C++: `std::unordered_map` uses chaining (each bucket hangs a linked list/nodes, each node separately `new`'d, cache-unfriendly); `std::map` is a red-black tree, O(log n) and sorted by key. Python's dict is an open-addressing hash table that additionally trades the "compact entries + sparse indices" split for an insertion-order guarantee — something neither a pure open-addressing nor a pure chaining implementation gets for free.

## 5. `set` and `frozenset`

```python
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}
print(a & b)   # {3, 4}              intersection
print(a | b)   # {1, 2, 3, 4, 5, 6}  union
print(a - b)   # {1, 2}              difference (in a, not in b)
print(a ^ b)   # {1, 2, 5, 6}        symmetric difference (in exactly one)

fs = frozenset([1, 2, 3])    # an immutable set
s = {fs, frozenset([4])}     # a frozenset is hashable, so it can go inside another set
print(len(s))                 # 2

sc = {x % 3 for x in range(10)}   # set comprehension
print(sc)                          # {0, 1, 2}
```

A common pattern — using a set to check "have I seen this before," averaging O(1), far faster than a list's O(n):

```python
seen = set()
data = [1, 2, 2, 3, 1]
for x in data:
    if x not in seen:
        seen.add(x)
print(seen)   # {1, 2, 3}
```

`set`'s implementation is highly similar to dict's — essentially "a hash table with keys but no values," likewise open-addressed and likewise requiring hashable elements. **Why set doesn't reuse dict's "indices + entries" split**: the entire point of that split is to trade the compact array's physical ordering for an insertion-order guarantee, and a set doesn't promise any order in the first place — it models a set in the mathematical sense, where elements have no relative ordering, so there's no reason to pay extra memory and an extra layer of indirection for a guarantee nobody needs. So set uses a table closer to the traditional sparse layout (an entry is directly hash + key), without introducing dict's dual-array structure. Don't rely on a set's iteration order — it carries no guarantee whatsoever.

**Why frozenset is needed**: a set is itself mutable, and therefore unhashable — it can't go inside another set or serve as a dict key, exactly the same rule as with tuple/list. But "needing an immutable, hashable set" is a real requirement — for example, using a whole group of tags together as a cache key, or deduplicating a "set of sets." `frozenset` exists specifically to fill that need: once created, it can no longer `add`/`remove`, so its hash value can be safely computed and cached, letting it serve as a dict key or go inside another set (as in the example above).

| Operation | Average complexity |
|------|-----------|
| `add` / `remove` / `in` | O(1) |
| intersection `a & b` | O(min(len a, len b)) — iterate the smaller set and probe the other |
| union `a` \| `b` | O(len a + len b) |
| difference `a - b` | O(len a) |

## 6. `collections.deque`: A Doubly-Linked List of Blocks

A list's head operations are O(n). When you need efficient insertion and removal at both ends (queues, sliding windows), use `collections.deque` — its usage is much like list's, but with the addition of `appendleft`/`popleft`:

```python
from collections import deque

dq = deque([1, 2, 3])
dq.append(4)         # append at the tail
dq.appendleft(0)      # append at the head
print(dq.pop())        # 4 -- pop from the tail
print(dq.popleft())    # 0 -- pop from the head
print(dq)               # deque([1, 2, 3])
```

The `maxlen` argument turns it into a fixed-length ring buffer, commonly used for "the most recent N records":

```python
from collections import deque

recent = deque(maxlen=3)
for x in [1, 2, 3, 4, 5]:
    recent.append(x)
print(recent)   # deque([3, 4, 5], maxlen=3) -- overflow automatically pushed out from the other end
```

Its implementation is **not** an ordinary doubly-linked list (which would mean one node per element and terrible cache behavior), but a **doubly-linked list of fixed-size blocks**:

```
block (64 pointers each)     block                block
+----------------+       +----------------+     +----------------+
| prev | items...|  <->  | prev | items...| <-> | prev | items...|
| next |         |       | next |         |     | next |         |
+----------------+       +----------------+     +----------------+
   ^ leftblock                                     ^ rightblock
```

**Why not one node per element, like a standard linked list**: every node in a plain linked list is a separate heap allocation, so elements aren't adjacent in memory and cache misses during iteration are severe; each node also has to store two extra pointers, `prev` and `next`, and for an element as small as an integer, that overhead is substantial relative to the payload. `deque` packs a number of elements into a fixed-size (64) block, so the linked-list structure exists only at the block level, while within a block it's a contiguous array — effectively "a linked list of arrays," trading a coarser linking granularity for most of the cache locality back, while still keeping O(1) insertion/removal at both ends (shift one position within the head/tail block, and chain on a new block when it's full, with no need to shift everything the way list does). The cost is that random indexing `dq[i]` becomes O(n) (you have to walk the block chain) — a direct consequence of the trade-off "optimize both ends, give up random access in the middle": by design, `deque` was never meant to be a container for random access.

| Operation | list | deque |
|------|------|-------|
| tail append/pop | O(1) amortized | O(1) |
| head insert/pop | O(n) | O(1) |
| random index `x[i]` | O(1) | O(n) |

|------|------|---------|---------|---------|---------|------|---------|

## 8. Summary of the Memory-Model Comparison with C

| C/C++ | Python | Key difference |
|-------|--------|---------|
| `int a[N]` | `list` | C stores values contiguously; a list stores pointers, values scattered on the heap |
| `std::vector<T>` | `list` | vector stores values and can hold small objects on the stack; a list always stores pointers |
| `struct {int; char*;}` | `tuple` / `namedtuple` / class | a struct lays out fields by offset; a tuple stores pointers by index |
| `std::unordered_map` | `dict` | chaining vs. open addressing; dict additionally preserves order |
| `std::unordered_set` | `set` | same as above |
| `std::deque` | `deque` | conceptually close, both block-based |

Core takeaway: **Python containers are containers of references.** Copying a container by default copies only pointers (a shallow copy), and multiple containers can alias the same batch of elements; true independence requires `deepcopy`. If you want C-style contiguous value storage, cache-friendliness, and vectorizable arrays, use [`array`/`memoryview`](11-标准库算法.md) or [`numpy`](12-常用第三方库.md).

## Summary

- `list` is a dynamic array of pointers to `PyObject`: an array was chosen over a linked list because random access and tail appends are far more common than middle insertion/deletion, and an array makes the former O(1) while a linked list can't; the mild 1/8 over-allocation trades off "number of reallocations" against "wasted memory"; a shallow copy duplicates only pointers, not elements.
- `tuple` inlines its pointer array at the tail of the object, allocated once and immutable; immutability is the precondition for it being usable as a dict key — a mutable object's hash value can drift as its content changes, breaking hash-table positioning, echoing exactly the mutability discussion from Chapter 03.
- `dict` implements an open-addressing hash table with two arrays — a sparse `indices` array (hash positioning) and a compact `entries` array (data storage, insertion-ordered): preserving order is a side effect of the compact array's physical layout, with no need to bolt on an extra linked list the way Java's `LinkedHashMap` does.
- `set` is a hash table with no values and no ordering guarantee, so it doesn't need dict's dual-array structure; `frozenset` exists for the real need of "an immutable, hashable set."
- `deque` is a doubly-linked list of blocks: fixed-size blocks buy back cache locality within a linked-list structure, giving O(1) at both ends and O(n) random access.

The next chapter moves into object orientation and the type system, where you'll see that a Python instance's attributes are, by default, also a `dict` — the same underlying hash-table mechanism as this chapter's.
