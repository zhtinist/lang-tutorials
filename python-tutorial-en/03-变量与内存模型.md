# Variables and the Memory Model ⭐

For C/C++ engineers, most of Python's "paranormal events"—changing one list and another changes with it, modifying a parameter inside a function and having the outside change too, a default argument remembering last time's value—all trace back to one thing: **a Python variable is not a memory slot, but a binding of a name to a heap object.** Once this model is nailed down, every seemingly bizarre behavior that follows becomes predictable. This chapter proves it repeatedly with `id()`, `is`, and `sys.getrefcount`, paired with memory diagrams throughout.

## 1. A Variable Is Not a Box, It's a Name Tag

C's mental model is "the box": `int x` is a named chunk of memory, `x = 3` stuffs a value into the box, `x = 5` overwrites it—the variable **is** that memory.

Python is nothing like this. Recall [The Execution Model](01-运行模型.md): every value is a `PyObject` on the heap. A variable is merely **a name in a namespace**, bound to some heap object—you can think of it as a "sticky note attached to an object," or: **a variable is a pointer holding an object's address, except you can never take its address, and you can't do pointer arithmetic on it.**

```
C's model: variable is memory       Python's model: name points to heap object
┌─────────┐                name x         heap
│  x: 3   │  ← box is the  ┌────┐        ┌──────────────┐
└─────────┘     value      │ ●──┼──────► │ int object: 3│
                           └────┘        └──────────────┘
                           sticky note     the real value lives here
```

**Why does Python choose this "name binding" model instead of C's "memory slot" model?** The root cause is, again, dynamic typing. In C, once `int x` is declared, this chunk of memory's size and how it's interpreted (4 bytes, read as a two's-complement integer) are both fixed at compile time—the variable and the value it stores are naturally one and the same thing. In Python, the same name can point first to an `int`, then a `str`, then any arbitrary custom object—these types have completely different sizes and internal layouts, so there's no way for a fixed-size "box" to hold a value of arbitrary type. The variable has to store only one uniformly sized thing (a pointer), leaving the actual data and type information in the heap object being pointed to. The name-binding model is the inevitable memory-level implementation of dynamic typing, not an arbitrary design preference.

`id(obj)` returns an object's identity, **which in CPython is the object's memory address** (the language only guarantees it's unique and constant within a single lifetime; the address is an implementation detail). It's the equivalent of `&`, except read-only and applied to objects rather than variables:

```python
x = 3
print(id(x))   # e.g. 4310721234, the heap address of that int object
```

## 2. What Does `a = b` Actually Copy

```python
a = [1, 2, 3]
b = a
b.append(4)
print(a)        # [1, 2, 3, 4]
```

A C++ engineer's first reaction is usually "didn't `b = a` make a copy? How does modifying b move a?"—because `b = a` **copies neither the object nor the data, only that pointer value**. After this executes, `a` and `b` are two sticky notes attached to **the same heap object**:

```
a = [1, 2, 3]
b = a

  name a ─┐
          ├──► ┌──────────────────────┐
  name b ─┘    │ list object          │
               │ [ ●, ●, ● ]          │ ← the same object
               └──┼──┼──┼─────────────┘
                  ▼  ▼  ▼
                  1  2  3   (each also a heap int object)
```

`b.append(4)` reaches that list object through b and **modifies it in place**; a points to the same object, so of course it looks changed from a's side too. This isn't a bug in copy semantics; it's **aliasing**—two names are two aliases of the same object. The C++ analogy: `b = a` is more like `int* b = a;` (copying a pointer), and never `std::vector<int> b = a;` (copy construction). **Python assignment is always "copy the reference," never a call to a copy constructor**—behind this is a deliberate design trade-off: if assignment defaulted to a deep copy, then any plain `b = a` where a happens to be a large object would silently imply an expensive full copy, when in most cases what the programmer actually wants is just "give this object another name." Python makes "cheap, always O(1)" the basic contract of assignment semantics; when you genuinely need a copy, you must explicitly call `copy`/`deepcopy` (see section 7)—the cost is stated up front in the semantics, which beats hiding an unpredictable performance cost in the default behavior.

`==` compares whether **values** are equal (calls `__eq__`); `is` compares whether it's **the same object** (compares `id`), equivalent to comparing two pointers in C:

```python
a = [1, 2, 3]
b = a
c = [1, 2, 3]          # build another list with identical contents

print(a == b)          # True, values equal
print(a is b)          # True, same object (aliases)
print(a == c)          # True, values equal
print(a is c)          # False, different objects
```

`a is b` being true while `a is c` is false proves that `b = a` created an alias, while `c = [1,2,3]` is a brand-new object. Use `is` to ask "is it the same thing," `==` to ask "are the values equal" (the `is None` idiom comes precisely from here—`None` is a singleton, so identity comparison is both correct and skips the cost of calling `__eq__`).

## 3. Reference Counting: Who Is Pointing at Me

Recall [The Execution Model](01-运行模型.md): every `PyObject` header carries `ob_refcnt`. Each additional name/container pointing at it adds 1; each one removed subtracts 1; hitting zero reclaims it immediately (CPython's primary memory-management mechanism—reference cycles are handled separately by a garbage collector, see [Error Handling and Resource Management](09-错误处理与资源管理.md)).

```python
import sys

x = object()
print(sys.getrefcount(x))   # e.g. 2

y = x                        # one more reference
print(sys.getrefcount(x))   # 3

del y                        # remove one reference (del unbinds a name, it does not delete the object!)
print(sys.getrefcount(x))   # back down to 2
```

`sys.getrefcount(x)`'s result is always one higher than intuition suggests, because passing `x` as an argument to the function itself temporarily adds one more reference. `del y` deletes the **name binding**, not the object—the object is only truly freed once its refcnt hits zero, which is completely unlike C++'s `delete` semantics; `del` is more like "letting go of one hand," and the object is only gone once the last hand lets go.

**Why does CPython choose reference counting as its primary reclamation mechanism, instead of relying purely on a tracing GC the way Java/Go do?** Reference counting's core advantage is **determinism**—an object is freed the instant its last reference disappears, giving a smooth memory-usage curve with no need for a dedicated GC pause to scan the whole heap. That mattered a lot in the 1990s, when memory was generally tight and Python was, from the very start, meant to interoperate with a huge amount of C extension code: a C extension can manage references directly with `Py_INCREF`/`Py_DECREF` without needing to understand a complex tracing collector. The cost is equally clear: reference counting itself has maintenance overhead (almost every assignment and argument pass has to update a count), and it **can't handle reference cycles** (two objects referencing each other will never have either count truly reach zero). CPython therefore layers a dedicated generational garbage collector for cycle detection on top of reference counting as a patch—the two work together rather than being an either/or choice.

## 4. Mutable vs. Immutable: the Line That Determines Aliasing Behavior

```python
a = 1
b = a
b = b + 1
print(a, b)     # 1 2 —— a didn't change
```

Why doesn't this have the "modifying b moves a" problem? The difference is that list is **mutable**, while int is **immutable**.

- **Mutable objects** (list, dict, set, most custom class instances): their contents can be modified in place; `b.append(4)`, `b[0] = 9` change the object itself, and every alias sees it.
- **Immutable objects** (int, float, str, tuple, bool, frozenset, None): their contents cannot change; any "modification" operation actually **builds a new object** and rebinds the name to point to it.

Under the hood of `b = b + 1`: `b + 1` computes a **new** int object `2`, and `b =` rebinds the name b to this new object; the original `1` object is left completely untouched, and a still points to it:

```
a = 1; b = a                b = b + 1

 a ─┐                        a ────► [ int 1 ]   ← a still points to the old object
    ├──► [ int 1 ]           b ────► [ int 2 ]   ← b now points to a new object
 b ─┘                        (newly allocated object)
```

**Why does Python design an "immutable object" category at all, instead of making everything mutable (which would fit more cleanly with the unified model of "everything is an object, a variable is just a reference")?** Immutability buys three concrete benefits, worth giving up the simplicity of "everything is uniformly mutable" for: first, immutable objects can naturally be shared safely—multiple names pointing at the same `int`/`str` object don't need to worry that some code elsewhere quietly changed it, and this is the precondition that lets sharing optimizations like small-integer caching and string interning (section 5) exist safely at all; second, immutable objects can safely be used as `dict` keys or put into a `set` (if a mutable object's hash value changed along with its contents, you'd lose track of it in a hash table the moment its contents changed, so Python simply disallows mutable types from being hashable by default); third, functional-programming style code is relieved of the mental burden of worrying "will the value I passed in get modified behind my back." The cost is that modifying the content requires constructing a new object, which has a real performance cost in scenarios like frequent string concatenation (avoidable with mutable containers in the `list`/`StringBuilder` style—see [Composite Data Structures](07-复合数据结构.md)).

Compare this with in-place modification of a mutable object:

```
a = [1,2,3]; b = a          b.append(4)

 a ─┐                        a ─┐
    ├──► [1,2,3]                ├──► [1,2,3,4]   ← the same object is modified
 b ─┘                        b ─┘                  both names see the change
```

The core rule: **in-place modification** of a mutable object affects every alias; **rebinding** (`b = ...`) of any object only changes where the name b itself points. Immutable objects have no "in-place modification" path at all, so their aliases are always safe.

One pitfall: `+=` behaves differently for mutable and immutable objects. `b += [4]` on a list is an in-place extension (equivalent to `extend`, affecting aliases); `b += 1` on an int is a rebind (not affecting aliases). The reason is that mutable objects implement an in-place `__iadd__`, while immutable objects don't, so `+=` falls back to plain `b = b + 1`:

```python
a = [1, 2]
b = a
a += [3]        # a list's += is an in-place extension (__iadd__), still the same object
print(b)        # [1, 2, 3]   alias b follows the change

a = a + [4]     # + builds a new list, then rebinds a to point at it
print(b)        # [1, 2, 3]   b no longer follows
```

## 5. Small-Integer Caching and String Interning

Both of these are CPython implementation details, not language semantics—but you will definitely run into them when using `is`.

CPython pre-creates every integer object from `-5` to `256` at startup and caches them for reuse; identical values within this range are **the same object**:

```python
a = 100
b = 100
print(a is b)      # True, both point to the one cached 100 object

m = int("257")
n = int("257")
print(m is n)      # False, the two int() calls each built a fresh 257 object on the heap
print(m == n)      # True, the values are of course equal
```

**Why do this caching at all?** Small integers get used extremely frequently in real code (loop counters, indices, and the machinery behind boolean checks are all small integers under the hood); pre-building them once and reusing them forever avoids the overhead of repeatedly allocating and freeing objects for the same common value—a classic case of trading a small chunk of permanently resident memory for avoiding a large number of repeated runtime allocations. The range `[-5, 256]` is a CPython implementation choice hardcoded into the interpreter, and other implementations or versions may differ—**always use `==` to test numeric equality, never `is`**; `is` merely happens to "work" for small integers, and outside that range it will give you a misleading `False`.

String interning works the same way: CPython interns certain strings (especially literals that look like identifiers) into a global table, so identical contents share the same object, letting dict key lookups and identifier comparisons check pointers (`is`) first for a fast equality test, rather than comparing content character by character:

```python
a = "hello"
b = "hello"
print(a is b)      # usually True, interned

import sys
u = "hello world!".upper().lower()
v = "hello world!".upper().lower()
print(u is v)                              # usually False, built at runtime, not interned
print(sys.intern(u) is sys.intern(v))      # True, after manual interning they share the same object
```

The conditions that trigger interning vary by version and context—**never rely on it**; here too, only ever use `==` to test equality.

## 6. Namespaces Are Dictionaries, but Local Variables Are Fast Locals

A module's global namespace and an object's attributes really are, under the hood, dicts (keys are name strings, values are object references):

```python
x = 10
print(globals()["x"])    # 10, the global namespace is just a dict
```

But local variables inside a function are an important exception. When CPython compiles a function, it can statically count exactly how many local variables there are and what each one is, so it places them into a **fixed-length array** ("fast locals"), accessed by index, rather than looking them up in a dict by string every time—recall the `LOAD_FAST` we saw with `dis` in [The Execution Model](01-运行模型.md):

```python
import dis

g = 1
def f(a):
    b = a + g       # a, b are locals (fast); g is global
    return b

dis.dis(f)
```

```
  4           RESUME                   0

  5           LOAD_FAST_BORROW         0 (a)
              LOAD_GLOBAL              0 (g)
              BINARY_OP                0 (+)
              STORE_FAST               1 (b)

  6           LOAD_FAST_BORROW         1 (b)
              RETURN_VALUE
```

(`LOAD_FAST_BORROW` is 3.14's optimized variant for reading a local—it "borrows" the reference, skipping refcnt increments/decrements, but it's still fundamentally an indexed local access underneath; globals are still `LOAD_GLOBAL`.)

`LOAD_FAST`/`STORE_FAST` use an array index, O(1), no hashing, no string comparison; `LOAD_GLOBAL` goes through a dict lookup (this module's globals first, then builtins)—noticeably more expensive. **Why optimize function locals into an array specifically, instead of uniformly using a dict the way the global namespace does?** A function body is statically known at compile time—which local variable names a function uses, and how many there are in total, can be determined by a single pass over the source, and this set never grows or shrinks dynamically at runtime (unlike module globals or object attributes, where new names can be added arbitrarily at runtime). Since the set of names is already fixed at compile time, there's no need to pay the cost of hashing and string comparison on every access—you can just allocate a fixed-length array and map each name to a fixed index at compile time. This is a concrete instance of the optimization principle "don't leave information to be recomputed at runtime if it can already be determined at compile time"; the module-level global namespace can't get this same optimization, because it has to support adding and removing names dynamically at runtime (e.g. via `exec` or a dynamic `import`).

This explains a common performance trick: in a hot loop, "caching" a frequently accessed global/attribute into a local variable is exactly swapping `LOAD_GLOBAL` (dict lookup) for `LOAD_FAST` (array lookup). It also explains why reading and writing a global inside a function requires a `global` declaration—the compiler decides whether to treat a name as local or global based on whether the function assigns to that name anywhere in its body, and this decision has to be made at compile time; it can't be deferred to runtime and decided situationally. The full story is in [Functions and the Call Stack](06-函数与调用栈.md).

## 7. Shallow Copy vs. Deep Copy

`b = a` only copies the pointer; if you actually want a copy of the data, use the `copy` module, and you need to distinguish shallow from deep copy—the same question as in C++: "is copy construction shallow or deep, do I need to write my own deep copy."

```python
import copy

original = [[1, 2], [3, 4]]

alias    = original                 # alias, same object
shallow  = copy.copy(original)      # shallow copy: new outer container, inner elements still shared
deep     = copy.deepcopy(original)  # deep copy: recursively copies every layer

original[0].append(99)

print(alias)    # [[1, 2, 99], [3, 4]]   changes too (same object)
print(shallow)  # [[1, 2, 99], [3, 4]]   changes too, the inner layer is shared
print(deep)     # [[1, 2], [3, 4]]       independent, unchanged
```

```
original ─► [ ●, ● ]            outer list
              │  │
              ▼  ▼
           [1,2] [3,4]          two inner lists

shallow = copy.copy(original):
shallow ─► [ ●, ● ]  ← a new outer object
             │  │
             └──┴──► point to the same [1,2] and [3,4]  (the inner layer wasn't copied)

deep = copy.deepcopy(original):
deep ─► [ ●, ● ]  ← new outer
          │  │
          ▼  ▼
       [1,2]'[3,4]'  ← even the inner layer is new objects, fully independent
```

**Why does the default shallow copy only copy one layer, instead of just always deep-copying, which would seem safer?** Nested structures can be arbitrarily deep; blindly deep-copying a large nested object (say, a big tree or graph) is expensive, and deep copy also needs to correctly handle reference cycles (an object referencing itself), which is more complex to implement. Shallow copy hands the decision of "how deep to copy" back to the caller—in most scenarios the container itself needs to be independent, but it's perfectly acceptable for the objects inside to remain shared (e.g. copying a "list of pointers," where what the pointers point to is meant to be shared anyway). This is a classic instance of the design convention "provide a cheap default operation; the expensive operation must be requested explicitly," the same principle behind "assignment defaults to copying a reference rather than deep-copying" from earlier.

Other common ways to produce a shallow copy—`list(x)`, `x[:]`, `dict(x)`, `x.copy()`—are all shallow, just like `copy.copy`; for full independence you must use `deepcopy`, which carries the cost of recursive traversal and extra allocation and shouldn't be reached for thoughtlessly.

## 8. The Mutable Default Argument Trap

With this chapter's model in hand, you can now make sense of one of Python's most famous pitfalls:

```python
def add_item(item, bucket=[]):    # dangerous: the default [] is created only once
    bucket.append(item)
    return bucket

print(add_item(1))   # [1]
print(add_item(2))   # [1, 2]   you might expect [2], but it remembered last time
```

The reason: the default value `[]` is a mutable object **created once when `def` executes**, held onto by the function object forever; every call that omits `bucket` reuses the same list, and the in-place `append` naturally accumulates. This is the inevitable result of three things converging: mutable object + aliasing + created only once. The `def` statement itself is a statement that executes at runtime (see section 3 of [The Execution Model](01-运行模型.md)), and the default-value expression is evaluated exactly once, at the moment that `def` statement executes—not re-evaluated on every call. The rationale behind this design, and the full mechanism, are covered in [Functions and the Call Stack](06-函数与调用栈.md).

## Summary

- A variable is a binding of a name to a heap object, not a memory box—this is the inevitable memory model that dynamic typing produces, not an arbitrary choice.
- `a = b` copies only the reference, never the object or the data—the result is aliasing; `is`/`id()` tell you whether it's the same object, `==` tells you whether the values are equal.
- Reference counting (`ob_refcnt`) is what CPython trades for "deterministic reclamation" and "easy C-extension integration," at the cost of needing an extra cyclic garbage collector as a patch; `del` unbinds a name, it doesn't delete the object.
- The mutable/immutable divide governs aliasing behavior: immutability is traded for safe sharing, hashability, and lower mental burden, at the cost that modifying content requires building a new object.
- Small-integer caching and string interning are CPython implementation optimizations that trade a small chunk of resident memory for avoiding repeated allocation—they are not language guarantees, so always test equality with `==`.
- Function local variables use a fast-locals array instead of a dict, a concrete instance of "don't recompute at runtime what's already known at compile time"; the module global namespace can't get the same optimization because it must support adding and removing names dynamically at runtime.
- Shallow copy copying only one layer reflects the convention "the default operation is cheap, the expensive operation must be requested explicitly"; the mutable default argument trap is a direct corollary of "the `def` statement executes only once, and the default value is only evaluated once."
