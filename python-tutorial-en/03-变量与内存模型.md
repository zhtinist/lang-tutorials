# Variables and the Memory Model ⭐

This is the most important chapter in the book. If you read only one chapter, read this one.

For C/C++ engineers, 90% of Python's "paranormal events"—changing one list and another changes too, modifying a parameter inside a function affects the outside, a default argument remembering last time's value—all stem from the same thing: **a Python variable is not a memory slot but a binding of a name to a heap object.** Once this model takes root in your mind, all the weirdness becomes predictable.

We'll repeatedly demonstrate this empirically with `id()`, `is`, and `sys.getrefcount`, and draw plenty of memory diagrams.

## 1. A Variable Is Not a Box, It's a Name Tag

C's mental model is "the box": `int x` is a named chunk of memory, `x = 3` stuffs a value into the box, and `x = 5` overwrites it. The variable **is** that memory.

Python is nothing like that. Recall [Chapter 01](01-运行模型.md): all values are `PyObject`s on the heap. A variable is merely **a name in a namespace**, bound to some heap object. You can picture it as a "sticky note attached to an object," or—to put it in terms you know—**a variable is a pointer holding an object's address, except you can never take its address and can't do pointer arithmetic on it**.

```
C's model: variable is memory       Python's model: name points to heap object
┌─────────┐                name x         heap
│  x: 3   │  ← box is the  ┌────┐        ┌──────────────┐
└─────────┘     value      │ ●──┼──────► │ int object: 3│
                           └────┘        └──────────────┘
                           sticky note     the real value is here
```

The precise reading of `x = 3` is not "store 3 into x" but "**bind the name x to the object representing 3**." This distinction is the foundation of everything in this chapter.

### Observing Object Identity with id()

`id(obj)` returns the object's identity; **in CPython it is the object's memory address** (the language only guarantees it's unique and constant within a single lifetime; the address is an implementation detail). It's the `&` you want—except it's read-only and applies to objects rather than variables.

```python
x = 3
print(id(x))   # e.g. 4310721234, the heap address of that int object
```

## 2. What Does a = b Actually Copy

Look at this and guess the result first:

```python
a = [1, 2, 3]
b = a
b.append(4)
print(a)        # ???
```

The answer is `[1, 2, 3, 4]`. A C++ engineer's first reaction is "didn't `b = a` make a copy? How does modifying b move a?"

Because `b = a` **copies neither the object nor the data, only that pointer value**. Afterward, `a` and `b` are two sticky notes attached to **the same heap object**.

```
a = [1, 2, 3]
b = a

  name a ─┐
          ├──► ┌──────────────────────┐
  name b ─┘    │ list object          │
               │ [ ●, ●, ● ]          │ ← the same object
               └──┼──┼──┼─────────────┘
                  ▼  ▼  ▼
                  1  2  3   (each also a heap int object)
```

`b.append(4)` reaches that list object through b and **modifies it in place**. a points to the same object, so of course it looks changed from a as well. This isn't a bug in copy semantics; it's **aliasing**—two names are two aliases of the same object.

Analogy with C++: `b = a` is more like `int* b = a;` (copying a pointer), and absolutely not `std::vector<int> b = a;` (copy construction, a deep copy). Python assignment is **always "copy the reference"**; it never calls any copy constructor.

### Proving Aliasing with is / == / id

- `==` compares whether the **values** are equal (calls `__eq__`).
- `is` compares whether they're **the same object** (compares `id`), equivalent to C's `a == b` comparing two pointers.

```python
a = [1, 2, 3]
b = a
c = [1, 2, 3]          # build another list with identical contents

print(a == b)          # True, values equal
print(a is b)          # True, same object (aliases)
print(id(a) == id(b))  # True

print(a == c)          # True, values equal
print(a is c)          # False! different objects
print(id(a), id(c))    # two different addresses
```

`a is b` being true while `a is c` is false proves in black and white that `b = a` created an alias whereas `c = [1,2,3]` is a brand-new object. To judge "is it the same thing" use `is`; for "are the values equal" use `==`. (The `is None` idiom comes precisely from here—`None` is a singleton, so comparing identity is the most fitting.)

## 3. Reference Counting: Who Is Pointing at Me

Recall [Chapter 01](01-运行模型.md): every `PyObject` header has `ob_refcnt`. Each additional name/container pointing at it adds 1, each removal subtracts 1, and reaching zero reclaims it immediately (CPython's primary memory-management mechanism; a separate cyclic garbage collector handles reference cycles, see [Chapter 09](09-错误处理与资源管理.md)). This is Python's "automatic free"—you don't call `free`; reference counting does it for you.

```python
import sys

x = object()
print(sys.getrefcount(x))   # e.g. 2

y = x                        # one more reference
print(sys.getrefcount(x))   # 3

del y                        # remove one reference (del unbinds the name, not the object!)
print(sys.getrefcount(x))   # 2, back down
```

Two key points:
1. `sys.getrefcount(x)` is **always one higher than your intuition**, because passing `x` as an argument to the function itself temporarily adds a reference.
2. `del y` **deletes the name binding**, not the object. The object is truly freed only when its refcnt hits zero. This is entirely different from C++'s `delete` semantics—`del` is more like "letting go of a hand"; the object is gone only when the last hand lets go.

```
del semantics                reclaimed only when refcnt hits zero
name y ──╳ (unbound)         object ← only x still points, refcnt=1, still alive
name x ─────► object         when x is also del'd, refcnt=0 → freed immediately
```

## 4. Mutable vs. Immutable: Governs All Aliasing Behavior

Now back to the "modifying b moves a" problem. Why does the following **not** have the same "issue"?

```python
a = 1
b = a
b = b + 1
print(a, b)     # 1 2   —— a didn't change!
```

The difference: list is **mutable**, int is **immutable**.

- **Mutable objects** (list, dict, set, and most custom class instances): their contents can be modified in place. `b.append(4)`, `b[0] = 9` change the object itself, and all aliases see it.
- **Immutable objects** (int, float, str, tuple, bool, frozenset, None): their contents can't change. Any "modification" operation actually **creates a new object** and rebinds the name to it.

Under the hood of `b = b + 1`: `b + 1` computes a **new** int object `2`, and then `b =` rebinds the name b to this new object. The original `1` object is untouched, and a still points to it.

```
a = 1; b = a                b = b + 1

 a ─┐                        a ────► [ int 1 ]   ← a still points to the old object
    ├──► [ int 1 ]           b ────► [ int 2 ]   ← b rebinds to a new object
 b ─┘                        (newly allocated object)
```

Compare with in-place modification of a mutable object:

```
a = [1,2,3]; b = a          b.append(4)

 a ─┐                        a ─┐
    ├──► [1,2,3]                ├──► [1,2,3,4]   ← the same object is modified
 b ─┘                        b ─┘                  both names see the change
```

**Core rules**:
- Modifying a mutable object **in place** (`append`/`b[0]=`/`.sort()`) affects all aliases.
- **Rebinding** any object (`b = ...`) only changes the name b's target, affecting no one else.
- Immutable objects have no "modify in place" path, so their aliases are always safe—which is why "using immutable types is more worry-free."

One pitfall: `+=` behaves differently for mutable and immutable. `b += [4]` on a list is an in-place extension (equivalent to `extend`, affecting aliases); `b += 1` on an int is a rebind (not affecting aliases). Because mutable objects implement an in-place `__iadd__`, whereas immutable ones fall back to `b = b + 1`.

Seeing is believing—with the very same `+=`, whether an alias follows depends on whether the object is mutable:

```python
a = [1, 2]
b = a
a += [3]        # list's += extends in place (__iadd__); still the same object
print(b)        # [1, 2, 3]   alias b follows the change

a = a + [4]     # + builds a new list, then rebinds a to it
print(b)        # [1, 2, 3]   b no longer follows—a now points elsewhere
```

## 5. Small-Integer Cache and String Interning

These two are pure **CPython implementation details**, but you'll surely bump into them while playing with `is`, and without explanation you'll be baffled.

### Small-Integer Cache [-5, 256]

At startup CPython pre-creates all integer objects from `-5` to `256` and caches them for reuse. So identical values in this range are **the same object**:

```python
a = 100
b = 100
print(a is b)      # True! both point to that unique cached 100 object

x = 257
y = 257
print(x is y)      # True! —— but this isn't due to the small-int cache

# Within the same compilation unit, the literal 257 is stored only once in the bytecode's
# co_consts constant table, and x, y reuse the same constant object, so even though 257 is
# outside the [-5, 256] cache range, x is y is still True.
# To truly demonstrate "beyond the cache, each is newly created," you must construct the
# objects at runtime, bypassing co_consts reuse:
m = int("257")
n = int("257")
print(m is n)      # False! the two int() calls each create a new 257 object on the heap
print(m == n)      # True, the values are of course equal
```

Why cache? Small integers are used extremely frequently; pre-building them avoids repeated allocation/deallocation, saving memory and time. The range `[-5, 256]` is a hardcoded implementation choice; other implementations or versions may differ.

> Lesson: **always use `==` to test numeric equality, never `is`.** `is` happens to "work" for small integers, but the moment you leave the range it deceives you.

### String Interning

CPython **interns** certain strings (especially identifier-looking literals and compile-time constants) into a global table, so identical contents share one object—this lets comparing identifiers or looking up dict keys first compare pointers (`is`) for a fast equality check.

```python
a = "hello"
b = "hello"
print(a is b)      # usually True, interned

s = "hello world!"
t = "hello world!"
print(s is t)      # True —— but again this is "two identical literals in the same compilation
                   # unit sharing one co_consts constant," which does NOT mean it was interned
                   # at runtime; strings with spaces/punctuation are not auto-interned by default

# To see "identical content but not the same object," construct at runtime, bypassing literal
# constant reuse:
u = "hello world!".upper().lower()
v = "hello world!".upper().lower()
print(u is v)      # usually False

import sys
print(sys.intern(u) is sys.intern(v))   # True —— after manual intern, they share one object
```

Interning is an optimization, and its triggering conditions vary by version and context; **never rely on it**. Likewise: test string equality with `==`.

## 6. Namespaces Are Dictionaries, but Local Variables Are Fast Locals

You may have heard "Python's namespaces are just dicts." Basically correct: a module's global namespace and an object's attributes are, under the hood, dicts (keys are name strings, values are object references).

```python
x = 10
print(globals()["x"])    # 10, the global namespace is just a dict
```

But **local variables inside functions are an important exception, and this is precisely a low-level optimization point**. When CPython compiles a function, it can statically count how many local variables there are and which is which, so it puts them in a **fixed-length array** ("fast locals") accessed by **index** rather than looking up a string in a dict every time. Recall the `LOAD_FAST` seen with `dis` in [Chapter 01](01-运行模型.md):

```python
import dis

g = 1
def f(a):
    b = a + g       # a, b are locals (fast), g is global
    return b

dis.dis(f)
```

Measured output on 3.14 here:

```
  4           RESUME                   0

  5           LOAD_FAST_BORROW         0 (a)
              LOAD_GLOBAL              0 (g)
              BINARY_OP                0 (+)
              STORE_FAST               1 (b)

  6           LOAD_FAST_BORROW         1 (b)
              RETURN_VALUE
```

(`LOAD_FAST_BORROW` is 3.14's optimized variant for reading a local—it "borrows" the reference, skipping refcnt bumps, but it's still an indexed local fetch; writes are still `STORE_FAST`, and globals are still `LOAD_GLOBAL`.)

Key instruction contrast:
- `LOAD_FAST 0 (a)` / `STORE_FAST 1 (b)`: local variables use an **array index**, O(1), no hashing, no string comparison. Fast.
- `LOAD_GLOBAL 1 (g)`: global variables use a **dict lookup** (this module's globals first, then builtins), slower.

```
Stack frame of function f
┌───────────────────────────┐
│ fast locals array:         │
│   [0] a ──► object         │  ← LOAD_FAST 0, indexed access, fast
│   [1] b ──► object         │
├───────────────────────────┤
│ evaluation stack           │  ← used by bytecode ops (see Chapter 01, stack machine)
└───────────────────────────┘
        global g lives in the module's globals dict; LOAD_GLOBAL looks up the dict
```

This explains a common performance trick: in a hot loop, "cache" a frequently accessed global/attribute in a local variable, i.e. replace `LOAD_GLOBAL` (dict lookup) with `LOAD_FAST` (array lookup). It also explains why writing to a global inside a function requires a `global` declaration—the compiler decides whether to treat a name as local or global based on whether the function assigns to it. The full story of scopes and stack frames is in [Chapter 06](06-函数与调用栈.md).

> Note the distinction: local variables using a fast-locals array is **CPython's implementation**; "functions have their own scope" is a language semantic.

## 7. Shallow Copy vs. Deep Copy

Since `b = a` only copies the pointer, what if you want a real copy? Use the `copy` module. But distinguish **shallow** from **deep**—this is the same question as in C++: "is the copy constructor shallow or deep, do I need to write my own deep copy."

```python
import copy

original = [[1, 2], [3, 4]]

alias    = original                 # alias, same object
shallow  = copy.copy(original)      # shallow copy: new outer, inner elements still shared
deep     = copy.deepcopy(original)  # deep copy: recursively copies every layer

original[0].append(99)              # modify the first inner sublist

print(alias)    # [[1, 2, 99], [3, 4]]   changes along (it's the same object)
print(shallow)  # [[1, 2, 99], [3, 4]]   changes along! the inner layer is shared
print(deep)     # [[1, 2], [3, 4]]       independent, unchanged
```

Memory diagram:

```
original ─► [ ●, ● ]            outer list
              │  │
              ▼  ▼
           [1,2] [3,4]          two inner lists

shallow = copy.copy(original):
shallow ─► [ ●, ● ]  ← a new outer object
             │  │
             └──┴──► point to the same [1,2] and [3,4]  (inner not copied!)

deep = copy.deepcopy(original):
deep ─► [ ●, ● ]  ← new outer
          │  │
          ▼  ▼
       [1,2]'[3,4]'  ← even the inner layer is new objects, fully independent
```

- **Shallow copy**: only the outermost container is newly created; what's inside is still **references to the original objects**. Modifying an inner mutable element affects the shallow copy too.
- **Deep copy**: recursively copies every layer into new objects, fully independent (and correctly handles reference cycles).

Other common ways to make a shallow copy: `list(x)`, `x[:]`, `dict(x)`, `x.copy()`. They're all shallow, just like `copy.copy`. For full independence you must use `deepcopy`. Deep copy has a cost (recursion + allocation); don't use it mindlessly.

## 8. The Mutable Default Argument Trap (Preview)

With this chapter's model, you can now understand what one of Python's most famous pitfalls looks like:

```python
def add_item(item, bucket=[]):    # dangerous! the default [] is created only once
    bucket.append(item)
    return bucket

print(add_item(1))   # [1]
print(add_item(2))   # [1, 2]  ← you might expect [2], but it remembered last time!
```

The reason is exactly what this chapter taught: the default value `[]` is **a single mutable object created once when `def` executes**, held by the function object forever; each call that omits `bucket` reuses **the same list**, and the in-place append naturally accumulates. This is the inevitable result of "mutable object + aliasing + created only once" all coming together.

The correct approach and full mechanism (where default values are stored, when `def` is evaluated) are covered in [Chapter 06](06-函数与调用栈.md)—but you already understand why this happens, which is precisely the power of this chapter's model.

## 9. Summary (Burn These Into Muscle Memory)

1. **A variable is a binding of a name to a heap object**, not a memory box; analogous to a read-only pointer that can't do arithmetic.
2. **`a = b` copies only the reference (pointer value)**, never the object or data; the result is an alias. `is`/`id()` can prove they're the same object; `==` only compares values.
3. **Reference counting** (`ob_refcnt`) manages reclamation, observable via `sys.getrefcount` (result is one higher); `del` unbinds a name rather than deleting the object.
4. **Mutable vs. immutable governs aliasing behavior**: modifying a mutable object in place affects all aliases; rebinding only changes the current name; an immutable object's "modification" always creates a new object.
5. **Small-integer `[-5,256]` caching and string interning** are CPython optimizations—always test equality with `==`, not `is`.
6. Namespaces are dicts, but **function local variables use a fast-locals array** (`LOAD_FAST`), an important performance point.
7. `copy.copy` is shallow, `copy.deepcopy` is deep; `b=a`/slicing/`list()` are all shallow only.
8. The mutable default argument trap is a direct corollary of this chapter's model; details in [Chapter 06](06-函数与调用栈.md).
