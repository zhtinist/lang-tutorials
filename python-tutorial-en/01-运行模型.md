# The Execution Model: How CPython Turns .py Into Behavior

As a C/C++ engineer, the mental model in your head is roughly this: `gcc` compiles `.c` into assembly, then assembles it into an object file, links it into an ELF/Mach-O executable, the OS `exec`s it into memory, and the CPU runs the machine code directly. There's no "runtime" translating anything in between; `int x = 3;` is just writing 4 bytes to some offset on the stack.

Python is nothing like this. Understanding its execution model is the foundation for understanding all the "weird" behavior that follows (aliasing, the GIL, the overhead of dynamic typing).

## 1. Running It

Python needs no separate compilation step:

```python
print("Hello, Python!")

x = 3          # assignment, no need to declare a type first like in C
x = x + 1
print(x)       # 4
```

Save it as `hello.py` and run `python3 hello.py` from the terminal—there's no "compile an executable first, then run it" two-step; the single program `python3` reads the file, processes it, and executes it all in one go. You can also just type `python3` to enter the interactive interpreter (the REPL) and see results line by line as you type.

Behind this "no separate compilation step" phenomenon is a real design choice: since its birth in 1991, Python has been positioned as a **dynamically typed scripting language**—a variable's type is only settled at runtime, and an expression like `x + 1` can only know whether `x` is an integer or a string, and which addition implementation to invoke, at the moment it actually executes. A statically compiled language can nail down type checks and method addresses at compile time and emit machine code that operates on memory directly; a dynamically typed language can't do this—type dispatch is inherently deferred to runtime, which means there has to be a program that stays resident at runtime, ready at any moment to interpret and dispatch. That program is the interpreter, covered next.

## 2. From Source to Behavior: CPython Is "an Interpreter Written in C"

First, two concepts worth separating:

- **Python** is a language—a language specification (semantics).
- **CPython** is the **reference implementation** of that language, written in C. What `brew install python` installs and what `python3` runs is, 99% of the time, CPython.

There are other implementations too: PyPy (with a JIT), Jython (runs on the JVM), GraalPy, and so on. Whenever this tutorial touches implementation details, it uses CPython as the reference and explicitly marks it "**in CPython**." Separating language semantics from implementation details matters—a lot of what gets called "a Python feature" is really just "a CPython implementation choice," and doesn't hold under a different implementation.

| Stage | C/C++ | CPython |
|------|-------|---------|
| Source | `.c` / `.cpp` | `.py` |
| Front end | Preprocessing + compilation | Lexing/parsing → AST |
| Intermediate artifact | `.o` object file (machine code) | `.pyc` bytecode |
| Linking | Linker produces an executable | No separate linking step |
| Execution | CPU runs machine code directly | Interpreter (virtual machine) eval loop interprets bytecode instruction by instruction |
| Type checking | Compile time | Runtime |

Once C finishes compiling, the compiler is out of the picture entirely—the program runs on its own, bare metal. The bytecode CPython compiles to **is not machine code**; it needs a resident interpreter (a giant loop written in C) to execute it instruction by instruction—that interpreter is the `python` executable itself. **Why does Python need this extra layer instead of compiling straight to machine code?** Dynamic typing means "how should this addition be done" can only be settled at runtime; compiling source directly into machine code that bakes in fixed type assumptions simply doesn't work. Bytecode is a compromise: statically resolvable work like parsing gets done up front and compiled into a compact intermediate representation, while type dispatch—which genuinely depends on runtime information—is left to the eval loop. That balances two goals at once: not having to re-parse the source every time, and leaving the final say on types to runtime.

## 3. Four Stages: Source → AST → Bytecode → Evaluation

```
foo.py (source text)
   │  lexical analysis (tokenize)
   ▼
Token stream
   │  parsing (parse)
   ▼
AST (abstract syntax tree)      <- viewable with the ast module
   │  compilation (compile)
   ▼
Code Object (contains bytecode) <- this is what's stored in .pyc
   │  interpretation (the eval loop in ceval.c)
   ▼
Program behavior / side effects
```

### 3.1 AST: the Syntax Tree

Python exposes this intermediate compilation state to user code, something that's normally only visible inside the compiler itself in the C toolchain:

```python
import ast

src = "x = a + b * 2"
tree = ast.parse(src)
print(ast.dump(tree, indent=2))
```

You'll see an `Assign` node whose target is `Name(id='x')` and whose value is `BinOp(left=Name('a'), op=Add, right=BinOp(...))`—exactly the syntax tree from a compiler-theory course. Internally, CPython compiles this same tree into bytecode.

### 3.2 Bytecode and `.pyc`

The compiled product is a **code object**, whose `co_code` is a string of bytecode instructions. When you `import` a module, CPython caches the compiled result to `__pycache__/module_name.cpython-<version>.pyc` (the version number follows the interpreter, e.g. CPython 3.14 produces `cpython-314.pyc`). On the next import, if the source hasn't changed (checked by mtime/hash), it loads the `.pyc` directly, skipping recompilation.

A `.pyc` **is not** machine code and can't run without the interpreter—it only saves the time of re-parsing and re-compiling, not the cost of interpreted execution. This is superficially similar to C's `.o` (both are cacheable intermediate artifacts), but a `.o` holds real machine code while a `.pyc` holds bytecode meant to be fed to a virtual machine—the reuse payoff is on a completely different scale. Running the main script directly with `python foo.py` does **not** produce a `.pyc` (only imported modules get cached, because the main script only ever executes once per run, so caching it buys nothing).

### 3.3 Viewing Bytecode with `dis`

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

Output (Python 3.12; instructions vary slightly across versions):

```
  2           RESUME                   0

  3           LOAD_FAST                0 (a)
              LOAD_FAST                1 (b)
              BINARY_OP                0 (+)
              RETURN_VALUE
```

`LOAD_FAST 0 (a)` pushes local variable `a` onto the **evaluation stack** (`FAST` means it's an array-index lookup rather than a dictionary lookup; see [Variables and the Memory Model](03-变量与内存模型.md)); `LOAD_FAST 1 (b)` does the same for `b`; `BINARY_OP` pops the top two objects off the stack, adds them, and pushes the result; `RETURN_VALUE` pops the top of the stack as the return value.

CPython is a **stack machine**, not a register machine—every operation revolves around pushing to and popping from an evaluation stack, the same design the JVM uses, unlike x86's register-plus-memory machine model. **Why a stack machine instead of a register machine?** A stack-based bytecode compiler is simpler (expression evaluation maps naturally onto push/pop order, with no register-allocation step needed), which fits with CPython's early priorities as a teaching/glue-language reference implementation: simplicity of implementation and ease of verifying correctness came first. The cost is that the same computation usually needs more instructions (a register-based VM like Lua's can reference operands directly by the register they live in, skipping explicit pushes and pops).

### 3.4 The Eval Loop

The core of CPython's bytecode execution is a giant loop in `Python/ceval.c`, in pseudocode:

```c
for (;;) {
    opcode = NEXT_INSTRUCTION();
    switch (opcode) {
        case LOAD_FAST:  ... push(...); break;
        case BINARY_OP:  ... ; break;
        case RETURN_VALUE: ...; return;
        ...
    }
}
```

Each turn of the loop interprets one instruction—conceptually this corresponds to the CPU's fetch-decode-execute cycle, except this layer is **simulated in software**. A single Python `a + b` goes through: fetch the bytecode → dispatch through the switch → look up the type of `a` → find `__add__` → call it → box the result. C's `a + b` has already settled which kind of addition it is at compile time, and at runtime it might just be a single `add` instruction. This extra dispatch overhead is the direct cost of "one more layer of interpretation," and it's also the inevitable consequence of the design choice from section 2—dynamic typing must defer "how to do this" to runtime. It isn't an implementation shortfall; the semantics demand it.

## 4. Why "Everything Is an Object, and It All Lives on the Heap"

In C, `int x = 3;` makes `x` a 4-byte slot on the stack with the value inlined directly inside it; taking its address `&x` gives you a stack address.

In Python, `x = 3` is an entirely different matter. In CPython, **every value is a `PyObject`, allocated on the heap**:

```c
typedef struct _object {
    Py_ssize_t ob_refcnt;   // reference count
    PyTypeObject *ob_type;  // pointer to the type object
} PyObject;
```

`ob_refcnt` records how many names/containers currently reference this object; it gets freed once this hits zero (see [Variables and the Memory Model](03-变量与内存模型.md)). `ob_type` points to the "type object," answering "who am I"—it records how to add, how to print, and so on. You can think of it as analogous to the vtable pointer in a C++ object, except that in Python **even integers and `None`—the "primitive values"—carry one**, and dispatch happens entirely through runtime lookup.

**Why does every value have to carry this object header, instead of inlining small integers directly like C does?** The root cause is, again, dynamic typing plus uniform runtime introspection—operations like `type(x)`, `isinstance(x, T)`, or calling `x.__add__` on an arbitrary value all require that "any value can answer what type it is and what operations it supports." C's `int` gets to skip carrying this information only because type resolution has already been settled once and for all by `gcc` at compile time and baked into the machine code—nothing needs to "ask" at runtime. Python defers type resolution to runtime, which means every value must carry enough self-describing information with it; the `PyObject` header is the minimal implementation of exactly that information.

A concrete `int` (`PyLongObject`) has the actual numeric digits following the `PyObject` header:

```
name x on the stack      PyLongObject on the heap
┌─────────┐            ┌──────────────────┐
│ pointer─┼──────────► │ ob_refcnt = 1    │
└─────────┘            │ ob_type   ──► int │
   (one machine word)   │ value: 3         │
                       └──────────────────┘
```

Compared to C's `int x = 3`:

```
C: on the stack   Python: name on stack + object on heap
┌───────┐        ┌───────┐        ┌───────────┐
│  3    │        │ ptr  ─┼──────► │ ...3(obj) │
└───────┘        └───────┘        └───────────┘
4 bytes, value    one pointer      one heap
is the data                        allocation
```

A Python "variable" is always a **reference to a heap object** (essentially a pointer)—it never inlines a value. There's no "value type on the stack"; even `1`, `True`, and `None` are heap objects (`None` is a globally unique singleton). An integer object on 64-bit CPython takes roughly 28 bytes rather than 4—that's the object header's doing—which explains why numerically intensive Python code pays a cost in both memory and speed, and it's the very reason NumPy exists (packing large numbers of values into one contiguous C array, sidestepping the per-object header overhead; see [Common Third-Party Libraries](12-常用第三方库.md)).

```python
import sys
print(sys.getsizeof(3))         # 28    a small int already takes 28 bytes
print(sys.getsizeof(2 ** 100))  # 40    arbitrary precision: bigger value, more digit words, bigger object
print(sys.getsizeof(2 ** 1000)) # 160
```

> Semantics vs. implementation: the language only guarantees the behavioral semantics of "everything is an object"; "it all lives on the heap" and "the object header is these two fields" are CPython's implementation. PyPy uses a different object representation, paired with escape analysis, so many short-lived objects never get a real heap allocation at all—which just underscores that "everything is an object" is the semantic contract, and "it all lives on the heap" is merely one way of satisfying that contract.

Small-integer caching (`[-5, 256]`) and string interning are another category of CPython implementation detail, dissected separately in [Variables and the Memory Model](03-变量与内存模型.md).

## 5. CPython vs. PyPy: Interpretation vs. JIT

The root cause of the slowness is "interpreting one instruction at a time in software," so the natural optimization is to just-in-time compile hot bytecode into machine code—which is exactly what **PyPy** does.

| | CPython | PyPy |
|--|---------|------|
| Execution | Pure interpretive eval loop | Tracing JIT: hot code compiled to machine code |
| First run / cold code | No warmup | Warmup overhead |
| Long loops / hot spots | Slow | Can be several to tens of times faster |
| C extension compatibility | Native (CPython API) | Weaker (cffi works better) |
| Object representation | Every object honestly heap-allocated | Can be optimized away after escape analysis |

The idea matches the JVM's HotSpot: interpret first, gather hot-spot statistics, then compile the hot paths into machine code with specialization (e.g. assuming that `a` and `b` in a loop are always `int`, skipping the type lookup on every iteration, and falling back to the generic path only if that assumption fails). Since 3.11, CPython has introduced a "specializing adaptive interpreter" (PEP 659) that does lightweight specialization at the bytecode level—same underlying idea, much smaller in scope: it doesn't generate machine code, it just caches common type combinations to speed up dispatch. Only 3.13+ has an experimental full JIT. **Why has CPython gone years without a full JIT, doing this lightweight specialization instead?** A JIT compiler is itself a subsystem with very high complexity and maintenance cost, and CPython carries decades of historical baggage in the form of a huge C-extension ecosystem (NumPy, Pandas, and others all operate directly on CPython's C API and its object memory layout). Any scheme that dynamically generates machine code and replaces the object representation risks breaking compatibility with these extensions—and this is the core reason PyPy, despite being faster, has never displaced CPython as the mainstream implementation: ecosystem compatibility outweighs raw execution speed.

## 6. The GIL: an Interpreter-Level Lock, Not Language Semantics

Is this eval loop thread-safe? Can multiple cores run pure Python code in parallel?

**In CPython there is a Global Interpreter Lock (GIL)**: within a single process, only one thread executes Python bytecode at any given moment. **Why have a GIL?** CPython manages memory with reference counting, and incrementing/decrementing `ob_refcnt` happens on almost every object pass, assignment, and function call. If multiple threads truly ran in parallel, reading and updating that counter would need a lock on every single object—not only slowing down single-threaded performance, but turning what should be a lightweight object operation into something heavyweight. When the GIL was introduced in 1992, single-core CPUs were the mainstream hardware, so trading away multi-core parallelism for single-thread simplicity and efficiency—and for letting every C extension skip its own thread-safety handling—was a reasonable call at the time: any extension written against the CPython C API could assume "only my code is touching Python objects right now" and never needed extra locking.

The cost is that pure Python computation can't saturate multiple cores through multithreading, which has increasingly become a real performance bottleneck now that multi-core CPUs are the default. That's what drove the experimental no-GIL build starting in Python 3.13 (PEP 703, i.e. free-threading)—essentially reworking reference counting to be thread-safe (e.g. via atomic operations or per-thread local counts), trading finer-grained synchronization for true parallelism, at the cost of extra overhead in single-threaded scenarios and C-extension compatibility problems. This trade-off is still being worked out. The full mechanism and the ways around it (multiprocessing, C extensions releasing the GIL, `asyncio`) are covered in [The Concurrency Model](10-并发模型.md).

## Summary

- CPython compiles `.py` into bytecode (cached as `.pyc`), then a C-written stack-machine eval loop interprets it instruction by instruction; this whole extra layer of software translation is the fundamental reason Python is slower than C, and it follows directly from the language design choice that dynamic typing must defer dispatch to runtime.
- Use `ast.parse` to see the syntax tree and `dis.dis` to see the bytecode—you can verify what's happening underneath at any time.
- Everything is an object, and in CPython everything is heap-allocated; every object carries `ob_refcnt` (reference count) and `ob_type` (like a vtable pointer)—this is the direct implementation of the requirement that "any value can answer what type it is at runtime." A variable is a reference to a heap object, never an inlined value, fundamentally unlike C's stack value types.
- Distinguish Python (semantics) from CPython (implementation): PyPy trades some C-extension compatibility for JIT speed; the GIL is a historical CPython trade-off to simplify reference counting and C-extension thread safety, at the cost of multi-core parallelism, and free-threading is now trying to trade back some of that parallelism through finer-grained synchronization.
