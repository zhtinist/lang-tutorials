# The Execution Model: How CPython Turns .py Into Behavior

As a C/C++ engineer, the mental model in your head is roughly this: `gcc` compiles `.c` into assembly, then assembles it into object files, links those into an ELF/Mach-O executable, and the operating system `exec`s it into memory where the CPU runs the machine code directly. There is no "runtime" translating in the middle; `int x = 3;` just writes 4 bytes to some offset on the stack.

Python is nothing like this. Understanding its execution model is the foundation for understanding all the "weird" behavior that follows (aliasing, the GIL, the overhead of dynamic typing). This chapter is about exactly one thing: **what sits between your source code and its behavior.**

## 1. A One-Line Comparison

| Stage | C/C++ | CPython |
|------|-------|---------|
| Source | `.c` / `.cpp` | `.py` |
| Front end | Preprocess + compile | Lexing/parsing → AST |
| Intermediate artifact | `.o` object file (machine code) | `.pyc` bytecode |
| Linking | Linker produces an executable | No separate linking step |
| Execution | CPU runs machine code directly | Interpreter (virtual machine) eval loop interprets bytecode one instruction at a time |
| Type checking | Compile time | Runtime |

The key difference: once C finishes compiling, the compiler is out of the picture and the program runs on its own on bare metal. The bytecode CPython emits **is not machine code**; it needs a resident "interpreter" (a giant loop written in C) to execute it instruction by instruction. That interpreter is the `python` executable itself.

## 2. CPython Is "an Interpreter Written in C"

First, distinguish two concepts:

- **Python** is a language, a language specification (semantics).
- **CPython** is the **reference implementation** of that language, written in C. What you install with `brew install python` and run as `python3` is, 99% of the time, CPython.

There are other implementations too: PyPy (with a JIT), Jython (runs on the JVM), GraalPy, and so on. Whenever this tutorial touches implementation details, it uses CPython as the reference and explicitly notes "**in CPython**." For someone like you who wants to dig into the low level, distinguishing language semantics from implementation details is extremely important—many "features of Python" are really just "implementation choices of CPython."

## 3. Four Stages: Source → AST → Bytecode → Evaluation

When you run `python foo.py`, CPython internally goes roughly through:

```
foo.py (source text)
   │  lexical analysis (tokenize)
   ▼
Token stream
   │  parsing (parse)
   ▼
AST (abstract syntax tree)   <- viewable with the ast module
   │  compilation (compile)
   ▼
Code Object (contains bytecode)   <- this is what's stored in .pyc
   │  interpretation (the eval loop in ceval.c)
   ▼
Program behavior / side effects
```

### 3.1 AST: the Syntax Tree

Python exposes its compilation intermediate state, which is a luxury in the C world.

```python
import ast

src = "x = a + b * 2"
tree = ast.parse(src)          # parse into an AST
print(ast.dump(tree, indent=2))
```

You'll see an `Assign` node whose target is `Name(id='x')` and whose value is `BinOp(left=Name('a'), op=Add, right=BinOp(...))`. This is exactly the syntax tree from your compiler-theory course. Internally, CPython compiles this tree into bytecode.

### 3.2 Bytecode and `.pyc`

The compilation product is a **code object**, in which `co_code` is a string of bytecode instructions. When you `import` a module, CPython caches the compiled result to `__pycache__/module_name.cpython-<version>.pyc` (the version number follows the interpreter, e.g. CPython 3.14 produces `cpython-314.pyc`). On the next import, if the source hasn't changed (compared by mtime/hash), it loads the `.pyc` directly and skips recompilation.

Note:
- A `.pyc` **is not** machine code and can't run without the interpreter. It's just "compiled bytecode"; it saves the time of re-parsing and re-compiling, not the cost of interpretive execution.
- Running the main script directly with `python foo.py` **does not** produce a `.pyc` (only imported modules get cached).
- This is analogous to C's `.o`: both are intermediate artifacts that can be cached and reused, but `.o` holds real machine code while `.pyc` holds bytecode fed to a virtual machine.

### 3.3 Viewing Bytecode with `dis`

The `dis` module disassembles bytecode into a human-readable form; it's your microscope for peering into the low level.

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

Output (Python 3.12; instructions vary slightly across versions):

```
  2           RESUME                   0

  3           LOAD_FAST                0 (a)
              LOAD_FAST                1 (b)
              BINARY_OP                0 (+)
              RETURN_VALUE
```

Instruction by instruction:
- `LOAD_FAST 0 (a)`: push local variable `a` onto the **evaluation stack**. (`FAST` means it uses an array index rather than a dictionary lookup; see [Chapter 03](03-变量与内存模型.md).)
- `LOAD_FAST 1 (b)`: push `b`.
- `BINARY_OP 0 (+)`: pop the top two objects, perform addition, push the result.
- `RETURN_VALUE`: pop the top of the stack as the return value.

The core insight here: **CPython is a stack machine**, not a register machine. Every operation revolves around pushing to and popping from an "evaluation stack" (value stack). This is similar to the JVM and different from the register-plus-memory machine model of x86. Every `LOAD_*` you see is pushing data onto this stack.

### 3.4 The Eval Loop

The heart of CPython's bytecode execution is a giant loop in `Python/ceval.c`, in pseudocode:

```c
for (;;) {
    opcode = NEXT_INSTRUCTION();
    switch (opcode) {
        case LOAD_FAST:  ... push(...); break;
        case BINARY_OP:  ... ; break;
        case RETURN_VALUE: ...; return;
        ...
    }
}
```

Each turn of the loop interprets one instruction. Compare it to the CPU's fetch-decode-execute cycle: the concept is the same, but this layer is **simulated in software**, so it's slow. A single Python `a + b` goes through: fetch bytecode → switch dispatch → look up the type of `a` → find `__add__` → call it → box the result. C's `a + b` might be a single `add` instruction. This is the fundamental source of Python being "slow"—the language isn't dumb; there's a whole extra layer of interpretation.

## 4. Why "Everything Is an Object, and It All Lives on the Heap"

This is the most fundamental divergence from C, and you must fully internalize it.

In C:
```c
int x = 3;   // x is a 4-byte slot on the stack containing the two's-complement of the integer 3
```
`x` is a chunk of memory with the value inlined directly. Taking its address `&x` yields a stack address.

In Python, `x = 3` is an entirely different matter. In CPython, **every value is a `PyObject`, allocated on the heap**. The simplified structure (see `Include/object.h`):

```c
typedef struct _object {
    Py_ssize_t ob_refcnt;   // reference count
    PyTypeObject *ob_type;  // pointer to the type object
} PyObject;
```

- `ob_refcnt`: how many names/containers currently reference me. When it hits zero, I'm freed (see [Chapter 03](03-变量与内存模型.md)).
- `ob_type`: points to the "type object," answering "who am I." An integer points to the `int` type object, which records how to add, how to print, and so on.

You can think of this `ob_type` pointer exactly like the **vtable pointer** in a C++ object: the object carries information about "how I should be operated on," and method calls are dispatched through it at runtime. The difference is that in Python **even integers and `None` carry it**, and dispatch is fully dynamic.

A concrete `int` (`PyLongObject`) carries the actual numeric digits after the `PyObject` header. So in memory:

```
name x on the stack     PyLongObject on the heap
┌─────────┐            ┌──────────────────┐
│ pointer─┼──────────► │ ob_refcnt = 1    │
└─────────┘            │ ob_type   ──► int │
   (one machine word)  │ value: 3         │
                       └──────────────────┘
```

Compared to C's `int x = 3`:

```
C: on the stack   Python: name on stack + object on heap
┌───────┐        ┌───────┐        ┌───────────┐
│  3    │        │ ptr  ─┼──────► │ ...3(obj) │
└───────┘        └───────┘        └───────────┘
4 bytes, value   one pointer      one heap allocation
is the data
```

Therefore:
- A Python "variable" is always a **reference to a heap object** (essentially a pointer); it never "inlines the value." `x = 3` means "make the name x point to the heap object representing 3."
- There are no "value types on the stack." Even `1`, `True`, and `None` are heap objects (`None` is a globally unique singleton).
- On 64-bit CPython an integer object takes about 28 bytes, not 4—because it carries the object header. This explains why Python pays in both memory and speed for numeric-intensive computation, and it's the reason NumPy exists (see [Chapter 12](12-常用第三方库.md)).

Use `sys.getsizeof` to see this "object-header tax" for yourself, and to confirm that arbitrary-precision integers are variable-length objects:

```python
import sys

# The object header (refcnt + type pointer) makes even the smallest int far larger than C's 4 bytes
print(sys.getsizeof(3))         # 28    a small int already takes 28 bytes
print(sys.getsizeof(2 ** 100))  # 40    arbitrary precision: bigger value, more digit words, bigger object
print(sys.getsizeof(2 ** 1000)) # 160
```

> Semantics vs. implementation: the language only guarantees the behavioral semantics of "everything is an object." "It all lives on the heap" and "the object header is these two fields" are **CPython's implementation**. PyPy uses a different object representation and escape analysis, so many short-lived objects are never truly heap-allocated.

Small integers actually have a caching optimization (`[-5, 256]`), and strings are interned; these too are CPython implementation details, dissected specifically in [Chapter 03](03-变量与内存模型.md), which is the main battleground for this book's memory model.

## 5. CPython vs. PyPy: Interpretation vs. JIT

Since the root cause of slowness is "interpreting one instruction at a time in software," the natural thought is: can we just-in-time compile hot bytecode into machine code? That's exactly what **PyPy** does.

| | CPython | PyPy |
|--|---------|------|
| Execution | Pure interpretive eval loop | Tracing JIT: hot code compiled to machine code |
| First run / cold code | No warmup | Warmup overhead |
| Long loops / hot spots | Slow | Can be several to tens of times faster |
| C extension compatibility | Native (CPython API) | Weaker (cffi is better) |
| Object representation | Every object honestly heap-allocated | Can be optimized away after escape analysis |

The JIT idea is familiar to you: just like the JVM's HotSpot, it first interprets, gathers hot-spot statistics, then compiles the hot paths into machine code with specialization (e.g. assuming `a` and `b` in a loop are always ints, so it skips the per-iteration type lookup, falling back once the assumption fails). Since 3.11 CPython also introduced a "specializing adaptive interpreter" (PEP 659) that does lightweight specialization at the bytecode level, but that's not yet a full JIT (3.13+ has an experimental JIT).

The practical implication for you: use CPython first (its ecosystem and C-extension support are the most complete); for pure-Python CPU-intensive long-running tasks that don't depend on heavy C extensions, PyPy may hand you a few times the performance for free.

## 6. The GIL: Laying the Groundwork

You're bound to ask: is this eval loop thread-safe? Can it run in parallel across cores?

The short answer: **in CPython there is a Global Interpreter Lock (GIL)**, and within a single process only one thread executes Python bytecode at any given moment. It exists so that operations like reference counting (incrementing/decrementing `ob_refcnt`) don't need locks everywhere, keeping the implementation simple and single-threaded execution fast. The price: pure Python computation can't saturate multiple cores via multithreading.

For someone used to the true parallelism of pthreads / `std::thread`, this is a big pitfall. But it's a CPython implementation choice (PyPy has one too; Jython doesn't), and since Python 3.13 there's an experimental no-GIL build. The full mechanism and workarounds (multiprocessing, C extensions releasing the GIL, `asyncio`) are covered specifically in [Chapter 10](10-并发模型.md); for now just remember: **the GIL is an interpreter-level lock, not a language semantic.**

## 7. Summary

- CPython compiles `.py` into **bytecode** (cached as `.pyc`), then a C-written **stack-machine eval loop** interprets it one instruction at a time—an entire extra layer of software translation, which is the root of the slowness.
- Use `ast.parse` to view the syntax tree and `dis.dis` to view the bytecode; you can verify what happens under the hood at any time.
- **Everything is an object, and in CPython it all lives on the heap**; every object carries `ob_refcnt` (reference count) and `ob_type` (like a vtable pointer). A variable is a reference to a heap object, never an inlined value—fundamentally different from C's stack value types.
- Distinguish Python (semantics) from CPython (implementation); PyPy uses a JIT for speed; the GIL is CPython's interpreter-level lock, with details in Chapter 10.
