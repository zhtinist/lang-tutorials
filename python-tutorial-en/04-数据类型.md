# Data Types

This chapter introduces Python's common built-in data types. **Each type is first covered from "how do you write it, how do you use it"—so you can get productive right away without understanding anything under the hood; then it moves into "under the hood, versus C"—what the type actually looks like inside CPython, and how it differs from the fixed-width C `int`, the IEEE-754 `double`, and `char*` you already know.** If you just want to use the type, reading the first half of each section is enough; if you want to understand performance and memory, keep reading into the second half.

The sentence running through this chapter: **in C, the type determines how many bytes a variable occupies; in Python, the type is a struct living on the object heap, and the variable is merely a pointer to it.**

---

## 4.1 Getting Started: a Quick Tour of Built-in Types and Literals

Before writing Python, get familiar first with how to "write out" the various values. None of the literals below need any import:

```python
n = 42              # int   integer, arbitrary size
f = 3.14            # float floating point
b = True            # bool  boolean (True / False)
z = 3 + 4j          # complex complex number (imaginary suffix j)
s = "café😀"        # str   string (Unicode text)
raw = b"hello"      # bytes byte sequence (prefix b)
nothing = None      # empty value
```

Common operations you can try right away:

```python
print(7 // 2)       # 3    —— floor division
print(7 % 2)        # 1    —— modulo
print(2 ** 10)      # 1024 —— exponentiation
print("ab" * 3)     # ababab —— string repetition
print(len("café"))  # 4    —— length (counts characters, not bytes)
print(type(42))     # <class 'int'> —— check the type
```

Use a type name as a function to convert:

```python
int("42")           # 42
float("3.14")       # 3.14
str(255)            # "255"
```

That's enough to start writing code. Below we go through each type in turn—**usage first, internals second**.

## 4.2 Going Deeper: Every Value Is a Heap Object

From this section on we move into "under the hood." To understand the memory overhead of each type coming up, first fix one fact in your mind: **in CPython, every value is an object on the heap**, and every one of them starts with a uniform "object header."

```c
// CPython source Include/object.h (simplified)
typedef struct _object {
    Py_ssize_t    ob_refcnt;    // reference count
    PyTypeObject *ob_type;      // pointer to the type object (equivalent to C++'s vptr)
} PyObject;

typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;         // "number of elements" for variable-length objects
} PyVarObject;
```

`ob_type` is the analog of C++'s virtual table pointer (vptr): it points to a `PyTypeObject` that stores "how this type does addition, how it prints, how it takes its length," and a whole pile of function pointers (called tp_slots). The difference is that C++'s vtable is fixed at compile time, whereas Python's type object is an ordinary object that's modifiable at runtime.

On 64-bit CPython this header is 16 bytes (`PyObject`) or 24 bytes (`PyVarObject`). **Any Python object is at least this big**—which is why Python's `int` easily runs to twenty or thirty bytes while C's `int` is only 4 bytes.

```python
import sys
print(sys.getsizeof(0))      # 28 (64-bit CPython 3.12+; earlier versions 24)
print(sys.getsizeof(1.0))    # 24
```

Keep this "object header" in mind—every type's "under the hood" subsection below will refer back to it.

## 4.3 `int`: Arbitrary-Precision Integers

### How to Use It

Python integers **have no upper size limit**—they can be as big as you like, and won't overflow the way C's do:

```python
x = 2 ** 100
print(x)            # 1267650600228229401496703205376, no overflow
print(x * x)        # keeps growing, still no problem

print(10 // 3)      # 3   —— floor division
print(10 % 3)       # 1   —— modulo
print(-7 // 2)      # -4  —— note: rounds toward negative infinity (not toward zero), unlike C!
print(int("ff", 16))# 255 —— parsed as base 16
```

> ⚠️ A trap that differs from C: Python's `//` and `%` **round toward negative infinity** (floor), so `-7 // 2 == -4`; C's integer division **truncates toward zero**, so `-7 / 2 == -3`.

### Under the Hood, Versus C

Python's `int` is **arbitrary precision**: it stores an integer as an array of "digits," expanding the number of digits on demand like doing long multiplication by hand in grade school—essentially the bignum you'd hand-write in C (what libraries like GMP do), just built in.

```c
// simplified conceptual structure
typedef struct {
    PyObject_VAR_HEAD          // ob_refcnt, ob_type, ob_size
    digit ob_digit[1];         // variable-length array, each digit holds 30 bits
} PyLongObject;
```

Key points:

- Each `digit` is a 32-bit unsigned integer (`uint32_t`), but **only 30 of its bits are used** (`PYLONG_BITS_IN_DIGIT == 30`). The spare bits leave room for carry/borrow arithmetic without overflowing C's 32 bits.
- The **absolute value** of `ob_size` is the digit count, and its **sign** indicates whether the integer is positive or negative; `ob_size == 0` represents the value 0. This is a "sign-magnitude" representation, not two's complement.

> Note: since CPython 3.12 the internals pack `ob_size` plus a flag bit into an `lv_tag` (an implementation detail), but the "30-bit digit array + sign" model is unchanged.

Memory layout sketch for `2**60` (needs 3 30-bit digits):

```
value = digit[0] * 2^0 + digit[1] * 2^30 + digit[2] * 2^60

+-------------------+
| ob_refcnt         |  \
| ob_type  --> int  |   > object header (PyVarObject)
| ob_size = 3       |  /   (positive sign + 3 digits)
+-------------------+
| ob_digit[0] (30b) |
| ob_digit[1] (30b) |
| ob_digit[2] (30b) |
+-------------------+
```

Compared with C:

| | C `int32_t` | Python `int` |
|---|---|---|
| Size | fixed 4 bytes | variable, grows with the value |
| Overflow | wraparound (UB/implementation-defined) | none, grows automatically |
| Negative representation | two's complement | sign bit + magnitude (digit array) |
| Location | can be on the stack/in a register | always on the heap; the variable holds a pointer |
| Operation | a single CPU instruction | a function call, processed digit by digit |

Observe the growth with `sys.getsizeof`:

```python
import sys
print(sys.getsizeof(0))          # 28 (empty-digit case, depends on version)
print(sys.getsizeof(2**29))      # 28  —— 1 digit
print(sys.getsizeof(2**30))      # 32  —— needs 2 digits
print(sys.getsizeof(2**60))      # 36  —— 3 digits
```

Each extra 30-bit digit adds roughly 4 bytes, matching the structure above exactly.

**Small-integer cache**: at startup CPython pre-creates every integer object in the range `[-5, 256]` and keeps them permanently resident. So:

```python
a = 100
b = 100
print(a is b)                       # True —— the same cached object
print(int("1000") is int("1000"))   # False: each is created separately at runtime, outside the cache range
```

**Never use `is` to compare numeric values**; use `==`.

## 4.4 `bool`: `True` / `False`

### How to Use It

```python
flag = True
print(not flag)         # False
print(True and False)   # False
print(3 > 2)            # True —— comparison operators return bool
```

### Under the Hood, Versus C

In Python, `bool` **inherits from `int`**; `True` and `False` are simply two singleton integer objects with values 1 and 0:

```python
print(isinstance(True, int))   # True
print(True + True)             # 2 —— treated as an integer in arithmetic
print(True == 1)               # True
```

This is entirely unlike C's `bool` (`_Bool`, usually a 1-byte value on the stack): Python's bool is a heap object that inherits from int. One practical consequence—you can sum booleans directly to count things:

```python
data = [1, -2, 3, -4, 5]
positives = sum(x > 0 for x in data)   # treats True/False as 1/0 and sums to 3
```

## 4.5 `float`: Double-Precision Floating Point

### How to Use It

```python
f = 3.14
print(f + 1)            # 4.140000000000001
print(2.0 ** 0.5)       # 1.4142135623730951 —— square root
print(round(3.14159, 2))# 3.14

print(0.1 + 0.2)        # 0.30000000000000004 —— floating-point error!
print(0.1 + 0.2 == 0.3) # False —— never compare floats with ==
```

For exact decimals (finance) use `decimal.Decimal`; for exact fractions use `fractions.Fraction`.

### Under the Hood, Versus C

Python's `float` is simply an IEEE-754 double-precision float, **boxed** inside an object:

```c
typedef struct {
    PyObject_HEAD
    double ob_fval;      // a genuine C double
} PyFloatObject;
```

So a float object = 16-byte object header + 8-byte double = 24 bytes:

```python
import sys
print(sys.getsizeof(3.14))   # 24
```

C's `double x = 3.14;` is a raw 8-byte value on the stack, and arithmetic goes straight to FPU instructions. In Python each float is a heap object, and `a + b` has to dereference two pointers to fetch `ob_fval`, add them, and then **allocate a new float object** to hold the result. This is exactly the root of pure Python being slow and memory-hungry under number-crunching workloads, and the reason NumPy exists—NumPy packs a hundred million doubles into one contiguous C array rather than a hundred million 24-byte objects ([see "Common Third-Party Libraries"](12-常用第三方库.md)).

All of floating point's "gotchas" are exactly the same as in C, because underneath it's the very same double: the `0.1 + 0.2 != 0.3` above is proof enough. The special values line up too: `float('inf')`, `float('nan')`, and `nan != nan` holds.

## 4.6 `complex`: Complex Numbers

### How to Use It

The imaginary-part literal suffix is `j`; it's a built-in type, no import needed:

```python
z = 3 + 4j
print(z.real, z.imag)   # 3.0 4.0
print(abs(z))           # 5.0 —— magnitude
```

For trigonometric/exponential operations on complex numbers, use the standard library `cmath` (not `math`):

```python
import cmath
z = 1 + 1j
print(cmath.phase(z))   # 0.7853981633974483 —— argument (π/4)
print((-1 + 0j) ** 0.5) # (6.12e-17+1j) —— √-1 ≈ i (the real part is floating-point noise)
```

### Under the Hood, Versus C

`complex` is simply an object holding two C doubles (the real and imaginary parts):

```c
typedef struct {
    PyObject_HEAD
    double real;
    double imag;
} PyComplexObject;
```

Same story as `float`, just with one extra double. Note that `(-1) ** 0.5` on a `float` returns `nan`, but on a `complex` it takes the complex square root and comes out to approximately `i`.

## 4.7 `str`: Unicode Text

### How to Use It

Strings are immutable Unicode text: indexable and sliceable, but never modifiable in place:

```python
s = "café😀"
print(s[0])             # c   —— index a single character
print(s[1:4])           # afé —— slicing
print(s.upper())        # CAFÉ😀
print("a" in s)         # False
print(len(s))           # 5   —— counts "characters (code points)", not bytes!

# s[0] = "C"            # TypeError: strings are immutable
s2 = s.replace("c", "C")# any "modification" produces a new string
```

When talking to the outside world (files, the network), `str` (text) and `bytes` (raw bytes) must be explicitly encoded/decoded:

```python
s = "café"
data = s.encode("utf-8")     # str -> bytes: b'caf\xc3\xa9'
back = data.decode("utf-8")  # bytes -> str: café
```

### Under the Hood, Versus C

C's `char*` is a run of bytes terminated by `\0`; Python's `str` is an **immutable sequence of Unicode code points**, and CPython picks the most economical internal encoding based on the content.

**PEP 393**: since Python 3.3, CPython looks at the **largest code point** in the string and picks one of three compact representations:

| Internal kind | Bytes per char | Representable range | Example |
|---|---|---|---|
| Latin-1 | 1 | U+0000–U+00FF | `"hello"`, `"café"` |
| UCS-2 | 2 | U+0000–U+FFFF | `"你好"` |
| UCS-4 | 4 | U+0000–U+10FFFF | `"😀"` |

As soon as a single high-code-point character shows up in the string, **the entire string** is promoted to the wider representation:

```python
import sys
print(sys.getsizeof("a"))       # 42 —— Latin-1, 1 byte per char
print(sys.getsizeof("你"))      # 60 —— UCS-2, 2 bytes per char
print(sys.getsizeof("😀"))      # 64 —— UCS-4, 4 bytes per char
```

Compared with C:

| | C `char*` | Python `str` |
|---|---|---|
| Unit | byte | Unicode code point |
| Length | via `\0` terminator, O(n) to compute | stored explicitly, O(1) |
| `len("😀")` | by byte count (4, UTF-8) | 1 (one code point) |
| Mutability | mutable | immutable |

`len()` counts **code points**, not bytes—a trap that regularly catches people used to C:

```python
s = "café😀"
print(len(s))            # 5 code points
print(len(s.encode()))   # 9 UTF-8 bytes: c/a/f are 1 byte each + é (2) + 😀 (4)
```

> **Don't conflate these two things**: what `.encode()` produces is a **variable-length UTF-8 encoding** (each character takes 1–4 bytes depending on its code point), which has **nothing to do** with how the string is stored in memory; what PEP 393's "the whole string is promoted to 4 bytes/char" refers to is the `str` **object's fixed-width in-memory buffer**. In memory it's stored **fixed-width** (so `s[i]` indexing is O(1)); `.encode()` serializes it to UTF-8 as **variable-width** (compact for storage/transport).

Because it's immutable, a string can cache its hash value, making it an efficient dict key. CPython also **interns** short string literals that look like identifiers, letting `is` compare quickly and saving memory—but comparing content should still always use `==`.

## 4.8 `bytes` / `bytearray`: the Real Counterpart to C's char Array

### How to Use It

If you want C's "a run of raw bytes," the counterparts are `bytes` (immutable) and `bytearray` (mutable), not `str`:

```python
b = b"hello"          # bytes literal, prefix b
print(b[0])           # 104 —— what you get out is an int (a byte value), not a character

ba = bytearray(b"hi") # a mutable version
ba.append(33)         # append '!' (0x21)
ba[0] = 72            # modify in place -> 'H'
print(ba)             # bytearray(b'Hi!')
```

When parsing a binary protocol, you often need to assemble bytes into an integer with a given endianness—in C you'd shift and `|`; in Python just use `int.from_bytes` / `int.to_bytes` directly:

```python
data = b'\x00\x00\x01\x00'
print(int.from_bytes(data, "big"))     # 256   —— big-endian
print(int.from_bytes(data, "little"))  # 65536 —— little-endian
print((1000).to_bytes(4, "big").hex()) # 000003e8
```

### Under the Hood, Versus C

- `bytes`: an immutable byte sequence; internally it's just a contiguous `unsigned char` buffer plus a length. Closest to C's `const char*` plus a length.
- `bytearray`: the mutable version; internally it's like C++'s `std::vector<uint8_t>`—it has capacity, and supports in-place insertion/deletion.

If you want C's "point a pointer into a buffer and read/write with zero copies," the counterpart is `memoryview`: it doesn't copy the underlying bytes, it's just a view opened onto an existing buffer, and slicing it is zero-copy too (analogous to C++'s `std::span` / `string_view`):

```python
ba = bytearray(b"hello world")
mv = memoryview(ba)          # zero-copy view
chunk = mv[6:11]             # a slice is still a view, pointing into the original buffer (no new allocation)
chunk[0] = ord('W')          # modify in place through the view
print(ba)                    # bytearray(b'hello World') —— the original object changed
```

When processing large binary blobs (network packets, images), this avoids the "slice once, copy once" cost of `data[a:b]`. In C, `char*` serves as both text and raw bytes, with encoding left entirely to convention; Python forces you to encode/decode explicitly at the boundary, with `str` handling text and `bytes` handling binary.

## 4.9 Singletons: `None`, `NotImplemented`, `Ellipsis`

These three are singleton objects, unique for the whole program's lifetime, and simple enough to use on their own.

`None` represents "no value / empty," analogous to C's `NULL`, but it's a genuine object (type `NoneType`), not a 0 pointer:

```python
x = None
print(x is None)        # the standard idiom for testing emptiness, always use is
```

`None` has exactly one instance for the entire program; `is None` is both faster and immune to interference from a custom `__eq__`. A function with no `return` statement returns `None` by default.

`NotImplemented` is used in operator overloading: when a type doesn't know how to operate with the other operand, it returns this (rather than raising an exception), letting Python try the other side's reflected method:

```python
class Vec:
    def __init__(self, x): self.x = x
    def __add__(self, other):
        if not isinstance(other, Vec):
            return NotImplemented    # hand off to the other operand
        return Vec(self.x + other.x)
```

The `Ellipsis` (`...`) literal is just three dots; it's often used as a placeholder, or by NumPy for multidimensional slicing (`arr[..., 0]`):

```python
def todo(): ...          # a legal function-body placeholder (equivalent to pass)
print(... is Ellipsis)   # True
```

## 4.10 Numeric and String Conversion

For explicit construction/conversion, use the type name as a function (it's actually calling the type object):

```python
int("42")           # 42        —— str -> int
int("ff", 16)       # 255       —— specifying a base
int(3.9)            # 3         —— float truncation (toward zero)
float("3.14")       # 3.14
str(255)            # "255"
bin(255)            # "0b11111111"
hex(255)            # "0xff"
```

Formatting (f-strings, 3.6+) is the most common way to convert numbers to strings:

```python
n = 255
print(f"{n:d} {n:b} {n:x} {n:08b}")   # 255 11111111 ff 11111111
pi = 3.14159
print(f"{pi:.2f}")                    # 3.14
print(f"{1234567:,}")                 # 1,234,567 —— thousands separator
```

A failed conversion raises `ValueError`, rather than silently returning 0 the way C's `atoi` does:

```python
try:
    int("abc")
except ValueError as e:
    print("conversion failed:", e)
```

This "raise an exception on error" orientation is a major stylistic difference from C's approach of returning error codes; see [see "Error Handling and Resource Management"](09-错误处理与资源管理.md) for more.

## Summary

- **At the usage level**: literals are written directly (`42` / `3.14` / `"text"` / `b"bytes"`); `//` is floor division and `**` is exponentiation; `len` counts code points; strings are immutable; conversion is done by calling a type name, and failure raises `ValueError`.
- **At the low level**: every value is a heap object sharing the `PyObject` header (a refcount plus a type pointer, roughly a vptr).
  - `int` is a variable-length bignum (a 30-bit digit array plus a sign), so it never overflows; `[-5,256]` has cached singletons.
  - `bool` is a subclass of `int`; `float`/`complex` are boxed C doubles (one/two of them).
  - `str` is immutable Unicode, stored compactly in 1/2/4 bytes chosen by the largest code point (PEP 393); `bytes`/`bytearray` are the real counterparts to C's byte array, and `memoryview` is the zero-copy view.

The next chapter moves on to control flow, where you'll see that Python's `for` is nothing like C's index-based loop at all, but is instead built on the iterator protocol.
