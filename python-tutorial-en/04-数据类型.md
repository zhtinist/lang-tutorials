# Data Types

In C, the type determines how many bytes a variable occupies and how those bytes are interpreted; once you declare `int x`, `x` is fixed at compile time as 4 bytes on the stack, read and written as two's complement. In Python, type information doesn't attach to the variable—it attaches to the object itself on the heap; a variable is always just a pointer to some object (see [Variables and the Memory Model](03-变量与内存模型.md)). This chapter goes through Python's built-in types one by one, looking at their real structure inside CPython, and exactly what problem each divergence from C solves, and what it gives up to solve it.

## 1. Built-in Types and Literals

```python
n = 42              # int   integer, arbitrary size
f = 3.14            # float floating point
b = True            # bool  boolean (True / False)
z = 3 + 4j          # complex complex number (imaginary suffix j)
s = "café😀"        # str   string (Unicode text)
raw = b"hello"      # bytes byte sequence (prefix b)
nothing = None      # empty value
```

```python
print(7 // 2)       # 3    —— floor division
print(7 % 2)        # 1    —— modulo
print(2 ** 10)      # 1024 —— exponentiation
print("ab" * 3)     # ababab —— string repetition
print(len("café"))  # 4    —— length (counts characters, not bytes)
print(type(42))     # <class 'int'> —— check the type
```

A type's name can itself be called as a function to perform a conversion, because a type name is a type object, and calling it is equivalent to constructing an instance of that type:

```python
int("42")           # 42
float("3.14")       # 3.14
str(255)            # "255"
```

## 2. The Object Header: the Structure Every Value Shares

In CPython, every value is an object on the heap, beginning with a uniform object header:

```c
// CPython source Include/object.h (simplified)
typedef struct _object {
    Py_ssize_t    ob_refcnt;    // reference count
    PyTypeObject *ob_type;      // pointer to the type object (equivalent to C++'s vptr)
} PyObject;

typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;         // "number of elements" for variable-length objects
} PyVarObject;
```

`ob_type` is the analog of C++'s vtable pointer (vptr): it points to a `PyTypeObject` that stores a whole batch of function pointers (called tp_slots) recording "how this type does addition, how it prints, how it takes its length." The difference is that C++'s vtable is baked into the object file at compile time, while Python's type object is an ordinary object that can be read—and even modified—at runtime. This is exactly the concrete realization of the requirement from [The Execution Model](01-运行模型.md) that "any value must be able to answer what type it is at runtime": `ob_type` is the carrier of that self-descriptive information.

On 64-bit CPython, this header alone is 16 bytes (`PyObject`) or 24 bytes (`PyVarObject`). This is a fixed cost every Python object has to carry, regardless of the actual size of its data:

```python
import sys
print(sys.getsizeof(0))      # 28 (64-bit CPython 3.12+; before 3.12 it's 24)
print(sys.getsizeof(1.0))    # 24
```

Every type's memory layout below is built on top of this object header.

## 3. `int`: Arbitrary-Precision Integers, Not a Fixed-Width Word

Python integers have no upper size limit:

```python
x = 2 ** 100
print(x)            # 1267650600228229401496703205376, no overflow
print(x * x)         # keeps growing, still no problem

print(10 // 3)      # 3   —— floor division
print(10 % 3)       # 1   —— modulo
print(-7 // 2)      # -4  —— rounds toward negative infinity (floor), not toward zero, unlike C
print(int("ff", 16))# 255 —— parsed as base 16
```

C's integer division truncates toward zero (`-7 / 2` is conceptually `-3`); Python's `//` floors. This isn't an arbitrary choice: flooring lets `//` and `%` satisfy one unified identity, `a == (a // b) * b + a % b`, with the remainder `a % b`'s sign always matching the divisor `b` (convenient for things like modular indexing, e.g. wrapping a negative offset around to the other end of an array). Truncating toward zero would instead make the remainder's sign follow the dividend—the identity still holds, but the sign rule is less intuitive, and different language implementations can even disagree with each other.

The way Python manages to make integers never overflow is by implementing `int` as an arbitrary-precision bignum—essentially what a C programmer would hand-write a big-number library (in the GMP family) to do, just built into the language:

```c
// simplified conceptual structure
typedef struct {
    PyObject_VAR_HEAD          // ob_refcnt, ob_type, ob_size
    digit ob_digit[1];         // variable-length array, each digit holds 30 bits
} PyLongObject;
```

- Each `digit` is a 32-bit unsigned integer (`uint32_t`), but only 30 of its bits are used (`PYLONG_BITS_IN_DIGIT == 30`). The 2 spare bits exist so that intermediate carry/borrow results don't overflow a C-level 32-bit register, so a chain of additions doesn't need extra overflow checks along the way.
- The absolute value of `ob_size` is the digit count, and its sign indicates whether the integer is positive or negative; `ob_size == 0` represents the value 0. This is a "sign-magnitude" representation, not two's complement—because an arbitrary-precision number has no fixed bit width, and two's complement's "invert and add one" trick only makes sense relative to a fixed width; storing the sign separately is the natural choice instead.

> Note: since CPython 3.12 the internals pack `ob_size` together with a flag bit into an `lv_tag` (an implementation detail), but the "30-bit digit array + sign" model is unchanged.

Memory layout sketch for `2**60` (needs 3 30-bit digits):

```
value = digit[0] * 2^0 + digit[1] * 2^30 + digit[2] * 2^60

+-------------------+
| ob_refcnt         |  \
| ob_type  --> int  |   > object header (PyVarObject)
| ob_size = 3       |  /   (positive sign + 3 digits)
+-------------------+
| ob_digit[0] (30b) |
| ob_digit[1] (30b) |
| ob_digit[2] (30b) |
+-------------------+
```

**Why choose a variable-length bignum instead of a fixed width like C?** If you stick with a fixed width (say, always 64 bits), you either stay silent on overflow like C does (the result wraps around; the C standard even makes signed overflow undefined behavior), or raise an error on overflow like some languages do—either way, the programmer has to introduce a separate big-number library and manually switch types between plain integers and bignums anywhere a value might grow large (factorials, combinatorics, cryptography, arbitrary-precision accumulation). Python makes "numbers can be arbitrarily large" the single, uniform semantics of `int`, eliminating this entire category of "might this overflow here" mental burden and its attendant bugs.

The cost is equally real: C's `int a + b` knows at compile time that both operands are fixed-width, register-sized integers, and compiles to a single hardware `ADD` instruction. Python's `a + b` can't assume at compile time that the operands have only one digit—it must call a general multi-precision addition algorithm: adding digit by digit through the digit arrays from low to high, handling carries, and the result might need one more digit than either operand, requiring a freshly allocated `PyLongObject` of the right size. Even when `a` and `b` are as small as `1 + 1`—a single digit each—the same general path is taken, just with the loop running only once. Since 3.11, CPython's specializing adaptive interpreter (PEP 659) can cache the fact that "last time this addition instruction saw two int operands," saving the repeated type-dispatch overhead, but it can't save the multi-precision addition that genuinely has to execute this time—dispatch overhead and algorithmic overhead are two different things; the former can be optimized away with caching, the latter is a fixed cost dictated by bignum semantics itself. This is one of the fundamental reasons Python is more than an order of magnitude slower than C for numerically intensive code, and it's the reason NumPy exists (packing numbers into fixed-width C arrays, sidestepping the per-element bignum dispatch)—see [Common Third-Party Libraries](12-常用第三方库.md).

Watch this variable-length structure grow directly with `sys.getsizeof`:

```python
import sys
print(sys.getsizeof(0))          # 28 (the empty-digit case, version-dependent)
print(sys.getsizeof(2**29))      # 28  —— 1 digit
print(sys.getsizeof(2**30))      # 32  —— needs 2 digits
print(sys.getsizeof(2**60))      # 36  —— 3 digits
```

Each extra 30-bit digit adds roughly 4 bytes, exactly matching the structure definition.

Compared with C:

| | C `int32_t` | Python `int` |
|---|---|---|
| Size | fixed 4 bytes | variable, grows with the value |
| Overflow | wraparound (UB/implementation-defined) | none, grows automatically |
| Negative representation | two's complement | sign + magnitude (digit array) |
| Location | can be on the stack/in a register | always on the heap; the variable holds a pointer |
| Operation | a single CPU instruction | a function call, processed digit by digit |

**Small-integer cache**: at startup, CPython pre-creates every integer object in the range `[-5, 256]` and keeps them permanently resident; identical values in this range are always the same object:

```python
a = 100
b = 100
print(a is b)                       # True —— the same cached object
print(int("1000") is int("1000"))   # False: each built fresh at runtime, outside the cache range
```

This cache exists purely to cut down the cost of repeatedly allocating and freeing objects for high-frequency small integers (loop counters, indices, the machinery behind booleans); it's an implementation detail, not language semantics—**never compare numeric values with `is`**, use `==`.

## 4. `bool`: a Subclass of `int`, Not an Independent Type

```python
flag = True
print(not flag)         # False
print(True and False)   # False
print(3 > 2)            # True —— comparison operators return bool
```

In Python, `bool` **inherits from `int`**; `True` and `False` are two singleton integer objects with values 1 and 0:

```python
print(isinstance(True, int))   # True
print(True + True)             # 2 —— treated as an integer in arithmetic
print(True == 1)                # True
```

**Why is `bool` a subclass of `int`, rather than an independent type unrelated to integers?** This is the joint result of historical baggage and engineering pragmatism. Python didn't gain a real `bool` type until 2.3 (2003, PEP 285); before that, Python (along with nearly every early tutorial and published codebase) had always represented truth values with plain `0`/`1`—a huge amount of existing code wrote things like `if x == 1`, `flag = 0`, and summed booleans as integer counts. If `bool` had been designed as a type completely unrelated to `int`, `True == 1` would have to be `False`, and all the old code relying on the established fact that "true/false just are 0/1" would have broken en masse. Making `bool` a subclass of `int` means the newly introduced `True`/`False` can seamlessly substitute for the old `1`/`0` anywhere an integer is expected—backward compatibility comes at zero cost. At the same time, subclassing directly reuses all of `int`'s already-written arithmetic, comparison, and hashing logic, without needing to implement a separate "boolean arithmetic table" for `bool`—`True + True == 2` and summing a list of booleans to count things both fall directly out of reusing `int`'s arithmetic, rather than being deliberately designed "features."

This also brings a consequence that's easy to overlook: because `bool` reuses `int`'s equality and hashing, `1` and `True` are treated as the same key in a `dict`/`set` context:

```python
d = {1: "one", True: "yes"}
print(d)                 # {1: 'yes'} —— True and 1 hash and compare equal, so they're the same key; the later write overwrites the earlier one
print(hash(True), hash(1))  # 1 1
```

This is entirely unlike C's `bool` (`_Bool`, typically a 1-byte stack value, a type completely unrelated to `int`): Python's `bool` is a heap object inheriting from `int`, and `sys.getsizeof(True)` is 28 bytes, the same as a small integer, rather than C's compact 1-byte flag.

## 5. `float`: a Boxed Double-Precision Float

```python
f = 3.14
print(f + 1)            # 4.140000000000001
print(2.0 ** 0.5)       # 1.4142135623730951 —— square root
print(round(3.14159, 2))# 3.14

print(0.1 + 0.2)        # 0.30000000000000004 —— floating-point error!
print(0.1 + 0.2 == 0.3) # False —— never compare floats with ==
```

For exact decimals (finance), use `decimal.Decimal`; for exact fractions, use `fractions.Fraction`.

Python's `float` is simply an IEEE-754 double-precision float, boxed inside an object:

```c
typedef struct {
    PyObject_HEAD
    double ob_fval;      // a genuine C double
} PyFloatObject;
```

A float object = 16-byte object header + 8-byte double = 24 bytes:

```python
import sys
print(sys.getsizeof(3.14))   # 24
```

C's `double x = 3.14;` is a raw 8-byte value on the stack, with arithmetic going straight to FPU instructions. In Python, every float is a heap object; `a + b` has to dereference two pointers to fetch their respective `ob_fval` fields, add them, then allocate a new `PyFloatObject` to hold the result—this is much the same situation as `int`, both being direct costs of the unified "everything is an object" model, and doesn't need a separate explanation for `float`. This is yet another root cause of pure Python being slow and memory-hungry under numerically intensive computation, and it's the reason NumPy exists—packing a hundred million doubles into one contiguous C array rather than a hundred million 24-byte objects (see [Common Third-Party Libraries](12-常用第三方库.md)).

All of floating point's pitfalls are exactly the same as in C, because underneath it's the very same IEEE-754 double: `0.1 + 0.2 != 0.3` is proof enough, and the special values line up too: `float('inf')`, `float('nan')`, with `nan != nan`.

## 6. `complex`: Complex Numbers

The imaginary-part literal suffix is `j`; it's a built-in type, no import needed:

```python
z = 3 + 4j
print(z.real, z.imag)   # 3.0 4.0
print(abs(z))            # 5.0 —— magnitude
```

For trigonometric/exponential operations on complex numbers, use the standard library `cmath` (not `math`):

```python
import cmath
z = 1 + 1j
print(cmath.phase(z))    # 0.7853981633974483 —— argument (π/4)
```

Internally, `complex` is simply an object holding two C `double`s (real and imaginary parts):

```c
typedef struct {
    PyObject_HEAD
    double real;
    double imag;
} PyComplexObject;
```

Same story as `float`, just with one more double stored. There's one behavior here that's frequently misremembered: `(-1) ** 0.5` does **not** error out the way `math.sqrt(-1)` does, and it doesn't come out to `nan` either—it **directly returns a `complex`**:

```python
print((-1) ** 0.5)          # (6.123233995736766e-17+1j), approximately i
print(type((-1) ** 0.5))    # <class 'complex'>

import math
try:
    math.sqrt(-1)
except ValueError as e:
    print("math.sqrt error:", e)   # math domain error
```

**Why does `**` specially handle a negative base with a non-integer exponent by returning a complex number, while the equivalent functions in the `math` module simply raise an error?** The `**` operator is implemented by `float.__pow__`, which explicitly checks for the case "the base is negative and the exponent is not an integer"—this kind of input genuinely has no solution within the reals (a negative number has no real square root), but it does have a well-defined principal value within the complex numbers. So `float.__pow__` proactively hands the computation off to complex-number arithmetic, returning a result that "looks strange but is mathematically correct," rather than papering over a problem that is actually solvable with `nan`. `math.pow`/`math.sqrt` do no such conversion at all, because the `math` module is designed from the ground up as **a thin wrapper over the C standard library `libm`**, whose contract is "operate only within the reals"—input outside the real domain should raise `ValueError`, and a module explicitly scoped to real numbers shouldn't quietly produce a complex-valued return. The two behaviors are two applications of the same underlying principle: `**` is a general-purpose operator that must cover every mathematically meaningful result; `math`, as a real-number-oriented library aiming to stay aligned with C semantics, would rather raise an error than let the result type jump unexpectedly. To genuinely work in the complex domain, use `cmath` explicitly.

## 7. `str`: Unicode Text and PEP 393

```python
s = "café😀"
print(s[0])             # c   —— index a single character
print(s[1:4])            # afé —— slicing
print(s.upper())         # CAFÉ😀
print("a" in s)          # False
print(len(s))             # 5   —— counts "characters (code points)," not bytes!

# s[0] = "C"            # TypeError: strings are immutable
s2 = s.replace("c", "C") # any "modification" produces a new string
```

Strings are immutable, for the reasons in section 4 of [Variables and the Memory Model](03-变量与内存模型.md)—immutability buys safe sharing, hashability, and freedom from defensive copies, at the cost that any "modification" must construct a new object.

C's `char*` is a run of bytes terminated by `\0`; Python's `str` is an immutable sequence of Unicode code points, and CPython chooses the most economical internal encoding based on the content. This scheme (**PEP 393**, introduced in Python 3.3) isn't the only possible design—before understanding it, it's worth seeing why two other routes were rejected:

- **Option one: always store every code point internally in 4 bytes (UCS-4/UTF-32)**. The upside is that any code point fits, and indexing by `s[i]` is always O(1) (fixed width, just multiply by 4 to find the offset). The downside is that the vast majority of real text is dominated by ASCII or Latin letters, where 1 byte per character would already suffice—yet for the sake of a rare case (say, a single emoji appearing somewhere in the string), **every character** would have to carry 4 times the memory.
- **Option two: store a variable-length encoding directly (e.g. UTF-8)**. The upside is good space efficiency, with ASCII characters still costing 1 byte. The downside is that under a variable-length encoding, the byte offset of the `i`-th code point in the buffer isn't a simple function of `i`—in UTF-8, different characters occupy anywhere from 1 to 4 bytes, so finding the `i`-th character requires scanning byte by byte from the start, degrading `s[i]` to O(n). This directly conflicts with Python's existing assumption that "string indexing and slicing are cheap operations," an assumption a large amount of existing code depends on.

PEP 393 chose a middle ground between the two: **dynamically pick a fixed-width representation based on the largest code point that actually appears in each string's own content**—any given string is internally always fixed-width (guaranteeing O(1) indexing), but different strings may choose different widths (so a handful of wide-character strings don't drag down the memory cost of every other string):

| Internal kind | Bytes per char | Representable range | Example |
|---|---|---|---|
| Latin-1 | 1 | U+0000–U+00FF | `"hello"`, `"café"` |
| UCS-2 | 2 | U+0000–U+FFFF | `"你好"` |
| UCS-4 | 4 | U+0000–U+10FFFF | `"😀"` |

The cost is explicit: constructing a string requires a scan up front to determine the largest code point (a one-time O(n) cost, usually merged with the scan the construction itself already does, so not really extra); the moment a single wider character appears anywhere in the string, **the entire string** has to be stored in the wider representation, even if only one character actually needs it; and concatenating a Latin-1 string with a UCS-4 string requires re-encoding the result into the wider representation (allocating a new buffer and copying every character over)—it isn't a simple memory concatenation.

```python
import sys
print(sys.getsizeof("a"))       # 42 —— Latin-1, 1 byte per char (3.12+; before 3.12 it was 50)
print(sys.getsizeof("你"))      # 60 —— UCS-2, 2 bytes per char (3.12+; before 3.12 it was 76)
print(sys.getsizeof("😀"))      # 64 —— UCS-4, 4 bytes per char (3.12+; before 3.12 it was 80)
```

The difference before and after 3.12 comes from PEP 623: earlier CPython kept an extra `wstr`/`wstr_length` cache field in every string object for legacy C-extension API compatibility; 3.12 removed this legacy cache purely as historical-baggage cleanup—it has nothing to do with PEP 393's core design (choosing width by largest code point).

Compared with C:

| | C `char*` | Python `str` |
|---|---|---|
| Unit | byte | Unicode code point |
| Length | via `\0` terminator, O(n) to compute | stored explicitly, O(1) |
| `len("😀")` | by byte count (4, UTF-8) | 1 (one code point) |
| Mutability | mutable | immutable |

`len()` counts code points, not bytes—a point that regularly trips up people used to C:

```python
s = "café😀"
print(len(s))            # 5 code points
print(len(s.encode()))   # 9 UTF-8 bytes: c/a/f are 1 byte each + é (2) + 😀 (4)
```

**Don't conflate these two things**: what `.encode()` produces is a variable-length UTF-8 encoding (each character takes 1–4 bytes depending on its code point)—this is the representation used when a string is **serialized out to the outside world**, and has nothing to do with how it's stored in memory. What PEP 393's "the whole string picks a width by its largest code point" refers to is the `str` object's **fixed-width in-memory buffer**, in service of the O(1)-indexing goal. In memory it's stored fixed-width (for cheap `s[i]`); `.encode()` serializes it variable-width (for compact storage/transport)—these are two separate encoding schemes optimizing for two different goals, and both happening to be called "encoding" is what makes them easy to confuse.

Because it's immutable, a string can cache its hash value, making it an efficient dict key. CPython also interns short string literals that look like identifiers, letting `is` compare quickly and saving memory—but comparing content should still always use `==`.

## 8. `bytes` / `bytearray`: Byte Sequences, Strictly Separate from `str`

If you want C's "a run of raw bytes," the counterparts are `bytes` (immutable) and `bytearray` (mutable), not `str`:

```python
b = b"hello"          # bytes literal, prefix b
print(b[0])            # 104 —— what you get out is an int (a byte value), not a character

ba = bytearray(b"hi") # a mutable version
ba.append(33)          # append '!' (0x21)
ba[0] = 72              # modify in place -> 'H'
print(ba)               # bytearray(b'Hi!')
```

When parsing a binary protocol, assembling bytes into an integer by endianness doesn't require hand-writing shifts and bitwise-ORs the way C does:

```python
data = b'\x00\x00\x01\x00'
print(int.from_bytes(data, "big"))     # 256   —— big-endian
print(int.from_bytes(data, "little"))  # 65536 —— little-endian
print((1000).to_bytes(4, "big").hex()) # 000003e8
```

`str` and `bytes`/`bytearray` are two types with no implicit conversion between them; any crossing over must be explicit `encode`/`decode`:

```python
s = "café"
data = s.encode("utf-8")     # str -> bytes: b'caf\xc3\xa9'
back = data.decode("utf-8")  # bytes -> str: café

try:
    b"x" + "y"
except TypeError as e:
    print(e)          # can't concat str to bytes
print(b"x" == "x")    # False —— doesn't error, but different types are simply judged unequal; never implicitly decoded
```

**Why keep text (`str`) and bytes (`bytes`) strictly separate at the type level, with conversion always required to be explicit?** This is a real historical lesson left over from Python 2. In Python 2, `str` itself was a run of bytes (with no fixed encoding), and `unicode` was the actual text type; when a `str` and a `unicode` got mixed in an operation (concatenation, comparison), Python 2 would **implicitly** try to decode the `str` into `unicode` using a default encoding (usually ASCII) and carry on. As long as the bytes flowing through a program happened to all be ASCII, this implicit conversion would succeed silently, and the code would work fine throughout development and testing; the moment a non-ASCII byte slipped in somewhere (a user typed Chinese, a file contained an accented character), that implicit decode would fail, raising a `UnicodeDecodeError`—and this kind of input tends to show up only in production, often much later, leading to an extremely long bug-latency period and a high cost to track down. This was one of the most complained-about pain points of the Python 2 era, and one of the core drivers of the community's migration to Python 3. Python 3's fix seals this gap off entirely at the type-system level: `str` is text (always a sequence of Unicode code points), `bytes` is bytes (carrying no encoding assumption whatsoever), and there is no default encoding between them—any mixed operation either raises `TypeError` outright (like concatenation), or, while not erroring, is judged unequal simply because the types differ (like `==`), and never silently performs an encode/decode on your behalf. The encoding to use (UTF-8? GBK?) must be explicitly specified by the programmer in the `encode`/`decode` call—turning "what encoding is this data actually in," a question that ought to be decided by a human, from an implicit assumption that could be ignored, work silently, and blow up one day out of nowhere, into an explicit, clearly visible contract in the code that must be handled or the code simply won't run.

`bytes` is an immutable byte sequence; internally it's just a contiguous `unsigned char` buffer plus a length, closest to C's `const char*` plus a length; `bytearray` is the mutable version, internally like C++'s `std::vector<uint8_t>`—it has capacity, and supports in-place insertion/deletion.

For C's "point a pointer into a buffer, zero-copy read/write," the counterpart is `memoryview`: it doesn't copy the underlying bytes, it's just a view opened onto an existing buffer, and slicing it is zero-copy too (analogous to C++'s `std::span`/`string_view`):

```python
ba = bytearray(b"hello world")
mv = memoryview(ba)          # zero-copy view
chunk = mv[6:11]              # a slice is still a view, pointing into the original buffer (no new allocation)
chunk[0] = ord('W')            # modify in place through the view
print(ba)                     # bytearray(b'hello World') —— the original object changed
```

When processing large binary blobs (network packets, images), this avoids the "slice once, copy once" cost of `data[a:b]`.

## 9. Singletons: `None`, `NotImplemented`, `Ellipsis`

`None` represents "no value/empty," analogous to C's `NULL`, but it's a genuine object (type `NoneType`), with exactly one instance for the entire program:

```python
x = None
print(x is None)        # the standard idiom for testing emptiness, always use is
```

There are two reasons for `is None` rather than `== None`: first, identity comparison (comparing pointers) is faster than calling `__eq__`; second, a custom class could override `__eq__` to make anything "equal" to `None`, whereas a singleton's identity can never be forged, making `is None` more reliable. A function with no `return` statement returns `None` by default.

`NotImplemented` is used in operator overloading: when a type doesn't know how to operate with the other operand's type, it returns this (rather than raising an exception), letting Python go try the other operand's reflected method (such as `__radd__`):

```python
class Vec:
    def __init__(self, x): self.x = x
    def __add__(self, other):
        if not isinstance(other, Vec):
            return NotImplemented    # hand off to the other operand
        return Vec(self.x + other.x)
```

**Why return a special value instead of just raising an exception?** When two mutually unfamiliar types perform an operation (say, `Vec + Matrix`), `Vec.__add__` has no way of knowing whether `Matrix` is capable of being added to it—if it raised an exception, Python would never get a chance to ask `Matrix.__radd__` whether it knows how to handle this. `NotImplemented` separates "I don't know how to compute this" from "this operation has definitively failed" into two independent judgments: the former is declared by each type on its own, and the latter is only actually raised as `TypeError` by Python once both sides have declared "I don't know." This gives two types that don't recognize each other at all a chance to still negotiate a way to interoperate, without either needing to know about the other in advance.

`Ellipsis` (`...`) is simply three dots, commonly used as a placeholder, or used by NumPy for multidimensional slicing (`arr[..., 0]`):

```python
def todo(): ...          # a legal function-body placeholder (equivalent to pass)
print(... is Ellipsis)   # True
```

## 10. Numeric and String Conversion

Explicit construction/conversion uses a type name as a function (which actually calls that type object's constructor):

```python
int("42")           # 42        —— str -> int
int("ff", 16)        # 255       —— specifying a base
int(3.9)              # 3         —— float truncation (toward zero)
float("3.14")        # 3.14
str(255)              # "255"
bin(255)              # "0b11111111"
hex(255)              # "0xff"
```

Formatting (f-strings, 3.6+) is the most common way to convert a number to a string:

```python
n = 255
print(f"{n:d} {n:b} {n:x} {n:08b}")   # 255 11111111 ff 11111111
pi = 3.14159
print(f"{pi:.2f}")                     # 3.14
print(f"{1234567:,}")                  # 1,234,567 —— thousands separator
```

A failed conversion raises `ValueError`, rather than silently returning 0 the way C's `atoi` does:

```python
try:
    int("abc")
except ValueError as e:
    print("conversion failed:", e)
```

This orientation of "raise an exception on error, refuse to fail silently" runs through the entire type system—division by zero raises `ZeroDivisionError`, a type mismatch raises `TypeError`, a failed conversion raises `ValueError`—covered in detail in [Error Handling and Resource Management](09-错误处理与资源管理.md).

## Summary

- Every value shares the `PyObject`/`PyVarObject` object header (reference count + type pointer), the direct realization, in memory layout, of the requirement that "any value must be able to answer what type it is at runtime."
- `int` is a variable-length bignum (a 30-bit digit array plus a sign), trading "every operation runs through a general multi-precision algorithm" for the semantics of "never overflows"; `[-5, 256]` has cached singletons—only use `==` to test equality.
- `bool` is a subclass of `int`, stemming from the need to stay compatible with existing "0/1 represents true/false" code when `bool` was only introduced in Python 2.3, incidentally reusing all of `int`'s arithmetic and hashing logic, at the cost that `1` and `True` get treated as the same key by dict/set.
- `float`/`complex` are one or two boxed C `double`s; `**` deliberately switches to computing a complex result for a negative base with a non-integer exponent, while the `math` module strictly stays within the reals and raises an error instead—the two serve different contracts.
- `str` is an immutable sequence of Unicode code points; PEP 393 dynamically picks a 1/2/4-byte fixed width based on each string's own largest code point, striking a middle ground—between "always 4 bytes, simple but memory-hungry" and "variable-length encoding, space-efficient but O(n) indexing"—that achieves O(1) indexing together with space efficiency.
- `bytes`/`bytearray` are the real counterparts to C's byte array, strictly separated from `str` with mandatory explicit conversion—a direct correction of the production-time `UnicodeDecodeError`s that Python 2's implicit encoding/decoding used to cause; `memoryview` provides a zero-copy view.
